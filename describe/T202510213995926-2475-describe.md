## 📖 一、总览

该仓库是rcore-rust家族中的一个实验性内核，旨在兼容Linux系统调用接口并支持双架构（riscv64, loongarch64）。其整体定位介于教学型rCore-Tutorial和完备Linux兼容内核之间：在进程管理、文件系统VFS、IPC和信号处理上大幅扩展，但代码质量和模块化仍处于原型阶段，体现了“先让关键功能跑起来”的工程策略。

该仓最值得注意的特点包括：第一，启动流程使用`compare_exchange`原子操作实现单次初始化守卫，并通过`fake_main`地址跳板解决不同架构的虚拟地址偏移，但对loongarch64的支持未完（二级核初始化被注释、内核页表映射缺失）。第二，内存管理模块内建System V共享内存反向查找和复杂区域拆分算法（`split2`/`split_in3`），是同类内核中少有的精细设计，但锁顺序不一致可能引入死锁。第三，调度层将CFS调度器作为可选特性，并组合Token-Ring多核处理器，全局任务管理依赖条件编译选择容器，无默认回退导致编译可能失败。

## 🎯 二、综合评价

### 整体定位
该仓库是rcore-rust家族中一个尝试扩展教学内核边界的实验性OS。与rCore-Tutorial相比，它在IPC（System V shm、futex、命名管道）、信号（SA_SIGINFO、SA_ONSTACK）、调度（CFS可选）和VFS（procfs条目、openat2等）上有显著扩充。但其代码仍大量采用`#[cfg]`条件编译和全局锁，模块间耦合度高，反映出“先实现、后优化”的开发策略。整体上，它更接近一个“Linux兼容性验证原型”，而非生产就绪的内核。

### 真正的创新或独特设计
启动流程是其中最严谨的设计：通过`IS_BOOT`原子变量和`compare_exchange`保证单核初始化（`[boot.ref:1]`），并使用`fake_main`地址跳板绕过KERNEL_BASE偏移（`[boot.ref:3]`）。内存管理的`Page`枚举统一匿名页、文件缓存和内联数据（`[mm.ref:4]`），并内置`addr2shmid`支持SysV shm反向查找（`[mm.ref:1]`）。IPC模块的futex实现包含了bitset、requeue和robust list支持（`[ipc.ref:10]`虽指出死锁风险，但功能完备），共享内存采用延迟删除策略（`[ipc.ref:5]`）。网络模块的TcpSocket使用原子状态机而非锁来管理状态迁移（`[net.ref:2]`），虽然存在饥饿风险，但设计思路独特。

### 取舍判断
开发团队将主要精力投入系统调用兼容性和多架构支持。262项系统调用覆盖了进程、文件、信号、IPC和网络核心分类，并包括`preadv2`、`execveat`等高级调用。为此，他们构建了自定义VFS、完整信号帧处理和Network socket抽象。但代价是代码质量被明显牺牲：超过30处unsafe块缺少安全注释（`[drivers.ref:12]`），全局互斥锁在驱动和网络模块中频繁使用（`[net.ref:4]`），大量`#[cfg]`依赖缺乏文档（`[task.ref:1]`）。驱动模块甚至放弃Rust trait，直接采用C风格的MMIO函数和固定大小数组（`[drivers.ref:1]`–`[drivers.ref:4]`），显然为快速验证硬件而放弃了抽象。

### 完成度与不足
整体完成度属于“核心功能可运行”，但严重缺陷遍布：loongarch64的`rust_main`未实现内核页表映射（`[mm.ref:5]`指出`new_kernel()`为空），驱动DMA描述符未正确分配TX缓冲区（`[drivers.ref:13]`），网络轮询无超时机制（`[net.ref:3]`），命名管道waiter使用LIFO导致饥饿（`[ipc.ref:1]`）。代码质量方面，`dump_system_info`死代码、FUTEX_WAKE_OP直接panic（`[ipc.ref:11]`）、地址翻译路径对用户态指针unwrap（`[net.ref:10]`）。这些表明代码尚未经过充分测试和清理，锁策略和错误处理有待系统重构。

### 1. 核心抽象：单次初始化守卫与双架构入口

启动流程模块的核心抽象是 `IS_BOOT: AtomicBool`（行89）配合 `compare_exchange(true, false, Ordering::SeqCst, Ordering::SeqCst)` 构成的**单次初始化守卫**。这一设计直接解决了多核（SMP）环境下多个 hart 同时进入 `rust_main` 的竞态问题：只有成功将 `IS_BOOT` 从 `true` 交换为 `false` 的 hart（即主引导核）执行完整的初始化路径，其余 hart 走轻量化的 `else` 分支（行141–149、199–207）。

与 rCore-Tutorial 通常使用的 `static mut INIT: bool` 加简单 `if !INIT` 检查相比，`compare_exchange` 方案具备**原子性语义保证**：即使多个 hart 在完全相同的时刻读到 `IS_BOOT == true`，CAS 指令也能确保只有一个成功。这是 SMP 启动的正确性关键，且无需额外的锁或忙等待。

两套 `#[cfg(target_arch = "riscv64")]`（行92）和 `#[cfg(target_arch = "loongarch64")]`（行154）条件编译的 `rust_main` 体现了**架构感知的入口分离**。riscv64 版本接收 `hart_id` 和 `dtb_address` 两个参数；loongarch64 版本仅接收 `hart_id`，DTB 或其他平台初始化通过内部 `bootstrap_init()` 完成。这种分离避免了在通用代码中插入大量架构特异的 `#[cfg]` 分支，但也意味着两个架构的初始化序列需要独立维护——例如 loongarch64 版本调用了 `bootstrap_init()` 和可选的 `pci::init()`，而 riscv64 版本则没有这些步骤。

### 2. 关键设计取舍：fake_main 跳板、SMP 竞态处理与架构差异

#### riscv64 的 fake_main 跳板

`fake_main`（行71–84）是 riscv64 架构独有的设计，被放置在 `.text.main` 段。它的职责是通过手动调整栈指针（`add sp, sp, KERNEL_BASE`）和函数地址（`la t0, rust_main; add t0, t0, KERNEL_BASE`），将执行流从 OpenSBI 等 bootloader 传入的物理地址空间跳转到内核虚拟地址空间。这实质上是一个**地址空间切换跳板**——放弃使用 Rust 的函数调用约定，而是通过内联汇编手动操作寄存器和跳转。这种设计的必要性在于：在进入 `rust_main` 之前，MMU 尚未完全初始化，但内核代码已被链接到虚拟地址 `KERNEL_BASE` 之上，因此必须通过跳板修正 PC 和 SP。

替代方案可以是在汇编入口中直接设置页表并跳转，但 `fake_main` 的方式将这部分逻辑保留在 Rust 中，减少了汇编代码量。代价是引入了额外的 `unsafe { asm!(...) }` 块，且调用约定与普通 Rust 函数不同（需要手动移动参数寄存器）。

#### SMP 次级核路径的简化

非主核的 `else` 分支（行141–149、199–207）显著简化：仅重新初始化 MMU（`mm::kernel_init()`）和 Trap（`trap::init()`），然后直接进入 `other_run_tasks`。这里**跳过了设备初始化、文件系统加载、计时器设置等多个步骤**，依赖主核已完成全局状态初始化。值得注意的是，`other_initproc(hart_id)` 在两架构的非主核路径中均被注释掉（行148、206），这意味着次级核没有独立的进程初始化——它们直接复用主核创建好的进程池。这是一个潜在的风险点：如果主核尚未完成 `boot_initproc`/`add_initproc`，次级核可能读到空的任务列表。

#### 架构差异的取舍

loongarch64 版本在初始化序列中多了 `bootstrap_init()`（行182、201）和条件编译的 `pci::init()`（行184–185）与 `ls7a_rtc_init()`（行187–188）。这些差异反映了两个平台硬件初始化需求的根本不同：riscv64 依赖 DTB 驱动的设备发现（`drivers::init_device(dtb_address)`），而 loongarch64 采用更传统的直接寄存器初始化。两种策略各有优劣——DTB 方式更灵活，但要求 bootloader 正确传递设备树；直接初始化更可控，但代码与特定 SoC 绑定。

### 3. 跨模块协同：子系统初始化顺序的依赖链

启动流程本质上是**一个精心排序的子系统初始化流水线**。以 riscv64 主路径（行113–140）为例：

```
clear_bss → DTB_BASE.lock() → logging::init → mm::init → trap::init →
init_device → sstatus::set_sum → trap::enable_timer_interrupt →
set_next_trigger → loader::list_apps → boot_initproc → run_tasks
```

这个顺序隐含了关键的依赖关系：
- **`clear_bss` 必须最先执行**（行113），因为后续所有全局变量的零初始化依赖 BSS 段已清零。
- **`logging::init` 早于所有可能输出日志的调用**（行115）。
- **`mm::init` 必须在 `trap::init` 之前**（行116–117），因为 Trap 初始化可能需要分配页表或设置 SATP。
- **`trap::init` 在 `sstatus::set_sum` 之前**（行117 vs 行126–128），因为 `set_sum` 修改了 S 态访问 U 态页面的权限位，必须在 Trap 处理程序就绪后执行，以防非法访问被误处理。
- **`loader::list_apps` 在 `boot_initproc` 之前**（行138–139），因为 `boot_initproc` 需要知道有哪些应用可加载。

`global_asm!(include_str!("link_app.S"))`（行57）将编译时生成的应用程序镜像直接链接进内核，这是 rCore 家族的经典做法——`list_apps` 遍历该镜像中的符号表枚举应用，而 `boot_initproc` 则通过 `loader` 模块将选定应用的 ELF 加载到内存。这种设计避免了文件系统依赖，简化了早期启动流程，但牺牲了运行时灵活性（无法动态添加新应用）。

### 4. 边角细节与不足

**<sup>[1](#mod-boot-ref-1)</sup> IS_BOOT 的 Ordering 选择**：`compare_exchange` 使用了 `Ordering::SeqCst` 而非更轻量的 `Acquire/Release`。这是最保守的内存序，保证了所有 hart 看到一致的初始化顺序。但在单核或简单的 SMP 场景下存在性能浪费，且文档未说明为何不降至 `AcqRel`。

**<sup>[2](#mod-boot-ref-2)</sup> 架构条件编译的覆盖风险**：`#[cfg(target_arch = "riscv64")]` 和 `#[cfg(target_arch = "loongarch64")]` 是两个独立的 `rust_main` 定义。如果未来新增架构（如 x86_64、aarch64），必须在 `target_arch` 条件中显式添加，否则编译会因缺少 `rust_main` 而失败。文件头部的注释 `// 目前只支持riscv64`（行40）与代码中实际支持的 loongarch64 已不一致，是工程文档的滞后信号。

**<sup>[3](#mod-boot-ref-3)</sup> fake_main 的安全契约缺失**：`fake_main` 中的 `unsafe { asm!(...) }` 块（行76–83）未附带 `// SAFETY:` 注释。该函数的关键假设——调用前 SP 指向可写内存、`KERNEL_BASE` 与链接脚本一致、`rust_main` 符号在预期地址——均未文档化，构成技术债务。

**<sup>[4](#mod-boot-ref-4)</sup> 注释掉的遗留代码**：行120–123 的 RTC 时间读取、行148 与 206 的 `other_initproc`、行210 的 `shutdown()` 均被注释掉而非删除。这表明启动流程在迭代过程中部分功能被搁置，但未清理。特别是 `shutdown()` 的缺失意味着 loongarch64 版本在 `panic!` 后无法优雅关机。

**<sup>[5](#mod-boot-ref-5)</sup> 错误处理路径单一**：两个架构的 `rust_main` 最终均通过 `panic!("shutdown machine")` 结束（行151、209）。如果 `run_tasks` 返回（理论上不会，因为它是 `!` 类型），内核将立即 panic，没有尝试重启或 fallback。这对于生产环境可能不够 robust。

**<sup>[6](#mod-boot-ref-6)</sup> `dump_system_info` 死代码**：行213–221 定义的 `dump_system_info` 函数在 main.rs 内未被任何代码调用，属于死代码。它依赖 `FRAME_ALLOCATOR` 和 `dentry_cache`，可能用于调试但未被集成到启动流程或控制台命令中，是早期实现残留。

**<sup>[7](#mod-boot-ref-7)</sup> 条件编译特性名的可维护性**：`#[cfg(feature = "smp")]`、`#[cfg(feature = "test")]`、`#[cfg(feature = "virt")]`、`#[cfg(feature = "la2000")]` 散布在启动流程中，但 Cargo.toml 中这些 feature 的定义与依赖关系未在 main.rs 中可见。评审者需要跳转到 Cargo.toml 才能理解 feature 间的互斥或依赖关系，增加了认知负担。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `MemorySet` 为核心抽象，管理每个进程的虚拟地址空间。地址空间由 `BTreeMap<VirtPageNum, MapArea>` 组织，`MapArea` 封装了映射类型（线性/分页/文件/堆/栈）、权限和物理页。`Page` 枚举统一了匿名页、文件缓存页和内联数据页。与 rCore-Tutorial 相比，本模块显著扩展了：System V 共享内存子系统（`ShmManager`）、显式 COW 处理路径、复杂区域拆分算法（`split2`/`split_in3`）、以及惰性分配的预置结构。主要设计取舍包括：riscv64 通过浅拷贝内核一级页表实现用户态内核栈可见、自定义 `MapPermission` 额外位（COW/Shared）、以及全局内核空间 + 独立页表的两阶段内核栈映射。

### 1. 核心抽象：MemorySet 与 MapArea 的层次化地址空间管理

本模块的核心抽象是 `MemorySet` 结构体 <sup>[1](#mod-mm-ref-1)</sup>，它管理一个进程的完整虚拟地址空间。地址空间被分割为若干 `MapArea`，存储在 `BTreeMap<VirtPageNum, MapArea>` 中，以起始虚拟页号（VPN）为键。这种设计相比链式 VMA 列表或区间树，在查找某个 VPN 属于哪个区域时需要 `range()` 查询，复杂度为 O(log n)；但分割/合并时需要重新建树，成本较高。

`MapArea` 的 `MapType` 枚举 <sup>[2](#mod-mm-ref-2)</sup> 定义了 7 种映射语义：`Linear`（线性偏移，用于内核直接映射）、`Framed`（匿名页）、`Filebe`/`FilebeRO`（文件后端）、`Stack`/`Heap`（惰性分配预留）。其中 `FilebeRO` 和 `Stack` 的 `map()` 方法直接 `panic!`，意味着它们只能通过缺页处理来建立映射—这是惰性分配的显式契约 <sup>[3](#mod-mm-ref-3)</sup>。

`Page` 结构体 <sup>[4](#mod-mm-ref-4)</sup> 是本模块的底层物理页抽象，使用 `PageKind` 枚举区分 `Framed`（纯匿名）、`Filebe`（文件缓存，含回写信息）和 `Inline`（ext4 内联数据）。它封装了物理帧的分配（通过 `frame_alloc_ppn`）和释放（通过 `Drop` 实现中的 `frame_dealloc`），并提供 `read`/`modify` 方法自动处理不同 `PageKind` 的锁语义。`Page::new_frameds` 批量分配连续物理帧，使用 `frame_alloc_range_any` 获取任意物理地址的帧 <sup>[5](#mod-mm-ref-5)</sup>。

### 2. 关键设计取舍

**riscv64 与 loongarch64 的架构分支**. `from_global()` 方法 <sup>[6](#mod-mm-ref-6)</sup> 仅在 `#[cfg(target_arch = "riscv64")]` 下编译，它通过 `PageTable::from_global()` 浅拷贝内核一级页表的后 256 项到用户空间，使用户态可通过内核栈的映射正确响应中断。而 loongarch64 版本使用 `new_bare()` 创建完全独立的页表，不共享内核映射。这种差异源于两种架构的页表层级设计不同：riscv64 Sv39 的一级页表最后一项映射了整个内核空间（0xffffffc000000000 以上），浅拷贝代价低；loongarch64 的页表结构不同，无法直接复用。

**内核栈二级页表帧的预分配**. 代码使用 `lazy_static!` 分配了一个 `kstack_second_level_frame` <sup>[7](#mod-mm-ref-7)</sup>，并手动写入内核根页表的第 511 项。注释明确说明：防止该帧被 drop 回收后内核栈映射失效。这是一个精妙但脆弱的 hack：它依赖 riscv64 Sv39 页表的具体结构（第 511 项映射 0xffffffc000000000–0xffffffffffffffff），且与 `PageTable::from_global()` 配合，确保用户空间也能看到内核栈映射。

**自定义 MapPermission 扩展位**. `MapPermission` <sup>[8](#mod-mm-ref-8)</sup> 在标准 RISC-V PTE 权限位（R/W/X/U/G/A/D）之外，定义了 `COW = 1 << 8` 和 `S = 1 << 9`。`COW` 位用于标注写时复制页，在 `pre_handle_cow_and_lazy_alloc` 中检查并处理；`S` 位标识共享映射。这两个扩展位是纯软件概念，不会写入硬件页表，`PTEFlags::from()` 转换时会屏蔽它们。

**惰性分配只完成了骨架**. 代码中有多处 `Todo: 支持areas的lazy allocation` 注释 <sup>[9](#mod-mm-ref-9)</sup> 和 `handle_lazy_allocation_area` 方法，表明惰性分配是计划中的特性但目前仅部分实现。`MapType::Stack` 和 `MapType::Heap` 的 `map()` 方法直接 panic，强制只能通过缺页处理来填充页。`from_elf_lazily` 函数实现了 ELF 加载时的惰性页分配原型，但手动解析 ELF 程序头的方式（读取 64 字节头部后再逐字段解析 <sup>[10](#mod-mm-ref-10)</sup>）相比使用 `xmas_elf` 库解析完整 ELF 更脆弱，且没有处理 `.interp` 段。

### 3. 跨模块协同

**与 Task 模块**. `MemorySet` 存储在 `TaskInner` 中，通过 `task.op_memory_set_mut()` 闭包访问 <sup>[11](#mod-mm-ref-11)</sup>，该闭包会在持有任务锁的情况下操作地址空间。这种设计避免了 `MemorySet` 内部自建锁，但要求所有调用方都通过 task 间接访问—syscall 实现中 `mmap`/`munmap`/`brk` 等均遵循此模式。

**与 Trap 处理**. 页故障处理在 `handle_recoverable_page_fault` <sup>[12](#mod-mm-ref-12)</sup> 中完成。它先尝试查找 PTE，如果是 COW 页则执行复制；否则调用 `handle_lazy_allocation_area`。COW 处理逻辑同时更新 PTE 和 `MapArea.pages` BTreeMap，并在完成后执行 `sfence_vma_vaddr` 刷新 TLB。注意这里的 COW 处理是**同步完成**的，不依赖硬件 dirty 位或异步扫描。

**与 FS 模块**. 文件映射通过 `MapArea.backend_file: Option<Arc<dyn FileOp>>` 关联文件对象。`MapArea::map()` 中 `MapType::Filebe` 调用 `file.get_pages(offset, count)` 批量获取页缓存 <sup>[13](#mod-mm-ref-13)</sup>；`Page::sync_with_address_space` 实现了脏页回写，能处理稀疏文件空洞（通过 `Ext4Extent` 分配新块并插入 extent 树）。

**与 System V 共享内存**. `ShmManager` 全局单例管理所有共享内存段 <sup>[14](#mod-mm-ref-14)</sup>。`MemorySet.addr2shmid` BTreeMap 维护反向映射（shm 地址 → shmid），用于 `shmdt` 时快速查找。`attach_shm_segment` 将 `ShmSegment.pages`（`Vec<Arc<Page>>`）中的页克隆 Arc 并插入进程的页表和 `areas` 中，实现了物理页的真正共享。引用计数为 0 且被 `IPC_RMID` 标记的段才会被销毁 <sup>[15](#mod-mm-ref-15)</sup>。

### 4. 边角细节与不足

**<sup>[1](#mod-mm-ref-1)</sup> 架构条件编译的天窗**. `from_global()` 仅 riscv64 有实现，loongarch64 无等价方法。若在 loongarch64 上调用 `from_elf`，将使用 `new_bare()` 创建无内核映射的页表，处理器无法响应中断（没映射内核栈），这是严重缺陷。同样 `new_kernel()` 的 loongarch64 版本是空实现 <sup>[16](#mod-mm-ref-16)</sup>，`#[cfg(target_arch = "loongarch64")] pub fn new_kernel() -> Self { let memory_set = Self::new_bare(); memory_set }` — 内核自身的地址空间映射完全缺失。

**<sup>[2](#mod-mm-ref-2)</sup> unwrap() 在 OOM 路径上直接 panic**. `frame_alloc().unwrap()` <sup>[17](#mod-mm-ref-17)</sup> 在物理内存耗尽时导致内核 panic，没有优雅降级或 oom_killer 机制。用户可以通过 `mmap` 耗尽内存触发此路径，构成 DoS 入口。

**<sup>[3](#mod-mm-ref-3)</sup> COW 位在跨架构移植中的兼容性**. `COW` 标志在 loongarch64 上也有定义，但 loongarch64 的 PTE 格式可能与 RISC-V 完全不同，`PTEFlags::from()` 转换时需要确认 loongarch64 的 D 位处理是否正确。代码中有 `#[cfg(target_arch = "loongarch64")] flags.insert(PTEFlags::D);` 这样的权宜补丁 <sup>[18](#mod-mm-ref-18)</sup>，表明 COW 处理在 loongarch64 上可能未充分验证。

**<sup>[4](#mod-mm-ref-4)</sup> 区域拆分算法的正确性依赖全序**. `split2` 和 `split_in3` 修改 `self.vpn_range.set_end()` 后，用 `retain` 将 pages 移动到新区域 <sup>[19](#mod-mm-ref-19)</sup>。但 `BTreeMap` 的迭代器在 retain 期间不能保证稳定，且如果 `split_vpn` 恰好等于 `start` 或 `end`，会创建零长度区域。调用方（`remap_area_with_overlap`）没有校验此情况。

**<sup>[5](#mod-mm-ref-5)</sup> 全局 SHM_MANAGER 中使用 `RwLock` + `Mutex` 组合锁，存在潜在死锁**. `attach_shm_segment` 获取 `SHM_MANAGER.segments.write()` 锁后，调用 `task.op_memory_set_mut` 获取任务锁；而 `detach_shm_segment` 反向获取任务锁后再获取 segments 写锁 <sup>[20](#mod-mm-ref-20)</sup>。虽然当前两路径不交织，但锁顺序不一致是死锁隐患。

**<sup>[6](#mod-mm-ref-6)</sup> from_elf_lazily 中硬编码路径**. 处理 `.sh` 和 `.py` 文件时，硬编码了 `/glibc/busybox` 和 `/python/bin/python3.10` 路径 <sup>[21](#mod-mm-ref-21)</sup>，没有用户可配置的 fallback。且通过文件名后缀判断脚本类型（而非读取 shebang），不符合 POSIX 规范。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块以 `Task` 结构体为核心抽象，封装了 Linux 兼容的进程/线程模型（支持 clone 标志位、线程组、pid/tgid/pgid、用户 ID、能力、资源限制等），通过 Token-Ring 风格的多核处理器（`Processor`）和按核隔离的调度器实例实现 SMP 调度。与 ArceOS 基线相比，显著扩展了信号处理、POSIX 定时器、futex、cgroup 等设施，并引入了 CFS 调度器作为可选特性（`#[cfg(feature="cfs")]`），从而支持更复杂的服务器或桌面场景。

### 1. 核心抽象与外部依赖

进程与任务调度的核心抽象是 `Task` 结构体（`os/src/task/task.rs:41-150`）。它使用 `#[repr(C)]` 布局，保证 `kstack` 和 `cpu_id` 在结构体头部，以适应汇编级别 `__switch` 对栈指针和核心 ID 的硬编码访问。`Task` 聚合了大量原子和锁保护的字段：`tid`/`tgid`/`pgid` 等标识，`status`（`TaskStatus`）表示状态机，`memory_set` 和 `fd_table` 负责资源隔离，`sig_handler`/`sig_pending` 支撑信号投递，以及 `itimerval`/`timers` 支持 POSIX 定时器。这些字段的粒度体现了设计取舍——高频访问的 `status` 使用 `Mutex`，而读多写少的 `tid` 使用 `RwLock`，原子类型用于性能关键路径（如 `sched_prio`）。这种设计为后续 clone/fork 时的资源共享/复制提供了清晰的边界。

模块通过 `Processor` 结构体（见 `os/src/task/processor.rs`）管理每个 CPU 核心的本地状态：当前任务、空闲任务、是否可抢占等。每核一个的 `PROCESSOR` 数组通过 `lazy_static` 初始化，运行时通过 `current_hart_id()` 获取索引。调度器实例同样按核隔离：`SCHEDULER`（`os/src/task/scheduler.rs:34-37`）是一个 `Vec<SyncUnsafeCell<Scheduler>>`，每个核心一个，利用 `SyncUnsafeCell` 避免锁竞争。这与 Linux 的 per-CPU runqueue 思想一致，但代价是跨核负载均衡的缺失——目前仅通过 `select_cpu` 静态轮询（`scheduler.rs:156-158`），无任务窃取机制。

外部依赖上，调度算法通过 `crate::sched` 模块提供，支持两种实现：`FIFOScheduler`（默认，基于双端队列的简单 FIFO）和 `CFSScheduler`（条件编译 `feature="cfs"` 激活）。CFS 的引入扩展了调度语义，支持虚拟运行时间 `vruntime` 和负载权重 `LoadWeight`，但代价是增加了调度器接口的抽象（`fetch_rt_task` 等函数根据 `#[cfg(feature="cfs")]` 分别处理实时与非实时任务）。

### 2. 关键设计取舍

**2.1 锁策略与并发安全**
`Task` 结构体内部混合使用 `Mutex`、`RwLock` 和原子变量。大锁（`spin::Mutex`）出现于 `status`、`sig_pending`、`fd_table` 等字段，而 `tid` 使用 `RwLock` 允许并发读。一种明显的风险是 `task_inner: RwLock<TaskInner>` 字段持有 `priority`、`policy`、`cpu_mask` 等调度相关属性，而调度器路径（`scheduler.rs`）频繁读取这些值，如果持锁时间过长（如遍历 `BTreeMap`）可能引发性能问题。此外，`TimeManager::split_off`（`manager.rs:320-347`）在持有 `alarms` 锁期间调用了回调函数（`cb()`），而回调可能再次访问同一锁（如 `wakeup` 函数尝试加任务锁），这构成潜在的 **锁重入风险** ——虽然当前回调仅做原子操作和唤醒，但设计上应谨慎。

**2.2 进程/线程模型**
`clone` 系统调用通过 `kernel_clone`（`task.rs:178-350`）实现，支持 `CloneFlags` 位掩码，包括 `CLONE_VM`、`CLONE_FILES`、`CLONE_SIGHAND` 等标准标志。创建线程时，共享父进程的 `thread_group`、`itimerval`、`rlimit` 等；创建进程时则复制或分配新资源。关键点在于 `CLONE_VM` 路径下直接克隆 `memory_set` 的 `Arc` 指针，而非深拷贝——这依赖写时复制机制（COW，由 `MemorySet::from_existed_user_lazily` 延迟分配）。同样，`CLONE_FILES` 直接 `clone` 文件描述符表的 `Arc`。这种共享避免了不必要的复制，但要求内核保证页表、文件表等资源的引用计数安全（通过 `Arc` 实现）。对比 rCore-Tutorial 的实现，此项目额外处理了 `CLONE_CHILD_SETTID`（通过 `copy_to_user` 写入用户空间）和 `CLONE_VFORK` 的同步阻塞（通过 `vfork` 标志和特别的状态机），体现了对 Linux 兼容性的细化。

**2.3 调度器多实例与 CFS 集成**
调度器按核独立（`SCHEDULER` 数组），通过 `unsafe` 访问 `SyncUnsafeCell` 来避免锁开销——这里的安全契约是每个核心仅访问自己的调度器，且调度路径保证串行化。`#[cfg(feature="cfs")]` 条件编译引入了两条路径：
- `add_task`：实时任务通过 `add_rt_task` 加入实时队列，普通任务加入 CFS 队列。
- `fetch_task`：优先检查实时队列，再走 CFS 的 `fetch`。
- `remove_task`：同时从实时队列和 CFS 队列移除。
这种设计复用了 ArceOS 的调度器泛型接口，但增加了 `rt_task` 的独立管理。值得注意的是，`yield_current_task` 在 CFS 版本（`scheduler.rs:231-281`）中始终将当前任务放回就绪队列（即使优先级高），而与 FIFO 版本中比较优先级的行为不同——CFS 的公平性要求每次调度选择最小 vruntime，因此无条件放回。这种分歧意味着 `yield_current_task` 的语义依赖于调度器类型，可能造成行为不一致。

### 3. 跨模块协同

**3.1 调度器与定时器/阻塞机制**
`schedule()` 函数（`scheduler.rs:164-225`）在执行用户任务前先调用 `handle_timeout()`（即 `TimeManager::split_off`），该函数遍历 `BTreeMap` 找到已过期的闹钟，移除并执行回调。回调通常是 `wakeup(tid)`，将任务从 WAIT 队列移回就绪队列。`wait_timeout`（`manager.rs:133-148`）先设置定时闹钟（`set_wait_alarm`），然后将当前任务加入 WAIT 队列（`WaitManager::add`），最后调用 `schedule()` 阻塞。超时或信号中断时，任务被唤醒继续执行，根据返回值判断唤醒原因。这种设计实现了 **计时阻塞与信号中断** 的统一框架，但注意 `wakeup` 回调在执行时持有 `TimeManager` 的锁，而 `wakeup` 内部可能尝试获取 `PriorityManager` 或任务锁——若锁协议不当，可能形成循环依赖。

**3.2 信号与任务状态机**
`Task` 含有 `interrupted: AtomicBool` 和 `sig_pending` 等字段。在 `wait()` 和 `wait_timeout()` 中，任务通过 `set_interruptable()` 标记为可中断睡眠（`TASK_INTERRUPTIBLE`）。当信号到达时，`handle_signal` 或其他路径会设置 `interrupted` 并在调度时通过 `is_interrupted()` 检查。`schedule()` 返回后，`wait_timeout` 检测到 `is_interrupted()` 返回 `-1`，从而支持 `EINTR` 语义。`delete_wait` 在任务异常退出时清理 WAIT 队列，避免悬挂引用。

**3.3 进程组管理与 wait 系统调用**
`ProcessGroupManager` 维护 `BTreeMap<pgid, Vec<Weak<Task>>>`，支持 `wait4` 等语义。系统调用（如 `waitpid`）需要查询子进程或同组进程的状态。`get_group` 返回弱引用列表并清理无效项，配合 `remove_group` 在进程退出时更新。然而，目前 `ProcessGroupManager::remove` 仅从组列表中移除自身，未处理组长退出后组消失的场景（Linux 的孤儿进程组），可能造成进程组孤立。

### 4. 边角细节与不足

- **[cfg] 条件编译歧义**: `TaskManager` 在 `#[cfg(feature="virt")]` 下使用 `HashMap`，`#[cfg(feature="board")]` 下使用 `BTreeMap`（`manager.rs:55-58`）。若两者同时未定义（例如测试环境），将编译失败，属于隐性硬约束。 `<sup>[1](#mod-task-ref-1)</sup>`
- **unsafe 块缺乏安全注释**: `SCHEDULER` 的访问使用 `unsafe { &mut *SCHEDULER[hart_id].get() }`（`scheduler.rs:70`），没有 `// SAFETY:` 注释解释为什么 per-hart 访问是安全的。同样，`__switch` 的 `unsafe` 块（`scheduler.rs:199`）也未申明其安全前提（栈指针有效、上下文正确）。 `<sup>[2](#mod-task-ref-2)</sup>`
- **可能的重入死锁**: `TimeManager::split_off` 在锁内执行回调 `cb()`（`manager.rs:340`），而回调可能调用 `wakeup`，后者又可能尝试获取 `TimeManager` 的锁（如 `remove_timer`），造成 A-B-B-A 死锁。当前代码未采取延迟释放或递归锁措施。 `<sup>[3](#mod-task-ref-3)</sup>`
- **Mutex 粒度粗**: `Task` 的 `status`、`sig_pending` 等字段各自使用独立的 `Mutex`，但 `task_inner` 使用 `RwLock`。然而，在调度路径中，`task.set_ready()`（更新 `status`）与读取 `priority` 等操作没有原子性保证，可能导致竞态（例如刚好在检查优先级后状态突变）。 `<sup>[4](#mod-task-ref-4)</sup>`
- **错误路径吞掉**: `WAIT_MANAGER.remove(tid)` 返回 `Err(())` 时，`wakeup` 函数直接忽略（`manager.rs:160`）。这意味着尝试唤醒一个不在 WAIT 队列的任务无任何反馈，可能掩盖调度错误。 `<sup>[5](#mod-task-ref-5)</sup>`
- **TODO 标记**: `ProcessGroupManager::add_group` 中有 `// Todo: 对于线程更改进程组需要更严格的限制`，表明对线程组的管理仍不完整。 `<sup>[6](#mod-task-ref-6)</sup>`
- **Magic number 与平台假设**: `MAX_POSIX_TIMER_COUNT = 16`（`timer.rs`），`RLIM_NLIMITS = 16`（`task.rs`）作为硬编码常数，未考虑 future 扩展或配置化。 `<sup>[7](#mod-task-ref-7)</sup>`
- **死代码**: `manager.rs` 中 `wait_event`、`wake_cond_one`、`wake_cond_all` 等条件阻塞函数被注释掉，但仍保留在源码中，显示设计曾考虑条件变量但未完成。 `<sup>[8](#mod-task-ref-8)</sup>`
- **生命周期假设**: `TimerManager` 的回调 `Box<dyn Fn() + Send>` 要求 `'static`，但实际回调捕获的 `Tid` 是整数因此安全；但 `wakeup` 等函数若未来捕获引用，可能违反 `Send` 约束。 `<sup>[9](#mod-task-ref-9)</sup>`

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块实现了类Linux VFS层，核心抽象包括 Dentry、InodeOp trait（inode操作）、FileOp trait（文件操作）以及路径解析器 Nameidata（类似 Linux 的 nameidata）。它解决了多文件系统（ext4、procfs、devfs、pipefs）在统一接口下的协调问题。与同族 rcore‑tutorial 基线相比，本仓库增加了完整的命名管道（FIFO）支持、更丰富的 procfs 条目（/proc/self/fd、/proc/sys/kernel/* 等），并引入了 O_TMPFILE 和 openat2 解析标志。

### 1. 核心抽象与外部依赖
文件系统模块的核心是 VFS 抽象层。`Dentry` 表示目录项缓存，通过 `lookup_dcache_with_absolute_path` 进行快速路径查找（namei.rs: `lookup_dentry`）。`InodeOp` trait（pipe.rs: `impl InodeOp for PipeInode`）定义了 inode 级别的操作（lookup、create、mkdir、getattr 等），而 `FileOp` trait 定义了文件描述符级别的操作（read、write、seek）。路径解析器 `Nameidata` 记录当前解析状态，包括 `path_segments`、当前 `dentry`、挂载点 `mnt` 和解析深度 `depth`。`Nameidata::new` 根据路径类型（绝对/相对）和 `dfd` 决定起始点，并检查是否目录（`can_lookup`）。

外部依赖方面，模块严重依赖 `task::current_task()` 获取当前进程的根目录、工作目录、umask 和文件描述符表；依赖 `ext4::inode::Ext4InodeDisk` 作为底层 inode 的磁盘表示（pipe.rs 和 procfs 也模拟这一结构）；依赖 `signal` 模块发送 `SIGPIPE`；依赖 `sync` 原语（`spin::Mutex`、`spin::RwLock`、`AtomicI32`）保证并发安全。

### 2. 关键设计取舍
**路径解析的循环与符号链接处理**：`open_last_lookups` 在主循环中逐个分量查找 dentry，对符号链接调用 `nd.resolve_symlink` 更新 `path_segments` 并重新遍历，最多允许 `MAX_SYMLINK_DEPTH = 40` 层（namei.rs）。此设计模仿 Linux 的 `link_path_walk`，但去除了 `struct path` 的临时绑定，直接修改 `Nameidata` 内部状态。取舍：简化了状态管理，但使符号链接的相对路径解析需要手动处理

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制模块实现了 Linux 兼容的 POSIX 信号处理，核心抽象包括 `SigPending`（待处理信号集与掩码）、`SigHandler`（64 个 `SigAction` 的动作表）和 `SigFrame`/`SigRTFrame`（注入用户栈的信号帧）。与 rCore-Tutorial 基线相比，本实现额外支持 `SA_SIGINFO`、`SA_ONSTACK`、`SA_RESTART`/`ERESTARTSYS` 重启语义以及 LoongArch64 与 RISC-V 双架构。模块与任务调度器通过 `receive_siginfo` 协同，在 trap 返回前由 `handle_signal` 修改 `TrapContext` 以重定向到信号处理函数，并通过 `rt_sigreturn` 恢复上下文。

### 1. 核心抽象与外部依赖

信号机制围绕三个核心数据结构展开：`SigSet`（位图信号集合）<sup>[1](#mod-signal-ref-1)</sup>、`SigPending`（待处理信号队列与掩码）<sup>[2](#mod-signal-ref-2)</sup> 和 `SigHandler`（信号动作表）<sup>[3](#mod-signal-ref-3)</sup>。`SigPending` 使用 `BTreeMap<i32, SigInfo>` 存储每个信号的详细信息，`fetch_signal` 通过 `find_signal` 在待处理集中按序号递增查找第一个未被掩码且非 `SIGKILL`/`SIGSTOP` 的信号，这符合 POSIX 未决信号传递顺序。`SigHandler` 维护了一个 64 个 `SigAction` 的数组，每个动作包含处理函数指针 `sa_handler`、标志位 `SigActionFlag` 以及临时阻塞掩码 `mask`。

模块依赖 `bitflags` crate 实现 `SigSet` 的位操作，依赖 `alloc` 进行动态内存分配，并通过 `log` 输出调试信息。与外部系统的交互通过 `Task` 结构体的方法委托完成：`op_sig_pending_mut` 访问任务的 `SigPending`，`op_sig_handler` 访问 `SigHandler`，`op_thread_group_mut` 遍历线程组。这种委托模式避免了全局信号管理器，将信号状态绑定到任务对象上，有利于 SMP 扩展。

### 2. 关键设计取舍

**信号传递模型**：`receive_siginfo` 根据 `thread_level` 参数区分线程级与进程级信号 <sup>[4](#mod-signal-ref-4)</sup>。进程级信号遍历整个线程组，将信号插入每个任务的 `SigPending`，并检查是否满足中断条件（任务可中断、信号未忽略）。这与 Linux 行为一致：`kill()` 发送进程级信号，`tkill()` 发送线程级信号。但实现中当 `SIGCONT` 到达停止的任务时直接调用 `continue_task`，未检查信号掩码，可能导致停止状态被意外恢复。

**重启语义处理**：`handle_signal` 中通过检查 `A0 == ERESTARTSYS` 和 `SA_RESTART` 标志决定是否回退 `sepc` 恢复系统调用 <sup>[5](#mod-signal-ref-5)</sup>。该段逻辑在 RISC-V 和 LoongArch64 分支下重复出现，且存在冗余的 `else if` 条件（`ERESTARTSYS` 分支被写了两次），实际上后一次永远不会执行。此外，`SA_RESTART` 与 `is_interrupted` 的组合处理中，先判断 `re_start` 再判断 `ERESTARTSYS`，但两个分支最终都执行 `sepc -= 4`，架构差异可能覆盖了系统调用指令长度的不同（RISC-V 的 `ebreak` 为 4 字节，LoongArch64 的 `syscall` 也为 4 字节，此处假设一致）。

**用户栈帧布局**：根据是否设置 `SA_SIGINFO` 标志，向用户栈依次压入 `SigFrame`（含 `SigContext`）或 `SigRTFrame`（含 `UContext` 和 `LinuxSigInfo`）<sup>[6](#mod-signal-ref-6)</sup>。`FrameFlags` 使用硬编码魔术数字 `0x66666666` 和 `0x77777777` 区分普通帧与实时帧，这增加了内核与用户空间约定错误的可能性，而 Linux 通常通过 `sigframe` 中的 `pretcode` 地址或 `rt_sigreturn` 的固定指令来识别。本实现采用 `FrameFlags` 的方式在 `rt_sigreturn` 中读取该标志以决定恢复方式。

**信号栈（sigaltstack）**：`SigStack` 结构体默认 `ss_flags` 为 `SS_DISABLE`，在 `handle_signal` 中对 `SA_ONSTACK` 的处理仅检查 `ss_flags == 0 || ss_flags == SS_ONSTACK` <sup>[7](#mod-signal-ref-7)</sup>，缺少对 `SS_AUTODISARM` 和 `SS_DISABLE` 的完整判断，可能导致在已禁用信号栈时错误使用。

### 3. 跨模块协同

**与任务调度器**：`receive_siginfo` 在添加信号后调用 `delete_wait`、`set_ready`、`add_task` 将任务重新加入调度队列 <sup>[4](#mod-signal-ref-4)</sup>。这种抢占式唤醒模型确保阻塞在 `wait4`、`read` 等可中断等待的任务能及时处理信号。`check_interrupt` 方法用于判断是否满足中断条件，被 `receive_siginfo` 调用。

**与 Trap 处理**：`handle_signal` 在系统调用返回或异常处理末尾被调用，它读取当前任务的 `TrapContext`，修改 `sepc` 指向信号处理函数、`ra` 指向 `sigreturn_trampoline`、`sp` 指向压好帧的用户栈 <sup>[5](#mod-signal-ref-5)</sup>。这种设计将信号传递完全封装在 `handle_signal` 函数中，trap 处理流程仅需在返回用户空间前插入一次调用即可。

**与内存管理**：通过 `copy_to_user` 将信号帧写入用户栈，依赖 MMU 的权限检查 <sup>[6](#mod-signal-ref-6)</sup>。如果写入失败（如用户栈非法），直接 `panic!`，这在生产环境中应该返回 `SIGSEGV` 给进程。

**与系统调用实现**：`rt_sigreturn` 系统调用会读取用户栈上的 `FrameFlags`，根据标志恢复 `SigContext` 或 `UContext`，包括寄存器、`sepc` 和信号掩码，然后清理栈帧 <sup>[8](#mod-signal-ref-8)</sup>。`sigreturn_trampoline` 是一段汇编代码，通常执行 `rt_sigreturn` 系统调用，其地址被写入 `ra`，确保信号处理函数返回时能正确恢复。

### 4. 边角细节与不足

- **<sup>[1](#mod-signal-ref-1)</sup> `SigSet` 位图定义中 `SYNCHRONOUS_MASK` 被注释掉**，但 `SigSet` 并未提供同步/异步分类支持，这可能导致同步信号（如 `SIGSEGV`）在掩码作用下被延迟处理，违反 POSIX 要求。
- **<sup>[2](#mod-signal-ref-2)</sup> `sig_struct.rs` 与 `sig_struct copy.rs` 两个文件并存**，后者额外包含 `interrupted`、`re_start`、`restore_mask` 字段和对应方法，但前者未使用。这种不一致可能是开发中的重构未清理，导致存在两份 `SigPending` 定义，容易引起混淆。
- **<sup>[3](#mod-signal-ref-3)</sup> `SigHandler::new()` 使用了 `MaybeUninit::uninit().assume_init()` 加 `transmute` 初始化 64 个 `SigAction` 数组**，未提供 `// SAFETY:` 注释。这一 `unsafe` 块假设 `MaybeUninit` 数组的布局与 `SigAction` 数组一致，但未考虑填充或对齐，是潜在的技术债。
- **<sup>[5](#mod-signal-ref-5)</sup> `handle_signal` 中 `while let ... fetch_signal` 循环每次 `break`**，实际只处理一个信号。如果多个信号同时待处理，只有第一个会被传递，剩余信号需等待下次 trap 事件。这可能影响实时信号的优先级表现，但简化了实现。
- **<sup>[6](#mod-signal-ref-6)</sup> `copy_to_user` 失败直接 `panic!`**，而 Linux 在无法传递信号帧时应保持静默或发送 `SIGSEGV`。这一硬 panic 可能被恶意用户空间构造非法栈地址触发拒绝服务。
- **<sup>[9](#mod-signal-ref-9)</sup> `core` 函数中直接 `panic!(

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 实现了管道（匿名/命名）、System V 共享内存和 futex 三类 IPC 机制。管道核心是环形缓冲区 `PipeRingBuffer` 和 `PipeInode`（实现 InodeOp 融入文件系统）；共享内存由 `ShmManager` 管理 `ShmSegment`，支持延迟删除与经典 Unix 权限；futex 基于哈希桶等待队列，支持私有/共享键、超时、bitset、requeue 和 robust list。相比 rCore-Tutorial，新增了命名管道阻塞语义、完整 shm 操作集和 futex 高级操作，但存在饥饿风险、死锁隐患和部分未实现操作（FUTEX_WAKE_OP）。

### 1. 核心抽象与外部依赖

进程间通信模块分为三个子系统：管道、System V 共享内存和 futex，它们都依赖 `spin::Mutex/RwLock` 保护共享状态、`alloc::sync::{Arc, Weak}` 管理生命周期、`crate::task::{wait, wakeup}` 实现阻塞唤醒、以及 `crate::signal` 传递 SIGPIPE。

**管道**的核心是 `PipeRingBuffer`（环形缓冲区）<sup>[1](#mod-ipc-ref-1)</sup> 和 `PipeInode`（实现 `InodeOp` trait）。`PipeRingBuffer` 使用 `Vec<u8>` 动态分配，支持运行时 `resize` 调整容量。每个 `Pipe` 包含 `read_end`/`write_end` 两个 `FileOp` 实现，通过 `Weak<dyn FileOp>` 检测对端关闭。命名管道通过 `is_named_pipe` 标记，打开时若对端未就绪则阻塞或返回 EAGAIN/ENXIO <sup>[13](#mod-ipc-ref-13)</sup>。

**共享内存**的核心是全局 `ShmManager`（`BTreeMap<usize, ShmSegment>`）<sup>[3](#mod-ipc-ref-3)</sup> 和 `ShmSegment`（包含页列表和 `IpcPerm` 权限）。`ShmId` 结构体完整映射 Linux `shmid_ds`，包括 `atime/dtime/ctime`、`cpid/lpid`、`nattch` 等字段。`attach_shm_segment` 通过调用 `MemorySet::attach_shm_segment` 将物理页映射到进程地址空间 <sup>[4](#mod-ipc-ref-4)</sup>。

**Futex** 的核心是 `FutexQ`（等待队列项，包含 `Weak<Task>` 和 `bitset`）<sup>[6](#mod-ipc-ref-6)</sup> 和 `FutexKey`（统一结构体，用 `ptr` 指代 mm 或 inode、`aligned` 页对齐地址、`offset` 页内偏移）<sup>[7](#mod-ipc-ref-7)</sup>。哈希桶数组 `FUTEXQUEUES` 由 `futex::queue` 模块提供，`futex_hash` 计算桶索引。

### 2. 关键设计取舍

**管道环形缓冲区的伸缩性**：`PipeRingBuffer` 使用 `Vec<u8>` 而非固定数组，允许通过 `resize` 动态扩容，且扩容时完全拷贝数据（不保留环形结构），简化了实现但造成 O(n) 开销。`resize` 不允许缩小到小于已使用空间（返回 EBUSY），避免数据丢失。阻塞模型采用“释放锁再等待”：在缓冲区满/空时 `add_waiter` 后 `drop(buffer)` 再 `wait()`，被唤醒后重新循环，这避免了持锁阻塞但引入了竞争窗口。

**共享内存的延迟删除**：`shmctl(IPC_RMID)` 将 `marked_for_deletion` 置 true，仅当 `nattch==0` 时才真正移除段 <sup>[5](#mod-ipc-ref-5)</sup>。这符合 POSIX 要求：标记后新 attach 仍可成功（因为段仍存在），直到所有进程 detach。权限检查采用两层模型：`check_shm_perm` 根据 euid/egid 匹配 user/group/other 权限位 <sup>[14](#mod-ipc-ref-14)</sup>；`check_shm_own` 检查是否拥有者或创建者（root 豁免）。

**Futex 键的扁平化设计**：`FutexKey` 使用 `u64 ptr + u64 aligned + u32 offset` 三个字段而非枚举，简化了比较和哈希。私有 futex 的 `ptr` 为 `mm` 指针，共享 futex 的 `ptr` 为 inode 指针（通过 `get_shared_mmaping_info` 获取）。这种设计避免了 match 分支性能开销，但在共享 futex 中 `inode_addr` 的计算需要谨慎（注释指出直接用 inode_num 可能因回收导致误匹配，故使用 inode 内存地址）。

**Futex 超时与信号处理**：`futex_wait` 中区分了 `FUTEX_WAIT` 和 `FUTEX_WAIT_BITSET`，后者支持 CLOCK_REALTIME 与 MONOTONIC。超时计算在 `do_futex` 中预处理，`futex_wait` 内部根据 `clock_id` 分支调用 `wait_timeout`。当 `wait()` 返回 -1 时视为被信号中断，从队列中移除任务并返回 EINTR；超时返回 ETIMEDOUT。但处理路径存在重复代码（信号唤醒后两分支均执行 retain 清理）<sup>[8](#mod-ipc-ref-8)</sup>。

### 3. 跨模块协同

**管道与文件系统**：`PipeInode` 实现 `InodeOp` trait，使 `fstat` 可获取管道元数据（大小、时间戳、权限）。匿名管道通过 `make_pipe` 返回两个 `Arc<Pipe>`，再包装成 fd。命名管道通过文件系统路径 `open` 创建，`PipeInode` 的 `inode_num` 用于区分匿名/命名管道（匿名 inode_num 固定为 0，命名管道使用真实 inode 号）。

**共享内存与内存管理**：`attach_shm_segment` 调用 `MemorySet::attach_shm_segment` 将 `ShmSegment.pages` 映射到指定虚拟地址。`addr2shmid` BTreeMap 维护映射关系，用于 `shmdt` 时反向查找。`detach_shm_segment` 解除映射并更新 `nattch`，若 `nattch` 降为 0 且标记删除则物理删除段。

**Futex 与任务调度**：`FutexQ` 持有 `Weak<Task>`，避免阻止任务退出。`futex_wake` 通过 `wakeup(task.tid())` 唤醒，`futex_wait` 在阻塞前将自身加入哈希桶。`futex_requeue` 和 `futex_cmp_requeue` 支持跨桶迁移 <sup>[10](#mod-ipc-ref-10)</sup>，当源桶与目标桶不同时需顺序锁两个桶（先源后目标），若哈希函数使它们映射到同一桶则直接返回 `futex_wake` 避免死锁 <sup>[10](#mod-ipc-ref-10)</sup>。

### 4. 边角细节与不足

**`<sup>[1](#mod-ipc-ref-1)</sup>` `PipeRingBuffer::get_one_waiter` 使用 LIFO 顺序**：`add_waiter` 是 `push`，`get_one_waiter` 是 `pop`，导致 waiter 按栈顺序唤醒，可能使部分等待者长期饥饿。Linux 管道使用 FIFO 队列，这是与基线行为的分歧。

**`<sup>[10](#mod-ipc-ref-10)</sup>` Futex requeue 的双重锁顺序可能死锁**：`futex_requeue` 先锁源桶，再锁目标桶，若另一个线程以相反顺序锁这两个桶，则死锁。虽然 `futex` 操作通常不嵌套，但多线程并发 requeue 不同键时仍可能触发。

**`<sup>[11](#mod-ipc-ref-11)</sup>` `FUTEX_WAKE_OP` 直接 panic**：do_futex 收到 `FUTEX_WAKE_OP` 命令时调用 `panic!`，未实现该操作。这可能被用户态程序意外触发导致内核崩溃。应返回 `ENOSYS`。

**`<sup>[5](#mod-ipc-ref-5)</sup>` `SHM_LOCK/SHM_UNLOCK` 是权限 stub**：`do_shmctl` 中这两个操作仅修改 `mode` 位（0o2000），未实际调用 `mlock` 锁定物理页，违反 POSIX 语义。

**`<sup>[8](#mod-ipc-ref-8)</sup>` `futex_wait` 中未使用的 `clock_id` 变量**：在超时分支中定义了 `clock_id` 并传递给了 `wait_timeout`，但 `wait_timeout` 签名未见定义，且 `clock_id` 仅用于计算剩余时间，但计算结果未使用。可能是开发中残留。

**`<sup>[12](#mod-ipc-ref-12)</sup>` 管道写入 SIGPIPE 的发送时机**：`write` 函数在每次循环中检查 `all_read_ends_closed()`，若关闭则调用 `receive_siginfo` 发送 SIGPIPE 并返回 EPIPE。但该检查位于循环内，如果写入过程中读端关闭，会正确发送信号。但 SIGPIPE 的默认行为是终止进程，若进程未处理 SIGPIPE，写入管道将导致进程退出。

**`<sup>[6](#mod-ipc-ref-6)</sup>` `FutexQ` 中 `bitset` 的用途**：`bitset` 仅在 `futex_wake_bitset` 中匹配，而 `futex_wait` 的 `bitset` 未用于唤醒匹配，因为 `FUTEX_WAIT` 默认使用 `FUTEX_BITSET_MATCH_ANY`（全 1）。实现正确但注释缺失。

**`<sup>[8](#mod-ipc-ref-8)</sup>` `futex_wait` 中存在 `unwarp()` 风险**：`hash_bucket.retain(|futex_q| futex_q.task.upgrade().unwrap().tid() != task.tid())`，若 `Weak` 已失效则 panic。理论上 retain 中任务仍存在，但并发下其他线程可能已移除该任务导致 Weak 失效。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 基于 smoltcp 库实现 TCP/UDP 协议栈，核心抽象是 Socket 枚举体（Tcp/Udp）和 TcpSocket 的原子状态机。全局 SocketSet 和 ListenTable 管理所有 socket 和监听连接。该模块在 rCore-Tutorial 基础上扩展了 AF_ALG、Unix 域套接字和多种硬件平台适配，但大量采用轮询阻塞和全局互斥锁，存在性能瓶颈和未完善的功能实现。

### 1. 核心抽象与外部依赖
网络模块以 smoltcp 作为基础 TCP/IP 栈，在 `Socket` 结构体中通过 `SocketInner` 枚举区分 TCP 和 UDP 两种传输层协议 <sup>[1](#mod-net-ref-1)</sup>。`Socket` 同时携带了 `Domain`、`SocketType`、`Protocol` 等枚举类型，用于系统调用参数校验。`TcpSocket` 实例通过 `UnsafeCell<Option<SocketHandle>>` 持有 smoltcp 内部的 socket handle，其生命周期由原子状态机管理。全局 `SOCKET_SET` 和 `LISTEN_TABLE` 分别用于管理所有 socket 实例的 smoltcp 端和监听端口的连接队列。

### 2. 关键设计取舍
- **原子状态机 vs 锁**：`TcpSocket` 使用 `AtomicU8` 和 `compare_exchange` 实现状态迁移（CLOSED→BUSY→CONNECTING→...）<sup>[2](#mod-net-ref-2)</sup>。这种无锁方案避免了互斥锁持有期间的上下文切换开销，但 `BUSY` 状态的设计意味着只有一条线程能进入临界区，其他线程会收到 `EISCONN` 错误，可能导致连接请求的饥饿。
- **轮询阻塞模型**：`block_on` 函数在循环中调用 `poll_interfaces` 和 `yield_current_task`，直到操作成功或返回非 EAGAIN 错误 <sup>[3](#mod-net-ref-3)</sup>。相比使用等待队列和条件变量，该实现简单但消耗 CPU 且延迟不可控（无超时机制）。对于高吞吐场景可能造成过多的上下文切换。
- **全局锁瓶颈**：`SOCKET_SET` 和 `LISTEN_TABLE` 均包裹在 `Mutex<OnceCell<...>>` 中，所有 socket 操作（connect、accept、recv、send）都需要获取这把全局锁 <sup>[4](#mod-net-ref-4)</sup>。在多核多线程环境下，该锁将成为严重争抢点，限制网络吞吐。
- **硬编码常量与平台适配**：IP 地址、缓冲区大小、DNS 服务器等直接写死为常量（如 `const IP: &str = "10.0.2.15"`）<sup>[5](#mod-net-ref-5)</sup>，通过 `#[cfg]` 条件编译选择不同初始化函数。这导致增加新平台时需要修改多处代码，且未覆盖架构（如 MIPS）会导致编译错误。
- **unsafe 块安全契约缺失**：多处使用 `unsafe { self.handle.get().read() }` 访问 `UnsafeCell`，但未提供 `// SAFETY:` 说明仅当线程处于特定状态时才可访问 <sup>[6](#mod-net-ref-6)</sup>。这违反了 Rust 的安全惯例，增加了 UB 风险。

### 3. 跨模块协同
网络栈的驱动与响应由 `poll_interfaces` 触发，该函数在 `mod.rs` 中定义，被 `TcpSocket::block_on` 循环调用 <sup>[3](#mod-net-ref-3)</sup>。`poll_interfaces` 会轮询 `ETH0` 和 `LOOPBACK` 网络接口，驱动 smoltcp 处理接收和发送事件。当 TCP 连接建立时，smoltcp 会将新连接放入监听 socket 对应的 `LISTEN_TABLE` 中，由 `TcpSocket::accept` 取出并创建一个新的 `TcpSocket` 实例 <sup>[7](#mod-net-ref-7)</sup>。`Socket` 中的 Unix 域套接字和 AF_ALG 功能则通过额外的数据结构（如 `socket_path_unix`、`socket_af_alg`）与文件系统、加密算法模块协同。

### 4. 边角细节与不足
- **`poll_stream` 的可读性逻辑异常**：当 `!socket.may_recv() || socket.can_recv()` 为 false 且 `!socket.may_send() || socket.can_send()` 为 false 时，代码将 `readable` 强制设为 true，这实际上让不可读变为可读，可能导致上层误判 <sup>[8](#mod-net-ref-8)</sup>。该行为可能是为了在特殊情况下避免死锁，但未加注释说明理由。
- **`block_on` 无超时保护**：循环仅在成功或非 EAGAIN 错误时退出，若网络卡死或丢包，任务将永久阻塞 <sup>[3](#mod-net-ref-3)</sup>。虽然用户态可通过 `O_NONBLOCK` 避免，但内核态其他操作（如 `accept` 中的次数计数器 `times>20`）也仅硬编码退出，未与外部超时信号联动。
- **`close_exec` 标志未生效**：`Socket` 结构体中定义了 `close_exec` 字段，但进程 fork 或 exec 时代码并未查询该标志 <sup>[9](#mod-net-ref-9)</sup>。因此 `SOCK_CLOEXEC` 实际未实现，属于未完成功能。
- **全局锁粒度过大且初始化冗余**：`SOCKET_SET` 的锁在每次操作时获取，包括 `poll_interfaces`，而 `poll_interfaces` 内部可能再次调用 socket 方法，存在重入风险（未实测）。同时，`init` 函数中多次对 `SOCKET_SET` 和 `LISTEN_TABLE` 调用 `get_or_init`，但未防止重复初始化 <sup>[4](#mod-net-ref-4)</sup>。
- **错误处理粗放**：多处使用 `unwrap()`（如 `get_peer_unix_file` 中对路径的 UTF-8 转换 <sup>[10](#mod-net-ref-10)</sup>），用户态输入的路径可直接触发内核 panic。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> StarFive JH7110 (VF2) GMAC 以太网驱动，采用直接 MMIO 寄存器操作的 C 风格编程模型，核心数据结构为 VisionfiveGmac 与 DmaDesc，提供寄存器读写、DMA 描述符环管理、MDIO 总线访问、时钟/PLL 配置等底层操作。与同家族基线（ArceOS 的 driver_common trait 体系）相比，本实现完全放弃 Rust trait 抽象，所有函数均为自由函数而非 trait 方法，且大量代码为从 C 驱动直译，保留了 C 风格的位操作与循环等待模式。

### 1. 核心抽象：C 风格 MMIO 编程 vs Rust 驱动 trait 体系

该模块的核心抽象并非 Rust 的 trait 或泛型接口，而是**一簇操作同一组 MMIO 寄存器的自由函数**。具体来说，`eth_mac_read_reg`<sup>[1](#mod-drivers-ref-1)</sup> 与 `eth_mac_write_reg`<sup>[2](#mod-drivers-ref-2)</sup> 构成了整个驱动的底层读写原语，所有上层配置函数（`eth_set_mtl`、`eth_set_mac`、`eth_set_dma` 等）均基于它们实现。这种设计直接将 C 语言中 `readl`/`writel` 的模式移植到 Rust，未做任何安全封装。

核心数据结构是 `VisionfiveGmac`<sup>[3](#mod-drivers-ref-3)</sup>，它持有了五个基地址（`MacBase`、`DmaBase`、`PhyBase`、`MtlBase`、`iobase`）以及两个固定大小为 300 的描述符指针数组（`TxDesc`/`RxDesc`）。注意这里**没有使用 Rust 的切片或 Vec，而是固定长度的 C 风格数组**，表明该结构是为直接与 DMA 硬件交互而设计。DMA 描述符 `DmaDesc`<sup>[4](#mod-drivers-ref-4)</sup> 是一个 `#[repr(C)]` 的四字结构体，与 Synopsys DesignWare GMAC 的硬件描述符格式严格对齐。

与 ArceOS 基线的典型驱动实现相比——后者通常通过 `driver_common` crate 的 `NetDriver` trait 抽象，或使用 `virtio-drivers` 的 `VirtIONetRaw` 泛型封装——本模块完全**没有 trait 抽象层**。这意味着任何上层网络栈若要使用该驱动，必须直接调用 `eth_dev.rs` 中的自由函数，无法通过统一的设备接口切换后端。

### 2. 关键设计取舍：硬编码、条件编译与中断策略

**取舍一：硬编码 SoC 细节 vs 设备树/ACPI 发现。**
整个驱动深度绑定了 StarFive JH7110 SoC 的寄存器布局。`VisionfiveGmac::init()`<sup>[5](#mod-drivers-ref-5)</sup> 中直接硬编码了 MAC 地址 `[0x00, 0x55, 0x7B, 0xB5, 0x7D, 0xF7]` 和 IO 基地址 `KERNEL_BASE + 0x16040000`，而 `gmac1_clk_init`<sup>[6](#mod-drivers-ref-6)</sup> 则通过 `KERNEL_SPACE` 直接映射物理地址 `0x13020000` 和 `0x13030000` 来访问时钟控制寄存器（SYS_CRG）。这种做法的代价是**完全不可移植**——换一颗 SoC 或不同板级配置就需要重新硬编码所有地址。

**取舍二：条件编译选择驱动后端。**
在 `mod.rs` 中，StarFive 驱动通过 `#[cfg(all(target_arch="riscv64", feature="vf2"))]`<sup>[7](#mod-drivers-ref-7)</sup> 条件编译。这意味着只有 RISC-V 64 且显式启用了 `vf2` feature 时才会编译该驱动。对于未列出的架构（如 x86_64、AArch64）或未开启该 feature 的 RISC-V 构建，此模块完全不可见。这种设计简单直接，但限制了在统一内核映像中同时支持多平台的可能性。

**取舍三：轮询模式 vs 中断驱动。**
在 `eth_set_rx_desc`<sup>[8](#mod-drivers-ref-8)</sup> 中，描述符的 `EQOS_DESC3_IOC` 位虽然被设置（允许完成中断），但代码注释明确写道 `//we don't use interrupt here`。同时 `eqos_mdio_read` 和 `eth_dma_reset` 均采用**忙等待循环**，而非利用中断同步。这在内核开发的早期阶段可以简化调试，但会带来 CPU 空转和响应延迟问题。

### 3. 跨模块协同：内存管理、平台抽象与 PLL 时钟树

该驱动虽然自身缺乏 trait 抽象，但与内核其他三个子系统存在紧密的协同：

**内存管理子系统**：在 `gmac1_clk_init`<sup>[6](#mod-drivers-ref-6)</sup> 中，函数直接调用 `KERNEL_SPACE.lock().push_with_offset()` 将两个 64KB 的物理地址区间映射到内核虚拟空间。这是对内核页表机制的**直接使用**，绕过了更上层的设备内存管理（如 `DeviceMemory` 区域分配器）。这也意味着如果内核的页表锁设计有变化，此处的调用链需要同步修改。

**平台抽象层**：驱动通过 `crate::drivers::net::starfive::platform` 模块引入了一组以 `plat_` 为前缀的平台函数（`plat_malloc_align`、`plat_virt_to_phys`、`plat_fence`、`plat_mdelay`）<sup>[9](#mod-drivers-ref-9)</sup>。这些函数是驱动与具体硬件平台之间的唯一适配层。例如，`plat_virt_to_phys` 用于将虚拟地址转换为 DMA 可用的物理地址，`plat_fence` 确保内存序对 DMA 可见。这种设计虽然简单，但缺少 trait 约束，更换平台时必须确保该模块提供了相同的函数签名。

**计时与任务子系统**：`wait_for_bits`<sup>[10](#mod-drivers-ref-10)</sup> 和 `eqos_mdio_read`<sup>[11](#mod-drivers-ref-11)</sup> 中使用了 `get_time_ms`（来自 `arch::timer`）和 `plat_mdelay` 来实现超时等待，以及 `crate::task::wait_timeout` 的引用（尽管实际代码中未使用后者）。这些依赖使得驱动无法在无计时器的裸机环境下独立工作。

### 4. 边角细节与不足

本节指出代码中评审者容易忽略但影响可靠性的五个风险点：

**<sup>[1](#mod-drivers-ref-1)</sup> 超时判断逻辑错误 (`wait_for_bits`)**：行 389 的条件 `start+timeout_ms > (get_time_ms()/10) as u32` 存在两个问题。第一，`get_time_ms()` 返回毫秒，除以 10 后变成**厘秒**，导致超时时间被放大约 10 倍；第二，`get_time_ms()` 的返回值类型未知（可能是 u64），转换为 u32 在长期运行的系统上可能回绕。更严重的是，该行的 `//todo` 注释暗示开发者自己知道这里有问题但未修复。

**<sup>[2](#mod-drivers-ref-2)</sup> TX 缓冲区分配被注释 (`eth_set_tx_desc`)**：行 252 的 `// buffer = unsafe { plat_malloc_align(4096, 32) };` 被注释掉，导致 `gmacdev.TxBuffer[i]` 始终为初始值 0。这意味着**发送路径的 DMA 缓冲区地址为 NULL**，任何实际的 TX 操作都会导致 DMA 访问地址 0，可能触发总线错误或静默数据损坏。这是一个明确的半成品标记。

**<sup>[11](#mod-drivers-ref-11)</sup> MDIO 读取中的无超时死循环**：行 416-420 的 `loop { ret=wait_for_bits(...); if ret>=0 { break; } }` 在 `wait_for_bits` 返回 -1（超时）后会无限重试，而不是返回错误。如果 PHY 芯片挂死或 MDIO 总线卡住，此循环将导致系统永久停滞。

**<sup>[12](#mod-drivers-ref-12)</sup> 几乎所有 `unsafe` 块缺少安全契约注释**：整个 `eth_dev.rs` 中超过 30 个 `unsafe` 块——包括 `read_volatile`、`write_volatile`、`plat_malloc_align`、`plat_virt_to_phys`、指针偏移等——均未提供 `// SAFETY:` 注释。这使得后续维护者无法判断每个 `unsafe` 的前提条件是否仍然成立。

**<sup>[13](#mod-drivers-ref-13)</sup> 描述符 FD/LD 标志被注释**：`eth_set_tx_desc`（行 266-270）和 `eth_set_rx_desc`（行 310-314）中，First Descriptor（FD）和 Last Descriptor（LD）标志的设置代码被注释掉。在标准的 DWMAC 驱动中，DMA 依靠 FD/LD 位来识别帧边界。缺少这些标志可能导致发送/接收一帧时 DMA 无法正确同步，尤其是在多描述符帧的场景下。

**<sup>[5](#mod-drivers-ref-5)</sup> MAC 地址与描述符数量硬编码**：`VisionfiveGmac::init()` 将 MAC 地址写死为 `[0x00, 0x55, 0x7B, 0xB5, 0x7D, 0xF7]`，未从设备 EEPROM/OTP 或设备树中读取；`GMAC_DESC_NUM = 300`<sup>[14](#mod-drivers-ref-14)</sup> 对大多数应用场景过大（每个描述符 16 字节，300 个共 4800 字节，且每个 RX/TX 各有 300 个），浪费内存且未参数化。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层模块作为用户态与内核态的桥梁，将Linux兼容的系统调用接口划分为fs、task、net、util四个子模块，各模块独立处理文件I/O、进程控制、网络通信和杂项请求。该模块通过临时内核缓冲区实现用户数据安全拷贝，并支持preadv2、execveat等高级系统调用，显著扩展了rCore-Tutorial的覆盖度，以满足LTP测试套件需求。与同家族基线相比，其独特之处在于深度Linux兼容性（包括AF_UNIX、AF_ALG部分实现）以及为调试保留的大量日志输出。

### 1. 架构划分与核心抽象
系统调用层位于`os/src/syscall/`目录，按功能划分为四个子模块：`fs.rs`处理文件与文件系统相关调用（如read、write、lseek、dup等）<sup>[1](#mod-syscall-ref-1)</sup>；`task.rs`负责进程控制（clone、execve、exit、wait等）<sup>[2](#mod-syscall-ref-2)</sup>；`net.rs`实现网络相关调用（socket、bind、connect、send等）<sup>[3](#mod-syscall-ref-3)</sup>；`util.rs`涵盖杂项（getrandom、rlimit、clock等）<sup>[4](#mod-syscall-ref-4)</sup>。每个系统调用函数接收原始usize参数，经过标志验证与用户态拷贝后，调用底层内核服务并返回`SyscallRet`（即`Result<isize, Errno>`）。

该模块不直接操作硬件或调度，而是委托给任务管理、VFS、网络栈等下层模块。例如`sys_read`通过`current_task()`获取当前任务的引用，再经文件描述符表获得`Arc<dyn FileOp>`对象，最后调用其`read`方法<sup>[1](#mod-syscall-ref-1)</sup>。设计上采用了典型的分层模式，各子模块职责清晰，但函数体较长，复用逻辑（如参数验证）存在重复。

### 2. 关键设计取舍
**全内核缓冲区拷贝**：`sys_read`与`sys_write`均先分配临时内核`Vec`，通过`copy_from_user`/`copy_to_user`间接转移数据<sup>[1](#mod-syscall-ref-1)</sup>。这确保了安全隔离（用户空间指针不会在内核态直接解引用），但额外引入了一次分配与拷贝，对大I/O可能影响性能。替代方案（如Linux直接映射用户页面）未采用，可能因架构抽象层`arch::mm`的设计限制。

**错误路径处理差异**：多数系统调用使用`?`传播错误，但部分地方只输出`log::error!`而不返回错误码（如`sys_read`中注释掉的`EISDIR`检查）<sup>[1](#mod-syscall-ref-1)</sup>。这可能导致用户调用`read`目录文件时收到部分数据而非明确错误，属于未完成的实现。

**Linux兼容性深度**：实现了`sys_preadv2`、`sys_pwritev2`、`sys_execveat`等较新调用，并处理`SOCK_NONBLOCK`、`SOCK_CLOEXEC`等标志<sup>[1](#mod-syscall-ref-1)</sup><sup>[3](#mod-syscall-ref-3)</sup>。但大量匹配分支带有`Todo`警告（如网络中的`flags`参数）<sup>[3](#mod-syscall-ref-3)</sup>，表明兼容性仍在演进。

**简化的权限检查**：`syscall_bind`仅检查`task.egid() != 0 || task.euid() != 0`<sup>[3](#mod-syscall-ref-3)</sup>，未实现Linux复杂的`CAP_NET_BIND_SERVICE`机制，可能导致低权限用户误绑定特权端口。

### 3. 跨模块协同
- **与任务管理**：`task.rs`中的系统调用频繁调用`current_task()`获取当前任务引用，并通过`Task`结构操作进程组、文件描述符表、线程组等。`sys_clone`调用`task.kernel_clone`创建新任务后需经`add_task`加入调度；`sys_execve`调用`kernel_execve_lazily`替换用户态映像<sup>[2](#mod-syscall-ref-2)</sup>。
- **与VFS层**：文件系统调用依赖`crate::fs::file::FileOp` trait，该trait统一了文件、socket、pipe等对象的读写接口。`sys_read`等函数通过文件描述符获取`Arc<dyn FileOp>`，再调用其多态方法<sup>[1](#mod-syscall-ref-1)</sup>。
- **与网络栈**：`net.rs`通过`Socket`对象（封装`smoltcp`栈）实现网络系统调用。`syscall_connect`解析用户地址后调用`socket.connect`，最终触发协议栈状态机<sup>[3](#mod-syscall-ref-3)</sup>。
- **与内存管理**：`copy_from_user`/`copy_to_user`最终映射到`arch::mm`模块，承担用户/内核空间间数据拷贝，可能涉及页表暂态映射，但具体实现对系统调用层透明。

### 4. 边角细节与不足
- **大量日志输出**：系统调用函数中充斥`log::error!`、`log::warn!`等日志，且多数未被条件编译剔除，生产环境会产生巨量输出并暴露内核路径。
- **未完成网络功能**：`net.rs`中`AF_UNIX`的connect对等端写检查使用`file.w_ready()`，若返回false则直接返回`EBADF`<sup>[3](#mod-syscall-ref-3)</sup>，未处理等待或非阻塞重试，可能导致合法场景失败。
- **硬编码忽略列表**：`task.rs`中的`IGNORE_TEST`数组<sup>[2](#mod-syscall-ref-2)</sup>及`sys_execve`中对`tst`前缀的特殊处理（直接exit 666）是专为LTP适配的临时代码，属于工程腐烂信号。
- **架构重复代码**：`sys_clone`对不同架构（riscv64/loongarch64）的实现几乎相同，仅`TrapContext`操作有少数差异，存在大量`#[cfg]`重复，可提取公共逻辑。
- **整数溢出风险**：`sys_readv`和`sys_writev`中逐项累加`iovec_total_len`，仅在循环外检查`> isize::MAX`，未在循环内防溢出，`iovec.len`来自用户空间，可能导致溢出绕过长度检查（尽管当前检查在最后，但可能被未捕获的溢出绕过）。<sup>[1](#mod-syscall-ref-1)</sup>
