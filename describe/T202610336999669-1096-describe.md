## 📖 一、总览

该仓库是一个面向 Linux 用户程序兼容的 Rust 宏内核，支持 RISC-V 和 LoongArch 多核架构。其设计目标超越教学型 rCore-Tutorial，旨在运行真实负载（如 musl libc、LTP 测试），在内存管理、调度、文件系统和网络方面进行了大量优化与兼容性打磨。核心架构整合了进程/线程统一模型、VFS 抽象、信号机制、System V IPC 和基于 smoltcp 的网络栈，整体功能广度接近可用原型。
值得注意的独特设计包括：1）启动流程通过 `rust_main_boot`/`rust_main_secondary` 分离架构差异，并结合原子标志实现多核同步，统一了 RISC-V 与 LoongArch 的启动路径；2）内存管理引入需求分页和 frame poison 隔离区调试，有效检测 use-after-free；3）调度器采用 EEVDF 与 FIFO 双轨设计，每个 CPU 独立调度并支持 work-stealing 负载均衡，锁序精心编排以避免死锁。

## 🎯 二、综合评价

### 整体定位
该仓库属于**实验性 Linux 兼容内核**，在 rcore-rust 家族中处于从教学演示向完整功能过渡的阶段。它用 Rust 实现了进程调度、信号、内存管理、文件系统、IPC 和网络等核心子系统，并针对多核（SMP）和多架构（RISC-V、LoongArch）做了统一抽象。与 rCore-Tutorial 相比，其代码规模（48 万行）和系统调用数量（378 项）显示了对真实兼容性的追求，但部分高级特性（如实时信号排队、SCHED_DEADLINE）仍未完成，体现出“够用优先，持续完善”的演进策略。

### 真正的创新点
1. **启动流程的多架构归一化**：通过 `rust_main_boot` 和 `rust_main_secondary` 将架构差异封装在 `arch::early_init` 之后，配合 `BOOT_HART_READY` 原子标志实现主-从核同步，并引入 `wait_all_harts_online` 全核屏障 `[boot.ref:1][boot.ref:3]`。这种设计避免了为每个架构重复初始化逻辑，简化了多核启动的复杂性。
2. **Frame Poison 调试机制**：在帧分配器中加入隔离区和毒化模式，释放时填入 `0xDEAD_BEEF` 并延迟回收，分配时校验完整性，能有效捕获 use-after-free 和野指针覆盖 `[mm.ref:1][mm.ref:3]`。该机制可通过 `FRAME_POISON` 开关控制，对调试内核内存错误具有实际价值。
3. **双轨调度器与 Work-Stealing**：每个 CPU 独立运行 EEVDF（普通任务）和 FIFO（实时任务）调度器，并通过 work-stealing 平衡负载。锁序上要求调度器锁先于任务锁，避免 ABBA 死锁 `[task.ref:2][task.ref:3]`。这种设计在保持调度公平性的同时，提供了低延迟的实时任务处理能力。
4. **信号投递的优化**：通过 `mask_has_deliverable` 过滤默认忽略的信号（如 SIGCHLD），避免不必要的 EINTR 打断阻塞系统调用，提升了 netperf 等网络应用的稳定性 `[signal.ref:3]`。此外，对终止信号的目标任务临时提升调度优先级，加速 hackbench 清理。
5. **网络 Loopback 快路径**：同进程 TCP 通信绕过 smoltcp 协议栈，使用 `VecDeque<u8>` 直接交换数据，显著降低延迟和 CPU 开销 `[net.ref:5]`。延迟关闭机制则通过自适应超时和全局缓存限制，解决了 TCP_CRR 场景下的内存 OOM 问题 `[net.ref:4]`。

### 取舍判断
开发团队将主要精力投入在**用户态兼容性**和**核心子系统可靠性**上：
- **文件系统**选择了 ext4（支持 extent 树和批量读）和 tmpfs（使用物理帧避免堆 OOM），但未实现 HTree 目录索引和 LRU 缓存，大目录场景下效率较低 `[fs.ref:4]`。
- **信号机制**完整实现了 POSIX 标准投递和 SA_SIGINFO 上下文，但实时信号（32–64）不排队，可能影响依赖可靠信号的应用程序 `[signal.ref:3]`。
- **调度器**未实现 SCHED_DEADLINE 和 NUMA 亲和性，侧重公平性和实时性基础支持 `[task.ref:5]`。
- **驱动框架**采用去中心化的 trait 体系，没有统一总线模型，但通过全局注册表实现了块/网络设备的热插拔兼容 `[drivers.ref:1]`。
这些取舍表明项目以“跑通主流 Linux 负载”为首要目标，而非追求完美的架构对称性。

### 完成度与不足
整体完成度较高，核心系统调用（fork、execve、mmap、futex、read/write、socket 等）均已实现并能通过 LTP 基本测试。但仍有明显不足：多处 unsafe 块缺少 // SAFETY 注释（如 `[task.ref:4]`），帧分配器的 `try_translated_ref` 未完整实现 `[mm.不足]`，实时信号不排队可能导致语义偏差，以及文件系统缓存策略简单影响性能。代码结构清晰，模块化良好，但技术债务（如死锁修复遗留的锁序注释、未使用的常量）需要系统性清理。综合来看，该仓库在创新性和功能广度上达到了级别 2（合格），代码质量同样为级别 2（合格），距离工业级稳定尚有距离，但作为实验性内核已展现出较高的完成度和设计智慧。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 本模块的核心抽象是通过 `rust_main_boot`/`rust_main_secondary` 两个 Rust 入口函数，配合 `global_asm!` 嵌入的汇编入口 `_start`，完成从机器上电到多核调度器启动的全过程。它解决了多核异构（RISC-V 与 LoongArch）启动路径的归一化问题，将架构差异封装在 `arch::early_init` 等 trait 之后，同时通过 `BOOT_HART_READY` 原子标志实现主-从核同步。与同家族基线（如 rCore-Tutorial）相比，本实现将 BSS 清零与蹦床映射分离、显式处理块设备中断窗口，并引入了 `wait_all_harts_online` 全核就绪屏障。

### 1. 入口点与架构抽象
启动流程的硬件入口是汇编文件（如 `arch/riscv64/entry.S`）中的 `_start`，但本模块的核心控制流已上移至 Rust 函数 `rust_main_boot` 和 `rust_main_secondary` <sup>[1](#mod-boot-ref-1)</sup>。这种设计将架构差异藏在 `arch::early_init` 调用之后：`early_init` 负责设置栈指针、清零 BSS、配置页表基址等机器相关操作，而后续的通用初始化序列（mm, trap, time 等）则完全与架构无关。`global_asm!` 的使用将汇编代码直接嵌入编译单元，避免了外部链接脚本的依赖，但也要求构建时正确选择架构对应的 entry.S（通过 OUT_DIR 下的 link_app.S 动态包含）<sup>[2](#mod-boot-ref-2)</sup>。

### 2. 多核启动协同
系统支持 SMP，通过两个独立入口处理主核与从核：
- **主核**：执行完整初始化，包括 `mm::init()`、`trap::init()`、设备驱动加载、文件系统挂载等，最后设置 `BOOT_HART_READY` 标志并调用 `wait_all_harts_online()` 等待所有从核就绪 <sup>[3](#mod-boot-ref-3)</sup>。
- **从核**：在 `rust_main_secondary` 中自旋等待 `BOOT_HART_READY` 被置起，随后执行最小初始化（`init_cpu`、`activate_kernel_space`、`trap::init`），通过 `mark_hart_online` 报告就绪，最后进入调度器 <sup>[4](#mod-boot-ref-4)</sup>。
这种不对称设计避免了多核同时初始化外设导致的竞态，但代价是从核在早期完全空闲等待，浪费了部分并行能力。

### 3. 子系统初始化顺序与依赖
启动序列严格遵循依赖关系：
1. **早期架构与 CPU 初始化**：`clear_bss()`（未加锁，只有主核执行）→ `arch::early_init` → `cpu::init_cpu`。
2. **内存与陷阱**：`mm::init()` 建立内核页表后，才能安全开启中断。`trap::init()` 安装中断向量，但外部中断使能推迟到平台设备初始化之后 <sup>[5](#mod-boot-ref-5)</sup>。
3. **设备与文件系统**：通过 `arch::has_platform_devices()` 条件编译，区分 QEMU 与真实硬件。特别注意 `flush_pending()` 的插入：注释详细说明了在 `net::init_eth0()` 前消费块设备 pending ISR，以避免 VirtIO 中断丢失导致死锁 <sup>[6](#mod-boot-ref-6)</sup>。这是一个典型的中断窗口修复，体现了对硬件交互细节的把握。
4. **时钟与任务**：时间子系统必须在驱动之后初始化，因为 `init_clocksource` 可能需要读取 RTC 或 MMIO 计时器。最后才创建初始进程并进入调度器。

### 4. 边角细节与不足
- **<sup>[7](#mod-boot-ref-7)</sup> `clear_bss()` 中的 `unsafe` 块缺少 SAFETY 注释**：虽然 `sbss`/`ebss` 由链接器提供，未写明安全契约，属于技术债。
- **<sup>[3](#mod-boot-ref-3)</sup> `wait_all_harts_online()` 使用忙等待**：若从核无法启动，主核将永久自旋，缺少超时机制，在用户态可控的调试场景下可能构成 DoS 入口。
- **<sup>[8](#mod-boot-ref-8)</sup> `BOOT_HART_READY` 与 `mark_hart_online` 的原子顺序**：从核在设置 `BOOT_HART_READY` 之前调用 `mark_hart_online`，但主核的 `wait_all_harts_online` 可能依赖后者计数，未显式保证内存序，存在理论上的弱一致性风险。
- **<sup>[9](#mod-boot-ref-9)</sup> 硬编码 `config::SMP_COUNT`**：板级配置由 `boards/config.rs` 定义，无法在运行时动态探测 CPU 数量，跨平台时若修改需同时 grep 该常量。
- **<sup>[10](#mod-boot-ref-10)</sup> `arch::has_platform_devices()` 与 `has_external_interrupts()` 的默认行为**：若条件编译未覆盖的架构（如 mips）错误返回 true 而未实现对应驱动，将导致 panic。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块以 rCore-Tutorial 的 MemorySet + PageTable 双核心抽象为基础，管理进程虚拟地址空间和物理帧。特色在于：需求分页（lazy allocation）、内核栈免清零批量分配、帧分配器 BTreeSet 优化（O(log n) 释放）、frame poison 调试设施（隔离区+物理帧只读陷阱）、LoongArch64 支持（PTE 编码、musl 调度桩补丁），以及用户态内存安全访问检查（显式权限验证 + fault-in）。

### 1. 核心抽象与物理帧管理

内存管理的底层以 `PhysPageNum` 和 `PhysAddr` 表示物理地址，`PageTableEntry` 封装多级页表项（支持 RISC-V SV39 和 LoongArch64 两种编码）。物理帧通过 RAII 包装 `FrameTracker` 管理：`frame_alloc` 分配清零帧，`frame_alloc_uninit` 分配不清零帧（专供内核栈）。分配器 `StackFrameAllocator` 使用 `BTreeSet` 管理回收帧，使 `dealloc` 的 double-free 检测和回收复杂度从 O(n) 降至 O(log n) <sup>[1](#mod-mm-ref-1)</sup>。为应对碎片化，`alloc_more` 设置了 `SCAN_CAP` 阈值：当连续块扫描超过 4096 条目且 fresh 区仍有余额时，放弃扫描直接走 fresh 路径，避免了因碎片导致接近 O(n) 退化 <sup>[2](#mod-mm-ref-2)</sup>。

帧分配器还包含一套可开关的 **frame poison** 调试机制（`FRAME_POISON = false`）：释放时将整帧填入毒模式 `0xDEAD_BEEF_DEAD_BEEF`，并放入隔离区（`QUARANTINE_CAP = 4096`），延迟回收；分配时校验毒模式是否被破坏，若破坏则记录肇事帧及写入偏移，随后扫描所有任务页表找出残留叶映射，精确定位 UAF 源头 <sup>[3](#mod-mm-ref-3)</sup>。在 RISC-V 上，free 期间还会通过 `set_identity_frame_writable` 将内核 identity 映射中对应页的 PTE W 位清除，使任何内核写该帧立即触发 StorePageFault，实现“抓现行” <sup>[4](#mod-mm-ref-4)</sup>。

### 2. 虚拟地址空间与页表

每个地址空间由 `MemorySet` 表示，包含一个 `PageTable` 和多个 `MapArea`。`PageTable` 的 `map` 方法容忍重复映射（仅打 warn 不 panic），这在 `brk` 扩展中重叠映射时有用 <sup>[5](#mod-mm-ref-5)</sup>。页表支持 COW：`mark_cow` 从 `&self` 操作（通过内部可变性的 `find_pte`），不要求独占引用，便于 fork 后批量处理。`PageTable::find_leaf_mappings_to` 是一个诊断接口，扫描整个页表找出指向目标物理帧的叶 PTE，用于 UAF 调试。

`MemorySet` 提供多种插入方式：`insert_framed_area` 立即分配帧；`insert_lazy_framed_area` 仅记录区间，页面在缺页时按需分配（demand paging）；`insert_kstack_area` 使用 `frame_alloc_batch_uninit` 批量分配不清零帧，避免逐页清零的开销 <sup>[6](#mod-mm-ref-6)</sup>。`unmap_range` 和 `extend_framed_area_ending_at` 分别用于 `munmap` 和 `brk` 的收缩与扩展，其中 `unmap_range` 支持分裂和部分重叠。

### 3. 关键优化与取舍

- **内核栈免清零**：`FrameTracker::new_uninit` 跳过清零，因为内核栈内容不暴露给用户。`insert_kstack_area` 批量分配帧时使用 `frame_alloc_batch_uninit`，在 pthread 高频创建/销毁场景下显著降低开销 <sup>[6](#mod-mm-ref-6)</sup>。
- **页表 token 缓存**：通过 `CACHED_KERNEL_TOKEN` 原子变量缓存内核页表 token，避免每次需要时都加锁访问 `KERNEL_SPACE`，减少锁争用 <sup>[7](#mod-mm-ref-7)</sup>。
- **ASID 支持**：在 `asid` feature 下，`token_for_current_hart` 返回嵌入 ASID 的 token，减少 TLB 刷新。
- **LoongArch64 特有补丁**：在加载 musl 动态链接器时，通过 `patch_la_musl_sched_stubs` 将 libc 中的 ENOSYS 桩替换为真正系统调用，解决 rt-tests 因用户态桩而失败的问题 <sup>[8](#mod-mm-ref-8)</sup>。
- **用户态内存安全**：`prepare_current_user_range` 在系统调用入口处主动 fault-in lazy/COW 页，使后续无锁翻译不会缺失；`user_range_writable` 和 `user_range_readable` 逐页检查 PTE 权限，模拟硬件缺页语义，防止内核通过物理直读绕过硬件权限 <sup>[9](#mod-mm-ref-9)</sup>。

### 4. 跨模块协同

内存管理与任务模块紧密配合：`prepare_current_user_range` 通过 `crate::task::current_task()` 获取当前任务的 `MemorySet`，对其执行 fault-in 和 COW 解析 <sup>[10](#mod-mm-ref-10)</sup>。文件系统模块被用于解释器路径解析（`resolve_interp_path` 查询 `MOUNT_TABLE`）。UAF 检测 `report_uaf` 调用 `crate::task::get_all_tasks()` 遍历所有任务，扫描其页表寻找残留映射。系统调用实现（如 mmap、brk）直接调用 `MemorySet` 的插入/扩展接口。

### 5. 边角细节与不足

- `FRAME_POISON` 默认关闭，但 `frame_alloc`、`frame_alloc_uninit` 等函数中依然存在 `if FRAME_POISON { report_uaf(); }` 的条件分支，虽会被常量折叠消除，但带来一定的代码可读性开销。
- `PageTable::find_pte` 在遇到空 `ppn` 时直接 panic，没有 go back 容错，这在调试诊断页表损坏时可能导致嵌套 panic <sup>[11](#mod-mm-ref-11)</sup>。
- `try_translated_ref` 函数体不完整，仅存注释和起始代码，实际未实现，影响系统调用如 `robust_list` 的处理。
- `slab` 模块仅存文件头注释，内容被截断，可能尚未完成。
- `set_identity_frame_writable` 仅在 RISC-V 下有实现，LoongArch64 下为空函数，frame poison 的“抓现行”能力在 LoongArch 上缺失 <sup>[4](#mod-mm-ref-4)</sup>。
- `MemorySet::unmap_range` 的分裂逻辑复杂，缺乏单元测试覆盖，容易引入地址空间重叠或丢失。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块定义了统一的进程/线程模型，核心是 `Task` 结构体（类似 Linux `task_struct`），通过 `TidHandle` 和 `tgid` 区分进程与线程。调度采用每个 CPU 独立的双轨调度器：`EevdfScheduler`（替代 CFS）用于普通任务，`FIFOScheduler` 用于实时任务，支持唤醒抢占和 work-stealing 负载均衡。锁序严格（调度器锁 < task 锁），避免死锁。与 rCore-Tutorial 相比，实现了更完整的 Linux 兼容性（线程组、信号、itimer、ptrace），并针对 SMP 优化了调度器公平性和退出路径的锁粒度。

### 1. 核心抽象：统一的 Task 结构体

该模块以 `Task` 作为进程和线程的统一抽象，通过 `tgid` 和 `tid` 区分：`tgid == tid` 表示进程主线程，否则为线程。`Task` 内部使用 `SpinNoIrqLock<TaskInner>` 保护可变状态，保证中断关闭下无锁竞争。内核栈 `KernelStack`、内存集 `MemorySet` 等资源通过 `Arc` 共享，线程组使用 `ThreadGroup` 管理生命周期。

`Task` 字段布局采用 `#[repr(C)]`，确保 `on_cpu` 紧随 `kstack_sp` 之后固定偏移，以便汇编切换代码（`switch.S`）直接访问<sup>[1](#mod-task-ref-1)</sup>。这种硬编码依赖要求任何字段顺序改动必须同步更新汇编，否则导致栈帧破坏。`VmMeta` 单独管理堆、mmap 区域、sealed 区间和 trap 上下文槽位，与 `MemorySet` 分离，减少锁粒度。

### 2. 调度策略：EEVDF 与 FIFO 双轨及 work-stealing

每个 CPU 维护一对调度器：`EevdfScheduler`（普通任务）和 `FIFOScheduler`（实时任务）。任务入队时根据 `policy` 选择，`SCHED_OTHER`/`SCHED_BATCH` 使用 EEVDF，`SCHED_FIFO`/`SCHED_RR` 使用 FIFO，`SCHED_IDLE` 退化 FIFO 优先级 0，`SCHED_DEADLINE` 阶段 0 未实现，降级 EEVDF<sup>[5](#mod-task-ref-5)</sup>。

唤醒时，`sched_add_task_to_cpu` 优先选择当前 CPU（cache 亲和），仅当负载严重失衡（阈值 `WAKE_IMBALANCE_THRESH=1`）才迁移。`select_target_cpu` 遍历在线 CPU 的 EEVDF 就绪队列长度，避免锁嵌套<sup>[2](#mod-task-ref-2)</sup>。`sched_wakeup_preempt_check` 比较 VRuntime/优先级，RT 任务可抢占非 RT，EEVDF 通过 `place_entity` 判定。

时钟中断调用 `check_cfs_slice`，VRuntime 到达 deadline 时触发重调度。`sched_task_sleeping` 在任务阻塞前保存 lag，唤醒时通过 `place_entity` 恢复公平位置。注意：必须先调用 `clear_curr_if` 再计算 `avg_v`，否则 lag 计算偏差，破坏公平性<sup>[2](#mod-task-ref-2)</sup>。

空队列时 `sched_fetch_task` 执行 work-stealing，按偏移遍历其他 CPU 的调度器。该路径曾因 `while-let` 误持锁导致 ABBA 死锁，现改为 `loop + let` 释放临时锁 guard<sup>[2](#mod-task-ref-2)</sup>。

### 3. 退出流程的锁序与生命周期管理

`exit_current_and_run_next` 负责单线程退出，协调整组（`request_thread_group_exit`）和孤儿 reparent。关键设计：
- **锁序**：必须避免 `inner` 锁嵌套 `ring` 锁导致 ABBA。退出时先将 `fd_table` 移出，释放 `inner` 后再 `drop`<sup>[3](#mod-task-ref-3)</sup>。
- **dying_task**：由于 KernelStack 的 Drop 顺序（栈页先于 inner 释放），直接 drop 会导致陷阱风暴。因此将任务插入 `cpu.dying_task`，由 idle 循环在空闲栈上安全析构<sup>[3](#mod-task-ref-3)</sup>。
- **孤儿自动 reap**：孤儿进程（reparented_to_init）退出时直接从 `initproc.children` 摘除并 `unregister`，避免 zombie 累积占帧池。initproc 退出则直接 shutdown<sup>[3](#mod-task-ref-3)</sup>。
- **SIGCHLD 投递**：根据 `exit_signal` 向父进程发送信号，支持 clone3 自定义信号。若父进程是 initproc 且子进程为孤儿，则跳过 waitpid 自回收。

### 4. 边角细节与不足

1. **#[repr(C)] 硬编码偏移**<sup>[1](#mod-task-ref-1)</sup>: `on_cpu` 的汇编访问假设字段顺序，未在文档中标注完整契约，修改 `Task` 字段布局可能导致静默崩溃。
2. **锁序约定依赖注释**<sup>[2](#mod-task-ref-2)</sup><sup>[3](#mod-task-ref-3)</sup>: 调度器锁须先于 task.inner，exit 中须先释放 inner 再操作 fd_table。虽有注释，但无编译期检查，易被未来修改破坏。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块基于 `VfsNode` trait 构建 VFS 抽象层，挂载了 tmpfs（内存文件系统）、procfs（进程信息）和 ext4（磁盘文件系统）三种后端。与标准 ArceOS 相比，该版本在 tmpfs 中使用物理帧后端而非堆分配以通过 lmbench 大文件测试，在 procfs 中实现了可写的 oom_score_adj、uid_map 等节点以通过 LTP 测试，在 ext4 中通过 extent 树解析和批量读优化了 ELF 加载性能。

### 1. 核心 VFS 抽象与三种后端

文件系统模块的核心是 `VfsNode` trait（定义于 `crate::fs::vfs`），它提供了 `read`、`write`、`stat`、`read_all`、`path`、`is_dir`、`get_offset`/`set_offset`、`set_times`、`set_mode`、`set_owner`、`set_xattr`/`get_xattr`/`list_xattr`/`remove_xattr`、`read_at`/`write_at`、`mmap`、`truncate`、`file_size`、`seek`、`inode_flags`/`set_inode_flags` 等丰富的方法。所有文件系统后端——tmpfs、procfs、ext4——都实现了该 trait，并通过挂载层 `MountedFs`（`crate::fs::mount`）将路径与后端绑定，`getdents64` 通过 `MountedFs::list_dir` 列举目录项 <sup>[8](#mod-fs-ref-8)</sup>。

**tmpfs** 是内存文件系统，挂载于 `/tmp` 和 `/var/tmp`。它使用 `Arc<Mutex<FileData>>` 作为文件共享数据，每个 `open` 调用返回的 `TmpFile` 拥有独立的 `AtomicUsize` 读写偏移。目录节点 `TmpDirNode` 通过 `TmpFsInner`（一个包含 `BTreeMap` 的目录树）管理子项，支持 nlink、mode、uid/gid、timestamps 和 xattr 的持久化存储。

**procfs** 是进程信息文件系统，通过 `ProcFile`（通用只读快照）、`ProcDir`（目录）及 `OomScoreAdjFile`、`UidMapFile`、`SetGroupsFile`、`UserFanotifyLimitFile`、`MqueueQueuesMaxFile`、`IpcNextIdFile`、`MsgMniFile` 等具体节点实现只读或可读写文件。`ProcFile` 的内容在 `open` 时快照生成，`read` 内部维护原子偏移游标，避免 `cat`/`ps` 循环读死循环。

**ext4** 是磁盘文件系统，在 EXT2 基础上支持 Extent 树和 64 位块组描述符。`Ext4Fs` 结构包含块设备引用、超级块、块大小、path 缓存（BTreeMap，最多 64 条）和 inode 缓存（最多 128 条）。其核心方法 `read_file_at` 通过 `collect_all_extents` 收集所有 extent 映射，然后批量读取连续物理块，大幅减少了 fork+exec 中的扇区级 I/O 次数。

### 2. 关键设计取舍

**① tmpfs 物理帧后端** <sup>[2](#mod-fs-ref-2)</sup>
传统 tmpfs 将文件数据放在内核堆的 `Vec<u8>` 中，但在 lmbench `lat_fs` 测试中，大量文件的创建/删除导致堆碎片化和 OOM panic。该实现改用物理帧（`frame_alloc`）存储数据，使 tmpfs 容量达到约 900 MB 帧池。稀疏后端——`pages: Vec<Option<Arc<FrameTracker>>>`——允许 `ftruncate` 到很大尺寸只扩展 None 槽位（每槽 ~8 B），不焦急分配物理帧。代价是每页需通过页表转换访问，且帧分配/释放路径比堆分配稍慢。

**② procfs 偏移游标与 sendfile 兼容** <sup>[1](#mod-fs-ref-1)</sup>
`ProcFile` 的 `read` 使用 `AtomicUsize` 推进偏移，但 `VfsNode` 默认的 `get_offset` 返回 0、`set_offset` 为 no-op。这导致 `sys_sendfile` 内部调用 `in_file.get_offset()` 恒为 0，且推进无效，每次 sendfile 都从头读取，使 `cat /proc/config.gz > /tmp/x` 无限循环挂死。修复是让 `get_offset`/`set_offset` 联动 `self.offset`，使 sendfile 第二次调用返回 0（EOF）。

**③ ext4 块缓存移除与批量读** <sup>[4](#mod-fs-ref-4)</sup>
最初有块级 LRU 缓存（1024 块，4 MB），但被移除：(1) LRU 扫描在并发工作集下抖动；(2) write-back 脏块管理在多文件写时可能导致读写旧数据。替代方案是直接磁盘 I/O，但利用 `read_fs_blocks` 一次性读多个连续扇区——在 `read_file_at` 中，可合并的连续块合并为 128 块（512 KB）一批，一次 virtio 往返读完，取代旧版逐块往返（fork+exec 中每 exec ~1700 次扇区读 / ~40 ms 的瓶颈即由此优化消除）。

**④ procfs 可写节点与 LTP 兼容** [ref:5, ref:6]
LTP 测试中 `set_oom_score_adj`、`userns` 等需要向 `/proc/<pid>/oom_score_adj`、`uid_map`、`gid_map`、`setgroups` 等写入并回读。该实现添加了 `OomScoreAdjFile`、`UidMapFile`、`SetGroupsFile` 等节点，将值持久化在进程控制块或 user namespace 中。`ProcFile::write` 对可写节点有 4096 字节长度限制（防 CVE-2022-4378），但对 `ip_local_reserved_ports` 节点允许全空白大写入以清空配置。这些修改解决了旧版本只读节点导致的系统性失分。

**⑤ tmpfs MAP_SHARED 帧共享** <sup>[3](#mod-fs-ref-3)</sup>
tmpfs 的 `mmap` 方法通过 `remap_shared_frames` 将进程页表与 inode 共享的物理帧直接映射，实现跨进程 MAP_SHARED 一致性。该功能通过 `SHARED_TMPFS_MMAP` 常量（默认 true）开关，若发现回归可快速回退到 eager 拷贝路径（返回 -38 → mm.rs 做旧拷贝）。这种设计兼顾了性能与稳定。

### 3. 跨模块协同与边角细节不足

**跨模块协同**：
- procfs 通过 `crate::task::task_of(pid)` 和 `crate::task::current_process()` 获取进程信息，通过 `nsproxy.user_ns` 访问 user namespace 数据 <sup>[5](#mod-fs-ref-5)</sup>。
- tmpfs 的 `mmap` 调用 `crate::mm::remap_shared_frames` 和 `crate::mm::VirtAddr`，与内存管理模块紧密耦合 <sup>[3](#mod-fs-ref-3)</sup>。
- ext4 依赖 `crate::drivers::block::BlockDevice` 进行块设备 I/O，并使用 `crate::ipc::posix_mqueue`、`crate::ipc::sysv::msg` 等提供 sysctl 后端。

**边角细节与不足**（逐条点出，隐藏风险）：

1. **ext4 缓存简单清空无 LRU** <sup>[7](#mod-fs-ref-7)</sup>：path 缓存和 inode 缓存使用 BTreeMap，满 64/128 条即全部清空。在 read-heavy 但文件集较大的工作负载中，缓存频繁失效，无法有效加速。若文件数超过 64，每次路径遍历都需完整走磁盘目录扫描。

2. **ext4 目录索引仅线性扫描** <sup>[9](#mod-fs-ref-9)</sup>：`dir_lookup` 注释提到支持 HTree，但实际只做线性扫描。对于包含数千文件的大目录，每次查找需读取整个目录文件（可能数个块），性能退化严重。LTP 中 `openat` 测试若涉及大目录可能触发此瓶颈。

3. **procfs 偏移原子操作的内存序** <sup>[10](#mod-fs-ref-10)</sup>：`ProcFile` 和 `ProcDir` 使用 `Ordering::Relaxed` 操作 `AtomicUsize` 偏移。多线程同时读同一文件时可能导致偏移跳跃或重复读取。不过 procfs 文件通常由单线程临时读取，且内容快照一次性生成，风险可控，但仍是隐患。

4. **tmpfs TOCTOU 在 append 写路径** <sup>[11](#mod-fs-ref-11)</sup>：`TmpFile::write` 在 append 模式下先释放锁再获取 `inner.len()`（实际代码在锁内获取，但 `off = if self.append { inner.len() } else { ... }` 在持锁后执行，因此安全。但注释提到历史版本存在该问题，当前代码已修复。）

5. **ext4 unsafe 块缺少 SAFETY 注释** <sup>[12](#mod-fs-ref-12)</sup>：`write_superblock`、`write_bgd`、`read_inode`、`write_inode` 等方法大量使用 `unsafe` 进行指针转换和切片构造，却未提供 `// SAFETY:` 注释说明不变式。例如行 170-173 将 `&self.sb` 强制转换为 `*const u8` 裸指针，依赖 `Ext2Superblock` 的布局与磁盘表示一致，但未注释此前提。这是一个技术债，增加维护风险。

6. **tmpfs SHARED_TMPFS_MMAP 缺乏回归测试** <sup>[3](#mod-fs-ref-3)</sup>：该常量默认开启，但注释称“若发现回归/腐败，置 false 即回退”，暗示该功能的正确性尚未在完整 LTP 套件中充分验证。若出现竞态条件导致数据腐败，可能难以调试。

这些不足大多标记为 TODO 或早期实现痕迹，表明文件系统模块仍在积极演进中。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 核心抽象为 StellarSig（信号编号）、SigMask（u64 位图）、SigAction（handler/flags/mask）、SigHandler（64 个 SigAction 表）和 SigPending（待处理集）。解决 POSIX 信号投递、用户态 handler 调度、阻塞 syscall 中断（EINTR）及 POSIX 定时器（timer_create）问题。与 rcce-rust 家族基线相比，增加了 SA_SIGINFO 完整 ucontext/siginfo 写入以支持 glibc/musl 异步取消；引入 mask_has_deliverable 过滤默认忽略信号（SIGCHLD）避免误 EINTR（修复 netperf 服务端退出缺陷）；POSIX 定时器使用原子计数 early-out 避免每 tick 取锁抖动；send_signal_from_kill 对终止信号提升目标调度策略为 FIFO 优先级 1 以加速 hackbench 清理；init 进程提供 SIGNAL_UNKILLABLE 免疫防止被组级 kill 误杀。

### 1. 核心抽象与数据类型

信号机制围绕四个核心类型构建。`StellarSig`（`sig_def.rs:14`）是信号编号的封装，提供 `default_action()` 方法（`sig_def.rs:79-94`）将 Linux 标准信号映射为 Terminate/Core/Stop/Ignore/Continue 五类默认动作。`SigMask`（`sig_def.rs:107`）是一个 `u64` newtype，位 N 对应信号 N+1，支持 `first_unblocked` 操作（`sig_def.rs:130-136`）以最低位优先原则按 Linux 顺序返回第一个未被屏蔽的待处理信号。`SigAction`（`sig_action.rs:26-35`）内存布局与 Linux `struct sigaction` 兼容（`#[repr(C)]`），包含 `handler`（SIG_DFL=0 / SIG_IGN=1 / 用户地址）、`flags`（`SigActionFlags` 涵盖 SA_RESTART、SA_NODEFER、SA_RESETHAND 等 8 个标志位）、`restorer` 和 `mask`。`SigHandler`（`sig_action.rs:76-97`）维护 64 个 `SigAction` 的数组，作为进程级信号处理表，`exec()` 时调用 `reset_on_exec()` 将所有用户 handler 重置为默认。待处理信号集 `SigPending`（`mod.rs:44-69`）本质是 `SigMask` 的 newtype，并明确标注“标准信号(1-31)不排队，实时信号(32-64)理论上需排队但 v1 暂不支持”<sup>[3](#mod-signal-ref-3)</sup>。

用于信号帧保存的 `SigContextSaved`（`sig_dispatch.rs:48-55`）以栈式 `Vec<SigContextSaved>` 存储在 `TaskInner` 中，替代旧版单槽设计，支持嵌套信号——外层 handler 被中断时内层帧可以独立弹出恢复，外层帧仍保留在栈中。`QueuedSigInfo` 随信号携带 `si_code/si_pid/si_uid/si_value` 载荷，通过 `rt_sigqueueinfo` 或 `pidfd_send_signal` 到达后存入 `TaskInner::sig_queued_info` map 中。`SigInfo` 结构体（`sig_dispatch.rs:20-32`）明确注释修正了 64 位下 `_sifields` union 偏移（`si_pid` 在 +16 而非旧版错误的 +12），确保与 glibc/musl 布局一致。

POSIX 定时器使用 `PosixTimerEntry`（`posix_timer.rs:58-71`）记录定时器属性，全局表 `POSIX_TIMERS` 由 `SpinNoIrqLock` 保护，最大 32 个条目。`POSIX_TIMER_COUNT` 原子计数器用于快速判定是否有活跃定时器，避免每 tick 取锁。

### 2. 信号分发与栈帧设计

信号分发的入口是 `handle_signal()`（`sig_dispatch.rs:133-172`），在每次 trap_return 之前调用。它在循环中每次只分发**一个**用户态 handler 信号：先通过 `check_pending_signal()` 找到当前 mask 下第一个未屏蔽的 pending 信号，调用 `dispatch_signal()`；若该信号配置了用户 handler，`setup_user_handler()` 会修改 `trap_cx` 使其 sepc 指向 handler、参数 a0/a1/a2 置为信号号及 siginfo/ucontext 地址、ra 指向 `SIGRETURN_TRAMPOLINE`（`sig_dispatch.rs:360`），然后必须立即返回用户态让 handler 执行；handler 返回时执行 `rt_sigreturn` 系统调用，`do_sigreturn()` 从栈顶弹出 `SigContextSaved` 恢复原始的 trap_cx 和 mask。内核不直接在 handler 返回后继续分发剩余信号，而是等待下一次 trap_return 再次进入 `handle_signal()`<sup>[4](#mod-signal-ref-4)</sup>。这种“一次一个”设计简化了信号重入问题，但代价是链式信号的整体延迟增加。

栈式 `sig_context` 的设计突破了传统单槽的限制：PUSH 操作在 `setup_user_handler` 末尾（`sig_dispatch.rs:312-316`）将当前 trap_cx 及 mask 压入 `Vec<SigContextSaved>`，POP 操作在 `do_sigreturn`（`sig_dispatch.rs:326-350`）中弹出。若 sigreturn 时还有外层帧存在（例如信号 handler 执行期间被更优先的信号中断），恢复的是被中断时的上下文，外层信号会在后续 trap_return 中继续分发。这避免了丢失外层信号帧的问题。

`SA_SIGINFO` 标志启用时，`setup_user_handler` 会在用户栈上依次分配 ucontext（1024 字节，含完整 GPR/FP 寄存器状态）和 siginfo_t（128 字节）。`write_native_mcontext`（`sig_dispatch.rs:620-655`）通过 `translated_refmut` 将 `TrapContext` 中的 32 个通用寄存器、浮点寄存器和 fcsr 逐一写入用户空间；该函数用 `#[cfg(target_arch = "riscv64")]` 条件编译，非 RISC-V 平台仅写入 PC 字段（`sig_dispatch.rs:658-664`）<sup>[11](#mod-signal-ref-11)</sup>。`do_sigreturn` 会从 ucontext+40 读取 `uc_sigmask`、从 ucontext+176 读取 `mc_pc`（可能被 async cancel handler 修改为重定向地址），吸收 libc 的线程取消需求。

### 3. 跨模块协同与性能优化

信号模块与任务（Task）、陷阱（Trap）、定时器（Timer）及系统调用子系统紧密协作。`TaskInner` 中暴露给信号使用的字段包括 `sig_pending`（SigPending）、`sig_mask`（SigMask）、`sig_handler`（Arc<SpinNoIrqLock<SigHandler>>）、`sig_context`（Vec<SigContextSaved>）、`sig_queued_info`（BTreeMap）、以及 itimer 相关字段。`handle_signal()` 被 `trap_return` 调用（在 arch 层）；`do_sigreturn()` 被 `rt_sigreturn` syscall 调用。`raise_expired_itimer()` 和 `deliver_itimer()` 处理 ITIMER_REAL 到期，前者供 socket 收发循环快路径调用，后者由 `check_timer` 在定时器堆 pop 时异步投递，两者互补避免阻塞 syscall 中 SIGALRM 永不产生。`check_posix_timers()`（`posix_timer.rs:230-277`）在每 10ms 周期时钟中断中调用，遍历全局 POSIX 定时器表，收集到期动作后释放锁再发送信号，防止锁嵌套<sup>[10](#mod-signal-ref-10)</sup>。

性能优化方面有三处突出设计。其一，`mask_has_deliverable`（`sig_dispatch.rs:93-109`）在判断是否打断阻塞 syscall 时，会进一步检查目标信号是否被默认忽略（SIGCHLD、SIGURG、SIGWINCH）或显式设为 SIG_IGN，仅当信号可“递送”（即真正需要用户 handler 或默认终止/停止）时才返回 true，避免 netserver 的 select 被无害 SIGCHLD 错误返回 EINTR 导致服务端退出<sup>[5](#mod-signal-ref-5)</sup>。其二，`POSIX_TIMER_COUNT` 原子计数（`posix_timer.rs:25`）在大部分无定时器进程的场景中让 `check_posix_timers` 和 `next_posix_timer_ns` 零开销 early-out，避免每 tick 取 `SpinNoIrqLock` 带来的固定延迟抖动（注释指出该抖动曾导致 lmbench 比例校准退化到 1 秒，整组测试变慢 40 倍）<sup>[9](#mod-signal-ref-9)</sup>。其三，`send_signal_from_kill_info`（`sig_dispatch.rs:740-775`）在检测到发送信号是致命/默认终止信号（SIGKILL 或默认动作为 Terminate/Core 且未被屏蔽）时，立即将目标任务的调度策略提升为 SCHED_FIFO 优先级 1，并在目标为 Ready 状态时执行 `sched_remove_task` + `sched_add_task` 强制重新入队（甚至从 EEVDF 迁移到 FIFO 队列），确保被 kill 的进程快速被调度执行退出逻辑。对 Stopped 状态的目标，`mark_stopped_sigkill_ready` 负责将其状态改为 Ready 并加入运行队列<sup>[6](#mod-signal-ref-6)</sup>。

### 4. 边角细节与不足

**实时信号不排队**<sup>[3](#mod-signal-ref-3)</sup>：`SigPending` 注释明确承认“实时信号（32-64）理论上需排队但 v1 暂不支持”。当前同号实时信号多次投递只记录一次，且在 `sig_queued_info` 中也仅保留最新载荷。这违反 POSIX 要求，可能影响依赖实时信号排队的应用。

**架构假设**：`write_native_mcontext` 的 RISC-V 分支硬编码了通用寄存器、浮点寄存器在 ucontext 中的偏移（`UCONTEXT_MCONTEXT_OFFSET = 176` 等），非 RISC-V 平台仅写入 PC 字段，其余寄存器为零（`sig_dispatch.rs:658-664`）<sup>[11](#mod-signal-ref-11)</sup>。若移植到 x86_64，glibc 的 ucontext 布局完全不同（需包含 xstate 扩展区），当前 fallback 路径会导出零寄存器，导致 libc unwind 近 NULL 崩溃。

**Unsafe 块安全契约缺失**：`write_ucontext` 和 `write_native_mcontext` 中大量使用 `translated_refmut`，但相关函数均未标注 `// SAFETY:` 注释（`sig_dispatch.rs:615-665`）。若用户提供的 siginfo/ucontext 地址指向未映射区域或只读页，`translated_refmut` 可能在 unsafe 块内触发缺页异常导致内核 panic（取决于该函数的实现，若其内部使用 `*mut` 写访问而不检查有效性）。

**资源泄漏风险防护**：`dispatch_signal` 的注释（`sig_dispatch.rs:236-238`）详细解释了为什么绝不能在调用可能退出任务的动作（`do_default_action` → `exit_current_and_run_next`）时仍在栈帧持有 `Arc<Task>`：否则已退出任务的 KernelStack 永远不会 drop 导致帧池被耗尽。当前代码通过独立作用域在进入分支前释放 Arc，规避了该问题<sup>[8](#mod-signal-ref-8)</sup>。

**初始化序列依赖**：`SIGRETURN_TRAMPOLINE` 由 `crate::config` 模块定义，但 `config.rs` 不可读。若该跳板页未正确映射或映射权限不含执行位，信号 handler 返回时会触发段错误。

**锁粒度**：`send_signal` 函数族在 `task.inner.exclusive_access()` 持锁期间调用 `wakeup_task`（可能触发调度），但 `wakeup_task` 不获取 task inner 锁（它操作运行队列锁），所以不会死锁。但 `send_signal_from_kill_info` 中持 inner 锁时调用了 `sched_remove_task` 和 `sched_add_task`，这些函数内部会获取调度器锁，符合锁顺序（inner 锁在外，调度锁在内），不会死锁。

**Ptrace 集成**：`handle_signal` 开头调用了 `check_ptrace_stop_needed` 和 `ptrace_signal_stop`，实现 ptrace signal-delivery-stop 语义。但当前实现仅在分发前做一次 stop 点，未支持 PTRACE_O_TRACESYSGOOD 等更细粒度的追踪事件。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现多种进程间通信机制：System V 信号量（semget/semop/semctl）、futex 快速用户空间互斥量、以及通过 Unix domain socket DGRAM 传递文件描述符（SCM_RIGHTS）。核心抽象包括 FutexKey、FutexQ、SemSet、SemManager 等。与同家族基线（如 StarryX）相比，本实现修复了 futex 超时精度、requeue 返回值等细节，并将 System V 信号量的忙等替换为真正睡眠等待，大幅提升了兼容性和正确性。

### 1. 核心抽象与外部依赖

进程间通信模块涵盖三种主要机制：futex、System V 信号量和 Unix domain socket 文件描述符传递。

- **Futex** 的核心抽象是 `FutexKey`，它唯一标识一个用户空间地址的等待队列。键的设计区分私有 futex（以进程内存描述符地址 + 虚拟页为命名空间）和共享 futex（以物理页为命名空间），使得映射同一物理页的不同进程线程能够互相唤醒。等待队列实现为全局哈希桶 `FUTEX_QUEUES`，每个桶由 `SpinNoIrqLock` 保护<sup>[1](#mod-ipc-ref-1)</sup>。
- **System V 信号量** 的核心抽象是 `SemSet`，包含一组信号量 `SemUnit`、IPC 权限、等待计数和 `WaitQueue`。全局管理器 `SemManager` 以 semid 索引，并与 IPC 命名空间绑定以支持 `CLONE_NEWIPC` 隔离<sup>[2](#mod-ipc-ref-2)</sup>。
- **Unix domain socket DGRAM** 的 SCM_RIGHTS 通过在 `sys_sendto` 和 `sys_recvfrom` 中处理 AF_UNIX 路径实现；辅助数据解析在 `configure_alg_cmsgs` 中（主要用于 ALG 套接字）<sup>[3](#mod-ipc-ref-3)</sup>。

外部依赖方面，futex 依赖任务调度（`block_current_and_run_next`、`wakeup_task_urgent`）、信号检查和定时器；System V 信号量依赖 `WaitQueue` 同步原语和 IPC 命名空间；Unix socket 依赖网络栈的 unix 模块。

### 2. 关键设计取舍

#### 2.1 Futex 超时处理

futex 的超时处理是调试最密集的部分。原实现曾将 `FUTEX_WAIT_BITSET` 的绝对超时误判为相对，导致 `pthread_cond_timedwait` 永不超时。现在 `do_futex` 中区分 `FUTEX_WAIT`（相对）和 `FUTEX_WAIT_BITSET`（绝对），根据 `FUTEX_CLOCK_REALTIME` 选择时钟基准，统一换算为单调纳秒截止时间。

等待循环被唤醒后先检查信号中断（返回 -EINTR），再检查线程组退出（显式返回 -EINTR 以便 `trap_return` 处理组退出），最后比较当前时间与截止时间：过期则返回 -ETIMEDOUT；否则重新入队等待，避免虚假唤醒导致提前返回<sup>[4](#mod-ipc-ref-4)</sup>。这种设计放弃简化处理，选择纳秒精度和重睡循环，以确保 LTP futex 测试通过。

#### 2.2 System V 信号量的阻塞替代

参考实现 StarryX 使用忙等（`core::hint::spin_loop`），这在单核下会导致死锁。本实现改为使用 `WaitQueue::wait_interruptible` 实现真睡眠，支持信号可中断和超时；为弥补可能的丢失唤醒，加入 5ms 轮询兜底。`WaitQueue` 唤醒调用 `wake_all`，但不在持信号量管理器锁时调用，避免 ABBA 死锁。权限校验完整移植 Linux `ipcperms()`，支持 root 绕过<sup>[5](#mod-ipc-ref-5)</sup>。这种设计的代价是增加了复杂性，且原子性通过先在临时副本模拟 `can_perform` 再执行确保，但操作集合间没有内核级锁保护用户地址，依赖用户态正确性。

#### 2.3 Unix domain socket DGRAM 发送

在 `sys_sendto` 中，对于 AF_UNIX 套接字，当目的地址非空时，读取 `sockaddr_un` 路径，调用 `unix.dgram_send`。TCP 连接状态下虽然 `dest_addr` 非空，但路径被忽略；DGRAM 未连接时根据路径路由。这种实现对齐 Linux 行为，但未实现 STREAM 模式的路径解析（已在 TODO 中？实际代码中 STREAM 已连接则走普通 send）<sup>[6](#mod-ipc-ref-6)</sup>。

### 3. 跨模块协同

- **Futex 与进程退出**：`exit_robust_list` 在线程退出时处理 robust futex 列表，确保挂起的等待者被唤醒。`futex_wake_from_task` 被 `CLONE_CHILD_CLEARTID` 路径调用，使用物理页和虚拟页两种 key 唤醒，兼顾共享和私有 futex<sup>[7](#mod-ipc-ref-7)</sup>。
- **System V 信号量的 SEM_UNDO**：在进程退出时通过 `clear_proc_sem` 钩子回滚操作。这需要 `task::exit` 调用 Manager 的清理函数，但现有代码中该钩子尚未挂接，可能仍为 TODO<sup>[8](#mod-ipc-ref-8)</sup>。

### 4. 边角细节与不足

- **`futex_get_value` 可能引发内核恐慌**：该函数直接调用 `*translated_ref(token, uaddr as *const u32)`，若输入地址无效且 `translated_ref` 内部 panic，则用户可通过非法地址造成内核 DoS。应使用 `try_translated_ref` 并返回 EFAULT<sup>[9](#mod-ipc-ref-9)</sup>。
- **`unsafe` 块缺乏安全说明**：`configure_alg_cmsgs` 中 `unsafe { core::ptr::read_unaligned }` 虽已在调用前有 `user_readable` 检查，但未提供 `// SAFETY:` 注释，是技术债务<sup>[10](#mod-ipc-ref-10)</sup>。
- **`current_task().unwrap()` 上下文假设**：多个地方（如 `futex_wait`、`cur_pid`）直接 unwrap，若在中断路径误调将导致内核 panic。应使用 `current_task()` 的 Option 版本并优雅处理<sup>[11](#mod-ipc-ref-11)</sup>。
- **`WaitQueue` 与中断锁配合风险**：`WaitQueue::wait_interruptible` 是否依赖中断使能？在单核下，`SpinNoIrqLock` 会关闭本地中断，而 `WaitQueue` 的等待可能需要开中断；但 futex 在释放 bucket 锁后才调用 `block_current_and_run_next`，sysv sem 在释放管理器锁后调用 `wq.wait_interruptible`，故目前安全，但须小心未来改动<sup>[12](#mod-ipc-ref-12)</sup>。
- **SEM_UNDO 清理钩子未接入**：`clear_proc_sem` 函数虽在 sem.rs 中定义（未显示），但 `task::exit` 并未调用，进程退出时 undo 记录不会被回滚，导致信号量状态不一致<sup>[13](#mod-ipc-ref-13)</sup>。

综上所述，本模块的 IPC 机制力求 Linux 兼容，修复了多个 LTP 测试失败的问题，但存在若干不安全代码和潜在 panic 路径需要加固。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块以 smoltcp 为协议栈内核，通过 SocketFile（实现 VfsNode + Socket trait）桥接 fd_table 与协议处理。核心抽象是 SocketInner 枚举（TCP/UDP/Raw/Unix），由 Socket trait 统一分发 syscall 请求。与同家族基线（ArceOS）相比，本仓独特之处包括：(1) 同进程 loopback TCP 快路径（FastTcpConn），绕过 smoltcp 全栈处理；(2) 延迟关闭机制（DEFERRED_CLOSES）配合自适应 SMP 超时，解决 TCP_CRR 高 churn 场景下的内存 OOM；(3) 纯用户态 netlink 路由仿真，支持 ip link/addr/route 的只读查询与动态修改。

### 1. 核心抽象：SocketFile 与 SocketInner 的分发架构

网络模块的核心抽象是 `SocketFile`（`os/src/net/core/socket.rs:150`），它同时实现 `VfsNode` trait（放入进程 fd_table）和 `Socket` trait（syscall 层直接调用）。`SocketFile` 内部持有一个 `Mutex<Arc<SocketInner>>`，`SocketInner` 是枚举体，包含 `Tcp`, `Udp`, `Raw`, `Unix` 四种变体 <sup>[1](#mod-net-ref-1)</sup>。这种“文件即 socket”的设计沿袭 Unix “一切皆文件”哲学，但与标准 Linux 不同——Linux 的 `struct socket` 和 `struct file` 是分离对象，通过 `sock_map_fd` 关联；本仓合并为单一结构，简化了引用计数和生命周期管理，代价是 `SocketInner` 的 `Mutex` 锁粒度覆盖整个协议类型切换（例如 `setsockopt` 和 `send` 都锁同一个 inner）。

`Socket` trait（`os/src/net/core/socket.rs:850` 附近）定义了 `bind`, `connect`, `send`, `recv`, `accept`, `shutdown` 等方法，每个方法内部先 clone 出 `Arc<SocketInner>` 快照再释放锁，避免持锁进入阻塞等待 <sup>[2](#mod-net-ref-2)</sup>——这是在 hackbench AF_UNIX socketpair 上实证过的死锁修复。`SocketFile` 还持有一个 per-socket `WaitQueue`（类似 Solaris squeue / Linux `sock->sk_wq`），创建时通过 `register_socket_waker` 注册到全局 `SOCKET_WAITERS`，`poll_interfaces()` 在 `iface.poll()` 后遍历唤醒所有阻塞任务 <sup>[3](#mod-net-ref-3)</sup>。

### 2. 关键设计取舍：延迟关闭、loopback 快路径与 interior mutability

**延迟关闭（Deferred Close）**：`tcp.rs` 的 `DEFERRED_CLOSES` 静态列表（行 38）记录已调用 `close()` 但尚未完成 FIN 握手的 socket。`cleanup_deferred_closes()`（行 122）在 `poll_interfaces()` 路径中被周期性调用，根据 socket 状态（TimeWait/Closed vs 其他）和 SMP 缩放因子自适应判断何时安全移除 <sup>[4](#mod-net-ref-4)</sup>。设计取舍在于：正常 iperf 场景下延迟关闭 socket 屈指可数，但 connect02 等测例在紧循环里建/关 1000+ socket，若不加 `MAX_DEFERRED_CLOSES=128` 上限（行 52），缓冲堆积会导致 128MB+ 堆 OOM panic。该上限在病态 churn 时强制移除最旧 socket（释放缓冲），仅影响异常路径。

**同进程 loopback 快路径（FastTcpConn）**：`tcp.rs` 的 `FastTcpConn`（行 64）为同一进程内的 loopback TCP 连接提供了绕开 smoltcp 协议栈的 shortcut——数据直接通过 `VecDeque<u8>` 交换，无需经过 smoltcp 的 socket 状态机和 IP 层封装 <sup>[5](#mod-net-ref-5)</sup>。`accept()` 路径上优先检查 `FAST_TCP_PENDING` 队列（行 770），server 端通过 `process_already_connected_to()` 区分控制连接与数据连接，避免在 `-P N` 并发场景下死等自己。该快路径的实现代价是增加了 `TcpSocket` 的 `fast: Mutex<Option<Arc<FastTcpSide>>>` 字段和维护两套 send/recv 路径的条件分支。

**Interior Mutability 与 `unsafe impl Sync`**：`TcpSocket`（行 246）大量使用 `Cell`（`handle`, `local_endpoint`, `recv_timeout_ms`, `hop_limit`）配合 `unsafe impl Sync` <sup>[6](#mod-net-ref-6)</sup>。安全契约是：`Cell` 写入仅发生在建立期（new/bind/setsockopt）或被全局 `SOCKETS` 锁串行化；热路径并发字段已使用 `AtomicBool`/`AtomicU32`。这比直接将 `&self` 强转为 `*mut Self`（旧版 rocketos 的做法）更安全，但仍需开发者手动维护契约——若未来某天在 `with_socket_mut` 外部写入 `Cell` 字段，将触发 TSO（Type Safety Violation）。

### 3. 跨模块协同：listen_table、signal 与 poll 调度

`LISTEN_TABLE`（`tcp.rs` 引用 `crate::net::listen_table`）是 TCP 监听表的核心协同点。`TcpSocket::listen()` 调用 `LISTEN_TABLE.lock().listen(port)` 注册，`accept()` 则通过 `listen_table.accept(port)` 取出已握手待 accept 的 socket handle <sup>[7](#mod-net-ref-7)</sup>。`backlog_handles(port)` 扫描预创建的 backlog sockets（`replenish_backlog()` 在 listen 时预分配多个 socket 以应对并发 SYN，避免 PARALLEL_TCP -P5 失败）。

信号中断机制跨 `tcp.rs` 与 `signal` 模块：`signal_interrupted()`（行 215）在每次阻塞等待后调用，先 `raise_expired_itimer()` 把到期的 ITIMER_REAL 转成 SIGALRM pending，再检查未屏蔽待处理信号。其变体 `signal_interrupted_strict()`（行 237）在 accept/connect 阻塞路径使用，不屏蔽已消费 itimer 残留的 SIGALRM——因为 netperf TCP_CRR 的 netserver 靠 watchdog alarm 的 SIGALRM 中断 accept 才能结束测试组，若被抑制会导致死锁 <sup>[8](#mod-net-ref-8)</sup>。

`poll_interfaces()` 是整个网络模块的调度引擎：它驱动 smoltcp `iface.poll()`、清理延迟关闭、唤醒 `SOCKET_WAITERS` 中所有任务。它在 `tcp.rs` 的每一处阻塞操作中被调用（yield_or_spin, yield_loopback_peer 等），同时在 syscall 返回路径上也由 syscall 层调用。

### 4. 边角细节与不足

<sup>[1](#mod-net-ref-1)</sup> `SocketInner` 枚举定义在 `os/src/net/core/socket.rs:55`，四个变体共享同一锁，UDP/Datagram 场景下 `sendto` 和 `recvfrom` 的锁开销不可忽略。

<sup>[2](#mod-net-ref-2)</sup> `SocketFile::send()` 行 1300 先 clone Arc 再释放 inner 锁的设计，解决了 hackbench 死锁，但每次 syscall 都多一次 Arc clone（原子引用计数增减 + 堆分配）。

<sup>[3](#mod-net-ref-3)</sup> `register_socket_waker` 使用全局原子 `NEXT_ID` 生成 waker ID（`socket.rs:175`），未复用已注销 ID，长期运行可能溢出（尽管 `AtomicUsize` 在 64 位平台几乎不可能）。

<sup>[4](#mod-net-ref-4)</sup> `cleanup_deferred_closes()` 的 SMP 缩放因子 `smp_scale * 4`（行 130）是启发式值，未考虑实际 hart 负载——若某 hart 被抢占，计数器增速不均匀，可能导致 TimeWait socket 过早移除或过晚残留。

<sup>[5](#mod-net-ref-5)</sup> `FastTcpConn` 的 `c2s`/`s2c` 队列类型为 `VecDeque<u8>`，每次拷贝一字节，大块数据传输时 cache 效率低于 smoltcp 的 buffer slice 方案。

<sup>[6](#mod-net-ref-6)</sup> `unsafe impl Sync for TcpSocket`（行 266）缺少 `// SAFETY:` 注释，是技术债——评审者需人工审计所有 `Cell` 写入点是否满足并发安全。

<sup>[7](#mod-net-ref-7)</sup> `LISTEN_TABLE.lock()` 在 `accept()` 路径上先后持 listen_table 锁和 SOCKETS 锁（通过 `with_socket`），可能构成锁顺序反转——当前代码通过 try_with_socket 避免 panic，但若被移除 handle 仍可能空转。

<sup>[8](#mod-net-ref-8)</sup> `signal_interrupted_strict()` 与 `signal_interrupted()` 的区分在行 237-243 的注释中明确说明，但这是手动维护的约定，缺乏编译期检查——未来重构若误用 `signal_interrupted` 替代 `signal_interrupted_strict`，netperf TCP_CRR 会偶发 hang。

<sup>[9](#mod-net-ref-9)</sup> `route_netlink.rs` 的 `NETLINK_READ_BUF_LIMIT = 1 << 20`（行 123）是硬编码上限，`recv_reply` 若真实回复超过此限制会静默截断（`out.len().min(avail)`），且 `report_full_len` 模式返回完整长度但不会分片重传。

<sup>[10](#mod-net-ref-10)</sup> `SocketFile::as_unix()`（行 386）返回 `&UnixSocket` 引用，依赖 unsafe 和“调用方保证不 replace_inner”的契约，但 `SocketFile` 的 `replace_inner` 方法是 pub(crate) 可访问的——若某天 syscall 层在持有 as_unix 引用的同时调用了 replace_inner，会触发 UAF。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该仓库的驱动框架是一个去中心化的设备驱动集合，没有统一的驱动框架 crate 或 bus 模型，而是通过三个正交抽象（BlockDevice trait、NetDevice trait、VfsNode trait）将不同类别的设备暴露给内核其余部分。块设备和网络设备各维护全局注册表（BLOCK_DEVICE Mutex<Option> / NET_DEVICES BTreeMap），而字符类设备（pty、loop）直接实现 VfsNode 挂接进伪文件系统。与 rCore-Tutorial 相比，核心差异在于：(1) 驱动设备通过 VfsNode 对用户态可见，无需单独的设备文件系统；(2) 引入了 loop device 桥接文件→块设备；(3) Net 驱动使用 smoltcp 协议栈 + 全局注册表而非单例。

### 1. 核心抽象：三层 trait 体系与设备模型

驱动框架没有采用 Linux 式的 bus-driver-device 模型或 ArceOS 的 `Driver` enum，而是由三个独立的 trait 构成契约层。

**`BlockDevice` trait**（`os/src/drivers/block/mod.rs:135-143`）是块设备最底层抽象，定义 `read_block`/`write_block`/`block_size`。其关键设计是 `Send + Sync` 约束——这意味着所有块设备实现必须能安全跨线程共享，因此 `VirtIOBlkDevice` 内部通过全局 `BLOCK_DEVICE` Mutex 序列化访问，而 `PartitionDevice` 仅做偏移转发，无锁开销 `<sup>[1](#mod-drivers-ref-1)</sup>`。`PartitionDevice` 的引入使得文件系统可以像操作原始分区一样操作 GPT/MBR 子分区，内核对分区表完全无感知。

**`NetDevice` trait**（定义在 `net/net_device.rs`，从 `net/mod.rs` 的引用可推断其接口包括 `transmit`/`receive`/`mac_address`/`capabilities`）提供了网络设备的统一面。与块设备的单例不同，网络设备使用 `BTreeMap` 注册表支持多设备（`eth0`/`eth1`...）`<sup>[2](#mod-drivers-ref-2)</sup>`。

**`VfsNode` trait**（来自 `crate::fs::vfs`）是驱动向用户态暴露的桥梁。`LoopDevice` 同时实现了 `BlockDevice` 和 `VfsNode` `<sup>[3](#mod-drivers-ref-3)</sup>`——这在族基线中罕见：rCore-Tutorial 的块设备通常仅通过内部接口暴露，而此处 loop 设备在 `/dev/loopN` 处同时支持 `read`/`write` 系统调用（通过 VfsNode）和文件系统的 `read_block`（通过 BlockDevice）。PTY 的 master/slave 两端也都实现 VfsNode，从而使用户态可通过标准的 `read`/`write`/`ioctl` 操作终端。

这三层 trait 没有统一的父 trait 或类型枚举，意味着驱动类型之间无法多态互转——loop 设备从 VfsNode 到 BlockDevice 的转换是显式的 `Arc<dyn BlockDevice>` 向下转型，依赖调用方知道具体类型。

### 2. 关键设计取舍

**同步 vs 异步路径**：`block/mod.rs:3-15` 明确选择了同步路径。virtio-drivers v0.7 提供了异步接口（`read_blocks_nb` + `complete_read_blocks`），但实现方需要持有 `BlkReq`/`BlkResp` 直到完成，状态管理复杂。当前实现走同步 `read_blocks`，由 virtio-drivers 内部完成 add→notify→wait→pop_used 的完整循环。代价是每次 I/O 都会阻塞当前核，且中断处理函数 (`handle_irq`) 仅做 `ack_interrupt()` 清设备 IRQ 标志，不做实际的数据搬运——这意味着 PLIC 的中断线在 I/O 期间可能反复触发，但 `try_lock` 失败时直接丢弃 `<sup>[4](#mod-drivers-ref-4)</sup>`。这个取舍在单核场景下合理，但在 SMP 下其他核可能因持锁而错过中断消费。

**VirtIO HAL 的地址翻译策略**：`virtio/hal.rs:1-20` 描述了恒等映射优先策略——内核堆和数据段位于 `[0, MEMORY_END)` 恒等映射区，`translate_kernel_va` 直接返回 `va` 作为物理地址（零拷贝）。仅当缓冲区落在 Framed 映射区（内核栈）且跨页时才走 bounce buffer 路径。这比 RocketOS 那种对所有 DMA 都做 bounce buffer 的方案减少了两次拷贝。但 `is_phys_contiguous`（`hal.rs:95-101`）对跨页栈缓冲区的处理是直接 panic 而非自动 fallback——虽然注释说“提示调用方改用堆缓冲区”，但 panic 在生产环境中不可恢复 `<sup>[9](#mod-drivers-ref-9)</sup>`。

**全局注册表的渐进迁移**：`net/mod.rs:96-110` 同时存在新旧两套注册表——`NET_DEVICE: Mutex<Option<VirtIONetDev>>`（旧单例）和 `NET_DEVICES: BTreeMap<String, Arc<Mutex<dyn NetDevice>>>`（新多设备）。`probe_net` 仍将设备写入旧注册表，而 smoltcp 适配层的 `DeviceWrapper` 也直接操作旧单例。新注册表目前仅提供 `register_net_device`/`get_net_device` 接口但尚无消费方。这种渐进式迁移在工程上合理，但新旧并存期间存在状态分裂：若两个路径同时操作不同设备，流量统计（`RX_PACKETS`/`TX_PACKETS`）只统计了经 `DeviceWrapper` 的路径。

**PTY 锁纪律**：`pty.rs:17-19` 明确规定了“队列锁内不阻塞、不发信号——信号在放锁后发”。`signal_fg` 函数（`pty.rs:148-155`）在释放 `input`/`output` 锁后调用，避免锁内发信号导致信号处理函数反向获取队列锁造成死锁。这是 socket-locking 同则在内核终端子系统的体现，与 Linux 的“锁外发信号”原则一致。

### 3. 跨模块协同

**VirtIO 生态的 HAL 依赖链**：`VirtioHal`（`hal.rs`）实现了 `virtio-drivers` crate 的 `Hal` trait，被 `block/mod.rs`、`net/mod.rs`、`rng/mod.rs` 中的 VirtIO 设备初始化时作为类型参数传入。三个设备子模块共享同一个 `VirtioHal` 实现，但各自维护独立的全局状态（`BLOCK_DEVICE`、`NET_DEVICE`、`VIRTIO_RNG`）。HAL 的 `dma_alloc` 分配的内存在恒等映射区，`share` 将内核 VA 转为设备可见的 PA——这套机制使得 virtio-drivers 无需感知内核页表结构。

**LoopDevice 的双重身份与路径分发**：`loop_device.rs` 的 `LoopDevice` 同时实现 `VfsNode` 和 `BlockDevice`。用户态通过 `/dev/loopN` 发起 `read`/`write` 系统调用时走 VfsNode 路径，经过 `UserBuffer` → 内核缓冲区的拷贝；文件系统（如 EXT2）通过 `BlockDevice::read_block` 访问时走内核态路径，不经过用户态拷贝。两者最终都调用 `read_at_kernel`/`write_at_kernel`，但 VfsNode 路径多了一层 `offset` 原子变量维护文件指针。`block_device_from_path` 函数（`loop_device.rs:615-622`）提供了从路径到 BlockDevice trait 对象的转换，使得文件系统可以挂载 loop 设备上的文件系统镜像。

**中断处理的分散模型**：与 Linux 的统一 IRQ 子系统不同，块设备和网络设备各自定义了 `handle_irq` 函数，由上层 `trap/mod.rs` 根据中断源分别调用。`block::handle_irq` 仅做 `ack_interrupt()`，`net::handle_irq` 类似。PLIC 的 `complete` 操作由 trap 层统一完成，这意味着设备驱动不持有 PLIC 的上下文，中断处理是半拉子的——只清设备级标志，不通知等待队列。对于同步路径这没问题（等待在 `read_blocks` 内部的自旋中），但未来要实现真正的异步 I/O，需要重构中断处理模型。

### 4. 边角细节与不足

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层采用函数式模块化设计，每个系统调用对应一个独立函数（如 sys_write/sys_read），通过 Arc<dyn VfsNode> 抽象文件对象、socket 对象则经 SocketFile 分发。核心解决了用户态指针安全校验、文件描述符表管理、路径解析挂载表分发三大问题。与同家族 rCore-Tutorial 基线相比，本仓加入了 RLIMIT_FSIZE 截断式限写、O_PATH fd 零分配标记、loopback TCP drain_after_close 同步排空、lib_fallback_path 库路径回退、以及大量 LTP 测试对齐的精细错误码优先级编排。

### 1. 核心抽象：fd_table + Arc<dyn VfsNode> 与 SocketFile 双模分发

系统调用层没有引入统一的 Syscall trait，而是采用 C 风格函数导出——每个系统调用一个 `pub fn sys_xxx(...) -> isize`。文件描述符表 `fd_table: Vec<Option<Arc<dyn VfsNode + Send + Sync>>>` 是进程内文件引用的中心索引 <sup>[1](#mod-syscall-ref-1)</sup>。对于网络 socket，特化为 `SocketFile`（是 VfsNode 的一种实现），同时在 sendto/recvfrom 内部通过 `get_socket` / `get_socket_file` 区分 smoltcp socket 层与 VfsNode 封装层 <sup>[2](#mod-syscall-ref-2)</sup>。这种双模设计源于网络栈的分层架构：smoltcp 提供无锁协议实现，而 SocketFile 提供 VfsNode 接口供 epoll/io_uring 等统一事件框架使用。

用户态内存访问封装在 `try_translated_ref` / `translated_byte_buffer` / `UserBuffer` 三层中。`try_translated_ref` 做精确的单指针校验，返回 None 表示 EFAULT；`translated_byte_buffer` 负责跨页 buffer 的物理页组装，但它对坏指针会静默走 DUMMY_PAGE，因此所有系统调用入口都先做一次 `try_translated_ref` 显式校验 <sup>[3](#mod-syscall-ref-3)</sup>。这一模式被设计者称为“坏指针铁律”，在 read/write/openat/pipe2 等 8 个 syscall 中均有体现。

### 2. 关键设计取舍：错误优先级、资源限制与死锁预防

**错误码优先级严格对齐 Linux 内核。** `sys_dup3` 内部按 `flags → oldfd==newfd → oldfd 有效性` 的顺序返回 EINVAL/EINVAL/EBADF <sup>[4](#mod-syscall-ref-4)</sup>，与 Linux `ksys_dup3` 完全一致，确保 LTP dup3_02 等用例通过。`sys_sendto` 的错误顺序为 `EBADF/ENOTSOCK → dest_addr EFAULT → addrlen EINVAL`。`sys_faccessat2` 对非法 flag 立即返回 EINVAL，且检查早于路径读取。这种编排不是偶然，每一处都有 LTP 测例名称注释（如“faccessat02 的负向测点”）。

**RLIMIT_FSIZE 截断而非报错。** `sys_write` 在写入前计算 `write_len = len.min(fsize_cur - cur_off)`，当写起点已超限时返回 EFBIG，但跨越限额的 mid-write 走 min 截断 <sup>[5](#mod-syscall-ref-5)</sup>。注释明确说明“setrlimit01 期望返截断字节数而非报错”。这是 Linux 行为：`write()` 返回写入字节数（可能小于请求数）而非 SIGXFSZ。对比 rCore-Tutorial 通常忽略 rlimit，本实现已覆盖 RLIMIT_FSIZE/RLIMIT_NOFILE 两项核心限制。

**O_PATH fd 的零分配策略。** 当 `flags & O_PATH` 时，`fd_table` 对应项保持 `None`，仅通过 `fd_flags[fd]` 中的 `FD_O_PATH_INTERNAL` 标记表示有效性 <sup>[6](#mod-syscall-ref-6)</sup>。这带来两个好处：（1）避免为空 fd 分配 `Arc<dyn VfsNode>` 的内存开销；（2）使 socket 系统调用对 O_PATH fd 自然返回 EBADF（而非 ENOTSOCK），与 Linux 语义一致。但这也迫使 dup/dup3/close_range 等函数在判断 fd 有效性时必须同时检查 `fd_table[fd].is_none() && !old_is_opath`，增加分支复杂度。

**ABBA 死锁预防是显式设计目标。** `sys_close` 的注释明确标注“注：fd 的 ABBA-安全 drop 已在上方完成”。关键模式是：`let closed = inner.fd_table[fd].take(); drop(inner); drop(closed);` <sup>[7](#mod-syscall-ref-7)</sup>。先取出文件引用，释放进程锁，再 drop 文件。因为 StarPipe::drop 会获取 ring 锁，而 pipe 阻塞侧持有 ring 锁等待 inner，如果不释放 inner 直接 drop 就会形成死锁。`sys_dup3` 和 `sys_close_range` 同样遵循此模式。这是一个在并发文件系统场景下容易被忽略的边角细节。

**MAX_RECV_BUF (1 MiB) 防 DoS。** `sys_recvfrom` 中 `let copy_cap = len.min(MAX_RECV_BUF)` <sup>[8](#mod-syscall-ref-8)</sup>。recv 缓冲分配上限是内核全局常量，防止用户传入 `len=usize::MAX` 导致内核 panic（无界 `alloc::vec![0u8; len]`）。但该上限会截断 stream socket 的接收长度，导致用户感知到短读——注释说明“真实测试 len 远小于上限不受影响”。这是一个安全与兼容性间的实用权衡。

**SCM_RIGHTS 禁止传递 UNIX socket 自身。** `collect_scm_rights` 中明确检查：如果待传递的 fd 指向一个 `SocketFile` 且其下是 UNIX socket，则返回 EINVAL <sup>[9](#mod-syscall-ref-9)</sup>。这是为了防止形成 Arc 循环引用导致内存泄漏（需要 unix_gc 才能回收）。Linux 通过垃圾回收机制处理这种情况，本实现选择直接拒绝，牺牲了与 Linux 的语义兼容性但换取了实现简单性。

### 3. 跨模块协同：MountTable、inotify/fanotify 与 loopback TCP drain

**MountTable 路径分发是全模块的依赖点。** `sys_mkdirat`、`sys_unlinkat`、`sys_faccessat`、`sys_fchmodat`、`sys_openat` 均通过 `MOUNT_TABLE.lookup(&full_path)` 获取文件系统实例和相对路径 <sup>[10](#mod-syscall-ref-10)</sup>。这取代了旧实现中硬编码的 if-else 分支，使 ext4/fat32/tmpfs 等文件系统可以动态挂载。但`lookup` 返回 `Arc<dyn FileSystem>` 后，各 syscall 还依赖 fs 的 `is_dir`、`access`、`stat`、`mkdir`、`unlink` 等方法，形成了一个隐式的 FileSystem trait 契约。

**inotify/fanotify 事件注入是跨切面的。** `sys_write` 产生 `IN_MODIFY`，`sys_read` 产生 `IN_ACCESS`，`sys_close` 根据是否可写投递 `IN_CLOSE_WRITE`/`IN_CLOSE_NOWRITE` <sup>[11](#mod-syscall-ref-11)</sup>。`sys_unlinkat` 调用 `fsnotify_self` 先发 `IN_ATTRIB` 再发 `IN_DELETE`，以对齐 inotify04 期望的事件序列。路径到监视点的映射通过 `Arc::as_ptr(&file) as *const () as usize` 作 key 在 `path_for` 表中查找。`suppress_fsnotify` 标志通过 `FD_FANOTIFY_EVENT_INTERNAL` 标记控制，避免内核内部操作（如 fanotify 自身的事件）触发递归通知。

**loopback TCP drain_after_close 是网络-任务-时钟三层协同的结果。** `sys_close` 在 drop socket 前取出 `loopback_tcp_handle`，drop 后调用 `drain_after_close` 同步轮询直到 FIN 被对端接收 <sup>[12](#mod-syscall-ref-12)</sup>。这个设计意图在注释中清晰说明：“让背靠背测试间的复位窗口‘撑开’，让 iperf3 -D 守护进程及时复位 ctrl_sck”。它仅在 loopback 场景生效，普通 fd 和真实网卡不受影响。这个机制与 `crate::net::tcp::graceful_teardown` 配合，后者在 `TcpSocket::Drop` 中将 FIN 投递给对端并唤醒 waiter。

### 4. 边角细节与不足

**`#[cfg]` 条件编译与架构假设。** <sup>[1](#mod-syscall-ref-1)</sup> 中 `fd_table` 使用 `Vec<Option<...>>` 和 `BTreeMap` 等通用容器，未出现任何架构相关的条件编译。但在 <sup>[13](#mod-syscall-ref-13)</sup> `user_buffer_within_one_page` 中硬编码了 `crate::config::PAGE_SIZE`，若移植到 mips（PAGE_SIZE=16K）或 riscv 的变种页大小需要检查该常量是否正确。目前没有看到对非 x86_64 架构的显式支持 gate。

**`unsafe` 块缺少 `// SAFETY:` 注释。** 例如 <sup>[14](#mod-syscall-ref-14)</sup> `translated_refmut(token, pipe_fd)` 和 `read_user_data`、`write_user_data` 等函数的大量调用点没有说明安全前提。特别是在 `collect_scm_rights` 中 `core::ptr::read_unaligned` 操作来自用户态的未校验对齐数据，虽然后面有长度检查但没有明确写出“用户数据是整数对齐的”这一安全假设。

**RLIMIT_FSIZE 在 close_range 中未处理。** <sup>[5](#mod-syscall-ref-5)</sup> 的截断逻辑只出现在 `sys_write` 中，而 `sys_close_range` 和其他批量操作路径中没有相应的 rlimit 检查。注释说明“fsize_cur 检查仅在 write 热路径”，但若用户通过 close_range 关闭 fd 后重新 open 并 write，限额可能已变化——这是正确的，但不一致可能让未来维护者困惑。

**`lib_fallback_path` 的硬编码假设。** <sup>[15](#mod-syscall-ref-15)</sup> 写死了 `/glibc/lib/` 和 `/musl/lib/` 作为回退路径，且检查顺序固定。如果系统使用 `/lib64/` 下的 glibc 但 `/glibc/lib/` 挂载了另一版本的 glibc，可能加载错误的动态链接库。回退机制也没有缓存，每次 openat 如果在标准库路径未命中都会触发两次额外的 lookup，对重负载动态链接程序的开销不可忽视。

**O_PATH fd 的 `AT_SYMLINK_NOFOLLOW` 处理不完整。** `openat_common` 中 O_PATH 分支对 symlink 的处理是通过 `readlink` 判断是否为 symlink，但对 `O_PATH|O_NOFOLLOW` 的目标如果是 dangling symlink 应仍然成功打开（Linux 允许）。当前代码在 `is_symlink` 检查后若 flags 含 O_NOFOLLOW 会返回 ELOOP，这不符合 Linux 语义。

**`configure_alg_cmsgs` 对未识别的 cmsg 类型静默忽略。** <sup>[16](#mod-syscall-ref-16)</sup> 函数内部检查 `cmsg_level == SOL_ALG` 后只处理三种类型，其他类型完全跳过。如果用户传入一个未来扩展的 cmsg_type，不会报错也不会生效。Linux 的 `ALG_SET_PUBKEY` 等类型在此被静默忽略，可能导致加密操作配置不完整。
