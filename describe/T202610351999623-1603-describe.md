## 📖 一、总览

该仓库是一个面向RISC-V单核平台的实验性教学内核，在rCore家族中属于教学演示型，但相较于基础的rCore-Tutorial，它扩展了对POSIX进程组、POSIX定时器、FUTEX、vfork等机制的支持，意图在保持内核最小化（零堆分配、静态表、关中断同步）的前提下，探索更广泛的系统调用覆盖。值得注意的是，它并未实现网络和复杂的驱动框架，而是将精力集中在任务调度和信号兼容性上。

最值得评审注意的三个特点：
1. **零堆分配的任务管理器**：TaskManager使用固定大小静态数组管理TCB和就绪队列，放弃动态灵活性换取内存安全可预测性，体现了单核嵌入式内核的典型取舍[task.ref:1]。
2. **基于物理帧的映射存储**：AddressSpace的MappingStore采用物理帧而非堆内存存储映射元数据，避免了启动后动态分配，但将最大映射数硬编码为5440，超出即panic[mm.ref:1]。
3. **信号处理的兼容性技巧**：fork时主动重置非SIG_IGN的handler为SIG_DFL，将SIGCHLD保留为内核内waitpid唤醒信号而不投递用户态，这些专门针对glibc/hackbench的兼容性调整，显示了项目对实际用户态程序运行质量的关注[signal.ref:2]。

## 🎯 二、综合评价

### 整体定位

这个仓库在rCore家族中定位为教学演示型，但比基础教学内核更深入。它试图在保持内核最小化（零堆分配、静态表、关中断同步）的前提下，覆盖更广泛的POSIX系统调用子集，特别是任务调度和信号处理方面。它专注于RISC-V单核平台，没有网络支持，驱动框架简陋，显示了项目对特定目标场景（可能是运行busybox或简单应用）的优化。整体上，它是一个“半教学半实用”的尝试，但离完整内核还有显著距离。

### 真正的创新点

- **零堆分配的全局数据结构**：TaskManager的TCB池（固定大小512）、IPC管道数组（1024）、VFS的FILE_TABLE等全部采用静态数组，完全避免运行时堆分配。这种设计在启动后内存使用完全可预测，但也导致容量硬编码，超出即panic或静默丢弃（如启动时最多6个内存区域[boot.ref:3]）。
- **基于物理帧的映射存储**：MappingStore使用物理帧而非堆内存存储映射条目，解决了在无堆分配器时的数据结构存储问题，但硬编码最大映射数5440，且超出时直接panic[mm.ref:1]，不适用于大量mmap的场景。
- **信号处理的务实调整**：fork时主动将非SIG_IGN的处理器重置为SIG_DFL[signal.ref:2]（避免hackbench级联），并将SIGCHLD用作内核内waitpid唤醒而不投递用户态[signal.ref:2]，这些针对glibc/busybox的兼容性技巧显示了项目对实际用户态需求的关注，而非单纯追求POSIX规范完备性。

### 取舍判断

项目将主要精力集中在了任务调度和信号机制的细节实现上，例如实现了FUTEX、POSIX定时器、进程组/会话、vfork等，但在其他方面做出了明显牺牲：
- **内存管理**：未实现缺页异常处理，brk延迟分配依赖缺页但无handler，访问未映射堆区域直接crash；映射数量硬限制5440。
- **文件系统**：VFS层使用关中断同步，仅适合单核；ext4块大小硬编码4096；mount仅支持MS_BIND；inode存在泄漏[fs.ref:3]。
- **驱动框架**：virtio_blk仅轮询，无中断优化，无统一抽象，UART和PLIC硬编码基地址[drivers.ref:5]。
- **网络**：完全未实现。
- **代码文档**：多处unsafe块缺少SAFETY注释，启动流程和驱动中的硬编码限制明显。
这些取舍表明项目更关注于任务和信号方面的POSIX扩展，而非打造一个完整的内核，适合运行有限的应用集。

### 完成度与不足

完成度方面，项目实现了约91个系统调用，但相当一部分是stub返回0（如SYS_UMOUNT2、SYS_MADVISE），核心路径如fork/execve/futex已实现，但内存管理和文件系统存在明显不足（缺页未实现、inode泄漏、mount不完整）。因此完成度评为2（合格），但距离完整内核还有较大差距。

创新性方面，零堆分配全局表、物理帧映射存储以及信号处理中的兼容性调整有一定的创新，但总体设计思路与rCore家族一致，创新性评为2（合格）。

代码质量方面，整体结构模块化，各模块职责清晰（如syscall层通过回调与arch解耦），但存在大量unsafe块缺乏注释、硬编码限制、溢出静默等问题，代码质量评为2（合格）。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块以 `BootInfo` 为核心抽象，将物理内存类型化为 7 种语义类别（Usable/Kernel/BootStack/Firmware/Device/Reserved/Initrd），通过双层解析策略（优先 DTB `/memory` 节点，失败回退到 QEMU virt 128MB 硬编码布局）构建启动期内存快照。与 rCore-Tutorial 基线相比，该实现放弃了对 SBI 查询接口的依赖，选择直接解析 Flattened Device Tree，并以静态数组（容量 6）代替动态分配，使得启动信息完全在单核、无堆环境下完成，但也因此引入了容量溢出静默丢弃和平台假设强耦合两个显著风险。

### 1. 核心抽象：类型化物理内存分区

启动流程模块的核心产出是 `BootInfo` 结构体 <sup>[1](#mod-boot-ref-1)</sup>，它并非简单的内存地址范围列表，而是将物理内存划分为 7 种语义类型（`MemoryKind` 枚举）的分区快照。这种设计模仿了编译器的**类型系统**：每种内存区域在启动后由不同的子系统消费——`Usable` 交给页分配器，`Kernel` 用于权限保护，`Firmware` 告知 MMIO 驱动避免触碰。`MemoryRegion` 以 `PhysAddr` 作为区间边界（左闭右开），配合 `&'static [MemoryRegion]` 切片实现零拷贝共享 <sup>[2](#mod-boot-ref-2)</sup>。

三个全局 `static mut` 变量（`MEMORY_REGIONS`、`MEMORY_REGION_LEN`、`BOOT_INFO`）构成一个简陋的启动期对象存储：汇编阶段写入栈和内核边界符号，Rust 阶段通过 `extern "C"` 链接符号读取 [_text_start 等，行 7-11]，再经由 `push_region` 逐步填充静态数组，最后由 `finalize_boot_info` 一次性地将切片引用注入 `BootInfo`。这个模式避免了启动期的动态分配，但代价是最大区域数被硬编码为 6 <sup>[3](#mod-boot-ref-3)</sup>，且所有操作必须在单核环境下串行完成。

### 2. 关键设计取舍：DTB 优先与静态回退

`init()` 函数（行 193-200）实现了典型的**双层降级策略**：
- **首选路径**：`rebuild_memory_map_from_fdt` 解析 DTB 的 `/memory` 节点，从中提取物理内存块，并用 `append_usable_region` 做 DTB 区域隔离（避免 DTB 自身占用的内存被标记为 Usable）。
- **回退路径**：`rebuild_fallback_memory_map` 使用 `PHYS_BASE=0x8000_0000`、`MEMORY_END=0x8800_0000` 的 128MB 硬编码布局，仅适配 QEMU virt 平台 <sup>[4](#mod-boot-ref-4)</sup>。

这个取舍的代价明确：如果 DTB 解析失败且不在 QEMU virt 上运行，内存布局完全错误。放弃 SBI 而选 DTB 的好处是减少了固件接口耦合——OpenSBI 的 `sbi_fw_base` 查询在部分模拟器上行为不同。但 `DTB_RESERVED_SIZE=0x20_000`（128KB）的硬编码假设了一个最大 DTB 尺寸 <sup>[5](#mod-boot-ref-5)</sup>，若 DTB 超过此范围，`append_usable_region` 的切割逻辑会产生错误的保留区间。

`append_usable_region` 的 DTB 切割逻辑（行 99-129）包含了 5 个条件分支 + 3 个早期返回，试图覆盖 DTB 区域与可用区域的各种重叠情况。然而其中一个分支（`dtb_start <= start && tail_start < end`）在 `tail_start >= end` 时会直接返回而不 push 任何区域——这是一个静默的可用内存丢失路径 <sup>[6](#mod-boot-ref-6)</sup>。

### 3. 跨模块协同：汇编入口到 Rust 启动链路

`arch/riscv64/boot/mod.rs` 通过 `core::arch::global_asm!(include_str!("entry.S"))` 将汇编入口嵌入内核 <sup>[7](#mod-boot-ref-7)</sup>。entry.S（未给出源码，但从 extern 符号可推断）负责：
1. 设置 `sp` 栈指针到 `_stack_top`；
2. 清零 BSS 段；
3. 读取 `a0`（hart_id）和 `a1`（dtb_pa）寄存器——这是 RISC-V 引导协议的标准参数传递方式；
4. 跳转到 Rust 侧的入口函数（推测为 `rust_main` 或类似），该函数随后调用 `hal::boot::init(hart_id, dtb_pa)`。

`hal/boot.rs` 与 `crate::hal::fdt` 模块的协同是关键：`rebuild_memory_map_from_fdt` 调用 `fdt::parse_memory_regions` 和 `fdt::count_cpus` <sup>[8](#mod-boot-ref-8)</sup>。fdt 模块负责最复杂的 DTB 遍历逻辑，而 boot.rs 只负责将解析结果映射到 `MemoryKind` 分类体系中——这是职责分离的典型案例。但 fdt 模块的签名要求 `MemoryBlock::empty()` 构造器存在且 `parse_memory_regions` 返回 `Result`，这意味着如果 fdt 模块的内部表示变化，boot.rs 需要同步调整。

`info()` 的懒初始化（行 202-212）提供了一个有趣的防御机制：即使 `init()` 从未被调用（例如在早期的测试或单核调试路径中），`info()` 也会自动构造一个最小回退布局。但这同时意味着 `info()` 可以在 `init()` 之前被调用并成功返回，可能掩盖初始化顺序的 bug。

### 4. 边角细节与不足

**<sup>[3](#mod-boot-ref-3)</sup> 容量硬编码风险**：`MEMORY_REGION_CAPACITY=6` 意味着最多记录 6 个区域。RISC-V 平台可能有多个非连续内存块（例如 HiFive Unleashed 有 3 块，加上 Kernel/BootStack/Firmware 可能超过 6）。`push_region` 在超限时直接 `return`（行 64），不返回错误码——这是一个 DoS 向量，如果 DTB 定义了多于 3 个 memory 节点，后续可用区域会被静默丢弃。

**<sup>[9](#mod-boot-ref-9)</sup> 所有 unsafe 块缺少 SAFETY 注释**：文件内 8 处 `unsafe` 均无 `// SAFETY:` 注释。`static mut` 的写操作在单核启动期是安全的，但 `info()` 函数中重新调用 `rebuild_fallback_memory_map` 修改 `MEMORY_REGION_LEN`（行 207）——如果两个 CPU hart 同时调用 `info()`（虽然启动期通常不会），存在数据竞争。契约未声明。

**<sup>[10](#mod-boot-ref-10)</sup> `MemoryKind::Initrd` 定义但未使用**：行 40-41 声明了 `Initrd` 变体，但在整个文件的任何 `push_region` 调用中从未出现。这是一个半成品标记——如果后续要支持 initrd，需要修改 `rebuild_*` 函数来解析 DTB 的 `/chosen/linux,initrd-start` 属性。

**<sup>[6](#mod-boot-ref-6)</sup> `append_usable_region` 的可用内存丢失路径**：当 `dtb_start <= start` 且 `tail_start >= end` 时（行 116-119 的分支条件不成立），函数会落到 `dtb_start >= end || dtb_end <= start` 分支判断；但如果 DTB 区域完全包裹了可用区域（`dtb_start < start && end < dtb_end`），所有分支都不会 push——一块可用内存被静默吞噬。

**<sup>[4](#mod-boot-ref-4)</sup> 回退布局的 128MB 硬编码**：`MEMORY_END = 0x8800_0000` 假设物理内存终止于 128MB 处。对于 512MB 的 virt 机器或 1GB 的 sifive_u，回退路径会丢失全部高位内存。虽然 DTB 解析正常时不会触发此路径，但 `info()` 的懒初始化（行 207）会以 `hart_id=0, dtb_pa=0` 调用回退路径，构造一个错误的布局——任何在 `init()` 之前调用 `info()` 或 `usable_memory()` 的代码都会读到截断的内存视图。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块的核心抽象是进程地址空间 `AddressSpace`，它封装了基于 RISC-V Sv39 的多级页表 `PageTable`、帧后备的映射记录 `MappingStore` 以及虚拟内存区域列表 `VmAreaList`。物理内存管理由 `frame.rs` 提供的位图+引用计数分配器负责。`AddressSpace` 创建时通过 `splice_kernel_mapping` 与内核页表共享映射。与同家族基线相比，此实现使用物理帧而非堆内存存储映射元数据，`mprotect` 中实现了基于引用计数的 COW 隔离，`fork` 时对可写页直接复制（安全优先），并针对 glibc 兼容性做了大量调整（如栈区域多留 32 页、不预建 TLS）。

### 1. 核心抽象与外部依赖

本模块的核心抽象是进程地址空间 `AddressSpace`，它封装了基于 RISC-V Sv39 的多级页表 `PageTable`、帧后备的映射记录 `MappingStore` 以及虚拟内存区域列表 `VmAreaList`。每个 `AddressSpace` 拥有独立的根页表，创建时通过 `splice_kernel_mapping` 将内核映射（RAM + MMIO）拼接进来，实现内核/用户地址空间隔离但不复制内核页表页。物理内存管理由 `frame.rs` 提供的位图+引用计数分配器负责，初始化时从 boot info 的最大 Usable 区域高端切出 metadata。`MappingStore` 使用分配的物理帧（而非堆内存）来存储 `UserMapping` 数组，每帧可存 170 个条目，按需增长，从而避免在 TCB 中内联大数组。`VmAreaList` 管理地址区域，支持拆分、收缩和 `swap_remove`，用于 `mmap`、`mprotect` 等操作。

### 2. 关键设计取舍

- **映射存储的物理帧后备方案**：`MappingStore` 用一个或多个物理帧存储映射记录，替代了常见的 `Vec<UserMapping>`。这样设计的前提是内核在用户态地址空间尚未启用堆分配器。但代价是映射数量受限：`MAX_STORE_FRAMES = 32`，即最多 5440 个映射。`push` 中 `assert!` 保证不超过该值，超过则 panic `<sup>[1](#mod-mm-ref-1)</sup>`。这对 iozone 类程序可能足够，但更极端的场景（如大量 mmap 的数据库）会触发 panic。

- **fork COW 策略**：`fork_cow` 中，对于可写页或已标记 COW 的页，直接分配新帧并拷贝，而非像 Linux 那样共享并延迟复制。注释明确表示这是“优先保证进程隔离正确性” `<sup>[2](#mod-mm-ref-2)</sup>`。这一选择简化了缺页处理（无需处理 COW 缺页），但显著增加了 fork 的内存开销和时间开销。在频繁 fork 的服务器场景下可能成为瓶颈。

- **mprotect 中的 COW 隔离**：`mprotect_range` 在添加写权限时，若物理页引用计数 > 1，会分配私有副本并拷贝，防止破坏进程隔离。这是除 fork 外唯一的 COW 分离路径。但缺页异常处理中未实现 COW，因此仅通过 mprotect 触发写权限变化的页面才会获得隔离。

- **帧分配器设计**：使用位图和引用计数，操作时禁用中断。搜索策略是环形扫描，从 `NEXT_HINT` 开始。释放时，仅当释放帧索引小于 hint 时才更新 hint，否则 hint 保持 `<sup>[4](#mod-mm-ref-4)</sup>`。这导致空闲帧集中在低地址时，搜索效率逐渐下降。此外，`alloc_contiguous_frames` 线性扫描位图，复杂度为 O(n²)。

- **`PageTable::map` 和 `unmap` 的 panic 策略**：当尝试映射已存在的 VPN 或解除未映射的 VPN 时，直接 panic，而非返回错误 `<sup>[3](#mod-mm-ref-3)</sup>`。这意味着调用者必须保证不重复映射，任何调用错误都会导致内核崩溃。在系统调用路径中，`AddressSpace` 的公共方法通常不会重复映射（如 `map_user_page` 先检查再映射），但 `mprotect_range` 中可能直接调用 `page_table.map`，若 VmArea 状态与页表不一致，可能触发 panic。

### 3. 跨模块协同

- **ELF 加载器与 AddressSpace**：`load_elf_from_pages` 创建 `AddressSpace`，调用 `map_user_page` 映射 PT_LOAD 段，注册 VmArea 为 Elf 类型，最后设置 brk。用户栈分配后，栈区域 VmArea 的 `end_vpn` 特意在 `USER_STACK_TOP` 之上多留 32 页，以适应 glibc 初始化时向上读取栈顶附近数据的访问模式。这一细节体现了对用户态运行时兼容性的深度考量。

- **系统调用层**：`sys_brk`、`sys_mmap`、`sys_munmap` 通过 `sched::manager::with_current_addr_space` 获取当前任务的 `AddressSpace`，然后调用其方法。对于文件映射，`sys_mmap` 先通过 `fs` 模块解析文件描述符，分配全局槽位（`mmap_file_alloc`），再将槽位与 `VmAreaType::MmapFile` 绑定。`sys_munmap` 使用三阶段流程 `<sup>[5](#mod-mm-ref-5)</sup>`：在中断禁用时收集共享页回写信息并释放非回写映射，然后启用中断执行 I/O 回写，最后禁中断清理回写页。这种设计避免了在中断禁用时执行 I/O，但阶段 1 中调用的 `mmap_file_free` 可能涉及内存操作，其安全性取决于该函数是否依赖中断。

- **TLB 刷新**：每次 `map_user_page`、`unmap_user_page`、`mprotect_range` 后都调用 `flush_tlb_all`。由于未使用 ASID，每次页表修改都需要全局刷新，代价较高。RISC-V 支持 ASID，但此实现未利用。

### 4. 边角细节与不足

- **硬编码限制**：`DEFAULT_BRK_START=0x20000` 和 `DEFAULT_MMAP_BASE=0x6000_0000` 是固定值，不支持参数化。若内核配置变化或地址空间布局需要调整，这些常量需手动修改。

- **`mprotect_range` 失败的一致性问题**：在修改 VmArea 后、遍历 VPN 更新页表时，若 COW 隔离中 `alloc_frame` 失败，函数返回 `-ENOMEM`，但此前可能已修改了 `areas`（拆分/收缩/改标志）。这导致地址空间处于不一致状态。此问题在代码中无回滚处理。

- **缺页异常处理缺失**：`set_brk` 仅更新 VmArea 边界，注释称“实际物理帧在首次访问时通过缺页异常按需分配”。但本仓库未见缺页处理代码。若用户访问堆区域，将触发 page fault，若未处理则导致内核 panic 或进程终止。这可能是尚未实现的部分。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以 `TaskManager` 单例为核心，管理全局固定大小 TCB 池（`[Option<TaskControlBlock>; 512]`）与循环就绪队列，实现单核时间片轮转调度（5 ticks × 10ms = 50ms）。核心抽象是零堆分配的 PID/TID 复用机制（RAII `PidHandle`）和六状态生命周期（UnInit→Ready⇄Running↕Blocked→Zombie→回收）。与 ArceOS 基线相比，本仓做了显著扩展：增加了 POSIX 进程组/会话（setpgid/setsid）、POSIX 定时器（itimers）、FUTEX 轻量级同步、vfork 语义、线程（CLONE_THREAD）支持，并将信号投递与定时器中断集成到调度循环中。整个模块按职责拆分为 manager/lifecycle/exec/sig_deliver/query/fd 六个子文件，但通过 `pub use` 统一在 manager.rs 重新导出，对外保持单一入口。

### 1. 核心抽象：零堆分配的单核任务管理器

调度模块的核心是 `TaskManager`——一个驻留在 BSS 段的单例结构体（`static mut MANAGER`），其内部所有数据结构均为固定大小的栈/静态数组，无任何堆分配。`tasks` 字段是 `[Option<TaskControlBlock>; MAX_TASKS]`（512 槽），PID 直接作为数组下标，实现 O(1) 查找。就绪队列是循环数组 `ready_queue: [usize; QUEUE_CAP]`，`QUEUE_CAP` 等于 `MAX_TASKS`（512）。这种设计放弃了动态扩容的灵活性，换取了内核栈空间的可预测性和零堆分配的安全性——对嵌入式/单核场景是合理取舍。<sup>[1](#mod-task-ref-1)</sup>

任务切换上下文 `TaskContext` 仅保存 callee-saved 寄存器（ra, sp, s0-s11），共 14×8=112 字节，通过 HAL 层的 `switch(cur_ctx, next_ctx)` 汇编函数执行实际切换。这个抽象与 rCore-Tutorial 一致，但 TCB 中额外存放了 `satp_val`（页表基址），使得 `activate()` 在切换上下文前即完成地址空间切换——这是单核模型下避免 TLB 刷新的关键优化。<sup>[3](#mod-task-ref-3)</sup>

PID 分配采用 `PidHandle` RAII 封装：`alloc_pid()` 返回 `PidHandle`，其 `Drop` 实现自动归还 PID 位图。线程退出时通过 `core::ptr::replace` 用 `PidHandle(0)` 替换原句柄并 `drop`，使 PID 立即复用，而 TCB 本体（含 64KB 内核栈）延迟到 `pending_reap` 在下一次 `schedule()` 时释放。这种“先归还 PID，后释放栈”的策略缩短了 PID 耗尽窗口，是值得注意的工程细节。<sup>[9](#mod-task-ref-9)</sup>

### 2. 关键设计取舍：抢占式轮转与模块化拆分

时间片轮转（RR）是默认策略，`DEFAULT_TIME_SLICE = 5` 个 tick，每个 tick 对应 10ms（100Hz 时钟）。`timer_tick(can_preempt: bool)` 是抢占入口：S-mode 下的时钟中断（can_preempt=false）仅重置 mtimecmp，不会扣减时间片；只有 U-mode 下返回前的 tick（can_preempt=true）才会真正扣减并触发抢占。这种两级设计防止了内核态临界区被意外抢占，但文档直言“S-mode timer trap 可能打断任意内核 Rust 栈帧”，因此 `process_timer_events()` 中遍历全局 TCB 池的操作必须保证在关中断环境下执行。<sup>[4](#mod-task-ref-4)</sup><sup>[5](#mod-task-ref-5)</sup>

调度器最微妙的机制是 `pending_reap` 延迟回收：`exit_current()` 在关中断下将 orphan TCB 的 PID 塞入 `pending_reap` 数组，然后通过 `__switch` 切换到新任务；切换到新任务的栈帧后，下一个调度周期的 `schedule()` 开头才会真正 `self.tasks[pid] = None` 释放 TCB。这是为了安全—不能在即将销毁的任务自己的栈上执行 `drop`。同一个模式在 `lifecycle.rs` 和 `manager.rs` 的 `yield_current` 和 `block_current` 中都有对称的回收逻辑，构成一个三层交织的清理契约。<sup>[6](#mod-task-ref-6)</sup>

模块化拆分是本仓与原始 ArceOS 基线最大的差异。`manager.rs` 仅保留调度核心（schedule/queue/timer_tick/spawn），生命周期（exit/waitpid）拆入 `lifecycle.rs`，fork/exec 拆入 `exec.rs`，信号投递拆入 `sig_deliver.rs`，查询函数拆入 `query.rs`，fd 操作拆入 `fd.rs`。全部通过 `pub use super::*` 在 `manager.rs` 统一 re-export，对外调用者无感知。这种拆分在 1500+ 行规模的调度模块中显著提升了可维护性。<sup>[7](#mod-task-ref-7)</sup>

### 3. 跨模块协同：futex、信号与进程组

FUTEX 是用户态锁的最底层阻塞机制，其实现横跨 `syscall/process.rs` 和 `sched/manager.rs`。`sys_futex(FUTEX_WAIT)` 在关中断临界区内双重检查 futex 变量（防止 lost-wakeup），然后调用 `block_current()` 阻塞当前任务；`FUTEX_WAKE` 则调用 `wakeup(pid)` 唤醒。两者之间的关键桥梁是 `FUTEX_WAITERS` 静态数组——用固定 512 槽存储等待者信息，在 `block_current` 之前入队、在 `wakeup` 路径上遍历匹配。这种设计避免了为 futex 分配额外的内核对象，但阻塞了 512 上限，大量并发 futex 等待会返回 `ENOMEM`。<sup>[10](#mod-task-ref-10)</sup>

信号投递与定时器的集成体现了最复杂的跨文件协作。`timer_tick` 在 U-mode 抢占路径上调用 `process_timer_events()`，该函数扫描全部 512 槽，将到期的 `posix_timer` 和 `itimer_real` 记录到 `TIMER_PENDING_SIGNALS` 暂存区，然后在开中断环境下调用 `sig_deliver::send_signal()` 实际投递。这种“先收集、后投递”的两阶段模式避免了对 `send_signal` 内部的锁需求，是单核模型下的典型技巧。但 `TIMER_PENDING_SIGNALS` 是固定 512 大小的静态数组，如果同一 tick 内有超过 512 个定时器到期，多余的信号会被静默丢弃。<sup>[8](#mod-task-ref-8)</sup>

进程组/会话支持是本仓相对于 rCore-Tutorial 的独特扩展。`sys_setpgid`、`sys_setsid`、`sys_getpgid`、`sys_getsid` 均作了最小 POSIX 合规实现，且与 `signal_process_group()` 函数配合——`sys_kill(0, sig)` 向整个进程组广播信号。`signal_process_group` 的实现很有意思：它会将目标进程组中除 PID 1（init）外的所有任务收集到 `[0usize; MAX_TASKS]` 栈数组（4KB 栈消耗），然后逐个调用 `send_signal`。这种暴力遍历在 512 槽上限下是可接受的，但若未来增大 `MAX_TASKS`，栈溢出风险需重新评估。<sup>[11](#mod-task-ref-11)</sup>

### 4. 边角细节与不足

**<sup>[2](#mod-task-ref-2)</sup> 条件编译与架构假设**：`manager.rs` 的 `idle_spin()` 调用 `wfi()`（RISC-V 特权指令），没有任何 `#[cfg]` guard。如果移植到 x86_64，此处的 `wfi` 应替换为 `hlt`，当前代码未提供架构抽象层。同文件多处使用 `hal::cpu::current_sp()` 和 `hal::time::read_ticks()`，这些 HAL 接口的实际实现架构依赖性极强，但 `manager.rs` 本身没有架构门控检查。

**<sup>[6](#mod-task-ref-6)</sup> 延迟回收的竞争窗口**：`pending_reap` 数组容量为 `MAX_TASKS`（512），`exit_current` 和 `yield_current` 中都有 `assert!` 检查溢出。但若极端情况下同时退出大量进程且调度器来不及处理回收，`pending_reap_count` 可能达到 512 并 panic。这是一个 DoS 入口——用户态可以通过 fork 大量子进程并立即 exit 来触发此断言。

**<sup>[8](#mod-task-ref-8)</sup> `TIMER_PENDING_SIGNALS` 静默丢失**：在 `process_posix_timers()` 中，当 `pending_count` 达到 `MAX_TASKS` 时，后续过期的定时器信号会被跳过（`if pending_count < MAX_TASKS` guard），没有任何 error 日志。同一问题存在于 `process_itimer_real()`。

**<sup>[10](#mod-task-ref-10)</sup> FUTEX 缺少 PI/REQUEUE 支持**：`sys_futex` 仅实现了 `FUTEX_WAIT` 和 `FUTEX_WAKE`（含 BITSET 变体），未实现 `FUTEX_CMP_REQUEUE`、`FUTEX_WAKE_OP`、PI-futex 等。更关键的是，`FUTEX_WAIT_BITSET` 的实现有一个代码路径 bug：它在 `FUTEX_WAIT` 分支中被匹配，但 `FUTEX_WAIT_BITSET`（op=9）与 `FUTEX_WAIT`（op=0）在语义上要求 clocksel 参数处理不同，当前代码完全忽略了这些差异。

**<sup>[11](#mod-task-ref-11)</sup> `signal_process_group` 栈上大数组**：函数内定义了 `let mut targets = [0usize; MAX_TASKS];`（4KB），位于关中断后的栈帧上。在 64KB 内核栈中这尚可接受，但如果调度器递归调用或在中断嵌套中触发，仍有栈溢出风险。注释中没有提及此栈消耗。

**<sup>[7](#mod-task-ref-7)</sup> `sys_sched_setscheduler` 等为 stub**：所有实时调度策略相关的系统调用（`sched_setscheduler`、`sched_getscheduler`、`sched_setparam`、`sched_getparam`、`sched_setaffinity`）均返回 0 或写入全零位图，不做任何实际策略调整。`cyclictest -p99` 可以“成功”运行但实际仍以 SCHED_OTHER 调度——这对实时性测试套具构成无声的语义违背。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该仓库的文件系统模块采用经典 VFS 架构，核心抽象为三个全局静态表（INODE_TABLE、FILE_TABLE、MOUNT_TABLE）和每个文件系统提供的一组函数指针结构体 FsOps。它解决了在无标准块设备或复杂 VFS 层下提供 POSIX 文件系统接口的问题，主要服务于 busybox 等用户态程序。与 ArceOS 基线相比，本实现采用固定大小表、关中断同步、手动 ext4 目录条目初始化，以及仅支持 MS_BIND 的 mount 子集，体现了早期轻量化内核的典型取舍。

### 1. 核心抽象与外部依赖

VFS 层以三个全局 `static mut` 数组为中心：`FILE_TABLE`（文件对象表）、`INODE_TABLE`（全局 inode 表）、`MOUNT_TABLE`（挂载表）<sup>[1](#mod-fs-ref-1)</sup>。每个表大小在编译时固定（`MAX_FILES`、`MAX_INODES`、`MAX_MOUNTS`）。文件系统类型通过 `FsOps` 结构体抽象——该结构体包含十个函数指针（read、write、lookup、create、unlink、readdir、stat、truncate、rename），每个已注册的文件系统（ramfs、devfs、ext4、procfs）提供一个 `static` 实例 <sup>[2](#mod-fs-ref-2)</sup>。路径解析委托给 `path.rs` 中的 `resolve_path_locked` 和 `resolve_parent_locked`，外部模块（如系统调用层）通过 `vfs_open`、`vfs_mkdir` 等高级接口操作 VFS。

### 2. 关键设计取舍

**同步策略**：整个 VFS 层使用 `with_interrupts_disabled` 来保护临界区，而非 `Mutex` 或 `RwLock`。这假设内核运行在单核或中断安全上下文中，在 SMP 上存在竞争风险。但这也避免了锁的开销和死锁可能性，适合早期原型。

**固定大小表**：`alloc_inode_locked` 在 inode 表满时直接 `panic!`<sup>[1](#mod-fs-ref-1)</sup>，`alloc_global_file_locked` 返回 `None`（调用方转为 `-EMFILE`）。前者意味着用户态通过连续打开文件可导致内核 panic，是一个 DoS 入口；后者则优雅返回错误。这种不对称处理反映了 VFS 层仍处于开发早期。

**Inode 生命周期**：`release_inode_locked` 仅当 inode 是 ext4 且非目录时，才在引用计数归零后释放表项 <sup>[3](#mod-fs-ref-3)</sup>。目录 inode 和 ramfs/devfs 的 inode 永不从表中移除，导致表项泄漏。注释表明这是针对 ext4 文件级映射的优化，但未考虑其他文件系统。

**Ext4 目录操作**：`ext4_create` 在创建新目录时手动写入 `.` 和 `..` 条目，并分配第一个数据块 <sup>[4](#mod-fs-ref-4)</sup>。`ext4_unlink` 在 nlink 降为 0 时释放 extent 块并回收 inode。这些操作直接操作原始磁盘块，没有事务保护，崩溃可能不一致。

**Mount 实现**：`sys_mount` 只支持 `MS_BIND` 标志，用于将普通文件绑定到另一路径；其它 mount 类型（如设备挂载）返回 0 但不做任何事，并输出 `kwarn!`<sup>[5](#mod-fs-ref-5)</sup>。这足以让 busybox 的 mount 命令通过，但真正的块设备挂载尚未实现。

### 3. 跨模块协同

系统调用层（`syscall/file.rs`）通过 `fs::vfs_open` 等接口调用 VFS。以 `sys_openat` 为例：先拷贝用户态路径，然后通过 `manager::resolve_dirfd` 将 `dirfd` 转换为全局 inode 索引，调用 `vfs_open` 获得 `GlobalFile` 索引（gfd），再调用 `manager::alloc_fd` 将 gfd 映射到进程 `FdKind::File` 中 <sup>[6](#mod-fs-ref-6)</sup>。`sys_read`/`sys_write` 则从 `FdKind::File` 中取出 gfd，再通过 `fs::file_read`/`file_write` 操作全局文件。

目录遍历 `sys_getdents64` 同样依赖 gfd：从 gfd 获取当前偏移和 inode，然后调用 `fs::file_readdir` 获取下一个目录项，组装成 `linux_dirent64` 格式写回用户空间。

进程切换时，cwd inode 保存在任务结构体中，由 `manager::current_cwd_inode` 和 `set_current_cwd_inode` 管理。`sys_chdir` 调用 `fs::resolve_path` 验证路径并在成功时更新 cwd。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 本项目信号机制围绕三组核心抽象构建：`SignalState`（每进程 pending/blocked 位图 + SigAction 表）、`SignalFrame`（用户栈上保存完整 TrapContext 快照 + trampoline 指令）、`SigAction`（与 Linux kernel_sigaction 兼容的 #[repr(C)] 结构）。`sig_deliver.rs` 实现信号发送、投递调度和 sigreturn，`signal.rs` 定义常量和默认动作，`syscall/sig.rs` 桥接系统调用。与 ArceOS/rcore-tutorial 基线相比，本项目在 fork 时主动将非 SIG_IGN handler 重置为 SIG_DFL（阻断 hackbench sigcatcher 级联），并将 SIGCHLD 留在内核内用作 waitpid 唤醒而不投递用户态。

### 1. 核心抽象与外部依赖

信号模块围绕三个数据抽象构建，分别位于 `signal.rs` 和 `sig_deliver.rs`。

**`SignalState`** <sup>[5](#mod-signal-ref-5)</sup>：每进程信号状态，包含 `pending`（64 位待处理位图）、`blocked`（屏蔽位图）和 `actions`（64 个 `SigAction` 数组）。64 位位图的设计直接对应 POSIX 信号 1-63 编号，`raise()` 和 `next_deliverable()` 均可在 O(1) 内完成，无需遍历链表。位图操作依赖于 `trailing_zeros()` 指令，在 RISC-V 上通过硬件 CTZ 指令高效实现。

**`SigAction`** <sup>[5](#mod-signal-ref-5)</sup>：`#[repr(C)]` 结构体，四个字段（`handler`/`flags`/`restorer`/`mask`）与 Linux `kernel_sigaction` 完全兼容，允许用户态通过 `rt_sigaction` 系统调用直接写入。`handler` 使用 SIG_DFL=0、SIG_IGN=1 作为特殊值，与 Linux 的 `SIG_DFL`/`SIG_IGN` 宏一致。

**`SignalFrame`** <sup>[5](#mod-signal-ref-5)</sup><sup>[8](#mod-signal-ref-8)</sup>：保存在用户栈上的帧结构，包含两字节能 `trampoline` 指令（`addi a7, x0, 139; ecall`）、信号编号、对齐填充、保存的 `blocked` 掩码和完整的 `UserTrapFrame` 快照。与 Linux 使用 `ucontext_t` + `siginfo_t` 的标准不同，本项目采用扁平结构，所有数据一次性拷贝写入用户栈。trampoline 嵌入栈帧的设计省去了 vDSO 或信号页的额外映射，但意味着栈上包含可执行指令，破坏了 W^X 隔离。

外部依赖方面，信号模块通过 `crate::hal::trap::UserTrapFrame` 与具体架构耦合，通过 `crate::sched::manager::MANAGER` 全局任务数组访问 TCB，通过 `crate::mm::address_space::AddressSpace` 配合 `write_to_user_va`/`read_from_user_va` 实现用户空间安全读写。

### 2. 关键设计取舍

**取舍 1：SIGCHLD 不投递到用户态。** `deliver_for_task` 中 SIGCHLD 被显式 `continue` 跳过 <sup>[2](#mod-signal-ref-2)</sup>，注释解释其仅作为 waitpid/block wakeup 提示。这种设计避免了实现完整 siginfo/ucontext ABI 的负担，但意味着依赖 SIGCHLD handler 的程序（如 `busybox ash`）无法正常工作。

**取舍 2：fork 时非标准 handler 重置。** `fork_clone()` <sup>[6](#mod-signal-ref-6)</sup> 将子进程中所有非 SIG_IGN 的 handler 重置为 SIG_DFL，与 POSIX "fork 后继承所有 handler" 的标准相悖。注释坦承这是为了阻断 hackbench sigcatcher 的链式传播。这种设计牺牲了子进程中用户态 handler 的延续性，但提高了 fork 后立即 exec 场景的安全性。

**取舍 3：trampoline 嵌入栈帧而非固定映射页。** `SignalFrame` 的 `trampoline` 字段直接嵌入用户栈 <sup>[8](#mod-signal-ref-8)</sup>，而非像 Linux 那样放在 vDSO 或信号页中。这省去了额外页表映射，但破坏了 W^X 原则：栈上包含可执行指令，且被篡改时可能导致任意代码执行。注释中的 `// Phase 7` 标记暗示开发者计划在后续阶段改用跳板页。

**取舍 4：SIGKILL 自杀的同步终止路径。** `send_signal` <sup>[1](#mod-signal-ref-1)</sup> 中，当目标为自身且信号为 SIGKILL 时，在释放中断锁后同步调用 `exit_current`。这意味着 `kill(getpid(), SIGKILL)` 永不返回。这种设计避免了在中断上下文终止进程的复杂性，但要求调用者确保 `exit_current` 不会持有其他锁。

**取舍 5：sigreturn 的零信任安全校验。** `do_sigreturn` <sup>[4](#mod-signal-ref-4)</sup> 在恢复上下文后执行三项强制校验：强制 `sstatus.SPP=0` 防止特权级提升、检查 `sepc` 是否指向内核空间（非法时直接 SIGSEGV 终止）、强制 `x[0] = 1`（绕过腐败帧检查）。注释明确“不信任用户态提供的值”，反映了对信号帧完整性的零信任模型。特别是 `sepc` 检查在非法时不使用 `raise(SIGSEGV)` 而是直接 `exit_current`，因为若 SIGSEGV 被阻塞将导致内核在 `sret` 时崩溃。

### 3. 跨模块协同

**trap handler → 信号生成。** `handle_user_fault()` <sup>[7](#mod-signal-ref-7)</sup> 由 arch trap handler 回调，将硬件异常（PageFault、IllegalInstruction 等）映射为 POSIX 信号。对于 PageFault，它先尝试 demand paging，若失败才发送 SIGSEGV。这种回调机制实现了 arch→sched 的解耦，但信号类型映射粒度较粗（AccessFault 和 Misaligned 均映射为 `TrapCause::Unknown` 进而统一产生 SIGBUS）。

**系统调用 → 信号状态修改。** `syscall/sig.rs` 中的 `sys_rt_sigaction` <sup>[10](#mod-signal-ref-10)</sup> 和 `sys_rt_sigprocmask` <sup>[11](#mod-signal-ref-11)</sup> 通过 `manager::with_current_signal()` 闭包修改当前任务的 `SignalState`。这些系统调用使用 `read_bytes_from_user`/`write_bytes_to_user` 安全地读写用户空间，并调用 `sanitize_mask` 确保 SIGKILL/SIGSTOP 不可被屏蔽。`sys_rt_sigsuspend` 临时替换屏蔽字后调用 `manager::block_current()` 阻塞当前任务，被唤醒后恢复原屏蔽字并返回 -EINTR。

**调度器 → 信号投递。** `deliver_pending_signals()` <sup>[1](#mod-signal-ref-1)</sup> 在返回用户态前被调用，是信号投递的最终触发点。投递结果分为 `Continue`/`Exit`/`StopCurrent` 三种。`Exit` 路径调用 `lifecycle::exit_current` 终止进程；`StopCurrent` 路径调用 `MANAGER.schedule()` 切换任务，并在调度后回收 `pending_reap` 列表中的僵尸进程。`send_signal` 在给 blocked 任务发送可投递信号时会清除 `sleep_deadline` 并设置 `sleep_interrupted = true`，确保阻塞系统调用（如 `nanosleep`/`waitpid`）被中断并返回 -EINTR。

### 4. 边角细节与不足

**<sup>[1](#mod-signal-ref-1)</sup> 中断禁用粒度过粗。** `send_signal` 和 `deliver_pending_signals` 使用 `irq::disable()` 保护临界区。持锁期间可能调用 `exit_current`（释放资源）或 `MANAGER.schedule()`（切换任务），这两个函数若尝试获取其他锁可能造成死锁。虽然没有死锁的明显证据，但持锁路径较长，存在重入风险。

**<sup>[3](#mod-signal-ref-3)</sup><sup>[4](#mod-signal-ref-4)</sup> unsafe 块缺乏完整安全契约。** `sig_deliver.rs` 中多处直接解引用裸指针（如 `trap_ctx_addr as *mut UserTrapFrame`），但缺少 `// SAFETY:` 注释说明不变量。特别是 `setup_signal_frame` 假设 `trap_ctx_addr` 始终有效且对齐，但没有验证 `kstack_top` 与 `ctx_size` 的减法不会下溢。

**<sup>[3](#mod-signal-ref-3)</sup> 信号帧布局的偏移手动计算。** `setup_signal_frame` 中逐个字段写入用户栈的偏移是手动计算的（`trampoline` + `sig` + `_pad` + `saved_blocked` + `saved_ctx`），而非使用 `offset_of!` 宏。`#[repr(C)]` 布局若因架构对齐调整，手动偏移可能与实际布局不一致。虽然 `SIGNAL_FRAME_SIZE` 通过了编译期计算，但字段级偏移仍是硬编码。

**<sup>[7](#mod-signal-ref-7)</sup> AccessFault/Misaligned 区分度不足。** `handle_user_fault` 将 `AccessFault` 和 `Misaligned` 映射为 `TrapCause::Unknown(0)` 和 `TrapCause::Unknown(1)`，最终统一产生 SIGBUS。对于调试而言，区分这两种异常有助于定位问题（例如 Misaligned 通常指示编程错误，而 AccessFault 可能指示硬件问题），当前实现丢失了这些信息。

**<sup>[2](#mod-signal-ref-2)</sup><sup>[5](#mod-signal-ref-5)</sup> 同步信号优先级未遵循 Linux 语义。** `next_deliverable()` 按位优先级（SIGKILL > SIGSTOP > 低位信号）投递，但 Linux 的 POSIX 信号模型要求同步信号（如 SIGSEGV、SIGBUS）优先于异步信号。例如，若进程同时收到 SIGSEGV 和 SIGTERM，当前实现会投递编号较小的信号而非同步信号。

**<sup>[6](#mod-signal-ref-6)</sup> fork_clone 与 POSIX 标准的不一致。** 如前所述，`fork_clone` 将非 SIG_IGN handler 重置为 SIG_DFL。虽然注释解释了动机，但这意味父进程设置了 SIGUSR1 handler 后，子进程 fork 后 handler 变为 SIG_DFL。如果在 fork 后、exec 前向子进程发送 SIGUSR1，行为与 Linux 不同——子进程将意外终止而非调用 handler。

### ⚠ 开放问题

- SIGCHLD 不投递用户态：依赖 SIGCHLD handler 的 shell（如 busybox ash）无法正常工作，需后续实现完整 siginfo/ucontext ABI。
- 信号帧偏移手动计算：setup_signal_frame 中 trampoline/sig/pad/blocked/ctx 偏移是硬编码而非 offset_of!，维护风险。
- AccessFault/Misaligned 区分度不足：handle_user_fault 将两者均映射为 SIGBUS，丢失调试信息。
- 同步信号优先级不符合 Linux：next_deliverable 按位优先而非同步信号优先（如 SIGSEGV 应优先于 SIGTERM）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 模块基于固定大小环形缓冲区实现 POSIX 管道，使用静态全局数组 `PIPES` 管理最多 1024 个管道对象，所有操作在关中断下执行以保证原子性。核心抽象是 `Pipe` 结构体，包含缓冲区、读写索引、引用计数及固定大小的阻塞 PID 队列。阻塞式读写通过 `sched::manager` 的 `block_current`/`wakeup` 实现，是内核提供的唯一 IPC 原语。与同家族基线（如 rCore-Tutorial）相比，本实现采用零堆分配的静态池设计，避免了动态内存分配，但限制了并发管道数；阻塞队列采用固定数组，队列满时降级为自旋 Yield，是一种激进的空间换时间权衡。

### 1. 核心抽象与外部依赖

进程间通信模块位于 `kernel/src/ipc/`，当前仅实现 POSIX 管道。核心数据结构是 `Pipe`（`kernel/src/ipc/pipe/mod.rs:16-29`），包含一个 4096 字节的环形缓冲区、读写位置、字节计数、读写端引用计数以及两个固定大小的阻塞 PID 队列。所有管道实例存储在全局 `static mut PIPES: [Pipe; 1024]` 中（`kernel/src/ipc/pipe/mod.rs:38-42`），通过索引访问，不依赖堆分配器。

模块的外部依赖集中在两处：一是 `crate::hal::irq::with_interrupts_disabled`，用于在单核环境下为所有管道操作提供原子性；二是 `crate::sched::manager`，提供 `block_current`、`wakeup` 和 `current_pid` 等调度原语。这种设计明确将管道层与调度器解耦——管道本身不管理等待队列，而是直接借用调度器的进程阻塞/唤醒机制。

与 rCore-Tutorial 中管道使用动态文件描述符和管理结构不同，本模块选择静态预分配数组。`ipc/mod.rs` 的注释表明设计者有意维持微内核风格：管道是内核提供的唯一 IPC 原语，高级机制（如消息队列、信号量）留给用户态实现。

### 2. 关键设计取舍：中断屏蔽、静态队列与不对称唤醒

**中断屏蔽作为锁**：所有公共 API（`pipe_create`、`pipe_read`、`pipe_write`、`pipe_close_*`、`pipe_poll`）均包裹在 `with_interrupts_disabled` 闭包内，关中断期间完成状态检查、数据拷贝和队列操作。这种做法的优点是实现简单、无锁开销，在单核下可保证原子性；缺点是关中断时间随数据拷贝量线性增长（最大 4096 字节），可能增大中断延迟。若未来支持多核，该设计需要替换为自旋锁或无锁技术。

**阻塞队列固定大小与 Yield 降级**：`MAX_BLOCKED_READERS=8` 和 `MAX_BLOCKED_WRITERS=64`（`kernel/src/ipc/pipe/mod.rs:12-13`）是硬编码常量。当等待队列已满时，`pipe_read_blocking` 并不阻塞，而是返回 `ReadAction::Yield`（`kernel/src/ipc/pipe/blocking.rs:67-69`），通过 `manager::yield_current` 让出 CPU 后重新尝试。这种设计避免了队列溢出，但引入活锁风险：若读者和写者都处于满队列状态，双方可能持续 Yield 而无法推进。hackbench 可能是主要基准，64 写者容量覆盖其每40 group 的场景，但通用负载下队列溢出后的行为文档未明确。

**不对称唤醒策略**：写操作完成时只唤醒一个等待的读者（`kernel/src/ipc/pipe/mod.rs:90-95`），而读操作完成后唤醒所有等待的写者（`kernel/src/ipc/pipe/mod.rs:120-125`）。这一设计基于管道字节流模型：一个读者消费数据后，所有因缓冲区满而阻塞的写者均可尝试写入；而一次写入仅需通知一个读者（EPOLLONESHOT 风格）。但若写者写入的数据量小于读者请求，只唤醒一个读者可能不够高效——读者可能仍需要更多数据。

### 3. 跨模块协同：调度器、中断控制与系统调用层

管道模块与 `sched::manager` 的交互集中在阻塞/唤醒路径。`pipe_read_blocking` 中先以 `with_interrupts_disabled` 判断条件，若需阻塞则调用 `manager::block_current()` 将当前进程标记为阻塞态，然后关中断返回后立即调用 `manager::wakeup(wake_pid)` 唤醒对方。注意：`block_current` 在关中断内调用，但 `wakeup` 放在关中断外——这是有意为之，避免在关中断期间执行调度器操作，减少中断屏蔽时间。

`remove_blocked_reader` 和 `remove_blocked_writer` 函数（`kernel/src/ipc/pipe/blocking.rs:8-32`）使用线性扫描移除 PID，复杂度 O(n)，但由于队列大小固定且很小，此开销可接受。它们同样在关中断下执行，防止并发修改。

系统调用层（`kernel/src/syscall/dispatch.rs`）通过 `sys_read`/`sys_write` 间接调用本模块的 `pipe_read_blocking`/`pipe_write_blocking`，但模块本身未暴露系统调用入口（直接暴露函数而非 syscall 编号）。这种间接层使得文件描述符抽象可以位于上层，保持 IPC 模块的纯净。

### 4. 边角细节与不足

**`unsafe` 块缺少 SAFETY 注释**：整个模块中所有 `unsafe` 块（如 `kernel/src/ipc/pipe/mod.rs:50-58`、`kernel/src/ipc/pipe/blocking.rs:14-32`）均未提供 `// SAFETY:` 说明。`static mut PIPES` 的并发安全依赖于关中断，但在多核环境下将不成立，缺乏文档使得后续维护者难以评估风险。

**`MAX_BLOCKED_READERS=8` 的激进假设**：注释说“单根管道最多同时阻塞的读者数量”（`kernel/src/ipc/pipe/mod.rs:13`），取 8 似乎基于典型场景。但若大量进程同时读同一管道（如生产者-多消费者模式），队列满后降级为 Yield 会导致持续空转。建议提供动态扩容或信号量机制。

**`pipe_poll` 中的硬编码常量**：POLLIN、POLLOUT、POLLHUP、POLLERR、POLLNVAL 的数值直接写死（`kernel/src/ipc/pipe/mod.rs:136-156`），未使用系统头文件或 crate 定义。虽然目前与 Linux 标准一致，但跨平台移植时需手工更新。

**`pipe_create` 无错误路径**：当所有 1024 个管道槽位均被占用时，`pipe_create` 返回 `None`（`kernel/src/ipc/pipe/mod.rs:56`），但调用者（如 `sys_pipe`）可能直接 `unwrap` 导致内核 panic。应返回合适的错误码。

**阻塞队列移除的悬崖情况**：`remove_blocked_reader` 在找到匹配 PID 后将其与最后一个元素交换，但在 `while` 循环中 `i` 不自增，可能再次检查交换后的元素，但已交换的元素是之前已检查过的，因此逻辑正确。不过，若 `pid` 在队列中出现多次（理论上不应发生），则只会移除最后出现的那个，但 `blocked_reader_count` 只减一次，可能造成计数不一。属于低概率边界。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 当前 `drivers` 模块仅包含一个 Virtio MMIO 块设备驱动（virtio_blk），采用轮询模式、全局可变静态变量和硬编码 MMIO 地址，没有统一的设备抽象层或注册机制。与同家族基线（ArceOS）相比，缺少 trait 抽象、设备树解析和中断驱动支持；UART 和 PLIC 驱动散落在 arch 目录，未纳入 drivers 管理。该驱动框架仍处于早期硬编码阶段，可移植性和扩展性受限。

### 1. 缺乏统一抽象：静态配置取代驱动框架

`kernel/src/drivers/mod.rs` 仅导出一个子模块 `virtio_blk`，没有定义任何设备 trait、注册表或 probe 接口<sup>[1](#mod-drivers-ref-1)</sup>。每个设备驱动自行暴露初始化函数（如 `virtio_blk::probe()`），由调用方在启动早期主动调用。UART 和 PLIC 驱动更是直接放在 `arch/riscv64/` 下，完全不经过 drivers 层<sup>[5](#mod-drivers-ref-5)</sup>。这反映出该仓库尚未构建一个可扩展的驱动框架，而是采用一种“随用随加”的静态配置方式。

相比之下，ArceOS 家族通常提供 `driver_common` crate，定义 `Device`、`BlockDevice` 等 trait，并通过设备树或总线枚举动态绑定驱动。当前代码放弃了这种灵活性，换来了极简的实现复杂度——对于只面向 QEMU virt 固定平台的早期原型，这是合理的取舍，但代价是移植到新平台时需要大量修改内核代码。

### 2. Virtio 块设备驱动的实现细节与取舍

virtio_blk 模块实现了 Virtio MMIO 协议的 v1（legacy）和 v2（modern）版本<sup>[2](#mod-drivers-ref-2)</sup>。核心机制是：

- **全局可变状态**：`BLK`、`VRING_MEM` 等均为 `static mut`，没有使用 `Mutex` 或 `RefCell`。多核安全依赖调用方在 `read_sector`/`write_sector` 中关中断<sup>[3](#mod-drivers-ref-3)</sup>。这种设计在单核下可行，但若未来支持多核并发 I/O，可能引入数据竞争。
- **轮询模式**：`block_io()` 使用自旋等待 `used.idx` 更新，每 500 万次重试后超时返回 false<sup>[4](#mod-drivers-ref-4)</sup>。超时后同步 `last_used_idx` 以保持状态一致，但设备侧后续可能仍会完成请求，导致下次 I/O 产生混乱。这种模式省去了中断处理程序的复杂性，但 CPU 占用高、实时性差。
- **硬编码参数**：队列深度固定为 8、MMIO 基地址 `0x1000_1000`、页大小 4096 等均写死<sup>[2](#mod-drivers-ref-2)</sup>。没有通过配置或设备树动态获取，与 QEMU virt 平台强耦合。
- **版本协商**：代码区分 v1 和 v2，为 v2 设置了 `VIRTIO_F_VERSION_1` 特性位<sup>[2](#mod-drivers-ref-2)</sup>。但未实现现代 virtio 的更多特性（如 indirect descriptors、event idx），这部分放弃的性能可通过轮询模式掩盖。

### 3. 跨模块协同：中断关闭与硬件抽象层

virtio_blk 的公开接口 `read_sector`/`write_sector` 通过调用 `crate::hal::irq::disable()` 来保护全局状态的读-改-写序列<sup>[3](#mod-drivers-ref-3)</sup>。`hal::irq` 模块又委托到 `arch::riscv64::trap::interrupt` 函数，最终操作 sstatus 寄存器关闭 S-mode 中断。这种设计将关中断作为同步原语使用，代价是全局中断关闭，影响系统实时性。

UART 的中断处理则展示了另一种跨模块协同：`plic::claim()` 返回 IRQ 号，`uart::handle_irq()` 通过轮询 LSR 寄存器读取所有就绪字节<sup>[5](#mod-drivers-ref-5)</sup>。此处没有使用 drivers 模块，而是直接在 trap handler 中调用。TTY 层通过 `tty::driver::register_backend` 注册 UART 的后端函数（`uart_putc`、`try_getc`）<sup>[6](#mod-drivers-ref-6)</sup>，这实际上是唯一一处轻量驱动抽象，但独立于 drivers 模块。

### 4. 边角细节与不足

- **`unsafe` 块缺乏安全注释**：virtio_blk 中几乎所有函数都是 `unsafe`，但未提供 `// SAFETY:` 说明<sup>[7](#mod-drivers-ref-7)</sup>。例如 `mmio_read32` 假设 base 地址有效且对齐，这些契约未文档化。评审者需要逐行推断安全条件。
- **死代码残留**：`read_sectors` 和 `write_sectors` 标有 `#[allow(dead_code)]`<sup>[8](#mod-drivers-ref-8)</sup>，说明它们未在调用链中使用。可能是不完整的批量 I/O 接口，或重构遗留。
- **TODO 标记的半成品**：UART 驱动注释明确表示“后续应将此模块移出内核，以用户态 UART server 替代”<sup>[5](#mod-drivers-ref-5)</sup>。这意味着当前的 UART 驱动只是临时方案，未来架构将彻底变更。
- **平台假设未隔离**：VIRTIO_MMIO_BASE 硬编码为 `0x1000_1000`，UART_BASE 为 `0x1000_0000`，PLIC_BASE 为 `0x0C00_0000`<sup>[2](#mod-drivers-ref-2)</sup><sup>[5](#mod-drivers-ref-5)</sup>。没有通过 DTB 或 SBI 查询，无法运行在非 QEMU virt 平台上。
- **错误吞没**：`probe()` 返回 bool，但内部 `init_device` 失败时只 log 错误，不提供详细错误码。块 I/O 超时只返回 false，调用方无法区分是设备故障还是超时<sup>[4](#mod-drivers-ref-4)</sup>。

这些不足反映了该仓库处于早期原型阶段，驱动框架尚未成型。后续若需支持真实硬件，至少需要引入设备树解析、中断驱动 I/O、统一的 Device trait 以及更安全的并发控制。

### ⚠ 开放问题

- Virtio 块设备驱动仅支持轮询模式，未实现中断驱动 I/O，在高延迟设备场景下 CPU 空转严重。
- UART 和 PLIC 驱动硬编码基地址，不解析设备树，无法运行于非 QEMU virt 平台。
- 全局可变静态变量 BLK 无同步原语，多核并发访问可能数据竞争（当前依赖关中断假设单核）。
- I/O 超时后 last_used_idx 同步机制脆弱，设备后续完成可能破坏状态一致性。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 本模块以 dispatch 函数为核心，通过 match 路由 Linux RISC-V ABI 系统调用编号到具体处理函数；与 trap handler 通过回调机制解耦；用户态指针验证采用两层检查（范围 + 页表权限）确保安全。对比 ArceOS 基线，本仓库实现了更完整的 syscall 列表（约 80 个），并对文件系统 inode 泄漏、SBI 定时器活性等边缘情形做了精细处理。

### 1. 回调驱动的分发架构
系统调用层的入口是 `dispatch()`，它根据 a7 寄存器中的编号将参数数组 `[a0..a5]` 路由到对应的 `sys_*` 函数<sup>[1](#mod-syscall-ref-1)</sup>。但该函数并非被直接调用，而是通过 trap handler 中注册的回调间接触发。`arch/riscv64/trap/handler.rs` 定义了 `TrapCallbacks` 结构体，其中的 `syscall_dispatch` 字段保存 `dispatch` 的函数指针；`handle_syscall()` 读出 a7 和 a0–a5，调用回调并将返回值写回 a0<sup>[3](#mod-syscall-ref-3)</sup>。这种回调设计将 arch 模块与内核策略层解耦，使得 arch 可以独立编译和测试。代价是每一次系统调用都经过一次间接函数调用，在热路径上可能有微小的性能损失，但对于 rcore 这类教学内核是可接受的。与 rCore-Tutorial 直接在 trap 中调用 syscall 函数相比，此处的回调是更灵活但更复杂的方案。

### 2. 用户态指针安全边界
所有来自用户态的指针必须在 syscall 入口处验证。`check.rs` 实现了两层检查：第一层 `checked_user_range()` 验证地址落在 Sv39 用户空间低半区 [`0`, `0x40_0000_0000`)，并处理溢出；第二层 `ensure_user_page()` 逐页检查页表中是否包含 `VALID | USER` 权限，且写操作同时要求 `WRITE` 或 `COW` 标志<sup>[4](#mod-syscall-ref-4)</sup>。这种设计避免了内核直接信任用户指针，防止恶意程序通过伪造指针破坏内核。但注意在 `io.rs` 的 `sys_write` 中，验证后立即使用 `unsafe { core::slice::from_raw_parts }` 创建切片，并未添加 `// SAFETY:` 注释说明安全前提<sup>[2](#mod-syscall-ref-2)</sup>。虽然验证逻辑保证了缓冲区可读，但缺少文档化是一种技术债。此外，验证与使用之间没有额外屏障，如果用户映射的页面在验证后被撤销（例如多核环境下另一线程调用 `munmap`），可能导致使用后漏洞；当前内核为单核设计，此风险尚可接受。

### 3. FD 分发与 I/O 路径
`io.rs` 中的 I/O 系统调用通过 `current_fd_kind()` 查询当前进程的 fd 表，将 `FdKind` 枚举分发到文件、pipe 或 socket 后端。例如 `sys_write` 匹配 `FdKind::File(gfd)` 后调用 `fs::file_write`，对 `PipeWrite` 调用 `pipe_write_blocking`。常量 `IO_MAX = 65536` 限制单次读写长度，防止长时间占用 CPU<sup>[2](#mod-syscall-ref-2)</sup>。值得注意的是 `sys_fstatat` 的注释解释为何使用 `open+fstat+close` 模式而非直接 `resolve_path+stat`：直接方式会导致 ext4 文件 inode 泄漏（refcount 永不释放），这是对 VFS 层实现细节的深刻理解<sup>[6](#mod-syscall-ref-6)</sup>。`sys_pipe2` 创建 pipe 后分配两个 fd，并处理 `O_CLOEXEC` 标志，错误路径会正确回滚已分配的资源。这种在错误处理中显式清理资源的风格在整个模块中一致。

### 4. 时间系统调用与硬件定时器协同
`time.rs` 实现了 `nanosleep` 和 `clock_nanosleep`，它们通过 `manager::block_current_until(deadline)` 阻塞当前任务，并依赖定时器子系统设置单次唤醒。`set_one_shot` 在 `arch/riscv64/trap/timer.rs` 中实现：仅当 `deadline` 早于当前 `mtimecmp` 时才更新 SBI 定时器，避免不必要的 ecall<sup>[5](#mod-syscall-ref-5)</sup>。定时器后端在初始化时进行活性探测：通过 SBI Base Extension 检查 `0x54494D45` 是否存在，并执行一次 liveness test 验证 `sip.STIP` 是否能正确触发，以防止 RustSBI 假成功问题<sup>[5](#mod-syscall-ref-5)</sup>。这一层探测使得内核能优雅回退到 legacy set_timer，是硬件适应性的良好范例。

### 5. 边角细节与不足
- **stub syscall**：`dispatch.rs` 中多个 syscall 被简化为 `stub_ok()` 或直接返回 0，例如 `SYS_UMOUNT2`、`SYS_MADVISE`、`SYS_GET_MEMPOLICY` 等。这些 stub 未返回正确错误码（如 `EOPNOTSUPP`），可能导致用户态程序误判<sup>[1](#mod-syscall-ref-1)</sup>。
- **unsafe 文档缺失**：`io.rs` 中每一个 `unsafe` 块都缺少 `// SAFETY:` 注释。虽然滥用风险较低（因为前面有验证），但不符合 Rust 社区的最佳实践<sup>[2](#mod-syscall-ref-2)</sup>。
- **用户验证的 COW 处理不完整**：`check.rs` 的 `ensure_user_page` 在写检查时遇到 `COW` 标志直接返回 `false`，而不是触发写时复制。这会导致 `mmap` 写操作在写时复制页面上的读写被错误拒绝<sup>[4](#mod-syscall-ref-4)</sup>。
- **错误码约定**：所有 syscall 返回负 errno，而非 POSIX 标准的返回 -1 并设置 errno。需要用户态库进行转换，增加了复杂度。
- **锁可见性**：在访问 `fd_table` 和地址空间时没有显式的锁注释；对于单核内核此问题不大，但不利于未来多核扩展。

总体而言，系统调用层在功能数量上覆盖了大部分常用 Linux syscall，并针对文件系统、定时器等具体问题做了精巧处理，但其早期实现的痕迹（stub、文档缺失）仍需进一步完善。
