## 📖 一、总览

该仓库是 rcore-rust 家族中的一个实验型内核，在架构上超越了 rCore-Tutorial 的教学范围，尝试用 Rust 覆盖更多 Linux 系统调用和机制。它同时支持 RISC-V 和 LoongArch64 双平台，通过条件编译和统一抽象（如 COW 标记位 9）实现跨架构兼容。整体定位是“验证 Rust 在内核中实现 POSIX 兼容子集的可能性”，而非生产级完整内核。

值得关注的三个特点：第一，多架构统一抽象——启动流程与内存管理均通过 global_asm 和条件编译适配两种架构，且引入统一位 9 的 COW 标记屏蔽页表差异，减少架构耦合；第二，丰富的 IPC 与同步机制——除了管道和信号量，还完整实现了 futex（含 requeue）、共享内存组管理、条件变量和邮箱 IPC，管道读写会协同信号检查并返回 ERESTART，体现 Linux 兼容深度；第三，系统调用覆盖纵向深入——116 个系统调用中核心进程管理、文件 I/O、信号、定时器等已实现，虽然部分以 stub 或 panic! 占位，但已能支持动态链接加载和基本信号递送，展示了 Rust 内核的潜力。

## 🎯 二、综合评价

### 整体定位
如果说 rCore-Tutorial 是“能跑通教学实验的玩具内核”，那么本仓库就是“尝试把玩具做成工具的早期原型”。它保留了单内核架构和异步模型，但模块化程度更高：VFS、IPC、信号、网络等模块相对独立，syscall 层按功能域拆分（fs/net/process），且采用了 Linux 式 PCB/TCB 分离 [task.ref:1] 并自动加载动态链接器 [mm.ref:4]。这些设计表明开发者有意接近 Linux 内核架构，但限于资源，许多部分仅实现了基本路径，整体处于实验验证阶段。

### 真正的创新点
项目在多个模块中引入了超越教学内核的独特设计：
- **统一 COW 标记位 9**：在 RISC-V 和 LoongArch 的 PTE 中均使用第 9 位作为写时复制标志，避免架构相关的 PTE 操作重复 [mm.ref:1]。
- **动态链接器自动加载**：在 ELF 加载过程中主动查找并映射 ld.so，突破了传统教学内核只支持静态链接的限制 [mm.ref:4]。
- **信号栈帧手动构造**：不依赖 trampoline，而是直接构建用户态栈帧并用魔数 0xdeadbeef 校验完整性 [signal.ref:4]（仅 riscv64 实现）。
- **TCP 原子状态机**：TcpSocket 使用 AtomicU8 和 CAS 实现无锁状态转换，平衡了并发安全与性能 [net.ref:4]。
- **futex 实现完整**：支持 FUTEX_WAIT/FUTEX_WAKE 以及 FUTEX_REQUEUE，键设计基于物理地址和 pid，是内核中较复杂的同步原语 [ipc.ref:3]。

### 取舍判断
开发者的精力分配清晰可见：他们优先实现了系统调用广度（116 个）、多架构支持（尽管不均衡）和机制完整性（如动态加载、COW、futex），但在代码规范、错误处理和安全性上做了大量省略。例如，大量 unsafe 块缺少 `// SAFETY` 注释（尤其在 fs 和 mm 模块 [fs.ref:1]），许多系统调用直接返回 `-1` 而非精确错误码（如 syscall.ref:5 指出的 pselect6 实现），管道缓冲区大小、网络常量等硬编码，还有大量被注释掉的死代码 [mm.ref:5]。这些取舍使内核在当前状态下难以用于生产，但对验证新特性（如 Rust 写时复制、用户态信号栈）是有价值的实验平台。

### 完成度与不足
综合来看，该内核在前沿特性上有较好表现，但各模块普遍存在完成度不均和质量问题。已完成的关键机制包括：多架构启动、分页内存管理（COW、需求分页、动态链接器加载）、PCB/TCB 分离的进程线程管理、基于 VFS 的 ext4 文件系统、信号递送与 SA_RESTART、futex 及共享内存 IPC、smoltcp 网络栈（仅 loopback）、VirtIO 驱动。不足之处也很突出：信号栈帧仅 riscv64 实现 [signal.ref:15]，网络仅 loopback 无实际网卡驱动，SIGSTOP/SIGCONT 未实现 [signal.ref:14]，共享内存映射冲突未检查 [ipc.ref:5]，exec 中 heap_id 泄漏 [task.ref:3]，大量 unsafe 无安全说明，系统调用错误路径粗糙，代码中存在`// im not sure`等不确定注释 [mm.ref:5]。总体来说，它是一个有价值但未成熟的原型，代码质量和完整性均有较大提升空间。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 核心抽象是使用 global_asm! 宏嵌入目标架构（riscv64/loongarch64）的早期启动汇编，设置栈指针后直接跳转到 Rust 编写的 main 函数。这解决了从 CPU 复位到高级语言运行时环境的过渡。与 rCore-Tutorial 相比，本仓库通过条件编译支持龙架构，但缺少对 cpu hart id 的传递，可能影响多核兼容性；同时采用 u128 块清零 BSS 段以提升性能，但缺少安全注释。

### 1. 核心抽象：从汇编入口到 Rust 主函数的桥接

启动流程的第一个核心抽象是 `boot.rs` 中的 `global_asm!` 宏。该宏在编译时根据 `#[cfg(target_arch = ...)]` 生成对应的汇编入口。对于 `riscv64`，代码极简：仅设置栈指针（`la sp, boot_stack_top`）后直接调用 `rust_main`（即 `super::main`）。对于 `loongarch64`，则多了一段处理器初始化：通过 `pcaddi` 和移位确定当前 PC 的高位，并写入 CSR 寄存器 `0x181`（ERA）和 `0x180`（PRMD），确保地址窗口在跳转到高地址后保持不变。<sup>[1](#mod-boot-ref-1)</sup> 这种设计将平台相关的早期初始化隐藏在内联汇编中，使得 Rust 侧的 `main` 函数可以保持平台无关。

`main` 函数的签名是 `pub fn main(cpu: usize) -> !`，但汇编调用时并未传递 `cpu` 参数。在 RISC-V 规范中，`_start` 入口的 `a0` 寄存器通常携带 hart ID，但此处未做读取；龙架构的调用同样未设置参数。<sup>[2](#mod-boot-ref-2)</sup> 这意味着 `cpu` 参数的值是未指定的——对于单核系统可能无害，但在多核启动场景下将是严重缺陷。

栈的分配采用 `static mut` 数组 `BOOT_STACK[4096*16]`，并放置在 `.bss.stack` 段中。汇编中引用的 `boot_stack_top` 和 `boot_stack_lower_bound` 标签都直接基于该段布局。<sup>[3](#mod-boot-ref-3)</sup> 这种方法简单可靠，但栈大小（64KB）是硬编码的，未来若需支持更多核心或复杂初始化可能需要调整。

### 2. 启动初始化流水线的设计权衡

进入 `main` 后，执行顺序如下：`clear_bss` → `logging::init` → `mm::init` → `trap::init` → `timer::set_next_trigger`（仅 RISC-V）→ `fs::list_apps` → `task::add_initproc` → `fs::init` → `net::net_init` → `enable_timer_interrupt` → `task::run_tasks`。<sup>[8](#mod-boot-ref-8)</sup> 这个顺序隐含了依赖关系：日志系统最先初始化以便输出；内存管理必须在陷阱处理之前，因为陷阱可能涉及页表切换；文件系统在网络初始化之前，因为网络可能依赖文件系统中的配置。

值得注意的取舍是：`clear_bss` 使用 `u128` 类型进行填零操作。<sup>[4](#mod-boot-ref-4)</sup> 这比逐字节或逐字填充效率更高，但要求 BSS 段的起始和结束地址均对齐到 16 字节。如果链接脚本未保证对齐，则会产生未定义行为。此外，`clear_bss` 的 `unsafe` 块未附带任何 `// SAFETY:` 注释，违背了本仓库的安全编码规范。<sup>[4](#mod-boot-ref-4)</sup>

另一个取舍是条件编译：`timer::set_next_trigger()` 仅在 `#[cfg(target_arch = "riscv64")]` 下执行，而龙架构的定时器初始化在 `hal::trap::init` 内部或通过后续的 `enable_timer_interrupt` 完成。<sup>[5](#mod-boot-ref-5)</sup> 这种分散的条件分支增加了维护成本——如果未来新增架构，需要同时修改多处。

此外，`main` 函数末尾的 `task::run_tasks()` 是**不返回**的（`!` 类型），因此在调用后立即 `panic!("Unreachable section for kernel!")` 作为安全网。但若 `run_tasks` 因错误返回（目前其实现应永远不返回），则 panic 会触发，这符合防御性编程原则。

### 3. 跨模块协同：boot.rs 与 board、config 的配合

`main.rs` 通过 `#[path = "boards/qemu.rs"] mod board;` 引入了板级配置模块。<sup>[6](#mod-boot-ref-6)</sup> 这意味着编译时针对 QEMU 模拟器做了专门适配，例如内存布局、外设地址等。`board` 模块应提供 `board_init()` 或类似函数，但当前启动流程中并未显式调用——而是隐式地通过 `mm::init` 等使用 `board` 中的常量。这种设计的优点是将板级细节隔离，但缺点是若更换硬件平台，必须替换整个 `board.rs`，且无法在编译时防止遗漏初始化。

`config` 模块被声明为 `#[macro_use] pub mod config;`，提供了 `FLAG` 等宏或常量。`main` 的第一行就打印了 `FLAG`，用于标识内核版本或构建信息。<sup>[7](#mod-boot-ref-7)</sup> 这虽然简单，但为调试和版本追踪提供了便利。

另一个跨模块协同体现在 `boot.rs` 中的汇编入口与链接脚本的配合。虽然链接脚本未直接给出，但 `.bss.stack` 段的存在暗示了链接脚本中需要包含该段定义。若未正确放置，可能导致栈区域被其他数据覆盖。这是隐含的契约，文档中未说明。

### 4. 边角细节与不足

在审查启动流程时，我们发现了以下值得关注的边角细节和潜在问题：

1. **`unsafe` 块缺少安全契约** <sup>[4](#mod-boot-ref-4)</sup>：`clear_bss` 中的 `unsafe` 块未书写 `// SAFETY:` 注释，违反了 Rust 安全最佳实践。此外，`BOOT_STACK` 被声明为 `static mut`，虽然在内核启动阶段通常安全，但任何不慎的并发访问都可能导致数据竞争。

2. **条件编译的覆盖缺口** <sup>[1](#mod-boot-ref-1)</sup>：`boot.rs` 只处理了 `riscv64` 和 `loongarch64`。若为其他架构（如 `x86_64`、`aarch64`）编译，将因没有匹配的 `global_asm!` 导致链接失败。虽然本项目可能只针对这两种架构，但缺乏优雅的编译错误处理。

3. **硬编码的栈尺寸** <sup>[3](#mod-boot-ref-3)</sup>：`BOOT_STACK` 的大小 `4096*16` 是魔术数字，没有通过配置文件或编译时参数暴露。若未来需要更大的栈，必须手动修改源码。

4. **被注释掉的定时器初始化** <sup>[5](#mod-boot-ref-5)</sup>：`main.rs` 中有一行 `//hal::trap::enable_timer_interrupt();` 被注释掉，但随后又在后面调用了 `hal::trap::enable_timer_interrupt();`。这可能是调试残留或早期版本的片段，应清除以免误导读者。

5. **`main` 参数未初始化** <sup>[2](#mod-boot-ref-2)</sup>：如前所述，`cpu` 参数在汇编入口中未赋值，其值不确定。对于多核系统，这会导致后续任务初始化可能绑定到错误的核上。

6. **缺少启动失败路径处理** <sup>[8](#mod-boot-ref-8)</sup>：`mm::init`、`fs::init` 等函数均返回 `()`，若内部发生错误（如内存不足），当前启动流程会直接 panic 或静默失败，没有错误传播或回滚机制。

7. **龙架构 CSR 操作缺少注释** <sup>[1](#mod-boot-ref-1)</sup>：loongarch64 的汇编片段中修改了 CSR 寄存器，但没有解释每个步骤的目的，给维护者带来理解困难。

以上分析表明，该启动流程实现聚焦于快速完成单核心初始化，并利用条件编译支持两个指令集。但其在参数传递、错误处理和安全文档方面存在明显的早期实现痕迹。与 rCore-Tutorial 相比，本仓库增加了龙架构支持，但移除了对多核 hart ID 的显式处理，可能是一处退化。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块基于分页虚拟内存管理，核心抽象为 `MemorySet`（地址空间）与 `MapArea`（连续区域），通过 `PageTable` 操作三层页表（SV39/LA-3L）。它解决了用户进程地址空间的创建、ELF 加载、堆栈管理、mmap/munmap、动态链接器加载、写时复制（COW）和信号返回等需求。与 rCore-Tutorial 基线相比，本模块增加了 LoongArch64 架构支持、动态链接器自动加载、共享内存组管理、用户态安全拷贝以及统一位 9 的 COW 标记，但部分实现（如内核段映射）仍留有未清理的死代码。

### 1. 核心抽象与外部依赖

本模块将虚拟内存管理抽象为三层：`PageTable` 负责硬件页表操作，`MapArea` 描述一段连续的虚拟地址区域（包括映射类型、权限、文件映射等），`MemorySet`（实际为 `MemorySetInner`）则整合页表和所有 `MapArea`，代表一个完整的地址空间 <sup>[10](#mod-mm-ref-10)</sup>。`MemorySet` 通过 `UPSafeCell` 提供内部可变性，但调用者需保证写独占。跨架构支持通过 `#[cfg]` 分支实现：`PTEFlags` 在 RISC-V 和 LoongArch 上定义不同，但 `MapPermission` 到 `PTEFlags` 的转换函数（`map_area.rs:190-225`）统一了权限语义 <sup>[7](#mod-mm-ref-7)</sup>，上层代码无需关心 PTE 位布局。物理帧分配依赖 `frame_alloc`（全局分配器），页表分配中间页表时也会调用它（`page_table.rs` 中 `find_pte_create` 在缺页表项时 `frame_alloc().unwrap()`）。

### 2. 关键设计取舍

**COW 标记的统一化**：项目选择在 `PageTableEntry` 中使用第 9 位作为写时复制标志（`is_cow`/`set_cow`），而不管 RISC-V 和 LoongArch 的 PTE 保留位是否相同 <sup>[1](#mod-mm-ref-1)</sup>。这一设计避免了在缺页处理中嵌入架构相关分支，简化了 `cow_page_fault` 的实现。代价是占用了可能由硬件使用的位。

**用户态数据拷贝策略**：`copy_from_user`/`copy_to_user` 通过 `PageTable::translate_va` 逐页翻译目标虚拟地址，然后直接以物理地址指针读写（使用 `PhysAddr::get_mut`/`get_ref`）<sup>[3](#mod-mm-ref-3)</sup>。相比传统的逐个字节翻译，这种方式在拷贝大块数据时效率更高。但安全前提是页表项必须有效且物理地址映射到内核地址空间——LoongArch 下通过固定窗口直接访问物理地址，RISC-V 下相同物理地址空间。

**动态链接器自动加载**：`load_dl_interp_if_needed` 解析 ELF 的 `.interp` 节，并尝试从预配置路径（musl/glibc）加载动态链接器 <sup>[4](#mod-mm-ref-4)</sup>。这比 rCore-Tutorial 的静态链接模型更接近 Linux 行为，但也引入了复杂错误处理——当动态链接器文件不存在或 ELF 损坏时需回退到静态。该函数还保留了对 musl 程序误用 glibc 链接器的警告。

**信号返回路径**：`map_trampoline` 不仅映射 `TRAMPOLINE`（trap 入口），还将 `SIGRETURN` 固定虚拟地址映射到 `__sigreturn` 函数 <sup>[5](#mod-mm-ref-5)</sup>。这使得信号处理完成后能通过跳转到固定地址返回到用户态，避免了内核与用户态之间的复杂切换。该设计在 rCore-Tutorial 中并未出现，属于本项目针对信号机制的增强。

**坏地址表**：`BAD_ADDRESS` 全局表记录被标记为坏地址的虚拟地址（如 mmap 失败或 unmap 后的地址）<sup>[6](#mod-mm-ref-6)</sup>。该表用于快速判断地址有效性，但存在多线程竞争（通过 `Mutex` 保护），且仅当显式调用 `insert_bad_address` 才记录，遗漏可能导致安全漏洞。

### 3. 跨模块协同

内存管理模块与系统调用、缺页处理、信号、文件系统紧密配合：

- **缺页处理**：`MemorySet` 导出了 `lazy_page_fault` 和 `cow_page_fault` 方法，缺页异常处理函数根据 `scause` 调用它们。在 `safe_translated_byte_buffer` 中，当从用户空间拷贝数据时若发现页表无效，也会主动触发 `lazy_page_fault` 来按需分配页面 <sup>[9](#mod-mm-ref-9)</sup>。这使得用户态内存访问的前置条件（页面存在）在拷贝过程中动态满足。

- **系统调用**：`mmap`/`munmap`/`mprotect`/`brk` 等系统调用最终调用 `MemorySet` 对应方法。例如 `mmap` 会根据 `MmapFlags` 决定是否创建共享组（`groupid`）<sup>[11](#mod-mm-ref-11)</sup>；`munmap` 通过 `munmap` 方法取消映射并清理帧。`mprotect` 则直接调用 `PageTable::set_map_flags` 修改权限。

- **信号模块**：`__sigreturn` 函数（定义在 `os/src/signal`）被映射到 `SIGRETURN` 地址，信号处理结束后 CPU 跳转到此地址恢复上下文。`map_trampoline` 还在内核地址空间映射了 `strampoline`（trap 入口），与 trap handler 配合完成用户态/内核态切换。

- **文件系统**：`mmap` 支持文件映射，`MapArea` 持有 `MmapFile`（包含 `Arc<OSInode>`），页面数据需从文件读取。但当前实现中文件映射的页面数据读取似乎尚未完整实现（`map_one` 中未处理 `MmapFile`）。

### 4. 边角细节与不足

1. **架构覆盖不完整**：`#[cfg]` 仅匹配 `riscv64` 和 `loongarch64`，若在其他架构（如 x86、aarch64）上编译，大量代码将被跳过，导致编译错误。未提供默认分支或编译时告警。
2. **unsafe 块缺乏安全文档**：`PhysAddr::get_ref` 和 `get_mut` 均无 `// SAFETY:` 注释（见 `address.rs`），调用者需自行保证物理地址有效且无别名。`PhysPageNum::get_pte_array` 同样缺少边界检查，传递无效页号可能导致越界访问。
3. **潜在用户态触发 panic**：`PageTable::map` 使用 `assert!(!pte.is_valid())`，若用户通过 `mmap` 重复映射同一虚拟地址，将导致内核 panic（`page_table.rs` `map` 方法）。Linux 对此返回 `-EEXIST` 或修改现有映射，本项目可能应该返回错误而非 panic。
4. **死代码残留**：`new_kernel` 中存在大片被 `//` 注释掉的旧映射代码（涉及 `s_sig_trap` 区域）<sup>[13](#mod-mm-ref-13)</sup>，表明开发过程中曾尝试将信号代码放在独立区域但未清理。这种技术债增加维护成本。
5. **TODO 与不确定实现**：`flush_tlb` 中 LoongArch 指令 `tlbflush` 旁有注释 `// im not sure` <sup>[12](#mod-mm-ref-12)</sup>，表明该指令的正确性未验证。此外，`new_kernel` 中多次出现 `debug!` 输出细节，发布时应转为 `trace!` 级别。
6. **共享组生命周期**：`MapArea` 的 `Drop` 实现在 `GROUP_SHARE.lock().del_area(self.groupid)`（`map_area.rs` Drop 实现），但 `MapArea` 可能被克隆（`from_another`），导致 groupid 被重复删除，可能引发共享组状态不一致。

        while bytes_copied < dst.len() {

> 💡 **TL;DR**
>
> 该模块以 Linux 式 PCB/TCB 分离模型为核心，通过 `ProcessControlBlock` 承载地址空间、文件表等进程级资源，`TaskControlBlock` 维护每个线程的内核栈与 trap 上下文。调度器采用简单的 FIFO 就绪队列，但预留了 `Stride` 优先级域。与 rCore-Tutorial 基线相比，实现了 `clone` 系统调用、线程组管理、`clear_child_tid`/futex 唤醒、`Tms` 时间统计以及更完整的 `exec` 参数布局，同时通过条件编译支持 RISC-V 和 LoongArch64 双架构。

### 1. 核心抽象：PCB 与 TCB 的职责分离

该模块将进程与线程明确分离：`ProcessControlBlock` (PCB) 管理进程级资源——内存集 (`memory_set`)、文件描述符表 (`fd_table`)、文件系统信息 (`fs_info`)、信号表 (`sig_table`)、互斥体/信号量/条件变量列表、优先级与 stride 值、时间统计 (`Tms`) 以及 `clear_child_tid` 和 `robust_list` 等 Linux 兼容字段 <sup>[1](#mod-task-ref-1)</sup>。`TaskControlBlock` (TCB) 则代表一个可调度实体，持有指向所属进程的 `Weak` 引用、内核栈 (`kstack`)、trap 上下文物理页号、任务上下文 (`task_cx`) 和状态，以及每个线程独立的信号屏蔽字 <sup>[2](#mod-task-ref-2)</sup>。

这种分离使得同一个进程的多个线程共享地址空间和打开文件表，而各自拥有独立的内核栈和 trap 上下文。与 rCore-Tutorial 将 PCB 和 TCB 合并在单一 `TaskControlBlock` 中的做法相比，本设计更贴近 Linux 的 `task_struct` 与 `signal_struct` 分离的哲学，但代价是增加了通过 `Weak` 引用访问进程的间接性，并引入了 `inner_exclusive_access()` 多层锁获取的开销。

### 2. 进程生命周期：fork/exec/exit 的实现细节

**fork** (`ProcessControlBlock::fork`) 实现了完整的 `clone` 语义，通过 `CloneFlags` 位标志控制资源共享。当未设置 `CLONE_VM` 时，通过 `MemorySet::from_existed_user` 创建独立的地址空间（写时复制由底层的页表复制实现）；当设置时则直接 `Arc::clone` 共享内存集 <sup>[5](#mod-task-ref-5)</sup>。对于子进程的 trap 上下文和用户栈，代码显式调用 `memory_set.clone_area` 将父进程对应页复制到子进程空间 <sup>[3](#mod-task-ref-3)</sup>。值得注意的是，当 `CLONE_THREAD` 被设置时，`fork` 不会创建新进程，而是仅在当前进程内创建一个新线程，共享全部资源，并将新 TCB 加入就绪队列。

**exec** (`ProcessControlBlock::exec`) 的实现在用户栈上手动构建了 Linux 风格的 argv/envp/aux 向量布局：依次将环境变量字符串、参数字符串写入用户栈指针递减后的位置，然后构造指针数组和辅助向量 (Aux)，最终设置 trap 上下文的入口地址和参数寄存器（a0-a3 或等效的 loongarch 寄存器） <sup>[4](#mod-task-ref-4)</sup>。这种手动布局避免了类似 rCore-Tutorial 中依赖 `UserStack` 封装的复杂性，但要求对栈布局有精确控制，且缺失对 `ELF` 辅助向量中 `AT_RANDOM` 等项目的安全检查。

**exit** (`exit_current_and_run_next`) 处理了 `clear_child_tid` 的 futex 唤醒：如果进程设置了 `CLONE_CHILD_CLEARTID`，退出时会在子进程 tid 地址写 0 并通过 `futex_wake_up` 唤醒等待在该地址的线程 <sup>[6](#mod-task-ref-6)</sup>。主线程退出时，进程变为僵尸，唤醒父进程，并回收所有线程的资源。这里一个微妙的设计是：对于非主线程（tid != 0），在 RISC-V 上通过 `add_stopping_task` 保留内核栈引用直到下次调度确保栈不被回收，而 LoongArch64 上直接 `drop(task)`，这在单核下可能安全，但代码注释说明不够清晰。

### 3. 调度器与多线程支持：简单 FIFO 与预留的 Stride

调度器核心是 `TaskManager`，维护一个 `VecDeque<Arc<TaskControlBlock>>` 作为就绪队列，`fetch` 从队首弹出 <sup>[5](#mod-task-ref-5)</sup>。此外还有一个按 pid 索引的 `block_map` 用于阻塞队列，以及一个 `stop_task` 槽位用于暂存停止中的任务。`Processor` 结构体保存当前运行的任务引用和 idle 任务上下文，`run_tasks` 循环不断 `fetch_task` 并调用 `__switch` 切换到下一个任务。当没有就绪任务时，调用 `check_timer` 处理定时器中断可能唤醒的任务。

PCB 中预留了 `priority` 和 `stride: Stride` 字段，`Stride` 结构体来自单独的 `stride.rs` 模块，表明开发者有意向实现优先级调度，但当前调度器完全忽略这些字段，仍是纯 FIFO。这是典型的「预留接口、后续实现」模式，但目前的调度器在长时间阻塞场景下可能缺乏公平性。

线程组通过全局 `THREAD_GROUP`（`Mutex<BTreeMap<usize, Vec<Arc<ProcessControlBlock>>>>`）维护，由 `insert_into_thread_group` 和 `remove_all_from_thread_group` 管理，主要用于 `exit` 时向整个线程组发送信号 <sup>[7](#mod-task-ref-7)</sup>。但该数据结构与 `PID2PCB` 和 `TID2TCB` 存在冗余，且锁类型不一致（`Mutex` vs `UPSafeCell`），可能造成死锁风险。

### 4. 跨模块协同：信号、futex、时间统计的集成

信号处理与调度模块紧密耦合：`check_signals_of_current` 读取当前进程的 `signals` 标志位，由中断处理或系统调用路径调用；`send_signal_to_thread_group` 通过线程组遍历所有线程设置挂起信号。TCB 中的 `sig_mask` 和 `sig_pending` 字段支持 per-thread 信号屏蔽，但 `exec` 中仅重置了 `sig_table` 而未清理 `sig_pending`，可能导致旧信号残留。

futex 机制在 `exit_current_and_run_next` 中通过 `futex_wake_up` 唤醒等待 `clear_child_tid` 地址的线程，这是 Linux 线程库（如 glibc）用于线程同步的重要原语。`wakeup_futex_task` 函数专门用于将 futex 阻塞的任务重新加入就绪队列 <sup>[6](#mod-task-ref-6)</sup>。

时间统计 `Tms` 在每次调度进入用户态时记录 `begin_urun_time`，在 trap 返回时计算 `utime` 和 `stime` 并累加到父进程的 `cutime`/`cstime` <sup>[9](#mod-task-ref-9)</sup>。但实现中存在竞争：`set_utime` 在持有子进程锁时又尝试获取父进程锁，若父进程同时访问子进程可能死锁（当前单核假设下因不可抢占而避免，但设计上脆弱）。

### 5. 边角细节与不足

- **`<sup>[10](#mod-task-ref-10)</sup>`** `UPSafeCell::new` 在多处使用 `unsafe` 块，但几乎全部缺少 `// SAFETY:` 注释（如 `process.rs:266`），违背了 Rust 安全规范。安全契约未文档化，后续维护者容易误引入未定义行为。
- **`<sup>[8](#mod-task-ref-8)</sup>`** `change_program_brk` 中 `grow < 0` 时执行 `inner.heap_top as isize + grow`，若 `grow` 的绝对值大于 `heap_top` 会导致整数下溢为很大的正数，虽然后续有 `shrink < inner.heap_bottom` 检查，但行为仍是错误的（应返回错误而非静默失败）。
- **`exec` 中 heap_id 泄漏**：每次 `exec` 调用 `heap_id_alloc()` 分配新 heap_id，但旧 `heap_id` 仅在内核栈退场时在 `exit_current_and_run_next` 中释放；若进程连续 `exec`，每次都会泄漏旧的堆区域。
- **`<sup>[2](#mod-task-ref-2)</sup>`** TCB 的 `dealloc_user_res` 在释放用户栈时假设 `ustack_top` 非零代表分配了栈，但 `fork` 中子线程可能设置 `stack != 0` 导致 `alloc_user_res(false)` 即 `ustack_top` 为零，此时不会释放用户栈，可能造成物理页泄漏。
- **`<sup>[5](#mod-task-ref-5)</sup>`** `TaskManager` 的 `block_map` 以 pid 为键，但一个进程可能有多线程，当某个线程阻塞时整个进程被加入阻塞映射，而同一进程的其他线程可能还在就绪队列，容易导致状态不一致。不过当前 `add_block_task` 仅在单线程场景下使用（`block_current_and_run_next`），尚未暴露问题。
- **`<sup>[6](#mod-task-ref-6)</sup>`** `exit_current_and_run_next` 中，非主线程（`tid != 0`）在 RISC-V 上通过 `add_stopping_task` 保留引用防止内核栈回收，但 LoongArch64 上直接 `drop(task)`，未见对应保护，若该线程在被调度前内核栈被回收将导致 use-after-free。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 文件系统模块围绕 VFS 抽象（Inode trait 和 File trait）构建，底层以 ext4 作为主要持久化文件系统（通过 lwext4_rust 绑定），同时支持设备文件、管道等抽象文件。与标准 ArceOS 相比，本仓库引入了独立的动态链接库路径映射、伪文件系统初始化（/proc、/dev 等）以及精细的符号链接控制（O_ASK_SYMLINK）。模块与进程管理、信号机制紧密耦合，管道读写中会检查待处理信号并可能返回 ERESTART。整体设计体现了 POSIX 兼容性，但存在不安全代码缺乏安全文档、硬编码常量、死代码和路径缓存一致性等早期实现特征。

### 1. VFS 架构与 Inode/File 抽象
核心抽象是 Inode trait 和 File trait。Inode 代表文件系统中的一个节点，提供 create、find、read_at、write_at、fstat、read_dentry 等操作。File 代表一个已打开的文件实例，提供 read、write、fstat、poll 等操作。二者分离使得 VFS 层可统一管理不同后端的文件。

Ext4Inode 是 ext4 的后端实现，内部包装 lwext4_rust 的 Ext4File 对象，并大量使用 `inner.get_unchecked_mut().f` 获取底层对象。这些 `unsafe` 调用全部缺少 `// SAFETY` 注释，违背了 Rust 安全契约 <sup>[1](#mod-fs-ref-1)</sup>。设备文件（如 /dev/zero、/dev/null）不经过 ext4，而是直接实现 File trait，并通过全局 DEVICES 注册表挂接到 VFS 层 <sup>[2](#mod-fs-ref-2)</sup>。管道使用固定大小（65536 字节）的环形缓冲区 PipeRingBuffer 实现，读写端共享一个 `Arc<Mutex<PipeRingBuffer>>` <sup>[3](#mod-fs-ref-3)</sup>。

真正的 ext4 文件与抽象文件（设备、管道）通过 FileClass 枚举（File/Abs/Sock 变体）统一，方便文件描述符表管理。FileClass 提供了 `file()`、`abs()`、`sock()` 等转换方法，若类型不匹配则返回 EINVAL。

### 2. 路径解析、符号链接与动态库映射
open 函数是文件访问的统一入口，其流程体现 VFS 灵活性：1）检测是否为已注册的设备文件，若是则返回对应抽象 File；2）对于动态链接库文件（如 .so），调用 `map_dynamic_link_file` 将路径映射到已知库路径（例如 ld-musl-riscv64-sf.so.1 回退到 libc.so）；3）从全局 inode_idx 缓存或根 inode 中查找。

Ext4Inode::find 是核心查找方法，它利用 lwext4_rust 的 `check_inode_exist` 判断类型。对于符号链接，若未设置 O_ASK_SYMLINK 标志，则递归解析（最多 5 次），超过返回 ELOOP <sup>[4](#mod-fs-ref-4)</sup>。open 内部还处理 O_DIRECTORY、O_APPEND、O_TRUNC 等标志，但 O_TMPFILE 定义却未实现。

全局 inode_idx 缓存（HashSet）用于加速重复路径查找，但 unlink 后缓存不自动删除，可能导致返回已删除的 inode <sup>[5](#mod-fs-ref-5)</sup>。这是明显的设计缺陷。

### 3. 文件描述符表与进程资源
每个进程维护一个 FdTable（文件描述符表）和一个 FsInfo（文件系统信息）。FdTable 内部是 `Vec<Option<FileDescriptor>>`，支持软硬限制（soft_limit=128, hard_limit=256）。`alloc_fd` 在限制内分配新描述符，`alloc_fd_larger_than` 用于 dup3 等需要指定最小 fd 的场景 <sup>[6](#mod-fs-ref-6)</sup>。每个 FileDescriptor 保存 OpenFlags 和 FileClass，支持 close-on-exec 和非阻塞标志。

FsInfo 存储进程的 cwd、exe、fd2path 映射和 umask。umask 默认为 022，可通过 `set_umask` 修改 <sup>[7](#mod-fs-ref-7)</sup>。fd2path 映射在 open/close 时由上层 syscall 维护，但 dup 等操作后可能不更新路径对应关系。

### 4. 管道实现与信号协同
管道的读写是阻塞的，但在执行阻塞前会检查当前任务是否有 pending 信号（除 SIGCHLD 外），若有则返回 ERESTART 让系统调用重启 <sup>[3](#mod-fs-ref-3)</sup>。写端在写入前检测读端是否全部关闭（通过弱引用），若关闭则发送 SIGPIPE 信号并返回 EPIPE。读端类似，写端全部关闭且缓冲区为空时返回 0（EOF）。

pipe 的 `poll` 实现正确检测 POLLIN、POLLOUT、POLLHUP、POLLERR 事件，支持 select/poll 系统调用。注意，管道未处理 O_NONBLOCK 标志，非阻塞模式下行为未定义，可能不符合 POSIX。

### 5. 边角细节与不足
（1）**不安全代码缺乏安全文档**：Ext4Inode 中几乎所有方法都使用 `inner.get_unchecked_mut()` 而无任何 SAFETY 注释，是重要的技术债务 <sup>[1](#mod-fs-ref-1)</sup>。
（2）**硬编码常量**：MAX_PATH_LEN=50、RING_BUFFER_SIZE=65536、MAX_LOOPTIMES=5，以及 fstat 时间修正中的魔法数字 4000000000，缺乏可移植性 <sup>[8](#mod-fs-ref-8)</sup>。
（3）**错误处理不稳健**：`create_init_files` 中多次 `unwrap()`，若文件创建失败会导致系统启动 panic <sup>[9](#mod-fs-ref-9)</sup>。
（4）**死代码**：inode.rs 中定义了 `LINK_CACHE` 但从未使用。
（5）**特性缺失**：O_TMPFILE、O_ASYNC 等标志定义但未实现；管道不支持非阻塞 I/O。
（6）**inode_idx 缓存与 VFS 一致性风险**：缓存只在 open 时插入，unlink 后未及时删除，可能返回已删除的 inode <sup>[5](#mod-fs-ref-5)</sup>。
（7）**动态库路径映射不通用**：仅预定义部分 .so 路径，当应用程序使用库文件不在列表中时仍返回原始路径会导致 ENOENT。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块实现了与 Linux 兼容的信号机制，核心抽象为 SignalFlags（位标志集合）、SigAction/KSigAction（信号动作描述）和 SigTable（进程级信号动作表）。它解决了用户态信号处理程序的注册、发送、屏蔽和递送问题，支持 SA_RESTART、SA_SIGINFO 等标志。与同家族基线（ArceOS/rCore）相比，本模块在信号栈帧布局上使用手动构造的帧（非 trampoline 段），并通过魔数 0xdeadbeef 校验帧完整性，同时 SIGSTOP/SIGCONT 等操作尚未实现。

### 1. 核心抽象与外部依赖

信号模块围绕三个核心数据结构构建：`SignalFlags`、`SigAction`/`KSigAction` 和 `SigTable`。`SignalFlags` 是一个 `bitflags` 类型，每个信号占用一个位（1～33），用于表示待决信号集合（`sig_pending`）和信号掩码（`sig_mask`）<sup>[1](#mod-signal-ref-1)</sup>。`SigAction` 是对标 Linux `struct sigaction` 的 C 兼容结构，包含 `sa_handler`（用户处理函数地址）、`sa_flags`（一组 `SigActionFlags`）、`sa_restore`（恢复函数地址）和 `sa_mask`（执行期间自动屏蔽的信号）。内核内部使用 `KSigAction` 包装 `SigAction`，增加 `customed` 字段标识是否已被用户自定义 <sup>[2](#mod-signal-ref-2)</sup>。`SigTable` 封装了长度为 `SIG_MAX_NUM+1`（34）的 `KSigAction` 数组，并附带一个 `group_exit_code` 用于进程组退出同步 <sup>[7](#mod-signal-ref-7)</sup>。

这些类型依赖外部同步原语 `UPSafeCell`（无锁内部可变性，假设单核或中断禁用）和内存翻译函数 `translated_ref`/`translated_refmut`（用于安全读写用户态栈空间）。Trap 上下文类型 `MachineContext` 和 `UserContext` 由 hal 层提供，用于保存/恢复寄存器状态。

### 2. 关键设计取舍

**架构耦合与条件编译**：`setup_frame` 函数被 `#[cfg(target_arch = "riscv64")]` 守卫，目前仅对 RISC‑V 生效 <sup>[3](#mod-signal-ref-3)</sup>。这意味着在其他架构（x86_64、aarch64、loongarch64）上编译时，自定义信号处理函数将不会被调用，用户进程可能无响应或崩溃。这是一个显式的早期实现取舍，尚未抽象出跨架构的信号帧布局接口。

**用户栈帧的手动构造**：模块没有使用内核辅助的 sigreturn trampoline（如 vDSO），而是在用户栈上直接压入 `MachineContext`、信号掩码、SA_SIGINFO 相关结构，最后写入魔数 `0xdeadbeef` 作为帧结束标记 <sup>[4](#mod-signal-ref-4)</sup>。`restore_frame` 通过检查该魔数来验证帧完整性。这种设计避免了额外代码段映射，但栈布局高度耦合于 RISC‑V 寄存器约定，且魔数硬编码为字面常量，若将来需要调整布局（例如加入浮点寄存器），必须同步修改两处校验逻辑。

**SA_RESTART 支持**：模块在 `setup_frame` 中检测当前异常是否为 `UserEnvCall` 且系统调用返回值等于 `ERESTART`，若设置了 `SA_RESTART` 则将 `sepc` 回退 4 字节并恢复 `a0`，使系统调用在信号处理后重新执行 <sup>[5](#mod-signal-ref-5)</sup>。这种实现在单线程场景下可行，但在多线程场景下，系统调用中断状态可能与其他线程的修改冲突，需要更细致的同步。

**默认操作与不可屏蔽信号**：`default_op` 函数按 Linux 惯例分类信号默认操作（Terminate、CoreDump、Ignore、Stop、Continue）<sup>[1](#mod-signal-ref-1)</sup>。但 `SIGKILL`（9）和 `SIGSTOP`（19）在 Linux 中不可被屏蔽或自定义，而本模块并未强制限制——用户可以通过 `rt_sigaction` 修改这些信号的动作，这可能导致语义偏差。同样，`SIGCONT` 的默认操作为 `Continue`，但模块未实现 `Stop` 和 `Continue` 状态机，相关代码以 TODO 标记 <sup>[6](#mod-signal-ref-6)</sup>。

### 3. 跨模块协同

信号机制与任务管理、系统调用、Trap 处理深度耦合：
- **信号发送**：系统调用 `kill`、`tkill`、`tgkill` 最终调用 `add_signal`，后者向目标 `TaskControlBlock` 的 `sig_pending` 添加信号位，并可能唤醒停止状态的任务（当前注释掉）<sup>[8](#mod-signal-ref-8)</sup>。`send_signal_to_thread_group` 遍历进程的所有线程逐一添加，这需要全局的 `THREAD_GROUP` 锁。
- **信号递送**：在从内核返回用户态之前（推测在 `os/src/trap/trap.S` 或 `trap_handler` 中），`check_if_any_sig_for_current_task` 被调用，若有待决且未屏蔽的信号，则调用 `handle_signal` 处理最高优先级的信号（通过 `peek_front` 取最低位）。`handle_signal` 根据 `KSigAction::customed` 决定是调用用户处理函数还是执行默认操作（如终止进程）<sup>[9](#mod-signal-ref-9)</sup>。
- **信号返回**：用户处理函数执行完毕后，会跳转到 `ra` 寄存器指向的 sigreturn 代码。若设置了 `SA_RESTORER`，则使用用户提供的 `sa_restore`；否则使用内核预置的 `SIGRETURN` 地址。该地址对应的 `__sigreturn` 汇编函数会触发 `rt_sigreturn` 系统调用，最终调用 `restore_frame` 恢复原始 trap 上下文 <sup>[5](#mod-signal-ref-5)</sup><sup>[10](#mod-signal-ref-10)</sup>。

`SigTable` 通过 `UPSafeCell` 实现进程级别的共享，`TaskControlBlock` 则存储每个线程独立的 `sig_pending` 和 `sig_mask`。这种“进程共享动作表、线程独立屏蔽字”的设计与 Linux 的线程信号模型一致。

### 4. 边角细节与不足

<sup>[11](#mod-signal-ref-11)</sup> **硬编码魔数**：`0xdeadbeef` 直接在 `setup_frame` 写入、`restore_frame` 校验，若用户态损坏该值会导致内核 `assert!` panic，构成 DoS 入口。

<sup>[12](#mod-signal-ref-12)</sup> **多处 `unwrap()` 风险**：`current_task().unwrap()`、`process.upgrade().unwrap()`、`SignalFlags::from_sig(signo).unwrap()` 等在无效 tid 或进程已退出时会直接 panic。用户态通过传递非法参数可能触发内核崩溃。

<sup>[13](#mod-signal-ref-13)</sup> **`assert!` 在 `restore_frame`**：`assert!(checkout == 0xdeadbeef, "restore frame checkout error!");` 可以被用户故意破坏栈帧触发，导致内核 panic。应当返回错误码而非直接 panic。

<sup>[14](#mod-signal-ref-14)</sup> **TODO 未完成功能**：`SigAction::new` 注释中 `TODO(ZMY): 添加Stop状态和相关函数` 表明 `SIGSTOP`/`SIGCONT`/`SIGTSTP` 等信号的暂停/继续行为尚未实现。相关处理仅占位为 `SigOp::Stop`，实际未挂起线程。

<sup>[15](#mod-signal-ref-15)</sup> **架构覆盖不足**：`handle_signal` 中只有 `#[cfg(target_arch = "riscv64")]` 分支调用 `setup_frame`，其他架构（包括 loongarch64 中已注册的 `estat` 模块）没有对应的帧构造代码。如果编译非 RISC‑V 内核且用户注册了自定义 handler，信号将静默忽略，导致用户程序行为异常。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本仓实现了涵盖管道(fifo)、futex、共享内存、信号量、条件变量、互斥锁、邮件等多种IPC原语，均集成在rCore-Rust内核中。管道通过File trait接入VFS，使用spin::Mutex保护的环形缓冲区实现阻塞读写，并感知信号中断；futex基于物理地址+pid的键和全局等待队列提供用户态快速互斥内核支持；共享内存通过页框分配和地址空间映射实现；信号量使用UPSafeCell和等待队列。与rCore-Tutorial相比，增加了futex的requeue操作、共享内存完整syscall、邮件IPC等，但部分机制（如管道缓冲区大小）仍为硬编码。

### 1. 多种IPC原语的分类与抽象集成
本仓的IPC模块并不集中于单一文件，而是分散在`fs/pipe.rs`（管道）、`task/futex.rs`、`mm/shm.rs`、`sync/semaphore.rs`以及未展示的condvar/mutex/mail中。这种分布体现了设计分层：**数据传递类**（管道、共享内存、邮件）依赖文件系统或内存管理抽象；**同步原语类**（futex、信号量、条件变量、互斥锁）直接与任务调度器交互。

管道通过实现`File` trait接入VFS<sup>[1](#mod-ipc-ref-1)</sup>，因此可以像普通文件一样被`read`/`write`/`poll`系统调用操作，这是Unix哲学的体现。共享内存则通过`MemorySet::shm`直接操作页表<sup>[2](#mod-ipc-ref-2)</sup>，绕过文件系统获得更高性能。futex和信号量均使用内核等待队列阻塞/唤醒任务<sup>[3](#mod-ipc-ref-3)</sup><sup>[4](#mod-ipc-ref-4)</sup>，但与任务控制块的引用方式不同：futex采用`Weak<TaskControlBlock>`避免循环引用，信号量则使用`Arc`。

### 2. 管道：环形缓冲区的阻塞I/O与信号中断协同
`Pipe`结构体包含`readable`/`writable`布尔标志和`Arc<Mutex<PipeRingBuffer>>`。环形缓冲区通过`head`/`tail`指针和`status`（Full/Empty/Normal）管理65536字节的`Vec<u8>`<sup>[5](#mod-ipc-ref-5)</sup>。

**阻塞逻辑**：在`read`和`write`中，首先检查当前任务是否有非SIGCHLD信号挂起，若有则返回`ERESTART`让用户态重试<sup>[6](#mod-ipc-ref-6)</sup>；否则检查缓冲区是否可用，若不可用且对端未关闭则调用`suspend_current_and_run_next()`永久阻塞<sup>[7](#mod-ipc-ref-7)</sup>。被唤醒后再次检查信号和缓冲区状态。这种循环保证了信号的及时响应。

**写端关闭检测**：通过`PipeRingBuffer`中保存的读端/写端弱指针，调用`all_write_ends_closed()`/`all_read_ends_closed()`判断对端是否全部关闭<sup>[8](#mod-ipc-ref-8)</sup>。若写端写入时读端已关闭，则发送`SIGPIPE`信号并返回`EPIPE`<sup>[9](#mod-ipc-ref-9)</sup>。这与POSIX行为一致。

**性能取舍**：使用`spin::Mutex`而非`UPSafeCell`，因为管道可能被多核访问，自旋锁在短临界区中效率更高。但缓冲区大小固定为64KB，无法根据负载调整，可能造成内存浪费或吞吐受限。

### 3. Futex：基于物理地址的细粒度等待队列
`FutexKey`由物理地址`PhysAddr`和进程PID组成<sup>[10](#mod-ipc-ref-10)</sup>，确保不同进程即使映射相同虚拟地址也不会互相干扰。全局等待队列`FUTEX_QUEUE`是一个`BTreeMap<FutexKey, VecDeque<Weak<TaskControlBlock>>>`<sup>[11](#mod-ipc-ref-11)</sup>。

`futex_wait`将当前任务以弱引用形式加入对应键的队列尾部，然后调用`block_current_and_run_next()`阻塞。被唤醒后检查信号位图，若存在未屏蔽信号则返回`EINTR`<sup>[12](#mod-ipc-ref-12)</sup>。`futex_wake_up`从队列头部弹出最多`max_num`个任务，通过`wakeup_futex_task`唤醒<sup>[13](#mod-ipc-ref-13)</sup>。`futex_requeue`实现更复杂的操作：先唤醒最多`max_num`个任务，再将剩余最多`max_num2`个任务转移到另一个键的队列<sup>[14](#mod-ipc-ref-14)</sup>，支持例如`FUTEX_REQUEUE`语义。

**设计决策**：使用弱引用是为了避免任务控制块被等待队列长期引用而无法释放。但全局一把`Mutex`可能成为高并发futex操作的热点，后续可考虑哈希分桶。

### 4. 共享内存与信号量：经典机制的内核实作
**共享内存**：`Shm`结构持有`Vec<Arc<FrameTracker>>`，每个`FrameTracker`代表一个物理页框。`ShmManager`的全局`BTreeMap`维护key到Shm的映射，key从1开始递增。`shm_create`根据大小分配`num`个页框<sup>[15](#mod-ipc-ref-15)</sup>，`shm_attach`将页框集合映射到当前进程地址空间<sup>[16](#mod-ipc-ref-16)</sup>（调用`MemorySet::shm`，该函数将页框插入页表并设置权限）。`shm_drop`仅从Manager中移除映射，但物理页框因`Arc`引用计数不会立即释放——若仍有进程attach，页框保持有效，这符合预期行为。

**信号量**：`SemaphoreInner`包含`count`和`VecDeque<Arc<TaskControlBlock>>`等待队列。`up`操作增加count，若count≤0（说明有等待者）则唤醒队列头部任务<sup>[17](#mod-ipc-ref-17)</sup>；`down`操作减少count，若count<0则将当前任务入队并阻塞<sup>[18](#mod-ipc-ref-18)</sup>。使用`UPSafeCell`保证单核安全，但在多核环境下需要外部同步（当前内核可能为单核）。

### 5. 边角细节与改进空间
- **<sup>[5](#mod-ipc-ref-5)</sup> 硬编码常量**：`RING_BUFFER_SIZE`固定为65536，既不能根据系统内存调整，也无法在运行时配置。大块连续`Vec`分配可能失败或浪费。
- **<sup>[11](#mod-ipc-ref-11)</sup> 全局锁粒度**：所有futex操作共用一把`Mutex<BTreeMap<...>>`，当大量线程同时进行futex_wait/wake时，锁竞争可能严重。替代方案是对FutexKey哈希分桶。
- **<sup>[10](#mod-ipc-ref-10)</sup> FutexKey使用PhysAddr**：物理地址在开启ASLR的系统中仍可能被预测，但内核保证物理页分配随机性有限；不过物理地址泄漏本身不是安全问题（用户态无法直接访问）。
- **<sup>[18](#mod-ipc-ref-18)</sup> 信号量等待队列使用Arc**：每个等待任务在队列中持有`Arc<TaskControlBlock>`，如果任务被唤醒后不移除（但up会pop_front），理论上可能导致任务控制块引用计数永远不为0（如果其他Arc还存在）。但实际中TaskControlBlock生命周期由进程管理，等待队列的Arc不会导致泄漏，因为唤醒后即移除。
- **未实现的特性**：管道不支持非阻塞标志（O_NONBLOCK）处理（pipe.rs的read/write总是阻塞）；共享内存未实现`shmctl`的IPC_RMID等操作；futex未支持`FUTEX_CMP_REQUEUE`和优先级翻转避免。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块基于 smoltcp 库，封装了 TcpSocket 和 UdpSocket 两个核心类型，通过全局 SocketSetWrapper 管理所有 socket 句柄。核心抽象是围绕 smoltcp 的 SocketHandle 和原子状态机实现的 POSIX 兼容 TCP/UDP 套接字。该模块解决了在单内核（unikernel）环境下提供标准 socket API 的问题，与 ArceOS 基线相比，增加了精细的原子状态机、非阻塞支持、预创建 listen 机制以及更丰富的错误映射。

### 1. 核心抽象与外部依赖

网络模块的核心依赖是 `smoltcp`（一个轻量级 TCP/IP 协议栈）。所有 socket 操作都通过 `smoltcp::iface::SocketSet` 间接完成。`SocketSetWrapper` 结构体包装了 `Mutex<SocketSet>`，提供线程安全的添加、查询和修改操作 <sup>[1](#mod-net-ref-1)</sup>。每个 TcpSocket 或 UdpSocket 持有一个 `SocketHandle`，通过 `SOCKET_SET.with_socket_mut` 方法来操作底层的 smoltcp socket <sup>[2](#mod-net-ref-2)</sup>。

TcpSocket 设计为可共享（`unsafe impl Sync`），其内部状态通过 `AtomicU8` 和 `UnsafeCell` 管理，以避免锁的开销 <sup>[3](#mod-net-ref-3)</sup>。状态机定义了 CLOSED、BUSY、CONNECTING、CONNECTED、LISTENING 五种状态，通过 `compare_exchange` 实现原子转换 <sup>[4](#mod-net-ref-4)</sup>。UdpSocket 则使用 `RwLock` 保护本地地址和远端地址，handle 固定不变 <sup>[5](#mod-net-ref-5)</sup>。

网络栈只初始化了 loopback 接口（127.0.0.1/8）<sup>[6](#mod-net-ref-6)</sup>，没有真实网卡驱动，因此通信仅限于本地进程。这可能是早期开发的简化选择，但也意味着外部网络连接完全不可用。

### 2. 关键设计取舍

**状态机 vs 锁**：TcpSocket 选择原子状态机而非简单 Mutex，是为了在多个线程共享同一个 socket 时（如被 Arc 包裹）避免每次操作都加锁，提高并发性能。代价是引入了 `unsafe` 代码和复杂的状态一致性保证。例如，`update_state` 方法通过 CAS 将状态置为 BUSY，执行操作后要么进入新状态要么回退，这要求调用者确保在 BUSY 期间没有其他线程访问 `UnsafeCell` 中的字段 <sup>[4](#mod-net-ref-4)</sup>。这种“用小心换性能”的做法在操作系统中常见，但需要严格的注释和审查。

**监听队列的预创建**：`snoop_tcp_packet` 函数在接收 TCP SYN 时会自动创建一个 smoltcp socket 并放入 `LISTEN_TABLE` <sup>[7](#mod-net-ref-7)</sup>。这样当用户调用 `accept` 时，连接可能已经建立好，直接返回即可。这减少了 accept 的延迟，但要求 `LISTEN_TABLE` 内部维护一个队列，且队列大小硬编码为 512 <sup>[8](#mod-net-ref-8)</sup>。如果瞬间涌入大量连接，可能导致队列溢出丢包。

**阻塞与非阻塞的统一**：TcpSocket 和 UdpSocket 的 `block_on` 方法实现了协作式阻塞：如果非阻塞模式则立即返回 `WouldBlock`；阻塞模式则循环调用 `poll_interfaces` 并 `suspend_current_and_run_next` 让出 CPU <sup>[9](#mod-net-ref-9)</sup>。这种设计依赖上层任务调度器（很可能是单线程协作式），与 Linux 内核的进程阻塞不同，但适合 unikernel 环境。

### 3. 跨模块协同

网络模块通过 `socket_impl.rs` 中的 `Socket` 枚举和 `Sock` trait 向上层系统调用层提供服务 <sup>[10](#mod-net-ref-10)</sup>。系统调用（如 sendto, recvfrom）最终调用 `Sock` 的方法，这些方法再分派到 `TcpSocket` 或 `UdpSocket`。注意这里使用了双重锁：`Socket::Tcp(Mutex<TcpSocket>)` 外层 Mutex 保护整个 socket 对象，内层 TcpSocket 又使用原子操作。这种冗余可能降低性能，但简化了代码结构。

与任务调度器的协作通过 `suspend_current_and_run_next` 实现，该函数将当前任务挂起并切换到下一个任务 <sup>[9](#mod-net-ref-9)</sup>。这是典型的协程切换模式，需要调度器支持 `yield`。此外，超时功能依赖 `get_time` 函数获取系统时钟 <sup>[11](#mod-net-ref-11)</sup>。

错误处理方面，`AxError` 枚举定义了丰富的错误类型，并通过 `From<SysErrNo>` 实现自动转换 <sup>[12](#mod-net-ref-12)</sup>。这保证了系统调用能返回正确的 POSIX errno。

### 4. 边角细节与不足

**<sup>[13](#mod-net-ref-13)</sup> TcpSocket 缺少 Drop 实现**：当 TcpSocket 被丢弃时，其 `SocketHandle` 不会从 `SOCKET_SET` 中移除，导致资源泄漏。而 UdpSocket 正确实现了 Drop。这可能是开发中的疏忽。

**<sup>[14](#mod-net-ref-14)</sup> UdpSocket 阻塞可能永久挂起**：UdpSocket 的 `block_on` 在数据未到达时会无限循环，没有超时机制。虽然提供了 `recv_from_timeout`，但 `recv_from` 本身可能被用于 DoS 攻击。

**<sup>[8](#mod-net-ref-8)</sup> 硬编码常量**：`LISTEN_QUEUE_SIZE` (512)、`TCP_RX_BUF_LEN` (64KB) 等硬编码，不能根据应用需求调整，可能在某些场景下成为瓶颈。

**<sup>[15](#mod-net-ref-15)</sup> UdpSocket bind_check 不检查 TCP**：`bind_check` 仅遍历 UDP socket 检查地址冲突，忽略了 TCP socket 的可能占用，可能导致 UDP 和 TCP 端口冲突。

**<sup>[16](#mod-net-ref-16)</sup> snoop_tcp_packet 错误被忽略**：`preprocess` 调用 `snoop_tcp_packet` 的结果通过 `ok()` 忽略，格式错误的 TCP 包不会通知上层。

**<sup>[17](#mod-net-ref-17)</sup> unsafe 块缺乏安全注释**：TcpSocket 的多个 `unsafe` 块缺少 `// SAFETY:` 说明，增加了代码审查难度。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架围绕 `BaseDriver` 和 `BlockDriver` trait 构建，以 `VirtIoHalImpl` 适配 virtio-drivers crate 的 Hal 接口，通过条件编译支持 riscv64 (MMIO) 和 loongarch64 (PCI)。与 ArceOS 基线相比，简化了驱动 trait，直接耦合 virtio 实现，保留了 PCI 枚举和线性映射偏移的硬编码假设。

### 1. 核心抽象与外部依赖
驱动框架以 `BaseDriver` trait 和 `BlockDriver` trait 为抽象核心，配合 `DeviceType` 枚举对设备分类。`BaseDriver` 仅提供设备名称和类型标识，`BlockDriver` 扩展了块设备操作（读、写、刷新）。这种精简的设计直接服务于系统所需的基本块/网络设备，省略了中断处理、电源管理等高级抽象，体现了“够用就好”的取舍。

外部依赖集中于 `virtio-drivers` crate，它封装了 VirtIO 协议的设备端实现。框架通过实现 `Hal` trait 完成平台适配：`VirtIoHalImpl` 负责 DMA 内存分配/释放、MMIO 物理地址到虚拟地址的转换，以及物理地址共享。特别是 `share` 函数，在 riscv64 上通过内核页表翻译获取设备 DMA 所需的物理地址，而在 loongarch64 上则直接返回虚拟地址（因为线性映射段物理地址等于虚拟地址减去偏移）<sup>[4](#mod-drivers-ref-4)</sup>。这种差异源于两种架构不同的内存映射模型，框架通过条件编译在函数内部处理，避免了 trait 接口的泛化开销。

### 2. 关键设计取舍
**传输层选择策略**：`mod.rs` 使用 `#[cfg(target_arch = ...)]` 静态决定采用 `MmioTransport` 还是 `PciTransport`，并固定设备基地址（riscv64 的 0x10001000，loongarch64 的 0x20000000）。这消除了运行时 PCI 枚举的开销，但将平台假设硬编码：若未来 QEMU 版本改变 virtio MMIO 地址，或需要支持非标准拓扑，必须修改源码。对比 ArceOS 原版中更灵活的 PCI 扫描（如通过 ACPI 表定位），此实现更简略但更适合教学场景。

**PCI 枚举代码重复**：`enumerate_pci` 函数和 `PciRangeAllocator` 在 `blk.rs` 和 `net.rs` 中完全重复出现 <sup>[1](#mod-drivers-ref-1)</sup>。两者逻辑一致（分配 BAR 寄存器、使能命令位），仅过滤的设备类型不同（Block vs Network）。这种重复源于两个驱动独立开发，未提取公共库。合并后可减少约 80 行重复代码，并防止一处修改另一处遗漏。

**错误处理妥协**：`VirtIoBlkDev::read_block` 采用重试机制：若第一次调用返回 `NotReady`，则直接填零返回；否则 ack 中断并重试一次，再失败则 panic <sup>[2](#mod-drivers-ref-2)</sup>。这种设计假设 `NotReady` 是临时状态且后续重试不可靠，但返回全零数据可能误导上层（如文件系统认为读到空块）。更稳健的做法是循环等待或返回错误码。

### 3. 跨模块协同
类型别名系统是模块间协同的纽带。`mod.rs` 定义 `BlockDeviceImpl` 和 `NetDeviceImpl`，它们分别实例化为带具体泛型参数的 `VirtIoBlkDev` 或 `VirtIoNetDev`。`new_device()` 函数封装了构造逻辑，调用者（如 `Disk`）只需依赖 `BlockDeviceImpl` 而无需关心底层传输层。

`VirtIoHalImpl` 的实现依赖于内核内存管理模块：它通过 `frame_alloc` 和 `frame_dealloc` 分配 DMA 页面，并将分配的 `FrameTracker` 存入全局 `QUEUE_FRAMES` 以阻止其被提早回收 <sup>[3](#mod-drivers-ref-3)</sup>。`share` 函数则调用 `KERNEL_SPACE.exclusive_access().inner.exclusive_access().page_table.translate_va` 将虚拟地址转为物理地址，这依赖两级锁（全局空间锁+页表锁），可能引入竞争风险，但当前单核环境下无问题。

网络驱动进一步封装为 `AxNetDevice`，提供 `receive`/`transmit`/`alloc_tx_buffer` 等高级方法，内部持有 `Mutex<VirtIONet>` 确保线程安全。`Disk` 结构体基于 `BlockDriver` trait 实现带游标的块访问，处理跨块边界的部分读写。

### 4. 边角细节与不足
- **<sup>[6](#mod-drivers-ref-6)</sup> 条件编译覆盖不足**：`mod.rs` 中仅处理 `riscv64` 和 `loongarch64`。若在 x86_64 或 aarch64 上编译，`BlockDeviceImpl` 等类型别名未定义，导致晦涩的编译错误。应增加 `compile_error!` 给出明确提示。
- **`unwrap` 恐慌风险**：PCI 枚举中多处使用 `.unwrap()`（如 `pci_root.bar_info().unwrap()`），若虚拟设备提供恶意 BAR 配置可触发 panic <sup>[5](#mod-drivers-ref-5)</sup>。
- **`unsafe` 块缺少安全注释**：`VirtIoHalImpl::mmio_phys_to_virt` 硬编码地址偏移 `0x80200000`，未说明此值仅当内核页表建立线性映射时才合法 <sup>[4](#mod-drivers-ref-4)</sup>。
- **`Mutex` 锁粒度**：`AxNetDevice` 的所有方法均获取 `Mutex` 锁，虽保证原子性，但在中断上下文（若中断处理程序调用这些方法）会死锁。当前中断处理程序不直接调用，但设计上未防范。
- **硬编码常量散落**：`VIRT_PCI_BASE`、`VIRT_PCI_SIZE`、`BLOCK_SIZE=512` 等常量在多个文件中重复定义 [ref:1, ref:7]，修改时需同步多处。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层位于 os/src/syscall/，按功能域拆分为 fs.rs（文件 I/O）、net.rs（网络）和 process.rs（进程管理）三个编译单元。核心抽象是 "每个系统调用一个独立 pub fn，返回 isize，负数表示负错误码"，统一通过 current_process().inner_exclusive_access() 获取进程控制块，再通过 fd_table 间接操作文件/socket。与同家族相比，本仓实现大量 Linux 特有 syscall（futex、pselect6、sendfile、renameat2 等），但多处存在直接返回 -1 替代精确错误码、unsafe 块缺少 SAFETY 注释、以及 stub 实现等问题。

### 1. 核心抽象与外部依赖

系统调用层的核心抽象极其直白：每个系统调用是一个独立的 `pub fn`，签名统一为 `fn(…args…) -> isize`，返回值中负数表示错误（负的错误码），0 或正数表示成功。没有用 trait 或虚表来抽象，没有统一的 dispatch 框架——这是典型的“手写 syscall table”风格，每个函数由 trap handler 直接调用。

三个文件按功能域切分：`fs.rs` 承担文件描述符操作（open/read/write/close/pipe/dup/seek/stat 等），`net.rs` 承担 socket 操作（socket/bind/connect/listen/accept/sendto/recvfrom/setsockopt 等），`process.rs` 承担进程生命周期管理（fork/exec/exit/waitpid 以及 futex、timer、resource limit 等）。这种切分与 Linux 的 syscall 分类一致，但与 rCore-Tutorial 的将所有 syscall 放在一个 `syscall.rs` 不同，这里更模块化。<sup>[1](#mod-syscall-ref-1)</sup><sup>[2](#mod-syscall-ref-2)</sup><sup>[3](#mod-syscall-ref-3)</sup>

每个 syscall 函数内部几乎都遵循同一模式：调用 `current_process()` 获取进程控制块 Arc，调用 `inner_exclusive_access()` 获得可变锁（`ProcessInner`）；通过 `inner.get_user_token()` 获取用户态页表 token；利用 `translated_ref`/`translated_refmut`/`translated_str`/`translated_byte_buffer` 族函数将用户提供的指针安全地翻译为内核可访问的地址；在需要长时间阻塞前手动 `drop(inner); drop(process);` 释放锁；最后返回 `isize`。

与同家族基线（rCore-Tutorial）相比，本仓的 syscall 层显著扩展了覆盖范围：实现了 `futex`（含 FUTEX_WAKE_OP）、`pselect6`（完整轮询+信号掩码）、`sendfile`（零拷贝语义）、`renameat2`、`sched_getaffinity`、`get_robust_list` 等 Linux 特定调用，体现了被用于运行 musl/glibc 程序的目标。<sup>[4](#mod-syscall-ref-4)</sup><sup>[5](#mod-syscall-ref-5)</sup>

### 2. 关键设计取舍

**取舍一：错误码路径的不一致。** 代码中使用 `SysErrNo` 枚举（如 `SysErrNo::EINVAL`）来返回精确错误码，但大量分支直接返回 `-1`。例如 `sys_read` 中所有错误路径都返回 `-1`（<sup>[6](#mod-syscall-ref-6)</sup>），`sys_write` 同样<sup>[7](#mod-syscall-ref-7)</sup>。这意味着应用层无法区分 EFAULT、EBADF、EISDIR——它们都被映射为 -1。这在调试阶段可能可接受，但对于生产级 POSIX 兼容性是个明显的 trade-off。替代方案是在 `SysErrNo` 上实现 `From<()>` 或统一错误映射，但当前代码选择了“先跑通”的策略。

**取舍二：统一文件描述符与 socket 的 FileClass 抽象。** 设计上，`fd_table` 中存储的是 `FileDescriptor`，而 `FileDescriptor` 内部包含 `FileClass` 枚举，其变体包括 `Abs(Arc<dyn File>)`（普通文件/pipe）和 `Sock(Arc<Socket>)`（网络 socket）。这允许 fs.rs 和 net.rs 共享同一套 fd 分配/释放逻辑，但代价是每次操作都需要在 `file()` 和 `sock()` 之间选择路径——例如 `sys_read` 和 `sys_sendto` 分别访问 fd_table 后走不同分支。<sup>[8](#mod-syscall-ref-8)</sup><sup>[9](#mod-syscall-ref-9)</sup> 这种设计的替代方案是让 Socket 也实现 `File` trait（像一些 BSD 内核那样），但当前设计保持了二者分离，可能是因为网络 socket 的异步语义（PollState、非阻塞）与文件差异较大。

**取舍三：手动锁释放模式。** 代码中常见的模式是 `drop(inner); drop(process);` 出现在 `for` 循环之前或阻塞调用之前。这是因为 `ProcessInner` 的锁（spin::Mutex）是进程粒度的，如果持锁调用 `suspend_current_and_run_next()` 会导致死锁——调度器可能选择同一进程的其他线程，而它们也在等待同一把锁。手动 drop 是显式逃逸分析，替代方案是引入作用域守卫（如 `{ let inner = ...; ... }`），但当前代码风格更偏好函数内统一释放。<sup>[10](#mod-syscall-ref-10)</sup> 这种模式有个隐患：如果提前 `drop` 后中间代码 panic，会导致 process Arc 引用计数泄漏或内存不安全，但当前代码中没有 panic 防护。

**取舍四：sys_pipe 中的直接物理地址写。** `sys_pipe` 在 `fs.rs` 中实现时没有使用 `translated_refmut` 写回用户空间的 fd 数组，而是手动查找页表、获取物理页号、加上硬编码的内核线性映射偏移（riscv64: `0x8020_0000`，loongarch64: `0x9000_0000_0000_0000`），然后 `unsafe` 写物理地址。这是一个极具架构依赖性的实现，且没有 `// SAFETY:` 注释。<sup>[11](#mod-syscall-ref-11)</sup><sup>[12](#mod-syscall-ref-12)</sup> 替代方案是使用 `UserBuffer::new(safe_translated_byte_buffer(...))` + 写操作，但当前选择直接操作物理映射，可能是因为想在裸机启动早期避免复杂的页表 walk（尽管同一函数已经调用了 `PageTable::from_token`）。这带来的风险是：如果内核线性映射基地址变更（例如为 KASLR 调整），这段代码会静默写错地址。

### 3. 跨模块协同

与 **task 模块**的协同是 syscall 层最密集的交互。每个 syscall 函数几乎都调用 `current_process()` / `current_task()` 获取执行上下文。更具体的协同场景有：

- **futex 机制**：`process.rs` 中的 `sys_futex` 解析 futex 操作码后，调用 `task` 模块的 `futex_wait` / `futex_wake_up` / `futex_requeue`。FutexKey 的构造需要物理地址（由 `memory_set.translate_va` 得到）和进程 pid（用于私有 futex），这要求 syscall 层同时持有进程锁和页表信息。<sup>[4](#mod-syscall-ref-4)</sup><sup>[13](#mod-syscall-ref-13)</sup>

- **信号与阻塞**：`sys_pselect6` 中，在轮询循环前通过 `task.get_sig_mask()` 保存旧信号掩码，循环中间如果信号到达（由 `check_if_any_sig_for_current_task` 检测），会设置新的信号掩码并返回。这个流程涉及 syscall 层、signal 模块、task 模块的三方协作。<sup>[14](#mod-syscall-ref-14)</sup>

- **进程生命周期**：`sys_fork` / `sys_exec` / `sys_exit` / `sys_waitpid` 之间通过 `ProcessInner` 的 `children` 列表和 `is_zombie` 标志协同。`sys_waitpid` 在发现僵尸子进程后，调用 `remove_all_from_thread_group` 和 `remove_from_pid2process` 清理全局数据结构，同时 `assert_eq!(Arc::strong_count(&child), 1)` 确保子进程被唯一持有——这是一个隐晦的内存量安全检查。<sup>[15](#mod-syscall-ref-15)</sup>

与 **mm 模块**的协同主要体现在用户内存访问封装。`translated_str` 从用户空间拷贝以 null 结尾的字符串（用于路径名），`translated_byte_buffer` 返回分片的 `Vec<&[u8]>` 用于批量数据读写，`translated_refmut` 在单次原子操作中读写用户空间的小对象（如 FdSet、TimeSpec）。这些封装隐藏了页表翻译的复杂性，但调用者需要自己保证跨 `drop(inner)` 边界后 token 仍然有效。<sup>[16](#mod-syscall-ref-16)</sup><sup>[17](#mod-syscall-ref-17)</sup>

与 **net 模块**的协同：socket 相关的 syscall 通过 `FileClass::Sock(Arc<Socket>)` 间接操作 `Socket` trait，实际的 TCP/UDP 状态机在 `net` 目录。例如 `sys_connect` 在解析完用户提供的 sockaddr_in 后调用 `socket_arc.connect(dest_addr)`，返回 `AxError` 并映射为 POSIX 错误码。<sup>[18](#mod-syscall-ref-18)</sup>

### 4. 边角细节与不足

**<sup>[19](#mod-syscall-ref-19)</sup> 不足：大量 unimplemented! 和 stub。** `sys_getpeername`、`sys_accept4`、`sys_sendmsg`、`sys_socketpair`、`sys_linkat` 等直接 panic，`sys_spawn` 被注释掉并返回 -1，`sys_get_robust_list` 返回 ENOSYS。这些 stub 表明该 OS 尚未达到完整 glibc 兼容性。

**<sup>[20](#mod-syscall-ref-20)</sup> 不足：unsafe 块缺少 SAFETY 注释。** 全文检索不到 `// SAFETY:`。`sys_pipe` 中对用户物理地址的直接写、`sys_sendto` 中的 `*(sockaddr_bytes.as_ptr() as *const SockaddrIn)` 等操作均无安全契约说明。这是明显的技术债，评审者应要求补全。

**<sup>[21](#mod-syscall-ref-21)</sup> 不足：整型符号错误与溢出风险。** `sys_pselect6` 中将 `fd as isize` 与 0 比较来判断负数（如 `if (readfds as isize) < 0`），这仅对 64 位架构有效（指针高位置位会被解读为负数）<sup>[22](#mod-syscall-ref-22)</sup>。在 32 位架构上，用户空间地址 0x80000000 会被误判为负从而拒绝合法请求。

**<sup>[23](#mod-syscall-ref-23)</sup> 潜在死锁路径。** `sys_waitpid` 中在持有当前进程锁时调用了 `block_current_and_run_next()`（在 else 分支中 `drop(inner)` 之后才调，但前面的循环内 `drop(inner)` 在 continue 前已执行，所以当前无问题）。但 `sys_futex` FUTEX_WAIT 路径中，`drop(inner); drop(process); futex_wait(key)` 的顺序是正确的（先释放再阻塞），但 `futex_wait` 内部可能尝试重入获取同一进程锁——验证需要查看 `futex_wait` 的实现。

**<sup>[24](#mod-syscall-ref-24)</sup> sys_exec 中硬编码的 musl/glibc 路径。** `sys_exec` 中检查路径是否包含 `/musl` 或 `/glibc` 来设置 cwd，并针对 .sh 文件特殊处理（强制使用 busybox）。这是典型的“先跑通后重构”的痕迹，直接将 musl/glibc 的路径策略硬编码到内核 syscall 中，破坏了内核与文件系统布局的解耦。

**<sup>[25](#mod-syscall-ref-25)</sup> 缺少 .gitignore 与 env 污染。** `sys_exec` 中向 argv/env 中硬插入了 `PATH=/:/bin:`、`LD_LIBRARY_PATH=...`、`ENOUGH=100000` 等环境变量，这些应该由用户态 loader 或 init 进程设置，而非内核 syscall 层注入。
