## 📖 一、总览

该仓库属于 ArceOS 家族中旨在提供 Linux 兼容性的实验型内核。它基于 axhal、axtask、axfs 等模块化 crate 构建，通过扩展这些组件增加了 Linux 进程模型、系统调用接口和驱动框架。与追求完整兼容的成熟项目不同，此仓更侧重快速原型验证：核心路径（fork、mmap、信号）已有实质实现，但大量系统调用仅以 stub 形式存在，且内部隐藏着简化假设和未完成的细节。整体定位介于教学演示与半成品研究之间，适合分析 ArceOS 的可扩展边界而非直接用于生产。

最值得注意的两个特点：1) 通过 `TaskExt` trait 将 Linux 进程/线程模型无缝注入 axtask 调度器[task.ref:1]，无需修改调度核心即实现进程间资源域隔离，这是 ArceOS 扩展点设计的典型范例。2) 内存管理的 `Backend` 枚举[mm.ref:2]将物理页面分配策略与地址空间操作解耦，实现了写时复制（COW）与惰性分配，其 `CowBackend` 配合全局帧表管理引用计数，提供了近乎 Unix 的 fork 语义。3) 系统调用层以近 300 分支的大 match 覆盖 265 个系统调用[sc.ref:1]，但约半数仅返回 0 或空操作，反映出广度优先、深度不足的取舍。

## 🎯 二、综合评价

### 整体定位

此仓是 ArceOS 生态中第一个系统尝试 Linux ABI 兼容的工程，但远未达到可用程度。它选择了一条“外围完整、核心堪用”的路径：系统调用枚举覆盖了 Linux 大多数接口（如 time 命名空间、POSIX 定时器），但实际执行时许多调用仅返回固定值或空操作（如 `sys_seccomp` 直返 0[sc.ref:5]），导致运行时兼容性极差。核心的进程创建、内存管理、文件操作虽有代码，却布满简化假设——例如 `sys_setitimer` 使用 `yield_now` 忙等待[sc.ref:?]，阻塞 IPC 直接返回 EAGAIN[ipc.ref:6]——这些都表明项目仍处于早期原型阶段。因此定位为“实验型+教学型”，适合学习 ArceOS 扩展机制，但绝不适合作为真实负载的内核。

### 真正的创新点

尽管完成度有限，该仓库在设计层面仍有若干值得关注的创新。

- **进程模型的非侵入注入**：通过 `TaskExt` trait 将 `Thread` 实例绑定到 `axtask::TaskInner`[task.ref:1]，使得 axtask 无需感知 Linux 语义即可调度任意扩展任务。同时利用 `on_enter`/`on_leave` 钩子[task.ref:?]触发的 `Scope` 资源域切换，实现了进程间 FD 表、FS 上下文、信号掩码的隔离。这一设计完全绕过了修改调度器的需要，是抽象边界清晰的典范。
- **Backend 枚举与惰性分配**：内存管理中的 `Backend` 枚举[mm.ref:2]将物理页分配策略（线性映射、匿名、写时复制）参数化到 `MemoryArea` 中，使得缺页处理、fork 复制等行为可通过枚举分派。`CowBackend::populate`[mm.ref:?] 在首次访问时才分配物理页，且通过全局帧表维护写时复制引用计数[mm.ref:2]，既实现了惰性分配，又避免了 page cache 的复杂性。
- **条件编译的 INIT_SCRIPT 机制**：启动流程中通过 feature 门控选择不同 init shell 脚本[boot.ref:4]，并内联构建 LTP 环境变量[boot.ref:5]，使得测试用例的自动化集成成为可能。该设计尽管粗糙，但展示了在嵌入式环境下灵活选择初始化策略的思路。

### 取舍判断

该仓库的精力主要投放在三个方面：**系统调用覆盖广度**、**ArceOS 扩展示例**、**POSIX 语义的部分模拟**。而以下方面被刻意省略或简化：

- **性能与可扩展性**：全局帧表（`FRAME_TABLE`[mm.ref:2]）使用 `SpinNoIrq` 锁，多核竞争严重；`MMAP_RECORDS`[mm.ref:4] 采用线性搜索 O(n)；共享 futex 表每 100 次操作才清理弱引用[ipc.ref:?]。这些设计直接导致多线程性能低下，但简化了实现。
- **完成度与严格性**：许多系统调用（如 `sys_keyctl`、`sys_seccomp`）直接返回 0[sc.ref:5]，阻塞路径（如消息队列、FIFO 写端关闭）未按 POSIX 要求挂起而是返回 EAGAIN，部分结构体字段（`/proc/stat`）返回硬编码或错误数据。项目显然优先让编译通过、简单测试通过，而非达到真正的 Linux 兼容。
- **安全性**：`unsafe` 块大量缺乏 `// SAFETY:` 注释[mm.ref:11][task.ref:1]，用户态指针通过 `VmPtr` 包装后仍存在未验证来源的 unsafe 操作[net.ref:?]。对信号、权限的简化处理（如 signalfd 丢失发送者身份[signal.ref:3]）可能被恶意程序利用。

### 完成度与不足

从模块级看，内存管理和任务调度具有相对最高的完成度：`fork`/`execve`/`mmap` 的代码路径完整，写时复制和地址空间切换可以工作。文件系统和网络模块实现了常用系统调用，但缺乏真实 I/O 结合的测试（如 poll 与信号中断的协同[fs.ref:5]）。信号、IPC、设备驱动则明显处于早期状态：信号集大小固定[signal.ref:1]、fanotify 队列硬编码[fs.ref:3]、sys_shmat 的 SHM_RND 标记预留未实现[ipc.ref:7]。启动流程也暴露了风险：`main` 返回将触发未定义行为[boot.ref:8]，平台 extern crate 仅覆盖 RISC-V VisionFive2[boot.ref:3]。整体估计，约 40% 的系统调用有实质实现，另 60% 是 stub 或极简实现，且已实现的部分也充满待办事项和简化假设。因此完成度属于“半成品”级别，创新性体现为架构层面的设计思考，而代码质量因大量 unsafe 无注释、硬编码、遗留 TODO 而明显不足。

### 1. 核心抽象与外部依赖

启动流程的核心抽象是 **Rust 入口点约定**：通过 `#[no_std]` 和 `#[no_main]` 避免标准运行时，并利用 `#[unsafe(no_mangle)]` 导出一个名为 `main` 的函数作为二进制入口<sup>[1](#mod-boot-ref-1)</sup>。这与典型 ArceOS 应用一致，但该仓库将实际内核初始化推给外部 crate `starry_kernel::entry::init`<sup>[2](#mod-boot-ref-2)</sup>。这种设计使启动骨架保持稳定，同时允许内核初始化逻辑独立演化。

硬件初始化依赖 axhal 系列 crate。代码中通过 `extern crate axplat_riscv64_visionfive2`（条件编译于 `#[cfg(feature = "vf2")]`）显式链接平台支持 crate<sup>[3](#mod-boot-ref-3)</sup>，该 crate 负责设置页表、陷阱向量、定时器等。启动流程本身不包含任何汇编代码（facts 中 `assembly_files` 为空），表明所有底层 CPU 初始化被封装在 axhal 内部。

### 2. 关键设计取舍

- **Why no_main?** 放弃标准库 main 的运行时初始化，换取完全控制权。代价是必须在启动早期手工设置堆（通过 `extern crate alloc`）和 panic 处理。

- **Why delegate to starry_kernel?** 将控制权转交而非直接内联初始化，使启动模块能够独立于内核实现变化。但这也增加了模块间耦合 —— 调用者必须信任 `starry_kernel::entry::init` 的安全合约。

- **Conditional init scripts**<sup>[4](#mod-boot-ref-4)</sup>: 通过三个 feature 镜像 (`timerfd-debug-init`, `shell-init`, 未设置时默认) 选择不同的 INIT_SCRIPT。这是一种编译期多态，避免了运行期判断，但 feature 冲突时最后匹配的 cfg 胜出，且没有编译告警。

- **LTP env 内联构建**<sup>[5](#mod-boot-ref-5)</sup>: 将 `ltp_testcases.sh` 解析为环境变量 `LTP_CASES`，绕过文件系统依赖。但这会产生巨大的环境字符串，可能触碰内核环境大小上限，也未做截断处理。

- **Hardcoded busybox path**<sup>[6](#mod-boot-ref-6)</sup>: `CMDLINE` 固定为 `/musl/busybox sh -c <init_script>`，假设根文件系统布局。这在跨平台测试中可能不通用。

### 3. 跨模块协同

启动流程与以下模块紧密配合：

- **starry_kernel::entry::init**：接收 `args` 和 `envs` 后，预计会完成剩余初始化（如挂载根文件系统、启动 init 进程）。facts 显示后续调用链包含 `handle_page_fault` 和 `irq_handler`，说明该函数会注册异常处理例程。

- **axhal / axplat**：`extern crate axplat_riscv64_visionfive2` 确保平台特定符号（如中断向量表、物理内存映射）被链接。axhal 的 `irq_handler`<sup>[7](#mod-boot-ref-7)</sup> 被注册后服务于整个运行周期。

- **测试框架**：`build_ltp_cases_env` 生成的 `LTP_CASES` 环境变量由后续 init 脚本消费，实现测试用例注入。这是 ArceOS 基线中没有的独特扩展，专用于 LTP 兼容性验证。

### 4. 边角细节与不足

**<sup>[1](#mod-boot-ref-1)</sup> 缺乏 unsafe 合约文档**：`main` 函数标记为 `#[unsafe(no_mangle)]` 但内部无 `unsafe` 块。对 `starry_kernel::entry::init` 的调用未显式约定了 safety 条件。（触发取舍清单第 3 条）

**<sup>[5](#mod-boot-ref-5)</sup> LTP 环境变量可能超限**：`build_ltp_cases_env` 使用 `String::push` 无限追加，未检测 `LTP_CASES` 长度。操作系统环境变量通常有上限（如 4KB 或 32KB），超过则悄无声息地截断或失败，但此处无任何检查。（触发取舍清单第 6 条硬编码常量 / magic number）

**<sup>[3](#mod-boot-ref-3)</sup> 平台 extern crate 不全面**：仅 `feature = "vf2"` 时链接 `axplat_riscv64_visionfive2`，其他平台（如 aarch64、x86_64）没有对应行。若在这些平台上编译，可能因找不到必要符号而链接失败。（触发取舍清单第 1 条 cfg 条件编译）

**<sup>[8](#mod-boot-ref-8)</sup> main 函数返回值被忽略**：`main` 返回 `()`，但 `starry_kernel::entry::init` 很可能不返回。若其意外返回，程序会执行完 main 进入未定义行为（usertrap 示例中常见 UB）。（触发取舍清单第 7 条错误路径）

**<sup>[2](#mod-boot-ref-2)</sup> 隐藏的内核初始化复杂度**：`starry_kernel::entry::init` 内部可能 panic 或死循环，但调用者无法感知错误。这属于半成品式委托，早期实现信号。（触发取舍清单第 8 条 TODO / 半成品）

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块的核心抽象是 `AddrSpace` 与 `Backend` 枚举，前者封装页表和映射区域集合，后者将物理页面管理策略（线性、匿名分配、写时复制）与地址空间操作解耦。与 ArceOS 基线相比，arceos-starry 引入了完整的用户态 sys_mmap/sys_mprotect 实现、基于全局映射记录的 /proc/pid/maps 支持、以及通过 `CowBackend` 实现的 fork 时写时复制机制。模块还通过 `access_user_memory` + 注册的缺页处理程序实现了内核态安全地访问用户内存。

### 1. 核心抽象：`AddrSpace` 与 `Backend` 的层次化设计

地址空间的主体是 `AddrSpace` 结构体，它包含一个 `VirtAddrRange` 描述有效范围、一个 `MemorySet<Backend>` 管理映射区域、以及一个 `PageTable` 硬件页表 <sup>[1](#mod-mm-ref-1)</sup>。`MemorySet` 来自 `memory_set` crate，而 `Backend` 是本模块新增的枚举，每个 `MemoryArea` 关联一个 `Backend` 实例。`Backend` 通过 `BackendOps` trait 定义 `map`、`unmap`、`populate`、`clone_map` 等操作，使得地址空间的增长、缺页处理、fork 复制等行为可以按后端类型分派。`Backend` 目前包含三种变体：`Linear`（线性映射，用于内核跳板和信号跳板）、`Cow`（写时复制，用于匿名映射、文件私有映射和堆/栈），其中 `Cow` 又细分为是否关联文件后端 <sup>[2](#mod-mm-ref-2)</sup>。这种设计将物理页面生命周期与虚拟区域分离——例如 `CowBackend` 的 `map` 方法仅在元数据层面记录区域，实际物理页面在 `populate`（缺页时）才分配，实现了真正的惰性分配。

### 2. 关键设计取舍

**COW 实现与全局帧表**：`CowBackend` 通过全局静态 `FRAME_TABLE`（`BTreeMap<PhysAddr, Arc<SpinNoIrq<FrameRefCnt>>>`）跟踪每个物理页面的引用计数 <sup>[3](#mod-mm-ref-3)</sup>。fork 时 `clone_map` 移除所有写权限并将同一物理帧映射到子进程，写入时触发 `handle_cow_fault`：若引用计数为 1 则直接升级权限，否则复制页面并递减原帧计数。这一设计避免了为每页分配独立的 `Arc`，但将帧生命周期管理集中到一张全局表中，`FRAME_TABLE` 的 `SpinNoIrq` 锁在高并发缺页时可能成为伸缩性瓶颈。同时，`FRAME_TABLE.init_frame` 中的 `assert!` 若因竞态条件触发（理论上不会，但无形式化保证）将直接 panic <sup>[3](#mod-mm-ref-3)</sup>。

**映射记录的双重维护**：`sys_mmap` 在修改 `AddrSpace` 之外，还将每次映射的起始、长度、权限、共享属性等记入全局 `MMAP_RECORDS` <sup>[4](#mod-mm-ref-4)</sup>。这一冗余用于生成 `/proc/pid/maps` 和 `mprotect` 权限更新，但 `MMAP_RECORDS` 是 `Vec<MapRecord>`，查找时线性扫描全部记录（`forget_mmap`、`recorded_prot` 等），随进程映射数增长性能劣化。此外，`addr_space.areas()` 已暴露所有区域，理论上可从中推导出大部分信息，但记录保留了用户态可见的原始 `prot` 标志（如 `GROWSUP`）和共享属性，这些无法从硬件页表标志完全恢复。

**GROWSDOWN 的全局列表**：可增长栈通过 `GROW_DOWN_AREAS` 全局列表标识，缺页时 `try_growdown_fault` 在列表中查找并调用 `aspace.map` 扩展 <sup>[5](#mod-mm-ref-5)</sup>。与标准 Linux 将 `VM_GROWSDOWN` 标志存储在 VMA 中不同，arceos-starry 将增长区域与地址空间解耦，这意味着进程退出时必须显式调用 `release_process_mmap_resources` 清理列表，否则 `pid` 可能被复用导致错误扩展。当前实现没有限制最大增长页面数，恶意程序可通过反复压栈触发无限内核内存分配。

**用户内存访问的缺页协同**：`access_user_memory` 设置线程标志后执行 `user_copy`，同时系统注册了 `PAGE_FAULT` 的 trap handler <sup>[6](#mod-mm-ref-6)</sup>。handler 检查 `is_accessing_user_memory` 标志，若为真则委托 `AddrSpace::handle_page_fault` 处理惰性分配。这一机制使得内核在 copy_from_user 时无需预分配所有页面，但 `check_region` 中提前调用了 `populate_area` 强制分配，实际上绕过了惰性设计——这是为了确保后续原子操作不触发缺页，但牺牲了按需分配的好处。

### 3. 跨模块协同

**与系统调用层**：`sys_mmap` 和 `sys_munmap`（`kernel/src/syscall/mm/mmap.rs`）直接操作当前进程的 `AddrSpace`，同时维护 `MMAP_RECORDS`、`GROW_DOWN_AREAS` 等全局记录。`sys_mprotect` 调用 `AddrSpace::protect` 更新页表权限，然后调用 `update_mmap_prot` 同步记录。这一双重更新路径存在时序窗口——若在 `protect` 后、记录更新前发生缺页，缺页处理可能使用旧的记录权限。

**与进程管理**：fork 通过 `AddrSpace::try_clone` 实现，内部遍历所有 `MemoryArea` 并调用 `Backend::clone_map`。`CowBackend::clone_map` 通过共享物理页并移除写权限实现写时复制，同时 `inherit_wipeonfork_ranges` 在子进程中用零填充标记为 `WIPEONFORK` 的区域 <sup>[7](#mod-mm-ref-7)</sup>。进程退出时 `release_process_mmap_resources` 清理 pid 相关的所有全局记录。

**与信号处理**：`map_trampoline` 在加载用户程序时将信号跳板映射到固定虚拟地址 `SIGNAL_TRAMPOLINE`，使用 `map_linear`（线性后端）确保跳板物理页与内核共享，避免在信号传递时复制 <sup>[8](#mod-mm-ref-8)</sup>。这一地址编译期固定，与 `axsignal` 的跳板代码布局紧密耦合。

**与文件系统**：文件映射通过 `CowBackend` 关联 `FileBackend` 实现。`alloc_new_at` 在映射缺页时从文件读取数据；`MAP_SHARED` 映射通过 `SIGBUS_AREAS` 跟踪写时 SIGBUS 语义——写入此类区域触发 SIGBUS 而非分配新页 <sup>[9](#mod-mm-ref-9)</sup>。`MAP_PRIVATE` 文件映射的写时复制与匿名 COW 共享同一套帧表，但文件后端导致 `populate` 时从指定偏移读取文件内容。

### 4. 边角细节与不足

- **TDOO 与 I-cache 刷新缺失**：`loader.rs` 中 `map_elf` 函数末尾注释 `// TDOO: flush the I-cache`，表明在执行 ELF 映射后忘记刷新指令缓存，在 icache 非一致性架构（如某些 ARM 实现）上可能导致执行旧指令 <sup>[10](#mod-mm-ref-10)</sup>。
- **unsafe 块缺少安全契约**：`cow.rs` 中多处 `core::ptr::copy_nonoverlapping` 和 `pt.map` 操作均未附 `// SAFETY:` 注释，例如 `phys_to_virt(paddr).as_mut_ptr()` 的转换假设物理页已映射到内核虚拟空间且无别名，这一假设若被破坏将导致内存安全漏洞 <sup>[11](#mod-mm-ref-11)</sup>。
- **硬编码架构假设**：`copy_from_kernel` 通过 `#[cfg(not(any(target_arch = "aarch64", target_arch = "loongarch64")))]` 条件编译，隐含其他架构（如 x86_64）需要将内核页表复制到用户页表。这一假设未在文档中说明，若移植到新架构可能被忽略 <sup>[12](#mod-mm-ref-12)</sup>。
- **全局锁竞争**：`MMAP_RECORDS`、`FRAME_TABLE`、`GROW_DOWN_AREAS` 均为全局 `Mutex` 或 `SpinNoIrq`，在多核密集 mmap/munmap 场景下锁竞争严重。`FRAME_TABLE` 使用 `SpinNoIrq` 而非 `Mutex`，在单核上不会阻塞，但多核上自旋导致 CPU 忙等。
- **错误路径兜底不足**：`mmap` 中 `PageIter4K::new(...).expect(...)` 若因无效地址范围失败将直接 panic，用户态可通过精心构造的 mmap 参数触发内核 panic <sup>[13](#mod-mm-ref-13)</sup>。类似地，`process_area_data` 中的 `expect` 同样存在此风险。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块基于 axtask 提供的任务管理体系，通过 TaskExt trait 将自定义的 Thread 和 ProcessData 注入用户任务，实现了 Linux 兼容的进程/线程模型。核心抽象包含三层：axtask::TaskInner（调度实体）、Thread（线程私有数据）、ProcessData（进程共享数据），并借助 starry_process 和 starry_signal 外部 crate 管理 PID 和信号。与 rCore-Tutorial 的独立调度器设计不同，本模块复用 axtask 的调度逻辑，通过 on_enter/on_leave 钩子与 scope_local::Scope 实现进程间资源域（FD 表、FS 上下文）隔离，无需修改调度器。

### 1. 核心抽象与外部依赖
本模块的进程/线程模型建立在 `axtask` crate 之上。`axtask` 提供最低限度的任务管理与调度（协作式），用户任务通过 `TaskExt` trait 扩展：`unsafe impl TaskExt for Box<Thread>` 将每个用户线程绑定到 `TaskInner`<sup>[1](#mod-task-ref-1)</sup>。`Thread` 结构体包含 per-thread 的进程数据引用（`Arc<ProcessData>`）、`clear_child_tid`、`robust_list_head`、线程级信号管理器等字段<sup>[2](#mod-task-ref-2)</sup>。`ProcessData` 聚合了进程地址空间（`Arc<Mutex<AddrSpace>>`）、资源域（`RwLock<Scope>`）、信号动作表、资源限制、子进程等待事件等<sup>[3](#mod-task-ref-3)</sup>。外部 crate `starry_process` 提供 `Process` 抽象，负责 PID 分配、进程组/会话管理、僵尸状态转换；`starry_signal` 提供 `ProcessSignalManager` 和 `ThreadSignalManager` 管理信号分发。这种分层使得调度核心不感知 Linux 语义，所有 POSIX 兼容逻辑集中在 `Thread`/`ProcessData` 中，与 ArceOS 原版的一体化设计形成对比。

### 2. 进程创建与资源隔离策略
进程创建入口为 `sys_clone`/`sys_fork`/`sys_clone3`，汇聚到 `CloneArgs::do_clone`<sup>[4](#mod-task-ref-4)</sup>。`CloneFlags` 完整建模了 Linux clone 标志集（包括 `CLONE_VM`、`CLONE_FILES`、`CLONE_SIGHAND` 等）。`do_clone` 根据 flags 决定共享或复制：若 `CLONE_VM` 则直接复用地址空间；否则调用 `try_clone` + `copy_from_kernel` 实现写时复制<sup>[5](#mod-task-ref-5)</sup>。信号动作在 `CLONE_SIGHAND` 时共享 `Arc<SpinNoIrq<SignalActions>>`，否则深度复制。文件描述符表和文件系统上下文的隔离通过 `scope_local::Scope` 实现：`FD_TABLE.scope_mut(&mut scope)` 和 `FS_CONTEXT.scope_mut(&mut scope)` 将当前进程的 Scope 与全局资源绑定<sup>[6](#mod-task-ref-6)</sup>。**关键创新**在于 `TaskExt::on_enter` 使用 `unsafe { ActiveScope::set(&scope) }` 在每次任务被调度时激活所属进程的 Scope，`on_leave` 恢复全局 Scope<sup>[7](#mod-task-ref-7)</sup>。代价是 `force_read_decrement` 绕过了 `RwLock` 读锁的安全递减，且两处 `unsafe` 均未提供 `// SAFETY:` 注释<sup>[8](#mod-task-ref-8)</sup>。此外，`CloneArgs::validate` 对 `CLONE_NEWPID` 等命名空间标志仅打印警告，不创建新命名空间<sup>[9](#mod-task-ref-9)</sup>；`sys_unshare` 同样只检查权限后直接返回 Ok(0)，未实际隔离任何资源<sup>[10](#mod-task-ref-10)</sup>。

### 3. 等待与僵尸进程回收
`sys_waitid`/`sys_waitpid` 实现了 Linux 的等待语义。核心状态 `ProcessWaitStatus` 枚举 `Stopped` 和 `Continued`，存储在 `ProcessData::wait_status` 中<sup>[11](#mod-task-ref-11)</sup>。`waitid_children` 使用 `block_on(interruptible(poll_fn(|cx| ...)))` 实现异步阻塞等待，通过 `child_exit_event` 注册 waker 以响应子进程状态变化<sup>[12](#mod-task-ref-12)</sup>。当子进程变为僵尸时，`reap_exited_child` 回收并累计 `child_maxrss_kb`<sup>[13](#mod-task-ref-13)</sup>。信号中断处理精细：`sys_waitpid` 根据信号是否可重启（`can_restart`）决定通过 `restart_syscall` 重试或返回 `EINTR`<sup>[14](#mod-task-ref-14)</sup>。但多线程父进程场景下 `WALL`/`WCLONE` 的支持被标记为 FIXME<sup>[15](#mod-task-ref-15)</sup>，说明进程组范围等待逻辑尚未完善。

### 4. 边角细节与不足
1. **命名空间 stub**：`CLONE_NEWNS`/`CLONE_NEWPID` 等标志在 `validate` 和 `sys_unshare` 中仅打印警告，不创建新命名空间<sup>[9](#mod-task-ref-9)</sup><sup>[10](#mod-task-ref-10)</sup>，容器化场景无法工作。
2. **`unsafe` 滥用**：`TaskExt::on_enter` 中 `unsafe { ActiveScope::set(&scope) }` 无安全注释；`on_leave` 中的 `force_read_decrement` 是不安全地绕过 Rust 借用检查的设计妥协<sup>[7](#mod-task-ref-7)</sup><sup>[8](#mod-task-ref-8)</sup>。
3. **硬编码背压阈值**：`THREAD_GROUP_BACKPRESSURE_THRESHOLD = 1024` 在嵌入式环境偏高，在服务器场景偏低，且不可配<sup>[16](#mod-task-ref-16)</sup>。
4. **proc 伪文件数据造假**：`/proc/stat` 返回全零运行统计<sup>[17](#mod-task-ref-17)</sup>；`/proc/pagemap` 对所有页面返回 present 标志<sup>[18](#mod-task-ref-18)</sup>；`/proc/meminfo` 的 available 等于 free<sup>[19](#mod-task-ref-19)</sup>，会误导依赖这些文件的工具。
5. **协作式调度局限**：模块未提供抢占式调度，`yield_now` 仅在 clone 背压时显式调用<sup>[20](#mod-task-ref-20)</sup>；`sched_policy` 字段未被底层调度器使用，`SCHED_FIFO`/`SCHED_RR` 无效。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块实现 Linux 文件系统相关的系统调用（open、mkdir、link、stat 等），核心抽象是 `FileLike` trait 和全局文件描述符表 `FD_TABLE`，依赖 `axfs` VFS 层进行路径解析和底层文件操作。它处理文件打开、创建、权限检查、目录遍历、文件监控（inotify/fanotify）等。与 ArceOS 基线相比，增加了保护硬链接/符号链接、粘滞位检查、fanotify 模拟以及更复杂的权限模型；但也存在硬编码队列大小、部分 unsafe 代码无安全注释、TODO 标记等不足。

### 1. 核心抽象与外部依赖

整个文件系统模块建立在三个核心抽象之上：`FileLike` trait、`FD_TABLE` 以及 `FsContext`。`FileLike` 定义了文件对象必须支持的操作，包括读取、写入、定位、状态查询和 I/O 控制，任何可以放入文件描述符表的实体（文件、目录、管道、套接字、fanotify 实例等）均实现该 trait。`FD_TABLE` 是一个全局的 `RwLock<BTreeMap<i32, Arc<dyn FileLike>>>`，存储所有打开的文件描述符，并提供了 `add_file_like`、`get_file_like` 等封装函数供系统调用使用。底层的文件系统操作则通过 `axfs::FsContext`（全局变量 `FS_CONTEXT`）委托给 VFS 层，包括路径解析、节点创建、元数据更新等。这种分层设计使得系统调用实现无需关心具体文件系统类型，只需调用统一的 `Location` 方法即可。例如，`with_fs` 辅助函数通过当前进程的根目录和当前工作目录构建 `FsContext`，然后执行闭包中的所有操作。

### 2. 关键设计取舍

**2.1 O_RDONLY|O_CREAT 兼容性处理**
Linux 允许使用 `O_RDONLY|O_CREAT` 打开一个只读的文件（若文件不存在则创建），而 `axfs::OpenOptions` 底层实现禁止同时设置只读和创建标志。为此，`sys_openat` 内部检测到这一组合后，会尝试先以只读方式打开；若因文件不存在失败，则以 `O_WRONLY|O_CREAT` 创建文件，再重新打开只读。这一 hack 增加了代码复杂度，但保持了与 Linux 的兼容性，避免了修改第三方 VFS crate 的代价。<sup>[1](#mod-fs-ref-1)</sup>

**2.2 保护硬链接/符号链接**
`ctl.rs` 中引入了 `PROTECTED_HARDLINKS` 和 `PROTECTED_SYMLINKS` 两个原子布尔标志，用于启用 Linux 内核的安全特性：在粘滞的世界可写目录中，非文件所有者不能删除/重命名符号链接；创建硬链接时，除非用户对目标文件有写权限，否则禁止。这通过 `check_protected_hardlink` 和 `check_protected_symlink_follow` 实现，在 `sys_linkat` 和 `sys_openat` 中调用。相比 ArceOS 原版，这个模块主动实现了这些安全检查。<sup>[2](#mod-fs-ref-2)</sup>

**2.3 fanotify 模拟**
`FanotifyFd` 完全在内核态模拟，使用全局 `RwLock<Vec<Weak<FanotifyFd>>>` 管理所有实例，每实例通过 `Mutex` 保护标记和事件队列。事件队列大小硬编码为 `FANOTIFY_MAX_QUEUED_EVENTS = 128`，无运行时调整能力。尽管模仿了 Linux 接口，但性能开销较大：每次文件操作都要遍历所有活跃的 `FanotifyFd` 并加锁匹配标记。<sup>[3](#mod-fs-ref-3)</sup>

**2.4 权限检查模型**
权限检查函数（`check_open_access`, `has_open_access`, `can_write_dir` 等）实现了与 Linux 一致的 uid/gid 位掩码比较，并且对 root (euid == 0) 做了特殊放行。然而，在部分路径（如 `check_create_parent_access`）中，仅检查了父目录的写搜索权限，忽略了 ACL 或 capabilities，这是简化实现。<sup>[1](#mod-fs-ref-1)</sup>

### 3. 跨模块协同

**3.1 文件监控与系统调用的耦合**
`inotify` 和 `fanotify` 的事件通知被直接嵌入在文件系统调用的各个分支中。例如，`sys_openat` 在打开成功后调用 `notify_inotify_file` 和 `notify_fanotify_open`；`sys_mkdirat` 成功创建目录后调用 `notify_inotify_dirent` 和 `notify_fanotify_create`。这种紧耦合确保了事件不会遗漏，但也意味着每次文件操作都多了一层通知，可能影响性能。<sup>[6](#mod-fs-ref-6)</sup>

**3.2 信号中断与 I/O 重试**
`io.rs` 中的 `read_file_like` 函数在循环中处理 `EINTR`：如果当前线程有信号待处理且所有信号都允许重启系统调用，则继续循环；否则返回错误。这需要与信号模块配合，读取 `thread.proc_data.signal` 的 pending 集合。<sup>[5](#mod-fs-ref-5)</sup>

**3.3 进程上下文访问**
几乎所有文件系统调用都通过 `current().as_thread().proc_data` 获取当前进程的 euid、egid、umask 等属性。进程的文件描述符表以 `Arc<Mutex<BTreeMap>>` 的形式存储于进程数据中，因此模块大量使用 `FD_TABLE` 的静态引用和进程私有数据之间的协作。<sup>[1](#mod-fs-ref-1)</sup>

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 本模块基于 `starry_signal` crate 实现了与 Linux 兼容的进程信号机制，包括信号发送、交付、阻塞、处理函数注册、备选信号栈、signalfd 等核心抽象。与 ArceOS 基线相比，本仓库自行实现了 `SignalSet`/`SignalInfo` 等类型而非复用 Linux 内核绑定，并利用 async poll 框架在微内核架构中实现信号等待，避免了抢占式内核的中断窗口问题。

### 1. 核心抽象与外部依赖
信号机制的核心抽象来自 `starry_signal` crate：`Signo`（信号编号枚举）、`SignalSet`（信号集位图）、`SignalInfo`（携带发送者信息的信号）、`SignalStack`（备选信号栈描述）、`SignalOSAction`（信号默认动作枚举）。本模块通过 `kernel/src/syscall/signal.rs` 中的系统调用处理函数与这些类型交互，同时通过 `kernel/src/task/signal.rs` 实现信号的检查与交付。与 Linux 完整信号模型相比，本模块省略了 `siginfo_t` 中的无关字段，仅保留必需项，这降低了跨平台适配成本但牺牲了部分兼容性。

`starry_signal` 本身是自实现的，未引用 Linux 内核头文件中的 sigset_t 布局，而是使用固定 64 位的 `SignalSet`（`repr(transparent)` 的 `u64` 包装）。这解释了 `check_sigset_size` 为何只允许 `size_of::<SignalSet>()` 或 0：位图宽度在编译期固定，不支持动态 sigsetsize 兼容老应用。

### 2. 关键设计取舍
**异步阻塞与轮询融合**：`sys_rt_sigtimedwait`、`sys_rt_sigsuspend`、`sys_pause` 均使用 `poll_fn` + `block_on` 组合，在内核栈上执行异步状态机。这种做法避免了为每个等待信号的任务分配独立内核线程，但导致这些系统调用会阻塞当前 CPU 核的整个调度器（因为 `block_on` 忙等待）。这是 ArceOS 非抢占式内核的典型取舍——信号等待被实现为同步阻塞，而非 Linux 的可中断睡眠。

**信号处理的递送点统一**：`check_signals` 是所有信号从挂起队列进入用户态处理的唯一入口。它被嵌入到 `sys_rt_sigtimedwait`、`sys_rt_sigsuspend`、`sys_pause` 的 poll 闭包中，也被 `syscall_entry` 的调度路径调用。这种集中设计简化了信号交付的完整性验证，但使 `check_signals` 成为巨大的匹配分支，且必须在每个可能被信号打断的操作中手动插入。

**ptrace 优先级高于常规处理**：`check_signals` 的开头检查 `ptrace_is_traced`，如果目标被跟踪，则信号不进入用户态处理而是直接转发给 tracer。这符合 Linux 的 ptrace 语义，但实现简单：它只拦截第一个未阻塞的信号，而非 Linux 中支持信号队列和 `PTRACE_INTERRUPT` 的完整模型。

**signalfd 的零拷贝限制**：`SignalfdSiginfo` 结构体使用 `zerocopy::IntoBytes` 直接写入用户缓冲区，避免了中间序列化。但 `from_signal_info` 中硬编码 `ssi_pid=0`、`ssi_uid=0`，因为线程的 `SignalInfo` 不保存发送者 PID/UID 信息。这意味着 signalfd 消费者无法获取信号发送者身份，降低了调试能力和某些工具（如 `strace` -p）的兼容性。

### 3. 跨模块协同
**信号发送链**：`sys_kill`/`sys_tkill` 在 `kernel/src/syscall/signal.rs` 中解析权限后，调用 `kernel/src/task/signal.rs` 的 `send_signal_to_process`/`send_signal_to_thread`。后者先处理特殊信号（`SIGCONT` 唤醒停止状态、`SIGKILL` 直接唤醒），然后调用 `proc_data.signal.send_signal(sig)` 将信号加入进程级队列。若该队列未指定目标线程，`send_signal` 返回一个 `Option<Pid>` 指示要唤醒的线程，然后通过 `task.interrupt()` 中断其等待状态。

**信号恢复与重新阻塞**：`sys_rt_sigreturn` 直接调用 `block_next_signal()` 设置原子标志，然后 `signal.restore(uctx)` 从用户上下文恢复寄存器。当下次 `check_signals` 被调用时，`unblock_next_signal()` 检查该标志并跳过信号检查一次，防止信号处理函数返回后立即再次被同一信号中断。这个机制与 `sys_rt_sigsuspend` 中使用 `check_signals(thr, uctx, Some(old_blocked))` 的临时阻塞恢复配合，构成了面向连接信号安全的完整沙箱。

**signalfd 与 poll 框架**：`Signalfd` 实现了 `Pollable` trait，其 `poll` 检查线程信号队列中是否有匹配挂起信号。当 `read` 被调用时，它使用 `poll_io` 异步等待直到有信号到达。信号到达是通过 `send_signal_thread_inner` 中的 `task.interrupt()` 唤醒的，这触发了 `Signalfd` 注册的 waker，从而退出 `poll_fn` 循环。这种设计将信号机制融入了 ArceOS 的统一 I/O 多路复用体系，但代价是 signalfd 只能读取当前线程的信号，跨线程信号需通过进程队列间接传递。

### 4. 边角细节与不足
1. **`check_sigset_size` 的硬编码大小假设**：`check_sigset_size` 只允许 `size_of::<SignalSet>()` 或 0，不兼容 `sigset_t` 大小变化的旧二进制程序。如果未来 `starry_signal` 改变位图宽度（例如扩展到 128 位），该函数需要同步修改，否则系统调用会错误返回 `EINVAL`。<sup>[1](#mod-signal-ref-1)</sup>
2. **`sys_rt_sigaction` 对 SIGKILL/SIGSTOP 的拒绝过于严格**：Linux 允许查询这些信号的动作（`oldact` 非空时），但当前实现直接返回 `EINVAL`。这违反了 POSIX 标准的读取语义。<sup>[2](#mod-signal-ref-2)</sup>
3. **signalfd 缺少发送者身份**：`SignalfdSiginfo::from_signal_info` 中 `ssi_pid` 和 `ssi_uid` 恒为 0，因为 `SignalInfo` 不保存发送者 PID。这意味着依赖 `si_pid` 的工具（如 `sigwaitinfo`）无法正常工作。<sup>[3](#mod-signal-ref-3)</sup>
4. **`sys_rt_sigtimedwait` 超时行为不精确**：超时后返回 `Err(AxError::WouldBlock)`，但 Linux 在超时且无信号时返回 `EAGAIN`（`WouldBlock` 等效）。然而这里没有清空信号屏蔽的恢复？仔细观察代码：超时分支中执行了 `signal.set_blocked(old_blocked)` 恢复屏蔽，但返回的错误码 `AxError::WouldBlock` 会被转换为 `EWOULDBLOCK` 而非 `EAGAIN`，两者数值相同，但语义上 `sigtimedwait` 应返回 `EAGAIN`，Linux 用 `EAGAIN` 表示超时。这算微小偏差。<sup>[4](#mod-signal-ref-4)</sup>
5. **`BLOCK_NEXT_SIGNAL_CHECK` 的全局性**：该标志是 `static AtomicBool`，对所有线程共享。如果两个线程同时陷入 `sys_rt_sigreturn`，可能会出现竞争条件导致一个线程的错误恢复被另一个线程的错误阻塞。由于 `sys_rt_sigreturn` 是串行执行的（单核非抢占），目前不会触发，但未来若引入 SMP 则必须改为 per-thread 变量。<sup>[5](#mod-signal-ref-5)</sup>
6. **SIGKILL 在信号队列中的处理**：`send_signal_to_thread` 中检测到 `SIGKILL` 时只调用 `stop_event.wake()`，但没有强制终止线程。真正的终止发生在 `check_signals` 的 `Terminate` 分支中，这要求信号必须被递送才能生效。如果线程处于不可中断的阻塞（如等待文件 I/O），`SIGKILL` 不会立即杀死它，这与 Linux 的强制行为不符。<sup>[6](#mod-signal-ref-6)</sup>

整体而言，该信号机制在微内核的异步模型下实现了 Linux 信号接口的核心子集，其取舍凝聚了非抢占调度、单内核栈、无中断上下文等架构约束下的工程智慧。

### ⚠ 开放问题

- `check_signals` 中 ptrace 分支在拦截信号后直接 `do_exit(signo, true)`，对于信号处理的默认动作为 Continue 的信号，ptrace 拦截后的行为不正确。
- signalfd 的 `update_mask` 调用 `self.poll_rx.wake()`，但唤醒后新的 mask 可能使先前匹配的信号不再匹配，导致 `read` 返回 WouldBlock 而非新的匹配信号。
- `sys_rt_sigtimedwait` 在超时后重新设置 `signal.set_blocked(old_blocked)`，但如果 `set` 参数包含未屏蔽的信号，超时后本应将屏蔽恢复为旧值，但代码中只恢复了一次，可能丢失后续修改（如信号处理函数内部调用 sigprocmask）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现了 System V 消息队列、System V 共享内存、管道/FIFO 和 Futex 四种 IPC 机制。核心抽象围绕等待队列（WaitQueue）、共享页（SharedPages）和环形缓冲区（HeapRb）构建，并通过 BiBTreeMap 等数据结构支持双向查找。与原始 ArceOS 相比，此仓新增了完整的 System V IPC 和 Futex 支持，遵循 Linux ABI，但部分阻塞路径和标志处理尚未完成。

### 1. 核心抽象与机制

进程间通信模块涵盖了四种不同语义的 IPC 原语，每种原语都围绕一个核心抽象构建。

**System V 消息队列** 的核心是 `MessageQueue` 结构体，内部使用 `BTreeMap<i64, Vec<Message>>` 按消息类型组织消息<sup>[5](#mod-ipc-ref-5)</sup>。这种设计直接映射了 Linux 的 `msgtyp` 选择语义：当 `msgtyp=0` 时取首条消息；`msgtyp>0` 时取指定类型；`msgtyp<0` 时取类型不大于 `|msgtyp|` 的最小类型。`MsgManager` 作为全局管理器，维护了从 key 到 msqid、从 msqid 到 `Arc<Mutex<MessageQueue>>` 的双层映射，并通过 `Mutex` 提供并发保护。

**System V 共享内存** 的核心是 `ShmInner`，它持有物理页引用 `Option<Arc<SharedPages>>` 以及每个附加进程的虚拟地址范围 `BTreeMap<Pid, VirtAddrRange>`<sup>[2](#mod-ipc-ref-2)</sup>。`ShmManager` 使用自定义的 `BiBTreeMap` 维护 key↔shmid、shmid→`ShmInner` 以及 pid→(shmid→vaddr) 的三层双向映射。`BiBTreeMap` 允许通过 key 或 value 互查，简化了 `shmdt` 和进程退出时的清理逻辑。

**管道/FIFO** 的核心是 `Shared` 结构体，它封装了 `Mutex<HeapRb<u8>>` 环形缓冲区以及三个 `PollSet`（分别用于读就绪、写就绪和关闭通知）<sup>[3](#mod-ipc-ref-3)</sup>。`Pipe` 实现了 `FileLike` 和 `Pollable`，因此可被 epoll/select 多路复用。FIFO 通过全局 `FIFO_PIPES`（`RwLock<BTreeMap<FifoKey, Weak<Shared>>>`）实现设备号+inode 到共享状态的绑定，当所有引用释放后自动清理。

**Futex** 的核心是 `WaitQueue`，它使用 `SpinNoIrq<VecDeque<(Arc<AtomicBool>, u32, Arc<PollSet>)>>` 存储等待者<sup>[9](#mod-ipc-ref-9)</sup>。每个等待者包含一个通知标志、一个 bitset 和一个 `PollSet`，支持带位掩码的唤醒（`FUTEX_WAKE_BITSET`）。`FutexTable` 将内存地址映射到 `Arc<FutexEntry>`，而 `FutexKey` 则区分私有 futex（仅同一进程可见）和共享 futex（跨进程共享）<sup>[4](#mod-ipc-ref-4)</sup>。共享 futex 通过全局 `SHARED_FUTEX_TABLES` 管理，每 100 次操作自动清理弱引用。

### 2. 关键设计取舍

**阻塞与非阻塞的权衡**：消息队列和 Futex 都面临阻塞等待的实现难题。消息队列的 `sys_msgsnd` 在队列满时直接返回 `EAGAIN` 并输出 `warn!("blocking send not implemented")`<sup>[6](#mod-ipc-ref-6)</sup>；`sys_msgrcv` 同样没有实现阻塞接收。这意味着依赖阻塞发送/接收的应用将无法正常工作。相比之下，Futex 通过 `block_on(poll_fn(...))` 和 `interruptible` 包装实现了完整的阻塞等待，支持超时和信号中断<sup>[10](#mod-ipc-ref-10)</sup>。这种差异源于历史演进：Futex 是后来实现的，采用了更成熟的异步框架。

**共享内存的映射策略**：`sys_shmat` 首次附加时创建 `SharedPages` 并调用 `aspace.map` 传入 `Backend::new_shared`，后续附加直接复用已有的 `phys_pages`<sup>[11](#mod-ipc-ref-11)</sup>。这种方法避免了内存拷贝，实现了真正的零拷贝共享。然而，`SHM_RND` 和 `SHM_REMAP` 标志被标记为 `TODO`<sup>[7](#mod-ipc-ref-7)</sup>，这意味着地址对齐和替换映射功能缺失，可能导致某些应用（如 PnP 库）行为异常。

**权限模型的复用**：System V IPC 统一使用 `IpcPerm` 结构体进行权限检查，与 Linux 的 `ipc_perm` 完全兼容<sup>[1](#mod-ipc-ref-1)</sup>。权限检查函数 `has_ipc_permission` 在 `super` 模块中定义，同时被消息队列和共享内存使用。这一设计避免了重复逻辑，但要求调用方正确传递当前进程的 euid/egid（从 proc_data 获取）。

**Futex 表的分离**：私有 futex 表存储在进程的 `ProcessData` 中，每个进程独立；共享 futex 表根据 `FutexKey` 中的弱指针地址归类到全局表<sup>[13](#mod-ipc-ref-13)</sup>。这种分离避免了私有 futex 的全局锁竞争，同时使得共享 futex 在底层物理页或文件被释放时自动失效。

### 3. 跨模块协同

**与虚拟内存模块的协同**：共享内存段依赖 `starry_vm` 的 `SharedPages` 和 `Backend::Shared`。在 `sys_shmat` 中，`aspace.map` 接受 `Backend` 对象，将虚拟页映射到物理页。当进程退出时，`clear_proc_shm` 遍历所有关联 shmid，调用 `detach_process` 并标记清理[ref:2 附件]。这种协同确保了物理页在最后一个附加者解除映射后才释放。

**与文件系统/I/O 多路复用的协同**：`Pipe` 实现了 `Pollable`，其 `register` 方法将 waker 注册到共享的 `PollSet` 中。当管道有数据可读或可写时，`try_read`/`try_write` 会调用 `poll_rx.wake()` 或 `poll_tx.wake()` 唤醒等待者。同时，`notify_async_io` 会通知 epoll 等异步引擎<sup>[8](#mod-ipc-ref-8)</sup>。这种设计使管道能无缝融入 I/O 事件循环。

**与信号机制的协同**：管道写入端检测到 `closed()`（即强引用计数为 1）时，通过 `send_signal_to_process` 向当前进程发送 `SIGPIPE` 信号<sup>[12](#mod-ipc-ref-12)</sup>。Futex 的 `owner_dead` 标志与 robust list 配合，当检测到所有者死亡时，`sys_futex` 返回 `EOWNERDEAD`<sup>[10](#mod-ipc-ref-10)</sup>。这些交互展示了 IPC 模块与信号、进程管理的深度绑定。

### 4. 边角细节与不足

以下结合 10 项取舍清单，列出 7 个具体风险点：

1. **阻塞路径缺失 <sup>[6](#mod-ipc-ref-6)</sup>**：消息队列的阻塞发送/接收未实现，只输出 `warn!` 后返回 `EAGAIN`。这违反了 Linux 语义，可能导致应用挂起或忙等待。用户态可控（大消息或大量消息）可轻松触发此路径，构成功能性 DoS。

2. **未完成的标志处理 <sup>[7](#mod-ipc-ref-7)</sup>**：`sys_shmat` 的 `SHM_RND` 和 `SHM_REMAP` 被标记为 `TODO`。`SHM_RND` 用于地址对齐，缺失会导致某些应用（如 Qt 的共享内存）出错；`SHM_REMAP` 缺失则无法在固定地址替换映射。

3. **Futex 表懒清理的竞争 <sup>[13](#mod-ipc-ref-13)</sup>**：共享 futex 表每 100 次操作才执行弱引用清理，但清理时仅移除强引用计数为 1 且 `is_empty()` 的表。若某 futex 表长期无操作但仍有空等待队列，它可能永久残留在全局映射中。这属于内存泄漏风险，但程度不大。

4. **环形缓冲区缩放的安全性 <sup>[3](#mod-ipc-ref-3)</sup>**：`Pipe::resize` 在 `new_size < buffer.occupied_len()` 时返回 `ResourceBusy`，但 Linux 的 `F_SETPIPE_SZ` 在缩小超出已占用数据时返回 `EBUSY`。此行为合理，但若用户频繁调用 `fcntl` 调整大小，可能导致数据丢失错觉。

5. **全局锁的粒度过粗**：`MSG_MANAGER` 和 `SHM_MANAGER` 均使用单一 `Mutex`，且 `sys_msgget` 等系统调用持有锁期间调用了 `next_ipc_id()`（可能涉及原子操作）和权限检查。虽然 IPC 操作频率不高，但多核场景下锁竞争可能成为瓶颈。

6. **`unsafe` 块的安全契约缺失**：`sys_msgsnd` 中使用 `unsafe { core::ptr::addr_of!((*msgp).mtype) }` 后调用 `vm_read()`，但未提供 `// SAFETY:` 注释说明指针来源可信。虽然 `msgp` 来自用户态且 `vm_read` 执行边界检查，但缺少显式安全注释属于技术债<sup>[1](#mod-ipc-ref-1)</sup>。

7. **FIFO 的 `PollSet` 未处理关闭事件**：`Pipe::register` 将三个 `PollSet` 都注册了 waker，但 `Drop` 实现只调用 `poll_close.wake()`。如果读端在写端之前被关闭，写端可能无法及时得到 `SIGPIPE` 之外的 `EPIPE` 返回，因为 `poll_io` 可能仍在等待 `OUT` 事件。

### 1. 核心抽象：Socket 作为 FileLike 与 axnet 的桥接

网络模块的核心抽象是 `Socket` 结构体（`kernel/src/file/net.rs`），它包装了来自 axnet crate 的 `Socket` 枚举（包含 TCP、UDP、Unix、VSOCK 变体），并通过实现 `FileLike` trait 接入内核的文件子系统。这使得 socket 可以像普通文件一样通过文件描述符表管理，复用 fd 表的生命周期、引用计数、非阻塞标志等机制。`Socket::from_fd` 方法通过 `get_file_like` 从当前进程的 fd 表获取 `Arc<dyn FileLike>`，再 `downcast_arc` 为 `Socket`，失败时返回 `NotASocket` 错误 <sup>[1](#mod-net-ref-1)</sup>。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层以 `handle_syscall` 为入口，通过 `Sysno` 枚举的大 match 分发到具体处理函数。参数通过 `UserContext` 提取，用户空间指针访问经 `starry_vm` 的 `VmPtr`/`VmMutPtr` 封装保证安全。与 ArceOS 基线相比，本模块大量模拟 Linux 系统调用（含已废弃接口），引入 time namespace 偏移、POSIX 定时器等复杂特性，但也存在多处 stub 实现和技术债（如 `sys_seccomp` 直返 0、`sys_keyctl` 极简化）。

### 1. 核心抽象：大匹配分发与参数提取

系统调用层的核心是 `handle_syscall` 函数（<sup>[1](#mod-syscall-ref-1)</sup>），它作为唯一入口由 trap handler 在用户态陷入后调用。函数首先通过 `Sysno::new` 验证系统调用号，然后执行一个涵盖近 300 个分支的 `match` 表达式。每个分支将参数从 `UserContext` 中取出并转为适当类型，调用对应的 `sys_*` 函数。这种大 match 模式在 Linux 兼容内核中常见，与函数指针表相比的取舍是：编译器能够对 match 进行跳转表优化，且每个分支的类型安全由编译器保证；代价是源代码行数膨胀（该文件达 857 行）和编译时间增加。

参数提取中值得注意的是 `split_offset` 函数（<sup>[2](#mod-syscall-ref-2)</sup>），它将两个 32 位参数合并为一个 64 位偏移，用于 `preadv2`/`pwritev2` 等需要 64 位参数的系统调用。这一设计直接反映出 x86_64 的寄存器传递约定：一个 64 位值被拆成两个 32 位寄存器，需要内核重新组装。

### 2. 条件编译与架构差异处理

由于 Linux 系统调用表在不同架构上并不一致，本模块大量使用 `#[cfg]` 属性进行条件编译（<sup>[3](#mod-syscall-ref-3)</sup>）。例如，`Sysno::mkdir` 仅在 x86_64 上定义，而 `Sysno::renameat` 被 `#[cfg(not(target_arch = "riscv64"))]` 排除。这种显式的 cfg 管理方式使得架构差异清晰可见，但维护负担较大：添加新架构时需要逐条调整。此外，部分系统调用的默认实现在未列出的架构上可能被错误地包含或遗漏，当前代码仅覆盖 x86_64 和 riscv64，对于 arm64 等尚未处理。

### 3. 用户空间安全访问与跨模块协同

系统调用处理函数通过 `starry_vm` 提供的 `VmPtr`/`VmMutPtr` 封装访问用户空间内存。以 `sys_clock_gettime`（<sup>[4](#mod-syscall-ref-4)</sup>）为例，它通过 `ts.vm_write()` 将内核计算的 `timespec` 写入用户指针，而无需手动进行页表切换或有效性检查。类似的，`vm_write_slice` 和 `vm_read_slice` 被用于批量数据传输（如 `sys_getrandom` <sup>[11](#mod-syscall-ref-11)</sup>）。这种封装将内存安全检查集中到 `starry_vm` 中，减少了每个系统调用处理函数中的 `unsafe` 块数量。

跨模块协同方面，`UserContext` 由 `axhal` 的中断入口构建，系统调用处理函数通过修改 `UserContext` 的返回值来影响用户态执行流（如 `uctx.set_retval()`）。信号相关的系统调用（如 `rt_sigreturn`）直接操作上下文以实现信号跳板。定时器子系统与 `axtask` 的异步机制协同：`sys_timer_settime` 通过 `spawn_posix_timer` 创建内核线程等待超时。

### 4. 边角细节与不足

尽管功能覆盖面广，模块中存在多处值得警惕的简化实现：

- **`sys_seccomp` 是空操作**（<sup>[8](#mod-syscall-ref-8)</sup>）：无论传入什么参数，直接返回 0。这意味着用户进程可以声称启用了 seccomp 但实际无任何过滤，攻击者可能绕过本应受限制的沙箱。
- **`sys_keyctl` 实现极简**：密钥标识硬编码为 1..5，无真正密钥管理，仅返回固定值。
- **进程记账硬编码**：`sys_acct` 中的 `write_acct_record` 将进程名写为 `b"acct02_helper"`（<sup>[9](#mod-syscall-ref-9)</sup>），忽略实际进程名，可能导致审计记录失真。
- **time 子系统的忙等待**：`sys_setitimer` 使用 `yield_now()` 循环等待 `try_borrow_mut` 成功（<sup>[7](#mod-syscall-ref-7)</sup>），在高竞争场景下可能引发优先级反转和 CPU 浪费。
- **内存序问题**：`TIMENS_OFFSETS_ACTIVE` 标志使用 `Ordering::Release` 写入但读取时使用 `Ordering::Relaxed`（time.rs 行 70 附近），可能读到陈旧值。
- **技术债标记**：多处 FIXME 注释（如 `// FIXME: AnyBitPattern`（<sup>[10](#mod-syscall-ref-10)</sup>）、`// FIXME: Zeroable`）表明部分类型约束未完全处理。
