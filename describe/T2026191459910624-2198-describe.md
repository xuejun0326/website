## 📖 一、总览

本仓库是 ArceOS 家族中面向 Linux 兼容性的实现，在保持微内核模块化设计的基础上，大幅扩展了进程模型、内存管理、文件系统、信号机制和进程间通信等子系统，旨在为未经修改的 Linux 用户态程序提供运行环境。与家族基线（AX 标准实现）相比，其核心定位是“功能完整性”——更注重 ABI 兼容性和系统调用覆盖度，而非教学演示或极致性能优化。

值得关注的架构特点包括：
- **从单进程到完整多进程模型**：通过 `Thread`/`ProcessData` 分层和 `TaskExt` trait 嵌入 `axtask` 调度框架，实现了 clone 系统调用、进程树管理、PID 分配、资源作用域隔离等，使内核能够支持多线程应用 [task.ref:1][task.ref:2]。
- **基于枚举的内存映射策略**：内存管理用 `Backend` 枚举（Linear/Cow/File/Shared）替代 trait object，内联存储后端变体，避免了虚函数调用和堆分配，同时支持线性映射、写时复制、文件后备和共享匿名映射 [mm.ref:2]。
- **用户空间指针的安全封装**：系统调用层使用 `UserPtr`/`VmPtr` 等类型封装用户指针，配合缺页异常转发，实现安全的用户内存访问，降低了内存安全风险 [syscall.ref:2][mm.ref:4]。

## 🎯 二、综合评价

### 整体定位
本仓库在 ArceOS 家族中属于“完整 Linux 兼容型”内核实现。与基线 ArceOS 的模块化 baremetal 设计相比，此仓库显著提升了功能完整性：不仅实现了 196 个系统调用，还引入了多进程模型、写时复制内存管理、POSIX 信号机制、System V IPC 和 epoll 等高级特性。其目标显然是运行真实的 Linux 用户态程序，而非仅作为教学或演示。启动流程采用 `main` 委托给 `starry_kernel::entry::init` 的模式 [boot.ref:1]，替代了 `axmain!` 宏，体现了对传统入口语义的回归。整体上，项目在 ArceOS 框架内最大化兼容了 Linux ABI，尽管部分实现仍处于早期阶段。

### 真正的创新点
尽管整体架构遵循 Linux 设计，但该仓在具体实现上有若干独特之处，值得评审注意：

- **内存管理的 `Backend` 枚举** [mm.ref:2]：用枚举体而非 trait object 实现多态映射策略，避免了动态分发开销和堆分配，同时保持了类型安全。这一设计在 Rust 内核中较为少见，展示了在嵌入式场景下对零成本抽象的追求。
- **用户空间指针的封装范式** [syscall.ref:2][mm.ref:4]：通过 `UserPtr`/`VmPtr` 类型统一管理用户指针的读写，自动进行页表转换和边界检查，并结合缺页异常转发（`accessing_user_memory` 标志 [mm.ref:4]）实现了内核态安全访问用户内存。这减少了裸指针 `unsafe` 的暴露面。
- **文件系统的分页缓存层** [fs.ref:3]：在 VFS 之上独立实现了 `PageCache`，默认 64 页容量的 LRU 缓存，对非 tmpfs 节点启用。这是对 ArceOS 基线的一个实质性增强，提升了文件 I/O 性能。
- **驱动的 `DeviceMmap::Cache` 变体** [drivers.ref:2]：允许将文件内容作为设备内存映射，扩展了设备驱动与文件系统的集成方式，在家族中未见类似实现。
- **信号机制的两级路由与异步检测** [signal.ref:1]：使用信号生成时遍历进程组 [signal.ref:3] 和线程级 `check_signals` 离散检查点，实现了与 Linux 类似的信号递送逻辑。

### 取舍判断
从代码结构和实现细节可以清晰看出该仓的设计优先级：
- **优先兼容性而非性能**：例如 `sys_mremap` 采用简单拷贝而非页表重映射 [mm.ref:10]；COW 帧引用计数限制为 u8 [mm.ref:15]；消息队列阻塞操作未实现 [ipc.ref:2][ipc.ref:3]。这些选择表明团队在有限资源下首先追求接口可用性。
- **优先功能广度而非深度**：系统调用覆盖 196 个，但 `sched_getscheduler`、`sync` 等为存根 [task.ref:15][fs.ref:7]；信号 `SIGSTOP` 被错误实现为 `do_exit` [signal.ref:9]；`CLONE_NEWNS` 等命名空间标志被静默忽略 [task.ref:5]。这些不足反映了重点在于核心流程（fork/execve/mmap/futex）。
- **安全与正确性的让步**：多处使用 `assert!` 可能导致内核 panic [mm.ref:13][task.ref:8]；不安全的 `assume_init` 缺乏 SAFETY 注释 [fs.ref:5][signal.ref:4]；权限检查欠缺失（如 `sys_shmget` [ipc.ref:4]）。这些技术债需要在后续迭代中清理。

### 完成度与不足
**完成度**：整体架构已成形，核心子系统（进程管理、内存管理、文件系统、信号、IPC、网络）均有覆盖。系统调用 196 个，关键组如 fork/execve/wait4/mmap/futex/kill/epoll 已实现。但细节完成度参差不齐：部分调用功能残缺（如信号 Stop）、阻塞操作未实现、riscv64 架构有约 20 个调用缺失 [syscall.ref:8]。

**代码质量**：模块划分清晰，抽象层次合理（如 `FileLike` trait、`Backend` 枚举、`SocketOps` trait）。但存在明显技术债：不安全代码文档缺失、错误处理不一致（有时返回 `0` 而非正确错误码）、部分算法效率较低（如 `check_null_terminated` 跨页反复获取锁 [mm.ref:6]）。总体属于“功能可用但有待打磨”的状态。

综合来看，这是一个有野心的项目，在 ArceOS 生态中推进了 Linux 兼容性的边界，值得关注。

## 🚀 三、启动流程

### 1. 核心抽象：从 `main` 到 `init` 的委托模式

`src/main.rs` 定义了整个系统的用户态入口函数 `main`，这是 Rust 裸机程序的经典写法：通过 `#[no_mangle]` 避免符号修饰，确保链接器能够正确识别。函数内部没有直接进行任何硬件初始化或内核服务启动，而是快速地构造了一个命令行参数切片，然后调用 `starry_kernel::entry::init(&args, &envs)` 将控制权转移给内核的初始化模块。这种“薄入口 + 厚委托”的设计将平台相关的启动细节完全封装在内核 crate 中，使得 `src/main.rs` 可以保持极简，便于在不同项目间复用。与 ArceOS 家族的典型实现（例如使用 `axmain!` 宏自动插入 `main` 并启动异步运行时）相比，本仓库显式地定义了 `main` 函数，并直接以同步方式调用 `init`，体现了对更传统操作系统入口语义的回归。

`starry_kernel::entry::init` 函数（位于 `crates/starry_kernel/src/entry.rs`，虽未直接提供源码，但根据 `call_chain` 和 `handlers` 可推断其职责）负责设置页表、中断向量、调度器以及执行用户态的初始进程。`handlers` 中注册的 `PAGE_FAULT` 和 `IRQ` 处理函数表明 `init` 内部会完成陷阱栈的建立。这种分层设计使得 `main` 函数完全与硬件隔离，便于单元测试和跨平台移植。

### 2. 编译时脚本集成与命令行构造

`CMDLINE` 常量的定义是启动流程中最具特色的部分：
```rust
pub const CMDLINE: &[&str] = &["/bin/sh", "-c", include_str!("init.sh")];
```
它使用 `include_str!` 宏在编译期读取项目根目录下的 `init.sh` 文件内容，直接嵌入到二进制中。这种做法带来了以下优点：
- **零运行时开销**：无需文件系统支持即可获得初始化脚本，适用于无盘嵌入式场景。
- **安全性**：脚本内容在编译时固定，无法被运行时篡改。
- **简化加载流程**：避免了解析外部命令行或读取环境变量的复杂性。

然而，这种设计也放弃了灵活性：任何对 `init.sh` 的修改都需要重新编译整个内核。在迭代开发阶段，这可能成为效率瓶颈。此外，`CMDLINE` 被硬编码为 `/bin/sh -c <script>`，意味着系统必须内置一个 POSIX shell 作为第一个用户态进程。如果目标平台没有 shell，则会导致启动失败。

`main` 函数随后通过 `iter().copied().map(str::to_owned).collect()` 将静态字符串切片转换为堆分配的 `Vec<String>`。这里使用了 `alloc` crate，表明系统至少已经具备堆分配能力。值得注意的是，`envs` 被定义为一个空数组 `[]`，环境变量完全未传递。这在早期启动阶段是合理的，但也意味着所有子进程的环境都依赖于 `init` 内部设置。

### 3. 条件编译与平台差异化

文件末尾的 `#[cfg(feature = "vf2")] extern crate axplat_riscv64_visionfive2;` 是一个针对特定 SBC（SiFive VisionFive 2）的平台支持注入点。通过条件编译，当启用 `vf2` feature 时，会引入该平台的外设驱动和板级初始化代码。这种机制允许同一内核二进制通过不同的 feature 组合适配多种硬件。

但这样做存在两个问题：
- **平台组合爆炸**：每增加一个平台就需要增加一个 feature，且所有平台的代码都会编译进内核（即使未启用），导致二进制膨胀。
- **缺少默认分支**：如果不指定任何 feature，`#[cfg]` 块不会引入任何平台依赖，此时启动流程能否工作完全取决于 `starry_kernel::entry::init` 是否具有合理的默认初始化逻辑。从代码看，没有为 x86_64 或 aarch64 等常见架构提供显式的 crate 导入，这可能意味着这些架构的支持要么不完整，要么通过其他 crate 实现（例如 `axhal` 本身的多架构支持）。`boot_style` 字段标记为 `axhal`，说明硬件抽象层由 `vendor/axhal` 提供，该 crate 应包含对不同架构的默认处理。

### 4. 边角细节与不足

**<sup>[1](#mod-boot-ref-1)</sup> `main` 函数的签名与缺少 `#[panic_handler]`**：`src/main.rs` 仅通过 `#![no_std]` 和 `#![no_main]` 声明了裸机环境，但并未提供 `#[panic_handler]` 或 `#[global_allocator]`。这些关键组件必须由 `starry_kernel` 或其他依赖 crate 提供。如果链接时找不到这些符号，编译器将报错。这属于隐式依赖，增加了项目配置的复杂度。

**<sup>[2](#mod-boot-ref-2)</sup> `CMDLINE` 中使用 `include_str!` 的路径依赖**：`include_str!("init.sh")` 是相对于 `src/main.rs` 的路径。如果构建系统修改了工作目录或文件被移动，会导致编译错误。同时，`init.sh` 的内容会直接暴露在二进制中，可能包含敏感信息。

**<sup>[3](#mod-boot-ref-3)</sup> 环境变量数组 `envs` 为空**：`let envs = [];` 将环境变量完全置空。虽然简化了实现，但 POSIX 标准要求 `init` 进程至少继承内核引导参数。当前实现会使得第一个用户态进程无法获知任何系统配置（如主机名、路径等），可能影响后续初始化。

**<sup>[4](#mod-boot-ref-4)</sup> 未处理的错误路径**：`starry_kernel::entry::init` 的返回值被忽略。如果内核初始化失败（例如内存不足、设备驱动错误），`main` 函数将直接返回，导致系统静默终止。没有任何错误提示或回退机制。

**<sup>[5](#mod-boot-ref-5)</sup> 缺乏对引导加载器的兼容**：`main` 函数以 Rust 标准入口点风格编写，但许多实际硬件（如 RISC‑V）期望一个更底层的入口（如 `_start`）。虽然这里使用了 `#[no_mangle]`，但链接脚本需要确保 `main` 被正确放置。没有提供 `.cargo/config.toml` 或 `linker.ld` 的引用，这给实际部署留下隐患。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `AddrSpace` 为核心抽象，封装了虚拟地址范围 (`VirtAddrRange`)、基于 `MemorySet<Backend>` 的内存区域集合和底层页表 (`PageTable`)。引入 `Backend` 枚举（Linear/Cow/File/Shared）代替 trait object 实现多态映射策略，解决了同一地址空间内不同语义映射（线性映射、写时复制、文件后备、共享匿名）的组合需求。与 ArceOS 基线相比，Starry 增加了完整的用户态地址空间管理、COW 机制、ELF 惰性加载、mmap 族系统调用实现，以及通过页面故障转发实现的内核态用户内存安全访问通路。

### 1. 核心抽象与外部依赖

内存管理的核心是 `AddrSpace` 结构体 <sup>[1](#mod-mm-ref-1)</sup>，它聚合了三要素：地址范围 (`va_range`)、区域集合 (`areas: MemorySet<Backend>`) 和页表 (`pt: PageTable`)。`MemorySet` 来自 `memory_set` crate，维护了按起始地址排序的 `MemoryArea<Backend>` 列表，支持 `find`、`map`、`unmap`、`protect` 等区域级操作。`PageTable` 来自 `axhal::paging`，是对硬件页表的封装（x86_64 PML4 / aarch64 TTBRx / riscv64 SATP），提供 `map`、`unmap`、`protect`、`query`、`cursor` 等底层接口。

`Backend` 是一个枚举而非 trait object，这是有意为之的取舍 <sup>[2](#mod-mm-ref-2)</sup>。枚举使得所有后端变体内联在 `MemoryArea` 中，避免了 trait object 的虚函数调用开销和堆分配，代价是增加后端变体时需要修改枚举定义和所有 match 位置。从 `cow.rs` 和 `mmap.rs` 可看到已实现的变体：`Backend::Cow`（写时复制，含匿名映射和文件后备）、`Backend::Linear`（线性映射，用于信号跳板）、`Backend::File`（文件共享映射）、`Backend::Shared`（共享匿名映射）。这种设计借鉴了 Linux 内核的 `vm_ops` 思想，但通过枚举静态化了虚表。

外部依赖方面，`axerrno` 提供错误类型，`axhal::paging` 提供页表抽象，`axfs::FileBackend` 提供文件 I/O 接口，`axsync::Mutex` 提供并发保护，`memory_addr` 提供地址计算原语，`memory_set` 提供区域集合。这些 crate 均为 ArceOS 生态组件。

### 2. 关键设计取舍

**CowBackend 的引用计数实现**：COW 机制的核心是 `FrameTableRefCount`——一个全局 `BTreeMap<PhysAddr, Arc<SpinNoIrq<FrameRefCnt>>>` <sup>[3](#mod-mm-ref-3)</sup>。每个物理帧对应一个 u8 引用计数。当 count 降为 0 时，先从 BTreeMap 中移除条目，再释放物理帧。这种设计的精妙之处在于：移除和释放的分步操作避免了 ABA 竞态——如果先释放再移除，另一个线程可能在同一物理地址上分配新帧，然后老的移除操作错误地将新帧从表中删除。代价是 u8 计数上限 255，CLONE 密集场景可能溢出（见不足部分）。

**用户内存访问的安全通路**：`access_user_memory()` 函数 <sup>[4](#mod-mm-ref-4)</sup> 设置当前线程的 `accessing_user_memory` 标志，然后执行用户内存操作。若访问触发页面故障，`handle_page_fault` 陷阱处理器 <sup>[5](#mod-mm-ref-5)</sup> 检查该标志并转发到 `AddrSpace::handle_page_fault`。这避免了在每个用户内存访问前手动 populate 页面的繁琐，实现了 lazy 缺页处理。但 `check_null_terminated` 函数 <sup>[6](#mod-mm-ref-6)</sup> 在每个页面边界重新获取 aspace 锁，导致多页扫描时锁竞争严重。

**架构相关的页表策略**：`copy_from_kernel` 函数 <sup>[7](#mod-mm-ref-7)</sup> 在 aarch64 和 loongarch64 上为空操作，因为这两种架构使用独立的用户/内核页表寄存器（TTBR0_EL1 / PGDL）。在其他架构（x86、riscv64）上，需要将内核页表条目复制到用户页表中，以确保用户态通过系统调用访问内核服务时不会缺页。这是一个典型的硬件特性适配。

**ELF 加载的两阶段设计**：`ElfLoader::load` <sup>[8](#mod-mm-ref-8)</sup> 分两个阶段执行：阶段一先扫描 ELF 的 INTERP 段，提前加载动态链接器 ld.so 到 LRU 缓存中；阶段二才清空地址空间并映射。如果 ld.so 加载失败，由于未清除地址空间，原进程内存得以保留，可以返回错误而非直接崩溃。这种设计体现了防御性编程：把不可逆操作（clear）推迟到必须操作。

**syscall 实现的取舍**：`sys_mmap` <sup>[9](#mod-mm-ref-9)</sup> 支持 MAP_SHARED、MAP_PRIVATE、MAP_FIXED、MAP_ANONYMOUS、MAP_POPULATE 等关键标志，并通过 `Backend` 枚举分发到不同的映射后端。但 `sys_mremap` 标有 TODO <sup>[10](#mod-mm-ref-10)</sup>，当前实现是简单的 unmap+mmap+copy，没有使用页表重映射来优化大块移动。`sys_madvise`、`sys_msync`、`sys_mlock2` 均为空操作（返回 Ok(0)），这在早期开发中是可接受的，但会影响依赖这些 syscall 的 runtime（如 Go 运行时）。

### 3. 跨模块协同

**页面故障处理链条**：用户态访问 -> CPU 缺页异常 -> `axhal::trap::handle_page_fault` -> 调用 `access.rs` 注册的 `handle_page_fault` 处理函数 <sup>[5](#mod-mm-ref-5)</sup> -> 检查 `thr.is_accessing_user_memory()` -> 获取进程 aspace 锁 -> `AddrSpace::handle_page_fault` <sup>[11](#mod-mm-ref-11)</sup> -> 查找对应 MemoryArea -> 调用 `Backend::populate` -> COW 模式下调用 `CowBackend::populate` <sup>[12](#mod-mm-ref-12)</sup> -> 触发 COW 或按需分配。整个链条在 IRQ 关闭上下文（`IrqSave`）中运行，aspace 锁与中断的配合需格外谨慎。

**ELF 加载与 Backend 联动**：`loader.rs` 中的 `map_elf` 函数 <sup>[13](#mod-mm-ref-13)</sup> 为每个 PT_LOAD 段创建 `CowBackend`，使用 `FileBackend::Cached` 作为文件数据源。COW 后端的 `alloc_new_at` 在首次缺页时从文件读取数据，实现了惰性加载。文件范围由 `ph.offset` 到 `ph.offset + ph.file_size` 限定，而 `ph.mem_size` 可能大于 `file_size`（对应 .bss 段），超出部分自动补零——因为 `alloc_new_at` 分配的是零初始化帧。

**mmap 与 Backend 创建**：`sys_mmap` <sup>[9](#mod-mm-ref-9)</sup> 根据映射类型和文件描述符选择 Backend 变体：MAP_SHARED + 文件 -> `Backend::new_file`（或 device mmap 路径）；MAP_PRIVATE + 文件 -> `Backend::new_cow`（文件后备 COW）；MAP_ANONYMOUS + MAP_SHARED -> `Backend::new_shared`；MAP_ANONYMOUS + MAP_PRIVATE -> `Backend::new_alloc`（等同于无文件 COW）。这种分发逻辑集中在 `mmap.rs`，保持了 aspace 核心的简洁。

**信号跳板的线性映射**：`map_trampoline` <sup>[14](#mod-mm-ref-14)</sup> 将信号处理跳板代码的物理地址线性映射到用户空间的固定虚拟地址 `SIGNAL_TRAMPOLINE`。线性后端直接存储虚拟地址与物理地址的偏移量，在 map 时计算映射关系。这避免了为跳板代码分配物理帧和拷贝的开销。

### 4. 边角细节与不足

- **<sup>[3](#mod-mm-ref-3)</sup> u8 溢出风险**：`FrameRefCnt` 使用 u8 作为引用计数（最大 255）。在 `clone_map` 中 <sup>[15](#mod-mm-ref-15)</sup>，每次 fork 递增计数，且溢出时仅 `warn!` 并返回错误。若有 255 个进程共享同一物理页，第 256 次 fork 将失败。实际场景中 255 次 fork 后能否到达该页取决于触发 COW 的时机，但理论上存在 DoS 风险。替代方案是使用 u16 或 u32。

- **<sup>[6](#mod-mm-ref-6)</sup> 空终止字符串检查的锁竞争**：`check_null_terminated` 函数在每次跨页时重新获取 aspace 锁，若字符串跨多个页面（如 100KB 的环境变量），会导致频繁的锁获取/释放。实际上只要一次 `can_access_range` 确认整个区域可访问即可，无需逐页检查。这是明显的性能问题。

- **<sup>[10](#mod-mm-ref-10)</sup> 半完成的 syscall**：`sys_mremap` 标记 `TODO: full implementation`，当前实现先 mmap 新区域、拷贝数据、再 munmap 旧区域，完全未利用页表重映射优化。`sys_madvise`、`sys_msync`、`sys_mlock2` 返回 Ok(0)，可能导致用户程序以为操作成功而做出错误假设（例如调用 mlock 期望页面锁定在内存中）。

- **<sup>[13](#mod-mm-ref-13)</sup> load_elf 中的 assert! 可能崩溃**：`assert_eq!(seg_pad, ph.offset as usize % PAGE_SIZE_4K)` 假设 ELF segment 的文件偏移与虚拟地址的页内偏移一致。用户提供的恶意 ELF 若违反此假设，会导致内核 panic。应当返回 `Err` 而非 panic。

- **<sup>[1](#mod-mm-ref-1)</sup> `Drop` 实现锁层级**：`AddrSpace::drop` 调用 `self.clear()`，而 `MemorySet::clear` 内部需要遍历后端并释放帧。COW 后端释放帧时需访问 `FRAME_TABLE` 锁。若持有 aspace 锁的线程 panic 导致 drop，且 panic handler 中又尝试获取 aspace 锁，则可能死锁。但当前设计中 aspace 通常通过 `Mutex` 访问，drop 发生在锁外，风险较低。

fn populate(&self, range: VirtAddrRange, flags: MappingFlags, access_flags: MappingFlags, pt: &mut PageTableCursor) -> AxResult<(usize, Option<PopulateCallback>)> {
    for addr in pages_in(range, self.size)? {
        match pt.query(addr) {
            Ok((paddr, page_flags, page_size)) => {
                if access_flags.contains(MappingFlags::WRITE) && !page_flags.contains(MappingFlags::WRITE) {

            Err(PagingError::NotMapped) => { self.alloc_new_at(addr, flags, pt)?; pages += 1; }
        }
    }
    Ok((pages, None))
}

- COW 帧引用计数为 u8（最大 255），clone_map 中溢出仅 warn! 并返回错误，频繁 fork 场景下可能导致非预期失败 [ref:15]

- sys_mremap 标有 TODO，当前实现是简单拷贝而非页表重映射优化 [ref:10]
- map_elf 中 assert_eq!(seg_pad, ph.offset % PAGE_SIZE_4K) 对恶意 ELF 会直接 panic，应改为返回 Err [ref:13]

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块以 `Thread` 结构体为核心抽象，通过 `TaskExt` trait 嵌入 `axtask` 调度框架，以 `ProcessData` 管理进程共享资源（地址空间、信号表、文件描述符作用域）。与 ArceOS 的裸机单进程不同，本仓库实现了完整的 Linux 兼容进程模型，包括多线程（CLONE_THREAD）、clone flag 验证、进程树管理、`/proc` 文件系统等，并通过 `Scope` 机制实现文件描述符和文件系统上下文的隔离。其独特之处在于以 `extern_trait` 宏将 Rust 堆分配的类型绑定到内核任务扩展点，以及使用 `WeakMap` 全局表进行任务查找。

### 1. 核心抽象与外部依赖

用户任务管理的核心是 `Thread` 结构体（<sup>[1](#mod-task-ref-1)</sup>），它封装了进程共享数据 `ProcessData`、线程级信号管理器 `ThreadSignalManager`、时间管理器 `TimeManager`、退出事件等。`Thread` 不直接实现调度，而是通过 `extern_trait` 宏实现 `axtask::TaskExt` trait（<sup>[2](#mod-task-ref-2)</sup>），将自身作为 `Box<Thread>` 挂载到 `TaskInner` 的扩展点上。`on_enter` 回调负责激活当前进程的 `Scope`（资源作用域），使文件描述符表和文件系统上下文自动切换；`on_leave` 则恢复默认全局作用域。

`ProcessData`（<sup>[3](#mod-task-ref-3)</sup>）是进程内所有线程共享的堆分配对象，包含 `Process`（来自 `starry_process` crate，管理进程树 pid/pgid/session/children）、地址空间 `AddrSpace`、信号动作表 `SignalActions`、`Scope`、以及资源限制 `Rlimits`。`clear_child_tid` 与 `robust_list_head` 字段直接对应 Linux 的 `set_tid_address` 和 robust futex 机制。

与 ArceOS 基线的对比：ArceOS 仅支持内核态任务（无用户进程概念），而本仓库从 `axtask` 的基础调度器出发，通过 `TaskExt` 扩展点嵌入了完整的用户态进程模型，实现了 Linux 语义的 `clone`、`exit`、`wait4` 等系统调用。

### 2. 关键设计取舍

#### 2.1 Clone 系统调用的统一实现

`CloneArgs` 结构体（<sup>[4](#mod-task-ref-4)</sup>）将 `clone`、`clone3`、`fork` 的不同参数模型统一为内部表示。`validate` 方法（<sup>[5](#mod-task-ref-5)</sup>）严格检查 Linux 内核文档中规定的 flag 组合约束，例如 `CLONE_THREAD` 必须同时设置 `CLONE_VM | CLONE_SIGHAND`，`CLONE_SIGHAND` 必须同时设置 `CLONE_VM`，`CLONE_VFORK` 不能与 `CLONE_THREAD` 共存等。这一验证放在早期，避免在资源分配中途失败。

`do_clone` 方法（<sup>[6](#mod-task-ref-6)</sup>）核心逻辑是依据 flag 决定资源共享还是深拷贝：
- **地址空间**：`CLONE_VM` 直接复制 `Arc<Mutex<AddrSpace>>`；否则调用 `try_clone` + `copy_from_kernel`（写时拷贝简化版）。
- **信号动作**：`CLONE_SIGHAND` 共享 `Arc<SpinNoIrq<SignalActions>>`；`CLEAR_SIGHAND` 创建空表；否则深拷贝。
- **文件描述符表**：通过 `Scope` 机制用 `FD_TABLE.scope_mut` 隔离子进程的 fd 表。`CLONE_FILES` 共享（整个 Scope 引用拷贝），否则复制表内容。
- **文件系统上下文**：`FS_CONTEXT` 类似处理。
- **进程树关系**：`CLONE_THREAD` 留在同一进程（通过 `old_proc_data.clone()` 共享 `ProcessData`），否则 fork 出新进程并设置父子关系。

这种设计放弃了简化实现的“全局数据结构”，换取了 Linux 兼容性，但代价是 clone 路径较长，且 `copy_from_kernel` 需要映射物理页，可能成为性能瓶颈。

#### 2.2 资源隔离的 Scope 机制

每个 `ProcessData` 拥有一个 `Scope`，用于隔离文件描述符表和文件系统上下文（<sup>[2](#mod-task-ref-2)</sup> 中的 `on_enter`/`on_leave`）。`Scope` 是由 `scope_local` crate 提供的动态作用域标记：进入用户态时 `ActiveScope::set(&scope)`，退出时恢复全局。这避免了在每次文件操作时传递进程参数，也无需在每个系统调用入口手动切换上下文。

该设计的取舍：它将进程标识与调度上下文解耦，但要求所有文件/fs 操作都通过 `FD_TABLE.scope()` 和 `FS_CONTEXT.scope()` 来访问当前作用域，增加了调用方的认知负担。此外，`Scope` 的引用计数通过 `force_read_decrement` 手动管理（<sup>[2](#mod-task-ref-2)</sup> 行 157），一旦 `on_enter`/`on_leave` 配对出错将导致内存泄漏。

#### 2.3 时间管理集成

`TimeManager`（<sup>[7](#mod-task-ref-7)</sup>）在每个线程的 `RefCell` 中维护，区分 `User`/`Kernel`/`None` 三种状态。上下文切换时通过 `axtask` 注册的定时器回调 `poll_timer`（`kernel/src/task/ops.rs` 行 35-47）调用 `TimeManager::poll`（<sup>[7](#mod-task-ref-7)</sup> 行 168-194），根据当前状态累加 `utime_ns` 或 `stime_ns`，并检查间隔定时器（`ITimer`）是否到期。到期的定时器通过 `send_signal_thread_inner` 发送 `SIGALRM`/`SIGVTALRM`/`SIGPROF`。

`alarm_task`（<sup>[8](#mod-task-ref-8)</sup>）是一个独立的内核线程，使用 `BinaryHeap` 管理绝对截止时间，等待 `EVENT_NEW_TIMER` 事件。这种方式避免了在每个 tick 中断中扫描所有线程的开销，但要求 `wall_time()` 单调递增且精度足够。

### 3. 跨模块协同

#### 3.1 procfs 与任务表的集成

`kernel/src/pseudofs/proc.rs` 的 `ProcFsHandler`（<sup>[9](#mod-task-ref-9)</sup>）调用 `tasks()` 枚举所有任务（来自 `TASK_TABLE`），为每个 PID 创建 `/proc/[pid]` 目录。`ThreadDir`（<sup>[10](#mod-task-ref-10)</sup>）提供 `stat`、`status`、`oom_score_adj`、`fd` 等文件。其中 `fd` 子目录通过 `FD_TABLE.scope` 读取当前线程的文件描述符，`stat` 字段从 `TaskStat` 转换而来，`status` 直接读取 `Thread` 的 `oom_score_adj`。

`ProcessTaskDir`（<sup>[11](#mod-task-ref-11)</sup>）在 `/proc/[pid]/task` 下列出进程的所有线程，通过 `process.threads()` 获取线程 TID 列表，再用 `get_task` 查找任务。这要求 `TASK_TABLE` 中必须包含所有用户线程（由 `add_task_to_table` 注册，<sup>[12](#mod-task-ref-12)</sup>）。

#### 3.2 信号发送与定时器的协同

信号发送最终由 `send_signal_thread_inner`（定义在 `signal.rs`，未直接提供 excerpt）完成，但可以从 `do_exit` 中看到其调用方式（<sup>[13](#mod-task-ref-13)</sup>）：当线程退出且是该进程最后一个线程时，向父进程发送 `exit_signal`（通常为 `SIGCHLD`），并唤醒父进程的 `child_exit_event`。类似地，`poll_timer` 在定时器到期时调用 `send_signal_thread_inner` 发送信号。

#### 3.3 wait4 与 exit 的同步

`wait.rs` 中的 `sys_waitpid`（<sup>[14](#mod-task-ref-14)</sup>）构造 `WaitPid` 枚举，从当前进程的 `children()` 中过滤目标子进程，然后通过轮询或阻塞等待 `child_exit_event`（一个 `PollSet`）。`do_exit`（<sup>[13](#mod-task-ref-13)</sup>）在最后一个线程退出时调用 `child_exit_event.wake()`，唤醒正在 `wait4` 中阻塞的父线程。此外，`clear_child_tid` 写入 0 并唤醒对应的 futex，满足 `set_tid_address` 的约定。

### 4. 边角细节与不足

- **<sup>[15](#mod-task-ref-15)</sup>** `sys_sched_getscheduler` 固定返回 `SCHED_RR`，`sys_sched_getparam` 无操作。这意味着优先级语义完全退化，后续若需要 CFS 或实时策略必须重构调度器层。`[N:3]`
- **<sup>[16](#mod-task-ref-16)</sup>** `sys_sched_setaffinity` 包含 `TODO: support other threads`，目前只支持 pid=0 设置当前线程。若用户对其他线程调用会返回 `EOPNOTSUPP`，可能破坏某些多线程应用的假设。`[N:3]`
- **等待系统调用的 WALL/WCLONE 缺失**：`wait.rs` 行 115 的 `FIXME` 注释明确说明因 `ProcessData` 生命周期问题暂时移除了 `WALL` 和 `WCLONE` 支持。这会导致 `waitpid(-1, ...)` 无法等待“克隆”子进程（如通过 `clone(CLONE_THREAD)` 创建的线程）。`[N:5]`
- **命名空间标志静默忽略**：`clone.rs` 行 87-91 检测到命名空间标志时只 `warn!()` 不返回错误，这不符合 Linux 语义（应返回 `EINVAL` 除非内核编译支持）。可能被容器框架误用。`[N:2]`
- **定时器任务的 assert! 风险**：`timer.rs` 行 239 的 `assert!(guard.pop().is_some_and(...))` 在触发时会使整个内核 panic，而该断言仅在 deadline 不匹配时失败，但由外部中断或竞争引起时难以排查。`[N:3]`
- **时间重新进入处理**：`ops.rs` 中 `poll_timer` 和 `set_timer_state` 使用 `try_borrow_mut`，若失败（由于重入，如 IRQ 嵌套），静默跳过定时器更新，可能导致 `SIGALRM` 丢失。`[N:1]`

```rust
pub fn poll(&mut self, emitter: impl Fn(Signo)) {
    let now_ns = monotonic_time_nanos() as usize;
    let delta = now_ns - self.last_wall_ns;
    match self.state {
        TimerState::User => { self.utime_ns += delta; self.update_itimer(ITimerType::Virtual, delta, &emitter); self.update_itimer(ITimerType::Prof, delta, &emitter); }
        TimerState::Kernel => { self.stime_ns += delta; self.update_itimer(ITimerType::Prof, delta, &emitter); }
        TimerState::None => {}
    }
    self.update_itimer(ITimerType::Real, delta, &emitter);
    self.last_wall_ns = now_ns;
}
```

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 文件系统模块基于 `axfs_ng_vfs` VFS 抽象，高层封装 `FsContext`（类似 `std::fs` 上下文）和 `File`（带 LRU 页缓存的文件句柄），系统调用层实现 Linux 兼容的 POSIX 接口。与 ArceOS 基线相比，本仓库引入了独立的 `CachedFile`/`PageCache` 页缓存层，新增了 epoll、sendfile 等高级 I/O 系统调用，并在路径解析中显式控制符号链接深度。

### 1. 核心抽象：从 VFS 到系统调用

文件系统模块的外部分层清晰。最底层依赖 `axfs_ng_vfs` crate，它定义了核心 VFS 类型 `Location`（文件系统节点句柄）、`FileNode`（节点上的文件操作）、`NodeType`、`VfsError` 等。高层封装位于 `axfs-ng-patched/src/highlevel/`，提供了 `FsContext` 和 `File` 两个主要类型。`FsContext` 封装了一个根目录和一个当前工作目录，提供 `resolve`、`read_dir`、`create_dir`、`remove_file` 等 `std::fs` 风格方法 <sup>[1](#mod-fs-ref-1)</sup>。`File` 结构体包装了 `FileBackend` 枚举（缓存或直通）和一个 `Mutex<u64>` 位置指针，实现了 `Read`、`Write`、`Seek` trait <sup>[8](#mod-fs-ref-8)</sup>。系统调用层在 `kernel/src/syscall/fs/` 下，通过 `FileLike` trait（由 `File`、`Epoll`、`Pipe` 等实现）实现多态，使用 `get_file_like(fd)` 从文件描述符表获取 trait 对象。文件描述符表的分配和查找由 `FileLike::add_to_fd_table` 等方法完成，但相关逻辑在 `kernel/src/file/mod.rs` 中，未直接给出。

与 ArceOS 基线相比，本仓库在 `axfs-ng-patched` 中新增了 `CachedFile` 和 `PageCache`，构建了一个独立的 LRU 页缓存层，位于 VFS 之上。原版 `axfs-ng` 的 `File` 可能直接调用 `Location` 的 `read_at`/`write_at`，而本仓库通过 `FileBackend::Cached` 在读写时经过页缓存，仅对特定类型的文件（字符设备、FIFO、socket）或显式指定 `direct` 时才绕过缓存 <sup>[2](#mod-fs-ref-2)</sup>。这一取舍在性能与内存占用之间做了平衡：对于普通文件，页缓存可以减少 VFS 层和底层驱动的 I/O 次数；对于不可缓存的特殊设备，则避免不必要的数据复制。

### 2. 页缓存与 I/O 策略

`CachedFile` 的核心是一棵 LRU 缓存树，每个缓存项是一个 4 KiB 的物理页（`PageCache`），通过 `global_allocator` 分配。缓存容量默认 64 页，但对 tmpfs 使用无界缓存 <sup>[3](#mod-fs-ref-3)</sup>。`with_page_or_insert` 方法实现按需加载：先查缓存，若未命中则从磁盘读取并插入；若缓存已满，则驱逐最近最少使用的页，写入脏页回磁盘 <sup>[9](#mod-fs-ref-9)</sup>。写操作通过 `write_at_locked` 进行，将修改标记为 dirty，并在驱逐或 `sync` 时写回。`sync` 方法遍历所有缓存页，逐页刷新。

一个重要设计是 `append_lock`：使用 `RwLock` 保护追加操作，允许多个线程同时读/写文件的不同位置，但仅允许一个线程进行追加（获取写锁）<sup>[8](#mod-fs-ref-8)</sup>。这是因为追加需要先获取文件长度再写入，若允许多个追加线程并发，会导致数据交错。

页缓存还支持注册逐出监听器（`EvictListener`），当用户态 DMA 或其他子系统需要跟踪缓存页状态时可以使用。`add_evict_listener` 返回一个句柄，需要调用 `remove_evict_listener` 移除，但该函数是 `unsafe` 的，且没有 `// SAFETY:` 注释，调用者必须保证句柄有效且未被重复移除 <sup>[5](#mod-fs-ref-5)</sup>。

### 3. epoll 实现与跨模块协同

epoll 子系统（`kernel/src/file/epoll.rs`）完全在用户空间 I/O 多路复用层面实现，不与 VFS 层直接交互，而是通过 `FileLike` trait 的 `poll` 和 `register` 方法。`Epoll` 结构体维护一个 `HashMap<EntryKey, Arc<EpollInterest>>` 来跟踪注册的文件描述符。每个 `EpollInterest` 包含触发模式（level-triggered / edge-triggered / one-shot）、事件掩码和一个原子标志 `in_ready_queue`。

当文件可读/可写时，文件系统调用之前注册的 waker（通过 `FileLike::register` 注册的 `InterestWaker`）。`InterestWaker::wake` 使用 `try_mark_in_queue` 原子地将 interest 加入 epoll 的 `ready_queue`，然后唤醒 epoll 的 `poll_ready` 标志。`Epoll::poll_events` 从队列中取出 interest，调用 `consume` 方法检查当前事件是否匹配，并根据模式决定是否将 interest 留在队列中或重新注册 waker <sup>[6](#mod-fs-ref-6)</sup>。

这一设计的关键在于锁的选择：epoll 内部使用 `SpinNoPreempt`（自旋锁），而非 `Mutex`。原因是 `InterestWaker` 可能在中断或信号上下文被调用，若使用可阻塞锁会导致死锁。自旋锁配合 `spin` crate 确保在持有锁期间不会被抢占，适用于原子性地操作软中断上下文。

`Epoll` 本身实现了 `FileLike` 和 `Pollable`，因此可以被其他 epoll 实例监视（尽管实际中较少使用）。它的 `path()` 方法返回 `"anon_inode:[eventpoll]"`，`poll()` 通过检查就绪队列是否为空来判断是否可读。

### 4. 边角细节与不足

**<sup>[7](#mod-fs-ref-7)</sup> dummy 同步**：`sys_sync` 和 `sys_syncfs` 的实现仅输出 `warn!` 日志，不执行任何真正的磁盘同步。这在开发阶段可以接受，但生产环境中会导致数据一致性风险。

**<sup>[5](#mod-fs-ref-5)</sup> unsafe 未审查**：`CachedFile::remove_evict_listener` 被标记为 `unsafe`，但缺少 `// SAFETY:` 注释。安全契约要求调用者传递 `add_evict_listener` 返回的句柄，且不能重复移除。若传入无效指针，可能导致 use-after-free。同样地，`DirBuffer::write_entry` 使用 `FIXME: safety` 标记，直接通过裸指针写入缓冲区，虽然提前检查了剩余空间，但未确认 `linux_dirent64` 的内存布局与目标架构兼容 <sup>[4](#mod-fs-ref-4)</sup>。

**<sup>[10](#mod-fs-ref-10)</sup> 用户态数据假设**：`sys_utime` 和 `sys_utimensat` 使用 `vm_read_uninit` 读取用户空间的 `timespec` 结构，然后调用 `assume_init()`。这假设用户空间传递的结构体已正确初始化，且不会导致未初始化内存的读传播。虽然在实际中用户空间传递的指针和结构通常是有效的，但恶意进程可以传递未映射的内存，导致 panic 而非优雅的 `EFAULT`。

**<sup>[11](#mod-fs-ref-11)</sup> 锁的粒度**：`File::position` 使用 `Mutex` 保护读写位置。在 `read` 和 `write` 方法中，先获取位置锁，然后调用 `self.read_at`（可能涉及 I/O 等待）。虽然位置锁的持有时间很短（仅用于更新位置），但若 `read_at` 因页缓存未命中而触发磁盘 I/O，整个位置锁会被持有，阻塞其他线程对同一文件句柄的 I/O。不过由于位置锁独立于文件数据锁，且一般文件描述符在应用层不共享，风险有限。

**<sup>[1](#mod-fs-ref-1)</sup> 上下文初始化**：`FS_CONTEXT` 使用 `scope_local!` 宏实现每个任务独立的 `Arc<Mutex<FsContext>>`，但初始值从 `ROOT_FS_CONTEXT`（一个 `Once`）克隆。若在 `init_filesystems` 之前尝试访问文件系统，会触发 panic。这种设计在单根文件系统环境下是合理的，但未来若支持容器或命名空间，可能需要改为按需创建。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块基于 `starry_signal` crate 的核心抽象（Signo、SignalSet、SignalInfo、SignalOSAction），实现了 Linux 兼容的信号生成、阻塞、排队、递送全套系统调用（kill/tkill/tgkill、rt_sigprocmask/action/pending/return/timedwait/suspend、sigaltstack、signalfd4）。与 ArceOS 基线相比，依赖 `starry_process` 实现了完整的进程组遍历，使得 kill(-1)、kill(-pgid) 能够正确路由。特点是使用异步中断驱动信号检测（task.interrupt() + check_signals 检查点）、两级进程/线程信号路由、以及基于异步 future 的信号等待（rt_sigtimedwait / rt_sigsuspend）。signalfd 接口兼容 Linux 规范，生成 128 字节 signalfd_siginfo。

### 1. 核心抽象与外部依赖

信号机制的核心抽象由 `starry_signal` crate 提供：`Signo`（信号编号枚举）、`SignalSet`（位掩码集，支持 `|`、`&`、`!` 等位运算）、`SignalInfo`（携带 siginfo_t 语义，包括 signo、code、errno、pid、uid 等字段）、`SignalStack`（备选信号栈描述符）、以及 `SignalOSAction`（枚举 Terminate / CoreDump / Stop / Continue / Handler）。系统调用层通过 `linux_raw_sys::general` 中的 Linux ABI 类型（`kernel_sigaction`、`siginfo`、`timespec`、`MINSIGSTKSZ`、`SI_USER`、`SI_TKILL`、`SIG_BLOCK` / `SIG_UNBLOCK` / `SIG_SETMASK`）与用户态接口，并在边界处使用 `Into`/`From` 转换为内部类型 <sup>[1](#mod-signal-ref-1)</sup>。每个线程（通过 `AsThread` trait 从 `axtask::TaskInner` 转换得到）持有一个 `signal` 字段，提供 `blocked()`、`set_blocked()`、`pending()`、`send_signal()`、`check_signals()`、`dequeue_signal()`、`restore()`、`stack()`、`set_stack()` 等方法，构成了信号操作的全部原语。

该模块解决的问题是：在单地址空间多线程的 Linux 兼容内核中，如何按照 POSIX 规范完成信号的生成、阻塞、排队、递送与处理。与同家族基线（原版 ArceOS）相比，这个仓库依赖 `starry_process` 实现了完整的进程组和会话管理，使得 `kill(-pgid, sig)` 和 `kill(-1, sig)` 能够正确遍历进程树，并跳过调用进程自身 <sup>[3](#mod-signal-ref-3)</sup>。此外，`starry_vm` 提供的 `VmPtr` / `VmMutPtr` 封装用于安全读写用户空间内存（`vm_read_uninit`、`vm_write`），所有用户态指针的访问都经过 `nullable()` 边界检查和 `unsafe` 契约封装。

### 2. 关键设计取舍

**异步中断驱动的信号检测**：信号发送（`send_signal_thread_inner`）并不立即执行用户注册的信号处理函数，而是通过 `task.interrupt()` 中断目标任务的执行 <sup>[11](#mod-signal-ref-11)</sup>。目标任务在下次系统调用返回路径上通过 `check_signals` 函数检查是否有待处理的信号 <sup>[8](#mod-signal-ref-8)</sup>。这避免了在任意用户指令流中插入信号处理逻辑的复杂性，但代价是信号递送存在延迟——如果线程在用户态执行长计算循环而不触发系统调用，信号可能无法及时得到处理。Linux 内核也采用类似的“信号检查点”模型，但这个仓的实现完全基于异步中断，没有使用 IPI 或硬件异常来强制信号处理。

**进程级信号的路由策略**：`send_signal_to_process` 调用 `proc_data.signal.send_signal(sig)`，该方法返回 `Option<Pid>` 指示由哪个线程处理信号 <sup>[12](#mod-signal-ref-12)</sup>。这与 Linux 的“信号可以被进程中任意未阻塞的线程处理”语义一致。相反，`send_signal_to_thread` 直接发送给指定线程 ID，用于 `tkill`/`tgkill`。这种两级路由机制允许灵活的信号分发策略——进程级 kill 自动选择一个未阻塞目标信号的线程，线程级 kill 精确控制目标。`starry_signal` 内部应实现了某种选择算法（如遍历线程查找未阻塞线程或使用主线程），但该细节在该仓库代码中不可见。

**signalfd 作为信号接口的替代方案**：signalfd 允许通过标准文件 I/O 接口接收信号，可与 `select`/`poll`/`epoll` 集成，避免了传统信号处理器的异步执行问题。该仓库的 signalfd 实现符合 Linux 规范，生成 128 字节的 `SignalfdSiginfo` 结构 <sup>[13](#mod-signal-ref-13)</sup>，并通过 `Pollable` trait 实现异步就绪通知 <sup>[15](#mod-signal-ref-15)</sup>。signalfd 的 `has_pending_signals` 和 `dequeue_signal` 方法直接操作当前线程的信号状态 <sup>[14](#mod-signal-ref-14)</sup>，隐含 signalfd 是线程特定的——这与 Linux 行为一致，signalfd 只能读取调用线程的信号。此外，`sys_signalfd4` 支持通过已存在的 fd 更新掩码（而非总是创建新 fd）<sup>[16](#mod-signal-ref-16)</sup>，这是 Linux 3.17+ 引入的优化。

**异步等待模式**：`sys_rt_sigtimedwait` 和 `sys_rt_sigsuspend` 使用了 `axtask::future` 的异步轮询机制 <sup>[6](#mod-signal-ref-6)</sup>。通过 `poll_fn` 闭包配合 `curr.poll_interrupt(cx)` 注册唤醒回调，实现了高效的、不占用 CPU 忙等的信号等待。`rt_sigtimedwait` 还支持超时，通过 `future::timeout` 包装实现。与传统的信号量 + 等待队列实现相比，这种基于 future 的写法更简洁且与 axtask 的协程调度无缝集成。

### 3. 跨模块协同

**系统调用层 ↔ 内部信号处理层**：`kernel/src/syscall/signal.rs` 中的系统调用函数（如 `sys_kill`）负责构建 `SignalInfo`（通过 `make_siginfo`），然后调用 `kernel/src/task/signal.rs` 中的内部函数（`send_signal_to_process`、`send_signal_to_thread`、`send_signal_to_process_group`）<sup>[2](#mod-signal-ref-2)</sup>。内部函数负责查找目标进程/线程（通过 `get_process_data`、`get_task`）、调用 `starry_signal` 的信号排队逻辑（`proc_data.signal.send_signal` / `thr.signal.send_signal`）、以及中断目标任务（`task.interrupt()`）。这种分层隔离了 Linux ABI 接口与内核内部实现，使得系统调用层只需关注参数解析和返回结果处理。

**信号检查点的协作**：`check_signals` 函数在系统调用返回路径上被调用，它调用 `thr.signal.check_signals(uctx, restore_blocked)` 获取待处理的信号及其动作 <sup>[8](#mod-signal-ref-8)</sup>。对于 `SignalOSAction::Handler`，`check_signals` 仅返回 `true` 表明信号已被处理——实际的上下文篡改（栈上构建 sigframe、修改 uctx 的 PC 指向用户信号处理函数）由 `starry_signal` crate 在 `check_signals` 内部完成，无需外部参与。对于 `SignalOSAction::Terminate` 和 `SignalOSAction::CoreDump`，`check_signals` 直接调用 `do_exit` 终止进程。`sys_rt_sigreturn` 则负责在用户信号处理函数返回时，通过 `signal.restore(uctx)` 恢复原始上下文 <sup>[5](#mod-signal-ref-5)</sup>。

**`rt_sigreturn` 与信号重入保护**：`sys_rt_sigreturn` 首先调用 `block_next_signal()` 设置一个全局原子标志 `BLOCK_NEXT_SIGNAL_CHECK` <sup>[10](#mod-signal-ref-10)</sup>，然后恢复用户上下文。这个标志用于防止信号处理函数一返回就被再次中断导致无限递归。该标志的检查点不在所提供的文件中，但推测位于 trap handler 或系统调用入口处，通过 `unblock_next_signal()` 消费该标志。这是典型的“一次信号递送循环”保护机制。

**signalfd 与文件描述符系统的集成**：`sys_signalfd4` 通过 `add_file_like` 将 `Signalfd` 对象注册到进程的文件描述符表中 <sup>[16](#mod-signal-ref-16)</sup>。当 `fd != -1` 时，通过 `Signalfd::from_fd(fd)` 从文件描述符表中查找已有的 signalfd 实例并更新其掩码，实现了运行时的掩码修改而无需关闭重建。`Signalfd` 实现了 `FileLike`（提供 read/write/poll/path）和 `Pollable`（提供 poll/register）两个 trait，使其可以无缝接入 axtask 的异步 I/O 框架。

### 4. 边角细节与不足

**未实现的信号动作**（TODO）：`SignalOSAction::Stop` 的处理直接调用 `do_exit(1)`，这与 POSIX 的“暂停线程”语义完全不符；`SignalOSAction::Continue` 是空操作，意味着 `SIGCONT` 无法唤醒暂停的线程；`SignalOSAction::CoreDump` 仅以 `128 + signo` 作为退出码，没有实际生成 coredump <sup>[9](#mod-signal-ref-9)</sup>。这三处都标注了 `TODO` 注释，表明信号处理尚未完全实现。

**`BLOCK_NEXT_SIGNAL_CHECK` 标志的检查点不透明**：`block_next_signal()` 和 `unblock_next_signal()` 提供了原子标志操作 <sup>[10](#mod-signal-ref-10)</sup>，但在所提供的四个文件中，没有任何代码调用 `unblock_next_signal()` 或检查该标志的值。这意味着该标志的消费者位于其他文件（很可能是 `kernel/src/trap.rs` 或 `kernel/src/syscall/mod.rs` 的返回路径），这个跨文件协作的契约完全依赖隐式约定，缺少可见性。

**信号集大小检查的兼容性处理**：`check_sigset_size` 接受 `size_of::<SignalSet>()` 或 `0` 作为合法大小 <sup>[1](#mod-signal-ref-1)</sup>。接受 `0` 是为了兼容旧版 `sigset_t`（64 位）与新版本（128 位）的差异，这是 Linux 内核的向后兼容性要求。但若 `starry_signal::SignalSet` 的大小与目标架构的 `sigset_t` 不匹配，这种检查可能隐藏类型不匹配的 bug。

**隐藏的用户态指针 `unsafe` 契约**：所有用户态指针的读取都通过 `unsafe { ptr.vm_read_uninit()?.assume_init() }` 进行 <sup>[4](#mod-signal-ref-4)</sup>，且没有 `// SAFETY:` 注释说明为何该访问是安全的（指针合法性由 `starry_vm` 的 `VmPtr` 封装保证，但 assume_init 要求指针指向已初始化的内存）。对于 `*mut SignalSet` 参数，如果用户传入未初始化内存，`assume_init()` 将导致未定义行为。这属于技术债务，需要补充安全契约注释。

**信号发送的原子性 gap**：`send_signal_thread_inner` 中，`thr.signal.send_signal(sig)` 与 `task.interrupt()` 不是原子操作 <sup>[11](#mod-signal-ref-11)</sup>。如果信号在线程被中断前已经被处理（例如在另一个核上运行），额外的中断仅造成一次不必要的系统调用返回路径执行，不会造成语义错误，属于可接受的折衷。但在信号量计数严格依赖的场景下，这种 gap 可能导致信号丢失——不过标准 Unix 信号不支持计数（相同信号合并），所以该竞态是可容忍的。

pub fn sys_sigaltstack(ss: *const SignalStack, old_ss: *mut SignalStack) -> AxResult<isize> {
    let curr = current();
    let sig = &curr.as_thread().signal;

    if let Some(old_ss) = old_ss.nullable() {

    }

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现了 Linux 兼容的 System V IPC（消息队列、共享内存）、管道和 futex。核心抽象包括全局静态管理器（MsgManager、ShmManager）、基于 BTreeMap 的消息队列、基于 SharedPages 的共享内存映射、基于 HeapRb 的管道循环缓冲区，以及基于 Waker 和 SpinNoIrq 的 futex 等待队列。与同家族基线（ArceOS）相比，此仓库添加了完整的 SysV IPC 和 futex 支持，旨在运行 Linux 用户态程序；但阻塞操作和部分标志位处理仍为早期实现。

### 1. 核心抽象与外部依赖

进程间通信模块涵盖四种机制：System V 消息队列、System V 共享内存、管道和 futex。

**消息队列**：核心数据结构是 `MessageQueue`，内部使用 `BTreeMap<i64, Vec<Message>>` 按消息类型分组存储，而非 Linux 内核的全局 FIFO 链表。全局管理器 `MsgManager` 维护 key→msqid 和 msqid→queue 两个映射，所有实例由 `static MSG_MANAGER: Mutex<MsgManager>` 保护 <sup>[19](#mod-ipc-ref-19)</sup>。依赖 `axsync::Mutex` 进行互斥，`starry_vm::VmPtr` 读写用户空间。

**共享内存**：核心是 `ShmInner`，包含物理页引用 `phys_pages: Option<Arc<SharedPages>>`。`ShmManager` 维护 key↔shmid 双向映射（使用 `BiBTreeMap`）和进程级 shmid→vaddr 映射。全局 `SHM_MANAGER` 同样由 `Mutex` 保护 <sup>[8](#mod-ipc-ref-8)</sup>。映射操作通过 `mm` 模块的 `AddrSpace::map` 和 `Backend::Shared` 完成。

**管道**：`Pipe` 封装 `ringbuf::HeapRb<u8>` 作为循环缓冲区，初始容量 64 KiB。通过 `PollSet` 实现事件驱动，集成 `axtask::future::poll_io` 支持非阻塞 I/O。实现 `FileLike` trait，可注册到进程文件描述符表 <sup>[13](#mod-ipc-ref-13)</sup>。

**Futex**：`WaitQueue` 使用 `SpinNoIrq<VecDeque<(Waker, u32)>>` 存储等待任务，支持 bitset 唤醒。`FutexTable` 是 `HashMap<usize, Arc<FutexEntry>>`，每个进程私有一张，共享区域使用全局 `SHARED_FUTEX_TABLES`。`FutexKey` 通过地址解析区分私有（进程内虚拟地址）和共享（基于 `Weak<SharedPages>` 或文件句柄）<sup>[6](#mod-ipc-ref-6)</sup>。

### 2. 关键设计取舍

**消息队列的类型分组 vs FIFO 顺序**：`MessageQueue` 使用 `BTreeMap` 按 `mtype` 组织消息，`find_first_message` 遍历返回类型最小的消息 <sup>[1](#mod-ipc-ref-1)</sup>。这与 Linux 的全局顺序不同——Linux 按消息到达时间排序。这一设计简化了按类型查找（如 `MSG_EXCEPT`、`msgtyp<0` 的搜索），但牺牲了公平性，可能影响依赖消息顺序的应用程序。

**阻塞操作未实现**：`sys_msgsnd` 和 `sys_msgrcv` 在队列满/空时仅 `warn!` 并返回 `EAGAIN`/`ENOMSG`，未实现等待队列 <sup>[2](#mod-ipc-ref-2)</sup><sup>[3](#mod-ipc-ref-3)</sup>。这意味着当前实现只能用于非阻塞场景，与 POSIX 语义有显著差距。

**共享内存权限检查缺失**：`sys_shmget` 直接将 `shmflg` 的低 9 位转为 `MappingFlags`，未调用 `has_ipc_permission` 检查调用者 uid/gid <sup>[4](#mod-ipc-ref-4)</sup>。相比之下，消息队列在 `sys_msgget` 中进行了权限校验。共享内存的权限控制完全依赖文件系统模式位，这可能允许未授权进程获取或创建共享段。

**`SHM_RND` 与 `SHM_REMAP` 未实现**：`sys_shmat` 中明确 `TODO: solve shmflg: SHM_RND and SHM_REMAP` <sup>[5](#mod-ipc-ref-5)</sup>。`SHM_RND` 用于将地址向下舍入到 SHMLBA 边界，`SHM_REMAP` 允许替换已有映射，缺失这些功能可能影响依赖它们的程序。

**Futex 的跨进程支持**：通过 `FutexKey::Shared` 使用 `Weak<SharedPages>` 标识共享内存区域，使不同进程的 `FutexTable` 共享同一等待队列 <sup>[6](#mod-ipc-ref-6)</sup>。这一设计避免了在共享内存中嵌入锁，但弱引用可能因物理页面释放而悬空；不过 `WaitQueue` 只保存 `Waker`，不阻止页面回收。

**管道缓冲区动态调整**：`Pipe::resize` 允许调整环形缓冲区大小，但只有新容量不小于已占用量时才成功，且会对齐到页大小 <sup>[7](#mod-ipc-ref-7)</sup>。这提供了灵活性，但调整操作会拷贝整个缓冲区，不适合高频调用。

### 3. 跨模块协同

**共享内存与虚拟内存映射**：`sys_shmat` 获取进程地址空间锁，调用 `aspace.map` 并传入 `Backend::new_shared`，将物理页面映射到用户指定或自动分配的虚拟地址 <sup>[9](#mod-ipc-ref-9)</sup>。若该物理页面是首次映射，则分配新 `SharedPages`；否则复用已有页面，实现跨进程共享。

**Futex 地址解析与 mm 协作**：`FutexKey::new` 通过 `AddrSpace::find_area` 检查地址所属的 `VmArea`，根据其后端类型（`Backend::Shared`、`Backend::File` 或其他）决定是私有还是共享 futex <sup>[10](#mod-ipc-ref-10)</sup>。这要求 `mm` 模块暴露 `area.backend()` 接口，实现了 IPC 与内存管理的紧密集成。

**进程退出时共享内存清理**：`ShmManager::clear_proc_shm` 在进程退出时被调用，遍历该进程关联的所有共享内存段，调用 `detach_process`，若段标记了 `IPC_RMID` 且附加计数归零，则彻底删除 <sup>[11](#mod-ipc-ref-11)</sup>。该函数与进程生命周期模块协同，防止资源泄漏。

**管道与信号系统协同**：写端检测到读端已关闭时，调用 `raise_pipe` 向当前进程发送 `SIGPIPE` <sup>[12](#mod-ipc-ref-12)</sup>。该函数通过 `send_signal_to_process` 与信号模块交互，符合 Linux 行为。

**管道与文件描述符表**：`sys_pipe2` 创建管道后，通过 `add_to_fd_table` 将读写端注册到当前进程的文件描述符表 <sup>[13](#mod-ipc-ref-13)</sup>，失败时自动回滚。这实现了 IPC 与通用文件系统的统一。

### 4. 边角细节与不足

- **`MSG_COPY` 语义偏差**：`sys_msgrcv` 中 `MSG_COPY` 使用 `get_message_by_index`，该函数按类型顺序索引消息 <sup>[14](#mod-ipc-ref-14)</sup>。Linux 中 `MSG_COPY` 按消息入队顺序索引，两者在有多类型消息时行为不一致。
- **共享内存权限检查遗漏**：`sys_shmget` 在 key 已存在时只验证 size 和 flags，未检查调用者权限 <sup>[15](#mod-ipc-ref-15)</sup>。攻击者可通过已知 key 附加到其他用户的共享段。
- **死代码**：`ShmManager::find_vaddr_by_shmid` 被 `#[allow(dead_code)]` 标记且未使用 <sup>[16](#mod-ipc-ref-16)</sup>，应是重构遗留。
- **全局 Futex 表清理策略**：`FutexTables` 每 100 次操作才清理空表，可能导致内存占用随短暂 futex 使用而增长 <sup>[17](#mod-ipc-ref-17)</sup>。
- **Unsafe 缺乏安全注释**：`pipe.rs` 中 `vacant_slices_mut` 返回的未初始化内存通过 `assume_init_mut` 处理，但未说明安全前提 <sup>[18](#mod-ipc-ref-18)</sup>。
- **IPC ID 无回绕**：`next_ipc_id` 使用 `AtomicI32::fetch_add` 单调递增，不回绕，可能导致 ID 溢出 <sup>[19](#mod-ipc-ref-19)</sup>。Linux 通过 seq 字段重用 ID。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块在 smoltcp 和 virtio-vsock 驱动之上构建了三套 socket 栈：TCP、UDP 和 VSOCK。核心抽象是通过 SocketOps/VsockTransportOps trait 统一暴露 POSIX 风格接口，底层分别由全局 SOCKET_SET（smoltcp socket 集合）和 LISTEN_TABLE（TCP 监听表）以及 VSOCK_CONN_MANAGER 管理。与 ArceOS 基线相比，此仓库增加了完整的 vsock 支持，并引入了 Pollable trait 实现 epoll 事件通知，但 vsock 的写就绪通知尚不完善。

### 1. 核心抽象：三套 Socket 栈的融合设计

网络模块围绕三个全局资源组织：smoltcp 的 `SOCKET_SET`（`tcp.rs`/`udp.rs` 中通过 `crate::SOCKET_SET` 引用）、自定义的 `LISTEN_TABLE`（TCP 监听队列）和 `VSOCK_CONN_MANAGER`（vsock 连接管理器）。TCP 和 UDP 共享 smoltcp 实例，但状态管理不同：TCP 使用 `StateLock` 状态机（Idle→Connecting→Connected→Closed），UDP 则是无状态设计，通过 `RwLock<Option<IpEndpoint>>` 记录本地和远端地址。

VSOCK 完全独立，基于 virtio-vsock 驱动，使用 `VsockConnectionManager` 管理 `Connection` 对象和 `ListenQueue`。每个 `Connection` 内部使用环形缓冲区 `HeapRb<u8>` 作为接收缓冲，并维护 `ConnectionState` 状态机。这种设计使 VSOCK 不依赖 smoltcp，可适应虚拟机场景的专用传输需求。

三种 socket 都通过 trait 暴露：`SocketOps`（TCP/UDP）和 `VsockTransportOps`（VSOCK），系统调用层通过动态分发调用对应方法。`Configurable` trait 统一处理 getsockopt/setsockopt，其中通用选项（如 SO_REUSEADDR）由 `GeneralOptions` 处理，协议特定选项（如 TCP_NODELAY、IP_TTL）由各实现覆写。

### 2. 关键设计取舍

**状态锁与轮询式阻塞**：TCP connect 使用 `self.general.send_poller` 循环调用 `poll_interfaces()` 并检查 `poll_connect()` 结果，直到连接建立或失败。这种同步非阻塞模型简化了并发，但引入了忙等风险——代码通过 `axtask::yield_now()` 让出 CPU。UDP 的 connect 仅记录远端地址，不产生网络交互，体现了 UDP 的无连接特性。

**全局端口分配**：TCP 和 UDP 使用共享的临时端口分配器（`get_ephemeral_port`），通过静态 `Mutex<u16>` 保护。TCP 版本在循环中检查 `LISTEN_TABLE.can_listen(port)` 以避免与监听端口冲突，UDP 版本仅简单递增。这种线性扫描在端口耗尽时可能退化为 16k 次循环，且全局锁在高并发下成为瓶颈。

**事件通知模型**：所有 socket 实现 `axpoll::Pollable` trait，通过 `register` 注册 waker，`poll` 返回就绪事件。TCP/UDP 的 poll 通过 `poll_interfaces()` 驱动 smoltcp 处理接收和发送能力；VSOCK 的 poll 直接查询 `Connection` 的接收缓冲区状态。但 vsock 的 `register` 对 `IoEvents::OUT` 事件仅打印 `warn`（`vsock/stream.rs` 约 330 行），导致 epoll EPOLLOUT 无法正确通知——这意味着应用层写就绪依赖超时或错误唤醒，是已知未完成特性。

**缓冲区硬编码**：TCP 收发缓冲区大小（`TCP_RX_BUF_LEN`/`TCP_TX_BUF_LEN`）、UDP 收发缓冲区（`UDP_RX_BUF_LEN`/`UDP_TX_BUF_LEN`）以及 VSOCK 接收缓冲区（`VSOCK_RX_BUFFER_SIZE=64KB`）均在 `consts` 模块或 `connection_manager.rs` 中硬编码。以太网设备的邻居缓存条目超时（`NEIGHBOR_TTL=60s`）也是定值。缺乏平台级参数化，修改需变更源码多处 grep。

**监听表与 accept 流程**：TCP 的 `LISTEN_TABLE` 独立于 smoltcp 的 socket set，在 `listen()` 时注册端口，`accept()` 时从中取出新连接的 `SocketHandle`，再创建 `TcpSocket::new_connected` 并设置状态为 `Connected`。这种设计将监听队列与业务 socket 分离，避免了 smoltcp 自身 accept 机制的复杂性，但要求 LISTEN_TABLE 内部使用 `Mutex<BTreeMap>` 保护。

### 3. 跨模块协同

**设备层驱动**：`EthernetDevice` 实现了 smoltcp 的 `Device` trait（`device/ethernet.rs` 约 30 行），封装 `AxNetDevice` 并提供 `send_to` 方法。`EthernetDevice` 管理者 ARP 邻居缓存和未解析的待发送包队列。TCP/UDP 通过 `poll_interfaces()` 函数触发 smoltcp 的 `iface.poll()`，由 smoltcp 调用 `EthernetDevice` 收发帧。

**VSOCK 驱动协作**：`VsockStreamTransport` 的方法（如 `connect`、`send`、`shutdown`）调用 `crate::device::*` 模块暴露的函数（`vsock_connect`、`vsock_send`、`vsock_disconnect` 等）。驱动层通过 `VsockConnectionManager::on_connection_request`、`on_data_received`、`on_disconnected`、`on_credit_update` 等回调将事件注入连接管理器。`start_vsock_poll()`/`stop_vsock_poll()` 控制驱动轮询的启动与停止，但 `remove_connection` 中无引用计数地调用 `stop_vsock_poll()` 可能导致多连接下提前停止轮询（见 §4）。

**系统调用分派**：虽然无法读取 `kernel/src/syscall/mod.rs`，但从系统调用列表（socket(198)、bind(200)、connect(203) 等）可知，每个 syscall 对应 socket 对象的相应方法。例如 `sys_socket` 根据 domain/type 创建 `Socket::Tcp`、`Socket::Udp` 或 `Socket::Vsock`，随后 `sys_bind` 等通过匹配调用对应实现的 trait 方法。`GeneralOptions` 中的 `reuse_address` 属性影响 `bind` 时是否跳过地址冲突检查，这是系统调用 `SO_REUSEADDR` 的直接映射。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架核心位于 `kernel/src/pseudofs/device.rs`，通过 `DeviceOps` trait 抽象设备操作，利用 `Device` 结构体将设备包装为 VFS 节点，实现与伪文件系统的集成。核心解决设备操作多态性与文件系统统一访问的问题。与 ArceOS 基线相比，引入 `DeviceMmap::Cache` 变体支持缓存文件映射，并通过 `inherit_methods_macro` 减少样板代码。

### 1. 核心抽象：DeviceOps trait 与设备多态

驱动框架的核心是 `DeviceOps` trait <sup>[1](#mod-drivers-ref-1)</sup>，它规定了设备必须实现的接口：`read_at`、`write_at`、`ioctl`、`as_any`，以及可选的 `as_pollable`、`mmap`、`flags`。该 trait 要求 `Send + Sync`，确保设备对象可以安全地在线程间共享。`as_any` 方法允许具体驱动向下转型（downcast），这是 Rust 中实现类型擦除的常见模式，但使用时需注意安全契约——调用方必须保证类型匹配，否则会 panic。

`DeviceMmap` 枚举 <sup>[2](#mod-drivers-ref-2)</sup> 定义了四种内存映射行为：`None`（不支持）、`Physical(PhysAddrRange)`（直接映射物理地址，适用于帧缓冲等设备）、`ReadOnly`（只读 CoW）、`Cache(CachedFile)`（映射到缓存文件）。`Cache` 变体是此仓库的独特扩展，允许将文件内容作为设备内存映射，这在原版 ArceOS 中未出现。`Physical` 变体依赖 `memory_addr::PhysAddrRange`，暗示平台支持物理地址直接映射（如 riscv 页表或 x86 线性映射）。

`Device` 结构体 <sup>[3](#mod-drivers-ref-3)</sup> 是 `DeviceOps` 与 VFS 的桥梁：内部持有 `SimpleFsNode`（伪文件系统节点）和 `Arc<dyn DeviceOps>`。通过 `#[inherit_methods(from = "self.node")]` 宏自动委托 `NodeOps` 的大部分方法到 `SimpleFsNode`，大幅减少样板代码。但 `sync` 方法直接返回 `Err(VfsError::InvalidInput)` <sup>[4](#mod-drivers-ref-4)</sup>，说明设备节点不支持持久化同步——这是一个有意取舍，因为伪文件系统中的设备通常是易失性的，无需落盘。

### 2. 关键设计取舍：设备节点与 VFS 的集成

驱动框架选择将设备暴露为伪文件系统中的节点，而非独立注册表。这种设计使得用户态可以通过标准的文件 I/O 接口（read、write、ioctl、mmap）访问设备，无需引入额外的系统调用。代价是每次设备操作都要经过 VFS 层的方法分发（动态分发），带来少量性能开销。

`FileNodeOps` 的实现 <sup>[5](#mod-drivers-ref-5)</sup> 直接委托给 `self.ops` 的对应方法。值得注意的是 `set_len` 的实现：它通过尝试 `self.write_at(b"", 0)` 是否成功来判断设备是否可写 <sup>[6](#mod-drivers-ref-6)</sup>。这是一种取巧的“可写性探测”，但可能产生副作用——某些设备在偏移 0 处写入空数据可能触发状态变更（如位置计数器移动或重置）。更安全的做法是让 `DeviceOps` 增加 `writable()` 查询方法。

`Pollable` 实现 <sup>[7](#mod-drivers-ref-7)</sup> 允许设备支持异步 I/O 事件。如果底层 ops 未实现 `as_pollable`，默认返回 `IoEvents::IN | IoEvents::OUT`，即始终认为设备可读写。这对简化普通设备（如随机数生成器）的实现很有用，但对某些阻塞设备（如串口）可能导致忙轮询。

### 3. 跨模块协同

此框架与多个外部 crate 配合：
- **`axfs_ng_vfs`**：提供 VFS 核心类型 `NodeOps`、`FileNodeOps`、`VfsError` 等。`Device` 实现了这两个 trait，使设备节点可以挂载到任意 VFS 树中。
- **`axpoll`**：提供 `IoEvents` 和 `Pollable`，使设备节点可被 epoll/select 机制监听。`Device` 的 `Pollable` 实现委托给 `DeviceOps::as_pollable`，允许驱动自定义事件行为。
- **`axfs::CachedFile`**：用于 `DeviceMmap::Cache` 变体，将缓存文件映射到设备内存空间，适用于虚拟设备或文件后端。
- **`memory_addr::PhysAddrRange`**：用于物理地址映射，与内核内存管理模块（如页表建立）协同，实现用户态直接访问硬件 MMIO 区域。

### 4. 边角细节与不足

- **<sup>[6](#mod-drivers-ref-6)</sup> `set_len` 的可写性探测**：`write_at(b"", 0)` 可能对设备产生不可预期的副作用，如改变内部状态或触发清空操作。建议增加显式的 `writable` 标志或方法。
- **<sup>[4](#mod-drivers-ref-4)</sup> `sync` 未实现**：始终返回 `InvalidInput`，意味着用户程序调用 `fsync` 或 `fdatasync` 时会失败。对于某些需要冲刷缓冲区的设备（如块设备），此实现不适用。
- **<sup>[2](#mod-drivers-ref-2)</sup> `DeviceMmap::ReadOnly` 未定义底层映射策略**：注释说“映射为 CoW”，但代码中仅定义了枚举变体，实际映射逻辑留给了调用方（如 mmap 系统调用处理）。这可能导致不同实现间的语义不一致。
- **缺少中断注册接口**：当前框架未提供任何中断处理程序的注册机制。真实 DMA 设备通常需要中断配合，此框架可能仅适用于轮询或无需中断的设备（如某些平台上的 UART 简单实现）。
- **`as_any` 的安全风险**：`DeviceOps::as_any` 返回 `&dyn Any`，调用方需自行维护类型契约。如果向下转型失败，会直接 panic，可能被用户态输入触发（例如通过 ioctl 传递不匹配的设备参数）。

总体而言，此驱动框架设计轻量，专注于将设备抽象为文件节点，但对复杂设备（中断、DMA、配置空间）的支持尚处于早期阶段。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层是用户态与内核交互的核心接口，基于 `syscalls::Sysno` 枚举实现大匹配分发，所有处理函数统一返回 `AxResult<isize>`。与 ArceOS 基线相比，本仓库（starry）扩展了大量 Linux 兼容调用（epoll、信号、futex 等），并采用 `UserPtr`/`UserConstPtr` 封装用户空间内存访问，集成了 `axpoll` 和 `axtask::future` 实现异步 I/O 多路复用。

### 1. 分发器设计：大匹配 vs. 间接表
`handle_syscall` 使用单个 `match` 语句将系统调用号映射到函数调用，类似编译器中的“解释器”模式。其优势在于简单直接、编译器可检查穷尽性，缺点是 match 分支庞大（约 200 个调用），影响编译时间和二进制大小。替代方案是函数指针表（如 Linux 的 `sys_call_table`），但需要 `unsafe` 的指针转换。此处选择了 Rust 风格的安全 match，体现了对类型安全的偏好。对于未识别的系统调用号，使用 `Sysno::new` 返回 `None` 并直接设置 `-ENOSYS`，避免了 panic 风险。<sup>[1](#mod-syscall-ref-1)</sup>

### 2. 用户空间指针的封装策略
几乎所有系统调用都需要读写用户提供的指针。本模块使用 `UserPtr<T>`、`UserConstPtr<T>`、`VmPtr<T>` 等类型（来自 `starry_vm` crate），提供 `get_as_ref()`、`get_as_mut()`、`vm_write()` 等方法，自动处理页表转换和边界检查。相比之下，ArceOS 原始版本可能直接使用 `*const T` 裸指针。这种封装降低了内存安全风险，但引入了额外的错误传播（`?` 返回 `AxError`），且在 `select.rs` 中仍需使用 `nullable!` 宏处理可选指针，增加了代码复杂度。<sup>[2](#mod-syscall-ref-2)</sup><sup>[3](#mod-syscall-ref-3)</sup>

### 3. I/O 多路复用的异步集成
`select`、`poll`、`epoll` 的实现都基于 `axpoll::IoEvents` 和 `axtask::future::poll_io` 函数。它们将文件描述符的轮询包装成 `FdPollSet`（实现 `Pollable` trait），然后通过 `block_on` 配合 `future::timeout` 实现阻塞等待。这种设计将异步运行时（`axtask`）与同步系统调用接口结合，允许在等待 I/O 时让出 CPU。但不足是所有轮询都发生在 `block_on` 中，若被轮询的文件描述符实现不当，可能长时间阻塞内核线程。<sup>[4](#mod-syscall-ref-4)</sup><sup>[5](#mod-syscall-ref-5)</sup>

### 4. 跨模块协同：epoll 与文件子系统
`sys_epoll_ctl` 和 `sys_epoll_wait` 调用了 `Epoll::from_fd` 和 `Epoll::poll_events`，这些类型定义在 `crate::file::epoll` 模块中。系统调用层不直接管理 epoll 实例，而是通过文件描述符表间接引用。这种设计解耦了系统调用逻辑与具体文件对象实现，但要求 `Epoll` 类型实现 `FileLike` trait，增加了接口约束。类似地，信号系统调用与 `starry_signal::SignalSet` 交互，时间调用与 `axhal::time` 交互，体现了模块化的分层架构。<sup>[6](#mod-syscall-ref-6)</sup><sup>[7](#mod-syscall-ref-7)</sup>

### 5. 边角细节与不足
- **条件编译的架构假设**：大量系统调用（如 `sys_open`、`sys_stat`、`sys_fork`、`sys_select`）仅对 `x86_64` 实现，而 `riscv64` 下使用不同名字（如 `openat` 替代 `open`）。`mod.rs` 中约有 20 处 `#[cfg(target_arch = "x86_64")]`，是潜在的兼容性风险。<sup>[8](#mod-syscall-ref-8)</sup>
- **unsafe 块缺少安全注释**：在 `select.rs` 中，`FD_ZERO` 和 `FD_SET` 调用使用了 `unsafe`，但没有任何 `// SAFETY:` 说明；`resources.rs` 中的 `vm_read_uninit()?.assume_init()` 也缺少安全契约。<sup>[9](#mod-syscall-ref-9)</sup><sup>[10](#mod-syscall-ref-10)</sup>
- **错误吞没**：`sys_prlimit64` 在权限不足时返回 `Ok(0)` 而不是错误码，资源限制被静默忽略；`sys_syslog` 和 `sys_seccomp` 是桩函数，直接返回成功，可能被绕过。<sup>[11](#mod-syscall-ref-11)</sup><sup>[12](#mod-syscall-ref-12)</sup>
- **未完成的功能**：`poll.rs` 中有 `// TODO: handle signal`，`resources.rs` 中有 `// TODO: patch resources`，表明这些实现尚未完全符合 Linux 语义。<sup>[13](#mod-syscall-ref-13)</sup><sup>[14](#mod-syscall-ref-14)</sup>
- **潜在死锁**：`select.rs` 中在持有 `FD_TABLE.read()` 锁的情况下进行 `block_on` 和 `poll_io`，若持锁时间过长，可能阻塞其他试图获取写锁的任务。<sup>[15](#mod-syscall-ref-15)</sup>
