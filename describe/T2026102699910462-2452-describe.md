## 📖 一、总览

该仓库是一个模块化、多架构的实验型内核，以 `myos-kernel` 为核心，围绕 `arch-riscv64`、`arch-loongarch64`、`myos-mm`、`myos-vfs` 等独立 crate 组织子系统。与教学版 rCore-Tutorial 相比，它更强调跨架构抽象和组件化，但完成度参差不齐。整体定位介于演示型与完整 Linux 兼容型之间：实现了进程/线程分离、VMA 管理、VFS 框架和多种文件系统，但网络、IPC 等功能仅搭建骨架。

最值得注意的三个特点：(1) **启动流程的两阶段抽象**（`BootContext` → `BootInfo`）将架构相关的启动参数转换为统一接口，是跨架构支持的关键设计 [boot.ref:1-3]。(2) **用户地址空间的借贷计数与延迟退役**（`RetirementBatch`）将 TLB 刷新与物理页释放原子化，避免了经典内核中的竞争窗口 [mm.ref:2-3]。(3) **网络 socket ID 通过文件 path 编码**，以极简方式复用 fd 机制，体现了“够用即可”的设计哲学 [net.ref:2]。

## 🎯 二、综合评价

### 整体定位
本仓库属于**实验型模块化内核**，旨在探索 Rust 语言在多架构（RISC-V、LoongArch）上的内核开发。它采用 crate 分割设计，将启动、内存管理、VFS、调度等子系统独立为可复用组件，与单体的 rCore-Tutorial 形成鲜明对比。基础事实显示 8 个 workspace member，syscall 覆盖任务（44）、文件（39）、内存（9）、网络（10）、信号（10）等领域，但实际深度不均：启动与内存管理较为完整，任务调度实现了抢占式 FIFO 和进程/线程分离，文件系统支持 tmpfs、procfs、sysfs 和只读 ext4；而网络仅提供 socket 系统调用的本地缓冲版本，未集成协议栈，IPC 除管道外仅暴露 futex 入口。总体而言，该仓库展示了模块化内核的潜力，但离生产级尚有距离。

### 真正的创新点
1. **启动流程的两阶段抽象**：`BootContext` 作为架构相关容器，在 LoongArch 中解析 EFI 表获取设备树 [boot.ref:1]，在 RISC-V 中直接使用 OpenSBI 寄存器 [boot.ref:2]；之后统一降级为架构无关的 `BootInfo` [boot.ref:3]。这种分离避免了启动代码在各架构间的重复，且不影响后续初始化。
2. **VMA 借贷计数与延迟退役**：用户页表创建时通过 `RuntimePageTable` 借用共享内核页表，并原子递增 borrower 计数 [mm.ref:2]；`RetirementBatch` 确保 TLB 刷新后才释放物理页 [mm.ref:3]。这与 Linux 的 RCU 或 ArceOS 的简单回收不同，是本仓库在并发安全上的独特尝试。
3. **网络 socket ID 的文件 path 编码**：`/dev/socket/<id>` 将 socket 映射为特殊的文件路径，从而自然复用 `close` 和 `epoll` 等文件操作 [net.ref:2]。尽管是权宜之计，但展示了资源异质化的设计思路。
4. **生成器模式集成 procfs/sysfs**：`procfs_ops` 等生成器函数在编译期生成 dentry 和 file_operations，提升了代码统一性 [task.ref:5]。

### 取舍判断
开发团队将精力集中在**核心机制与多架构支持**上，有意简化或省略了以下部分：
- **网络协议栈**：socket 层只有本地缓冲，未集成 smoltcp，导致 sendto/recvfrom 不实际收发数据 [net.ref:2]。这是最明显的功能缺失，表明网络尚属概念验证。
- **调度灵活性**：任务表容量硬编码 128 [task.ref:6]，调度策略仅为 FIFO 轮转，无优先级或负载均衡。线程间依赖管理（set_parent）标记为 dead_code，fork 路径不完整。
- **信号与 IPC**：信号仅支持 6 种，实时信号和线程级信号未实现 [signal.ref:3]；IPC 除管道外仅有 futex 系统调用入口，无共享内存或消息队列。
- **错误处理**：多处使用 expect 直接 panic，如驱动中的 DMA 分配失败 [drivers.ref:11]、路径解析中的空路径检测，缺乏弹性降级。
这些取舍使系统可以在有限代码量下跑通基本流程（如运行静态链接程序），但牺牲了健壮性和兼容性。

### 完成度与不足
- **启动流程**：完成度高，双架构均有实现，且设计了回滚机制 [boot.ref:4]。但 LoongArch 启动页表未建立实际映射，后期需补充；unsafe 块缺少 SAFETY 注释。
- **内存管理**：VMA、页表、物理分配三层清晰，借贷计数与延迟退役是亮点。但 mprotect 的 PTE 降级路径不更新 VMA 元数据 [mm.ref:4]，`copy_to_user` 不支持文件映射 [mm.ref:5]。
- **任务调度**：进程/线程分离、上下文切换接口设计合理 [task.ref:5]。但容量固定、调度策略单一、进程关系管理未完全集成。
- **文件系统**：VFS 框架完整，支持多种伪文件系统，ext4 只读快照有预算防耗尽 [fs.ref:1,d]。但 vfat 未实现，错误路径过多 panic。
- **网络驱动**：VirtIO 块设备驱动集成较好，但网络设备驱动与 socket 层脱节，实际网络功能缺失。
- **整体代码质量**：模块化结构清晰，但 unsafe 块普遍缺乏安全论证，锁层次依赖 `LockClass` 但验证不充分，panic 路径控制不足。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块实现了从引导器（QEMU direct boot / OpenSBI）到内核 Rust 入口的过渡。核心抽象是 `BootContext`（架构相关）到 `BootInfo`（架构无关）的两阶段转换。LoongArch 通过解析 EFI system table 提取设备树，RISC-V 直接使用 OpenSBI 寄存器约定。启动页表采用不同的预分配策略：LoongArch 分配四级无效表并利用 `invalid_global` PTE 避免 TLB 干扰，RISC-V 仅分配根页表。模块通过 `phys_access` 子模块直接读写物理地址，在分页未开启时完成元数据收集和页表初始化。

### 1. 核心抽象与外部依赖

启动流程模块围绕 `BootContext` → `BootInfo` 的两阶段抽象设计。`BootContext` 是架构相关的启动参数容器，在 LoongArch 中由 `arch/loongarch64/src/platform/qemu_virt/boot.rs` 的 `boot_context()` 函数构造 <sup>[1](#mod-boot-ref-1)</sup>，在 RISC-V 中由 `arch/riscv64/src/boot.rs` 的 `from_raw()` 直接填充 <sup>[2](#mod-boot-ref-2)</sup>。`BootInfo` 定义在 `boot/src/info.rs`，是一个架构无关的公共启动信息结构，保存原始参数、可选设备树地址、命令行地址等 <sup>[3](#mod-boot-ref-3)</sup>。两阶段分离使得上层初始化代码只需依赖 `BootInfo`，无需感知架构细节。

启动页表抽象则分为 `BootPageTable`，它封装了尚未发布给页表寄存器的根页表物理帧。LoongArch 的 `BootPageTable` 位于 `arch/loongarch64/src/memory/paging/boot.rs`，RISC-V 的对应实现位于 `arch/riscv64/src/memory/paging/boot.rs` [ref:4,5]。两者均依赖外部 crate `myos_mm` 提供的 `EarlyFrameAllocator` 来分配物理帧，并通过 `phys_access` 模块的直接物理内存访问能力写入页表内容。

### 2. 关键设计取舍

**LoongArch 的四级无效表 vs RISC-V 的单级空表**。LoongArch 的 `BootPageTable::allocate` 一次性分配了 4 个物理页（PTE、PMD、PUD、PGD），并用 `invalid_global` 项填充最低级 PTE <sup>[6](#mod-boot-ref-6)</sup>。`LeafPageTableEntry::invalid_global()` 返回一个仅含 `GLOBAL` 位的空表项，这个设计模仿 Linux 的做法：LoongArch 的 TLB 以相邻奇偶页组成一对，如果内核全局映射旁边的 PTE 完全为零，硬件会清除整对 TLB 项的 global 属性，导致性能下降；设置 `GLOBAL` 位可避免此问题 <sup>[7](#mod-boot-ref-7)</sup>。而 RISC-V 的页表在缺失时会触发缺页异常，不需要预留无效项，因此 `BootPageTable` 只分配了根页表帧，初始化为全零 <sup>[5](#mod-boot-ref-5)</sup>。这一取舍直接体现了两个架构对 TLB 一致性模型的不同要求。

**LoongArch 的 EFI system table 解析**。`boot_context()` 通过 QEMU direct boot 传入的 `a2` 指针获取 EFI-style system table，然后遍历 `ConfigurationTable` 查找 `DEVICE_TREE_GUID` 对应的 FDT <sup>[8](#mod-boot-ref-8)</sup>。这种设计使得内核能够在不依赖完整 UEFI 实现的情况下获得设备树，但增加了启动路径的复杂度：需要手动验证签名、限制表项数量防止遍历过深（`MAX_CONFIGURATION_TABLES = 64`），并通过编译器检查结构体大小对齐 <sup>[9](#mod-boot-ref-9)</sup>。RISC-V 直接从 `a1` 寄存器获取 FDT 地址，无需额外解析，体现了 OpenSBI 启动约定的简洁性 <sup>[2](#mod-boot-ref-2)</sup>。

**物理地址直接访问的安全性**。在分页尚未启用时，QEMU 传入的物理地址可以直接用作 CPU 地址（LoongArch 的 DMW 直达，RISC-V 的直接映射）。`read_value()` 使用 `core::ptr::read_volatile` 读取这些物理地址上的 C 布局结构体，并依赖 `phys_access::ram_ptr` 将物理地址转换为可解引用指针 <sup>[10](#mod-boot-ref-10)</sup>。代码注释明确说明“当前尚未开启分页”，但没有使用 `SAFETY` 注释块，这是一个轻微的技术债。`BootPageTable` 初始化也通过 `ram_mut_ptr` 获取裸指针后使用 `write_bytes` 和 `fill` 直接写入，同样依赖该前提。

### 3. 跨模块协同

启动流程模块与内存管理、平台抽象、固件接口三个子系统紧密协作：

- **与 `phys_access` 模块**：`boot.rs` 中的 `read_value` 和页表初始化函数调用 `crate::memory::phys_access::ram_ptr` / `ram_mut_ptr` 将物理地址转换为指针。这是整个启动阶段访问固件数据结构（EFI system table、FDT）和初始化页表的唯一途径。
- **与 `myos_mm` crate**：`BootPageTable` 使用 `EarlyFrameAllocator` 分配物理帧，并借助其 checkpoint/restore 机制保证分配失败时回滚 <sup>[4](#mod-boot-ref-4)</sup>。`LeafPageTableEntry` 接受 `PhysFrame` 和 `MappingOptions` 参数创建页表项，这些类型和枚举来自 `myos_mm`。
- **与平台模块**：LoongArch 的 `crate::platform::boot_context()` 函数（在 `qemu_virt/boot.rs` 中实现）被 `arch/loongarch64/src/boot.rs` 的 `from_raw` 调用，完成架构级启动参数到 `BootContext` 的填充 <sup>[1](#mod-boot-ref-1)</sup>。这种回调机制允许不同平台（如 QEMU virt 与真实硬件）提供各自的参数解析逻辑。
- **与上层初始化层**：`BootContext::into_boot_info()` 将架构相关的 `BootContext` 转换为无架构偏好的 `BootInfo`，随后被传递到 `axhal` 或类似组件进行设备树解析、内存探测等操作 <sup>[11](#mod-boot-ref-11)</sup>。

### 4. 边角细节与不足

1. **`MAX_CONFIGURATION_TABLES` 硬编码** <sup>[9](#mod-boot-ref-9)</sup>：LoongArch 的 `find_device_tree` 用 `64` 作为最大遍历数。这个值来源于经验，但若固件提供了更多配置表，设备树将被忽略。更好的做法是接受固件报告的数值并动态限制，或直接从头部结构的总大小计算。

2. **缺少 `SAFETY` 注释**：`read_value` 和 `initialize_filled_table` 中的 `unsafe` 块没有 `// SAFETY:` 注释说明前置条件 [ref:10,12]。虽然当前上下文合理（分页未启用、帧独占），但违反了 Rust 社区推荐的安全文档惯例，可能在未来修改中被误用。

3. **LoongArch dirty fault 追踪未实现**：`LeafPageTableEntry::new` 的注释明确写道“当前尚未实现 dirty fault 追踪”，因此初始可写映射直接设置硬件 D 和软件 MODIFIED 位 <sup>[13](#mod-boot-ref-13)</sup>。这意味着交换到磁盘时无法通过 dirty 位跟踪脏页，限制了后续虚拟内存系统的功能。

4. **RISC-V 不支持 `MemoryType::Uncached`**：`entry.rs` 中 `leaf()` 函数对 `Uncached` 类型直接返回 `Err(UnsupportedMemoryType)` <sup>[14](#mod-boot-ref-14)</sup>。这简化了初始实现，但若将来需要映射设备内存为 uncached（例如 framebuffer），需要依赖 Svpbmt 扩展或平台 PMA，代码未预留扩展点。

5. **`unwrap` 潜在风险**：`BootPageTable::allocate` 在调用 `initialize_zero_table` 时使用 `.expect()` 处理 `ram_mut_ptr` 的错误 <sup>[12](#mod-boot-ref-12)</sup>。如果 `phys_access` 因某种原因（如地址超出 DMW 范围）失败，内核将直接 panic。虽然概率低，但作为系统初始化代码，更优雅的错误上报（如返回 `Result`）可能更健壮。

```rust
pub(crate) fn boot_context(arg0: usize, arg1: usize, arg2: usize) -> BootContext {
    let mut context = BootContext::new([arg0, arg1, arg2]).with_command_line(arg1);
    if arg2 == 0 { return context; }
    context = context.with_system_table(arg2);
    if let Some(device_tree) = find_device_tree(arg2) {
        context = context.with_device_tree(device_tree);
    }
    context
}
```

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 用户态内存管理基于 myos_mm::UserAddressSpace 抽象 VMA 布局，配合 RuntimePageTable 操作硬件页表，物理内存通过 BuddyAllocator 管理。与同族基线（ArceOS/rCore）相比，本仓引入了共享内核映射的借贷计数、延迟退役（RetirementBatch）机制，以及 mprotect 的 PTE 降级回退。

### 核心抽象与分层架构

内存管理模块的核心抽象是 `UserAddressSpace`（来自 `myos_mm` crate），它封装了 VMA 的插入、分裂、合并以及 TLB 请求规划。与之配套的 `RuntimePageTable`（`kernel/src/runtime_page_table.rs`）负责硬件页表的构建、修改与回收。物理内存则由 `myos_mm::BuddyAllocator`（`mm/src/buddy/allocator.rs`）以伙伴算法管理。这种三层分层（VMA 策略、页表实现、物理分配）将不同关注点解耦，但也在性能关键路径（如缺页处理）上引入了多次锁获取和跨层调用的开销。

`UserMm` 结构体（<sup>[1](#mod-mm-ref-1)</sup>）通过 `IrqSpinLock<UserMmState>` 保护内部状态，其中包含 `UserAddressSpace` 实例、一个 `Option<RuntimePageTable>` 和已映射页面的 `Vec<MappedPage>`。之所以让 `page_table` 可选，是因为在 `new` 创建失败路径上可能尚未分配根页表，避免 `Nullable` 优化；但所有对外操作都通过 `ok_or(NotMapped)` 要求 `page_table` 存在，否则返回错误。这种设计使 `UserMm` 始终处于部分初始化状态的可能性被消除。

### 关键设计取舍

**共享内核映射的借贷机制**：RISC-V 上用户页表通过拷贝高半部分顶级页表项来共享内核映射。`RuntimePageTable::new_user`（<sup>[2](#mod-mm-ref-2)</sup>）分配一个零化根页表，然后从内核页表拷贝高半项。同时，通过 `borrow_shared_kernel_root()` 对全局计数器 `SHARED_KERNEL_ROOT_BORROWERS` 做原子递增。当用户页表释放空表时（`reclaim_empty_tables`），若还有 active borrower 且要回收的页表不在 `USER_RANGE`，则跳过回收（<sup>[2](#mod-mm-ref-2)</sup> 中 `shared_kernel_tables_are_borrowed` 检查）。这避免了提前释放被共享的页表内核级节点，但计数器的 increment 和拷贝之间没有内存屏障来保证顺序一致性，在弱序架构下可能存在可见性问题。同族标准实现（如 ArceOS）多采用直接拷贝整个内核页表副本，本仓的借贷设计减少了页表内存占用，但增加了并发复杂度。

**延迟退役（RetirementBatch）**：`set_program_break` 和 `unmap_range` 等操作调用 `retire_range_locked` 收集需要释放的物理页和页表页，并生成 TLB 请求。`RetirementBatch` 结构体（<sup>[3](#mod-mm-ref-3)</sup>）将 TLB 请求与退役分配绑定，强制调用者在释放物理资源前先完成 TLB 刷新。这种显式的手续避免了传统 OS 中“修改页表 → 释放旧页 → TLB 未刷新”的竞态窗口。缺点是每个需要释放的路径都必须手动构造 `RetirementBatch`，增加了代码侵入性；但在本模块中，所有路径都遵循此契约，一致性较好。

**mprotect 的 PTE 降级回退**：当 VMA 容量耗尽且操作是权限降级（不含 `WRITE`）时，`protect_range` 不分裂 VMA，而是直接修改对应页表项（<sup>[4](#mod-mm-ref-4)</sup>）。该回退路径使用临时只读 VMA 推导映射选项，调用 `protect_page` 更新 PTE，但不更新 `UserAddressSpace` 中的 VMA layout。这意味着后续通过 `layout().find_area()` 查询该区域权限会得到旧值，可能导致策略检查错误（例如 `copy_from_user` 的 `validate_range` 可能放行实际已写保护的页面）。这是工程上的 trade-off：牺牲语义一致性换取 VMA 容量不足时的可用性。修复思路是为 `VmArea` 增加“PTE 孤立”标记，但本模块尚未实现。

### 跨模块协同

用户数据拷贝是典型的跨模块协同场景。`copy_to_user`（<sup>[5](#mod-mm-ref-5)</sup>）先主动调用 `populate_page` 触发缺页分配零页，模拟 Linux 的 uaccess 行为。`populate_page` 内部获取 `state` 锁，通过 `UserAddressSpace::find_area` 确认区域可达，然后分配物理页并调用 `RuntimePageTable::map_page` 建立映射。拷贝阶段再次获取锁，通过 `RuntimePageTable::translate` 获取物理地址，最后通过架构相关的 `phys_access::ram_mut_ptr` 进行直接物理内存访问。这里两次获取同一把锁（`state` 锁）导致 `populate_page` 释放锁后 `copy_to_user` 再次加锁，中间可能被其他 thread 抢占。虽然当前是单核或关中断的，但未来多核环境下需要更细粒度的锁设计。

缺页处理框架在 `UserMm` 中仅定义了 `UserFaultResolution` 枚举（<sup>[6](#mod-mm-ref-6)</sup>），实际处理逻辑未在此模块出现，而是由 trap handler 通过 `UserAddressSpace::handle_page_fault`（来自 myos_mm）处理。本模块未直接实现缺页处理，而是由外部调用 `UserAddressSpace` 的 fault 方法，并通过 `UserMm` 的公共接口进行后续操作。这导致缺页路径上的页表修改（如 cow 拆分）可能绕过 `RetirementBatch` 机制，形成 TLB 刷新缺失。

### 边角细节与不足

- **PTE 回退导致 VMA 元数据不一致**：如 <sup>[4](#mod-mm-ref-4)</sup> 分析，`mprotect` 的 PTE-only 回退不更新 VMA layout，使得后续权限查询（如 `validate_range`）可能依赖错误信息。<sup>[1](#mod-mm-ref-1)</sup> 中 `validate_range` 直接使用 `layout().find_area` 查询 VMA，而未检查实际 PTE，因此存在误判风险。
- **copy_to_user 仅支持匿名页**：`populate_page` 分配零页并映射，对于文件映射 mmap（如 `MAP_SHARED`），应读取文件内容而非零页。当前 mmap 系统调用（`kernel/src/user.rs:4058`）显然支持 `MAP_ANONYMOUS` 标志，但 `populate_page` 未区分，可能导致文件映射数据丢失。<sup>[5](#mod-mm-ref-5)</sup> 展示了 `copy_to_user` 中的缺页触发代码。
- **共享内核页表借贷的并发不确定性**：<sup>[2](#mod-mm-ref-2)</sup> 中的原子计数仅保证不 underflow，但未与 TLB 刷新或页表拷贝同步。在弱序架构上，如果一个 CPU 正在通过拷贝项访问内核页表，而另一个 CPU 正在回收该页表，缺乏足够的屏障。当前代码通过 `Ordering::AcqRel` 提供了顺序，但 `synchronize_kernel_mappings` 的写入和 `reclaim_empty_tables` 的读取之间缺少显式同步。
- **`RuntimePageTable::reclaim_empty_tables` 忽略内核页表回收**：当页表位于内核区域且仍有 borrower 时，函数直接返回 Ok（<sup>[2](#mod-mm-ref-2)</sup> 中 `return Ok(())`）。这意味着内核页表空页在存在 borrower 时永远不会被回收，只有等到最后一个用户页表释放 borrower 计数为零时，才可能通过后续的释放路径回收。这可能导致内核页表内存被长期占用；但在典型场景下，内核页表几乎不产生空表，因此实际影响有限。

总体而言，该模块在 VMA 和页表操作之间引入了清晰的抽象边界，但某些优化路径（PTE 降级、借贷机制）暴露了语义一致性风险，需要在后续迭代中通过更完善的元数据同步来弥补。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本仓的进程与任务调度模块基于一个全局 `IrqSpinLock` 保护的 `Scheduler` 结构体，实现了抢占式、每 CPU FIFO 轮转调度派发器。核心抽象是 `Task`（绑定 `kernel` 层的执行上下文）和 `Thread`（绑定 `process` 层的用户态控制流），二者通过 `Arc<Process>` 和原子 ID 双向链接。与 ArceOS 基线相比，独特之处在于：(1) 引入完整的进程/线程分离模型（支持 CLONE_VM/CLONE_THREAD）；(2) 使用侵入式等待队列和 Completion 原语，并在上下文切换的尾路径处理唤醒竞争；(3) 拥有丰富的 `#[cfg(debug_assertions)]` 验证套件，覆盖抢占、等待、完成、切换竞争和远端 TLB 刷写。

### 1. 核心抽象与调度模型

调度器定义在 `kernel/src/task/mod.rs` 中，围绕三个核心类型构建：**`TaskId`**（newtype over `usize`）、**`Task`**（任务控制块）和 **`Scheduler`**（全局调度器）。`Task` 拥有 `kind`（Idle/KernelThread/SystemThread/UserThread）和 `state`（Runnable/Running/SwitchingOut/Blocked/Idle/Exited）<sup>[4](#mod-task-ref-4)</sup>。`Task` 直接内联架构上下文 `crate::arch::task::Context` 和一个可选的 `KernelStack`，避免了为栈分配额外独立对象<sup>[5](#mod-task-ref-5)</sup>。用户线程通过 `user_thread: Option<Arc<Thread>>` 与 `kernel/src/process.rs` 中的 `Thread` 结构体关联，而 `Thread` 通过 `Arc<Process>` 持有进程级别的资源（地址空间、文件表、信号等）<sup>[12](#mod-task-ref-12)</sup>。

`Scheduler` 结构体包含一个容量为 `MAX_TASKS=128` 的稠密任务表 `Vec<Option<Task>>` 和 `MAX_CPUS` 个 `CpuScheduler` 实例。每个 `CpuScheduler` 持有 `current`、`idle`、`run_queue`（`VecDeque<TaskId>`）、时间片剩余计数（`timeslice_remaining: u32`）以及抢占/中断嵌套计数等核心调度状态<sup>[2](#mod-task-ref-2)</sup><sup>[5](#mod-task-ref-5)</sup>。调度策略是 **preemptive per-CPU FIFO round-robin**，时间片固定为 4 个 timer tick（`DEFAULT_TIME_SLICE_TICKS=4`）<sup>[7](#mod-task-ref-7)</sup>。当任务时间片耗尽（`on_timer_ticks`）或显式 `yield_now` 时，`prepare_yield`/`prepare_preempt` 被调用，它们持有全局锁并返回一个 `ContextSwitch` 指针对（指向当前和下一个任务的 `Context`）<sup>[9](#mod-task-ref-9)</sup>。上下文切换在关中断下执行，避免与中断路径冲突。

与 ArceOS 基线的关键区别是，本仓明确区分了内核线程和用户线程，且 `UserThread` 必须绑定一个 `Arc<Thread>`，该绑定在 `Scheduler::spawn_user` 调用 `thread.bind_scheduler_task(id)` 时建立<sup>[13](#mod-task-ref-13)</sup>。这为后续的信号传递、退出状态回收提供了基础。此外，`SystemThread` 是一种不计入 live 计数的特殊内核线程（如 idle 任务和 reaper 主线程），使用 `counted_kernel_thread` 机制区分<sup>[3](#mod-task-ref-3)</sup>。

### 2. 关键设计取舍

**全局锁 vs. 每 CPU 锁**：整个调度器由一个 `IrqSpinLock` 保护。这是一种粗粒度策略，优点是实现简单，避免了复杂的锁层次和死锁问题。代价是在多核环境下，每次调度操作（yield、block、wake、设置亲和性）都会竞争全局锁，可能成为扩展瓶颈。尽管如此，`IrqSpinLock` 在持有期间会禁用本地中断，因此不会与 irq 路径冲突，但加重了中断响应延迟。

**固定任务表大小**：`MAX_TASKS=128` 硬编码，`allocate_task_id` 在表满时直接 `panic!`<sup>[6](#mod-task-ref-6)</sup>。这在嵌入式/实时场景中可能是合理的（避免动态分配失败），但对于通用负载可能不够。没有提供运行时扩展机制。

**时间片固定**：`DEFAULT_TIME_SLICE_TICKS=4` 不可调节，也没有实现调度类（如 CFS 或实时优先级）。任务只能通过 `sched_setscheduler` 等系统调用改变调度策略，但底层实际仍然是 FIFO 轮转（代码中未实现其他调度类）。

**工作窃取策略**：`steal_runnable` 顺序遍历 CPU 从低到高，只窃取无亲和性限制的任务，且只在本地队列为空时尝试<sup>[11](#mod-task-ref-11)</sup>。这是一种延迟窃取模式，避免在每次调度时检查所有 CPU，但可能导致负载不均衡。

**上下文切换与 MMU 协同**：`switch_mm_irqs_off` 在本地中断禁用、持全局锁的情况下，处理用户态地址空间的切换。它检查当前 CPU 的 `loaded_mm` 是否与即将运行的任务匹配，只有 mismatch 才执行 `deactivate`/`activate`，并维护 `mm_switches` 计数器<sup>[8](#mod-task-ref-8)</sup><sup>[18](#mod-task-ref-18)</sup>。这种设计避免了对同一地址空间的重复 TLB 刷新。

**侵入式等待队列**：`WaitQueue` 使用任务自身的 `wait_channel`、`wait_prev`、`wait_next` 字段构建双向链表，不额外分配节点。`Completion` 基于 `WaitQueue` 实现一次性/全部通知。`wake_one`/`wake_all` 在持有全局调度锁的条件下完成，并自动设置 `need_resched`。`wait_until` 在释放锁之前会重新检查条件，防止丢失唤醒。

### 3. 跨模块协同

**Task 与 Thread/Process 的绑定**：当 `Scheduler::spawn_user` 被调用（通常由 `clone` 或 `execve` 系统调用触发），它创建一个 `Task::user_thread`，并在 `Task` 中存入 `Arc<Thread>`。同时，`Thread::bind_scheduler_task` 将 `TaskId` 写入 `Thread.scheduler_task`（原子变量），形成双向引用<sup>[13](#mod-task-ref-13)</sup>。`Thread` 退出时，通过 `Process::detach_thread` 通知进程，并在 `Drop` 中调用 `parent.mark_child_zombie` 将子进程加入僵尸列表<sup>[17](#mod-task-ref-17)</sup>。

**用户线程执行流**：`user_thread_bootstrap` 作为 `Task` 的入口函数，先调用 `finish_switch` 完成上下文切换的尾工作，然后启用中断，调用 `crate::user::run_scheduled_thread(&thread)`。该函数负责安装用户 trap frame 并 `iret` 到用户态。当用户线程因系统调用或异常返回内核后，`run_scheduled_thread` 返回，接着清理 robust list 和 clear_child_tid，调用 `thread.exit` 设置状态，最后 `exit_current` 切换到其他任务<sup>[15](#mod-task-ref-15)</sup><sup>[16](#mod-task-ref-16)</sup>。

**信号与等待队列**：`Process` 中的 `child_wait: WaitQueue` 用于 `wait4` 系统调用。`Process::mark_child_zombie` 在添加僵尸后调用 `wake_all`，而 `wait_zombie_child` 通过 `child_wait.wait` 阻塞当前任务。`WaitQueue` 同样被用于 TLB 验证（M4C2）的同步屏障<sup>[17](#mod-task-ref-17)</sup>。

**验证基础设施**：`m4c_verify` 和 `m4c2_verify` 模块仅在 `#[cfg(debug_assertions)]` 下编译，它们创建内核线程执行抢占测试、等待队列测试、Completion 测试、切换竞争测试和远端 TLB shootdown 验证。这些测试使用全局静态变量和 `WaitQueue` 协调，并在结束时检查资源泄漏（空闲页面数不变）<sup>[14](#mod-task-ref-14)</sup>。这是该仓区别于 ArceOS 基线的重要特点：它提供了可重入的 scheduler 自检。

### 4. 边角细节与不足

1. **硬编码任务容量**：`MAX_TASKS = 128` 在系统调用 `clone` 或 `fork` 超过此数时会直接 panic，而不是返回错误，构成 DoS 风险<sup>[1](#mod-task-ref-1)</sup><sup>[6](#mod-task-ref-6)</sup>。

2. **锁的粒度和可重入**：全局 `IrqSpinLock` 的临界区较长（包括 dequeue、activate、switch_mm），持锁期间调用 `KernelStack::destroy` 等外部函数可能引发问题。但 `destroy` 在 `Task::destroy_resources` 中调用，该函数在任务被 reaper 回收时才触发，此时任务已不在调度表中，锁已释放。

3. **死代码与技术债**：`Process::set_parent`、`add_child` 使用了 `#[allow(dead_code)]`，表明进程树管理可能尚未完全实现<sup>[19](#mod-task-ref-19)</sup>。类似地，`Thread::prepare_entry` 等函数也未在主要代码路径中调用。

4. **验证仅 debug**：所有 M4C 和 M4C2 验证测试仅在 `#[cfg(debug_assertions)]` 下存在。发布版不含任何运行时自检，如果调度器逻辑存在 bug，可能在生产环境中难以发现。

5. **时间片固定**：调度时没有考虑优先级，所有非 idle 任务同等对待。`sched_setscheduler` 等系统调用虽然有 `stub`，但底层调度类未实现。

6. **亲和性绑定检查**：`enqueue` 中对 `affinity` 的断言是 `assert_eq!(affinity, cpu)`，如果用户通过 `sched_setaffinity` 将任务迁移到其他 CPU，但 `Scheduler` 尚未知晓，可能导致 panic。目前 `clear_current_affinity` 只在迁移测试中使用，正常路径可能缺乏支持。

总之，该模块设计注重正确性和验证，但在可配置性和资源弹性上做了取舍。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块实现了一个基于内存节点（Node）的虚拟文件系统（VFS），核心抽象包括文件元节点、文件操作特征（FileOperations）和挂载表，支持 tmpfs、devtmpfs、proc、sysfs 和只读 ext4 快照。与同一家族的标准实现（如 ArceOS）相比，本模块自行管理 VFS 锁层次（IrqSpinLock + LockClass），并引入了 ext4 只读快照机制，通过 NodeBudget 限制递归深度和节点总数，防止恶意镜像导致的内存或栈溢出。同时采用生成器模式集成 procfs 和 sysfs。不足之处包括 ext4 为只读快照、vfat 挂载占位未实现、部分错误路径使用 expect 直接 panic。

### 1. 核心抽象：Node 与 FileOperations
VFS 的核心是 `Node` 结构体（<sup>[1](#mod-fs-ref-1)</sup>），它封装了 inode 号、文件模式、读写标记、链接计数以及状态（目录/普通文件/符号链接/设备等）。所有文件和目录操作（读、写、定位、统计、目录遍历、ioctl、轮询）都通过 `FileOperations` trait（<sup>[2](#mod-fs-ref-2)</sup>）多态分派，每个 `File` 持有 `Arc<dyn FileOperations>` 和路径信息。这种设计允许不同类型文件使用不同实现：`RegularFile` 将数据存储在 `Vec<u8>` 中，`DirectoryFile` 使用已排序的子节点列表，`DeviceFile` 映射到 Null、Zero 等特殊设备，`BlockDeviceFile` 则通过 BufferCache 读写裸块设备。路径解析函数 `resolve_path` 拼接当前目录和绝对路径，并检测空路径和内嵌空字节。

### 2. 路径解析与并发控制
路径解析依赖 `TREE` 全局锁（<sup>[3](#mod-fs-ref-3)</sup>）保护目录树结构的并发修改。锁层次按照 `LockRank::Vfs` 及其子级排序，以避免死锁。符号链接跟随深度限制为 `MAX_SYMLINK_FOLLOWS = 40`（<sup>[4](#mod-fs-ref-4)</sup>），防止递归循环导致栈溢出。`lookup_follow` 在每次跟随符号链接时递减计数器，超过限制返回 `ELOOP`。节点本身的状态（如目录子节点列表）由 `NODE_LOCK` 级别的 `IrqSpinLock` 保护，与树锁形成两阶段锁协议。挂载表由 `MOUNTS` 锁保护。这种设计在单核或关中断环境下有效，但全局树锁可能成为多核下的性能瓶颈。

### 3. ext4 只读快照与内存文件系统
ext4 文件系统以只读快照方式集成（<sup>[5](#mod-fs-ref-5)</sup>）。`Ext4FileSystem::open` 读取超级块并校验不兼容特性，仅支持 `EXT4_FEATURE_INCOMPAT_FILETYPE|EXTENTS|64BIT|FLEX_BG`。`load_inode_tree` 递归加载 inode 树，使用 `NodeBudget`（<sup>[6](#mod-fs-ref-6)</sup>）限制总节点数不超过 8192，深度不超过 16，extent 树深度不超过 5，以防止畸形镜像造成内存或栈耗尽。快照在 VFS 层表现为一个完整的子树，可通过 `install_ext4_snapshot` 挂载到任意目录（<sup>[7](#mod-fs-ref-7)</sup>）。与直接面向块设备的读写驱动不同，快照模式适合系统初始化和检查，但不支持写操作。挂载时若目标目录非空，会清空其原有子节点（见 mount 函数中 `*target_node.state.lock() = NodeState::Directory(Vec::new())`）。

### 4. 挂载管理与多文件系统集成
`mount` 函数根据 `filesystem_type` 字符串选择不同行为（<sup>[8](#mod-fs-ref-8)</sup>）。`Ext4` 和 `Vfat` 需要块设备源，而 `Proc`、`Sysfs` 和 `Tmpfs` 无需设备。Proc 和 Sysfs 通过生成器函数 `populate_proc_root` / `populate_sysfs_root` 动态创建目录结构和 proc_file_node。`umount` 仅从挂载表移除条目，对 ext4 额外调用 `clear_ext4_snapshot`（实际是清除快照节点，但不写回设备）。`umount2` 标志 `MNT_FORCE`、`MNT_DETACH`、`MNT_EXPIRE` 被接收但行为相同，未实现懒卸载或强制卸载语义。`install_standard_fds` 将 `/dev/console` 打开为 stdin/out/err，挂载表初始化时注册 `/` 为 tmpfs、`/dev` 为 devtmpfs。

### 5. 边角细节与不足
- **<sup>[4](#mod-fs-ref-4)</sup> `MAX_SYMLINK_FOLLOWS = 40`**: 硬编码限制，若用户态通过大量符号链接任意深度可达限制返回 ELOOP，但可能因嵌套导致内核栈溢出（递归跟随实现未展示，但此限制缓解风险）。`<sup>[4](#mod-fs-ref-4)</sup>`
- **<sup>[6](#mod-fs-ref-6)</sup> `NodeBudget` 上限 8192**: ext4 快照总节点数受限，极深层目录树或大目录可能加载失败返回 `Unsupported`。`<sup>[6](#mod-fs-ref-6)</sup>`
- **<sup>[5](#mod-fs-ref-5)</sup> ext4 只读**: 未实现写操作，任何修改尝试（如 `write`、`truncate`）返回 `EROFS` 或 `EINVAL`（取决于调用路径）。`<sup>[5](#mod-fs-ref-5)</sup>`
- **vfat 占位挂载**: `mount` 对 `MountFsType::Vfat` 只插入挂载条目，未实际加载 FAT 表，后续路径解析可能找不到节点。`<sup>[8](#mod-fs-ref-8)</sup>`
- **锁粒度**: `TREE` 全局锁在目录创建/删除期间持有，可能阻塞其他无关操作；`NODE_LOCK` 上的 `IrqSpinLock` 在中断关闭时自动禁用中断，长时间持有会增大中断延迟。`<sup>[3](#mod-fs-ref-3)</sup>`
- **错误处理**: `initialize` 中多个 `insert_child` 调用使用 `expect` 直接 panic，若内存不足或设备注册失败将导致内核崩溃。`<sup>[9](#mod-fs-ref-9)</sup>`
- **umount 简化**: `umount` 仅移除挂载表项，对 ext4 调用 `clear_ext4_snapshot` 但未释放块设备，也未同步脏数据。`<sup>[10](#mod-fs-ref-10)</sup>`

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 本模块实现精简的 POSIX 信号基础设施，核心抽象为 `KernelSigAction`（`#[repr(C)]` 结构体）与 `u64` 位掩码信号集。只定义了 6 个标准信号（SIGINT/9/11/13/15/17），依赖 `process` 模块的 `signals().add_pending()` 完成信号入队。相比 Linux 完整信号模型，在信号数量（64 位掩码限制 63 个信号）、线程级发送（`send_signal_to_thread` 为 dead_code）、实时信号支持上做了大幅取舍，并嵌入 LoongArch64 平台特定 SIGALRM 跟踪。

### 1. 核心抽象与外部依赖

本模块的信号机制围绕两个核心抽象展开。

**`KernelSigAction` 结构体**（第17-23行）是用户态信号处理动作的内核表示。它被显式标记为 `#[repr(C)]`，强制使用 C 语言内存布局，表明该结构体直接与用户空间共享 ABI —— 系统调用 `rt_sigaction` 在内核与用户之间传递的正是此结构体。四个字段 `handler`（信号处理函数地址）、`flags`（SA_* 标志位）、`restorer`（信号返回时调用的恢复代码地址）、`mask`（处理期间要阻塞的信号集）完整覆盖了 `sigaction` 的语义。使用 `usize` 而非特定指针类型，暗示在 64 位与 32 位平台间通过指针宽度适配 <sup>[3](#mod-signal-ref-3)</sup>。

**信号集使用 `u64` 位掩码**（第77-81行），`signal_bit()` 将信号编号 s 映射为 `1_u64 << (s-1)`。这一设计将最大信号编号硬限制为 63（信号 0 被排除，信号 ≥64 返回 `None`），相比 Linux 的 `sigset_t`（通常 128 位或更大）大幅压缩了状态空间。好处是掩码操作（置位/清除/测试）可在单条 ALU 指令内完成，且 `update_mask()` 函数（第66-75行）无需循环。坏处是放弃了与 Linux 完整信号集的兼容性——无法表示 SIGRTMIN~SIGRTMAX（Linux 信号 34~64）之外的信号 <sup>[9](#mod-signal-ref-9)</sup>。

模块的外部依赖集中在 `send_signal()` 函数（第45-59行）：它先调用 `validate_signal()` 检查信号编号合法性，然后通过 `crate::process::lookup_process(pid)` 查找目标进程，最后调用 `process.signals().add_pending(signal)` 将信号挂入进程的待处理队列。`signals()` 返回的信号管理器类型不在本模块定义，说明信号入队与信号交付（delivery）的逻辑完全托管在 `process` 模块中。这种分离使 `signal.rs` 保持轻量，仅承担验证与查找的职责 <sup>[5](#mod-signal-ref-5)</sup>。

### 2. 关键设计取舍

**进程级 vs 线程级信号发送。** `send_signal_to_thread()`（第62-64行）被标记为 `#[allow(dead_code)]`，且其实现直接退化为 `send_signal(thread.process().id(), signal)`——它完全忽略了线程参数，将信号发往整个进程。这意味着内核目前仅支持进程级信号发送，`tkill`/`tgkill` 系统调用虽在 syscall 表中注册，但其内核路径最终会落入进程级发送。这是对 Linux 线程信号模型的显著简化，放弃了将信号精确投递到特定线程的能力，但避免了线程组内信号竞争与转发逻辑 <sup>[7](#mod-signal-ref-7)</sup>。

**不可屏蔽信号的强制过滤。** `unblockable_mask()`（第83-85行）将 SIGKILL(9) 和 SIGSEGV(11) 标记为不可屏蔽。`update_mask()` 在每次更新掩码后执行 `next & !unblockable_mask()`，确保无论用户如何调用 `rt_sigprocmask`，这两个信号始终不被阻塞。SIGKILL 不可屏蔽是 POSIX 强制的正确选择；但 SIGSEGV 在 Linux 上实际是可阻塞的（尽管阻塞会导致不可预测行为），这里额外将其加入黑名单，是更保守的安全策略——防止用户意外屏蔽段错误信号导致进程无法被内核终止。替代方案是在检查点（signal delivery）判断，而非修改掩码时强制清除，当前设计更早暴露错误但修改了用户传入的 mask 值 <sup>[10](#mod-signal-ref-10)</sup>。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块以 `NetDevice` trait 为设备抽象层，以全局 `SOCKET_TABLE`（`IrqSpinLock<BTreeMap<usize, SocketInner>>`）为 socket 状态存储，实现了 Linux 兼容的 10 个 socket 系统调用（socket/bind/connect/accept/listen/sendto/recvfrom/shutdown/setsockopt/getsockopt）。socket 层与 VirtIO-Net 驱动之间存在断层：sendto/recvfrom 仅在进程本地 `recv_buf` 上操作，未集成 smoltcp 协议栈，实际网络 I/O 缺失。与 ArceOS 原版相比，本实现仅搭建了系统调用骨架，状态机高度简化（无 TCP TIME_WAIT 等），且通过文件 path 编码 socket ID 实现 fd 复用，这是较独特的工程设计。

### 1. 核心抽象：NetDevice trait 与 Socket 状态机

网络子系统定义了两层抽象。设备层通过 `NetDevice` trait（`mod.rs:19-31`）<sup>[1](#mod-net-ref-1)</sup> 描述网络接口的基本操作：`transmit`/`receive`/`poll_receive`/`mac_address`/`mtu`。该 trait 要求 `Send + Sync + 'static`，允许将具体驱动（如 VirtIO-Net）作为 `Arc<dyn NetDevice>` 全局注册。socket 层则通过 `SocketInner` 结构体（`socket.rs:35-42`）<sup>[2](#mod-net-ref-2)</sup> 维护每个 socket 的状态机：`SocketType` 枚举分为 `Tcp(TcpState)` 和 `Udp(UdpState)`，其中 `TcpState` 包含 `Created`→`Bound`→`Listening`/`Connected` 的简化状态转换。这种设计是显式的取舍——放弃完整的 TCP FSM（无 SYN_SENT/ESTABLISHED/CLOSE_WAIT 等），换取内核实现的极低复杂度。决定将地址结构固定为 `SockAddrIn`（仅 IPv4，16 字节 `sockaddr_in`），`AF_INET6` 常量虽声明但从未在实际分支中使用（`socket.rs:28`），属死代码。

### 2. 系统调用接口的设计取舍

socket 系统调用采用“全局表 + path 编码”架构。每个 socket 在创建时由 `allocate_socket_id()` 分配全局唯一 ID，并作为 `File` 对象的 path 字段（`"socket:{id}"`）编码（`socket.rs:256`）；后续系统调用通过 `get_socket_id_from_fd()`（`socket.rs:201-215`）解析 path 字符串得到 socket ID。这种做法复用 VFS 的 fd 分配和文件查找机制，无需单独维护 socket fd→ID 映射表，但字符串解析和堆分配引入额外开销。`sys_socket` 中通过位掩码提取 `SOCK_CLOEXEC`/`SOCK_NONBLOCK`（`socket.rs:243-244`），但 `ioctl` 实现（`socket.rs:163-175`）直接忽略 FIONBIO 设置——非阻塞是默认行为，这是故意降低复杂度。`sendto`/`recvfrom` 的 flags 校验（`socket.rs:555-565`）仅接受 `MSG_DONTWAIT`/`MSG_PEEK`/`MSG_NOSIGNAL` 等子集，其余标志被拒绝，限制了应用兼容性。

### 3. 跨模块协同的断裂

当前实现中存在明显的设计断裂：设备层与 socket 层之间没有数据通路。`VirtioNetDevice`（`virtio_net.rs:30-78`）完整实现了 `NetDevice` trait，其 `transmit` 调用 `VirtIONetRaw::send`，`receive` 通过 `poll_receive` + `receive_wait` 驱动 VirtIO 队列<sup>[5](#mod-net-ref-5)</sup>。然而 socket 层的 `sys_sendto`（`socket.rs:550-580`）仅从用户空间拷贝数据到局部缓冲区并返回，既不调用 `NetDevice::transmit`，也不向 `recv_buf` 写入任何数据；`sys_recvfrom`（`socket.rs:581-637`）从 `inner.recv_buf` 读取，但该缓冲区从未被填充（因为没有中断/轮询机制接收网络帧）。设备注册接口 `register_interface`（`mod.rs:73-82`）只是将设备存入 `INTERFACES` 静态列表，socket 层完全无视该列表。这意味着即使物理网卡收到数据包，也没有任何代码将其路由到对应 socket 的 `recv_buf`。`struct SocketFile` 的 `read`/`write` 实现（`socket.rs:104-130`）同样只操作进程本地缓冲区，不触发任何网络传输<sup>[3](#mod-net-ref-3)</sup>。这种断裂使得当前网络模块只能用于编译时链接通过，无法完成任何实际通信。

### 4. 边角细节与不足

- **`unsafe` 块缺乏安全契约**：`SockAddrIn::as_bytes`（`socket.rs:66-69`）和 `from_bytes`（`socket.rs:88-93`）使用 `unsafe` 将自身指针转为字节切片，却未标注 `// SAFETY:`。这类转换虽因 `#[repr(C)]` 而合法，但缺少注释将让后续维护者难以判断内存在传递过程中是否保持了布局不变性<sup>[6](#mod-net-ref-6)</sup>。
- **锁粒度与潜在死锁**：`SOCKET_TABLE` 使用全局 `IrqSpinLock`（`socket.rs:194-196`$），所有 socket 操作（包括 `SocketFile::read`）都竞争同一把锁。`read` 在持有锁期间调用 `buf.push`（`socket.rs:113`），若 `MutableIoBuffer` 背靠用户页面且触发缺页，在 spinlock 内部睡眠将导致死锁<sup>[3](#mod-net-ref-3)</sup>。`IrqSpinLock` 虽然关中断，但缺页异常可能来自其他 CPU。
- **硬编码常量**：VirtIO-Net 驱动的 MTU 固定为 1500（`virtio_net.rs:41`）、RX 缓冲区固定 2048 字节（`virtio_net.rs:15`），未根据设备能力或用户配置动态调整。`SOCK_CLOEXEC`/`SOCK_NONBLOCK` 掩码值（`socket.rs:243-244`）直接写死八进制字面量，跨架构需额外注意 endian 问题。
- **未使用的参数与半成品代码**：`sys_sendto` 中 `flags` 被校验但未用于实际行为，`_dest_addr_ptr` 和 `_addr_len` 完全忽略（`socket.rs:550-567`）；`sys_recvfrom` 同理。`sys_shutdown`（`socket.rs:633-637`）仅设置 `peer_closed` 标志，但 no 通知驱动或协议栈执行 FIN/RST 握手<sup>[7](#mod-net-ref-7)</sup>。
- **状态机不完整**：UDP 的 `Bound` 状态不支持多 peer；TCP 的 `Listening` 状态从未真正监听端口；`accept` 创建的已连接 socket 使用假 peer 地址 `127.0.0.1:0`（`socket.rs:460`），且不执行三次握手。这些 `TODO` 级别的缺陷使得模块无法通过任何网络连通性测试。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块由三层组成：`device.rs` 提供 Linux 风格的总线/设备/驱动注册与自动匹配机制；`block.rs` 定义块设备抽象 `BlockDevice` 并实现基于时钟算法的缓存层；`virtio.rs` 通过 `SudoHal` 实现 `virtio-drivers` 的 `Hal` trait，探测 MMIO 和 PCI 设备并注册到子系统中。与 rCore-Tutorial 的简单设备表相比，本仓库引入了完整的驱动模型、DMA32 专用分配器和时钟算法缓存，但也存在全局锁粒度粗、panic 路径多等早期实现问题。

### 1. 核心抽象：三层次的驱动框架

`device.rs` 定义了 `Device`、`Driver` trait 和 `Bus` trait，并维护全局注册表 `DEVICES`、`DRIVERS`、`BUSES`<sup>[1](#mod-drivers-ref-1)</sup>。`Device` 持有名称、类型、MMIO/IRQ 资源和类型擦除的私有数据；`Driver::probe()` 负责初始化硬件；`Bus::match_driver()` 提供灵活的设备—驱动匹配逻辑。注册新设备或驱动后会立即遍历已存在列表尝试绑定，模拟热插拔语义<sup>[2](#mod-drivers-ref-2)</sup>。

块设备层在 `block.rs` 中通过 `BlockDevice` trait 定义统一接口（`read_block`、`write_block`、`block_size`、`block_count`）。在该 trait 之上，`BufferCache` 使用 8 个槽位的时钟算法缓存，`PageCache` 将页请求拆分为块请求后委托给 `BufferCache`<sup>[3](#mod-drivers-ref-3)</sup>。

VirtIO 驱动在 `virtio.rs` 中通过 `SudoHal` 实现 `virtio_drivers::Hal`，提供 DMA 分配/释放、MMIO 物理到虚拟地址转换、缓冲区共享/取消共享等底层操作<sup>[4](#mod-drivers-ref-4)</sup>。`VirtioBlockDevice` 包装 `VirtIOBlk` 并实现 `BlockDevice`，通过 `block::register_device()` 注册到全局块设备表。值得注意的是，virtio 块设备并未走 `device.rs` 的总线模型，而是直接对接 `block.rs`，两条注册路径并存但未统一<sup>[5](#mod-drivers-ref-5)</sup>。

### 2. 关键设计取舍

**DMA 分配策略**：`SudoHal::dma_alloc` 从 DMA32 区（< 4GB）分配连续零化物理页，记录到全局表 `DMA_ALLOCATIONS`，放弃使用 `virtio-drivers` 的默认 `DmaHal`。自行管理生命周期可以精确跟踪每一块 DMA 缓冲区，便于调试和泄露检查，但代价是 `dma_dealloc` 需要线性搜索全局表，性能随分配次数线性下降<sup>[6](#mod-drivers-ref-6)</sup>。

**缓存替换算法**：`BufferCache` 使用简单的时钟算法——维护单调递增的 `tick`，驱逐时选择 `(valid, age)` 最小的条目。相比 LRU 实现更简单且无需链表，但年龄通过 `wrapping_add` 处理可能翻转，且驱逐逻辑仅依赖锁内状态无原子性保护<sup>[7](#mod-drivers-ref-7)</sup>。

**锁粒度**：框架大量使用 `IrqSpinLock`（禁用中断的自旋锁）。`DMA_ALLOCATIONS`、`BLOCK_DEVICES`、`DEVICES` 等都是全局锁，持锁期间可能调用 `page_alloc::allocate`（会获取页分配器内部的锁），违背锁顺序原则，在极端竞争下有死锁风险<sup>[8](#mod-drivers-ref-8)</sup>。

### 3. 跨模块协同：从硬件探测到 I/O 路径

`virtio::initialize()` 接收 FDT 解析出的 `MmioRegions` 和 `PciHostBridges`，遍历每个 MMIO 区域或 PCI 设备，根据设备类型分发：
- 块设备：创建 `VirtioBlockDevice`，调用 `block::register_device()` 写入全局块设备表；为 `vda` 自动注册 `/dev/vda2` 兼容别名<sup>[9](#mod-drivers-ref-9)</sup><sup>[10](#mod-drivers-ref-10)</sup>。
- 网络设备：通过 `net::register_interface()` 注册到网络栈。
- 熵源：包装为闭包注册到 `crate::rng` 子系统。

I/O 路径上，`VirtioBlockDevice::read_block` 分配临时 DMA 缓冲区，调用 `VirtIOBlk` 执行传输，再释放 DMA 页。`BufferCache` 在此基础上提供块缓存，`PageCache` 进一步向文件系统屏蔽块大小差异。

### 4. 边角细节与不足

- **panic 风险**：`SudoHal::share` 中若 `virtual_to_physical` 返回 `None` 直接 panic，用户态传入无效缓冲区指针可触发内核崩溃<sup>[11](#mod-drivers-ref-11)</sup>。
- **竞态条件**：`device::register_device` 在释放全局锁后调用 `bus.on_device_added`，此时设备可能已被其他线程移除<sup>[12](#mod-drivers-ref-12)</sup>。
- **PCI ECAM 映射泄漏**：`probe_pci_host` 通过 `mmio_virtual_address` 映射 ECAM 窗口，但未保存映射以便后续释放<sup>[13](#mod-drivers-ref-13)</sup>。
- **设备名硬限制**：`validate_device_name` 限制 31 字符且仅允许字母数字、`-`、`_`，Linux 设备名如 `nvme0n1` 无法注册<sup>[14](#mod-drivers-ref-14)</sup>。
- **虚函数空实现**：`Driver::remove` 默认返回 `Ok(())`，总线未实现解绑，热拔插不安全。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层以 `syscall::Request` 和 `syscall::abi` 为核心抽象，将架构相关的寄存器布局统一为与 Linux 兼容的编号和参数数组。它通过 ABI 与策略分离的设计，将解码/编码逻辑集中在 `syscall` 模块，而分发策略保留在 `kernel::user` 模块，避免了架构细节泄漏。与同家族基线相比，本仓库显式区分了 `number` 常量和 `errno` 编码，并提供了 RISC-V 64 和 LoongArch64 双架构的条件编译支持，同时通过 `verify_contract()` 运行时常量验证防范配置漂移。

### 1. 核心抽象：从架构寄存器到策略无关的 Request

`syscall::Request` 是一个紧凑的结构体，包含 `number`（usize）和 `arguments`（[usize; 6]），抹平了所有架构差异。`syscall::abi` 模块提供三个核心函数：`decode` 从架构特定的 `TrapFrame` 中提取 `Request`，`advance` 将 PC 移至下一条指令（避免重复执行 ecall），`set_result` 将返回值写回陷阱帧。每个函数都通过 `#[cfg(target_arch = ...)]` 实现多架构支持，当前仅覆盖 RISC-V 64 和 LoongArch64 <sup>[1](#mod-syscall-ref-1)</sup>。

`syscall::number` 模块定义了 100 多个与 Linux asm-generic 兼容的系统调用号，涵盖文件 I/O（如 `OPENAT`、`READ`、`WRITE`）、进程控制（`CLONE`、`EXECVE`、`EXIT`）、信号（`RT_SIGACTION`、`RT_SIGRETURN`）、内存管理（`MMAP`、`MUNMAP`）以及网络 socket（`SOCKET`、`CONNECT`）等类别。`syscall::errno` 模块定义常见错误码并提供了 `encode`（取负）和 `is_error`（范围检查）辅助函数，与 Linux 内核 `-errno` 约定一致 <sup>[2](#mod-syscall-ref-2)</sup>。`SYSCALL_INSTRUCTION_BYTES` 硬编码为 4，符合 RISC-V 和 LoongArch64 的 ecall/syscall 指令长度，若移植到变长指令架构（如 ARM Thumb）则需调整。

### 2. 关键设计取舍

**ABI 与策略分离**：模块注释明确“本模块只拥有 ABI 事实，策略在分发器中” <sup>[3](#mod-syscall-ref-3)</sup>。与 Linux 内核的 `sys_call_table` 全局数组相比，此处的 `number` 常量是静态的，分发逻辑完全由 `kernel::user` 模块的 match 语句或表驱动实现。这种分离避免了在添加系统调用时意外修改寄存器编解码逻辑，但代价是增加了一次间接调用开销。如果未来需要动态注册系统调用（如内核模块），当前设计需要扩展。

**运行时常量验证**：`verify_contract()` 通过 `assert_eq!` 检查所有常量的正确性 <sup>[4](#mod-syscall-ref-4)</sup>。这是一种防御性编程，确保编译时优化的常量替换不会引入错误。但缺点有三：一是启动时间增加，在嵌入式环境可能触发 panic；二是每个常量值被硬编码两次（一次在 `number` 模块，一次在验证函数），存在同步风险；三是只验证了常量的子集（约 40 个），遗漏了 `io_uring`、`futex` 等调用。

**错误编码为负值**：`errno::encode` 返回负值，`is_error` 通过范围 `[-4095, -1]` 判断。这与 Linux 的 `-errno` 指针技巧一致，但没有使用 `ERR_PTR` 宏，简化了实现。但 `is_error` 对合法负值返回值的误判风险 <sup>[5](#mod-syscall-ref-5)</sup> 在用户态可控输入中可能导致拒绝服务（例如返回负文件描述符被误认为错误），尽管 Linux 约定合法返回值不在此范围，但模块未显式声明此约定。

**固定参数数量**：`Request` 使用 6 个参数，这是 Linux 系统调用 ABI 的最大值，足以覆盖 `mmap`（6 参数）等调用。更复杂的功能如 `io_uring` 通过系统调用链实现，不需要扩展参数结构。

### 3. 跨模块协同

`kernel/src/trap.rs` 中的 `kernel_arch_trap` 函数是异常处理的入口。当陷阱原因为 `USER_ECALL`（RISC-V）或 `ECODE_SYSCALL`（LoongArch）且先前模式为用户态时，直接调用 `crate::user::handle_syscall(frame)` <sup>[6](#mod-syscall-ref-6)</sup>。此函数首先调用 `syscall::abi::decode` 提取 `Request`，然后根据 `number` 分发到具体的处理逻辑，最后通过 `set_result` 写回结果。由于 `syscall::Request` 是 `Copy` 类型，解码无需堆分配，路径上的开销仅为寄存器复制。
