## 📖 一、总览

PulseOS 是一个基于 ArceOS 内核框架构建的实验性操作系统，旨在提供 Linux 兼容的 ABI 和功能集。与同家族的教学型或演示型实现不同，PulseOS 在保留 ArceOS 微内核架构的同时，显著扩展了进程模型、文件系统、信号和 IPC 等子系统，试图接近完整 Linux 内核的功能。其整体定位是**探索型兼容内核**——在 Rust 生态中验证复杂 POSIX 语义的实现可行性。

最值得评审注意的两个特点：**1）任务模型通过 `task_ext` 指针将 `Thread` 嵌入 `axtask::TaskInner`，并放弃 per-CPU 变量改用全局注册表**（[task.ref:1][task.ref:2]），这一设计在保持调度器轻量的同时支持跨 CPU 信号投递；**2）文件系统采用 `ext4plus` 并强制 `#![forbid(unsafe_code)]**（[fs.ref:1]），在 Rust 中实现了完整的 ext4 读写，包括 extent tree、HTree 目录索引等，但其 journal/recovery 检查被注释掉（[fs.ref:2]），反映出安全性与完整性之间的权衡。此外，信号机制实现了三层数据结构（`SignalShared`/`ThreadSignal`/派送上下文）和精确的 `siginfo_t` 传递，但核心的 core dump 生成仍为骨架（[signal.ref:5]）。

## 🎯 二、综合评价

### 整体定位
PulseOS 在 arceos-starry 家族中属于**实验型完整 Linux 兼容内核**。它从 ArceOS 的模块化架构出发，但远超基础教学范围：实现了多进程、写时复制、System V IPC、ext4 持久化文件系统、完整信号语义和网络协议栈。启动流程直接内联 init 闭包（[boot.ref:3]）而非独立 init 程序，这降低了初始复杂度，但也剥夺了用户态启动灵活性——这是教学痕迹与实用目标的折中。整体上，PulseOS 不是为了演示某一特性，而是试图用 Rust 构建一个可运行 shell 和基本 POSIX 应用的平台，但诸多未完成项（如信号核心转储、RPL 安全、Sv32 页表支持）表明它仍是**演进的实验品**。

### 真正的创新点
PulseOS 有几个不显眼但关键的独特设计：

**1. task_ext 桥接与全局线程注册表**（[task.ref:1][task.ref:2]）：与 ArceOS 原版通过 `CURRENT_THREAD` per-CPU 变量直接获取当前线程不同，PulseOS 将 `ThreadHandle` 指针存入 `TaskInner` 的扩展域，并通过全局 `THREAD_REGISTRY` 哈希表管理生命周期。这意味着每次获取当前线程都需要从调度器数据中解析任务再解引用——看似多了一次间接，但**消除了跨 CPU 迁移时 per-CPU 变量失效的问题**，使信号投递能安全地在不同核心上唤醒线程。

**2. 信号的三层数据结构和 CAS 无锁出队**（[signal.ref:1][signal.ref:5]）：进程级共享层 (`SignalShared`) 使用 `SpinNoIrq<[SigAction; 65]>` 而非哈希表存储动作表，既保证性能又避免中断死锁；线程级私有层 (`ThreadSignal`) 用 `AtomicU64` 位掩码管理未决/阻塞信号；派送路径中采用 CAS 循环实现无锁出队（`dequeue_process_unblocked`），但无退避机制带来活锁风险。

**3. ext4plus 的安全优先策略**（[fs.ref:1]）：在文件系统实现中强制 `#![forbid(unsafe_code)]`，并启用 `clippy::arithmetic_side_effects` 等 lint。这在 Rust 系 ext4 实现中罕见，但其代价是注释掉了 journal 恢复检查（[fs.ref:2]），意味着写入时可能忽略不可恢复错误。

**4. 共享内存的连续物理页映射**（[ipc.ref:5]）：System V 共享内存通过 `virt_to_phys` + `map_linear` 分配连续物理页并线性映射到用户空间，大幅简化了实现，但放弃了离散映射的灵活性，且 `ShmInner` 的 `Send/Sync` 实现缺少 `// SAFETY` 注释（[ipc.ref:2]）。

### 取舍判断
PulseOS 明显将精力集中在**核心语言生态和常用 POSIX 语义**上，而选择性地简化或省略非关键路径：

- **启动简化**：内联 init 闭包（[boot.ref:3]）省去 init 进程编排，但硬编码栈大小 0x8000（[boot.ref:5]），且 `expect` 在无线程上下文时直接 panic（[boot.ref:4]）。
- **调度与信号**：放弃 per-CPU 变量换取跨 CPU 安全（[task.ref:2]），但锁粒度较粗（`SpinNoIrq` 保护线程状态表，[task.ref:4]），潜在死锁风险。
- **内存管理**：增加 COW 和 `Backend` 抽象（[mm.ref:2][mm.ref:4]），但 `copy_cow_range_recursive` 只支持 3/4 级页表（[mm.ref:2]），Sv32 直接 panic。brk 使用预分配 chunk 减少 syscall 次数（[mm.ref:3]）。
- **资源限制**：大部分设为 `u64::MAX`（仅栈和 fd 数受限）（[syscall.ref:5]），表明作者将正确性置于资源管控之前。
- **网络**：依赖 smoltcp 并可选拥塞控制（Reno/Cubic）（[net.ref:5]），但 RPL 安全消息为 `todo!()`（[net.ref:4]），且 TCP socket 缓冲区满时直接 panic（[net.ref:1]）。

**被刻意省略的领域**包括：无完整 audit/namespaces/cgroups、无 eBPF、无用户态驱动、无图形子系统。这些都是向完整 Linux 兼容前进时的自然取舍。

### 完成度与不足
从模块分析看，PulseOS 完成度较高：188 个系统调用覆盖了文件 I/O、进程控制、信号、网络、IPC 等主要类别，核心功能（fork/execve/futex/信号派送/ext4 读写）可工作。但各模块都存在明显的**技术债务**：

- **启动**：硬编码栈大小、未撤销钩子、阻塞式 `join` 无法响应信号（[boot.ref:5][boot.ref:4]）。
- **内存管理**：`bits64.rs` 缺少 SAFETY 注释、Sv32 支持缺失、`sys_brk` 回滚不完整（[mm.ref:5]）。
- **任务**：实时调度参数仅存储不生效（[task.ref:9]）、`unsafe` 块无安全契约（[task.ref:1]）。
- **文件系统**：ext4 journal 检查注释掉（[fs.ref:2]）、extent 初始化跳过预留块（[fs.ref:3]）、挂载源设备启发式猜测（[fs.ref:5]）。
- **信号**：`ucontext_t` 偏移硬编码（[signal.ref:5]）、SIGCHLD 通知未实现、core dump 无实际写入。
- **IPC**：`SemManager` 惊群效应（[ipc.ref:4]）、`BiBTreeMap` 死代码（[ipc.ref:3]）、权限检查每次克隆 `groups` 向量。
- **网络**：RPL 安全 `todo!()`（[net.ref:4]）、DHCPv4 缺少强制初始化（[net.ref:2]）、TCP 缓冲区 panic（[net.ref:1]）。
- **驱动**：`write_desc` unsafe 块无注释（[drivers.ref:3]）、`VirtIOBlk` 队列大小硬编码（[drivers.ref:1]）、`add_direct` 空 buffer panic。
- **系统调用**：`query_user_page_slice` 无 SAFETY 注释（[syscall.ref:3]）、`sys_syslog` 返回静态字符串（[syscall.ref:11]）、`sys_getrandom` 依赖设备文件（[syscall.ref:12]）。

这些不足不致命，但表明 PulseOS 尚未达到 alpha 质量，更适合作为**研究原型**或**教育平台**。

### 1. 核心抽象与外部依赖

启动流程围绕 `src/main.rs:14` 的 `main` 函数展开，该函数是 `#![no_std]` 和 `#![no_main]` 环境下的唯一入口。它依赖五个关键外部 crate：`axhal`（硬件抽象层）、`axruntime`（运行时初始化）、`pulse_core`（进程/线程/信号核心）、`pulse_syscalls`（系统调用实现）和 `starry_vdso`（VDSO 支持）。<sup>[1](#mod-boot-ref-1)</sup> 启动序列分为三个阶段：(1) 基础设施初始化——VDSO 数据准备、itimer/procfs/TTY 钩子注册、trap 初始化 <sup>[2](#mod-boot-ref-2)</sup>；(2) 内核线程与用户进程构造——使用 `axtask::TaskInner` 创建一个名为 `pulse_init` 的内核线程，其闭包负责加载 `/bin/sh` 并切换到用户态 <sup>[3](#mod-boot-ref-3)</sup>；(3) 等待 init 完成并关机——通过 `join()` 阻塞直到 shell 退出，然后同步文件系统并调用 `axhal::power::system_off()` <sup>[7](#mod-boot-ref-7)</sup>。

### 2. 关键设计取舍

**直接内联 init 逻辑而非用户态 init 程序**：`main` 的闭包直接调用 `proc.load_elf(SHELL_ELF_PATH)`, 替代方案是像 Linux 那样启动一个用户态 init（如 `/sbin/init`），但这里选择在内核线程中完成 ELF 加载和地址空间激活，省去一次 execve 系统调用。<sup>[3](#mod-boot-ref-3)</sup> 代价是这个闭包必须拥有完整的进程/线程上下文，`current_thread().expect(...)` 如果在线程未就绪时调用会直接 panic。<sup>[4](#mod-boot-ref-4)</sup>

**硬编码常量**：`SHELL_ELF_PATH` 固定为 `/bin/sh` <sup>[3](#mod-boot-ref-3)</sup>，`TaskInner::new` 的栈大小硬编码为 `0x8000`（32 KiB）<sup>[5](#mod-boot-ref-5)</sup>。这些值无法通过命令行或设备树覆盖，跨平台时可能需要参数化。

**进程与任务的分步构造**：先创建 `TaskInner`（内核调度实体），再调用 `Process::new_uspace` 创建用户进程，然后通过 `inner.ctx_mut().set_page_table_root(pt_root, asid)` 将页表绑定到任务上下文 <sup>[6](#mod-boot-ref-6)</sup>。这种分离允许在进程创建前配置任务属性（如栈大小、名称），但要求调用方手工保持两者状态一致。

**条件编译影响**：`cfg!(feature = "testcode")` 控制两处行为——禁用 stdin 轮询（行 22）和传递不同的 shell 参数（行 44-48）。这意味着测试模式下的启动路径与非测试模式不同，可能隐藏与 stdin 相关的竞态问题。<sup>[2](#mod-boot-ref-2)</sup>

### 3. 跨模块协同

**VDSO 协同**：`starry_vdso::vdso::init_vdso_data()` 初始化 VDSO 数据，然后 `axruntime::vdso::set_update_hook` 注册回调，使内核在更新 VDSO 时能同步到用户空间。<sup>[1](#mod-boot-ref-1)</sup>

**Trap 初始化与处理**：`pulse_core::trap::init()`（行 32）安装异常向量表，与 `pulse_core/src/trap.rs` 中的 `handle_illegal_instruction`、`handle_address_error`、`handle_page_fault` 等处理函数配合。由于 trap 模块在启动时即初始化，后续 ELF 加载过程中的缺页异常能得到正确路由。

**任务与进程的绑定**：通过 `init_task_ext(ThreadHandle)`（行 80）将用户态线程句柄附加到内核任务，使调度器在切换任务时能同步用户态上下文。`register_task_ref`（行 82）建立进程到任务的引用，确保进程结束时能清理关联的任务。

**错误路径的关机协议**：无论 init 进程创建成功或失败，最终都会调用 `pulse_syscalls::sys_sync()` 和 `axhal::power::system_off()` <sup>[7](#mod-boot-ref-7)</sup> 确保文件系统一致性和安全关机。但注意，如果 `Process::new_uspace` 失败，之前注册的 itimer/procfs 等钩子未被撤销，这可能在后续关机流程中产生副作用。

### 4. 边角细节与不足

- **<sup>[4](#mod-boot-ref-4)</sup> 潜在的 panic**：`current_thread().expect(...)` 假设闭包一定在 `Thread` 上下文中执行。如果 `TaskInner` 被调度时 `ThreadHandle` 尚未关联（例如构造顺序错误），将直接 panic，没有优雅降级路径。

- **<sup>[5](#mod-boot-ref-5)</sup> 硬编码栈大小**：`0x8000` 作为 magic number，未根据不同架构（如 x86_64 vs aarch64）差异化。如果 `/bin/sh` 需要更大栈，可能因栈溢出导致难以调试的崩溃。

- **<sup>[7](#mod-boot-ref-7)</sup> join 结果处理粗糙**：`init_task.join()` 返回 `Option<ExitCode>`，代码仅区分 0/非 0/None，丢失了退出码的具体语义。同时，`join()` 本身是阻塞调用，在这期间如果信号到达或需要处理其他事件，当前模型无法响应。

- **<sup>[6](#mod-boot-ref-6)</sup> 资源泄漏风险**：如果进程创建成功但后续 `axtask::spawn_task` 失败（虽然该函数当前不会失败，但未来实现可能返回 Result），则 `register_thread_global` 和 `register_process` 已注册的条目不会被清理，导致内核状态残留。

- **缺乏初始化时序文档**：代码注释仅包含 info! 日志，没有说明各子系统的初始化依赖顺序（例如 trap 必须在 ELF 加载之前完成），维护者修改顺序时易引入隐式依赖破坏。

总体而言，启动流程采用单体内核风格，将 init 逻辑集中在一个函数中，简洁但牺牲了灵活性和错误隔离。与 ArceOS 基线相比，本仓库最大的修改是将 init 进程创建从单独的服务移入 `main` 闭包，同时引入 `pulse_core` 的进程/线程模型和 VDSO 支持。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块的核心抽象是泛型页表 `PageTable64`，通过 `PagingMetaData`、`GenericPTE`、`PagingHandler` 三个 trait 解耦架构差异。地址类型由 `MemoryAddr` trait 和 `PhysAddr`/`VirtAddr` 提供。在此基础上实现了 Linux 兼容的地址空间管理，包括 brk、mmap、mprotect 等系统调用。与 ArceOS 基线相比，本仓增加了写时复制（`copy_cow_range`）和 `Backend` 映射抽象，以支持更复杂的映射语义（如共享文件、栈增长）。

### 1. 页表核心抽象及其多架构支持
本仓的内存管理建立在 `page_table_multiarch` crate 的 `PageTable64` 结构体上。该结构体通过三个泛型参数将页表操作与架构解耦：`M: PagingMetaData` 提供页表级数和虚拟地址类型；`PTE: GenericPTE` 统一页表项格式；`H: PagingHandler` 负责帧分配和物理-虚拟地址转换。`PageTable64` 提供了 `map`、`unmap`、`query` 等基本操作，并支持大页映射 (`map_region`) 和区域保护 (`protect_region`)。

代码中通过 `p4_index`、`p3_index` 等辅助函数显式假设了 4 级页表结构（x86-64 及 Sv48），而 `M::LEVELS` 若为 3 则跳过顶层。这种设计固然灵活，但在 `copy_cow_range_recursive` 中出现了 `unreachable!()` 分支并明确只处理 LEVELS 为 3 或 4 的情况 <sup>[2](#mod-mm-ref-2)</sup>，意味着如果尝试在只支持 2 级页表（如 RISC-V Sv32）的平台上使用，将直接 panic。这是架构支持上未完成的牺牲。

页表的物理帧管理依赖 `PagingHandler::alloc_frame` 和 `phys_to_virt`，在 `table_of` 和 `table_of_mut` 中使用 unsafe 从物理地址直接构造切片引用，但两处均未提供 `// SAFETY:` 注释 <sup>[6](#mod-mm-ref-6)</sup>，这违反了 Rust 的安全惯例，是潜在维护负担。

### 2. 用户态地址空间的系统调用实现
`pulse_syscalls/src/impls/mm.rs` 实现了 Linux 风格的内存管理 syscall，核心是 brk、mmap、mprotect、munmap、mlock 等。每个 syscall 首先获取当前进程对象，通过 `proc.aspace_handle()` 获取地址空间锁，并调用相应的 `aspace` 操作方法。

`sys_brk` 的实现体现了优化设计：当堆需要扩展时，以 256KB 的块 (`const BRK_CHUNK_SIZE: usize = 256 * 1024;`) 整体预映射 <sup>[3](#mod-mm-ref-3)</sup>，而非按页细粒度映射，从而减少系统调用次数和页表分裂。收缩时同样以块为单位整块释放。这种预分配策略在吞吐量和内存使用之间做了折中，但如果程序频繁小尺度调整堆，可能导致内部碎片。此外，brk 的全部操作都在 `proc.brk_lock` 和 `aspace` 写锁的保护下完成，与 mmap 等操作不会产生竞争，但锁持有的临界区较长（包含 `maybe_lock_future_range` 调用），可能在高并发下带来压力。

`sys_mmap` 更全面地覆盖了 Linux flag 的语义，包括 MAP_FIXED、MAP_ANONYMOUS、MAP_SHARED、MAP_PRIVATE、MAP_GROWSDOWN 等。在映射后端上，代码引入了 `axmm::Backend` 抽象 <sup>[4](#mod-mm-ref-4)</sup>，将文件映射、匿名映射、共享映射等区分对待。特别地，`MAP_GROWSDOWN` 使用了 `Backend::new_alloc_grows_down`，该 backend 尚未集成于主线 ArceOS，可能是本仓的独有扩展。但 `Backend::new_alloc_grows_down` 的具体实现是否存在以及能否正确支持栈自动增长，需要进一步检查（未在本次片段中看到对应定义）。

### 3. 关键设计取舍：写时复制与映射抽象
本仓区别于标准 ArceOS 的一个显著特征是实现了写时复制 (COW) fork 支持。`PageTable64::copy_cow_range` 方法 <sup>[2](#mod-mm-ref-2)</sup> 递归地在两个页表间复制指定范围的映射，同时通过 `is_cow` 标志降低源页表项的写权限，并通过引用计数 `inc_ref` 记录共享物理帧。这避免了在 fork 时立即拷贝物理内存，仅在实际写入时触发缺页处理，是 Linux 经典 COW 机制的复现。

另一个重要抽象是 `Backend`，它封装了页的后备存储。匿名映射对应 `Backend` 的 `Alloc` 变体，文件映射对应 `Backend` 的 `File` 变体，共享匿名映射使用 `Backend::new_shared`。这种抽象使得地址空间层无需关心具体页的分配和释放策略，只需调用 `map_with_backend`。它允许在同一个地址空间中混用不同后端，并且后端可以在未来支持诸如设备内存、持久内存等。

### 4. 边角细节与不足
**SAFETY 注释缺失**：`table_of` 和 `table_of_mut` 使用 unsafe 直接从物理地址创建引用，但未写明调用者必须保证物理地址对应的映射有效、生命周期合理等条件 <sup>[6](#mod-mm-ref-6)</sup>。在涉及多处理器或动态映射解除时，这种假设可能被违反，导致安全漏洞。
**技术债务标记**：`memory_addr` crate 的 `offset` 方法内包含 `// todo: use strict_add_signed when it's stable.` <sup>[5](#mod-mm-ref-5)</sup>，表明开发者明知有更合适的 API 但尚未更新，属于中等程度的技术债。
**架构假设硬编码**：`copy_cow_range_recursive` 中通过 `unreachable!()` 排除了 LEVELS 不等于 3 或 4 的情况 <sup>[2](#mod-mm-ref-2)</sup>，这意味着 RISC-V Sv32 (2级页表) 等平台无法使用此功能，属于已知的早期实现限制。
**错误恢复粗糙**：`sys_brk` 在调用 `maybe_lock_future_range` 失败后，尝试回滚 unmap，如果该回滚也失败，只会记录警告 <sup>[7](#mod-mm-ref-7)</sup> 而不会返回错误或重试，这可能导致地址空间残留未完全映射的区域，使进程进入不一致状态。
**功能未完成**：`sys_mmap` 中对 `MAP_GROWSDOWN` 的处理使用了 `Backend::new_alloc_grows_down`，但该 backends 的定义在现有片段中不可见，且缺乏缺页处理与之配合，栈自动增长的具体机制尚存疑。必须结合 trap handler 的缺页异常处理才能生效，本模块未显示此联动逻辑。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以 `Process` 和 `Thread` 为核心抽象，通过 `ThreadHandle` 嵌入 `axtask::TaskInner` 的 `task_ext` 指针，实现了与底层调度器（axtask）的解耦。它解决了在 ArceOS 微内核上提供 Linux 兼容多进程模型的需求，包括多线程、信号、POSIX 定时器、资源限制和凭证管理。与同家族基线相比，本仓库将 per-CPU 的线程查找替换为全局哈希表（PROCESS_REGISTRY/THREAD_REGISTRY），以支持跨 CPU 信号投递，并新增了完整的 itimer 和 posix timer 实现。

### 1. 核心抽象：Process 与 Thread 的分离及 task_ext 桥接

本模块将 Linux 的进程/线程模型拆为两层：`Process` 持有共享资源（地址空间、文件描述符表、信号共享状态），`Thread` 代表可调度实体。但底层调度器 `axtask` 只认识 `TaskInner`，并不知晓 `Thread`。桥接的关键是 `ThreadHandle`——一个内嵌在 `TaskInner` 扩展域（`task_ext_ptr`）中的指针<sup>[1](#mod-task-ref-1)</sup>。`thread_handle_from_task` 通过 `unsafe` 将 `*const c_void` 转换为 `&ThreadHandle`，从而在任意上下文（包括中断）中恢复出 `Thread` 引用。

对比 ArceOS 原版设计，这里放弃 per-CPU 的 `CURRENT_THREAD` 变量<sup>[2](#mod-task-ref-2)</sup>，改为每次调用 `axtask::current() + task_ext_ptr` 解析当前线程。这意味着每次 `current_thread()` 都要从当前 CPU 的调度器数据中获取任务再解引用，虽然增加了一次指针间接，但消除了跨 CPU 迁移时 per-CPU 变量失效的问题——信号投递可能在不同 CPU 上唤醒线程。

`ThreadState` 枚举 (`Pending` / `Active(AxTaskRef)`) 反映了线程从创建到绑定调度器的两阶段状态<sup>[4](#mod-task-ref-4)</sup>。创建时先插入 `Pending`，待底层 `AxTaskRef` 就绪后调用 `register_task_ref` 升级为 `Active`<sup>[6](#mod-task-ref-6)</sup>。这种设计允许系统在调度器准备好之前安全地持有线程结构。

### 2. 全局注册表的设计取舍

`PROCESS_REGISTRY` 和 `THREAD_REGISTRY` 均为 `SpinNoIrq<HashMap>`，加锁时关闭中断<sup>[2](#mod-task-ref-2)</sup>。这是一个清晰的取舍：用全局锁换取 pid/tid 跨 CPU 查询的简单性。代价是每次 `process_by_pid` 或 `thread_by_tid_global` 都会中断当前 CPU 的抢占，在频繁 kill/pidfd_open 的场景下可能成为性能瓶颈。

`THREAD_REGISTRY` 存储 `Weak<Thread>` 而非 `Arc`<sup>[3](#mod-task-ref-3)</sup>，这是合理的：线程退出时自动清理注册表条目，避免内存泄漏。但这也意味着调用 `thread_by_tid_global` 后必须立即 `upgrade()`，否则 `Weak` 可能因线程退出而失效，增加上层错误处理负担。

`PROCESS_REGISTRY` 则存储 `Arc<Process>`，因为进程生命周期通常更长，且退出后可能还有子进程或 procfs 查询引用它。`INIT_PROCESS` 作为 `Once` 只初始化一次，用于引导阶段的特殊处理。

### 3. 调度系统调用的桥接与模拟局限

`pulse_syscalls/src/impls/task/schedule.rs` 将 Linux 调度 API 翻译为 `axtask` 调用。核心矛盾在于：Linux 的调度策略（SCHED_FIFO/SCHED_RR/SCHED_DEADLINE）和优先级（1-99）是强实时语义，而 `axtask` 的 `set_priority` 只接受一个 `isize` 且不保证对应任何实时策略。

妥协体现为：`sys_sched_setattr` 完整解析 `SchedAttr` 并验证参数（包括 SCHED_DEADLINE 的运行时间/截止期/周期约束<sup>[9](#mod-task-ref-9)</sup>），但最终只调用 `axtask::set_priority(priority as isize)` 或将非实时策略映射为 `-100`<sup>[10](#mod-task-ref-10)</sup>。所有调度参数（`sched_flags`、`sched_runtime`、`sched_deadline`、`sched_period`）仅存储在 `Thread` 的原子字段中供 `sched_getattr` 读取，底层调度器完全不感知。这意味着用户程序通过 `sched_setattr` 设置的 Deadline 调度参数实际上**不会**改变调度行为——这是一个语义黑洞，但对尚未真正需要实时调度的应用场景是可接受的早期实现。

`sched_setaffinity` 的实现有趣地调用了 `axtask::set_current_affinity`，说明底层调度器确实支持 CPU 亲和性<sup>[11](#mod-task-ref-11)</sup>。但 `check_pid` 只允许操作当前线程（pid != 0 && pid != tid 返回 EPERM），限制了 `sched_setaffinity` 对其他线程的影响。

### 4. 跨模块协同：定时器、信号与文件系统

**定时器与 tick 钩子**：`init_itimer_hook` 将 `itimer_tick_hook` 注册到 `axtask` 的定时器钩子链中<sup>[14](#mod-task-ref-14)</sup>。每 tick，该钩子计算时间差并检查当前进程的 ITIMER_VIRT 和 ITIMER_PROF 是否到期<sup>[17](#mod-task-ref-17)</sup>。ITIMER_REAL 则通过 `axtask::set_generic_timer` 调度一个一次性回调<sup>[15](#mod-task-ref-15)</sup>，到期后发送 SIGALRM 并可选重设。POSIX 定时器同样复用 `set_generic_timer`，但支持绝对时间调整（`adjust_absolute_timers`）和 overrun 计数<sup>[16](#mod-task-ref-16)</sup>。

**信号投递路径**：`queue_signal_to_process` 和 `queue_signal_to_thread` 在 `signal.rs`（未完整读取）中实现，但 `mod.rs` 导出了它们。`sys_kill`/`sys_tkill`/`sys_tgkill` 借助 `can_signal` 进行权限检查后调用这些函数<sup>[12](#mod-task-ref-12)</sup>。信号投递与线程调度配合的关键是：目标线程可能在休眠，因此 `begin_group_exit` 会醒来所有线程并通知信号等待队列<sup>[13](#mod-task-ref-13)</sup>。

**文件系统集成**：`ProcfsProcessProvider` 实现了 `axfs::ProcfsProcessProvider` trait<sup>[17](#mod-task-ref-17)</sup>，使得 `/proc` 文件系统可以查询进程状态（status、stat、maps、pagemap 等）。这需要进程模块提供 `processes_snapshot` 和 `process_by_pid` 等查询接口，而注册表设计正好满足。

**地址空间协同**：`Process.aspace` 是 `Arc<RwLock<AddrSpace>>`，由 `axmm` crate 管理。`exec` 时通过 `load_user_app` 重置整个地址空间<sup>[12](#mod-task-ref-12)</sup>，`mmap`/`munmap` 等操作通过 `aspace_handle` 读写。僵尸进程的地址空间被释放给 `ZOMBIE_ASPACE_HANDLE` 共享，避免内存泄漏<sup>[8](#mod-task-ref-8)</sup>。

### 5. 边角细节与不足

- **<sup>[10](#mod-task-ref-10)</sup> 负优先级魔法**：`sys_sched_setparam` 中 `sched_priority > 0` 调用 `axtask::set_priority`，否则设为 `-100`。`axtask` 的优先级范围未文档化，`-100` 可能表示“非实时”或“最低”，但这个魔数缺乏硬约束，若 `axtask` 改变实现可能静默失效。
- **<sup>[9](#mod-task-ref-9)</sup> Deadline 调度形同虚设**：`sys_sched_setattr` 对 `SCHED_DEADLINE` 做了严格的参数校验（runtime ≤ deadline ≤ period），但校验后只存储不生效。用户程序会误以为 deadline 调度已激活，可能造成实时任务时序违反。
- **<sup>[4](#mod-task-ref-4)</sup> `SpinNoIrq` 保护进程线程表**：`self.threads` 和 `self.children` 使用 `SpinNoIrq`，这意味着在持有锁期间发生缺页异常或用户态访问可能死锁（因为中断被关了但缺页可能需要 I/O 等待）。当前实现中 `task_ref_by_tid` 等操作都很短，风险可控，但若未来加入长路径操作需要改用 `RwLock`。
- **<sup>[1](#mod-task-ref-1)</sup> `unsafe` 缺少 SAFETY 注释**：`thread_handle_from_task` 中两次 `unsafe` 操作（`task_ext_ptr` 判空后的指针转换）没有 `// SAFETY:` 注释。契约是 `task_ext_ptr` 必须指向 `ThreadHandle` 或为空，但谁负责设置这个指针没有形式化文档。
- **`THREAD_REGISTRY` 的 `Weak` 竞争**：`register_thread_global` 存储 `Weak`，而 `thread_by_tid_global` 返回 `Option<Arc<Thread>>`。在并发退出场景中，调用者获取 `Arc` 后线程可能已被回收，但 Arc 的存在阻止了内存释放——这与 Linux 中 `get_pid_task` 的 `rcu_read_lock` 保护类似，但这里没有 RCU，仅靠 Arc 的临时引用保证对象存活。
- **`exec.rs` 中 `check_txt_busy` 的覆盖率**：该函数遍历所有进程的文件描述符表，检查是否有写打开的文件匹配目标文件的 device/inode。但只检查了 `fd_table` 中的常规文件，未检查内存映射文件（通过 `mmap` MAP_SHARED 写入），可能导致 ETXTBSY 遗漏。
- **`sys_sched_getattr` 的 size 检查**：要求 `size < SCHED_ATTR_SIZE` 时返回 EINVAL<sup>[11](#mod-task-ref-11)</sup>，这与 Linux 行为一致。但 `SCHED_ATTR_SIZE` 是编译时常量，未考虑未来扩展的 `sched_attr` 新字段，可能兼容性隐患。

总之，本模块在 ArceOS 的基础上做了大量 Linux 兼容工作，其抽象层次清晰（Process vs Thread 分离、task_ext 桥接），但调度系统调用的实现仍处于“语义模拟”阶段，实时策略和 deadline 调度尚未真正连接到底层调度器，属于可预期的早期限制。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块提供完整的文件系统支持，核心抽象是 VFS 层（axfs/axfs_ng_vfs）与具体文件系统 ext4plus。它解决了从用户态系统调用到磁盘数据存取的全路径问题，支持 ext4 读写、挂载传播、权限检查等 Linux 语义。与原始的 ArceOS 仅支持 ramfs 相比，本模块引入了真实的块设备文件系统，并实现了大量的 POSIX 系统调用。

### 1. 双层 VFS 架构与核心抽象
文件系统模块采用分层设计：上层是系统调用处理层（`pulse_syscalls/src/impls/fs/path.rs`），中层是虚拟文件系统抽象层（`axfs` / `axfs_ng_vfs`），底层是具体文件系统实现（如 `ext4plus`）。VFS 层提供了 `Path`、`Location`、`Entry`、`Filesystem` 等核心 trait，将路径解析、权限验证、挂载管理封装为统一接口<sup>[1](#mod-fs-ref-1)</sup>。系统调用处理函数通过 `Context` 对象与 VFS 交互，无需关心底层是 ext4 还是 tmpfs。

具体文件系统方面，`ext4plus` crate 是一个完整的 ext4 读写实现，支持 extent tree、HTree 目录索引、元数据校验和等特性。其 `Ext4` 结构体通过 `PtrPrimitive` 实现引用计数的克隆，便于在多线程/协程环境中共享。ext4plus 强调内存安全，强制禁用 `unsafe` 代码<sup>[1](#mod-fs-ref-1)</sup>，并且通过 `#[cfg]` 支持异步/同步两种模式。

### 2. 关键设计取舍
**安全性第一**：`ext4plus` 声明了 `#![forbid(unsafe_code)]`，并启用了 `clippy::arithmetic_side_effects` 等 lint，将所有潜在的算术溢出和未定义行为扼杀在编译期。这虽然牺牲了部分性能（如手工 bitmap 操作），但符合本项目作为系统软件的高可靠性要求。

**异步兼容**：通过 `#[maybe_async]` 宏，`ext4plus` 可以同时编译为异步和同步版本。当前项目实际使用的是同步路径，异步接口更多是为 Tokio 等运行时预留。这种设计增加了维护成本，但在初期保持了灵活性。

**完整的挂载语义**：`path.rs` 中的 `sys_mount_propagation` 实现了 Linux 的传播标志（MS_SHARED/MS_SLAVE/MS_PRIVATE/MS_UNBINDABLE）以及递归版本（MS_REC）<sup>[4](#mod-fs-ref-4)</sup>。此外还支持 `mount --move` 操作。这些功能在同类嵌入式 VFS 中较少见，表明项目追求高 Linux 兼容性。

**精细的只读检测**：在 `sys_openat` 和 `sys_mkdirat` 中，都会对目标路径所在文件系统进行只读检查。如果写操作尝试发生在只读挂载点，会提前返回 `EROFS`，避免数据损坏<sup>[4](#mod-fs-ref-4)</sup>。

### 3. 跨模块协同：从系统调用到磁盘块
以 `sys_openat` 为例展示完整的协同流程：
1. 用户态路径通过 `read_user_cstring` 读入内核<sup>[5](#mod-fs-ref-5)</sup>；
2. 根据路径是否绝对路径，决定使用 `dirfd` 或 `AT_FDCWD` 作为基准；
3. 通过 `context_for_dirfd` 获取当前进程的 VFS 上下文；
4. 进行只读检查、权限验证（O_NOATIME、ETXTBSY 等）<sup>[4](#mod-fs-ref-4)</sup>；
5. 调用 `options.open(&ctx, path)`，触发 VFS 层的路径解析：它逐级查找目录，最终调用底层文件系统的 `lookup` 或 `create`；
6. 若底层是 ext4，则会调用 `Ext4::read_inode`、`extent_tree` 解析文件块，最终通过 `Ext4Read` trait 读取磁盘；
7. 打开成功后，通过 `open_result_to_entry` 将文件/目录包装为 `FdEntry`，注册到当前进程的文件描述表中。

整个流程涉及用户数据拷贝（`read_user_cstring`）、VFS 路径解析、文件系统后端、进程 fd 表，代码分布在多个 crate 中，通过 trait 接口解耦。

### 4. 边角细节与不足
- **extent_tree 初始化 TODO**：`ExtentTree::initialize` 函数中明确标注 `// TODO: linux claims some initial blocks for the extent tree`，表明初始化时没有处理 inode 中预留的 extent 块。这意味着对于使用内联 extent 的早期 ext4 文件系统，读取可能失败<sup>[3](#mod-fs-ref-3)</sup>。
- **注释掉的 journal 检查**：在 `Ext4::load_with_writer` 中，有一段检查 journal、recovery 等特性的代码被完全注释掉<sup>[2](#mod-fs-ref-2)</sup>。这表明写入功能尚未经过完整测试，若在包含 journal 的文件系统上写入，可能导致数据不一致。
- **启发式设备路径猜测**：`mount_source_candidates` 函数会尝试去除设备路径末尾的数字（如 `/dev/sda1` → `/dev/sda`）作为候选。这种启发式可能错误地将分区设备映射为整个磁盘，导致挂载失败或数据损坏<sup>[5](#mod-fs-ref-5)</sup>。
- **线性查找 extent 节点**：`find_child_index` 使用 `O(n)` 线性扫描来定位 extent 内部节点中的子项<sup>[6](#mod-fs-ref-6)</sup>。对于深度较大的 extent 树（如高度 5），性能可能退化，但在典型 ext4 卷中节点数通常很少，此取舍可以接受。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制基于 per-thread AtomicU64 位掩码实现未决/阻塞信号管理，以 SignalShared 跨线程共享信号动作表。核心抽象是 ThreadSignal（线程级信号状态）与 SignalShared（进程级信号表+未决信号），通过 check_signals_and_deliver 在 trap 返回路径中统一派送。与 ArceOS 基线相比，本仓实现了完整的 siginfo_t 传递、SA_RESETHAND/SA_NODEFER 语义、sigaltstack 骨架、以及 riscv64/loongarch64/x86_64 三架构的 ucontext 保存恢复，但缺失 SIGCHLD 通知和 core dump 实际落盘。

### 1. 核心抽象与数据结构分层
信号模块的数据结构分为三层：进程级共享层、线程级私有层、以及派送上下文层。

**进程级共享层**由 `SignalShared` 承担，内部包含一个 `Arc<SignalHandlers>`（信号动作表，64 项 `SigAction`，SpinNoIrq 保护）和一个 `AtomicU64` 进程级未决信号位图，以及 `Mutex<BTreeMap<usize, [u8; 128]>>` 存放进程级 siginfo_t 数据<sup>[1](#mod-signal-ref-1)</sup>。`SignalHandlers` 的 `actions` 字段用 `SpinNoIrq<[SigAction; 65]>` 数组而非 `Mutex<BTreeMap>`，这是出于性能考虑——信号动作表访问频繁且大小固定，数组 + SpinLock 的开销远小于哈希表，且 `SpinNoIrq` 保证在中断关闭下不会死锁。`SignalShared` 提供了 `clone_sighand_only` 和 `clone_actions_only` 两种克隆方式，分别对应 `CLONE_SIGHAND` 和 `CLONE_THREAD` 的语义差异<sup>[2](#mod-signal-ref-2)</sup>。

**线程级私有层**由 `ThreadSignal` 构成，拥有 `thread_pending`（AtomicU64）、`blocked`（AtomicU64）、`in_handler`（AtomicBool）、`skip_once`（AtomicBool）、`saved_ctx`（Mutex<Option<SavedSignalContext>>）、`altstack`（Mutex<SignalAltStack>）等字段<sup>[3](#mod-signal-ref-3)</sup>。`await_queue` 使用 `axtask::WaitQueue` 实现信号等待（被 `sigsuspend` 等系统调用使用），通过 `notify_waiters` 在信号入队时唤醒。每个字段的锁粒度都很细——`pending` 和 `blocked` 用原子操作无锁读写，只有 `saved_ctx` 和 `altstack` 用 Mutex，因为它们在信号处理上下文中访问频率较低。

**派送上下文层**中，`SavedSignalContext` 保存进入信号处理前的完整 `TrapFrame`、旧信号掩码和用户态 ucontext 地址，供 `sigreturn` 恢复使用。`SignalDelivery` 是一个简单的返回值类型，将信号编号与解析后的动作传递给上层调用者（trap handler 或进程管理模块）。

### 2. 信号队列与递送的关键设计取舍
信号队列的核心是一个分层 CAS 循环机制，设计上模仿 Linux 内核的双层 pending 设计。

**入队路径**：`queue_signal_to_process` 和 `queue_signal_to_thread` 是公开入口。它们首先处理 SIGSTOP/SIGCONT 的特殊副作用（设置 `stopped_signal_pending` 等进程级标志并通知父进程的 `child_exit_event`）<sup>[4](#mod-signal-ref-4)</sup>，然后调用 `queue_process_signal` 或 `queue_thread_signal` 执行原子 `fetch_or` 在位图中置位。值得注意的是，即使信号已经 pending，`queue_signal_to_process` 也会强制通知所有线程的等待队列和信号等待标志——注释写的是 "Always notify even if already queued, to ensure blocked tasks re-check signals"，这是一个偏保守的设计，以额外唤醒开销换取正确性。

**出队路径**：`dequeue_unblocked_with_info` 使用 CAS 循环原子取出线程级未决信号中最低位且未被阻塞的信号，如果线程级没有则尝试从进程级取出。CAS 循环的设计避免了加锁，但高并发下可能出现活锁（虽然概率极低）。`dequeue_process_unblocked` 在 `SignalShared` 中实现了一个 `loop { load; compute ready; CAS }` 模式，这是 Rust 中 lock-free 编程的典型写法<sup>[5](#mod-signal-ref-5)</sup>。

**阻塞掩码处理**：`sanitize_mask` 函数强制清除 SIGKILL 和 SIGSTOP 对应的位，确保这两个信号永远不能被阻塞（POSIX 要求）<sup>[6](#mod-signal-ref-6)</sup>。`set_blocked_mask` 调用 `sanitize_mask` 对用户传入的 mask 进行消毒后再写入。这是一个重要的安全边界——如果漏掉这步，用户通过 `rt_sigprocmask` 意外阻塞 SIGKILL 将导致系统无法正常终止进程。

**信号动作解析**：`resolve_action` 将 `SigAction.handler` (usize) 值与 `SIG_DFL=0` 和 `SIG_IGN=1` 做比较来确定是默认/忽略/用户处理。这种用特殊地址值表示默认/忽略的做法是 Linux 内核的惯例（SIG_DFL=0, SIG_IGN=1），但存在歧义风险——如果用户故意注册 handler 为 0 或 1，会被错误解释。不过实践中用户空间不会把代码映射到 0 或 1 地址，且 `sigaction` 系统调用实现中应当验证。

### 3. 信号递送路径与跨模块协同
信号递送的主入口是 `check_signals_and_deliver`，它被 trap handler 在每次 syscall/异常返回用户态前调用（通过 `pulse_syscalls` 中的系统调用处理逻辑触发）。

**递送流程**：
1. `clear_skip_once()`：如果 `skip_once` 标志被设置（由 `restore_from_sigreturn` 置位），跳过本次递送。这个机制防止信号处理程序刚返回就再次被同一个信号中断。
2. `has_pending_unblocked()`：检查线程级和进程级的 pending 位图与 blocked 掩码的交集。
3. `dequeue_unblocked_with_info()`：原子取出一个未决信号及其关联的 siginfo_t 数据。
4. `resolve_action()`：查表确定动作类型。
5. 根据动作类型分支：
   - Ignore/Default：恢复 sigsuspend 掩码，返回 `SignalDelivery` 让上层处理（Terminate/CoreDump 由进程管理器执行）。
   - Handler：这是最复杂的路径。首先计算新掩码（旧掩码 | act.mask | 当前信号位，除非设置了 SA_NODEFER）<sup>[7](#mod-signal-ref-7)</sup>，然后调用 `write_user_signal_frame` 在用户栈上写入 siginfo_t 和 ucontext_t 结构体（大小分别为 128 和 1024 字节，硬编码常量），保存当前 TrapFrame 和 ucontext 地址，设置参数寄存器（a0=sig, a1=si_addr, a2=uc_addr），设置 `ra` 为信号跳板地址（`thread.process().signal_trampoline()`），`pc` 为用户信号处理函数地址。

**跨模块协同点**：
- `write_user_signal_frame` 使用 `process.write_user_bytes` 写入用户空间内存，这依赖于 `Process` 的页表/内存管理模块<sup>[8](#mod-signal-ref-8)</sup>。
- `signal_trampoline()` 来自 `Process` 结构，信号跳板由进程初始化时设置（通常在 mmap 固定地址）。
- 调用 `thread.signal_wait_queue().notify_all(true)` 和 `thread.notify_signal_pending()` 唤醒可能因 `sigsuspend`/`sigwaitinfo` 阻塞的线程。
- `restore_from_sigreturn` 在 `rt_sigreturn` 系统调用处理函数中被调用，它从 `saved_ctx` 恢复 TrapFrame、掩码，并从用户态 ucontext 恢复 PC（但会检查 `pc < KERNEL_ASPACE_BASE` 防止内核地址劫持）<sup>[9](#mod-signal-ref-9)</sup>。 这里还有一个有趣的安全检查：`if let Ok(pc) = read_user_signal_pc(...)` 之后，代码检查 `pc < axconfig::plat::KERNEL_ASPACE_BASE`，这是防止用户通过构造恶意 ucontext 跳转到内核空间的缓解措施。

### 4. 架构抽象与条件编译
为了支持多架构，模块使用了大量的 `#[cfg]` 条件编译。以寄存器访问函数为例：`set_ip`、`set_ra`、`set_arg0..2`、`current_ip`、`current_sp`、`set_sp`、`signal_return_value` 都根据 `target_arch` 有不同的实现<sup>[10](#mod-signal-ref-10)</sup>。

其中显式支持了 riscv64 和 loongarch64，而 fallback (`#[cfg(not(any(target_arch = "riscv64", target_arch = "loongarch64")))]`) 假定 x86_64 ABI（访问 `tf.rip`、`tf.rdi`、`tf.rsi`、`tf.rdx`、`tf.rsp`、`tf.rax`）。这意味着在不支持 riscv64/loongarch64 的其他架构上（如 aarch64），编译会通过但行为错误——`TrapFrame` 结构体可能根本没有这些字段。

`write_user_signal_frame` 和 `restore_from_sigreturn` 中有两块巨大的架构特定代码（各约 30 行），分别保存/恢复 32 个通用寄存器到/从 ucontext_t 的 __gregs 段。riscv64 的 gregs 偏移是 176，loongarch64 的 gregs 偏移是 184，这些偏移量都是硬编码常量且没有文档说明来源（应该来自 Linux 内核的 `ucontext_t` 定义）。

### 5. 边角细节与不足

**<sup>[1](#mod-signal-ref-1)</sup> `SpinNoIrq` 的使用场景**：`SignalHandlers.actions` 使用 `SpinNoIrq` 而非 `Mutex`，这是个显式的性能取舍——`SpinNoIrq` 要求在中断关闭上下文中使用，如果调用路径意外开启中断会造成死锁。但信号动作表主要在系统调用处理路径（`rt_sigaction`）和信号递送路径（中断关闭）中访问，所以这个选择是合理的。没有 `// SAFETY` 注释说明调用者必须保持中断关闭。<sup>[1](#mod-signal-ref-1)</sup>

**<sup>[2](#mod-signal-ref-2)</sup> 原子 CAS 循环的活锁风险**：`dequeue_process_unblocked` 和 `dequeue_process_from_mask` 中的 CAS 循环（`loop { load; compute; CAS; if ok break }`）在极端高并发下可能活锁（多个线程竞争同一个 pending 位）。Linux 内核中类似场景使用的是原子 `test_and_set_bit` + 自旋锁的组合。这里选择纯 CAS 是为了避免锁的开销，但缺少退避机制。<sup>[5](#mod-signal-ref-5)</sup>

**<sup>[3](#mod-signal-ref-3)</sup> 硬编码的 ucontext 偏移**：UCONTEXT_SIGMASK_OFFSET=40、UCONTEXT_PC_OFFSET=176、UCONTEXT_FRAME_SIZE=1024 都是硬编码常量，直接假设 Linux 的 ucontext_t 布局。如果将来适配 musl 或其他 libc（布局可能不同），这些值需要重新验证。没有 grep 友好的命名前缀或跨平台抽象。<sup>[8](#mod-signal-ref-8)</sup>

**<sup>[4](#mod-signal-ref-4)</sup> 缺省的 SAFETY 注释**：第 456 行 `unsafe { core::slice::from_raw_parts_mut(gregs.as_mut_ptr() as *mut u8, 256) }` 没有 `// SAFETY:` 注释。虽然这里 `gregs` 是栈上数组且长度正确，安全契约是“用户提供的 ucontext 地址可读且大小至少 256 字节”，但过程通过 `process.read_user_bytes` 返回值检查了可读性，`unsafe` 块实际只负责类型转换，安全边界是清晰的——但仍应写明。<sup>[8](#mod-signal-ref-8)</sup>

**<sup>[5](#mod-signal-ref-5)</sup> fallback 架构假设**：`#[cfg(not(any(target_arch = "riscv64", target_arch = "loongarch64")))]` 分支假定 `TrapFrame` 有 `rip`、`rdi`、`rsi`、`rdx`、`rsp`、`rax` 字段（即 x86_64）。如果编译到 aarch64 或其他架构，这些字段不存在，会导致编译错误或运行时恐慌。<sup>[10](#mod-signal-ref-10)</sup>

**<sup>[6](#mod-signal-ref-6)</sup> 死代码**：`pick_thread_for_process_signal` 标记了 `#[allow(dead_code)]`，说明当前没有被使用。这是为未来实现 `kill` 选择目标线程预留的骨架。同样，`list_threads_for_signal` 也是 `#[allow(dead_code)]`，但它在 `queue_signal_to_process` 中被实际调用。<sup>[4](#mod-signal-ref-4)</sup>

**<sup>[7](#mod-signal-ref-7)</sup> 测试不足**：文件末尾有一个测试 `reset_on_exec_preserves_blocked_mask`，只测试了 `reset_on_exec` 后 blocked mask 保持不变、thread_pending 清零。缺少对核心路径（如 CAS 出队、sanitize_mask、write_user_signal_frame 边界条件）的单元测试。<sup>[11](#mod-signal-ref-11)</sup>

**<sup>[8](#mod-signal-ref-8)</sup> CoreDump 无实现**：`DefaultSignalAction::CoreDump` 变体存在，但注释写着“终止进程并设置 core dump 标志位（不需要实际写文件）”，意味着 core dump 功能被有意降级——只设置标志位，不生成 core 文件。这在嵌入式场景可以接受，但与 POSIX 标准有差距。<sup>[12](#mod-signal-ref-12)</sup>

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了 Linux 兼容的 System V IPC（共享内存、信号量）和 futex 原语。核心抽象是全局的 ShmManager / SemManager 单例，通过 key 标识 IPC 对象，配合标准权限检查模型。与同家族基线相比，独特之处在于：(1) 完整实现 SEM_UNDO、IPC_RMID 延迟删除等语义；(2) 使用 axtask::WaitQueue 实现信号量阻塞；(3) 共享内存通过内核全局分配器分配连续物理页再线性映射到用户空间，简化了实现但放弃了离散映射的灵活性。

### 1. 核心抽象：基于 key 的全局 Manager 与 ABI 兼容结构

System V IPC 的核心机制是通过一个全局键值（key）来标识和访问 IPC 对象。ShmManager 和 SemManager 均采用 BiBTreeMap 维护 key 到内部 ID 的双向映射，同时用 BTreeMap 记录 ID 到 Arc<Mutex<内部状态>> 的映射，使通过 key 查找 ID 以及通过 ID 访问对象都是 O(log n) 操作 <sup>[1](#mod-ipc-ref-1)</sup>。

共享内存段的状态由 ShmInner 表示，包含内核虚拟地址 addr、页数、映射标志以及 Linux ABI 兼容的 shmid_ds 结构。信号量集由 SemSetInner 表示，包含信号量值数组和 axtask::WaitQueue。这些结构均按 Linux C ABI 布局（#[repr(C)]），可以直接通过 write_user_plain / read_user_plain 在用户态与内核间传输。

权限检查函数 ipc_has_permission 实现了标准 Linux 逻辑：超级用户（euid == 0）总是通过；否则按 owner → group → other 三级检查，并考虑了 supplementary groups。共享内存和信号量在系统调用入口处均执行此检查。

### 2. 关键设计取舍

**共享内存物理页的延迟分配与线性映射**。物理页在第一次 shmat 时才分配（而非 shmget 时），alloc_pages 通过全局分配器分配连续 4K 页，然后用 unsafe write_bytes 清零。在 sys_shmat 中，使用 axhal::mem::virt_to_phys 将内核虚拟地址转换为物理地址，再通过 aspace.map_linear 映射到用户空间 <sup>[5](#mod-ipc-ref-5)</sup>。这意味着物理内存必须连续且内核直接映射区域必须存在。这种设计简化了页表管理，但放弃了离散映射的灵活性，对大段共享内存可能面临连续物理内存不足的风险。

**信号量操作的循环等待模型**。semtimedop 采用 loop：持锁时通过 check_ops 模拟操作判断是否可执行；若可执行则 apply_ops 并返回；否则记录等待计数（semncnt/semzcnt），释放锁，在 WaitQueue 上等待 <sup>[4](#mod-ipc-ref-4)</sup>。被唤醒后重试。这种模式简单直接，但有惊群效应（notify_all 唤醒所有等待者但仅一个能成功）和潜在的高重试次数。没有实现优先级继承或公平性保证，极端负载下可能表现不佳。

**IPC_RMID 的延迟删除**。共享内存的 IPC_RMID 仅设置 rmid 标志，实际删除发生在最后一个进程 shmdt 时 <sup>[6](#mod-ipc-ref-6)</sup>。这依赖 Arc 引用计数：shm_inner_arc 在进程的 shared_memory 表中持有引用，当所有进程 detach 且 rmid 为 true 时，Arc 被 drop 触发 ShmInner::Drop 释放物理页。信号量的 IPC_RMID 则立即设置 removed 标志、唤醒所有等待者（返回 EIDRM）并从全局管理器移除。

**Futex 委托给进程对象**。sys_futex 和 sys_futex_waitv 只做参数验证与时间转换，实际逻辑委托给 process 的 futex_wait / futex_wake / futex_requeue / futex_waitv 方法。这意味着 futex 状态由每个进程的地址空间管理，而非全局表，更接近 Linux 基于用户地址的等待队列设计。

### 3. 跨模块协同

**IPC 核心与 syscall 层的分离**。核心数据结构（ShmManager、SemManager、ShmInner、SemSetInner）定义在 pulse_core::ipc 中，系统调用处理逻辑在 pulse_syscalls::impls::ipc 中。syscall 层通过导入 SHM_MANAGER 和 SEM_MANAGER 全局单例访问核心状态。

**进程退出与 SEM_UNDO 回滚**。exit_sem_undos 在进程退出时遍历其 SEM_UNDO 条目，对每个条目对应的信号量值执行反向操作并通知等待者 <sup>[8](#mod-ipc-ref-8)</sup>。该函数与进程的 ipc.sem_undos 列表协同，该列表在 semtimedop 执行 SEM_UNDO 时被更新。

**共享内存与进程地址空间的交互**。shmat 调用 aspace.map_linear 将共享物理页映射到进程虚拟地址空间，并将 Arc<Mutex<ShmInner>> 记录在 proc.ipc.shared_memory 中。shmdt 从注册表中移除记录并调用 aspace.unmap。进程退出时可遍历该表自动 detach 所有段。

**信号量等待队列与调度器的配合**。SemSetInner 的 wait_queue 来自 axtask crate。在 semtimedop 中线程通过 queue.wait() 或 queue.wait_timeout(dur) 阻塞。其他进程的 semop 或 semctl SETVAL/SETALL 成功时调用 notify_all(true) 唤醒所有等待者，参数 true 表示中断等待者（对应 EIDRM 或信号中断）。

### 4. 边角细节与不足

**unsafe impl Send/Sync 缺少安全契约**<sup>[2](#mod-ipc-ref-2)</sup>。ShmInner 包含内核虚拟地址 addr，注释说“因为内核虚拟地址不是 Send+Sync，但包在 Arc<Mutex<>> 中”，然后直接 unsafe impl Send 和 Sync。虽然实际上通过 Mutex 保护是安全的，但缺少 // SAFETY: 注释，是明确的技术债务。

**BiBTreeMap 死代码**<sup>[3](#mod-ipc-ref-3)</sup>。get_by_value 被 #[allow(dead_code)] 标记，表明当前未被使用。该方法是双向映射的反查接口，但实际代码未调用，可能是未来预留或未完成。

**权限检查中 groups 克隆的开销**<sup>[7](#mod-ipc-ref-7)</sup>。每次 shmget/shmat/shmdt/shmctl 等系统调用均执行 `let groups = credentials.groups.clone()`，在持锁期间克隆 supplementary groups 列表。高频调用场景下可能造成不必要的内存分配，可考虑 RCU 或引用计数共享。

**semtimedop 的惊群效应**。notify_all 唤醒所有等待在同一信号量集上的线程，但仅一个能成功，其余重新等待。高竞争下导致 O(n²) 上下文切换。

**futex_waitv 的早期实现限制**<sup>[9](#mod-ipc-ref-9)</sup>。flags 必须为 0，nr_futexes 限制在 128 以内，且有严格对齐要求，可能阻止依赖现代 futex2 特性的应用运行。

**共享内存连续物理页风险**。alloc_pages 分配连续页，长时间运行的系统内存碎片化后大块分配可能失败，对需要大段共享内存的应用（如数据库）构成稳定性隐患。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块以 smoltcp crate 为核心，在无 OS 环境下实现了完整的 TCP/IP 协议栈及部分 IoT 协议（RPL、DHCPv4）。核心抽象是 TCP/UDP socket，通过环形缓冲区和状态机实现有限状态自动机。与原生 ArceOS 相比，本仓库增加了可选的拥塞控制算法（Reno/Cubic）并通过 feature gate 控制，同时扩展了 RPL 支持。系统调用层在 pulse_syscalls 中直接映射 POSIX socket 操作到 smoltcp 接口，形成用户态网络栈。

### 1. 核心抽象与外部依赖
网络模块的核心是 smoltcp crate，一个用 Rust 编写的嵌入式 TCP/IP 栈。它提供了 `socket::tcp::Socket` 结构体来管理 TCP 连接，其状态机遵循 RFC 793，包含 Closed、Listen、Established 等 11 个状态。每个 socket 持有固定大小的接收和发送环形缓冲区（`RingBuffer<u8>`），以及一个用于 TCP 重排序的 `Assembler`。这种设计避免了动态内存分配，但要求用户预先分配缓冲区，适合资源受限环境。`Socket` 还集成了 RTT 估计器（`RttEstimator`）、定时器（`Timer`）和可替换的拥塞控制控制器（`congestion::AnyController`），支持延迟确认（Delayed ACK）和 Nagle 算法。

除了 TCP，smoltcp 还实现了 UDP、ARP、DHCPv4（`wire/dhcpv4.rs`）和 RPL（`wire/rpl.rs`）等协议。DHCPv4 数据包结构遵循 RFC 2131，支持通过 `options()` 迭代器解析 DHCP 选项。RPL 实现了一个基本的 DODAG 信息处理框架，但安全消息处理仍为 `todo!()` <sup>[4](#mod-net-ref-4)</sup>。

系统调用层位于 `pulse_syscalls/src/handler.rs`，将用户空间的 socket 调用（如 `socket`、`bind`、`sendto`）转发到 smoltcp 对应方法。例如，`socket` 系统调用会创建一个 smoltcp 的 TCP 或 UDP socket 实例并注册到网络接口。这种架构使协议栈完全独立于硬件，便于移植。

### 2. 关键设计取舍
- **选择 smoltcp 而非 LwIP**：完全用 Rust 编写，无缝融入 Rust 生态，且天然支持 `#![no_std]`。但功能相比完整 Linux 栈较弱，例如缺乏 IPv6 高级路由支持，RPL 仅为初始实现。
- **可配置拥塞控制**：通过 `set_congestion_control()` 可设置 None、Reno（需 `socket-tcp-reno` 特性）或 Cubic（需 `socket-tcp-cubic` 特性）。Cubic 使用 `f64` 浮点运算，在无 FPU 的 Cortex-M0 上会显著增加开销；Reno 的数学复杂度低但带宽利用率不如 Cubic。默认不启用任何特性，此时实际为无拥塞控制，可能在高丢包网络下造成拥塞崩溃 <sup>[5](#mod-net-ref-5)</sup>。
- **固定缓冲区策略**：`Socket::new()` 要求传入固定大小的缓冲区，并在容量超过 1 GiB 时直接 `panic!`（`rx_capacity > (1 << 30)`）<sup>[1](#mod-net-ref-1)</sup>。这防止了超大缓冲区占用，但也意味着用户不可控的输入（如通过合理构造的 socket 参数）可能导致内核 panic，构成 DoS 风险。
- **默认开启 Nagle 和延迟 ACK**：`nagle` 字段初始化为 `true`，`ack_delay` 默认为 10ms。两者协同减少小包数量，但会增加交互式应用（如 SSH）的延迟。用户可通过 socket 选项手动关闭，但 smoltcp 未提供标准 `TCP_NODELAY` 映射，需直接操作内部字段。
- **RTT 估计器的保守初始值**：`RTTE_INITIAL_RTT` 为 300ms，`RTTE_INITIAL_DEV` 为 100ms，导致首次重传超时（RTO）高达 700ms。这在第一次传输后很快收敛，但早期连接可能因超时等待而变慢。

### 3. 跨模块协同
系统调用处理与 smoltcp 的协作体现在以下几个层面：
- **Socket 生命周期管理**：`pulse_syscalls` 为每个 socket 分配一个文件描述符，并维护一个映射到 smoltcp socket 的集合。`close` 系统调用需从接口中注销 socket，而 `listen` 和 `accept` 需要底层支持多监听队列——smoltcp 本身不提供 backlog，因此可能在上层实现了简易队列。
- **数据路径优化**：`sendto`/`recvfrom` 直接调用 smoltcp socket 的 `send_slice()`/`recv_slice()`，绕过内核拷贝。由于 smoltcp 工作在用户态，用户空间到栈间的数据拷贝可通过共享缓冲区减少。
- **异步化钩子**：tcp.rs 中通过 `#[cfg(feature = "async")]` 集成 `WakerRegistration`，允许 socket 在等待数据时注册 waker，配合异步调度器实现非阻塞系统调用。但 pulse_syscalls 的系统调用默认同步，实际未启用异步特性 <sup>[3](#mod-net-ref-3)</sup>。

DHCPv4 模块通常与网络接口绑定，在启动时由设备模块调用 `dhcpv4::Client` 获取 IP 地址。但提供的代码仅包含数据包格式，未体现客户端逻辑。

### 4. 边角细节与不足
- **Panic 安全**：`Socket::new()` 中的 `panic!`（`rx_capacity > (1 << 30)`）是唯一受用户输入影响的 panic 点。虽然 1 GiB 在实际中不易达到，但恶意应用可通过传递超大缓冲区大小（如通过 `SO_RCVBUF` 选项）触发，应改为返回 `Result` <sup>[1](#mod-net-ref-1)</sup>。
- **安全消息未实现**：RPL 的 `options_mut()` 方法对安全子类型直接 `todo!()`，导致尝试发送安全 RPL 消息时会崩溃。虽然 RPL 安全在实际部署中不常用，但该遗留项影响合规性 <sup>[4](#mod-net-ref-4)</sup>。
- **未初始化字段风险**：DHCPv4 的 `set_sname_and_boot_file_to_zero` 文档声明“必须调用，否则未定义”，但未通过类型系统或断言强制。用户若忘记调用，`sname` 和 `file` 字段将包含栈垃圾，可能泄露信息或导致错误的 DHCP 响应 <sup>[2](#mod-net-ref-2)</sup>。
- **硬编码常量不可调整**：`ACK_DELAY_DEFAULT`（10ms）、`CLOSE_DELAY`（10s）、`RTTE_MIN_RTO`（10ms）等均为常值，无法在运行时透过 socket 选项修改。这对于不同网络场景（如卫星链路需更大 RTO）缺少灵活性。
- **Feature 间依赖不明确**：`CongestionControl::Reno` 和 `Cubic` 仅在对应特性启用时可用，但 `set_congestion_control()` 接受任何变体，在未启用时静默回退到 `None`。用户代码无法在编译期检测到无效配置，可能导致运行时行为与预期不符 <sup>[5](#mod-net-ref-5)</sup>。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块以 `crates/virtio-drivers` 为核心，提供了一套泛型化的 VirtIO 设备驱动框架。核心抽象是 `Hal`（硬件抽象层，封装 DMA 分配与缓存一致性操作）和 `Transport`（传输层抽象，封装设备配置与通知机制）。框架通过 `VirtQueue` 实现了完整的 split virtqueue 管理，支持 legacy/modern 两种 DMA 布局、间接描述符和事件索引通知抑制。与同家族基线（原版 ArceOS）相比，此实现引入了 const generic 参数化队列大小、细粒度 `BufferDirection` 区分、以及更灵活的 `VirtQueueLayout` 设计，大幅减少了运行时检查并提升了缓存一致性控制的精度。

### 1. 核心抽象：Hal 与 Transport 双 trait 设计

驱动框架通过两个泛型 trait 实现了平台无关性。`Hal` 抽象了 DMA 内存分配（`Dma::new`）、物理地址查询、缓存一致性操作（`unshare`）以及忙等待 yield 指令<sup>[1](#mod-drivers-ref-1)</sup>。`Transport` 抽象了 VirtIO 设备的能力协商（`begin_init`/`finish_init`）、队列配置（`queue_set`/`queue_unset`）、通知（`notify`）和中断确认（`ack_interrupt`）<sup>[5](#mod-drivers-ref-5)</sup>。这种设计使得同一个 `VirtQueue` 实现能够无缝运行于 x86_64 的 PCI 传输层或 RISC-V 的 MMIO 传输层上——只需更换具体的 `Transport` 实现即可。

值得注意的是，`Hal::unshare` 被设计为双向操作：`BufferDirection::DriverToDevice` 方向执行缓存刷新（保证设备可见），`DeviceToDriver` 方向执行缓存失效（保证驱动读取设备写入的数据）<sup>[4](#mod-drivers-ref-4)</sup>。这要求 `VirtQueue` 在回收描述符时精确记录每个 buffer 的方向，框架通过 `InputOutputIter` 迭代器来保持方向与 buffer 的对应关系。

### 2. VirtQueue：Split VirtQueue 的完整实现

`VirtQueue<H, SIZE>` 是框架的核心数据结构，它管理三种内存区域：描述符表（`desc`）、可用环（`avail`）和已用环（`used`）<sup>[1](#mod-drivers-ref-1)</sup>。设计上的关键取舍体现在 `VirtQueueLayout` 上：

- **Legacy 布局**（`allocate_legacy`）：将所有区域打包进一个连续 DMA 区域，兼容旧版规范，但无法区分数据方向，导致缓存一致性操作需要保守地双向执行。
- **Modern 布局**（`allocate_flexible`）：将描述符表+驱动区域与设备区域分离成两个独立的 DMA 区域，允许 `Hal` 分别指定 `BufferDirection::DriverToDevice` 和 `BufferDirection::DeviceToDriver`，从而减少不必要的缓存刷新<sup>[2](#mod-drivers-ref-2)</sup>。

框架默认使用 Modern 布局（通过 `Transport::requires_legacy_layout()` 判断），仅当设备必须使用 legacy 接口时才回退。这种选择以略微增加的分配复杂度换取了更好的多核缓存性能。

描述符管理采用自由链表和影子副本（`desc_shadow`）机制：驱动总是先写入影子副本，再通过 `write_desc` 批量同步到设备可见区域<sup>[7](#mod-drivers-ref-7)</sup>。这避免了在每次修改描述符时都需要考虑设备并发读的风险。`const SIZE: usize` 泛型参数在编译期强制队列大小必须为 2 的幂且不超过 `u16::MAX`<sup>[7](#mod-drivers-ref-7)</sup>，排除了大量运行时边界检查。

### 3. 设备驱动实例：同步与异步的组合模式

`VirtIOBlk` 展示了典型的同步驱动写法：`new` 方法中依次调用 `transport.begin_init`、`VirtQueue::new`、`transport.finish_init` 完成设备初始化<sup>[5](#mod-drivers-ref-5)</sup>；读写操作通过 `add_notify_wait_pop` 阻塞等待设备完成。该函数内部循环调用 `H::busy_wait_yield()` 和 `spin_loop()`，适用于中断未启用或轮询模式的场景。

同时，框架也提供了非阻塞接口 `read_blocks_nb`/`complete_read_blocks`，允许调用者自己管理令牌（token）并轮询 `peek_used`<sup>[6](#mod-drivers-ref-6)</sup>。这种双模设计让上层可以根据优先级灵活选择同步等待或异步 IO 多路复用。

对比之下，`VsockConnectionManager` 采用了更复杂的事件驱动模型：其 `poll` 方法接收一个闭包，由 `VirtIOSocket::poll` 在持有事件时回调，闭包内更新连接状态和环形缓冲区<sup>[9](#mod-drivers-ref-9)</sup>。这种回调机制避免了在每个连接上循环遍历，但要求闭包不能持有对外部锁的引用（否则可能死锁），代码中通过将连接列表的可变引用传入闭包来规避此风险。

两个设备驱动的共同特点是它们都在 `Drop` 中调用 `transport.queue_unset(QUEUE)` 来清空队列指针，防止设备在驱动释放 DMA 内存后仍然访问<sup>[5](#mod-drivers-ref-5)</sup>。

### 4. 安全性与边缘细节

框架中有大量 `unsafe` 块，主要集中于：
- 对 `desc`、`avail`、`used` 指针的解引用（这些指针指向映射后的 DMA 物理内存，设备可能并发写入 used ring，但规范保证写入的是已使用元素，框架通过 `last_used_idx` 隔离了读写位置）<sup>[3](#mod-drivers-ref-3)</sup>。
- `recycle_descriptors` 中调用 `H::unshare` 时需要保证 buffer 仍有效且匹配原始方向<sup>[4](#mod-drivers-ref-4)</sup>。

`unsafe impl Send/Sync for VirtQueue` 基于的假设是：`VirtQueue` 的所有内部指针都指向 DMA 内存，且访问模式满足线程安全（对 `desc` 的写是独占的；对 `avail` 的写是独占的；对 `used` 的读通过 `last_used_idx` 与设备的写分离）<sup>[10](#mod-drivers-ref-10)</sup>。这个假设在单生产者、单消费者模型下成立，但如果驱动层从多个线程并发调用 `add`，则会出现数据竞争——框架依赖调用者保证 `&mut self` 的互斥性。

### 5. 边角细节与不足

**`<sup>[7](#mod-drivers-ref-7)</sup>` `assert!` 风险**：`add_direct` 和 `add_indirect` 中使用了 `assert_ne!(buffer.len(), 0)`，如果上层传入空 buffer 会导致 panic。虽然这在驱动内部调用中可控，但如果 `VirtIOBlk` 对外暴露的 `read_blocks` 被传入空 buffer，同样会 panic——尽管上层文档要求长度非零。更好的做法是返回 `Error::InvalidParam`。

**`<sup>[2](#mod-drivers-ref-2)</sup>` 硬编码队列大小**：`VirtIOBlk` 硬编码 `QUEUE_SIZE = 16`，而 VirtIO 规范允许设备支持更大的队列。当设备实际最大队列小于 16 时，`VirtQueue::new` 会返回 `Error::InvalidParam`，但驱动没有尝试降级。这可能导致一些设备无法被驱动。

**`<sup>[8](#mod-drivers-ref-8)</sup>` `cfg(feature = "alloc")` 依赖**：间接描述符和间接描述符列表依赖 `alloc` feature。如果禁用 `alloc`，`add_indirect` 不可用，但 `add_direct` 在 `descriptors_needed > SIZE` 时会直接失败（虽然 `SIZE` 通常为 16，写入大于 16 个 buffer 的请求会被拒绝）。这限制了驱动处理大数据块的能力。

**`<sup>[3](#mod-drivers-ref-3)</sup>` 缺少 SAFETY 注释**：`write_desc` 方法中的 `unsafe` 块缺少 `// SAFETY:` 注释<sup>[7](#mod-drivers-ref-7)</sup>，虽然从上下文看是安全的，但未遵守 `#![deny(unsafe_op_in_unsafe_fn)]` 要求（该 crate 顶部有 `#![deny(unsafe_op_in_unsafe_fn)]`，但 `write_desc` 不是 `unsafe fn`，所以不会触发警告）。不过，审计者仍需确认 `desc` 指针和 `desc_shadow` 的有效性。

**`<sup>[9](#mod-drivers-ref-9)</sup>` 回调中 panic 风险**：`VsockConnectionManager::poll` 的回调闭包如果 panic，会传播到 `VirtIOSocket::poll`，导致驱动状态不一致。但回调中使用了 `unwrap`（如 `connections.last_mut().unwrap()`），如果 `ConnectionRequest` 事件上下文不满足预期，可能触发 panic。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层是 PulseOS 用户态与内核之间的接口层，基于 Linux ABI 标准实现了约 80 个系统调用，分布在文件 I/O、进程控制、信号管理、网络选项和杂项等子模块。核心抽象围绕 pulse_core 提供的文件描述符表 (fd_table)、进程/线程 (task) 及用户态安全访问 (uaccess) 展开，每个 sys_* 函数独立处理参数校验、权限检查和结果返回。与同家族基线 ArceOS 相比，本层显著扩展了系统调用覆盖面（特别是信号、资源限制和 polling），并引入了针对直接 I/O 和管道的零拷贝优化路径。

### 1. 核心抽象与外部依赖

系统调用层的实现分布在 `pulse_syscalls/src/impls/` 下的 `fs/io.rs`、`misc.rs` 和 `net/opt.rs` 三个文件中，每个文件封装一类系统调用。所有函数遵循一致的返回约定：成功返回非负值，失败返回 `-LinuxError::code()` 的负数。

核心抽象来自 `pulse_core` crate：<sup>[1](#mod-syscall-ref-1)</sup> 文件 I/O 通过 `pulse_core::fd_table` 提供的 `FileObject` trait（封装 `axio::Io` 等底层 I/O 操作）和 `FdObject`（提供 `as_any().downcast_ref` 实现运行时类型判断）。<sup>[2](#mod-syscall-ref-2)</sup> 用户态内存访问通过 `pulse_core::task::uaccess` 模块的 `read_user_plain`/`write_user_plain`/`read_user_bytes`/`write_user_bytes` 系列函数实现，这些函数以当前进程的页表上下文执行安全拷贝。

跨模块协同体现在：每个 syscall 函数通过 `get_fd_entry(fd)` 从当前进程的文件描述符表中获取 `FdEntry`，然后对 `entry.object`（`Arc<dyn FileObject>`）调用多态方法。这种设计将系统调用层与底层文件系统/网络栈解耦——`FileObject` 的具体实现来自 `axfs` 或 `axnet`，但系统调用层只需依赖 trait 方法。

辅助工具函数集中在 `utils` 子模块（从 `use crate::impls::utils::*` 可见），提供 `alloc_uninit_bytes`、`query_user_page_slice`、`read_user_iovec_array` 等跨系统调用共享的基础设施。<sup>[1](#mod-syscall-ref-1)</sup>

### 2. 关键设计取舍

**取舍一：零拷贝 vs 安全 fallback。** 文件读写路径中包含两条并行的数据路径：
- 快速路径：调用 `query_user_page_slice` 尝试直接获取用户缓冲区页面的 `&mut [u8]` 指针，如果成功则直接传递给底层 `object.read()`/`object.write()`，避免了一次内核缓冲区的拷贝。<sup>[3](#mod-syscall-ref-3)</sup>
- 慢速路径：分配临时内核缓冲区，通过 `read_user_bytes`/`write_user_bytes` 进行逐字节拷贝。

这一设计的代价是 `query_user_page_slice` 返回的指针必须在 `unsafe` 块中解引用，且代码中没有 `// SAFETY:` 注释（见 <sup>[3](#mod-syscall-ref-3)</sup> 第 2 行），这是显眼的技术债。

**取舍二：管道大块传输的专用零拷贝路径。** `sys_read` 和 `sys_write` 中针对 `PipeObject` 做了特殊优化：当用户缓冲区页面对齐（`buf % 4096 == 0`）、大小超过 64KB 且也是页面对齐时，调用 `pipe.read_zerocopy(buf, count)` 直接通过物理页交换完成数据传输。<sup>[4](#mod-syscall-ref-4)</sup> 这放弃了通用性换取了高性能 IPC，但如果用户缓冲区未对齐则静默回退到通用路径。

**取舍三：直接 I/O 的对齐强制。** 对于 `is_direct_regular_file()` 返回 true 的文件对象，系统调用层强制执行对齐检查：文件偏移、用户缓冲区地址和传输长度都必须按块大小对齐，否则返回 `EINVAL`。<sup>[5](#mod-syscall-ref-5)</sup> 这放弃了灵活性（非对齐小 I/O 会失败），但避免了底层驱动因非对齐请求而返回错误或产生性能下降。

**取舍四：poll 的混合等待策略。** `sys_ppoll` 和 `sys_pselect6` 实现了两级等待：前 64 轮使用 `axtask::yield_now()` 忙等待，之后切换到 `axtask::sleep(POLL_SLEEP_QUANTUM = 100μs)` 的休眠等待。<sup>[6](#mod-syscall-ref-6)</sup> 这种设计在低延迟 IPC 场景（前 64 轮快速响应）和低功耗场景（之后让出 CPU）之间做了权衡，但由于缺乏事件驱动机制（如 epoll），所有 poll 变体本质上是轮询，CPU 开销随 fd 数量线性增长。

**取舍五：资源限制的硬编码宽松策略。** `rlimit_for` 函数中对 `RLIMIT_CPU`、`RLIMIT_FSIZE`、`RLIMIT_DATA` 等大部分资源返回 `rlim_cur = u64::MAX`（无限），只有 `RLIMIT_STACK` 和 `RLIMIT_NOFILE` 使用实际配置值。<sup>[7](#mod-syscall-ref-7)</sup> 这保证了最大兼容性（不会因资源限制杀死未知应用），但放弃了安全隔离能力——一个恶意进程可以无限制消耗 CPU 时间或文件系统空间。

### 3. 跨模块协同

**文件描述符生命周期管理。** `sys_pipe2` 展示了跨模块回滚的协同模式：<sup>[8](#mod-syscall-ref-8)</sup> 先调用 `pipe_entries` 创建一对管道端点，再通过 `process.insert_fd_entry` 依次分配读端和写端的 fd 号。如果写端插入失败，代码会手动回滚读端 fd 的插入；如果后续 `write_user_bytes` 写入用户空间失败，则回滚两个 fd。这种模式在 `pulse_core` 的 `fd_table` 层不提供事务性支持时是必要的防御性编程。

**信号处理与线程状态同步。** `sys_rt_sigaction` 和 `sys_rt_sigprocmask` 等信号系统调用需要与 `pulse_core::task::Thread` 的信号屏蔽字和 `SignalShared` 状态协同：<sup>[9](#mod-syscall-ref-9)</sup> `sys_rt_sigaction` 通过 `process.signal_shared()` 获取进程共享的信号动作表，而 `sys_rt_sigprocmask` 通过 `thread.set_signal_blocked_mask()` 修改线程私有的屏蔽字。`sys_rt_sigreturn` 则直接调用 `thread.restore_from_sigreturn(tf)`，将 trap frame 恢复到信号处理前的状态。这种分工意味着信号处理的状态机分散在系统调用层、pulse_core 的 task 模块和 axhal 的 trap 层之间。

**网络选项的 stub 模式。** `getsockopt`/`setsockopt` 的实现大量使用 stub 返回：<sup>[10](#mod-syscall-ref-10)</sup> 对于 `SO_KEEPALIVE`、`SO_BROADCAST`、`SO_DONTROUTE` 等选项，setsockopt 直接返回 0 不做任何操作，getsockopt 返回硬编码 0；对于 `SO_SNDBUF`/`SO_RCVBUF` 返回固定值 65536。这种模式保证了常见应用程序不会因 `ENOPROTOOPT` 崩溃，但实际配置被静默丢弃，可能引发网络栈行为不符合用户期望的隐晦 bug。

**系统信息的主机名共享。** `sys_uname` 和 `sys_sethostname` 通过 `process.hostname_handle()` 共享进程内的 `RwLock<[u8; 64]>` 缓冲区：<sup>[11](#mod-syscall-ref-11)</sup> uname 读取时获取读锁，sethostname 写入时获取写锁。这种设计避免了全局可变的静态变量，但将主机名限制在单个进程内——所有进程看到的 uname 可能不同，这与 Linux 的全局主机名语义存在偏差。

### 4. 边角细节与不足

**不足一：`query_user_page_slice` 的安全契约缺失。** <sup>[3](#mod-syscall-ref-3)</sup> 中第 2 行的 `unsafe { &mut *slice_ptr }` 和类似写法没有附带 `// SAFETY:` 注释。安全契约是：`query_user_page_slice` 保证返回的指针在调用者当前线程的生命周期内有效，且指向的页面在用户地址空间中可读/可写。但没有任何文档或断言验证这一契约——如果底层页表发生并发修改（如另一个线程调用了 `munmap`），该指针将成为悬挂指针。这是 <sup>[2](#mod-syscall-ref-2)</sup> 中 `unsafe` 使用的典型技术债。 [re:1]

**不足二：Syslog 占位符是功能性缺陷。** `sys_syslog` 函数使用静态常量 `KMSG_PLACEHOLDER = b"PulseOS kernel log buffer is not persisted yet.\n"` 代替真正的内核消息缓冲区。<sup>[12](#mod-syscall-ref-12)</sup> 这意味着 `dmesg` 等工具永远看不到真正的内核日志。代码用 `AtomicBool` 记录是否已发出警告，但仅此而已——没有计划中的持久化实现。这是开发者明确知悉的功能差距。 [re:2]

**不足三：`sys_getrandom` 绕过了内核随机数生成器。** <sup>[13](#mod-syscall-ref-13)</sup> 显示 `getrandom` 通过 `FS_CONTEXT.lock().read_prefix(path, buflen)` 读取 `/dev/urandom` 或 `/dev/random` 文件，而不是调用硬件 RNG 或内核 CSPRNG。这意味着随机数生成依赖于文件系统挂载了相应的设备文件，且路径字符串比较在每次调用时都要执行，性能开销大。在无设备文件的嵌入式场景下，该调用会静默失败。 [re:3]

**不足四：信号相关 `unsafe` 块缺乏安全注释。** <sup>[14](#mod-syscall-ref-14)</sup> 中 `sys_rt_sigaction` 使用 `core::mem::transmute::<_, usize>(new_act.sa_handler)` 将函数指针转为整数，以及 `core::mem::transmute(old.handler)` 的逆转换，均无安全注释。这些转换的安全性依赖于 `SigAction` 的 `handler` 字段始终存储有效的函数指针或特殊常量（如 `SIG_IGN`/`SIG_DFL`），但没有任何运行时检查确保这一点。 [re:4]

**不足五：`sys_ppoll` 的信号检查在最终 poll 阶段可能漏检。** <sup>[6](#mod-syscall-ref-6)</sup> 中 ppoll 的混合等待路径在每次轮询后检查 `thread.has_pending_signal()`，但在最终写入回 `pollfd` 之前没有再次检查。<sup>[6](#mod-syscall-ref-6)</sup> 第 15-18 行显示返回前有一个单独的 `write_back` 调用，如果在 `write_back` 执行期间信号到达，进程会返回就绪状态而非 `EINTR`，这违反了 POSIX 规范（信号应在 poll 返回前优先处理）。
