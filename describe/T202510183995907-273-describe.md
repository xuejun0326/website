## 📖 一、总览

该仓库以教学内核 xv6-riscv 为蓝本，但大幅扩展了线程模型、信号处理和文件系统，试图向 Linux 兼容性靠拢。其架构引入线程组/进程分离、SV39 页表上的 VMA、基于 lwext4 的 ext4 支持，以及多核启动的多架构适配（RISC-V/LoongArch），功能集合远超原始 xv6，但也带来显著的复杂性增长和完成度参差。

最值得注意的特点：
1. **线程与信号框架的完整性**：不仅将调度单位下沉到线程，还实现了完整的信号处理流程（sigaction/sigpending/sigreturn）和 futex 同步原语，在教学内核中鲜见，但多处存在锁竞态和内存泄漏（见信号模块）。
2. **ext4 集成与 VFS 抽象**：通过 lwext4 库引入 ext4，并用 VFS 层兼容 xv6 原生文件系统，文件系统能力大幅提升，但日志功能和并发保护不完整。
3. **跨架构启动多样性**：同时支持 RISC-V 和 LoongArch，后者利用硬件直接映射窗口简化启动，但时钟中断未初始化带来风险。

## 🎯 二、综合评价

### 整体定位：功能扩展型教学内核
该内核在 xv6 的简洁框架上叠加了多线程、信号、ext4 等复杂子系统，定位从“教学演示”向“Linux 兼容实验”倾斜。它试图证明教学内核也能承载完整 OS 服务，但代价是代码量翻倍（~35k LOC）和大量悬而未决的 TODO。与同家族其他实现相比，它最接近“实验型内核”：敢于引入线程组、VMA 和跨架构启动，却因工期或设计取舍留下了明显漏洞。

### 真正的创新点：线程与信号的全新设计
最突出的创新是**将调度单位从进程下沉到线程**，并围绕此重构了 proc 和 tcb 结构（[task.ref:1][task.ref:2]）。这要求 fork、exit 等核心路径与线程创建原子配合，页表操作（如 uvm_thread_trapframe）必须同步完成，远比 xv6 单线程复杂。信号机制同样如此——它实现了 **POSIX 实时信号的完整框架**，包括 sigpending 排队、sigaction 表、sigreturn 回跳（[signal.ref:4][signal.ref:5]），但在教学中通常只做 stub。这两个设计在 xv6 生态中属于超前尝试，尽管具体实现有瑕疵。

另外，**多文件系统共存的 VFS 抽象**（[fs.ref:1]）和 **LoongArch 窗口映射启动**（[boot.ref:3][boot.ref:4]）也是亮点：前者为 ext4 与 xv6 原生 fs 提供统一调用，后者利用硬件特性简化早期页表，体现了架构感知的实践。

### 取舍判断：广度优先，深度不足
开发团队明显将精力投向了**功能覆盖**而非**正确性与性能**。例如：
- 为了快速支持 ext4，直接嵌入 lwext4 库，但日志模块标为 `__unused`（[fs.ref:4]），每个 ext4 系统调用仅返回 -1 而未设 errno（[fs.ref:1]）。
- 信号处理中 SIG_IGN 路径导致内存泄漏（[signal.ref:1]），sig_first_member 越界风险（[signal.ref:2]），反映出对边角情况的忽略。
- 多核安全方面，大量锁使用不一致：futex 等待队列无锁保护（[ipc.ref:4]），mmap 的 VMA 分配与锁释放顺序错误（[mm.ref:1]）。

被刻意省略的部分包括：写时复制（fork 直接拷贝）、TLB 刷新（uvmunmap 未调用 sfence）、用户栈 ASLR（VMA 地址固定）以及时钟中断初始化（timerinit 被注释）。这些取舍让内核在功能表上好看，但离稳定运行尚远。

### 完成度与不足
从模块看，**进程调度和内存管理基本完成**，能跑多线程及缺页；**信号和 IPC 框架存在**，但脏路径明显；**文件系统 ext4 挂载可读，但写操作和日志几乎未验**；**启动流程多架构适配成功**，但遗留调试注释。整体完成度约 50%，大量 FIXME 和 TODO 表明处于早期开发阶段。最大的不足在于**没有 syscall 测试集**（基础事实 syscall_count=0 暗示编译/运行环境未集成），很难判断核心路径是否真正工作。此外，代码规范较差（硬编码常量、全局裸指针、注释不统一），为后续维护埋下隐患。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块实现了从硬件复位到 C 内核入口 `start()` 的过渡，核心抽象包括汇编入口 `_entry`、多核栈 `stack0`、架构相关的页表模式（LoongArch 直接映射窗口 vs RISC-V 关闭分页）以及统一 C 入口 `start()`。LoongArch 在汇编阶段通过 CSR_DMW 建立固定窗口映射，省去早期页表构建；RISC-V 则在 `start()` 中关闭分页并开启 S 态中断。模块通过 `entry.S`（两种架构）与 `start.c` 的约定接口协同：汇编设置栈和基本特权级，C 继续配置中断/分页状态，最终调用 `main()`。与原版 xv6-riscv 相比，本内核新增了 LoongArch 支持，采用窗口映射简化启动，但 `timerinit()` 被完全注释掉，时钟中断初始化可能依赖其他路径，存在遗留风险。

### 1. 核心抽象：从硬件复位到 C 环境的建立

启动流程模块的核心抽象是“从 CPU 复位向量到 C 内核入口 `main()` 的过渡”，它封装了架构相关的寄存器初始化、栈设置和页表模式选择。两种架构共享同一套 C 入口 `start()`，但汇编阶段差异显著。

在 RISC-V 中，`_entry`（`kernel/arch/riscv/entry.S`）利用 SBI 传入的 hartid（保存在 a0），将其复制到 tp 寄存器作为 CPU 标识，然后计算栈指针：`sp = stack0 + (hartid+1) * 4096` <sup>[5](#mod-boot-ref-5)</sup>。注释“内核栈是否需要加大”暗示栈尺寸选择是权衡——固定 4KB 每核简化了内存布局，但可能不足。随后直接调用 `start()`。

在 LoongArch 中，`_entry`（`kernel/arch/loongarch/entry.S`）首先配置 CSR：CRMD 写入 0xb0（PLV=0, IE=0, PG=1）立即启用分页 <sup>[3](#mod-boot-ref-3)</sup>；PRMD 写入 0x04（PLV=0, PIE=1, PWE=0）；EUEN 写入 0x00 关闭扩展单元。然后通过 CSR_DMWIN0 和 CSR_DMWIN1 设置两个直接映射窗口，将虚拟地址区间 0x8000xxxxxxxxxxxx 等映射到物理地址 <sup>[4](#mod-boot-ref-4)</sup>。这一设计使得整个内核在虚拟地址空间运行，无需早期页表构建。栈设置使用 `la.abs $sp, stack0+4096` 并加上 CPUID*4096 偏移 <sup>[7](#mod-boot-ref-7)</sup>，与 RISC-V 思路一致但实现更简洁。

两种架构最终都跳转到 C 函数 `start()`，该函数在 `kernel/init/start.c` 中定义 <sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>，是架构无关与架构相关代码的分水岭。

### 2. 关键设计取舍：页表模式的早期切换策略

启动阶段页表处理是核心设计决策。本内核在两种架构上采用了截然不同的策略，反映了对硬件特性的适应。

**LoongArch：硬件窗口映射**。LoongArch 的 CSR_DMW 机制允许在 MMU 使能后直接定义几个固定窗口，将虚拟地址高位匹配的区域直通到物理地址，无需软件页表。`entry.S` 中设置 DMW0 和 DMW1 后，内核就可以在虚拟地址空间执行，`start()` 中无需再处理 satp 或页表 <sup>[1](#mod-boot-ref-1)</sup>。这一选择放弃了页级权限控制和细粒度映射，但启动代码极简，且避免了早期页表分配和 TLB 维护。代价是地址空间布局受限（窗口固定），但本内核仅需映射内核所在区域，足够使用。

**RISC-V：关闭分页，纯物理执行**。RISC-V 的 `start()` 首先执行 `w_satp(0)` 关闭分页 <sup>[1](#mod-boot-ref-1)</sup>，确保后续指令在物理地址空间执行。同时使能 S 态中断（SSTATUS_SIE 和 SIE 寄存器）。这种选择保留了页表初始化的完全灵活性——`main()` 可以通过 `kvminit()` 建立任意映射，但要求内核在关闭分页期间代码和数据都位于物理地址的低 1GiB 内（QEMU 默认），且需小心避免取指超出物理范围。该策略与 xv6-riscv 原版一致，但本内核的 `start()` 中注释掉了 `timerinit()` <sup>[6](#mod-boot-ref-6)</sup>，这将在 §3 讨论。

两种策略都避免了在汇编中构建复杂页表，但取舍分明：LoongArch 依赖硬件固定映射，RISC-V 依赖物理连续且地址可知。

### 3. 跨模块协同：汇编、C 与失落的时钟中断

启动流程的跨文件协同体现在三个阶段：
1. **汇编层**（`entry.S`）：初始化栈指针、基本 CSR/寄存器，确保调用 C 时栈有效。LoongArch 额外配置 DMW，使 `start()` 可在虚拟地址运行。
2. **C 层**（`start.c`）：`start()` 根据架构完成中断和分页状态的最后设置。RISC-V 版本打开 S 态中断 <sup>[1](#mod-boot-ref-1)</sup>；LoongArch 版本为空（仅注释）<sup>[2](#mod-boot-ref-2)</sup>。然后调用 `main()`。
3. **后续层**（`main()` 及之后）：`main()` 负责剩余的初始化（如控制台、物理内存分配器、进程表等）。原版 xv6 中 `start()` 会调用 `timerinit()` 设置机器态时钟中断，但本内核将其整个函数体注释掉 <sup>[6](#mod-boot-ref-6)</sup>。这意味着时钟中断的初始化被推迟到 `main()` 或完全由其他机制替代（如 SBI 或平台定时器）。这导致一个潜在的缺憾：在多核系统中，每个核的时钟中断 scratch 空间未配置，`timervec` 也不会被注册，可能影响调度器定时中断。

汇编与 C 的接口通过全局符号 `stack0` 和 `timer_scratch` 共享数据 <sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>。栈数组定义在 `start.c` 中，entry.S 通过 `la.abs` 或 `la` 指令访问。这种跨文件协同是早期内核的典型风格——汇编直接引用 C 符号。

### 4. 边角细节与不足

基于代码审查发现以下风险点和不足（每点均对应具体行引用）：

- **栈尺寸风险**：RISC-V entry 中使用 `li t0, 1024*4` 作为每核栈大小，注释“内核栈是否需要加大” <sup>[5](#mod-boot-ref-5)</sup>。4KB 对于早期启动足够，但若内核复杂化（如深调用链、大量局部变量）容易溢出，且无保护机制（如栈 canary）。
- **hartid/CPUID 未校验**：两种架构都直接使用 hartid（RISC-V 的 a0，LoongArch 的 CSR 0x20）作为栈偏移索引，未验证是否小于 NCPU。若硬件返回无效值，将导致越界写栈 <sup>[5](#mod-boot-ref-5)</sup><sup>[7](#mod-boot-ref-7)</sup>。
- **LoongArch CSR 硬编码**：CRMD、PRMD、EUEN 写入的值使用立即数（0xb0、0x04、0x00），而非命名的宏定义 <sup>[3](#mod-boot-ref-3)</sup>。这降低可读性，且若硬件版本差异需逐一修改。
- **LoongArch `start()` 空实现**：`#elif defined(__ARCH_LOONGARCH)` 分支仅包含注释“do nothing here, just for compatibility” <sup>[2](#mod-boot-ref-2)</sup>。缺少对 S 态中断的使能（如设置 CSR.EEN bit），可能中断无法到达 S 态。
- **时钟中断初始化缺失**：整个 `timerinit()` 函数被 `#if 0` 注释掉 <sup>[6](#mod-boot-ref-6)</sup>。若 `main()` 也未初始化，系统将无法产生时钟中断，导致时间片轮转失效。
- **RISC-V 栈计算幻数**：`addi a0, a0, 1` 将 hartid 加 1 再乘以栈大小，可能是为了避开 CPU0 栈底重叠，但未注释原因 <sup>[5](#mod-boot-ref-5)</sup>。
- **多核竞态**：启动阶段理论上只有一个核执行，但 `start.c` 中无显式同步，若未来支持 SMP 热插拔，`stack0` 和 `timer_scratch` 的写入可能产生竞争。

这些不足多为早期实现的原生遗留，但在生产级内核中需逐一修复。

### ⚠ 开放问题

- timerinit() 被完整注释，时钟中断未初始化，可能导致调度器无法获得定时中断。
- LoongArch start() 空实现未使能 S 态中断，若 main() 也未处理则中断无法到达。
- 栈大小 4KB 每核可能不足，RISC-V entry 注释已提示此疑虑，且无溢出检测。
- hartid/CPUID 未与 NCPU 比较，越界时可写坏其他核栈或无关内存。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块实现了基于 xv6 的物理页面分配器、Sv39 三级页表管理以及用户级 mmap 支持。核心抽象包括空闲列表分配器（kalloc.c）、页表遍历（walk/mappages）和 VMA 描述符（vma_struct）。与原始 xv6 相比，扩展了多线程内核栈映射、文件映射 mmap（包括脏页写回）以及每个进程的 VMA 链表。分配器使用单一全局空闲列表和自旋锁，页表操作在缺页时惰性分配中间节点，用户堆通过 uvmalloc 立即分配。

### 1. 核心抽象：物理页面、页表与 VMA

物理内存分配器（`kalloc.c`）维护一个由自旋锁 `kmem.lock` 保护的单链表空闲列表。`kalloc()` 从表头摘取页面并清零，`kfree()` 将页面回插并用 `0x01` 填充以检测悬空指针 <sup>[1](#mod-mm-ref-1)</sup>。页表层（`vm.c`）为 RISC-V Sv39 提供三级遍历：`walk()` 按需分配中间页表，`mappages()` 为连续虚拟地址范围创建 PTE <sup>[2](#mod-mm-ref-2)</sup>。用户层使用 `mm_struct`（`mm.h`）封装每个进程的 VMA 链表和用户页表，`vma_struct`（`mmap.h`）记录映射基址、长度、保护位、标志及关联文件 <sup>[3](#mod-mm-ref-3)</sup><sup>[4](#mod-mm-ref-4)</sup>。

### 2. 关键设计取舍：惰性映射与立即拷贝

在 `fork` 时，内核采用立即物理页面复制而非写时复制：`uvmcopy()` 为每个用户页分配新页并 `memmove` 内容，简化了缺页逻辑但牺牲了性能 <sup>[5](#mod-mm-ref-5)</sup>。相比之下，`mmap` 支持采用惰性分配：`do_mmap()` 仅创建一个 VMA 条目，不分配物理内存；实际页面在首次访问时由缺页处理程序按需分配 <sup>[6](#mod-mm-ref-6)</sup>。这一选择与 xv6 的 `sbrk`（`uvmalloc` 立即分配）形成对比。多线程扩展下，`uvm_thread_trapframe()` 为每个线程的陷阱帧映射专用页面，内核栈在启动时预映射并配有守卫页 <sup>[7](#mod-mm-ref-7)</sup>。

### 3. 跨模块协同：文件系统与缺页处理

文件映射通过 VMA 中的 `struct file` 指针与文件系统层集成。`mmap_writeback_unmapf()` 在取消映射时遍历 VMA 的页面，检查 `PTE_D` 脏位，对共享文件映射调用 `filewrite` 写回，再通过 `uvmunmap` 释放物理页 <sup>[8](#mod-mm-ref-8)</sup>。`proc_copy_vma()` 在 `fork` 时复制 VMA 列表并递增文件引用计数，确保共享映射的正确生命周期管理 <sup>[9](#mod-mm-ref-9)</sup>。系统调用接口 `sys_mmap`/`sys_munmap`/`sys_mprotect` 解析用户参数并委托给内部实现；例如 `do_mprotect` 通过 `setperm` 逐 PTE 修改权限位 <sup>[10](#mod-mm-ref-10)</sup>。

### 4. 边角细节与不足

- **脏位模拟缺失**：`mmap_writeback_unmapf` 依赖硬件设置的 `PTE_D` 位，但 RISC-V Sv39 规范不强制硬件更新该位；若缺页处理程序未模拟脏位，写回可能遗漏脏页 <sup>[8](#mod-mm-ref-8)</sup>。
- **max_vma 耗尽**：`do_mmap` 中 `max_vma` 单调递减，若进程反复 mmap 可能导致其低于 `MMAP_END_ADDRESS`；代码虽有运行时检查但会泄漏已分配的 VMA <sup>[11](#mod-mm-ref-11)</sup>。
- **竞争条件**：`mmap_writeback_unmapf` 遍历 VMA 页面时不持有 `mm.lock`，并发 `do_munmap` 可能释放 VMA 导致释放后使用；`do_munmap` 自身也含锁顺序 TODO，可能引发与文件 inode 锁的死锁 <sup>[12](#mod-mm-ref-12)</sup>。
- **中断安全性**：`kalloc`/`kfree` 获取 `kmem.lock` 时不关闭中断，若缺页处理程序在中断上下文中分配内存将导致自旋锁死锁 <sup>[1](#mod-mm-ref-1)</sup>。
- **桩函数限制**：`kmalloc` 和 `kcalloc` 对大于一页的请求返回 0，调用者未检查返回值可能引发静默错误 <sup>[13](#mod-mm-ref-13)</sup>。
- **TLB 刷新遗漏**：`uvmunmap` 清除用户 PTE 后未执行 `sfence_vma`，多核环境下其他核心可能缓存过时翻译 <sup>[14](#mod-mm-ref-14)</sup>。
- **TODO 积压**：`do_mmap` 包含 `max_vma` 管理 TODO，`do_munmap` 包含锁顺序 TODO，均标示已知未解决问题 <sup>[15](#mod-mm-ref-15)</sup>。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块的核心抽象是线程组（thread_group），将进程（struct proc）视为资源容器，而调度单位下沉到线程（struct tcb）。它解决了xv6单线程进程模型无法支持多线程应用的问题，并扩展了信号、futex、clone等机制。与原始xv6-riscv相比，引入了完整的线程控制块、队列状态管理、进程家族树以及/proc文件系统接口。

### 1. 核心抽象：线程组与进程解耦

本模块最根本的设计决策是将“调度实体”从进程下沉到线程。进程控制块 `struct proc`（<sup>[1](#mod-task-ref-1)</sup>）不再包含调度状态（RUNNABLE/RUNNING），而是退化为资源容器，内部嵌入一个 `struct thread_group tg` 来管理所有线程。线程控制块 `struct tcb`（<sup>[2](#mod-task-ref-2)</sup>）持有自己的状态、内核栈、上下文和 trapframe，并作为调度队列的基本元素。

这种解耦的代价是复杂性显著上升：原本 xv6 里一个进程一个栈、一个 trapframe 的简洁结构被打破。进程的页表由所有线程共享（`p->mm.pagetable`），但每个线程又需要独立的 trapframe 映射在固定的虚拟地址 `THREAD_TRAPFRAME(t->tidx)`（<sup>[2](#mod-task-ref-2)</sup> line 36-40）。为了实现这一点，`proc_join_thread()` 在 `uvm_thread_trapframe()` 中为每个线程分配并映射一页，这要求页表操作必须与线程创建原子配合。

进程家族树通过 `parent`、`first_child` 和 `sibling_list` 组织，与 xv6 的儿童链表不同，它采用更灵活的“第一个孩子+兄弟链表”结构（<sup>[1](#mod-task-ref-1)</sup> line 47-52），但这也意味着插入和删除需要小心处理边界情况（如 `delete_child` 中的空判断）。

### 2. 调度器设计与上下文切换

调度器 `thread_scheduler()`（<sup>[3](#mod-task-ref-3)</sup>）是一个无限循环，每次从全局 `runnable_t_queue` 中弹出一个线程，设置 `cpu->thread`，然后通过 `swtch()` 切换到该线程的上下文。线程主动让出 CPU 的路径是 `thread_yield()` → `tcb_q_change_state(t, TCB_RUNNABLE)` → `thread_sched()`（<sup>[4](#mod-task-ref-4)</sup>）。`thread_sched()` 会检查锁嵌套深度（`mycpu()->noff == 1`）和中断状态（`intr_get()` 为假），确保在关中断且仅持有线程锁的情况下切换。

一个值得注意的细节是 `tcb_q_change_state()` 对 `TCB_RUNNING` 状态的特殊处理：当线程正在运行时，它不在任何状态队列中，因此从 RUNNING 切换到其他状态时，不能从旧队列中移除，而是直接设置状态，然后由 `thread_scheduler` 或 `thread_sched` 负责入队新队列（<sup>[5](#mod-task-ref-5)</sup> line 65-68）。这种设计避免了在 `swtch` 后立即操作队列的竞态，但要求状态切换点必须精确同步。

上下文切换 `swtch()` 只保存/恢复 callee-saved 寄存器（`struct context` 中的 s0-s11 和 ra/sp），这与 xv6 一致。但本模块的上下文切换发生在线程层，因此 `cpu->context` 是调度器的上下文，而 `t->context` 是线程的上下文（<sup>[3](#mod-task-ref-3)</sup> line 33-37）。

### 3. 跨模块协同：创建、退出与信号传递

进程/线程的创建是跨文件合作的典型例子。`allocproc()`（<sup>[6](#mod-task-ref-6)</sup>）仅初始化进程的固定字段（pid、pgid、队列等），而 `create_proc()`（<sup>[7](#mod-task-ref-7)</sup>）进一步调用 `alloc_thread()` 分配一个初始线程作为 `group_leader`，再通过 `proc_join_thread()` 将两者绑定。`alloc_thread()`（<sup>[8](#mod-task-ref-8)</sup>）分配线程时并不立即映射 trapframe——那是在 `proc_join_thread()` 中通过 `uvm_thread_trapframe()` 完成的，这要求进程的页表已经存在。

线程退出 `thread_exit()` 递减线程组的原子计数，当计数降到 1 时调用 `proc_exit()`（<sup>[9](#mod-task-ref-9)</sup>）。`proc_exit()` 会关闭所有文件描述符、释放用户内存、重设子进程的父进程为 init、向父进程发送 `SIGCHLD`，然后将自身状态转为 ZOMBIE。信号发送 `thread_send_signal()`（<sup>[10](#mod-task-ref-10)</sup>）会检查目标线程是否在 SLEEPING 状态，如果是则调用 `thread_wakeup_specific()` 强制唤醒——这种设计避免了信号被无限期挂起。

文件系统接口 `procfs` 通过注册自定义 `file_ops` 暴露 `/proc/interrupts` 等伪文件，读取时遍历全局中断计数数组 `intrcnt`。虽然只是简单统计，但它演示了进程模块如何与 VFS 层配合：`procfs_read()` 内部调用 `copyout` 将格式化字符串复制到用户空间，需要持有目标进程的页表（`myproc()->mm.pagetable`）。

### 4. 边角细节与不足

尽管模块整体设计合理，但代码中存在多处值得警惕的细节：

- **<sup>[1](#mod-task-ref-1)</sup> 线程 ID 搜索的边界隐患**：`thread_kill()` 和 `thread_group_kill()` 检查 `tid < 0 || tid >= NTHREADS`，但 `tid` 是由 `alloctid()` 全局递增分配的，可能大于 `NTHREADS`。不过搜索时遍历整个 `tcb_pool`（固定大小），若 tid 未找到会返回 -1，因此不会越界。但若线程数量超过 `NTHREADS`，`alloc_thread()` 会因 `queue_pop_atomic` 返回 NULL 而失败，这实际上是容量限制，而非边界错误。
- **<sup>[2](#mod-task-ref-2)</sup> `do_clone` 中 `CLONE_VM` 共享页表的缺陷**：当 `flags & CLONE_VM` 时，新进程直接借用父进程的页表（`np->mm.pagetable = p->mm.pagetable`），但 `freeproc()` 在回收进程时会无条件调用 `proc_freepagetable()` 解除映射并释放物理页。如果子进程先退出，父进程的页表会被撤销，后续父进程访问用户内存将引发致命错误。目前该路径未实现引用计数，共享页表极不安全。
- **<sup>[3](#mod-task-ref-3)</sup> 全局队列的锁竞争**：所有 CPU 共用同一组 `g_tcb_queues`，每次调度器取任务和状态切换都涉及原子队列操作。虽然使用了 `queue_push_back_atomic` / `queue_pop_atomic`，但全局锁可能成为多核扩展的瓶颈。xv6 原版采用每个 CPU 的进程表轮询，此处未采用 per-CPU 队列是明显的性能取舍。
- **<sup>[4](#mod-task-ref-4)</sup> `thread_sleep` 中的超时字段与 chan 的重置时机**：在 `thread_sleep()` 中，`t->timeout` 在睡眠前设置，但唤醒后立即清零；而 `t->chan` 在 `thread_sched()` 返回后清零。但 `thread_wakeup_chan_timeout()` 同时匹配 chan 和 timeout，如果超时和 chan 唤醒同时发生，可能导致竞态。代码中并未使用原子操作保护 `chan` 和 `timeout` 的检查/修改序列。
- **<sup>[5](#mod-task-ref-5)</sup> `freeproc` 中锁获取顺序风险**：`freeproc()` 在遍历线程组时先获取 `lth_exitlock`，再获取 `cur_proc->tg.lock`，而 `thread_exit()` 路径中获取锁的顺序可能相反（先 `tg.lock` 后 `lth_exitlock`），这构成了潜在的死锁。尽管当前版本中 `freeproc` 仅在进程最后一个线程退出时被调用，`thread_exit` 也会在调用 `proc_exit` 前递减计数，但锁顺序不一致是经典的死锁诱因。
- **死代码与未完成实现**：`proc.c` 中大量注释掉的代码（如 `mm_init`、`pid_lock`、`trapframe` 相关），以及 `do_clone` 中 `CLONE_FS`、`CLONE_FILES` 只是打印而未实际实现（<sup>[6](#mod-task-ref-6)</sup> line 658-662），这些都是早期开发的残留信号。此外，`sighandinit` 在 fork 和 clone 时被多次调用，但并未考虑 `CLONE_SIGHAND` 情况下的引用计数共享（已经在 do_clone 中处理，但 fork 中未处理）。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 文件系统模块基于 lwext4 轻量级 ext4 库，通过 VFS 层抽象支持 ext4 和 xv6 原生文件系统共存。核心抽象包括块设备注册、挂载点管理、JBD 日志、extent 树和 VFS 操作表。相比 xv6 原版，本内核集成了完整的 ext4 支持，提供了更多系统调用（dup3、readv、linkat 等），并实现了日志恢复机制。该模块解决了对大容量、稳健文件系统的需求，同时保持与 xv6 进程管理的兼容。

### 1. 核心抽象：VFS 层与多文件系统支持

本文件系统模块的核心抽象是 VFS（虚拟文件系统）层，它通过 `struct vfs_filesystem` 和 `fsops` 函数指针表统一管理不同文件系统实现。`sysfile.c` 中的系统调用（如 `sys_link`、`sys_unlink`、`sys_mkdir`）都遵循同一模式：先通过 `vfs_resolve_fs` 解析绝对路径所属的文件系统实例，再调用对应的 `fsops->link`、`fsops->unlink` 等操作<sup>[1](#mod-fs-ref-1)</sup>。这种设计允许 ext4 和 xv6 原生文件系统（由 `xv6fs.h` 暗示）共存。 ext4 部分则由 `ext4.c` 提供挂载/卸载（`ext4_mount`、`ext4_umount`）、目录项操作（`ext4_link`、`ext4_unlink`）等高层接口，内部依赖 `ext4_blockdev`、`ext4_bcache` 等底层结构。块设备通过 `ext4_device_register` 注册到全局静态数组 `s_bdevices`，挂载点则记录在 `s_mp` 数组中，两者均受限于编译期常量 `CONFIG_EXT4_BLOCKDEVS_COUNT` 和 `CONFIG_EXT4_MOUNTPOINTS_COUNT`<sup>[2](#mod-fs-ref-2)</sup>。

### 2. 关键设计取舍：lwext4 库集成与 JBD 日志

选择 lwext4 而非自研文件系统意味着快速获得成熟、经过测试的 ext4 实现，但引入了外部依赖和额外的内存开销（如块缓存、目录索引）。lwext4 采用可移植的 C 语言编写，通过宏和回调与内核适配（例如 `EXT4_MP_LOCK` 宏调用操作系统锁），但并未使用内核的休眠锁或自旋锁机制，而是依赖用户自定义的 `os_locks` 回调<sup>[3](#mod-fs-ref-3)</sup>。日志子系统实现了标准的 JBD（Journaling Block Device）协议，支持校验和（CRC32C）、撤销块（revoke）和事务恢复。`ext4_journal.c` 负责 JBD 超级块的读写、事务开始/停止、以及崩溃恢复（`jbd_recover`）。值得注意的是，日志功能默认未完全启用——`__ext4_journal_start` 和 `__ext4_journal_stop` 被标记为 `__unused`，表明开发者可能尚未在常规挂载路径中集成日志事务<sup>[4](#mod-fs-ref-4)</sup>。此外，extent 树（`ext4_extent.c`）支持大文件和稀疏存储，通过 `ext4_extent_tree_init` 初始化根头，但需要 `CONFIG_EXTENTS_ENABLE` 编译选项启用。

### 3. 跨模块协同：系统调用到 VFS 到 ext4 的调用链

文件 I/O 系统调用（如 `sys_read`、`sys_write`）在 `sysfile.c` 中通过 `argfd` 获取文件描述符对应的 `struct file`，然后调用 `fileread`/`filewrite`（位于 `file.c`，未提供片段）。这些函数最终会通过 `struct file` 中的 `type` 和 `ip` 字段找到具体的 inode 操作。对于 ext4 文件，路径解析和元数据操作（如 `link`、`unlink`）则由 `generic_link` 等函数代理到 VFS 层。`generic_link` 调用 `vfs_resolve_fs` 得到 `struct vfs_filesystem`，再调用 `fsops->link`，该指针实际指向 `ext4_link`<sup>[1](#mod-fs-ref-1)</sup>。`ext4_link` 内部调用 `ext4_dir_add_entry` 添加目录项，并处理 `..` 和 `.` 的链接计数。跨文件协同的关键在于 VFS 层作为中间人解耦了系统调用和具体文件系统实现。

### 4. 边角细节与不足

代码中存在多处早期实现的痕迹和潜在风险：
- **TODO 时间戳**：`ext4_unlink` 中注释了父目录和 inode 的时间戳更新（`// TODO: Update timestamps of the parent`），这意味着 `unlink` 后不会更新 ctime/mtime，可能影响依赖时间戳的应用程序<sup>[5](#mod-fs-ref-5)</sup>。
- **未使用的日志函数**：`__ext4_journal_start` 等函数被 `__unused` 修饰，且 `__ext4_recover` 中的日志恢复路径分配了动态内存但只在特定编译选项下生效，实际挂载时并未调用，崩溃恢复能力可能缺失<sup>[4](#mod-fs-ref-4)</sup>。
- **固定数组越界风险**：`s_bdevices` 和 `s_mp` 数组大小由编译常量决定，`ext4_device_register` 和 `ext4_mount` 未对超出范围的索引做保护，仅返回 `ENOSPC`，但如果调用方忽略错误仍继续使用，可能导致未定义行为<sup>[2](#mod-fs-ref-2)</sup>。
- **错误处理不完整**：`sysfile.c` 中许多错误路径仅 `printf` 并返回 -1，未设置 `errno`。例如 `generic_link` 在 `fsops->link` 失败时打印消息并返回 -1，上层 `sys_link` 直接返回该值，用户进程无法通过 `errno` 获取具体错误原因<sup>[1](#mod-fs-ref-1)</sup>。
- **并发安全隐忧**：`ext4.c` 使用 `EXT4_MP_LOCK` 宏保护挂载点，但在系统调用路径（如 `sys_link`）中，`vfs_resolve_fs` 和后续 `fsops->link` 之间没有显式加锁，多线程并发操作同一挂载点可能产生竞态。
- **FIXME 注释**：`ext4_journal.c` 的 `jbd_get_fs` 中存在 `/* FIXME: detection on existance of journal bdev is missing.*/`，表明外部日志设备检测未实现，当前仅支持内联日志<sup>[6](#mod-fs-ref-6)</sup>。

这些不足表明文件系统模块尚处于早期集成阶段，日志和错误处理机制有待完善。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块实现了 POSIX 风格的信号处理框架，核心抽象包括 sigset_t（位掩码信号集）、sigaction（信号处理动作）、sighand（线程级信号动作表）、sigpending（待处理信号队列）和 rt_sigframe（信号帧）。它解决了用户态信号分发、阻塞、排队和自定义处理的问题。与 xv6 原版相比，该实现增加了完整信号处理流程：支持 SA_SIGINFO 实时信号、sigprocmask 阻塞、sigtimedwait 等待和 sigreturn 恢复，并通过两级锁（sighand.siglock 和 sigpending.siglock）隔离信号配置与发送路径。架构上通过 sigret.S 中的 trampoline 实现用户态返回后的内核重新介入。

### 1. 核心抽象与数据结构

信号模块以 `sigset_t` 为基石，定义为 `uint64` 位掩码（signal.h 第57行），最多覆盖 64 个信号，通过 `sig_gen_mask` 等宏进行集合运算。每个信号的动作由 `struct sigaction` 描述（signal.h 第119-127行），其联合体 `sa_handler` / `sa_sigaction` 支持传统单参数处理器和 SA_SIGINFO 三参数处理器。每个线程（`struct tcb`）拥有独立的 `struct sighand` 指针 `t->sigs`，其中包含长度为 `_NSIG`（64）的 `actions` 数组（signal.h 第131-134行）和一把 `siglock` 保护配置变更。

待处理信号的排队通过 `struct sigpending` 实现（signal.h 第136-140行），包含链表头 `list`、累积位掩码 `signal` 和自身的 `siglock`。`struct sigqueue` 节点（signal.h 第142-146行）挂载在 `list` 上，携带 `siginfo_t` 详细信息。这种设计将**信号配置（sighand）**与**信号发送（sigpending）**分离：前者由 `siglock` 保护，后者由 `sigpending.siglock` 保护，减少了锁竞争。

信号帧 `struct rt_sigframe`（signal.h 第206-210行）包含三部分：标准的 `struct siginfo info`、符合 RISC-V ABI 的 `ucontext_t uc_riscv`（内含通用寄存器 `mcontext_t`），以及内核自定义的 `struct ucontext uc`（内含 `struct sigcontext` 即 `trapframe` 拷贝和 `sigprocessing` 字段）。这种嵌套设计兼容用户态 libc 的 `ucontext_t` 期望，同时保存了内核需要的完整执行上下文。

### 2. 关键设计取舍

**跳板地址编译期固定**：`setup_rt_frame` 将返回地址 `tf->ra` 设置为 `SIGRETURN`（signal.c 第196行），这是一个在 `memlayout.h` 中定义的固定虚拟地址。对应的 `sigret.S`（riscv 和 loongarch）提供了一段执行 `ecall` 的代码，以 `SYS_rt_sigreturn`（139）陷入内核。选择固定地址而非栈上注入代码，避免了栈执行权限问题，但也要求内核在页表中映射该跳板页，且用户态不可修改。这是牺牲了灵活性（无法动态选择跳板位置）换取了安全性和简单性。

**信号帧复制而非引用**：`signal_frame_setup`（signal.c 第252-269行）使用 `copyout` 将整个 `struct ucontext` 和 `ucontext_t` 写入用户栈，`signal_frame_restore`（signal.c 第276-297行）则用 `copyin` 读回。这意味着每次信号处理都要在栈上复制完整的寄存器状态（trapframe 大小约 280 字节）。替代方案是在内核中保留一份副本并在 sigreturn 时直接恢复，但那样需要为每个待处理信号分配内核内存，且延长了持有锁的时间。当前方案用两次拷贝避免了内核态的内存生命周期管理。

**信号忽略路径的“懒删除”**：`signal_handle` 中，当 `sig_act.sa_handler == SIG_IGN` 时，代码直接 `continue` 而不删除队列项（signal.c 第144-146行注释）。这实际上是一个**未完成的实现**——被忽略的信号会一直留在队列中，`pending_cnt` 不减，导致 `signal_handle` 每次被调用都会空转遍历已忽略的信号。从设计角度看，这可能是为了后续支持 `sigwait` 而有意保留信号在队列中（等待函数可以解除忽略），但当前实现既未删除也未提供重新触发的机制，造成资源泄漏。

**锁的嵌套顺序**：`signal_send` 被注释要求“call with the thread‘s lock held”（signal.c 第356行），但函数内部又 `acquire(&t->sig_pending.siglock)`。这意味着调用者必须先持有了某个锁（可能是 `t->lock`），然后函数内再获取 `sig_pending.siglock`。`do_sigtimedwait` 则在持有 `sig_pending.siglock` 时调用 `thread_sleep` 释放锁（signal.c 第441行），醒来后重新获取。这种模式避免了持锁睡眠，但要求所有发送路径严格遵守相同的锁顺序，否则可能死锁。

### 3. 跨模块协同

信号模块是连接**异常/中断处理**与**用户态程序**的桥梁。当硬件中断或系统调用返回用户态前（通常在 `trap.c` 的 `usertrapret` 中），内核检查当前线程的 `pending_cnt` 并调用 `signal_handle`。`signal_handle` 根据动作类型执行：对于 `SIG_DFL` 直接调用 `signal_default` 以 `thread_setkilled` 结束线程（signal.c 第79-95行）；对于自定义处理器则通过 `setup_rt_frame` 修改 `trapframe` 的 `epc`（或 `era`）、`sp`、`ra`、`a0-a2` 等寄存器（signal.c 第197-211行），从而在用户态返回时直接执行信号处理函数。

信号处理返回时，`SIGRETURN` 跳板执行 `ecall` 进入 `sys_sigreturn`，该调用最终调回 `signal_frame_restore`。后者从用户栈的 `rt_sigframe` 中 `copyin` 回 `t->blocked`、`t->trapframe` 和 `sigprocessing`（signal.c 第280-296行），恢复执行原始程序的断点。这一过程依赖于 `vm.c` 的 `copyin`/`copyout` 通过进程页表进行地址转换，涉及到页表权限检查。

此外，`do_sigaction` 和 `do_sigprocmask` 被系统调用直接调用（如 `sys_sigaction`、`sys_sigprocmask`），它们通过 `mythread()` 获取当前线程，并操作 `t->sigs` 和 `t->blocked`。`do_sigtimedwait` 则使用了 `thread_sleep` 同步机制，将当前线程挂起直到目标信号到达或超时，这依赖于 `thread.c` 的调度模块。

### 4. 边角细节与不足

**`<sup>[1](#mod-signal-ref-1)</sup>` SIG_IGN 路径的内存泄漏**：`signal_handle` 中忽略信号时，删除和释放队列节点的代码被注释掉（signal.c 第144-146行），导致被忽略的信号节点永久驻留在 `sig_pending.list` 中，`pending_cnt` 不会递减。若频率性地发送被忽略的信号，内核堆将耗尽。

**`<sup>[2](#mod-signal-ref-2)</sup>` `sig_first_member` 空集越界**：宏 `sig_first_member`（signal.h 第281-288行）在集合为空时回退到循环外的 `i + 1`，其中 `i == 64`，返回 `65`，超出有效信号编号范围。`do_sigtimedwait` 用它获取信号号时未检查返回值，可能导致后续 `signal_handle` 收到非法信号号而 `panic`。

**`<sup>[3](#mod-signal-ref-3)</sup>` `signal_send` 中 killed 标记缺少锁保护**：对于 `SIGKILL`/`SIGSTOP`/`SIGTERM`，`signal_send` 在获取 `sig_pending.siglock` 之前直接设置 `t->killed = 1`（signal.c 第374行）。若 `killed` 字段受其他锁保护（如 `t->lock`），则此赋值存在竞态，可能丢失信号或写入过期值。

**`<sup>[4](#mod-signal-ref-4)</sup>` `get_sigframe` 缺少栈下溢检查**：`get_sigframe`（signal.c 第239行）计算 `sp -= framesize` 时不检查 `sp` 是否小于 `framesize`，对于无效栈指针（例如 `sp=0`）可能产生巨大的正数地址。虽然注释提到了“bogus address”但代码并未实现上溢/下溢检测，攻击者可构造恶意 `sp` 劫持信号帧到任意内核地址。

**`<sup>[5](#mod-signal-ref-5)</sup>` trapframe 拷贝泄露内核敏感状态**：`signal_frame_setup` 将整个 `trapframe`（包含 `kernel_sp`、`kernel_trap` 等内核栈指针）复制到用户可读的栈上（signal.c 第258行）。虽然用户态通常无法访问刚写入的栈帧（除非信号处理函数主动读取），但这违反了最小权限原则，属于内核信息泄露风险。

**`<sup>[6](#mod-signal-ref-6)</sup>` `do_sigtimedwait` 中释放锁后的竞态窗口**：从 `do_sigtimedwait` 的 `while` 循环退出到 `signal_handle` 调用之间，`sig_pending.siglock` 被释放（signal.c 第447行）后重新获取（在 `signal_handle` 内部第117行）。在此间隙中，其他核可能消费或修改目标信号，导致 `signal_handle` 返回 `-1` 或处理错误的信号。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该内核的进程间通信模块实现了管道（pipe）、futex、信号（signal）和信号量（semaphore）四种机制。核心抽象包括 `struct pipe`（环形缓冲区与自旋锁）、`struct futex`（用户地址与等待队列）、`struct sigaction`（信号处理函数与掩码）及 `struct semaphore`（整数值与条件变量）。与 xv6-riscv 原版相比，显著扩展了 futex（支持 FUTEX_WAIT/WAKE，但 WAIT_BITSET/REQUEUE/PI 等复杂操作为桩函数）和 POSIX 信号（通过 `sys_sigaction`/`sys_rt_sigprocmask`/`sys_sigtimedwait` 实现），并引入了基于条件变量的信号量。管道实现基本复用 xv6 设计，但将读写标志改为 flags 位操作。整体设计体现了教学内核“够用即可”的取舍，但存在多处未实现路径、锁使用不一致及潜在的数据竞争。

### 1. 核心抽象与外部依赖

进程间通信模块由四个子模块组成：管道、futex、信号和信号量。它们共享内核基础设施如自旋锁（`spinlock`）、线程调度（`thread_sleep`/`thread_wakeup_chan`）、内存分配（`kalloc`/`kmalloc`）以及用户态内存访问（`copyin`/`copyout`）。

**管道** (`kernel/ipc/pipe.c`) 以 `struct pipe` 为核心，包含一个 512 字节环形缓冲区、读写指针和两个打开计数。它通过文件系统 VFS 层暴露（`struct file`），`pipealloc` 分配两个 `file` 结构分别用于读端和写端，设置 `FD_PIPE` 类型和读写标志（使用 `SET_READABLE`/`UNSET_WRITABLE` 等宏代替原 xv6 的直接赋值）<sup>[1](#mod-ipc-ref-1)</sup>。读写在持有管道自旋锁的条件下进行，缓冲区满时写者睡眠在 `&pi->nwrite` 等待队列，空时读者睡眠在 `&pi->nread` 等待队列，关闭时唤醒对方<sup>[2](#mod-ipc-ref-2)</sup><sup>[3](#mod-ipc-ref-3)</sup>。

**futex** (`kernel/ipc/futex.c`) 实现了 Linux 风格的快速用户空间互斥。核心数据结构是 `struct futex`，包含一个（未实际使用的）自旋锁和一个等待队列。futex 实例通过哈希表组织，哈希键为用户虚拟地址（`uaddr`），每个哈希桶由链表串接 `struct futex_hash_node`。全局哈希表 `futex_hashtable` 的大小固定为 32（`FUTEX_NUM`）<sup>[4](#mod-ipc-ref-4)</sup>。操作接口 `do_futex` 根据 `futex_op` 分发到 `futex_wait` 或 `futex_wake`，其他复杂操作如 `FUTEX_WAIT_BITSET`、`FUTEX_REQUEUE`、`FUTEX_LOCK_PI` 均被注释为未实现<sup>[5](#mod-ipc-ref-5)</sup>。

**信号** (`kernel/ipc/syssig.c`) 提供 `sys_sigaction`、`sys_rt_sigprocmask`、`sys_sigreturn` 和 `sys_sigtimedwait` 系统调用。它们通过 `copyin`/`copyout` 在用户空间和内核 `struct sigaction`/`sigset_t`/`siginfo_t` 之间传输数据，并委托给 `do_sigaction`、`do_sigprocmask`、`do_sigtimedwait`（实现在 `kernel/signal.c`，未出现在片段中）<sup>[6](#mod-ipc-ref-6)</sup>。`sys_sigreturn` 从 trapframe 的 sp 指针读取信号帧，调用 `signal_frame_restore` 恢复上下文。

**信号量** (`kernel/atomic/semaphore.c`) 基于条件变量实现经典 Dijkstra 信号量。`struct semaphore` 包含整数值 `value`、唤醒计数 `wakeup`、自旋锁和条件变量。P 操作（`sema_wait`）递减 value，若为负则循环等待条件变量；V 操作（`sema_signal`）递增 value，若小于等于 0 则唤醒一个等待者<sup>[7](#mod-ipc-ref-7)</sup>。条件变量本身由 `kernel/cond.c` 提供（未给出片段，但通过 `cond_wait`/`cond_signal` 接口可知）。

### 2. 关键设计取舍

**2.1 管道：保留 xv6 环形缓冲区，但修改标志位处理**

原 xv6 使用 `readable`/`writable` 布尔字段，此实现将其替换为 `flags` 位操作（`SET_READABLE`/`UNSET_WRITABLE`），这可能是为了与统一文件标志体系兼容。但该改动未影响核心逻辑，环形缓冲区仍为 512 字节固定大小，读写指针为 `uint` 类型且未处理回绕——当 `nwrite` 溢出后，`pi->nwrite == pi->nread + PIPESIZE` 条件可能错误判定缓冲区满，这是原 xv6 遗留问题<sup>[8](#mod-ipc-ref-8)</sup>。

**2.2 Futex：轻量实现但锁模型存在隐患**

futex 的设计目标是最小化内核干预。`futex_wait` 首先通过 `copyin` 从用户地址读取当前值，若与期望值不等则直接返回 0（无需睡眠），否则获取或创建 `struct futex` 后修改线程状态为 `TCB_SLEEPING` 并加入等待队列<sup>[9](#mod-ipc-ref-9)</sup>。`futex_wake` 遍历等待队列唤醒最多 `nr_wake` 个线程，若队列变空则释放 futex 结构。然而，`struct futex` 中定义了自旋锁 `fp->lock`，但代码中从未获取它——所有对等待队列的操作（`queue_push_back`、`queue_pop_atomic`）均依赖哈希表锁来保护<sup>[10](#mod-ipc-ref-10)</sup>。注释“using p->lock is ok”暗示设计者认为可以通过进程锁代替，但实际上 `futex_wake` 可能和 `futex_wait` 并发访问同一 futex，而它们仅短时持有哈希表锁，释放后对 `fp->waiting_queue` 的操作便无保护，导致数据竞争。此外，`get_futex` 在持有哈希表锁时调用 `kmalloc`，若 `kmalloc` 可能睡眠（例如需要页分配），则违反自旋锁不可睡眠的规则<sup>[11](#mod-ipc-ref-11)</sup>。

**2.3 信号：部分 POSIX 支持，但存在复制粘贴 bug**

信号子系统实现了 `sigaction`、`sigprocmask` 和 `sigtimedwait`，但缺失 `kill`、`raise` 等生成信号的系统调用（可能在其他文件中）。`sys_sigaction` 中，当需要返回旧动作时，`copyout` 的源地址错误地传入了 `&oldact_addr`（指向用户地址的指针）而非 `&oldact`（内核栈上的结构体），导致拷贝的是指针值而非实际的 `oldact` 内容——这是一个典型的复制粘贴错误<sup>[12](#mod-ipc-ref-12)</sup>。同样问题也出现在 `sys_rt_sigprocmask` 中（`copyout` 使用 `&oldset` 正确，但 `sys_sigaction` 错误）。

**2.4 信号量：基于条件变量的简洁实现**

信号量使用条件变量和自旋锁，避免忙等待。`sema_wait` 中 `do { cond_wait(...); } while (s->wakeup == 0);` 是典型的条件变量使用模式，但 `cond_wait` 内部会释放锁并重新获取，因此循环检查 `wakeup` 是必要的<sup>[7](#mod-ipc-ref-7)</sup>。不过，`sema_wait` 在首次获取锁后递减 value，若 value 为负则等待，但释放锁后其他线程可能改变 wakeup，循环保证了正确性。该实现简单但有效。

### 3. 跨模块协同

- **管道与文件系统**：`pipealloc` 调用 `filealloc()` 分配两个 `struct file`，并将 `file.pipe` 指向同一个 `struct pipe`。文件读写操作通过 `fileread`/`filewrite` 分发到 `piperead`/`pipewrite`。管道关闭时通过 `fileclose` 调用 `pipeclose`，后者在双方都关闭后释放 pipe 内存。
- **Futex 与线程调度**：`futex_wait` 调用 `tcb_q_change_state(t, TCB_SLEEPING)` 和 `queue_push_back(&fp->waiting_queue, t)` 将线程挂起到指定等待队列，然后调用 `thread_sched()` 触发调度。`futex_wake` 则从队列中弹出线程并设置 `TCB_RUNNABLE`。这种机制与普通 `thread_sleep`/`thread_wakeup_chan` 不同，futex 维护了自己的等待队列，而非使用内核通用等待通道。
- **信号与陷阱处理**：信号处理需要与用户态栈帧交互。`sys_sigreturn` 从 `trapframe->sp` 读取 `rt_sigframe`，调用 `signal_frame_restore` 恢复寄存器，然后清除当前正在处理的信号掩码。这要求信号递送时内核在用户栈上构建了信号帧（由 `trap.c` 中的 `sig_handle` 完成，未在片段中）。
- **信号量与条件变量**：`kernel/atomic/semaphore.c` 依赖 `kernel/cond.c` 的条件变量原语。条件变量内部使用自旋锁和等待队列，与 futex 的等待队列复用同一套 `queue` 结构。

### 4. 边角细节与不足

1. **Futex 竞争条件 <sup>[10](#mod-ipc-ref-10)</sup>**：`struct futex` 的自旋锁 `lock` 未被任何代码获取，对 `waiting_queue` 的访问仅在 `futex_wait`/`futex_wake` 中通过哈希表锁保护，但这两个函数在哈希表锁释放后仍操作 `fp->waiting_queue`（`futex_wait` 在释放锁后调用了 `thread_sched`，而 `futex_wake` 在释放锁前操作队列），实际上在 `thread_sched` 返回后，`fp` 可能已被其他线程释放，造成 UAF。

2. **错误的内存释放逻辑 <sup>[13](#mod-ipc-ref-13)</sup>**：`futex_free` 调用 `hash_delete`，后者执行 `kfree(fp)` 和 `kfree(fhn)`。但 `futex_free` 仅在 `futex_wake` 中发现队列为空时调用，而 `futex_wait` 从未主动释放 futex。若一个 futex 被创建后没有线程 wake，或者 wake 时队列为空但之后又有线程 wait，会造成泄露。此外，`get_futex` 在失败路径上错误地释放了 `fp` 和 `fhn`，但若 `hash_insert` 失败（例如节点已存在），也会导致已插入的节点被释放。

3. **信号系统调用中的 copyout 错误 <sup>[12](#mod-ipc-ref-12)</sup>**：`sys_sigaction` 第 55 行 `copyout(..., (char*) &oldact_addr, ...)` 应改为 `(char*) &oldact`，这是明显的复制粘贴 bug，可能导致用户态获得无效的 `oldact` 数据。

4. **未实现的 futex 操作 <sup>[5](#mod-ipc-ref-5)</sup>**：`FUTEX_WAIT_BITSET`、`FUTEX_REQUEUE`、`FUTEX_CMP_REQUEUE`、`FUTEX_WAKE_OP` 及所有 PI 相关操作均被注释掉，仅返回 -1 或 panic。依赖这些功能的用户程序将无法正常工作。

5. **硬编码哈希表大小 <sup>[4](#mod-ipc-ref-4)</sup>**：`FUTEX_NUM` 固定为 32，当系统中有大量并发 futex 时哈希冲突严重，且哈希表不支持动态扩容。

6. **整数回绕风险 <sup>[8](#mod-ipc-ref-8)</sup>**：管道读写指针 `nread`/`nwrite` 为 `uint`，在持续读写后可能回绕，导致 `nwrite == nread + PIPESIZE` 条件失效。原 xv6 也存在此问题，未做改进。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架模块涵盖了xv6内核中设备驱动的抽象层与具体实现。核心抽象包括：(1) 基于主设备号的块设备开关表 bdev_table (device.c)，提供 register_bdev/unregister_bdev/bdev_open 接口；(2) 基于 virtio MMIO 的磁盘驱动 (virtio_disk.c)，使用描述符链与 QEMU 交互，支持 DMA 传输；(3) 适配 ext4 文件系统的块设备层 (blockdev.c)，桥接 ext4_blockdev 与 xv6 的 buf 缓存；(4) 16550A UART 驱动 (uart.c) 与 RISC-V PLIC 中断控制器驱动 (plic.c)。与原始 xv6 相比，本内核引入了多线程睡眠同步 (thread_sleep/wakeup)、ext4 支持以及 AHCI 驱动框架 (arch/loongarch)，但存在接口不统一（直接调用 vs 注册表）、锁遗漏、硬编码常量等问题。

### 1. 核心抽象与层次划分

驱动框架模块在xv6内核中呈现为三层结构：

- **设备开关表层**：`device.c` 定义了一个全局的 `struct bdev_ops *bdeventry[MAXBDEV]` 数组 (device.c:13-17)，通过 `register_bdev` / `unregister_bdev` 注册块设备操作接口，`bdev_open` 根据 inode 中的主设备号查表调用对应驱动。这是一个极简的设备抽象，仅支持按 major 号索引，minor 号透传给驱动 `open` 函数。
- **块设备适配层**：`blockdev.c` 通过 `EXT4_BLOCKDEV_STATIC_INSTANCE` 宏 (blockdev.c:32) 创建了一个 `ext4_blockdev` 实例，其 `bread`/`bwrite` 回调将 ext4 的块请求转换为对 xv6 缓存层 (`bread`/`bwrite`/`bget`) 的调用。这一层使得 ext4 文件系统能在 xv6 上运行，但 lock/unlock 回调为空实现 (blockdev.c:90-98)，存在并发风险。
- **具体硬件驱动层**：包括 virtio 磁盘驱动 (virtio_disk.c)、UART 驱动 (uart.c)、AHCI 驱动 (ahci_block_device.c) 以及 PLIC 中断控制器 (plic.c)。它们直接操作 MMIO 寄存器或 DMA 描述符，并注册中断处理函数与顶层调用接口。

值得注意的是，这三层之间存在断裂：virtio 驱动并不通过 bdev_table 注册，而是被文件系统 bio.c 直接调用（通过 `virtio_disk_rw`）；uart 驱动也不使用 bdev_table，而是被 console 子系统通过函数名直接调用。这导致两种并行的块设备访问路径：一条是 xv6 原生路径（bio.c → virtio_disk.c），另一条是 ext4 路径（ext4 → blockdev.c → buf.c → virtio_disk.c）。设计上未强制统一抽象，增加了维护负担。

### 2. 关键设计取舍

**2.1 内存映射 I/O 与 DMA 描述符管理**

virtio 驱动使用 MMIO 寄存器基址 `VIRTIO0` 操作设备，宏 `R(r)` 直接计算物理地址并强制转换为 `volatile uint32*` <sup>[1](#mod-drivers-ref-1)</sup>。这种裸指针访问在内核中普遍存在，但未做有效性校验（依赖平台布局已知）。描述符链采用固定三个描述符（命令头、数据块、状态字节），通过 `alloc3_desc` 分配三个连续索引（virtio_disk.c:100-110），若分配失败则睡眠等待。`free_chain` 沿 `next` 指针释放整个链（virtio_disk.c:80-91），但未处理环形链表或重复释放。

**2.2 锁与中断同步模型**

virtio 驱动使用自旋锁 `vdisk_lock` 保护整个 `disk` 结构体。在 `virtio_disk_rw` 中，分配描述符时如果失败，调用 `thread_sleep(&disk.free[0], &disk.vdisk_lock, NULL)`释放锁并睡眠；而在中断处理函数 `virtio_disk_intr` 中，获取锁后遍历 used ring，并通过 `thread_wakeup_chan(b)` 唤醒等待的线程 <sup>[2](#mod-drivers-ref-2)</sup>。这种“锁内睡眠-中断中唤醒”模式要求自旋锁在睡眠前释放，中断中获取锁时不发生死锁。从 spinlock.c 可以看到，`acquire` 先调用 `push_off` 关闭中断，防止中断处理函数与持有锁的代码造成死锁 <sup>[3](#mod-drivers-ref-3)</sup>。但 `thread_sleep` 内部需要重新获取锁（在唤醒后），因此锁的释放与重获必须与 `thread_sleep` 原子配合，`virtio_disk_rw` 的做法符合规范。

**2.3 中断路由与 PLIC**

`plic.c` 初始化时设置 UART0 和 VIRTIO0 中断优先级非零 (plic.c:11-13)，并在每个 hart 上启用这两个中断 (plic.c:18-20)。中断发生后，`plic_claim` 返回中断号 (plic.c:27-29)，`plic_complete` 通知完成 (plic.c:33-35)。这要求上层中断分发函数（如 `devintr`）正确调用 uartintr / virtio_disk_intr，并确保中断现场保存。

**2.4 UART 同步与异步混合输出**

uart 驱动提供了两种输出路径：
- `uartputc`：带缓冲、可睡眠，用于 write 系统调用。它使用 `uart_tx_lock` 保护缓冲区，当缓冲区满时调用 `thread_sleep` (uart.c:62-68)。
- `uartputc_sync`：轮询等待 THR 空，不获取锁，用于内核 panic 和早期初始化 (uart.c:85-92)。
这种设计兼顾了效率和可靠性，但 `uartputc_sync` 中不关闭中断（只调用 `push_off` / `pop_off`），在持有锁时可能被中断打断，但函数本身只读 LSR 和写 THR，无并发问题。

### 3. 跨模块协同

**3.1 文件系统与块设备**

xv6 原生文件系统通过 `buf.c` 的 `bread`/`bwrite` 调用 `virtio_disk_rw`。`virtio_disk_rw` 内部使用 `b->blockno` 计算扇区号（`sector = b->blockno * (BSIZE / 512)`）<sup>[4](#mod-drivers-ref-4)</sup>。当 I/O 完成后，`virtio_disk_intr` 将 `b->disk` 置 0 并唤醒等待线程。`blockdev.c` 则作为 ext4 的适配层，同样通过 `bread`/`brelse` 操作 xv6 缓存，实现了代码复用。

**3.2 中断处理与进程调度**

`virtio_disk_intr` 在中断上下文中调用 `thread_wakeup_chan(b)`，这要求 `thread_wakeup_chan` 在中断中安全（不获取可能被进程持有的锁）。从代码看，`thread_wakeup_chan` 内部操作等待队列，可能关闭中断，但中断中调用需确保不会嵌套。同样，`uartintr` 中调用 `consoleintr` 处理输入，并调用 `uartstart` 发送输出，这两者都可能访问全局数据结构。

**3.3 多平台 AHCI 驱动框架**

`ahci_block_device.c` 提供了 `block_read`/`block_write` 接口，通过 `block_make_request` 调用 `ahci_read`/`ahci_write`。该实现注释掉了请求队列和任务结构体，直接发命令 <sup>[5](#mod-drivers-ref-5)</sup>。这暴露了一个早期实现的问题：未使用中断，完全轮询等待。

### 4. 边角细节与不足

以下风险点按 [ref:N] 标记：

1. **<sup>[6](#mod-drivers-ref-6)</sup> 数组越界风险**：`free_desc` 函数检查 `i >= NUM` 但未检查 `i < 0`。若传入负数，`disk.free[i]` 会越界访问（`disk.free` 是 char 数组，位于结构体开头，负索引可能访问结构体前的内容）。同样，`alloc_desc` 返回 -1 时调用者使用前会检查，但 `free_chain` 直接使用来自 `disk.desc[i].next` 的值，若设备返回恶意值或链中有循环，可能导致无限循环或越界。
2. **<sup>[7](#mod-drivers-ref-7)</sup> 锁保护遗漏**：`virtio_disk_rw` 中在释放锁之前调用了 `__sync_synchronize()` 和 `*R(VIRTIO_MMIO_QUEUE_NOTIFY) = 0;`，但 `R` 宏访问的 MMIO 寄存器未受锁保护（锁仅保护数据结构，不保护 MMIO 访问）。不过 MMIO 原子性由硬件保证，但写寄存器与数据结构更新之间的顺序依赖 `__sync_synchronize` 保障。更严重的是，`disk.avail->ring[disk.avail->idx % NUM] = idx[0];` 和 `disk.avail->idx += 1;` 位于加锁区域内，但 device 可能通过 DMA 读写 `avail` 结构，若锁未正确保障 DMA 一致性，可能出现竞争。
3. **<sup>[8](#mod-drivers-ref-8)</sup> 桩函数与并发风险**：`blockdev.c` 中 `blockdev_open`、`blockdev_close`、`blockdev_lock`、`blockdev_unlock` 均为空实现（`return EOK`）。`lock`/`unlock` 为空意味着 ext4 文件系统在多线程并发访问时缺乏保护，可能导致元数据损坏。虽然 xv6 文件系统本身可能只在一个线程中运行，但 ext4 代码可能期望锁有效。
4. **<sup>[9](#mod-drivers-ref-9)</sup> 硬编码队列深度**：`virtio.h` 中 `#define NUM 8`。该值决定了最大并发请求数，如果文件系统同时发起超过 8 个请求，`alloc_desc` 将持续失败导致睡眠。但 xv6 的 `NBUF` 通常也为 30，`NUM` 设为 8 可能成为瓶颈。且该值未在初始化时从设备能力（`VIRTIO_MMIO_QUEUE_NUM_MAX`）检查兼容性——如果设备支持少于 8 会 panic，但支持多于 8 也只用 8。
5. **<sup>[10](#mod-drivers-ref-10)</sup> 中断上下文调用 `thread_sleep` 风险**：`uartputc` 在缓冲区满时调用 `thread_sleep`，但该函数仅用于进程上下文（write 系统调用）。如果未来在中断中误用，会导致死锁。代码注释已说明“因可能阻塞，不可从中断调用”，但未做断言或运行时检查。

总体而言，驱动框架模块为 xv6 提供了丰富的硬件支持，但接口设计沿用 xv6 直接调用风格，尚未形成统一的设备驱动模型，锁与同步机制虽有原始创新（如线程睡眠原语），但仍存在若干早期实现的粗糙点。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层核心抽象是静态系统调用表 `syscalls[]`，通过 `#include "sysfunc.h"` 在编译期生成（`syscall.c` 第110-113行），由 `syscall()` 函数根据 `trapframe->a7` 中的系统调用号分发（第121-127行）。参数提取通过 `argraw()` 从 trapframe 的 a0-a5 寄存器读取（第22-39行），提供 `argint`/`argaddr`/`argstr` 等类型安全封装。与 xv6-riscv 原版相比，本项目显著扩展了 Linux 兼容性：新增 clone/futex/tkill/tgkill/exit_group/prlimit64/nanosleep/clock_nanosleep/set_tid_address 等 20+ 系统调用，引入线程组和进程组概念，并支持 futex 同步原语。具体实现在 `sysproc.c`（进程控制类）中完成，通过 `syscall.c` 统一分发。

### 1. 核心抽象：编译期系统调用表与寄存器参数传递

系统调用层的核心骨架由 `kernel/syscall.c` 完成。

**系统调用表**采用编译期生成的方式：`syscalls[]` 数组通过 `#include "sysfunc.h"` 填充（`syscall.c` 第111-113行）<sup>[1](#mod-syscall-ref-1)</sup>，该头文件由构建脚本生成或手动维护，包含所有系统调用函数的函数指针。`syscall()` 分发函数（第117-136行）从 `tcb->trapframe->a7` 读取系统调用号，在边界检查 `num > 0 && num < NELEM(syscalls) && syscalls[num]` 后调用对应函数，返回值写入 `trapframe->a0`（第127行）<sup>[2](#mod-syscall-ref-2)</sup>。这种设计与 xv6-riscv 原版一致，但差异在于使用线程控制块 `tcb` 而非 `proc` 作为当前执行上下文——体现了本项目对多线程的支持（`mythread()` vs `myproc()`）。

**参数提取**通过 `argraw()` 函数（第22-39行）实现，它根据参数索引 n 从 `t->trapframe->a0`~`a5` 中读取原始值。上层封装 `argint()` `argaddr()` `argstr()` 等函数提供类型安全接口。值得注意的是，`arguint64()` 和 `argulong()` 的引入（第76-78行）表明本项目需要处理 64 位无符号参数（如文件系统地址或 `clone` 的 `tls` 指针），而 xv6 原版仅提供 `argint` 和 `argaddr`。`fetchaddr()` 和 `fetchstr()` 则通过 `copyin`/`copyinstr` 从用户空间读取数据，依赖进程页表进行地址转换。`fetchaddr` 在访问前做了明确的溢出检查：`if(addr >= p->sz || addr+sizeof(uint64) > p->sz)`（第13行）<sup>[3](#mod-syscall-ref-3)</sup>，这是应对恶意或错误用户地址的标准做法。

### 2. Linux 兼容性扩展的设计取舍

`kernel/sysproc.c` 实现了大量 xv6 原版没有的 Linux 兼容系统调用，反映出项目定位从教学内核向实用内核的演进。

**线程组与进程组支持**：通过 `clone` 系统调用（第342行）创建线程，返回 `do_clone()` 的结果；配套实现了 `sys_gettid`（第537行）、`sys_getpgid`（第550行）、`sys_setpgid`（第585行）和 `sys_tkill`/`sys_tgkill`（第430行、第441行），这些函数在 xv6 原版中完全不存在。实现上的一个关键取舍是：`clone` 的 `flags` 参数直接透传给 `do_clone`，但 `sys_clone` 的参数布局与 Linux 略有不同（`parent_tid`、`tls`、`child_tid` 的顺序与 Linux 的 `clone` 不一致），这可能导致用户态兼容性问题。

**睡眠类系统调用的冗余**：项目同时实现了 `sys_sleep`（第390行）、`sys_nanosleep`（第85行）和 `sys_clock_nanosleep`（第36行）三个功能重叠的接口。`sys_sleep` 以 10 倍 tick 为单位进行睡眠，是 xv6 遗留接口；`sys_nanosleep` 使用 `TIMESPEC2TICKS` 转换时间；`sys_clock_nanosleep` 则额外支持 `clock_id` 和 `flags` 参数。代码复用不足：三个函数都独立实现了 while 循环 + `thread_sleep` 的等待逻辑，没有提炼出公共内核函数。特别是 `sys_clock_nanosleep` 中处理 `req.tv_sec == INT_MAX` 的逻辑（第74行）<sup>[8](#mod-syscall-ref-8)</sup> 直接修改请求时间并标记 `is_invalid`，最后调用 `thread_exit(0)` 而非返回正确错误码，这种异常路径的处理方式非常危险。

**资源限制接口**：`sys_prlimit64`（第500行）通过 `do_prlimit` 操作 `proc` 结构体中的 `rlim` 数组，并借助 `rlim64_to_rlim`/`rlim_to_rlim64` 函数在 `rlimit`（32 位兼容）和 `rlimit64`（64 位原生）之间转换。这一兼容层增加了代码复杂度，但换取了对 64 位资源上限的支持（如 `RLIM64_INFINITY`）。

### 3. 跨模块协同：trapframe、线程调度与页表

系统调用层是用户态与内核态的桥梁，其正确运作依赖多个子系统的紧密配合：

**与 trap 处理器的协同**：当用户态执行 `ecall` 指令时，trap 处理器（`usertrap`）将用户寄存器保存到 `trapframe`，设置 `a7` 为系统调用号，然后调用 `syscall()`。`syscall()` 函数的返回值被写入 `t->trapframe->a0`（第127行），当 `usertrap` 返回时，`sret` 指令从 `trapframe` 恢复寄存器，从而将返回值传递回用户态。这是 xv6 的标准约定，但本项目通过 `tcb->trapframe` 而非 `proc->trapframe` 访问，支持每个线程独立的 trapframe。

**与线程调度器的协同**：多个系统调用（`sys_nanosleep`、`sys_clock_nanosleep`、`sys_sleep`、`sys_futex`）调用 `thread_sleep(&ticks, &tickslock, NULL)` 实现阻塞等待。`thread_sleep` 在释放 `tickslock` 的同时让出 CPU，被唤醒后重新获取锁。这里的关键设计是：`tickslock` 必须在中断关闭下持有，因为 `ticks` 变量在定时器中断处理函数中递增（通常在 `devintr` 或 `clockintr` 中），如果不关中断则可能发生数据竞争。代码中 `acquire(&tickslock)` 在 spinlock 实现内部会关中断，但调用 `thread_sleep` 前需要确保锁被持有且中断已关闭。

**与内存管理的协同**：系统调用从用户空间读取结构体（如 `timespec`、`rlimit64`）时，统一使用 `copyin(p->mm.pagetable, ...)` 通过进程页表进行地址翻译（第46行、第98行等）。写入返回数据则使用 `copyout`。这一接口抽象了页表遍历逻辑，但需要调用方确保 `pagetable` 字段在系统调用执行期间保持有效——在 `sys_getpgid` 和 `sys_setpgid` 中，通过 `acquire(&p->lock)` 持有目标进程锁来确保这一点。

### 4. 边角细节与不足

尽管系统调用层整体框架较为健壮，但代码中存在多处早期实现的痕迹和安全性缺陷：

**缺陷一：`sys_getpgid` 越界访问** <sup>[4](#mod-syscall-ref-4)</sup>。第557行 `p = &proc[pid]` 在范围检查 `if (pid < 0 || pid >= NPROC)` 之前执行。若传入负数或大于等于 `NPROC` 的 pid，该行代码将访问 `proc[]` 数组的越界地址，可能造成内核崩溃或信息泄露。正确的做法是先将 `pid` 合法性检查前置，或使用安全索引宏。

**缺陷二：`sys_setpgid` 重复获取锁** <sup>[5](#mod-syscall-ref-5)</sup>。第601行 `acquire(&p->lock)` 之后，第604行又再次调用 `acquire(&p->lock)`，且中间未释放。这会导致自旋锁递归死锁（xv6 的 spinlock 不支持递归持有）。更可能的原因是复制粘贴遗漏：第一个 `acquire` 是多余的，应当删除。
