## 📖 一、总览

本仓库是 ArceOS-Starry 家族中一个面向 Linux 二进制兼容的实验性内核实现。它在 ArceOS 模块化基线（axhal、axdriver 等）之上，自主实现了进程管理、内存管理、信号系统、文件系统、futex、网络 syscall 入口和系统调用分发层，核心定位是在单核无中断协作式调度环境下以最小化代码变更模拟 Linux ABI，而非构建完整的内核分离层。整体架构呈现“薄下层 + 厚中层”特征：底层启动和驱动完全沿用 ArceOS 基线，中层由 VOS 组件（如 MemoryState、Process、FutexTable、VfsRoot）组合出 Linux 语义，上层通过一个约 120 臂的 syscall 分发器将 150+ 系统调用路由到各子模块。最值得评审注意的三个特点是：(1) 进程级内存台账的精细化设计——特别是 brk 堆扩缩对 loongarch64 musl 的特殊路径（mm.ref:4）和 System V 共享内存的全套实现；(2) 信号机制的完整生命周流——从信号投递、队列阻塞、用户态 handler 注入到 rt_sigreturn 的帧恢复，全部在单核调度器内编排，并包含了 SA_RESTART、线程组停止等 Linux 语义（signal.ref:1）；(3) 系统调用层的架构解耦与内核级等待原语——通过 ArchSyscallAbi trait 将架构差异限制在 arch 目录，并实现了通用 signal 感知阻塞原语（syscall.ref:3），同时为 eventfd、timerfd 等高级 syscall 配置了轻量 probe 策略。

## 🎯 二、综合评价

### 整体定位

本仓的架构定位可以用“兼容性替换实验”来概括。它没有在 ArceOS 之外另起炉灶，而是在保持 axhal、axdriver 等底层模块不变的前提下，替换和扩展了内核中最复杂的中层：进程模型、内存管理、信号、文件系统、futex 和网络 syscall 入口。这种设计选择意味着该仓的工作重点集中在“用户可见”的 ABI 兼容，而非内核内部的重构。与其说它是一个操作系统内核，不如说它是一个 **Linux 系统调用的模拟层**，运行在 ArceOS 提供的单核无中断运行环境之上。因此，该仓最适合用于快速原型验证和教学演示，而不适合直接部署到多核生产场景。

### 真正的创新点

尽管整体思路是模拟 Linux，但该仓在细粒度设计上展现出几个值得关注的独特之处。

- **brk 对 loongarch64 musl 的适配**：`mm.ref:4` 展示了 syscall/memory.rs 中硬编码的 `MUSL_ERRNO_OFFSET_FROM_TP = 0xa4`，使 brk 在 loongarch64 上能够正确设置用户态 errno。这种直接操纵 libc 私有数据的做法在传统 OS 实现中很少见，体现了为特定 libc ABI 深度定制的工程智慧，但也带来 ABI 版本脆弱性。

- **信号单帧恢复限制**：`signal.ref:3` 指出本仓仅维护一个信号恢复帧，不实现嵌套帧栈。这意味着在信号 handler 执行期间到达的另一个信号会直接覆盖恢复帧，简化了调度器的状态机，但牺牲了 POSIX 要求的可重入性。这是一个明确的“教学优先”取舍——单帧足以演示信号机制，但无法支撑复杂信号密集型应用。

- **Futex 的 Crowd Timeout Deferral**：`ipc.ref:1` 展示了 futex 模块中硬编码的批量超时发布阈值（128 个 waiter / 120 个超时 / 1秒间隔）。这种设计旨在避免“惊群超时”导致调度器开销激增，是一种原创的拥堵控制策略，虽然参数尚不可配置。

此外，`syscall.ref:3` 中定义的通用信号感知阻塞原语 `InterruptibleBlock`，将 waitqueue、signal 检查和超时统一到一个循环中，是 syscall 层复用率最高的基础设施。

### 取舍判断

本仓在工程资源分配上展现出明确的优先级：

- **重点投入**：进程管理（fork/execve/wait）、内存管理（mmap/brk/mremap）、信号系统、futex。这些模块的代码量最大，且均有真实状态机和完整的 syscall 入口。
- **刻意简化**：网络协议栈——本仓不实现 TCP/UDP 协议，而是将 socket syscall 转发到 axnet 或创建模拟套接字；文件系统——采用 ramfs + ext4 组合，但 O_TMPFILE 和 openat2 等高级接口仅定义 stub；BPF 和 seccomp——前者全局共享状态无隔离，后者用 7-syscall 白名单模拟。
- **完全沿用**：启动流程（axhal）和驱动框架（axdriver），模块摘要明确表示为“基线沿用”。

这种取舍反映出该仓的设计者将精力集中在“最经常被用户感知”的 syscall 上，而那些在内核内部优化或硬件相关的部分则选择信任 ArceOS 基线。

### 完成度与不足

从功能覆盖来看，该仓实现了 Linux 兼容内核的支柱：进程生命周期、地址空间操作、信号投递、文件系统、futex 同步、socket syscall 入口。核心路径（如 mmap、fork、nanosleep、futex_wait）能够工作，基本可以运行 musl libc 编译的静态链接 ELF。

然而，审视每个模块的“不足/疑问”列表会发现大量半成品：

- 内存管理：惰性分配函数已定义但未启用（`mm.ref:5`）；mlock 仅记录台账不锁定物理页；`MUSL_ERRNO_OFFSET_FROM_TP` 无版本校验。
- 信号：x86_64 下 default_signal_restorer 返回 None，导致用户自定义 handler 无法使用（`signal.ref:6`）；单帧限制下 handler 执行期间的第二个信号可能破坏恢复现场。
- 系统调用层：`dispatch_minimal_syscall` 中多处 unsafe 块缺少安全注释（`syscall.ref:2`）；BPF 全局数组 `BPF_STATE` 可能因 UnsafeCell 非 Copy 导致编译失败（`ipc.ref:5`）。
- 网络：`write_sockaddr_un` 的 unsafe 块无安全注释，存在 padding 泄露风险（net.ref:8）；sendto/recvfrom 不支持 flags。

代码质量方面，整体组织清晰，模块间耦合良好，但安全注释缺失、硬编码常量（0x2000 等）和未及时清理的 workload 专用注入点（signal.ref:12）拉低了工程标准。

综合评分：
- 完成度：2（核心功能可用，但细节缺失较多）
- 创新性：2（在模拟框架下有局部原创设计）
- 代码质量：2（结构良好但安全和文档需加强）

这个分数反映了该仓作为“实验性原型”的合理水平：它不是一个产品级实现，但很好地说明了如何在单核无中断内核上构建 Linux 兼容层。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块实现了用户态内存管理的核心抽象与系统调用入口。核心抽象为 `MemoryState` 进程级台账，包含 `UserVmLayout`、`BrkState`、`MmapRegion` 和 `MappingSource` 枚举，覆盖 brk 堆扩缩、匿名/文件 mmap、System V 共享内存等。与 ArceOS 基线相比，独特点包括：完整的 brk 扩缩（含 loongarch64 musl 特殊路径）、MAP_SHARED 文件映射的 Drop 时写回、System V 共享内存全套（shmget/shmat/shmdt/shmctl），以及 mlock/munlock 等辅助系统调用的台账骨架。但惰性分配、mremap 固定地址、mlock 物理锁定等仍为半成品。

### 1. 进程内存台账：从布局到映射的综合视图

用户态内存管理的基础是每个进程拥有一份独立的虚拟地址空间描述。`kernel/src/task/memory.rs` 定义了 `UserVmLayout`、`BrkState`、`MmapRegion` 和 `MemoryState` 等数据类型，共同构成进程级内存台账。

`UserVmLayout` 记录 exec 后用户空间的四个关键地标：`image_end`、`heap_base`、`mmap_base`、`stack_bottom`/`stack_top`。这些值在进程创建时由 exec 模块设定，后续 brk 和 mmap 都以此为基准。特别地，`mmap_base` 被设置在堆区和栈区中间，给 brk 留出连续增长空间 <sup>[1](#mod-mm-ref-1)</sup>。

`BrkState` 封装了堆的 base、current 和 limit。其 `growth_range` 和 `shrink_range` 方法将请求的对齐后地址转换为实际需要映射或解除映射的范围，避免每次都进行完整页表扫描 <sup>[2](#mod-mm-ref-2)</sup>。

`MmapRegion` 记录了每段通过 mmap（或堆扩展）建立的映射的起始、长度、保护位、标志和来源。来源由 `MappingSource` 枚举描述，分为 `Anonymous`、`FileBacked`、`SharedFileBacked`、`SysvShared` 四类。这种设计使得在 munmap/mprotect 时可以精准地区分哪些区域需要特殊处理（如共享文件写回标记、System V 段生命周期管理）。

`MemoryState` 聚合了所有信息：布局、堆状态、映射区域列表、锁定范围、present 范围、System V 段和附件。它通过 `overlay_region` 方法实现区域插入与自动分割，确保映射台账与页表状态一致 <sup>[3](#mod-mm-ref-3)</sup>。

### 2. 系统调用实现中的关键设计取舍

`kernel/src/syscall/memory.rs` 将上述台账暴露为 Linux 系统调用。

**brk 实现**：`apply_brk` 函数通过两步完成堆扩缩：先计算增长/收缩的范围（`growth_range`/`shrink_range`），然后直接调用 `aspace.map_alloc` 或 `aspace.unmap` 操作页表，最后更新 `BrkState`。这一步没有使用额外的锁，因为操作在 `with_memory_state` 闭包内完成，且 `aspace` 操作本身是线程安全的。对于 loongarch64，还提供了 `syscall_vos_musl_brk` 和 `syscall_vos_musl_sbrk`，通过计算 `tp - 0xa4` 将 errno 写入用户态 TLS，以兼容 musl libc 的特别约定 <sup>[4](#mod-mm-ref-4)</sup>。

**mmap 与惰性分配**：匿名 mmap 通过 `aspace.map_alloc` 立即分配物理页面，但代码定义了 `should_lazy_large_private_anon_mmap` 函数，当请求大小超过 1MB、私有且可写时，理论上应当延迟到缺页时分配。然而，当前实现并未真正启用惰性——页表映射仍然立即建立，物理页面在缺页时可能由 `axmm` 的缺页处理来分配，但台账侧没有体现延迟决策 <sup>[5](#mod-mm-ref-5)</sup>。这导致大匿名映射会立即占用物理内存，与 Linux 默认行为不符。

**共享文件映射写回机制**：`MappingSource::SharedFileBacked` 对应的 `SharedFileMapping` 在 `Drop` 中执行同步写回。它通过 `pages.snapshot_prefix` 获取页面快照，然后循环调用 `node.write_at` 将数据写回文件 <sup>[6](#mod-mm-ref-6)</sup>。这种设计避免了在系统调用路径上直接执行文件 I/O，但将写回延迟到最后一个引用释放的时刻。如果映射被长时间持有，写回可能推迟，且 Drop 发生在任意上下文（如信号处理），可能引入延迟敏感问题。

**mremap 限制**：`MREMAP_SUPPORTED` 只包含 `MREMAP_MAYMOVE`，`MREMAP_FIXED` 和 `MREMAP_DONTUNMAP` 直接返回 EINVAL <sup>[7](#mod-mm-ref-7)</sup>。即使对于 `MAYMOVE`，移动操作也仅支持匿名私有映射（`find_exact_anon_private_region`），共享或文件映射的移动未实现。

**mlock 家族**：`mlock`、`munlock`、`mlockall`、`munlockall` 仅更新 `MemoryState` 中的 `locked_ranges` 台账，未调用任何底层页面锁定原语 <sup>[8](#mod-mm-ref-8)</sup>。这意味着锁定仅存在于软件记账层面，物理页面仍可能被回收。`lock_current_mappings` 在 exec 后调用，遍历所有映射区域并记录锁定范围，但并未实际固定页面。

### 3. 跨模块协同：与 axmm、axhal 和 VFS 的接口依赖

内存管理系统不直接操作页表，而是通过 `Thread::with_aspace` 访问 `axmm::AddrSpace`。`aspace.map_alloc`、`map_shared`、`unmap`、`protect` 等方法封装了页表级别的操作。`MappingSource` 中的 `SharedPages` 来自 `axmm` crate，表示一段可以被多个地址空间共享的物理页面集合。

当创建 MAP_SHARED 文件映射时，`map_shared_file` 函数将每个 4K 页面转换为一个 `Arc<SharedFileMapping>`，并依次调用 `aspace.map_shared` 将同一组页面映射到多个地址空间。这实现了写时共享（COW）和回写同步。

与 VFS 的交互体现在共享文件映射的回写：`SharedFileMapping::drop` 调用 `node.write_at`，需要 `axfs_vfs` 层支持。此外，`MmapFileBacking` 结构体包含 `VfsNodeRef`，在 map_shared_file 中通过 `backing.data` 读取文件内容。

System V 共享内存的 sysv_shm 段使用 `SysvShmSegment` 中的 `Arc<SharedPages>` 作为物理页面持有者。`shmat` 通过 `aspace.map_shared` 将同一组页面映射到多个进程地址空间，而段的生命周期由 `MemoryState` 中的引用计数和 `marked_for_removal` 标志管理 <sup>[9](#mod-mm-ref-9)</sup>。

### 4. 边角细节与不足

当前实现中存在多处早期或不完整的痕迹，以下列出关键风险点：

<sup>[4](#mod-mm-ref-4)</sup> **`MUSL_ERRNO_OFFSET_FROM_TP` 的硬编码假设**：`0xa4` 偏移量写死在代码中，如果 musl 版本更新导致线程指针布局变化，将直接导致 errno 设置错误。该常量仅在 loongarch64 下编译，其他架构不受影响。

<sup>[5](#mod-mm-ref-5)</sup> **惰性分配仅停留在定义层面**：`should_lazy_large_private_anon_mmap` 函数已定义，但 `syscall_mmap` 中对于匿名映射并未使用该判断结果来推迟页面分配。实际效果是超过 1MB 的私有匿名映射仍会立即触发物理页面分配，违背了 Linux 对较大映射采用惰性分配以减少物理内存压力的期望。

<sup>[8](#mod-mm-ref-8)</sup> **mlock 只有记账没有实际的页面锁定**：`lock_range` 和 `unlock_range` 仅更新 `MemoryState.locked_ranges` 向量，没有调用页表级的锁定原语（如 `mlock` 对应的内核锁定操作）。对于依赖 `mlock` 保证页面不被换出的实时应用，此实现会​​直接​​失效。

<sup>[6](#mod-mm-ref-6)</sup> **共享文件写回可能在不合适的上下文中执行**：`SharedFileMapping::drop` 中的 `write_at` 调用是同步文件 I/O，如果最后一个引用在中断上下文或持有内部锁时被释放，可能导致阻塞或死锁。代码中没有采用异步或延迟写回机制。

<sup>[7](#mod-mm-ref-7)</sup> **mremap 功能极度受限**：仅支持 `MREMAP_MAYMOVE`，且移动仅针对匿名私有映射。`MREMAP_FIXED` 和 `MREMAP_DONTUNMAP` 直接返回 EINVAL，与 Linux 兼容性差距较大。

<sup>[10](#mod-mm-ref-10)</sup> **`present_ranges` 更新不全面**：`mark_present_range` 仅在 brk 增长和 mmap 时被调用，用户态缺页处理路径未更新该台账。因此 `mincore` 系统调用（依赖 `present_vec`）可能返回过时的 present 状态，导致用户态误判页面驻留情况。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块基于 ArceOS 无中断内核环境，实现了最小 Linux 兼容进程模型与调度层。核心抽象为 `Process` 与 `Thread`，通过 `Arc` 共享状态，并配合全局的 `ReadyThreadQueue`、`StoppedThreadQueue` 及 `InterruptibleBlockQueue` 实现阻塞与唤醒。与 ArceOS 原版相比，引入了更完整的 Linux ABI 模拟（如进程组、信号、wait、epoll、pipe、socket 等），但当前聚焦于单核 bring-up 场景，调度为简单 FIFO，许多功能以 stub 或最小状态机替代。

### 1. 核心抽象与外部依赖

进程模型的核心是 `Process` 与 `Thread` 结构（定义于 `mod.rs` 及子模块）。`Process` 持有退出状态、资源限制、能力集、信号状态、fd 表等，通过 `Arc` 多线程共享。`Thread` 持有进程引用、trap frame、线程特定状态，是调度基本单位。调度层使用三个全局静态队列管理线程状态：`READY_THREADS` (可运行)、`STOPPED_THREADS` (作业控制停止)、`INTERRUPTIBLE_BLOCKS` (可中断阻塞，如 sleep、poll、wait)。队列操作通过 `UnsafeCell<Vec<Thread>>` 进行，配合 `Ordering::Release/Acquire` 实现轻量同步。

外部依赖包括 `axfs_vfs` (文件系统句柄)、`axnet` (TCP/UDP socket)、`axsync::Mutex`、`axstd::collections::VecDeque`、`axhal::time`、`axhal::context`、`axmm::AddrSpace` 等。所有打开资源抽象为 `OpenResource` 枚举，统一了 `Stdio`、`VfsNode`、`Pipe`、`Socket`（含真实套接字与模拟 stub）、`EventFd`、`Epoll`、`PidFd` 等类型。

### 2. 关键设计取舍

- **全局队列与单核假设**：调度队列定义为全局静态，没有 per-CPU 分区，驱动依赖禁用中断或单核串行化。若未来支持多核，需引入 per-CPU 队列或 spinlock，当前设计将不适用（见 `<sup>[1](#mod-task-ref-1)</sup>`）。
- **UnsafeCell 同步**：队列内部用 `UnsafeCell` 并通过 `unsafe impl Sync` 标记同步，但未提供显式互斥。实际正确性依赖调用者保证同一时间仅一个执行上下文（如中断处理或系统调用）访问队列。`// SAFETY:` 注释缺失，构成技术债务（见 `<sup>[2](#mod-task-ref-2)</sup>`）。
- **一体化阻塞框架**：`BlockKind` 枚举覆盖了所有可中断阻塞场景，提供统一入口 `park_interruptible_block`。这简化了代码，但扩展新阻塞类型必须修改枚举，且每个阻塞类型需处理信号唤醒。缺点是如果某类型有特殊中断处理（如 futex），会混入此框架（见 `<sup>[3](#mod-task-ref-3)</sup>`）。
- **模拟 stub 增强兼容性**：对于无 NIC 或 unix socket，`InetStubSocket` 与 `UnixSocket` 提供了纯用户态模拟，使 LTP 等测试可不依赖硬件通过。代价是可能掩盖真实的网络问题，且模拟逻辑与真实 socket 路径分离（见 `<sup>[4](#mod-task-ref-4)</sup>`）。
- **Linux 常量硬编码**：大量 Linux ABI 常量（如 `O_*`、`SOCK_*`、`RLIMIT_*`、`CLONE_*`）在 `fd.rs` 和 `syscall/process.rs` 中直接定义，增加了平台相关性和维护成本。这些常量与 `axconv` 或 `linux-raw-sys` 可能重复（见 `<sup>[5](#mod-task-ref-5)</sup>`）。

### 3. 跨模块协同

- **系统调用与任务管理**：`kernel/src/syscall/process.rs` 处理 `getpid`、`execve`、`wait4`、`exit` 等系统调用，调用 task 模块函数（如 `wait_child_nonblocking`、`exit_thread_group`、`schedule_next_ready`）。`execve` 进一步调用 `exec` 模块的 `load_exec_image` 和 `install_exec`，说明 exec 与进程状态重置的协同模式（见 `<sup>[6](#mod-task-ref-6)</sup>`）。
- **信号交付与阻塞唤醒**：`wait.rs` 中的 `park_interruptible_block` 在阻塞前检查 `should_interrupt_interruptible_block`（来自 `signal.rs`），若信号挂起则返回 `EINTR`。外部信号交付（如 `deliver_local_signal`）会通过 `ReschedFlag` 或直接替换阻塞状态唤醒等待线程。这种信号与阻塞框架的耦合是模拟 Linux `EINTR` 语义的关键（见 `<sup>[7](#mod-task-ref-7)</sup>`）。
- **fd 操作与 VFS 集成**：`fd.rs` 中 `FdTable` 的 `read`/`write` 方法承接用户缓冲区与 `OpenResource`。对于 `VfsNode` 句柄，直接调用 `axfs_vfs` 的 `read_at`/`write_at`，偏移由 `VfsHandle` 管理。与 `axnet` 的 socket 操作类似，`SocketResource::Tcp` 等调用 `axnet::TcpSocket` 方法。模拟 stub 则提供纯内存实现（见 `<sup>[8](#mod-task-ref-8)</sup>`）。
- **wait 机制与全局结果缓存**：单线程 wait 中使用全局 `LAST_WAIT_OUTCOME` 快速传递子进程退出结果。`wait_child_nonblocking` 检查此缓存，如果命中则立即返回，否则父线程通过 `park_waiting_thread` 阻塞于 `InterruptibleBlockQueue`，待子进程 exit 时由 `wake_waiting_parent_for_exit` 唤醒（见 `<sup>[9](#mod-task-ref-9)</sup>`）。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 核心抽象是基于 axfs_vfs trait 的 VfsRoot 组合器，聚合 ext4、devfs、procfs、sysfs 和运行时 ramfs 构成多层可覆盖根文件系统。与 ArceOS 基线相比，引入了容量受限 ramfs 写保护、独立元数据管理层以补偿下层文件系统属性缺失，并为 glibc 运行时预置内建 locale/gconv 资产。

### 1. 核心抽象：VfsRoot 组合器

VfsRoot 结构体（rootfs.rs:42-49）持有五个文件系统句柄：持久化 ext4、设备 devfs、/proc procfs、/sys sysfs 和运行时 ramfs。这一设计放弃了单一文件系统树，转而采用显式叠加：路径查找时优先检查运行时 ramfs（覆盖层），然后依次查询其他虚拟文件系统和底层 ext4。<sup>[1](#mod-fs-ref-1)</sup> 每个文件系统类型实现 axfs_vfs 的 VfsOps 和 VfsNodeOps trait，使得 ext4 适配器可以独立于内核框架开发；事实上仓内定制了 Ext4FileSystem、ProcFileSystem 和 SysFileSystem，均未采用 ArceOS 原生的 fatfs 方案。<sup>[2](#mod-fs-ref-2)</sup>

### 2. 关键设计取舍：容量控制、元数据上提与覆盖语义

- **ramfs 容量限额**：全局原子计数器 `CAPACITY_LIMITED_RAMFS_IN_FLIGHT_BYTES`（rootfs.rs:16-18）配合 RAII 守卫 `CapacityLimitedRamfsWriteGuard` 实现写入字节计量，超过 512 MB 上限后拒绝分配。<sup>[3](#mod-fs-ref-3)</sup> 此设计在避免 OOM 的同时牺牲了性能，每次写操作都需 fetch_add/fetch_sub，对小块写入不友好，且未区分用户进程（全局计数器无法按 cgroup 隔离）。
- **元数据独立存储**：VfsRoot 内部的 `MetadataStore`（rootfs.rs:160-165）保存权限、所有权、时间戳及数据 extent 地图。底层 ramfs 和部分虚拟文件系统不维护完整 inode 属性，所以需要上层统一管理。但这也造成了第一次路径查找时需做元数据合成（touch_created_vfs_path），增加了 init 路径的复杂度。
- **白名单与符号链接**：`whiteouts` 和 `symlinks` 列表（rootfs.rs:57-58）独立于各个文件系统，使 ramfs 覆盖层可删除或重写 ext4 下的文件，实现类似 overlayfs 的效果。符号链接遍历深度限制为 40 次（rootfs.rs:17），是一个安全保护，防止恶意链接链导致内核栈溢出。
- **mount 管理**：`mounts` 列表（rootfs.rs:56-57）记录所有挂载点及其属性，路径查找时根据最长匹配前缀切换到对应文件系统。这与 Linux VFS 的 mount tree 类似，但简化为线性扫描且无缓存，高频访问时可能成为瓶颈。

### 3. 与系统调用层的集成

syscall/fs.rs 中的文件操作（openat、read、write 等）全部委托给 VfsRoot 暴露的 lookup_path、create_file、read 等方法。<sup>[5](#mod-fs-ref-5)</sup> 此外，系统调用层实现了 Linux 文件控制语义，包括 `O_DIRECT` 退化为缓冲 I/O（syscall/fs.rs:33-35）、`O_TMPFILE` 部分支持（syscall/fs.rs:41-42），以及完备的 advisory lock 子系统。`ADVISORY_LOCKS` 全局列表（syscall/fs.rs:174-191）配合进程等待/唤醒机制（park_advisory_lock、wake_advisory_lock_waiters）通过回调 `try_set_advisory_lock_for_waiter` 实现非忙等阻塞。<sup>[4](#mod-fs-ref-4)</sup> 这些锁操作与 VfsRoot 无直接交互，完全通过 i-node 号（从 VfsRoot 元数据获得）关联，保持了模块化。

### 4. 运行时资产的注入

runtime_assets.rs 通过调用 VfsRoot 的 mount 和 create_file 方法，在系统初始化时注入 glibc 运行时所需的 locale、gconv 模块和少量系统配置文件。<sup>[6](#mod-fs-ref-6)</sup> 这些资产以 `include_bytes!` 嵌入内核，构建出 `/glibc/usr/lib/locale`、`/glibc/lib/gconv` 等只读挂载点。自实现的 `RuntimeAssetDirNode` 和 `RuntimeAssetFileNode`（runtime_assets.rs:190-290）组合成纯内存文件系统树，避免了构建 squashfs 等镜像的复杂性，且对静态编译环境友好。值得注意的是，这些资产只在 `/glibc` 存在时才安装，否则静默跳过（runtime_assets.rs:66-69），体现了嵌入式场景下的条件部署设计。

### 5. 边角细节与不足

- <sup>[7](#mod-fs-ref-7)</sup> 硬件架构硬编码：`MUSL_ERRNO_OFFSET_FROM_TP` 等常量仅针对 riscv64（syscall/fs.rs:129-131），未提供 x86_64 或其他架构的 fallback，若编译时切换架构会直接编译失败。
- <sup>[8](#mod-fs-ref-8)</sup> 错误路径隐藏：`lookup_path` 调用后多处使用 `.map_err(|_| ...)` 掩盖底层错误细节（如 rootfs.rs:112），可能使调试困难；正常情况下应保留原始错误。
- <sup>[9](#mod-fs-ref-9)</sup> 并发锁设计：`metadata`、`mounts`、`symlinks` 各持有独立 Mutex（rootfs.rs:53-58），但路径查找中需访问多个全局状态且不持有顺序锁，可能产生不一致的视图；例如在检查 whiteout 的同时获取 mount 列表。
- <sup>[10](#mod-fs-ref-10)</sup> 死代码：syscall/fs.rs 定义了 `RESOLVE_NO_XDEV` 等 openat2 标志位（syscall/fs.rs:101-107），但当前未实现 openat2 系统调用，这些常量属于未使用的代码，会增加维护负担。
- 容量上限 `DEFAULT_RAMFS_CAPACITY_CEILING` 固定为 512 MB，不适用于低内存设备或需要动态调整的场景。

尽管存在上述早期实现痕迹，该文件系统模块已能够支撑 busybox 和 glibc 运行时，展示了丰富的覆盖语义和国产操作系统的实践。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 本模块在 ArceOS-Starry 中实现 Linux 兼容的最小信号运行时（signal runtime）。核心抽象是 `SignalSet`（u64 位图）、`SignalAction`（handler+flags+mask）和 `SignalFrame`（用户栈上的恢复帧）。模块解决了从信号投递、用户态 handler 注入、到 rt_sigreturn 恢复的完整生命周期，并集成进统一阻塞队列与调度器。与 ArceOS 基线相比，本仓不做简单的 SIG_DFL 终止，而是实现了线程组停止/继续、SA_RESTART、futex 打断、musl 线程取消信号抑制等 Linux 语义，且通过编译期条件（riscv64/loongarch64）区分了信号恢复 trampoline 的映射策略。

### 1. 核心抽象：位图驱动的信号状态机

信号模块的核心数据结构是 `SignalSet`（u64 位图）<sup>[1](#mod-signal-ref-1)</sup>。设计者选择用 u64 原生类型而不是 `[u64; 2]`（Linux 内核用 `unsigned long sig[2]` 支持 64 个信号），因为本项目只支持 1..=64 号信号（见 `SIGNAL_SLOTS = 64`），省去了跨字操作的复杂性。代价是将来若要支持实时信号扩展（>64）必须改数据结构。`PendingSignalInfo` 携带 `si_code`、`sender_pid`、`sender_uid`，存储在 `Thread` 的一个 `[Option<PendingSignalInfo>; SIGNAL_SLOTS]` 数组中——每个信号最多一个信息，不符合 Linux 的排队语义，但满足 POSIX 最小要求。

`SignalAction` 是线程组共享的策略表，存放在 `SignalState` 中。`SIG_DFL`（0）和 `SIG_IGN`（1）是特殊地址值，用户 handler 地址必须 > 1（由 `is_supported_handler` 检查）<sup>[2](#mod-signal-ref-2)</sup>。`SignalResumeAction` 枚举指导调度器在回用户态前的下一步动作：`ContinueUser`（无事发生）、`PreparedHandler`（已压帧，跳转 handler）、`StopGroup`（停止线程组）、`TerminateGroup`（终止线程组）。状态机的转换在 `prepare_signal_delivery` 中完成，该函数每次回用户态前被调用一次 <sup>[3](#mod-signal-ref-3)</sup>。

### 2. 关键设计取舍：最小运行时与 musl 深度耦合

**取舍一：单帧限制 vs. 嵌套交付。** 代码注释明确指出“当前最小 signal 运行时只跟踪一个 frame；已有 handler 未 rt_sigreturn 前，先延后其他 pending signal” <sup>[3](#mod-signal-ref-3)</sup>。这是极大的简化：避免管理多帧栈、避免重入问题、避免信号栈溢出。代价是如果一个 handler 执行时收到另一个不可屏蔽的致命信号（如 SIGSEGV），旧帧会被覆盖，恢复现场丢失。Linux 内核维护一个信号帧栈，此处放弃了。

**取舍二：musl `struct pthread` 固定偏移。** `musl_canceldisable_state` 从 `tp` 寄存器减去硬编码的 `MUSL_CANCEL_DISABLE_OFFSET_FROM_TP = 152` 读取线程取消禁用状态 <sup>[4](#mod-signal-ref-4)</sup>。这个 152 字节偏移是 musl 1.2.x RV64/LA64 的 ABI 细节，若 musl 版本升级或用户使用 glibc，该偏移不存在，`read_user_value` 会产生 EFAULT，导致取消信号抑制逻辑静默失效（`.flatten()` 吞掉错误）。替代方案是让用户态提供 `cancellation` 检查 syscall，但那样会多一次陷入，此处选择性能。

**取舍三：SA_RESTORER 与 trampoline 映射。** Linux 上每个架构的 sigaction ABI 不同：riscv64 的 `kernel_sigaction` 有 `restorer` 字段，loongarch64 则没有 <sup>[5](#mod-signal-ref-5)</sup>。当用户不提供 `SA_RESTORER` 时，模块通过 `ensure_default_signal_restorer` 调用 `map_rt_sigreturn_trampoline` 在用户地址空间映射一个固定的 `rt_sigreturn` 蹦床（地址为 `RT_SIGRETURN_TRAMPOLINE_ADDR`）。但该路径仅在 `cfg(any(target_arch = "riscv64", target_arch = "loongarch64"))` 下有效，其他架构（如 x86_64）的 `default_signal_restorer` 返回 `None`，导致 `deliver_user_signal` 直接返回 `ContinueUser`，信号 handler 不会被调用 <sup>[6](#mod-signal-ref-6)</sup>。这是一个明显的架构未适配信号。

### 3. 跨模块协同：投递、打断与调度器集成

信号投递入口有 6 个 syscall（kill/tkill/tgkill/rt_sigaction/rt_sigprocmask/rt_sigreturn）和内部接口（`notify_parent_sigchld`）。投递函数 `deliver_thread_signal` 通过 `wake_signal_wait_thread_where`（优先唤醒在 `rt_sigtimedwait` 中等待该信号的线程）、`wake_blocked_thread_tid`（打断可中断阻塞，如 wait4/ppoll/nanosleep）、最后回退到排队（`visit_threads_mut`）的顺序查找目标线程 <sup>[7](#mod-signal-ref-7)</sup>。这个顺序模拟了 Linux 的“同步投递优先于挂起”语义。

`prepare_signal_delivery` 被调度器在回用户态前调用。返回值 `SignalResumeAction` 驱动调度器做不同动作：`StopGroup` 让线程组停止并通知父进程，`TerminateGroup` 触发线程组终止流程 `finish_terminated_group`（清理队列、释放文件描述符、通知父进程等）<sup>[8](#mod-signal-ref-8)</sup>。`PreparedHandler` 意味着用户栈上已经压入了一个 `SignalFrame`，TrapFrame 的 PC 已改为 handler 地址，返回用户态即执行 handler。`restore_from_signal_frame` 被 syscall `rt_sigreturn` 调用，它从 `frame_addr` 读回保存的 TrapFrame、ucontext、信号掩码，并重置 `SignalDeliveryState`，实现现场恢复。注意 `apply_to_trap_frame` 是架构相关的（RV64 vs LA64），它通过条件编译的两个 `LinuxMContext` 实现来适配不同寄存器布局 <sup>[9](#mod-signal-ref-9)</sup>。

`syscall_rt_sigprocmask` 中有一个精妙的设计：当解除对信号 32/33（musl 的取消信号）的屏蔽时，若有 pending 的取消信号，会调用 `defer_cancel_signal_delivery_once` 和 `mark_signal_ucontext_syscall_rewind`，延迟一次投递并做 syscall IP 回卷 <sup>[10](#mod-signal-ref-10)</sup>。这是配合 musl 线程取消协议的特殊处理，普通操作系统教程很少涉及。

### 4. 边角细节与不足

**<sup>[1](#mod-signal-ref-1)</sup> `SignalSet` 的位图 API**：`from_signal` 和 `contains` 都调用 `signal_bit`，该函数检查范围并返回 `Option<u64>`。当 sig 超出 1..=64 时返回 `None`，调用者（如 `insert`/`remove`）静默返回 `false` 不报错——这意味着 `sigaction` 设置错误信号号时，错误码从 syscall 层 `is_supported_signal` 产生，但内部路径如 `enqueue_signal` 的错误被吞掉。若有人直接调用内部函数传非法 sig，可能产生数据竞争式静默失败。

**<sup>[4](#mod-signal-ref-4)</sup> musl 偏移硬编码风险**：如取舍二中所述，`MUSL_CANCEL_DISABLE_OFFSET_FROM_TP = 152` 是一个 magic number，没有任何 run-time 验证。若内核链接了不同 musl 版本的 C 库编译的应用程序，偏移可能不正确，导致非法内存读取或漏掉取消信号抑制。

**<sup>[6](#mod-signal-ref-6)</sup> x86_64 未适配**：`default_signal_restorer` 对非 RV64/LA64 返回 `None`，导致用户自定义 handler 无法正常调用（`deliver_user_signal` 在 `signal_restorer` 为 None 时返回 `ContinueUser`）。这意味着本模块在 x86_64 上只能执行默认动作（终止/忽略），完全无法支持用户态信号 handler。

**<sup>[7](#mod-signal-ref-7)</sup> 投递顺序的锁粒度**：`deliver_thread_signal_where` 内的 `visit_threads_mut` 逐个检查所有线程，但检查期间持有外部锁（由 `CurrentThread::with`、`visit_ready_threads_mut` 等隐含）。当线程数多时，遍历所有状态（ready/blocked/futex）的 `visit_threads_mut` 可能引入显著的调度延迟 <sup>[11](#mod-signal-ref-11)</sup>。且 `poll_interruptible_timeouts` 和 `futex::poll_futex_timeouts` 在遍历路径中也被调用，它们是 O(n) 超时清理，与信号投递耦合在一起。

**<sup>[8](#mod-signal-ref-8)</sup> `finish_terminated_thread_group` 的清理顺序**：该函数先调用 `prepare_thread_exit_cleanup`，再 `close_all_fds_for_exit`，然后 `finish_exit_group_status`。但 `wake_waiting_parent_for_exit` 在 `finish_exit_group_status` 之后才调用，可能导致父进程在 wait4 中读到子进程已终止状态时，子进程的某些资源（advisory locks）尚未释放。

**<sup>[10](#mod-signal-ref-10)</sup> `syscall_rt_sigprocmask` 的 defer 机制**：函数中判断 `pending_cancel_signal_unblocked` 时只检查了信号 32 和 33（`SIGRTMIN+1` 和 `SIGRTMIN+2`），这是 musl 线程取消的约定。但 Linux 不保证这两个信号一定用于取消，若用户态 libc 使用其他实时信号做取消，这段代码就不工作。这属于特定 libc 实现耦合。

**[死代码/半成品]** 模块中存在 `trace_mmapstress_signal_prepare` 和 `trace_mmapstress_sigreturn` 调用，它们是跟特定 mmap-stress 测试用例绑定的 trace 点，不属于通用信号机制，也没有 feature gate 控制，生产环境应剥离 <sup>[12](#mod-signal-ref-12)</sup>。另外 `prepare_signal_delivery` 中 `SIGALRM` 的处理路径专门打了这两个 trace 点，暗示这是为了调试某特定 workload 的临时注入。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了基于 futex 的用户态同步原语（FutexTable），由哈希桶（FutexBucket）和等待队列（FutexWaiter）构成，提供 FUTEX_WAIT / FUTEX_WAKE / FUTEX_REQUEUE 等系统调用。核心设计利用单核协作式调度免去锁竞争，以 UnsafeCell 实现内部可变性，并通过全局静态表管理 256 个桶。与标准 ArceOS 相比，增加了超时等待、批量超时发布、crowd timeout deferral 机制以及 LTP 跟踪功能，实现了进程退出时的 waiter 清理。syscall 接口在 kernel/src/syscall/futex.rs 中，负责参数验证、时间解析并与调度器协同切换线程。

### 1. 核心抽象与外部依赖
Futex 的核心抽象是 `FutexTable`——一个包含 256 个 `FutexBucket` 的静态哈希表，每个桶内维护一个 `VecDeque<FutexWaiter>` 作为等待队列。`FutexWaiter` 存储用户态地址、线程句柄和可选的超时截止时间。这种设计将用户地址空间中的同步点映射到内核态队列，使得等待/唤醒操作仅涉及哈希计算和队列操作。

外部依赖包括：（1）`super::wait` 模块，`FutexBucket` 引用了 `drain_deque_threads` 等通用队列操作函数；（2）`ReschedFlag`，用于标记超时发布后需要重新调度；（3）`axhal::time`，获取单调时间和挂钟时间计算超时剩余时间；（4）调度器 `enqueue_runnable` 和 `schedule_next_ready`，负责线程状态切换。

### 2. 关键设计取舍
**使用哈希桶而非全局队列**：通过地址的乘法哈希 `addr.wrapping_mul(0x9e3779b97f4a7c15) >> 32 % 256` 分散等待者，减少热点桶的竞争。256 个桶在典型应用场景下足够分散，且取模运算可被编译器优化为位掩码。

**采用 `VecDeque` 而非排序链表**：先进先出确保唤醒顺序的公平性；对于超时场景，`publish_expired` 线性扫描整个桶，未考虑时间排序，可能在大量等待者时性能下降，但单核协作式调度下线性扫描成本可控。

**使用 `UnsafeCell` 而非锁**：在当前单核协作式调度器中，不存在并行访问，因此采用 `UnsafeCell` 加上手动 `Sync` 实现，避免了锁开销。安全契约是：所有对桶的访问都必须在持有调度器排他上下文中进行，且不允许中断重入。`// SAFETY:` 注释缺失，这是一个潜在的技术债。

**超时处理：deferral 机制**：当同一地址的等待者超过 128 个（`FUTEX_CROWD_TIMEOUT_GRACE_THRESHOLD`）时，`should_defer_crowded_timeout` 会延长其超时，避免大量等待者同时超时导致硬实时尖峰。这个设计是一种启发式，牺牲精度换取平滑性。

**批量超时发布**：`poll_futex_timeouts` 每次最多处理 64 个超时等待者，并使用 `FUTEX_TIMEOUT_BATCH_IN_FLIGHT` 原子标记防止重入。在单核中该标记更像状态机，但代码中对双核假设下的并发路径有一定防御逻辑。

### 3. 跨模块协同
**syscall 接口**：`syscall_futex` 在 `kernel/src/syscall/futex.rs` 中根据操作码分发。其中，`FUTEX_WAIT` 和 `FUTEX_WAIT_BITSET` 先检查用户态值是否匹配，若匹配则调用 `park_futex_waiter` 将当前线程移入等待队列，然后经由 `capture_syscall_context` 保存 trap frame，最后通过 `schedule_next_ready` 切换到下一个就绪线程。唤醒后，返回值在 `set_user_retval(0)` 中预设。

**时间解析**：`read_futex_deadline` 将用户态 `timespec` 转换为 `FutexDeadline`，并处理相对/绝对时间、实时时钟和单调时钟。使用 `axhal::time` 的 `wall_time` 和 `monotonic_time`。

**进程退出清理**：`remove_process_waiters` 和 `drain_process_waiters` 由进程管理模块（如 `do_exit`）调用，移除终止进程的所有 futex 等待者，防止悬垂指针。

**调度器标记**：`take_futex_timeout_resched_pending` 和 `futex_timeout_resched_pending` 被调度器用于检查是否需要因超时发布而重新调度，实现了两次检查：一次在 `poll_futex_timeouts` 中设置标记，一次在调度循环中消费。

**LTP 跟踪**：`dump_waiters` 在低噪 LTP trace 启用时输出等待者状态，帮助调试测试失败。

### 4. 边角细节与不足

- **`UnsafeCell` 缺少 `// SAFETY:` 注释** <sup>[6](#mod-ipc-ref-6)</sup>：`FutexBucket(UnsafeCell<...>)` 与 `unsafe impl Sync for FutexBucket {}` 没有解释为何跨线程共享是安全的。虽然单核下不会并发，但未来多核移植时存在风险。

- **全局静态数组可能无法编译** <sup>[5](#mod-ipc-ref-5)</sup>：`static FUTEX_TABLE` 的初始化使用了 `[EMPTY_BUCKET; 256]`，但 `UnsafeCell` 不实现 `Copy`，此写法在标准 Rust 中非法。注释声称 “const 数组初始化做 bitwise copy”，但 `UnsafeCell` 禁止按位复制。如果编译器允许，可能依赖于未稳定语义，构成了可移植性隐患。

- **`should_defer_crowded_timeout` 中的线性搜索和可变借用意图** <sup>[4](#mod-ipc-ref-4)</sup>：函数签名接收 `&mut VecDeque`，但实际调用时通过 `FutexBucket.with_mut` 提供的 `&mut`。遍历中索引可能因 `defer_from_now` 改变 deadline 而非移动节点，在密集地址场景下可能仍有竞争，但单核无虞。

- **批处理标记的并发语义不一致** <sup>[2](#mod-ipc-ref-2)</sup>：`futex_timeout_batch_in_flight` 中，当 `!has_ready_threads()` 时清除标记，但两个检查之间可能插入新超时发布，导致短时间忽略超时。在协作式单核中不影响正确性，但设计意图不明确。

- **`FUTEX_WAIT_BITSET` 未实现 bitset 过滤** <sup>[7](#mod-ipc-ref-7)</sup>：syscall 中若 bitset 不是 `FUTEX_BITSET_MATCH_ANY`，仅打印一条消息，实际仍视为全匹配。这与 Linux 语义不符，但符合简单早期实现。

- **`FUTEX_CROWD_TIMEOUT_MAX_DEFERRALS` 硬编码为 120** <sup>[1](#mod-ipc-ref-1)</sup>：若大量等待者持续超时但无法被调度，deferral 最多推迟 120 秒（每段 1 秒），可能造成应用长时间阻塞。可配置的参数化设计更灵活。

- **桶索引计算使用 `% 256` 而非 `& 255`**：编译器会优化，但显式掩码可减少歧义。黄金乘法哈希的常数选择适合地址空间分布，但桶数必须是 2 的幂才能发挥掩码优势，此处 `% 256` 仍正确。

- **`dump_waiters` 的 `assert_kernel_stack_margin` 仅部署在 `visit_waiters_mut` 中** <sup>[3](#mod-ipc-ref-3)</sup>：其他桶操作函数（如 `publish_expired`）未进行栈边界检查，若在深层调用链中可能触发栈溢出。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 该模块是 Linux 兼容 socket syscall 的入口层，位于 `kernel/src/syscall/net.rs`（904 行）。核心抽象是将 15+ 个 Linux socket 系统调用（socket/bind/connect/listen/accept/sendto/recvfrom/shutdown/setsockopt/getsockopt/getsockname/getpeername/socketpair）转换为对 `Thread` 类型方法的调用，同时负责用户态地址结构体（sockaddr_in/sockaddr_in6/sockaddr_un/sockaddr_alg）的读取/写入和协议组合校验。与典型 ArceOS 直接暴露内部网络 API 不同，此模块在 syscall 层做参数字段级校验和阻塞式线程调度，支持 AF_UNIX socketpair 和 AF_ALG 加密算法 socket，但尚未实现 sendmsg/recvmsg/poll 等高阶接口。

### 1. 核心抽象：syscall 入口与 Thread 代理

本模块不直接实现网络协议栈，而是作为 Linux 兼容层的最外层——将用户态 syscall 请求转化为对 `crate::task::Thread` 类型的方法调用。文件开头的文档注释明确说明了其“最小网络 syscall 入口”定位，承接 `network-perf-suite` 的前两步：资源层创建进程私有 fd resource，以及 socket 行为面调用 <sup>[1](#mod-net-ref-1)</sup>。每个 syscall 处理函数（如 `syscall_socket`、`syscall_bind`）都是一个 `pub(super)` 函数，接收 `[usize; 6]` 参数数组和当前线程 `current: Thread`，返回 `SyscallReturn`（即 `Result<(Thread, usize), (Thread, LinuxError)>`）。这种设计将参数解析和错误处理集中在入口层，底层 `Thread` 方法不需要关心用户态地址的安全检查。

用户态网络地址结构体通过 `#[repr(C)]` 的 Rust 类型（`LinuxSockAddrIn`、`LinuxSockAddrIn6`、`LinuxSockAddrUn`、`LinuxSockAddrAlg`）直接映射，并实现 `From` trait 与 `core::net::SocketAddrV4`/`SocketAddrV6` 互转 <sup>[2](#mod-net-ref-2)</sup>。转换中注意了网络字节序（`to_be()` / `from_be()`）和 `sin_zero` 清零，但未对 `sin_len` 字段（Linux 特有）做处理——这是一个取舍：放弃了对老旧 BSD 带长度 sockaddr 的兼容。

### 2. 关键设计取舍：协议校验前置、阻塞调度与不完整 flags

**协议校验前置**：`syscall_socket` 在调用 `current.create_socket_fd` 之前先执行 `validate_socket_protocol`，对 domain/type/protocol 组合做白名单检查 <sup>[3](#mod-net-ref-3)</sup>。这种设计的优点是在资源分配前早早拒绝非法组合，避免底层 `Thread` 方法需要做重复检查；代价是白名单需要与底层实际支持的协议栈保持一致——当前白名单允许 `IPPROTO_SCTP`（132），但底层是否真的实现了 SCTP 存疑（后面没有看到任何 SCTP 相关实现代码）。

**阻塞调度的嵌入**：`syscall_recvfrom` 和 `syscall_accept`/`syscall_accept4` 的处理中，当底层返回 `EAGAIN` 时，代码检查 socket 的 `SOCK_NONBLOCK` 标志，若是非阻塞模式则直接返回 EAGAIN，否则调用 `crate::task::park_socket_recvfrom` / `crate::task::park_socket_accept` 将当前线程挂起，然后通过 `unsafe { crate::task::schedule_next_ready(trap_stack_top) }` 主动让出 CPU <sup>[4](#mod-net-ref-4)</sup>。这是一个典型的“协作式阻塞”模式——在单核/无抢占环境中，线程主动 park 并让调度器选择下一个就绪线程，而不是依赖内核态抢占。这个设计选择放弃了多核并发场景下的性能优化（如直接使用 mutex+condvar），但换来了实现简单性和与现有单核调度器的无缝集成。

**flags 的大幅缩减**：`syscall_sendto` 和 `syscall_recvfrom` 中 `flags != 0` 直接返回 `EOPNOTSUPP` <sup>[5](#mod-net-ref-5)</sup>。这意味着 `MSG_PEEK`、`MSG_DONTWAIT`、`MSG_WAITALL` 等标准 Linux flag 全都不支持。这是早期实现的明显标志——先做通基本路径，再逐步补充 flags。

### 3. 跨模块协同：Thread 方法与调度器契约

本模块与 `crate::task` 模块的协同是理解其架构的关键。从调用模式可以推断出以下契约：

- **`Thread::create_socket_fd(domain, type, protocol, cloexec, nonblocking) -> Result<usize, LinuxError>`**：底层负责创建 socket 文件描述符，返回 fd 号。
- **`Thread::bind_socket_fd(fd, SocketAddr)` / `Thread::connect_socket_fd(fd, SocketAddr)`**：绑定/连接，不同地址族走不同方法（如 `bind_unix_socket_fd`、`bind_alg_socket_fd`）<sup>[6](#mod-net-ref-6)</sup>。
- **`Thread::accept_socket_fd(fd) -> Result<(usize, SocketPeerName), LinuxError>`**：接受连接，返回新 fd 和对端地址。
- **`Thread::recvfrom_socket_fd(fd, buf, len) -> Result<(usize, Option<SocketPeerName>), LinuxError>`**：接收数据。
- **`crate::task::park_socket_recvfrom(current, tf, fd, buf, len, addr_ptr, addrlen_ptr)`**：将线程挂起在特定 socket 事件上，等待调度器在数据到达时唤醒。

这种分层将用户态内存安全（地址校验、数据拷贝）与底层网络栈逻辑分离。本模块负责从用户态读取地址结构体并转换为 `SocketPeerName`/`SocketLocalName` 枚举，这些枚举定义在 `crate::task` 中 <sup>[7](#mod-net-ref-7)</sup>。

### 4. 边角细节与不足

**<sup>[8](#mod-net-ref-8)</sup> `unsafe` 块缺少 SAFETY 注释**：`write_sockaddr_un` 函数中有一个将 `&LinuxSockAddrUn` 转为 `*const u8` 切片的核心操作 `core::slice::from_raw_parts(&sockaddr as *const LinuxSockAddrUn as *const u8, ...)`，**没有 `// SAFETY:` 注释**。安全契约是：必须确保 `LinuxSockAddrUn` 是 `#[repr(C)]` 且不含 padding（确认是 C 布局，但 `u16 + [u8;108]` 可能存在对齐 padding）。如果编译器在 `sun_family` 和 `sun_path` 之间插入了 padding，这个转换会读取未初始化字节泄露到用户态。

**<sup>[9](#mod-net-ref-9)</sup> 硬编码常量与 magic number**：`SOCK_NONBLOCK = 0o000_4000`、`SOCK_CLOEXEC = 0o2000000`、`AF_UNIX = 1`、`AF_INET = 2` 等大量 Linux 常量直接硬编码，没有使用 `libc` crate 或统一的常量定义模块。若架构变化或需要支持新协议，修改点分散且无 grep 友好命名（如 `AF_ALG = 38` 只有一处引用）。

**<sup>[10](#mod-net-ref-10)</sup> 断言与 unwrap 风险**：`read_sockaddr_un_path` 中 `unwrap_or(bytes.len().saturating_sub(1))` 和 `unwrap_or(bytes.len())` 是安全的 fallback，但 `AF_UNIX` 路径解析逻辑中 `bytes.first() == Some(&0)` 的分支处理了抽象 socket 地址（以 `@` 开头），如果用户传入一个长度在 family 之后立即为 0 的地址，路径会错误触发抽象 socket 分支。

**<sup>[11](#mod-net-ref-11)</sup> 不完整的错误路径**：`syscall_getsockopt` 和 `syscall_setsockopt` 直接透传给 `current` 的方法 <sup>[12](#mod-net-ref-12)</sup>，但 `sendto` 中若 `addr_ptr == 0` 使用 `None` 表示无目标地址——这在 `SOCK_STREAM`（已连接）下是正确的，但在 `SOCK_DGRAM` 下 Linux 应该返回 `EDESTADDRREQ`，这里简单忽略。

**<sup>[13](#mod-net-ref-13)</sup> TODO 暗示的半成品状态**：文件头注释说“当前承接 network-perf-suite 的前两步”，暗示还有第 3 步（可能是 IO 多路复用或高级 socket 选项）。`SOCK_SEQPACKET` 常量被定义但在 `validate_socket_protocol` 中没有匹配分支，说明 SOCK_SEQPACKET 基本不可用。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块实现了一个面向 Linux 用户态的最小 syscall 分发层，核心抽象是 `LinuxSyscallInvocation`——它将架构相关的 trap 寄存器布局（通过 `ArchSyscallAbi` trait 支持 RISC-V / LoongArch64）解码为架构无关的 nr+args 表示，再由 `dispatch_minimal_syscall` 中的巨型 match 派发到 fs、process、memory、signal、time、poll、net 等子模块。与原生 ArceOS 相比，本仓增加了 seccomp 模拟检查、signal 感知的阻塞等待原语（`InterruptibleBlock` / `SignalAwareDeadlineWait`）、以及一批“VOS 私有 syscall”用于 musl libc 兼容。大量“高级”syscall（eventfd2、timerfd_create、inotify_init1 等）被实现为返回 epoll fd 的轻量 probe，仅 BPF 和 mount 系列拥有真实状态机。

### 1. 核心抽象：LinuxSyscallInvocation 与架构解耦

本模块最基础的抽象是 `LinuxSyscallInvocation`，它持有一个 syscall 号 `nr: usize` 和 6 个参数 `args: [usize; 6]`<sup>[1](#mod-syscall-ref-1)</sup>。其构造方法 `from_trap_frame` 通过 `decode_syscall_abi` 将架构相关的 TrapFrame 寄存器布局解码为统一的 `ArchSyscallAbi`<sup>[1](#mod-syscall-ref-1)</sup>。这个设计的核心价值在于：**架构差异被限制在 `crate::arch` 内**，syscall 分发逻辑可以完全与寄存器布局无关地书写。`dispatch_minimal_syscall` 函数接收 `Thread` 所有权、`TrapFrame` 引用、`VirtAddr` 和 `LinuxSyscallInvocation`，通过一个约 120 个 arm 的 match 将调用派发到各个子模块<sup>[2](#mod-syscall-ref-2)</sup>。

特别值得注意的是，match 的返回类型是 `SyscallReturn`，即 `Result<(Thread, usize), (Thread, LinuxError)>`<sup>[11](#mod-syscall-ref-11)</sup>。这种设计强制每个子模块处理完 syscall 后将 `Thread` 所有权交回，由分发层通过 `CurrentThread::install` 重新安装。这保证了 **线程生命周期从用户态入口到 syscall 返回始终保持唯一拥有者**，避免了全局调度器状态的不一致。

分发层还嵌入了一个 seccomp 白名单检查：`seccomp_violation_signal` 在进入实际 handler 前检查当前进程的 seccomp 模式（disabled / strict / filter），对于不合法 syscall 直接通过 `exit_thread_group` 发送致命信号终止进程<sup>[7](#mod-syscall-ref-7)</sup>。对于 `SECCOMP_MODE_STRICT`，白名单只包含 read、write、exit、rt_sigreturn 四个号；`SECCOMP_MODE_FILTER` 的白名单稍宽但仍然极其有限。这是对 Linux seccomp 机制的纯粹模拟——没有 BPF 程序，只有硬编码白名单。

### 2. Signal 感知的阻塞 syscall 等待原语

本模块实现了一套可复用的阻塞等待机制，用于需要被信号中断的 syscall（如 nanosleep、poll、select、sigsuspend）。核心是 `InterruptibleBlock`（携带可选的 deadline）和 `SignalAwareDeadlineWait<T>` 枚举（Complete / TimedOut / Interrupted）。`wait_with_signal_deadline` 函数封装了循环逻辑<sup>[3](#mod-syscall-ref-3)</sup>：

1. 先检查 `should_interrupt_interruptible_block`，如果当前线程有挂起的非屏蔽信号则立即返回 EINTR；
2. 调用 `try_complete` 闭包尝试完成操作；
3. 如果 deadline 已过期则返回 TimedOut；
4. 否则调用 `on_idle` 等待重试。

这个原语在 poll.rs 中被深度定制。`syscall_poll` 和 `syscall_select` 在进入通用等待前，还会尝试多条**快速路径**<sup>[4](#mod-syscall-ref-4)</sup>：如果所有被 watch 的 fd 都是 pipe 读端、或都是 headless stdin、或都是“local read wait”fd，则分别使用 `park_poll_wait` / `park_select_wait` / `park_interruptible_block` 进行精确的阻塞，而不是走通用的忙等循环。这样可以减少无效的上下文切换。

`poll_idle_once` 函数进一步优化了 deadline 场景<sup>[12](#mod-syscall-ref-12)</sup>：如果 deadline 即将到达（距离开销小于 `POLL_DEADLINE_TAIL_SPIN`，LoongArch64 上 8ms，其余 1ms<sup>[8](#mod-syscall-ref-8)</sup>），改用 `core::hint::spin_loop()` 忙等而非 yield，以降低 deadline 刚过的调度延时；否则调用 `poll_precise_once`（poll 网络接口 + spin_loop）或 `poll_once`（poll 网络接口 + yield）。这是对实时性需求的一个折中：忙等带来功耗代价，但提高了 deadline 精度。

### 3. 跨模块协同：fdprobe 的“伪装”策略与真实状态机

`fdprobe` 子模块实现了各种“高级”但本系统只有桩实现的 syscall。其核心策略是对无实质实现需求的 fd 类 syscall 统一使用 `create_probe_fd` 返回一个 epoll fd 作为“有效但无具体语义”的 fd 句柄<sup>[5](#mod-syscall-ref-5)</sup>——包括 `eventfd2`（实际有 counter 语义）、`signalfd4`、`timerfd_create`、`pidfd_open`、`fanotify_init`、`inotify_init1`、`userfaultfd`、`perf_event_open`、`io_uring_setup`。这使得调用这些 syscall 的用户态库（如 glibc 的 `eventfd` 封装）可以拿到一个合法 fd 而不会立即崩溃，同时实际 I/O 操作（如 read/write）则会走通用的 fd 路径返回适当错误。

但 BPF 和 mount 系列并非桩。`bpf()` 实现了最小状态机，支持 `BPF_MAP_CREATE`、`BPF_MAP_LOOKUP_ELEM`、`BPF_MAP_UPDATE_ELEM`、`BPF_PROG_LOAD` 四个命令<sup>[6](#mod-syscall-ref-6)</sup>。Map 类型支持 HASH、ARRAY、RINGBUF，但只存储一次性元素（hash 只保存一个 entry，array 是线性存储）；Prog 加载则实现了 4 类假程序（Noop、StoreOne、StoreArithmeticResult、StoreDivModZeroResults），用于通过 LTP bpf 相关测试。mount 系列 syscall（`fsopen`、`fsconfig`、`fsmount`、`open_tree`、`move_mount`、`mount_setattr`）则实现了完整的配置状态机<sup>[9](#mod-syscall-ref-9)</sup>：`fsopen` 创建 `MountConfigState` 记录 fs_name 和 source；`fsconfig` 解析 FSCONFIG_SET_FLAG/SET_STRING/SET_BINARY/SET_PATH/SET_FD/CMD_CREATE 等命令；`fsmount` 最终在目标路径上挂载 ramfs 并保存 `DetachedMountState`。

这些状态通过全局 `LazyInit<Mutex<BpfState>>` 管理，意味着它们**不隔离进程**——同一个内核内的所有用户进程共享同一个 BPF map 空间和 mount 配置空间。

### 4. 边角细节与不足

**<sup>[10](#mod-syscall-ref-10)</sup> VOS 私有 syscall 桥接**：代码中定义了 `VOS_MUSL_PATHCONF_SYSCALL`（`usize::MAX - 30`）等几个畸形 syscall 号。这些不是标准 Linux syscall，而是 ArceOS-Starry 特制的 musl libc 兼容补丁入口。当内存中打过补丁的 musl libc 调用这些号时，会直接跳转到内核的专用快捷处理（如 pathconf、gethostname、brk/sbrk）。这意味着用户态 libc 与内核存在非标准耦合。

**Seccomp 白名单不足**：`seccomp_violation_signal` 的 `SECCOMP_MODE_FILTER` 白名单只有 7 个 syscall<sup>[7](#mod-syscall-ref-7)</sup>。如果用户态通过 `prctl(PR_SET_SECCOMP, SECCOMP_MODE_FILTER, ...)` 设置了 filter，但期望 BPF 程序实际运行的 syscall 不在白名单中，会收到 `SIGSYS`。这是对 Linux 复杂 seccomp 机制的严重简化。

**`unsafe` 块缺少 SAFETY 注释**：在 `dispatch_minimal_syscall` 中对 `exit_thread_group` 的 unsafe 调用（第 708 行附近）以及在 poll.rs 中对 `schedule_next_ready` 的多个 unsafe 调用（如第 173、740 行）均没有 `// SAFETY:` 说明<sup>[2](#mod-syscall-ref-2)</sup><sup>[4](#mod-syscall-ref-4)</sup>。这违反了 Rust 安全编码规范，构成技术债。

**BPF 状态机无进程隔离**：全局 `BPF_STATE`<sup>[6](#mod-syscall-ref-6)</sup> 使得进程 A 可以访问进程 B 创建的 BPF map 和 mount 配置。虽然当前实现只用于 LTP 通过测试，但这在实际多进程场景下是安全漏洞。

**Syscall 号架构硬编码**：所有 syscall 号常量（如 `READ_SYSCALL = 63`）是 Linux asm-generic 值，主要适配 RISC-V。若移植到 x86_64（如 x86 的 read 是 0）或 ARM64（read 是 63 但 accept 是 43 而非 202），需要系统性重写所有常量。`#[cfg(target_arch)]` 目前只覆盖 riscv64 和 loongarch64。
