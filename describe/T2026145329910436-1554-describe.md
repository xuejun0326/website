## 📖 一、总览

该仓库属于arceos-starry家族中的实验型Linux兼容内核，基于ArceOS模块化框架构建，通过axhal硬件抽象层实现裸机引导，并在应用层自定义main入口。整体架构分为内核核心（starry-kernel）与系统整体（starryos）两部分，以内核模块形式组织内存管理、任务调度、文件系统、网络、信号、IPC等子系统，并提供了约200个系统调用，覆盖进程、文件、网络、IPC、信号等多个领域。与基线ArceOS仅提供基础示例不同，本仓付出了大量工程努力以实现用户态API兼容，体现在：

1. 内存管理引入Backend枚举[mm.ref:3]，支持线性、COW、共享等多种映射策略，并配合全局帧引用计数表实现fork时物理页共享，这是ArceOS家族中首次出现的完整虚拟内存方案。
2. 系统调用层集成了异步I/O多路复用（epoll/poll/select），通过axtask的waker机制将文件就绪事件传递到等待线程[fs.ref:4]，而非简单轮询。
3. 信号机制采用进程/线程双层分发与原子标志防重入[signal.ref:5]，并引入了signalfd文件接口，实现了与Linux基本一致的不透明信号处理。

然而，该仓的工程完成度仍处于实验阶段：许多阻塞操作被简化为非阻塞或stub，权限模型过度简化，代码质量存在unsafe缺少注释、潜在UB等问题。

## 🎯 二、综合评价

### 整体定位：面向Linux兼容的实验内核

本仓在arceos-starry家族中属于追求Linux兼容的“完整型”实现，但当前状态更接近实验开发版。它以ArceOS的模块化框架（axhal/axtask/axmm等）为起点，将系统调用覆盖数从数十扩展至约200，并增加了对动态链接、信号、SysV IPC、epoll等特性的支持。与ArceOS官方仓库相比，本仓更关注用户态可运行性，而非教学或演示。其启动流程采用条件编译支持LTP、OSCOMP等测试场景，体现对验证的重视。

### 真正的设计亮点

尽管整体处于初级阶段，但部分设计展现了合理的抽象与创新：

- **内存后端策略**：Backend枚举[mm.ref:3]是核心创新，它将虚拟内存映射的物理页管理策略（线性、COW、共享、文件映射）编码为类型安全的变体，这与Linux的struct vm_area_struct相比更接近函数式设计，且易于验证。全局帧引用计数表[mm.ref:3]使得fork无需复制物理页，而是共享后写时复制，提升了性能。
- **用户指针封装**：UserPtr<T>[mm.ref:2]将裸指针转换为带权限检查的引用，减少系统调用内对页表的重复查询。尽管'static生命周期有风险，但接口设计思路值得借鉴。
- **系统调用异步化**：epoll实现[fs.ref:4]利用了axtask的异步waker，使得I/O等待不阻塞整个调度器，这是ArceOS生态中该机制的早期应用。
- **信号双层分发**：信号发送时先定位进程，再由进程信号管理器选择合适线程[signal.ref:5]，并通过原子标志防止重入，设计清晰。

### 取舍与局限

本仓致力于广度而不是深度。它实现了大量系统调用，但许多仅提供基本功能：SysV IPC的阻塞操作被省略[ipc.ref:1]，信号CoreDump未尽，syncfs是dummy。这种取舍让它能快速启动busybox和部分LTP测试，但离生产级兼容尚有距离。权限模型在IPC[ipc.ref:1]中几乎旁路，命名空间也仅打印警告，降低了容器运行潜力。整体上，开发者的精力放在搭建框架和覆盖关键路径上，而非完善细节。

### 完成度与代码质量

完成度方面，约200个系统调用中约有40%为stub或简化实现，核心路径（fork/exec/read/write/mmap/信号）基本完整。但异步信号、作业控制、多核扩展等薄弱。代码质量方面，模块组织清晰，每个子系统独立，AxResult统一错误处理等设计良好。但存在多处无需SAFETY注释的unsafe（如AssumeSync[task.ref:1]、remove_evict_listener），以及明显的逻辑错误（sys_reboot调用system_off[syscall.ref:18]），整体编码严谨度有待提升。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 该模块以 axhal 为硬件抽象层，在 src/main.rs 中定义 Rust 入口函数 main，通过条件编译选择 LTP/custom/OSCOMP 三种启动脚本，收集参数后调用 starry_kernel::entry::init 完成内核初始化。与标准 ArceOS 相比，本仓将入口从 axhal 的主函数下沉到应用级 main，并引入了多 feature 启动配置机制，体现了编译器项目类型的模块化启动策略。

### 1. 核心抽象：axhal 引导下的 Rust main 入口

启动流程的核心抽象是 **axhal (ArceOS Hardware Abstraction Layer)** 提供的运行时环境。`src/main.rs` 通过 `#![no_std]` 和 `#![no_main]` 声明这是一个裸机二进制程序，不依赖标准库入口点，而是由 axhal 在完成底层初始化（页表、中断、控制寄存器等）后跳转到 `#[unsafe(no_mangle)] fn main()`<sup>[1](#mod-boot-ref-1)</sup><sup>[4](#mod-boot-ref-4)</sup>。这种设计将硬件初始化与业务逻辑解耦：axhal 负责架构相关的引导（如 RISC-V 的 `_start` 或 x86 的 `_start`），而 Rust 层面只需关注操作系统自身的初始化流程。

与标准 ArceOS 的 `axmain` 宏不同，本仓直接定义裸 `main` 函数并手动调用 `starry_kernel::entry::init`，这是一种更显式的控制流表达——放弃宏封装带来的简洁性，换取了调用前参数预处理（如 `CMDLINE` 收集）的灵活性。boot facts 也印证了这一点：call_chain 显示参数通过 `iter` → `copied` → `map` 链式转换后再传入内核入口<sup>[5](#mod-boot-ref-5)</sup>。

### 2. 条件编译与启动配置策略

`CMDLINE` 常量通过三重 `#[cfg]` 分支选择不同的 init 脚本<sup>[2](#mod-boot-ref-2)</sup><sup>[3](#mod-boot-ref-3)</sup>：
- **`ltp-only`**：运行 LTP (Linux Test Project) 测试套件，用于验证系统调用兼容性；
- **`custom`**：用户自定义启动脚本，供开发调试；
- **默认（oscomp）**：OS 竞赛场景的标准测试脚本。

三者的公共前缀都是 `/musl/busybox sh -c`，说明用户态 init 统一由 musl 版 busybox 的 shell 解释执行。`include_str!` 在编译期将 `.sh` 脚本嵌入二进制，避免了文件系统未就绪时无法读取 init 程序的鸡生蛋问题<sup>[2](#mod-boot-ref-2)</sup>。这种设计本质上是一种 **编译期预置 initramfs** 的轻量替代——相比于构建完整的 initramfs 映像，此处直接把脚本内容硬编码进数据段，代价是修改脚本需要重新编译整个内核。

需要注意的是，`#[cfg]` 分支使用了 `not(any(...))` 的取反逻辑，这意味着如果同时启用 `ltp-only` 和 `custom`（例如 `--features "ltp-only custom"`），两个分支都不匹配，会回退到默认分支。这可能是 Feature 互斥设计的疏忽——更安全的做法是加 `#[cfg(not(any(feature = "ltp-only", feature = "custom")))]` 的语义本身正确，但若未来新增第三个 Feature 则需同步修改取反条件，维护成本较高。

### 3. 跨模块协同：从 main 到内核 init 的调用链

`main` 函数仅做两件事：构造参数、调用 `starry_kernel::entry::init`<sup>[6](#mod-boot-ref-6)</sup>。这层薄封装背后涉及多个模块的协同：

1. **参数传递**：`CMDLINE` 定义为 `&[&str]`，通过 `iter().copied().map(str::to_owned)` 转为 `Vec<String>`，从静态切片过渡到堆分配的所有权类型。`envs` 被设为空数组 `[]`，这意味着当前启动环境不传递任何环境变量。

2. **内核入口**：`starry_kernel::entry::init` 接收 `&[String]` 和 `&[&str]`，其内部应完成内存管理初始化（如堆分配器）、进程/线程创建（将 args 作为 init 进程的参数）、中断注册等。boot facts 中列出的两个 handler——`PAGE_FAULT` 由 `kernel/src/mm/access.rs:258` 的 `handle_page_fault` 处理，`IRQ` 由 `vendor/axhal/src/irq.rs:35` 的 `irq_handler` 处理——说明在入口 `init` 内部或之前的 axhal 引导阶段已经完成了中断向量表的安装。

3. **模块分层**：`starry_kernel` 是内核 crate，而 `src/main.rs` 属于顶层 binary crate。这种分离使得内核逻辑可以独立于启动入口进行单元测试，也允许将来替换入口（如 UEFI 启动、多核 AP 启动）而无需修改内核。

### 4. 边角细节与不足

以下是审查中发现的若干具体风险点和技术债务：

- **<sup>[4](#mod-boot-ref-4)</sup> `#[unsafe(no_mangle)]` 的安全契约**：Rust 2024 edition 将 `no_mangle` 标记为 unsafe，因为错误的名字修饰可能导致符号冲突或被外部代码意外调用。此处虽无直接危险，但缺少 `// SAFETY:` 注释说明为何该符号可以被安全地导出，构成隐性的技术债。若未来链接了其他包含同名 `main` 的对象，将引发链接错误。

- **<sup>[6](#mod-boot-ref-6)</sup> 不可达错误路径**：`starry_kernel::entry::init` 的返回类型未在 `main` 中处理——若 init 返回错误或 panic，`main` 函数将直接返回，而 axhal 通常不会处理 main 返回后的状态（boot facts 中 assembly_files 为空，表明无返回路径清理代码）。这意味着 init 失败后将进入未定义行为（通常为死循环或崩溃）。

- **<sup>[5](#mod-boot-ref-5)</sup> 硬编码路径假设**：`/musl/busybox` 的路径硬编码在 CMDLINE 中，且未通过任何配置项参数化。若文件系统布局变更（如使用 musl 以外的 libc），需要修改三处分支。此外，该路径假定 init 程序位于 `/musl/` 目录下，与标准 Linux `/bin/busybox` 不同，这体现了本项目使用自定义 musl 工具链的意图，但也构成了与非 musl 环境的耦合。

- **<sup>[1](#mod-boot-ref-1)</sup> `extern crate alloc;` 的隐式假设**：虽然 Rust 2018+ 已不再需要 `extern crate`，此处仍保留是为了确保 alloc 在 `no_std` 环境下可用。但若未来某个 feature 组合禁用了全局分配器（如 `alloc_error_handler` 未定义），此声明会静默编译失败，错误信息晦涩。

- **<sup>[3](#mod-boot-ref-3)</sup> 默认分支的 fallback 覆盖问题**：`#[cfg(not(any(feature = "ltp-only", feature = "custom")))]` 分支覆盖了所有其他情况，包括同时启用两个 feature 的情形。这本身是安全的，但若新增第三个 feature 但忘记更新此条件，新增 feature 将意外走默认 oscomp 路径，直到测试才暴露。

- **空 envs 的环境隔离缺失**：`let envs = [];` 表明 init 进程接收的环境变量为空。这在大多数测试场景下是可接受的，但某些程序（如依赖 `PATH` 的 shell 脚本）可能行为异常。虽然 busybox 本身能在无 PATH 时工作，但 LTP 测试用例可能依赖特定环境变量。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 AddrSpace 为核心抽象，组合 MemorySet<Backend> 和 PageTable 实现用户进程地址空间管理。Backend 枚举封装线性映射、COW、共享等后端策略，通过全局帧引用计数表实现物理页共享。与家族基线相比，增加了完整的动态链接器加载、用户内存安全访问封装（UserPtr）以及基于页错误的 COW 按需分页机制。

### 1. 核心抽象与外部依赖

地址空间核心是 `AddrSpace` 结构体 <sup>[1](#mod-mm-ref-1)</sup>，包含三个字段：`va_range` 描述虚拟地址范围，`areas` 是 `MemorySet<Backend>` 管理已映射区域，`pt` 是硬件页表。依赖来自 `axhal::paging` 的 `PageTable`、`MappingFlags` 和 `PageFaultFlags`，以及 `memory_addr` 和 `memory_set` crate。`Backend` 定义在 `aspace/backend` 模块中，是一个枚举，包含 `Linear`、`Cow`、`Shared`、`File` 等变体，每个实现 `BackendOps` trait 提供映射、取消映射、填充等操作。该设计将地址空间逻辑与具体物理内存管理解耦，允许不同映射策略共存于同一地址空间。

用户内存访问通过 `UserPtr<T>` 和 `UserConstPtr<T>` 封装 <sup>[2](#mod-mm-ref-2)</sup>，它们将裸指针包装为类型安全的用户空间引用，内部通过 `check_region` 验证地址范围、权限和页表存在性，再返回 `&'static mut T`。这种设计避免了系统调用中反复进行页表查询，但 `'static` 生命周期假设要求调用者保证在进程上下文内使用。

### 2. 关键设计取舍

**COW 后端与全局帧引用计数**：`CowBackend` 实现 `MAP_PRIVATE` 语义 <sup>[3](#mod-mm-ref-3)</sup>。每个物理帧在 `FRAME_TABLE`（全局 `BTreeMap<PhysAddr, Arc<SpinNoIrq<FrameRefCnt>>>`）中维护一个引用计数。COW 发生时，若引用计数 >1 则复制帧并更新页表；若 =1 则直接提升权限。使用全局表而非区域绑定计数，允许不同地址空间的同一个物理帧共享计数，但引入了锁竞争（`SpinNoIrq`）。替代方案是将计数嵌入 `MemoryArea`，但跨地址空间共享时需要更复杂的消息传递。这一取舍优化了 `fork()` 后父子进程共享页的性能（见 `try_clone` <sup>[4](#mod-mm-ref-4)</sup>），代价是每次页错误都需要查询全局表。

**用户内存访问的双重保护**：`access_user_memory` 设置线程级标志 <sup>[5](#mod-mm-ref-5)</sup>，然后在 `check_region` 中先锁住地址空间检查权限并预填充（`populate_area`），随后才直接访问指针。若访问引发页错误，`PAGE_FAULT` 陷阱处理程序 <sup>[6](#mod-mm-ref-6)</sup> 检查标志位并调用 `AddrSpace::handle_page_fault` 进行按需分页。这种“先检查后访问”比每次访问都走 `copy_from_user` 的逐页拷贝效率更高，但危险在于 `check_region` 与后续访问之间 TLB 可能失效（内核不会让其他线程修改当前页表），因此安全前提是单线程处理。

**动态链接器加载的地址分配**：在 `load_user_app` 中，动态链接器（ldso）映射到地址 `USER_INTERP_BASE`，主程序映射到 `USER_SPACE_BASE` <sup>[7](#mod-mm-ref-7)</sup>。这种静态基地址分配避免了 `mmap` 的随机化，简化了 ELF 加载器的实现。但牺牲了 ASLR，且当程序要求固定地址时可能冲突。

### 3. 跨模块协同

`sys_mmap` <sup>[8](#mod-mm-ref-8)</sup> 是用户可见的映射入口，它将 `MAP_SHARED`/`MAP_PRIVATE`/`MAP_ANONYMOUS` 等 Linux 语义转化为 `Backend` 变体：`MAP_PRIVATE` 且带文件时创建 `CowBackend`；`MAP_SHARED` 且带文件时创建 `FileBackend`（支持缓存或直接设备）；匿名映射则创建 `SharedBackend` 或 `AllocBackend`。映射前通过 `AddrSpace::find_free_area` 寻找合适空洞（支持 `MAP_FIXED` 强制覆盖），随后调用 `AddrSpace::map` 插入区域并更新页表。`mmap` 的实现错误路径处理较为完善，但 `mremap` 仍是“先 mmap 新区域再 memcpy 后 munmap”的朴素版本（带有 TODO）。

页错误处理体现深度协同：`access.rs` 注册的 `handle_page_fault` 在 `PAGE_FAULT` 时调用 <sup>[6](#mod-mm-ref-6)</sup>，它从 `current_may_uninit` 获取当前线程，检查 `is_accessing_user_memory` 标志，然后锁定地址空间并调用 `AddrSpace::handle_page_fault`。后者在对应区域的后端上调用 `populate`：对 `CowBackend`，若缺页则分配零页或从文件读取；若 COW 写则处理帧复制。整个过程贯穿了线程、地址空间、后端三个层级。

内存跟踪设备 <sup>[9](#mod-mm-ref-9)</sup> 通过 `axalloc::allocations_in` 枚举内核堆分配，与 ELF 加载器等关联类别，实现对进程创建、文件缓存等内存使用的分类统计，这对调试内存泄漏有价值。

### 4. 边角细节与不足

1. **<sup>[10](#mod-mm-ref-10)</sup> `unsafe` 的 `'static` 假设**：`UserPtr::get_as_mut` 返回 `&'static mut T`，但引用生命期实际受线程执行上下文限制。若跨线程传递引用，将导致 UAF。代码中未提供 `SAFETY` 注释，且用户态系统调用相对安全，但仍是潜在隐患。
2. **<sup>[11](#mod-mm-ref-11)</sup> 全局帧表锁竞争**：`FRAME_TABLE` 使用 `SpinNoIrq`，在 COW 密集型负载（如频繁 fork）下可能成为瓶颈。小尺寸（`BTreeMap`）且操作短，但在多核场景仍有压力。
3. **<sup>[12](#mod-mm-ref-12)</sup> `check_null_terminated` 的效率问题**：代码中明确标记 TODO 并注释 `unsafe` 循环中的页锁定，每次迭代都重新锁 `aspace`，导致 `O(len)` 次锁获取。替代方案可一次查询整个范围的页表项。
4. **<sup>[13](#mod-mm-ref-13)</sup> 架构条件编译的隐蔽性**：`copy_from_kernel` 使用 `#[cfg(not(any(target_arch = "aarch64", target_arch = "loongarch64")))]`，假设其他架构（x86_64、riscv64）需要复制内核页表。但若未来增加新架构且也使用独立页表，会静默地多复制一份，可能导致性能下降。
5. **<sup>[14](#mod-mm-ref-14)</sup> `mremap` 和 `mprotect` 未完成**：`mremap` 标注 TODO，实现为“拷贝后删除”，未处理 remap 的原地扩展或移动语义；`mprotect` 不支持 `PROT_GROWSDOWN` 和 `PROT_GROWSUP`。这些信号会导致系统调用返回 `ENOSYS`，对依赖此类功能的用户程序构成兼容性问题。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 模块以 `Thread`（线程）和 `ProcessData`（进程共享数据）为核心抽象，通过 `axtask::TaskExt` 挂接到 `TaskInner`，实现 1:N 线程模型。它解决多线程用户进程的创建、调度、退出、信号、定时器及权限管理等完整生命周期问题。与通用 ArceOS 相比，增加了 `CloneFlags` 细粒度控制、分层信号传递、每线程定时器、基于 `Scope` 的资源隔离以及简化但 POSIX 兼容的 credentials 管理。

### 1. 核心抽象与线程模型

模块以 `Thread` 和 `ProcessData` 为核心数据结构。`Thread`（mod.rs:60-82）包含线程私有状态：`clear_child_tid`、`robust_list_head`、线程级信号管理器 `signal`、时间管理器 `time`（包装在 `AssumeSync<RefCell<TimeManager>>` 中）、`exit` 标志和 `exit_event`。`ProcessData`（mod.rs:156-190）存储进程共享资源：`Arc<Process>`、地址空间 `aspace`、`Scope`（用于资源隔离）、futex 表、信号 actions、资源限制 rlim、credentials（uid/gid 等）。`Thread` 通过 `proc_data: Arc<ProcessData>` 引用共享数据，形成 1:N 关系——同一进程的多个线程共享同一个 `ProcessData`。

`TaskExt` trait 的实现（mod.rs:140-154）利用 `on_enter`/`on_leave` 回调在上下文切换时自动切换 `ActiveScope`。每个 `Thread` 存储了一个引用计数 `Scope`，`on_enter` 将 `ActiveScope` 设置为该 scope，`on_leave` 恢复为全局 scope 并递减引用计数。这个设计使得文件描述符表、FS 上下文等资源可以按进程隔离，无需在每个系统调用中手动切换。

`AsThread` trait（mod.rs:126-137）提供了从 `TaskInner` 向下转型到 `&Thread` 的安全接口。`try_as_thread` 检查 `task_ext` 是否存在，`as_thread` 则直接 panic。这个 trait 被 `ops.rs`、`clone.rs`、`ctl.rs` 广泛使用，是获取线程上下文的统一入口。

### 2. 关键设计取舍

**进程线程关系**：Linux 的 clone 系统调用通过 `CLONE_THREAD` 标志决定是否创建新线程（共享 ProcessData）还是新进程（复制）。本模块在 `CloneArgs::do_clone`（clone.rs:148-250）中实现：若 `CLONE_THREAD` 设置，则直接复用 `old_proc_data`；否则通过 `fork` 创建新 `Process`，复制或共享地址空间（由 `CLONE_VM` 控制），并复制或共享信号 actions（`CLONE_SIGHAND`）。地址空间复制使用 `try_clone` 和 `copy_from_kernel`（clone.rs:197-200），支持写时复制？目前 `try_clone` 可能只是简单复制页表。命名空间标志（NEWNS/NEWPID 等）目前仅打印警告（clone.rs:128），实际未实现——这是已知简化，会影响容器场景。

**信号与定时器协同**：信号管理分为进程级和线程级。`ProcessSignalManager` 管理进程共享的 `SignalActions` 和挂起信号；`ThreadSignalManager` 管理线程级信号掩码和挂起。定时器由 `TimeManager` 管理（timer.rs:111-124），每个线程一个实例，在上下文切换时通过 `poll_timer`（ops.rs:148-158）检查是否到期。`TimeManager` 内部使用 `ITimer` 数组（Real/Virtual/Prof），`poll` 方法计算 delta 时间并更新 utime/stime，到期后通过闭包发送信号。全局 `alarm_task`（timer.rs:262-277）处理 `ITimerType::Real` 的触发，使用 `BinaryHeap` 和 `Event` 实现异步等待。

**Credentials 管理**：`ProcessData` 存储六个 UID/GID 字段（uid/euid/suid/gid/egid/sgid），使用原子操作。`set_resuid`（mod.rs:251-281）实现了 Linux `setresuid` 的权限检查：如果 euid != 0（非 root），则新值必须存在于当前 real/effective/saved 集合中。但实际中 euid==0 检查太宽松——真正的 `CAP_SETUID` 能力检查未实现，这是一个安全简化。

**定时器安全包装**：`TimeManager` 放在 `RefCell` 中，但被 `AssumeSync` 包装（mod.rs:47-57）以通过 `Sync` 检查。作者注释称只有上下文切换时独占访问，因此假设安全。但 `AssumeSync` 没有 SAFETY 注释，如果将来引入并发访问，将成为未定义行为源——这是技术债。

### 3. 跨模块协同

**任务表与系统调用**：`ops.rs` 维护全局 `TASK_TABLE`（`WeakMap<Pid, WeakAxTaskRef>`）。`add_task_to_table` 在 `do_clone` 成功后调用（clone.rs:246），同时将进程、进程组、会话插入相应表。系统调用如 `getpid`/`gettid` 通过 `current().as_thread().proc_data.proc.pid()` 获得，但 `get_task` 通过 TID 查表用于 `wait4`、`kill` 等。`processes()` 和 `get_process_data` 用于发送信号和 procfs。

**进程退出流程**：`do_exit`（ops.rs:218-240）依次执行：清除 `clear_child_tid` 并唤醒 futex（`wq.wake`），退出 robust list，减少进程线程计数（`process.exit_thread`），若为最后一个线程则调用 `process.exit()`、发送 `exit_signal`（通常是 SIGCHLD）给父进程、唤醒 `child_exit_event`，清理共享内存（`SHM_MANAGER`）。若 `group_exit` 为 true，则向所有线程发送 SIGKILL。最后设置 `thr.exit` 标志。

**procfs 集成**：`pseudofs/proc.rs` 中的 `ProcFsHandler` 通过 `tasks()` 获取所有任务列表，生成 `/proc/[tid]` 目录。每个目录包含 `stat`、`status`、`oom_score_adj`、`fd`、`cmdline`、`exe` 等文件。`fd` 目录通过 `FD_TABLE.scope(&task.as_thread().proc_data.scope.read())` 访问对应进程的文件描述符表，展示了 scope 隔离的实际应用。`oom_score_adj` 支持读写，通过 `Thread::oom_score_adj` 原子操作实现。

### 4. 边角细节与不足

- **AssumeSync 无 SAFETY 注释**（mod.rs:47-57）：非安全 `Sync` 实现，若 `RefCell` 被多核同时访问将导致数据竞争。当前仅在单核或中断禁用时安全，但未文档化。
- **CAPABILITY_VERSION_3 硬编码**（ctl.rs:14）：固定版本号，若用户空间传递不同版本则返回 `EINVAL`。根据 Linux 行为，应更新版本字段并重试，当前实现直接失败。
- **TODO(mivik): preempting does not change timer state**（timer.rs:92）：注释承认当线程被抢占时定时器状态（USER/KERNEL）不更新，导致 utime/stime 统计不准确。
- **命名空间 stub**（clone.rs:128）：只打印警告，实际未创建新命名空间，可能影响依赖该功能的应用程序。
- **定时器 alarm_task 中的 unwrap**（timer.rs:272）：使用 `assert!(guard.pop()...)`，若堆顶任务被意外移除或 deadline 不匹配则 panic。用户空间无法直接修改内核对象，风险较低但仍是隐患。
- **exit_robust_list 用户输入依赖**（ops.rs:193-215）：读取用户空间的 `RobustList` 指针，虽然有限制 `ROBUST_LIST_LIMIT` 和 `yield_now`，但若用户提供循环链表仍可能消耗大量 CPU 时间。
- **ProcessData 的 scope 字段有 TODO: scopify 注释**（mod.rs:165）：表明 scope 的使用可能不完整，目前只是简单的 `RwLock<Scope>`，未来可能改为更严格的资源隔离方案。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块基于 `axfs-ng-vfs` 抽象层，提供 POSIX 文件系统系统调用的实现。核心抽象是 `FsContext`（任务局部 FS 上下文）与 `CachedFile`/`DirectFile` 双后端文件句柄。相比原版 ArceOS，关键修改包括：引入 LRU 页缓存、epoll 支持、更完整的权限处理、以及针对特殊文件（字符设备/FIFO）的直接 I/O 路径。解决了多任务文件并发访问与 I/O 多路复用问题。

### 1. 核心抽象与外部依赖

文件系统模块建立在 `axfs-ng-vfs` 提供的 `Location`、`FileNode` 等基础抽象之上。`Location` 封装了一个文件系统节点（文件/目录/符号链接），支持元数据读写、目录遍历、链接操作等。`axfs-ng` 在高层次上提供了 `FsContext` 结构体，它封装了根目录与当前工作目录，并提供类似 `std::fs` 的接口（`resolve`、`read_dir`、`create_dir` 等）<sup>[1](#mod-fs-ref-1)</sup>。

每个进程拥有一个任务局部的 `FS_CONTEXT`，通过 `scope_local!` 宏实现，默认克隆自全局 `ROOT_FS_CONTEXT` <sup>[2](#mod-fs-ref-2)</sup>。这种设计避免了全局锁竞争，同时允许 `chroot` 等操作改变特定任务的根目录。文件句柄分为 `File` 枚举，内部使用 `CachedFile`（带 LRU 页缓存）或 `DirectFile`（绕过缓存直接读写）后端，由 `OpenOptions::_open` 根据文件类型与标志自动选择 <sup>[3](#mod-fs-ref-3)</sup>。

系统调用实现集中在 `kernel/src/syscall/fs/` 下，通过 `get_file_like` 获取文件描述符对应的 `FileLike` trait 对象，调用其读写、ioctl、poll 等方法。`FileLike` trait 同时被普通文件、目录、epoll、管道等实现，实现了统一的 I/O 多路复用接口。

### 2. 关键设计取舍

**页缓存策略**：`CachedFile` 内部维护一个 `LruCache<u32, PageCache>`，默认容量为 64 页（256KB）。对于 tmpfs（内存文件系统），则使用无界缓存，因为已经占用了物理内存，无需限制。缓存在 evict 时回写脏页，并通过 `EvictListener` 机制允许外部注册回调（例如块设备刷新）。这一设计在性能和内存占用间取得平衡，但缺少预读和批量回写，可能影响顺序读性能。

**直接 I/O 路径**：对于字符设备、FIFO、Socket 等特殊文件，以及标记了 `NON_CACHEABLE` 或 `direct` 标志的文件，`_open` 会选择 `FileBackend::new_direct`，绕过页缓存直接读写。这是因为这些文件通常不支持随机访问或需要即时传输，缓存反而会引入一致性问题。此判断逻辑在 `file.rs:217-231`，使用了 `matches!` 宏检测节点类型。

**权限与安全**：`OpenOptions` 支持 `user` 字段，在 `open` 系统调用中传递进程的 euid/egid，使新创建的文件具有正确的属主。`mkdirat` 也会在创建后显式 `update_metadata` 设置 owner，弥补底层 `create_dir` 不接收用户的缺陷 <sup>[4](#mod-fs-ref-4)</sup>。这反映了对 POSIX 权限模型的仔细实现。

**epoll 实现**：`kernel/src/file/epoll.rs` 完整实现了 epoll 的 level-triggered、edge-triggered 和 one-shot 模式。它利用 `Pollable::register` 在文件上注册自定义 Waker，当文件就绪时通过 `InterestWaker` 将兴趣添加到 epoll 的 ready 队列，再唤醒等待的 epoll 实例。这里使用了 `Weak` 引用避免循环引用导致的内存泄漏；同时，`consume` 方法会检查模式并返回 `ConsumeResult::EventAndKeep` 或 `EventAndRemove`，正确处理 LT/ET/ONESHOT 的语义 <sup>[5](#mod-fs-ref-5)</sup>。

### 3. 跨模块协同

**系统调用与 FS 上下文**：系统调用函数（如 `sys_openat`、`sys_read`、`sys_epoll_ctl`）通过 `FS_CONTEXT.lock()` 获取当前任务的 `FsContext`，然后调用其方法进行路径解析或文件操作。例如 `sys_chdir` 调用 `fs.resolve(path)` 得到 `Location`，再调用 `fs.set_current_dir` <sup>[6](#mod-fs-ref-6)</sup>。路径解析过程会跟随符号链接，最大深度 `SYMLINKS_MAX=40`，防止循环。

**文件描述符与 FileLike**：`get_file_like(fd)` 返回 `Arc<dyn FileLike>`，既可以指向 `File`（普通文件），也可以指向 `Epoll` 或 `Pipe`。这使得 `poll` 系统调用可以统一处理多种类型。`Epoll` 自身实现了 `FileLike`，并暴露 `poll_events` 方法供 `epoll_wait` 调用。

**页缓存与全局分配器**：`PageCache` 通过 `axalloc::global_allocator().alloc_pages` 分配物理页面，标记为 `UsageKind::PageCache`。这使得内存管理器可以统计文件缓存占用的物理内存，并可能在未来支持回收。`Drop` 实现中会释放页面，但若页面仍为脏，则仅打印警告，不会自动回写，因此必须确保 evict 时写回 <sup>[7](#mod-fs-ref-7)</sup>。

### 4. 边角细节与不足

1. **`unsafe` 块缺乏安全注释**：`CachedFile::remove_evict_listener` 标记为 `unsafe`，但缺少 `// SAFETY:` 注释，调用者必须保证 handle 合法，否则可能导致 use-after-free <sup>[8](#mod-fs-ref-8)</sup>。同样，`DirBuffer::write_entry` 中的 `unsafe` 块也没有说明内存布局的合法前提。
2. **`sync` 系统调用为 dummy**：`sys_sync` 和 `sys_syncfs` 只打印警告并返回成功，不实际刷新文件系统缓存。这在崩溃场景下可能导致数据丢失 <sup>[9](#mod-fs-ref-9)</sup>。
3. **页缓存写回时机不完整**：`PageCache::drop` 如果 `dirty` 为真，只打 `warn!` 而不写回。正常情况下 evict 时会写回，但若 `CachedFile` 在脏页未 evict 时被销毁，数据将丢失 <sup>[7](#mod-fs-ref-7)</sup>。
4. **`page_or_insert` 的 TODO**：代码中存在 `// TODO: Matching the result of `get_mut` confuses compiler.`，表明这里可能有更好的写法，当前使用 `cache.contains` + `cache.get_mut`，存在两次查找，效率略低 <sup>[10](#mod-fs-ref-10)</sup>。
5. **`ioctl` 安全漏洞风险**：在 `sys_ioctl` 中，读取用户态指针 `(arg as *const u8).vm_read()` 可能由于恶意的 `arg` 导致错误，但 `vm_read` 应能处理。不过对于 TIOCGWINSZ 的静默忽略可能导致应用程序误以为操作成功 <sup>[11](#mod-fs-ref-11)</sup>。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制以自研crate `starry_signal` 为核心抽象，提供 `SignalSet`、`SignalInfo`、`Signo` 等基本类型，系统调用处理集中在 `kernel/src/syscall/signal.rs`，信号发送与检查逻辑在 `kernel/src/task/signal.rs`，并以 `signalfd` 作为文件接口与 I/O 多路复用集成。相比原版 ArceOS（无信号支持），该仓完整实现了 Linux RT 信号集，包括 `rt_sigaction`、`rt_sigprocmask`、`signalfd4` 等 16 个系统调用，并采用进程/线程双层分发设计与原子标志防重入。

### 1. 核心抽象与外部依赖

信号机制的核心类型来自外部 crate `starry_signal`，包括 `SignalSet`（信号集）、`SignalInfo`（信号信息及发送者）、`Signo`（信号编号枚举）、`SignalStack`（替代信号栈）以及 `kernel_sigaction`（内核信号动作）<sup>[1](#mod-signal-ref-1)</sup>。这些类型在系统调用层与任务层之间传递，构成了整个信号子系统的“语法”。

与 Linux 兼容的 `linux_raw_sys` crate 提供了 `SI_USER`、`SIG_BLOCK` 等常量与 `kernel_sigaction` 的原始定义，但 `starry_signal` 对其进行了 Rust 封装，例如 `Signo::from_repr()` 用于安全转换 `u32` 到枚举 <sup>[9](#mod-signal-ref-9)</sup>。这种分层避免了内核直接操作 C 结构体，但也意味着 `starry_signal` 必须与 `linux_raw_sys` 布局严格对齐——若未来 Linux 头文件变更，需同步更新两处。

任务层通过 `axtask::current()` 获取当前线程，再通过 `AsThread` trait 将 `TaskInner` 转为 `Thread`。`Thread` 中包含 `signal` 字段（类型为 `starry_signal` 中的某信号控制结构），以及 `proc_data` 中的 `signal`（进程级信号管理）。这种双层设计（线程+进程）是 Linux 信号模型的直接映射。

### 2. 关键设计取舍

**【信号发送路径的二分法】**

信号可以定向到线程（`tkill`/`tgkill`）或进程（`kill`/`rt_sigqueueinfo`）。线程定向直接调用 `send_signal_to_thread`，通过 `get_task(tid)` 找到目标 `TaskInner`，然后调用 `send_signal_thread_inner` 将信号插入线程信号队列并 `task.interrupt()` 唤醒 <sup>[3](#mod-signal-ref-3)</sup>。进程定向则通过 `send_signal_to_process` 访问进程级 `proc_data.signal.send_signal(sig)`，该方法返回一个选中的线程 tid（可能基于优先级或当前线程），再对该线程执行 interrupt <sup>[3](#mod-signal-ref-3)</sup>。这种设计允许进程信号“自动选择一个线程处理”，符合 POSIX 要求，但选择算法隐藏在 `starry_signal` 内部，内核无法干预——若未来需支持线程组停止/继续语义，可能需要暴露更多控制。

**【`check_signals` 的驱动与重入防护】**

信号的实际处理发生在 `check_signals` 函数中，它被系统调用返回路径、sigreturn、sigsuspend/sigtimedwait 的等待循环中调用 <sup>[2](#mod-signal-ref-2)</sup>。该函数从 `thr.signal.check_signals(uctx, restore_blocked)` 获取一个待处理信号及其默认动作。对于 `Terminate`、`CoreDump` 等动作，直接调用 `do_exit`；对于 `Handler`，则返回 `true` 表示已设置用户态跳板（具体跳板逻辑封装在 `starry_signal` 的 `check_signals` 实现中，内核只负责调用）。

一个精妙但容易忽视的点是 `BLOCK_NEXT_SIGNAL_CHECK` 原子标志 <sup>[10](#mod-signal-ref-10)</sup>。在 `sys_rt_sigreturn` 入口处调用 `block_next_signal()` 设置该标志，随后在信号恢复期间禁止再次触发 `check_signals`，避免信号处理函数返回时因嵌套信号导致上下文错乱。这种“一次检查”模式简单有效，但使用 `SeqCst` 排序可能在内核高频信号路径上产生不必要的屏障开销——不过对于信号不频繁的通用场景，可接受。

**【锁粒度权衡】**

信号动作表 `actions` 使用 `Mutex` 保护（`sys_rt_sigaction` 中 `curr.as_thread().proc_data.signal.actions.lock()`），这是进程级共享数据，锁持有时间很短（仅复制结构体），不会阻塞其他系统调用。但若同一进程的多线程并发修改不同信号的动作，仍会串行化——在极端高频注册场景下可能成为瓶颈，但符合 Linux 行为（`sigaction` 本身是进程级同步点）。

### 3. 跨模块协同

**【系统调用 ↔ 任务信号层】**

所有信号相关系统调用（`sys_rt_sigprocmask`、`sys_kill` 等）位于 `kernel/src/syscall/signal.rs`，它们通过 `crate::task::` 模块导出函数与任务层交互。例如 `sys_kill` 根据 PID 语义（正数、0、-1、负数）分别调用 `send_signal_to_process`、`send_signal_to_process_group` 或遍历进程列表 <sup>[4](#mod-signal-ref-4)</sup>。这种分层使系统调用处理只关注参数解析和用户空间指针安全，而信号队列管理和线程选择下沉到任务层。

**【signalfd 与 I/O 多路复用集成】**

`Signalfd` 实现了 `FileLike` 和 `Pollable` trait，使其可以被 `poll`/`epoll` 监控。其 `read` 方法通过 `block_on(poll_io(...))` 阻塞等待，内部调用 `dequeue_signal` 从当前线程信号队列取出匹配 mask 的信号，并转换为 128 字节的 `SignalfdSiginfo` 结构 <sup>[7](#mod-signal-ref-7)</sup>。`Pollable::poll` 直接查询 `has_pending_signals()`，无需阻塞。这种方式将信号事件纳入了内核统一的 I/O 事件框架，但与 Linux 不同的是，Linux 的 signalfd 在进程/线程共享信号队列时行为复杂，此实现仅读取当前线程的信号——若用户期望读取整个进程的信号，可能需要额外线程协作。

**【信号返回与用户上下文恢复】**

`sys_rt_sigreturn` 调用 `signal.restore(uctx)`，该函数由 `starry_signal` 实现，负责从用户栈上的 `SignalFrame` 恢复寄存器并返回 <sup>[5](#mod-signal-ref-5)</sup>。注意其先调用了 `block_next_signal()` 再 `restore`，且通过 `uctx.retval()` 将返回值设为 `-EINTR`（在 sigsuspend 场景下），这要求 `UserContext` 与信号栈帧格式严格匹配——任何布局偏差都会导致用户态崩溃。

### 4. 边角细节与不足

**<sup>[1](#mod-signal-ref-1)</sup> unsafe 块的安全契约**：多处使用 `unsafe { set.vm_read_uninit()?.assume_init() }` 读取用户空间指针 <sup>[9](#mod-signal-ref-9)</sup>。安全前提是 `VmPtr` 封装已确保指针合法（通过页表检查），且 `SignalSet` 是 `MaybeUninit` 兼容的整数类型。但若 `starry_vm` 的实现存在竞态（例如指针在检查后被取消映射），则可能导致内核访问无效内存——这是所有用户空间指针解引用的通病，该实现未使用 `copy_from_user` 的逐字节检查，而是依赖 `VmPtr` 的原子性，缺乏双重检查。

**<sup>[2](#mod-signal-ref-2)</sup> TODO 未实现项**：`check_signals` 中 `SignalOSAction::CoreDump`、`Stop`、`Continue` 均以 `do_exit` 或空操作代替 <sup>[2](#mod-signal-ref-2)</sup>。这意味着收到 `SIGSTOP` 的进程会直接退出而非暂停，严重不符合 POSIX。对于依赖作业控制的应用（如 shell），这是功能缺失。

**<sup>[6](#mod-signal-ref-6)</sup> SignalfdSiginfo 字段硬编码为 0**：`from_signal_info` 实现中，`ssi_pid`、`ssi_uid`、`ssi_fd` 等大量字段被设为 0 或 -1 <sup>[6](#mod-signal-ref-6)</sup>。虽然 `SignalInfo` 可能不包含这些信息，但 Linux 的 `signalfd` 期望这些字段准确（例如 `ssi_pid` 应为发送者 PID）。这将导致用户空间程序（如使用 `signalfd` 监控 `SIGCHLD`）获取到不完整的进程信息。

**<sup>[11](#mod-signal-ref-11)</sup> MINSIGSTKSZ 硬编码**：`sys_sigaltstack` 中直接使用 `MINSIGSTKSZ`（来自 `linux_raw_sys`）与用户提供的 `ss.size` 比较 <sup>[11](#mod-signal-ref-11)</sup>。该常量的值取决于架构（x86_64 为 2048），但代码中无 `cfg` 条件，若移植到 RISC-V 等架构可能不匹配。不过 `linux_raw_sys` 会根据目标架构导出正确值，所以实际风险较低，但若未来添加自定义架构则需要额外注意。

**<sup>[10](#mod-signal-ref-10)</sup> 原子排序的保守使用**：`BLOCK_NEXT_SIGNAL_CHECK` 的所有操作使用 `Ordering::SeqCst`，这提供了最强的全局顺序保证，但对于仅用于单核或信号路径的场景，使用 `Acquire/Release` 对即可满足互斥，`SeqCst` 会在多核系统上产生内存屏障开销，属于过度防御。

**<sup>[8](#mod-signal-ref-8)</sup> poll_fn 中的调度点**：`sys_rt_sigtimedwait` 和 `sys_rt_sigsuspend` 使用 `poll_fn` + `block_on` 在异步上下文中等待信号 <sup>[8](#mod-signal-ref-8)</sup>。这种方法允许信号到来时通过 `task.interrupt()` 唤醒，但若 `timeout` 为 `None`，将永久阻塞。值得注意的是，`sys_rt_sigsuspend` 在等待循环中先设置 `uctx.set_retval(-EINTR)`，然后调用 `check_signals`，若信号处理函数返回，`check_signals` 返回 `true` 且 `restore_blocked` 会将阻塞集恢复，然后函数返回 `Err(AxError::Interrupted)`。这正确模拟了 Linux 行为，但返回值 `-EINTR` 已写入 `uctx`，一旦 `check_signals` 发现信号并处理，`uctx` 已被修改，此时再次返回 `Err` 可能导致用户态看到双重 `EINTR`？实际上 `check_signals` 内若为 `Handler`，会设置跳板并修改 `uctx` 为信号处理函数入口，此时 `sigsuspend` 不会继续执行，而是直接返回至用户态跳板，因此不会冲突——这是通过 `poll_fn` 的 `Ready` 提前退出实现的。该实现依赖于 `check_signals` 返回 `true` 时已经修改了 `uctx`，协作细节较为隐晦。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现了 System V IPC（消息队列、共享内存）、匿名管道和 futex 同步原语。核心抽象是利用全局静态管理器（MSG_MANAGER/SHM_MANAGER）配合 `Arc<Mutex<…>>` 封装共享资源，通过原子 ID 分配唯一标识。管道基于 ringbuf 环形缓冲区，futex 维护等待队列并借助 async waker 实现唤醒。与同家族基线相比，补齐了 SysV IPC 和 futex 系统调用，但阻塞操作均未实现且权限模型过于简化。

### 1. 核心抽象：全局管理器 + 锁保护资源

System V IPC 是典型的“用户空间不可见、内核维护持久对象”的机制。该模块为消息队列和共享内存分别设置了全局单例管理器 `MsgManager` 和 `ShmManager` <sup>[15](#mod-ipc-ref-15)</sup><sup>[16](#mod-ipc-ref-16)</sup>。它们均被 `axsync::Mutex` 保护，内部使用 `BTreeMap` 建立 key 与 ID、ID 与对象（`MessageQueue`、`ShmInner`）之间的映射。每个对象又被 `Arc<Mutex<…>>` 包裹，实现多线程安全——读操作仅持有管理器锁，写操作在拿到对象锁后快速释放管理器锁。管道则采用 `Arc<Shared>` 结构，内部包含 `Mutex<HeapRb<u8>>` 环形缓冲区以及用于 poll 的 `PollSet` <sup>[3](#mod-ipc-ref-3)</sup>。futex 的等待队列 `WaitQueue` 用 `SpinNoIrq<VecDeque<(Waker, u32)>>` 实现，省去关闭中断的开销 <sup>[4](#mod-ipc-ref-4)</sup>。所有 IPC 对象通过原子计数器 `IPC_ID` 生成单调递增 ID，简单但可能溢出。

### 2. 关键设计取舍

**阻塞操作未实现**：`sys_msgsnd` 在队列满时不挂起当前任务，直接打印 `warn!` 并返回 `EAGAIN` <sup>[5](#mod-ipc-ref-5)</sup>；`sys_msgrcv` 同理。这意味着依赖阻塞模式的程序（如非 `IPC_NOWAIT` 的 `msgsnd`/`msgrcv`）会直接失败——这是明显的早期实现痕迹。

**共享内存映射策略**：`sys_shmat` 支持指定虚拟地址，但完全忽略 `SHM_RND` 和 `SHM_REMAP` 标志 <sup>[6](#mod-ipc-ref-6)</sup>。代码仅调用 `align_down_4k` 进行基本对齐，且用 `assert!` 确保进程未重复 attach 同一 shm，这在多线程场景下可能触发 panic。

**futex 键设计**：`FutexKey` 在构造时遍历进程地址空间，通过 `Backend::Shared` 或 `Backend::File` 判断地址是否属于共享区域，从而区分私有和共享 futex <sup>[7](#mod-ipc-ref-7)</sup>。这种做法相比纯按地址哈希更安全，避免不同进程相同虚拟地址冲突，但弱引用的使用要求共享区域存活期长于 futex 表。

**权限检查的简化**：`has_ipc_permission` 仅在 root 时放行，且所有 IPC 对象初始 uid/gid 设为 0<sup>[1](#mod-ipc-ref-1)</sup>。这意味着非根用户环境下权限模型无法正确工作，也没有 IPC 命名空间隔离。

### 3. 跨模块协同

**共享内存与地址空间**：`ShmInner.phys_pages` 记录所有附着进程共享的物理页。首个进程 attach 时分配 `SharedPages` 并通过 `Backend::new_shared` 映射；后续进程复用。进程退出时 `clear_proc_shm` 遍历附着记录，更新 attach 计数，若标记 `rmid` 且计数为 0 则删除共享段 <sup>[6](#mod-ipc-ref-6)</sup>。futex 也通过 `FutexKey::new` 中的 `Backend::Shared` 分支判断共享 futex 并路由到正确的 `FutexTable` <sup>[7](#mod-ipc-ref-7)</sup>。

**管道与信号**：写端在检测到读端关闭时调用 `raise_pipe` 向当前进程发送 `SIGPIPE` <sup>[9](#mod-ipc-ref-9)</sup>，符合 POSIX 要求。`Pipe` 实现了 `Pollable` trait，使得 `poll` 可以异步监听读写事件。

**futex 与 robust list**：`sys_get_robust_list` 和 `sys_set_robust_list` 仅实现系统调用接口，存储头指针但未在进程退出时扫描并清理 futex <sup>[10](#mod-ipc-ref-10)</sup>。因此 `FutexEntry.owner_dead` 标志极少被置位。

### 4. 边角细节与不足

1. **<sup>[5](#mod-ipc-ref-5)</sup> 消息队列阻塞操作全部未实现**：`sys_msgsnd` 和 `sys_msgrcv` 在需要等待时直接返回 `EAGAIN`，依赖阻塞模式的应用程序无法正常使用。
2. **<sup>[6](#mod-ipc-ref-6)</sup> 共享内存标志缺失**：`SHM_RND`/`SHM_REMAP` 被 TODO 标记，且 `sys_shmat` 用 `assert!` 防止重复 attach，而非返回 `EINVAL`。
3. **<sup>[11](#mod-ipc-ref-11)</sup> `msg_qbytes` 双用途错误**：注释声称“both byte count and message count are limited by msg_qbytes”，而 Linux 中 `msg_qbytes` 只限字节数，代码同时检查 `total_bytes` 和 `msg_qnum` 可能提前拒绝合法消息。
4. **<sup>[10](#mod-ipc-ref-10)</sup> robust list 功能残缺**：只实现 set/get 系统调用，未在进程退出时清理 futex。
5. **<sup>[12](#mod-ipc-ref-12)</sup> `BiBTreeMap` 位置不当**：通用双射映射定义在 `shm.rs` 中，作者自注“不知放哪”，反映模块结构考虑不周。
6. **<sup>[13](#mod-ipc-ref-13)</sup> futex `CMP_REQUEUE` value2 参数提取错误**：使用 `timeout.addr() as u32` 而非单独参数，违反 Linux ABI。
7. **<sup>[14](#mod-ipc-ref-14)</sup> `UserMsgbuf` 零长度数组**：`mtext: [u8; 0]` 在 Rust 中属 UB，虽通过 `vm_load` 绕过，但仍依赖平台行为。

这些不足表明当前 IPC 模块处于“可用但粗糙”阶段，需大量完善。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块封装了 `axnet` crate 提供的多种地址族（IPv4、IPv6、Unix、Vsock）套接字能力，通过 `SocketAddrExt` trait 统一用户态 `sockaddr` 与内核 `SocketAddr` 间的转换，并以 `Socket` 结构体实现 `FileLike` 和 `Pollable` 接口，使套接字能融入内核通用文件表与异步 I/O 框架。与 ArceOS 基线相比，该模块增加了对 Unix 域套接字（`SOCK_STREAM`/`SOCK_DGRAM`）、VSOCK 的完整支持，并通过 `socketpair` 调用提供了进程间通信能力。

### 1. 核心抽象与外部依赖
网络模块的核心抽象是 `SocketAddrExt` trait（`addr.rs:9-15`），它定义了 `read_from_user` 和 `write_to_user` 方法，将用户空间传入的 `sockaddr` 指针安全地转换为内核可操作的 `SocketAddr` 对象。该 trait 为 `SocketAddrV4`、`SocketAddrV6`、`UnixSocketAddr` 和 `VsockAddr` 分别实现，通过 `SocketAddrEx` 枚举统一分派（`addr.rs:178-194`）。这种设计避免了动态分发带来的性能开销，并保持了类型安全。

套接字本身由 `Socket` 结构体（`net.rs:10`）表示，它内部持有 `axnet::Socket` 枚举，支持 `Tcp`、`Udp`、`Unix`、`Vsock` 四种变体。`Socket` 实现了 `Deref`（`net.rs:19-23`）以直接访问底层方法，以及 `FileLike`（`net.rs:26-40`）和 `Pollable` 接口，从而能够被插入到进程文件描述符表，并参与基于 `axpoll` 的异步事件循环。

### 2. 关键设计取舍
- **地址族扩展性**：`SocketAddrEx` 枚举在 `read_from_user` 中根据 `AF_INET`/`AF_INET6`/`AF_UNIX`/`AF_VSOCK` 分派（`addr.rs:179-187`）。这比 trait 对象更简单，但拓展新地址族需修改枚举定义，不具备二进制兼容性。
- **非阻塞连接**：`sys_connect` 将 `WouldBlock` 错误转换为 `InProgress`（`socket.rs:39-44`），符合 POSIX `EINPROGRESS` 语义，但这也意味着调用方无法区分真正的阻塞超时和连接初始未就绪。
- **套接字选项调度**：`opt.rs` 使用 `call_dispatch!` 宏将 `(level, optname)` 对直接映射到 `GetSocketOption`/`SetSocketOption` 变体。这种方法用编译期 macto 展开替代了运行时分发，提高了性能，但宏内的类型转换（如 `Int<usize>` 到 `i32`）如果未覆盖所有合法值，可能引入未定义行为。
- **控制消息（cmsg）**：`cmsg.rs` 中的 `CMsg` 和 `CMsgBuilder` 仅支持 `SCM_RIGHTS` 类型，`parse` 函数直接读取用户空间中的 `cmsghdr`（`cmsg.rs:19-25`），依赖用户提供正确的 `cmsg_len`。恶意用户可能构造过小的 `cmsg_len` 导致 `get_as_slice` 读取越界或产生未初始化数据。
- **文件描述符集成**：`sys_socket` 和 `sys_socketpair` 在创建套接字后立即调用 `add_to_fd_table`，并支持 `O_CLOEXEC` 和 `O_NONBLOCK` 标志（`socket.rs:40-44`）。这种“创建即绑定”的方式简化了生命周期管理，但无法在创建后单独设置执行时关闭标志。

### 3. 跨模块协同
- **地址转换链**：`sys_bind`/`sys_connect` 首先调用 `SocketAddrExt::read_from_user`（`addr.rs`）解析用户态地址，再传给 `axnet` 的 `bind`/`connect`。返回时，`sys_accept4` 通过 `write_to_user` 将远端地址写回用户空间（`socket.rs:82-84`）。
- **I/O 与控制消息**：`sys_sendmsg` 和 `sys_recvmsg` 在 `io.rs` 中分别调用 `send_impl` 和 `recv_impl`。发送时解析用户提供的 `msghdr`，遍历 `msg_control` 链表，通过 `CMsg::parse`（`cmsg.rs`）生成控制消息；接收时则用 `CMsgBuilder` 将内核生成的 `CMsg` 序列化回用户空间（`io.rs:130-142`）。
- **选项获取与设置**：`sys_getsockopt`/`sys_setsockopt`（`opt.rs`）通过 `Socket` 结构体转发到 `axnet` 的 `GetSocketOption`/`SetSocketOption` 枚举，`call_dispatch!` 宏将 `(level, optname)` 匹配到具体选项变体，再通过 `dispatch!` 宏调用对应的类型转换函数（如 `Duration::sys_to_rust`）。
- **文件表与异步轮询**：`net.rs` 中 `Socket` 实现了 `Pollable` trait，使其支持 `register` 和 `poll` 方法，能与内核的 `epoll`/`axpoll` 框架协作。`nonblocking` 方法通过 `GetSocketOption::NonBlocking` 间接查询（`net.rs:50-55`），避免了为每个套接字维护独立的非阻塞标志。

### 4. 边角细节与不足
- **`fill_addr` 的长度覆盖**：`fill_addr`（`addr.rs:43-49`）在写入用户缓冲区后，用 `data.len()` 覆盖 `*addrlen`。但输入 `addrlen` 可能比数据短（通过 `min` 截断），此时覆盖会导致用户看到比实际写入更长的长度，可能引发用户态缓冲区溢出。更好的设计是保持 `*addrlen` 为实际写入长度，而非数据总长度。
- **`read_family` 的严格性**：`read_family`（`addr.rs:28-30`）要求 `addrlen >= size_of::<__kernel_sa_family_t>()`，但 `sockaddr` 通常包含 `sa_family` 和 `sa_data`，此检查过于宽松；然而对后续 `read_from_user` 中 `addrlen` 的精确匹配（如 IPv4 要求恰好 `size_of::<sockaddr_in>`）又过于严格，可能拒绝某些 glibc 传入的延长结构。
- **VSOCK 的早期实现**：`vsock` 特性仅在 `AF_VSOCK + SOCK_STREAM` 组合下生效（`socket.rs:33-35`），且协议号未校验。`sockaddr_vm` 结构体（`addr.rs:149-157`）注释提到“linux-raw-sys 缺少此类型”，使用了手动定义的 `#[repr(C)]` 结构，未来上游合并后可能产生冲突。
- **`stat` 的固定权限**：`Socket::stat`（`net.rs:44-49`）返回 `S_IFSOCK | 0o777`，不反映实际权限。恶意程序可能通过 `fstat` 间接获取到错误的文件模式。
- **`SO_PEERCRED` 的假设**：`opt.rs:76-79` 将 `ucred` 直接转换为 `UnixCredentials`，假设用户填充了正确的 `pid`/`uid`/`gid`。但 Unix 域套接字对等凭据需要内核填充，此处依赖上层 `axnet` 的实现正确性。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块实现了 Linux 兼容的系统调用分发与处理。核心抽象是 `handle_syscall()` 入口函数，通过 `syscalls::Sysno` 枚举进行匹配，分派到 `fs`、`mm`、`signal`、`task`、`time` 等子模块。所有系统调用实现统一返回 `AxResult<isize>`，错误通过 `axerrno::LinuxError` 转换为负值。与 ArceOS 基线相比，本仓大幅扩展了系统调用覆盖（约 150+ 个），引入了 `starry_vm::VmPtr` 用户态指针封装、异步 I/O 多路复用（epoll/poll/select）的 `async` 实现、以及基于 `axtask` 的协程调度集成。

### 1. 核心抽象与外部依赖

系统调用层的核心是 `kernel/src/syscall/mod.rs` 中的 `handle_syscall()` 函数。它接收 `axhal::uspace::UserContext` 的可变引用，通过 `syscalls::Sysno::new()` 解析系统调用号，然后进入一个巨型 `match` 表达式进行分派<sup>[1](#mod-syscall-ref-1)</sup><sup>[2](#mod-syscall-ref-2)</sup>。每个子模块（`fs`、`io_mpx`、`ipc`、`mm`、`net`、`resources`、`signal`、`sync`、`sys`、`task`、`time`）都通过 `pub use self::{...}::*` 将内部定义的 `sys_*` 函数暴露给主调度器<sup>[3](#mod-syscall-ref-3)</sup>。

返回值约定是统一的 `AxResult<isize>`。成功时返回非负值；失败时通过 `.unwrap_or_else(|err| -LinuxError::from(err).code() as _)` 转换为 Linux 风格的负错误码，最终通过 `uctx.set_retval()` 写回用户寄存器<sup>[4](#mod-syscall-ref-4)</sup>。这一设计将 AxError 与 Linux 错误空间解耦：内核内部使用 `AxError`（如 `InvalidInput`、`BadFileDescriptor`），每个模块只需关心 AxError，而主调度器负责将其映射为 `-ENOSYS`、`-EINVAL` 等。

用户态内存访问通过 `starry_vm` 提供的 `VmPtr`/`VmMutPtr` 封装完成。例如 `sys_uname` 中 `name.vm_write(UTSNAME)?`，`sys_getrusage` 中 `usage.vm_write(result.into())?`<sup>[5](#mod-syscall-ref-5)</sup>。这些封装内部处理了用户态指针的合法性检查（边界、权限），但部分接口（如 `vm_read_uninit`）仍标记为 `unsafe` 且缺少完整的 `// SAFETY:` 注释，构成技术债。

### 2. 关键设计取舍

**取舍一：条件编译区分架构系统调用表。** 模块中大量使用了 `#[cfg(target_arch = "x86_64")]` 来包裹传统系统调用（如 `open`、`stat`、`fork`、`pipe`、`select`、`poll`）<sup>[6](#mod-syscall-ref-6)</sup>。而 riscv64 和 aarch64 仅暴露对应的新式系统调用（`openat`、`fstatat`、`clone3`、`pipe2`、`ppoll`）。这一取舍的直接后果是：如果用户程序直接调用 `sys_open`（而不是 `sys_openat`），在 riscv64 上会返回 `ENOSYS`。这放弃了与旧二进制兼容性，换取了更小的内核体积和更少的代码路径。但从代码中可以看到，`#[cfg(not(target_arch = "riscv64"))]` 用于 `renameat` 时，riscv64 仅支持 `renameat2`<sup>[7](#mod-syscall-ref-7)</sup>——这反映了 Linux 内核在新架构上只实现新系统调用的惯例。

**取舍二：`async` 驱动的 I/O 多路复用。** `io_mpx` 子模块（poll、select、epoll）没有采用传统的阻塞循环或中断驱动，而是基于 `axtask::future::block_on` + `poll_io` 的协程模型<sup>[8](#mod-syscall-ref-8)</sup>。以 `do_select` 为例，它构造一个 `FdPollSet`（实现了 `Pollable` trait），通过 `block_on(future::timeout(...))` 驱动就绪事件轮询；当没有事件时返回 `AxError::WouldBlock`，由 `poll_io` 负责挂起当前任务<sup>[9](#mod-syscall-ref-9)</sup>。这一设计的隐含前提是整个调度器是协作式的——如果某个任务不 yield，其他任务无法运行。这与 Linux 内核的抢占式 `select`/`epoll` 模型有本质区别，但在单核或低并发场景下可以降低实现复杂度。

**取舍三：`FdPollSet` 的简单实现。** `io_mpx/mod.rs` 中的 `FdPollSet` 接受一个 `Vec<(Arc<dyn FileLike>, IoEvents)>`，其 `register()` 方法遍历所有文件描述符调用 `file.register(context, *events)`<sup>[10](#mod-syscall-ref-10)</sup>。这意味着每次 poll 都会重新注册所有感兴趣的 fd 到同一个 waker，没有增量注册/注销机制。对于 epoll 这种需要大量 fd 的场景，这种线性扫描会随 fd 数量线性增长，而 Linux 内核的 epoll 使用红黑树实现 O(log n) 的查找。这是目前的一个性能盲点。

**取舍四：系统调用的 stubbing 策略。** 许多 Linux 系统调用被显式标记为“dummy”。例如 `timerfd_create`、`bpf`、`io_uring_setup`、`perf_event_open` 等通过 `sys_dummy_fd` 返回一个伪造的 fd<sup>[11](#mod-syscall-ref-11)</sup>；`timer_create`/`timer_gettime`/`timer_settime` 直接返回 `Ok(0)`<sup>[12](#mod-syscall-ref-12)</sup>；`sys_seccomp` 只打印 `warn!` 并返回 0<sup>[13](#mod-syscall-ref-13)</sup>。这暴露了项目的早期状态：优先保证常见 syscall（文件、进程、信号、网络）可用，而性能敏感或安全敏感的特性（seccomp、io_uring）被推迟。对用户程序而言，如果依赖这些 syscall 的行为（如 seccomp 过滤），程序可能会在无任何错误提示的情况下忽略限制。

### 3. 跨模块协同

系统调用层与以下模块紧密耦合：

- **任务管理（`axtask` + `task` 子模块）**：`sys_clone`/`sys_fork`/`sys_execve` 等涉及创建新任务的操作通过 `axtask` 的调度器创建新线程，同时需要修改 `proc_data`（进程级元数据）<sup>[2](#mod-syscall-ref-2)</sup>。`sys_getrusage` 遍历进程内所有线程的 `time.borrow().output()` 累加 CPU 时间，展示了线程级统计如何汇总到进程级<sup>[14](#mod-syscall-ref-14)</sup>。
- **虚拟内存（`starry_vm`）**：所有系统调用参数中的用户态指针（`*mut T`、`*const T`）都通过 `VmPtr`/`VmMutPtr` 进行安全转换。`sys_mmap`/`sys_munmap`/`sys_mprotect` 直接操作 `VmArea` 集合<sup>[2](#mod-syscall-ref-2)</sup>。
- **信号子系统（`starry_signal` + `signal` 子模块）**：信号相关 syscall（`rt_sigprocmask`、`rt_sigaction`、`kill` 等）与 `starry_signal::SignalSet` 和内核信号处理框架交互。`do_select` 和 `do_epoll_wait` 都通过 `with_blocked_signals()` 处理信号屏蔽字<sup>[15](#mod-syscall-ref-15)</sup>。
- **文件系统（`axfs` + `fs` 子模块）**：`sys_read`/`sys_write`/`sys_openat` 等通过 `FD_TABLE`（全局文件描述符表）查找 `FileLike` trait 对象。`sys_getrandom` 甚至直接通过 `FS_CONTEXT.lock().resolve(path)` 打开 `/dev/urandom` 文件<sup>[16](#mod-syscall-ref-16)</sup>。
- **时间（`axhal::time` + `time` 子模块）**：`sys_clock_gettime` 根据时钟 ID 调用 `wall_time()` 或 `monotonic_time()`，同时支持 `CLOCK_PROCESS_CPUTIME_ID` 从线程级时间累加器读取<sup>[17](#mod-syscall-ref-17)</sup>。

### 4. 边角细节与不足

**`#[cfg]` 条件编译的架构覆盖风险 <sup>[6](#mod-syscall-ref-6)</sup><sup>[7](#mod-syscall-ref-7)</sup>**：大量 syscall 被 `#[cfg(target_arch = "x86_64")]` 包裹，但未显式处理 `aarch64`。当编译到 aarch64 时，这些 syscall 将静默消失，用户程序收到 `ENOSYS`。虽然部分 aarch64 系统在 `mod.rs` 末尾的 `_ =>` 分支会被兜底，但缺乏显式的架构门控测试。

**`unsafe` 块缺少 SAFETY 注释 <sup>[5](#mod-syscall-ref-5)</sup>**：`sys_setitimer` 中 `unsafe { new_value.vm_read_uninit()?.assume_init() }` 没有 `// SAFETY:` 注释。该操作假设用户提供的指针指向合法的、已初始化的内存；但如果用户传入非法指针或未初始化内存，将导致未定义行为。`vm_write_slice` 等函数内部同样有 unsafe 操作。

**`sys_reboot` 的 RESTART 命令调用 `system_off()` <sup>[18](#mod-syscall-ref-18)</sup>**：`LINUX_REBOOT_CMD_RESTART` 的处理代码调用了 `axhal::power::system_off()`（关机）而不是重启函数。这应该是复制粘贴错误——重启命令实际执行了关机操作。

**硬编码常量与 Magic Number 的集中程度 <sup>[6](#mod-syscall-ref-6)</sup>**：系统调用号由 `syscalls::Sysno` crate 管理，但架构相关的条件编译散布在整个 match 表达式中。如果想移除某个架构的支持，需要逐个删除 `#[cfg]` 标记，而不是通过 Cargo feature 统一控制。

**`FdPollSet` 的 `unreachable!()` 占位 <sup>[10](#mod-syscall-ref-10)</sup>**：`Pollable` trait 的 `poll()` 方法被实现为 `unreachable!()`，这意味着 `FdPollSet` 不能作为独立的 pollable 使用，只能通过 `poll_io` 宏或外部包装来驱动。如果未来需要独立 poll 测试，这个空实现会立即 panic。

**`sys_getgroups` 的简化实现 <sup>[19](#mod-syscall-ref-19)</sup>**：总是返回 `[0]`，实际是硬编码了只存在一个辅助组（gid=0）。如果用户程序依赖 `getgroups` 获取真正的组成员信息，将收到不正确的数据。

**`sys_prlimit64` 的资源限制检查缺失 <sup>[20](#mod-syscall-ref-20)</sup>**：当新限制的 `rlim_max` 超过当前 `max` 时，代码注释标注了 `// TODO: patch resources` 并直接返回 `Ok(0)`（静默忽略）。这可能导致用户程序误以为资源限制已放宽。
