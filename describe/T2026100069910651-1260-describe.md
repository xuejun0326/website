## 📖 一、总览

该仓库是基于 ArceOS 组件化内核框架构建的 Linux 兼容性实验原型。它并非追求完整的 POSIX 合规，而是以“能跑起来”为首要目标，通过工作区多 crate 组合（ourkernel2026 系列）快速拼装出文件系统、网络、信号、进程间通信等关键子系统。其整体架构遵循 ArceOS 的 axhal 启动与陷阱处理模型，但在系统调用层采用集中式静态派发，并大量依赖硬编码、stub 和外部 C 库（如 lwext4）来加速开发。这种设计使其成为“赛题驱动”的演示内核：在有限时间内验证 Rust 组件化内核支撑 Linux 应用的可行性，而非构建通用操作系统。

该仓库最值得评审注意的三个特点：其一，以 VFS 为载体硬编码 procfs 信息，用静态目录树模拟 /proc 节点，以零运行时代价欺骗 ps/top 等工具，但这种“表面兼容”也导致进程监控工具完全失效（见任务模块）；其二，信号处理与陷阱返回路径的巧妙集成，通过 POST_TRAP 钩子在每次陷入返回前注入信号帧，几乎透明地支持用户态信号 handler，但 OS 级动作（CoreDump/Stop）缺失限制了作业控制（见信号模块）；其三，系统调用层的 stub_bypass 策略将数十个安全敏感调用静默处理为成功，既降低了适配成本，也暴露了原型系统在安全隔离上的根本性妥协（见系统调用模块）。

## 🎯 二、综合评价

### 整体定位
该仓库属于 arceos-starry 家族中的“演示型”项目，旨在探索用 Rust 组件化框架快速搭建一个能跑 Linux 二进制文件的内核原型。与追求完整性的 Linux 兼容层不同，此仓库明显受赛题场景约束：它必须在短时间内让一组特定应用运行，因此处处体现出“够用就好”的工程哲学。从启动流程 [boot.ref:1] 到文件系统 [fs.ref:1]，几乎所有子系统都建立在 ArceOS 提供的基础组件之上，并通过 ourkernel2026_api 等 crate 添加 Linux 兼容胶水。这种架构使得内核核心保持小而精，但同时也意味着大量功能需要通过外部库（如 lwext4）或 stub 来补齐，整体完成度处于“概念验证”阶段。

### 真正的创新点
**1. procfs 的“表面兼容”设计**  
任务模块 [task.ref:3,4] 展示了该仓库最鲜明的设计倾向：通过 Builder 模式构建一个完全静态的 /proc 树，其中 PID 目录、进程状态、CPU 信息全部硬编码为常量字符串。这种“欺骗”用户态工具的手法虽然粗糙，却在开发成本与实用性之间取得了巧妙平衡：它使得 ps、top 等能顺利执行，而不需要实现完整的进程管理子系统。然而，这种设计本质上是原型速成的捷径，一旦进程数、CPU 型号等动态信息需要体现，便会立即失效。

**2. 信号与陷阱的透明协作**  
信号模块 [signal.ref:2,4] 利用 POST_TRAP 钩子，在每次从用户态陷入返回前调用 axsignal::check_signals。如果存在待处理信号，axsignal 会直接修改陷阱帧的返回地址和寄存器，将用户态执行流导向 signal handler，上层 syscall 层无需感知这一过程。这种设计将复杂的栈帧注入逻辑下沉到 axsignal crate，使得系统调用层代码极其简洁。但代价是 OS 级动作（CoreDump、Stop）完全未实现 [signal.ref:2]，导致内核只能执行 SIGTERM 等终止动作，无法支持作业控制。

**3. 系统调用层的集中派发与妥协**  
系统调用层 [syscall.ref:1,5] 用一个巨大的 match 语句处理 180+ 系统调用，通过条件编译应对不同架构的 ABI 差异。这种集中式派发相比原版 ArceOS 的动态注册表模式开销更低，但灵活性差。更值得关注的是其 stub 策略：对于 setuid、chroot、unshare 等安全敏感调用，直接通过 stub_bypass 返回 0 [syscall.ref:5]，这虽让应用认为操作成功，却制造了虚假的隔离感，可能埋下严重安全隐患。

### 取舍判断
该仓的精力明显集中在能让应用“运行”的表面功能上：文件系统实现了 tmpfs 和 DynamicFs 以满足基本文件 I/O 和 proc/sys 伪文件系统需求 [fs.ref:1,5]；网络提供了 TCP/UDP socket 的 FileLike 封装 [net.ref:1]；IPC 则通过 Futex 和 Pipe 支持了基本的并发同步 [ipc.ref:1]。而真正内核深度功能被刻意省略：内存管理虽然支持 mmap/munmap，但共享内存管理器存在锁序问题 [ipc.ref:2] 且无权限检查；信号仅处理终止动作；ext4 实现大量 TODO [fs.ref:6,13]；驱动框架锁定单线程使用 [drivers.ref:5]。这种“重面子、轻里子”的取舍使得系统能快速通过赛题验证，但距离成为安全、健壮的通用内核还有遥远距离。

### 完成度与不足
总体完成度较低，各子系统均遗留明显缺口。信号模块缺失权限检查和 OS 动作 [signal模块不足]；procfs 双版本代码完全重复，维护负担重 [task模块不足]；IPC 的 Pipe 存在两份实现且旧版本残留 [ipc模块不足]；系统调用层重复分支导致 x86_64 编译失败 [syscall模块不足]；多处 unsafe 无注释 [mm模块不足, drivers模块不足, ipc模块不足]。这些现象表明该仓库目前更接近功能演示而非工程产品，若需投入实际使用，需在多方面进行重构和补全。

### 1. 整体架构: 嵌入式内核的启动骨架

`src/main.rs` 同时承担了 crate root 和启动入口两个角色. 文件顶部的 `#![no_std]` 与 `#![no_main]` 声明<sup>[1](#mod-boot-ref-1)</sup>表明这里没有标准 Rust 运行时——没有栈展开, 没有 `std::process::exit`, `main` 也不是被 CRT/crt0 调用的符号. 实际的硬件入口由 `axhal` 提供 (boot facts 记录 `boot_style: "axhal"`), 它负责 CPU 特权级切换、MMU 启用、中断控制器初始化, 最终以 C ABI 调用 `#[unsafe(no_mangle)] fn main()` <sup>[3](#mod-boot-ref-3)</sup>. 这种"运行时先初始化, 再回调内核 main"的模式是 ArceOS 家族的标志性设计, 让内核代码可以像写普通 `fn main()` 一样组织, 而无需手动编写汇编启动代码.

`extern crate alloc;` 和 `#[macro_use] extern crate axlog;` 两行<sup>[1](#mod-boot-ref-1)</sup>暴露了两个隐含依赖: (1) 堆分配器已在 axhal 阶段完成初始化, 否则任何 `Box`/`Vec` 操作都会崩溃; (2) `axlog` 的日志宏 (`info!`, `ax_println!`) 依赖 axhal 提供的早期串口输出, 这是内核启动阶段唯一的诊断通道. 注意 `#[macro_use]` 是 2015 edition 风格, 与 Rust 2024 的 `#[unsafe(no_mangle)]` 并存<sup>[3](#mod-boot-ref-3)</sup>, 暗示代码库的 edition 迁移尚未完成.

### 2. 启动序列: 从文件系统到评测 Shell

启动流程可精确切分为六个阶段, 每阶段都对应一行或一个代码块 <sup>[4](#mod-boot-ref-4)</sup>:

**阶段 A — 根目录重定向 (行 23-24)**: `FS_CONTEXT` 是 `axfs_ng` 提供的全局文件系统上下文, 内部包含一个 `Mutex<FsContext>`. 代码先锁定获取 `root_dir` 的克隆, 释放锁, 再锁定调用 `change_root`. 两段式锁定意味着中间存在一个临界区间隙: 若内核未来支持多核, 另一个 hart 可在间隙中修改 FS_CONTEXT, 导致 `root_dir` 成为悬垂引用. 当前单核启动环境下安全, 但这是一个需要文档化的隐式契约.

**阶段 B — 文件描述符表初始化 (行 25)**: `FD_TABLE.init_new(FdTable::new())` 将一个全局 `FdTable` 置入 `FD_TABLE` 静态变量. 这是 `ourkernel2026_api` 提供的抽象——在 ArceOS 原本的 `axfs` 中, 文件描述符管理通常分散在各模块中, 这里显式集中到一个全局表中, 为 syscall 处理提供统一入口 <sup>[7](#mod-boot-ref-7)</sup>.

**阶段 C — 评测脚本扫描 (行 27-32)**: `OscompRunner::new(&fs).plan()` 在文件系统中搜索 `*_testcode.sh` 脚本 (根据 `NoScriptsFound` 分支的错误消息推断 <sup>[4](#mod-boot-ref-4)</sup>). 如果扫描失败, 系统会 panic 而不是降级运行——这是一个刻意的设计选择: 评测系统没有脚本就等于没有工作负载, 继续启动无意义.

**阶段 D — 文件系统挂载 (行 33)**: `mount_all()` 在阶段 C *之后* 调用. 这个顺序值得关注: 如果 `plan()` 依赖 `/proc` 或 `/dev` 等挂载点, 它就会在挂载前运行并失败. 从代码行为反推, `plan()` 只需要根文件系统上的只读扫描, 而 `mount_all()` 添加的是辅助文件系统. 但这种隐含的排序约束没有任何注释说明.

**阶段 E — 路径分发 (行 35-40)**: `match plan` 有两个分支: `LaunchShell` 将解析出的命令传给 `OscompRunner::shell_invocation`, 生成 `args`/`envs`; `NoScriptsFound` 打印消息后直接调用 `axhal::misc::terminate()`. 这里有一个 Rust 类型系统的细节: `terminate()` 的返回类型应该是 `!` (never type), 否则 match 臂的类型不匹配会导致编译错误.

**阶段 F — 用户态执行与终止 (行 42-44)**: `run_user_app(&args, &envs)` 将控制权交给用户态 Shell (很可能是 busybox ash). 当 Shell 退出时, `exit_code` 被记录到日志, 然后系统再次调用 `terminate()` 关机. 评测结果必须在此前完成写盘, 因为 `terminate()` 不执行任何清理.

### 3. 关键设计取舍: 全局锁、unwrap 语义与硬终止

**全局 FS_CONTEXT 锁**: `FS_CONTEXT` 是一个全局 `Mutex`, 启动过程中被锁定三次 <sup>[4](#mod-boot-ref-4)</sup>. 这种"细粒度短持锁"的模式在单核启动时几乎零开销, 但放弃了一个重要的保障——原子性. 如果未来引入中断处理或后台任务 (例如 IRQ handler 中访问文件系统), 当前的锁模式没有死锁风险 (每次锁定都是独立作用域), 但存在 TOCTOU 竞态. 替代方案是持一把横跨整个启动序列的锁, 代价是锁粒度变粗、持锁时间长.

**`unwrap()`/`expect()` 即致命错误**: 代码中有两处可能 panic 的调用: `change_root().unwrap()` <sup>[5](#mod-boot-ref-5)</sup> 和 `plan().expect(...)` <sup>[4](#mod-boot-ref-4)</sup>. 在通用操作系统中这是不可接受的——用户插一个损坏的磁盘不应导致内核崩溃. 但在这个评测系统的语境下, panic 是合理的: 评测环境是受控的, 文件系统由评测框架预置, 任何初始化失败都意味着环境配置错误, 此时"快速失败、大声崩溃"比静默降级更容易诊断. 不过, 错误信息的质量值得商榷: `"Failed to scan test scripts"` 没有携带根因 (是 I/O error? 解析错误? 权限错误?), 现场调试时需要依赖串口日志的其他部分来推断.

**`axhal::misc::terminate()` 作为唯一退出路径**: 系统在两处调用 `terminate()` (行 39 和行 44 <sup>[4](#mod-boot-ref-4)</sup>), 没有任何卸载文件系统、回写缓冲区、关闭设备的步骤. 这在 ArceOS unikernel 中是惯用法——unikernel 假设单一应用, 应用退出即机器关闭, 无需清理. 但对于一个写入评测结果到磁盘的系统, `terminate()` 之前缺少显式的 `sync`/`flush` 可能导致数据丢失.

### 4. 跨模块协同: 陷阱处理与启动的隐式耦合

boot facts 列出了四个注册的 trap handler <sup>[7](#mod-boot-ref-7)</sup>:
- **SYSCALL** → `handle_syscall` (`src/syscall.rs:28`): 依赖启动阶段初始化的 `FD_TABLE` 来解析文件描述符操作.
- **PAGE_FAULT** → `handle_page_fault` (`src/mm.rs:13`): 依赖 axhal 在启动早期建立的页表结构.
- **POST_TRAP** → `post_trap_callback` (`api/src/imp/task/signal.rs:76`): 信号处理钩子, 在每次陷阱返回前检查是否有待递送信号.
- **IRQ** → `handler_irq` (`arceos/modules/axhal/src/irq.rs:38`): 标准中断分发.

这些 handler 的注册不在 `main.rs` 中——它们由 axhal 的 trap 框架在调用 `main()` 之前完成绑定. 这构成了一个重要的隐式契约: `main()` 中初始化的数据结构 (FD_TABLE, FS_CONTEXT) 必须在第一个 syscall 到达之前就绪. 由于 `main()` 是单线程同步执行的, 且用户态 Shell 直到 `run_user_app` 才启动, 这个契约自然满足. 但如果将来在启动期间启用了中断或后台任务, 这个假设就会破灭.

### 5. 边角细节与不足

1. **<sup>[5](#mod-boot-ref-5)</sup> `unwrap()` 丢弃错误信息**: `change_root` 返回的 `Result` 被 `unwrap()` 吞掉, 如果失败, 只有一行 panic 消息, 没有文件系统状态、inode 编号等诊断上下文. 对于需要远程调试的评测集群, 这增加了故障定位成本.

2. **<sup>[4](#mod-boot-ref-4)</sup> `mount_all()` 排序缺乏文档化**: `plan()` 在 `mount_all()` 之前执行, 意味着脚本扫描不能依赖任何非根文件系统. 这个排序约束没有注释, 未来维护者可能错误地将 `mount_all()` 提前, 无意间改变 `plan()` 的运行环境.

3. **<sup>[6](#mod-boot-ref-6)</sup> 缺少 `// SAFETY:` 注释**: `#[unsafe(no_mangle)]` 是 Rust 2024 的新要求——`no_mangle` 因为可能导致符号冲突而被标记为 `unsafe`. 代码中没有任何 SAFETY 注释说明符号 `main` 的调用者 (axhal) 遵守了哪些约定 (ABI、栈对齐、参数传递). 这是技术债.

4. **<sup>[4](#mod-boot-ref-4)</sup> 双重 `terminate()` 调用点**: `main()` 中有两个独立的 `terminate()` 调用 (无脚本时与 Shell 退出后). 如果未来需要添加"优雅关机"逻辑 (如写评测结果、unmount), 必须同时修改两处, 容易遗漏. 更好的做法是将关机逻辑收敛到一个函数.

5. **<sup>[4](#mod-boot-ref-4)</sup> FS_CONTEXT 三次加锁的竞态隐患**: 三次 `lock()` 调用之间没有原子性保证. 当前单核安全, 但代码没有用 `debug_assert!` 或类型系统 (如 `&mut` 借用) 来防御未来多核场景. 如果在启动后期引入 timer interrupt, 一个中断处理程序可能在锁间隙中修改 FS_CONTEXT.

6. **<sup>[5](#mod-boot-ref-5)</sup> `FD_TABLE.init_new()` 没有错误处理**: 虽然 `FdTable::new()` 大概率是 infallible 的, 但 `init_new` 的语义不明确——它可能返回 `Result` 也可能 panic. 从代码看它没有使用 `unwrap()`/`expect()`, 说明它要么绝对安全, 要么内部自行 panic. 这个接口的语义需要明确.

7. **<sup>[1](#mod-boot-ref-1)</sup> `extern crate alloc;` 依赖隐式分配器**: 代码中没有显式的 `#[global_allocator]` 声明, 堆分配器必须由 axhal 或另一个模块在 `main()` 之前初始化. 如果这个隐式依赖被破坏 (例如更换 axhal 版本), `FdTable::new()` 中的 `Box` 分配会直接导致 undefined behavior.

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 内存管理模块基于泛型页表抽象 PageTable64<M, PTE, H>，通过 PagingMetaData (架构常量)、GenericPTE (页表项格式) 和 PagingHandler (OS 帧分配器) 三个 trait 解耦硬件差异。其上 AddrSpace 封装进程地址空间操作，向上提供 mmap/munmap/mprotect 等系统调用。相比 ArceOS 基线，增加了大页支持、脏/访问位追踪、跨地址空间页表拷贝、以及 percpu 标志安全访问用户内存等特性。

### 1. 分层抽象: 从硬件页表到地址空间

内存管理的核心是 `PageTable64<M, PTE, H>` 泛型结构<sup>[1](#mod-mm-ref-1)</sup>，它接受三个 trait 参数：`PagingMetaData` 定义页表层数、物理/虚拟地址位宽、TLB 刷新方法等架构常量<sup>[2](#mod-mm-ref-2)</sup>；`GenericPTE` 抽象页表项 flags 读写，由 `page_table_entry` crate 提供 RISCV/LA64 具体实现；`PagingHandler` 暴露 OS 侧的物理帧分配 (`alloc_frame`/`dealloc_frame`) 和 `phys_to_virt` 转换<sup>[2](#mod-mm-ref-2)</sup>。这使得同一份页表逻辑可运行于 RISC-V (Sv39/Sv48) 和 LoongArch64 等不同架构。

在此之上，`axmm::AddrSpace` (未展示源码但大量使用) 封装了进程级地址空间操作：`map_alloc` (按需分配物理页)、`map_linear` (直接映射已知物理地址)、`map_file` (文件映射)、`find_free_area` (查找空闲虚拟区间) 等。系统调用层 (`sys_mmap`/`sys_munmap`/`sys_mprotect`) 将 Linux ABI 标志位转换为内部类型，完成参数校验后委托给 `AddrSpace` 执行<sup>[3](#mod-mm-ref-3)</sup>。

### 2. 关键设计取舍

**大页自动提升**：`map_region` 在 `allow_huge` 为真且虚拟地址、物理地址、长度均对齐时，自动尝试 1G/2M 大页映射<sup>[4](#mod-mm-ref-4)</sup>。这能显著减少 TLB miss，但要求 `unmap_region` 按实际映射页大小逐页解除——代码中通过 `assert!(page_size.is_aligned(vaddr_usize))` 强制对齐<sup>[5](#mod-mm-ref-5)</sup>，若页表损坏则内核 panic。这种 assert 在页表由内核全权控制时合理，但若未来支持用户态页表操作则可能成为攻击面。

**TLB 刷新控制**：`TlbFlush` 和 `TlbFlushAll` 是 `must_use` 类型<sup>[2](#mod-mm-ref-2)</sup>，强迫调用者在修改映射后显式 `flush()` 或 `ignore()`。`map_region` 等批量操作提供 `flush_tlb_by_page` 参数，允许延迟刷新以分摊开销<sup>[4](#mod-mm-ref-4)</sup>。这一设计避免了未刷新 TLB 的常见 bug，但要求调用者正确理解 ignore 的语义——例如在 `map_region` 循环中，若 `flush_tlb_by_page` 为 false，每条 `tlb.ignore()` 只是抑制警告，最终的批量刷新由 `TlbFlushAll` 完成。

**跨地址空间页表共享**：`copy_from` 直接通过 `copy_from_slice` 拷贝顶级页表项<sup>[6](#mod-mm-ref-6)</sup>，用于将内核映射复制到用户地址空间。在 aarch64/loongarch64 上因使用独立页表基址寄存器 (TTBR0_EL1/PGDL) 而跳过此步<sup>[7](#mod-mm-ref-7)</sup>。这种共享避免了重复映射的开销，但引入双重释放风险：若源和目标页表先后 drop，中间级页表会被释放两次，因为 `Drop` 实现遍历并释放所有非叶子页表帧<sup>[11](#mod-mm-ref-11)</sup>。

**安全用户内存访问**：`ACCESSING_USER_MEM` percpu 标志允许内核在访问用户空间地址时容忍缺页<sup>[8](#mod-mm-ref-8)</sup>。这是一把作用域锁：设置后方可安全读取用户传入的指针，完成后立即清除。但当前实现是简单的 bool 切换，不支持嵌套访问（如缺页处理中再次访问用户内存）。

### 3. 跨模块协同

**mmap 与设备内存**：当 mmap 作用于设备文件时 (如 framebuffer)，`try_get_device_memory` 通过 `virt_to_phys` 获取物理地址<sup>[9](#mod-mm-ref-9)</sup>，再调用 `aspace.map_linear` 直接映射到用户空间<sup>[3](#mod-mm-ref-3)</sup>，实现零拷贝帧缓冲访问。如果请求长度超过设备内存，剩余部分会通过 `map_alloc` 补充分配匿名页。

**ELF 加载**：`load_user_app` 解析 ELF 的 LOAD 段，对每个段依次调用 `uspace.map_alloc` 分配虚拟空间、`uspace.write` 写入段数据<sup>[7](#mod-mm-ref-7)</sup>。同时映射信号跳板 (通过 `virt_to_phys` 获取物理地址后线性映射) 和栈/堆区域。期间还处理 shebang 脚本和 ELF interpreter 递归加载。

**共享内存**：`SharedMemoryManager` 维护全局 key→Arc<SharedMemory> 映射<sup>[10](#mod-mm-ref-10)</sup>，创建时通过全局分配器分配连续物理页。但 `Drop` 时直接释放物理帧，不检查是否有进程仍在映射该段，存在 use-after-free 风险。

### 4. 边角细节与不足

- **`unsafe` 无 SAFETY 注释**：`table_of` 和 `table_of_mut` 通过 `phys_to_virt` 构造裸指针切片<sup>[12](#mod-mm-ref-12)</sup>，但未说明安全前提（物理地址必须指向已分配的页表页且生命周期合法）。
- **`unreachable!()` 限制页表层数**：`top_level_idx_range`、`get_entry` 等函数在 `LEVELS` 不为 3 或 4 时触发 `unreachable!()`<sup>[13](#mod-mm-ref-13)</sup>，无法支持未来可能的 5 级页表。
- **TODO 标记的未完成逻辑**：`sys_mmap` 中 MAP_SHARED 分支的零初始化注释 `// TODO: 仅在MAP_ANONYMOUS时才zero`<sup>[14](#mod-mm-ref-14)</sup>，当前对共享映射一律填零，可能错误破坏非匿名共享映射内容。
- **断言可能 panic**：`unmap_region` 的 `assert!(page_size.is_aligned(vaddr_usize))` 缺乏错误处理<sup>[5](#mod-mm-ref-5)</sup>，若页表元数据损坏将直接内核 panic，而非常规错误返回。
- **共享内存生命周期缺失**：`SharedMemory` 的 Drop 直接释放物理页<sup>[10](#mod-mm-ref-10)</sup>，但未追踪映射这些页的进程地址空间，一旦最后一个 Arc 引用被清除，正在使用该共享内存的进程将访问已释放内存。
- **`copy_from` 无冲突检测**：直接覆盖目标页表项<sup>[6](#mod-mm-ref-6)</sup>，若目标区间已有映射，旧的中间页表帧会泄漏。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块的核心抽象是通过 /proc 伪文件系统向用户态暴露进程与调度信息, 底层依托 ourkernel2026_vfs 的 DynamicFs/DynamicDir/SimpleFile 机制构建. 它解决的是应用兼容性问题: 大量 Linux 工具(ps/top/htop)依赖 /proc 读取进程状态、CPU 信息、内存统计等, 本模块以全量硬编码 stub 数据满足这些读取期望. 与同家族 ArceOS 基线相比, 该仓库增加了两层 proc 实现(api/ 与 src/), 其中 api 版本额外集成了 axtask 的实时中断计数与 proc_observe 观测报告.

### 1. 核心抽象: 以 VFS 为载体的进程信息暴露

进程与任务调度的可观测性在本仓库中通过 `/proc` 伪文件系统实现, 而非传统内核中 task_struct 链表遍历. 入口函数 `new_procfs()` 返回一个 `Filesystem<RawMutex>` 对象, 该对象可被挂载到根文件系统供用户态访问 <sup>[1](#mod-task-ref-1)</sup><sup>[7](#mod-task-ref-7)</sup>. 其构建过程采用 Builder 模式: 创建 `DynamicDir` 根目录, 逐层添加 `sys/kernel`、`sys/fs`、`self` 等子目录, 每个叶子节点是一个 `SimpleFile`, 其内容由闭包或常量字符串提供 <sup>[2](#mod-task-ref-2)</sup>.

这一设计的核心取舍是 **用空间换开发速度**: 所有 `/proc` 条目均为编译期硬编码的 `const &str`, 完全放弃动态进程信息追踪. 好处是零运行时开销、无锁竞争、无需维护进程描述符; 代价是 `ps aux` 看到的永远是 PID 1 的 systemd 和 PID 10 的 bash, 且进程状态永远为 S(sleeping) <sup>[3](#mod-task-ref-3)</sup><sup>[4](#mod-task-ref-4)</sup>. 对于赛题场景中需要 `procfs` 节点存在性检查的应用, 这已足够; 但对于依赖准确进程信息的工具链, 这构成功能缺陷.

### 2. 关键设计取舍: 双版本 proc 实现的工程意图

仓库中存在两份高度相似的 proc 实现:

- `src/fs/imp/proc.rs`: 最小化版本, 仅依赖 `crate::fs::dynamic` 内部 VFS 抽象, 提供 core_pattern、cpuinfo、meminfo、mounts、self/status、self/maps 等基础条目 <sup>[5](#mod-task-ref-5)</sup>.
- `api/src/core/fs/imp/proc.rs`: 增强版本, 额外依赖 `crate::core::fs::pseudo` VFS 抽象, 并导入 `proc_observe` 模块的三个报告函数 (`file_page_cache_report`, `runtime_metrics_report`, `oscomp_status_report`) 以及 `axtask::get_irq_count()` 用于 `/proc/interrupts` <sup>[6](#mod-task-ref-6)</sup><sup>[8](#mod-task-ref-8)</sup>.

两份代码共享完全相同的常量定义 (PID_MAX、SHMMAX、STAT、CPUINFO 等), 但 import 路径不同反映出两套 VFS 实现 (`pseudo::dynamic` vs `dynamic::dynamic`). 这暗示 api 层可能是面向外部应用的更高层封装, 而 src 层是内核内部使用的精简版. 这种双轨制避免了将观测/诊断功能强行打入核心内核, 但也带来了 **常量同步维护成本**: 若需修改 CPUINFO 字符串, 必须同时改动两处 <sup>[9](#mod-task-ref-9)</sup>.

### 3. 跨模块协同: 与 axtask、proc_observe、syscall 的配合

该模块并非孤立存在. 从 api 版本可见三条协同路径:

- **与 axtask 的配合**: `/proc/interrupts` 条目不是硬编码, 而是通过 `axtask::get_irq_count()` 动态获取中断计数 <sup>[10](#mod-task-ref-10)</sup>. 这是整个 proc 实现中唯一实时数据点, 说明底层 ArceOS 的 axtask crate 维护了 percpu 中断统计.
- **与 proc_observe 的配合**: `file_page_cache_report` 等三个函数从 `proc_observe` 模块导入, 提供文件页缓存、运行时指标、OS 竞赛状态的观测报告 <sup>[6](#mod-task-ref-6)</sup>. 这些是赛事特定的诊断接口, 超出标准 Linux /proc 范畴.
- **与 syscall 层的配合**: 系统调用列表显示 clone(220)、fork、execve(221)、exit(93) 等核心进程操作均已实现, 指向 `src/syscall.rs` <sup>[11](#mod-task-ref-11)</sup>. 这些 syscall 的实际实现(无法直接阅读)必然与 axtask 的任务管理交互, 而 /proc 层则是这些操作结果的"只读视图". 但当前 /proc 完全不反映 syscall 的副作用——fork 一个新进程不会在 /proc 下创建对应目录.

### 4. 边角细节与不足

**<sup>[9](#mod-task-ref-9)</sup> 双份代码维护风险**: `api/src/core/fs/imp/proc.rs` 与 `src/fs/imp/proc.rs` 的常量部分完全重复. 修改 CPUINFO 或 STAT 字符串时容易遗漏其一, 导致两套 proc 挂载点呈现不一致的信息. 建议将共享常量抽取到公共模块.

**<sup>[3](#mod-task-ref-3)</sup> 硬编码 PID 目录**: `/proc/1` 和 `/proc/10` 是 builder 中手动添加的固定目录, 无论系统实际运行多少任务, 用户态 `ls /proc` 只能看到 1 和 10 两个数字目录. 任何遍历 /proc 以枚举进程的工具(如 `killall`)将完全失效.

**<sup>[4](#mod-task-ref-4)</sup> 虚假进程状态**: STAT 常量描述的是"systemd"进程且状态为 S(sleeping), STATUS 描述的是"bash"进程. 两个不同 PID 目录下挂载的 stat 内容完全相同——这会导致 `cat /proc/1/stat` 和 `cat /proc/10/stat` 返回一样的数据, 明显不符合 POSIX 语义.

**<sup>[8](#mod-task-ref-8)</sup> api 与 src 的 VFS 依赖不对称**: api 版本引入了 `proc_observe` 和 `axtask` 依赖, 但 src 版本没有. 若构建系统选择编译 src 版本, `/proc/interrupts` 和观测报告将完全缺失, 且编译不会报错——这是隐式的功能退化.

**<sup>[10](#mod-task-ref-10)</sup> 唯一动态条目的线程安全**: `axtask::get_irq_count()` 在每次读取 `/proc/interrupts` 时被调用. 若该函数内部持锁, 而用户态高频读取该文件, 可能构成中断统计锁的争用热点. 代码中未见任何缓存或防抖机制.

**<sup>[12](#mod-task-ref-12)</sup> 架构/平台假设**: CPUINFO 硬编码为 AMD Ryzen 7 7840HS (family 25 model 116), 且包含 AVX-512 等指令集标志. 若实际运行在 Intel 平台或较旧的 AMD 平台上, `/proc/cpuinfo` 将报告错误信息. `#[cfg]` 条件编译未按 CPU 厂商分流.

**<sup>[7](#mod-task-ref-7)</sup> 文件系统魔术数**: `DynamicFs::new_with` 的第二个参数 `0x9fa0` 是 ext2/ext4 的默认魔术数 PROC_SUPER_MAGIC, 但注释缺失, 维护者可能不理解该值的来源.

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块基于自研 `ourkernel2026_vfs` 的泛型特质体系构建，核心抽象是 `FilesystemOps`、`NodeOps`、`FileNodeOps`、`DirNodeOps` 四个 trait，泛型参数 `M: RawMutex` 允许替换同步原语。实现了三类文件系统：基于 Slab 分配的内存文件系统 tmpfs (MemoryFs)、面向 pseudo/proc 的动态文件系统 DynamicFs、以及通过 lwext4 绑定提供块级操作的 ext4 支持。与同家族基线相比，本仓引入了 `DynamicDirBuilder` 构建器模式来简化伪文件系统的目录树构造，并额外实现了 ext4 的底层 I/O 与符号链接支持。

### 1. VFS 泛型特质体系与 DirNode 缓存层

`ourkernel2026_vfs` 以 `RawMutex` 为泛型参数构建了一套可替换锁机制的文件系统抽象 <sup>[4](#mod-fs-ref-4)</sup>。`FilesystemOps<M>` 定义了文件系统级操作（`root_dir`、`stat`），`NodeOps<M>` 则提供节点通用接口（`inode`、`metadata`、`update_metadata`）<sup>[1](#mod-fs-ref-1)</sup>。`FileNodeOps<M>` 与 `DirNodeOps<M>` 分别扩展出文件读写（`read_at`、`write_at`）与目录操作（`lookup`、`create`、`unlink`、`rename`）<sup>[16](#mod-fs-ref-16)</sup>。`DirNode<M>` 作为缓存层包裹 `DirNodeOps<M>`，内部维护一个 `BTreeMap<String, DirEntry<M>>` 缓存 <sup>[4](#mod-fs-ref-4)</sup>，并提供 `open_file_or_create`、`rename` 等高层语义方法，负责 `verify_entry_name` 校验 <sup>[10](#mod-fs-ref-10)</sup> 和缓存失效 <sup>[4](#mod-fs-ref-4)</sup>。这种“trait 定义核心逻辑 + DirNode 提供缓存与校验”的分层，使得具体文件系统实现只需关注存储原语，而不用重复处理路径合法性、缓存一致性等横切关注点。

### 2. tmpfs：基于 Slab 的 inode 分配与 n_link 引用计数

MemoryFs 以 `Slab<Arc<Inode>>` 管理 inode <sup>[1](#mod-fs-ref-1)</sup>，分配时使用 `vacant_entry` 得到递增的 ino；`Inode` 内嵌 `Mutex<Metadata>` 与 `NodeContent` 枚举（`File(FileContent)` 或 `Dir(DirContent)`）<sup>[1](#mod-fs-ref-1)</sup>。文件内容直接存为 `Vec<u8>`，目录内容存为 `BTreeMap<FileName, InodeRef>`，其中 `FileName` 自定义了 `Ord`，使 `.` 与 `..` 排序到最前 <sup>[1](#mod-fs-ref-1)</sup>。节点引用通过 `InodeRef` 新类型管理：构造时递增 `n_link` <sup>[11](#mod-fs-ref-11)</sup>，析构时调用 `release_inode` 递减 `n_link` 并尝试回收 <sup>[2](#mod-fs-ref-2)</sup>。《》

`release_inode` 的回收条件为 `n_link == 0 && Arc::strong_count(inode) == 2` <sup>[2](#mod-fs-ref-2)</sup>，这一硬编码的假设（`Slab` 与当前调用者各持一个 `Arc`）极其脆弱：任何外部缓存或异步任务若额外持有 `Arc`，inode 将永久泄漏；反之，若 `strong_count` 意外小于 2，则可能提前释放导致 use-after-free。此外，`MemoryNode::new` 中显式 `drop(inodes)` 以避免重入锁 <sup>[1](#mod-fs-ref-1)</sup>，这种手动锁释放是死锁防御的典型标志。`rename` 方法前的 `// TODO: atomicity` 注释 <sup>[3](#mod-fs-ref-3)</sup> 表明当前实现仅简单地先 `remove` 再 `insert`，中间无任何日志或回滚机制，crash-consistent 无保证。

### 3. DynamicFs：面向 pseudo 文件系统的构建器模式

DynamicFs 针对 `/proc`、`/sys` 等伪文件系统设计，核心思想是“预先声明目录树结构，延迟实例化节点”。`DynamicDirBuilder` 允许逐层添加子节点，每个子节点可以是 `DirMaker`（闭包，在访问时创建子目录）或 `Arc<dyn FileNodeOps>`（文件操作对象）<sup>[5](#mod-fs-ref-5)</sup>。`DynamicDir::read_dir` 通过链式组合 `DOT`/`DOTDOT`、预设子节点列表与 `pseudo_ops.list_children()` 的结果，统一呈现目录条目 <sup>[5](#mod-fs-ref-5)</sup>。《》

`PseudoDirOps` trait 提供 `list_children` 与 `get_child` 两个方法，允许动态生成目录项（例如 `/proc/self`），而无需预先填充完整的 `children` 表 <sup>[5](#mod-fs-ref-5)</sup>。`DynamicDir::lookup` 依次查找静态子节点与 `pseudo_ops`，这种“静态优先、动态兜底”的顺序保证了静态定义的节点不会被动态生成覆盖。但 `create`、`unlink`、`rename` 等变更操作一律返回 `VfsError::EPERM` <sup>[8](#mod-fs-ref-8)</sup>，明确了伪文件系统的只读语义 —— 不过这也意味着不存在任何原子性隐患。

### 4. ext4 块级操作与原子性缺失

ext4 的 `InodeRef<Hal>` 实现了块级 `read_at` 与 `write_at`，核心技巧是合并连续的物理块进行批量 I/O <sup>[7](#mod-fs-ref-7)</sup>。`WritebackGuard::new(bdev)` <sup>[7](#mod-fs-ref-7)</sup> 创建一个 RAII guard 来标记块设备的写回区间，确保多块写入的缓存一致性。符号链接的实现采用双路径：短链接（< `size_of::<[u32; 15]>()`）直接存储在 `ext4_inode` 的 `blocks` 字段内 <sup>[6](#mod-fs-ref-6)</sup>；长链接则分配数据块并通过 `ext4_fs_append_inode_dblk` 写入。`set_symlink` 中直接通过裸指针 `self.inner.inode as *mut u8` 操作结构体字段 <sup>[6](#mod-fs-ref-6)</sup>，没有任何 SAFETY 注释，这依赖于对 lwext4 C 结构体布局的硬编码假设。

`set_len` 的扩展逻辑仍有 `// TODO: correct implementation` <sup>[13](#mod-fs-ref-13)</sup>，当前简单地将新分配的块填零，未考虑文件系统的碎片整理或 extent 优化。`read_at` 与 `write_at` 大量使用 `unsafe` 调用 C 函数（如 `ext4_block_readbytes`），但同样缺乏契约注释。

### 5. 边角细节与不足

- **<sup>[2](#mod-fs-ref-2)</sup> `release_inode` 引用计数假设脆弱**：`Arc::strong_count(inode) == 2` 是一个隐式契约，无编译期保证，容易因重构引入泄漏或 double-free。
- **<sup>[3](#mod-fs-ref-3)</sup> rename 非原子**：tmpfs 的 `rename` 在一个锁内执行 `remove` 和 `insert`，但未考虑进程在两者之间被调度或系统崩溃；注释明示未实现原子性。
- **<sup>[13](#mod-fs-ref-13)</sup> ext4 `set_len` 未完成**：TODO 标记表明扩展文件时直接写零块是占位实现，可能影响大文件性能与空间利用率。
- **<sup>[6](#mod-fs-ref-6)</sup> ext4 多处裸指针操作无 SAFETY 注释**：`set_symlink` 等函数通过 `as *mut u8` 直接写入 `ext4_inode` 字段，依赖 C 结构体布局，未声明安全前提。
- **<sup>[9](#mod-fs-ref-9)</sup> 双份 tmpfs 代码**：`api/src/core/fs/imp/tmp.rs` 与 `src/fs/imp/tmp.rs` 内容完全相同，可能是复制粘贴或构建脚本冗余，增加了维护负担。
- **<sup>[15](#mod-fs-ref-15)</sup> 硬编码魔数**：`MemoryFs::stat` 返回 `0x01021994` 作为 tmpfs 的 f_type，该值未命名且与标准 tmpfs 魔数 (`0x01021994` 或 `TMPFS_MAGIC`) 可能不一致。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 本模块为 arceos-starry 内核提供 Linux 兼容的信号系统调用层, 核心依赖 axsignal crate 实现信号存储/掩码/队列/栈帧操作, 通过 POST_TRAP 钩子在每次从用户态陷入返回前检查并派发信号. 与 ArceOS 基线相比, starry 新增了完整的 14 个信号相关 syscall, 但 OS 级动作 (CoreDump/Stop/Continue) 均未实现, 用户自定义 handler 的实际栈帧注入委托给 axsignal, 本层只处理 Terminate 等默认动作.

### 1. 信号架构的分层设计: axsignal 内核与 syscall 适配层

该模块并非从零实现信号机制, 而是在 `axsignal` crate 之上构建 Linux syscall 兼容层 <sup>[1](#mod-signal-ref-1)</sup>. `axsignal` 负责信号队列、阻塞掩码、信号栈 (sigaltstack) 管理以及最关键的用户态 handler 栈帧注入; 本模块则处理三件事: (a) 将 Linux 风格的 `sigaction`/`sigset_t` 结构转换为 `axsignal` 内部表示; (b) 实现 `kill`/`tkill`/`tgkill` 等发送路径; (c) 在陷入返回路径上调用 `check_signals` 并处理 `axsignal` 返回的 OS 级动作.

这种分层带来了清晰的职责边界, 但也导致一个容易被误读的现象: `check_signals` 中对 `SignalOSAction::Handler` 分支是空操作 <sup>[2](#mod-signal-ref-2)</sup>. 因为 `axsignal::signal.check_signals(tf, restore_blocked)` 已经在返回前修改了 TrapFrame——将用户态 PC 指向 handler、压入 sigframe、设置返回地址为 restorer (通常是 `__kernel_rt_sigreturn`). 本模块只需处理那些不涉及用户态代码跳转的系统动作 (Terminate/CoreDump/Stop/Continue).

`KernelSigAction` 结构体是 Linux ABI 与内部 `SignalAction` 之间的桥梁 <sup>[3](#mod-signal-ref-3)</sup>. 它通过 `#[repr(C)]` 保证与 Linux `struct sigaction` 内存布局一致, `signal_action_from_kernel`/`signal_action_to_kernel` 完成双向转换. 其中对 `SIG_DFL`/`SIG_IGN` 的辨别使用了指针值比较: `handler == 0` 为 Default, `handler as usize == 1` 为 Ignore <sup>[8](#mod-signal-ref-8)</sup>, 这直接对应 Linux 内核对这两个常量的定义.

### 2. POST_TRAP 钩子与信号派发时机

信号检查发生在每次陷入返回前夕, 通过 `#[register_trap_handler(POST_TRAP)]` 注册的 `post_trap_callback` 实现 <sup>[4](#mod-signal-ref-4)</sup>. 该回调只在 `from_user == true` 时才调用 `check_signals(tf, None)`, 即仅当陷入来自用户态时才检查——内核态发生的异常不应触发用户信号. 这是标准 Unix 信号异步派发模型在 ArceOS 陷入框架上的映射.

`check_signals` 的第二个参数 `restore_blocked` 支持在信号处理完毕后恢复被替换的阻塞掩码——这是 `sys_rt_sigsuspend` 的关键机制: 该 syscall 用 `mem::replace` 将当前掩码临时替换为用户指定的 `set` <sup>[9](#mod-signal-ref-9)</sup>, 然后在 `check_signals` 被触发时通过 `Some(old_blocked)` 恢复原掩码. 循环中的 `current_process_data().signal.wait_signal()` 负责将线程置入阻塞态, 避免忙等 <sup>[10](#mod-signal-ref-10)</sup>.

### 3. 信号路由: 进程 / 线程 / 进程组三级发送

发送侧实现了三种粒度的路由 <sup>[5](#mod-signal-ref-5)</sup>:
- `send_signal_thread(tid, sig)`: 按线程 ID 精确投递, 用于 `tkill`/`tgkill`.
- `send_signal_process(pid, sig)`: 投递到进程的线程组, 由 `axsignal` 内部选择合适的线程.
- `send_signal_process_group(pgid, sig)`: 遍历进程组内所有进程分别投递, 失败不中断遍历.

`sys_kill` 对 `pid` 的四路分发是 POSIX 标准语义 <sup>[6](#mod-signal-ref-6)</sup>: `pid > 0` 发给指定进程; `pid == 0` 发给同进程组; `pid == -1` 广播给所有进程; `pid < -1` 发给进程组 `-pid`. 值得注意的是 `pid == -1` 的广播路径有一个 TODO 注释提示应跳过 init 进程 <sup>[11](#mod-signal-ref-11)</sup>, 否则用户态可能意外杀死 PID 1 导致系统崩溃.

`sys_rt_sigqueueinfo` 和 `sys_rt_tgsigqueueinfo` 支持携带自定义 `siginfo` 的排队信号, 但有权限限制: 若 `sig.code() != SI_USER` 且发送者与目标不属于同一进程, 则返回 `EPERM` <sup>[12](#mod-signal-ref-12)</sup>. 这是 POSIX 实时信号扩展的安全要求.

### 4. 边角细节与不足

**5 条具体风险点:**

1. **权限检查缺失** <sup>[7](#mod-signal-ref-7)</sup>: `sys_kill`、`sys_tkill`、`sys_tgkill` 中均有 `// TODO: should also check permissions` 注释. 当前任何进程都可以向任意其他进程/线程发送信号, 这违反了 Linux 的权限模型 (需 uid 匹配或有 `CAP_KILL`), 构成安全漏洞.

2. **CoreDump / Stop / Continue 未实现** <sup>[13](#mod-signal-ref-13)</sup>: `SignalOSAction::CoreDump` 和 `Stop` 被 fallback 为 `sys_exit_impl`, `Continue` 完全是空操作. 这意味着 SIGSTOP/SIGCONT 的作业控制功能缺失, SIGSEGV/SIGABRT 等不会生成 core dump. 对于依赖这些信号的 shell 和调试器, 行为不符合预期.

3. **unsafe 块无 SAFETY 注释** <sup>[14](#mod-signal-ref-14)</sup>: 多处 `unsafe { *oldset = *blocked }`、`unsafe { *act }` 等直接解引用用户态指针. 虽然 `UserPtr::get()` / `UserConstPtr::get()` 提供了可读性验证, 但指针的 `Sync`/`Send` 安全性、TOCTOU 竞态 (用户态可能在验证后修改映射) 均未说明. 至少 8 处 unsafe 缺少 SAFETY 文档.

4. **`sys_rt_sigsuspend` 的 TOCTOU 窗口** <sup>[9](#mod-signal-ref-9)</sup><sup>[10](#mod-signal-ref-10)</sup>: 该函数先调用 `check_signals` 检查待处理信号, 若无信号再调用 `wait_signal()` 阻塞. 如果信号在 `check_signals` 返回 false 之后、`wait_signal()` 进入阻塞之前到达, 线程可能永远阻塞. Linux 通过原子化的 `schedule()` 配合信号检查解决此问题, 当前实现取决于 `wait_signal()` 内部是否使用了自旋锁 + 条件变量来闭合这个窗口.

5. **信号排队容量无上限** <sup>[15](#mod-signal-ref-15)</sup>: `send_signal_thread`/`send_signal_process` 直接调用 `axsignal` 的 `send_signal`, 代码中未见对排队深度的限制. Linux 对标准信号只保留一个 pending 位, 对实时信号有队列长度限制 (受 `RLIMIT_SIGPENDING` 控制). 如果攻击者高速发送实时信号, 可能导致内存耗尽.

另外, `sys_sigaltstack` 在设置新栈前验证了用户态指针可读性 <sup>[16](#mod-signal-ref-16)</sup> (`stack_ptr.get_as_array(ss.size)`), 并检查 `ss.size > MINSIGSTKSZ`, 这个防御性编程值得肯定, 但没有记录为何要同时检查读权限——推测是为了防止进程指定一个未映射地址导致后续 handler 执行时缺页.

- 权限检查全线缺失: sys_kill/sys_tkill/sys_tgkill 均有 TODO 标记, 任意进程可发信号给任意目标 (安全漏洞).
- CoreDump/Stop/Continue 三个 OS 动作未实现: CoreDump 和 Stop 退化为 exit, Continue 为 no-op, 作业控制功能不可用.

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块通过 Pipe、Futex、Shared Memory 三条路径实现进程间通信。Pipe 以 FileLike trait 接入 VFS，基于 64KB 环形缓冲区与 task_yield 协作调度；Futex 用 WaitQueue 加 10ms 轮询实现跨线程同步，并集成信号中断感知；共享内存通过全局 SHARED_MEMORY_MANAGER + 页表线性映射暴露物理页。与同家族基线相比，该仓新增了非阻塞管道语义、FUTEX_REQUEUE 及信号感知等待，但存在两套 Pipe 实现并存、锁序颠倒、硬编码限制、bitset futex 未完整支持等工程债务。消息队列、信号量、消息传递等 SysV IPC 虽在 syscall 分发表中声明，其实现文件未在本文分析范围内。

### 1. 核心抽象：FileLike、WaitQueue 与全局共享内存管理器

进程间通信子系统围绕三条不同的内核抽象展开。

**Pipe** 的核心是将一个 64KB 的环形缓冲区 `PipeRingBuffer` 包装为 `FileLike` trait 的实现 <sup>[1](#mod-ipc-ref-1)</sup>。该 trait 来自 `ourkernel2026_vfs` 生态，使得管道可以像普通文件一样通过 fd 表管理——`read()`/`write()`/`poll()` 均通过 trait 方法暴露。`PipeShared` 结构体承载实际数据：`Mutex<PipeRingBuffer>` 保证读写互斥，`readers`/`writers` 两个 `AtomicUsize` 追踪两端存活状态以判断 EOF/EPIPE 条件 <sup>[2](#mod-ipc-ref-2)</sup>。这种"Arc 共享 + 原子计数"的模式在 `Pipe::new()` 中构造：读写两端各自持有同一 `Arc<PipeShared>` 的克隆，析构时通过 `fetch_sub` 递减计数 <sup>[3](#mod-ipc-ref-3)</sup>。

**Futex** 的抽象层次不同：它用一个 per-process 的 `HashMap<usize, Arc<WaitQueue>>`（即 `futex_table`）将用户态地址映射到内核等待队列 <sup>[4](#mod-ipc-ref-4)</sup>。`WaitQueue` 来自 `axtask` crate，提供 `wait_timeout_until` 和 `notify_one`/`requeue` 接口。关键设计在于：futex 并不在内核中存储"期望值"——每次被定时器唤醒后都重新读取用户态内存以判断条件是否满足，这正是 POSIX futex 的语义 <sup>[5](#mod-ipc-ref-5)</sup>。

**共享内存** 的抽象是一个全局单例 `SHARED_MEMORY_MANAGER`（定义于 `starry_core`），内部维护 `mem_map: Mutex<BTreeMap>` 管理所有已创建的共享内存段 <sup>[6](#mod-ipc-ref-6)</sup>。每个段记录 `key`、物理地址、页数。attach 时通过 `virt_to_phys` 获取物理地址，再调用 `addr_space.map_linear` 将物理页线性映射到进程地址空间的空闲区域 <sup>[7](#mod-ipc-ref-7)</sup>。

### 2. 关键设计取舍：两套 Pipe 实现的演进与 FIFO 语义

代码库中存在两份近乎重复的 Pipe 实现——`api/src/core/file/pipe.rs`（"新版本"）和 `api/src/core/fs/pipe.rs`（"旧版本"）。二者共享完全相同的 `PipeRingBuffer` 和 `PipeShared` 结构，差异集中在 `Pipe` 结构体本身。

旧版本将 `file_flags` 存储为普通 `FileFlags` 值，注释中写道 `// TODO: non-blocking support, flags support, etc.` <sup>[8](#mod-ipc-ref-8)</sup>。其 `set_flags()` 直接 `todo!()`，`read()` 中非阻塞逻辑被注释掉（`/* || non_block */`），`write()` 完全不处理 `O_NONBLOCK` 语义 <sup>[9](#mod-ipc-ref-9)</sup>。新版本将 `file_flags` 改为 `Mutex<FileFlags>`，在 `Pipe::new()` 中为读端追加 `READ` 标志、写端追加 `WRITE` 标志 <sup>[10](#mod-ipc-ref-10)</sup>。`read()` 和 `write()` 均实现了完整的 POSIX 非阻塞语义：读端在缓冲区空且写端未关闭时，若 `O_NONBLOCK` 则返回 `EAGAIN` 而非 yield <sup>[11](#mod-ipc-ref-11)</sup>；写端区分请求字节数是否超过 `PIPE_BUF`（即 64KB），小于等于时整笔写入不可分割——若空间不足直接 `EAGAIN`，大于时尽力写入部分数据 <sup>[12](#mod-ipc-ref-12)</sup>。

系统调用入口 `api/src/imp/fs/pipe.rs` 中 `sys_pipe_impl` 明确导入的是 `crate::core::file::pipe::Pipe`（新版本）<sup>[13](#mod-ipc-ref-13)</sup>。因此旧版本处于"已编译但未连接"的半废弃状态。这暗示开发过程中有过重构——从 `core/fs` 模块搬迁到 `core/file` 模块——但旧文件未被清理，成为技术债。

Futex 一侧最显著的取舍是**轮询替代真正阻塞**。理想情况下，futex 等待者应通过 WaitQueue 的 `wait_until` 被解锁线程精确唤醒；但本实现将 `FUTEX_SIGNAL_POLL_INTERVAL` 设为 10ms，在循环中反复 `wait_timeout_until` <sup>[14](#mod-ipc-ref-14)</sup>。每次超时后检查三个条件：值是否改变、是否有未阻塞信号、总超时是否到达。这种设计简化了与信号系统的集成（不必在 signal delivery 路径中显式唤醒 futex waiter），代价是引入最多 10ms 的调度延迟和周期性 CPU 唤醒开销。

共享内存的全局管理器被硬编码上限为 4096 个段（`mem_map.lock().len() > 4096` 即返回 `ENOSPC`）<sup>[15](#mod-ipc-ref-15)</sup>。这是典型的早期实现占位：没有考虑动态扩容、没有区分用户配额、也没有清理机制——已删除的段是否从 map 中移除取决于 `delete()` 的实现（不在可见代码中）。

### 3. 跨模块协同：fd 管理、信号中断与页表操作

**Pipe 与 fd 表**：`sys_pipe_impl` 调用 `fd_add` 将两个 `Arc<Pipe>` 注册到当前进程的 fd 表，并处理 `O_CLOEXEC` 标志映射 <sup>[13](#mod-ipc-ref-13)</sup>。注意 `write_fd` 分配失败时的回滚：使用 `inspect_err` 闭包调用 `fd_remove(read_fd)`，防止 fd 泄漏 <sup>[16](#mod-ipc-ref-16)</sup>。这是少见的正确错误恢复路径。

**Futex 与信号系统**：`wait_futex_interruptible` 在每次迭代中调用 `has_unblocked_pending_signal()`（来自 `crate::imp::task::signal`），若存在未阻塞的待处理信号则返回 `EINTR` <sup>[17](#mod-ipc-ref-17)</sup>。这实现了 POSIX 要求的"futex 可被信号中断"语义。但实现方式是主动轮询而非被动唤醒——信号投递路径并不会主动通知 futex waiter，依赖 10ms 轮询周期来发现信号。

**共享内存与页表**：`sys_shmat` 调用 `addr_space.find_free_area` 在进程地址空间中寻找空闲虚拟地址区域，随后通过 `virt_to_phys` 将共享内存段的物理地址转换为物理页框号，最后 `map_linear` 建立页表映射 <sup>[7](#mod-ipc-ref-7)</sup>。这里的关键假设是共享内存段分配在**线性映射**区域（即物理地址直接映射到内核虚拟地址），这依赖于 `starry_core` 的内存布局设计。attach 成功后，`process_data.shared_memory` 中插入 `(VirtAddr, Arc<SharedMemory>)` 条目以备 detach 时使用 <sup>[18](#mod-ipc-ref-18)</sup>。

### 4. 边角细节与不足

**1. 两套 Pipe 实现并存** <sup>[19](#mod-ipc-ref-19)</sup><sup>[8](#mod-ipc-ref-8)</sup>：`core/file/pipe.rs` 与 `core/fs/pipe.rs` 功能差异显著（非阻塞支持、flags 线程安全），后者 `set_flags` 为 `todo!()`，且不被任何调用路径引用。存在"代码腐烂"风险——未来维护者可能误改旧版本。

**2. 锁序颠倒导致潜在死锁** <sup>[20](#mod-ipc-ref-20)</sup><sup>[21](#mod-ipc-ref-21)</sup>：`sys_shmat` 先锁 `addr_space` 再锁 `shared_memory`；`sys_shmdt` 先锁 `shared_memory` 再锁 `addr_space`。同一进程内若两个线程分别调用 attach 和 detach，可形成 ABBA 死锁。

**3. 用户指针无效时的静默回退** <sup>[22](#mod-ipc-ref-22)</sup>：`futex_value_changed` 在 `uaddr.get_as_ref()` 失败时返回 `unwrap_or(true)`，即将非法地址视为"值已改变"。这会导致 futex waiter 被虚假唤醒，虽然符合"spurious wakeup 允许"的 POSIX 约定，但掩盖了真正的内存访问错误。

**4. 硬编码常量缺少参数化** <sup>[23](#mod-ipc-ref-23)</sup><sup>[24](#mod-ipc-ref-24)</sup><sup>[25](#mod-ipc-ref-25)</sup>：共享内存段上限 4096、管道容量 65536、futex 轮询间隔 10ms、pipe metadata 的 uid/gid 固定为 1000——这些值分散在代码中，没有集中配置点，跨架构移植时难以调整。

**5. FUTEX_WAIT_BITSET/WAKE_BITSET 仅支持 MATCH_ANY** <sup>[26](#mod-ipc-ref-26)</sup>：bitset 参数被检查，若不为 `FUTEX_BITSET_MATCH_ANY` 则返回 `ENOSYS` 并打印 warning。这在运行 glibc 的 `pthread_cond_t` 时会触发，因为 glibc 使用特定 bitset 来区分条件变量。标记为 TODO 但未实现。

**6. 权限检查缺失** <sup>[27](#mod-ipc-ref-27)</sup>：`sys_shmget` 和 `sys_shmat` 均有 `// TODO: permission check`，`shmctl` 的 IPC_STAT 命令返回 `ENOSYS`（整段结构体填充逻辑被注释掉）。当前任何进程均可访问任意共享内存段。

**7. sys_shmat 重复获取 current_process_data** <sup>[28](#mod-ipc-ref-28)</sup>：函数开头获取一次用于锁 `addr_space`，末尾再次获取用于锁 `shared_memory`。虽然两次调用返回同一对象（per-cpu 指针），但风格不一致且容易在重构中引入 bug。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 该模块以 `Socket` 枚举为核心抽象, 内部用 `Mutex<axnet::TcpSocket>` / `Mutex<axnet::UdpSocket>` 封装 axnet 协议栈, 并通过实现 `FileLike` trait 将 socket 接入 fd 表. 配套的 `SockAddr` 提供从原始 `sockaddr` 到 Rust `SocketAddr` 的安全转换, 适配自 rustix. 整体定位是 IPv4-first 的早期实现: IPv6 只在本地回环/未指定地址上做兼容映射, 多数 socket option 以 `warn!` 吞掉, `listen` 的 backlog 与 `shutdown` 的 how 参数均被忽略.

### 1. 核心抽象: Socket 枚举与 FileLike 集成
网络模块的核心是将 `axnet` 的 TCP/UDP socket 包装为统一的文件描述符. `Socket` 枚举 <sup>[1](#mod-net-ref-1)</sup> 的两个变体分别持有 `Mutex<UdpSocket>` 和 `Mutex<TcpSocket>`, 所有 I/O 操作 (`send`/`recv`/`bind`/`connect` 等) 通过模式匹配分发到底层类型, 并在持锁期间调用 axnet 方法. 这一设计使得 socket 可以经 `fd_add` 注册到全局 fd 表 <sup>[2](#mod-net-ref-2)</sup>, 后续通过 `Socket::from_fd` 做类型安全的逆查找 (利用 `Arc::into_any().downcast`) <sup>[3](#mod-net-ref-3)</sup>.

`FileLike` trait 的实现 <sup>[4](#mod-net-ref-4)</sup> 将 `read`/`write` 分别映射到 `recv`/`send`, 从而使 `poll`/`select` 等复用机制也能透明作用于 socket. 注意 `FileLike::poll` 直接委托给 `Socket::poll` (同名 inherent method), Rust 的方法解析规则保证此处无递归 <sup>[5](#mod-net-ref-5)</sup>.

### 2. 关键设计取舍
**IPv4-only 策略.** 系统在 `sys_socket` 入口处将 `AF_UNSPEC` 和 `AF_INET6` 静默改写为 `AF_INET` <sup>[6](#mod-net-ref-6)</sup>; `from_sockaddr` 对 IPv6 地址仅放行 `::` 和 `::1` (映射为 IPv4 对应地址), 其余一律返回 `EAFNOSUPPORT` <sup>[7](#mod-net-ref-7)</sup>; `into_sockaddr` 遇到 `SocketAddr::V6` 直接 `panic!` <sup>[8](#mod-net-ref-8)</sup>. 这说明 IPv6 支持仅用于兼容 probing 工具, 而非真正的 dual-stack.

**锁粒度与阻塞语义.** 每个 socket 持有一把 `Mutex`, 所有操作 (包括可能阻塞的 `recv`/`accept`) 在持锁期间执行. 这简化了并发安全, 但意味着阻塞 I/O 会长期占有锁, 阻塞同一 socket 上的其他操作. `sys_sendto` 和 `sys_send` 在发送后调用 `axtask::yield_now()` <sup>[9](#mod-net-ref-9)</sup> 主动让出 CPU, 这是一个轻量级的协作调度提示.

**错误映射与吞没.** `sys_connect` 将底层 `EAGAIN` 映射为 `EINPROGRESS` <sup>[10](#mod-net-ref-10)</sup>, 这是非阻塞 connect 的标准语义. 但 `setsockopt` 大量使用 `warn!("unimplemented ...")` 后直接 `Ok(())` <sup>[11](#mod-net-ref-11)</sup>, 导致调用者误以为设置成功而实际未生效 — 这是典型的早期实现权衡.

**SockAddr 安全抽象.** `SockAddr` <sup>[12](#mod-net-ref-12)</sup> 维护三个 invariant: `len` 在 `sa_family_t` 大小与完整 `sockaddr` 之间; 前 `len` 字节已初始化. 它用 `MaybeUninit<sockaddr>` 存储原始字节, 通过 `unsafe` 的 `Send`/`Sync` 实现 <sup>[13](#mod-net-ref-13)</sup> 支持跨线程传递, 注释说明了 bindgen 生成的 union 不包含指针. 到 Rust `SocketAddrV4`/`V6` 的转换通过 `TryFrom` 实现, 在 family 不匹配时返回 `EAFNOSUPPORT` <sup>[14](#mod-net-ref-14)</sup>.

### 3. 跨模块协同
Socket 的生命周期涉及多个 crate 的协作:
- **fd 表 (`arceos_posix_api` + `ourkernel2026_vfs`)**: `fd_add` 将 `Arc<Socket>` 存入全局表, `fd_lookup` 按 fd 号取回. `FileLike` trait 使 socket 可被 `poll`/`select` 统一轮询.
- **axnet (底层网络栈)**: 提供 `TcpSocket`/`UdpSocket` 的实际协议实现, 包括 `bind`/`connect`/`send`/`recv`/`listen`/`accept` 以及 `dns_query`.
- **axsync**: 提供 `Mutex`, 所有 socket 操作串行化.
- **axerrno**: 用 `LinuxError` 枚举做错误码映射, `?` 操作符贯穿整个 syscall 层.
- **axtask**: `yield_now()` 在 send 后触发重调度.
`getaddrinfo` <sup>[15](#mod-net-ref-15)</sup> 还涉及 DNS 解析: 先尝试将 nodename 解析为 `IpAddr`, 失败则调用 `axnet::dns_query`. 返回的链表用 `Vec<aibuf>` 在堆上分配, 通过 `mem::forget` 泄露给调用者, 由配套的 `sys_freeaddrinfo` 用 `Vec::from_raw_parts` 回收 <sup>[16](#mod-net-ref-16)</sup>.

### 4. 边角细节与不足
- **<sup>[11](#mod-net-ref-11)</sup> 静默吞掉的 option**: `setsockopt` 对 `SO_RCVBUF`/`SO_SNDBUF` 等常见选项仅打印 warn 后返回 Ok, 应用层无法检测到功能缺失. 这是最可能引发兼容性问题的点.
- **<sup>[8](#mod-net-ref-8)</sup> IPv6 panic 风险**: `into_sockaddr` 对 `SocketAddr::V6` 直接 `panic!`, 如果上层代码 (如 `getsockname` 返回了一个 IPv6 映射地址) 触发此路径, 内核将崩溃.
- **<sup>[15](#mod-net-ref-15)</sup> getaddrinfo 硬编码**: `ai_socktype` 和 `ai_protocol` 被固定为 TCP (`SOCK_STREAM`/`IPPROTO_TCP`), 完全忽略 `hints` 参数; UDP 应用通过 `getaddrinfo` 查询会得到错误的 socket 类型.
- **<sup>[17](#mod-net-ref-17)</sup> shutdown 忽略 how**: `sys_shutdown` 的 `flag` 参数标注 `currently not used`, 内部 `Socket::shutdown` 先检查 `peer_addr()` (已连接) 再无条件关闭 — 不支持 `SHUT_RD`/`SHUT_WR` 的半关闭语义.
- **<sup>[18](#mod-net-ref-18)</sup> listen backlog 被忽略**: `sys_listen` 的 `backlog` 参数未传递给底层, 注释标记 `currently not used`.
- **<sup>[16](#mod-net-ref-16)</sup> freeaddrinfo 的锁 TODO**: `sys_freeaddrinfo` 中 `Vec::from_raw_parts` 旁有 `// TODO: lock` 注释, 暗示当前实现在并发释放时缺乏同步保护.
- **<sup>[19](#mod-net-ref-19)</sup> 重复 debug! 调用**: `sys_recvfrom` 函数体内有连续两条完全相同的 `debug!` 宏调用, 属于复制粘贴残留.
- **<sup>[20](#mod-net-ref-20)</sup> Socket::status 返回假数据**: `status()` 返回 `Metadata::default()`, `mode` 虽有注释中的 `S_IFSOCK` 但实际被 `default()` 覆盖为 0, `stat` 调用无法获取正确的 socket inode 信息.

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块提供一个最小化的块设备抽象层, 核心为 `BlockDevice` trait (含 read_blocks/write_blocks/num_blocks 三个方法). 其唯一实现者是泛型 `Ext4BlockDevice<Dev>`, 负责将 Rust trait 实现桥接到 lwext4 C 库的 `ext4_blockdev` 结构体, 通过四个 `extern "C"` 回调 (open/bread/bwrite/close) 完成 FFI 双向调用. 与 ArceOS 原版 `driver_block` 相比, 此仓直接面向 ext4 文件系统需求裁剪了驱动接口, 放弃了通用块设备层的多层抽象, 换取与 lwext4 C 库的紧耦合集成.

### 1. 核心抽象: `BlockDevice` trait 与 `Ext4BlockDevice` 适配器

驱动框架的核心是一个极简的 `BlockDevice` trait <sup>[1](#mod-drivers-ref-1)</sup>, 仅定义三个方法: 按块 ID 读写和查询块总数. 这个 trait 不涉及任何异步、DMA 或分散-聚集操作, 是最低限度的同步块设备契约. 任何实现了该 trait 的类型 (如 RAM disk、virtio-blk 包装) 都可以作为 ext4 的后端存储.

泛型 `Ext4BlockDevice<Dev: BlockDevice>` <sup>[2](#mod-drivers-ref-2)</sup> 扮演适配器角色: 它在构造时分配 C 库所需的 `ext4_blockdev`、`ext4_blockdev_iface` 和 `ext4_bcache` 结构体, 将 Rust 函数指针填入 C 的回调表 (`open`/`bread`/`bwrite`/`close`), 并将 `Dev` 的裸指针存入 `p_user` 字段供回调反向获取 <sup>[3](#mod-drivers-ref-3)</sup>. 这种"Rust 持有所有权、C 持有裸指针"的模式在 FFI 中常见, 但安全性完全依赖调用者确保指针生命周期.

### 2. 关键设计取舍

**取舍 1: 放弃 ArceOS 的 `BlockDevice` 抽象层次**. ArceOS 原版在 `driver_block` 中定义了一个更通用的 `BlockDevice` trait (常含 `read_block`/`write_block` 异步方法), 并通过 `AxDisk` 等中间层解耦. 此仓直接面向 lwext4 的 C 接口定义 trait, 省去了中间层的转换开销, 但也使该 trait 与 ext4 强绑定, 无法复用于其他文件系统.

**取舍 2: RAII 资源管理 vs 显式生命周期标注**. 所有 C 结构体 (`ext4_blockdev`, `ext4_blockdev_iface`, `ext4_bcache`) 及其依赖的 Rust 端数据统一装箱放入 `ResourceGuard` <sup>[4](#mod-drivers-ref-4)</sup>. 该结构体标记 `#[allow(dead_code)]` 且字段永不被读取 —— 它的唯一职责是当 `Ext4BlockDevice` 被 drop 时自动释放所有堆分配. 这避免了在 `Drop` 实现中手动 `free` 每个字段, 但也引入了不必要的 `Box` 层 (如 `ext4_bcache` 已在 `Box` 中, 又被 `ResourceGuard` 的 `Box` 包装一层).

**取舍 3: 同步阻塞接口, 无并发支持**. `block_dev_iface` 中的 `lock` 和 `unlock` 字段被显式设为 `None` <sup>[5](#mod-drivers-ref-5)</sup>, 意味着 lwext4 库内部对块设备的访问不加锁. 这在单核/协作式调度场景下可接受, 但若未来引入抢占或多核, 对同一设备的并发读写将导致数据竞争. 此外注释掉的 `// drop_cache(); // sync` <sup>[6](#mod-drivers-ref-6)</sup> 暗示回写和缓存刷新策略尚未完实.

**取舍 4: 硬编码 `EXT4_DEV_BSIZE = 512`**. 物理块大小被编译期固定为 512 字节 <sup>[7](#mod-drivers-ref-7)</sup>, 与大部分传统磁盘的扇区大小兼容, 但无法适配 4K 原生扇区的现代 NVMe 设备. 在 `dev_bread`/`dev_bwrite` 中 `buf_len` 按 `ph_bsize * blk_cnt` 计算, 其中 `ph_bsize` 虽可运行时从 `ext4_blockdev_iface` 读取, 但 `block_buf` 的大小在编译期锁死, 若运行时改变 `ph_bsize` 将导致缓冲区溢出.

**取舍 5: `p_user` 的指针稳定性依赖 `ResourceGuard` 不移动语义**. 在 `new()` 中, `p_user` 被赋值为 `dev.as_mut() as *mut _ as *mut c_void` <sup>[8](#mod-drivers-ref-8)</sup>. 由于 `dev` 是 `Box<Dev>`, 且 `Box` 本身被 `ResourceGuard` 持有而不再移动, 该指针在设备生命周期内保持有效. 但这一契约未在任何注释中说明, 仅是实现层面的隐式约定.

### 3. 跨模块协同

该模块处于 lwext4 C 库与上层 Rust 存储后端之间. 其上游是 C 头文件 `ffi.rs` (定义 `ext4_blockdev`、`ext4_bcache` 等结构体布局及 `ext4_block_init`/`ext4_block_fini` 等 FFI 函数签名), 其下游是实现了 `BlockDevice` trait 的具体设备驱动 (如 virtio-blk、RAM disk). 回调链路为: C 代码调用 `ext4_blockdev.bdif->bread(...)` → 经过 `dev_bread` 的 `unsafe extern "C"` 函数桥 → 通过 `dev_read_fields` 还原 `&mut Dev` → 调用 Rust trait 方法 `dev.read_blocks(...)` <sup>[9](#mod-drivers-ref-9)</sup>. 这种"C 调 Rust 再调具体驱动"的双向 FFI 在 OS 开发中常见, 但此实现的一个精巧处在于 `dev_read_fields` 一次性从 `ext4_blockdev` 裸指针中还原出三个可变引用 <sup>[10](#mod-drivers-ref-10)</sup>, 避免了分步解引用的代码膨胀和潜在的不一致窗口.

### 4. 边角细节与不足

- **[未写 SAFETY 注释]** 文件中出现 7 处 `unsafe` 块/函数, 无一附带 `// SAFETY:` 说明 <sup>[11](#mod-drivers-ref-11)</sup>. 最关键的 `dev_read_fields` 从 `p_user` 裸指针强转为 `&mut Dev` —— 其正确性依赖调用时 C 端确实持有唯一所有权, 但无文档说明此契约. 这是可审计性层面最严重的技术债.
- **[错误信息写错]** `dev_bwrite` 中写失败时 log 的是 `"read_blocks failed: {err:?}"` <sup>[12](#mod-drivers-ref-12)</sup>, 明显是 copy-paste 遗留, 故障排查时会误导.
- **[`dev_close` 为空操作]** `dev_close` 仅打印一条 debug 日志便返回 `EOK` <sup>[13](#mod-drivers-ref-13)</sup>, 未调用任何设备关闭或刷新逻辑. 若具体 `BlockDevice` 实现持有需要显式释放的资源 (如文件描述符), 资源泄漏不可避免.
- **[`mem::zeroed()` 用于非纯数据类型的风险]** `ext4_bcache` 通过 `mem::zeroed()` 零初始化 <sup>[14](#mod-drivers-ref-14)</sup>, 假设全零比特对该 C 结构体是合法初始态. 若 lwext4 未来版本在该结构体中引入需要非零默认值的字段 (如链表指针、引用计数), 此假设将导致 UB.
- **[无溢出检查]** `buf_len = (bdif.ph_bsize * blk_cnt) as usize` <sup>[15](#mod-drivers-ref-15)</sup> 在 `blk_cnt` 极大时可能乘法溢出 (虽然 32 位 `blk_cnt * u32` 在 64 位 usize 下通常安全, 但缺少显式 `checked_mul`).
- **[`#[allow(dead_code)]` 掩盖设计意图]** `ResourceGuard` 上的 `#[allow(dead_code)]` <sup>[4](#mod-drivers-ref-4)</sup> 是一种权宜之计 —— 字段确实"活着"只为 RAII, 但更清晰的方案是用 `ManuallyDrop` 或分离资源持有与借出的生命周期, 而非压制编译器警告.
- **[注释掉的 `drop_cache`/`sync`]** <sup>[6](#mod-drivers-ref-6)</sup> 表明写操作后缺少缓存刷新和同步步骤, 这在异常掉电场景下可能导致数据丢失.
- **[单生产者假设]** `lock`/`unlock` 置 `None` <sup>[5](#mod-drivers-ref-5)</sup> 表明设计者假设 lwext4 对块设备单线程访问. 若 lwext4 内部使用多线程 (如日志提交线程), 这里缺少底层同步机制会直接导致竞态.

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层通过 `#[register_trap_handler(SYSCALL)]` 过程宏将 `handle_syscall` 注册为全局陷阱入口，利用 `syscalls::Sysno` 枚举在一个巨大的 `match` 块中静态派发 180+ 个 Linux 系统调用。与 ArceOS 原版分离注册不同，该实现采用集中式分发，并通过条件编译适配 x86_64/riscv64/aarch64/loongarch64 的 ABI 差异。同一仓库同时暴露出未完成实现的妥协：大量安全敏感调用以 `stub_bypass` 静默成功。

### 1. 系统调用派发的核心抽象与架构

系统调用层以 `#[register_trap_handler(SYSCALL)]` 标记的 `handle_syscall` 函数为唯一入口，使用 `syscalls` crate 提供的 `Sysno` 枚举将号数转换为类型安全的分支条件 <sup>[1](#mod-syscall-ref-1)</sup>。该设计将硬件陷阱处理（`axhal::trap::SYSCALL`）与上层实现（`ourkernel2026_api`）完全解耦：陷阱帧由 `axhal::arch::TrapFrame` 统一抽象，参数通过 `tf.arg0()` 至 `tf.arg5()` 以 `as` 和 `.into()` 转换后传递给各 `sys_*` 函数 <sup>[2](#mod-syscall-ref-2)</sup>。

与 ArceOS 原版基线（`axsyscall` 模块的注册表模式）相比，此仓库放弃了动态注册的灵活性，选择编译期静态分发以获得更低的调用开销。缺点是新增系统调用时必须修改此文件并重新编译，无法做到运行时插拔。

### 2. 跨平台条件编译的取舍

派发表中存在大量 `#[cfg(target_arch)]` 条件分支，最典型的为 `clone` 系统调用：在 `x86_64` / `loongarch64` 上只接收 5 个参数，而在 `riscv64` / `aarch64` 上则额外传入 `tf: &mut TrapFrame` <sup>[3](#mod-syscall-ref-3)</sup>。这反映了架构间 ABI 的差异 — riscv64 和 aarch64 的 `clone` 实现需要在 API 层直接操作陷阱帧设置子进程的栈指针与返回值，而 x86_64 则将此职责放在更底层。

另一值得注意的模式是 x86_64 独占的传统系统调用（`fork`、`pipe`、`rename`、`stat` 等）。非 x86_64 平台只能使用对应的 `*at` 变体（如 `renameat`）或完全缺失 <sup>[4](#mod-syscall-ref-4)</sup>。这并非架构限制，而是开发者选择将有限的适配精力集中在 x86_64 兼容性上。

### 3. 未实现系统调用的分级伪装策略

代码中针对未实现系统调用设置了三个信任层级：
- **stub_unimplemented**: 返回 `-ENOSYS`，用于完全没有实现痕迹的调用 <sup>[5](#mod-syscall-ref-5)</sup>。
- **硬编码错误码**: 对部分调用返回特定错误（如 `msgrcv` 返回 `ENOMSG`，`setpriority` 返回 `ESRCH`）<sup>[6](#mod-syscall-ref-6)</sup>。
- **stub_bypass**: 返回 `Ok(0)`，让调用者误以为操作成功 <sup>[7](#mod-syscall-ref-7)</sup>。

`stub_bypass` 的滥用是本模块最大的设计缺陷。诸如 `chroot`、`pivot_root`、`unshare`、`setuid`、`setgid` 等安全敏感系统调用均被静默成功 <sup>[8](#mod-syscall-ref-8)</sup>。容器运行时或特权分离程序会因虚假的成功反馈而错误地认为已获得隔离环境，构成隐蔽的安全漏洞。此外，`sync` / `fsync` 的“实现”仅调用 `axtask::yield_now()` 并返回 0，完全未同步数据 <sup>[9](#mod-syscall-ref-9)</sup>。

### 4. 与信号模块的深度耦合及时间统计

系统调用层通过 `set_trap_frame(tf)` 将当前陷阱帧指针注入任务上下文，供信号处理（`sys_rt_sigreturn`）恢复用户态寄存器 <sup>[10](#mod-syscall-ref-10)</sup>。系统调用返回前，代码检查 `has_unblocked_pending_signal()` 与 `EINTR` 的交互：若系统调用被信号中断且存在未屏蔽的待处理信号，则在 riscv64 上将指令指针减去 4 字节以触发系统调用重启 <sup>[11](#mod-syscall-ref-11)</sup>。

时间统计通过 `time_stat_from_user_to_kernel()` 与 `time_stat_from_kernel_to_user()` 包裹整个派发过程，提供粗粒度的内核态耗时记录 <sup>[12](#mod-syscall-ref-12)</sup>。这一位置选择牺牲了精细度（无法区分不同系统调用的开销），但避免了在每个 `sys_*` 实现中插入计时代码的复杂性。

### 5. 边角细节与不足

1. **重复的 `access` 派发分支**：第 148 行与第 416 行各有一个 `#[cfg(target_arch = "x86_64")]` 限定的 `Sysno::access` 分支，分别调用 `sys_access` 与 `stub_bypass`（后者带有 `// TODO: implement access` 注释）<sup>[13](#mod-syscall-ref-13)</sup>。在 x86_64 编译时这会导致不可达模式错误，是两个开发版本合并时的冲突遗留。

2. **无效系统调用号泄漏**：第 31–34 行对 `sysno.is_none()` 仅打印警告，随后以 `unwrap_or(Sysno::memfd_secret)` 将非法号数映射到 `sys_openstub()`，而非返回 `-ENOSYS` <sup>[14](#mod-syscall-ref-14)</sup>。被注释掉的 `return` 语句表明这是早期实现阶段的暂时妥协，但至今未被修正。

3. **死代码 `stub_kill`**：第 554–558 行定义的 `stub_kill` 函数从未在任何匹配分支中被调用，属于残留代码 <sup>[15](#mod-syscall-ref-15)</sup>。

4. **硬编码指令长度假设**：riscv64 的系统调用重启逻辑以 `saturating_sub(4)` 回退指令指针，假设 `ECALL` 固定为 4 字节。虽在标准 RISC-V 中成立，但缺乏对压缩指令扩展的注释说明，存在维护风险 <sup>[11](#mod-syscall-ref-11)</sup>。

5. **锁安全与重入隐忧**：`handle_syscall` 持有 `TrapFrame` 引用的同时可能触发任务调度（如 `sched_yield`），而信号系统调用（`rt_sigsuspend`）直接将 `tf` 传入 API 层进行上下文修改，若 API 层内部获取任务锁，可能引发锁顺序问题。
