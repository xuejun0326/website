## 📖 一、总览

该仓库是 ArceOS-Starry 家族中一个实验性多进程内核，旨在通过模块化设计实现 Linux 兼容的系统调用接口。其定位介于教学演示与测试验证之间：拥有152个系统调用覆盖进程、文件、网络、IPC和信号，但大量实现为 stub 或简化版本，优先确保 LTP 等测试套件的通过。整体架构沿用 ArceOS 的组件化风格，但增加了许多新模块，如独立的内存管理（混合分配器）、多文件系统支持（ext4/ramfs/fat32）、System V IPC。该仓库最值得注意的特点包括：(1) 内存管理引入 hybrid 分配器，在 buddy system 基础上叠加 large allocator 的大内存池，以64KB阈值分流大块，减少外部碎片（`[mm.ref:3]`）；(2) 驱动框架采用两级中断注册流程，并支持 LoongArch64 下通过 PCI 总线枚举避免依赖 FDT（`[drv.ref:5]`）；(3) 信号机制从位图定义到 syscall 接口完整实现，但分发执行流程尚未完成，反映出系统调用覆盖与内部实现之间的落差（`[sig.ref:1]`）。这些特点表明该仓库在追求广泛兼容性的同时，策略性地选择了实现深度，形成一个功能覆盖广但细节多未打磨的实验性内核。

## 🎯 二、综合评价

### 整体定位与架构分析

该仓库的设计目标是在 ArceOS 的基础上扩展多进程和 Linux 兼容性，形成一个能够运行简化 Linux 用户程序的运行环境。它不是一个完整的操作系统内核，而是一组库的集合，通过 executor 协作调度。其架构高度模块化，但模块间存在紧密依赖，例如 task 模块直接使用 runtime crate 的内存管理，syscall 层依赖 vfscore 文件系统。这种设计使得添加新系统调用较为直接，但也导致了一些问题：内核锁的顺序未统一（`[task.ref:11]` 指出 thread_exit 中锁顺序不一致可能导致死锁），以及错误处理模式多样。对比基线 ArceOS，该仓库在内存管理上从简单分配器升级为支持多进程虚拟内存；在文件系统上从单一 ramfs 扩展为支持三种类型；在驱动上增加了 LoongArch64 支持和中断注册层次。但整体而言，代码质量痕迹显示这是一个快速迭代的原型，而非精炼的产品。

### 真正的创新或独特设计

尽管该仓库遵循 Linux 接口，但有些设计决策在类似 unikernel 中并不常见：

- **混合内存分配器**：`HybridAllocator` 将 buddy system 和专用大内存池结合，分配时若大小超过64KB则从大池分配，否则从 buddy 分配（`[mm.ref:3]`）。这旨在降低大块分配对 buddy 的碎片化影响，特别针对网络测试的大内存场景。这是一个实践驱动的优化，并非理论创新，但体现了实际问题考量。

- **COW fork 的紧凑实现**：在 `cow_fork` 中，通过将非共享区域页表设置为只读（URX），共享区域保持读写（URWX），缺页时自动复制（`[task.ref:5]`）。这种写时复制机制虽然标准，但在 unikernel 环境中需要对用户地址空间有精细控制，该仓库通过 `MemSet` 和 `MemArea` 实现了这一功能，完善了多进程支持。

- **驱动两级中断注册**：`register_device_irqs` 和 `regist_devices_irq` 将设备中断绑定和中断控制器初始化分离，使得不同架构的中断拓扑可以灵活配置（`[drv.ref:5]`）。此外，LoongArch64 的 PCI VirtIO 发现不依赖 FDT，而是通过 PCI 总线枚举，这在 RISC-V 或 x86 上并不常见，体现了对较新架构的适配。

- **ramfs 使用物理页帧**：ramfs 内容存储在 `FrameTracker` 中，直接映射物理页（`[fs.ref:2]`）。这避免了堆分配带来的复杂性和 fragmentation，但也意味着每个文件占用整数个4KB页，小文件浪费空间。这是一个独特的空间换简化设计的例子。

### 取舍判断：广度优先还是深度优先

该仓库明显选择了广度优先。系统调用覆盖广（152个），但许多是空实现或总是成功：例如权限相关调用直接返回0，`sched_getaffinity` 返回0但不设置（`[task.ref:21]`）。这种取舍使内核能通过存在性检查，但掩盖了逻辑缺陷。文件系统模块虽然提供了三种实现，但每个都存在明显问题：ext4 重试可能掩盖永久错误（`[fs.ref:1]`），ramfs 在内存不足时部分写入（`[fs.ref:3]`），fat32 逐扇区读写性能极差（`[fs.ref:4]`）。网络模块引入端口重用时替换 fd 的特殊语义，但 `setsockopt` 为空实现。总体而言，开发者优先保证接口存在和基本流程，而将语义正确性和性能优化后置。这种策略适合快速验证 Linux 兼容性，但离生产使用尚有距离。

### 完成度与不足

核心框架基本完成，但细节实现大量不完整。内存管理中的 `mprotect`/`msync` 为空实现；任务调度仍为协作式，`sys_exit_group` 仅退出当前线程（`[task.ref:22]`）；信号机制虽定义了完整系统调用，但分发和执行流程未实现（`[sig.ref:5]` 表明 `SignalOps` trait 无实现）；驱动中 `virtio_input` 为 todo（`[drv.ref:12]`），ramdisk 仅1字节（`[drv.ref:13]`）。代码质量方面，多处 `unsafe` 缺少 SAFETY 注释（`[syscall.ref:2]`），错误处理不一致（如 `getdents64` 失败返回0而非错误码，`[syscall.ref:4]`），锁使用风格未统一。整体完成度为早期实验状态，适合教学和探索，不适合直接用于稳定系统。

**评分**：完成度2（框架全面但细节缺失），创新性2（混合分配器等有实用价值但非突破），代码质量1（unsafe 多缺安全说明，错误处理不一致）。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块采用三层架构管理内存：物理层通过 `FrameAllocator`（位图）+ `FrameTracker`（RAII）管理页帧；堆层通过 `HybridAllocator`（buddy system + 大内存池）提供全局分配，以 64KB 阈值分流大块缓解碎片；虚拟层通过 `MemSet`/`MemArea` 记录进程地址空间，配合 `PageTable` 完成映射。解决的问题是为多进程提供类 Linux 的 brk/mmap/munmap/mprotect/msync 接口。与 ArceOS 基线相比，独特修改包括：实现了完整的用户态虚拟内存管理（文件映射、共享映射、匿名映射）；引入 hybrid 分配器针对网络测试大内存场景优化；但 mprotect/msync 仍为空实现，/proc/meminfo 为硬编码存根。

### 1. 核心抽象：三层内存管理架构

本模块的物理页帧管理以 **位图 + RAII 包装** 为核心。`FrameRegionMap` 用 `Vec<usize>` 的 bit array 跟踪一段连续物理内存的分配状态，每个 bit 代表一个 4KB 页帧 <sup>[1](#mod-mm-ref-1)</sup>。`FrameAllocator` 聚合多个 `FrameRegionMap`（对应不同物理内存区间），提供 `alloc` / `alloc_much`（连续多页）/ `dealloc` 接口。`alloc_much` 采用滑动窗口算法在 O(n) 时间内查找足够大的连续空闲块，失败时输出碎片化诊断信息 <sup>[1](#mod-mm-ref-1)</sup>。分配结果通过 `FrameTracker(PhysAddr)` 包装，其 `Drop` 实现自动 `clear()` 清空页面并归还给分配器 <sup>[2](#mod-mm-ref-2)</sup>。这一 RAII 模式确保页帧生命周期与 Rust 所有权绑定，避免人为忘记释放。

堆内存使用 **混合分配器** `HybridAllocator` 实现 `#[global_allocator]`。内部组合了 `buddy_system_allocator::LockedHeap<32>`（·最大 order 32，支持最大 4GB 块）和自实现的 `LargeAllocator`（管理 128MB 大内存池）。分配决策逻辑：若 Layout 的 size > 64KB（`LARGE_ALLOC_THRESHOLD`），优先尝试大内存池，失败则回退到 buddy <sup>[3](#mod-mm-ref-3)</sup>。大内存池采用 **First-fit + 懒惰压缩** 策略：`LargeAllocator` 维护 `Vec<(offset, size, is_free)>` 记录，分配时先遍历空闲块，未找到则尝试 `compact_memory()`（仅移除 `is_free=true` 的记录，不移动数据），最后在末尾追加。

虚拟内存管理以 `MemSet`（即 `Vec<MemArea>`）为核心数据结构，每个 `MemArea` 记录一段连续的虚拟地址空间属性：类型（CodeSection/Stack/Mmap/Shared/ShareFile）、映射跟踪列表 `mtrackers`、关联文件句柄等 <sup>[4](#mod-mm-ref-4)</sup>。`MemArea::sub()` 实现区域分割/裁剪，配合 `PageTable::unmap_page()` 解除页映射，返回分裂出的新 `MemArea` <sup>[5](#mod-mm-ref-5)</sup>。系统调用层（`syscall/mm.rs`）通过 `self.task.pcb.lock().memset` 和 `self.task.page_table` 协同完成地址空间操作。

### 2. 关键设计取舍

**（一）混合分配器的 64KB 阈值。** buddy system 在大块分配时容易产生内部碎片，且 buddy 的 order 有限。本项目设置 64KB 阈值，将大块请求导向专用大内存池，减少 buddy 的碎片化。代价是引入了第二套分配器，增加了 alloc/dealloc 路径的分支开销（每次分配/释放都要检查指针是否落入大内存池范围）。

**（二）物理页帧位图 vs. buddy 分配器。** 原版 ArceOS 使用 buddy 管理物理页帧，本项目改用位图。位图的优势是分配/释放为 O(bit-scan) 常数时间，且连续多页分配更直观。劣势是缺乏 buddy 的合并机制，碎片化程度更高——`alloc_much` 失败时会输出 "largest consecutive block" 诊断信息，侧面说明碎片是已知问题 <sup>[1](#mod-mm-ref-1)</sup>。

**（三）大内存池的假压缩。** `compact_memory()` 仅移除已经标记为 `is_free` 的记录，不迁移数据。这意味着实际物理碎片无法被消除，只是缩小了元数据体积。当所有空闲块都被已分配块隔断时，该函数无效果。注释中“内存压缩”名称有误导性 <sup>[6](#mod-mm-ref-6)</sup>。

**（四）sys_mprotect / sys_msync 空实现。** `sys_mprotect` 直接 `Ok(0)` 不修改任何页表权限，`sys_msync` 同样 `Ok(0)`。这意味着依赖 mprotect 做内存保护（如 JIT 代码页的 RW→RX 切换）的程序在此内核上会静默成功但实际没有保护，构成安全缺口 <sup>[7](#mod-mm-ref-7)</sup>。

**（五）/proc/meminfo 硬编码。** `MemInfo::readat` 返回固定的 8GB MemTotal / 4GB MemFree 等常量，不查询 `FRAME_ALLOCATOR` 的实际空闲页数。这使 LTP 测试能够解析到符合格式的内容，但数值完全不反映真实内存状态 <sup>[8](#mod-mm-ref-8)</sup>。

### 3. 跨模块协同

**系统调用 ↔ MemSet + PageTable。** 以 `sys_mmap` 为例：先解析 flags/prot，检查 MAP_FIXED 或地址重叠（调用 `memset.overlapping()`），然后根据映射类型走不同路径——匿名私有映射仅 `memset.push(MemArea)`；共享匿名映射调用 `self.task.frame_alloc()` 分配物理帧并逐页 `map()`；文件共享映射调用 `self.task.map_frames()` 并传入 `file.get_bare_file()` 做 I/O <sup>[9](#mod-mm-ref-9)</sup>。`MemSet::sub_area` 在 munmap 时被调用，它遍历所有 `MemArea`，对重叠区域调用 `MemArea::sub()`，该函数内部通过 `PageTable::unmap_page()` 解除页映射，并通过 `write_page()` 将脏数据写回文件（ShareFile 类型）<sup>[5](#mod-mm-ref-5)</sup>。这构成了“系统调用 → 虚拟地址空间元数据更新 → 页表操作 → 文件回写”的完整链路。

**FrameTracker Drop 链。** 当 `MemArea` 被移除或进程退出时，`mtrackers` 中的 `Arc<FrameTracker>` 引用计数归零触发 `Drop`：先 `clear()` 清空页面（写零），再归还给 `FRAME_ALLOCATOR`。对于 ShareFile 类型，`MemArea::drop()` 会在释放前将数据写回文件 <sup>[4](#mod-mm-ref-4)</sup>。页帧的生命周期由此完全由 Rust 所有权驱动，无需手动 free。

**堆分配器与物理内存的间接联系。** 堆内存来自 `.bss.heap` / `.bss.large_pool` 静态数组，由链接器在加载时保留物理内存。`alloc_error_handler` 在 OOM 时调用 `get_free_pages()` 显示可用物理页数 <sup>[2](#mod-mm-ref-2)</sup>，为用户态调试提供关联信息。

### 4. 边角细节与不足

**<sup>[10](#mod-mm-ref-10)</sup> emergency_reset 策略3 可能损坏分配状态。** 当大内存池 allocation 数量超 500 且前三策略无效时，`self.allocations.truncate(keep_count)` 直接丢弃后半段记录。这些记录可能包含仍在使用的分配（`is_free=false`），truncate 后对应的内存块失去追踪，后续 dealloc 找不到记录返回 warn（"not found"），导致内存泄漏；若该物理地址被重新分配则引发 use-after-free。该策略的注释“可能已经释放但没有正确标记”暴露了设计者对状态不一致的默认容忍。

**<sup>[11](#mod-mm-ref-11)</sup> sys_get_robust_list 中 unsafe 写用户空间。** 代码通过 `paddr.get_mut_ptr::<usize>().write(0)` 直接向翻译后的物理地址写入，绕过页表权限检查。若该页被标记为只读，写操作会静默破坏页表属性不变量。且 `translate` 返回的 `(paddr, flags)` 未检查 flags 中的 W 位，写入只读页会触发缺页异常。

**<sup>[12](#mod-mm-ref-12)</sup> 条件编译未覆盖的架构。** `#[cfg(target_arch = "loongarch64")]` 块检查用户地址是否超过 `0x7F_FFFF_FFFF`。对于未列出的架构（如 x86_64，其用户空间上限为 `0x7FF_FFFF_FFFF_FFFF`），该检查不会被编译，使用 `MAP_FIXED` 映射到内核地址不会返回错误，可能导致内核崩溃。

**<sup>[3](#mod-mm-ref-3)</sup> LargeAllocator::dealloc 的 fallback 逻辑模糊。** 当精确匹配失败时，代码遍历寻找“包含该地址”的分配并置 `is_free=true`。若同一大内存块被多次部分释放，会导致元数据混乱：一个 `MemArea` 被部分 `munmap` 但 LargeAllocator 的元数据层面无法感知部分释放，`find_free_block` 会使用错误的 offset 计算区间。

**<sup>[5](#mod-mm-ref-5)</sup> MemArea::sub 中 extract_if 的使用。** `mtrackers.extract_if(|x| new_area_range.contains(&x.vaddr.raw()))` 假设 `extract_if` 返回的元素可以直接构造新 `MemArea`，但 `mtrackers` 中 MapTrack 的顺序不一定与虚拟地址顺序一致，分裂后的新 `MemArea` 的 `mtrackers` 列表可能是乱序的，影响后续 `contains` 等操作的正确性。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块围绕 `UserTask` 结构体构建进程与线程抽象，使用 `ProcessControlBlock`（PCB）管理进程级资源，`ThreadControlBlock`（TCB）管理线程级上下文。通过 `executor` crate 的协程式调度（`yield_now`/`select`）实现协作式多任务。与标准 ArceOS 基线相比，本仓引入了完整的 `CloneFlags` 兼容的 `fork`/`thread_clone`、写时复制（COW）分叉、futex 同步原语以及信号传递框架，但调度器仍为协作式，未实现时间片轮转抢占。

### 1. 核心抽象：进程/线程统一模型

本模块使用单一类型 `UserTask` 表示进程和线程，通过 `task_id` 和 `process_id` 的关系区分：当 `task_id == process_id` 时视为主线程（进程），否则为附属线程 `<sup>[1](#mod-task-ref-1)</sup>`。进程级状态封装在 `ProcessControlBlock` 中，包括地址空间（`MemSet`）、文件描述符表、信号动作表、futex 表、子进程列表、计时器、资源限制等 `<sup>[2](#mod-task-ref-2)</sup>`。线程级状态封装在 `ThreadControlBlock` 中，包括陷阱帧（`TrapFrame`）、信号掩码、信号列表、`clear_child_tid` 机制、线程退出码等 `<sup>[3](#mod-task-ref-3)</sup>`。这种分离使多个线程共享同一个 `ProcessControlBlock`（通过 `Arc`），而各自的 `ThreadControlBlock` 独立存储，与 Linux 内核的 `task_struct` 设计思路一致。

`UserTask::new()` 初始化时自动分配新页表（`PageTableWrapper::alloc()`），每个任务拥有独立的地址空间；进程与线程通过 `cow_fork()` 和 `thread_clone()` 两种方式创建 `<sup>[4](#mod-task-ref-4)</sup>`。`cow_fork()` 克隆页表并通过只读映射实现写时复制，`thread_clone()` 则直接共享页表（`self.page_table.clone()`）和 PCB，仅分配新的 TCB，这与 Linux `clone` 系统调用中 `CLONE_VM` 标志的语义一致。

### 2. 关键设计取舍

**2.1 写时复制（COW）fork**
`cow_fork()` 实现时，对非 `Shared`/`ShareFile` 类型的内存区域映射为只读（`URX`），父进程和子进程共享物理页，写访问触发缺页异常后复制 `<sup>[5](#mod-task-ref-5)</sup>`。但代码中有一段 `#[cfg(never)]` 禁用的逻辑，试图填补内存区域间的空隙，开发者在注释中明确指出“这个逻辑创建了没有实际物理页面映射的内存区域，当程序访问这些区域时会触发 SIGSEGV”，因此将其禁用 `<sup>[6](#mod-task-ref-6)</sup>`。这是对早期不成熟实现的修正，反映了项目的演进历史。

**2.2 协作式调度与阻塞原语**
本模块的调度依赖 `executor` crate 提供的 `yield_now()` 和 `select()` 协程原语，无硬件定时器驱动的抢占 `<sup>[7](#mod-task-ref-7)</sup>`。`sys_sched_yield` 直接调用 `yield_now().await`。同步原语 `futex` 的 `FUTEX_WAIT` 通过 `WaitFutex` 结构体在 `select` 中等待被唤醒或超时 `<sup>[8](#mod-task-ref-8)</sup>`。这种设计实现简单，但不适用于 CPU 密集型负载：若用户线程不主动 yield，其他线程将无法运行。这也是协作式调度在通用操作系统中的常见缺陷。

**2.3 信号传递与上下文恢复**
信号通过 `SignalList` 在每个线程的 TCB 中排队，`sys_kill` 向目标任务添加信号后立即 `yield_now().await`，让接收线程在下次进入内核时处理信号 `<sup>[9](#mod-task-ref-9)</sup>`。`sys_sigreturn` 从用户栈读取保存的 `SignalUserContext` 恢复寄存器、PC 和信号掩码 `<sup>[10](#mod-task-ref-10)</sup>`。信号处理函数运行在用户栈上，内核通过修改陷阱帧的 PC 跳转到信号处理函数。这种机制与 Linux 的 sigreturn 设计类似。

**2.4 线程退出与进程清理**
`thread_exit()` 处理线程退出：如果该线程是主线程且没有其他活跃线程，则会清理整个进程（清除地址空间、文件表、子进程列表，并向父进程发送 `SIGCHLD`）`<sup>[11](#mod-task-ref-11)</sup>`。否则仅从线程列表中移除自身。这个逻辑比简单版本更安全，避免了多线程程序中主线程退出导致进程过早终止的问题。但存在锁顺序风险：`thread_exit` 在持有 `tcb.write()` 锁时调用 `futex_wake`，而 `futex_wake` 内部会锁住 `futex_table` 互斥量；若其他线程以相反顺序加锁，可能死锁。

### 3. 跨文件协同

本模块与 `executor` crate、`signal` crate、`fs` crate 紧密协作。`executor` 提供 `AsyncTask` trait、`TASK_MAP` 全局映射、`thread::spawn`、`yield_now` 等核心调度基础设施 `<sup>[12](#mod-task-ref-12)</sup>`。`initproc.rs` 中的 `command` 函数创建 `UserTask` 后调用 `thread::spawn(task.clone(), user_entry())` 启动用户态执行 `<sup>[13](#mod-task-ref-13)</sup>`。系统调用处理时，`syscall/task.rs` 通过 `tid2task` 从 `TASK_MAP` 查找任务 `<sup>[14](#mod-task-ref-14)</sup>`。`UserTaskContainer` 结构体（保存 `task` 引用和 `tid`）作为系统调用处理的上下文。

futex 实现需要与 `fs::TimeSpec`、`user::useref::UserRef` 协同。`WaitFutex` 是 `executor` 中定义的 `Future`，它不断检查 `futex_table` 中是否有该线程的等待项。`select` 用于实现带超时的 `FUTEX_WAIT` `<sup>[15](#mod-task-ref-15)</sup>`。信号的发送（`sys_kill`）通过 `tid2task` 找到目标任务，然后向其 TCB 的 `signal` 列表添加信号；信号处理在下次内核入口（如定时器或系统调用返回）时由 trap handler 检查，但该部分代码未包含在提供的片段中。

### 4. 边角细节与不足

以下列出观察到的风险点与工程缺陷，每点后附引用。

**(a) 条件编译的架构覆盖不全**：`initproc.rs` 仅对 `loongarch64` 和非 loongarch64 进行分支，未显式处理 x86_64、riscv64 等架构。在非 loongarch64 分支中硬编码了 RISC-V 的动态链接器路径 `/lib/ld-musl-riscv64.so.1`，若在 x86_64 上运行会执行错误的 `ld.so` `<sup>[16](#mod-task-ref-16)</sup>`。

**(b) unsafe 块缺乏安全契约**：`force_cx_ref()` 直接返回 `&'static mut TrapFrame`，未提供 `// SAFETY:` 注释说明调用者必须确保线程未并发访问 `<sup>[17](#mod-task-ref-17)</sup>`。同样，`thread_exit` 中的 `addr.get_mut_ptr::<u32>().write(0)` 虽已有地址翻译检查，但未处理翻译后的物理地址可能被其他线程同时写入的情况。

**(c) 锁粒度与潜在死锁**：在 `thread_exit` 中，先获取 `tcb.write()` 锁，然后调用 `futex_wake`（内部锁住 `futex_table`）；而其他代码路径（如 `sys_futex` 的 `FUTEX_WAIT`）先获取 `futex_table` 锁，然后修改 TCB。这种不一致的锁顺序可能导致 ABBA 死锁 `<sup>[11](#mod-task-ref-11)</sup>`。此外，PCB 锁（`Mutex`）在信号处理等路径中可能被持有较长时间，阻塞其他线程。

**(d) 半成品标记与死代码**：`initproc.rs` 文件开头有 `#![allow(dead_code)]` 和 `#![allow(unused_imports)]`，表明大量代码未实际使用 `<sup>[18](#mod-task-ref-18)</sup>`。`command_iperf` 函数带有 `#[allow(dead_code)]` 推断，但其内部的 `kill_all_tasks` 和内存屏障循环（`for _ in 0..100000 { core::hint::spin_loop(); }`）是极其低效的“同步”方式，无实际意义 `<sup>[19](#mod-task-ref-19)</sup>`。

**(e) 硬编码路径与平台假设**：`set_libc_path`、`set_dyn_path` 的默认值硬编码在 `initproc` 中，且 `#[cfg(not(target_arch = "loongarch64"))]` 分支假定使用 musl RISC-V 动态链接器，无法灵活适配不同 libc 实现 `<sup>[16](#mod-task-ref-16)</sup>`。

**(f) 错误处理不完整**：`sys_execve` 中使用 `unwrap()` 处理用户指针解析，用户可传入无效指针导致内核 panic `<sup>[20](#mod-task-ref-20)</sup>`。`sys_sched_getaffinity` 直接返回 `Ok(0)`，未实际设置 CPU 亲和性 `<sup>[21](#mod-task-ref-21)</sup>`。`sys_exit_group` 的注释中保留了旧代码但被注释掉，实际实现只退出当前线程而非整个线程组 `<sup>[22](#mod-task-ref-22)</sup>`。

总体而言，本模块提供了一个功能较为完整的进程/线程框架，实现了 `clone`/`fork`/`execve`/`wait4`/`futex` 等关键 POSIX 原语，并支持信号传递。主要不足在于调度器仍为协作式、锁顺序风险、以及大量工程半成品标记。若计划用于多核场景或实时负载，需引入抢占式调度并重构锁设计。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 文件系统模块基于 vfscore crate 定义的 INodeInterface 抽象，提供了 ext4、ramfs、fat32 三种具体实现。核心抽象是 FileSystem 和 INodeInterface trait，mount 点通过 dentry 模块管理。与同类 ArceOS 相比，该仓库自行实现了 ext4 和 ramfs 的细节，尤其是 ramfs 直接使用帧分配器而非 Vec<u8>，以及对符号链接的递归解析。模块在错误处理、锁粒度、unsafe 安全契约方面存在多个早期实现的不足。

### 1. 核心抽象与外部依赖
文件系统的统一抽象是 vfscore crate 中的 `FileSystem` 和 `INodeInterface` trait，各文件系统都通过实现这两个 trait 来提供操作。`FileSystem::root_dir` 返回 `Arc<dyn INodeInterface>` 作为根节点，后续的路径解析、文件操作全部通过 INodeInterface 的多态分派完成。**这保证了 VFS 层可以无缝混用不同文件系统，是典型的组合模式。**

外部依赖上，ext4 通过 `lwext4_rust` 绑定 C 库 `/lwext4` 实现，ramfs 完全自研并直接调用 `runtime::frame::frame_alloc` 分配物理页，fat32 则依赖 `fatfs` Rust 库。三者对块设备的访问都通过 `devices::get_blk_device()` 获取，该函数返回一个 `BlockDevice` trait 对象，底层可能是 virtio-blk 或 ramdisk。

### 2. 关键设计取舍
**a) ext4_shim 的重试逻辑与错误映射**
`readat` 中实现了最多 3 次重试的循环，每次失败后重新打开文件并 seek<sup>[1](#mod-fs-ref-1)</sup>。这种设计假设临时错误（如并发写入导致的元数据锁竞争）可被重试修复，但这同时掩盖了永久的 IO 错误。`map_ext4_err` 直接使用 `Errno::new(err)` 转换，但 lwext4 返回的负值错误码可能与 POSIX Errno 不完全兼容，评审时应关注此映射的正确性。

**b) ramfs 的页分配与内存压力处理**
ramfs 的文件内容不存储在 Vec<u8> 中，而是管理一组 `FrameTracker` 页<sup>[2](#mod-fs-ref-2)</sup>，每个页对应一个物理帧。`writeat` 中动态分配帧时，若分配失败会尝试部分写入，并记录详细日志<sup>[3](#mod-fs-ref-3)</sup>。这比直接使用 Vec 更节省内存（仅分配实际使用的页），但增加了碎片风险。`truncate` 中的 `clear()` 和 `clear_len` 操作假设帧内容可安全清零，但并未检查帧是否已被其他模块映射——这是潜在的安全漏洞。

**c) fatfs_shim 的 DiskCursor 逐扇区读写**
DiskCursor 的 `read` 和 `write` 只处理一个扇区（512 字节）的读写，而将跨扇区连续性交给 fatfs 库的 `read_exact` / `write_all` 处理<sup>[4](#mod-fs-ref-4)</sup>。这简化了实现，但每次调用都重复读取扇区（未缓冲），性能较差。同时 `seek` 中 `SeekFrom::End` 分支直接 `unreachable!()`，说明 fatfs 库本身不会使用该模式。

### 3. 跨模块协同：路径解析与符号链接
`file.rs` 的 `File::open` 函数是 VFS 层与具体文件系统的桥梁。它调用 `dentry::get_mounted` 获取挂载点对应的根目录，然后逐级 `lookup` 路径组件。**关键协同点在于符号链接的递归解析**：<sup>[5](#mod-fs-ref-5)</sup> 当发现当前节点类型为 `Link` 时，调用 `resolve_link()` 获取目标路径，然后递归调用 `Self::open` 重新开始路径解析。这种设计允许符号链接跨文件系统（因为递归重新走 `get_mounted`），但**缺少循环链接检测**，可能导致无限递归栈溢出。

此外，`O_CREAT` 和 `O_DIRECTORY` 标志在路径遍历完成后才被检查，这意味着创建文件时会先 lookup 最后一层父目录，再调用 `create`，最后再次 `lookup` 新节点。这种“lookup 两次”模式在并发创建时可能出现竞态（第一次 lookup 失败后创建，但第二次 lookup 时可能被其他任务删除）。

### 4. 边角细节与不足
- **<sup>[6](#mod-fs-ref-6)</sup>** ext4_shim 的 `create` 中硬编码文件权限为 777（`0o100777`），且通过 `ext4_mode_set` 直接设置底层 inode 模式位。这绕过了 umask 机制，且不支持 Linux 的默认权限掩码，是明显的简化实现。
- **<sup>[7](#mod-fs-ref-7)</sup>** ext4_shim 的 `readat` 重试循环中，若所有尝试都失败，最后一行 `Err(map_ext4_err(5))` 直接返回 EIO（错误码 5），但 5 在 POSIX 中是 EIO，在 lwext4 的错误定义中是否相同？该行没有注释说明魔术数字的来源。
- **<sup>[8](#mod-fs-ref-8)</sup>** ramfs 的 `RamDir::rmdir` 使用 `extract_if` 删除子目录，但**不检查目录是否非空**，这与 POSIX 要求返回 ENOTEMPTY 不符。虽然代码中有 `// TODO: identify whether the dir is empty`，但未实现。
- **<sup>[9](#mod-fs-ref-9)</sup>** 多个结构体（如 `FatFile`、`FatDir`、`Ext4FileSystem`）使用了 `unsafe impl Send/Sync`，但相关 `SAFETY` 注释缺失。例如 `FatFile` 内部的 `Mutex<FatFileInner>` 实际上提供了线程安全，但锁保护的数据存在 `'static` 生命周期引用（`File<'static, ...>`），需要仔细确认该生命周期是否真的安全。
- **<sup>[10](#mod-fs-ref-10)</sup>** `File::getdents` 中加入了 `max_entries = 50` 的保护限制，防止卡死（日志中提到“防止死锁”），但根本原因是 `read_dir` 后的遍历缺少退出条件——如果 `read_dir` 返回大量条目但用户缓冲区不足，常规实现应返回 `EINVAL` 或 `ENOSPC`，而非硬编码上限。

    }

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制的实现以 `crates/signal` 为核心库，提供 `SignalFlags` 位图、`SigAction` 结构体、`SigProcMask` 屏蔽字等基础类型，并定义 `SignalOps` trait 作为抽象接口。`kernel/src/tasks/signal.rs` 中的 `SignalList` 实现挂起信号队列；`kernel/src/syscall/signal.rs` 实现 `rt_sigprocmask`、`rt_sigaction`、`rt_sigsuspend` 等系统调用；`kernel/src/syscall/types/signal.rs` 提供四架构（x86_64/riscv64/aarch64/loongarch64）的 `SignalUserContext` 用于上下文保存与恢复。与标准 ArceOS 基线相比，本仓库添加了完整的 Linux 信号系统调用接口（kill/tkill/tgkill/信号掩码/信号处理注册），弥补了 unikernel 设计中通常缺失的信号机制，但信号的分发与执行流程（trap 入口检测→用户态处理函数跳转→sigreturn）尚未完整实现。

### 1. 核心抽象：从位图到 trait 的层次设计

信号机制的数据核心位于 `crates/signal/src/lib.rs`，这是一个 `#![no_std]` 的独立 crate，通过 `bitflags!` 宏定义 `SignalFlags: u64` 类型，将 64 位整数的每一位映射为一个标准/实时信号 <sup>[1](#mod-signal-ref-1)</sup>。这种位图设计直接对标 Linux 内核的 `sigset_t`，优势在于：① 集合运算（并/交/补）可退化为按位操作，性能极佳；② 64 位的容量恰好覆盖标准信号（1-31）和 POSIX 实时信号（32-63），不需动态分配；③ 通过 `bit_field::BitField` 的特化方法可以按位读写而不引入分支。代价是硬编码了最大信号数 64——若未来需要扩展（如同 4.4BSD 的 `SIGRTMAX` 超过 63），整个位图布局需要重新设计。

`SignalFlags` 之上抽象了 `SignalOps` trait，定义了 `add_signal`、`handle_signal`、`check_signal`、`sigmask` 四个方法 <sup>[5](#mod-signal-ref-5)</sup>。这是一个有趣的早期设计意图：trait 提供了统一的信号操作接口，允许不同的调度器或线程模型注入各自的实现。但实际代码中，`kernel/src/tasks/signal.rs` 的 `SignalList` 结构体完全没有实现这个 trait，而是定义了自己的一套 `add_signal`/`has_signal`/`try_get_signal`/`remove_signal`/`mask` 方法 <sup>[12](#mod-signal-ref-12)</sup>。trait 与实现之间的断连表明该模块仍处于半工程化状态——`SignalOps` 可能是重构残留，也可能是未来策略模式的预留桩，但当前没有任何代码引用它。

`SigProcMask` 的 `masked` 方法值得特别审视：`(self.mask >> signum) & 1 == 0` 返回 `true` 表示信号**未被**屏蔽 <sup>[3](#mod-signal-ref-3)</sup>。这个语义是反向的——英文 `masked` 通常意味着“被屏蔽”，而此处实际是“未被屏蔽”。调用者必须记住这个逆语义，否则在条件判断中容易引入逻辑错误。Linux 内核使用 `sigisemptyset`、`sigismember` 等命名，语义更清晰。

### 2. 架构感知的上下文保存与 restore

信号处理要求对用户态执行上下文进行快照（`ucontext_t`），以便信号处理函数返回后恢复。`kernel/src/syscall/types/signal.rs` 使用 `cfg_if!` 宏为四种架构分别定义了 `SignalUserContext` 结构体 <sup>[10](#mod-signal-ref-10)</sup>。该设计放弃了 trait 多态，选择了编译期单体化（monomorphization），避免了虚函数开销，但也意味着增加新架构需要复制整个结构体定义。

各架构的 `store_ctx`/`restore_ctx` 实现差异反映了寄存器布局的不同：
- **x86_64** 需要手动映射 17 个通用寄存器，且顺序与 `TrapFrame` 中的字段顺序不一一对应（如 `gregs[0]` 存 `r8` 而非 `rax`）<sup>[10](#mod-signal-ref-10)</sup>。这种硬编码映射是架构特定的，任何对 `TrapFrame` 字段顺序的调整都会悄然破坏信号上下文。
- **riscv64** 直接写成 `self.gregs = ctx.x`，因为 RISC-V 的通用寄存器数组 `x[32]` 与 `TrapFrame` 布局一致 <sup>[11](#mod-signal-ref-11)</sup>，代码简洁且不易出错。
- **aarch64** 和 **loongarch64** 也采用整体赋值，但 loongarch64 的特殊之处在于 `_reserved: [usize; 16]` 字段的注释“减少到16个usize以避免栈溢出”<sup>[10](#mod-signal-ref-10)</sup>——这暗示了原来的大小可能在栈上分配时超过了某个阈值（可能是 4KB 内核栈），这是对栈资源紧张环境的有意识妥协。

`SignalStack` 和 `SignalStackFlags` 实现了备选信号栈（`sigaltstack`）的类型定义，但当前代码中栈切换逻辑尚未与 trap 处理流程集成。

### 3. 系统调用实现与信号模型语义

`kernel/src/syscall/signal.rs` 在 `UserTaskContainer` 上实现了四个关键系统调用，其语义遵循 Linux NPTL 线程信号模型，这一点在 `sys_sigaction` 上方的注释中明确说明：“每个线程都有自己独立的 signal mask，但所有线程共享进程的 signal action”<sup>[7](#mod-signal-ref-7)</sup>。这意味着：

- **`sys_sigprocmask`** 操作线程控制块（`tcb.sigmask`），使用 `SigProcMask::handle` 实现阻塞/解除/设置三种操作 <sup>[6](#mod-signal-ref-6)</sup>。它通过 `UserRef<SigProcMask>` 从用户空间读取参数，通过写锁（`tcb.write()`）保护屏蔽字的修改，完成后显式 `drop(tcb)` 释放锁。注意 `SigMaskHow::from_usize` 返回 `Option` 并用 `.ok_or(Errno::EINVAL)?` 处理非法参数——这是正确的错误传播，但调用者传入非法 `how` 值时系统调用直接返回 `EINVAL`，不会 panic。

- **`sys_sigaction`** 操作进程控制块（`pcb.lock().sigaction[sig]`），直接使用 `sig` 作为数组下标 <sup>[7](#mod-signal-ref-7)</sup>。这里存在越界风险：`sig` 作为 `usize` 从用户态传入，如果用户传入 0 或大于 64 的值，会直接访问数组越界。Linux 内核在 `copy_from_user` 后会对 `sig` 做范围校验（`signal_valid`）。此处缺失校验是安全性缺口。

- **`sys_sigsuspend`** 使用了一个无保护的 `loop { yield_now().await }` 轮询 `tcb.signal.has_signal()` <sup>[8](#mod-signal-ref-8)</sup>。这是协程友好的忙等待（不阻塞内核线程），但缺少超时机制：如果信号永不抵达（例如所有信号被屏蔽），调用将永远挂起。标准 Linux `sigsuspend` 会原子性地替换临时屏蔽字并挂起，但当前实现完全忽略了 `sigset` 参数——它只用于调试打印，实际的屏蔽字操作完全没有实现。

- **`sys_sigtimedwait`** 明确标记为 `TODO: finish sigtimedwait`，当前实现只调用了 `WaitSignal` 然后返回 0，完全不解析超时参数 <sup>[9](#mod-signal-ref-9)</sup>。这是 stub 级实现。

### 4. 边角细节与不足

以下是评审中需要关注的 7 个具体风险点 [ref:N]：

1. **<sup>[7](#mod-signal-ref-7)</sup> 数组越界风险**：`sys_sigaction` 中 `self.task.pcb.lock().sigaction[sig]` 对用户传入的 `sig` 无边界校验。攻击者或恶意用户程序传入 `sig=0` 或 `sig=64` 可导致内核越界访问（panic 或信息泄露）。

2. **<sup>[9](#mod-signal-ref-9)</sup> 半成品系统调用**：`sys_sigtimedwait` 带有 `TODO` 注释且逻辑不完整；`sys_sigsuspend` 忽略了 `sigset` 参数，无临时屏蔽字替换。

3. **<sup>[5](#mod-signal-ref-5)</sup> 未使用的 trait 接口**：`SignalOps` trait 定义在公共 crate 中，但没有任何类型实现它（`SignalList` 使用自己的独立方法）。这是否意味着模块拆分时遗留的抽象层？当前找不到任何 `impl SignalOps for ...`。

4. **<sup>[3](#mod-signal-ref-3)</sup> 命名倒置**：`SigProcMask::masked(signum)` 返回 `true` 表示信号**未屏蔽**，这与英文词义和调用者直觉相反。在复杂的信号处理逻辑中，这个倒置容易引入条件判断错误。

5. **<sup>[10](#mod-signal-ref-10)</sup> 栈大小隐含假设**：loongarch64 的 `_reserved` 字段从原始的完整大小缩减到 `[usize; 16]`，注释直言“避免栈溢出”。这说明该结构体的栈分配已接近内核栈上限（通常 4KB/8KB），任何后续字段添加都可能触发栈溢出。

6. **<sup>[1](#mod-signal-ref-1)</sup> 实时信号索引截断**：`SignalFlags::is_real_time` 使用魔数 `0xFFFFFFFE00000000` 判断位 33-63，但若信号编号超过 64，`from_num` 中的 `1 << (num-1)` 会发生移位溢出（`num=0` 时 `1 << (usize::MAX)` 是 UB）。

7. **信号分发流程缺失**：当前代码仅实现了信号注册和屏蔽的系统调用，但完整的信号处理还需要 trap 入口在返回用户态前检查 `SignalList`、构建 `SignalUserContext`、修改栈帧跳转到用户态信号处理函数、以及 `sigreturn` 恢复上下文。这些关键环节在已读文件中未见实现——`rt_sigreturn` 虽然在系统调用表中注册（`mod.rs:253`），但 `UserTaskContainer` 上没有对应的方法。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本仓库的进程间通信模块在 `kernel/src/syscall/shm.rs` 和 `kernel/src/tasks/shm.rs` 中实现了 System V IPC 的子集：共享内存（shmget/shmat/shmctl）、信号量（semget/semctl/semop）和消息队列（msgget/msgctl/msgsnd/msgrcv）。核心抽象围绕全局静态的 `BTreeMap` 和 `Arc<Mutex<T>>` 构建，每个 IPC 对象以键值对形式管理。与标准 ArceOS 相比，该仓独立添加了完整的 System V IPC 基础设施，包括用户态指针检查（`UserRef`）和简化的阻塞处理。模块尚处早期阶段，存在大量简化实现和 TODO，如 `semop` 仅返回成功，消息队列阻塞模式未真正实现。

### 1. 核心抽象与外部依赖

进程间通信模块的核心抽象是三个全局静态表：`SHARED_MEMORY`、`SEMAPHORES`、`MESSAGE_QUEUES`，均为 `Mutex<BTreeMap<usize, Arc<Mutex<T>>>>` 结构。每个 IPC 对象通过 `key`（用户指定的整数）索引，通过 `Arc` 进行引用计数共享，这使得多个进程可以通过 fork 或 exec 共享同一 IPC 对象。共享内存的具体物理页由 `FrameTracker` 管理，通过 `runtime::frame::frame_alloc_much` 批量分配 <sup>[1](#mod-ipc-ref-1)</sup>。信号量和消息队列的数据结构由 `Semaphore` 和 `MessageQueue` 直接定义，内部使用 `Vec` 存储计数或消息列表。这些设计体现了对 Linux System V IPC 接口的浅层兼容性，但未引入任何复杂的状态机或等待队列。

模块依赖 `sync::Mutex`（自旋锁）保护全局表，但未使用更高级的同步原语（如 `RwLock`），导致读操作也需要独占锁，这在多进程高频查询时可能造成性能瓶颈。此外，共享内存的 `FrameTracker` 通过 `MapedSharedMemory` 绑定到进程 PCB，并在进程退出时通过 `Drop` 实现引用计数递减与清理 <sup>[2](#mod-ipc-ref-2)</sup>。

### 2. 关键设计取舍

**2.1 键值分配策略**：当用户传入 `key == 0`（IPC_PRIVATE）时，模块自动生成一个递增的整数作为新键 <sup>[3](#mod-ipc-ref-3)</sup>。这种做法简单但存在缺陷：系统重启后键值可能重复，并且无法保证唯一性（若用户主动创建大量 IPC 对象可能导致溢出）。Linux 内核使用独立的 IDR 机制管理 IPC 标识符，这里用 `max().unwrap_or(0) + 1` 替代，放弃了 IDR 的预分配与回收能力。

**2.2 信号量 `semop` 的空实现**：`sys_semop` 仅检查信号量集合是否存在，直接返回 `Ok(0)` <sup>[4](#mod-ipc-ref-4)</sup>，完全忽略 `sembuf` 结构体中的操作。这意味着任何依赖信号量计数的用户程序（如 `lmbench`）都将得到虚假的成功。这种取舍是为了快速通过基本测试，但会隐藏并发错误：内核不会阻塞或唤醒任何进程，导致 `sem_wait` / `sem_post` 语义完全失效。

**2.3 消息队列阻塞处理的简化**：`sys_msgsnd` 和 `sys_msgrcv` 在队列满/空且未设置 `IPC_NOWAIT` 时本应阻塞，但当前实现直接返回成功或 `ENOMSG` <sup>[5](#mod-ipc-ref-5)</sup>。阻塞需要引入等待队列与调度器交互，而当前代码未实现任何睡眠/唤醒机制。这导致消息队列的流控完全失效，但满足了 LTP（Linux Test Project）测试的快速通过需求（许多测试不会验证阻塞行为）。

**2.4 共享内存映射地址的选择**：`sys_shmat` 中当用户未指定地址（`shmaddr == 0`）时，内核从 `task.get_last_free_addr()` 开始分配，但有一个下限 `va!(0x4000_0000)` <sup>[6](#mod-ipc-ref-6)</sup>。这个硬编码地址假设用户空间堆栈不会低于此值，但在不同架构（如 32 位）或定制链接脚本下可能冲突。更安全的做法是使用进程空闲区域（如通过 `mmap` 接口）。

**2.5 共享内存引用计数与删除**：`MapedSharedMemory` 的 `Drop` 实现了引用计数清理：仅在 `Arc` 强引用计数为 1 且 `deleted` 标记为 true 时才从全局表中移除 <sup>[2](#mod-ipc-ref-2)</sup>。这模仿了 Linux 的僵尸共享内存机制：即使 `shmctl(IPC_RMID)` 被调用，物理页也仅在所有进程分离后才释放。但当前 `deleted` 标记仅通过 `shmctl` 设置，未处理进程异常退出场景（如 `exec` 或崩溃），可能导致内存泄漏。

### 3. 跨模块协同

**3.1 与任务模块的绑定**：共享内存映射信息存储在 `TaskPCB.shms`（一个 `Vec<MapedSharedMemory>`）中，在 `sys_shmat` 时追加 <sup>[7](#mod-ipc-ref-7)</sup>。当进程通过 `exec` 或 `exit` 销毁时，PCB 的释放会自动触发 `MapedSharedMemory::drop`，从而管理引用计数。但这种设计未考虑 `fork` 场景：子进程会复制父进程的 `shms` 列表，导致子进程也“持有”相同的 `Arc<SharedMemory>`，引用计数正确增加，但子进程可能无法直接访问映射（因为地址空间被刷新）。目前未看到 `fork` 后对 `shms` 的调整。

**3.2 用户态指针的验证**：消息队列的读写操作通过 `UserRef::<u8>::from(msgp)` 构造用户空间指针引用，并调用 `is_valid()` 检查 <sup>[8](#mod-ipc-ref-8)</sup>。这使用了内核中通用的用户指针验证机制（`utils/useref`），防止用户传递无效地址导致内核崩溃。但注意 `slice_mut_with_len` 在长度超出时可能 panic，不过当前调用时长度是可控的（如 `msgsz`）。

**3.3 文件系统层模拟**：`filesystem/devfs/src/shm.rs` 提供了一个伪目录节点 `Shm`，使 `/dev/shm` 可以被挂载，但仅返回静态的 stat 信息 <sup>[9](#mod-ipc-ref-9)</sup>。这实现了 POSIX 共享内存文件系统接口的最小外观，但实际共享内存操作完全绕过了 VFS，直接走系统调用。这种分离避免了复杂的文件层抽象，但导致 `shm_open` 等标准函数无法在此内核上工作。

### 4. 边角细节与不足

**4.1 硬编码魔术数字**：`max_bytes: 16384`（MSGMNB）、`max_messages: 40` 硬编码在 `MessageQueue::new` 中 <sup>[10](#mod-ipc-ref-10)</sup>。这些值取自 Linux 默认值，但未被参数化，在多核大内存系统中可能偏小。修改这些值需要搜索所有引用，易遗漏。

**4.2 信号量 `semctl(IPC_RMID)` 的死锁风险**：在 `sys_semctl` 的 `cmd == 0` 分支中，先 `drop(sem)` 再 `drop(sems)`，然后重新获取 `SEMAPHORES.lock()` 并删除 <sup>[11](#mod-ipc-ref-11)</sup>。这种显式 `drop` 避免了在持有锁时移除自身导致的死锁，但增加了代码复杂性。若未来添加更多持有锁的路径，容易引入死锁。

**4.3 消息队列 `msgrcv` 的截断处理**：当消息实际大小大于用户提供的 `msgsz` 且未设置 `MSG_NOERROR` 时返回 `E2BIG`，但已写入前 `msgsz` 字节到用户空间 <sup>[12](#mod-ipc-ref-12)</sup>。这符合 POSIX 规范，但内核应先检查截断条件再写入，否则会污染用户部分缓冲区。当前实现顺序是先写入再检查，属于轻微规范违反。

**4.4 `shared_memory::deleted` 标记的并发安全**：`deleted` 是 `Mutex<bool>`，但 `MapedSharedMemory::drop` 中读取和判断是在持有 `self.mem` 的 `Arc` 引用后进行的，没有额外的锁保护。如果有两个进程同时 `shmdt`，可能同时进入 `drop` 并同时判断 `Arc::strong_count(&self.mem) == 1 && *self.mem.deleted.lock() == true`，导致两个进程都尝试 `remove`，但第二个会失败（已删除）。这不算严重，但缺少注释说明预期行为。

**4.5 `sys_shmctl` 中 `IPC_RMID` 只是设置 `deleted` 标记**，并未立即解除所有映射 <sup>[13](#mod-ipc-ref-13)</sup>。这是正确的（僵尸实现），但 `deleted` 标记没有与 `SHARED_MEMORY` 的锁同步：设置标记时持有 `SHARED_MEMORY.lock()`，但其他进程在 `shmat` 时并不检查 `deleted` 标记，导致已标记删除的共享内存仍可被新进程映射，破坏了语义。

**4.6 用户指针有效性检查的漏洞**：`UserRef::from(msgp)` 在 `sys_msgsnd` 和 `sys_msgrcv` 中分别调用，但未确保 `msgp + 8 + msgsz` 不越界。如果 `msgsz` 过大，`slice_mut_with_len` 内部会使用 `UserRef` 构造大范围，可能跨越有效用户区域，返回无效片但 `is_valid` 无法检测（仅检查起始地址）。这可能导致内核访问任意用户内存。

### 总结
该 IPC 模块实现了 System V IPC 的基本框架，通过全局表 + 引用计数实现跨进程共享，但存在大量早期简化：信号量操作空实现、消息队列无阻塞、用户指针检查不完整。共享内存和消息队列的引用计数与删除逻辑有设计亮点（僵尸实现），但细节处理粗糙。作为 `compiler` 类型系统软件中的 IPC 组件，其设计偏向快速迭代测试，距离生产级尚有较大差距。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块以 `Socket` 结构体为核心抽象，通过包装 `lose_net_stack` crate 的 `dyn SocketInterface` trait 对象，并实现 `INodeInterface` 适配 VFS 框架，使 socket 可当作文件描述符操作。系统调用层在 `UserTaskContainer` 中实现 15 个 socket 相关 syscall，依赖全局单例 `NET_SERVER` 管理协议栈。与 ArceOS 基线相比，该实现采用 `lose_net_stack`（而非 smoltcp），并引入端口重用时替换 fd 表条目的特殊语义，但存在多处 `unwrap`/`expect` 可能 panic、`setsockopt` 空实现、Drop 中强计数魔法数字等弱点。

### 1. 核心抽象与外部依赖

网络模块的核心抽象是 `Socket` 结构体 <sup>[10](#mod-net-ref-10)</sup>，它包装了 `lose_net_stack` crate 的 `dyn SocketInterface` trait 对象。这个设计将底层网络协议栈（TCP/UDP）与星瑞内核的 VFS 文件系统框架解耦。`Socket` 通过实现 `INodeInterface` trait 提供 `readat`/`writeat`/`poll`/`stat` 方法 <sup>[15](#mod-net-ref-15)</sup>，使得 socket 可以被当作文件描述符通过标准的 `read`/`write` 系统调用访问。同时，系统调用处理函数（如 `sys_recvfrom`、`sys_sendto`）直接调用 socket 的原生方法，绕过 VFS 层以获取更精细的控制（如获取对端地址）。

外部依赖主要是 `lose_net_stack` crate，它提供 `NetServer`、`NetInterface`、`SocketInterface` 等核心类型。全局的 `NET_SERVER` 是一个 `Lazy<Arc<NetServer<NetMod>>>` <sup>[2](#mod-net-ref-2)</sup>，其中 `NetMod` 实现了 `NetInterface` trait <sup>[1](#mod-net-ref-1)</sup>，封装了物理网卡发送和 MAC 地址获取。`NetMod::send` 直接调用 `get_net_device(0).send(data)`，绕过任何协议栈内部的发送队列，依赖底层网卡驱动提供同步发送接口。这种设计假设编号 0 的设备始终存在且可发送，不支持多网卡热插拔。

Socket 的创建通过 `Socket::new(domain, net_type)` <sup>[13](#mod-net-ref-13)</sup> 完成，根据 `NetType` 枚举（STEAM/DGRAME/RAW）调用 `NET_SERVER.blank_tcp()` 或 `blank_udp()` 创建协议栈内部连接对象。`NetType` 枚举的变体名存在拼写错误——`STEAM` 应为 `STREAM`，`DGRAME` 应为 `DGRAM` <sup>[9](#mod-net-ref-9)</sup>，这在内部使用不影响功能，但给调试和后续维护带来困惑。RAW 分支直接 `panic!`，表明原始套接字完全未实现。

### 2. 关键设计取舍

**2.1 Socket 作为 VFS 节点的代价。** 选择让 `Socket` 实现 `INodeInterface` 意味着 socket 操作可以复用 VFS 的文件描述符分配、poll 等成熟机制，但也带来语义不匹配问题。`readat` 需要处理单次读取与网络分包的差异——它引入了一个内部 buffer `self.buf` <sup>[15](#mod-net-ref-15)</sup> 来缓存 `recvfrom` 剩余的字节，确保后续 `read` 能返回连续的流。这个 buffer 的存在增加了复杂度：`readat` 中先检查 buffer 是否有残留数据，如果没有则调用 `recvfrom` 获取新数据，再从中拷贝请求长度，最后将未拷贝完的部分存回 buffer。如果 `recvfrom` 返回的数据比请求的多，buffer 机制能工作；但如果多次 `read` 之间没有新的数据到达，`recvfrom` 会阻塞（或返回 EWOULDBLOCK），此时 `readat` 依赖 buffer 中的残留数据可能已经陈旧。

**2.2 端口重用的“替换”语义。** `sys_bind` <sup>[4](#mod-net-ref-4)</sup> 中，如果目标端口已被占用，不是返回 `EADDRINUSE` 错误或等待，而是通过 `socket.reuse(port)` <sup>[14](#mod-net-ref-14)</sup> 创建一个全新的 `Socket` 实例并替换当前 fd 表中的条目。`reuse` 方法根据协议类型从 `NET_SERVER` 获取已绑定的内部连接对象。这意味着调用者持有的旧 `Arc<Socket>` 引用将指向一个被遗弃的 socket，而 fd 表已指向新 socket。这种设计避免了实现 `SO_REUSEADDR` 选项的复杂性，但破坏了 socket 生命周期的透明性——用户态程序无法感知 fd 背后的 socket 对象已被替换，如果应用程序持有通过 `dup` 或 `clone` 获得的多个 fd 引用同一个 socket，替换操作只会影响当前 fd。

**2.3 阻塞与非阻塞的异步模型。** 所有可能阻塞的系统调用（`accept`、`connect`、`recvfrom`）在循环中调用 `yield_now().await` <sup>[5](#mod-net-ref-5)</sup>，主动让出 CPU 等待网络事件。这是单线程内核中避免阻塞中断的关键设计，但同时也意味着这些调用是协作式阻塞——如果网络栈没有数据，任务会持续被调度但立即 yield，产生空转。对于非阻塞模式，`sys_recvfrom` 检查 `O_NONBLOCK` 标志并返回 `EAGAIN`，但 `sys_accept` 和 `sys_connect` 并未完整支持非阻塞：`sys_accept` 会忙等直到有连接，没有非阻塞的快速路径；`sys_connect` 会循环直到连接成功，不返回 `EINPROGRESS`。`sys_accept4` 的实现相对完整，先尝试 accept，如果阻塞则根据 `O_NONBLOCK` 返回 `EAGAIN` 或继续等待。

**2.4 `setsockopt` 与 `getsockopt` 的实现策略。** `sys_setsockopt` <sup>[8](#mod-net-ref-8)</sup> 直接返回 `Ok(0)`，不做任何操作。注释中包含一段中文讨论网络游戏体验和 `SO_ATTACH_FILTER`，表明开发者意识到这个空实现会影响实时通信，但未实现任何选项。`sys_getsockopt` <sup>[6](#mod-net-ref-6)</sup> 对发送缓冲 (0x7) 和接收缓冲 (0x8) 硬编码返回 32000，对选项 0x2 返回 2000，对 0x4 返回 `EPERM`。这些值无法通过 `setsockopt` 修改，导致网络性能调优完全失效。依赖 socket 选项的应用程序（如设置超时、缓冲区大小、keepalive）将静默失败或得到固定值。

### 3. 跨模块协同

**3.1 与 VFS 层的协同。** `Socket` 通过 `INodeInterface` 的四个方法暴露给文件系统层。`stat` 设置 `StatMode::SOCKET` 标识文件类型。`poll` <sup>[15](#mod-net-ref-15)</sup> 根据 `is_closed`、`get_remote` 和 `readable` 状态返回 `POLLOUT`/`POLLIN` 事件，实现了 `select`/`poll`/`epoll` 的基础支持。但 `poll` 中直接 `unwrap()` 调用 `is_closed()` 和 `readable()`，如果这些方法返回 `Err` 会导致内核 panic。系统调用处理函数（如 `sys_recvfrom`）优先使用 socket 的原生 `recv_from` 方法以获取对端地址，而通过 fd 的 `read` 系统调用会经过 `readat`，这意味着同一个 socket 可以通过两种路径读取数据，可能造成数据竞争——`readat` 和 `recvfrom` 都操作内部状态（`self.buf` 和 `self.inner`），尽管 `Mutex` 保护了 buffer，但 `recvfrom` 直接调用 `self.inner.recv_from()` 而不经过 buffer 缓存，可能导致 `readat` 中缓存的旧数据与新到达的数据交错。

**3.2 与任务调度器的协同。** 阻塞系统调用中的 `yield_now().await` <sup>[5](#mod-net-ref-5)</sup> 依赖于内核的异步运行时。每次循环迭代都会检查信号（如 `has_signal()` 返回 `EINTR`），使得被信号中断的系统调用能够及时返回。`sys_accept` 中先 `accept` 再检查信号的顺序意味着如果信号在 `accept` 阻塞期间到达，需要等到下一次循环才能响应，延迟在一个调度周期内。

**3.3 与设备驱动层的协同。** `NetMod::send` 直接调用 `get_net_device(0).send(data)` <sup>[1](#mod-net-ref-1)</sup>，没有经过协议栈的发送队列。接收路径则完全由 `lose_net_stack` 内部管理（未展示）。这种设计假设网卡驱动提供同步阻塞发送接口，且 `send` 调用成功后才返回。如果底层设备支持异步发送（如 DMA 中断完成），`NetMod::send` 会阻塞等待，与协作式调度模型产生张力。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架基于一组设备 trait（Driver、BlkDriver、NetDriver 等）和编译期注册机制（linkme 分布式切片 + DRIVER_REGS 映射），实现设备的统一发现、初始化和中断注册。核心创新在于引入两级中断注册流程（register_device_irqs → regist_devices_irq）以及 LoongArch64 下通过 PCI 总线枚举替代 FDT 的 VirtIO 发现路径。与 ArceOS 基线相比，该仓库增加了对 LoongArch64 平台 PLIC 的 S-mode 偏移处理、PCI BAR 分配逻辑，并保留了大量调试代码，反映了实验阶段的工程状态。

### 1. 核心抽象：基于 trait 的多态设备模型与编译期初始化注册

驱动框架的核心定义在 `crates/devices` crate 中。它提供了一组 trait：`Driver` 作为所有设备的基 trait，以及 `BlkDriver`、`NetDriver`、`UartDriver`、`IntDriver`、`InputDriver`、`RtcDriver` 等子 trait，分别对应不同功能的设备<sup>[1](#mod-drivers-ref-1)</sup>（`crates/devices/src/device.rs:18-28`）。`DeviceSet` 结构体持有各类设备的 `Vec`，并通过 `add_device` 方法根据 `get_device_wrapper()` 返回的 `DeviceType` 枚举自动分配到相应的向量中，同时将中断控制器和主 UART 设置到全局单例 `INT_DEVICE` 和 `MAIN_UART`<sup>[2](#mod-drivers-ref-2)</sup>（`crates/devices/src/device.rs:30-48`）。

驱动注册采用两种路径：
- **基于设备树的惰性注册**：`driver_define!(compatible_string, init_fn)` 宏将初始化函数与 FDT compatible 字符串关联，存入 `DRIVER_REGS` 全局映射。当 `try_to_add_device` 被调用时，遍历节点的 compatibles，找到匹配的函数并创建设备实例<sup>[3](#mod-drivers-ref-3)</sup>（`crates/devices/src/lib.rs:112-130`）。
- **直接初始化切片**：`driver_define!({...})` 形式会将闭包直接放入 `#[linkme::distributed_slice]` 声明的 `DRIVERS_INIT` 切片中。所有切片函数在 `prepare_drivers` 被同步调用，返回 `Option<Arc<dyn Driver>>` 并加入 `ALL_DEVICES`<sup>[4](#mod-drivers-ref-4)</sup>（`crates/devices/src/lib.rs:31,68-74`）。

这种双重注册机制兼顾了两种场景：对于有设备树的平台（如 RISC-V、ARM），通过 FDT 被动发现设备；对于需要主动枚举总线的平台（如 LoongArch64 的 PCI VirtIO），使用初始化切片直接执行设备发现逻辑。

### 2. 关键设计取舍：VirtIO 硬件适配层的平台特定假设

VirtIO 驱动的核心适配代码位于 `driver/kvirtio/src/virtio_impl.rs`，通过实现 `virtio-drivers` crate 的 `Hal` trait 完成 DMA 内存管理、物理地址转换和 MMIO 映射。这里暴露了几项重要的设计取舍：

- **MMIO 物理地址到虚拟地址的转换**：`mmio_phys_to_virt` 在 LoongArch64 上直接将物理地址与 `0x8000_0000_0000_0000` 进行按位或，得到 uncached 访问的虚拟地址<sup>[6](#mod-drivers-ref-6)</sup>（`driver/kvirtio/src/virtio_impl.rs:37-47`）。这是 LoongArch 特有的特性，在其他架构上不可用。该函数还包含一个测试读操作来验证映射，但若硬件未正确映射该区域，会导致 panic。
- **Buffer 共享的内存模型假设**：`share` 函数优先使用 `virt_to_phys` 通过页表翻译，但回退到 `raw_ptr & (!VIRT_ADDR_START)`，这是基于线性映射的简化做法<sup>[7](#mod-drivers-ref-7)</sup>（`driver/kvirtio/src/virtio_impl.rs:49-55`）。对于不使用线性映射的平台，一旦 `virt_to_phys` 失败（如页表未初始化），回退逻辑会错误地忽略高地址位，导致传输错误的数据物理地址。这实质上是假设系统在启用虚拟地址后仍存在恒等映射区域，暴露了早期开发的痕迹。
- **DMA 内存生命周期管理**：`dma_alloc` 调用 `frame_alloc_much` 分配物理连续页，然后将 `FrameTracker` 存储到全局 `VIRTIO_CONTAINER` 中防止回收。但 `dma_dealloc` 仅通过 `retain` 移除，不支持部分解除分配，且线性搜索效率低<sup>[5](#mod-drivers-ref-5)</sup>（`driver/kvirtio/src/virtio_impl.rs:8-35`）。注释掉的 `drain_filter` 表明开发者曾试图改进，但未完成。

### 3. 跨模块协同：设备树解析、中断注册与平台差异化流程

驱动框架与设备树解析、中断管理等多个模块紧密协同：

- **设备发现**：`try_to_add_device` 在解析 FDT 过程中被调用，通过 `DRIVER_REGS` 映射找到对应的驱动初始化函数<sup>[8](#mod-drivers-ref-8)</sup>（`crates/devices/src/lib.rs:76-88`）。对于 VirtIO MMIO 设备，`driver_define!("virtio,mmio", init_mmio)` 注册处理函数，该函数通过 MMIO 传输层创建设备，并根据设备类型分派到具体的子驱动<sup>[9](#mod-drivers-ref-9)</sup>（`driver/kvirtio/src/lib.rs:58-69,146`）。
- **中断注册链**：子驱动初始化时调用 `register_device_irqs` 将自身和中断号插入 `IRQ_MANAGER`；随后在 `regist_devices_irq` 中，通过 `INT_DEVICE`（即 PLIC 或其他中断控制器）的 `register_irq` 将设备注册到硬件中断控制器<sup>[10](#mod-drivers-ref-10)</sup>（`crates/devices/src/lib.rs:90-96`）。这个两段式设计将中断源收集与硬件编程分离，降低了驱动的耦合度。
- **LoongArch64 特例**：该架构下 VirtIO 不使用 FDT，而是通过 `driver_define!({...})` 直接执行 PCI 枚举<sup>[11](#mod-drivers-ref-11)</sup>（`driver/kvirtio/src/lib.rs:148-153`）。`enumerate_pci` 函数遍历 PCI 总线，对 VirtIO 兼容设备进行 BAR 分配、配置空间验证和传输层创建。这种平台差异化通过 `#[cfg(target_arch = "loongarch64")]` 条件编译实现，但也导致代码中存在大量重复的调试函数（如 `verify_pci_configuration`, `test_register_access`），说明该部分仍处于实验阶段。

### 4. 边角细节与不足

- **半成品 Input 驱动**：`virtio_input.rs` 中所有 `InputDriver` 方法均为 `todo!()`，整个驱动不可用<sup>[12](#mod-drivers-ref-12)</sup>（`driver/kvirtio/src/virtio_input.rs:34-48`）。
- **Ramdisk 镜像被完全注释**：`kramdisk/build.rs` 中原本包含 `MOUNT_IMG_PATH` 环境变量和 `.incbin` 生成逻辑，现已被替换为生成仅含一个字节的空段，导致 `ramdisk_start` 和 `ramdisk_end` 实际上没有包含任何镜像数据，`RamDiskBlock` 设备容量为 1 字节<sup>[13](#mod-drivers-ref-13)</sup>（`driver/kramdisk/build.rs:22-29`）。
- **UART 波特率硬编码**：`ns16550a.rs` 中调用 `init` 时使用 `Divisor::BAUD1200`，实际上 ns16550a crate 的 `Divisor` 枚举可能包含多个速率选择，但这里硬编码为 1200，与现代 Linux 默认的 115200 不符，可能导致串口通信错误<sup>[15](#mod-drivers-ref-15)</sup>（`driver/ns16550a/src/lib.rs:58`）。
- **`unsafe` 块的 SAFETY 注释缺失**：几乎所有驱动寄存器的直接写入（如 PLIC、Goldfish RTC、NS16550a）都使用了 `unsafe` 块，但没有任何 `// SAFETY:` 注释说明满足的安全契约，属于技术债<sup>[16](#mod-drivers-ref-16)</sup>（`driver/general-plic/src/plic.rs:15-25` 等）。
- **平台条件编译的风险**：`#[cfg(not(any(target_arch = "loongarch64")))]` 分支覆盖了除 loongarch64 外的所有架构，但未显式列出支持的架构（如 x86_64、riscv64、aarch64）。如果未来添加新架构（如 mips），该分支会静默选中，可能触发未测试的代码路径<sup>[9](#mod-drivers-ref-9)</sup>（`driver/kvirtio/src/lib.rs:146`）。
- **`prepare_drivers` 的错误处理**：该函数仅遍历 `DRIVERS_INIT` 并忽略返回 `None` 的初始化函数，若某驱动初始化失败仅输出 `debug` 日志，不会传播错误<sup>[4](#mod-drivers-ref-4)</sup>（`crates/devices/src/lib.rs:68-74`）。在 LoongArch64 的 `virtio_device_probe` 中，`unwrap()` 直接导致 panic，错误处理不一致。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层围绕 `UserTaskContainer` 结构体实现 150+ 个 Linux 兼容的系统调用，通过 `mod.rs` 中的 `syscall` 方法分发。核心抽象是使用 `UserRef<T>` 安全访问用户空间内存，并将文件描述符操作委托给 VFS。与同家族基线（ArceOS）相比，此版本大量采用 stub 实现和宽松错误处理（如 setuid 直接返回 0、mount 跳过真正挂载），目标是兼容 LTP 测试套件而非严格语义正确性。

### 1. 核心抽象与外部依赖
系统调用层没有自成一体的抽象层，而是将所有处理函数作为 `UserTaskContainer` 的方法实现。该结构体代表一个用户任务容器，封装了任务的文件描述符表、内存空间、trap 帧引用等。`mod.rs` 中的 `syscall` 方法根据 `Sysno` 枚举分发到具体的 `sys_*` 方法，参数通过 `UserRef<T>` 封装，避免直接解引用用户空间裸指针。文件描述符操作（如 `get_fd`、`alloc_fd`、`set_fd`）委托给内部 `Task` 对象，底层依赖 `fs` crate 的 VFS 实现。网络系统调用通过 `socket.rs` 模块和全局 `NET_SERVER` 与协议栈交互，信号系统调用则与 trap handler 配合（如 `arch_prctl` 通过 `force_cx_ref` 修改 trap 帧中的 TLS 寄存器<sup>[7](#mod-syscall-ref-7)</sup>）。这种设计将系统调用逻辑与内核其他部分解耦，但导致大量重复的 async 方法签名和类型转换。

### 2. 关键设计取舍
#### 2.1 Stub 实现优先于语义正确
为通过 LTP 测试，许多语义敏感的系统调用被简化为空操作或直接返回成功。例如 `sys_setuid`、`sys_setgid`、`sys_getuid` 等权限相关调用均返回 `Ok(0)`<sup>[9](#mod-syscall-ref-9)</sup>；`sys_mount` 对 tmpfs、proc 等类型直接返回成功而不实际挂载<sup>[10](#mod-syscall-ref-10)</sup>；`sys_faccessat2` 干脆写死 `Ok(0)`<sup>[8](#mod-syscall-ref-8)</sup>。设计者明确用 `warn!` 日志标记这些 bypass 行为，表明这是有意识的技术债务，而非疏忽。
#### 2.2 条件编译与架构兼容性
`mod.rs` 使用大量 `#[cfg(target_arch = "x86_64")]` 区分架构差异（如 x86_64 用 `open`、`stat`，其他架构用 `openat`、`newfstatat`）。但部分条件编译列表不完整：`chown` 系列系统调用的允许架构列表包含了 x86、mips、powerpc、sparc、s390x、arm 等，却遗漏了 riscv64<sup>[1](#mod-syscall-ref-1)</sup>。这导致 riscv64 上 `chown` 调用会落入 `_ => Err(Errno::EPERM)`，影响 LTP 测试覆盖。
#### 2.3 异步与方法签名不一致
多数系统调用方法声明为 `async fn`，但部分如 `sys_lseek` 是同步的（未标注 async）。参数类型也缺乏统一：`sys_read` 使用 `UserRef<u8>`，而 `sys_write` 却使用 `VirtAddr`<sup>[6](#mod-syscall-ref-6)</sup>，这迫使调用方在不同形状的指针封装间转换，增加了误用风险。

### 3. 跨模块协同
#### 3.1 与内存管理模块配合
系统调用通过 `UserRef` 请求内核验证并读写用户空间内存。`UserRef::get_mut()` 和 `slice_mut_with_len()` 内部调用页表检查（实现位于 `crate::utils::useref`），确保访问合法后通过直接指针操作完成。这种方式避免了每个系统调用重复实现内存检查，但 `UserRef::from(usize)` 构造器可能缺乏校验，若调用方传入任意地址可能导致内核访问非法内存。
#### 3.2 与任务调度器协调
`sys_sched_yield` 通过 `executor::yield_now` 主动让出 CPU；`sys_nanosleep` 等阻塞调用在 `time.rs` 中通过 `WaitHandleAbleSignal` 实现异步等待；`sys_wait4` 则在 `task.rs` 中等待子进程状态变化。这些跨模块接口依赖 `crate::tasks` 和 `executor` crate，但代码中大量使用 `.await`，最终由内核主调度器驱动。
#### 3.3 信号与 trap 帧交互
信号系统调用（`rt_sigaction`、`rt_sigprocmask`、`rt_sigreturn`）通过修改任务的内核信号屏蔽字和处理函数表来工作。`sys_arch_prctl` 直接修改 trap 帧中的 TLS 字段<sup>[7](#mod-syscall-ref-7)</sup>，以支持线程本地存储设置，这需要与用户态跳板代码紧密配合。

### 4. 边角细节与不足
- **条件编译遗漏架构**：`chown` 系列未包含 riscv64，可能导致该架构上相关调用失败<sup>[1](#mod-syscall-ref-1)</sup>。
- **不安全代码缺少文档**：`sys_getrandom` 使用 `static mut SEED` 且无 `// SAFETY` 注释，多核并发时存在数据竞争<sup>[2](#mod-syscall-ref-2)</sup>。
- **错误信息被吞没**：`sys_getdents64` 在 `getdents` 失败时返回 `Ok(0)` 而非错误码，用户程序会误以为目录结束<sup>[4](#mod-syscall-ref-4)</sup>。
- **实现不符合 POSIX 语义**：`sys_renameat2` 通过打开旧文件、读取内容、写入新文件来实现“重命名”，这既不原子也不支持目录重命名<sup>[5](#mod-syscall-ref-5)</sup>。`sys_mount` 对内存文件系统直接 `Ok(0)` 而不实际挂载，后续文件操作可能失败，但 mount 本身不会报错<sup>[10](#mod-syscall-ref-10)</sup>。
- **硬编码常量**：`AT_FDCWD` 定义为 -100 而非引用内核头文件，一旦内核常数变化需手动同步<sup>[3](#mod-syscall-ref-3)</sup>。
- **参数类型不一致**：`sys_write` 和 `sys_pwrite` 用 `VirtAddr`，而 `sys_read` 和 `sys_pread` 用 `UserRef<u8>`，迫使调用方做多余转换<sup>[6](#mod-syscall-ref-6)</sup>。
- **同步与异步混合**：`sys_lseek` 是同步非 async 函数，与模块整体 async 风格不符，可能在异步上下文中导致阻塞。
