## 📖 一、总览

本项目定位为基于 ArceOS/StarOS 的教学型内核变体，核心工程动机是将内核改造为自动化测试框架，而非实现完整 Linux ABI。它重写启动流程为测试用例调度器（[boot.ref:1]），围绕此目标扩展了进程、内存、文件、信号、IPC、网络等子系统，但大多数实现停留在“可用但不完善”的状态。整体架构上复用 ArceOS 的底层驱动和核心（axhal、axmm、axtask），在上层构建自己的抽象层（starry_core、starry_api、axprocess、axsignal），属于实验性、模块化的系统。最值得注意的三个特点：

1. **启动即测试**：将内核入口 `main` 重写为测试运行器，通过自动扫描或配置测试文件，顺序执行用例后关机（[boot.ref:1-4]）。这种设计在同类内核中极为罕见，将测试融入内核主线。
2. **两级信号管理器**：精确映射 Linux 进程共享动作表、线程独立阻塞的语义，并通过 `POST_TRAP` 钩子自动检查信号，无需系统调用显式触发（[signal.ref:2-4]）。
3. **分层进程扩展**：采用 `axtask::def_task_ext!` 宏将进程扩展数据附着在内核任务上，保持调度器纯净（[task.ref:1]）。

## 🎯 二、综合评价

### 整体定位与架构
该仓库是 arceos-starry 家族中一个实验性分支，核心贡献在于集成与测试。它打破了传统内核“启动 → shell”的模式，转向测试框架驱动。这种定位决定了子系统必须支持测试用例的产生与执行：内存模块需支持 ELF 加载和目标代码运行，进程模块需支持 fork/clone，文件系统需提供 proc 等虚拟文件。整体架构复用 ArceOS 的驱动和核心（axhal、axmm、axtask），在上层构建了自己的抽象层（starry_core、starry_api、axprocess、axsignal），各模块独立开发且通过测试验证。

### 真正的创新点
1. **启动即测试**：将 `main` 重写为测试调度器，自动扫描目录或读取环境变量配置测试列表，顺序执行后关机（[boot.ref:3-4]）。这体现了“测试内嵌于内核”的思想，但返回值未检查、默认配置为提示字符串等缺陷削弱了可靠性（[boot.ref:9]）。
2. **两级信号管理**：`ProcessSignalManager` 持有进程级动作表和 pending 队列，`ThreadSignalManager` 持有线程级阻塞掩码和 pending，并引用进程级管理器（[signal.ref:2-3]）。通过 `POST_TRAP` 回调自动检查信号，无需在 syscall 路径中显式插入（[signal.ref:4]）。
3. **分层进程扩展**：`TaskExt` 通过 `def_task_ext!` 宏附着到每个内核任务，再通过类型擦除 `data` 字段挂接 `ProcessData` 和 `ThreadData`（[task.ref:1-3]）。这种方式使 `axtask` 调度器无需知道进程语义，是架构解耦的典范。

### 关键取舍判断
- **精力集中点**：内存（ELF 加载、COW fork、用户地址空间管理）、文件系统（`FileLike` 统一接口、硬链接管理器、epoll）、进程（写时复制、pidfd）和系统调用层（150+ 个 syscall）。驱动直接沿用 ArceOS 基线，未重新实现。
- **刻意省略点**：信号恢复（`restore`）为空、调度系统调用无效、网络 `setsockopt` 空壳、shmctl 仅支持 `IPC_RMID`、管道采用 busy-wait 而非阻塞同步、shm 使用全局锁简化并发（[ipc.ref:1]）。这些要么是 TODO，要么是已知临时方案。
- **质量取舍**：多处返回值未检查（[boot.ref:9]）、`unsafe` 块缺少安全注释（[net.ref:7]）、条件编译假设不清晰（[mm.ref:1]）、TODO 遍布。代码处于快速迭代阶段，未经过严格审查。

### 完成度与不足
- **完成度**：166 个系统调用覆盖了主要子系统，核心 fork/execve/mmap/futex/pipe/shmat 等已提供基本功能。但信号恢复、调度策略、setsockopt 等大量为桩，管道和网络部分依赖 busy-wait，整体可运行 musl 动态/静态程序，但复杂负载可能崩溃。**评分：2**。
- **创新性**：两级信号管理和分层进程扩展有一定新颖性，启动即测试是一个有趣的切入点，但整体属于模块化集成而非原创理论突破。**评分：2**。
- **代码质量**：未检查错误、硬编码假设、忙等待循环、安全注释缺失。可读性尚可，但健壮性不足，生产不可用。**评分：1**。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块以 `main` 函数为内核入口（被 axruntime 调用），核心抽象是 `TestAction` 枚举（Marker/Command/CheckedCommand），用于描述测试动作。它通过 `axprocess::Process::new_init` 创建 init 进程，扫描或配置测试用例列表，顺序执行每个用例，最后调用 `axhal::misc::terminate()` 关机。与 ArceOS 基线的区别在于：基线直接启动 Shell 或 init 进程，而此版本将整个启动重写为测试运行器，大量使用文件系统准备和自动化测试调度逻辑。

### 1. 入口点与核心依赖

启动流程的入口是 `#[unsafe(no_mangle)] fn main()`（`src/main.rs:199`）<sup>[1](#mod-boot-ref-1)</sup>，这是一个在 `no_std` + `no_main` 环境下被 `axruntime` 调用的函数。它依赖以下外部 crate 完成初始化：

- `axprocess`：创建内核 init 进程（`Process::new_init`，行 201）<sup>[2](#mod-boot-ref-2)</sup>；
- `axtask`：获取当前任务 ID 作为进程标识；
- `axfs::api`：操作文件系统（创建目录、写 `/proc/` 文件、检查文件存在性）；
- `axhal::misc::terminate()`：测试完成后关闭系统；
- `starry_api::path::{FilePath, HARDLINK_MANAGER}`：为 busybox applet 创建硬链接。

`main` 的第一步是创建 init 进程：`axprocess::Process::new_init(axtask::current().id().as_u64() as _).build()`。这里 `build()` 的返回值未被检查，若初始化失败（如内存不足）会静默忽略——这是早期实现的一个风险点 <sup>[9](#mod-boot-ref-9)</sup>。

### 2. 测试用例调度：发现 vs 配置

启动流程的核心职责是确定要运行哪些测试用例。它支持两种模式：

1. **环境变量 `OSCOMP_RUNTIME_DISCOVERY=1` 触发自动发现**（`discover_testcases`，行 261）<sup>[3](#mod-boot-ref-3)</sup>：递归扫描 `/` 下最多 3 层的目录，检查是否存在 `basic/`、`busybox`、`lua` 等标记文件/目录，然后为每个套件添加批量测试（`add_basic_tests`、`add_busybox_tests` 等）。
2. **否则使用 `AX_TESTCASES_LIST` 环境变量**（`configured_testcases`，行 245）<sup>[4](#mod-boot-ref-4)</sup>：该变量未设置时会触发 `unwrap_or_else`，打印固定消息并 panic <sup>[5](#mod-boot-ref-5)</sup>。

```rust
let testcases = if option_env!("OSCOMP_RUNTIME_DISCOVERY") == Some("1") {
    discover_testcases()
} else {
    configured_testcases()
};
```

这种设计权衡了灵活性与确定性：发现模式适合 CI 环境自动化扩展测试集，配置模式适合精确定义回归用例。但发现模式依赖文件系统布局，若磁盘结构被恶意构造可能导致递归过深（虽然 depth=3 限制缓解了栈溢出风险）<sup>[6](#mod-boot-ref-6)</sup>。

所有测试用例被转换为 `TestAction` 枚举（Marker/Command/CheckedCommand），然后**顺序执行**。`CheckedCommand` 会检查退出码并输出 `success/fail`，但即使子进程失败也不会阻断后续用例——这是典型的“收集尽可能多结果”的测试哲学。

### 3. 跨模块协同：进程、文件系统与硬链接

启动流程需要多个模块紧密配合：

- **进程创建与执行**：`run_command`（行 225）<sup>[7](#mod-boot-ref-7)</sup> 调用 `entry::run_user_app(args, &envs)` 加载并运行用户程序。该函数期望返回 `Result<Option<i32>>`，其中 `Some(exit_code)` 表示正常退出，`None` 或 `Err` 表示执行失败。失败时仅记录 `error!` 并返回 `None`，不中断整体测试。这意味着用户态可控的 `args` 可以通过让 `run_user_app` 返回 `Err` 来触发大量 error 日志，但不会导致 DoS（因为主循环继续）。
- **文件系统准备**：`prepare_runtime_files()`（行 641 起）<sup>[8](#mod-boot-ref-8)</sup> 创建 `/tmp`、`/var/tmp`，并写入模拟的 `/proc/mounts` 和 `/proc/meminfo`。这些伪文件的内容是硬编码的，跨平台时可能不匹配实际内存大小——若用户态测试读取 `/proc/meminfo` 并依赖精确值，会导致误测。
- **硬链接批量创建**：`prepare_busybox_dir` 使用 `starry_api::HARDLINK_MANAGER` 为 busybox 的每个 applet 创建硬链接。这避免了磁盘空间浪费，但依赖 `HARDLINK_MANAGER` 的实现正确性；若链接已存在则跳过（检查 `metadata`）。

此外，`libctest` 测试通过 `dynamic_libctest_enabled()` 函数（行 343）<sup>[10](#mod-boot-ref-10)</sup> 在 `loongarch64` 架构上禁用动态链接测试，因为该架构的 musl 工具链可能缺少动态库支持——这是 `#[cfg]` 条件编译的一个防御性用例。

### 4. 边角细节与不足

以下是代码审查中发现的具体风险与设计瑕疵：

**1. `unsafe` 块缺少安全注释**：`#[unsafe(no_mangle)]` 使用了 `unsafe` 限定符（Rust 2024 风格），但未提供 `// SAFETY:` 注释说明其合理性。虽然 `no_mangle` 通常安全，但 `unsafe` 关键字暗示可能存在其它不安全操作，这会降低评审效率 <sup>[1](#mod-boot-ref-1)</sup>。

**2. 配置缺失时的 panic 路径**：`option_env!("AX_TESTCASES_LIST").unwrap_or_else(|| "Please specify...")` 实际上返回的默认值是**一个字符串字面量**，它会被当作单个测试用例名称处理（因为它不是有效的 `shlex` 拆分），导致 `configured_testcases` 返回空列表 <sup>[4](#mod-boot-ref-4)</sup>。更合理的做法是 `unwrap_or_else(|| "")` 然后过滤空字符串。目前如果未设置环境变量且未启用运行时发现，测试列表将是空的，且不会报错——这可能让开发者误以为测试通过。

**3. 硬编码常量与路径假设**：测试列表（`BASIC_TESTS`、`LUA_TESTS` 等）和路径 `/musl`、`/glibc` 是硬编码的 <sup>[8](#mod-boot-ref-8)</sup>。如果文件系统布局与预期不符（如 `/musl` 不存在），相关测试组会被静默跳过（`add_basic_tests` 中检查 `metadata` 后返回）。这种“沉默失败”在自动化测试中会掩盖配置错误。

**4. 递归深度有限但无扩展性**：`collect_suite_dirs` 硬限制深度为 3 层 <sup>[6](#mod-boot-ref-6)</sup>。如果测试套件被组织在更深目录中（如 `/musl/extra/deep/basic`），将不会被发现。虽然当前测试套件结构满足该限制，但未来扩展时需要修改常量。

**5. 错误吞没**：`run_command` 在 `entry::run_user_app` 返回 `Err` 时只记录 `error!` 并返回 `None`，调用者（主循环）仅通过 `is_some_and` 检查退出码 <sup>[7](#mod-boot-ref-7)</sup>。这意味着**加载失败（如 ELF 格式错误）不会让测试标记为失败**，因为 `CheckedCommand` 只在 `code == 0 || accept_nonzero` 时认为成功。如果 `run_user_app` 返回 `Err`，则 `code` 是 `None`，导致 `is_some_and` 返回 `false`，标记为 fail——但这依赖于 `run_user_app` 的错误语义，当前未在文档中明确。

**6. `#[cfg]` 的不完整架构覆盖**：`dynamic_libctest_enabled` 排除了 `loongarch64`，但未考虑 `riscv64`、`mips` 等。如果将来在 riscv64 上运行，动态链接测试可能错误地启用并崩溃 <sup>[10](#mod-boot-ref-10)</sup>。

**7. 没有测试失败的总计数**：主循环只逐个打印结果，未统计通过/失败总数。最终的 `#### OS COMP TESTS FINISHED ####` 标记无法直接判断整体成功与否，必须依赖外部脚本解析输出。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `axmm::AddrSpace` 为核心抽象，封装页表操作，为星云OS提供用户态地址空间管理能力。它解决了多进程虚拟地址空间隔离、ELF加载、堆栈布局、mmap/mprotect等POSIX内存系统调用兼容问题。与ArceOS基线相比，该模块引入了完整的用户地址空间生命周期管理、条件编译的内核映射复制策略、信号跳板固定映射，以及基于percpu标志的用户内存访问安全机制。

### 1. 核心抽象与外部依赖

本模块围绕 `axmm::AddrSpace` 构建，该类型来自外部crate `axmm`，封装了页表(Pagetable)操作，如 `map_alloc`、`unmap`、`protect`、`handle_page_fault`、`write`、`read` 等。`AddrSpace` 实例与每个进程绑定，存储在 `process_data.aspace` 中，并通过 `Mutex` 保护以支持并发访问 <sup>[5](#mod-mm-ref-5)</sup>。

地址空间的基本参数（基址、大小、堆栈区域）由 `axconfig::plat` 平台模块提供，例如 `USER_SPACE_BASE`、`USER_STACK_TOP`、`USER_HEAP_BASE` 等 <sup>[1](#mod-mm-ref-1)</sup>。这种分层设计使得跨架构移植时只需修改平台配置，而不影响逻辑。

此外，模块依赖 `axhal` 提供的物理/虚拟地址转换、页表标志定义([`MappingFlags`])，以及 `percpu` crate 提供的 per-CPU 变量机制，用于标记当前是否正在访问用户内存。

### 2. 关键设计取舍

**地址空间隔离与内核映射复制策略**

`copy_from_kernel` 函数根据目标架构决定是否将内核地址空间映射复制到用户页表 <sup>[1](#mod-mm-ref-1)</sup>：
- 在 aarch64 和 loongarch64 上，硬件使用独立页表寄存器（TTBR0_EL1 / PGDL），用户页表无需包含内核映射，因此直接跳过复制。
- 在其他架构（x86_64, riscv64）上，用户和内核共享同一页表层次结构，必须复制内核映射以保证系统调用和中断处理程序可达。

此策略利用条件编译 (`#[cfg]`) 做出了正确的硬件假设，但若未来支持新架构（如 x86_32 或更多 RISC-V 变体），需手动更新此列表，否则可能导致缺页或权限错误。

**用户内存访问安全保护**

`ACCESSING_USER_MEM` 是一个 per-CPU 布尔标志 <sup>[2](#mod-mm-ref-2)</sup>，通过 `access_user_memory()` 闭包临时置位。当该标志为 `true` 时，页错误处理程序才会将用户态发起的缺页错误交给 `aspace.handle_page_fault()` 处理 <sup>[8](#mod-mm-ref-8)</sup>；否则，内核缺页会被视为真正的故障（返回 `false`，可能触发 oops）。这一机制避免了内核在遍历用户页表时误吞错误。

**信号跳板的固定地址映射**

`map_trampoline` 使用 `map_linear` 将信号处理跳板映射到固定虚拟地址 `SIGNAL_TRAMPOLINE` <sup>[3](#mod-mm-ref-3)</sup>。该地址在编译期即确定，无需在每次信号递送时查询页表，提升了信号处理的性能。代价是牺牲了一段固定虚拟地址空间，且需要确保 `virt_to_phys` 转换正确。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块实现了兼容 Linux 的进程与线程管理，核心抽象基于 `axprocess` crate 的 `Process`、`Thread`、`ProcessGroup`、`Session` 类型，并通过 `starry_core::task` 中的 `TaskExt`、`ProcessData`、`ThreadData` 扩展附加调度、信号、地址空间、futex 等上下文。与 ArceOS 基线相比，该仓库将进程模型从单内核任务扩展到支持多线程、写时复制 fork、命名空间和 pidfd，但调度系统调用仍为桩实现，未与底层 `axtask` 调度器联动，且 exit 时命名空间资源未释放。

### 1. 核心抽象：分层进程模型与类型擦除扩展

进程模型采用三层抽象：底层 `axtask` crate 提供通用的内核任务调度（`TaskInner`、`WaitQueue`），中层 `axprocess` crate 定义 `Process`、`Thread`、`ProcessGroup`、`Session` 的进程树结构，上层 `starry_core::task` 通过 `axtask::def_task_ext!` 将 `TaskExt` 附加到每个内核任务，再通过类型擦除的 `data` 字段将 `ProcessData` 和 `ThreadData` 挂接到 `axprocess` 的对象上 <sup>[1](#mod-task-ref-1)</sup>。这种分层避免了进程逻辑侵入调度器核心：调度器只负责 `TaskInner` 的上下文切换和时间片，而进程退出、信号传递、地址空间管理完全在扩展层实现。

`ProcessData` 持有地址空间 (`Arc<Mutex<AddrSpace>>`)、信号管理器、futex 表、子进程等待队列和 SysV 共享内存映射表 <sup>[2](#mod-task-ref-2)</sup>。`ThreadData` 持有 `clear_child_tid`、`robust_list_head` 和线程级信号管理器 <sup>[3](#mod-task-ref-3)</sup>。两者均通过 `Once<Box<dyn Any + Send + Sync>>` 实现类型擦除，使 `axprocess` 本身不依赖具体扩展类型，保持接口清洁。

### 2. 关键设计取舍：写时复制、命名空间与调度桩

**fork 与写时复制 (COW)**：`sys_clone` 在创建进程时（非 `CLONE_THREAD`），调用 `aspace.try_clone()` 创建地址空间副本，随后通过 `copy_from_kernel` 复制内核映射 <sup>[4](#mod-task-ref-4)</sup>。`try_clone` 实现了 COW 页表 —— 父子进程共享物理页，在首次写入时触发缺页复制。这一设计放弃了立即拷贝所有用户内存的开销，但要求 MMU 支持脏页追踪。对于不支持 COW 的架构（如部分 RISC-V 实现），该路径可能回退到完整拷贝，但当前代码未提供 fallback。

**命名空间隔离**：每个 `ProcessData` 携带独立的 `AxNamespace`，用于隔离文件描述符表、当前目录等资源。`clone.rs` 在创建进程时根据 `CLONE_FILES`/`CLONE_FS` 标志决定是共享还是复制这些资源，通过 `deref_from` 绑定到命名空间 <sup>[5](#mod-task-ref-5)</sup>。然而 `do_exit` 中的 `FD_TABLE.clear()` 直接清空当前命名空间的表，而注释 `// TODO: clear namespace resources` 和 `// FIXME: axns should drop all the resources` 表明命名空间的完整生命周期管理尚未实现 <sup>[6](#mod-task-ref-6)</sup>。

**调度系统调用实现**：`sched_getscheduler`、`sched_setscheduler`、`sched_getparam` 等均读写全局静态变量 (`SCHED_POLICY`、`SCHED_PRIORITY` 等)，未与 `axtask` 调度器交互 <sup>[7](#mod-task-ref-7)</sup>。这意味着用户程序设置的调度策略和优先级实际不起作用，`sched_setaffinity` 也仅验证 `mask[0] & 1` 非零而跳过亲和性设置。这一取舍换取了系统调用兼容性（libc 不会崩溃），但放弃了实时调度特性，属于早期桩实现。

### 3. 跨模块协同：创建、退出与等待的完整链路

**进程创建** (`sys_clone`)：解析 `CloneFlags` 后，通过 `new_user_task` 创建 `axtask::TaskInner`，其入口闭包执行用户上下文切换。对于新进程，调用 `parent.fork(tid)` 构建 `axprocess::Process` 骨架，再通过 `ProcessData::new` 附加地址空间、信号动作和命名空间。若 `CLONE_VM` 未设置，则克隆地址空间；若 `CLONE_SIGHAND` 未设置，则创建新的信号操作表 <sup>[8](#mod-task-ref-8)</sup>。最后 `add_thread_to_table` 将线程、进程、进程组和会话注册到全局 `WeakMap` 表中，`ThreadBuilder::build` 将线程加入进程线程列表。

**退出** (`do_exit`)：首先处理 `clear_child_tid`：将用户空间 `tid` 清 0 并唤醒可能的 futex 等待者。然后遍历健壮 futex 列表，通过原子 CAS 将 futex 字标记为 `FUTEX_OWNER_DIED` <sup>[9](#mod-task-ref-9)</sup>。`thread.exit()` 检查是否所有线程已退出，若是则标记进程为僵尸，释放地址空间的用户映射，向父进程发送 `exit_signal`，并通知父进程的 `child_exit_wq`。`group_exit` 时向同组所有线程发送 `SIGKILL`。

**等待** (`sys_waitpid`/`sys_waitid`)：遍历父进程的 `children()`，根据 `WaitPid` 选择器和 `WCLONE`/`WALL` 过滤。若找到僵尸进程则读取 `exit_code`，并视 `WNOWAIT` 决定是否 `free()`。若无僵尸且 `WNOHANG` 未设置，则阻塞在 `ProcessData::child_exit_wq` 上，等待子进程退出通知 <sup>[10](#mod-task-ref-10)</sup>。该等待队列基于 `axtask::WaitQueue`，但不能被信号唤醒（与 Linux `wait4` 语义不符）。

### 4. 边角细节与不足

- **调度系统调用无实际效果**：全局 `SCHED_POLICY` 等变量仅用于 get/set，`axtask` 调度器始终使用默认策略 <sup>[7](#mod-task-ref-7)</sup>。
- **`unsafe` 缺少安全注释**：`exit.rs` 中 `let futex = unsafe { &*futex_ptr.cast::<AtomicU32>() };` 在用户地址上构造原子引用，依赖 `UserPtr::get_as_mut` 的合法性检查，但未提供 `// SAFETY:` 说明 <sup>[11](#mod-task-ref-11)</sup>。
- **nanosleep 不支持信号唤醒**：`sys_nanosleep` 和 `sys_clock_nanosleep` 使用 `axtask::sleep` 阻塞，但注释 `TODO: should be woken by signals, and set errno` 表明信号唤醒未实现 <sup>[12](#mod-task-ref-12)</sup>。
- **架构特定条件编译**：`Drop for ProcessData` 仅在非 aarch64 和非 loongarch64 上清除内核映射，假设这些架构的内核地址空间无需清理 <sup>[13](#mod-task-ref-13)</sup>。该假设可能随内核布局变化失效。
- **命名空间资源泄漏**：`do_exit` 中 `FD_TABLE.clear()` 后显式标记 `// TODO` 和 `// FIXME`，命名空间自身的内存和对象未回收 <sup>[6](#mod-task-ref-6)</sup>。
- **全局表使用 `WeakMap` 但无驱逐机制**：`THREAD_TABLE` 等使用 `Weak` 值，`Arc` 释放后条目自动失效。但若进程频繁创建退出，表体积可能膨胀，因为失效的 `Weak` 条目不会被主动清除（除非 `WeakMap` 内部有惰性清理）。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块通过 `FileLike` trait 为用户态系统调用提供统一的文件描述符抽象，封装了文件、目录、管道、eventfd、timerfd、epoll 等对象，并实现了一套用户态的硬链接管理器。与 ArceOS 基线不同，本仓库在 `axfs::fops` 上层增加了动态分发的 `FileLike` 接口，引入独立的硬链接重定向层，并实现了 Linux 兼容的 eventfd/timerfd/epoll 机制。

### 1. 核心抽象：FileLike trait 与统一文件描述符模型

文件系统模块的核心抽象是 `FileLike` trait（定义在 `api/src/file.rs`，未提供片段，但从各子模块的调用可推断）。该 trait 定义了 `read`、`write`、`read_at`、`write_at`、`seek`、`stat`、`poll`、`set_nonblocking`、`add_seals`、`get_seals`、`descriptor_flags`、`status_flags`、`truncate`、`flush`、`into_any` 等方法。所有可被文件描述符引用的对象——包括文件 (`File`)、目录 (`Directory`)、管道 (`Pipe`)、eventfd (`EventFd`)、timerfd (`TimerFd`)、epoll 实例 (`EpollInstance`)——都实现了此 trait。文件描述符表 `FD_TABLE`（类型为 `RwLock<...>`，见 `fd_ops.rs:127` 的 `FD_TABLE.write()` 调用）以 `Arc<dyn FileLike>` 存储，这实现了统一的动态分发：用户态系统调用仅通过 `fd` 索引到 `Arc<dyn FileLike>`，再调用相应方法，无需区分具体类型。

这种设计借鉴了 Linux 的“一切皆文件”哲学，但与 ArceOS 基线不同：基线中 `axfs::fops::File` 和 `axfs::fops::Directory` 是独立的，系统调用代码需分别 `match`；本模块通过 `FileLike` 统一，简化了 syscall 实现。代价是增加了一次 vtable 间接跳转，且 trait 方法无法内联。

`FileLike` 的实现与底层 `axfs::fops` 接口耦合。例如 `File` 结构体内部持有 `axfs::fops::File` 和路径字符串；`Directory` 内部持有 `axfs::fops::Directory`。`fd_ops.rs:69-98` 的 `sys_openat` 中，如果路径对应目录则创建 `Directory`，否则创建 `File`，并通过 `add_to_fd_table` 注册到 `FD_TABLE`。

### 2. 关键设计取舍：用户态指针安全、硬链接管理器、同步原语

**用户态指针封装**：`UserPtr<T>` 和 `UserConstPtr<T>`（定义在 `api/src/ptr.rs`，未提供片段）是安全访问用户态内存的封装。调用 `get_as_mut_slice`、`get_as_slice`、`get_as_ref`、`get_as_mut`、`get_as_str` 等方法时，会通过地址检查验证指针在当前进程的用户空间范围内，防止内核访问无效内存。例如 `io.rs:21` 的 `buf.get_as_mut_slice(len)?`。这种设计避免了在每个系统调用中手动检查和转换，但也要求这些方法在内部正确处理页边界和未映射区域。

**硬链接管理器**：`path.rs` 中的 `HardlinkManager` 是一个纯用户态实现的硬链接层。它使用 `BTreeMap<String, String>` 记录软/硬链接映射，并通过引用计数管理实际文件的删除：当链接计数归零时，才调用 `axfs::api::remove_file` 删除目标文件（`path.rs:251-258`）。这与 Linux 内核的 VFS 层硬链接不同，后者依赖底层文件系统（如 ext4）的 inode 链接计数。此实现的局限性在于：链接信息仅存储在内存中的 `RwLock` 内，系统崩溃或重启后丢失；且不支持跨文件系统的硬链接。但作为轻量级兼容方案，它满足了基本链接语义。

**同步原语**：模块内部大量使用 `axsync::Mutex` 和 `spin::RwLock` 以及 `AtomicBool`。例如 `EventFd` 和 `TimerFd` 用 `Mutex` 保护内部状态（`eventpoll.rs:30, 309`），`HardlinkManager` 用 `RwLock` 保护链接表（`path.rs:114`），`EpollInstance` 用 `Mutex` 保护注册事件列表（`eventpoll.rs:341`）。锁粒度方面，`epoll_wait_inner` 遍历全部注册 fd 时，仅短暂持锁获取快照，但等待循环中对每个 fd 单独调用 `get_file_like`（持全局 `FD_TABLE` 读锁）和 `file.poll()`，这些操作在锁外完成，避免了长时持锁。但全局 `FD_TABLE` 是一个单一 `RwLock`，当频繁打开/关闭/查询时，会成为共享瓶颈。

### 3. 跨模块协同：epoll 轮询与事件驱动

`epoll` 的实现展示了文件系统模块与其他模块（如设备、时间）的深度协同。`EpollInstance` 的 `sys_epoll_ctl`（`eventpoll.rs:388-426`）向 `epoll` 注册一个 fd 及其关注的事件（EPOLLIN/EPOLLOUT/EPOLLET/EPOLLONESHOT）。`sys_epoll_pwait` 调用 `epoll_wait_inner`，该函数循环执行以下步骤：
1. 持 `epoll` 的锁获取事件快照 (`eventpoll.rs:471`)；
2. 对每个 fd 调用 `get_file_like(fd)?.poll()` 获取当前状态；
3. 根据注册事件和边缘触发（EPOLLET）逻辑决定是否上报；
4. 若收集到事件，回写用户空间 `epoll_event` 数组并返回；
5. 若无事件且未超时，调用 `axtask::yield_now()` 让出 CPU（`eventpoll.rs:534`）。

`FileLike::poll` 方法连接了具体对象的就绪状态。例如 `EventFd::poll` 返回 `counter > 0` 作为可读标志（`eventpoll.rs:114-117`），`TimerFd::poll` 调用 `check_expiration` 检查是否有到期（`eventpoll.rs:332-335`）。普通的文件对象（File/Directory）的 `poll` 行为由底层 `axfs::fops` 定义。

这一协同模式体现了“轮询统一接口”的设计：所有实现了 `FileLike` 的对象都可以被 epoll 监控，无需为每种特殊文件编写独立的等待队列。缺点是阻塞式读写（如 `EventFd::read` 和 `TimerFd::read`）内部也使用 `axtask::yield_now()` 忙等待，而非高效的等待队列，这会导致 CPU 忙耗。

### 4. 边角细节与不足

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 基于 axsignal crate 实现的两级信号管理器（ProcessSignalManager + ThreadSignalManager），提供 Linux 兼容的阻塞掩码、动作表、待处理队列和 signalfd 支持。核心抽象是 ThreadSignalManager 持有线程级 pending 队列和 blocked 掩码，并引用进程级 ProcessSignalManager 共享 actions 和进程级 pending。与基线 ArceOS 相比，本实现增加了 SignalFd 文件描述符和完整的系统调用集，但信号处理函数执行（restore）和 Stop/Continue 动作仍为占位。

### 1. 核心抽象与外部依赖

信号模块的核心抽象是**两级信号管理器**：`ProcessSignalManager` 负责进程级的动作表（`SignalActions`）和待处理信号队列；`ThreadSignalManager` 负责线程级的阻塞掩码、当前信号栈以及线程级待处理队列，并通过 `Arc` 引用进程级管理器 <sup>[2](#mod-signal-ref-2)</sup><sup>[3](#mod-signal-ref-3)</sup>。这种分层设计直接映射了 Linux 中进程共享动作、线程独立阻塞的语义。信号集 (`SignalSet`) 用 `u64` 位图表示，支持 `BitOrAssign`/`BitAndAssign` 等操作；`Signo` 枚举覆盖了 Linux 标准的 19 个信号，但缺少 `SIGRTMIN`~`SIGRTMAX` 实时信号 <sup>[8](#mod-signal-ref-8)</sup>。

外部依赖集中在三个方向：
- **trap 框架**：`api/src/signal.rs` 中的 `post_trap_callback` 通过 `#[register_trap_handler(POST_TRAP)]` 注册到 axhal 的 trap 处理链，每次从用户态返回时自动调用 `check_signals` <sup>[4](#mod-signal-ref-4)</sup>。
- **进程/线程数据**：`ProcessSignalManager` 与 `ThreadSignalManager` 分别存放在 `ProcessData` 和 `ThreadData` 中，通过 `current().task_ext().process_data() / thread_data()` 访问 <sup>[3](#mod-signal-ref-3)</sup>。
- **用户态指针**：系统调用实现大量使用 `UserPtr` / `UserConstPtr` 安全地读写用户空间，并通过 `nullable!` 宏处理空指针 <sup>[1](#mod-signal-ref-1)</sup>。

`axsignal` crate 还定义了通用 `WaitQueue` trait，允许替换等待原语（当前默认实现未提供，由上层注入），增加可测试性。

### 2. 关键设计取舍

**取舍 1：两级 pending 队列 vs 单一队列**。Linux 规定某些信号（如 SIGCHLD）必须发送到进程，某些（如 SIGSEGV）必须发送到引发异常的线程。本实现为每个线程保留独立 `pending` 队列，同时进程级也维护一个 `pending`，`ThreadSignalManager::pending()` 合并两者 <sup>[7](#mod-signal-ref-7)</sup>。这带来了 `take_matching_signal` 时的搜索开销（需要线性扫描两个 `VecDeque`），但正确实现了语义。

**取舍 2：SpinMutex 粗粒度并发**。`ThreadSignalManager` 的 `blocked`、`pending`、`stack` 分别用 `SpinMutex` 保护，而 `ProcessSignalManager` 的 `actions` 使用 `Mutex`（可能是休眠锁）。在 `check_signals` 和 `with_blocked_mut` 中会嵌套锁定（先锁 blocked，再锁 pending），但锁顺序一致（blocked → pending），未导致死锁。然而 `SpinMutex` 在中断上下文中危险——`post_trap_callback` 运行在 trap 处理中，如果抢占被禁用，spin 循环可能导致死等。当前没有评测是否会在中断中竞争，这是一个潜在风险点 <sup>[6](#mod-signal-ref-6)</sup>。

**取舍 3：信号跳板页编译期固定**。`axsignal::arch` 模块声明一个 4096 字节对齐的静态数组作为跳板页，地址通过 `signal_trampoline_address()` 返回 <sup>[9](#mod-signal-ref-9)</sup>。这避免了动态内存分配和页表映射开销，但假设所有进程共享同一个跳板虚拟地址，限制了地址空间隔离（例如 KASLR 场景）。

**取舍 4：SignalFd 基于当前线程**。`SignalFd::read` 和 `SignalFd::poll` 始终调用 `current().task_ext().thread_data().signal` 获取当前线程的 `ThreadSignalManager`，而不是存储打开 fd 时的目标线程 <sup>[5](#mod-signal-ref-5)</sup>。这意味着如果一个线程创建 signalfd 后传递给另一个线程读取，实际检查的是读取线程而非创建线程的待处理信号——这违反了 Linux 中 signalfd 与打开时线程/进程绑定的语义。`SignalFd::from_fd` 也未实现（只看到 `SignalFd::new`），`from_fd` 用于更新已有 fd 的标志，但代码中未定义，暗示可能未完整实现。

**取舍 5：对不可阻塞/不可改变信号的处理**。`sys_rt_sigaction` 拒绝对 SIGKILL 和 SIGSTOP 修改动作 <sup>[1](#mod-signal-ref-1)</sup>；`sys_rt_sigsuspend` 从临时掩码中移除这两者；`check_signals` 的动作映射将 SIGKILL/SIGTERM/SIGSEGV 映射为 `Terminate` <sup>[6](#mod-signal-ref-6)</sup>。这些硬编码符合 POSIX，但 `SignalOSAction::Stop` 和 `Continue` 的 handler 分支未实现真实逻辑（见 §4）。

### 3. 跨模块协同与 trap 集成

信号检查的入口是 `api/src/signal.rs` 中的 `post_trap_callback`，它被注册为 axhal 的 `POST_TRAP` 回调 <sup>[4](#mod-signal-ref-4)</sup>。每次从用户态返回时，该函数调用 `check_signals(tf, None)`。`check_signals` 内部调用 `ThreadSignalManager::check_signals`，取出当前线程的第一个未阻塞信号，并根据信号类型决定动作 <sup>[6](#mod-signal-ref-6)</sup>。

动作决策后：
- `Terminate` → 调用 `do_exit(128+signo, true)`，进程退出。
- `Stop` / `Continue` → 目前仅打印 TODO 或直接忽略。
- `Handler` → 不执行任何操作，意味着信号处理函数从未被实际调用。

信号发送链：`sys_kill` / `sys_tkill` / `sys_tgkill` 等系统调用先构造 `SignalInfo`，然后调用 `send_signal_process` / `send_signal_thread`。这些函数通过 `proc.data::<ProcessData>()` 或 `thr.data::<ThreadData>()` 获取对应的管理器，并调用其 `send_signal` 方法 <sup>[3](#mod-signal-ref-3)</sup><sup>[7](#mod-signal-ref-7)</sup>。`send_signal` 将信号压入 `pending` 队列后调用 `wait_queue.notify_one()` 唤醒可能阻塞在 `sigtimedwait` 或 `sigsuspend` 的线程。

`sys_rt_sigreturn` 是信号处理函数返回的关键，但它的实现仅委托给 `ThreadSignalManager::restore(tf)`，而该函数体为空（`{}`）<sup>[10](#mod-signal-ref-10)</sup>。这意味着信号处理函数执行后无法正确恢复上下文，整个用户态信号处理机制实际上断裂。没有看到 sigframe 的构建和栈切换逻辑，这与基线 ArceOS 的早期状态一致。

### 4. 边角细节与不足

- **<sup>[3](#mod-signal-ref-3)</sup> TODO 权限检查**：`sys_kill` 和 `sys_tkill` 注释 `// TODO: should also check permissions`，当前不验证发送者是否有权发送信号（如跨用户发送可能导致未授权信号投递）。
- **<sup>[6](#mod-signal-ref-6)</sup> 阻塞信号与非阻塞信号冲突**：`check_signals` 中 `take_unblocked_signal` 只考虑线程级 blocked，但未检查进程级 blocked（虽然 Linux 不支持进程级阻塞掩码，但此实现中 `ProcessSignalManager` 没有 blocked 字段，也许正确）。
- **<sup>[5](#mod-signal-ref-5)</sup> SignalFd 线程绑定错误**：`SignalFd::poll` 和 `read` 使用 `current()` 而非保存的线程引用，导致 signalfd 语义错误；`from_fd` 未实现，`sys_signalfd4` 中 `fd >= 0` 分支调用 `SignalFd::from_fd(fd)?` 但该函数未定义，应无法编译（可能是 `starry_core` 中的关联函数？）。
- **<sup>[10](#mod-signal-ref-10)</sup> restore 为空**：`ThreadSignalManager::restore` 函数体为空，导致 `sys_rt_sigreturn` 无法恢复栈帧，信号处理函数实际上不可用。
- **<sup>[9](#mod-signal-ref-9)</sup> 跳板页只读假设**：`SignalTrampolinePage` 未标记为 `const`，但静态变量存储在 `.bss`？实际上 `static` 默认可变，但代码从未写入，应该标记为 `const` 以避免意外修改。
- **<sup>[1](#mod-signal-ref-1)</sup> 对 SIGSTOP 的阻塞掩码操作可能被忽略**：`sys_rt_sigprocmask` 没有拒绝阻塞 SIGSTOP/SIGKILL（虽然 Linux 也会忽略），但 `sys_rt_sigsuspend` 手动移除它们，存在不一致。
- **实时信号缺失**：`Signo` 枚举仅到 19，缺失 `SIGRTMIN` (32) 到 `SIGRTMAX` 的实时信号。

总体而言，该模块提供了信号机制的基础骨架，但核心的信号处理函数执行路径和进程管理交互（stop/continue）仍为占位，距离可用还有较大距离。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了三种IPC原语：System V共享内存、匿名管道和futex快速互斥。共享内存基于全局ShmManager管理，通过axmm::SharedPages实现零拷贝映射；管道采用固定大小(256B)环形缓冲区，以忙等待(yield_now)同步，注释标记TODO待优化；futex通过进程私有的FutexTable将用户地址映射到axtask::WaitQueue，支持WAIT/WAKE/REQUEUE。与ArceOS基线相比，该仓显著扩展了IPC覆盖：增加了完整的shm系列和futex系统调用，并引入进程粒度的futex表自动清理机制。

### 1. 核心抽象与外部依赖
本模块的三种IPC机制各自依赖不同的抽象层。System V共享内存的核心是全局`ShmManager`<sup>[1](#mod-ipc-ref-1)</sup>，它持有两个BTreeMap分别按key和id索引`ShmSegment`，每个段包含一个`Arc<SharedPages>`<sup>[2](#mod-ipc-ref-2)</sup>——来自axmm crate的共享物理页抽象，支持通过`aspace.map_shared`映射到多进程地址空间实现零拷贝。管道基于`PipeRingBuffer`循环队列<sup>[3](#mod-ipc-ref-3)</sup>，其大小256字节编译期固定，通过`Arc<Mutex<PipeRingBuffer>>`在读写端共享。Futex底层依赖`axtask::WaitQueue`，`FutexTable`将用户地址映射到`Arc<WaitQueue>`<sup>[4](#mod-ipc-ref-4)</sup>，每个进程维护自己的`futex_table`。

### 2. 关键设计取舍
**共享内存全局锁 vs 进程本地锁**：`SHM_MANAGER`是全局`static Mutex`<sup>[1](#mod-ipc-ref-1)</sup>，所有shm操作必须获取它。这简化了并发控制，但可能成为瓶颈。替代方案（每个段独立锁）复杂度更高，设计者选择全局锁，并将attach/detach中的地址空间操作放在进程本地`aspace.lock`下以缩小临界区。

**管道固定缓冲区与忙等待**：管道选择256B固定缓冲区适合小数据IPC，但不适用流式大数据。最值得关注的是同步方式：`read`和`write`在数据未就绪时循环调用`axtask::yield_now()`<sup>[5](#mod-ipc-ref-5)</sup>，这是纯粹的忙等待（协程友好但非阻塞），没有条件变量或waker。注释明确标记`// TODO: use synconize primitive`，表明作者已知此临时方案。在单核协作调度下可用，但多核/抢占环境会导致CPU空转。

**Futex生命周期管理**：`FutexTable`使用`BTreeMap`存储地址到`WaitQueue`的映射，`WaitQueueGuard`通过RAII在Drop时自动清理空队列<sup>[6](#mod-ipc-ref-6)</sup>。它检查`Arc::strong_count == 1`（仅自己被引用）且队列无等待者才移除映射，避免了手动管理，但引入`current()`调用和加锁，可能触发死锁（见不足）。

### 3. 跨模块协同
**共享内存与内存管理**：`sys_shmat`通过`process_data().aspace.lock()`获取地址空间锁，调用`find_free_area`找空闲区域，然后`map_shared`映射共享页<sup>[7](#mod-ipc-ref-7)</sup>。`map_shared`来自axmm，其内部维护页表并引用`SharedPages`。`sys_shmdt`调用`unmap`后手动刷新TLB（`axhal::arch::flush_tlb`）<sup>[8](#mod-ipc-ref-8)</sup>，展示了mm模块向上层IPC提供通用共享映射。

**Futex与任务调度**：`FutexTable`的`WaitQueue`来自axtask，`wq.wait()`和`wq.notify_one()`直接挂起/唤醒任务。`sys_futex`在`FUTEX_WAIT`中调用`wq.wait_timeout`<sup>[9](#mod-ipc-ref-9)</sup>，在持用户地址锁时出让CPU。注意，`FutexTable`存储在`process_data`，故futex默认进程内；要支持跨进程需将表放在共享内存，目前未实现。

**管道与文件描述符系统**：`sys_pipe2`创建一对`Pipe`，通过`add_to_fd_table`注册到当前进程fd表<sup>[10](#mod-ipc-ref-10)</sup>。`Pipe`实现了`FileLike` trait，可通过统一read/write接口使用，与普通文件一致。

### 4. 边角细节与不足
1. **管道忙等待**：`axtask::yield_now()`循环让出CPU，无阻塞机制，在无其他任务时空转<sup>[5](#mod-ipc-ref-5)</sup>。
2. **管道缓冲区硬编码**：`RING_BUFFER_SIZE = 256`固定，不可配置<sup>[3](#mod-ipc-ref-3)</sup>。
3. **shmctl功能缺失**：仅支持`IPC_RMID`，其他命令返回`EINVAL`<sup>[11](#mod-ipc-ref-11)</sup>。
4. **shmat不支持SHM_REMAP**：直接返回`EINVAL`，而Linux允许替换已有映射<sup>[12](#mod-ipc-ref-12)</sup>。
5. **futex忽略私有标志**：`FUTEX_CMD_MASK`丢弃`FUTEX_PRIVATE_FLAG`，进程内和共享futex行为相同，可能误唤醒<sup>[13](#mod-ipc-ref-13)</sup>。
6. **WaitQueueGuard Drop潜在死锁**：`drop`中获取`current()`并锁定`futex_table`，若当前任务在持有其他锁时释放guard，可能锁顺序反转<sup>[6](#mod-ipc-ref-6)</sup>。
7. **sys_pipe2忽略flags**：包括`O_NONBLOCK`、`O_CLOEXEC`被忽略<sup>[10](#mod-ipc-ref-10)</sup>。

这些不足表明IPC实现处于早期可用阶段，许多Linux语义尚未完备。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块通过 `Socket` 枚举（Tcp/Udp/Unix）统一管理三种 socket 类型，依赖 `axnet` crate 实现 TCP/UDP 协议栈，并自实现 Unix domain socket 的 socketpair。地址处理使用 `SocketAddrExt` trait 在用户/内核空间间序列化 `sockaddr`。与 ArceOS 基线相比，本模块增加了 AF_UNIX 支持，但 `setsockopt` 为空壳，且 Unix socket 阻塞模型采用 busy-wait 而非条件变量。

### 1. 核心抽象：Socket 枚举与 SocketAddrExt trait

网络模块的核心抽象是 `Socket` 枚举<sup>[1](#mod-net-ref-1)</sup>，它统一了三种 socket 类型：TCP、UDP 和 Unix domain socket。TCP 和 UDP 直接封装自 `axnet` crate 的 `TcpSocket` 和 `UdpSocket`<sup>[9](#mod-net-ref-9)</sup>，而 Unix domain socket 则是完全自实现的 `UnixSocket` 结构体。每个 socket 变体都包裹在 `Mutex`（或 `AtomicBool`）中，提供内部可变性以支持多线程共享。

地址处理层面定义了 `SocketAddrExt` trait<sup>[10](#mod-net-ref-10)</sup>，为 `SocketAddr`、`SocketAddrV4`、`SocketAddrV6` 添加了 `read_from_user` 和 `write_to_user` 方法。这解决了用户空间与内核空间之间 sockaddr 结构的序列化/反序列化问题，避免了为每种地址类型重复编写拷贝逻辑。底层使用 `copy_sockaddr_from_user` 配合 `MaybeUninit` 暂存数据<sup>[7](#mod-net-ref-7)</sup>，通过 unsafe 的 `copy_nonoverlapping` 完成内核栈上的拷贝。

### 2. 关键设计取舍

**Unix domain socket 的自实现**：与 ArceOS 基线仅支持 AF_INET/AF_INET6 不同，本模块通过 `UnixSocket` 结构体实现了 AF_UNIX SOCK_STREAM/SOCK_DGRAM 的 socketpair<sup>[5](#mod-net-ref-5)</sup>。它使用两个 `Arc<Mutex<VecDeque<u8>>>` 作为双工管道，通过引用计数检测对端关闭（`Arc::strong_count == 1` 时返回 EOF 或 EPIPE）。这个实现轻量但功能有限：不支持命名 socket（`bind`/`connect` 返回 `EOPNOTSUPP`），也不支持辅助数据（ancillary data）。

**Busy-wait 阻塞模型**：`UnixSocket::recv` 在无数据且非非阻塞模式下会循环 `axtask::yield_now()` 直至数据到来<sup>[6](#mod-net-ref-6)</sup>。这本质上是 busy-wait，没有使用条件变量或 epoll 就绪通知。优点是不依赖内核通知机制，在单线程协程调度下可行；缺点是 CPU 空转，且与 `axnet` 底层的事件驱动模型存在范式冲突。

**setsockopt 的空壳实现**：`sys_setsockopt` 仅验证 fd 有效性，所有选项（`SO_REUSEADDR`、`SO_RCVBUF` 等）均被静默忽略<sup>[3](#mod-net-ref-3)</sup>。这能通过许多未检查选项的应用程序，但对于依赖特定 socket 选项（如 `TCP_NODELAY`）的程序会造成功能缺失。`getsockopt` 也仅返回硬编码值：`SO_TYPE` 根据变体返回 `SOCK_STREAM` 或 `SOCK_DGRAM`<sup>[4](#mod-net-ref-4)</sup>，`SO_SNDBUF`/`SO_RCVBUF` 固定返回 64KB。

### 3. 跨模块协同

系统调用层（`imp/net.rs`）通过 `Socket::from_fd(fd)?` 从全局 fd 表获取 `Socket` 实例，然后调用其方法。`Socket` 的方法（如 `sendto`、`recvfrom`）通过 match 分派到具体变体的实现。这种桥接模式将 Linux 系统调用语义映射到 `axnet` 的接口。

地址拷贝路径中，`copy_sockaddr_from_user` 使用 `unsafe` 的 `copy_nonoverlapping` 在用户和内核空间之间传输 sockaddr 数据<sup>[7](#mod-net-ref-7)</sup>。长度检查在调用前完成，确保不超过 `size_of::<sockaddr>()`。`recvmsg` 的 `gather_iovs` 函数实现了 scatter-gather 操作，将用户空间的 iovec 数组拷贝为连续缓冲区，但硬编码了 `iovlen` 上限 1024。

### 4. 边角细节与不足

<sup>[2](#mod-net-ref-2)</sup> `sys_recvmsg` 对 datagram socket 的特殊处理：在读完第一个 iov 后立即 break，避免跨 datagram 边界。这是正确的语义，但注释缺位。

<sup>[4](#mod-net-ref-4)</sup> `getsockopt` 中 Unix socket 的 `SO_TYPE` 返回 `SOCK_STREAM`，符合 Linux 惯例，但未区分 Unix SOCK_DGRAM（本模块支持但未暴露）。

<sup>[8](#mod-net-ref-8)</sup> `stat` 方法返回硬编码的 `0o777` 权限，与真实访问控制无关，注释明确标注“not really implemented”。

不足主要体现在：

* **setsockopt 空壳**<sup>[3](#mod-net-ref-3)</sup>：任何选项设置都不报错也不生效，可能静默破坏应用行为。
* **UnixSocket busy-wait**<sup>[6](#mod-net-ref-6)</sup>：无事件通知机制，高并发场景下 CPU 浪费严重，且与 `axnet` 的异步模型不协调。
* **unsafe 缺少 SAFETY 注释**<sup>[7](#mod-net-ref-7)</sup>：`copy_sockaddr_from_user` 中的 `unsafe` 块未说明安全不变式，依赖外部长度检查但未在代码中显式声明。
* **Unix socket 功能缺失**：不支持命名 socket、辅助数据、`SO_PASSCRED` 等，`listen`/`accept` 返回 `EOPNOTSUPP`。
* **`sys_recvmsg` 的 iovlen 上限 1024**：硬编码常量，可能不兼容某些大量 scatter-gather 的应用。

```rust
fn recv(&self, buf: &mut [u8]) -> LinuxResult<usize> {
    if buf.is_empty() { return Ok(0); }
    loop {
        let mut incoming = self.incoming.lock();
        let read_size = incoming.len().min(buf.len());
        if read_size > 0 {
            for slot in buf.iter_mut().take(read_size) {
                *slot = incoming.pop_front().unwrap_or(0);
            }
            return Ok(read_size);
        }
        if Arc::strong_count(&self.incoming) == 1 { return Ok(0); }
        if self.nonblocking.load(Ordering::Relaxed) { return Err(LinuxError::EAGAIN); }
        drop(incoming);
        axtask::yield_now();
    }
}
```

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层是用户态与内核态之间的桥梁，核心抽象为一个由 `#[register_trap_handler(SYSCALL)]` 宏注册的集中式分发函数 `handle_syscall`，它通过 `syscalls::Sysno` 枚举将系统调用号映射到 `starry_api` crate 中具体的 `sys_*` 实现函数。所有参数通过 `TrapFrame::arg0()`–`arg5()` 访问，并使用 `LinuxError` 统一处理错误（返回负错误码）。与 ArceOS 基线相比，本模块显式地支持了 150+ 个 Linux 系统调用，覆盖文件系统、I/O、网络、信号、进程管理、内存管理等完整子系统，并引入了 `time_stat_from_user_to_kernel`/`time_stat_from_kernel_to_user` 调用链用于内核态时间统计埋点。

### 1. 核心抽象：基于宏注册的集中式系统调用分发器

系统调用层的入口是一个以 `#[register_trap_handler(SYSCALL)]` 属性标注的函数 `handle_syscall` <sup>[1](#mod-syscall-ref-1)</sup>。这个宏来自 `axhal::trap` 模块，它将函数注册为 CPU 系统调用中断（通常为 x86_64 的 `int 0x80` / `syscall` 指令或 riscv64 的 `ecall`）的处理例程。设计上回避了动态注册或链表遍历：`register_trap_handler` 大概率在编译期将函数指针填入中断向量表，使得 syscall 从陷入到分发的路径仅需一次间接跳转。这与 Linux 内核的 `sys_call_table` 思路一致，但省去了查表偏移计算的开销 <sup>[2](#mod-syscall-ref-2)</sup>。

函数签名 `fn handle_syscall(tf: &mut TrapFrame, syscall_num: usize) -> isize` 暴露了两个关键抽象：`TrapFrame` 封装了 CPU 寄存器状态，`syscall_num` 是用户态传入的系统调用号。内部通过 `Sysno::from(syscall_num as u32)` 将原始整数提升为类型安全的 `syscalls::Sysno` 枚举 <sup>[3](#mod-syscall-ref-3)</sup>——这个枚举来自 `syscalls` crate，为每个 Linux 系统调用定义了命名常量，避免了 magic number 散落在代码中。随后一个巨大的 `match sysno` 将枚举变体映射到对应的 `sys_*` 函数调用，这些函数全部来自 `starry_api::*`（glob import）<sup>[4](#mod-syscall-ref-4)</sup>。

### 2. 关键设计取舍：架构适配、错误处理与参数传递

**架构条件编译。** 代码中大量使用 `#[cfg(target_arch = "x86_64")]` 来 gate 特定系统调用（如 `rename`、`link`、`unlink`、`open`、`dup2`、`pipe`、`stat`、`lstat`、`newfstatat`、`fork`、`getrlimit`、`arch_prctl`）<sup>[5](#mod-syscall-ref-5)</sup>。x86_64 有自己的一套系统调用号分配，而 riscv64/aarch64 使用不同的编号体系，且某些调用（如 `arch_prctl`）是 x86 特有的。这里的选择是：在非 x86_64 架构上不注册这些分支，而是依赖 `Sysno::fstatat`（行 229-232 有 `#[cfg(not(target_arch = "x86_64"))]` 的 fstatat 分支）等替代入口。这种做法的代价是如果未来需要支持 mips 或 loongarch，必须为每个新架构重复相同的 cfg 模式——缺少一个表格驱动（table-driven）的架构抽象层。

**错误处理策略。** 所有 `sys_*` 函数返回 `Result<_, LinuxError>`。主函数通过 `result.unwrap_or_else(|err| -err.code() as _)` 将 `Err` 转换为负数（Linux 内核约定：负的错误码表示失败），`Ok` 值则直接传递 <sup>[6](#mod-syscall-ref-6)</sup>。未匹配的系统调用走默认分支 `_ => { warn!(...); Err(LinuxError::ENOSYS) }` <sup>[7](#mod-syscall-ref-7)</sup>。这里有一个细微但重要的问题：`unwrap_or_else` 没有区分“调用成功但返回负数”的情况——Linux 有些系统调用（如 `getpriority`）返回值本身可以是负数，但这里会将任何 `Err` 都当作错误处理，而 `Ok(-1)` 会被正确传递。只要 `sys_*` 实现不把合法负值包在 `Err` 里，这个设计就是安全的。

**参数传递模式。** 代码在参数转换上混合了两种风格：`tf.arg0() as _` 和 `tf.arg1().into()`。`as _` 用于整数类型转换（如 `i32` → `usize`、`usize` → `i32`），可能截断（例如 64 位系统上 `usize` → `i32` 会丢失高位）。`.into()` 调用 `From` trait，常见于指针/缓冲区参数向用户态指针类型的转换（如 `*const u8` 转 `UserPtr`）。这种混合说明 `TrapFrame::argN()` 返回的是一个通用整数类型（如 `usize`），需要下游根据语义做转换——整数用 `as`，指针用 `into()`。问题在于 `as _` 静默截断：如果用户传入的地址高位非零（虽然实际用户空间地址通常 < 48 位），`as i32` 会触发未定义行为？实际上在 Rust 中 `as` 是定义良好的 wrapping 转换，但语义上可能不符合 POSIX 规范（如 `ftruncate` 的 `length` 参数在 32-bit 兼容模式下需要被截断，但在 64-bit 模式下不应截断）。

**`pipe` 与 `pipe2` 的合并。** 行 117-118 将 x86_64 上的 `Sysno::pipe` 映射到 `sys_pipe2(tf.arg0().into(), 0)`（flags=0），从而复用 `pipe2` 实现。这是一个典型的“旧调用转新调用”简化策略，避免了为 `pipe` 单独编写处理函数 <sup>[8](#mod-syscall-ref-8)</sup>。

**模式匹配的语法糖运用。** 行 168 使用 `Sysno::fsync | Sysno::fdatasync => sys_fsync(tf.arg0() as _)`，Rust 的 `|` 模式允许两个枚举变体共享同一处理分支。这比写两个重复分支更简洁，且编译期会检查枚举覆盖完备性。

### 3. 跨模块协同：TrapFrame、starry_api 与时间统计的配合

系统调用层本身只负责路由，不实现任何具体逻辑。真正的文件操作、网络传输、进程调度等全部委托给 `starry_api` crate——这是一个明显的分层设计：系统调用层处理“如何从用户态接收请求”，`starry_api` 处理“这个请求的内核逻辑是什么”。这使得在不修改系统调用分发代码的情况下可以独立替换 `starry_api` 的实现。

与 `axhal::trap` 的协同有两个层面：一是 `SYSCALL` 常量和 `register_trap_handler` 宏决定了 `handle_syscall` 被链接到哪个中断向量；二是 `TrapFrame` 的 `arg0()`–`arg5()` 方法屏蔽了不同架构的寄存器参数传递约定（x86_64 用 `rdi, rsi, rdx, r10, r8, r9`，riscv64 用 `a0–a5`），使得上层的 match 代码与架构无关。

`execve`、`clone`、`clone3`、`fork` 和 `rt_sigreturn` 等系统调用特殊地接收了 `tf: &mut TrapFrame` 本身，而非仅使用其参数访问方法 <sup>[10](#mod-syscall-ref-10)</sup>。这是因为这些调用需要修改 CPU 上下文（如设置新任务的寄存器状态、信号返回时的栈帧修复），这通常需要直接操作 `TrapFrame` 中的寄存器字段。
