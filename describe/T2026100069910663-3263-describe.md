## 📖 一、总览

该仓库是 rCore-Rust 家族中一个独特的教学-实验混合型内核，它以运行特定 musl libc 用户态为目标，在单体内核框架下实现了从启动到多线程 IPC 的完整操作系统链。该内核选择在功能广度上大力扩展（176 个 syscall），同时刻意放弃多核支持、高性能并发模型和严格安全隔离，从而在单一硬件线程上实现了类 Linux 的复杂语义。内核的启动流程将汇编嵌入 Rust 源码，内存管理提供了 SV39 页表、COW、惰性分配等现代特性[mm.ref:2][mm.ref:3]，进程调度则采用全局锁与中心化定时器管理各种等待队列[task.ref:1]。其最值得注意的特点有三：一是通过合成信号 SIGCANCEL 与 musl libc 线程取消的深度集成，这在内核与 libc 协同设计上迈出了探索性的一步；二是文件系统与网络子系统的“双栈并存”架构——一套功能完整的 RamFS 和自研 TCP/UDP loopback 绕过协议栈，另一套基于 smoltcp 的标准栈却未被集成[fs.ref:4][net.ref:3]，折射出开发过程的实验性；三是在驱动层从零实现 VirtIO 块设备，未依赖社区 crate，但完全采用轮询和全局串行锁，体现了“够用即可”的效率权衡[drivers.ref:1][drivers.ref:5]。这些设计使得该内核成为一个考察 Rust 语言在受限环境下实现 POSIX 语义的生动样本，而非一个可实际部署的系统。

## 🎯 二、综合评价

### 整体定位
该内核处于教学与实用之间的模糊地带。与纯教学性质的 rCore-Tutorial 相比，它大幅超越了简单的用户态 shell 演示，可以运行多线程测试套件；但与追求 Linux 兼容的完整内核相比，它又充满硬编码限制（如 64KB socket 缓冲、480 MiB mmap 范围[mm.ref:1]）和功能缺失（无多核调度）。实际上，该内核更像一个“musl libc 适配器”：许多细节（如硬编码 TLS 偏移、SIGCANCEL、robust futex 清理）都指向了与特定 libc 实现的共生关系。这种定位使得其完成度不能以通用 OS 标准衡量，而应看作一个探索特定 libc 兼容性的实验平台。

### 真正的创新点
1. **合成信号与线程取消协议**：信号模块引入的非标准 SIGCANCEL(33) 是亮点。该信号由内核生成，用于配合 musl 的 pthread_cancel，在不完全依赖用户态 setjmp/longjmp 的情况下实现线程取消。同时，信号动作表支持 fork 深拷贝与 pthread 共享两种克隆语义[signal.ref:2]，这在同类内核中鲜有完整实现。
2. **自研 VirtIO 块驱动**：与 rCore-Tutorial 直接使用 virtio-drivers crate 不同，本内核手写了从寄存器操作到队列管理的全套逻辑[drivers.ref:2]，虽然仅支持轮询模式（PollingOnly），但展现了在无外部依赖下对接硬件的能力，并为后续优化保留了控制点。
3. **物理帧引用计数与共享内存生命周期**：在 IPC 模块中，共享内存的 FrameTracker 通过 .clone() 实现引用传递[ipc.ref:5]，使得在 IPC_RMID 后附着页表仍能有效访问，这一设计在不引入垃圾回收的前提下正确处理了共享资源的生命周期，体现了 Rust 所有权模型的妙用。

### 取舍判断
该内核显然将绝大部分精力投入到系统调用的语义实现和线程/信号机制上，而在以下方面做了明显妥协：
- **并发模型**：全局共享锁（USER_PROCESSES、SHM_STATE、文件系统锁[fs.ref:4]）贯穿始终，调度器每次状态变更都要全量重建索引[task.ref:1]，完全放弃多核伸缩性。
- **网络栈**：实现了基于 SocketTable 的进程内 IPC 以通过套接字测试，但忽略了真正的协议栈集成[net.ref:3]，形成“两个世界”的割裂。
- **错误处理**：多处使用 .unwrap() 和 .expect()，在驱动初始化失败或资源耗尽时选择 panic 而非优雅降级[drivers 不足][fs 模块不足]，降低了系统鲁棒性。
- **安全边界**：chroot 等安全特性仅做表面校验，实质未提供隔离；robust futex 链表截断可能留下死锁[task.ref:4]。
这些取舍清晰地反映出“跑通测试高于一切”的开发导向，对于教学演示或许可以接受，但严重限制了其向生产环境演进的可能性。

### 完成度与不足
从 syscall 覆盖度看，176 项中涵盖了进程控制、文件操作、高级信号、futex、socket 等，表面成绩显著，但深入使用即发现皱纹：
- 内存管理：惰性分配无超额保护，可能耗尽内存后直接 OOM；TLB 刷新单核无广播，多核下隐患[mm.ref:2]。
- 信号：单槽待决信号导致并发信号丢失[signal.ref:3]；sigtimedwait 几乎未实现。
- 网络：loopback 绕过协议栈，TCP 连接建立仅为端口配对[net.ref:5]；Unix 域套接字 connect 始终返回错误。
- 驱动：sync() 为空，无 flush 命令，掉电可能数据丢失。
这些不足表明，内核的“80% 功能”多处于能用但远未正确或完整的阶段。代码质量方面，大量硬编码常量、未标注 safety 的 unsafe 块、测试中隐含的 musl 版本绑定，使得维护和移植困难。因此，该内核适合作为观察 Rust 语言在内核开发中可能的语法和模式探索，但若要成为可靠系统，仍需大量重构与完善。

### 1. 核心抽象: global_asm! 作为架构入口的载体

本模块的核心抽象是将平台启动汇编代码直接嵌入 Rust 源文件, 通过 `core::arch::global_asm!` 宏实现 <sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>. 这样做的直接好处是: 不需要在构建系统中额外声明 `.S` 文件, 也不需要维护汇编器与 Rust 编译器之间的符号可见性约定. `_start` 被放置在 `.text.entry` 段中, 由链接脚本确保其位于最终二进制的最前端, 满足固件/bootloader 的入口跳转约定.

RISC-V 与 LoongArch 的入口实现均遵循相同模式: (1) 加载启动栈顶地址到栈指针; (2) 调用外部符号 `rust_entry`; (3) `rust_entry` 返回后进入死循环 <sup>[3](#mod-boot-ref-3)</sup><sup>[4](#mod-boot-ref-4)</sup>. 但两版存在微妙差异: LoongArch 版在调用前显式将 `$a0` 与 `$a1` 清零, 而 RISC-V 版直接保留了 SBI 传入的 `a0` (hart ID) 和 `a1` (DTB 指针) <sup>[5](#mod-boot-ref-5)</sup>. 这暗示 `rust_entry` 的签名可能不接受参数 —— 两版都丢弃了 bootloader 传入的信息, 但方式不同: RISC-V 版是

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 MemorySet / MapArea / AddressSpace 为核心抽象，通过 ArchPageTable 对接 RISC‑V SV39 页表硬件，支持惰性分配、写时复制、文件后盾映射及线程栈预留等策略。与标准 rCore 实现相比，引入了 SharedAnonymous、ThreadStackReservation 等细粒度 AreaKind，并实现了文件回写页失效 (invalidate_file_backed_pages) 等机制，但 mmap 地址空间范围受限，TLB 刷新未考虑多核一致性，mprotect 共享页处理存在隐患。

### 1. 核心抽象：MemorySet、MapArea 与地址空间隔离
内存管理围绕 MemorySet 展开，它表示一个完整地址空间，内部维护按起始地址排序的 MapArea 向量<sup>[3](#mod-mm-ref-3)</sup>。每个 MapArea 记录一段虚拟地址区间、权限（MapPerm）、映射类型（MapType::Identical 或 Framed）以及区域种类（AreaKind）。AreaKind 区分匿名内存、文件映射、线程栈等，并通过 is_mmap_area 方法实现统一管理<sup>[8](#mod-mm-ref-8)</sup>。AddressSpace 在 MemorySet 外封装 SpinLock，为并发访问提供线程安全<sup>[4](#mod-mm-ref-4)</sup>。
相比 rCore‑Tutorial，本仓库引入了 ThreadStackReservation、NoAccessAnonymous 等种类，以支持多线程栈预留和 mmap 的 PROT_NONE 语义。这种设计使 MemorySet 能够映射复杂的用户态内存布局，但也增加了区域合并与分割时的状态复杂度。

### 2. 页表抽象与硬件协同：SV39 实现
ArchPageTable 封装 RISC‑V SV39 页表操作，包括 map、unmap、protect、translate 等<sup>[2](#mod-mm-ref-2)</sup>。它支持 L1 叶（2 MiB 大页）映射，加速内核直接映射区，条件为 vpn == ppn、2 MiB 对齐且非用户态（!perm.contains(MapPerm::U)）<sup>[7](#mod-mm-ref-7)</sup>。这一限制使得用户态无法使用大页，与主流 OS 一致：用户态大页需要额外 COW 处理，而当前分配器粒度不足以支撑。
页表操作通过直接物理地址访问（page_from_ppn / page_from_ppn_mut）实现，依赖内核恒等映射。所有 unsafe 块均标注 SAFETY 理由，但缺乏全局 unsafe 审计注释，仅凭本地说明难以完全保证安全性。TLB 刷新使用 sfence.vma，但仅刷新当前核<sup>[2](#mod-mm-ref-2)</sup>。若多核共享地址空间（如多线程），缺少 TLB shootdown 会导致其他核使用过时映射，这在单核或独占地址空间下安全，但为 SMP 场景埋下风险<sup>[6](#mod-mm-ref-6)</sup>。

### 3. 惰性映射与写入时复制
MapArea 的 lazy 标志允许创建区域时不分配物理页<sup>[3](#mod-mm-ref-3)</sup>。缺页时，trap handler 调用 AddressSpace::handle_lazy_page_fault，由 MemorySet 完成分配和映射<sup>[4](#mod-mm-ref-4)</sup>。这减少了进程创建开销，并支持超额申请。
COW 通过 split_shared_frame 实现（位于 area.rs 的 cow 子模块）。当 protect_range 添加写权限或写时拷贝触发时，ensure_private_before_write_protect 会检查 Framed 区域并执行私有化分裂<sup>[6](#mod-mm-ref-6)</sup>。然而，分裂逻辑未在本文片段完全展示；若处理不当，可能导致 FrameTracker 引用计数混乱。
文件后盾失效 invalidate_file_backed_pages 遍历虚拟页，卸载已映射的页表项并移除帧记录<sup>[9](#mod-mm-ref-9)</sup>。但它未处理脏页写回，且只操作当前 MemorySet，若同一文件被多区域或跨进程映射，其他映射可能残留过时内容，存在数据丢失风险。

### 4. 并发控制与 AddressSpace 封装
AddressSpace 使用 SpinLock<MemorySet> 保护所有操作，简化了并发模型。但缺页处理也需要获取锁，多线程争抢时会成为瓶颈。替代方案如 RwLock 或 per‑area 锁可提升并发度，但当前设计以性能换取了正确性。
线程栈镜像操作 mirror_thread_stack_range_from 先锁源空间再锁目标空间，并通过 core::ptr::eq 避免自镜像。这种跨地址空间锁定若使用不当可能引起死锁，但实际场景中路径一致，风险较低。

### 5. 边角细节与不足
<sup>[1](#mod-mm-ref-1)</sup> 用户 mmap 地址范围硬编码为 0x2000_0000–0x3e00_0000（480 MiB），远小于 64 位空间，大型应用可能提前失败。
<sup>[2](#mod-mm-ref-2)</sup> 大页映射要求 vpn == ppn，意味着内核映射必须是恒等映射。若有后期非恒等动态映射试图使用大页，会静默退化为 4 K 页，可能造成性能意外退化。
<sup>[3](#mod-mm-ref-3)</sup> 惰性分配未做超额保护，缺页时 alloc_frame 失败直接导致 OutOfMemory，进程被信号杀死，缺少 OOM killer 或回收机制。
<sup>[5](#mod-mm-ref-5)</sup> 内核堆 128 MiB，bump allocator + free list 可能产生外部碎片；dealloc 对无效指针静默返回，掩盖内存错误。
<sup>[6](#mod-mm-ref-6)</sup> protect_range 在 no_access 时卸载页表项并调用 remove_frame，未检查该帧是否被其他区域引用，若为共享映射则留下悬垂。
<sup>[9](#mod-mm-ref-9)</sup> 文件后盾失效只 unmap 本进程页表，未写回脏页，若文件曾被写入，数据将丢失。

- TLB 刷新仅使用 sfence.vma 刷新当前核，多核共享地址空间时其他核可能使用过时映射，存在潜在数据不一致。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以 `UserProcessTable` 为核心抽象，通过 `UserTaskState` 状态机管理所有用户线程的运行、等待、休眠。采用全局 `USER_PROCESSES` 互斥锁保护内部任务数组与各类等待队列（futex、管道、套接字、子进程），以统一的截止时间定时器驱动超时唤醒。相较于典型 rCore 教学实现，本仓增加了 robust futex 清理、信号中断睡眠剩余时间报告等接近 Linux 语义的细节，但也因此引入了频繁的索引重建与较粗粒度的锁。

### 1. 任务状态机与阻塞原语
用户可见的每个线程对应 `UserProcessTable` 中的一个 `UserTask`，其状态由 `UserTaskState` 枚举控制（Running／Waiting／Sleeping／Zombie 等）。系统调用通过 `current_index()` 获取当前任务在 `tasks` 数组中的位置，并在修改状态后立即调用 `rebuild_indexes()` 刷新 pid→index 的查找结构 <sup>[1](#mod-task-ref-1)</sup>。一段典型的阻塞路径是：调用 `sleep_current_until(deadline)` 将当前任务置为 Sleeping，同时向全局 `timers` 注册截止时间，最后系统调用返回 `SYSCALL_BLOCKED` 令调度器重新选择任务 <sup>[7](#mod-task-ref-7)</sup>。待超时或信号到达时，`wake_sleepers` 遍历到期定时器，将对应的任务改回 Ready，并取消其在 futex／管道／套接字上的等待 <sup>[2](#mod-task-ref-2)</sup>。这种“状态机 + 定时器 + 外挂队列”的组合避免了每个等待类型实现独立的超时逻辑，但也导致每次状态变更都必须重算索引，构成明显的性能代价。

### 2. 统一超时与多类型等待队列的协同
`UserProcessTable` 内部维护了多种等待目标：futex（`futex_table`）、管道读写（`pipe_read_waits`／`pipe_write_waits`）、套接字读写以及子进程（`child_wait`）。所有等待在概念上都支持可选的截止时间——当为 futex 或管道调用 `wait_current_for_futex_until` 或 `wait_current_for_pipe` 时，同时向 `timers` 注册一个超时事件 <sup>[5](#mod-task-ref-5)</sup>。超时触发时，`wake_sleepers` 一次性处理所有类型的等待：先调用 `cancel_futex_waiter`、`cancel_pipe_waiter` 和 `cancel_socket_waiter` 将其从对应队列中删除，再把任务置为 Ready 并推入 `WakeEvent` 以通知调用方超时发生 <sup>[2](#mod-task-ref-2)</sup>。这种“中心化清理”的设计保证了即便一个任务在多个队列上等待（例如同时等待 futex 和管道），超时也能彻底解除所有关联，避免了悬挂的等待项。代价是每次超时扫描需要访问多个独立性队列，且必须连续持有全局锁，在多核下可能成为瓶颈。

### 3. 信号驱动的等待中断与剩余时间语义
`nanosleep` 等系统调用要求在被信号中断时报告剩余休眠时间。本模块通过 `interrupt_wait_for_signal` 实现这一语义：当信号到达时，先调用 `timers.cancel` 以及各队列取消函数，然后检查任务是否处于 Sleeping 状态且 `sleep_reports_remaining` 为 true；若是，则计算 `deadline.saturating_sub(now).max(1)` 存入 `interrupted_sleep_remaining_micros`，否则清零 <sup>[3](#mod-task-ref-3)</sup>。随后将任务状态强制切为 Ready 并生成 `SignalPending` 唤醒事件。`sys_nanosleep` 在重新获得 cpu 后会读取该剩余时间，通过 `rem_addr` 写回用户空间并返回 `EINTR`。整个流程中 `sleep_reports_remaining` 标志由 `sleep_current_until_reporting_remaining` 设置，确保只有显式请求报告剩余时间的调用者（如 `nanosleep`）才需要维护这一额外字段。这种按需记录的思路避免了为所有等待类型都保存剩余时间，但引入了若干微妙组合：当信号和超时几乎同时发生时，`take_wake_reason` 的调用顺序可能决定最终返回码，需要配合信号掩码测试才能完全验证正确性。

### 4. 边角细节与不足
- **<sup>[4](#mod-task-ref-4)</sup> robust futex 硬编码限制**：`mark_robust_futexes_for_exit` 中 `ROBUST_LIST_LIMIT` 固定为 2048，若用户态构造超长链则仅处理前 2048 个节点，可能导致部分 robust 锁未被标记为 `OWNER_DIED`，违反 POSIX 健壮性语义。
- **<sup>[8](#mod-task-ref-8)</sup><sup>[10](#mod-task-ref-10)</sup> 时间子系统的简化实现**：`CLOCK_PROCESS_CPUTIME_ID`／`CLOCK_THREAD_CPUTIME_ID` 直接返回 `EOPNOTSUPP`；`adjtimex` 只实现了查询路径，写入模式虽校验参数但并未实际调整时钟，依赖这些调用的程序（如 NTP）会获得错误的反馈。
- **<sup>[11](#mod-task-ref-11)</sup> 单一 CPU 的亲和性假设**：`sys_getcpu` 硬编码 cpu=0，不反映任何真实硬件拓扑，且 `sched_setaffinity` 等调用未实现实际迁移，只能通过返回成功欺骗用户态。
- **<sup>[9](#mod-task-ref-9)</sup><sup>[12](#mod-task-ref-12)</sup> 时钟偏移与系统信息的不精确**：`realtime_micros` 用 `AtomicI64` 保存相对于 `uptime` 的偏移，但 `settimeofday` 不存在全局单调时钟调整，多核下可能出现短暂的不一致；`sys_sysinfo` 仅填充了 `totalram`／`freeram` 等基本字段，共享内存、缓冲区等字段保持 0，可能导致 `top` 等工具显示异常。
- **<sup>[1](#mod-task-ref-1)</sup> 频繁的 `rebuild_indexes`**：几乎所有状态变更都会触发该函数，它需要重建 `tasks` 中 pid→index 的映射，若无增量更新或缓存，在任务数量较多时系统调用延迟将线性增长。

以上多数问题源于“最小兼容”策略——优先通过 LTP 测试而非实现完整语义——在原型阶段可以接受，但若要成为生产级内核则需要逐步替换为准确实现。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块实现了三层 VFS 架构 (syscall 翻译 → VFS API → RamFS 后端) 与独立的 loopback-only socket 子系统。核心抽象为 RamFS 内存文件系统 (Vec<RamNode> 全节点存储) 和固定槽位 SocketTable (64 槽)。解决了用户态 Linux syscall 到内核文件/网络操作的完整映射, 包含凭证校验、umask、mount readonly 保护。与同家族基线 (rCore-Tutorial) 相比, 自研了 RamFS 替代简单的文件列表, 并加入 TCP/UDP loopback 栈和完整的 *at 系列 syscall 支持。

### 1. 核心抽象: 三层 VFS + 独立 Socket 子系统

文件系统模块采用分层架构, 自上而下为:

- **Syscall 翻译层** (`syscall/fs/path.rs`): 将 Linux syscall 语义 (openat/mkdirat/unlinkat/fchownat 等 30+ 系统调用) 转换为 VFS 调用。每条 syscall 承担路径读取 (`read_user_cstring`)、dirfd 解析 (`dir_base`)、凭证提取 (`euid/egid`)、标志位验证、umask 屏蔽等职责 <sup>[6](#mod-fs-ref-6)</sup>。
- **VFS API 层** (`fs/vfs/api.rs`): 提供路径正规化、目录项查找、权限检查的统一接口。通过 `path_lookup()` 和 `ramfs_write_path()` 两个入口对接后端, 暴露 `resolve`/`open`/`mkdir`/`unlink`/`rename` 等 POSIX 语义操作。权限模型区分搜索权限 (目录遍历) 与访问权限 (最终对象), 并接入 mount readonly 检查 <sup>[10](#mod-fs-ref-10)</sup>。
- **RamFS 后端** (`fs/ramfs/store.rs`): 纯内存文件系统, 以 `Vec<RamNode>` 存储所有节点, 支持目录、普通文件、符号链接、硬链接、设备节点、FIFO。每个 `RamNode` 包含路径、父节点索引、文件名、链接别名列表、文件类型、权限模式、uid/gid、时间戳和可变数据 (`RamNodeData`)<sup>[4](#mod-fs-ref-4)</sup>。

Socket 子系统独立于文件系统树, 通过全局 `SocketTable` (固定 `MAX_SOCKETS=64` 槽位) 管理所有套接字状态 <sup>[1](#mod-fs-ref-1)</sup>。用户态通过 `SocketEndpoint` 句柄操作套接字, `Drop` 实现自动 close (含对端唤醒)<sup>[2](#mod-fs-ref-2)</sup>。

### 2. 关键设计取舍: 全局锁、线性搜索与协议栈简化

**全局粗粒度锁** 是整个模块最显著的取舍。RamFS 通过 `static RAMFS: SpinLock<RamFs>` 保护全部节点状态, Socket 子系统通过 `static SOCKET_TABLE: SpinLock<SocketTable>` 保护所有套接字 <sup>[4](#mod-fs-ref-4)</sup><sup>[2](#mod-fs-ref-2)</sup>。这消除了锁排序和死锁风险, 代价是所有文件/socket 操作完全串行化——在多核场景下构成瓶颈。替代方案 (per-node RWLock、RCU) 的复杂度被刻意规避, 符合 "最小可行内核" 的定位。

**路径解析采用 O(n) 线性扫描**: `lookup_path_location` 遍历 `nodes` 向量逐条比对路径字符串, 同时对硬链接别名做二次扫描 <sup>[5](#mod-fs-ref-5)</sup>。无任何索引加速 (哈希表/B 树), 也无法利用目录层级剪枝。在节点数可控的嵌入式场景可接受, 但频繁操作下退化明显。

**Socket 子系统被限制为纯 loopback**: `bind()` 只接受 `addr == 0` 或 `addr == LOOPBACK_ADDR` (127.0.0.1), `connect()` 同样拒绝非本地地址 <sup>[3](#mod-fs-ref-3)</sup>。这裁掉了完整 IP 协议栈——无 ARP/路由/网卡驱动, TCP 状态机简化为内存队列操作。对于单机 IPC, 这提供了比 Unix domain socket 更标准的 API 外观; 跨机通信完全不可用。

**固定槽位硬约束**: `MAX_SOCKETS=64` <sup>[1](#mod-fs-ref-1)</sup>, 超限返回 `EMFILE`。端口分配通过线性探测 ephemeral 范围 (49152-60999), 耗尽后返回 `EADDRINUSE` <sup>[9](#mod-fs-ref-9)</sup>。Datagram 队列受 `MAX_DGRAM_QUEUE` (16K 条) 和 `MAX_DGRAM_BYTES` (64KB) 双层限制, 防止内存膨胀。

### 3. 跨模块协同: 凭证传播与只读挂载保护

`sys_openat` 展示了完整的跨层协同链路 <sup>[6](#mod-fs-ref-6)</sup>: (1) 用户态路径读取 → (2) dirfd 解析 → (3) 凭证提取 → (4) 根据 `O_CREAT`/`O_EXCL`/`O_NOFOLLOW`/`O_TMPFILE`/`O_DIRECTORY` 等标志分流到不同 VFS 路径 (如 `vfs::create_file_owned_with_credentials` 或 `vfs::open_with_search_creds`) → (5) 新建文件时应用 umask 与 setgid 清除规则 → (6) 检查 mount readonly → (7) 将 `OpenFile` 插入 fd table 并同步到线程组。

凭证 (`euid`, `egid`) 从进程上下文中提取后贯穿整个调用链: VFS API → `path_lookup().lookup_object_with_search_creds()` → RamFS 元数据创建 (`insert_node` 记录 owner uid/gid)。权限检查分两阶段: 目录遍历时的搜索权限 (execute on directory) 和最终对象访问权限 (read/write/execute), 由 `access_path` 统一执行 <sup>[10](#mod-fs-ref-10)</sup>。

Mount readonly 保护以字符串前缀匹配实现: 任何修改操作 (chmod/chown/unlink/rename 等) 在 VFS 层调用 `kernel_mount::is_user_mount_readonly_path` 拦截, 返回 `EROFS`。此机制比传统内核 mount flag 检查简单但同样有效, 且与 chmod/chown 的凭证校验独立运作。

Socket 的 `close()` 通过 `close_event()` 构造 `SocketCloseEvent`, 携带对端 `wake_reader`/`wake_writer` ID, 供上层 epoll/poll 唤醒阻塞的读写者。TCP 流关闭时对端被标记 `recv_shutdown = true`, 这是半关闭语义的内核侧实现。

### 4. 边角细节与不足

- **<sup>[7](#mod-fs-ref-7)</sup> `chroot` 是空壳**: `sys_chroot` 仅校验目标为目录, 注释写明 "最小兼容不实现完整 jail 语义"。依赖 chroot 隔离的应用将获得虚假安全感。
- **<sup>[8](#mod-fs-ref-8)</sup> `flock` 同样是空壳**: 只验证 fd 与操作码, 不实际加锁——因为 "当前单进程语义"。多线程并发文件写入完全无保护。
- **`decode_utimens_timespec` 中的隐式 `unwrap()`**: `raw[0..8].try_into().unwrap()` 依赖调用方 `read_utimens_times` 保证 32 字节缓冲区长度。若未来调用路径传入不足 16 字节的切片则 panic, 缺乏编译期长度检查或 `// SAFETY` 注释。
- **端口分配可能饥饿**: `allocate_ephemeral_port` 线性递增, 10108 个端口耗尽后遍历一圈返回 `EADDRINUSE` <sup>[9](#mod-fs-ref-9)</sup>, 无随机化起始点, 也无重试/超时机制。
- **RamFS 节点回收依赖 open_count**: 被删节点变为 tombstone (`removed=true`), 仅当 `open_count==0` 时被替换 [ref:4 附近]。若进程长期持有 fd, 对应内存永不释放——POSIX 正确语义但内存受限环境下是隐患。
- **无 dentry cache**: 每次 `lookup_path_location` 从头扫描全表, 频繁路径解析产生 O(n²) 效应 <sup>[5](#mod-fs-ref-5)</sup>。
- **硬链接 promotion 可能遗漏父目录更新**: `remove_primary_dentry` 将第一个硬链接提升为主路径, 但未更新原父目录的子节点关联, 可能导致 `lookup_child_location` 查找失败。
- **`with_two_sockets_mut` 静默忽略相同索引**: 通过 `split_at_mut` 获取两个可变引用时, `first == second` 则直接返回不操作 [ref:2 附近], 若调用方错误传入相同索引, 无任何 panic 或 log, 可能掩盖 bug。

### ⚠ 开放问题

- chroot(2) 仅校验路径为目录, 不实现 jail 语义 (syscall/fs/path.rs:636), 依赖 chroot 隔离的应用会误判安全。

- RamFS remove_primary_dentry 提升硬链接为主路径时未更新父目录子节点关联, 可能导致 lookup_child_location 查找失败。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块基于单槽 pending_sync_signal + u64 掩码 + SignalActions 动作表实现 POSIX 信号子系统, 核心创新在于引入 SIGCANCEL(33) 作为线程取消的合成信号并与 musl libc 的 pthread 取消状态深度耦合. 信号投递采用“同步检查”模型 (trap 返回前轮询 pending_sync_signal), 不同于 Linux 的异步队列; 这简化了内核但限制了并发信号保序能力. 与 ArceOS 基线相比, 本仓新增了完整的 rt_sigreturn/sigsuspend/sigtimedwait 及线程组信号动作传播.

### 1. 信号掩码与动作表: 核心数据抽象
信号子系统的状态载体是三个字段: `signal_mask`(u64 位图), `signal_actions`(SignalActions), `pending_sync_signal`(Option<usize>). 掩码支持 1-64 号信号, 每信号占一个 bit<sup>[4](#mod-signal-ref-4)</sup>. 动作表用 `Arc<SpinLock<[KernelSigAction; 65]>>` 封装, 提供 `fork_copy`(深拷贝) 和 `shared_clone`(Arc clone) 两种语义以适配 fork 与 pthread_create 的不同共享需求<sup>[2](#mod-signal-ref-2)</sup>. KernelSigAction 的四元组 (handler/flags/restorer/mask) 在架构间分化: 当 `crate::arch::user_sigaction_has_restorer()` 为 false 时用户态 sigaction 只有 24 字节 (无 restorer 字段), 否则为 32 字节<sup>[5](#mod-signal-ref-5)</sup>; 这个编译期分支避免了运行时 ABI 检测开销.

### 2. “同步单槽”投递模型: 取舍与代价
信号投递全程在 `USER_PROCESSES` 锁内完成. `send_signal_to_task_index` 的决策树为: SIGKILL 立即终止; SIGPIPE/SIGTERM 若 handler 为默认且未屏蔽则终止, 若被屏蔽则写入 `pending_sync_signal`; 其余信号若被屏蔽也写 `pending_sync_signal`, 否则调用 `deliver_signal_handler_to_task` 立即构造栈帧<sup>[1](#mod-signal-ref-1)</sup>. 关键设计取舍在于 **只有单个待决槽**: `pending_sync_signal` 是 `Option<usize>`, 后到的信号会覆盖先到的<sup>[3](#mod-signal-ref-3)</sup>. Linux 使用每线程 pending 队列 + 共享 pending 队列的二级模型保证并发信号不丢; 本实现放弃保序换取简洁性, 依赖“用户态信号处理器返回前内核不会再次投递同类型信号”的 POSIX 语义作为部分补偿.

投递本身通过 `build_sigcancel_frame` 在用户栈上构造 siginfo + ucontext 帧, 将原 trap frame 保存到 `signal_restore_frame`, 修改当前 trap frame 的 PC/SP/参数寄存器使其跳转到 handler<sup>[6](#mod-signal-ref-6)</sup>. 返回地址设为 restorer (优先用户态提供的, 否则用架构默认值), handler 返回时通过调用 restorer 进入 `sys_rt_sigreturn` 恢复原上下文<sup>[7](#mod-signal-ref-7)</sup>.

### 3. SIGCANCEL 与 musl 线程取消的深度耦合
信号 33 (SIGCANCEL) 是本模块最具特色的设计. 它用于 musl libc 的 `pthread_cancel`, 投递逻辑在 `deliver_signal_handler_to_task` 中特殊处理: 若目标线程的 `pthread_base` 指向的 TLS 中 cancel state 为 `MUSL_PTHREAD_CANCEL_DISABLE`(值为 1), 则信号被推迟 — 重新写回 `pending_sync_signal` 而非立即投递<sup>[8](#mod-signal-ref-8)</sup>. 这个 check 通过 `read_user_u8` 直接读取用户态内存 `pthread_base + MUSL_PTHREAD_CANCEL_STATE_OFFSET` 完成<sup>[9](#mod-signal-ref-9)</sup>. 此外 `should_interrupt_wait_for_signal` 对 SIGCANCEL 有额外判断: 如果线程在 Waiting/Sleeping 状态且 cancel 被禁用, 则不中断等待 — 让线程自然醒来后再处理取消请求. 这种内核直读 musl 内部状态的做法避免了新增 syscall, 但也将内核与特定 libc 版本锁定.

### 4. 跨模块协同
信号模块通过四个渠道与内核其它部分交互: (a) `crate::arch::TrapFrameAccess` trait 提供 `user_pc()/user_sp()/set_entry()` 等 trap frame 操作, 使投递代码与具体架构解耦; (b) `crate::arch::signal_ucontext_pc_offset()` 提供 ucontext 中 PC 字段偏移, 支撑 `rt_sigreturn` 的跨平台兼容; (c) 进程表 `USER_PROCESSES` 锁贯穿所有 syscall, `sys_rt_sigaction` 额外遍历同 tgid 任务传播动作设置<sup>[5](#mod-signal-ref-5)</sup>; (d) 唤醒机制: `sys_rt_sigsuspend` 将任务置为 Waiting 并检查 `WakeReason::SignalPending`, 与 `wait_queue` 模块配合实现阻塞-唤醒循环.

### 5. 边角细节与不足
- **<sup>[3](#mod-signal-ref-3)</sup> 单槽待决信号丢失**: `pending_sync_signal` 只能存一个信号, 并发场景下后到的信号直接覆盖前者. 对实时信号应用是不可接受的.
- **<sup>[10](#mod-signal-ref-10)</sup> `sys_rt_sigtimedwait` 仅实现 SIGCHLD**: 其他信号一律返回 EAGAIN, 等同于未实现. 且超时参数仅识别 `{0,0}`(零超时), 非零超时直接阻塞 — 没有定时器唤醒.
- **<sup>[11](#mod-signal-ref-11)</sup> 硬编码 musl TLS 地址**: `trace.rs` 中 `MUSL_TL_WAITERS = 0x1200_a2548` 和 `MUSL_THREAD_LIST_LOCK = 0x1200_a27c8` 是编译 musl 时固定的链接地址, 换 musl 版本或编译选项即失效.
- **unwrap 恐慌点**: `read_sigaction` 内 `raw[0..8].try_into().unwrap()` 和 `read_optional_timeout` 内 `raw[0..8].try_into().unwrap()` — 虽然数组长度由常量保证, 但若未来修改常量未同步更新将导致内核 panic, 属于防御性编程不足.
- **持锁分配**: `sys_kill` 处理负 pid(进程组广播) 时在持有 `USER_PROCESSES` 锁期间 `collect::<Vec<_>>()` 分配堆内存, 若分配器内部需要锁可能死锁; 且锁临界区遍历全表 O(n).
- **无 SA_RESTART 支持**: `KernelSigAction.flags` 字段存在但未被任何代码检查 — 被信号中断的 syscall 统一返回 EINTR, 不区分是否应自动重启.

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现三套 IPC 机制: System V 共享内存(shmget/shmat/shmdt/shmctl)、futex(含 WAIT/WAKE/REQUEUE/BITSET 及 membarrier)、以及匿名管道(PipeEndpoint). 共享内存通过 FrameTracker 克隆语义使物理帧在 IPC_RMID 后仍被已映射页表持有; futex 与进程表 USER_PROCESSES 深度耦合, 通过 64 桶哈希 FutexTable 管理等待队列; 管道基于 Arc<SpinLock<>> 共享环形缓冲区, 按读写端 Drop 时递减引用计数以检测 EOF/EPIPE.

### 1. 三种 IPC 机制的分工与抽象层次

该内核同时提供三套进程间通信原语, 各自面向不同场景: System V 共享内存用于大块数据的零拷贝共享, futex 支撑用户态 pthread 互斥锁/条件变量等同步原语, 匿名管道则覆盖流式生产者-消费者模式. 三者在实现上几乎无共享代码 — 共享内存操作 `SHM_STATE` 全局锁, futex 操作 `USER_PROCESSES` 进程表锁, 管道则通过 `Arc<SpinLock<PipeState>>` 在读写端之间传递所有权. 这种隔离避免了 IPC 机制间的相互干扰, 但也意味着各模块重复实现了资源配额、ID 分配等基础设施 <sup>[1](#mod-ipc-ref-1)</sup><sup>[2](#mod-ipc-ref-2)</sup><sup>[3](#mod-ipc-ref-3)</sup>.

### 2. 共享内存: 物理帧生命周期与全局锁

System V 实现的核心抽象是 `ShmState`, 它在一个 `SpinLock` 后维护 `segments: Vec<ShmSegment>` 和 `attachments: Vec<ShmAttachment>`. 每个 `ShmSegment` 拥有一组 `FrameTracker`, 代表该段申请的全部物理页面 <sup>[4](#mod-ipc-ref-4)</sup>. 关键设计在于 `segment_snapshot` 方法: 它通过 `.iter().cloned()` 深拷贝 `FrameTracker` 向量, 交予 `shmat` 进行页表映射 <sup>[5](#mod-ipc-ref-5)</sup>. 由于 `FrameTracker` 疑似内部引用计数 (从 `Clone` 语义推断), 克隆操作会递增物理帧引用, 使得即便后续 `IPC_RMID` 移除段并 drop 原始 `FrameTracker`, 已建立的页表映射仍然持有有效物理页 — 自测用例 `sysv_shm_private_segment_survives_immediate_rmid` 明确验证了这一行为 <sup>[6](#mod-ipc-ref-6)</sup>.

取舍方面, 全局 `SHM_STATE` 锁在 `shmat` 期间跨越了页表操作 (`map_shared_frames`), 这简化了并发模型但将潜在的页表缺页处理延迟放大为锁持有时间. 此外 `shmctl` 仅实现 `IPC_RMID`, `IPC_STAT`/`IPC_SET`/`SHM_LOCK` 等均返回 `EINVAL`, 表明这是一个面向内部最小可用集的裁剪 <sup>[7](#mod-ipc-ref-7)</sup>.

### 3. Futex: 进程表深度集成与调度优化

Futex 是 Linux 用户态同步的基石系统调用. 本实现的 `sys_futex` 支持 `FUTEX_WAIT`/`FUTEX_WAKE`/`FUTEX_REQUEUE` 以及带 bitset 的变体, 外加 `membarrier` 三个命令 <sup>[8](#mod-ipc-ref-8)</sup>. 其核心流程: `sys_futex_wait` 先获取 `USER_PROCESSES` 大锁, 检查信号/超时/虚假唤醒标志, 再用 `UserBuffer` 读取用户态 futex 字并与预期值比较, 最后通过 `table.wait_current_for_futex` 将当前任务挂入 `FutexTable` <sup>[9](#mod-ipc-ref-9)</sup>.

`FutexTable` 是一个 64 桶的开放哈希表, 键为 `FutexKey { address_space_id, aligned_uaddr, shared }`. 哈希函数混合了地址空间 ID 与共享标志 `shared` 以降低跨进程碰撞 <sup>[10](#mod-ipc-ref-10)</sup>. 每个桶是一个 `Vec<FutexWaitQueue>`, 而 `FutexWaitQueue` 封装了 `WaitQueue` — 来自 `task::wait_queue` 的通用等待队列. 这种分层使 futex 等待可复用统一的调度基础设施.

一个值得注意的优化是 `should_schedule_after_futex`: 当 `FUTEX_WAKE` 返回值 >0 且另有可运行任务时, 本应触发重调度; 但若被唤醒的 futex 地址恰好是当前线程的 pthread TID 或 detach 状态字 (通过 `task.pthread_base + MUSL_PTHREAD_TID_OFFSET` 判断), 则跳过调度 — 因为当前线程刚刚释放锁, 继续运行可以立即获取下一把锁, 避免不必要的上下文切换 <sup>[11](#mod-ipc-ref-11)</sup>. 这直接依赖 `USER_PROCESSES` 中存储的 `pthread_base` 元数据, 是 musl libc 深度适配的体现.

### 4. 管道: Arc 共享的环形缓冲区与非阻塞语义

`PipeEndpoint` 以 `Arc<SpinLock<PipeState>>` 共享一个 64KB 环形缓冲区. 读写端各自持有 `PipeEndpointMode::Read` 或 `Write`, 并在 `Drop` 时递减对应的 `readers`/`writers` 计数 <sup>[12](#mod-ipc-ref-12)</sup>. 这种设计使得 EOF 和 EPIPE 检测完全由引用计数驱动 — 当所有写端关闭时 `read` 返回 `Eof`, 当所有读端关闭时 `write` 返回 `EPIPE`.

管道的读写均为非阻塞: `read` 在缓冲区空但仍有写端时返回 `WouldBlock`, `write` 在缓冲区满时返回 `Ok(0)`. 这与传统 UNIX 阻塞管道语义不同, 更适合配合 `epoll`/`poll` 使用, 但当前代码中 `PipeReady` 结构体虽已定义却未见被任何事件通知机制消费 <sup>[13](#mod-ipc-ref-13)</sup>. 此外读写操作通过 `copy_from_slice` 直接拷贝用户数据到内核缓冲区, 未使用 `UserBuffer` 等用户态安全访问封装, 入参 `input: &[u8]` 表明拷贝已由上层调用者完成.

### 5. 边角细节与不足

1. **硬编码 musl 地址** <sup>[14](#mod-ipc-ref-14)</sup>: `trace_musl_thread_lock_state` 直接读取固定虚拟地址 `0x1200_a2548` 和 `0x1200_a27c8`. 这些是特定 musl 链接脚本下的线程局部存储偏移, 换用其他 libc 或 musl 编译选项即会读到无效数据甚至触发页错误. 生产环境中该函数由 `KERNEL_TRACE_SYSCALL` 环境变量在编译期关闭, 但地址本身没有任何 guard.

2. **Futex 检查-执行竞态** <sup>[9](#mod-ipc-ref-9)</sup>: `sys_futex_wait` 在读取用户态 futex 字之后、挂入等待队列之前, 存在时间窗口: 若另一线程恰在此时修改 futex 值并发起 `FUTEX_WAKE`, 本次 `WAIT` 将丢失唤醒, 导致调用线程永久阻塞. Linux 通过原子比较并挂入等待队列来解决此问题; 本实现未见等价机制.

3. **全局锁持有 page table 操作** <sup>[15](#mod-ipc-ref-15)</sup>: `sys_shmat` 在持有 `SHM_STATE.lock()` 期间调用 `task.memory.map_shared_frames()`. 页表修改可能触发 TLB 刷新或其他锁获取, 若这些操作反向竞争 `SHM_STATE` 则形成死锁. 当前代码路径中未观察到反向依赖, 但缺乏文档保证.

4. **管道写满返回 0 而非阻塞** <sup>[16](#mod-ipc-ref-16)</sup>: `write` 在缓冲区满时返回 `Ok(0)`, 而 POSIX 期望对于阻塞文件描述符, `write` 应阻塞直到有空间. 调用者若未处理短写, 可能陷入忙等循环.

5. **SHM ID 回绕** <sup>[17](#mod-ipc-ref-17)</sup>: `create_segment` 中 `next_id` 通过 `checked_add(1).unwrap_or(1)` 在溢出时回绕到 1. 虽然 `MAX_SHM_SEGMENTS=64` 大幅降低了 ID 冲突概率, 但理论上长期运行后删除旧段再创建新段可能复用 ID, 若用户空间仍持有旧 ID 则可能误操作非预期段.

6. **FUTEX_REQUEUE 未校验源值** <sup>[18](#mod-ipc-ref-18)</sup>: `sys_futex_requeue` 仅检查 `uaddr2` 非零且不等于 `uaddr` 后直接调用 `table.requeue_futex_waiters`. Linux 的 `FUTEX_CMP_REQUEUE` 要求原子比较源 futex 值, 本实现虽仅支持 `FUTEX_REQUEUE` (非 CMP 变体), 但缺乏值校验意味着无法区分虚假唤醒与真实唤醒, 可能破坏条件变量的语义正确性.

7. **trace_la_shared_words_locked 未定义** <sup>[9](#mod-ipc-ref-9)</sup>: `sys_futex_wait` 和 `sys_futex_wake` 均调用了 `trace_la_shared_words_locked`, 该函数来自 `super::super::table`, 但未在提供的源码片段中出现. 若该函数 panic 或持有额外锁, 会在 `USER_PROCESSES` 已锁定的上下文中引入风险.

## 🌐 九、网络

> 💡 **TL;DR**
>
> 该内核的网络子系统并非传统协议栈，而是一个基于全局 SocketTable 的进程内 IPC 机制。核心抽象是 `SocketTable`（维护在 `kernel/src/fs/socket/` 中），支持 SOCK_STREAM 与 SOCK_DGRAM 语义，但仅限 127.0.0.1 回环。系统调用入口位于 `kernel/src/syscall/socket.rs`，通过 `UserBuffer` 与进程调度协同实现阻塞 I/O。仓库中另有一套基于 smoltcp 的 `LoopbackStack`（`kernel/src/net/`），但未被系统调用路径集成，属于孤立代码。与同家族 rCore-Tutorial 基线相比，该实现自建了绕过协议栈的直接内存传递路径，而非完全依赖 smoltcp。

### 1. 核心抽象：双栈并存下的 SocketTable IPC

该内核的网络子系统中存在两套独立实现：一套是 `kernel/src/fs/socket/` 下的 `SocketTable`，另一套是 `kernel/src/net/` 下基于 smoltcp 的 `LoopbackStack`。实际承担系统调用服务的只有前者。`SocketTable` 维护一个 `Vec` 表，每条目代表一个套接字端点，记录其域（仅 AF_INET 有效）、类型（SOCK_STREAM/SOCK_DGRAM）、绑定与连接状态、本地/对端地址、收发关闭标志以及数据包队列。套接字通过文件描述符暴露给用户态，`fd_table` 中存储 `vfs::OpenFileKind::Socket` 变体，`socket_id()` 辅助函数负责从 fd 提取套接字索引，非套接字 fd 返回 ENOTSOCK <sup>[1](#mod-net-ref-1)</sup><sup>[2](#mod-net-ref-2)</sup>。

`LoopbackStack` 则是一套完整的 smoltcp 集成：拥有 `Interface`、`LoopbackDevice`、`SocketSet`，可实现标准 TCP/UDP 语义。但代码库中没有任何系统调用路径引用 `crate::net` 或 `LoopbackStack`——该栈孤立于主数据通路之外，仅可能被 `selftest` 模块使用 <sup>[3](#mod-net-ref-3)</sup>。这一“双栈并存”的格局是理解该模块当前状态的钥匙。

### 2. 关键设计取舍：绕过协议栈的直接内存传递

`SocketTable` 的数据路径做出了一个激进取舍：完全跳过 TCP/IP 封装，直接在套接字之间拷贝数据。流套接字发送时，`send_stream` 首先校验源端已连接且未 shutdown，接着通过 `stream_peer_index` 查找目标——匹配条件是目标端口等于源对端端口、且目标对端地址等于源本地地址——然后直接调用目标的 `push_packet` 将数据写入其接收队列 <sup>[4](#mod-net-ref-4)</sup><sup>[5](#mod-net-ref-5)</sup>。数据包套接字类似，但目标匹配更灵活：优先精确匹配对端地址的已绑定套接字，回落到端口匹配但对端为空的套接字（未 connect 的 UDP 套接字）<sup>[6](#mod-net-ref-6)</sup>。

这一设计的优势是极低的延迟和实现简洁，代价则十分明显：（1）所有非 127.0.0.1 的地址均返回 `EADDRNOTAVAIL`，无法与外界通信 <sup>[7](#mod-net-ref-7)</sup>；（2）缺失 TCP 状态机、拥塞控制、分段重组等成熟机制，流控制仅靠缓冲区大小硬限制；（3）连接建立被简化为端口配对查找，缺少 SYN/ACK 语义，可能导致边缘行为与 POSIX 预期不一致。

阻塞 I/O 方面，系统调用层采用经典的“挂起-唤醒”模型：当操作遇 `EAGAIN` 且 fd 非 O_NONBLOCK 时，调用 `current.wait_for_socket_read(socket_id)` 挂起任务；发送路径在完成写入后调用 `current.wake_socket_readers(socket_id)` 唤醒对端阻塞的读者 <sup>[8](#mod-net-ref-8)</sup><sup>[9](#mod-net-ref-9)</sup>。唤醒点分散在 `sendto`、`connect`、`shutdown` 等多处——这是一种“点对点”的唤醒策略，而非统一的事件通知机制。

### 3. 跨模块协同：系统调用层、VFS 与进程调度

`kernel/src/syscall/socket.rs` 是整个网络子系统的入口与协调层，其职责包括：

- **用户态内存接入**：所有跨地址空间访问均通过 `UserBuffer` 封装，该类型在读写前调用 `validate_write`/`validate_read` 检查页面表合法性，将非法访问转换为 EFAULT <sup>[10](#mod-net-ref-10)</sup>。
- **sockaddr 序列化**：`read_sockaddr_in` 解析 16 字节 IPv4 地址结构，`write_sockaddr_in` 反之；`read_socklen`/`write_socklen` 处理地址长度字段。这些函数在 syscall 路径的多个位置复用，避免了重复代码 <sup>[11](#mod-net-ref-11)</sup>。
- **错误码映射**：socket 模块内部返回 `Result<T, isize>`，isize 直接对应 POSIX errno，在 syscall 层通过 `unwrap_or_else(err)` 统一抛出。

与 VFS/文件描述符的协同点集中在 `socket_id()`：它从 `fd_table` 获取 `OpenFile`，匹配其 `OpenFileKind::Socket` 变体，提取内部套接字 ID。这使套接字可以混入普通文件、管道的 fd 空间，且 `fcntl` 的 O_NONBLOCK/O_CLOEXEC 标志也可通过 fd 表作用于套接字 <sup>[2](#mod-net-ref-2)</sup>。

与进程调度的协同体现为阻塞语义的闭环：`accept_with_flags`、`sys_sendto`、`sys_recvfrom` 均使用 `process::with_current_user_context` 闭包，该闭包允许返回 `process::SYSCALL_BLOCKED` 暂停当前任务。`SocketSend`/`SocketRecv` 结构体携带 `wake_reader`/`wake_writer` 字段，指示需要唤醒的套接字 ID。以 `sys_sendto` 为例：发送成功后若返回了 `wake_reader`，则调用 `current.wake_socket_readers(socket_id)` 唤醒阻塞在 `recvfrom` 上的任务 <sup>[12](#mod-net-ref-12)</sup>。

### 4. 边角细节与不足

**AF_UNIX 存根** <sup>[13](#mod-net-ref-13)</sup>：`connect_unix_path_probe` 解析 Unix 域路径并通过 `vfs::resolve` 在文件系统中查找，但最终始终返回 `ECONNREFUSED`。这意味着 Unix 域套接字被识别但完全不可用——connect 必然失败，属于明确的半成品。

**panic 脆弱性** <sup>[14](#mod-net-ref-14)</sup>：`LoopbackStack` 多处使用 `.expect()` 处理可能失败的操作（如 IP 地址表 `push`、`listen`、`send`）。若内部状态不变量被打破，内核将直接 panic。虽然回环场景下难以触发，但不符合内核容错原则。

**孤立 smoltcp 栈** <sup>[15](#mod-net-ref-15)</sup>：`LoopbackStack` 与 `LoopbackDevice` 实现了完整的 TCP/IP over loopback，但系统调用路径完全不使用它们。该模块可能是早期原型或为未来扩展预留，目前属于事实上的死代码。

**硬编码限制** <sup>[16](#mod-net-ref-16)</sup>：`MAX_SOCKET_IO` 固定为 64KB，TCP/UDP 缓冲区同为 64KB，用户无法通过 `setsockopt` 调整。超限发送直接返回 `EMSGSIZE`，构成硬性吞吐瓶颈。

**不可靠的 unwrap** <sup>[17](#mod-net-ref-17)</sup>：`read_sockaddr_in` 从固定长度数组构造 `u16`/`u32` 时使用 `.try_into().unwrap()`，虽因前置长度检查不会失败，但若后续重构引入不一致，这些点将变成隐蔽的 panic 源。

**无真实网络设备**：套接字表层硬编码拒绝非 127.0.0.1 地址 <sup>[7](#mod-net-ref-7)</sup>；`LoopbackDevice` 虽实现了 `phy::Device` trait 可接入真实网卡，但无任何非回环设备注册。整个网络子系统是封闭的进程内通信设施。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块以 BlockDeviceOps trait 为核心抽象, 通过 BlockDeviceRegistry 管理至多 4 个块设备的注册与主设备选举; 底层实现为 VirtIO-blk 驱动, 同时支持 MMIO (legacy/modern) 与 PCI modern 三种传输路径. 上层覆以直接映射写直达缓存 (64 行 × 4096 字节), 使用自定义 SpinLock 保护全局单例. 与 rCore-Tutorial 的 virtio-drivers crate 方案不同, 本仓从零手写 VirtIO 寄存器交互与队列管理, 未依赖外部设备 crate, 且中断处理仅实现了轮询模式.

### 1. 核心抽象: BlockDeviceOps 与设备注册

驱动框架的基石是 `BlockDeviceOps` trait <sup>[1](#mod-drivers-ref-1)</sup>, 它仅定义五个方法: 就绪检查、扇区读、扇区写、同步与名称查询. 该 trait 要求实现者满足 `Sync` 约束, 这意味着所有块设备实现必须是线程安全的 — 锁的职责下沉到实现侧, 而非由框架统一加锁. `VirtioBlockOps` 结构体 <sup>[2](#mod-drivers-ref-2)</sup> 是一个零大小标记类型, 其所有方法委托到 `virtio::blk` 模块的静态函数, 而这些函数内部各自获取 `DEVICE` 和 `QUEUE` 两把全局自旋锁.

`BlockDeviceRegistry` <sup>[3](#mod-drivers-ref-3)</sup> 是一个容量为 4 的静态注册表, 内部以 `Option<&'static dyn BlockDeviceOps>` 数组存储. "主设备"(primary) 概念的引入简化了单盘系统的调用路径: `block_device()` 直接返回主设备引用, 调用者无需遍历注册表. 主设备选举逻辑 `selected_primary_index()` <sup>[4](#mod-drivers-ref-4)</sup> 优先检查已标记的主设备是否就绪, 若未就绪则线性扫描第一个就绪设备; 若全部未就绪, 回退到原始 primary_index (可能指向一个未就绪设备). 这种 fallback 机制在设备热移除场景下提供了一定韧性, 但 `primary_index` 字段本身不会因设备状态变化而更新, 构成潜在的状态不一致.

### 2. VirtIO 传输层: 三路径初始化与 DMA 地址处理

VirtIO-blk 驱动需要应对三种传输路径, 由 `TransportKind` 枚举 <sup>[5](#mod-drivers-ref-5)</sup> 区分: `Mmio` (含 legacy 与 modern 子版本) 和 `PciModern`. 初始化入口 `init_first()` <sup>[6](#mod-drivers-ref-6)</sup> 通过 `bus::device_probe().first_virtio_block_device()` 获取设备信息后, 根据设备枚举类型分发到 `init_mmio()` 或 `init_pci_modern()`. 

MMIO 路径的版本判别依赖 `version` 字段: `VIRTIO_MMIO_VERSION_LEGACY` (1) 走 legacy 队列初始化 — 要求队列内存页对齐, 并通过 `PAGE_SIZE` 寄存器通告页大小; modern 路径则先协商 `VIRTIO_F_VERSION_1` feature bit. PCI modern 路径通过 capability 链表 <sup>[7](#mod-drivers-ref-7)</sup> 定位 common_cfg / notify_cfg / device_cfg 三个 BAR 区域, 并读出 notify_offset 用于后续 doorbell 写入.

一个值得注意的细节是 `dma_addr()` 方法 <sup>[8](#mod-drivers-ref-8)</sup>: 对于 `TransportKind::Mmio`, 地址原样返回 (假设内核使用恒等映射); 对于 `PciModern`, 地址与 `DMA_PHYS_MASK` (48 位) 做按位与. 这个掩码硬编码了 x86-64 的物理地址宽度, 在 RISC-V (Sv39 为 39 位) 或 ARM64 (可配置 44/48 位) 上既可能错误截断也可能未充分屏蔽.

### 3. 缓存层: 直接映射写直达的取舍

`BlockCache` <sup>[9](#mod-drivers-ref-9)</sup> 是一个直接映射 (direct-mapped) 写直达 (write-through) 缓存, 64 行 × 4096 字节 = 256KB. 替换策略为轮转法: `self.next` 在 0..63 间循环, 每次 miss 时覆盖 victim 行. 没有 LRU/CLOCK 等更精细的淘汰算法, 也没有脏标记 — 因为所有写入直接穿透到设备 <sup>[10](#mod-drivers-ref-10)</sup>, 缓存行始终与设备同步.

写路径的设计顺序值得推敲: `write_sector()` 首先调用 `block_device().write_sector()` 将数据写入物理设备, 成功后再扫描全部缓存行, 将匹配的缓存行数据更新为输入内容. 这里隐含假设设备写入是同步的 (在当前轮询模型下成立), 若未来改为 DMA 异步写, 缓存行可能在设备完成前就被读取, 导致脏读. 此外, `PERF_COUNTERS_ENABLED` 编译期开关 <sup>[11](#mod-drivers-ref-11)</sup> 仅在 selftest 或 syscall trace 时启用缓存命中/缺失/淘汰统计, 避免生产路径的性能开销.

### 4. 跨模块协同: 从总线枚举到扇区 I/O 的全链路

初始化序列由 `drivers::init()` <sup>[12](#mod-drivers-ref-12)</sup> 串联: 先 `block::init()` 将静态 `VirtioBlockOps` 实例注册为主设备, 再 `virtio::init()` 调用 `blk::init_first()` 完成实际硬件初始化. 总线探测层 `DeviceProbe` trait <sup>[13](#mod-drivers-ref-13)</sup> 封装了传输偏好: `preferred_virtio_block_transport()` 查询架构模块 (`crate::arch`), 按偏好顺序先后尝试 PCI 与 MMIO. PCI 枚举 <sup>[14](#mod-drivers-ref-14)</sup> 遍历 ECAM 空间, 匹配 VirtIO vendor ID (0x1af4) 与 block device ID (modern 0x1042 / transitional 0x1001 + subsystem 2), 启用 bus mastering 后分配内存 BAR 地址并解析 VirtIO capability; MMIO 枚举 <sup>[15](#mod-drivers-ref-15)</sup> 则简单扫描 magic number (0x74726976) 与 device ID (2).

一次完整的读扇区调用链为: `block::read_sector()` → 获取 `CACHE` 锁 → `BlockCache::read_sector()` → `get_or_fill_index()` (查缓存, miss 时从设备填充整条缓存行) → `block_device().read_sector()` → `VirtioBlockOps::read_sector()` → 同时获取 `DEVICE` 锁与 `QUEUE` 锁 → `dev.read_sector()` → `request::prepare_read()` 填充三个 descriptor (header + data + status) → `submit_request()` → `publish()` 写入 avail ring + SeqCst fence → `notify_queue()` doorbell 写 → `wait_for_used()` 轮询 used ring. 整个路径中 `CACHE` 锁与 `DEVICE`/`QUEUE` 锁形成两段式加锁, 虽无死锁, 但缓存锁在设备 I/O 期间持续持有, 阻塞了所有并发缓存访问.

### 5. 边角细节与不足

- **中断处理完全缺失** <sup>[16](#mod-drivers-ref-16)</sup>: `PciInterruptHandling` 枚举仅有 `PollingOnly` 变体, `wait_for_used()` <sup>[17](#mod-drivers-ref-17)</sup> 以 `spin_loop()` 忙等. 生产环境下这浪费 CPU 周期, 且无法支持多队列并发.
- **硬编码 DMA 物理地址掩码** <sup>[8](#mod-drivers-ref-8)</sup>: `DMA_PHYS_MASK = (1usize << 48) - 1` 假设 x86-64 地址宽度, 跨架构移植时需参数化.
- **sync() 为空操作** <sup>[18](#mod-drivers-ref-18)</sup>: `VirtioBlockOps::sync()` 直接返回 `Ok(())`, 未发送 `VIRTIO_BLK_T_FLUSH` 命令. 文件系统依赖 sync 保证元数据持久性, 此实现下掉电可能丢数据.
- **粗粒度锁** <sup>[19](#mod-drivers-ref-19)</sup>: 每次扇区读写同时持有 `DEVICE` 与 `QUEUE` 两把锁, 加上外层 `CACHE` 锁, 整个块 I/O 子系统完全串行化.
- **队列深度固定为 8** <sup>[20](#mod-drivers-ref-20)</sup>: `VIRTQ_LEN = 8` 且每个请求占 3 个 descriptor, 理论最大并发仅 2 个请求; 对于未来 DMA 批量传输构成瓶颈.
- **block_device() 的 panic 风险** <sup>[21](#mod-drivers-ref-21)</sup>: `block_device()` 内部调用 `.expect()`, 若设备注册表为空或在初始化完成前调用将直接 panic.
- **PCI BAR 分配简单线性递增** <sup>[22](#mod-drivers-ref-22)</sup>: `allocate_memory_bars` 使用 `next_bar_addr` 线性累加, 未查询固件预留区域, 在真实硬件上可能与固件分配的地址冲突.
- **写直达缓存缺乏写屏障** <sup>[10](#mod-drivers-ref-10)</sup>: `write_sector` 中设备写入与缓存更新间无 fence, 在弱内存序架构上可能导致缓存行先于设备写入对其他观察者可见.

- 中断处理仅 PollingOnly, wait_for_used 以 spin_loop 忙等, 浪费 CPU 且无法支持并发 I/O 完成通知.
- DMA_PHYS_MASK 硬编码为 48 位 (x86-64), 在 RISC-V/ARM64 平台需参数化, 否则可能错误截断物理地址.

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块以集中式分发表为核心抽象, 通过 TrapFrameAccess trait 隔离架构差异, 采用 PC 回退协议统一处理阻塞式系统调用. 与同家族基线相比, 本实现扩展了信号递交、定时器抢占启发式以及 select/poll 的精细等待队列机制, 但巨量 match 分支与硬编码常量带来了维护风险.

### 1. 核心抽象: 集中式分发与上下文传递
系统调用入口统一由 `syscall_enter()` 承接 <sup>[1](#mod-syscall-ref-1)</sup>, 它从架构相关的 `UserTrapFrame` 中提取调用号与参数, 随后分派至两大路径: 特殊处理的 `clone`/`execve` 以及通用的 `syscall_dispatch_with_context` <sup>[2](#mod-syscall-ref-2)</sup>. 通用分派是一个超过 200 个分支的 giant match, 按子系统划归到 `fs`, `mm`, `process`, `socket`, `shm`, `select` 等模块 <sup>[3](#mod-syscall-ref-3)</sup>. 分派时携带一个 `SyscallContext`, 当前仅携带 `user_sp` 用于 `readlinkat` 等需感知用户栈的系统调用 <sup>[4](#mod-syscall-ref-4)</sup>. 这种集中分派节省了每个子系统自行解析参数的开销, 代价是分发表极其臃肿, 且参数类型转换 (如 `as isize`, `as i32`) 分散在各分支, 缺乏统一的类型安全层.

### 2. 关键设计取舍: 阻塞协议与调度决策
I/O 操作可能无法立即完成. 本模块未采用内核线程或 async 状态机, 而是通过 `SYSCALL_BLOCKED` 哨兵值实现协作式阻塞 <sup>[5](#mod-syscall-ref-5)</sup>. 当处理函数返回该值时, `syscall_enter` 将指令指针回退 (`retreat_pc`), 使该调用在任务被唤醒时重新执行 <sup>[6](#mod-syscall-ref-6)</sup>. 这一设计避免了维护跨调用的恢复上下文, 但要求每个阻塞路径必须正确注册等待队列 (例如 `wait_for_pipe_write` 或 `wait_for_socket_write`), 并在被唤醒后从零开始重试整个调用.

调度决策发生在两个层面: 系统调用返回后 (`should_schedule_after_syscall`) 与定时器中断中 (`should_schedule_after_user_timer_interrupt`). 前者对 exit/clone/futex/I/O/sched_yield 做特判 <sup>[7](#mod-syscall-ref-7)</sup>; 后者采用启发式: 若用户态连续在相同 PC 上运行超过 10 个时钟滴答 (`TIMER_PREEMPT_SAME_PC_TICKS`), 则强制调度 <sup>[8](#mod-syscall-ref-8)</sup>. 这种启发式避免了紧凑循环导致的饥饿, 但计数饱和 (saturating_add) 可能掩盖极长循环.

### 3. 跨模块协同: 从 Trap 到信号递交
LoongArch 的 trap 入口 `loongarch64_trap_handler` 通过异常码 `EXCEPTION_SYSCALL` 识别系统调用, 调用 `handle_user_syscall` -> `syscall_enter` <sup>[9](#mod-syscall-ref-9)</sup>. 整个路径无锁传递 `TrapFrame` 可变引用, 因为 trap 处理在关闭中断的上下文中执行. 系统调用返回后, `syscall_enter` 进行三层信号递送: (a) 由本次系统调用引发的同步信号 (`current_sync_signal_after_syscall`), (b) 其他待决信号, (c) 对 execve 成功后的调度 <sup>[10](#mod-syscall-ref-10)</sup>. 信号递送会直接修改 trap frame 以进入用户态信号处理函数, 此种“在 trap 返回路径中变轨”设计复用了已有的上下文切换机制, 但增加了控制流复杂度.

`select` 和 `poll` 模块 (`select.rs`) 展示了完备的等待队列协作: 它们通过 `WakeReason` 枚举区分信号唤醒、超时唤醒、管道/套接字事件唤醒, 在每次进入系统调用时优先检查这些唤醒原因, 避免重复轮询 <sup>[11](#mod-syscall-ref-11)</sup>. 当没有事件就绪且需要阻塞时, 它们收集 `FdWaitRegistration` 集合并调用 `wait_for_fd_events`, 最终返回 `SYSCALL_BLOCKED` 使调用方休眠 <sup>[12](#mod-syscall-ref-12)</sup>. 这种设计使得 select/poll 可以同时等待多种事件源, 但 `MAX_POLL_FDS=64` 和 `MAX_SELECT_NFDS=1024` 的硬编码上限限制了可扩展性 <sup>[13](#mod-syscall-ref-13)</sup>.

### 4. 边角细节与不足
- **`SyscallContext::default()` 的空 `user_sp`**: 若 `readlinkat` 通过非 `syscall_enter` 路径 (如测试桩) 调用 `syscall_dispatch`, 将获得 `user_sp: None`, 可能导致用户栈指针错误. 正常路径因 `syscall_enter` 总是注入 `Some(...)` 而安全, 但 API 设计留下了误用可能 <sup>[4](#mod-syscall-ref-4)</sup><sup>[14](#mod-syscall-ref-14)</sup>.
- **`current_sync_signal_after_syscall` 的语义歧义**: 该函数在 kill/tkill/tgkill 成功 (`ret==0`) 时直接返回所发信号编号, 意图是为自杀式信号提供同步递送. 但若目标进程不是当前进程, `deliver_current_pending_signal_handler` 将因无待决信号而静默跳过. 这依赖了待决信号掩码的正确性, 万一出现信号误投递, 会导致当前进程错误地收到不应有的信号 <sup>[10](#mod-syscall-ref-10)</sup>.
- **I/O 分派的不对称**: `SYS_WRITE` 被定向到 `super::io::sys_write`, 而 `SYS_READ` 却定向到 `fs::sys_read`. 尽管二者最终可能调用相同的底层, 但这种分派不一致增加了理解成本, 且缺乏注释说明 <sup>[3](#mod-syscall-ref-3)</sup><sup>[15](#mod-syscall-ref-15)</sup>.
- **硬编码缓冲区与 ioctl 风险**: `sys_ioctl` 使用固定大小 `RTC_TIME_SIZE` 作为 ioctl 缓冲, 若某个 inode 的 ioctl 返回尺寸超过该值, 将发生栈溢出. 同时 `FS_IOC_SETFLAGS` 分支仅读取 `FS_IOC_FLAGS_LEN` 字节, 未针对其他请求类型做长度校验 <sup>[16](#mod-syscall-ref-16)</sup>.
- **定时器抢占启发式的磨损**: `SAME_TIMER_PC_TICKS` 在每次检测到相同 PC 时递增并饱和, 但只要 PC 变化一次就会重置. 若用户态在包含多条指令的紧凑循环中执行, 每次 PC 可能略微漂移, 导致启发式失效而永不调度 <sup>[8](#mod-syscall-ref-8)</sup>.
- **分发表缺乏返回值校验**: 分发的所有分支均直接调用子系统函数并返回其结果, 但子系统函数返回 `isize`, 分发表将其作为 `SyscallRet` 原样传递. 若某子系统错误地返回了正数但语义不符 (例如返回系统调用号), 调用方可能误判为成功字节数. 缺少一个统一的返回码校验层 <sup>[3](#mod-syscall-ref-3)</sup>.

### ⚠ 开放问题

- SyscallContext::default() 将 user_sp 设为 None, 非 syscall_enter 路径调用 readlinkat 会传递无效栈指针.
- current_sync_signal_after_syscall 对 kill/tkill 不验证目标进程, 依赖待决信号掩码的正确性防止误递送.
- MAX_POLL_FDS=64 和 MAX_SELECT_NFDS=1024 硬编码上限, 不符合 Linux 的弹性限制, 可能阻断高并发应用.
- ioctl 使用固定栈缓冲区 RTC_TIME_SIZE, 若驱动的 ioctl 返回更大数据将导致栈内存破坏.
