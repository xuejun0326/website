## 📖 一、总览

该仓库是ArceOS-Starry家族中专注于Linux系统调用兼容性的微内核实现。它继承ArceOS的模块化设计（axhal、axmm、axtask等），但重构了进程管理、文件系统和系统调用层，以提供更完整的POSIX接口。与基线相比，启动流程被显式化，采用两级路由策略将内核初始化与运行时路径分离（[boot.ref:1]），从而支持自动化评测和交互式Shell两种模式。整体架构围绕“通过竞赛测试”这一目标展开，许多决策反映了对评测场景的深度定制。

该仓库最值得关注的特点有三：1）系统调用覆盖广泛（190项），核心的fork/execve/futex/epoll已正确实现，但内存管理（mremap、madvise）和信号处理（handler未调用）仍为桩或简化版本（[mm.ref:6][signal.ref:1]）；2）内存管理引入LRU ELF缓存和写时复制后端，提升了多进程加载效率（[mm.ref:2]），但部分语义不完整；3）文件系统完整实现了epoll LT/ET/OneShot和close_range UNSHARE（[fs.ref:4]），但存在unsafe规范问题（[fs.ref:10]）和返回值偏差（[fs.ref:15]）。这些特点凸显了项目在“广度优先”与“细节深度”之间的权衡。

## 🎯 二、综合评价

### 整体定位与架构继承
该仓库是ArceOS-Starry家族中致力于Linux系统调用兼容性的微内核实现。它继承了ArceOS的模块化设计（axhal、axmm、axtask等），但重构了进程管理、文件系统和系统调用层，以提供更完整的POSIX接口。与基线ArceOS相比，主要差异在于：启动流程被显式化，采用两级路由策略将内核初始化与运行时路径分离（[boot.ref:1]）；进程模型从单线程扩展到多线程，通过Thread和ProcessData双层结构支持资源共享（[task.ref:1]）；文件系统从简单VFS升级为三层架构并支持epoll（[fs.ref:1]）。系统调用层以包含190个分支的match表组织（[syscall.ref:1]），结合条件编译支持多架构但增加了维护复杂度。整体上，该仓库在ArceOS骨架上添加了Linux兼容的血肉，但其核心目标是为操作系统竞赛提供评测环境，许多决策围绕评测场景优化。

### 跨模块协同的创新设计
在遵循Linux规范的同时，该仓库展现了若干独特设计：- **信号与I/O多路复用的融合**：通过signalfd将信号事件注入epoll/poll循环（[signal.ref:5]），使得用户态进程可统一处理I/O和信号，无需单独信号线程。这种集成在微内核中较为少见。
- **ELF缓存与用户指针安全抽象**：`ElfLoader`使用容量固定的LRU缓存（[mm.ref:2]）避免重复解析ELF；`UserPtr`抽象（[mm.ref:4]）在访问用户空间指针时检查地址对齐、权限并确保物理页已分配，降低了unsafe操作的风险。
- **进程文件表隔离的轻量实现**：`close_range`的`CLOSE_RANGE_UNSHARE`通过关中断与unsafe引用计数递减（[fs.ref:4]），在无完整RCU支持下实现了文件表隔离，是一种精巧的工程简化。
- **统一的异步阻塞模型**：系统调用层使用`axtask::future::block_on`配合`axpoll::Pollable`（[syscall.ref:3]）实现阻塞式syscall（select、epoll_wait等）的休眠与唤醒，避免了忙等待，且该模型与信号协同（通过`pselect6`的信号掩码参数）一致。这些设计在保持代码简洁的同时，满足了竞赛场景对性能与正确性的要求。

### 工程取舍与能力边界
项目的目标决定了其取舍。明显省略的方面包括：- **多架构支持的滞后**：loongarch64支持被整体注释（[boot.ref:5]），部分系统调用仅对x86_64编译（[syscall.ref:2]），目前实际测试平台限于x86_64。
- **信号处理的不完整**：用户注册的信号处理函数从未被调用（[signal.ref:1]），仅实现了信号发送、掩码和default动作（终止/忽略），依赖信号处理器的程序将失败。
- **vfork的语义降级**：`CLONE_VFORK`被降级为地址空间复制，父进程不阻塞（[task模块不足]），虽可简化实现但破坏了POSIX语义。
- **低级系统调用的简化**：`mremap`未处理`MREMAP_MAYMOVE`等标志（[mm.ref:6]），`madvise`直接返回0（[mm.ref:10]），`prlimit64`忽略权限检查。这些简化减少了代码量，但限制了功能完整性。
- **调试与错误处理**：启动流程中所有fallible操作返回值均被丢弃（[boot.ref:5]）；`sys_rt_sigreturn`中`unwrap()`可能导致panic（[signal模块不足]）。这些反映仍处于原型阶段。这些取舍表明，项目优先保证关键路径（进程创建、文件IO、网络通信）的正确性，而将次要功能推迟。

### 完成度与代码质量评估
从功能覆盖看，190项系统调用涵盖了主要类别，但执行质量不均衡。内存管理和信号模块的桩调用较多（mremap、madvise、signal handler），网络存在逻辑bug（accept返回错误对端地址[net.ref:5]），文件系统部分返回值不规范（getcwd [fs.ref:15]）。代码质量方面，unsafe块普遍缺少安全契约（[fs.ref:10][net.ref:3]），全局变量设计可能引入竞态（[signal.ref:4]），refcell的假设同步缺少安全说明（[task.ref:1]）。这些缺陷使代码处于原型等级，适用于功能验证但不适合直接合入生产环境。

**评分**：
- completeness 2（核心功能基本实现，但细节缺失，信号处理未完成）。
- innovation 2（跨模块协同设计有亮点，但整体未脱离Linux框架）。
- code_quality 1（unsafe使用不规范，缺乏文档，存在潜在编程错误，如accept bug）。

### 1. 核心抽象：两级路由与编译时决策

启动流程的核心抽象是 **两级路由**：第一级是固定的 `starry_api::init()` 调用（行22-24），第二级是基于 `option_env!("AX_CMDLINE")` 的条件分支（行28-33）<sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>。这种设计将引导参数的决定权从运行时（如多阶段 bootloader）完全移交给了构建系统——启动后能且仅能根据 `AX_CMDLINE` 中是否包含子串 `"run_test"` 来选择路径。

第二级路由的两个分支代表了两种完全不同的使用场景：
- **自动化评测路径**（行35-38）：无交互地运行 `test::run_all_contest_tests()`，完成后直接调用 `axtask::exit(0)` 退出，整个生命周期在几毫秒内结束，适合 CI/CD 管线快速验证<sup>[3](#mod-boot-ref-3)</sup>。
- **本地调试路径**（行41-55）：通过 `entry::run_initproc` 以指定环境变量拉起 `/musl/busybox sh`，提供完整的交互式 Shell，允许开发者手工验证系统调用和文件系统行为<sup>[4](#mod-boot-ref-4)</sup>。

值得注意的是，`AX_CMDLINE` 的获取使用了 `option_env!` 加 `unwrap_or("")` 的惯用法（行29），这意味着如果构建系统忘记注入该环境变量，启动将静默地进入调试分支，而不是报告错误。这与 ArceOS 基线中普遍采用的 `env!`（编译期硬失败）形成对比，是一种有意的“容错”设计，但同时也可能掩盖配置遗漏。

### 2. 关键设计取舍：灵活性与安全性的权衡

**取舍一：编译时字符串匹配而非结构化配置。** 使用 `contains("run_test")` 的字符串包含判断（行31）而非枚举或布尔标志，使得构建系统只需在命令行中嵌入任意包含该子串的字符串即可触发测试模式。替代方案可以是独立的 feature flag 或 `#[cfg]` 属性。当前设计牺牲了类型安全，但换来了构建脚本的灵活性——例如 `"run_test_extra"` 也会被匹配，这既是 feature 也是风险。

**取舍二：同步阻塞式退出而非异步优雅停机。** 两条路径最终都调用 `flush_file_system()`（行121-124）然后通过 `axtask::exit(0)` 强行退出<sup>[5](#mod-boot-ref-5)</sup>。`flush_file_system` 内部依次调用了 `axfs::sync_filesystems()`、`FS_CONTEXT.lock()` 后的 `unmount_all()` 和 `filesystem().flush()`。全部返回值的 `let _ =` 丢弃（行121-124）意味着任何 IO 错误都会被无声忽略。在 ArceOS 基线中，文件系统的卸载错误通常通过 `?` 传播或触发 panic，而这里选择了“尽力而为”的容错策略。

**取舍三：基础设施架构的显式模块化 vs 注释占位。** 文件末尾的 `extern crate` 声明（行126-129）使用 `#[cfg(feature = "vf2")]` 和 `#[cfg(feature = "2k1000la")]` 条件引入平台级 crate，与第58-117行被整块注释的 `loongarch64` 测试代码形成强烈反差<sup>[6](#mod-boot-ref-6)</sup><sup>[7](#mod-boot-ref-7)</sup>。feature gate 保留了扩展点，但实际实现被注释掉，表明作者优先完成了 RISC-V 64 路径，而 LoongArch64 仍为“半成品”。这种“先占位后填充”的策略在原型阶段合理，但需要明确 TODO 标记。

### 3. 跨模块协同：从初始化到退出的完整链路

启动流程并非孤立存在，而是与以下模块紧密配合：
- **starry_api::init**（行23）：推测来自 `api` crate，负责建立全局数据结构（如 percpu 区域、中断向量表、内存分配器）。调用点位于 `main` 的第一条语句，确保后续所有操作（包括打印、内存分配、文件系统访问）的前提条件已满足。
- **test::run_all_contest_tests**（行35）：测试模块的入口，内部可能使用 `axtask` 创建多个用户进程，依赖 `axhal` 的定时器中断进行调度。启动流程将控制权完全让渡给测试框架，直到返回后才执行清理。
- **entry::run_initproc**（行51）：用户进程启动器，接收参数和环境变量。它与 `test` 模块不同在于：它只创建一个进程（busybox），而 `test` 模块会运行多轮进程。两者最终都通过文件系统同步路径退出。
- **flush_file_system**（行118-124）：作为退出前的最后关卡，它保证了脏缓存落盘，依赖 `axfs` 的同步原语。但错误忽略的设计意味着即使落盘失败，系统也会继续调用 `axtask::exit`，可能造成评测脚本检测到不完整的数据。

这种“初始化 → 路由 → 执行 → 清理 → 退出”的线性模型没有考虑热重启或故障恢复（例如测试过程中 panic），属于典型的单次运行的 unikernel 风格。

### 4. 边角细节与不足

以下问题在代码审查中值得特别关注：

- **<sup>[2](#mod-boot-ref-2)</sup> `option_env!` 默认分支风险**：当 `AX_CMDLINE` 未定义时，`unwrap_or("")` 返回空字符串，不包含 `"run_test"`，因此默认走调试路径。如果构建流程在非交互环境中（如 CI runner）忘记注入该变量，系统会卡在等待 Shell 输入的阶段，导致评测超时。更安全的做法是使用 `expect` 或编译期断言。
- **<sup>[5](#mod-boot-ref-5)</sup> 错误被吞噬**：`flush_file_system` 中四个 fallible 操作全部使用 `let _ =` 忽略结果。特别是 `unmount_all()` 的失败可能表明仍有打开的文件句柄，直接退出会导致文件系统损坏。ArceOS 基线的 `axfs` 通常要求先关闭所有文件，此处未做检查。
- **<sup>[6](#mod-boot-ref-6)</sup> 大规模死代码**：第58-117行的大块注释（约60行）包含了完整的 loongarch64 测试流程，被 `//` 注释而非 `#[cfg]` 条件编译。这意味着这些代码永远不被编译，且与下方 `#[cfg(feature = "2k1000la")]` 的 extern crate 自相矛盾——平台 crate 被引入但无用。这是明显的工程腐烂信号，应标记为 `TODO` 或删除。
- **<sup>[3](#mod-boot-ref-3)</sup> 测试路径的无条件退出**：`test::run_all_contest_tests()` 返回后直接 `exit(0)`，未检查测试结果。如果测试中有失败断言（非 panic），返回值仍为 0，评测系统无法区分成功与失败。可考虑让 `run_all_contest_tests` 返回 `bool` 或 `i32` 退出码。
- **`unsafe` 块缺失 SAFETY 注释**：虽然 `main` 函数本身未直接使用 `unsafe`，但 `#[unsafe(no_mangle)]` 属性（行22）隐含了对符号名的断言。Rust 2024 版要求 `unsafe` 属性必须附 `// SAFETY:` 注释，此处未提供，构成技术债。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 axmm::AddrSpace 为核心为用户进程管理虚拟地址空间，结合 axhal::paging 和 axmm::backend 实现页表操作。它解决了用户程序加载、动态内存映射（mmap/brk）、缺页按需分配等核心问题，并通过 ELF 缓存、信号跳板映射、用户指针安全检查等机制提升性能与安全性。与标准 ArceOS 相比，本仓库增加了写时复制（CoW）后端、LRU ELF 缓存、更完善的 mmap 语义（共享/私有/设备映射）以及基于 AtomicBool 的缺页过滤机制。

### 1. 核心抽象：基于 axmm 的地址空间框架

本模块以外部 crate `axmm` 提供的 `AddrSpace` 为基石，每个用户进程持有一个 `AddrSpace` 实例，封装了虚拟地址范围、页表操作（map/unmap/protect）及缺页按需分配等底层功能。在 `core/src/mm.rs` 中，`new_user_aspace_empty` 通过指定用户空间基址与大小创建空白地址空间<sup>[1](#mod-mm-ref-1)</sup>；`map_trampoline` 将信号跳板物理页以线性映射方式固定到 `SIGNAL_TRAMPOLINE` 虚拟地址，使得信号处理无需内核切换页表<sup>[1](#mod-mm-ref-1)</sup>。与标准 ArceOS 相比，本模块补充了完整的用户程序加载流程（`load_user_app`）和 ELF 缓存机制，并且通过 `Backend` 抽象支持多种物理内存后端（分配、文件 CoW、设备直接映射）。

### 2. 关键设计取舍

**ELF 缓存**：`ElfLoader` 内部使用 `LRUCache<ElfCacheEntry, 32>` 缓存最近加载的 ELF 文件，避免重复解析文件系统和频繁的 I/O<sup>[2](#mod-mm-ref-2)</sup>。缓存条目 `ElfCacheEntry` 借助 `ouroboros` 宏实现自引用，同时持有 `CachedFile` 和解析后的 `ELFParser`。取舍：缓存显著加速了动态链接程序（尤其是多次启动同一程序）的加载时间，但占用了内存；清理函数 `clear_elf_cache` 需要手动调用（用于内存泄漏检测）。

**用户内存访问安全**：`UserPtr<T>` 和 `UserConstPtr<T>` 封装了用户空间指针，它们在解引用前先调用 `check_region` 检查对齐、访问权限，并通过 `populate_area` 确保物理页已分配<sup>[4](#mod-mm-ref-4)</sup>。该设计避免了内核直接解引用用户指针可能触发的错误，但返回的 `&'static mut T` 隐含了“地址空间不会并发变化”的假设，否则可能导致 use-after-free。

**缺页与 access_user_memory 标志**：内核访问用户内存时（如拷贝数据），先通过 `access_user_memory` 设置全局 `AtomicBool` 标志，再执行内存操作。缺页处理程序 `handle_page_fault` 检查该标志，仅在标志为真时才处理用户空间缺页（即按需分配物理页），否则直接返回 false 交给系统默认处理<sup>[3](#mod-mm-ref-3)</sup><sup>[5](#mod-mm-ref-5)</sup>。这一设计避免了复杂的内核堆栈切换，并防止了内核逻辑误触发用户缺页导致的递归。

**mmap 系统调用**：`sys_mmap` 支持丰富的映射类型：匿名映射（`Backend::new_alloc`）、文件私有映射（`Backend::new_cow` 写时复制）、文件共享映射（`Backend::new_file` 或基于设备特性）、以及大页（2M/1G）<sup>[6](#mod-mm-ref-6)</sup>。私有映射采用 CoW 后端，只有写入时才复制页面，节省物理内存。设备映射则根据设备的 `DeviceMmap` 枚举选择直接物理映射或缓存文件。

### 3. 跨模块协同

缺页处理流程横跨 `core/src/mm.rs`、`api/src/mm.rs` 和 `axmm` crate。`api/src/mm.rs` 通过 `#[register_trap_handler(PAGE_FAULT)]` 注册缺页处理函数，该函数首先读取 `is_accessing_user_memory()` 标志，若为真则获取当前线程的 `AddrSpace` 并调用 `handle_page_fault`<sup>[5](#mod-mm-ref-5)</sup>。`AddrSpace` 的实现来自 `axmm` crate，它遍历页表、分配物理页并更新映射。标志 `ACCESSING_USER_MEM` 在 `core/src/mm.rs` 中作为全局状态管理，被 `VmIo`  trait（用于 `user_copy`）和 `check_null_terminated` 等函数使用<sup>[3](#mod-mm-ref-3)</sup>。

资源映射方面，`core/src/mm.rs` 的 `map_elf` 使用 `FileBackend::Cached` 作为 `Backend::new_cow` 的后端，实现可执行文件按需加载和写时复制；而 `api/src/syscall/mm/mmap.rs` 在文件映射时复用相同的 `Backend` 和 `SharedPages` 机制。此外，`api/src/mm.rs` 中的 `VmBytes` / `VmBytesMut` 实现了 `axio::Read` / `Write` trait，使得用户内存可以像文件一样被高层次的 I/O 操作使用，实现了 `core` 与 `api` 层之间的协同。

### 4. 边角细节与不足

- **架构假设与条件编译**：`copy_from_kernel` 在 aarch64 和 loongarch64 上跳过（因独立页表，硬件隔离），其他架构则复制内核映射<sup>[9](#mod-mm-ref-9)</sup>。`flush_icache` 仅在 RISC-V 和 LoongArch64 上执行，因为 x86_64 硬件保证指令缓存一致性<sup>[8](#mod-mm-ref-8)</sup>。这些条件编译是正确的，但若增加新架构（如 mips）需显式处理。
- **sys_mremap 简化实现**：使用先 mmap 新址、拷贝数据、再 munmap 旧址的方式，未处理地址重叠、未实现 `MREMAP_MAYMOVE` 等标志<sup>[6](#mod-mm-ref-6)</sup>。
- **桩子系统调用**：`sys_madvise` 直接返回 0；`sys_mlock2` / `sys_membarrier` 仅完成最低功能<sup>[10](#mod-mm-ref-10)</sup>。
- **性能隐患**：`check_null_terminated` 每字节检查且每次循环都获取 `aspace` 锁（因为可能触发缺页），注释中明确承认效率低<sup>[7](#mod-mm-ref-7)</sup>。
- **死代码**：`core/src/mm.rs` 中 `Vm(IrqSave)` 结构体被 `#[allow(dead_code)]` 标记，可能是未清理的遗留代码。
- **生命周期假设**：`UserPtr::get_as_mut` 返回 `&'static mut T`，但仅在下一次地址空间修改前有效，若调用者长期持有则存在 UAF 风险。
- **memfd 占位实现**：使用临时文件而非匿名内存，不符合 Linux 语义。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块基于 axtask crate 实现了用户态进程/线程抽象，以 Thread 结构为核心（通过 TaskExt trait 附加到 axtask 的 TaskInner 上），通过 ProcessData 共享地址空间、文件表、信号等资源。采用协作式调度（yield）与异步阻塞（poll_fn + waker）结合，系统调用 clone/execve/exit 均集中实现。相比原始 ArceOS，本仓增加了完整的多线程支持、健壮列表、futex 和 /proc 文件系统交互，但存在 vfork 语义不完整、退出等待机制粗糙等早期实现痕迹。

### 1. 核心抽象：Thread 与 ProcessData 的双层任务模型

模块没有直接复用 axtask 的原始任务对象，而是在其上包裹了一层 `Thread`（`core/src/task.rs:52-84`）。`Thread` 通过 `TaskExt` trait 与 `TaskInner` 关联：`on_enter` 时设置 `ActiveScope`（用于资源隔离），`on_leave` 时恢复为全局作用域 <sup>[1](#mod-task-ref-1)</sup>。这种设计使得同一进程内的线程共享 `ProcessData`（包含地址空间 `AddrSpace`、信号管理器、futex 表等），而每个线程又有独立的 `ThreadSignalManager`、`clear_child_tid`、`robust_list_head` 和 `TimeManager`。

`ProcessData` 使用了 `Arc<Mutex<AddrSpace>>` 实现写时共享——当 `clone` 时不带 `CLONE_VM` 标志时，会通过 `try_clone()` 复制并写入映射 <sup>[9](#mod-task-ref-9)</sup>；否则直接克隆 Arc，实现地址空间共享。这种设计直接映射了 Linux 的 `CLONE_VM` 语义，但代价是必须在使用 `clone` 系统调用时手动处理页表根替换（`set_page_table_root`），否则新线程会看到旧的页表。

### 2. 关键设计取舍

- **阻塞等待的异步化**：`sys_waitpid`（`api/src/syscall/task/wait.rs`）没有像传统内核那样直接挂起当前任务并用调度器唤醒，而是使用 `block_on(interruptible(poll_fn(...)))` 模式，将等待事件（`child_exit_event`）注册到 waker。这使得等待过程可以被打断（例如收到信号），从而在用户态表现为返回 `EINTR`。但该实现选择在被打断且非 `WNOHANG` 时自动重试而不是返回错误 <sup>[10](#mod-task-ref-10)</sup>，偏离了 POSIX 规范，但避免了用户态处理 `EINTR` 的复杂性。
- **vfork 的简化处理**：`clone` 系统调用中遇到 `CLONE_VFORK` 标志时，仅简单地移除 `CLONE_VM` 标志 <sup>[8](#mod-task-ref-8)</sup>，转而复制地址空间，而不是像 Linux 那样阻塞父进程直到子进程 exec/exit。这违反了 vfork 的语义，但简化了实现，且对于不使用 vfork 的程序无影响。
- **健壮列表的有限支持**：`exit_robust_list` 在 `api/src/task.rs:105-122` 实现了 futex 健壮列表清理，但限制循环次数为 `ROBUST_LIST_LIMIT`，并在每次迭代中调用 `yield_now` 防止饥饿。这是对 Linux `kernel/futex/core.c` 的简化移植，缺少竞争处理和 priority-inheritance 逻辑。
- **per‑thread 时间管理器**：`TimeManager` 使用 `AssumeSync<RefCell<TimeManager>>` 实现线程本地可变借用，并明确假设只在上下文切换时独占借用 <sup>[2](#mod-task-ref-2)</sup>。中断上下文中使用 `try_borrow_mut` 避免重入 panic——这是一个精巧的折中，但需要调用者保证不会在持有 RefCell 时换出。

### 3. 跨模块协同

- **信号传递与任务打断**：`send_signal_thread_inner`（`core/src/task.rs:304-308`）在调用 `thr.signal.send_signal` 成功后执行 `task.interrupt()`，中断正在用户态循环的任务。该中断在 `new_user_task` 的主循环顶部被 `curr.clear_interrupt()` 消费，随后进入信号处理路径 <sup>[5](#mod-task-ref-5)</sup>。
- **/proc 文件系统暴露**：`api/src/vfs/proc.rs` 通过读取全局 `TASK_TABLE` 和 `PROCESS_TABLE` 生成 `/proc/[pid]` 和 `/proc/[tid]` 目录，使外部可以查看任务统计、命令行、fd 等状态。这依赖 `starry_core::task::tasks()` 和 `get_task()` 等公共接口。
- **futex 表共享与隔离**：`ProcessData::futex_table_for` 根据 `FutexKey` 是私有（`Private`）还是共享（`Shared`）决定返回进程内 futex 表还是全局共享表（通过 `Weak` 指针索引）<sup>[7](#mod-task-ref-7)</sup>。共享表由 `SHARED_FUTEX_TABLES` 全局 `Mutex` 管理，并每 100 次操作清理一次过期条目。
- **execve 的文件系统交互**：`sys_execve` 使用 `axfs::File::open` 打开可执行文件路径，并配合 `FS_CONTEXT` 和 `FD_TABLE` 的 scope 机制实现资源隔离。进程的 umask 和 brk 边界在 exec 后通过 `ProcessData` 的原子变量重置。

### 4. 边角细节与不足

1. **`AssumeSync` 缺少 `// SAFETY:` 注释** <sup>[1](#mod-task-ref-1)</sup>：`unsafe impl Sync for AssumeSync<T>` 没有解释为什么 `RefCell<TimeManager>` 在并发访问下是安全的。当前实现依赖“只在上下文切换时独占借用”的约定，但没有任何强制机制；如果未来有代码在多个线程中同时调用 `time.borrow_mut()`，将导致 `RefCell` panic。
2. **`add_task_to_table` 的非原子检查** <sup>[3](#mod-task-ref-3)</sup>：插入多个全局表时分别用 `contains_key` 检查，但检查与插入之间不是原子的。高并发下可能重复插入导致 `WeakMap` 条目冗余或条件竞争。
3. **`kill_all_non_init_processes` 的忙等待** <sup>[4](#mod-task-ref-4)</sup>：使用 50 次 yield 等待进程退出，不保证在时限内所有进程都已僵尸化。如果某些进程卡在不可中断状态，该函数会过早进入回收阶段，留下仍然存活的进程。
4. **vfork 语义缺失** <sup>[8](#mod-task-ref-8)</sup>：前文已述，`CLONE_VFORK` 被当作普通 clone 处理，父进程未挂起。这可能导致依赖于 vfork 同步的程序（如 glibc 的 posix_spawn 实现）出现竞态。
5. **等待系统调用的信号打断自动重试** <sup>[10](#mod-task-ref-10)</sup>：非 `WNOHANG` 时，信号打断后不返回 `EINTR` 而是重试，这会使用户态永远收不到 `EINTR`，可能被某些程序误用于实现无中断的等待。

总体而言，该模块在实用性和可理解性之间做了取舍，大量逻辑直接映射 Linux 系统调用的语义，但多个边缘路径（vfork、健壮列表、精确等待）以简化实现为主，留下了不少早期开发痕迹。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块以 `axfs_ng_vfs` 的 VFS trait 体系（FilesystemOps、NodeOps、FileNodeOps、DirNodeOps）为核心抽象，自研了内存文件系统 `MemoryFs`（tmpfs），并在 `FileLike` trait 上统一了常规文件、目录、pipe、epoll 等对象。系统调用层完整实现了 Linux 兼容的 openat、read/write、ioctl、epoll、getdents64 等 60+ 文件系统调用。与 ArceOS 基线相比，显著扩展了设备文件特殊处理（/dev/ptmx 自动创建 Pty、/dev/tty 重定向）、epoll LT/ET/OneShot 完整实现、以及 close_range 的 UNSHARE 进程文件表隔离机制。

### 1. 核心抽象：三层 VFS 架构

文件系统的根基是 `axfs_ng_vfs` crate 定义的 VFS trait 体系。底层 `FilesystemOps` 描述一个挂载点实例的行为（获取根目录、查询 statfs）<sup>[1](#mod-fs-ref-1)</sup>；`NodeOps` 描述单个 inode 的元数据操作；`FileNodeOps` 与 `DirNodeOps` 分别规定文件读写和目录遍历的语义。该体系的核心设计在于 `DirEntry` 作为统一引用句柄——它同时持有 `Arc<DirNode>` 或 `Arc<FileNode>` 以及一个 `Reference`（包含父目录和文件名），使得路径解析与 inode 操作解耦。

中间层 `axfs` crate 在此基础上构建了 `FsContext`（进程级虚拟文件系统状态，包含当前工作目录和挂载表）、`OpenOptions`（open 标志转换器）、以及 `FileBackend` 枚举（区分直接定位与页缓存代理）。特别地，`FS_CONTEXT` 是一个全局 `RwLock<FsContext>`，所有路径解析都通过它进行——这意味着 chroot 的实现仅需替换这个上下文<sup>[7](#mod-fs-ref-7)</sup>。

最上层是本仓库特有的系统调用实现层。核心 trait `FileLike` 统一了所有可放入文件描述符表的对象：`File`（常规文件）、`Directory`（目录）、`Pipe`（管道）、`Epoll`（事件轮询）以及 `DummyFd`（用于 qemu 用户态回退）<sup>[12](#mod-fs-ref-12)</sup>。通过 `FD_TABLE`（进程级文件描述符表），系统调用层以 `get_file_like(fd)` 获得 `Arc<dyn FileLike>`，然后调用其 `read`/`write`/`ioctl` 等方法，实现了面向 Linux ABI 的适配层。

### 2. 关键设计取舍

**自研 tmpfs（MemoryFs）而非复用现有实现**。`api/src/vfs/tmp.rs` 实现了一个从零构建的内存文件系统：inode 分配依赖 `slab::Slab`（O(1) 分配回收），目录项用 `HashMap<FileName, InodeRef>` 存储。一个独特之处是 `FileName` 的排序策略——`Ord` 实现中将 `.` 排在第 0 位、`..` 排在第 1 位、普通文件排在其后<sup>[13](#mod-fs-ref-13)</sup>。这确保了 `getdents64` 返回的目录流中 `.` 和 `..` 总是最先出现，符合 POSIX 语义但不依赖底层哈希表顺序。引用计数管理使用 `Arc<Inode>` + `nlink` 双重计数：`release_inode` 在 nlink 归零且 `Arc::strong_count` 仅剩 2（inode slab slot + 当前持有者）时回收 inode <sup>[14](#mod-fs-ref-14)</sup>。这种设计的代价是 `InodeRef` 的 `Drop` 必须调用 `release_inode`，而 `MemoryNode` 的 `Drop` 也参与释放链，一旦形成循环引用（如硬链接环）就会泄漏。

**epoll 的完整实现**。`api/src/file/epoll.rs` 实现了 Level-Triggered（LT）、Edge-Triggered（ET）和 OneShot 三种触发模式<sup>[8](#mod-fs-ref-8)</sup>。ET 与 OneShot 都在 `consume` 时从就绪队列移除，LT 则保留以持续通知。去重使用 `AtomicBool` + `compare_exchange` 防止同一兴趣项被多次加入就绪队列。`InterestWaker` 的 `wake_by_ref` 采用弱引用链（`Weak<EpollInner>` + `Weak<EpollInterest>`），优雅处理 epoll 实例或兴趣项已销毁的情况<sup>[9](#mod-fs-ref-9)</sup>。与标准 Linux epoll 的关键差异在于：本实现没有红黑树组织兴趣项，而是用 `HashMap<EntryKey, Arc<EpollInterest>>`，其中 `EntryKey` 同时以 fd 和 `Weak<dyn FileLike>` 指针为哈希输入，确保同一个 fd 被替换后不会误命中旧项。

**`sys_close_range` 的 UNSHARE 机制**。当传入 `CLOSE_RANGE_UNSHARE` 标志时，该调用会执行一个精心编排的锁协议：先在读锁下 clone 全局 `FD_TABLE`，然后关中断、释放 `scope` 读锁（利用 `scope_local` 的 `force_read_decrement`）、获取写锁替换为新表<sup>[4](#mod-fs-ref-4)</sup>。这种做法的目的是在不中断其他线程读访问的前提下，为当前进程创建一个独立的文件表副本，断开与父进程的共享。风险是 `force_read_decrement` 是一个 unsafe 操作，依赖调用者确保没有其他并发读取者——代码通过 `IrqSave`（关中断）来保证原子性，但这在多核上并不可靠（它只屏蔽了当前核的中断）。

### 3. 跨模块协同

文件系统模块与以下子系统深度配合：

- **`starry_vm::VmPtr`**: 所有用户空间指针（`*const c_char`、`*mut u8`）都通过 `vm_load_string`、`vm_write_slice`、`VmBytes`/`VmBytesMut` 进行安全访问。例如 `sys_getdents64` 最终调用 `vm_write_slice(buf, &buffer.buf)` 将内核填充好的 dirent64 结构写回用户空间<sup>[5](#mod-fs-ref-5)</sup>。
- **`starry_core::task::AsThread`**: 通过 `current().as_thread()` 获取当前进程的 umask（`sys_mkdirat` 中 `mode & !umask()`）、以及进程组、会话和终端信息（用于 `/dev/tty` 重定向）<sup>[3](#mod-fs-ref-3)</sup>。
- **`axhal::time::wall_time`**: `sys_utime`/`sys_utimensat` 在 `times` 参数为 NULL 时使用墙上时间作为当前时间。
- **`axpoll::{IoEvents, PollSet, Pollable}`**: epoll 复用 `Pollable` trait 监听 fd 状态变化，`PollSet` 作为跨任务唤醒的底层机制。

一个典型的协同流程：`sys_openat` 解析路径后，在 `add_to_fd` 中检测到底层文件是否是一个 `Device`。若是 `tty::Ptmx`，则自动创建 pty master/slave 对，通过 `FS_CONTEXT` 解析 `/dev/pts/N` 路径获得 slave 的 `Location`，最后封装为 `axfs::File` 并插入 `FD_TABLE`<sup>[3](#mod-fs-ref-3)</sup>。这个过程串联了 VFS 路径解析、设备驱动抽象层、终端子系统、以及文件描述符表管理。

### 4. 边角细节与不足

**不安全代码契约缺失**：`DirBuffer::write_entry` 中的 unsafe 块只写了 `// FIXME: safety` 注释，未说明安全前提（调用者必须保证 buf 已初始化、offset 在合法范围内、name 不包含内部 NUL 等）<sup>[10](#mod-fs-ref-10)</sup>。这不是生产级代码应有的状态。

**`sys_chroot` 的粗暴实现**：`*fs = FsContext::new(loc)` 直接丢弃旧的 `FsContext` 实例，不会递归 unmount 子文件系统，也不会释放关联的 dentry 缓存<sup>[7](#mod-fs-ref-7)</sup>。对运行中的多线程进程调用 chroot，其他线程的 `FS_CONTEXT.lock()` 竞争可能导致观察到不完整的文件系统状态。

**`sys_getcwd` 的返回值错误**：代码注释 `// FIXME: it is said that this should return 0` 表明已知在缓冲区足够时应返回拷贝的长度而非 `buf` 指针，但当前返回的是指针值<sup>[15](#mod-fs-ref-15)</sup>。这违反了 Linux 的 `getcwd` 规范，会导致 glibc 的 `getcwd(NULL, 0)` 自动分配路径时错误。

**`sys_utime` 中的 FIXME**：`unsafe { times.vm_read_uninit()?.assume_init() }` 未实现 `AnyBitPattern` trait，依赖 `FIXME` 标记<sup>[11](#mod-fs-ref-11)</sup>。`vm_read_uninit` 的语义不安全——若用户空间提供的 `utimbuf` 包含未初始化填充字节，`assume_init` 会导致立即 UB。

**锁粒度的潜在死锁**：`sys_close_range` 的 UNSHARE 实现中，持 `IrqSave` 期间调用了 `scope.write()`，而 `scope` 的读写锁实现如果是 spinlock，在多核上关中断不会阻止其他核的并发访问。同时，`removed` Vec 在锁外 drop 的设计虽然避免了重入死锁，但若 `FileLike::Drop` 实现再次调用 `FD_TABLE.write()`（如关闭 pipe 时清理 epoll 注册），仍可能死锁——因为 `FD_TABLE` 可能已经被该线程的上层调用持锁。

**`sys_ioctl` 的 TIOCGWINSZ 静默忽略**：当 ioctl 返回 `NotATty` 且 cmd 为 TIOCGWINSZ 时直接 return，仅对非 tty 文件上调用 glibc 的常见行为做兼容<sup>[6](#mod-fs-ref-6)</sup>。这是实用的工程妥协，但意味着对所有非 tty 字符设备的 TIOCGWINSZ 查询都会静默成功（无错误输出），可能掩盖底层驱动的问题。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制基于`starry_signal` crate 提供的信号集、信号信息、信号动作等抽象，通过线程本地信号状态和进程级动作表管理信号传递与处理。系统调用层在 `api/src/syscall/signal.rs` 实现标准 POSIX 信号接口，`api/src/signal.rs` 提供信号检查与动作分派，`api/src/file/signalfd.rs` 将信号桥接到 I/O 多路复用框架。与同家族基线相比，本实现更完整地覆盖了 `signalfd`、`rt_sigqueueinfo` 等扩展，但信号处理程序（handler）的实际用户态跳板尚未实现，多处 `TODO` 标志着早期开发状态。

### 1. 核心抽象：跨任务信号传递模型

信号机制的核心抽象由 `starry_signal` crate 提供，包括 `SignalSet`（信号掩码）、`SignalInfo`（信号信息，含信号编号、发送者 PID、错误码等）、`Signo`（强类型信号编号枚举）、`SignalStack`（备选信号栈）以及 `SignalOSAction`（信号处置动作枚举，如终止、Core Dump、停止、继续、用户处理）。内核通过 `axtask::current` 获取当前任务，并通过 `AsThread` trait 访问线程的 `signal` 字段——一个类型为 `starry_signal::SignalContext` 的结构体，包含 `blocked` 掩码、`pending` 队列以及指向进程级动作表的引用。

系统调用层在 `api/src/syscall/signal.rs` 中实现了 11 个信号相关系统调用：`rt_sigprocmask`、`rt_sigaction`、`rt_sigpending`、`kill`、`tkill`、`tgkill`、`rt_sigqueueinfo`、`rt_tgsigqueueinfo`、`rt_sigreturn`、`rt_sigtimedwait`、`rt_sigsuspend` 以及 `sigaltstack`。这些 syscall 通过 `send_signal_to_process` 等函数（位于 `starry_core::task` 模块）分发信号，信号发送逻辑目前隐藏在该模块内，但调用点显示其支持按 PID、线程 ID、进程组广播。

信号检查在 `api/src/signal.rs` 的 `check_signals` 函数中完成。该函数从线程信号上下文中取一个待处理信号，根据其动作执行 `do_exit`（终止）、`do_exit` 带转储码（Core Dump）、或者留空（Stop/Continue/Handler）。值得注意的是，`SignalOSAction::Handler` 分支目前仅作 `// do nothing` 注释，这意味着用户注册的信号处理函数尚未被真正调用——这是一个关键的功能缺口。

### 2. 系统调用实现中的同步与异步

信号相关系统调用在设计上兼顾了同步掩码操作和异步等待。`sys_rt_sigprocmask` 直接通过线程的 `signal.set_blocked()` 修改阻塞掩码，同时将旧掩码写回用户空间。`sys_rt_sigaction` 通过锁保护进程级 `actions` 数组，但锁的粒度仅限于单个 syscall 执行期间，且未使用原子变量，在并发注册时可能存在短暂的不一致。

`sys_rt_sigtimedwait` 和 `sys_rt_sigsuspend` 使用 `poll_fn` + `block_on` 实现异步等待。前者会临时解除对触发信号的阻塞（`old_blocked & !set`），等待时通过 `poll_interrupt` 挂起任务；后者则用指定掩码替换当前掩码，并始终返回 `-EINTR`。这种设计模拟了 Linux 原语，但依赖 `block_on` 阻塞当前线程，可能阻塞同进程的其他任务（如果内核是单线程调度模型则无问题）。

`sys_rt_sigreturn` 是信号处理程序返回的入口。它在恢复用户上下文前调用 `block_next_signal()`（基于全局 `AtomicBool`），然后调用 `signal.restore()`。全局标志的引入是为了防止信号处理函数执行期间被另一个信号嵌套检查，但多核环境下全局变量可能导致竞态——更好的方案可能是 per-thread 标志或嵌套计数。

### 3. 多模块协同：从系统调用到信号消费

信号机制的完整流程跨越多个模块。以 `kill(pid, sig)` 为例：用户态调用 `kill`，陷入内核执行 `sys_kill`，后者调用 `send_signal_to_process`（位于 `starry_core::task`）。信号被放入目标进程的信号链表（pending），并在目标线程下次检查信号或从内核返回用户态时触发。

信号检查的触发点有两类：显式等待系统调用（`rt_sigtimedwait`、`rt_sigsuspend`）和系统调用返回前的隐含检查。隐含检查在 `syscall_handler` 中完成，但当前未包含在提供的片段中，推测在 `axhal::trap` 中调用 `check_signals`。`check_signals` 根据 `SignalOSAction` 分派：终止动作调用 `do_exit`（位于 `api/src/task.rs`），Handler 动作理论上应修改 `UserContext` 的返回地址以跳转到用户态信号处理函数，但当前被忽略。

`signalfd` 展示了信号与 I/O 系统的集成。`sys_signalfd4` 创建一个 `Signalfd` 对象，注册为文件描述符，并被 `epoll`/`select` 框架轮询。`Signalfd::read` 通过 `poll_io` 等待信号到来，从线程的 pending 信号中出队匹配掩码的信号，转换为 128 字节的 `SignalfdSiginfo` 返回。这使得信号可像普通文件描述符一样被异步多路复用，与 Linux 完全兼容。

### 4. 边角细节与不足

- **<sup>[1](#mod-signal-ref-1)</sup> 信号处理程序未实现**：`check_signals` 对 `SignalOSAction::Handler` 仅 `// do nothing`，这意味着所有通过 `sigaction` 注册的用户处理函数永远不会被调用。要实现它，需要在 `UserContext` 中修改指令指针指向用户注册的 `sa_handler`，并压入返回地址为 `sigreturn` 的框架。当前缺失使得大部分用户态信号处理程序（如 `SIGALRM`、`SIGUSR1`）无法工作。
- **<sup>[2](#mod-signal-ref-2)</sup> 多个 TODO 标记**：CoreDump、Stop、Continue 动作均只打印注释而未实现。`sys_rt_sigreturn` 中有 `block_next_signal()` 但无对应的 `unblock` 显式调用（通过 `restore` 隐式处理？）。`sys_rt_sigqueueinfo` 中的权限检查仅对比 pid，未处理更细粒度的权能检查。
- **<sup>[3](#mod-signal-ref-3)</sup> 锁与原子性风险**：`sys_rt_sigaction` 使用 `num_actions.lock()` 保护动作数组，但 `sys_rt_sigpending` 和 `kill` 等读取 pending 信号时未加锁，依赖 `SignalSet` 内部的原子操作。若 `SignalSet` 基于整数而非原子整数，则并发读写可能导致撕裂。
- **<sup>[4](#mod-signal-ref-4)</sup> 全局 `AtomicBool` 的适用性**：`block_next_signal` 使用全局 `static` 变量，在多核或多线程环境下，一个线程的 `block_next_signal()` 可能错误地阻塞另一个线程的信号检查。理想设计应该是 per-thread 标志。
- **<sup>[5](#mod-signal-ref-5)</sup> 错误处理省略**：`sys_rt_sigreturn` 中的 `unwrap()` 调用（`ok()` 后解包）若 `restore` 失败会直接 panic，没有向上传播错误。此外，`check_sigset_size` 对 `size == 0` 也返回 `Ok(())`，与 Linux 行为可能不一致（Linux 要求 `sigsetsize` 必须等于 `sizeof(sigset_t)`）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现了 System V 共享内存、匿名管道和 futex 快速用户态锁三种 IPC 机制。共享内存以 `ShmInner` 封装段元数据，通过全局 `ShmManager` 管理 key-id 映射和进程附着关系，物理页面延迟到第一次 `shmat` 时分配；管道基于 `HeapRb` 环形缓冲区，集成 `axpoll` 实现异步 I/O 并支持容量动态调整；futex 使用 `FutexKey` 区分私有/共享地址，每个地址空间持有独立的 `FutexTable`，`WaitQueue` 支持超时、bitset 和 requeue 操作。与基线 ArceOS 相比，本仓增加了跨进程共享内存后端支持，futex 支持共享类型（通过 `SharedPages` 引用），并引入了进程退出时的自动清理回调。

### 1. 核心抽象与外部依赖

进程间通信模块由三个独立子系统构成：共享内存（`shm`）、管道（`pipe`）和 futex。

**共享内存**采用 System V 风格 API（`shmget`/`shmat`/`shmdt`/`shmctl`）。核心数据结构 `ShmInner` 封装段标识符、页数、物理页面引用（`Arc<SharedPages>`）和 Type-C 兼容的 `ShmidDs` 状态。全局 `ShmManager` 通过 `BiBTreeMap` 维护 key↔shmid 双向映射，并通过 `pid_shmid_vaddr` 跟踪每个进程的附加地址。物理页面分配推迟到首次 `sys_shmat`，利用 `axmm::backend::SharedPages` 实现跨进程共享。外部依赖包括 `axhal::paging`（映射标志）、`axmm::AddrSpace`（地址空间操作）和 `axsync::Mutex`（全局锁）<sup>[1](#mod-ipc-ref-1)</sup>。

**管道**基于 `axio::Buf/BufMut` 和 `ringbuf::HeapRb` 环形缓冲区，初始容量 64 KiB。`Pipe` 结构体封装读写端标志、共享状态和原子非阻塞标志。通过 `axpoll::PollSet` 实现异步唤醒，读写操作使用 `block_on(poll_io(...))` 模式，依赖 `axtask::future` 和 `axsync::Mutex`。管道容量可按页对齐调整，但需保证不丢失已有数据<sup>[2](#mod-ipc-ref-2)</sup>。

**Futex** 实现核心是 `WaitQueue`（基于 `SpinNoIrq<VecDeque>`）和 `FutexTable`（进程私有 `Mutex<HashMap>`）。`FutexKey` 区分私有（纯虚拟地址）和共享（共享内存区域偏移+后端弱引用）两种类型，后者允许跨进程同步。`FutexEntry` 包含等待队列和所有者死亡标志（用于 robust list）。外部依赖涉及 `axmm::AddrSpace` 查询区域后端、`axsync::Mutex` 和 `axtask` 的异步运行时<sup>[3](#mod-ipc-ref-3)</sup>。

### 2. 跨模块协同与数据流

**共享内存与地址空间**：`sys_shmat` 获取 `ShmInner` 后，检查 `phys_pages`：若已存在（其他进程已附加），则用 `Backend::new_shared` 重用；否则分配新 `SharedPages`。映射通过 `aspace.map()` 完成，标志继承自 `shmget` 时的权限（可被 `SHM_RDONLY` 降级）。分离时 `sys_shmdt` 调用 `aspace.unmap()` 并更新 `ShmManager` 的跟踪映射。进程退出时，`clear_proc_shm` 遍历该进程附加的所有段，递减引用计数，对标记 `rmid` 且计数值归零的段执行最终删除<sup>[4](#mod-ipc-ref-4)</sup><sup>[5](#mod-ipc-ref-5)</sup>。

**管道与信号**：当写端检测到读端全部关闭（`Arc::strong_count == 1`），调用 `raise_pipe()` 通过 `starry_core::task::send_signal_to_process` 向当前进程发送 `SIGPIPE`。这与常规 Unix 语义一致，但依赖于信号模块的存在。`ioctl(FIONREAD)` 通过 `vm_write` 将占用长度写入用户空间指针<sup>[6](#mod-ipc-ref-6)</sup>。

**Futex 与进程数据**：`sys_futex` 通过 `FutexKey::new_current` 查询当前地址空间，基于区域后端决定 futex 类型。私有 futex 存储在当前进程的 `FutexTable` 中（通过 `proc_data.futex_table_for` 获取）。共享 futex 的 key 包含对 `SharedPages` 的弱引用，使得跨进程同步时使用同一 `FutexEntry`。`FutexGuard` 的析构函数在引用计数 ≤2 且等待队列为空时从表中移除条目，避免泄漏<sup>[7](#mod-ipc-ref-7)</sup>。

### 3. 关键设计取舍

**物理页面延迟分配**：与 Linux 在 `shmget` 时分配物理资源不同，本实现在第一次 `shmat` 时才创建 `SharedPages`。这减少了未附加段的资源占用，但增加了 `shmat` 的延迟和失败风险。替代方案（在 `shmget` 时预分配）会增加未使用段的内存压力。

**FutexKey 的共享类型设计**：通过 `FutexKey::Shared` 携带 `Weak<SharedPages>`，同一个共享内存区域内的 futex 在不同进程中映射到相同的 key（基于偏移）。这允许跨进程进程同步，但弱引用在物理页面释放后变为 `None`，此时 futex 退化？当前代码在 `FutexKey::new` 中未处理 `Weak::upgrade` 失败的情况，可能导致悬空指针<sup>[8](#mod-ipc-ref-8)</sup>。

**管道容量动态调整**：`resize` 方法允许改变环形缓冲区大小，但限制新容量不小于已占用长度且为页大小倍数。这提供了灵活性，但调用者需确保无并发 I/O，否则可能出现数据竞赛（未加锁保护整个操作）。

**IPC ID 生成**：使用简单的原子递增，不回收已删除 ID。长运行系统会耗尽 ID 空间（i32 范围，约 20 亿），但嵌入式场景不易触发。替代方案（如 ID 复用）引入复杂性且价值有限。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块的核心抽象是 `SocketAddrExt` trait 与 `Socket` newtype，通过 trait 多态统一 IPv4/IPv6/Unix/VSock 地址族的用户空间编解码，`Socket` 包装 `axnet::Socket` 并实现 `FileLike` 和 `Pollable`，使 socket 融入文件描述符框架。与同家族相比，本仓在 `sys_connect` 中显式将 `WouldBlock` 映射为 `InProgress` 以兼容 Linux 非阻塞语义，同时通过自定义 `CMsgBuilder` 在无 libc 依赖下实现 `SCM_RIGHTS` 文件描述符传递。但存在 `sys_accept` 误用 `local_addr()` 代替 `peer_addr()` 的 bug，以及 `stat` 实现为 TODO 等问题。

### 1. 地址抽象层：`SocketAddrExt` 与地址族的统一

网络模块面临的首要问题是 Linux socket 地址族（AF_INET、AF_INET6、AF_UNIX、AF_VSOCK）在用户空间与内核空间之间的传递格式各不相同。`api/src/socket.rs` 定义了 `SocketAddrExt` trait，提供 `read_from_user`/`write_to_user` 方法，将用户空间的 `sockaddr` 裸指针转化为 Rust 类型安全的 `SocketAddr`、`UnixSocketAddr`、`VsockAddr`。`<sup>[1](#mod-net-ref-1)</sup>` 该 trait 内部使用 `read_family` 读取前两个字节的地址族，然后根据族值分派到具体实现。对于 `UnixSocketAddr`，代码额外处理了 `AF_UNIX` 的三种变体：Unnamed、Abstract（以 \0 开头）、Path（以绝对路径字符串结尾），其中 Abstract 地址是现代 Linux 特性，实现通过检查首字节是否为 `0` 来区分。`<sup>[11](#mod-net-ref-11)</sup>` 这种设计将底层地址布局完全封装在 trait 实现中，上层系统调用如 `sys_bind` 只需调用 `SocketAddrEx::read_from_user` 即可获得抽象地址，无需关心细节。`SocketAddrEx` 是一个枚举，其 `read_from_user` 进一步根据 `read_family` 的结果将调用分派给 `SocketAddr`、`UnixSocketAddr` 或（当 `vsock` 特性启用时）`VsockAddr`。`<sup>[1](#mod-net-ref-1)</sup>` 这样做的好处是新增地址族只需要添加新的枚举变体和 `SocketAddrExt` 实现，不修改现有系统调用代码；代价是增加了 trait 的动态分派开销，但在系统调用上下文中可忽略。

### 2. 系统调用实现：分层、错误语义与 Linux 兼容性

网络系统调用在 `api/src/syscall/net/` 下按功能分为四个文件：`socket.rs`（创建、绑定、连接、监听、接受、关闭）、`io.rs`（发送与接收）、`opt.rs`（getsockopt/setsockopt）、`name.rs`（getsockname/getpeername）。这种分层与 Linux 系统调用分类一致。

`sys_socket` 是入口，根据 `domain` 和 `ty` 的组合实例化对应的 `axnet::Socket` 变体（Tcp/Udp/Unix/Vsock）。`<sup>[2](#mod-net-ref-2)</sup>` 其中 `ty` 的低 8 位与 `O_NONBLOCK`/`O_CLOEXEC` 标志位耦合，代码先提取 `ty = raw_ty & 0xFF` 用于匹配，再检查高位标志，这在 Linux 中是标准做法。但需要注意 `raw_ty` 可能包含其他标志（如 `SOCK_NONBLOCK` 在 Linux 上定义为 `O_NONBLOCK`），目前代码只处理了 `O_NONBLOCK` 和 `O_CLOEXEC`，其他标志被静默忽略，可能引入兼容性问题。`<sup>[2](#mod-net-ref-2)</sup>`

`sys_connect` 中有一个精妙的错误转换：当底层返回 `AxError::WouldBlock` 时，代码将其转为 `AxError::InProgress`。`<sup>[6](#mod-net-ref-6)</sup>` 这是因为 Linux 对非阻塞 TCP connect 的规范是返回 `EINPROGRESS`（对应 `InProgress`），而不是 `EWOULDBLOCK`。这个转换保证了与 glibc 等用户态库的兼容性，否则用户程序会误判连接状态。

`sys_accept4` 的实现存在一个**明显的 bug**：第 122 行调用的是 `socket.local_addr()` 而不是 `socket.peer_addr()`。`<sup>[5](#mod-net-ref-5)</sup>` Linux 的 accept 系统调用返回的是对端地址（peer address），而这里错误地传回了本地地址。这将导致用户程序（如 httpd）获得错误的远程地址。代码注释和变量名写的是 `remote_addr`，但实际调用了 `local_addr()`，这是一个应被标记为 `BUG` 的实现缺陷。

`sys_listen` 对 `backlog` 的校验也过于简单：仅当 `backlog < 0 && backlog != -1` 时返回 `InvalidInput`，而 Linux 通常将负值视为 0，且建议对最大值设置上限（如 `SOMAXCONN`）。`<sup>[4](#mod-net-ref-4)</sup>` 当前实现在 backlog 为 `-1` 时将其直接传递给下层，可能导致下层处理意外值。

### 3. 跨模块协同：Socket 作为文件描述符与 axnet 的绑定

`api/src/file/net.rs` 定义了 `Socket` newtype，其唯一字段 `axnet::Socket` 通过 `Deref` 暴露所有 socket 操作。`<sup>[9](#mod-net-ref-9)</sup>` 同时，`Socket` 实现了 `FileLike` trait，将 `read`/`write` 委托给 `recv`/`send`，使得 socket 能够像普通文件一样被 `sys_read`/`sys_write` 处理。更关键的是 `Pollable` 实现，它委托给 `axnet::Socket` 的 `poll`/`register`，使得 socket 可以被 epoll 等 I/O 多路复用机制监控。`<sup>[9](#mod-net-ref-9)</sup>` 这种设计复用现有的文件描述符表（`add_to_fd_table`）和文件引用计数（`Arc`），无需为 socket 单独建立新的管理结构。

控制消息（cmsg）的处理是另一个跨模块协同的典型。`api/src/syscall/net/cmsg.rs` 实现了 `CMsg` 和 `CMsgBuilder`，在无 libc 依赖的情况下解析和构造 `cmsghdr`。`<sup>[8](#mod-net-ref-8)</sup>` 特别地，`SCM_RIGHTS` 用于在进程间传递文件描述符：解析时从用户空间读取文件描述符整数，通过 `get_file_like` 获取对应的 `Arc<dyn FileLike>`；构造时通过 `add_file_like` 将本进程的文件对象插入目标进程的描述符表。`<sup>[8](#mod-net-ref-8)</sup>` 这个机制依赖 `axtask::current()` 获取当前进程上下文，体现了与任务模块的协作。

`sockopt` 的处理通过宏 `call_dispatch!` 生成模式匹配，将 `(level, optname)` 映射到 `GetSocketOption`/`SetSocketOption` 枚举值。`<sup>[10](#mod-net-ref-10)</sup>` 这种宏驱动的注册方式便于扩展新选项，但编译时所有选项都被编译进来，没有条件编译，可能会在非 Linux 平台引入未定义选项。目前所有选项都是 Linux 标准选项，没有平台特定的优化。

### 4. 边角细节与不足

- **`unsafe` 块缺乏安全契约**：`api/src/socket.rs:36-38` 的 `cast_to_slice` 函数使用裸指针将任意引用转换为字节切片，但没有 `// SAFETY:` 注释，调用者必须自行保证类型无填充且指针合法。`<sup>[3](#mod-net-ref-3)</sup>` 当前调用点 `fill_addr` 中传递的是栈上分配的 `sockaddr_*` 结构体，但未来若传入堆分配或外部数据，可能触发未定义行为。
- **`sys_accept` 对端地址错误**：如前所述，`local_addr()` 应为 `peer_addr()`，这是显著的逻辑错误。`<sup>[5](#mod-net-ref-5)</sup>`
- **`stat` 实现为 TODO**：`api/src/file/net.rs:26` 标记 `// TODO(mivik): implement stat for sockets`，目前返回硬编码 `S_IFSOCK | 0o777`，这意味着 `fstat` 系统调用无法获取 socket 的真实信息（如 inode 号、大小）。`<sup>[7](#mod-net-ref-7)</sup>` 这会影响依赖 `fstat` 的用户程序。
- **`sys_listen` 的 backlog 检查有歧义**：`backlog < 0 && backlog != -1` 这个条件意味着 `-1` 被放行但 `-2` 被拒绝，但 Linux 中 `backlog` 为负数应被视为 0 或按内核默认值处理。`<sup>[4](#mod-net-ref-4)</sup>` 当前行为与 Linux 不完全一致。
- **`cfg(feature = "vsock")` 的覆盖风险**：vsock 支持通过特性门控，在 `api/src/socket.rs` 和 `api/src/syscall/net/socket.rs` 中都有 `#[cfg(feature = "vsock")]` 分支。如果未来新增地址族（如 `AF_BLUETOOTH`），需要同步修改多处，但没有统一的注册点，可能导致遗漏。
- **`sys_connect` 的 `WouldBlock` 转换可能遗漏其他阻塞场景**：只有 connect 会返回 `WouldBlock`，但理论上 `send`/`recv` 也可能返回该错误，而 `sys_sendto`/`sys_recvfrom` 没有做类似转换。`<sup>[10](#mod-net-ref-10)</sup>` 这可能是因为 Linux 只在 connect 上要求 `EINPROGRESS`，但其他操作的非阻塞语义仍可能触发 `WouldBlock`，目前上层会直接返回该错误，用户态需要自行处理。

总体而言，网络模块在架构上清晰地将地址转换、系统调用实现、文件描述符集成分层，但在细节处存在多处与 Linux 语义的偏差和未完成的实现，需要在后续迭代中逐步修正。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块实现Linux兼容的系统调用分派与处理，核心抽象是基于Sysno枚举的大型match分发表，将用户态系统调用请求路由到对应子模块（fs、mm、net、signal等）的处理函数。它解决了在ArceOS微内核上为用户程序提供标准Linux系统调用接口的问题，通过异步阻塞模型（axtask::future::block_on + axpoll::Pollable）实现I/O多路复用syscall。与基线ArceOS相比，本仓库增加了大量系统调用实现（超过200个），并引入了starry-core/starry-vm等依赖，实现了进程、信号、资源限制等完整POSIX语义。

### 1. 系统调用分发架构与抽象层次

核心入口`handle_syscall`接收`UserContext`引用，从中提取系统调用号并匹配`Sysno`枚举<sup>[1](#mod-syscall-ref-1)</sup>。匹配臂覆盖了文件操作、内存管理、网络、信号、进程控制等类别，每个臂调用对应的子模块函数（如`sys_read -> sys_read`）。这种大规模match设计优点是直截了当，便于添加新syscall；缺点是match分支庞大（超过200个），编译耗时且不易维护。替代方案可以是使用虚表或动态分发，但match更适用于已知全集且零开销。

子模块函数返回`AxResult<isize>`，由`handle_syscall`统一设置返回值。无效系统调用号返回`-ENOSYS`。注意，部分syscall被条件编译限制在x86_64或riscv64<sup>[2](#mod-syscall-ref-2)</sup>，这反映了对多架构的支持策略：x86_64作为主架构，riscv64初步支持，而aarch64可能暂未实现某些旧式syscall（如`open`、`stat`）——这些架构走`openat`、`fstatat`等新式接口。

### 2. I/O多路复用的异步阻塞模型

`select`、`pselect6`、`poll`、`ppoll`、`epoll_wait`等syscall共享一个设计模式：将文件描述符集合封装为`FdPollSet`（实现`Pollable` trait），然后通过`axtask::future::block_on(future::timeout(timeout, poll_io(…)))`阻塞当前任务直到事件就绪或超时<sup>[3](#mod-syscall-ref-3)</sup><sup>[4](#mod-syscall-ref-4)</sup>。这种“异步轮询 + 显式阻塞”的混合模型利用了ArceOS的协程调度器，允许在等待I/O时不浪费CPU，但每个syscall块一个任务，与原生Linux的线程阻塞语义一致。

`epoll`内部使用`Epoll`对象（文件系统抽象的子类），事件循环同样复用`poll_io`。注意，`epoll_pwait`和`pselect6`接受用户态信号掩码，通过`with_replacen_blocked`临时替换当前信号掩码<sup>[5](#mod-syscall-ref-5)</sup>，这是与`starry_signal`模块协同的关键点。

### 3. 时间、资源与进程管理的协同设计

时间相关syscall（`clock_gettime`、`gettimeofday`、`times`）直接调用`axhal::time`获取墙钟或单调时间，并利用`starry_core::task::AsThread` trait访问线程的时间会计<sup>[6](#mod-syscall-ref-6)</sup>。资源限制`prlimit64`通过`get_process_data`获取进程级`rlim`数组，读写时加锁<sup>[7](#mod-syscall-ref-7)</sup>。`getrusage`则遍历进程的所有线程，累加`utime/stime`<sup>[8](#mod-syscall-ref-8)</sup>。

这些实现依赖`starry_core`和`starry_vm`提供的内核抽象（如进程树、虚拟内存指针），与基线ArceOS只有简单任务管理的做法不同，本仓库构建了完整的进程模型。

### 4. 边角细节与不足

- 条件编译裸奔：`#[cfg(target_arch = "x86_64")]`散落在各处，aarch64默认分支可能误覆盖<sup>[2](#mod-syscall-ref-2)</sup>（如`Sysno::pipe`直接调用`pipe2`，未检查是否支持`pipe`）。
- `sys_prlimit64`中当`new_limit.rlim_max > limit.max`时仅返回`Ok(0)`而不报错<sup>[7](#mod-syscall-ref-7)</sup>，这忽略了权限校验，可能被用户绕过限制。
- `sys_sysinfo`用`core::mem::zeroed()`初始化`sysinfo`（注释`FIXME: Zeroable`）<sup>[9](#mod-syscall-ref-9)</sup>，如果结构体含有非零位模式的有效位，可能引发UB。
- `sys_setitimer`使用`unsafe { new_value.vm_read_uninit()?.assume_init() }`<sup>[10](#mod-syscall-ref-10)</sup>，没有`#[derive(Zeroable)]`，读未初始化内存是未定义行为。
- `sys_getrandom`通过文件系统读取`/dev/random`或`/dev/urandom`<sup>[11](#mod-syscall-ref-11)</sup>，这会引入文件系统依赖并可能阻塞，而原生Linux的`getrandom`是直接调用内核随机数生成器，不受挂载影响。
- 很多`unwrap()`和`expect()`未在路径中处理，如`FD_TABLE.read()`<sup>[12](#mod-syscall-ref-12)</sup>的锁获取若失败会panic，但用户态输入可间接触发资源耗尽？

这些不足多处于早期实现阶段，体现了“先功能后正确”的工程折衷。
