## 📖 一、总览

该仓库是一个面向教学与实验的类 Linux 内核，定位于单任务运行时（singletask runtime），整体风格偏向“用 Rust 安全地实现确定性与零成本抽象”。与 rCore-Tutorial 或 ArceOS 相比，它更激进地采用静态全局数组和无锁原子操作管理全部内核资源（任务槽、VMA、文件描述符、套接字、IPC 对象），放弃了动态分配、通用扩展性和多核并发保护，换取了编译期确定性和极小的间接开销。它明确以 Linux ABI 为兼容目标，实现了 252 项系统调用，但并非完整覆盖，而是聚焦于进程控制、文件系统和内存管理核心路径，网络和高级信号等模块尚处于模拟或 stub 状态。

值得评审注意的独特特点有两个：一是基于固定容量（64 槽）的 EDF 调度器，所有任务状态通过全局静态 AtomicUsize 数组维护，实现完全无锁且无动态分配；二是三层文件系统栈（ext4 只读根卷 + 内存可写覆盖层 + 编译期硬编码的 builtin 虚拟文件系统），通过 copy-up-on-write 实现写入，而 /proc、/dev、/sys 全部为编译期静态数据，不支持运行时修改。这些设计在 rcore-rust 家族中非常罕见，体现了对极致确定性和最小运行时开销的追求。

## 🎯 二、综合评价

### 整体定位

本仓库是一个**教学/演示型内核**，并非生产级 Linux 兼容实现。它尝试在 Rust 的类型系统和编译期能力范围内，构建一个尽可能确定性的单任务内核。所有核心数据结构都使用固定容量的静态数组和 `AtomicUsize` 实现无锁并发控制（如 EDF 调度器的任务槽 `[ref: task.ref:1]`、VMA 快照的 `[usize; VMA_SLOT_COUNT]` `[ref: mm.ref:1]`、套接字表 `[ref: net.ref:2]` 等）。这与 ArceOS 或 rCore-Tutorial 的多进程、动态地址空间模型形成鲜明对比：后者依赖堆分配和锁保护，灵活性高但确定性低。本仓库的定位是“在特定约束下（单任务、固定容量、单核）用 Rust 证明内核的可靠性”。它暴露了 252 个 Linux 系统调用，但很多仅通过 `SyscallVfs` 的默认 `UnsupportedPath` 返回错误 `[ref: fs.ref:1]`，真正的覆盖范围远小于数字所示。

### 真正的创新点

1. **EDF 调度器与无锁任务状态机**：在 rcore-rust 家族中首次实现最早截止时间优先调度，所有任务状态通过全局原子数组维护，没有 `SpinLock` 或 `Mutex`。等待机制细分为 19 种类型，每种有独立的等待槽位数组 `[ref: task.ref:4]`，调度器通过 `WaitChannel` 枚举统一所有阻塞原因 `[ref: drivers.ref:1]`。这种设计在用户态内核中极为罕见，体现了对“确定性调度”的极致追求。

2. **三层文件系统与 copy-up-on-write**：根文件系统只读挂载 ext4，通过 `WritableOverlay` 内存层实现写入，builtin 虚拟文件系统（procfs/devfs/sysfs）完全在编译期由常量生成 `[ref: fs.ref:4]`。这一模型放弃了动态写回永久存储的能力，但使得文件系统元数据完全确定，适合无盘嵌入式场景。

3. **SyscallOutcome 显式状态枚举**：系统调用结果不再是 `Result<isize>`，而是 `SyscallOutcome` 枚举，明确区分 `Return`、`Blocked`、`Fork`、`Exec`、`Exit` 等 `[ref: syscall.ref:2]`。调度器可以直接 match 决定下一步，无需回调或隐式状态机。`SignalMask` 字段与阻塞直接集成，使 `pselect6` 等可中断等待的正确性得到类型保证 `[ref: syscall.ref:3]`。

### 取舍判断

仓库团队将精力集中在**确定性、零间接成本和单核无锁设计**上，而非功能完备性、多核支持或通用性。具体取舍包括：
- **放弃动态扩展**：所有资源池（任务、VMA、套接字、IPC 对象、overlay 文件）均硬编码容量，超限静默丢弃或返回错误。这降低了代码复杂度，但限制了可扩展性。
- **放弃多核锁同步**：全局 `static mut` 变量无锁保护（如 `ACTIVE_FD_TABLE` `[ref: syscall.ref:不足1]`），当前假设单核运行。若引入多核，数据竞争会引发 UB。
- **放弃完整网络协议栈**：套接字用固定包池模拟数据流，无 TCP/UDP 协议实现，仅支持本地域通信 `[ref: net.ref:不足3]`。这在教学场景可接受，但无法用于真实网络。
- **放弃 per-process 独立地址空间**：所有用户内存状态集中在全局 `UserMemorySlot` 数组中，而非每个任务独立 VMA 结构 `[ref: mm.ref:1]`。这简化了实现，但违背了传统多进程隔离模型。

### 完成度与不足

从功能覆盖看，核心路径（fork/execve/exit、mmap/brk、open/read/write、sigaction/rt_sigreturn）均已实现，能承载简单的 Linux 用户程序（例如编译测例中的 musl 静态二进制）。然而，模块中暴露了多处不足：
- VMA 快照在容量超限时静默丢弃条目 `[ref: mm.ref:2]`，可能导致物理页泄漏或地址空间不一致。
- 信号机制缺少 `SIGSTOP/SIGCONT` 的进程暂停/继续状态机 `[ref: signal.ref:9]`，仅返回 `NotSupported`。
- `SysV` 信号量和消息队列的操作无全局锁，存在并发竞争窗口 `[ref: ipc.ref:不足1]`。
- `unsafe` 块普遍缺少 `// SAFETY` 注释（如 overlay copy-up 路径 `[ref: fs.ref:5]`），违反 Rust 安全最佳实践。
- 硬编码常量过多（如 `COMPLETION_SPIN_LIMIT=1e8` `[ref: drivers.ref:5]`），缺乏运行时配置手段。

整体而言，该仓库在单任务、固定容量的约束下完成了较高的完成度，但距离一个通用、可移植的内核仍有显著差距。它是 rcore-rust 家族中一次有趣的实验，证明了 Rust 原子操作和编译期常量在构建确定性内核方面的能力，但生产使用者需要仔细评估其限制。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块由三部分组成：架构无关的契约层（BspServices、EarlyBootInfo）、架构特定实现（RISCV64/LoongArch64 的 boot.rs）和内核编排层（kernel/boot.rs）。核心抽象是以函数指针聚合所有 BSP 回调的 BspServices 结构体，配合 BoundaryMode 三级安全模式，将硬件操作封装成幂等状态机。与同家族标准 ArceOS 相比，本仓放弃了全局 AxConfig 风格，改用显式函数指针组合，牺牲了动态可扩展性但获得了编译期类型安全和零间接开销。FDT 解析由 RISCV64 自包含实现（约 200 行）和 LoongArch64 复用共享模块两种策略共存，反映了架构团队尚未统一启动发现逻辑。

### 1. 核心抽象：跨架构启动契约

启动流程的核心抽象是 `BspServices`——一个包含 11 个函数指针字段的结构体<sup>[1](#mod-boot-ref-1)</sup>，涵盖 trap 安装、timer 启动、MMU 激活、用户态进入、上下文切换、块设备、内存发现、init 路径解析和停机等所有 BSP（板级支持包）回调。与同家族标准 ArceOS 的 `AxConfig` 全局单例不同，本仓选择纯函数指针组合，这意味着：

- **零间接开销**：每个 BSP 服务调用都是直接的函数调用，没有 vtable 或 switch；
- **编译期完备性**：`BspServices::new` 是 `const fn`，所有回调在编译时确定，无法在运行时替换；
- **状态显式化**：`BspSnapshot` 记录了每个硬件组件的 `HardwareReadiness` 状态<sup>[2](#mod-boot-ref-2)</sup>，内核层可以在启动过程中查询哪些子系统已就绪。

`EarlyBootInfo` 是固件参数的架构中立打包<sup>[3](#mod-boot-ref-3)</sup>，包含 `architecture` 枚举（Riscv64/LoongArch64）、`primary_cpu_id` 和两个不透明固件参数。每个架构在入口点（如 `arch/riscv64/boot.rs::early_boot_info`）完成填充。

### 2. 关键设计取舍：FDT 解析、函数指针与安全模式

**2a. FDT 解析策略分歧**

RISCV64 实现在 `arch/riscv64/boot.rs` 中自建了完整的最小 FDT 解析器<sup>[4](#mod-boot-ref-4)</sup>，包括头部验证、结构块遍历、`/memory` 节点 `reg` 属性解析、`/chosen/bootargs` 中 `init=` 参数提取。而 LoongArch64 实现则通过 `crate::arch::contract::fdt` 模块调用共享的 `fdt::discover_memory` 和 `fdt::discover_boot_init_path`<sup>[5](#mod-boot-ref-5)</sup>。这种分歧意味着：

- RISCV64 版本不依赖外部 libfdt，但函数约 200 行，仅支持 memory 和 chosen 两个节点，不支持中断控制器等复杂设备树；
- LoongArch64 版本复用共享模块，但该模块的 file_excerpts 未提供，推测其实现与 RISCV64 类似但封装更通用；
- 两个架构没有统一 FDT 解析入口，违反了 DRY 原则，是早期架构分叉的典型技术债务。

**2b. 函数指针 vs. Trait 对象**

`BspServices` 使用裸函数指针而非 `dyn Trait`，优势是调用路径无动态分发、结构体可 `Copy` 和 `const` 构造；劣势是添加新服务需要修改五个位置（结构体字段、构造函数、调用站点、架构实现、契约定义）。作为对比，同家族仓库 rCore-Tutorial 使用 `Trait` + `Box<dyn>` 实现设备驱动，本仓的设计更适合启动阶段（无堆分配），但在内核完整态下扩展性不足。

**2c. BoundaryMode 三级安全防线**

每个硬件操作都接受 `BoundaryMode` 参数<sup>[6](#mod-boot-ref-6)</sup>，分为 `Inspect`（仅检查可行性）、`Prepare`（执行非破坏性准备）、`ApplyUnsafe`（真正操作硬件）。例如 `install_trap_vector` 在 `Inspect` 模式下返回 `TrapInstallState::Feasible` 而不写入 CSR，使得内核可以在安全模式下预先验证整个启动计划，再以 `ApplyUnsafe` 统一提交。这种设计借鉴了 Rust 的 `unsafe` 契约思想：安全代码可以枚举所有可能状态，只有 `ApplyUnsafe` 分支才需要 `unsafe` 块。

### 3. 跨模块协同：从固件到 kernel_start

启动流程的完整调用链为：

1. **固件入口**：OpenSBI 将控制权传给内核，传递 hart_id 和 FDT 指针；
2. **架构层打包**：`arch/riscv64/boot.rs::early_boot_info` 构造 `EarlyBootInfo`<sup>[3](#mod-boot-ref-3)</sup>；
3. **服务组合**：`bsp_services` 查询各子模块的 `readiness()`，构造 `BspServices`<sup>[7](#mod-boot-ref-7)</sup>；
4. **内核编排**：`kernel/boot.rs::kernel_start` 接收 `BspServices`，按序执行<sup>[8](#mod-boot-ref-8)</sup>：
   - 安装 trap 向量（`install_trap_vector`）；
   - 启动硬件 timer（`start_timer`）；
   - 发现物理内存（`discover_boot_memory` → 回调到架构的 FDT 解析）；
   - 激活内核 MMU（`activate_kernel_mmu`）；
   - 解析 boot init 路径（`discover_boot_init_path` → 再次 FDT 解析）；
   - 记录 init 路径到全局静态变量（`record_init_trace`）；
5. **停滞**：当前实现记录完路径后直接 `halt`，没有真正执行 init 程序。

其中 `discover_boot_memory` 和 `discover_boot_init_path` 以函数指针的形式存储在 `BspServices` 中<sup>[1](#mod-boot-ref-1)</sup>，使内核层无需关心具体平台如何解析 FDT。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 内存管理模块以单任务运行时（singletask）为核心假设，通过固定大小的 VMA 快照（VmaSnapshot）和页表直接操作接口（ActiveUserMemory）构建用户态虚拟地址空间。与多进程的 ArceOS 基线不同，本仓库将所有用户内存状态集中到全局的 UserMemorySlot 数组中，并依赖原子变量实现 VMA 元数据的内核态-用户态共享。系统调用层通过 UserMemoryMapper/Reader/Writer 三个 trait 将架构无关逻辑与 RISC-V 页表操作解耦，COW 和共享文件页面的缺页解析路径统一收敛到 resolve_active_user_copy_fault。

### 1. 核心抽象与单任务内存模型

本模块的核心假设是系统一次只运行一个用户任务（singletask runtime），因此所有用户内存管理状态都集中在全局的 `UserMemorySlot` 结构体中，而非 per-process 的独立地址空间对象。`UserMemorySlot` 通过一组 `AtomicUsize` 字段维护 VMA 元数据（起始、结束、权限、backing 类型、文件索引等），`VmaSnapshot` 则在每次操作时从这些原子变量中加载一份全量快照，再修改后写回 <sup>[1](#mod-mm-ref-1)</sup>。这种“读快照-修改-写回”模式避免了持有锁跨多个 slot 字段，但要求调用方在快照生存期内不能持有任何可能阻塞的引用。

`VmaSnapshot` 内部使用 `[usize; VMA_SLOT_COUNT]` 这样的固定长度数组，而非动态链表或红黑树 <sup>[1](#mod-mm-ref-1)</sup>。这是一个显式的“用容量换确定性”的取舍：最大 VMA 数量在编译期确定，任何插入超出容量的操作都会被静默丢弃（push_sorted 中 `if self.len >= self.entries.len() { return; }`）<sup>[2](#mod-mm-ref-2)</sup>。对于嵌入式或轻量场景这可以接受，但如果用户程序通过 mmap 创建大量匿名映射，内核会静默丢失 VMA 记录，导致后续缺页或 unmap 行为不一致。

`AddressSpace` 结构体仅持有 `HardwareRootReadiness` 和 `KernelGlobalMappings`，前者可能处于 `NotReady` 状态（帧分配失败时）<sup>[3](#mod-mm-ref-3)</sup>。这种设计将页表根的分配延迟到首次需要时，但 `NotReady` 变体没有被调用方妥善处理的风险——例如 `sys_mmap` 路径中直接调用 `ActiveUserMemory` 的方法，若帧分配器未就绪则返回 `UserMapError::NotReady`，进而映射为 `ENODEV`，这在 Linux 语义中不常见。

### 2. 地址空间构建与用户加载段

用户程序的加载通过 `UserAddressSpacePlan` 和 `UserLoadSegment` 完成。`UserAddressSpacePlan::new` 在构造时执行大量校验：text 段必须可读可执行不可写，stack 段必须可读可写不可执行，entry 点必须在 text 段内，初始栈指针必须在 stack 范围内 <sup>[4](#mod-mm-ref-4)</sup>。这些校验在编译期（`const fn`）执行，但 `const fn` 内的 `Result` 无法传播 panic，因此错误路径依赖于调用方检查返回值——如果调用方 `unwrap()` 则会 panic。

`UserLoadSegment` 的 `page_copy` 方法计算从源 ELF 到目标页面的逐页复制参数 <sup>[5](#mod-mm-ref-5)</sup>。它通过 `max_usize`/`min_usize` 精确处理文件大小不足一页的场景，并检查地址溢出。但 `source_base` 默认参数为 0，调用者若传入非零基地址可能导致 `PayloadRangeOutsideSource` 误判（`source_end > source_len` 检查依赖 `source_base` 与 `file_end` 的和）。

`UserStackInitPages` 最多允许 `MAX_USER_STACK_INIT_PAGES`（18）个预初始化页，超过则返回 `TooManyStackInitPages` <sup>[6](#mod-mm-ref-6)</sup>。这个限制直接来自 `address_space.rs` 的常量，但未在文档中解释选择 18 的缘由——可能是为了覆盖 argv/envp 的典型大小，但硬限制在极端场景下可能成为隐蔽的故障点。

### 3. 跨模块协同：缺页路径与系统调用

缺页处理是跨文件协同最密集的区域。用户态访问未映射页时，`ActiveUserMemory` 的 `read_user`/`write_user` 方法（架构相关）首先通过 `translate_user_access_from_root` 遍历当前 SATP 指向的页表 <sup>[7](#mod-mm-ref-7)</sup>。如果页表项不存在或权限不足，则回退到 `resolve_active_user_copy_fault`，该函数调用 `single_classify_user_page_fault`（在 `memory_runtime.rs` 中）来在 VMA 快照中查找地址所属区域 <sup>[8](#mod-mm-ref-8)</sup>。

`single_classify_user_page_fault` 的返回结果直接指导后续操作：如果是 `MappedRegion` 且 backing 支持 DemandZero，就调用 `map_active_zeroed_user_pages_with_flags` 分配物理页并填入页表；如果是 COW 页，则调用 `resolve_active_cow_user_page` 复制帧并更新 PTE <sup>[9](#mod-mm-ref-9)</sup>。这种设计将 VMA 元数据（权限、backing 类型）与页表操作分离，使得架构无关的缺页策略逻辑集中在 `memory_runtime.rs`，而页表遍历和帧分配留在 `user_memory.rs`。

系统调用层（`syscall/memory.rs`）通过三个 trait 与底层交互：`UserMemoryMapper` 提供 map/unmap/protect，`UserMemoryReader` 和 `UserMemoryWriter` 提供拷贝。例如 `sys_brk` 调用 `single_register_user_vma` 更新 VMA 快照，然后调用 `memory.unmap_user_pages` 执行实际的页表解除 <sup>[10](#mod-mm-ref-10)</sup>。这种双层架构确保系统调用逻辑可以同时用于单任务和未来可能的多进程模型——但当前实现中 `ActiveUserMemory` 是唯一的具象类型，trait 的泛化尚未被充分利用。

### 4. 边角细节与不足

- **<sup>[1](#mod-mm-ref-1)</sup> 固定容量 VMA 数组**：`VMA_SLOT_COUNT` 编译期常量，当用户进程 VMA 数量超过此值时，`push_sorted` 静默丢弃条目。无日志或错误返回，可能导致内存映射状态与内核记录不一致。应至少提供 `debug_assert!` 或返回 `Result`。
- **<sup>[2](#mod-mm-ref-2)</sup> 缺少 `active_user_root` 失败传播**：`active_user_root` 在 SATP 模式不为 8 时返回 `Err(UserMapError::NotReady)` <sup>[11](#mod-mm-ref-11)</sup>。但某些调用点（如 `map_active_user_page_with_bytes`）在获取根帧后直接 `unwrap` 或 `?`，若帧分配器未初始化则整个系统调用返回 `ENODEV`，这在 Linux ABI 中语义不匹配。
- **<sup>[8](#mod-mm-ref-8)</sup> `resolve_active_user_copy_fault` 无条件 fallback 到 DemandZero**：即使 VMA 的 backing 不是 `DemandZero`（例如 `UserFaultBacking::SharedFile`），代码最后也会尝试 `map_active_zeroed_user_pages_with_flags` <sup>[12](#mod-mm-ref-12)</sup>。这会错误地在共享文件区域填入匿名零页，导致后续共享文件写回时数据丢失。该路径仅在 `backing.can_zero_fill()` 为 false 时提前返回 `NotMapped`，但若 backing 是 `Cow`（需要 COW 解析）也可能被错误覆盖。
- **<sup>[4](#mod-mm-ref-4)</sup> `UserAddressSpacePlan::new` 的 `const fn` 限制**：虽然构造器是 `const fn`，但内部调用了 `text.overlaps(stack)` 等非 `const fn`（因为 `overlaps` 是 `const fn`，但 `region_overlaps_kernel_globals` 可能不是）。实际代码中 `region_overlaps_kernel_globals` 是普通函数，因此 `const fn` 标注具有误导性——编译器会在非 const 上下文中退化为运行时调用，但 const 上下文中会编译失败。
- **<sup>[10](#mod-mm-ref-10)</sup> `sys_brk` 中 `single_set_program_break` 与 VMA 操作的原子性**：`sys_brk` 先检查 `single_register_user_vma` 再调用 `single_set_program_break`，但两者不是原子操作。如果在注册 VMA 后、设置 program_break 前发生线程切换（即使单任务运行时也可能在中断处理中触发），`single_classify_user_page_fault` 可能看到不一致的状态。
- **未发现明显的 TODO 注释**，但 `user_memory.rs` 中 `resolve_active_shared_file_write_user_page` 在入口处检查 `if pte_is_cow(raw)` 返回 `Unsupported`，缺少对 PTE 已经是可写的情况的处理——可能遗漏了 page 已 dirty 但未标记的场景。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块以固定槽位（EDF_TASK_CAPACITY=64）的无锁 EDF 调度器为核心，所有任务状态存储在全局静态 AtomicUsize 数组中，避免锁竞争。调度器支持 6 种任务状态和 19 种等待类型（子进程退出、管道、futex、信号量等），通过独立的等待槽位数组实现唤醒。进程控制系统调用兼容层实现了 setpgid、get/set priority、capabilities、pidfd 等数十个 Linux 系统调用，使用全局单例函数（如 single_pid、credentials_for_process）访问内核状态。与同家族基线（ArceOS / rCore-Tutorial）相比，独特之处在于：(1) 调度策略为 EDF 而非 round-robin 或 CFS；(2) 完全无锁设计，通过原子操作维护任务状态机；(3) 等待机制异常丰富，覆盖了 Linux 常见的阻塞原语。

### 1. 核心抽象：EDF 调度器与任务槽位

调度器（`scheduler.rs`）采用最早截止时间优先（EDF）算法，将任务状态存储在全局静态数组中。`EDF_TASK_CAPACITY` 固定为 64，意味着内核最多同时管理 64 个任务 <sup>[1](#mod-task-ref-1)</sup>。每个任务在数组中占据一个槽位（slot），通过 `TASK_PID`、`TASK_STATE`、`TASK_DEADLINE_MICROS` 等并行数组记录属性，所有数组均为 `AtomicUsize`，通过 `Ordering::Acquire`/`Release` 保证无锁下的顺序一致性 <sup>[2](#mod-task-ref-2)</sup>。任务状态机包含 `EMPTY`、`PREPARED`、`RUNNABLE`、`RUNNING`、`SLEEPING`、`WAITING` 六种状态 <sup>[3](#mod-task-ref-3)</sup>，其中 `WAITING` 进一步细分为 19 种等待类型（如 `WAIT_CHILD_EXIT`、`WAIT_FUTEX`、`WAIT_SOCKET_READABLE` 等），每种等待类型拥有独立的等待槽位数组（例如 `CHILD_EXIT_WAIT_SLOTS`）<sup>[4](#mod-task-ref-4)</sup>。这种设计将等待队列与调度器核心解耦：任务通过 `plan_wait_child_block` 等函数进入等待队列，调度器在上下文切换时仅需检查 `TASK_WAIT_KIND` 和 `TASK_WAIT_KEY0/1` 字段即可确定阻塞原因。EDF 调度策略通过 `pick_next_task` 实现，该函数扫描所有槽位，选择 `deadline_micros` 最小的 `RUNNABLE` 任务 <sup>[5](#mod-task-ref-5)</sup>。值得注意的是，`configure_deadline_task` 强制要求 `runtime <= relative_deadline <= period`，否则返回 `InvalidDeadline`，这是对 EDF 可调度性条件的编译期检查（尽管是运行时）<sup>[6](#mod-task-ref-6)</sup>。

### 2. 进程控制系统调用兼容层

`process_control.rs` 实现了数十个 Linux 进程控制系统调用，包括进程组/会话管理（`sys_setpgid`、`sys_getsid`）、用户/组 ID 操作（`sys_getuid`、`sys_setresuid`）、capability 访问（`sys_capget`、`sys_capset`）、优先级（`sys_getpriority`、`sys_ioprio_set`）、personality（`sys_personality_process`）、pidfd 操作（`sys_pidfd_open`、`sys_pidfd_send_signal`）等。每个系统调用处理函数遵循统一的模式：从 `SyscallFrame` 解析参数，调用全局单例函数（如 `single_pid`、`credentials_for_process`、`single_set_process_group`）访问内核状态，最后将错误转换为 Linux errno <sup>[7](#mod-task-ref-7)</sup>。一个关键设计选择是：全局单例函数（如 `credentials_for_process`）直接返回进程的凭据对象，而不进行错误检查。例如，`sys_getuid` 直接调用 `credentials_for_process(current_pid.value()).real_uid()`，若进程不存在则可能 panic 或返回垃圾值 <sup>[8](#mod-task-ref-8)</sup>。类似地，`sys_setfsuid` 在更新凭据时使用 `let _ = replace_process_credentials(...)` 忽略了替换可能失败的事实 <sup>[9](#mod-task-ref-9)</sup>。这些点表明系统调用兼容层假设全局单例函数总是成功，这在单任务测试中可能成立，但在多任务或竞争条件下可能引发生成问题。此外，capability 操作涉及用户态内存读写（通过 `UserMemoryReader`/`UserMemoryWriter` trait），`sys_capget` 和 `sys_capset` 在读写用户指针前进行了严格的空指针和越界检查 <sup>[10](#mod-task-ref-10)</sup>。

### 3. 跨模块协同：调度器、进程管理与 procfs

任务创建与销毁需要调度器与进程管理模块的紧密协作。虽然进程创建（clone/fork）的核心逻辑在别处实现，但调度器提供了 `register_task`、`remove_task`、`configure_deadline_task` 等接口供进程管理调用。例如，进程退出时需调用 `remove_task` 清除调度器槽位和等待状态 <sup>[11](#mod-task-ref-11)</sup>。上下文切换由 `plan_context_switch` 生成计划，该函数验证当前和下一个任务的 `saved_stack_pointer_slot` 和 `kernel_stack_pointer` 不重复，防止栈损坏 <sup>[12](#mod-task-ref-12)</sup>。procfs 模块（`proc.rs`）通过调用 `current_proc_snapshot()` 获取当前进程的快照，生成 `/proc/self/status`、`/proc/self/stat` 等文件内容 <sup>[13](#mod-task-ref-13)</sup>。`current_proc_snapshot` 内部调用 `single_pid()` 和 `single_proc_snapshot(pid)`，后者返回 `SingleProcSnapshot`，包含进程 ID、父 PID、是否已退出等信息 <sup>[14](#mod-task-ref-14)</sup>。procfs 与调度器的交互是单向的：procfs 读取调度器状态（如通过 `snapshot()` 获取任务计数），但从不修改调度器数据。这种解耦使得 procfs 的实现相对简单，但也意味着 procfs 显示的 `procs_running` 等字段是调度器状态的瞬时快照，可能不精确 <sup>[15](#mod-task-ref-15)</sup>。

### 4. 边角细节与不足

该模块在设计上做出了若干取舍，但也暴露出一些值得关注的风险点：

**<sup>[8](#mod-task-ref-8)</sup> `credentials_for_process` 无错误处理风险**：`sys_getuid`、`sys_geteuid` 等函数直接调用 `credentials_for_process(current_pid.value()).real_uid()`，未检查返回的凭据是否有效。若 `credentials_for_process` 因进程不存在而 panic 或返回默认值，将导致信息泄露或静默错误 <sup>[8](#mod-task-ref-8)</sup>。

**<sup>[9](#mod-task-ref-9)</sup> `sys_setfsuid` 忽略替换错误**：在 `sys_setfsuid` 中，`let _ = replace_process_credentials(...)` 丢弃了返回值 <sup>[9](#mod-task-ref-9)</sup>。`replace_process_credentials` 返回 `bool`，若替换失败（如内存不足），错误将被无声地吞掉，导致进程认为操作成功但实际上凭据未更新。

**<sup>[3](#mod-task-ref-3)</sup> 调度器槽位耗尽**：`EDF_TASK_CAPACITY` 硬编码为 64，若系统同时运行超过 64 个任务，`register_task` 将返回 `QueueFull` 错误 <sup>[1](#mod-task-ref-1)</sup><sup>[3](#mod-task-ref-3)</sup>。当前代码中许多调用方未处理此错误（例如 clone 系统调用可能直接忽略）。在典型的工作负载下 64 可能足够，但恶意用户可通过 fork bomb 耗尽槽位。

**<sup>[4](#mod-task-ref-4)</sup> 等待槽位竞争**：每种等待类型使用独立的等待槽位数组，数组大小同为 `EDF_TASK_CAPACITY`。当多个任务等待同一资源（如同一个 futex 地址）时，可能发生槽位竞争。调度器使用 `SYSV_SEMAPHORE_WAIT_ANY` 等方式处理信号量，但缺乏 CAS 循环的详细注释，可能存在 ABA 问题 <sup>[4](#mod-task-ref-4)</sup>。

**<sup>[6](#mod-task-ref-6)</sup> EDF 参数验证不完整**：`configure_deadline_task` 验证 `runtime <= relative_deadline <= period`，但未检查 `runtime_micros` 是否合理（如是否小于一个时钟 tick）。若 runtime 设为 1 微秒而时钟周期为 10ms，调度器可能频繁抢占，导致活锁 <sup>[6](#mod-task-ref-6)</sup>。此外，`configure_normal_task` 使用了硬编码的 `DEFAULT_TASK_PERIOD_MICROS`（10ms），对于 I/O 密集型任务可能过短。

**<sup>[12](#mod-task-ref-12)</sup> 上下文切换计划中的竞争**：`plan_context_switch` 检查 `saved_stack_pointer_slot` 和 `kernel_stack_pointer` 是否重复，但这些检查在读取后到实际切换前可能失效（TOCTOU）。不过由于调度器是单核假设（全局静态数组暗示无 SMP 支持），这种竞争可能性较低，但若未来扩展为多核，则存在隐患 <sup>[12](#mod-task-ref-12)</sup>。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本仓库的文件系统采用分层VFS架构，核心抽象是`source/core/fs/vfs.rs`中定义的`SyscallVfs` trait，它将系统调用与具体文件系统后端解耦。从下到上依次为：ext4只读根文件系统（`RootExt4Volume`）、纯内存的`WritableOverlay`可写覆盖层、以及编译期硬编码的builtin虚拟文件系统（procfs/devfs/sysfs）。`source/kernel/syscall_runtime/vfs_backend.rs`中的`ActiveSyscallVfs`实现了该trait，并在其中串联权限检查、copy-up-on-write、inotify通知、文件锁等完整机制。与同家族基线（ArceOS）相比，本仓最独特的是采用"ext4-only根文件系统+内存覆盖层"双栈模型，而非多后端VFS；builtin层模拟了大量`/proc/sys`内核参数，但所有数据编译期固定，不支持运行时写入。

### 1. 核心抽象：SyscallVfs trait 与三层文件栈

文件系统模块的核心抽象是 `SyscallVfs` trait（`source/core/fs/vfs.rs:547-...`），它定义了约60多个方法，覆盖路径解析、打开/读写、元数据操作、xattr、挂载、epoll、signalfd、timerfd 等全部文件相关系统调用。`SyscallVfs` 的设计目标是让系统调用分发层（`syscall_runtime`）完全与后端解耦——其默认方法大多数返回 `Err(VfsError::UnsupportedPath)` 或 `Err(VfsError::RootfsSourceMissing)`，后端可以选择性地覆盖 <sup>[1](#mod-fs-ref-1)</sup>。

实际运行时的实现是 `ActiveSyscallVfs`（`source/kernel/syscall_runtime/vfs_backend.rs:23`），一个零大小结构体 `#[derive(Clone, Copy)]`。它不直接持有状态，而是通过 `active_overlay_mut()`、`active_fd_table_mut()`、`active_rootfs_mut()` 等一系列 `active_*` 访问器获取全局可变状态——这是一种隐式依赖注入模式，将状态管理外移到 `PerCpu` 或 `TaskLocal` 存储中 <sup>[2](#mod-fs-ref-2)</sup>。与此对照，`NoSyscallVfs`（同文件行 ~1200）作为"空后端"存在，当未挂载根文件系统时所有方法返回 `RootfsSourceMissing` 或 `BadFileDescriptor`，确保早期引导阶段不会悄悄吞掉错误。

文件栈本身是三层结构，在 `source/core/fs/vfs.rs:41-47` 通过 `mod builtin`、`mod overlay`、`mod proc` 等子模块组织：
- **Ext4 只读层**：`RootExt4Volume`（`source/core/fs/vfs.rs:2400-2485`）封装了 `Ext4Volume`，使用 `BlockCache` 和 `BlockProvider` 实现块设备访问。实际文件数据只读，不可写。
- **WritableOverlay 层**：`pub use overlay::WritableOverlay` 是一个纯内存的可写覆盖层，存储目录条目、文件块（4096字节）、白名单（whiteout）和扩展属性。其容量通过 `VFS_OVERLAY_DIR_CAPACITY=128`、`VFS_OVERLAY_FILE_CAPACITY=128`、`VFS_OVERLAY_FILE_BLOCKS_PER_FILE=2048` 等编译期常量严格限定 <sup>[3](#mod-fs-ref-3)</sup>。
- **Builtin 虚拟层**：`mod builtin` 和 `mod proc` 实现了大量 `VFS_BUILTIN_PROC_*_INODE` 常量（`0x7000_0001` 到 `0x7000_007e`），对应 `/proc`、`/dev`、`/sys` 下的静态文件。例如 `PROC_CPUINFO_BYTES` 根据 `#[cfg(target_arch)]` 条件编译不同架构的 CPU 信息 <sup>[4](#mod-fs-ref-4)</sup>。

### 2. 关键设计取舍：copy-up-on-write、硬编码builtin 与 单根文件系统

**取舍一：copy-up-on-write vs. 直接写入ext4**

当用户程序写入一个原本来自 ext4 根文件系统的文件时，`open_path` 方法（`source/kernel/syscall_runtime/vfs_backend.rs:310-390`）先检查 overlay 是否持有该文件。若没有，调用 `copy_up_rootfs_regular()` 将 ext4 中的数据逐块复制到 overlay 层，然后才在 overlay 上打开写 <sup>[5](#mod-fs-ref-5)</sup>。这种设计放弃了"原地修改ext4"的能力——ext4 始终保持只读基线，所有写入都是 overlay 层的新副本。代价是内存膨胀（每个被修改的文件都需要复制），好处是实现简单且无需处理写时 ext4 的日志/一致性。

**取舍二：builtin 文件系统全部编译期硬编码**

所有 `/proc/cpuinfo`、`/proc/net/tcp`、`/proc/sys/kernel/hostname` 等虚拟文件的数据都以 `&[u8]` 常量形式编译进内核（`source/core/fs/vfs.rs:300-500`）。例如 `PROC_SYSCTL_HOSTNAME_BYTES` 固定为 `b"uestc-source\n"`——这意味着 `/proc/sys/kernel/hostname` 的 `write` 操作不被支持。虽然 `VFS_BUILTIN_WRITABLE_REG_MODE = 0o100644` 暗示某些 builtin 文件拥有写权限位，但实际覆盖 `builtin_write` hook 的路径有限 <sup>[6](#mod-fs-ref-6)</sup>。这种设计放弃了动态 procfs/sysfs 的能力，换取了零运行时开销和确定性行为。与 Linux 完整实现相比，`sysctl` 修改无法持久化，部分依赖于 procfs 配置的应用程序将无法正常工作。

**取舍三：单根文件系统 vs. 多挂载点 namespace**

`VFS_MOUNT_CAPACITY` 被设为 8，限制最多同时挂载 8 个文件系统实例 <sup>[3](#mod-fs-ref-3)</sup>。挂载表 `VfsMountTable` 存储在 `mount` 子模块中，但实际只有根文件系统被挂载——`mount()` 系统调用仅支持 `"ext4"` 和 `"bind"` 两种类型。这与 Linux VFS 的通用挂载模型相距甚远，但对于单进程沙盒场景（此仓库的目标场景）已足够。

### 3. 跨模块协同：syscall 分发 → VFS 后端 → 各层 I/O

分布式协作的关键路径体现在以下三个层面：

**(a) 路径解析与权限检查链**：当用户调用 `openat` 时，系统调用分发层调用 `ActiveSyscallVfs::open_path_at()`，该函数调用 `resolve_runtime_path_from_active_cwd()` 解析路径（涉及当前工作目录 `active_cwd`、符号链处理 `RuntimeFinalSymlink`）。然后通过 `authorize_active_open()` 执行 `VFS_ACCESS_*` 权限检查——它借用 `ProcessCredentials`（来自 task 模块）。若创建新文件，还需 `active_creation_mode_and_gid()` 决定创建 mode 和 gid。最终，实际 I/O 分发到三层中的一个：builtin（`open_builtin_link_path`）、overlay（`active_overlay_mut().open()`）或 ext4（`active_rootfs_mut().open_with_options()`）<sup>[7](#mod-fs-ref-7)</sup>。

**(b) inotify 通知集成**：`vfs_backend.rs` 的 `notify` 子模块提供了 `notify_inotify_path()` 和 `notify_inotify_description()` 函数。几乎每个修改操作（create、unlink、rename、write、truncate、chmod等）都会在操作前后调用这些函数。`wake_socket_readable` 和 `wake_socket_writable` 来自 `source/core/syscall/scheduler`（跨模块调用），说明 inotify 事件醒来与 scheduler 的等待队列协同 <sup>[8](#mod-fs-ref-8)</sup>。

**(c) 文件锁与 scheduler**：`flock()` 和 `fcntl()` 记录锁的实现中，操作完成后调用 `scheduler::wake_flock_waiters(key.inode(), key.kind())`——文件锁释放时主动唤醒等待者。这避免了轮询，但引入了对 scheduler 模块的编译时依赖 <sup>[9](#mod-fs-ref-9)</sup>。

### 4. 边角细节与不足

**不足 1：builtin inode 空间分配缺少冲突检测** <sup>[4](#mod-fs-ref-4)</sup> 定义了大量 `VFS_BUILTIN_PROC_*_INODE` 常量，从 `0x7000_0001` 到 `0x7000_007e`，中间跳跃式分配（例如跳过 `0x7000_006c-006f`）。但不存在任何编译期断言检查是否有重复或超出 `VFS_BUILTIN_FD_LINK_INODE_BASE` 范围。新增内置文件时，开发者必须手动检查常量是否冲突。

**不足 2：`ActiveSyscallVfs` 是无状态零大小结构体** <sup>[2](#mod-fs-ref-2)</sup> 所有访问都通过全局 `active_*()` 函数获取可变引用。这种模式意味着：1) 不可测试——无法轻松 mock 或替换 overlay；2) 在单线程模型下虽可行，但若未来引入多线程，这些 `active_*()` 函数内部使用 `RefCell` 或 `Mutex` 的方式会成为并发瓶颈。当前没有 `// SAFETY:` 注释或同步原语可见。

**不足 3：overlay 容量硬编码可能导致 DoS** <sup>[3](#mod-fs-ref-3)</sup> `VFS_OVERLAY_FILE_CAPACITY=128`、`VFS_OVERLAY_FILE_BLOCKS_PER_FILE=2048` 等常量决定可写层总存储为 128 文件 × 2048 块/文件 × 4096 字节/块 = 1GiB。但多打开几个写文件即可快速耗尽目录或文件槽位——用户进程可以通过触发大量 copy-up 消耗内存。没有逐出策略或 OOM 回退。

**不足 4：`unsafe` 块缺失安全契约** <sup>[5](#mod-fs-ref-5)</sup> `copy_up_rootfs_regular()` 内部会操作 overlay 的块表和 page cache，部分路径通过 `active_overlay_mut()` 获得 `&mut` 引用。该函数位于 `vfs_backend.rs` 中，但其中存在文件 I/O 和内存拷贝，没有与之相关的 `// SAFETY:` 注释。如果 overlay 内部 `FileBlock` 索引越界（`VFS_OVERLAY_FILE_BLOCK_NONE = usize::MAX` 作为哨兵），可能造成未定义行为。

**不足 5：cfg gate 背后的架构假设** <sup>[10](#mod-fs-ref-10)</sup> `PROC_CPUINFO_BYTES` 使用 #[cfg(target_arch = "riscv64")] 和 `#[cfg(target_arch = "loongarch64")]` 条件编译，默认分支是 `"model name: uestc source virtual cpu"`。这意味着如果编译目标为 aarch64、x86_64 以外的架构（如 mips、powerpc），编译不会报错，但 `/proc/cpuinfo` 会显示错误的默认值——这是一个静默语义错误。

**不足 6：`stat_runtime_path_no_follow()` 在错误路径中 unwrap** <sup>[7](#mod-fs-ref-7)</sup> 在 `open_path` 中调用 `open_builtin_link_path()` 成功后，使用 `active_fd_table_mut().get().map_err` 获取描述符，但紧接着调用 `notify_inotify_path` 时传入 `description.is_directory()`——这里如果 description 为 `None`（实际上面已经检查过 Ok 分支），会 panic。严格来说这不是问题，但用 `map` 链式处理更安全。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制模块基于 `SignalState` 状态机实现 POSIX 信号的核心语义，使用 `usize` 位图管理最多 64 个信号的屏蔽与挂起，通过 `SignalDelivery` 类型保证递送过程的原子性。与同家族基线（ArceOS / rCore）相比，本仓库在架构适配层显式区分了 LoongArch64 和 RISC-V，信号帧内嵌了完整的 ucontext 和 trap frame 保存/恢复，并支持通过 `rt_sigreturn` trampoline 实现用户态信号返回。其独特代价是架构相关的常量手动计算，缺乏自动化布局验证。

### 1. 核心抽象与状态机

信号机制的核心是每个任务（线程/进程）持有的 `SignalState` 结构体<sup>[2](#mod-signal-ref-2)</sup>。它封装了四个关键域：动作表（`actions: [SignalAction; 65]`，索引 1-64，下标 0 未使用）、信号栈（`stack: SignalStack`）、阻塞掩码（`blocked_mask: usize`）和挂起掩码（`pending_mask: usize`）。每个信号的动作由 `SignalAction` 描述，包含 `handler`（用户函数地址、`SIG_DFL=0` 或 `SIG_IGN=1`）、`flags`（`SA_ONSTACK`、`SA_NODEFER`、`SA_RESETHAND`）、`restorer` 和 `mask`。挂起信号附带 `QueuedSignalInfo`，可记录发送者 PID、UID、code 和 value，这些信息在写入用户态 siginfo 时被布局到信号帧中<sup>[5](#mod-signal-ref-5)</sup>。

信号递送通过 `begin_delivery()` 触发，该方法从挂起信号中选出优先级最高的信号（通过 `first_signal_in_mask` 使用 `trailing_zeros`），计算出 `SignalEffect`（Ignore/Pending/Handle/Terminate/Stop/Continue），然后更新阻塞掩码（加入 action 的 mask 和当前信号自身，除非设置了 `SA_NODEFER`）<sup>[3](#mod-signal-ref-3)</sup>。如果设置了 `SA_RESETHAND`，递送后将动作重置为 `SIG_DFL`。`complete_delivery()` 负责恢复之前的阻塞掩码，`abort_delivery()` 则恢复掩码并重新挂起信号。这种设计确保了递送过程中断（例如信号帧写入失败）不会丢失信号状态<sup>[2](#mod-signal-ref-2)</sup>。

### 2. 关键设计取舍

**位图掩码 vs. 完整 sigset_t**：本仓库使用 `usize`（64 位）作为掩码，`SIGNAL_COUNT` 固定为 64<sup>[1](#mod-signal-ref-1)</sup>。这与 Linux 的 64 位 sigset_t 一致，但省略了实时信号（SIGRTMIN=32..64）。选择位图而非全数组判断（如遍历 1..=SIGNAL_COUNT）使 `first_signal_in_mask` 只需一条 `trailing_zeros` 指令，性能优势明显，代价是失去了对 >64 信号的扩展能力。

**不可变信号**：`IMMUTABLE_SIGNAL_MASK` 硬编码屏蔽了 `SIGKILL` 和 `SIGSTOP`<sup>[11](#mod-signal-ref-11)</sup>。`sanitize_signal_mask()` 始终清除这两位，因此即使 `rt_sigprocmask` 尝试阻塞这两个信号也会被静默忽略<sup>[12](#mod-signal-ref-12)</sup>。`validate_mutable_signal()` 在修改动作时直接返回 `ImmutableSignal` 错误。这是对 POSIX 规范的精确遵守——即 SIGKILL 和 SIGSTOP 不能被捕获或阻塞——但在实现上，`sanitize_signal_mask` 的“静默清除”策略比 Linux 内核的“直接返回错误”更宽容，可能导致用户态感知不到其意图被覆盖。

**递送原子性与交付帧**：架构相关的 `apply_pending_before_user_return()` 在 trap 返回用户空间前检查挂起信号<sup>[7](#mod-signal-ref-7)</sup>。它调用 `single_begin_signal_delivery`（当前任务的单例）以原子方式取出一个待递送信号。如果效果是 `Handle`，则调用 `write_delivery_frame()` 在用户栈上构建信号帧<sup>[5](#mod-signal-ref-5)</sup>。帧结构包含：magic 字（用于完整性检查）、信号编号、previous_mask、restorer 地址、siginfo 块、ucontext（含栈快照和寄存器集）、完整的 trap frame 副本，以及 trampoline（当用户未提供 restorer 时使用）。`install_signal_handler` 将 trap frame 的 `era`（epc）设为 handler，寄存器 `$ra`/`$r1` 设为 restorer，`$sp` 设为帧地址，`$a0`/`$a1`/`$a2` 设为信号编号、siginfo 和 ucontext 地址<sup>[6](#mod-signal-ref-6)</sup>。这种设计避免了在用户态准备额外的信号帧分配——内核直接在用户栈上布局，消除了内核-用户之间的数据复制。

**Trampoline 的可执行化**：当用户未提供 restorer (0) 时，trampoline 嵌入栈帧末尾。`make_trampoline_executable` 将 trampoline 所在的页面标记为 rwx<sup>[5](#mod-signal-ref-5)</sup>。这是权衡后的选择：虽然将栈页标记为可执行违反 W^X，但 trampoline 代码短小（8 字节 `addi.d $a7, $zero, 139; syscall`）且生命周期与信号帧相同。RISC-V 和 LoongArch64 分别在 `sync_trampoline` 中使用 `fence.i` 和 `ibar` 保证指令一致性<sup>[5](#mod-signal-ref-5)</sup>。对于其他架构，同步函数为空，这是一个潜在的隐患。

### 3. 跨模块协同：从系统调用到 trap 处理

系统调用层（`source/core/syscall/signal/`）负责参数校验和路由。例如 `sys_kill_process` 解析 PID 和信号，通过 `validate_kill_target` 区分进程、进程组和全部进程<sup>[13](#mod-signal-ref-13)</sup>。从 kill 到实际信号挂起经历了多层间接：对于当前进程，直接调用 `process.apply_signal()`；对于远程进程，调用 `single_apply_signal_to_pid()` 等单例函数，这些函数内部会访问目标进程的 `SignalState` 并调用 `apply_signal_info`。

关键的单例层（`single_*` 函数，如 `single_begin_signal_delivery`）承担了“当前任务”的访问抽象。由于本仓库支持多线程但使用单核心模型（推测），`single_` 前缀的函数提供对当前 CPU/线程的 `SignalState` 的访问。`apply_pending_before_user_return` 在每次 trap 返回用户空间时被调用，从而在不侵入调度器的情况下实现信号递送。

信号返回由 `rt_sigreturn` 系统调用处理。架构相关的 `consume_rt_sigreturn` 从用户栈读取信号帧，校验 magic 字，恢复 trap frame，并调用 `single_complete_signal_delivery(previous_mask)` 回复阻塞掩码<sup>[8](#mod-signal-ref-8)</sup>。注意它使用 `ReturnToUserWithoutSignalCheck` 标志，避免在返回后立即再次检查挂起信号（防止无限循环）。

跨模块协同最完整的体现是 `SignalDelivery` 的生命周期：`begin_delivery`（task/signal.rs）→ `write_delivery_frame`（arch/signal.rs）→ 用户 handler 执行 → `rt_sigreturn`（arch/signal.rs）→ `complete_delivery`（task/signal.rs）。这一闭环完全由内核控制，用户代码仅在 handler 中执行。

### 4. 边角细节与不足

- **信号帧布局依赖硬编码常量**：架构信号文件中充满了手动计算的偏移量（如 `SIGNAL_UCONTEXT_OFFSET`、`LINUX_UCONTEXT_MCONTEXT_OFFSET`）<sup>[5](#mod-signal-ref-5)</sup>。这些值基于结构体大小的编译时猜测，如果未来调整 `QueuedSignalInfo` 的字节数或添加新字段，需要同步更新所有架构常量，否则将导致用户栈数据错位。`SIGNAL_FRAME_MAGIC` 虽能检测严重错位，但若偏移量碰巧导致部分覆盖，则难以发现。

- **`SIGNAL_RESTORER_REQUIRED = false`**：两个架构都将该常量设为 `false`<sup>[10](#mod-signal-ref-10)</sup>，意味着 `begin_delivery` 不会因 restorer 缺失而返回错误。这与 Linux 的 `SA_RESTORER` 约定不同——即使用户未提供 restorer，内核会在栈上插入 trampoline。但若用户显式提供了非法 restorer，`restorer_address` 会直接使用它，后续 `make_trampoline_executable` 不会被执行，导致 `rt_sigreturn` 时跳转到非法地址。应当增加 restorer 的有效性检查（至少验证指向用户空间且可执行）。

- **`SIGSTOP`/`SIGCONT` 未实现**：`signal_default_effect` 对 `SIGSTOP`/`SIGTSTP` 等返回 `Stop`，对 `SIGCONT` 返回 `Continue`<sup>[3](#mod-signal-ref-3)</sup>。但在 `signal_effect_outcome_remote` 中，`Stop` 和 `Continue` 均返回 `NotSupported`<sup>[9](#mod-signal-ref-9)</sup>。这意味着试图通过 kill 发送停止/继续信号会得到 `ENOSYS`，而挂起状态机完全缺失线程暂停/恢复的逻辑。这是一个未完成的半成品区域。

- **`SA_ONSTACK` 使用不完整**：`SignalDelivery::uses_signal_stack()` 依赖 `SA_ONSTACK` 标志<sup>[2](#mod-signal-ref-2)</sup>，`signal_stack_top` 在 deliver 时会检查信号栈是否有效<sup>[5](#mod-signal-ref-5)</sup>。但 `SignalStack::new` 的验证中，如果 `SS_ONSTACK` 被设置则返回 `CurrentlyOnStack` 错误，这意味着用户不能通过 `sigaltstack` 设置 `SS_ONSTACK`——该标志应由内核在用户当前在信号栈上执行时设置，而不能由用户主动请求。但这里直接拒绝任何带 `SS_ONSTACK` 的配置，偏离了 POSIX 行为。

- **挂起信号信息覆盖**：`apply_signal_info` 总是覆盖 `pending_info[signal]` 的最新信息<sup>[2](#mod-signal-ref-2)</sup>。如果同一信号快速到达多次，只有最后一次的信息被保留。对于标准信号，POSIX 允许合并（即只挂起一次），但对于 `SIGCHLD` 等携带进程退出的信号，信息丢失可能导致父进程无法正确获取子进程状态。本实现未尝试排队或保留历史信息。

- **`sync_trampoline` 的空实现**：对于非 LoongArch64/RISC-V 架构（如 x86_64），编译时使用空函数<sup>[5](#mod-signal-ref-5)</sup>。如果未来适配 x86，需要实现 `wbinvd` 或 `CLFLUSH` 来确保指令缓存一致性，当前空实现可能导致信号处理程序执行损坏的 trampoline。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该仓库的进程间通信模块实现了 System V 信号量、System V 消息队列和 POSIX 消息队列三种经典 IPC 机制，均以内核态系统调用的形式暴露。核心抽象是无锁原子变量数组（`AtomicUsize`）管理的静态资源池，通过 `scheduler::WaitChannel` 与任务调度器集成实现阻塞唤醒。与 ArceOS 基线相比，该仓库完整实现了 POSIX 消息队列的名称查找和权限模型，而基线通常只提供信号量的简单 stub。实现中存在明显的并发安全问题：System V 信号量和消息队列的操作缺乏全局锁，仅依赖原子变量的非组合更新，可能在高竞争下产生数据竞争。

### 1. 核心抽象与外部依赖

进程间通信模块的核心抽象是**基于静态数组的内核对象池**。对于 System V 信号量和消息队列，分别使用 `SEM_SETS: [SemSetSlot; 16]` 和 `MSG_QUEUES: [MsgQueueSlot; 16]` 两个全局静态数组管理所有内核 IPC 对象<sup>[1](#mod-ipc-ref-1)</sup><sup>[2](#mod-ipc-ref-2)</sup>。每个槽位通过 `AtomicUsize` 字段（如 `active`, `sequence`, `key`）实现无锁状态维护。POSIX 消息队列则使用 `MessageQueueTable` 结构体，内部包含固定容量的 `MessageQueue` 数组和 `QueuedMessage` 数组，并通过 `&mut self` API 暴露操作，暗示外部需要锁保护<sup>[7](#mod-ipc-ref-7)</sup>。

模块依赖的外部接口包括：`scheduler::WaitChannel` 用于阻塞/唤醒等待信号量或消息的进程<sup>[3](#mod-ipc-ref-3)</sup>，`scheduler::wake_sysv_semaphore_waiters` 等函数在操作成功后唤醒等待者；`UserMemoryReader`/`UserMemoryWriter` trait 用于安全地读写用户空间数据<sup>[4](#mod-ipc-ref-4)</sup>；`ProcessCredentials` 用于权限检查<sup>[5](#mod-ipc-ref-5)</sup>。队列发送操作可能返回 `SyscallBlock` 让进程进入阻塞状态<sup>[6](#mod-ipc-ref-6)</sup>。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 本模块实现了Linux套接字系统调用的ABI兼容层及底层无锁套接字表。核心抽象是静态分配的SocketTable与SocketPacket池，通过系统调用解析层（syscall/socket.rs）和VFS实现层（fs/fd/socket.rs）分离实现。该设计放弃了动态堆分配和完整协议栈，以固定容量和无锁操作换取确定性，适合单线程或轻量级应用。与rcore-rust家族其他实现相比，该方案采用包池模拟数据流，而非依赖完整TCP/IP协议栈，显著降低了实现复杂度，但牺牲了网络协议兼容性。

### 1. 核心抽象：基于静态表的无锁套接字
本模块的核心是SocketTable——一个由固定容量数组构成的无锁数据结构<sup>[2](#mod-net-ref-2)</sup>。SocketTable内部分为两个区域：sockets数组用于存储套接字状态，packets数组作为数据包池。这种设计避免了动态内存分配和锁竞争，但限制了最大并发连接数和数据包总量。每个套接字可以关联一个包链表，但具体队列管理逻辑需参考后续代码（未完全展示）。SocketEndpoint枚举支持Inet和Unix域地址<sup>[2](#mod-net-ref-2)</sup>，但Inet地址仅存储IP和端口，没有真实网络协议栈驱动。SocketType区分Stream、Datagram和SeqPacket<sup>[2](#mod-net-ref-2)</sup>，但实际行为可能仅模拟本地通信。

### 2. 关键设计取舍：常量集中与容量限制
模块定义了庞大的ABI常量集合（从AF_UNIX到TCP_TX_DELAY）<sup>[1](#mod-net-ref-1)</sup>，集中管理避免了硬编码漂移，但也使模块与Linux ABI紧密耦合。固定容量设计（SOCKET_TABLE_CAPACITY和SOCKET_PACKET_POOL_CAPACITY）是双刃剑：一方面，它确保内存用量可预测，无需堆分配；另一方面，它放弃了灵活性，当负载超过容量时返回错误。包池分配采用线性扫描<sup>[3](#mod-net-ref-3)</sup>，时间复杂度O(n)，在高频创建/销毁套接字场景下可能成为瓶颈。错误处理上，函数通过返回Err并映射为Linux errno<sup>[4](#mod-net-ref-4)</sup>，但缺乏日志记录，可能导致调试困难。此外，部分常量（如SOCKET_DEFAULT_TCP_KEEPIDLE）直接硬编码<sup>[2](#mod-net-ref-2)</sup>，跨平台时可能需要参数化。

### 3. 跨模块协同：系统调用层与VFS层的接口
syscall/socket.rs中的sys_socket<sup>[4](#mod-net-ref-4)</sup>等函数解析用户态参数后，通过SyscallVfs trait调用VFS层实现。VFS层（fs/fd/socket.rs）提供socket/bind/connect等具体操作<sup>[4](#mod-net-ref-4)</sup>。一个关键协同点是套接字等待机制：sys_accept和sys_connect/recv等可能阻塞，socket_wait_outcome<sup>[5](#mod-net-ref-5)</sup>通过scheduler::WaitChannel将任务挂起，直到套接字可读/可写。WaitChannel的创建依赖SocketID<sup>[5](#mod-net-ref-5)</sup>，但该ID的生成与SocketTable的索引映射关系值得关注（需要后续代码确认）。此外，返回文件描述符时需检查nofile_limit<sup>[4](#mod-net-ref-4)</sup>，体现了与文件描述符空间的协作。系统调用层通过UserMemoryReader/Writer trait处理用户态拷贝<sup>[4](#mod-net-ref-4)</sup>，避免直接访问未验证指针。

### 4. 边角细节与不足
1. 缺少全局同步机制：SocketTable本身是纯数据结构，若在多核环境下无锁保护，多个CPU同时操作会导致数据竞争。目前未看到Mutex使用，推测当前假设为单次单线程。
2. 包池耗尽：SOCKET_PACKET_POOL_CAPACITY仅为SOCKET_TABLE_CAPACITY的一半，若大量socket同时接收数据，包池可能耗尽，且线性分配可能导致碎片<sup>[3](#mod-net-ref-3)</sup>。
3. 真正的网络协议缺失：尽管支持Inet域，但无对应网卡驱动或协议栈，套接字数据包仅在本地进程间通过包池传递，实质上是进程间通信（IPC）的简易模拟。
4. 错误路径静默：例如map_vfs_runtime_error<sup>[4](#mod-net-ref-4)</sup>仅返回errno，不记录日志，可能导致无法跟踪的静默失败。
5. 不安全的常量推导：部分常量如SOCKET_PACKET_BYTES（32KB）硬编码<sup>[2](#mod-net-ref-2)</sup>，若实际网络负载更大，需手动调整且重新编译。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该仓库的驱动框架并未提炼出独立的 trait 或总线抽象层，而是通过架构级的具体驱动（block.rs）配合调度阻塞机制（blocking.rs）和陷阱慢路径（trap_block.rs）实现设备 I/O 与进程等待的协同。核心抽象是 WaitChannel 枚举（涵盖 16 种阻塞原因）和调度器的三阶段协议（plan/commit/rollback），所有 I/O 等待统一通过该通道表达。与同家族基线 ArceOS 相比，本仓将设备发现、virtqueue 初始化、同步 I/O 全部实现在 LoongArch64 目录下，没有复用 virtuio-driver crate，且调度阻塞采用静态枚举而非动态派发，扩展性受限。

### 1. 核心抽象：等待通道枚举与阻塞三阶段协议

驱动框架的核心不在设备管理，而在进程因 I/O 而阻塞的通用机制。`source/core/scheduler/blocking.rs` 中定义了 `WaitChannel` 枚举 <sup>[1](#mod-drivers-ref-1)</sup>，它枚举了 16 种阻塞原因：ChildExit、PipeReadable、Futex、Signal 等。每个变体携带对应的资源标识符（如 pipe_id、futex_key），通过 `kind()`、`key0()`、`key1()` 三个方法映射为定长整数，用于哈希/索引操作 <sup>[2](#mod-drivers-ref-2)</sup>。这个设计将内核中所有“等待某个条件满足”的场景收敛到一个类型中，避免了为每种 I/O 设备单独实现等待队列。

配合 WaitChannel 的是三阶段阻塞协议：`plan_*` 检查当前状态并选择是否立即切换上下文；`commit_*` 将任务挂起并记录等待信息；`rollback_*` 在出错时恢复原状 <sup>[3](#mod-drivers-ref-3)</sup>。三阶段的拆分确保调度器在上下文切换前可以决定目标，且错误路径能安全回滚。`trap_block.rs` 的 `consume_blocked_syscall` 函数正是这个协议的主要消费者：它根据 `SyscallBlockReason` 构造对应的 `WaitChannelRequest`，依次调用 plan、commit，最后通过 `switch` 模块执行实际上下文切换 <sup>[4](#mod-drivers-ref-4)</sup>。

### 2. 关键设计取舍：枚举分派 vs trait 对象，同步轮询 vs 中断，全局可变状态

**取舍一：静态枚举而非 trait 对象。** `WaitChannel` 是 `#[derive(Clone, Copy)]` 的枚举，所有等待原因在编译期确定。优点是无动态分派开销、可嵌入 `SchedulerWaitChannelRequest` 这种小结构（Copy）；缺点是新增等待类型必须修改核心调度器代码，无法由外部驱动模块动态注册。这适合微内核或教学内核场景，但不适合插件式驱动框架。

**取舍二：同步轮询而非中断驱动。** `source/arch/loongarch64/block.rs` 中的 `read_sector_inner` 使用 `while (used_idx == start)` 忙等待，最多自旋 `COMPLETION_SPIN_LIMIT = 100_000_000` 次后超时返回 <sup>[5](#mod-drivers-ref-5)</sup>。这绕过了中断控制器和 MSI-X 的复杂配置，但浪费 CPU，且 boot 路径结束后无法用于性能敏感的块设备。代码中明确将 MSI-X 向量设为 `VIRTIO_MSI_NO_VECTOR` <sup>[6](#mod-drivers-ref-6)</sup>，表明作者有意放弃了中断。

**取舍三：全局可变静态变量。** `block.rs` 使用 `static mut DESC_TABLE`、`static mut AVAIL_RING` 等全局变量存储 virtqueue 状态 <sup>[7](#mod-drivers-ref-7)</sup>。所有操作都在 `unsafe` 块中进行，依赖单核/单线程假设。没有使用 `Mutex` 或 `RefCell`，这意味着该驱动不支持并发访问。这在 boot 阶段可行，但未来若启动多核或异步 I/O 路径，必须重写为 per-CPU 或锁保护。

**取舍四：PCI 设备发现直接操作 ECAM。** 代码通过硬编码 `PCI_ECAM_BASE = 0x2000_0000` 访问 PCI 配置空间 <sup>[8](#mod-drivers-ref-8)</sup>，不使用 ACPI 或设备树。这减少了依赖链，但仅适用于 QEMU virt 机器的固定地址，移植到真实硬件时需要参数化。

**取舍五：三阶段协议的重复性。** `trap_block.rs` 中每种阻塞类型（sleep、wait_child、fd_wait 等）都重复相同的 plan-match-commit 模式，代码高度冗余 <sup>[9](#mod-drivers-ref-9)</sup>。虽然可以通过宏消除，但当前写法导致约 40 行逻辑被复制 8 次，增加了维护负担。

### 3. 跨模块协同：trap_block 与调度器、syscall 的协作

整个 I/O 等待的完整流程跨越三个模块：

1. 系统调用层（`core::syscall`）将用户态请求转换为 `SyscallBlock`，其中包含 `SyscallBlockReason` 枚举和 `previous_single_signal_mask`。
2. `trap_block.rs` 的 `consume_blocked_syscall` 根据 reason 类型调用不同的 `consume_*_block` 函数，这些函数构造 `SchedulerWaitChannelRequest` 并调用调度器的 plan/commit 接口 <sup>[4](#mod-drivers-ref-4)</sup>。
3. 调度器（`blocking.rs`）的 `commit_wait_channel_block` 最终调用 `wait_task_on_channel_until`，它将任务状态设为 `STATE_WAITING`，并调用 `insert_wait_index` 将任务注册到全局等待哈希表 <sup>[10](#mod-drivers-ref-10)</sup>。
4. 当设备可读/可写时（如 pipe 写入完成），驱动通过 scheduler 的 `wake_task` 或 `wake_waiting_slot` 唤醒对应 channel 上的任务 <sup>[11](#mod-drivers-ref-11)</sup>。
5. 唤醒后，`trap_block.rs` 的 `complete_committed_wait_channel_idle` 调用 `complete_*_after_wait` 系列函数（来自 syscall 模块或 kernel syscall_runtime）完成系统调用剩余操作（如拷贝数据、设置返回值）<sup>[12](#mod-drivers-ref-12)</sup>。

这种模式下，具体驱动（如块设备）不需要直接与调度器交互，只需要在 I/O 完成时调用 scheduler 的唤醒接口即可。但当前块设备驱动 `block.rs` 是同步轮询的，不涉及唤醒操作，因此该协同目前只体现在管道、socket 等伪设备（PD）驱动中。

### 4. 边角细节与不足

**<sup>[5](#mod-drivers-ref-5)</sup> `COMPLETION_SPIN_LIMIT = 100_000_000`**：这是一个硬编码的轮询退避上限。在 QEMU 上，一次 virtio 块请求通常只需几十微秒，因此极少触发；但在仿真环境或慢速设备上，这个值可能不够大，且没有提供配置接口。修改时必须同时修改 `RESET_SPIN_LIMIT`、`CAPABILITY_WALK_LIMIT` 等常量，但 grep 可以找到所有引用。

**<sup>[7](#mod-drivers-ref-7)</sup> `static mut` 没有 `Sync` 约束**：所有全局变量都不是 `Sync`，但代码没有 `Send` 或 `Sync` 的显式实现。如果未来尝试在多核上并发出块请求，会导致未定义行为。当前仅在 boot 单核路径下安全。

**<sup>[13](#mod-drivers-ref-13)</sup> `dma_fence` 的实现**：代码调用 `dma_fence()`，但从 `core::sync::atomic::compiler_fence` 导入 <sup>[13](#mod-drivers-ref-13)</sup>。编译器屏障只阻止编译器重排，对硬件（如 PCIe 桥的写缓冲）无效。在 LoongArch64 上需要结合 `ibar`/`dbar` 指令才能保证 DMA 一致性。这可能导致在真实硬件上出现设备看到过期的描述符状态。

**<sup>[14](#mod-drivers-ref-14)</sup> `unwrap()` 和 `ok_or()` 的错误吞没**：`read_sector_inner` 中 `core::ptr::read_volatile` 等操作未检查有效性；`ensure_ready()` 的 `ok_or` 在失败时返回 `ProviderMissing`，但实际错误可能是 `FeatureNegotiationFailed`，错误信息丢失 <sup>[14](#mod-drivers-ref-14)</sup>。

**<sup>[15](#mod-drivers-ref-15)</sup> 缺少 `TODO` 注释**：代码中几乎未见 `TODO` 或 `FIXME` 标记。但这并非优点——它暗示开发者没有记录已知限制。例如，中断支持、多队列、异步接口等均未实现但也未标记。

**<sup>[10](#mod-drivers-ref-10)</sup> `insert_wait_index` 的哈希碰撞风险**：调度器使用 `TASK_WAIT_KIND`、`TASK_WAIT_KEY0`、`TASK_WAIT_KEY1` 作为等待索引，多个 channel 可能映射到相同的 (kind, key0, key1) 组合（如不同 pipe_id 的 high bits 相同），但代码没有显式处理碰撞，依赖 `key0()` 返回的唯一标识。对于 `Futex` 变体，`key` 是用户空间地址，碰撞概率极低但非零 <sup>[2](#mod-drivers-ref-2)</sup>。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 本模块实现了 Linux 兼容的系统调用处理框架，核心抽象是 `SyscallFrame`（封装系统调用号和参数）和 `SyscallOutcome`（枚举所有可能的内核响应，包括返回值、阻塞、Fork、Exec 等）。设计上采用双层架构：`core/syscall` 定义协议和 trait 接口，`kernel/syscall_runtime` 提供全局运行时状态（FD_TABLE、VFS、epoll 表等）并注入具体实现。与 rCore-Tutorial 或 ArceOS 相比，本实现将阻塞系统调用与信号屏蔽字交织的语义显式编码进 `SyscallBlock::signal_mask` 字段，并通过 `InterruptibleSignalState` trait 统一进程级和线程级信号状态管理，使 `pselect6`/`ppoll` 等可中断等待的正确性在类型层面得到保证。

### 1. 核心抽象与分派机制

系统调用层的入口是 `SyscallFrame` <sup>[1](#mod-syscall-ref-1)</sup>，它携带调用号（`number`）和最多 6 个参数（`args: [usize; 6]`）。`SyscallOutcome` <sup>[2](#mod-syscall-ref-2)</sup> 枚举了系统调用的所有可能结果：
- `Return(isize)`：同步返回值；
- `Exit(ExitState)` 和 `RemoteExit`：进程退出；
- `Fork(ForkRequest)` 和 `Exec(ExecRequest)`：进程创建/替换；
- `Blocked(SyscallBlock)`：等待某事件（I/O、futex、信号等）；
- `Yield(Pid)` 和 `Reschedule`：主动让出 CPU。

相比于简单的 `Result<isize, Errno>`，`SyscallOutcome` 显式地将“需要内核进一步处理”的请求编码为枚举变体，这样调度器可以直接 match 来决定下一步动作，无需通过额外的回调或状态机。

分派函数分为两组：
- `dispatch_single_*` 系列（不带 `&mut Process`）用于处理不修改进程状态的调用（如 `getpid`）；
- `dispatch_*` 系列需要 `&mut Process` 来处理可能改变进程状态的调用（如 `sigaction`）。

这两组函数最终都通过 `dispatch_single_with_runtime` 或 `dispatch_with_runtime` 将控制权交给 `core/syscall/dispatch` 子模块，后者根据 `frame.number()` 查找对应的处理函数。`kernel/syscall_runtime.rs` 中的 `dispatch_with_memory` <sup>[4](#mod-syscall-ref-4)</sup> 是一个典型例子：它实例化 `ActiveSyscallVfs`（一个零大小结构体，实现了 `SyscallVfs` trait），然后调用 `dispatch_with_runtime`，将 VFS 操作委托给全局静态表。

### 2. 双层架构：核心协议与运行时实现

`core/syscall` 目录下的代码是纯协议层：它定义 `SyscallFrame`、`SyscallOutcome`、`SyscallBlock` 以及各个子模块（`signal`、`sched`、`fs_path` 等）中的系统调用处理函数。这些函数不直接访问全局状态，而是通过 trait 参数（`UserMemoryReader`、`UserMemoryWriter`、`UserMemoryMapper`、`SyscallVfs`）来执行用户内存读写、页表映射和文件系统操作。这种设计使得核心逻辑可以在不依赖具体运行时的情况下进行单元测试。

`kernel/syscall_runtime` 层则扮演“实现注入器”的角色：它通过 `ActiveSyscallVfs` 结构体将 VFS 操作委托给全局的文件描述符表、打开文件表、挂载表等。这些全局变量全部声明为 `static mut` <sup>[7](#mod-syscall-ref-7)</sup>，例如 `ACTIVE_FD_TABLE` 和 `OPEN_FILE_TABLE`。在单核环境下，`static mut` 是安全的，但这也意味着当前设计假设了单核抢占式调度（没有其他核心同时访问这些表）。`ActiveSyscallVfs` 本身是一个零大小结构体，其方法直接操作这些全局表，并调用 `core/fs` 模块的底层函数。

这种分层的一个关键优点是：当需要支持多核时，只需将 `static mut` 替换为 `PerCpu` 或加锁封装，而 `core/syscall` 的代码完全不需要改动。

### 3. 信号中断与阻塞系统调用的集成

`SyscallBlock` 结构体 <sup>[3](#mod-syscall-ref-3)</sup> 不仅包含阻塞原因（`SyscallBlockReason`），还包含一个 `signal_mask` 字段，用于处理可中断的系统调用。`InterruptibleSignalState` trait <sup>[5](#mod-syscall-ref-5)</sup> 抽象了信号屏蔽字的读取和替换操作，并为 `Process`（进程级）和 `SingleSignalState`（线程级）提供了实现。

`activate_temporary_signal_mask` 函数 <sup>[5](#mod-syscall-ref-5)</sup> 实现了 Linux 中 `pselect6`/`ppoll` 的语义：在阻塞之前临时替换信号屏蔽字，如果在替换后发现挂起信号与新建屏蔽字有交集，则立即返回 `EINTR`。该函数在替换前后都会检查 `pending_mask & !blocked_mask`，以确保信号不会丢失。当阻塞结束后，`restore_single_temporary_signal_mask` 或 `restore_process_temporary_signal_mask` 负责恢复原屏蔽字。

`complete_interruptible_wait` <sup>[8](#mod-syscall-ref-8)</sup> 在唤醒后从调度器获取唤醒原因，如果是因为信号而被唤醒，则将返回值置为 `EINTR`。这套机制将信号递送与系统调用阻塞有机地结合，避免了竞态条件。

### 4. 运行时上下文切换与调度器协作

系统调用层通过 `TaskRuntimeSwitch` 机制与调度器协作。`prepare_task_runtime_switch` <sup>[6](#mod-syscall-ref-6)</sup> 在调度器切换任务前被调用，它保存当前任务的运行时快照（包括 FD 表、cwd、挂载等），并预取下个任务的快照。`enter_next_task_runtime` 和 `restore_current_task_runtime` 则在切换完成后恢复对应上下文的全局变量。

`TASK_RUNTIME_PID` 和 `TASK_RUNTIME_FD_GROUP` 是两个原子数组 <sup>[7](#mod-syscall-ref-7)</sup>，用于跟踪每个调度槽关联的 PID 和 FD 组。FD 组允许多个任务共享文件描述符表（类似 Linux 的 `CLONE_FILES`），当组内最后一个任务释放时，FD 表才会被关闭。

`resolve_shared_file_user_page_fault` <sup>[10](#mod-syscall-ref-10)</sup> 展示了系统调用层如何与缺页异常处理协作：它通过 `single_user_shared_file_vma_at` 查询当前任务映射的共享文件 VMA，然后调用 VFS 的 `read_mapped_file` 读取文件内容，最后通过 `memory.map_user_page_with_bytes` 将页面映射到用户空间。这实现了 MMAP 文件的按需分页。

### 5. 边角细节与不足

**硬编码架构常量**：`USER_MMAP_BASE = 0x1_0000_0000` <sup>[9](#mod-syscall-ref-9)</sup> 是 x86-64 的典型值，但在 32 位或 riscv64 上可能无效。虽然没有 `#[cfg]` 分支，但 `linux_abi.rs` 中的 `USER_MMAP_LIMIT = (1usize << 37) - PAGE_SIZE` <sup>[11](#mod-syscall-ref-11)</sup> 同样依赖 64 位地址空间。如果移植到 32 位平台，这些常量会导致整数溢出。

**无锁全局状态**：所有运行时表（`ACTIVE_FD_TABLE`、`ACTIVE_OVERLAY` 等）都是 `static mut`，在单核环境下可以工作，但没有任何并发保护。代码中也没有看到 `Mutex` 或 `RwLock`，这意味着当前设计假设单核运行。

**信号状态恢复的 `unsafe` 缺失注释**：`restore_single_temporary_signal_mask` 调用 `single_replace_signal_mask`，后者很可能通过全局原子变量操作。但函数签名中没有 `unsafe`，其内部也未注明安全契约。

**未实现的系统调用常量**：`linux_abi.rs` 定义了 `MADV_COLD`、`MADV_PAGEOUT`、`MPOL_*` 等高级常量，但对应的系统调用处理函数在 `core/syscall` 子模块中可能尚未实现（需要进一步核查）。这些常量的存在是为了 ABI 兼容性，但过早暴露可能导致调用者误以为功能已支持。
