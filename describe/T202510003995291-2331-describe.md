## 📖 一、总览

该仓库属于 ArceOS-Starry 家族，定位于 **Linux 兼容性内核**，目标是在 ArceOS 微内核基础上提供接近 Linux 的系统调用接口，以便运行未经修改的 Linux 用户程序（如 busybox）。与标准 ArceOS 的教学/演示型不同，本仓库强调功能覆盖和兼容性，但处于快速演进阶段，许多细节尚未完善。

最值得评审注意的特点包括：
1. **基于 POST_TRAP 回调的信号递送机制**：通过注册 `axhal::trap::POST_TRAP` 钩子，在每次从内核返回用户态前递送信号，避免了标准 ArceOS 轮询模型的缺陷，实现了真正的用户态信号处理 [signal.ref:1]。
2. **两套独立的 procfs 实现**：`proc.rs` 与 `proc1/mod.rs` 并存，后者正在添加动态进程目录和符号链接支持，表明文件系统子系统处于重构阶段 [fs.ref:5]。
3. **多架构条件编译**：同时支持 x86_64、riscv64、aarch64、loongarch64，但在关键位置（如地址空间拷贝）条件编译未覆盖所有架构，存在遗漏风险 [mm.ref:2]。

## 🎯 二、综合评价

### 整体定位
该仓库是 ArceOS-Starry 家族中一个**功能完整的 Linux 兼容层尝试**。与标准 ArceOS 的模块化教学内核不同，本仓库将启动逻辑上移至应用层，引入 `Process`/`Thread` 分离模型 [boot.ref:3]，并实现了超过 200 个系统调用。其设计定位介于“演示型”与“实验型”之间：足以运行 busybox shell 等静态链接程序，但动态链接、网络选项、信号完整语义等尚未完善。整体上，它是一个**能力原型**，展示了在微内核上构建 POSIX 兼容层的可行路径，但距离生产使用仍有较大距离。

### 真正的创新点
- **POST_TRAP 信号递送机制**：通过注册 `axhal::trap::POST_TRAP` 钩子，starry 实现了每次用户态返回前的信号检查 [signal.ref:1]，这与 Linux 的处理时机一致。标准 ArceOS 仅在轮询中处理信号，无法处理同步信号。这一设计使得信号响应更加及时，且为 `SIGCHLD`、`SIGPIPE` 等异步信号提供了正确基础。
- **TaskExt 嵌入与进程/线程分离**：通过 `def_task_ext!` 宏将 POSIX 线程数据嵌入 `axtask::TaskInner` [task.ref:1]，在不修改调度器的情况下扩展了语义。`Process` 和 `Thread` 的 1:N 关系支持了多线程，虽然当前尚未完全实现 `CLONE_VM` 语义，但架构已就绪。
- **两代 procfs 的演进设计**：`proc.rs` 使用静态硬编码数据，而 `proc1/mod.rs` 开始引入动态进程目录和符号链接 [fs.ref:5]。这种并存状态表明开发者正在从“快速原型”转向“正确实现”，是开放式重构的证据。

### 取舍判断
存储库的精力明显集中在**广度覆盖**而非**深度正确**上。大量系统调用通过 `stub_bypass` 或 `stub_unimplemented` 返回假值（如 `access` 永远返回 0）[syscall.ref:4]。网络模块的 `setsockopt` 全为桩，`getaddrinfo` 存在内存泄漏风险 [net.ref:5]。内存管理省略了 `mmap_min_addr` 低地址检查 [mm.ref:3]。这些妥协使得 coreutils 等程序能运行，但安全性和健壮性不足。

代码组织上，`api/src/imp/` 放置系统调用实现，`core/src/` 放置核心状态，这种分离清晰，但也导致模块间循环依赖倾向（如 fs 模块依赖 task 获取当前进程）。此外，**代码重复**是一个显著问题：两套 procfs 的大量重叠，以及 mm 模块中 `copy_from_kernel` 的条件编译逻辑在多个位置重复。

### 完成度与不足
**完成度 (2/3)**：核心系统调用（fork/execve/exit/wait、mmap/brk、open/read/write、signal 三件套）全部实现且可运行 busybox shell。但高级功能普遍缺失：信号 Stop/CoreDump 降级为终止 [signal.ref:4]，futex 仅支持 `FUTEX_WAIT_PRIVATE`，网络地址转换仅限 IPv4。整体表现为“核心路径可用，边缘情况未经测试”。

**创新性 (2/3)**：POST_TRAP 信号机制和 TaskExt 嵌入是不同于 ArceOS 标准的独创设计，但整体架构仍大量复用 ArceOS 的 axhal/axtask/axmm 等基础 crate。核心创新点数量有限，但方向合理。

**代码质量 (1/3)**：存在明显问题：magic number 2333 无解释 [boot.ref:2]，多处 `unsafe` 缺少安全注释 [signal.ref:2]，条件编译遗漏 riscv64 导致潜在未定义行为 [mm.ref:2]，panic 无法返回错误 [fs.ref:4]。代码风格尚可，但 rigor 不足。因此给予基础分。

整体上，starry 是一个令人兴奋但未完工的项目，其设计思路值得后续项目借鉴。

## 🚀 三、启动流程

>

### 1. 启动入口与初始化流水线

启动流程的顶层入口是 `src/main.rs` 中的 `#[unsafe(no_mangle)] fn main()`（行 22-42）。该函数在 no_std / no_main 环境下作为内核的 Rust 入口点，承担了三阶段初始化：

**阶段一：内核基础设施准备。** 首先获取根目录并切换（`FS_CONTEXT.lock().change_root(root_dir).unwrap()`，行 24），然后初始化全局文件描述符表（`FD_TABLE.init_new(FdTable::new())`，行 25），最后挂载所有文件系统（`mount_all().expect(...)`，行 26）。这三步构成了文件系统子系统的完整启动依赖链。值得注意的是，`FS_CONTEXT` 来自 `axfs_ng::api`，而非旧版 `axfs`，表明该仓库正在迁移到新的文件系统框架。

**阶段二：测试用例注入。** 通过 `include_str!(env!("AX_TESTCASES_FILE"))`（行 29）在编译期将测试脚本嵌入内核，构造 `busybox sh -c <command>` 的命令行参数（行 30-32）。这种设计将测试逻辑固化在构建时，避免了运行时从磁盘寻找 init 程序的复杂性，但代价是更换测试用例需要重新编译内核。

**阶段三：用户进程启动。** 调用 `entry::run_user_app(&args, &envs)`（行 37）启动 shell 进程，并打印退出码。`run_user_app` 返回 `Option<i32>`，使用 `info!` 而非 `panic!` 处理退出，体现了对用户进程正常退出的容忍。

### 2. 用户空间构造与任务生成机制

`core/src/entry.rs` 中的 `run_user_app` 函数（行 11-68）是启动流程的核心实现，它完成了一个用户态进程从地址空间创建到调度器提交的全生命周期构建。其内部可分解为四个子阶段：

**子阶段 A：地址空间初始化。** 调用 `new_user_aspace_empty()` 创建空地址空间，随后通过 `.and_then()` 链式调用 `copy_from_kernel()` 和 `map_trampoline()`（行 15-21）。`copy_from_kernel` 将内核必要的数据（如信号处理代码）映射到用户空间，而 `map_trampoline` 建立信号跳板的线性映射——这是信号机制能够在用户态与内核态之间切换的基础。`<sup>[1](#mod-boot-ref-1)</sup>` 证明了跳板映射被整合进了启动流水线中，而非由信号模块单独初始化，这是一种"启动即就绪"的设计哲学。

**子阶段 B：程序加载与上下文创建。** `load_user_app()`（行 33-35）将 ELF 文件解析到地址空间，返回入口地址和用户栈顶。随后 `UspaceContext::new(entry_vaddr.into(), ustack_top, 2333)`（行 37）创建首次进入用户态所需的 CPU 上下文。其中第三个参数 `2333` 是一个硬编码的 magic number，在标准 ArceOS 中通常传入 `0` 或环境指针——这里使用 2333 可能是一个 TODO 遗留或调试占位符。`<sup>[2](#mod-boot-ref-2)</sup>`

**子阶段 C：进程/线程模型实例化。** 通过 `Process::spawn_process()` 创建进程对象，再从进程获取主线程（行 42-43）。随后构建 `ProcessData` 和 `thread_data`（行 46-52），`ProcessData` 持有地址空间的 `Arc<Mutex<...>>` 和 `SIGCHLD` 信号掩码。这一步将 ArceOS 原生的"单任务"模型扩展为"进程-线程"两层结构，`<sup>[3](#mod-boot-ref-3)</sup>` 中 `Process` 来自 `undefined_process::process::Process` 而非标准的 `axprocess`，表明这是该仓库独有的抽象。

**子阶段 D：namespace 隔离与任务提交。** 通过 `FD_TABLE.deref_from(&thread_data.namespace).init_new(...)` 等三行（行 54-60），将全局 FD 表、当前目录路径克隆到新线程的 namespace 中。这种"克隆 + 隔离"的模式使得子线程拥有独立的文件描述符和当前目录快照，但初始内容与父进程一致——这是实现 `fork` 语义的基础。最后调用 `axtask::spawn_task(user_task)` 将任务提交给调度器，并通过 `user_task.join()` 阻塞等待其结束（行 65-67）。

### 3. 跨模块协同与设计定位

启动流程并非孤立的代码段，它与内核多个子系统存在精密的协同关系：

**(a) 与 `axhal` 的协同：** `UspaceContext::new` 是架构相关的结构体（来自 `axhal::arch`），其内部封装了通用寄存器、页表基地址等 CPU 状态。`<sup>[4](#mod-boot-ref-4)</sup>` 中通过 `set_page_table_root` 将地址空间的页表根写入任务的上下文，这是 MMU 切换与任务调度之间的桥梁——调度器在切换任务时会加载该页表根。

**(b) 与 `axsignal` 的协同：** `map_trampoline` 在启动阶段就完成了信号跳板的映射，`ProcessData::new` 的第四个参数 `Some(Signo::SIGCHLD)` 设置了子进程退出时向父进程发送的信号。这保证了后续信号传递路径在进程创建时已就绪。`<sup>[5](#mod-boot-ref-5)</sup>`

**(c) 与 `axtask` 调度器的协同：** `create_user_task` 创建的任务在被 `init_task_ext` 附加了 `TaskExt`（包含线程和进程数据）后才提交给 `spawn_task`。调度器感知到的 `TaskExt` 使得在上下文切换时能够正确切换 FD 表 namespace、信号掩码等每任务元数据。`<sup>[6](#mod-boot-ref-6)</sup>`

**(d) 与 `axfs` 的协同：** 主函数中先 `change_root` 再调用 `run_user_app`，而 `run_user_app` 内部又通过 `axfs::api::set_current_dir` 将 CWD 设置为可执行文件的父目录。这种分层保证了文件系统命名空间的一致视图。`<sup>[7](#mod-boot-ref-7)</sup>`

### 4. 边角细节与不足

**`<sup>[8](#mod-boot-ref-8)</sup>` 2333 魔法数：** `UspaceContext::new` 的第三个参数硬编码为 `2333`，无注释说明其含义。在 ArceOS 主线实现中，该参数通常为 `0` 或指向环境变量的指针。若这是一个线程 ID 或调试标志，则缺少 cfg 条件化或常量定义，跨架构移植时可能产生未定义行为。

**`<sup>[9](#mod-boot-ref-9)</sup>` 密度过高的 unwrap：** `run_user_app` 中连续调用了 3 次 `expect`/`unwrap`（行 21、29、35）和 1 次 `unwrap_or_else`（行 34），任何磁盘 I/O 错误或无效 ELF 都将导致内核 panic。用户提供的 `args[0]` 若路径非法，`FilePath::new` 后直接 `unwrap()`（行 29）构成 DoS 入口。

**`<sup>[10](#mod-boot-ref-10)</sup>` namespace 隔离的完整性疑点：** `FD_TABLE.copy_inner()` 执行的是浅拷贝还是深拷贝？若 `FdTable` 内部持有 `Arc` 引用，则子线程修改 FD 可能意外传播到父进程。代码未提供 `FdTable::copy_inner` 的实现细节，无法确认隔离语义的完备性。

**`<sup>[11](#mod-boot-ref-11)</sup>` 注释掉的 `LD_LIBRARY_PATH`：** 行 34-35 中注释了两行环境变量设置，表明开发者曾尝试加载动态链接库但未完成。若 busybox 为动态链接版本，缺少 `LD_LIBRARY_PATH` 会导致运行时加载器失败。这是一个未完成的集成工作。

**`<sup>[12](#mod-boot-ref-12)</sup>` 信号跳板的后门假设：** `map_trampoline` 的实现依赖于架构特定的跳板代码（通常是 sigreturn 例程）。若该跳板物理页被分配在低地址区域且用户进程具有写权限，则存在用户态篡改跳板以提权的风险。当前代码未对跳板设置只读/执行权限或进行完整性校验。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `axmm::AddrSpace` 为地址空间核心抽象，封装页表操作，提供用户进程地址空间创建、ELF 加载、堆栈初始化等功能。系统调用层实现了 mmap、munmap、mprotect、brk 等 POSIX 内存接口。与 ArceOS 基线相比，新增了信号跳板固定映射、percpu 用户内存访问标记、huge page 支持以及脚本解释器递归加载等特性。模块尚处早期开发，存在多处未实现分支和条件编译覆盖不全的问题。

### 1. 核心抽象与外部依赖

内存管理模块围绕 `axmm::AddrSpace` 构建，该类型封装了架构相关的页表操作，提供 `map_alloc`、`map_linear`、`unmap`、`protect`、`find_free_area` 等方法。`core/src/mm.rs` 中的顶层函数利用此抽象构造用户地址空间：`new_user_aspace_empty` 创建空用户地址空间（基于 `USER_SPACE_BASE` 和 `USER_SPACE_SIZE`）<sup>[1](#mod-mm-ref-1)</sup>；`copy_from_kernel` 按条件拷贝内核映射；`map_trampoline` 固定映射信号跳板；`load_user_app` 完成 ELF 加载、堆栈设置和参数布局。系统调用实现（`api/src/imp/mm/mmap.rs` 与 `api/src/imp/mm/brk.rs`）通过 `current_process_data()` 获取当前进程的 `AddrSpace` 锁，并调用其方法。模块依赖 `axhal::paging` 定义页大小和权限标志，依赖 `axconfig::plat` 提供平台布局常量（堆基址、栈顶等），并通过 `axfs_ng::api::FS_CONTEXT` 读取程序文件。

### 2. 关键设计取舍

**2.1 地址空间分离策略**：`copy_from_kernel` 通过 `#[cfg(not(any(target_arch = "aarch64", target_arch = "loongarch64")))]` 条件编译选择是否拷贝内核页表到用户地址空间 <sup>[2](#mod-mm-ref-2)</sup>。AArch64 和 LoongArch64 使用独立页表基址寄存器（TTBR0/TTBR1 和 PGDL/PGDH），用户页表完全独立，故无需拷贝；其他架构（如 x86_64）则需拷贝内核映射以保持一致性。这一取舍降低了上下文切换时 TLB 刷新开销，但假设了目标架构的页表拆分机制，且未覆盖 RISC-V（它通常也使用独立页表）。

**2.2 信号跳板零拷贝映射**：`map_trampoline` 将 `axsignal::arch::signal_trampoline_address()` 返回的物理地址通过 `map_linear` 映射到固定虚拟地址 `SIGNAL_TRAMPOLINE`，权限为 `READ | EXECUTE | USER` <sup>[3](#mod-mm-ref-3)</sup>。每次信号处理无需动态分配跳板页，但永久占用了该虚拟地址段（与其他模块固定映射冲突的风险）。

**2.3 用户内存访问标记**：通过 percpu 变量 `ACCESSING_USER_MEM` 记录内核是否处于“访问用户内存”的区域内。`access_user_memory` 将其置 `true`，执行闭包后恢复 <sup>[4](#mod-mm-ref-4)</sup>。此标记被设计为与 trap handler 协同：当内核态发生页错误时，若该标记为 `true`，则错误由内核自行消化（返回 `EFAULT` 等），否则视为内核 bug。这类似 Linux 的 `pagefault_disable` 机制，但粒度更粗（整个闭包期间禁用用户态 page fault 转发）。

**2.4 堆管理简化**：`sys_brk` 仅检查请求地址是否在预分配堆区域（`USER_HEAP_BASE` + `USER_HEAP_SIZE`）内，并更新进程的堆顶 <sup>[5](#mod-mm-ref-5)</sup>。整个堆在 `load_user_app` 时已被 `map_alloc` 预映射，故 `brk` 退化为一个指针调整，不涉及页分配。这在牺牲空间（始终占用最大堆大小）的同时换取了速度，且无法支持高地址堆扩展。

### 3. 跨模块协同

**3.1 与信号子系统**：`map_trampoline` 直接引用 `axsignal::arch::signal_trampoline_address()` 获取物理地址，将跳板页固定在约定虚拟地址。信号交付时，处理器将跳转到该处执行 sigreturn 序列。这要求 `axsignal` 模块的信号 trampoline 代码必须位于该固定地址，形成编译期耦合。

**3.2 与文件系统**：`load_user_app` 使用 `axfs_ng::api::FS_CONTEXT` 读取用户程序二进制 <sup>[6](#mod-mm-ref-6)</sup>。`sys_mmap` 在文件映射（非匿名）时，通过 `File::from_fd` 获取文件对象，并用 `file.read_at` 填充已分配页面 <sup>[7](#mod-mm-ref-7)</sup>。值得注意的是，写映射情况下仅打印错误而未返回错误码，说明该路径未完全实现。

**3.3 与页错误处理**：虽然本仓库未提供 trap handler 源码，但 `ACCESSING_USER_MEM` percpu 标记的存在表明它将在 `do_page_fault` 中被检查。当标记为 `true` 时，缺页错误由内核内部处理（例如通过 `copy_from_user` 等函数返回 `-EFAULT`），否则传递到用户信号。这避免了对用户内存访问时 double fault。

### 4. 边角细节与不足

**4.1 `TDOO` 拼写错误与 I-cache 缺失**：`map_elf` 加载代码段后注释为 `TDOO: flush the I-cache` <sup>[8](#mod-mm-ref-8)</sup>。指令缓存未刷新意味着自修改代码或动态代码生成（如 JIT）可能失效。

**4.2 `MAP_SHARED` 误清零**：`sys_mmap` 在 MAP_SHARED 路径中调用 `map_shared(start_addr, aligned_length, map_permission, true, page_size)`，第四个参数 `is_zero` 固定为 `true` <sup>[9](#mod-mm-ref-9)</sup>。但注释明确说“仅在 MAP_ANONYMOUS 时才 zero”，可能导致文件共享映射的页面被错误清零。

**4.3 文件映射写错误处理不完整**：当 `writeable` 为真时，代码仅调用 `error!` 输出日志，却继续使用 `file.read_at` 读取数据 <sup>[10](#mod-mm-ref-10)</sup>。应该返回 `-EINVAL` 或实现写映射的语义。

**4.4 缺少低地址检查**：`find_free_area` 搜索范围是整个地址空间，未参照 Linux 的 `/proc/sys/vm/mmap_min_addr` 限制低地址映射 <sup>[11](#mod-mm-ref-11)</sup>。恶意程序可以通过 `MAP_FIXED` 映射零页获得内核权限。

**4.5 `PROT_GROWSDOWN`/`PROT_GROWSUP` 未实现**：`sys_mprotect` 直接返回 `EINVAL` <sup>[12](#mod-mm-ref-12)</sup>，而 Linux 允许这些标志用于栈的 guard 页扩展。

**4.6 huge page 支持脆弱**：`sys_mmap` 仅支持通过 `MAP_HUGE_2MB` 或 `MAP_HUGE_1GB` 字面量指定页大小 <sup>[13](#mod-mm-ref-13)</sup>，未考虑系统实际支持情况（如硬件不支持 1G 页时直接拒绝）；且 `MAP_HUGETLB` 不指定具体大小时直接报错，不符合 Linux 行为（应使用默认大页大小）。

**4.7 架构覆盖不全**：`copy_from_kernel` 的条件编译未包含 `target_arch = "riscv64"` <sup>[2](#mod-mm-ref-2)</sup>。RISC-V Linux 通常也使用 S 模式下的独立页表（`satp` CSR 切换），按此逻辑也应跳过拷贝，否则会错误地复制内核映射。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 模块以 `Process`/`Thread` 作为 POSIX 进程/线程抽象，通过 `TaskExt`（嵌入 `axtask::TaskInner`）将用户态线程与内核调度实体绑定。它解决了多进程、多线程、进程组、会话、信号、futex、wait 等 Linux 兼容需求。与 ArceOS 基线相比，新增了完整的进程/线程生命周期、信号机制、进程组/会话管理，并重新实现了 clone/exit/wait 等系统调用。

### 1. 核心抽象与外部依赖

进程与任务调度的核心是两层抽象：底层使用 `axtask` crate 提供的 `TaskInner` 作为可调度实体，上层通过 `TaskExt`（定义于 `core/src/task.rs`）嵌入 POSIX 语义：一个 `TaskExt` 包含一个 `Arc<Thread>` 和 `Arc<ThreadData>`，分别代表 POSIX 线程标识和运行态数据（命名空间、信号状态、地址空间等）。`Thread` 又关联至 `Process`，形成 1:N 的线程-进程关系。`Process` 维护线程映射、进程组、父/子关系、僵尸状态；`ProcessGroup` 和 `Session` 实现会话管理。

这一分层的关键取舍是复用 `axtask` 的调度接口（`WaitQueue`、yield、spawn）而非自研调度器，代价是需要通过 `unsafe` 的 `task_ext_ptr()` 从 `axtask::current()` 获取扩展数据。`def_task_ext!` 宏自动实现 `TaskExtRef` trait，但需要调用方确保 `TaskExt` 在任务生命周期内合法。

### 2. 关键设计取舍

**Clone 实现的分支逻辑**：`sys_clone_impl` 根据 `CLONE_THREAD` 标志决定创建线程还是进程。创建线程时共享地址空间（仅增加引用），创建进程时根据 `CLONE_VM` 标志选择共享或深拷贝 `AddrSpace`。深拷贝调用 `try_clone()` 后还需 `copy_from_kernel` 确认页表数据已复制到新空间。这种设计在性能与语义正确性间做了平衡：`CLONE_VM` 不加 `CLONE_THREAD` 其实是不合法的（Linux 手册），但本实现允许了此组合，可能隐含未处理的情况。

**信号管理的双层架构**：信号处理同时存在于进程级（`ProcessSignalManager`）和线程级（`ThreadSignalManager`）。`check_signals` 函数（定义于 `api/src/imp/task/signal.rs`）在每次 `POST_TRAP` 回调中被调用，实现从内核返用户态前的信号投递。`sys_rt_sigaction` 操作进程级 `signal.actions`，而 `sys_rt_sigprocmask` 操作线程级 `signal.blocked`。这种分离使得线程可独立屏蔽信号，但 `SIGKILL/SIGSTOP` 强制不可屏蔽的检查仅在 `sys_rt_sigaction` 中执行，而 `sys_rt_sigsuspend` 中手动移除它们，一致性存在风险。

**进程退出的委托链**：`Thread::exit` 调用 `Process::remove_thread`，当线程列表为空时自动触发 `Process::exit`。`Process::exit` 将子进程移交给 init 进程（`pid==1`），然后自身成为僵尸。父进程通过 `wait4` 系统调用回收僵尸：`sys_wait4` 在循环中检查子进程的 `is_zombie` 标志，然后调用 `release` 从全局表中移除。这一链式设计减少了外部协调，但 `get_child_reaper` 目前硬编码为 pid==1 且未检查 init 本身是否为僵尸（见 TODO）。

**Futex 表的进程级存放**：`futex_table` 存放在 `ProcessData` 中，每个进程持有 `BTreeMap<usize, Arc<WaitQueue>>`。`sys_futex` 锁定表的粒度是 `Mutex` 包裹整个 BTreeMap，在 `FUTEX_REQUEUE` 操作中先获取锁查找两个地址的 waitqueue，再释放锁后调用 `notify_one` 和 `requeue`。此设计避免了持锁期间调用可能阻塞的 wait，但存在 ABA 问题（两次锁之间表可能变化），不过 Linux 用户态 futex 也有类似风险。

### 3. 跨模块协同

**进程创建与调度**：`clone.rs` 调用 `core/src/task.rs` 的 `create_user_task` 生成 `TaskInner`，并将 `TaskExt` 通过 `init_task_ext` 绑定。`create_user_task` 的闭包中执行 `enter_uspace`，该函数依赖于 `axhal` 的架构层，最终跳入用户态。调度由 `axtask::spawn_task` 触发，由内核定时器或自愿 yiled 驱动上下文切换。

**信号投递与进程退出**：`exit.rs` 在线程退出时通过 `send_signal_process` 向父进程发送 `SIGCHLD`，并唤醒 `child_exit_wq`。`wait4` 系统调用正是阻塞在该 waitqueue 上。信号发送又跨模块调用 `api/src/imp/task/signal.rs` 的 `send_signal_process`，最终调用 `ProcessSignalManager::send_signal`。此处信号发送链无状态回滚，若父进程在收到信号前已退出（孤儿进程），`get_process_data` 返回 `None`，信号被静默丢弃。

**时间统计与调度**：`core/src/task.rs` 提供 `time_stat_from_kernel_to_user` / `from_user_to_kernel`，在 `axhal` 的 trap 入口/出口处调用（通过 `POST_TRAP` 回调或显式钩子）。这些统计值通过 `time_stat_output` 被 `sys_times` 使用。时间统计使用单调时钟，单位纳秒，输出时转换为微秒。该实现依赖 `monotonic_time_nanos`，与 wall clock 无关联。

### 4. 边角细节与不足

1. **`#[cfg]` 条件编译的风险**：在 `Drop for ProcessData`（`core/src/process.rs` 第 68-73 行）中，清理地址空间映射跳过了 aarch64 和 loongarch64，注释引用`crate::new_user_aspace`。若未来添加其他架构（如 riscv），该清理会被错误跳过，导致 kernel space 映射残留。需要改为显式 check 或使用 feature flag。<sup>[4](#mod-task-ref-4)</sup>

2. **`assert!` 造成的 DoS**：`Process::exit` 中 `assert!(!self.is_zombie())`（`process/src/process.rs` 第 165 行），如果并发调用 `exit` 或竞争条件导致重复退出，会直接 panic 终止内核。类似 panic 在 `Process::release` 中也存在。用户态通过恶意线程重复调用 exit 可能触发此漏洞。<sup>[5](#mod-task-ref-5)</sup>

3. **`unsafe` 缺少 SAFETY 注释**：`read_trapframe_from_kstack` 和 `write_trapframe_to_kstack` 在 `core/src/task.rs` 第 99-107 行直接构造裸指针，仅依赖 `kstack_top` 参数合法。但若调用方传入非法栈顶（如用户态可控制的数值），可导致任意内核内存读写。没有 `// SAFETY:` 解释安全契约。<sup>[6](#mod-task-ref-6)</sup>

4. **硬编码的 init 进程 ID**：`get_child_reaper` 直接返回 `pid==1` 的进程（`process/src/process.rs` 第 164 行），未检查该进程是否已是僵尸。如果 init 进程异常退出且未被回收，其子进程将永远无法被领养，导致系统无法正常退出。<sup>[7](#mod-task-ref-7)</sup>

5. **TODO 半成品标记**：多处 TODO 注释显示未实现的功能：CoreDump 仅调用 exit（`api/src/imp/task/signal.rs` 第 125 行），Stop/Continue 信号处理仅为 stub；`child_reaper` 未实现真正的 sub-reaper 机制；FUTEX_WAIT_BITSET 的 bitset 支持完全跳过（`api/src/imp/task/futex.rs` 第 140 行）。这些限制可能导致部分 Linux 应用程序异常。<sup>[8](#mod-task-ref-8)</sup>

6. **死代码与工程腐烂**：`api/src/imp/task/thread.rs` 中 `sys_setpgid` 的 `// I don't know why it's here` 注释表明代码遗留。`core/src/process.rs` 的 `assert!(weak_thread_data.strong_count() > 0)` 在 `get_thread_data` 中，若 refcount 为 0 说明已经被 drop，但该断言之前已经通过 `upgrade()` 获取了 Arc，若 empty 则 `upgrade` 返回 None，断言实际上不会触发。可能是编程失误。<sup>[9](#mod-task-ref-9)</sup>

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 模块基于 `undefined_vfs` crate 提供的伪文件系统框架（DynamicFs、DynamicDir、SimpleFile）构建内存中的虚拟文件系统，通过 FileLike trait 和 FdTable 实现类 Linux 文件描述符抽象。核心问题是：在不依赖真实块设备的情况下，为 Linux 兼容层提供 `/proc`、`/dev`、`/tmp` 等虚拟文件系统并支持 pipe、epoll 等特殊文件语义。与普通 ArceOS 基线相比，独特之处在于：(1) 存在两套独立的 procfs 实现（`proc.rs` 与 `proc1/mod.rs`），后者添加了动态进程目录生成和符号链接支持；(2) procfs 数据大量采用硬编码快照（~15KB CPUINFO）而非动态采集，这是一种偏重启动速度但牺牲准确性的取舍。

### 1. 伪文件系统框架与 `undefined_vfs` 抽象

该模块并未自行实现 VFS 层，而是复用外部 crate `undefined_vfs` 提供的内存伪文件系统框架。核心抽象包括 `DynamicFs`（动态文件系统）、`DynamicDir`（动态目录构造器）、`SimpleFile`（闭包驱动的伪文件）和 `DirMaker`（目录构建产物）。这些类型通过 `DynamicFs::new_with(name, magic, builder)` 组合：`builder` 是一个 `fn(Arc<DynamicFs>) -> DirMaker` 回调，在其中用 `DynamicDir::builder(fs.clone())` 创建目录树并挂载 `SimpleFile` 叶子节点 <sup>[1](#mod-fs-ref-1)</sup>。

这种设计的核心取舍是 **完全在内存中构造目录树，无块设备交互**。替代方案是挂载真实文件系统（如 FAT32），但这会引入块驱动和 FUSE 的复杂性。`undefined_vfs` 的选择让 /proc、/dev 等伪文件系统能以极低成本实现，但代价是文件内容必须由闭包每次重新生成——对动态数据（如中断计数）这是优点，对静态数据（如 cpuinfo）则成为不必要的开销。

在文件描述符层，`fd.rs` 定义了 `FileLike` trait 和 `FdTable`。`FdTable` 管理从 `i32` 文件描述符到 `Arc<dyn FileLike>` 的映射，并提供 `fd_lookup`、`fd_add`、`fd_remove` 等全局函数。`FileLike` 定义了 `read`/`write`/`status`/`poll`/`get_flags`/`into_any` 等统一接口，所有可打开对象（File、Directory、Pipe、EpollInstance）都实现此 trait，使得 `read()` 系统调用可以多态地分发到不同实现。

### 2. 两个 procfs 实现：演进中的设计

仓库中存在两个独立的 procfs 实现：`imp/proc.rs` 与 `imp/proc1/mod.rs`。两者都通过 `new_procfs() -> Filesystem<RawMutex>` 函数导出，但设计理念存在显著差异。

**`proc.rs`（较老版本）**：完全静态。`/proc/self` 是一个预填充的静态目录，包含 maps、status、stat 三个硬编码文件 <sup>[7](#mod-fs-ref-7)</sup>。PID 目录（如 `/proc/1`、`/proc/10`）也是预先创建的静态目录，每个只包含一个 stat 文件 <sup>[10](#mod-fs-ref-10)</sup>。没有符号链接，没有动态进程发现。这种实现的优点是构建简单，缺点是无法支持不同进程查看自身 `/proc/self` 时获得不同内容。

**`proc1/mod.rs`（较新版本）**：引入动态机制。`/proc/self` 被实现为 `SimpleFile::create(..., NodeType::Symlink, ..., || current_process().get_pid().to_string())`，即每次读取时闭包调用 `current_process()` 获取当前线程的 PID 并返回字符串形式的符号链接目标 <sup>[5](#mod-fs-ref-5)</sup>。更关键的是，通过 `root.set_pseudo_ops(ProcessInfoDir::new(fs.clone()))` 注册了 `PseudoDirOps` 实现 <sup>[6](#mod-fs-ref-6)</sup>；当用户访问 `/proc/[pid]/` 时，`ProcessInfoDir` 会动态查找当前系统中活动的进程并生成对应的 `task_stat` 等信息。这更接近真实 Linux 的 procfs 行为。

两个版本的代码存在大量重复（cpuinfo 常量、sys 目录结构等），且 `proc1/mod.rs` 声明了 `mod process; mod task_stat;` 子模块而 `proc.rs` 没有 <sup>[8](#mod-fs-ref-8)</sup>。这强烈暗示 `proc.rs` 是早期实现，`proc1` 是重构中的替代品，但 `proc.rs` 尚未被删除。这种中间状态构成技术债务——未来维护者需要确认是否两个版本都会被编译，还是通过 `#[cfg]` 选择。

### 3. 跨模块协同与动态数据源

procfs 虽然以静态数据为主，但有三处与内核其他模块的动态协同值得注意：

1. **`/proc/interrupts` 与 `axtask`**：`proc.rs` 的该文件通过 `format!("0: {}", axtask::get_irq_count())` 生成 <sup>[4](#mod-fs-ref-4)</sup>。这要求 `axtask` 模块在每次文件读取时提供当前 IRQ 计数，使得 `/proc/interrupts` 能够反映真实中断活跃度。但闭包在每次 `read()` 时都重新分配字符串，对频繁读取的场景会产生不小的分配压力。

2. **`/proc/self` 与 `starry_core::task`**：`proc1` 版本中 `/proc/self` 符号链接的目标由 `current_process().get_pid()` 动态决定 <sup>[5](#mod-fs-ref-5)</sup>。这要求 `starry_core::task` 模块暴露当前进程信息，并且符号链接内容在每次读取时刷新，而非创建时固定。

3. **`Filesystem<RawMutex>` 与 `axsync`**：整个 VFS 实例使用 `axsync::RawMutex` 作为锁类型，确保多线程下目录树操作安全 <sup>[1](#mod-fs-ref-1)</sup>。但由于 `SimpleFile` 的内容闭包自身不持有锁（内容在闭包内生成），数据竞争风险被限制在目录结构变更（如 mount/umount）上。

在文件描述符侧，`FdTable` 与 `FileLike` 的协同更为基础：`sys_read()` 通过 `fd_lookup(fd)` 获取 `Arc<dyn FileLike>`，然后多态调用 `read()`。`Pipe` 和 `EpollInstance` 分别实现环形缓冲区和事件轮询语义，与 VFS 层的目录/文件实现共享同一套 trait 接口。

### 4. 边角细节与不足

**硬编码快照的数据腐坏风险 <sup>[3](#mod-fs-ref-3)</sup>**：`CPUINFO` 常量超过 15KB，完整包含 16 核 AMD Ryzen 7 7840HS 的 `/proc/cpuinfo` 输出，包括 flags、bugs、TLB size 等细节。`DUMMY_MEMINFO` 包含 54 项内存统计，`STAT` 包含完整的 bash 进程状态。这些数据来自某台特定开发机，若应用程序（例如 `lscpu`）依赖其中的 cpu MHz 或 cache size，将得到误导性结果。没有 `#[cfg]` 开关允许平台替换这些数据。

**`/proc/self` 语义不一致 <sup>[5](#mod-fs-ref-5)</sup><sup>[7](#mod-fs-ref-7)</sup>**：`proc` 实现中 `/proc/self` 是静态目录，`proc1` 实现中是符号链接。如果两个版本通过 feature flag 切换，用户空间可能观察到截然不同的行为——某些脚本可能期望 `/proc/self` 是 symlink（真实 Linux 行为）。应当统一为符号链接方案。

**错误路径缺失 <sup>[4](#mod-fs-ref-4)</sup><sup>[5](#mod-fs-ref-5)</sup>**：`SimpleFile` 的闭包签名是 `|| -> String`，没有错误返回路径。若 `axtask::get_irq_count()` 或 `current_process().get_pid()` 失败（例如未初始化），闭包将 panic 或返回空字符串。伪文件系统无法向上层返回 EIO 错误。

**`SimpleFile::create` 的第二个参数类型 <sup>[5](#mod-fs-ref-5)</sup>**：`proc1/mod.rs` 中调用 `SimpleFile::create(fs.clone(), NodeType::Symlink, NodePermission::default(), ...)`。查看 `undefined_vfs` 的 API 签名，`SimpleFile::create` 接受 `NodeType` 枚举，但 `NodeType::Symlink` 究竟被解释为什么？如果底层框架对 symlink 不特殊处理（即不会跟随链接），那么这个符号链接仅仅是字符串形式上的模拟，真正的 `readlink()` 系统调用可能无法工作。

**两个 procfs 的可达性 <sup>[8](#mod-fs-ref-8)</sup>**：`proc1/mod.rs` 中 `mod process; mod task_stat;` 声明了子模块，但 `ProcessInfoDir` 引用路径是 `crate::core::fs::imp::proc::process::ProcessInfoDir`——注意路径中 `proc` 而非 `proc1`，说明 `ProcessInfoDir` 结构体定义在 `proc` 模块内。这意味着 `proc1` 依赖 `proc` 中定义的结构，两个实现并非完全独立，增加了重构的耦合成本。

**`/proc/mounts` 的静态伪造 <sup>[4](#mod-fs-ref-4)</sup>**：mounts 文件直接返回硬编码的 `"proc /proc proc rw,nosuid,nodev,noexec,relatime 0 0\n"` 字符串，完全忽略实际挂载状态。如果系统同时挂载了 tmpfs 或 devfs，这些信息将不会出现在 `/proc/mounts` 中，可能误导依赖此文件的监控工具。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块实现了 Linux 信号系统调用的核心逻辑，基于 `axsignal` crate 提供的 `Signo`、`SignalSet`、`SignalInfo`、`SignalStack`、`SignalOSAction` 等抽象，在 `arceos-starry` 内核中为用户进程提供信号发送、阻塞、处理、返回等完整生命周期管理。与基线 ArceOS 的轮询式信号检查不同，starry 通过注册 `POST_TRAP` 回调实现每次用户态返回时的信号递送；同时将 `sigaction` 表存放在进程级数据中、未决信号和阻塞掩码存放在线程级数据中，为多线程信号语义提供了基础支撑。当前实现存在多处简化（Stop/Continue 未实现、权限检查缺失、core dump 空壳），但已经覆盖了 14 个信号相关系统调用，能够运行依赖信号机制的用户态程序。

### 1. 核心抽象与外部依赖

本模块不定义自己的信号类型，而是完全依赖外部 crate `axsignal` 提供的基础抽象：`Signo`（信号编号枚举）、`SignalSet`（位图表示的信号集）、`SignalInfo`（包含信号编号和码值的详细信息）、`SignalStack`（备选栈描述符）以及 `SignalOSAction`（信号处理方式枚举：Terminate/CoreDump/Stop/Continue/Handler）<sup>[1](#mod-signal-ref-1)</sup>。这种设计避免了内核中重复定义信号常量，与 Linux 源码中的 `include/uapi/asm-generic/signal.h` 解耦。同时，模块使用 `linux_raw_sys::general` 中的 `SIG_BLOCK`、`SIG_UNBLOCK`、`SIG_SETMASK` 和 `kernel_sigaction` 结构体，直接与用户态传递的二进制布局对接，减少了序列化开销。

更关键的依赖是 `axhal::trap::register_trap_handler` 和 `POST_TRAP` 钩子<sup>[1](#mod-signal-ref-1)</sup>。通过 `#[register_trap_handler(POST_TRAP)]` 宏，`post_trap_callback` 被注册为所有异常/中断处理完毕后、恢复用户态上下文前调用的回调。这意味着信号检查并非由独立的内核线程或定时器驱动，而是完全附着在已有的 trap 处理流水线上，实现了零额外上下文切换的信号递送路径。

线程/进程级信号的存储通过 `starry_core::task::current_thread_data().signal` 和 `current_process_data().signal` 访问。模块通过 `with_blocked_mut`、`with_stack_mut` 等函数式 API 对信号掩码和备选栈进行原子化读写，避免了直接暴露锁的语义<sup>[2](#mod-signal-ref-2)</sup><sup>[9](#mod-signal-ref-9)</sup>。

### 2. 关键设计取舍

**信号检查时机**：选择 `POST_TRAP` 回调而非像 Linux 那样在 `syscall_exit_work` 中检查的 `TIF_SIGPENDING` 标志位。这种设计简化了架构依赖——只需在 trap 返回路径中插入一个统一的回调，无需为每个架构的异常返回代码添加条件分支。代价是信号无法在中断上下文中立即递送，必须等到下一次用户态 -> 内核 -> 用户态的切换周期。但在单核调度模型中，这一延迟通常可以接受。

**信号动作的降级处理**：`check_signals` 函数根据 `SignalOSAction` 分发<sup>[3](#mod-signal-ref-3)</sup>。对于 `Terminate` 和 `CoreDump`，当前实现直接调用 `sys_exit_impl` 终止当前进程；`CoreDump` 虽然定义了退出码 `0x80 + signo`，但实际 core dump 写入功能被标记为 `TODO`。`Stop` 和 `Continue` 同样未实现：`Stop` 被简化为终止进程，`Continue` 则直接空操作（即忽略 SIGCONT 恢复信号）。这是早期开发阶段的合理裁剪——在缺少进程组作业控制之前，Stop/Continue 无法被正确测试。

**阻塞掩码的持久化与恢复**：`sys_rt_sigsuspend` 的实现值得关注<sup>[8](#mod-signal-ref-8)</sup>。它首先保存当前阻塞掩码，然后替换为用户传入的掩码（并强制移除 SIGKILL 和 SIGSTOP），然后在一个循环中反复调用 `check_signals`。如果检查无可用信号，则调用 `current_process_data().signal.wait_signal()` 阻塞等待。当信号到来时，`check_signals` 会恢复旧阻塞掩码（通过 `restore_blocked` 参数），并使用 `EINTR` 作为系统调用的返回值。这一模式精确模拟了 Linux 的 `sigsuspend` 原子操作。

**权限检查的缺失**：多个发送信号的系统调用（`sys_kill`、`sys_tkill`、`sys_tgkill`、`sys_rt_sigqueueinfo`）都包含 `// TODO: should also check permissions` 注释<sup>[6](#mod-signal-ref-6)</sup>。当前实现假设调用者拥有发送任意信号的权限（忽略 EUID/RUID 检查），这构成了一个显著的安全漏洞——非特权用户进程可以向 init 进程发送 SIGKILL。

### 3. 跨模块协同

**与 trap 处理器的协同**：`post_trap_callback` 被 `register_trap_handler(POST_TRAP)` 宏注册<sup>[1](#mod-signal-ref-1)</sup>。该宏展开后会将被装饰函数的函数指针插入到 `POST_TRAP` 等级的钩子链表中。当 CPU 从用户态陷入内核（无论是 syscall、page fault 还是外部中断），完成处理后，`axhal` 的 trap 退出路径会遍历所有 `POST_TRAP` 钩子并依次调用。`from_user` 参数确保仅在实际从用户态返回时才执行信号检查，避免内核线程误触发。

**与 syscall 分发的协同**：所有 `sys_rt_sig*`、`sys_kill`、`sys_tkill`、`sys_tgkill`、`sys_sigaltstack` 函数均由 `api/src/syscall.rs` 中的系统调用分发表调用（参考用户提供的已实现系统调用列表）。每个函数的签名都对齐 Linux x86_64 的 ABI：第一个参数通过寄存器传递，返回 `LinuxResult<isize>`。`sys_rt_sigreturn` 比较特殊，它接收 `&mut TrapFrame` 参数，由分发器在调用前传入当前 trap 帧的引用，以便函数直接改写寄存器状态。

**与进程/线程管理的协同**：信号发送函数 `send_signal_thread`、`send_signal_process`、`send_signal_process_group` 通过 `undefined_process::process::get_thread_data` 等函数获取目标线程/进程数据。`send_signal_thread` 直接调用 `thread_data.signal.send_signal(sig)`<sup>[7](#mod-signal-ref-7)</sup>，这会将信号追加到线程的未决信号队列中，并可能唤醒等待 `wait_signal` 的线程。而 `sys_rt_sigaction` 操作的是进程级的 `actions` 表，通过 `current_process_data().signal.actions.lock()` 获取互斥锁保护，这保证了同一进程内的所有线程共享信号处理函数配置，符合 POSIX 规范。

**与用户态内存安全的协同**：所有用户态指针都经过 `UserPtr<T>` / `UserConstPtr<T>` 封装的 `ptr::get()` 方法访问。该方法在内部调用 `validate_user_ptr` 并返回 `Result<&mut T>`。如果指针无效或未对齐，则系统调用直接返回 `EFAULT`（通过 `?` 传播）。`nullable` 变体允许空指针（即 `oldact = NULL` 表示只设置不读取）。这一设计将内核空间与用户空间的内存隔离强类型化，避免了手动 `copy_from_user` 容易产生的漏洞。

### 4. 边角细节与不足

**`check_signals` 中的危险代码**：`match os_action` 分支中，`Terminate`、`CoreDump`、`Stop` 都调用了 `sys_exit_impl(0, ..., true)`<sup>[3](#mod-signal-ref-3)</sup><sup>[4](#mod-signal-ref-4)</sup><sup>[5](#mod-signal-ref-5)</sup>。注意 `Stop` 分支直接调用了 `exit` 而非暂停线程，这在当前实现中是一个显著的降级。更严重的是，`sys_exit_impl` 在被调用后不会返回——它会在内部执行线程清理并进入 idle 循环。但 `check_signals` 的返回类型是 `bool`，调用方（`post_trap_callback` 或 `sys_rt_sigsuspend` 的循环）在 `sys_exit_impl` 返回后还会继续执行？实际上 `sys_exit_impl` 内部会调用 `loop {}` 或 `!` 类型函数，因此不会返回。但代码没有使用 `std::process::exit` 或 `unreachable!()` 标记，导致静态检查器可能误认为后续的 `true` 返回值会被执行。

**`sys_rt_sigsuspend` 的无超时循环**：<sup>[8](#mod-signal-ref-8)</sup> 中的 `loop { if check_signals(...) { break; } current_process_data().signal.wait_signal(); }` 缺乏超时机制。`wait_signal()` 被定义为阻塞等待直到有信号入队列。如果信号从未到达（例如被其他线程同步屏蔽），当前线程将永久阻塞。这与 `sigsuspend` 的规范一致（总是等待直到处理一个信号），但在无法预读信号的早期调试阶段，这可能造成难以诊断的挂起。

**`send_signal_process_group` 的部分成功问题**：<sup>[7](#mod-signal-ref-7)</sup> 中 `send_signal_process_group` 遍历进程组内的所有进程并调用 `send_signal_process`，但仅仅累计成功计数，不向调用者报告哪些进程发送失败。更糟糕的是，如果 `send_signal_process` 返回 `Err(LinuxError::EPERM)`（由于线程/进程不存在），该错误会被静默忽略（在 `as usize` 转换中 `false` 被转化为 0）。这违背了 `kill(-pgid, sig)` 应当返回 `EPERM`（如果没有任何进程收到信号）的 POSIX 要求。

**`sys_rt_sigaction` 的锁持有范围**：<sup>[2](#mod-signal-ref-2)</sup> 中 `signal.actions.lock()` 的锁作用域覆盖了整个函数，包括用户态内存读写（`unsafe { &mut *oldact }` 和 `unsafe { (*act).try_into()? }`）。如果用户态内存访问触发 page fault（例如传入的指针指向交换出的页面），内核将在持锁状态下进入 page fault handler。若该 handler 尝试获取同一把锁（例如通过另一个系统调用间接调用 `sigaction`），将导致死锁。虽然当前实现中 page fault handler 不会递归获取 actions 锁，但这是一个潜在的脆弱点，应当缩小锁范围或将用户态拷贝移到锁外。

**静态分析契约缺失**：多处 `unsafe` 块没有 `// SAFETY:` 注释，例如 `unsafe { *oldset = *blocked }`、`unsafe { (*act).try_into()? }`<sup>[2](#mod-signal-ref-2)</sup><sup>[8](#mod-signal-ref-8)</sup>。虽然这些操作的合理性可以推断（`oldset` 是用户态有效指针，`blocked` 已借出），但缺乏安全注释在团队协作中构成技术债，也给未来的 audit 增加难度。

**架构特定硬编码**：代码使用了 `linux_raw_sys::general` 中的常量（如 `SI_TKILL`、`SI_USER`、`MINSIGSTKSZ`）和类型（`kernel_sigaction`、`siginfo`、`timespec`），这些在 `linux_raw_sys` 中针对 x86_64 架构定义。如果移植到 AArch64 或 RISC-V，需要确认这些头文件的兼容性。此外，`TrapFrame` 来自 `axhal::arch`，其结构是架构相关的——`tf.retval()` 和 `tf.set_retval()` 的语义在 x86_64 上对应 RAX，在其他架构上未必正确。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块以 System V 共享内存为核心，提供基于 key 的共享内存段创建、获取、删除机制。核心抽象是 SharedMemoryManager，封装了 BTreeMap 管理的 Arc<SharedMemory> 键值对，并通过全局静态 SHARED_MEMORY_MANAGER 暴露。区别于完整 Linux IPC（信号量、消息队列、futex），当前仅实现了共享内存子系统，且地址映射采用内核线性分配（无用户态页表映射），存在明显的早期简化痕迹。

### 1. 核心抽象：基于 key 的共享内存管理器

共享内存子系统的入口是全局静态 `SHARED_MEMORY_MANAGER`<sup>[1](#mod-ipc-ref-1)</sup>，类型为 `SharedMemoryManager`。该结构体内部持有 `Mutex<BTreeMap<u32, Arc<SharedMemory>>>` 用于以 key 索引共享内存段，以及一个 `AtomicU32` 类型的 `next_key` 用于生成新 key。每个 `SharedMemory` 实例记录 key、内核虚拟地址和页数，其 `Drop` 实现自动归还页面到全局分配器<sup>[2](#mod-ipc-ref-2)</sup>。

`create` 方法通过 `global_allocator().alloc_pages` 分配连续物理页，返回 `LinuxResult<Arc<SharedMemory>>`。`get` 和 `delete` 分别封装 BTreeMap 的查询与删除。这种设计使得共享内存的生命周期由 `Arc` 引用计数托管：当所有引用释放时自动回收内存，这与 Linux 内核中基于引用计数的 `shm_file` 机制思路一致，但实现上跳过了页表映射与进程地址空间挂载。

### 2. 关键设计取舍与未竟之处

**线性分配 vs. 进程地址空间映射**：`create` 直接调用 `global_allocator().alloc_pages` 获取一段内核线性地址<sup>[3](#mod-ipc-ref-3)</sup>。这意味着共享内存仅存在于内核空间，用户进程无法通过 `shmat` 将其附加到自己的虚拟地址空间——当前实现只是“分配了一块内核内存并记录 key”，真正的 `shmat` 系统调用需要将这块内存映射到用户页表，而这部分逻辑尚未出现。这是一个明显的简化取舍：放弃了用户态访问能力，换取了实现复杂度的大幅降低。

**key 生成算法缺陷**：`next_available_key` 的实现存在逻辑错误。它先 `fetch_add` 自增原子计数器，再取当前 map 中最大 key 加上自增后的 `next_key` 作为新 key<sup>[4](#mod-ipc-ref-4)</sup>。这导致 key 不是简单递增，而是 `max(existing_keys) + next_key`，且 `fetch_add` 的副作用可能造成 key 跳跃增长。更严重的是，如果 map 为空，`keys().max()` 返回 `None`，此时 `unwrap_or(&0)` 得到 0，加上 `next_key`（最小为 1），结果从 1 开始。但连续调用时，由于 `fetch_add` 不断自增，key 值会快速膨胀到不合理的数字。这很可能是一个 bug——预期行为应是返回当前最大 key+1 或简单使用原子递增的序列号。该函数目前未被 `create` 调用（`create` 要求调用者传入 key），因此暂时不会触发问题，但 `shmget` 的 IPC_PRIVATE 或 key=IPC_PRIVATE 路径若使用此函数则会出错。

**错误检查缺失**：`create` 中明确标注 `// TODO: more error checking`<sup>[5](#mod-ipc-ref-5)</sup>，目前只检查了 `alloc_pages` 失败（返回 ENOMEM），没有检查 key 是否已存在（重复创建会直接覆盖旧条目，导致旧段泄漏），也没有检查 `size` 是否为 0 或超过系统上限。

**全局锁粒度**：整个 `mem_map` 使用一个 `Mutex` 保护，在 `get`、`create`、`delete` 操作期间都持有锁。由于 `create` 内部调用 `alloc_pages`（可能阻塞），持锁期间不应执行可能阻塞的操作——但当前 `alloc_pages` 在锁内调用，可能引起锁持有时间过长。更优设计是 `create` 先分配再插入，但当前顺序如此。

### 3. 与系统调用层的协同

根据系统调用表，共享内存相关的四个系统调用 `shmget(194)`, `shmat(196)`, `shmctl(195)`, `shmdt(197)` 已被声明<sup>[6](#mod-ipc-ref-6)</sup>。其中 `shmget` 应调用 `SHARED_MEMORY_MANAGER.create` 或 `get`；`shmctl` 应调用 `delete`；`shmat` 和 `shmdt` 需要将内核地址映射到用户页表，但当前 `SharedMemory` 只持有内核线性地址，没有提供 `mmap` 接口或进程页表操作。系统调用表还列出了信号量集合（`semget`/`semctl`，行 485/502）和消息队列（`msgget`/`msgctl`/`msgsnd`/`msgrcv`，行 477-493）以及 POSIX 消息队列（`mq_open`/`mq_unlink`/`mq_timedsend`/`mq_timedreceive`，行 459-463）和 `futex`（行 208）<sup>[7](#mod-ipc-ref-7)</sup>。这些系统调用的存在暗示内核有更完整的 IPC 框架意图，但当前代码库中仅共享内存模块有实际实现，其他可能处于 stub 或未完成状态。

futex 的实现通常需要与内核调度器和用户态原子操作紧密配合。系统调用表标注 `futex(98) → src/syscall.rs:208`，但 grep 该文件不在可读范围内，无法确认实现深度。同样，`get_robust_list`/`set_robust_list`（行 422-423）是 futex 的配套机制，用于处理线程退出时的锁清理。这些系统调用的存在表明该仓库可能已经在进行 futex 的支持，但代码尚未成熟。

### 4. 边角细节与不足

- **<sup>[1](#mod-ipc-ref-1)</sup> 全局静态变量**：`SHARED_MEMORY_MANAGER` 定义为 `static` 而非 `static mut`，使用了 `Mutex` 内部可变性，这是 Rust 安全实践。但 `SharedMemoryManager::new` 是 `const fn`，允许静态初始化，这是合理的。
- **<sup>[2](#mod-ipc-ref-2)</sup> Drop 中的 info!**：`SharedMemory::drop` 中打印 info 日志，这在生产环境中可能产生大量输出，但有利于调试。注意 `drop` 在 `Arc` 引用归零时自动调用，无需显式 `delete`。
- **<sup>[4](#mod-ipc-ref-4)</sup> key 生成逻辑**：如前所述，`next_available_key` 的算法异常，是明显的技术债。将来若实现 `IPC_PRIVATE` 必须修复。
- **<sup>[5](#mod-ipc-ref-5)</sup> TODO 注释**：`create` 中的 TODO 直接指出了不完整性。此外没有对 `size` 的范围检查，如果用户传 0，`div_ceil(0, PAGE_SIZE_4K)` 返回 0，分配 0 页可能返回错误或成功分配空内存，行为未定义。
- **<sup>[8](#mod-ipc-ref-8)</sup> 缺乏进程隔离**：当前共享内存仅存在于内核地址空间，没有与任何进程的页表关联。`shmat` 系统调用需要执行 `mmap` 操作将共享内存映射到用户空间，但 `SharedMemoryManager` 不提供任何页表操作 API。这意味着共享内存实际上不可被用户态进程访问——模块名“共享内存”名不副实，实际上仅是“内核内存”。

总体而言，该模块处于早期开发阶段，仅完成了共享内存的框架性实现，其他 IPC 机制仅有系统调用声明而无实质代码。

- 信号量、消息队列、futex 系统调用已声明但对应模块代码缺失或不可见。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块通过 `Socket` 枚举封装 `axnet` crate 的 `TcpSocket` 和 `UdpSocket`，实现 Linux socket 系统调用。核心抽象是 `Socket` 类型，它实现了 `FileLike` trait，从而与通用 fd 表集成。与 ArceOS 基线相比，本模块在 `sendto`/`recvfrom` 中自动绑定回环地址，并提供了带 `ref_` 计数的 `getaddrinfo` 实现，但大量选项（如 `setsockopt`）仍为桩实现，且存在内存泄漏风险。

### 1. 核心抽象：Socket 枚举与 FileLike 集成

本模块的核心是 `api/src/imp/net/socket.rs` 中的 `Socket` 枚举，它包含两个变体：`Udp(Mutex<UdpSocket>)` 和 `Tcp(Mutex<TcpSocket>)` <sup>[1](#mod-net-ref-1)</sup>。这种设计将底层网络栈（`axnet` crate）的套接字包装为文件描述符。`Socket` 实现了 `FileLike` trait（文件抽象接口），通过 `read`/`write` 委托给 `recv`/`send`，并实现 `poll`、`status` 等 <sup>[2](#mod-net-ref-2)</sup>。这使得套接字可以无缝接入 `fd_add`/`fd_lookup` 机制，被通用文件系统管理。

`Socket` 的方法（如 `bind`、`connect`）通过内部模式匹配分派到具体的 `TcpSocket` 或 `UdpSocket`，使用 `Mutex` 保证线程安全。这种设计与 ArceOS 基线一致，但本模块在 `sendto` 和 `recvfrom` 中引入了自动绑定到回环地址的独特处理。

### 2. 关键设计取舍：自动绑定、错误码映射与地址转换

**自动绑定**：在 `sendto` 中（`sys_sendto` 调用），如果套接字是 UDP 类型且尚未绑定，模块会强制绑定到 `127.0.0.1:0` <sup>[3](#mod-net-ref-3)</sup>。这是因为底层 `axnet::UdpSocket` 在发送前必须绑定，而标准 Linux 允许未绑定的 `sendto` 由内核自动分配地址。这种取舍简化了实现，但限制了出站 UDP 的源地址为回环，在多网卡场景下可能不符合预期。

**错误码映射**：模块将 `axnet` 的错误（如 `axerrno::AxError`）通过 `?` 操作符转为 `LinuxError`，但在多处使用了 `.map_err(|_| LinuxError::EISCONN)` 覆盖底层错误 <sup>[4](#mod-net-ref-4)</sup>。例如 `sendto` 中 `bind` 失败也被映射为 `EISCONN`，这不够精确。

**地址转换**：`from_sockaddr` 和 `into_sockaddr` 函数在 `sockaddr_in` 和 `SocketAddr` 之间转换 <sup>[5](#mod-net-ref-5)</sup>。它们通过指针强制转换和内存复制实现，其中 `into_sockaddr` 使用 `unsafe` 块将 `sockaddr_in` 转换为 `sockaddr`，但未检查内存对齐或大小。`from_sockaddr` 只支持 IPv4，IPv6 会引发 `panic!`。

### 3. 跨模块协同：fd 表、axnet 与 axsync

网络模块与文件描述符表 (`crate::core::file::fd`) 紧密协作：`Socket::add_to_fd_table` 调用 `fd_add` 将 `Arc<Socket>` 插入全局 fd 表 <sup>[6](#mod-net-ref-6)</sup>；系统调用通过 `fd_lookup` 获取 `Arc<Socket>`，再 `downcast` 为 `Socket` <sup>[7](#mod-net-ref-7)</sup>。这要求 `Socket` 实现 `Send + Sync` 且存储为 `Arc`。

底层通信依赖 `axnet::TcpSocket` 和 `axnet::UdpSocket`，它们封装了 `smoltcp` 或类似协议栈。模块未直接操作 `smoltcp`，而是通过 `axnet` 的 API。`axsync::Mutex` 替代了 `std::sync::Mutex`，适用于无标准库环境。

`getaddrinfo`（`sys_getaddrinfo`）与 DNS 查询集成，通过 `axnet::dns_query` 解析域名 <sup>[8](#mod-net-ref-8)</sup>。该函数返回 `Vec<IpAddr>`，模块仅取前 `MAXADDRS` 个，并填充到 `addrinfo` 链表。内存管理采用 `Vec::forget` 配合 `sys_freeaddrinfo` 中的 `Vec::from_raw_parts` 释放 <sup>[9](#mod-net-ref-9)</sup>，但未加锁，存在并发竞争。

### 4. 边角细节与不足 [ref:N]

**1. `unsafe` 缺少安全注释**：`into_sockaddr` 中的 `unsafe` 块 <sup>[5](#mod-net-ref-5)</sup> 没有 `// SAFETY:` 注释，违反安全契约。指针强制转换假设 `sockaddr_in` 布局与 `sockaddr` 兼容，但这依赖于 ABI 细节。

**2. `setsockopt` 是完整桩**：`Socket::setsockopt` 对所有级别和所有套接字都只打印 `warn!` 并返回 `Ok(())` <sup>[10](#mod-net-ref-10)</sup>。这意味着 `SO_REUSEADDR`、`TCP_NODELAY` 等常见选项被静默忽略，可能导致网络行为异常。

**3. `getaddrinfo` 内存泄漏风险**：`sys_getaddrinfo` 使用 `core::mem::forget` 避免过早释放 <sup>[9](#mod-net-ref-9)</sup>，但若 `sys_freeaddrinfo` 未被正确调用（例如用户进程崩溃），泄漏的内存无法回收。此外，`Vec::from_raw_parts` 未考虑内存对齐和 `lock` 字段，存在未定义行为。

**4. `shutdown` 实现不合理**：`Socket::shutdown` 先调用 `peer_addr()` 检查是否已连接，但若未连接则返回错误，而 Linux 的 `shutdown` 允许未连接套接字 <sup>[11](#mod-net-ref-11)</sup>（行为未定义但通常不返回错误）。这导致对未连接套接字调用 `shutdown` 会返回意外错误。

**5. 硬编码地址与仅 IPv4**：`sendto` 自动绑定写死 `127.0.0.1:0` <sup>[3](#mod-net-ref-3)</sup>，`from_sockaddr` 和 `into_sockaddr` 只支持 IPv4，IPv6 会 panic <sup>[5](#mod-net-ref-5)</sup>。这与现代 Linux 的全面支持差距大。

**6. 死代码与未实现**：`FileLike::get_flags` 返回空标志 <sup>[12](#mod-net-ref-12)</sup>；`set_flags` 只打印警告；`set_nonblocking` 方法被注释掉 <sup>[13](#mod-net-ref-13)</sup>。这些功能对网络套接字是必要的，缺失会影响非阻塞 I/O 和 fd 标志正确性。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层是 Starry 内核的用户态入口，通过 `#[register_trap_handler(SYSCALL)]` 将 `handle_syscall` 函数注册为 trap 处理函数。核心抽象是一个大型 match 表，将 Linux 系统调用号 (`Sysno`) 映射到 `api/src/imp/` 下各模块的具体实现函数。与 ArceOS 基线相比，Starry 系统调用层实现了超过 100 个系统调用，覆盖文件系统、网络、信号、进程管理、内存管理等领域，并引入了 `stub_bypass` 和 `stub_unimplemented` 策略处理未就绪的调用，同时通过条件编译适配 x86_64、riscv64、aarch64、loongarch64 等架构的 ABI 差异。

### 1. 入口分发与统一错误模型

系统调用层的核心是 `src/syscall.rs` 中的 `handle_syscall` 函数。它通过 `#[register_trap_handler(SYSCALL)]` 属性宏连接到 axhal 的 trap 子系统，每次用户态发起系统调用时自动被调用。函数签名接收 `&mut TrapFrame` 和系统调用号 `syscall_num`，返回 `isize`。

系统调用号首先通过 `syscalls::Sysno::new` 转换为枚举值，若无效则默认 fallback 到 `Sysno::memfd_secret`（一个无害的 stub），避免未定义行为。随后通过一个巨大的 `match` 语句分发到各个具体实现，每个分支返回 `LinuxResult<isize>`。最终结果通过 `unwrap_or_else` 将错误转换为负的 Linux errno 值（如 `-EINVAL`）。这种设计统一了错误处理，但丢失了错误来源的上下文信息，且没有使用 `map_err` 链式附加调试信息。

系统调用的进入和退出均通过 `info!` 日志记录，并调用 `time_stat_from_user_to_kernel` 和 `time_stat_from_kernel_to_user` 进行性能统计，体现了对可观测性的重视。

### 2. 架构条件编译与 stub 策略

Linux 系统调用的参数布局在不同架构上存在差异，Starry 通过 `#[cfg]` 条件编译处理。例如 `clone` 系统调用在 x86_64/loongarch64 上的参数顺序与 riscv64/aarch64 不同 <sup>[1](#mod-syscall-ref-1)</sup>；`dup2` 仅在 x86_64 上直接实现，其他架构依赖 `dup3` 模拟 <sup>[7](#mod-syscall-ref-7)</sup>。这种条件编译正确反映了 ABI 差异，但也意味着某些系统调用在非 x86_64 架构上可能缺失：例如 `access` 在 x86_64 上被标记为 `stub_bypass`（返回 0），而在其他架构上通过 `faccessat` 实现，这种不对称可能导致兼容性问题。

对于大量尚未实现的系统调用，代码采用了三种 stub 策略：
- `stub_bypass`：返回 `Ok(0)`，假装成功，适用于 `sync`、`fsync`、`setuid` 等对正确性影响不大的调用 <sup>[4](#mod-syscall-ref-4)</sup>。
- `stub_unimplemented`：返回 `ENOSYS`，适用于真正未实现的调用。
- 直接返回特定错误码：如 `getsockopt` 返回 `EFAULT`，`getpeername` 返回 `EFAULT`，这些调用如果被库频繁调用可能导致应用异常退出。

这种分层策略显示了开发者的取舍：优先保证常见 workload 的通过率，而对可能暴露实现缺陷的调用则严格拒绝。

### 3. 跨模块协同：文件描述符、信号、内存

系统调用层本身是薄调度层，核心逻辑委托给 `api/src/imp/` 下的专业模块。三个关键的协同模式如下：

**文件描述符系统**：`sys_socket` 在创建 `Socket` 后调用 `add_to_fd_table` 将其注册到进程的文件描述符表 <sup>[2](#mod-syscall-ref-2)</sup>。该表由 `api/src/core/file/fd` 管理，所有文件类对象（普通文件、socket、epoll 等）均实现 `FileLike` trait，使得系统调用如 `read`、`write`、`poll` 可以统一通过 `fd_lookup` 获取对象再调用对应方法。

**信号处理**：信号系统通过 `#[register_trap_handler(POST_TRAP)]` 注册 `post_trap_callback`，在每次 trap 返回用户态前检查待处理信号 <sup>[5](#mod-syscall-ref-5)</sup>。`check_signals` 函数从当前线程的信号状态中取出信号，根据 `SignalOSAction` 决定终止、core dump、停止或调用用户态 handler。这种机制与 Linux 的 `do_signal` 逻辑对应，但实现尚有不完整之处（如 `CoreDump` 和 `Stop` 直接调用 `sys_exit_impl` 而非真正 dump 或暂停）。

**内存管理**：`sys_mmap` 操作进程地址空间（通过 `current_process_data().addr_space.lock()` 获取），利用 `axhal::paging` 模块进行页表操作 <sup>[6](#mod-syscall-ref-6)</sup>。特别地，它支持通过 `get_device_by_fd` 获取设备内存并直接线性映射，避免拷贝。但 `MAP_SHARED` 分支下的 TODO 注释表明当前实现未区分匿名映射和文件映射，总是填充零页，可能造成性能浪费。

### 4. 边角细节与待改进之处

通过逐行审查代码，发现以下值得注意的细节与不足：

- **`stub_bypass` 的安全隐患**：`sync`、`setuid` 等调用被绕过（返回 0），若应用依赖其语义（如数据库要求 fsync 持久化），则数据完整性无法保证。虽然多数用户态程序不直接调用这些系统调用，但通过 libc 可能会间接触发。

- **信号处理中的 TODO**：`sys_kill` 对 `pid == -1` 的分支未跳过 init 进程（注释 `// TODO: skip init process?`），可能误杀关键进程；且权限检查被完全省略。

- **mmap 对 MAP_SHARED 的零填充**：代码中注释 `// TODO: 仅在MAP_ANONYMOUS时才zero`，但实际实现无论是否匿名都调用 `map_shared` 并填充零。对于共享文件映射，这会导致已映射的文件内容被零覆盖，是明显的行为错误。

- **架构条件编译的不一致**：`dup2` 只在 x86_64 上定义，但 `dup3` 在所有架构上存在且可以模拟 `dup2`。若 riscv64 上的应用直接调用 `dup2` 系统调用（而非 libc 封装的 `dup2`），将进入 `stub_unimplemented` 返回 ENOSYS。同样，`access` 在 x86_64 上被 stub，但在其他架构上通过 `faccessat` 实现，这种不对称可能导致跨架构测试遗漏。

- **`#[syscall_trace]` 与日志重复**：部分函数（如 `sys_kill`、`sys_mmap`）标记了 `#[syscall_trace]` 属性，推测用于自动记录系统调用参数。但 `handle_syscall` 已经通过 `info!` 记录了所有调用的进入和返回，两者功能重叠，且 `syscall_trace` 宏的定义未在当前代码库中出现，可能存在维护隐患。
