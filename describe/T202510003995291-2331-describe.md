## 📖 一、总览

该仓库在 ArceOS 的 Unikernel 基础上构建了一套 Linux 系统调用兼容层，通过引入进程模型、信号机制和 VFS 框架，尝试支持复杂用户态程序（如 busybox）。相比家族基线，其核心突破在于通过 undefined_process 实现了进程生命周期管理（fork/clone/execve）和惰性信号递送，并在文件系统层以 DynamicFs 快速搭建伪文件系统。然而，实现中大量保留桩代码，关键机制如 CoW、进程组信号、setsockopt 等或缺失或返回假成功，整体呈现“原型验证”阶段的实验型系统特征，尚未达到生产级可靠性。最值得评审注意的三个特点：1）利用 POST_TRAP 钩子与信号跳板（trampoline）机制，在用户态返回路径上惰性递送信号，且支持进程组广播，这在 Unikernel 场景下是极少见的工程实践；2）通过 FileLike 统一抽象文件、socket、pipe 等资源，使得 I/O 多路复用可以一致处理，同时保留两个 procfs 实现（静态/动态）展示了从硬编码向动态信息采集过渡的尝试；3）有意识的“桩策略”——212 项系统调用中近半数仅返回成功，却聚焦实现了 fork/execve/futex/mmap 等核心子集，体现了“先跑通再完善”的竞赛式开发思维，但 syscall 层的宽进宽出也留下了数据完整性与安全性隐患。

## 🎯 二、综合评价

### 整体定位

该仓属于比赛驱动型原型系统，目标是在有限时间内使 ArceOS 能够运行 busybox 以通过特定测试用例。它不是完整 Linux 兼容层，而是选择性实现系统调用子集，并大量依赖静态链接和 Unikernel 假设（如单一地址空间、急切内存分配）。与教学型 OS 不同，其代码组织复杂，包含多 crate workspace，引入了进程、信号、VFS 等重型子系统，但成熟度不足，多处存在未完成标记和架构性缺陷。整体上，它将 ArceOS 从“裸机运行时”强行推向了“多进程操作系统”的方向，但距离稳定可用仍有明显差距。

### 真正的创新点

1. **惰性信号递送与跳板机制**：与仅在系统调用返回时检查信号不同，本仓利用 POST_TRAP 钩子在每次异常/中断返回用户态时检查并递送信号[signal.ref:3]，并专门映射了信号跳板页以支持用户态信号处理函数[mm.ref:5]。这使得信号处理可以与调度、进程组等机制联动，比 ArceOS 家族中常见的简单占位符实现要完整得多。
2. **进程模型的集成**：通过 undefined_process 引入了进程组、会话、父子关系等 POSIX 概念，并实现了 fork/clone 的命名空间共享标志。这标志着从单任务的 Unikernel 向多进程 OS 的跨越，在 arceos-starry 家族中属于罕见尝试。
3. **可组合的 VFS 与双 procfs 尝试**：以 FileLike 统一抽象，使得文件、socket、管道等均可用文件描述符表管理[fs.ref:1]。同时存在两个 procfs 实现（静态/动态）[fs.ref:5]，展示了从硬编码向动态信息采集过渡的工程意识，尽管动态版本目前仍大量回退到硬编码，但模块化设计为未来改进留出了空间。

### 取舍判断

- **内存管理：急切分配，放弃 CoW/按需调页**。所有 mmap 均立即分配物理页，brk 仅移动指针，直接导致物理内存利用率极低，但换取了实现简单性[mm.ref:4]。这种取舍在原型阶段可以接受，但若扩展到多进程场景，内存浪费将急剧膨胀。
- **网络：功能残缺，仅够 TCP/UDP 基础通信**。setsockopt 几乎全部空实现，UDP 发送端强制绑定本地地址，非阻塞模式被注释掉，明显只追求“能发包”而忽略 POSIX 完备性，反映出开发者优先保障核心路径的应试策略。
- **系统调用：宽进宽出，桩策略激进**。stub_bypass 对 sync、fsync 等关键调用返回成功[syscall.ref:5]，setsockopt 对未知 level 直接 panic，提示开发者倾向于快速通过测试而非保证正确性，这在比赛环境中可理解但留下了严重隐患。
- **文件系统：依赖静态数据**。`/proc/cpuinfo`、`meminfo` 等均为硬编码，在非 x86 平台上提供虚假信息，进一步印证了“面向测试”的实现哲学。

### 完成度与不足

完成度方面，212 项系统调用中约 40% 为桩，但核心的进程管理、内存映射、基本文件 I/O 和 futex 已实现，足以运行静态链接的 busybox。不足集中在：信号处理中 Stop/Continue/CoreDump 均未实现，权限检查缺失[signal.ref:5]；进程模型存在全局锁竞争、硬编码 PID 等粗糙设计；代码质量堪忧，多处 `unwrap()` 可能导致内核 panic，大量 TODO/FIXME，两个 procfs 并存增加维护负担；异步 I/O、共享内存等子系统仅有骨架，远非可用状态。总体来看，该系统处于“可演示”但“不可靠”的阶段，若要进化到实用水平，需对内存管理、信号、网络等核心模块进行深度重构。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 本模块实现 unikernel 风格的启动流程: 内核入口 `main()` 直接初始化 OS 服务(根文件系统、fd 表、挂载点)并运行单个用户应用. 核心抽象是 `run_user_app` —— 它串联地址空间创建 (含信号跳板映射)、ELF 加载、进程/线程构造、per-process 命名空间初始化, 最终通过 axtask 孵化任务并阻塞等待退出. 与 ArceOS 基线相比, starry 家族引入了 `undefined_process` 进程模型、基于命名空间的 FD_TABLE/CURRENT_DIR 隔离, 以及 axsignal 信号跳板 (trampoline) 机制, 启动流程因此多了进程 spawn、thread_data 组装等步骤.

### 1. 入口点与最小化平台初始化

内核入口 `main()` 位于 `src/main.rs:27` <sup>[1](#mod-boot-ref-1)</sup>, 以 `#[unsafe(no_mangle)]` 导出, 确保链接器能找到该符号. 函数首先完成三个关键初始化: (1) 通过 `FS_CONTEXT.lock().change_root(root_dir)` 设置根文件系统上下文 <sup>[2](#mod-boot-ref-2)</sup>; (2) `FD_TABLE.init_new(FdTable::new())` 创建全局 fd 表; (3) `mount_all()` 挂载所有文件系统. 这一顺序隐含了对 axfs_ng 的依赖 —— 根目录必须先就位, fd 表才能基于挂载点解析路径.

与标准 ArceOS 的 `rust_main` 不同, 此处直接以 `main` 为符号名, 且未显式调用 axhal 的平台初始化 (`axhal::platform_init`). 实际上平台初始化已在更早的阶段 (crt0 / bootloader) 完成, `main` 只负责 OS 层面的服务就绪. 随后从编译期环境变量 `AX_TESTCASES_FILE` 读取 shell 命令 <sup>[9](#mod-boot-ref-9)</sup>, 构造 `args`/`envs` 后调用 `run_user_app`.

### 2. 用户地址空间的构建与跳板机制

`run_user_app` (位于 `core/src/entry.rs:11`) 是启动流程的核心. 地址空间构建分为三步 <sup>[3](#mod-boot-ref-3)</sup>:

1. `new_user_aspace_empty()` 创建空用户页表;
2. `copy_from_kernel()` 将内核数据 (可能是 vDSO 或辅助向量) 拷贝到用户空间;
3. `map_trampoline()` 映射信号跳板页.

跳板 (trampoline) 是 starry 家族的独特设计: 当信号需要递送时, 内核操纵用户栈, 使信号处理函数返回后自动执行跳板代码, 跳板再通过 `sigreturn` 系统调用恢复上下文. 跳板必须在用户空间有一个固定的可执行映射, `map_trampoline` 正是为此预留虚拟地址. 这避免了内核在信号递送时动态分配用户页的复杂性, 但也牺牲了 ASLR 的彻底性 <sup>[3](#mod-boot-ref-3)</sup>.

地址空间就绪后, 设置当前目录 (`axfs::api::set_current_dir`) 并加载 ELF: `load_user_app` 返回入口虚拟地址和用户栈顶. `UspaceContext::new(entry_vaddr.into(), ustack_top, 2333)` <sup>[5](#mod-boot-ref-5)</sup> 将这两个地址封装为硬件相关的用户上下文; 第三个参数 `2333` 是一个硬编码的 PID 占位符, 实际进程 ID 由后续的 `Process::spawn_process()` 分配 <sup>[6](#mod-boot-ref-6)</sup>, 此处存在语义重叠.

### 3. 进程/线程模型与命名空间隔离

starry 引入了两层抽象: `undefined_process::process::Process` (进程) 和 axtask (调度任务). 启动流程中, `Process::spawn_process()` 创建进程对象, 获取主线程 <sup>[6](#mod-boot-ref-6)</sup>; 同时 `create_user_task` 基于 `UspaceContext` 构造 axtask 任务 <sup>[10](#mod-boot-ref-10)</sup>. 两者通过 `TaskExt` 桥接: `TaskExt::new(thread, thread_data)` 将进程的线程信息与 per-task 扩展数据绑定.

`ProcessData` 封装了命令行参数、用户地址空间 (Arc<Mutex<>>)、信号配置 (默认 SIGCHLD) <sup>[7](#mod-boot-ref-7)</sup>. `create_thread_data` 进一步生成线程级元数据, 其内嵌的 `namespace` 用于初始化 per-process 的 `FD_TABLE`、`CURRENT_DIR`、`CURRENT_DIR_PATH` <sup>[8](#mod-boot-ref-8)</sup> — 这实现了 POSIX 语义下的文件描述符和当前目录隔离. `deref_from` 模式使得这些全局变量能在不同进程/线程间切换, 但依赖 `thread_data` 的生命周期, 若线程提前销毁可能导致悬垂.

### 4. 跨模块协同全景

启动流程串联了至少六个外部 crate:
- **axhal**: 提供 `UspaceContext`, 封装架构相关的用户态寄存器布局;
- **axtask**: 任务 spawn 与 `join` 阻塞等待;
- **axfs / axfs_ng**: 文件系统挂载 (`mount_all`) 与目录切换;
- **axsignal**: 信号编号 (`Signo::SIGCHLD`) 用于进程间通信;
- **axmm**: 内核地址空间 (`kernel_aspace`), 用户态页表根物理地址的设置;
- **undefined_process / undefined_os_api**: 提供 `Process`、`FdTable`、命名空间等 POSIX 抽象.

`main()` 中 `FS_CONTEXT.lock().change_root(root_dir).unwrap()` 的调用看似冗余 (将 root_dir 再设为 root), 实际可能是为触发 axfs_ng 内部路径解析缓存的重置, 或初始化当前线程的 fs 上下文 <sup>[2](#mod-boot-ref-2)</sup>. 若 `mount_all` 依赖 `FS_CONTEXT` 已处于有效状态, 这个顺序就是强制约束.

### 5. 边角细节与不足

1. **硬编码 PID** <sup>[5](#mod-boot-ref-5)</sup>: `UspaceContext::new` 的第三个参数 `2333` 是 magic number, 注释缺失. 若后续进程 ID 分配与此冲突, 可能导致调试混乱. 应改为从 `Process` 获取实际 pid.

2. **panic 密集的错误处理**: `run_user_app` 对地址空间创建、ELF 加载、目录设置均使用 `expect`/`unwrap` <sup>[3](#mod-boot-ref-3)</sup><sup>[4](#mod-boot-ref-4)</sup>. 一旦 musl/busybox 路径不存在或 ELF 格式异常, 内核直接 panic, 无任何恢复或错误报告路径. 这在教学场景可接受, 但在需要健壮性的场景应返回 `Err` 并清理资源.

3. **`path.parent().unwrap()` 风险** <sup>[4](#mod-boot-ref-4)</sup>: 若 `args[0]` 是根路径 `/`, `parent()` 返回 `None`, 触发 panic. 虽然实际测试用例不太可能传 `/` 作为 busybox 路径, 但代码未做防御.

4. **冗余的 change_root** <sup>[2](#mod-boot-ref-2)</sup>: `FS_CONTEXT.lock().change_root(root_dir).unwrap()` 将根目录设置为自身, 疑似初始化 side effect 的 workaround. 若 `change_root` 有内部状态变更 (如更新 inode cache), 应抽离为显式的 `init_root` 接口.

5. **被注释的 LD_LIBRARY_PATH** <sup>[9](#mod-boot-ref-9)</sup>: `envs` 中 `LD_LIBRARY_PATH` 和 `LD_DEBUG` 被注释, 暗示动态链接器调试曾有问题. 若 musl 的动态链接依赖这些环境变量, 其缺失可能导致某些动态库加载失败, 目前靠 busybox 的静态链接规避.

6. **`#[unsafe(no_mangle)]` 语法** <sup>[1](#mod-boot-ref-1)</sup>: 这是 Rust 2024 edition 的 unstable 特性. 生产环境若稳定版编译器不支持, 需回退到 `#[no_mangle]` + `extern "C"` 的组合, 或在 `Cargo.toml` 中锁定 nightly.

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块以 `axmm::AddrSpace` 为核心抽象, 封装进程虚拟地址空间的创建、映射与销毁. 对外暴露 Linux 兼容的 mmap/munmap/mprotect/brk 系统调用, 内部通过 `MmapProt`/`MmapFlags` 两层 bitflags 将 Linux 语义翻译为 `axmm` 的 `map_alloc`/`map_linear`/`unmap`/`protect` 原语. 相比 ArceOS 基线, 本仓库新增了设备内存线性映射快路径、shebang 脚本递归加载、percpu 用户态内存访问门控(允许内核态触发用户页错误)以及大页支持支架. 核心取舍是采用急切分配策略(无按需调页、无 CoW), 以简化实现换取兼容性折损.

### 1. 核心抽象: AddrSpace 与两级语义翻译

内存管理的中心类型是 `axmm::AddrSpace`——一个 per-process 的虚拟地址空间句柄, 内部封装了页表. 每个进程通过 `current_process_data().addr_space.lock()` 获取 `Mutex` 保护的地址空间引用, 随后通过三类原语操作 <sup>[1](#mod-mm-ref-1)</sup>:

- `map_alloc`: 分配物理帧并映射到虚拟地址 (用于匿名映射与 ELF 加载)
- `map_linear`: 将已知物理地址线性映射 (用于设备内存与信号跳板)
- `map_shared`: 共享映射 (当前实现与 map_alloc 几乎一致, 标注 TODO)

Linux 语义由两个 bitflags 结构体承担协议转换: `MmapProt` (保护位) 与 `MmapFlags` (映射标志). 其中 `MmapProt` 通过 `From` trait 转换为 `axhal::paging::MappingFlags` 时自动附加 `USER` 标志, 使调用方无需显式设置用户态访问权限 <sup>[2](#mod-mm-ref-2)</sup>. 这种分层将平台无关的 Linux ABI 与平台相关的页表操作解耦, 但代价是 `MmapFlags` 中的 `MAP_HUGETLB`/`MAP_STACK`/`MAP_NORESERVE` 等标志大部分未被实际处理——仅大页标志影响了 `PageSize` 选择, 其余被静默忽略.

### 2. 关键设计取舍

**2.1 架构差异的门控**

`copy_from_kernel` 体现了 aarch64/loongarch64 与其他架构的根本分歧: 前者使用独立的用户态页表寄存器 (TTBR0_EL1 / PGDL), 内核映射无需复制到用户页表; 而 x86_64 等共享页表的架构必须将内核部分拷贝 <sup>[3](#mod-mm-ref-3)</sup>. 这个 `#[cfg]` 门是硬编码的——默认分支覆盖了 riscv64 等同样使用独立页表的架构, 会导致内核映射被错误复制到用户空间. 新增架构支持时需在此处显式添加排除项.

**2.2 急切分配: 无按需调页, 无写时复制**

所有映射均采用 eager allocation: `map_alloc` 立即分配物理帧, `MAP_PRIVATE` 并未实现 CoW 语义——它仅调用 `map_alloc(populate=true)` 在映射时即填充文件内容 <sup>[4](#mod-mm-ref-4)</sup>. 后果: (a) 大文件 mmap 立即占满物理内存; (b) fork 后父子进程页面修改互相可见. 这是为简化实现做的明确取舍, 代价是 JIT 引擎等依赖 CoW 的应用无法正确运行.

**2.3 信号跳板的零拷贝设计**

`map_trampoline` 将信号返回代码通过 `map_linear` 直接映射到固定虚拟地址 `SIGNAL_TRAMPOLINE`, 物理地址来自 `axsignal::arch::signal_trampoline_address()` <sup>[5](#mod-mm-ref-5)</sup>. 这避免了将跳板代码复制到每个进程的栈, 实现了编译期固定地址 + 每进程零额外内存. 但这也硬编码了跳板地址, 所有用户进程必须共享同一虚拟地址.

**2.4 brk() 的"预留式堆"**

`sys_brk` 仅更新进程元数据中的 `heap_top` 指针, 不执行任何页表操作 <sup>[6](#mod-mm-ref-6)</sup>. 真正的堆空间在 `load_user_app` 阶段已通过 `map_alloc` 预分配了 `USER_HEAP_SIZE` 的物理内存 <sup>[7](#mod-mm-ref-7)</sup>. 这完全避免了运行时缺页, 但每个进程即使只用 4KB 堆也占满全部堆物理内存.

### 3. 跨模块协同

**3.1 设备内存映射快路径**

`sys_mmap` 在 fd 指向设备文件时走特殊路径: 通过 `get_device_by_fd` → `device.ops().get_device_mem()` 获取物理地址, 随后用 `map_linear` 直接建立用户态虚拟地址到设备物理内存的映射 <sup>[8](#mod-mm-ref-8)</sup>. 这绕过了文件系统层的缓冲 I/O, 实现零拷贝. 若请求长度超出设备内存, 超出部分用 `map_alloc` 填零 <sup>[9](#mod-mm-ref-9)</sup>——此行为是否符合 POSIX 需存疑.

**3.2 ELF 加载与 Shebang 递归**

`load_user_app` 实现了完整的 ELF 加载: 读文件 → 解析 ELF → 处理 INTERP 段 (动态链接器) → 映射段 → 构建栈与堆. 当文件不是合法 ELF 但以 `#!` 开头时, 提取解释器路径后递归调用自身 <sup>[10](#mod-mm-ref-10)</sup>. 这种递归没有深度限制——恶意构造的脚本链可触发栈溢出.

**3.3 用户态内存访问的安全围栏**

`access_user_memory` 与 `is_accessing_user_memory` 构成 percpu 门控: 内核在需要读取用户态内存 (如 syscall 参数指向的缓冲区) 前设置 `ACCESSING_USER_MEM` 标志, 访问后清除 <sup>[11](#mod-mm-ref-11)</sup>. 页错误处理程序据此区分"内核自身缺页"与"替用户态访问触发的缺页", 后者应修复用户页表而非 panic. 这是 arceos-starry 对原版 ArceOS 的显著扩展.

### 4. 边角细节与不足

- **[不足] 标志校验不完整**: `sys_mmap` 仅检查 `MAP_PRIVATE|MAP_SHARED` 互斥, 注释标注 `// TODO: more checks` <sup>[12](#mod-mm-ref-12)</sup>. 未校验的组合 (如 `MAP_FIXED` 与 `addr=0` 同时出现) 可能导致静默错误.
- **[风险] 用户输入触发 assert!**: `map_elf` 中 `assert_eq!(seg_pad, segment.offset % PAGE_SIZE_4K)` 在畸形 ELF 下直接 panic, 杀死内核而非返回 `EINVAL` <sup>[13](#mod-mm-ref-13)</sup>.
- **[半成品] TDOO 拼写错误**: `map_elf` 末尾 `// TDOO: flush the I-cache` <sup>[14](#mod-mm-ref-14)</sup>——I-cache 刷新在加载可执行段后是必要的, 尤其在 ARM 架构上缺失可能执行到旧指令.
- **[缺陷] PROT_WRITE+fd 的错误处理**: 当 `populate && writeable` 时仅调用 `error!()` 记录日志, 不返回错误码, 调用者收到 `Ok(addr)` 而数据未写入 <sup>[15](#mod-mm-ref-15)</sup>.
- **[语义错误] MAP_SHARED 零初始化**: 注释 `// TODO: 仅在MAP_ANONYMOUS时才zero` 指出对所有 `MAP_SHARED` 都做了零初始化, 文件映射的共享区域不应被清零 <sup>[16](#mod-mm-ref-16)</sup>.
- **[死锁风险] 持锁 I/O**: mmap/munmap/mprotect 持有 `addr_space` 互斥锁期间可能执行文件 I/O (`file.read_at`). 若文件系统层也尝试获取同一把锁则构成死锁.
- **[缺失] PROT_GROWSUP/PROT_GROWSDOWN**: `sys_mprotect` 直接拒绝这两个标志 <sup>[17](#mod-mm-ref-17)</sup>, 而 Linux 栈自增长机制依赖它们.
- **[未限制] Shebang 递归深度**: 无迭代上限, 恶意脚本可触发栈溢出 <sup>[10](#mod-mm-ref-10)</sup>.

## 🧵 五、进程与任务调度

> ⚠ **降级模式** · 主流程 JSON 解析失败 (`json_parse_failed_with_fallback`)

> 以下为兜底分析, refs / 数据结构 / 接口表均为空。

> ⚠ 此章为 JSON 解析失败后的降级兜底分析, 引证较少

**模块定位**  
进程与任务调度模块负责内核中进程/线程的生命周期管理、调度上下文切换、信号处理以及进程关系维护，是系统调用的核心支撑。其实现分布在多个文件中：`core/src/process.rs` 与 `core/src/task.rs` 定义进程/线程数据结构及调度相关函数，如 `ProcessData`、`ThreadData`、`TaskExt` 以及时间统计 `time_stat_from_kernel_to_user` 等；`process/src/process.rs` 专门处理进程组、会话、父子关系等 POSIX 语义；`api/src/imp/task/clone.rs` 提供 `clone` 系统调用的逻辑，组合各种命名空间共享标志；`api/src/imp/task/signal.rs` 则实现信号检查与递送，并通过 trap handler 钩子完成信号处理。

**核心抽象**  
模块采用 `Process` 与 `Thread` 的分层模型。`Process`（`process/src/process.rs`）维护进程 ID、线程表、进程组、会话、父子关系等，进程组与会话通过独立对象 `ProcessGroup` 和 `Session` 抽象，支持作业控制。`core/src/process.rs` 中的 `ProcessData` 和 `ThreadData` 作为内核内部的数据载体，通过 `Arc` 共享，分离了 POSIX 高层语义与调度核心。调度器通过 `task.rs` 中的 `current_thread`/`current_process` 提供即时上下文，`TaskExt` 包装线程执行实体并携带等待队列等资源。这一设计使得进程关系操作（如 `create_group`、`move_to_group`）可安全地跨异步边界，同时调度核心无需感知进程组等上层概念。

**代码中的具体不足与 TODO**  
第一，信号处理仍有多处未实现。在 `api/src/imp/task/signal.rs` 中，处理 `SignalOSAction::CoreDump`、`Stop`、`Continue` 的分支仅包含注释 `// TODO: implement core dump`、`// TODO: implement stop`、`// TODO: implement continue`，表明当前仅完成了终结型和忽略型动作，缺少核心转储、作业停止与继续的关键逻辑。  
第二，`process/src/process.rs` 中 `Process` 结构体末尾标记 `// TODO: sub reaper`，表示尚未支持 Linux 的“子收割者”特性，这可能导致在复杂的进程层级（如容器场景）下孤儿进程回收行为不符合 POSIX 预期。  
第三（隐含），`clone.rs` 中可见的 `CloneFlags` 虽定义了众多标志位，但部分标志（如 `CLONE_NEWNS`）其底层命名空间实现可能在 `AxNamespaceImpl` 中仅为骨架，实际隔离尚未完备，在系统调用路径上缺少对这些未实现能力的显式 `-EINVAL` 返回，可能导致静默失败。

**跨文件协同**  
整个模块通过共享 `Arc<ProcessData>` 和 `Arc<ThreadData>` 实现跨文件协同。`api/src/imp/task/clone.rs` 在创建新线程时调用 `starry_core::process::create_thread_data` 和 `starry_core::task::create_user_task`，并依据 `CloneFlags` 决定共享 `FD_TABLE`、`FS_CONTEXT` 等资源，将进程关系更新交由 `process/src/process.rs` 中的函数完成。信号处理依赖 `starry_core::task::current_thread_data` 获取当前线程的信号队列，`check_signals` 在其中提取未决信号并调用 `sys_exit_impl` 终止进程，同时通过 `#[register_trap_handler(POST_TRAP)]` 注册的 `post_trap_callback` 钩子在每次从用户态陷入内核后自动检查并递送信号。这种钩子机制解耦了信号检查与具体的系统调用路径，保持了内核退出逻辑的集中性。另外，`process.rs` 中进程组与会话的操作依赖于 `process_group` 和 `session` 模块，它们在进程 `fork` 及 `setsid` 等场景中被 `process.rs` 调用，形成清晰的层次依赖。然而，由于 `sub reaper` 等特性缺失，当前进程树维护和等待子进程的语义仍较脆弱，跨文件的孤儿回收逻辑可能需要进一步补全。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块基于 `undefined_vfs` 构建可组合的 VFS 层，通过 `FileLike` trait 统一抽象文件、目录、管道、epoll 等资源，并由 `FdTable` 管理进程级文件描述符。其独特之处在于同时保留了两个 procfs 实现：一个全硬编码的静态版本（`proc.rs`）用于早期原型，另一个通过 `starry_core::task` 动态获取进程信息的版本（`proc1/mod.rs`），呈现从原型向生产过渡的工程痕迹。与 ArceOS 基线相比，该仓引入了大量 Linux 兼容系统调用，并借助 `DynamicFs`/`DirMaker` 模式快速构建伪文件系统，但 procfs 实现仍以硬编码为主，真实信息采集不足。

### 1. 核心抽象：`FileLike` 统一接口与 `undefined_vfs` 框架
文件系统模块围绕 `FileLike` trait 构建统一接口，使得所有文件类型（如 `File`、`Directory`、`Pipe`、`EpollInstance`）都能被 `FdTable` 统一管理。底层 VFS 抽象由 `undefined_vfs` crate 提供，定义了 `Filesystem`、`Inode`、`DirEntry` 等概念。procfs 的实现展示了如何使用 `DynamicFs` 和 `DirMaker` 模式快速构建伪文件系统：`DynamicFs::new_with` 接受一个名称和一个根 inode 编号，以及一个 `builder` 闭包，该闭包通过 `DynamicDir::builder` 组装目录树，并使用 `SimpleFile::new` 添加只读文件节点。这种声明式的构建方式使得在不实现完整磁盘文件系统的情况下即可提供 Linux 兼容的 procfs。<sup>[1](#mod-fs-ref-1)</sup><sup>[2](#mod-fs-ref-2)</sup>

proc.rs 的 `builder` 函数展示了这一模式的典型用法：它创建根目录，添加 `sys`、`sysvipc` 等子目录，并在其中填充诸如 `cpuinfo`、`meminfo`、`mounts` 等文件，每个文件的内容均来自预定义的常量字符串。`SimpleFile::new` 接受一个闭包，在读取时返回字符串内容，这提供了灵活的惰性求值能力。<sup>[3](#mod-fs-ref-3)</sup>

### 2. 关键设计取舍：双 procfs 实现与硬编码数据
该仓库中存在两个功能重叠的 procfs 实现：`api/src/core/fs/imp/proc.rs` 和 `api/src/core/fs/imp/proc1/mod.rs`。二者的 `new_procfs` 函数签名完全相同，且 `builder` 结构高度相似。区别在于，`proc.rs` 完全依赖硬编码常量，连 `/proc/self` 都是一个固定目录（内容等同于 `/proc/1`），而 `proc1` 开始引入动态元素：`/proc/self` 被实现为符号链接，通过 `current_process().get_pid()` 动态获取目标；根目录上设置了 `ProcessInfoDir` 作为伪目录操作，允许按需枚举 `/proc/<pid>/` 子目录。<sup>[4](#mod-fs-ref-4)</sup><sup>[5](#mod-fs-ref-5)</sup>

这种双实现并存的现象暗示了开发过程中的过渡策略：早期为了快速验证 Linux 工具链对 `/proc` 的依赖，采用了全硬编码方式；随着对进程管理模块（`starry_core::task`）的完善，开始向动态 procfs 迁移。然而迁移并不彻底：`proc1` 中仍然使用了大量硬编码常量（如 `CPUINFO`、`DUMMY_MEMINFO`、`STATUS`、`STAT`），且两个文件中的常量定义几乎完全重复，形成了明显的技术债。<sup>[6](#mod-fs-ref-6)</sup>

### 3. 跨模块协同：文件系统与任务管理的集成
`proc1` 的实现体现了文件系统模块与任务管理模块的紧密协同。`ProcessInfoDir`（定义于 `process.rs`）实现了 `PseudoDirOps` trait，使得 `/proc` 根目录可以动态列出当前所有进程的 ID 目录。它的 `list_children` 方法通过调用 `starry_core` 的任务管理接口枚举进程，而 `get_child` 方法则按需创建对应 `<pid>` 子目录的内容。这种设计避免了预先创建所有进程目录的开销，符合伪文件系统的按需生成理念。<sup>[7](#mod-fs-ref-7)</sup>

此外，`task_stat.rs` 中的 `TaskStat` 结构实现了 `Display` trait，用于格式化 `/proc/<pid>/stat` 的内容，但其当前实现仍返回硬编码的伪造数据，尚未与真实的进程控制块打通。

在宏系统调用层面，文件描述符表 (`FdTable`) 提供了全局的 `fd_lookup` 等辅助函数，使得 `openat`、`read`、`write` 等系统调用能够快速将 fd 解析为 `Arc<dyn FileLike>`，而无需每个系统调用都持有锁或遍历表。这种设计平衡了性能与安全性。

### 4. 边角细节与不足
1. **硬编码 CPU 信息**：`CPUINFO` 常量包含 "AMD Ryzen 7 7840HS" 等特定硬件数据，在任何非 AMD 平台上都会暴露错误信息，严重破坏 Linux 兼容性的假象。<sup>[8](#mod-fs-ref-8)</sup>
2. **`/proc/self` 语义错误**：`proc.rs` 中将 `self` 实现为普通目录，而非指向当前进程 PID 的符号链接，违反 Linux 文件系统标准，可能导致依赖该行为的程序崩溃。<sup>[4](#mod-fs-ref-4)</sup>
3. **Magic number `0x9fa0`**：`DynamicFs::new_with` 的第二个参数 `0x9fa0` 含义不明，疑似根 inode 编号，但缺乏文档说明，且在两个实现中硬编码相同值，若与其它文件系统的 inode 编号冲突可能引发问题。<sup>[9](#mod-fs-ref-9)</sup>
4. **重复代码与技术债**：两个 procfs 实现的 `builder` 函数和常量定义大量重复，`proc.rs` 明显是废弃代码但未清理，增加了维护者困惑和 bug 引入风险。<sup>[10](#mod-fs-ref-10)</sup>
5. **动态信息缺失**：`proc1` 中的 `status`、`stat` 等关键文件仍然使用硬编码常量，`TaskStat::from_process(pid)` 虽然接收真实 pid 却返回假数据，说明与进程管理模块的对接尚未完成。
6. **锁安全性未验证**：`new_procfs` 返回 `Filesystem<RawMutex>`，但 `RawMutex` 的具体语义（是否可重入？在中断上下文中是否安全？）未在文件中显式约定，与 `current_process` 内部可能持有的其他锁的组合安全性缺乏论证。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块实现 Linux 信号系统调用与内核信号分发机制的胶水层, 核心抽象是将 `axsignal` crate 的类型 (`Signo`, `SignalInfo`, `SignalSet`, `SignalStack`, `SignalOSAction`) 桥接到 POSIX 语义. 它解决两个问题: (1) 将用户态信号操作翻译为 per-process/per-thread 的状态变更; (2) 通过 POST_TRAP 钩子在返回用户态时惰性递送信号. 与 ArceOS 基线相比, 本仓引入了进程组广播 (`sys_kill` 的 pid≤0 分支)、线程定向信号 (`tkill`/`tgkill`)、以及 `sigqueueinfo` 的权限校验雏形, 但 Stop/Continue/CoreDump 均未完整实现.

### 1. 核心抽象与外部依赖

信号模块的核心职责是将用户态 Linux 信号 API 映射到内核内部信号状态机. 它依赖 `axsignal` crate 提供基础类型: `Signo` (信号编号枚举), `SignalInfo` (信号实例, 含编号与 si_code), `SignalSet` (位图掩码), `SignalStack` (备选栈描述), `SignalOSAction` (信号处置: Terminate/CoreDump/Stop/Continue/Handler). 这些类型通过 `linux_raw_sys::general` 的 `kernel_sigaction` 和 `siginfo` 与 C 结构体互转 <sup>[1](#mod-signal-ref-1)</sup>.

进程/线程维度的状态存储委托给 `starry_core::task` 的 `current_thread_data().signal` 和 `current_process_data().signal`, 前者持有 per-thread 的阻塞集与 pending 集, 后者持有 per-process 的 action 表 (由 `Mutex` 保护) <sup>[2](#mod-signal-ref-2)</sup>. 这种分离精确对应 POSIX 语义: 信号处理器 (`sigaction`) 是进程级共享资源, 而阻塞掩码 (`sigprocmask`) 是线程私有状态.

### 2. 关键设计取舍

**惰性递送与陷阱钩子**. 模块不实现抢占式信号递送, 而是通过在每次用户态陷阱返回时调用 `post_trap_callback` 触发 `check_signals` <sup>[3](#mod-signal-ref-3)</sup>. 这避免了内核态处理信号的复杂上下文切换, 代价是信号延迟不可控 — 若目标线程长时间陷于内核 (如阻塞 I/O), 信号将一直不递送. `sys_rt_sigsuspend` 的等待循环进一步放大了这一问题: 循环体内先调用 `check_signals` 再调用 `current_process_data().signal.wait_signal()` <sup>[4](#mod-signal-ref-4)</sup>, 但若信号在 `check_signals` 返回 false 后、`wait_signal` 进入前到达, 该信号可能被错过或延迟到下一次循环.

**信号处置的有限实现**. `check_signals` 对五种 `SignalOSAction` 的分发明显处于早期阶段: Terminate 正确调用 `sys_exit_impl`; CoreDump 标记为 TODO 但实际行为与 Terminate 相同, 仅加了硬编码的 `0x80` 偏移 <sup>[5](#mod-signal-ref-5)</sup>; Stop 同样是 TODO 却退化为终止 <sup>[6](#mod-signal-ref-6)</sup>; Continue 完全为空操作 <sup>[7](#mod-signal-ref-7)</sup>; Handler 分支也是空的 — 实际的用户态处理器跳转逻辑藏在 `signal.check_signals(tf, ...)` 内部对陷阱帧的篡改中, 本文件不感知.

**sigaltstack 的保守校验**. `sys_sigaltstack` 在设置备选栈前执行了两层验证: 先用 `MINSIGSTKSZ` 检查栈大小下限, 再通过 `stack_ptr.get_as_array(ss.size)` 验证用户提供的地址区域可访问 <sup>[8](#mod-signal-ref-8)</sup>. 后者实际上触发了页表遍历和缺页处理, 能在设置阶段提前暴露非法地址, 避免信号递送时因栈不可用而二次崩溃. 代价是每次 `sigaltstack` 调用都额外产生一次用户内存探测.

**信号定向的三层粒度**. `sys_kill` 根据 pid 参数实现进程/进程组/广播三种语义, `sys_tkill` 直接向线程发送, `sys_tgkill` 额外校验线程所属的线程组 <sup>[9](#mod-signal-ref-9)</sup>. `check_thread` 函数通过 `undefined_process::thread::get_thread` 获取线程对象再比对 tgid, 这一校验路径依赖外部 crate 的线程查找表, 若该表存在 TOCTOU 问题则校验可能被绕过.

### 3. 跨模块协同

信号生命周期跨越多个子系统:

1. **发送路径**: `sys_kill` / `sys_tkill` / `sys_rt_sigqueueinfo` 等系统调用构造 `SignalInfo`, 调用 `send_signal_process` / `send_signal_thread` 将信号注入目标进程或线程的 pending 集. 这些函数通过 `starry_core::process` 的 `get_process_data` / `get_thread_data` 查找目标, 失败时统一返回 `EPERM` 而非 `ESRCH` <sup>[10](#mod-signal-ref-10)</sup>.

2. **递送路径**: `axhal::trap` 的 `register_trap_handler(POST_TRAP)` 宏将 `post_trap_callback` 注册为每次从用户态陷阱返回时的回调. 该函数调用 `check_signals`, 后者再调用 `current_thread_data().signal.check_signals(tf, restore_blocked)` 完成实际的信号选择与处理器跳转.

3. **清理路径**: 当处置为 Terminate/CoreDump/Stop 时, `check_signals` 调用 `sys_exit_impl(0, exit_code, true)` 终止进程. 第三个参数 `true` 指示这是信号导致的退出, 可能影响父进程的 `WIFSIGNALED` 判断.

4. **等待路径**: `sys_rt_sigtimedwait` 和 `sys_rt_sigsuspend` 依赖 `current_thread_data().signal.wait_timeout()` 和 `current_process_data().signal.wait_signal()` 提供的阻塞原语, 这些原语内部应基于条件变量或等待队列实现, 但本文件不暴露其实现细节.

### 4. 边角细节与不足

**信号编号截断** <sup>[11](#mod-signal-ref-11)</sup>: `parse_signo` 将 `u32` 的 signo 截断为 `u8` 再通过 `Signo::from_repr` 查找. 虽然 POSIX 标准信号编号在 1-31 范围内, Linux 实时信号可达 64, 此实现上限 255 目前足够, 但截断操作丢失了未来扩展的可能性.

**权限检查缺失** <sup>[12](#mod-signal-ref-12)</sup>: `sys_kill`, `sys_tkill`, `sys_tgkill` 三处均有 `// TODO: should also check permissions` 注释. Linux 要求发送者必须具有与目标相同的 ruid/euid, 或有 `CAP_KILL` 能力. 当前实现允许任意进程向任意目标发送信号, 构成权限绕过.

**广播信号的 init 豁免缺失** <sup>[13](#mod-signal-ref-13)</sup>: `sys_kill` 的 `-1` 分支遍历所有进程并广播信号, 注释 `// TODO: skip init process?` 表明开发者意识到应跳过 init (PID 1), 否则某些信号可能导致系统崩溃.

**`send_signal_process_group` 的错误吞没** <sup>[14](#mod-signal-ref-14)</sup>: 该函数通过 `.is_ok() as usize` 将每次发送结果二值化为计数, 丢失了失败原因. 若部分进程已退出或权限不足, 调用者 (如 `sys_kill`) 仅知道成功数 > 0, 无法区分"全部成功"与"多数失败但至少一个成功".

**`unsafe` 块缺乏 SAFETY 文档** <sup>[15](#mod-signal-ref-15)</sup>: 至少 8 处用户指针解引用 (`*oldset = *blocked`, `*act`, `*set.get()?`, `sig.get()?.read()`, `*info = sig.0` 等) 均未附 `// SAFETY:` 注释. 这些操作的契约在于调用方已验证指针非空且可访问, 但缺少显式文档增加了审查难度.

**`SignalOSAction::Stop` 退化为终止** <sup>[6](#mod-signal-ref-6)</sup>: 注释 `// TODO: implement stop` 后直接调用 `sys_exit_impl`, 这意味着 `SIGSTOP`/`SIGTSTP` 等信号会杀死进程而非挂起, 严重违反 POSIX 预期.

**`SignalOSAction::Continue` 为空操作** <sup>[7](#mod-signal-ref-7)</sup>: 被停止的进程无法通过 `SIGCONT` 恢复, 因为 Continue 分支完全未实现 — 既无 TODO 注释后的 fallback, 也无任何状态变更.

**`check_signals` 的布尔返回值语义模糊** <sup>[16](#mod-signal-ref-16)</sup>: 返回 `true` 表示有信号被递送, 但调用方 (`post_trap_callback` 与 `sys_rt_sigsuspend`) 使用方式不同: 前者忽略返回值, 后者用它决定是否继续循环. 统一性欠缺.

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现 System V 共享内存子系统的内核管理部分，核心抽象为 `SharedMemory` 结构体（代表一块连续物理页）及全局 `SharedMemoryManager`（以 `BTreeMap<u32, Arc<SharedMemory>>` 维护 key 到共享内存的映射）。其余 IPC 机制（futex、消息队列、信号量）在 syscall 层有入口但实现代码未暴露。与 ArceOS 基线相比，本仓库新增了共享内存管理器，采用全局 Mutex + Arc 的同步模型，并在内存分配上直接依赖 `axalloc` 的全局分配器，但缺少用户态映射安全的完整保护。

### 1. 核心抽象与外部依赖

共享内存子系统围绕两个核心类型展开。`SharedMemory` 结构体<sup>[1](#mod-ipc-ref-1)</sup>持有一个内核虚拟地址 `addr`、页数 `page_count` 和 System V 风格的 `key`。它不直接描述用户态映射，而是一个内核态句柄：物理页由全局分配器分配，虚拟地址仅用于内核释放或内部访问。`SharedMemoryManager` 是全局单例<sup>[7](#mod-ipc-ref-7)</sup>，内部使用 `Mutex<BTreeMap<u32, Arc<SharedMemory>>>` 管理所有段，`next_key` 字段为 `AtomicU32` 用于帮助生成新 key。

外部依赖方面：内存分配通过 `axalloc::global_allocator()` 分配连续物理页，这对共享内存的物理连续性假设极强<sup>[4](#mod-ipc-ref-4)</sup>；同步使用 `axsync::Mutex`（可能是自旋锁或睡眠锁，取决于内核配置）；错误处理统一转换为 `axerrno::LinuxError`。这些依赖将模块与 ArceOS 的体系紧密绑定，不利于移植到其他内核。

### 2. 关键设计取舍

**全局管理器 vs 进程本地**：选择全局静态 `SHARED_MEMORY_MANAGER` 简化了跨进程查找，但代价是全局锁竞争。所有 `shmget`/`shmat`/`shmdt` 操作都要经过同一把 `Mutex`<sup>[2](#mod-ipc-ref-2)</sup>。替代方案（如 per-namespace 管理器或 lock-free map）在此处被放弃，推测是为了快速实现原型。

**BTreeMap 而非 HashMap**：使用 `BTreeMap` 而非 `HashMap` 避免了哈希函数选择问题，且能够按 key 排序，但插入/查找退化为 O(log n)。在共享内存段数量通常不大的场景下可接受，但代码中 `keys().max()`<sup>[3](#mod-ipc-ref-3)</sup> 暗示了对有序性的依赖——虽然该调用方式实际利用了迭代器，并未真正受益于 BTree。

**Key 生成策略的隐患**：`next_available_key` 的实现<sup>[3](#mod-ipc-ref-3)</sup>意图结合原子递增和当前最大 key 来生成新 key，但逻辑存在缺陷。`fetch_add` 与后续 `lock()` 之间并非原子，且 `keys().max().unwrap_or(0)` 若当前 map 为空则导致 key 从 1 开始，但后续若插入其他 key，生成的值会跳跃。这不会导致重复 key（因为 insert 时若重复会覆盖），但造成 key 空间浪费且行为难以预测。

**物理页直接分配**：`create` 方法<sup>[4](#mod-ipc-ref-4)</sup>直接调用 `alloc_pages` 分配连续物理内存，未通过页表映射回用户空间——这部分职责落在 syscall 层的 `shmat` 实现中。这种分离是合理的，但 `create` 并未记录分配的物理地址，而是保存内核虚拟地址 `addr`，这要求分配器返回的内核虚拟地址在释放时仍有效，且不能移动。

### 3. 跨模块协同（推断）

虽然 syscall 代码不可见，从系统调用列表可推断 `shmat`（行 355）会调用 `SHARED_MEMORY_MANAGER.get(key)` 获取 `Arc<SharedMemory>`，然后将 `addr` 对应的物理页映射到调用进程的用户地址空间。`shmdt`（行 357）则解除映射但不应立即释放物理页——释放由 `drop` 触发，当所有 `Arc` 引用消失时才会回收<sup>[5](#mod-ipc-ref-5)</sup>。然而，这里存在一个严重缺口：用户态映射计数与内核 `Arc` 引用计数割裂。如果进程通过 `shmdt` 解除映射但内核 `Arc` 仍被另一进程持有，物理页不会释放；反之若内核 `Arc` 计数因某种原因降至零（比如 `delete` 移除并丢弃 `Arc`），`drop` 会立即释放页面，而仍映射该区域的进程将面临 UAF。这种安全契约目前没有代码层面的保障。

`futex` 机制位于 syscall 第 208 行，可能调用独立的 futex 模块（不在可见范围内），但其实现通常依赖 per-process 等待队列和原子操作。`msgget`/`msgsnd`/`msgrcv` 和 `semget`/`semctl` 等同样分散在 syscall 后端，可能各自维护全局管理器。

### 4. 边角细节与不足

- **<sup>[6](#mod-ipc-ref-6)</sup> TODO 标记**：`create` 函数第 52 行注释 `// TODO: more error checking`，表明参数合法性（如 size 为 0）未验证，可能触发分配器内部断言。
- **`Drop` 的安全假设**<sup>[5](#mod-ipc-ref-5)</sup>：`drop` 直接 `dealloc_pages`，假定此时没有任何进程仍映射该内存。若 `Arc` 计数不反映实际映射数，将构成 UAF。正确的设计应引入引用计数包装物理页，或由 syscall 层手动管理映射计数。
- **`next_available_key` 的不直观行为**<sup>[3](#mod-ipc-ref-3)</sup>：在高并发下，`fetch_add` 与 `lock` 之间可能插入其他 `create`，导致生成的 key 并非严格递增，也可能与用户通过 `shmget` 指定的 key 冲突（虽然 System V 允许用户指定 key，若重复则 `shmget` 只返回现有段，但此处未区分）。
- **全局锁粒度**<sup>[7](#mod-ipc-ref-7)</sup>：`BTreeMap` 受 `Mutex` 保护，所有操作串行化。当共享内存段数量较多或频繁 attach/detach 时，锁竞争可能成为明显瓶颈。
- **硬编码 PAGE_SIZE_4K**<sup>[4](#mod-ipc-ref-4)</sup>：分配页数计算基于 `PAGE_SIZE_4K`，未适配 64K 或 2M 大页架构，缺乏平台可移植性。
- **缺乏命名空间隔离**：全局 `SHARED_MEMORY_MANAGER` 是单一实例，所有进程共享同一个 key 空间，未实现 Linux 的 IPC 命名空间特性，安全隔离性弱。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块以 `Socket` 枚举为核心抽象, 将 `axnet` crate 的 `TcpSocket`/`UdpSocket` 封装在 `Mutex` 后, 并通过实现 `FileLike` trait 集成到 fd 表中. 它解决了在无标准库的 OS 内核中提供 Linux 兼容 socket 系统调用 (socket/bind/connect/sendto/recvfrom/listen/accept 等) 的问题. 与 arceos-starry 家族基线相比, 本仓库额外实现了 `socketaddr.rs` 子模块 (从 rustix 移植的 `SockAddr` 安全抽象), 但该模块与 syscall 层的地址转换函数 (`into_sockaddr`/`from_sockaddr`) 存在重复实现且未被统一使用; 此外 `sendto` 对 UDP 做了强制 bind 到 127.0.0.1:0 的非标准行为修改.

### 1. 核心抽象与外部依赖

网络模块的核心是 `Socket` 枚举 <sup>[1](#mod-net-ref-1)</sup>, 它只有两个变体: `Udp(Mutex<UdpSocket>)` 与 `Tcp(Mutex<TcpSocket>)`. 每个 socket 实例被 `Mutex` 包裹, 这意味着所有 I/O 操作 (send/recv/bind/connect/shutdown) 都需要先获取锁 — 这是一种粗粒度锁策略, 简单但可能在多线程高并发场景下造成争用.

`Socket` 实现了 `FileLike` trait <sup>[2](#mod-net-ref-2)</sup>, 这使得 socket fd 可以复用文件描述符表的基础设施 (`fd_add`/`fd_lookup`). `read` 映射到 `recv`, `write` 映射到 `send`, `poll` 直接委托给底层 socket 的 `poll`. 这种统一抽象让 `select`/`poll`/`epoll` 等 I/O 多路复用机制可以一致地处理文件和 socket.

地址抽象存在两条路径: (1) `socket.rs` 内部的 `into_sockaddr`/`from_sockaddr` 函数, 直接操作原始 `sockaddr` 指针, 仅支持 IPv4 <sup>[7](#mod-net-ref-7)</sup>; (2) `socketaddr.rs` 子模块提供了从 rustix 移植的 `SockAddr` 类型 <sup>[8](#mod-net-ref-8)</sup>, 它用 `MaybeUninit<sockaddr>` 存储任意协议族地址, 支持 IPv4/IPv6 双向转换, 且为 `Send`/`Sync` 写了 SAFETY 注释 <sup>[9](#mod-net-ref-9)</sup>. 然而 syscall 层并未使用 `SockAddr`, 两条路径并存暴露了模块尚处于演进中.

### 2. 关键设计取舍

**粗粒度 Mutex vs 细粒度锁**. 选择每 socket 一个 `Mutex` 而非更细粒度的读写锁, 简化了实现但牺牲了并发读的能力. 在 `sendto` 实现中, 持锁期间还执行了额外的 `bind` 调用 <sup>[3](#mod-net-ref-3)</sup>, 延长了临界区.

**UDP sendto 的隐含 bind**. 代码注释标记了 `// diff: must bind before sendto` <sup>[3](#mod-net-ref-3)</sup>: 在对 UDP socket 调用 `sendto` 时, 模块会先将 socket 绑定到 `127.0.0.1:0` (任意端口), 然后才发送数据. 这与 POSIX 语义不符 — 标准允许未 bind 的 UDP socket 直接 sendto, 内核自动分配源地址. 这个 workaround 可能是为了绕过底层 `axnet` 的限制, 但副作用是: 如果用户已经显式 bind 了其他地址, 第二次 bind 会返回 `EISCONN` (代码将其吞掉并继续 send).

**setsockopt 全线空实现**. `setsockopt` 方法仅按 level 分类输出 `warn!("unimplemented setsockopt for ...")` 后返回 `Ok(())` <sup>[4](#mod-net-ref-4)</sup>. 这意味着所有 socket 选项设置都被静默忽略 — 包括 `SO_REUSEADDR`, `SO_KEEPALIVE`, `TCP_NODELAY` 等关键选项. 上层应用可能因此出现难以调试的行为异常.

**非阻塞与 cloexec 的遗留**. 文件顶部定义了 `SOCK_NONBLOCK` 和 `SOCK_CLOEXEC` 常量 <sup>[5](#mod-net-ref-5)</sup>, `sys_socket` 中也提取了这些标志位, 但相应的 `set_nonblocking` 调用被整段注释掉 <sup>[2](#mod-net-ref-2)</sup>, `set_close_on_exec` 标记为 TODO. 这意味着 `socket(AF_INET, SOCK_STREAM | SOCK_NONBLOCK, 0)` 会成功创建 socket 但忽略非阻塞标志, 返回的 fd 实际上是阻塞的.

**listen backlog 被忽略**. `sys_listen` 接收 `backlog` 参数但标注 `// currently not used` <sup>[10](#mod-net-ref-10)</sup>, 直接委托给底层 `tcpsocket.lock().listen()`. TCP 握手队列长度完全由 `axnet` 的默认值决定, 用户无法控制.

### 3. 跨模块协同

**与 fd 表的集成**. `Socket::add_to_fd_table` 将自身包装为 `Arc<Self>` 后通过 `fd_add` 注册到全局 fd 表 <sup>[1](#mod-net-ref-1)</sup>, `from_fd` 则通过 `fd_lookup` + `downcast` 取回. 这里 `downcast` 失败返回 `EINVAL` 而非 `EBADF` 或 `ENOTSOCK`, 对错误的分类不够精确.

**与 axnet 的委托关系**. 所有实质网络操作最终落到 `axnet` crate 的 `TcpSocket`/`UdpSocket` 方法上. 错误传播路径是 `axnet` 的错误 → `?` → `LinuxError`. 但代码中有多处 `.map_err(|_| LinuxError::EISCONN)` 将底层具体错误吞掉 <sup>[6](#mod-net-ref-6)</sup>, 统一映射为 `EISCONN`, 丢失了原始错误信息.

**地址转换的跨边界协同**. `sys_recvfrom` 和 `sys_accept` 需要向用户空间写回 `sockaddr`. 它们先通过 `Socket::local_addr()`/`peer_addr()` 获取 Rust 的 `SocketAddr`, 然后调用 `into_sockaddr` 转换为 C 结构体, 最后通过 `unsafe` 指针写入用户空间 <sup>[11](#mod-net-ref-11)</sup>. `sys_getaddrinfo` 则更复杂: 它通过 `axnet::dns_query` 解析域名, 手动构建 `addrinfo` 链表, 用 `mem::forget` 转移所有权, 最终由 `sys_freeaddrinfo` 通过 `Vec::from_raw_parts` 回收 <sup>[12](#mod-net-ref-12)</sup>. 这个所有权传递链条依赖调用者正确调用 `freeaddrinfo`, 否则会内存泄漏.

### 4. 边角细节与不足

**<sup>[3](#mod-net-ref-3)</sup> sendto 的隐含 bind — 语义破坏**. `Socket::sendto` 对 UDP 无条件执行 `bind(SocketAddr::new(IpAddr::V4(Ipv4Addr::new(127,0,0,1)), 0))`, 注释声明这是 "diff". 如果 socket 已 connect 到远端 (即已有 peer_addr), 这个 bind 会失败, 代码将错误映射为 `EISCONN`. 而如果 bind 成功, 后续 `send_to` 失败也被统一映射为 `EISCONN` — 两种截然不同的失败原因被合并.

**<sup>[4](#mod-net-ref-4)</sup> setsockopt 全线空实现**. 不仅 TCP/UDP 的所有 level 都返回 `Ok(())`, 而且 `SocketOptionLevel::try_from` 对未知 level 直接 `unimplemented!()` — 这会触发 panic, 而用户态可控的 `level` 参数可以轻易构造出未覆盖的枚举值 <sup>[13](#mod-net-ref-13)</sup>. 这是一个 DoS 入口.

**<sup>[12](#mod-net-ref-12)</sup> sys_freeaddrinfo 的 assert 炸弹**. `sys_freeaddrinfo` 内部有 `assert_eq!(unsafe { *aibuf_ptr }.slot, 0)` 和 `assert!(len > 0)`. 如果用户传递了被篡改的 `res` 指针或多次释放同一指针, 这些 assert 会在内核态 panic, 直接导致系统崩溃.

**<sup>[7](#mod-net-ref-7)</sup> from_sockaddr 仅支持 IPv4**. `from_sockaddr` 将 `addrlen` 与 `size_of::<sockaddr>()` 严格比较, 且只检查 `AF_INET`. IPv6 地址 (使用更大的 `sockaddr_in6`) 会被 `EINVAL` 拒绝. 而 `socketaddr.rs` 中的 `SockAddr` 完全支持 IPv6, 但 syscall 层未使用它 — 这意味着即使底层 `axnet` 支持 IPv6, 用户态也无法通过 syscall 使用.

**<sup>[14](#mod-net-ref-14)</sup> shutdown 要求 peer_addr 存在**. `Socket::shutdown` 在调用底层 `shutdown()` 前先执行 `peer_addr()?`, 如果对端地址不存在则返回错误. 这对监听 socket 或未 connect 的 UDP socket 而言不合理 — POSIX 允许 shutdown 一个未连接的 socket (应返回 `ENOTCONN`, 而非本实现的 `EINVAL` 或底层错误).

**<sup>[15](#mod-net-ref-15)</sup> get_flags/set_flags 缺失**. `FileLike::get_flags` 返回 `FileFlags::empty()`, `set_flags` 仅打印 warn. 这意味着 `fcntl` 对 socket fd 的操作 (如设置 `O_NONBLOCK`) 无效, 且不会返回错误, 用户态无法感知.

**<sup>[16](#mod-net-ref-16)</sup> unsafe 块缺乏 SAFETY 注释**. `socket.rs` 中大量 `unsafe` 块 (如 `from_sockaddr` 的裸指针解引用、`sys_recvfrom` 的用户空间写入、`sys_getaddrinfo` 的指针链表构建) 均未提供 SAFETY 契约说明. 相比之下 `socketaddr.rs` 的两处 `unsafe impl Send/Sync` 均有简洁注释 <sup>[9](#mod-net-ref-9)</sup>, 风格更规范.

- sendto 对 UDP 强制 bind 127.0.0.1:0 破坏 POSIX 语义, 且 bind 失败与 send_to 失败均映射为 EISCONN [ref:3]

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层以 `src/syscall.rs` 中的 `handle_syscall` 为核心, 通过 `#[register_trap_handler(SYSCALL)]` 注册为异常处理入口, 利用 `syscalls::Sysno` 枚举进行分派. 实际实现分布在 `api/src/imp/` 各子模块 (文件系统、内存、网络、任务、信号等), 并借助 `UserPtr` 系列封装实现安全的用户态内存访问. 与 arceos 基线相比, 本模块引入了 starry 的进程模型 (`starry_core`)、信号机制 (`axsignal`) 以及更完整的 Linux ABI 适配 (含架构特定调用规约), 但大量系统调用仍为返回成功的桩实现, 存在数据完整性风险与功能缺失.

### 1. 核心抽象：异常陷阱与分派机制

系统调用层的入口 `handle_syscall` 通过 `axhal::trap::register_trap_handler` 宏绑定 `SYSCALL` 异常, 接收 `&mut TrapFrame` 与系统调用号 <sup>[1](#mod-syscall-ref-1)</sup>. 分派逻辑构建在 `syscalls::Sysno` 枚举之上, 将调用号转换为变体后, 通过穷尽的 `match` 表达式映射到 `undefined_os_api` 提供的 `sys_*` 函数 <sup>[1](#mod-syscall-ref-1)</sup>. 参数提取自 `TrapFrame` 的 `arg0()`~`arg5()` 方法, 再经由 `UserPtr`、`UserInPtr`、`UserOutPtr` 等封装转换, 确保用户态指针在访问前进行合法性检查 <sup>[3](#mod-syscall-ref-3)</sup>. 这一设计在保持类似 Linux `sys_call_table` 的集中分派风格的同时, 利用 Rust 枚举获得编译期穷尽性保证. 对于无效调用号, 代码并未提前返回错误, 而是回退到 `Sysno::memfd_secret`——注释中甚至保留了一行被注释掉的 `return -EINVAL`, 表明开发者曾有过更严谨的考虑, 但最终选择了宽松策略 <sup>[2](#mod-syscall-ref-2)</sup>.

### 2. 关键取舍：架构适配与桩策略

模块大量使用 `#[cfg(target_arch)]` 处理不同架构的 ABI 差异. 典型如 `clone` 系统调用: 在 x86_64/loongarch64 上参数顺序为 `(flags, stack, parent_tid, child_tid, tls)`, 而 riscv64/aarch64 上为 `(flags, stack, parent_tid, tls, child_tid)` <sup>[4](#mod-syscall-ref-4)</sup>. 代码通过两套 `#[cfg]` 分支分别匹配, 未覆盖的架构会落入 `_ => stub_unimplemented` 返回 `ENOSYS`, 这种保守策略虽安全, 却也意味着每新增架构都需手动添加参数适配, 移植成本较高. 对于尚未实现的系统调用, 模块提供了三种桩函数: `stub_unimplemented` 返回 `ENOSYS`; `stub_bypass` 返回 `Ok(0)` 并记录警告; 以及 `sys_openstub` 返回 `ENOSYS` 用于较新的 Linux 调用 <sup>[5](#mod-syscall-ref-5)</sup><sup>[6](#mod-syscall-ref-6)</sup>. `stub_bypass` 的使用尤其值得警惕——`sync`、`fsync`、`flock`、`fallocate` 等安全关键调用均静默返回成功, 用户程序可能误以为数据已持久化或文件锁已获取, 构成数据完整性风险 <sup>[6](#mod-syscall-ref-6)</sup>. 与之对照, 部分调用如 `getsockopt`、`msgrcv` 等返回明确的错误码 (`EFAULT`、`ENOMSG`), 表明团队对不同调用的危害性有区别认知 <sup>[5](#mod-syscall-ref-5)</sup>.

### 3. 跨模块协同：信号、进程与地址空间

系统调用层通过 `TrapFrame` 与其他子系统深度耦合. 分派前调用 `set_trap_frame(tf)` 将陷阱帧指针存入线程局部存储, 使得信号处理、内存管理等模块能够在任意位置访问和修改当前上下文 <sup>[3](#mod-syscall-ref-3)</sup>. `execve` 系统调用直接接收 `&mut TrapFrame`, 用于替换进程映像并重置用户态栈和入口 <sup>[7](#mod-syscall-ref-7)</sup>. 信号子系统通过 `register_trap_handler(POST_TRAP)` 注册 `post_trap_callback`, 在每次从用户态陷入后调用 `check_signals`; 该函数可根据信号动作直接调用 `sys_exit_impl` 终止进程, 或设置返回地址以交付信号处理函数 <sup>[8](#mod-syscall-ref-8)</sup>. 这种"后置钩子"设计使得信号能够抢占任何系统调用的返回路径, 符合 POSIX 语义. 内存管理方面, `sys_mmap` 通过 `starry_core::task::current_process_data()` 获取当前进程地址空间锁, 支持匿名映射、文件映射、大页以及设备内存的直接映射 <sup>[9](#mod-syscall-ref-9)</sup>, 但文件映射的写保护 (PROT_WRITE) 尚未实现, 仅打印错误日志而不阻止操作 <sup>[9](#mod-syscall-ref-9)</sup>.

### 4. 边角细节与不足

- **无效系统调用号静默处理** <sup>[2](#mod-syscall-ref-2)</sup>: `Sysno::new` 返回 `None` 后未提前返回, 而是回退到 `memfd_secret`, 隐藏了用户程序的错误调用.
- **`stub_bypass` 静默成功** <sup>[6](#mod-syscall-ref-6)</sup>: `sync`/`fsync` 等关键调用返回 0, 可能导致文件系统不一致或数据丢失.
- **架构分支不完备** <sup>[4](#mod-syscall-ref-4)</sup>: `clone` 的 cfg 分支仅覆盖四种架构, 移植到 mips/powerpc 时需手写新的参数适配.
- **`setsockopt` 的 `unimplemented!()` 恐慌** <sup>[10](#mod-syscall-ref-10)</sup>: 对不支持的 `level` 直接 `unimplemented!()`, 运行时 panic 构成 DoS 风险, 应返回 `EINVAL`.
- **文件映射写保护缺失** <sup>[9](#mod-syscall-ref-9)</sup>: `sys_mmap` 对文件映射请求 PROT_WRITE 时仅打日志, 未返回错误, 可能允许非法写入.
- **信号处理 TODO** <sup>[8](#mod-syscall-ref-8)</sup>: `CoreDump`、`Stop`、`Continue` 动作均标注 TODO, 目前仅简单退出或忽略, 不符合完整 POSIX 信号语义.
- **日志噪声** <sup>[3](#mod-syscall-ref-3)</sup>: 每次系统调用打印 `info!` 日志, 生产环境中可能造成性能损耗和日志洪水.
