## 📖 一、总览

该内核是一个面向自动化评测的自研实验内核，定位介于教学原型（如xv6）与完整Linux兼容层之间。它以RISC-V/LoongArch为主要平台，从C入口kmain直接初始化硬件、构建内存系统、挂载只读EXT4文件系统并执行预置测试脚本，而非启动交互式Shell。设计核心围绕“让未修改的Linux二进制可运行”：系统调用层对154项调用大量采用桩实现（返回0或-ENOSYS），但关键调用如fork/execve/mmap/futex等提供实际功能，形成了一个“能跑就行”的兼容表面。该内核将开发精力投入到进程模型的深度（clone线程、COW）和文件系统广度（EXT4只读、ramfs、管道）上，而在并发安全、错误处理、信号/网络等方面做出显著省略。最值得评审注意的三点：一是启动期即运行测试脚本的闭环设计，将内核验证自动化；二是采用巨型switch系统调用分发器快速覆盖Linux ABI；三是全局数据结构普遍无锁，构成本内核的主要稳定性风险。

## 🎯 二、综合评价

### 整体定位
该内核并非单纯的课堂练习，也远未达到可部署水平，其真实定位是“实验性用户程序执行环境+自动化测试平台”。静态进程表（[task.ref:1]）、固定大小的内存池、简陋的调度器透露出它从未以多用户、长时间运行为目标。相反，syscall层通过巨型switch和大量桩（[syscall.ref:1]）实现了对Linux ABI的广度覆盖，目的就是让标准交叉编译的二进制能在其上运行，从而验证进程、文件、内存等子系统的核心逻辑。启动流程（[boot.ref:3]）中直接扫描ext4测试脚本并按阶段执行，更表明其首要客户是自动化测试套件，而非人类用户。

### 真正的创新点
与xv6等经典教学内核相比，本做品有两个亮点：一是对现代Linux进程模型的逼近——通过引用计数实现的COW（[mm.ref:2][mm.ref:3]）和clone系统调用的CLONE_VM/CLONE_THREAD支持，使其能运行多线程程序，这在自研教学内核中颇为少见；二是将测试框架直接融入启动流程（[boot.ref:3]），形成“构建-启动-评测”的闭环，可快速迭代验证修改，这种设计思路在同类开源内核中较为独特。此外，只读EXT4驱动支持extent布局，也是一项实用的工程扩展。

### 取舍判断
作者清醒地选择了广度优先策略：在并发安全上几乎完全放弃——物理页分配器（[mm.ref:1]）和管道（[ipc.ref:4]）均无锁保护，sleep/wakeup调用不持锁导致丢失唤醒风险；在内存管理上接受硬编码限制（MAX_REF_PAGES仅1GiB，refcnt为u16易溢出）；在设备驱动上采用轮询静态变量，放弃中断和多设备支持。这些牺牲换来的是代码量可控（约8000行）、开发注意力集中，短期内实现了丰富的功能表面。系统的核心矛盾在于：它披着一层Linux兼容的薄纱，但底层的可靠性假设却停留在单核、单用户、短时运行的实验室环境。

### 完成度与不足
从完整度看，核心控制流（进程创建、切换、文件读写）基本完备，已能支撑简单用户程序；但信号、网络、完整IPC、持久化存储等关键领域缺失，大量syscall桩使得依赖高级特性的程序必然失败。从质量看，全局无锁、参数校验缺失、错误传播断裂（如sys_brk内存泄漏、block driver缺边界检查）等问题普遍存在，表明代码尚处于“能跑通测试”的原型阶段，距离生产级或可信教学参考尚有显著距离。若后续改进，最紧迫的是引入锁机制保护关键数据结构，并补全信号/等待/非阻塞I/O等调度基础设施，否则稍复杂的并发负载即可触发崩溃。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 内核 C 入口 `kmain(hartid, dtb)` 承接汇编引导阶段，串联物理内存发现 → 分页/堆初始化 → 进程/陷阱/定时器 → 块设备/ext4 挂载 → 测试脚本扫描与调度的完整启动链。核心数据结构为 `script_list`（定长 64 条目）与 `scan_ctx`（递归深度上限 4）。相比 xv6，本内核在启动后期嵌入了一个面向自动化评测的 ext4 遍历测试框架，按预定义阶段排序并逐脚本执行。

### 1. C 入口与架构分派

`kmain(u64 hartid, u64 dtb)` 是整个内核的 C 语言入口点，由架构相关汇编引导代码（`arch/riscv/boot.S` 或 `arch/loongarch/boot.S`）在完成栈指针设置、清零 BSS 后跳转而来 <sup>[1](#mod-boot-ref-1)</sup>。该函数接收两个参数：`hartid`（硬件线程 ID）和 `dtb`（设备树二进制对象指针），这是 RISC-V SBI 规范的典型约定；LoongArch 路径上参数语义可能不同，但代码通过 `#ifdef ARCH_RISCV` 编译期分派来隔离差异 <sup>[2](#mod-boot-ref-2)</sup>。

与 xv6-riscv 原版 `main.c` 相比，本内核的 `kmain` 额外承担了测试编排职责——初始化完成后直接扫描 ext4 测试盘上的 `*_testcode.sh` 脚本并按阶段排序执行，而非启动 shell。这意味着内核启动的"成功"标准不再是进入用户态交互，而是完成一批自动化脚本的运行。

### 2. 初始化序列：从裸机到多任务就绪

`kmain` 的初始化调用链严格遵循依赖拓扑 <sup>[3](#mod-boot-ref-3)</sup>：

1. **`arch_early_init(hartid, dtb)`** — 架构早期初始化（如 PLIC/中断控制器、定时器基地址配置），必须先于任何设备操作。
2. **物理内存发现** — RISC-V 路径尝试从 FDT 解析 `/memory` 节点；若缺失则回退到硬编码 128MB@`0x80000000` 并打印警告。LoongArch 路径完全硬编码为 256MB@`0x9000000000000000` <sup>[4](#mod-boot-ref-4)</sup>。两种路径都依赖链接器导出的 `__bss_end` 符号确定内核镜像尾端，`free_start = max(base, (u64)__bss_end)` 保留内核占用区域。
3. **`pmm_init(free_start, ram_end)`** — 物理页分配器初始化，建立空闲链表。
4. **`kvm_init()` + `kvm_enable()`** — 创建内核页表并启用分页。必须在任何堆分配之前完成，因为 `kheap_init` 依赖虚拟地址映射。
5. **`kheap_init()`** — 内核堆初始化，为后续 `proc_init`、文件系统等提供动态内存。
6. **`proc_init()` / `trap_init()` / `timer_init()`** — 进程表、陷阱向量、定时器中断，构建多任务基础。
7. **`file_init()` / `blk_init()` / `ext4_mount()`** — 文件与块设备层。

一个关键设计取舍：步骤 2-6 中所有初始化函数返回值均为 `void` <sup>[5](#mod-boot-ref-5)</sup>。调用方无法感知初始化失败——这与 xv6 的教学传统一致，但在生产场景中，`pmm_init` 若因内存不足无法建立空闲链表，后续 `kheap_init` 将直接触发缺页异常而崩溃，且无任何诊断信息。

### 3. 测试框架：ext4 遍历与脚本调度

初始化完成后，`run_test_scripts()` 接管控制流 <sup>[6](#mod-boot-ref-6)</sup>：

- **递归扫描**：`scan_dir` 从 ext4 根目录出发，深度上限 `MAX_SCAN_DEPTH=4`，通过 `ext4_readdir` 注册回调 `scan_cb`。回调中匹配 `*_testcode.sh` 后缀的文件，对目录则递归深入。
- **ext4 类型判断的二次查询**：`scan_cb` 中当目录项类型既非 `EXT4_FT_DIR` 也非 `EXT4_FT_REG_FILE` 时，额外调用 `ext4_is_dir(ino)` 做磁盘级二次判断 <sup>[7](#mod-boot-ref-7)</sup>。这可能是为了兼容某些不设置目录类型标志的 ext4 变体，但引入了隐式的磁盘 I/O 开销。
- **阶段排序**：收集到的脚本按 `script_phase()` 返回的阶段号（basic=0, busybox=1, cyclictest=2, …, ltp=10）排序，同阶段按路径字典序。排序使用冒泡算法 O(n²) <sup>[8](#mod-boot-ref-8)</sup>，在 n≤64 的评测场景中可接受，但缺乏可扩展性。
- **执行**：每个脚本通过 `find_busybox_for` 定位 busybox 可执行文件（依次尝试脚本所在目录、`/busybox`、`/bin/busybox`），构造 `busybox sh <script>` 命令行，调用 `exec_user_cwd` 加载执行 <sup>[9](#mod-boot-ref-9)</sup>。每次执行后调用 `scheduler()` 让出 CPU 等待子进程结束。

若磁盘上无匹配脚本，代码回退到执行 `/init`，模拟传统 Unix 启动行为。最后调用 `arch_shutdown()` 关机。

### 4. 边角细节与不足

1. **初始化失败不可检测** <sup>[5](#mod-boot-ref-5)</sup>：`pmm_init`、`kvm_init`、`kvm_enable`、`kheap_init`、`proc_init`、`trap_init`、`timer_init` 均返回 `void`。任何早期初始化失败都会导致后续代码在未定义状态下崩溃，且无法给出有意义的错误信息。
2. **LoongArch 内存硬编码** <sup>[4](#mod-boot-ref-4)</sup>：LoongArch 路径完全跳过 FDT 解析，将内存硬编码为 256MB@`0x9000000000000000`。若 QEMU 配置不同内存量（如 512MB 或 1GB），内核会浪费上半部分物理内存或因基址不匹配而无法启动。
3. **路径拼接的边界条件** <sup>[10](#mod-boot-ref-10)</sup>：`join_path` 内部循环以 `n < max - 1` 为边界，在 `max ≤ 1` 时立即返回 -1 不写入任何数据，行为正确但调用方 `scan_cb` 以栈上 `char path[MAX_TEST_PATH]` 传入，安全性完全依赖编译期常量 `MAX_TEST_PATH=160` 足够容纳任意路径。
4. **busybox 查找的静默回退** <sup>[9](#mod-boot-ref-9)</sup>：`find_busybox_for` 依次尝试三个路径后无条件回退到 `/bin/busybox`，即使该路径也不存在。后续 `exec_user_cwd` 失败时仅报告 "failed to start"，不区分"busybox 缺失"与"脚本错误"。
5. **无 SMP 初始化** <sup>[1](#mod-boot-ref-1)</sup>：`kmain` 接收 `hartid` 参数但未见任何次级硬件线程（secondary hart）的启动逻辑。内核全程单核运行，评测场景可接受但限制并发能力。
6. **`sort_scripts` 结构体整体拷贝** <sup>[8](#mod-boot-ref-8)</sup>：冒泡交换时使用 `memcpy(&tmp, &list->scripts[j-1], sizeof(tmp))` 做整体结构体拷贝（每个 `test_script` 约 288 字节），虽正确但不如交换指针高效。
7. **`run_user_program` 忽略 `exec_user_cwd` 的错误码细节** <sup>[6](#mod-boot-ref-6)</sup>：函数将 `exec_user_cwd` 的返回值仅做 `< 0` 判断，丢失了具体的 errno（如 `-ENOENT` vs `-ENOMEM`），上层无法区分失败原因。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该内核的内存管理由三层构成: (1) 物理页分配器 (pmm.c) — 基于空闲链表, 复用空闲页自身存储指针, 并附加静态数组 refcnt[] 实现引用计数以支撑 COW; (2) 内核堆分配器 (kheap.c) — 地址有序的空闲链表, 分配时分裂、释放时合并, 按需从物理分配器申请单页增长; (3) 虚拟内存子系统 (vm.h) — 声明了页表操作、copyin/copyout、COW fork (uvmcopy_cow/uvmcow) 与进程页表管理接口. 相比 xv6-riscv, 本模块新增了 COW、mmap/munmap/mprotect 系列 POSIX 系统调用, 以及 NUMA 相关的 mbind/migrate_pages 等桩调用, 表明内核向上提供了更完整的内存管理语义.

### 1. 三层内存管理架构

物理页分配器 (`pmm.c`) 是整个内存子系统的基石. 它采用经典的空闲链表设计: 每个空闲 4 KiB 页的前几个字节存储指向下一个空闲页的指针 (`struct run`), 因此分配器自身不额外消耗内存元数据 <sup>[1](#mod-mm-ref-1)</sup>. 与纯空闲链表不同, 这里额外引入了一个静态数组 `refcnt[MAX_REF_PAGES]` 用于跟踪每页的引用计数 <sup>[2](#mod-mm-ref-2)</sup>. 这个数组是 COW 机制的前提 — 当多个进程共享同一物理页时, 通过 `kref_page` 递增计数; 释放时 `kfree_page` 仅当计数归零才真正归还至空闲链表 <sup>[3](#mod-mm-ref-3)</sup>.

内核堆分配器 (`kheap.c`) 构建在物理分配器之上. 它是一个地址有序的 best-fit 分配器, 每个内存块头部嵌入 `struct hdr` 记录块大小与链表指针 <sup>[6](#mod-mm-ref-6)</sup>. 分配时遍历空闲链表寻找第一个足够大的块, 若剩余空间大于 `MINCHUNK` 则分裂; 释放时通过 `insert_free` 按地址插入并尝试与前驱和后继合并 <sup>[8](#mod-mm-ref-8)</sup>. 堆的扩展以单页为单位调用 `kalloc_page`, 不预先储备多个页面, 这简化了实现但导致多页分配需多次触发 `grow()` <sup>[9](#mod-mm-ref-9)</sup>.

虚拟内存层 (`vm.h` + 未展示的 `arch/riscv/vm.c`) 提供页表操作原语. 权限位直接使用 RISC-V Sv39 PTE 编码 (`PERM_R/W/X/U`) <sup>[11](#mod-mm-ref-11)</sup>, 地址空间布局沿用 xv6-riscv 惯例: 跳板页 (TRAMPOLINE) 和陷阱帧 (TRAPFRAME) 映射在 256 GiB 虚拟地址空间顶端, 用户栈紧邻其下 <sup>[12](#mod-mm-ref-12)</sup>. 值得关注的是 `uvmcopy_cow`、`uvmcow`、`uvmcopy_page`、`uvmshare_page` 四个 COW 相关声明, 它们构成了 fork 时零拷贝共享、写时复制分离的完整路径.

### 2. 物理页引用计数: COW 的隐形支柱

`pmm.c` 的引用计数机制是理解整个内存管理设计的关键. 数组 `refcnt` 使用 `u16` 类型, 以物理页号索引 <sup>[2](#mod-mm-ref-2)</sup>. 初始化流程值得细究: `pmm_init` 遍历 RAM 区间, 先将每页的 `refcnt` 置 1, 然后调用 `kfree_page` — 后者会递减计数, 若归零则加入空闲链表 <sup>[5](#mod-mm-ref-5)</sup>. 这种"先设为 1 再走释放路径"的做法看似迂回, 实则是为了复用 `kfree_page` 中的边界检查、对齐断言与毒化逻辑, 避免在初始化路径中重复代码.

COW 的生命周期如下: (1) `fork()` 时, `uvmcopy_cow` 将父进程的页表项标记为只读, 子进程映射同一物理页, 并调用 `kref_page` 递增 `refcnt`; (2) 当任一进程写该页时, 触发页故障, `usertrap` 调用 `uvmcow` 分配新物理页、复制内容、更新相关进程的页表, 然后对旧页调用 `kfree_page` 递减计数; (3) 最终计数归零时, 物理页被毒化 (填充 0x1) 后归还空闲链表 <sup>[3](#mod-mm-ref-3)</sup>. 毒化操作 `memset(pa, 1, PGSIZE)` 是调试辅助, 可捕获 use-after-free, 但仅在早期验证阶段有效 <sup>[3](#mod-mm-ref-3)</sup>.

`kref_page` 的防御式检查尤为严格: 它断言地址对齐、地址在管理范围内且 `refcnt` 非零 (即页面必须已被分配) [ref:2-impl]. 任何违反都会触发 `panic`, 这在开发阶段能快速暴露错误, 但在生产环境中缺乏优雅降级.

### 3. 内核堆分配器的取舍: 简洁与局限

`kheap.c` 的设计目标明确: 为内核小对象提供快速分配, 超过一页的请求直接拒绝 (`panic`) <sup>[9](#mod-mm-ref-9)</sup>. 这迫使调用方对页表、进程控制块等大对象直接使用 `kalloc_page`, 堆仅服务于 `struct file`、`struct inode`、路径名缓冲区等小对象. 这种分离避免了通用分配器处理大块时的碎片化问题, 但也要求开发者明确知晓分配上限.

地址有序插入 (`insert_free`) 是减少碎片的关键 <sup>[8](#mod-mm-ref-8)</sup>. 它保证空闲链表按地址递增排列, 使合并操作仅需检查相邻块, 不需要遍历整个链表. 但合并逻辑使用了 `container_of` 宏从 `next` 字段地址反推包含它的 `struct hdr` <sup>[8](#mod-mm-ref-8)</sup>, 这依赖于编译器对结构体成员偏移的计算, 且需额外处理 `pp == &freelist` 的特例 (因为 `freelist` 不是任何 `struct hdr` 的成员). 这种技巧虽然精巧, 但可读性差, 容易在重构时引入 bug.

`kmalloc` 的两次尝试循环 (`attempt < 2`) 是值得注意的细节 <sup>[9](#mod-mm-ref-9)</sup>: 第一次遍历若找不到足够大的块, 则调用 `grow()` 申请一页, 第二次遍历应当成功. 但若请求大小恰好接近 `PGSIZE - HDR`, 而 `grow()` 只提供一页, 则新页扣除头部后剩余空间刚好满足需求, 不存在失败可能. 真正会失败的只有物理内存耗尽, 此时 `kmalloc` 返回 `NULL`, 调用方需自行检查 — 但内核中大量调用点可能忽略此检查.

### 4. 跨模块协同: 从 trap 到 mmap

内存管理并非孤立运转. 以 COW 页故障为例, 完整流程跨越三个模块:
- **trap 处理** (`usertrap` 在 `arch/riscv/` 中, 由 `vm.h` 声明 <sup>[11](#mod-mm-ref-11)</sup>): 读取 `stval` 获取故障地址, 识别为页面写保护故障;
- **VM 层** (`uvmcow`): 分配新物理页 (`kalloc_page`), 复制内容 (`memcpy`), 修改源和目标页表项权限, 对旧页调用 `kfree_page` 递减引用;
- **物理分配器** (`kfree_page`): 引用计数减 1, 若归零则毒化并回收.

`copyin/copyout` 是另一个关键跨模块接口 <sup>[11](#mod-mm-ref-11)</sup>. 它们在内核态访问用户空间地址, 必须走页表遍历 (`walkaddr`) 获取物理地址. 这要求用户页表在系统调用期间保持活跃 — 内核不切换到内核页表 (或通过 trampoline 切换), 这是 xv6 式内核的基本约定.

内存布局常量 (`memlayout.h`) 被汇编和 C 共享 <sup>[12](#mod-mm-ref-12)</sup>, `#ifdef __ASSEMBLER__` 分支确保 `MAXVA` 在汇编上下文中使用立即数而非 UL 后缀. `USTACK_PAGES` (1024) 定义了用户栈的最大虚拟地址窗口, 但初始只映射 16 页 (`USTACK_INITIAL_PAGES`), 剩余页面通过缺页按需分配 — 这与 xv6 的 `uvmalloc` 惰性栈增长机制一致.

### 5. 边角细节与不足

**<sup>[2](#mod-mm-ref-2)</sup> 引用计数溢出风险**: `refcnt[]` 使用 `u16` 类型, 最大 65535. 虽然单页被数百个进程共享的场景极为罕见, 但在容器或大量线程场景下, 若 COW 页面被反复 fork 而不写入, 理论上可能溢出. 溢出后回绕至 0, 导致过早释放 — 这是严重的安全漏洞.

**<sup>[5](#mod-mm-ref-5)</sup> MAX_REF_PAGES 硬编码**: 静态数组 `refcnt[262144]` 恰好对应 1 GiB 物理内存 (262144 × 4096). `pmm_init` 在 `page_count > MAX_REF_PAGES` 时直接 `panic`. 对于配置超过 1 GiB RAM 的系统, 内核无法启动. 虽可通过重新编译调整, 但缺乏运行时动态分配能力.

**[无锁竞争] 全局空闲链表无并发保护**: `pmm.c` 和 `kheap.c` 的 `freelist` 均为全局变量, 未使用自旋锁或关中断保护. 在多核环境下, 并发的 `kalloc_page` / `kfree_page` 会导致链表损坏. xv6-riscv 原版使用 `kmem.lock` 保护空闲链表, 本实现缺失此机制, 可能是尚未完成 SMP 适配.

**<sup>[10](#mod-mm-ref-10)</sup> kfree 不毒化内存**: `kfree_page` 在回收物理页时用 0x1 填充以暴露 use-after-free <sup>[3](#mod-mm-ref-3)</sup>, 但 `kfree` 直接调用 `insert_free` 而不清理被释放的堆内存. 这使堆对象的 use-after-free 更难检测. 一致性不足.

**<sup>[9](#mod-mm-ref-9)</sup> kmalloc 对超大请求直接 panic**: 当 `need > PGSIZE` 时, `kmalloc` 调用 `panic` 而非返回 NULL. 这意味着任何超过约 4080 字节的内核堆分配将导致系统崩溃, 而非优雅失败. 更健壮的做法是返回 NULL 让调用方处理.

**<sup>[8](#mod-mm-ref-8)</sup> insert_free 的 container_of 依赖**: 合并前驱时使用 `container_of(pp, struct hdr, next)` 从 `next` 成员指针反推结构体. 这要求 `struct hdr` 的 `next` 字段偏移量正确, 且编译器未对结构体布局做出意外调整. 虽然 C 标准保证首个成员与结构体地址相同, 但 `next` 并非首个成员 — `size` 才是. 若启用了结构体打包或非标准对齐, 此宏可能返回错误地址.

**[桩系统调用]**: 从 `syscall.h` 可知 `mbind`(235)、`get_mempolicy`(236)、`set_mempolicy`(237)、`migrate_pages`(238)、`move_pages`(239) 均注册了系统调用号, 但其实现极可能为桩函数 (直接返回 -ENOSYS 或 0). 这些 NUMA 相关接口在单节点内核中无实际意义, 注册它们可能是为了兼容性占位.

    struct hdr **pp = &freelist;
    while (*pp && *pp < blk)
        pp = &(*pp)->next;

    /* coalesce with successor */
    if (*pp && (char *)blk + blk->size == (char *)*pp) {
        blk->size += (*pp)->size;
        blk->next = (*pp)->next;
    } else {
        blk->next = *pp;
    }

    struct hdr *prev = (pp == &freelist) ? NULL : container_of(pp, struct hdr, next);

### ⚠ 开放问题

- 全局空闲链表 (pmm freelist / kheap freelist) 无锁保护, 多核并发分配/释放将导致链表损坏.
- MAX_REF_PAGES 硬编码 262144 (1 GiB), 超过此容量的物理内存将触发 panic 拒绝启动.
- refcnt[] 为 u16 类型, 极端 COW 场景下可能溢出回绕导致页面过早释放.
- kmalloc 对超过 PGSIZE 的请求直接 panic 而非返回 NULL, 缺乏优雅降级.
- kfree 不毒化释放的内存, 与 kfree_page 的行为不一致, 削弱 use-after-free 检测能力.

> 💡 **TL;DR**
>
> 本模块管理全部用户进程与内核线程的生命周期, 核心抽象为静态进程表 `proctab[NPROC]` 与同规模的内核栈池. 调度器采用简单轮转 (round-robin), 配合 `sleep`/`wakeup` 实现基于 `chan` 指针的阻塞同步. 与 xv6-riscv 相比, 本实现最大扩展在于 `clone()` 系统调用: 支持 `CLONE_VM`/`CLONE_THREAD` 等 Linux 标志, 通过独立页表根 + 物理页引用计数实现线程间共享用户页, 非共享区域则采用写时复制 (COW). 此外新增了进程组/会话管理、`proc_handle_page_fault` 缺页处理、以及初步的信号终止机制; 但非终止信号的实际递交尚未实现, 调度策略也缺少优先级支持.

### 1. 核心抽象: 静态进程表与轮转调度

所有进程与内核线程统一由 `struct proc` 表示, 静态分配在 `proctab[NPROC]` 中, 每个槽位预分配一块独立的内核栈 `kstacks[NPROC][KSTACK_SIZE]` <sup>[1](#mod-task-ref-1)</sup>. 这种设计放弃了动态分配灵活性, 换取了可预测的内存占用和免除内核态 `malloc` 的复杂度 — 但也将系统并发度硬限制在 `NPROC` 以内. 分配新进程通过 `proc_alloc()` 线性扫描 `UNUSED` 槽位, 失败时直接返回 `NULL` 而非阻塞等待 <sup>[2](#mod-task-ref-2)</sup>; 这意味着调用者必须处理进程表满的 `-EAGAIN` 错误, 而无法像 Linux 那样依赖 `fork()` 的 `ENOMEM` 信号. 进程标识 `next_pid` 简单递增, 既不回收也不检查冲突, 若系统长期运行存在理论上的回绕风险 <sup>[2](#mod-task-ref-2)</sup>.

调度器 `scheduler()` 构成典型的 xv6 风格轮转循环: 关中断 → 遍历进程表 → 选中第一个 `RUNNABLE` 进程 → 通过 `swtch()` 切换上下文 <sup>[3](#mod-task-ref-3)</sup>. 与 xv6 原版不同的是, 当没有可运行进程时, 本调度器会检查是否仍有 `SLEEPING` 进程 — 若没有则直接 `return`, 这是为 Phase-1 内核线程演示设计的退出路径, 在完整操作系统中调度器不应返回 <sup>[3](#mod-task-ref-3)</sup><sup>[4](#mod-task-ref-4)</sup>. 闲置期间通过 `arch_idle()` 等待中断, 保持中断打开以允许定时器或其他设备唤醒进程. 阻塞原语 `sleep(void *chan)` 将当前进程的 `chan` 字段设置为等待通道、状态切为 `SLEEPING`, 然后调用 `sched()` 让出 CPU; 唤醒者调用 `wakeup(chan)` 将所有等待同一通道的 `SLEEPING` 进程批量置为 `RUNNABLE` <sup>[5](#mod-task-ref-5)</sup><sup>[6](#mod-task-ref-6)</sup>. 这里的 `chan` 是任意指针 (常为内核对象地址), 这种无类型等待通道是 xv6 的经典取舍: 简单、快速, 但缺乏超时和优先级唤醒, 且容易出现"惊群" (所有等待者同时被唤醒).

### 2. clone() 与线程模型: 超越 xv6 的 Linux 兼容层

本模块最大的工程扩展在于 `clone()` 系统调用, 它远远超出了 xv6 的 `fork()` 语义. `clone()` 支持 `CLONE_VM`、`CLONE_THREAD`、`CLONE_SETTLS`、`CLONE_CHILD_CLEARTID` 等六个 Linux 标志 <sup>[7](#mod-task-ref-7)</sup>, 允许创建共享地址空间的用户线程.

其核心取舍在于"共享地址空间"的实现策略: 当 `CLONE_VM` 置位时, 父子进程各自持有独立的页表根 (`proc_pagetable()` 为每个进程分配独立根页表), 但用户态物理页面通过 `uvmshare_page()` 共享, 并依赖物理页引用计数防止提前释放 <sup>[8](#mod-task-ref-8)</sup>. 这样做的原因是每个线程需要自己的 `TRAPFRAME` 映射 (用于保存用户态陷入上下文), 而 xv6 的 `TRAPFRAME` 通常映射在固定虚拟地址 — 若共享同一页表根, 多个线程会互相覆盖陷入帧. 因此本实现选择了"独立页表根 + 共享用户叶子"的折中, 这比 Linux 的完全共享页表更安全, 但增加了页表内存开销和 TLB 压力. 对于非 `CLONE_VM` 的 `fork()`, 则采用写时复制 (`uvmcopy_cow`), 避免传统 xv6 的完整内存拷贝 <sup>[8](#mod-task-ref-8)</sup>.

`clone()` 的出错处理路径值得注意: 在分配 `trapframe` 或页表失败时, 函数直接回填 `np->state = UNUSED` 并释放已分配资源; 但在 `CLONE_PARENT_SETTID`/`CLONE_CHILD_SETTID` 的 `copyout` 失败时, 它不会立即放弃, 而是尝试通过 `uvmprotect()` 为目标地址补充写权限后重试一次 <sup>[9](#mod-task-ref-9)</sup>. 这种"自动修复权限"的行为对用户态透明, 但也掩盖了潜在的非法地址错误: 若两次 `copyout` 均失败, 虽然通过 `clone_cleanup()` 正确释放了子进程资源, 但父进程的 `ptid` 指向的地址已经被第一次 `copyout` 部分写入过(尽管返回值指示失败), 可能存在残留数据风险.

### 3. 进程生命周期与信号机制的早期形态

进程退出路径分为两条: 正常 `proc_exit(code)` 和信号触发的 `proc_exit_signal(sig)`. 两者的共同点是均先关闭中断, 然后判断是否为线程 — 线程直接释放资源 (`proc_release_thread`) 并切换走, 进程则调用 `proc_zombify()` 进入 `ZOMBIE` 状态 <sup>[10](#mod-task-ref-10)</sup><sup>[11](#mod-task-ref-11)</sup>. `proc_zombify()` 做三件事: 关闭文件描述符、将子进程过继给 `initproc`、唤醒父进程. 这是 xv6 标准流程, 但增加了一个细节: 若被过继的子进程已是 `ZOMBIE`, 会立即唤醒 `initproc` 以便其回收 <sup>[12](#mod-task-ref-12)</sup>.

`proc_kill()` 实现了 `kill(2)` 的进程组匹配语义 (`pid>0` / `pid==0` / `pid==-1` / `pid<-1`), 但信号递送存在明显的未完成痕迹 <sup>[13](#mod-task-ref-13)</sup>. 对于终止信号 (`SIGHUP`/`SIGINT`/`SIGQUIT`/`SIGABRT`/`SIGKILL`/`SIGPIPE`/`SIGALRM`/`SIGTERM`), 函数递归调用 `proc_terminate_tree()` 遍历进程树设置 `killed` 标志并唤醒睡眠进程; 对于非终止信号 (如 `SIGSTOP`、`SIGCONT`、`SIGUSR1` 等), 则**完全不做任何动作** — 仅设置 `found=1` 然后继续循环 <sup>[13](#mod-task-ref-13)</sup>. 这意味着用户态注册的信号处理器永远不会被调用, 信号队列和 `sigaction` 机制均未实现. 当前信号支持仅停留在"可终止进程"的层次, `proc_exit_signal` 通过 `XSTATE_SIGNAL` 标志位将信号编号编码到退出状态中, 由 `wait4()` 解码返回给父进程 <sup>[14](#mod-task-ref-14)</sup>.

`wait4()` 在此基础上支持 `WNOHANG` (`options & 1`), 并正确实现了 `pid` 参数的 BSD 语义 <sup>[14](#mod-task-ref-14)</sup>. 但循环始终在关中断状态下运行 — 包括调用 `sleep(p)` 进入阻塞. `sleep()` 自身在返回时**不恢复中断**, 这依赖调用者 `wait4` 的循环始终保持关中断 (注释明确注明 "interrupts stay disabled") <sup>[5](#mod-task-ref-5)</sup><sup>[14](#mod-task-ref-14)</sup>. 这种设计减少了竞态窗口, 但要求所有 `sleep` 调用方都理解这一约定, 极易因疏忽导致中断永久关闭.

### 4. 缺页处理与 COW 的跨模块协同

`proc_handle_page_fault()` 是调度模块与虚拟内存子系统之间最紧密的耦合点. 当陷入处理函数识别出缺页异常时, 控制流进入此函数, 它执行四层回退策略 <sup>[15](#mod-task-ref-15)</sup>:

1. **COW 解析**: 调用 `uvmcow()` 尝试为当前页表复制共享页面;
2. **线程组共享**: 若 COW 失败且页面已映射 (`walkaddr` 返回非零), 说明页表项存在但其权限不允许访问; 此时遍历同 `tgid` 的其他进程, 若某线程的页表中该地址已映射且可访问, 则通过 `uvmshare_page()` 将物理页共享到当前页表, 并更新 `mmap_top` 和 `sz` <sup>[15](#mod-task-ref-15)</sup>;
3. **栈守卫页**: 对 `USTACK_TOP - USTACK_PAGES*PGSIZE` 之上的地址, 通过 `uvmprotect` 直接授予读写权限, 实现按需增长的栈;
4. **VMA 权限恢复**: 检查进程的 `vmas[]` 数组, 若匹配到已注册的映射区域, 则按记录的 `perm` 恢复页表项权限;
5. **线程组宽恕**: 再次遍历线程组, 对任何 `a < q->sz` 的页授予读写权限.

这套逻辑的巧妙之处在于: 它利用了 `mmap_top` 作为用户栈与 mmap 区域的分界线, 将栈自动增长与 COW 解析统一在同一缺页路径中. 但另一方面, 该函数在扫描全局 `proctab` 时未持有任何锁 — 当前单核设计下安全, 但代码中已出现 `mycpu()` 的调用, 暗示未来可能支持多核. 若引入 SMP, 此处的无锁遍历将直接导致竞态 <sup>[15](#mod-task-ref-15)</sup>.

### 5. 边角细节与不足

- **<sup>[16](#mod-task-ref-16)</sup> 中断关闭传播约定**: `sleep()` 进入时关中断、返回时维持关中断, 依赖于调用者 (`wait4`、`proc_exit` 等) 正确管理中断状态. 若某新调用点误以为 `sleep()` 返回后中断已恢复, 将导致系统静默死锁. 这种隐式约定缺乏静态检查手段.
- **<sup>[17](#mod-task-ref-17)</sup> `kthread_create` 与 `proc_alloc` 的代码重复**: 内核线程创建函数自行遍历 `proctab` 并逐字段初始化, 与 `proc_alloc()` 的逻辑大量重叠. 若后续新增进程字段 (如 cgroup、命名空间), 极易遗漏其中一处, 导致内核线程与用户进程的初始化不一致.
- **<sup>[13](#mod-task-ref-13)</sup> 非终止信号无实际递送**: `proc_kill()` 对 `SIGSTOP`/`SIGCONT` 等信号仅记录 `found=1`, 既不设置 `p->killed` 也不触发任何用户态处理. 这导致依赖信号的应用程序 (如 `kill -STOP`) 完全无效.
- **<sup>[18](#mod-task-ref-18)</sup> 长时间关中断的风险**: `proc_kill()` 在关中断后调用 `proc_terminate_tree()`, 后者递归遍历进程树并逐个修改状态. 若进程层次深 (如 shell 脚本产生的中度 fork 链), 中断关闭时长可能超过定时器间隔, 造成时钟丢失或外部设备数据溢出.
- **<sup>[3](#mod-task-ref-3)</sup> 调度器退出逻辑**: `scheduler()` 在无 `RUNNABLE` 且无 `SLEEPING` 进程时直接 `return`, 这是 Phase-1 演示的临时设计. 在完整操作系统中, 调度器应永远不返回 (即使所有进程退出也应 idle 或 halt). 目前该路径仅被 `phase1_demo()` 调用, 但若未来将 `scheduler()` 作为通用调度入口, 此行为需彻底重写.
- **<sup>[19](#mod-task-ref-19)</sup> `next_pid` 无回绕保护**: PID 计数器单调递增, 未检测是否已分配过或已回绕. 虽然 32 位有符号整数溢出需极长时间, 但在嵌入式或长时间运行场景中并非不可能.
- **<sup>[20](#mod-task-ref-20)</sup> `groups[8]` 硬编码限制**: 组成员数组固定为 8 个元素, 超出部分被静默截断, 且无任何错误返回. 这与 Linux 的 `ngroups_max` 可配置性形成对比.
- **<sup>[21](#mod-task-ref-21)</sup> 演示代码残留**: `demo_thread()` 和 `phase1_demo()` 函数包含忙等待循环和调试输出, 属于早期原型阶段的产物, 应被条件编译隔离或移除.

void scheduler(void)
{
    struct cpu *c = mycpu();
    c->proc = NULL;

    for (;;) {
        arch_intr_off();
        int ran = 0;
        for (int i = 0; i < NPROC; i++) {
            struct proc *p = &proctab[i];
            if (p->state != RUNNABLE)
                continue;
            p->state = RUNNING;
            c->proc = p;
            swtch(&c->ctx, &p->ctx);
            c->proc = NULL;
            ran = 1;
        }
        if (ran)
            continue;
        int sleeping = 0;
        for (int i = 0; i < NPROC; i++)
            if (proctab[i].state == SLEEPING)
                sleeping = 1;
        if (!sleeping)
            return;
        arch_intr_on();
        arch_idle();
    }
}
```

int wakeup_n(void *chan, int max)
{
    int was = arch_intr_get();
    arch_intr_off();
    int count = 0;
    for (int i = 0; i < NPROC; i++) {
        struct proc *p = &proctab[i];
        if (p->state == SLEEPING && p->chan == chan) {
            p->state = RUNNABLE;
            count++;
            if (max > 0 && count >= max)
                break;
        }
    }
    if (was)
        arch_intr_on();
    return count;
}
```

### ⚠ 开放问题

- proc_kill 对非终止信号 (SIGSTOP/SIGCONT 等) 无任何递送动作, 仅返回成功; 信号机制严重不完整.
- sleep() 返回后中断保持关闭的隐式约定无编译期或运行时检查, 易导致调用者永久关中断.
- kthread_create 与 proc_alloc 存在大量字段初始化重复, 新增进程字段时极易遗漏同步.
- scheduler() 在无进程时直接 return 的退出路径仅适用于 Phase-1 演示, 正式系统需改写为永久 idle 循环.

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块通过struct file统一抽象实现混合文件系统：控制台设备、只读EXT4、内存ramfs、管道及简单套接字。file_init在ramfs中预置类Unix目录树，ext4_mount挂载只读磁盘。文件操作借助copyin/copyout与进程交互。与xv6相比，扩展了EXT4只读支持和内存文件系统，但未提供持久化写入与并发保护。

### 1. 统一文件抽象与混合设计
`struct file`通过`type`字段区分FD_DEVICE、FD_INODE、FD_RAMFS、FD_PIPE、FD_SOCKET等类型 <sup>[1](#mod-fs-ref-1)</sup>；`file_alloc`从固定数组`ftable`分配，`file_close`减少引用计数并回收资源 <sup>[2](#mod-fs-ref-2)</sup>。全局无锁，多核下存在竞态。`file_init`在ramfs中创建`/dev`、`/bin`、`/proc`及设备节点种子 <sup>[3](#mod-fs-ref-3)</sup>，使内核自举不依赖外部持久化，但目录结构硬编码且写数据重启丢失。

### 2. 只读EXT4驱动：缓存与扩展树
`ext4_mount`读取超块，验证魔数0xef53，计算块大小 <sup>[4](#mod-fs-ref-4)</sup>。驱动采用hand替换的固定大小块缓存 <sup>[5](#mod-fs-ref-5)</sup>，无LRU。`extent_map`遍历扩展树，最多8层深度防护，返回0表示空洞 <sup>[6](#mod-fs-ref-6)</sup>；仅支持extent布局，放弃间接块以简化实现。目录查找`ext4_lookup`使用`dir_find`配合`lookup_cache` <sup>[7](#mod-fs-ref-7)</sup>。

### 3. ramfs：内存文件系统
基于`ramtab`数组（容量1024），节点`struct ramnode`以绝对路径索引 <sup>[8](#mod-fs-ref-8)</sup>。文件数据按页分配（`ramfs_page_at`），最大8MB <sup>[9](#mod-fs-ref-9)</sup>。`ramfs_rename`全表扫描重命名子节点 <sup>[10](#mod-fs-ref-10)</sup>。`ramfs_readdir`通过回调列出目录项 <sup>[11](#mod-fs-ref-11)</sup>，性能与节点数线性相关。

### 4. 跨模块协同
I/O通过`copyin/copyout`与用户空间交互，需要当前进程页表且依赖`myproc()` <sup>[12](#mod-fs-ref-12)</sup>。`ramfs_read_user`处理文件空洞时填零 <sup>[13](#mod-fs-ref-13)</sup>。设备文件如`/dev/random`直接生成数据 `random_read_user` <sup>[14](#mod-fs-ref-14)</sup>。管道/套接字使用环形缓冲，但缺少并发保护。

### 5. 边角细节与不足
(1) 并发安全缺失：全局表操作无锁，多核分配/释放可能错乱 [issue:1]。
(2) 整数溢出风险：`ext4_mount`中`1024u << le32(...)`未限制移位量 <sup>[15](#mod-fs-ref-15)</sup>。
(3) 缓存一致性：lookup_cache及block_cache无失效机制，虽只读但未来扩展风险高。
(4) 桩函数`socket_unregister_file`为空 `(void)f;` <sup>[16](#mod-fs-ref-16)</sup>，套接字关闭未完成资源释放。
(5) 路径解析：ramfs依赖路径前缀匹配，无明确父指针，目录遍历可能误判嵌套。

> 💡 **TL;DR**

> 本模块以 Unix 管道为核心 IPC 机制，基于 512 字节环形缓冲区与 sleep/wakeup 阻塞同步，通过文件描述符暴露给用户态。系统调用表中另有 futex、set_robust_list 及 shm* 系列声明，但实现代码未在可见范围内。

### 1. 核心抽象：基于环形缓冲区的阻塞管道

管道由 `struct pipe` 定义 <sup>[1](#mod-ipc-ref-1)</sup>，内含固定 512 字节缓冲区 `buf`，64 位计数器 `nread`/`nwrite` 跟踪已消费/已生产字节数，以及 `readopen`/`writeopen` 状态标志。通过取模运算 `% PIPESIZE`（第 10 行）映射到缓冲区，避免显式回绕逻辑，但 512 字节固定容量限制了大块 I/O 的效率 <sup>[7](#mod-ipc-ref-7)</sup>。

`pipe_alloc` 创建一对 `struct file` 对象分别作为读端和写端，共享同一个管道结构 <sup>[2](#mod-ipc-ref-2)</sup>。这种设计使管道完全融入 Unix “一切皆文件” 模型，用户态通过标准 `read`/`write` 系统调用即可操作。

### 2. 同步原语：sleep/wakeup 的误用与竞态窗口

读写阻塞分别等待 `&pi->nread` 和 `&pi->nwrite` <sup>[4](#mod-ipc-ref-4)</sup><sup>[5](#mod-ipc-ref-5)</sup>，语义上：读者在缓冲区空且写端未关时睡眠；写者在缓冲区满且读端未关时睡眠。致命缺陷在于整个 `pipe` 结构完全无锁保护。原版 xv6 在调用 `sleep` 前持有自旋锁，`sleep` 原子释放锁并进入睡眠，消除检查条件与睡眠间的竞态窗口。本实现直接检查共享字段后调用 `sleep`，存在经典 **丢失唤醒**：写者更新 `nwrite` 并 `wakeup` 可能发生在读者检查条件之后、`sleep` 之前，导致读者永久阻塞。

同样，`pipe_close` <sup>[3](#mod-ipc-ref-3)</sup> 在无锁情况下检查 `readopen==0 && writeopen==0` 后释放管道内存，并发关闭可触发 double-free。

### 3. 跨模块协同与内核态变体

`pipe_alloc` 调用 `file_alloc`（文件系统）、`kzalloc`（内存管理）并挂接文件对象。读写操作依赖 `copyin`/`copyout` 操作用户态虚拟地址，需要当前进程页表 (`myproc()->pagetable`) 且假设用户地址合法。`pipe_write_k` <sup>[6](#mod-ipc-ref-6)</sup> 为内核态提供直接内存访问版本，绕过 `copyin`，用于内核组件间通信，但源指针有效性完全依赖调用者。阻塞时 `sleep` 让出 CPU，依赖调度器在条件满足时通过 `wakeup` 唤醒。

### 4. 边角细节与不足

- **无锁保护**：已述，丢失唤醒与 double-free 风险。
- **错误处理模糊**：`copyin` 失败时 `pipe_write` 仅 `break` 返回已写字节数，未传递错误原因 <sup>[8](#mod-ipc-ref-8)</sup>；`copyout` 同理 <sup>[9](#mod-ipc-ref-9)</sup>。
- **逐字节复制**：读写循环每次仅处理一个字节并调用 `copyin`/`copyout`，效率低下；可批量复制至内核栈再移入缓冲区，减少陷入次数。
- **无 O_NONBLOCK 支持**：管道始终阻塞，缺乏 poll/select 集成。
- **固定缓冲区 512 字节**：容量偏小，频繁上下文切换。
- **缺失的 shm/futex 实现**：系统调用表声明了 `futex`、`set_robust_list` 及 `shm*` 系列，但实现代码未在可见范围内，IPC 功能不完整。

### ⚠ 开放问题

- 整个 pipe 结构无锁保护：并发 read/write/close 存在丢失唤醒导致永久阻塞的风险。
- pipe_close 在无锁下检查 open 标志后 free，并发关闭可能 double-free。

- 读写循环逐字节 copyin/copyout，效率低，未批量操作。
- 未实现 O_NONBLOCK 非阻塞模式及 poll/select 支持。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 本模块提供块设备抽象层 (blkdev.h) 及唯一一个 virtio-blk 驱动程序 (virtio_blk.c)。核心接口为 blk_init / blk_read / blk_write 三个函数，通过轮询 virtqueue 完成同步 I/O，不依赖中断。驱动同时支持 legacy (版本 1) 与 modern (版本 2) 的 virtio-mmio 传输，并为 LoongArch 平台提供 PCI legacy 路径。整体设计追求最小化：无设备模型、无队列深度利用、无 DMA 映射层，依赖身份映射内存假设，是教学内核中典型的“够用即可”的块驱动实现。

### 1. 核心抽象与接口设计

驱动框架的对外合约定义在 `include/drivers/blkdev.h` 中：三个函数 `blk_init`、`blk_read`、`blk_write` 构成完整的块设备操作集合 <sup>[2](#mod-drivers-ref-2)</sup>。初始化函数负责探测设备并配置 virtqueue，读写函数各处理一个 512 字节扇区。这一抽象极度简化——没有异步 I/O、没有设备号、没有分区表、没有多个设备句柄，整个内核假设只存在一个块设备。

驱动实现 `drivers/virtio_blk.c` 围绕 virtio 的 vring 结构展开：`struct virtq_desc`、`struct virtq_avail`、`struct virtq_used` 组成标准的分割式 virtqueue <sup>[3](#mod-drivers-ref-3)</sup>。传输层通过 `vbase`（MMIO 寄存器基址）或 `pci_iobase`（PCI I/O 基址）访问设备寄存器，用 `volatile` 指针强制每次访问生成实际内存指令 <sup>[4](#mod-drivers-ref-4)</sup>。请求结构体 `virtio_blk_req` 仅携带类型、保留字段和扇区号 <sup>[5](#mod-drivers-ref-5)</sup>，省去了 virtio-blk 协议中可选的 writeback 等高级字段，进一步缩减复杂度。

### 2. 关键设计取舍

**轮询而非中断。** 整个内核没有实现中断处理，驱动通过 `while (used->idx != avail->idx)` 忙等设备完成 I/O，超时设为 5 亿次空循环 <sup>[6](#mod-drivers-ref-6)</sup>。这放弃了 CPU 效率与 I/O 并发，换来极简的代码路径——无需中断控制器初始化、无需 IDT/异常向量、无需在驱动与 trap handler 间传递上下文。对于教学内核的单线程批处理场景，这是合理的权衡。

**身份映射而非动态 DMA 映射。** 头注释明确说明“Memory is identity-mapped”，因此在 legacy 初始化中直接将 `legacy_region` 的虚拟地址右移 12 位作为 PFN 写入 `QUEUE_PFN` <sup>[7](#mod-drivers-ref-7)</sup>。这与 xv6-riscv 的做法一致：内核页表将物理地址等同映射到虚拟地址，省去 `kva_to_pa` 转换。但代码并未完全遵守这一约定：在 PCI legacy 路径中又使用了 `kva_to_pa` 进行转换 <sup>[8](#mod-drivers-ref-8)</sup>，暗示身份映射可能并非所有平台都成立，两条路径的不一致是潜在的维护陷阱。

**静态全局变量替代动态分配。** 所有 vring 数据结构——`legacy_region`、`desc_modern`、`avail_modern`、`used_modern`——均为编译期静态数组 <sup>[9](#mod-drivers-ref-9)</sup>。这避免了内核分配器的依赖，但将设备队列深度锁死在 `NUM=256`，且 legacy 与 modern 两套 buffer 同时存在时浪费内存。更关键的是，请求结构体 `req` 和状态字节 `vstatus` 也被声明为 `static` <sup>[10](#mod-drivers-ref-10)</sup>，这意味着整个内核只允许一个正在进行的 I/O 请求，多核并发调用将产生数据竞争。

**Legacy 与现代传输的双路径。** 驱动在 `blk_init_mmio` 中通过读取 version 寄存器判断走 legacy (v1) 还是 modern (v2) 初始化流程 <sup>[11](#mod-drivers-ref-11)</sup>。两条路径对 vring 的布置方式不同：legacy 要求 desc/avail/used 放在同一物理页对齐区域，modern 允许独立布局。这一兼容性设计使内核能在 QEMU 默认的 legacy 模式和较新的 modern 模式下均可运行，但代价是代码膨胀——两套初始化序列、两套全局 buffer，且 LoongArch 的 PCI legacy 路径几乎重复了 MMIO 的初始化逻辑。

### 3. 跨模块协同

块设备驱动主要与文件系统层交互：文件系统通过 `blk_read` / `blk_write` 读取超级块、inode、位图及数据块。调用方传入的缓冲区指针直接作为 DMA 目标地址传递给设备 <sup>[12](#mod-drivers-ref-12)</sup>，要求该缓冲区位于身份映射区且至少 512 字节——这一前置条件并未在接口文档或运行时检查中体现，错误传入栈上小缓冲区将导致内存损坏。

驱动依赖 `kernel/addr.h` 中的 `kva_to_pa` 与 `pa_to_kva` 进行地址转换 <sup>[1](#mod-drivers-ref-1)</sup>。在 PCI 路径中，`pa_to_kva` 将硬编码的物理地址 `LA_PCI_CFG_BASE`、`LA_PCI_IO_BASE` 转换为内核可访问的虚拟地址 <sup>[8](#mod-drivers-ref-8)</sup>，这些宏与 LoongArch 的物理内存布局强绑定，不具备可移植性。

`printk` 被用于输出探测信息和错误诊断 <sup>[13](#mod-drivers-ref-13)</sup>，是驱动唯一依赖的内核服务。没有使用自旋锁或任何同步原语，体现了内核的单核假设。

### 4. 边角细节与不足

**<sup>[14](#mod-drivers-ref-14)</sup> 并发不安全：** `virtio_rw` 使用 `static` 局部变量 `req` 和 `vstatus` 存放 I/O 请求与状态。若两个 CPU 同时调用 `blk_read`，后一次请求会覆盖前一次的 `req` 和描述符链，导致数据错乱。内核当前可能仅在单核运行，但代码未加任何锁或 percpu 保护，是多核迁移的隐性债务。

**<sup>[15](#mod-drivers-ref-15)</sup> 缓冲区溢出风险：** `blk_read` 和 `blk_write` 接收调用方传入的 `void *buf`，直接作为 DMA 缓冲区使用，但不对其长度做任何校验。若文件系统层传入小于 512 字节的缓冲区，设备将写入超出缓冲区的内存。同样，`legacy_region` 中被手动计算偏移的 `avail` 和 `used` 指针依赖于正确的 `PGROUNDUP` 宏定义，如果该宏实现有误，可能造成结构体跨越页边界。

**<sup>[16](#mod-drivers-ref-16)</sup> 平台地址硬编码：** `VIRTIO_MMIO_BASE 0x10001000UL` 和 `LA_PCI_CFG_BASE 0x20000000UL` 等常量直接嵌入代码，未通过设备树或 ACPI 获取。这意味着内核只能在特定版本的 QEMU 虚拟机或特定 LoongArch 开发板上运行，换一个平台就需要修改源码。

**<sup>[17](#mod-drivers-ref-17)</sup> 错误处理不完整：** 设备初始化失败时 `blk_init` 返回 -1，但调用方 `main` 可能忽略该返回值继续启动，导致后续所有 `blk_read` 在 `vbase==NULL` 时解引用空指针。`virtio_rw` 的超时返回 -1 同样可能被上层忽略，文件系统可能将垃圾数据当作有效块内容。

**<sup>[18](#mod-drivers-ref-18)</sup> 内存屏障使用不一致：** vring 操作中使用了 `__sync_synchronize()` 全屏障 <sup>[19](#mod-drivers-ref-19)</sup>，但 `avail->idx++` 并未使用原子加操作。在弱一致性模型 CPU 上，设备可能看到过时的 `idx` 值。同时，`used->idx` 的读取在 while 循环中仅靠 `__sync_synchronize()` 保证刷新，没有 `volatile` 修饰——虽然 `used` 指向的 `virtq_used` 结构体成员并非 volatile，编译器可能将 `used->idx` 提升到循环外，导致死循环。

**<sup>[20](#mod-drivers-ref-20)</sup> 未实现的扩展点：** `blkdev.h` 注释中“single virtio-blk disk for now”暗示未来可能支持多设备，但当前接口完全没有预留设备句柄参数。如果要增加第二块磁盘，必须修改所有现有调用点。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 以 `void syscall(void)` 巨型 switch 语句为核心的系统调用分发器，位于 `kernel/syscall.c`。通过 `p->trapframe->a0..a7` 寄存器获取调用号与参数，分发至 80+ 个文件、进程、内存、网络、信号等兼容 Linux 调用号的 handler。大量调用以桩实现（返回 0 或 `-ENOSYS`），目标并非 POSIX 完备性，而是让未修改的 Linux 二进制直接运行。相比 xv6 原版约 20 个 syscall，本内核扩展了 mmap/futex/socket/nanosleep 等现代 Linux 接口，但分发器采用 switch 而非函数表，且多数桩未实际实现功能。

### 1. 核心抽象：巨型 switch 分发器取代函数表

传统 Unix 内核（含 xv6-riscv）使用静态函数指针数组 `syscalls[]` 按调用号索引分发，本内核改用单一 `switch (num)` 语句，case 总数超过 80 个 <sup>[1](#mod-syscall-ref-1)</sup>。模块头部注释明确宣称"让未修改的 libc 二进制能跑起来"，这直接解释了分发器的组织方式：case 按功能分组（文件、进程、杂项），而非按编号排序。

这一取舍放弃了查表的 O(1) 常数开销以及表驱动的可扩展性，但换取了两个实际优势：(1) 可在 case 块内直接嵌入简单逻辑而不必为每个桩立独立函数——例如 `SYS_set_tid_address` 仅做一次赋值 `p->clear_child_tid = p->trapframe->a0` 即返回；(2) case 排列顺序隐含热路径优先级，文件 I/O 类调用排在进程管理类之前。代价是分发器本身长达约 190 行，维护新 syscall 时需在 switch 内增补 case，相比表驱动更容易引入合并冲突，且编译器无法对 switch 做间接跳转优化（case 稀疏）。

### 2. 关键设计取舍：Linux 二进制兼容的桩策略

该内核的设计目标不是 POSIX 完备性，而是"让未修改的 Linux 二进制能跑起来"。为此，大量系统调用被实现为桩：`sync`/`fsync`/`umount2`/`mount` 直接返回 0；`sys_statfs_common` 伪造 ext 文件系统统计信息（`f_type=0xef53`、`f_blocks=8192`）而完全不查询底层文件系统 <sup>[2](#mod-syscall-ref-2)</sup>；`sys_getrandom` 用线性同余 `(0x9e*(i+1)+0x3b)` 生成确定性字节序列，充当伪随机数源 <sup>[3](#mod-syscall-ref-3)</sup>；`sys_timer_create_compat` 始终返回定时器 ID 为 1。少数关键调用明确返回 `-ENOSYS`（`sendmsg`/`recvmsg`），使依赖这些调用的程序能感知失败而非静默错乱。

与 xv6 原版仅支持约 20 个系统调用相比，该内核扩展至 80+ 个，调用号与 Linux 保持一致（如 `SYS_openat=56`、`SYS_futex=98`），可直接运行动态链接的 Linux 用户程序。信号相关调用（`rt_sigaction`/`rt_sigprocmask`）全部返回 0 且未安装任何信号处理器 <sup>[10](#mod-syscall-ref-10)</sup>，这意味着内核在收到信号时实际不会触发用户态 handler，仅提供"看起来支持"的假象。

### 3. 跨模块协同：trapframe 传参与 copyin/copyout 屏障

系统调用号及参数全部通过 `p->trapframe->a0..a7` 寄存器传递，遵循 RISC-V 调用约定 <sup>[4](#mod-syscall-ref-4)</sup>。分发器从 `a7` 获取调用号，各 handler 自行从 `a0`-`a5` 提取参数。所有用户态提供的指针（路径名、缓冲区、结构体地址）必须经 `copyin`/`copyout` 访问——这两个函数（定义于 `kernel/vm.c`）接受进程页表 `p->pagetable` 做地址翻译，并校验页表项 U 标志，失败返回 -1，调用方统一转为 `-EFAULT` <sup>[5](#mod-syscall-ref-5)</sup>。

值得特别指出的是 `sys_futex`：它在读取用户态 futex 字前先调用 `arch_intr_off()` 关闭中断，检查值与期望一致后才调用 `sleep()`，唤醒后根据之前保存的中断状态决定是否恢复 <sup>[6](#mod-syscall-ref-6)</sup>。这保证了在单核（UP）环境下检查-睡眠序列的原子性，防止 wakeup 丢失。但该实现未使用自旋锁保护 `sleep`/`wakeup` 的内部等待队列，暗示内核目前可能仅支持单核。

`sys_socketpair_compat` 展示了巧妙的跨调用复用：它临时修改 `trapframe->a0`/`a1` 的值，调用 `sys_pipe2` 创建管道对来模拟 socketpair，随后恢复原始寄存器——利用 trapframe 的可写性避免了在 pipe2 内部增加条件分支 <sup>[11](#mod-syscall-ref-11)</sup>。

### 4. 边角细节与不足

- **指针边界校验缺失**：`sys_getrandom` 从用户态接收 `len` 后直接以 `for (u64 i = 0; i < len; i++)` 循环写回，未设置上限。恶意程序传入 `U64_MAX` 将造成极长时间的忙等循环 <sup>[3](#mod-syscall-ref-3)</sup>。
- **整数溢出**：`copy_single_node_mask` 计算 `words = (maxnode + 63) / 64` 时，若用户态传入接近 `U64_MAX` 的 `maxnode`，加法溢出导致 `words` 变为 0，后续循环被跳过而无错误提示 <sup>[7](#mod-syscall-ref-7)</sup>。
- **资源泄漏**：`sys_brk` 处理堆缩减时仅更新 `p->brk = addr`，未调用 `uvmdealloc` 回收页表项和物理页，造成内存泄漏 <sup>[8](#mod-syscall-ref-8)</sup>。
- **竞态条件**：`sys_migrate_pages_compat` 等调用 `proc_find(pid)` 检查进程存在性后，未持任何锁即信任该结果，与目标进程并发退出形成 TOCTOU 竞态 <sup>[9](#mod-syscall-ref-9)</sup>。
- **硬编码魔术数字**：`sys_statfs_common` 返回的 `f_type`、`f_blocks`、`f_bfree` 等均为编译期常量，与底层文件系统实际状态无关 <sup>[2](#mod-syscall-ref-2)</sup>。
- **未完成实现**：`SYS_sendmsg`/`SYS_recvmsg` 返回 `-ENOSYS`，`SYS_truncate` 直接返回 0（无实际操作），信号处理（`rt_sigaction`/`rt_sigprocmask`）一律返回 0 且未安装任何信号处理器 <sup>[10](#mod-syscall-ref-10)</sup>。
- **错误码忽略**：`sys_ioctl_generic` 中 `copyout` 的返回值被丢弃，`TIOCGWINSZ` 和 `RTC_RD_TIME` 失败时调用方无法感知 <sup>[12](#mod-syscall-ref-12)</sup>。
- **无调用号上界检查**：分发器对未识别的调用号仅打印日志并返回 `-ENOSYS`，无任何审计或速率限制，可能被用于侧信道探测内核接口 <sup>[10](#mod-syscall-ref-10)</sup>。
