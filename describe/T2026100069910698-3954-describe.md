## 📖 一、总览

该内核是一个面向 RISC-V 与 LoongArch 双架构的自研 Unix-like 操作系统，整体呈现“教学基底 + 准生产兼容层”的混合面貌。其核心基础（内存管理、任务调度、中断处理）承袭了 xv6 的单核与静态数组设计，但在此之上堆叠了 400+ Linux 系统调用号定义、一个只读 ext4 后端与内存文件系统双层结构、以及规划中的 BSD socket 接口；155 个已实现系统调用中，相当部分为返回 -ENOSYS 或仅完成资源表登记的桩实现。因此，该内核的定位并非完整 Linux 兼容型，而更接近“探讨如何以最小单核内核承载较复杂用户空间”的实验型/演示型项目。

最值得评审关注的三个特性：其一，文件系统采用 memfs 为主、ext4 为辅的 copy-on-open 架构——ext4 中的常规文件首次被打开时整体复制到内存文件系统，后续读写完全在 memfs 进行，以工程便捷性换取了可写支持，同时规避了 ext4 写入的复杂度 (见 fs 模块)。其二，系统调用层通过静态资源表（管道、socket 等）与引用计数，在无动态内存分配的条件下管理多种内核对象，这是对教学内核资源模型的一种轻量扩展，但也因此引入了静态上限与并发安全风险 (见 syscall 模块)。其三，尽管提供了双架构启动代码，但几乎所有核心子系统（内存页分配器、调度器、文件系统全局变量）均未考虑多核互斥，形成鲜明“扩展不均”的剖面：兼容接口的广度与基础实现的工程严谨性之间存在明显鸿沟。

## 🎯 二、综合评价

### 整体定位：单核教学内核的“兼容外衣”
该内核的结构清晰显示其“教学内核躯体”+“ Linux 兼容皮肤”的混合策略。核心内存管理采用基于 Sv39 的简单页表，物理页分配器是单链表的伙伴前身，引用计数仅为 COW 服务 [mm.ref:3]；任务调度是完全协作式轮转，全局 `struct Task` 数组无任何锁保护 [task.ref:1][task.ref:3]——这些都与 xv6 一脉相承。然而，在其上构建的系统调用层却定义了逾 400 个 Linux 调用号 [syscall.ref:2]，并通过静态资源表提供了管道、socket、eventfd 等对象的分配机制 [syscall.ref:3]。这种“厚皮薄底”的组合，使它无法像 xv6 那样作为简洁教学范例，又远未达到可运行主流 Linux 用户栈的完成度，本质上是一个研究“如何以最简内核表面支持较复杂应用程序”的实验平台。

### 设计亮点：双层文件系统与资源表管理的折衷创新
在有限基础之上，该内核做出了两个值得关注的原创设计。其一是 memfs + ext4 只读后端 + copy-on-open 的双层文件系统 [fs.ref:1][fs.ref:2]。所有可写操作被收敛到纯内存的 memfs 中，ext4 仅用于启动时预热可执行文件或只读数据 [fs.ref:3]。这一设计绕过了 ext4 写操作所需的日志、块分配等巨大复杂性，却能直接从标准 ext4 磁盘映像引导并修改文件，为教学内核实现“可引导、可写”提供了一条轻量级路径。其二是系统调用资源表的统一管理模式：管道、socket、eventfd 等对象均通过带 `used` 标志和引用计数的固定大小数组管理生命周期 [syscall.ref:3]，在无动态内存分配器的前提下实现了多种资源的复用与共享，虽然伴随静态上限风险，但思路简洁有效。

### 工程取舍：广度优先，基础后补
观察整个代码库，可以明显看出作者优先将精力投放在“用户可见的 API 宽度”而非“内核核心的深度”。系统调用列表从 17 到 435 剧烈扩展 [syscall.ref:2]，甚至包含了 timerfd、inotify、memfd_create 等较新的 Linux 接口，但同时内存管理硬编码物理内存为 256 MiB [mm.ref:19]，启动栈仅 16～64 KiB [boot.ref:3][boot.ref:4]，任务调度无时间片抢占，所有核心数据结构均无并发原语。这种取舍使得内核能够迅速支持一套相对丰富的用户空间程序集（如带 busybox 的 Shell），但在任何稍碰边界的场景（多核、大内存、长时间运行）下都将快速暴露不足。此外，网络模块同样体现了“先占位、后实现”的思路：16 个 socket 系统调用已注册 [net.ref:3]，但底层仅有一个原始包收发接口，没有任何协议栈实现，属于典型的“接口先行”策略。

### 完成度与关键不足
综合判断，该内核目前处于早期实验阶段，核心子系统基本可用但健壮性不足。155 项系统调用中，有相当数量为返回 `-ENOSYS` 的占位实现（如多数 net 族调用），信号处理虽完整但跳板代码硬编码了系统调用号 [task.ref:17]。代码质量方面，全局变量的竞态风险几乎遍及所有模块（文件系统、内存分配、任务管理），且用户地址校验函数 `user_range_ok` 仅声明未实现 [syscall.ref:5]，导致安全边界模糊。驱动层的 VirtIO 代码在三处重复拷贝 [drivers.ref:1][drivers.ref:2][drivers.ref:3]，缺乏公共提炼。尽管存在这些工程缺陷，该内核所展现的“以最小单核内核运行复杂用户空间”的集成思路，以及双层文件系统和资源表管理的具体实践，为同类教学内核的扩展提供了有参考价值的案例。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程是内核的汇编入口, 由架构相关的 `boot.S` 实现 (RISC-V64 与 LoongArch64 各一份). 核心职责: (1) 设置启动栈指针; (2) 清零 BSS 段; (3) 跳转 C 入口 `kernel_main`. 整体采用经典的「汇编引导→C 主函数」模式, 与 xv6-riscv 类似但更精简——省略了多核启动、特权级切换及固件参数保存, 且栈空间极小 (16 KiB/64 KiB).

### 1. 启动序列与核心职责

操作系统的启动是「从硬件到 C 运行时的桥梁」。两个 `boot.S` 各自承担该职责：处理器上电（或由前级 bootloader 移交）后从 `_start` 开始执行，此时没有栈、BSS 段是脏数据。启动代码必须顺序完成三个步骤。

**第一步：设置栈指针**。RISC-V 版通过 `la sp, boot_stack_top` 将栈顶载入 `sp` <sup>[2](#mod-boot-ref-2)</sup>，LoongArch64 版用 `la.local $sp, boot_stack_top` <sup>[1](#mod-boot-ref-1)</sup>。栈位于 `.bss.stack` 段，由 loader 在加载时分配，不占用 ELF 镜像空间。RISC-V 栈仅 16 KiB，LoongArch64 栈 64 KiB，两者差异显著 <sup>[3](#mod-boot-ref-3)</sup><sup>[4](#mod-boot-ref-4)</sup>。

**第二步：清零 BSS**。这是 C 语言运行时要求的——未初始化的全局变量必须为 0。RISC-V 采用逐字节清零 (`sb zero`)，步长为 1 <sup>[2](#mod-boot-ref-2)</sup>；LoongArch64 以 8 字节为单位 (`st.d $zero`)，步长为 8 <sup>[1](#mod-boot-ref-1)</sup>。前者对任何对齐均安全，后者更高效但要求 BSS 8 字节对齐（链接脚本应保证）。两者都循环比较 `__bss_start` 与 `__bss_end`，这两个符号由链接脚本导出。

**第三步：转入 C 代码**。RISC-V 用 `call kernel_main` <sup>[2](#mod-boot-ref-2)</sup>，LoongArch64 用 `bl la_kernel_main` <sup>[1](#mod-boot-ref-1)</sup>。C 函数理论上不应返回；若返回，则进入 `halt` 循环——RISC-V 执行 `wfi` 等待中断，LoongArch64 执行 `idle 0` 进入低功耗状态 <sup>[5](#mod-boot-ref-5)</sup><sup>[6](#mod-boot-ref-6)</sup>。

### 2. 关键设计取舍

**栈放在 BSS 段**：将 `boot_stack` 声明在 `.bss.stack` 段（`aw` 属性 + `@nobits`）使得栈内存在 ELF 中不占实际空间，节约镜像体积。代价是依赖 loader 正确解释 BSS 并分配物理内存。这比放在 `.data` 段或动态分配更简洁。

**无多核支持**：两个实现均只声明单个栈，未读取 hart ID / CPU ID 来分流。所有核心到达 `_start` 后将共享同一栈顶，引发灾难性竞态。在 RISC-V 平台上，OpenSBI 默认唤醒所有 hart，因此这是一个 **严重的实际缺陷**——除非运行环境的固件已将其他 hart 置于睡眠或该内核刻意设计为单核。

**不保存固件上下文**：RISC-V SBI 在 `a1` 中传递设备树指针 (DTB)，LoongArch 固件也有类似约定。此处直接丢弃这些寄存器，意味着 `kernel_main` 必须通过其他手段（硬编码、独立探测）获取硬件信息，可移植性受损。

**无早期陷阱向量**：跳转 C 前未设置 `stvec`（RISC-V）或类似 CSR。若在 `kernel_main` 安装 handler 之前发生中断，CPU 将跳转到未定义地址。这依赖于固件启动时保持中断禁用，但若存在 NMI 或固件未屏蔽中断，后果不可预料。

**调用距离限制**：RISC-V 的 `call` 伪指令有 ±2 GiB 范围，通常足够。LoongArch64 的 `bl` 仅 ±128 MiB；若 `la_kernel_main` 链接较远，链接器需插入跳板。好在二者都是局部符号，链接器可见全局布局。

**与 xv6-riscv 对比**：原版 xv6 的 `entry.S` 会为每个 hart 分配独立栈、设置机器模式、写入 `mstatus` 和 `mepc` 并 `mret` 到 S 模式。本实现完全省略了这些步骤，假设进入 `_start` 时已在期望的特权级（很可能是 S 模式），大幅简化但牺牲了通用性。

### 3. 跨模块协同

启动代码虽短，却与链接脚本及 C 初始化代码紧密耦合：

- **链接脚本**提供 `__bss_start`、`__bss_end`、`boot_stack_top` 及栈大小定义。若脚本未正确定义这些符号，链接阶段即失败。
- **`kernel_main` / `la_kernel_main`** 是 C 入口，负责后续所有初始化：页表、陷阱向量、内存管理、进程调度等。该函数必须永不返回。
- **`trap.S`** 通常包含陷阱向量表，`kernel_main` 会在初始化时将其地址写入 `stvec`。若此步骤遗漏，任何中断均导致崩溃。
- **`user.S`** 暗示存在用户态进程支持，启动代码建立的初始内核栈是后续上下文切换的根基。

由于无法读取链接脚本及 C 源码，以上协同基于 Unix-like 内核惯例推断，实际代码可能有所出入。

### 4. 边角细节与不足

以下是从汇编代码中识别出的具体风险点与工程缺陷：

1. **多核竞争** `<sup>[2](#mod-boot-ref-2)</sup><sup>[1](#mod-boot-ref-1)</sup>`：两版本均未读取 hart/CPU ID，多个核心并发执行 `_start` 时共用同一栈，导致不可预测的栈覆写。这是单核假设的直接表现。
2. **栈溢出无保护** `<sup>[3](#mod-boot-ref-3)</sup><sup>[4](#mod-boot-ref-4)</sup>`：栈大小仅 16 KiB (RISC-V) / 64 KiB (LoongArch64)，且前后无 guard page。`kernel_main` 或后续初始化若栈使用过量，将静默覆写相邻数据。
3. **未保存固件参数** `<sup>[2](#mod-boot-ref-2)</sup><sup>[1](#mod-boot-ref-1)</sup>`：两个版本均在跳转前丢弃所有通用寄存器，导致 DTB 地址等关键信息丢失。内核可能要硬编码外设地址。
4. **RISC-V BSS 清零效率低** `<sup>[2](#mod-boot-ref-2)</sup>`：逐字节 `sb` 方式，对于数十 KiB 的 BSS 会产生大量 store 指令，拖慢启动。应采用 8 字节对齐 + 尾处理。
5. **缺少早期调试输出**：启动全程没有任何向串口写字符的操作，若在清 BSS 或跳转阶段卡死，完全无法定位。这是调试友好性的明显不足。
6. **符号命名不一致**：RISC-V 调用 `kernel_main`，LoongArch64 调用 `la_kernel_main`。两者若非同一函数的条件编译别名，则可能指向不同实现，增加维护开销。
7. **硬编码栈大小** `<sup>[3](#mod-boot-ref-3)</sup><sup>[4](#mod-boot-ref-4)</sup>`：栈大小以立即数写入 `.space` 指令（16384 / 65536），未使用宏或头文件常量。调整时需改动汇编源码，易遗漏。
8. **`wfi` 后无内存屏障**：RISC-V `halt` 循环仅 `wfi` + `j halt`，未使用 `fence`。若唤醒后需保证之前的 store 可见，缺乏屏障可能导致调试困难。

### ⚠ 开放问题

- 无多核启动支持：所有 hart/核心共享同一 boot_stack，并发进入 _start 将导致栈冲突。

- 栈大小硬编码且无 guard page，16 KiB/64 KiB 对于复杂内核可能不足，溢出无检测机制。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块实现基于 Sv39 的虚拟内存管理，包含伙伴系统前身的单链表物理页分配器（带引用计数）、页表遍历/映射/解映射、写时复制（COW） fork、共享内存（PTE_SHM）及用户地址合法性校验。核心数据结构为 PageInfo（每物理页的 ref/next/on_free）与 pagetable_t（三级页表根）。与 xv6-riscv 相比，增加了 vm_protect 页级权限修改、COW 页引用计数（允许多进程共享只读后按需复制）以及内联 COW 解决的 vm_user_range_ok 校验路径。整个模块面向单核设计，未实现多核 TLB 击落，物理内存大小硬编码为 256 MiB。

### 1. 核心抽象：物理页分配器与 Sv39 页表

物理内存管理采用定长数组 `page_infos[VM_PAGE_COUNT]` 记录每个物理页元数据 <sup>[1](#mod-mm-ref-1)</sup>。空闲页通过单向链表（`free_page_head` + `PageInfo.next`）串联，分配与释放分别由 `page_list_pop`/`page_list_push` 操作 <sup>[2](#mod-mm-ref-2)</sup>。每个物理页维护 16 位引用计数 `ref`，用于支持 COW；`vm_incref_page`/`vm_decref_page` 在页表映射用户页时增减计数，计数归零时回收至空闲链表 <sup>[3](#mod-mm-ref-3)</sup>。

分配器存在“早期”阶段：在 `page_allocator_ready` 置位前，`vm_alloc_page` 使用线性起泡指针 `early_free` 从内核结束地址向上分配，不纳入引用计数管理 <sup>[4](#mod-mm-ref-4)</sup>；`vm_init` 调用 `vm_page_allocator_init` 后，已使用的早期页被标记为 ref=1，后续分配切换至空闲链表 <sup>[5](#mod-mm-ref-5)</sup>。这种两阶段设计使得内核在未初始化元数据数组时即可获得页面，代价是早期页无法复用且缺乏统计。

虚拟地址空间基于 Sv39 三级页表。`vm_walk` 按软件遍历 L2→L1→L0，支持按需分配中间页表（alloc=1） <sup>[6](#mod-mm-ref-6)</sup>。`vm_map` 逐页调用 `vm_walk` 插入最终 PTE，同时设置 A/D 位并可选增加引用计数 <sup>[7](#mod-mm-ref-7)</sup>。`vm_unmap` 清除 PTE 并递减引用计数后执行 `sfence_vma` <sup>[8](#mod-mm-ref-8)</sup>。内核页表采用直接映射（identify mapping）覆盖全部物理 RAM、UART 与 VIRTIO MMIO 区域 <sup>[9](#mod-mm-ref-9)</sup>，并置位 SUM 比特允许内核访问用户地址 <sup>[10](#mod-mm-ref-10)</sup>。

### 2. 关键设计取舍：COW 与共享内存

fork 时通过 `vm_clone_user_pagetable` 浅拷贝父进程用户地址空间：`vm_clone_user_leaf` 将可写页的 PTE 权限去 W 并置 COW 标志（位 8），同时保留原始物理页，父子 PTE 均指向同一物理页 <sup>[11](#mod-mm-ref-11)</sup>。如果页面被标记为 `PTE_SHM`（共享内存），则保留可写权限且不置 COW，父子进程直接共享 <sup>[12](#mod-mm-ref-12)</sup>。

COW 缺页由 `vm_handle_cow_fault` 处理：若物理页引用计数 ≤1（无其他进程共享），直接清除 COW 并添加 W 权限，避免不必要拷贝 <sup>[13](#mod-mm-ref-13)</sup>；否则分配新页、memcpy 旧内容、更新 PTE 指向新页、递减旧页引用并递增新页引用 <sup>[14](#mod-mm-ref-14)</sup>。这种“快路径”减少了 fork 后立即 exec 的开销，与 Linux 内核的思路一致。

`vm_user_range_ok` 在校验用户地址可写性时会主动调用 `vm_handle_cow_fault` 解决 COW <sup>[15](#mod-mm-ref-15)</sup>。这意味着 `copyin`/`copyout` 路径不必单独处理 COW——但副作用是校验函数会修改页表，若后续操作失败可能留下已拆分的页面。这是性能与简单性的折中：避免了在每条系统调用入口进行 COW 处理的代码重复。

`vm_protect` 提供页级权限修改（类似 mprotect），会保留 COW 状态——如果原始 PTE 有 COW，则新权限仍去 W 并置 COW <sup>[16](#mod-mm-ref-16)</sup>。但该实现不检查权限变更是否与 COW 状态兼容：例如将只读页改为可写却仍保留 COW，会导致后续写触发 COW 缺页，未必符合调用者预期。

### 3. 跨模块协同：缺页处理与系统调用

虽然未直接读取 trap 处理代码，但从 `vm_handle_cow_fault` 和 `vm_user_range_ok` 的接口可推断协作模式：

- **缺页中断**：硬件在用户态写 COW 页时产生 Store/Page Fault，trap handler 应当调用 `vm_handle_cow_fault`（传入当前进程页表与故障地址），成功则返回重试 <sup>[13](#mod-mm-ref-13)</sup><sup>[14](#mod-mm-ref-14)</sup>。
- **系统调用地址校验**：`sys_mmap`/`sys_munmap` 等使用 `vm_user_range_ok` 验证用户传入的地址范围是否已映射且权限匹配，同时触发 COW 解决 <sup>[15](#mod-mm-ref-15)</sup>。
- **fork 实现**：`sys_fork` 调用 `vm_clone_user_pagetable` 创建子进程页表，COW 浅拷贝 <sup>[11](#mod-mm-ref-11)</sup>。
- **页表销毁**：`vm_destroy_user_pagetable` 递归释放用户页并递减引用计数，避免物理页泄漏 <sup>[17](#mod-mm-ref-17)</sup>。

内核在 `vm_init` 中映射外设寄存器（UART/VIRTIO）并启用分页 <sup>[9](#mod-mm-ref-9)</sup>；`vm_create_user_pagetable` 为新进程创建页表时同样映射这些设备，保证内核在用户态陷入后能直接访问设备 <sup>[18](#mod-mm-ref-18)</sup>。该做法简化了设备驱动，因为内核在任何上下文下均使用同一物理地址访问 MMIO。

### 4. 边角细节与不足

1. **多核 TLB 一致性问题**：<sup>[8](#mod-mm-ref-8)</sup><sup>[13](#mod-mm-ref-13)</sup> 中 `sfence_vma` 仅刷新当前 hart 的 TLB，缺少 IPI 机制通知其他核心。若内核运行在多核环境，其他核心可能持有陈旧的 COW 页表项缓存，导致写操作绕过 COW 拷贝。
2. **整数溢出风险**：`vm_map` 未检查 `va + size` 回绕，仅判断 `size == 0` <sup>[7](#mod-mm-ref-7)</sup>；对比 `vm_protect` 与 `vm_user_range_ok` 明确检查了 `va + size < va` <sup>[16](#mod-mm-ref-16)</sup><sup>[15](#mod-mm-ref-15)</sup>。攻击者可能通过超大 size 参数绕过校验，映射越界区域。
3. **部分映射失败无回滚**：`vm_map` 在逐页映射中途失败（如 `vm_walk` 返回 NULL）直接返回 -ENOMEM，已映射的页面未被解除 <sup>[7](#mod-mm-ref-7)</sup>。这可能导致页表残存半映射，后续 `vm_unmap` 无法清理干净。
4. **早期分配器无引用计数**：<sup>[4](#mod-mm-ref-4)</sup> 中早期页面直接返回且不记录元数据；`vm_page_allocator_init` 将已用页面 ref 置 1 <sup>[5](#mod-mm-ref-5)</sup>，但若早期页面被误传入 `vm_free_page`，将触发 `pa_is_managed` 检查而跳过释放，导致泄漏且无警告。
5. **物理内存硬编码**：`memlayout.h` 中 `PHYS_MEM_SIZE` 硬编码为 0x10000000（256 MiB）<sup>[19](#mod-mm-ref-19)</sup>。若实际 RAM 超出该值，多余内存不可用；若不足，`vm_page_allocator_init` 会 panic。缺少设备树或 ATAGS 探测。
6. **`vm_protect` 权限语义模糊**：当修改包含 COW 的页时，新 PTE 强制去 W 并置 COW <sup>[16](#mod-mm-ref-16)</sup>，但未检查 `perm` 参数是否合法；调用者若请求 RW 却得到只读页可能引发难以调试的错误。
7. **共享内存页在 COW 缺页中未特殊处理**：`vm_handle_cow_fault` 未检查 `PTE_SHM` <sup>[13](#mod-mm-ref-13)</sup>，但 SHM 页在 fork 时不被标记为 COW <sup>[12](#mod-mm-ref-12)</sup>，因此 COW 缺页不会发生在共享页上；然而若未来扩展导致共享页被错误置 COW，缺页处理将按普通 COW 拷贝，破坏共享语义。
8. **竞争条件**：物理页分配器（空闲链表操作）与引用计数增减均无锁保护 <sup>[2](#mod-mm-ref-2)</sup><sup>[3](#mod-mm-ref-3)</sup>。单核下安全，但若内核未来支持多核，需添加自旋锁。
9. **错误码忽略**：`vm_create_user_pagetable` 调用 `vm_map` 映射 UART/VIRTIO 时仅检查返回值，失败即返回 NULL <sup>[18](#mod-mm-ref-18)</sup>；但调用方可能未检查 NULL，导致后续页表操作崩溃。
10. **已分配页未初始化 CoW 标志**：`vm_alloc_page` 通过 `memset(page, 0, PAGE_SIZE)` 清零 <sup>[4](#mod-mm-ref-4)</sup>，因此新页的 PTE 不会残留 COW 标志；但若某条路径未清零（如未来改用池化），可能导致虚假 COW 缺页。

### ⚠ 开放问题

- vm_map 未检查 va+size 回绕，可能导致映射越界 [ref:7]。
- 多核下缺少 TLB shootdown，其他核心可能使用陈旧 COW 缓存 [ref:8][ref:13]。
- vm_map 部分映射失败无回滚，页表可能残存不完整映射 [ref:7]。
- 物理内存硬编码 256 MiB，不支持探测实际 RAM 大小 [ref:19]。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块以静态数组 `struct Task tasks[TASK_MAX]` 为任务表, 每个任务由 `struct Task` 描述, 包含 pid/tgid/ppid、页表、陷入帧、信号处理表、文件描述符、futex/管道/socket 等待状态等. 调度采用协作式轮转: 所有调度点集中在 `task_on_trap_return`, 由 `need_resched` 标记驱动, 无时间片抢占. 进程创建通过 `task_clone_current` 支持 clone/flags (CLONE_VM/CLONE_THREAD/CLONE_FILES 等), 实现了线程组与共享页表. 信号机制完整: 支持 64 种信号, 含 rt_sigframe 跳板、可选 sigaltstack、信号掩码与 SA_RESTORER. 整体是单核设计, 无锁, 全局状态直接读写.

### 1. 核心抽象: Task 结构体与全局任务表

整个进程模型围绕 `struct Task` 构建 <sup>[1](#mod-task-ref-1)</sup>. 该结构体将 pid、tgid、ppid 组织成进程树, 并内联了页表指针 `pagetable`、陷入帧 `tf`、信号处理数组 `signal_handler[65]`、文件描述符数组 `fds[128]` 以及用于 futex/管道/socket 等待的辅助字段 <sup>[2](#mod-task-ref-2)</sup>. 所有任务存放在全局静态数组 `tasks[TASK_MAX]` (最大 256) 中, 当前任务由 `current` 指针指向 <sup>[3](#mod-task-ref-3)</sup>. 

这种“大结构体 + 静态数组”的设计与 xv6 一脉相承, 优点是简单、无动态分配开销, 缺点是伸缩性受编译期常量限制. 特别的是, 文件描述符表的归属通过 `files_id` 字段而非直接共享指针实现: 当 `CLONE_FILES` 时子进程继承父进程的 `files_id`, 从而共享同一组 fd 而无需引用计数 <sup>[4](#mod-task-ref-4)</sup>. 这种方案简洁, 但在 `task_finish` 中必须遍历整个任务表来判断文件描述符是否仍被使用 <sup>[5](#mod-task-ref-5)</sup>, 时间复杂度 O(n).

### 2. 调度机制: 协作式轮转与定时唤醒

调度算法实现在 `task_pick_next` 中 <sup>[6](#mod-task-ref-6)</sup>. 它从 `current` 位置开始线性扫描, 寻找第一个 `TASK_RUNNABLE` 任务, 形成一个简单轮转. 每次扫描前会检查所有 `TASK_SLEEPING` 和 `TASK_FUTEX` 任务, 若其 `wake_time` 已到期则将其唤醒 <sup>[7](#mod-task-ref-7)</sup>. 此外 `task_next_deadline` 计算最近到期时间, 供 idle 循环使用.

关键的调度入口是 `task_on_trap_return` <sup>[8](#mod-task-ref-8)</sup>, 它在每次陷入返回前被调用. 该函数先尝试向当前任务递送信号, 然后保存陷入帧; 若 `need_resched` 被置位或任务状态不是 `TASK_RUNNING`, 则触发调度. 为防止同一线程组内频繁互相抢占, 引入了 `skip_preempt_tgid` 临时跳过同 tgid 的其他任务 <sup>[9](#mod-task-ref-9)</sup>. 当没有任何可运行任务时, 内核进入忙等/`wfi` 循环直到超时或有新任务就绪.

这个协作式调度放弃了时间片抢占, 换取实现的极度简化: 内核无需维护定时器中断, 上下文切换仅发生在系统调用/异常返回处. 代价是恶意或死循环用户进程会永久占用 CPU. 此外, `need_resched` 的置位与检查之间没有原子性保证, 若中断处理程序在检查与调度之间修改了状态, 可能丢失一次重新调度机会——但在当前单核、中断关闭的假设下尚可工作.

### 3. 进程创建: clone 系统调用与线程组

`task_clone_current` 是该模块的核心创建函数 <sup>[10](#mod-task-ref-10)</sup>. 它解析陷入帧 `a0` 中的 clone flags, 支持 `CLONE_VM` (共享页表)、`CLONE_THREAD` (共享 tgid)、`CLONE_FILES` (共享文件描述符表) 等. 当 `CLONE_VM` 置位时, 子进程直接继承父进程页表并将 `shared_vm` 标记为 1, 同时递增页表引用计数 <sup>[11](#mod-task-ref-11)</sup>; 否则通过 `vm_clone_user_pagetable` 复制一份独立的用户页表.

线程组的语义通过 `tgid` 实现: 线程的 `tgid` 等于主线程的 pid, 而 `ppid` 也继承自主线程 <sup>[12](#mod-task-ref-12)</sup>. 这与 Linux 的线程模型一致. 文件描述符的共享通过 `files_id` 实现: 若 `CLONE_FILES` 置位, 子进程直接继承父进程的 `files_id`, 此后所有共享该 id 的任务看到同一组 fd <sup>[4](#mod-task-ref-4)</sup>. `CLONE_CHILD_CLEARTID` 机制也得到支持: 子进程退出时会向用户态地址写入 0 并唤醒等待该地址的 futex <sup>[13](#mod-task-ref-13)</sup>, 这是实现 pthread_join 的关键.

### 4. 信号处理: 从排队到用户态跳板的完整路径

信号系统实现了 POSIX 实时信号的子集. 信号排队通过 `task_queue_signal` 将信号位写入 `signal_pending_mask` 并置 `signal_pending` 标志 <sup>[14](#mod-task-ref-14)</sup>. 若目标任务处于可中断睡眠 (TASK_SLEEPING/BLOCKED/FUTEX/PIPE/SOCKET), `task_interrupt_for_signal` 将其唤醒, 修改 `sepc` 并设置返回值为 `-EINTR` <sup>[15](#mod-task-ref-15)</sup>.

实际递送发生在 `task_on_trap_return` 中调用的 `task_deliver_signal` <sup>[16](#mod-task-ref-16)</sup>. 它从 `signal_pending_mask & ~signal_mask` 中选出第一个未屏蔽信号, 若用户注册了处理函数 (非 SIG_DFL/SIG_IGN), 则在用户栈上构造 `UserRtSigFrame` 结构体, 包含保存的寄存器上下文、信号信息和一个两指令的跳板 `[0x08b00893, 0x00000073]` (即 `li a7, 139; ecall`, 用于 rt_sigreturn) <sup>[17](#mod-task-ref-17)</sup>. 跳板地址被写入 `tf->ra`, 这样信号处理函数返回时自动执行 rt_sigreturn, 恢复寄存器并解除信号掩码.

值得注意的设计: 交替信号栈 (`SA_ONSTACK`) 的支持通过检查 `sigaltstack` 状态实现, 并在栈上预留帧 <sup>[18](#mod-task-ref-18)</sup>; 但跳板代码被硬编码为两条 RISC-V 指令, 假设 `rt_sigreturn` 的系统调用号为 139 <sup>[17](#mod-task-ref-17)</sup>, 一旦调整系统调用表编号就会失效.

### 5. 边角细节与不足

1. **无并发保护** <sup>[3](#mod-task-ref-3)</sup><sup>[6](#mod-task-ref-6)</sup>: 全局 `tasks[]`、`current`、`need_resched`、`next_pid` 均无锁保护. 代码假定单核执行且中断在关键路径关闭; 若未来扩展 SMP 或允许中断处理程序触碰这些变量, 将出现竞态.
2. **用户态指针校验不足** <sup>[19](#mod-task-ref-19)</sup>: `task_copy_to_user` 和 `task_copy_from_user` 通过 `vm_translate` 获取物理地址后直接解引用, 未验证物理地址是否在合法 RAM 范围. 若页表被破坏, 可能写入任意物理内存.
3. **跳板指令硬编码** <sup>[17](#mod-task-ref-17)</sup>: 信号跳板的两条指令硬编码为特定系统调用号 139. 若系统调用表重新编号, 信号返回将静默失败, 导致进程崩溃.
4. **静态限制无错误恢复** <sup>[20](#mod-task-ref-20)</sup>: `task_alloc_slot` 在 `TASK_MAX` 满时返回 NULL, 调用方 `task_clone_current` 返回 `-ENOMEM`, 但用户态可能无法区分是临时资源不足还是系统硬限制, 缺乏动态扩展或负载反馈.
5. **协作式调度无时间片** <sup>[8](#mod-task-ref-8)</sup>: 调度仅在陷入返回时发生, 无抢占定时器. 一个死循环的用户进程会永久占用 CPU, 无看门狗或 NMI 兜底.
6. **共享页表退出时可能泄漏** <sup>[21](#mod-task-ref-21)</sup>: 当 `shared_vm` 任务退出时, `task_finish` 仅对页表做 `vm_decref_page`, 并未销毁. 若引用计数逻辑有误或所有线程退出后计数未归零, 页表内存将泄漏.
7. **空闲循环忙等** <sup>[22](#mod-task-ref-22)</sup>: `task_on_trap_return` 的空闲分支使用 `do { rdtime; } while (now < deadline)` 忙等, 虽然后续有 `wfi`, 但若 deadline 较远, 忙等浪费功耗且可能因编译器优化导致死循环 (缺少 `volatile` 或 barrier).

### ⚠ 开放问题

- 全局任务状态完全无锁，扩展 SMP 或允许中断处理程序触碰 task 状态将导致竞态 [ref:3].
- 信号跳板指令硬编码系统调用号 139，若重排系统调用表则信号返回崩溃 [ref:17].
- task_copy_to_user 通过物理地址直接写入，未校验物理地址合法性，页表损坏可导致任意内存写入 [ref:19].
- 共享页表退出时仅减引用计数而不销毁页表，若引用计数逻辑有误则页表内存泄漏 [ref:21].

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块实现了一个双层文件系统: 上层是内存文件系统 memfs (基于定长 `struct MemFile` 数组与线性内存池), 下层是只读 ext4 驱动 (通过 `block_read_sector` 访问块设备). 核心机制是 copy-on-open: ext4 中的常规文件在首次被打开时复制到 memfs, 此后所有读写均在 memfs 中进行. 目录结构通过扁平命名空间 + 字符串前缀匹配模拟, 避免了树状索引的复杂度. 文件描述符绑定到 `Task.fds[]` 数组, ext4 inode 与 memfs file_id 通过高位标志位统一编址. 相比 xv6 原版, 该模块引入了 ext4 只读支持、文件缓存预热、busybox 别名等实用扩展, 但缺乏并发保护与写回能力.

### 1. 双栈架构: memfs 为主、ext4 为辅的只读后端

该文件系统采用"内存优先、ext4 只读后端"的双层设计. memfs 是所有可写操作的实际载体, 其数据存储在一个 64 MiB 的静态字符数组 `memfs_pool` 中, 元数据由最多 2048 个 `struct MemFile` 槽位管理 <sup>[1](#mod-fs-ref-1)</sup>. ext4 仅作为可执行文件、脚本等静态内容的初始来源, 在 `fs_open()` 路径中, 若 memfs 未命中且 ext4 中存在该文件, 则调用 `memfs_copy_from_ext4()` 将其内容整体拉入 memfs <sup>[2](#mod-fs-ref-2)</sup>. 此后应用层的 `read`/`write` 均直接操作 memfs, ext4 完全保持只读. 这一取舍避免了在 ext4 上实现写操作 (需处理日志、位图、块分配等) 的巨大工程复杂度, 同时保留了从标准 ext4 镜像引导的能力.

`fs_init()` 启动时预先创建 `.`、`mnt`、`tmp`、`proc`、`dev` 等 POSIX 惯用目录, 并填充 `proc/mounts`、`dev/zero`、`dev/null` 等伪文件 <sup>[3](#mod-fs-ref-3)</sup>. 随后 `preload_ext4_root_files()` 批量预热: 遍历 ext4 根目录, 将常规文件读入 memfs, 并为 busybox 创建 `ls`、`sh`、`cp` 等别名——这些别名通过 `memfs_alias_file()` 实现, 它们共享同一块 `data` 指针, 零额外内存开销 <sup>[4](#mod-fs-ref-4)</sup>.

### 2. 扁平命名空间与目录模拟

memfs 未维护树状 inode/dentry 结构, 而是将所有文件以规范化绝对路径为键存入扁平数组. 目录枚举通过遍历全数组实现: `memfs_getdents_dir()` 对每个 `memfs[i]` 调用 `path_parent_eq()` 判断其是否为目标目录的直接子节点 <sup>[5](#mod-fs-ref-5)</sup>. `path_parent_eq()` 的核心逻辑是: 若父路径为 `.`, 则子路径不含 `/` 即为直接子; 否则要求子路径以 `parent/` 为前缀且剩余部分不含 `/` <sup>[6](#mod-fs-ref-6)</sup>. 这是一种以 O(n) 时间换 O(1) 空间的典型教学内核取舍——在几十个文件的规模下完全可接受.

路径规范化由 `normalize_path()` 完成, 它处理 `.`、`..` 以及连续斜杠的消解 <sup>[7](#mod-fs-ref-7)</sup>. 其中 `..` 的回溯逻辑在 `while (oi > 0 && out[oi - 1] == '/')` 处可能过度清理尾部斜杠, 且未对符号链接做任何处理 (该文件系统不支持符号链接). `fs_resolve_task_path()` 在 `normalize_path` 之前将相对路径与任务当前工作目录 (`task->cwd`) 拼接 <sup>[8](#mod-fs-ref-8)</sup>.

### 3. 跨模块协同: 块设备 → ext4 → memfs → 用户态

数据从磁盘到用户缓冲区的完整路径涉及三个模块:

- **块设备层**: ext4 驱动通过 `block_read_sector()` 读取物理扇区. `read_bytes()` 封装了跨扇区读取, 处理非对齐偏移 <sup>[9](#mod-fs-ref-9)</sup>. `read_block()` 将块号转换为扇区号后调用 `read_sectors()` <sup>[10](#mod-fs-ref-10)</sup>.
- **ext4 驱动**: `ext4_init()` 解析超级块获取几何参数 (block_size, inode_size, desc_size 等), 验证魔数 `0xef53`, 然后预热文件名缓存和文件内容缓存 <sup>[11](#mod-fs-ref-11)</sup>. 路径解析 `ext4_lookup_path()` 从根 inode (2) 出发逐级 `lookup_child()`, 后者遍历目录块的 ext4 目录项结构并维护一个 64 条目的名称缓存 `name_cache[]` 加速重复查找 <sup>[12](#mod-fs-ref-12)</sup>. 文件数据读取支持 extent 树和间接块两种模式 <sup>[13](#mod-fs-ref-13)</sup>.
- **文件缓存**: `ext4_read_all()` 在真正读盘前先查询 `file_cache[]` (最多 8 个条目, 每个 2 MiB) <sup>[14](#mod-fs-ref-14)</sup>. 若命中则直接返回缓存中的指针, 实现零拷贝. `ext4_init()` 末尾预热根目录下所有常规文件, 使得后续 exec 加载 busybox 等可执行文件无需磁盘 I/O.

### 4. 边角细节与不足

**[并发安全缺失]** 整个文件系统模块未使用任何自旋锁或互斥锁. `memfs` 全局数组、`memfs_pool` 分配器、`name_cache` 和 `file_cache` 均直接暴露于并发访问之下 <sup>[15](#mod-fs-ref-15)</sup>. 在 SMP 或多任务抢占内核中, 两个任务同时 `fs_open()` 可能竞态修改 `memfs_pool_used` 导致分配器状态错乱. 若该系统确为单核协作式调度则无问题, 但代码中无任何注释声明此假设. 此外, 内核若在持有自旋锁时关闭中断, 此处未体现——意味着文件系统操作不可在中断上下文中安全调用.

**[ext4 全局缓冲区不可重入]** `ext4_read_all()` 回退路径将超大文件 (超过 2 MiB 缓存阈值) 读入静态全局数组 `ext4_file_buf` (8 MiB) 并返回其地址 <sup>[16](#mod-fs-ref-16)</sup>. 并发调用会互相覆盖数据. `extent_lookup_rec()` 递归解析 extent 树时同样使用静态 `child[4096]` 缓冲区 <sup>[17](#mod-fs-ref-17)</sup>, 递归路径不可重入.

**[内存池压缩的别名一致性问题]** `memfs_compact_pool()` 通过先检测是否已有其他条目指向同一旧地址来避免重复移动 <sup>[18](#mod-fs-ref-18)</sup>, 但该检测依赖于 `memfs[j].data == old_data` 的指针相等性. 若别名条目恰好在 `i` 之前已被更新为 `new_data` (因其被先遍历到), 则后续别名条目将被错误地认为"已移动过"而跳过更新, 导致其 `data` 仍指向已废弃的旧地址.

**[边界检查与指针安全]** `memfs_fill_file()` 和 `memfs_alias_file()` 调用 `memfs_get(memfs_create_raw(...))`, 若 `memfs_create_raw` 返回 -1 则 `memfs_get` 直接返回 NULL, 后续解引用 `node->data` 将导致空指针访问 <sup>[19](#mod-fs-ref-19)</sup>. `path_leaf()` 返回指向输入字符串内部的裸指针, 调用者须自行保证输入生命周期长于返回值——在 `fs_stat_path()` 中使用 `path_leaf(path)` 时 `path` 是栈上 `resolved` 缓冲区的别名, 安全但依赖调用约定 <sup>[20](#mod-fs-ref-20)</sup>.

**[TODO / 早期实现痕迹]** `fs_write_file()` 对 ext4 id 直接返回 `-EBADF`, 意味着 ext4 文件经 copy-up 后虽可读, 但任何试图通过 ext4 inode 直接写入的操作均被拒绝 <sup>[21](#mod-fs-ref-21)</sup>. 目录项的 `open_refs` 引用计数仅在 memfs 侧维护, ext4 目录打开后关闭时 `fs_close_ref_id` 虽有范围检查但不会触发 ext4 侧资源释放 <sup>[22](#mod-fs-ref-22)</sup>. `file_cache_insert` 在缓存满时简单覆盖 `file_cache_next` 而非真正 LRU, 频繁访问的文件可能反复被驱逐 <sup>[23](#mod-fs-ref-23)</sup>.

**[整数溢出与截断]** `memfs_ensure_capacity` 中容量倍增 `new_capacity *= 2` 受 `FS_DATA_SIZE / 2` 约束, 实际风险低 <sup>[24](#mod-fs-ref-24)</sup>. `read_bytes` 中 `offset + done` 为 `u64` 加法, 若镜像极大理论上可回绕, 但实用中不会触发. `tmpfile_seq` 为全局递增 `int`, 创建超过 2^31 个临时文件后会溢出回负, 影响文件名生成 <sup>[25](#mod-fs-ref-25)</sup>.

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块目前仅暴露了一个极简的原始数据包收发接口 (net_send_packet / net_recv_packet) 和一个硬编码的 MTU 常量, 定位为底层链路层抽象. 上层的 socket 系统调用 (socket/bind/listen/accept 等 16 个) 已在 syscall 表中注册, 但实现侧未见对应的 socket 层数据结构或协议栈. 与 xv6 原版不具备网络能力相比, 本内核至少规划了完整的 BSD socket 接口, 但截至当前分析仅能确认底层收发的桩定义, 整体处于早期阶段.

### 1. 核心抽象：原始数据包接口

网络模块的唯一公开接口定义于 `src/include/net.h`<sup>[1](#mod-net-ref-1)</sup>. 它提供四个操作: `net_init` 完成设备/协议栈的初始化; `net_available` 轮询是否有待接收数据; `net_send_packet` 和 `net_recv_packet` 分别完成原始数据包的发送与接收. 这一设计放弃了任何分层抽象 (无 sk_buff 那样的元数据容器, 无协议族/地址族概念), 直接暴露 `const void *data` + `usize len` 的裸指针缓冲区. 相比 Linux 的 `dev_queue_xmit` 或 lwIP 的 `netif->output`, 这里没有链路层地址解析 (ARP) 的介入点, 也没有 DMA 描述符或分散/聚集 I/O 的支持, 是典型的裸机轮询式网卡驱动的朴素封装<sup>[1](#mod-net-ref-1)</sup>. 硬编码常量 `NET_MTU 1514UL` 暗示假设以太网链路, 但未提供 MTU 协商或 Jumbo Frame 的扩展可能<sup>[2](#mod-net-ref-2)</sup>.

### 2. 与 socket 系统调用的断层

虽然 `src/include/syscall.h` 中注册了 16 个 socket 相关系统调用 (socket 到 accept4)<sup>[3](#mod-net-ref-3)</sup>, 但当前可见的头文件中完全没有对应的 socket 结构体、协议族枚举或地址结构 (如 `struct sockaddr`). 这意味着 socket 层的实现可能存放在其他未暴露的源文件中, 或者尚未完成. 若 socket 调用当前被实现为直接返回 `-ENOSYS` 的占位桩, 则规划与实际交付之间存在显著差距. 反之, 若存在一个独立的 socket 实现 (例如移植 lwIP 或 uIP), 该实现必然需要调用 `net_send_packet` / `net_recv_packet` 来完成实际的数据收发, 但目前未见任何将 socket 缓冲区与底层数据包关联起来的 `sk_buff` 或自定义 `mbuf` 结构, 也未见从网络中断到 socket 接收队列的数据路径描述<sup>[4](#mod-net-ref-4)</sup>.

### 3. 关键设计取舍：同步轮询 vs 中断驱动

`net_available` 返回布尔值的设计强烈暗示当前网络栈采用同步轮询模型 (用户态或内核态线程主动检查是否有数据), 而非基于中断的唤醒/阻塞模型. 这一选择简化了锁和上下文切换的设计, 但代价是实时性差、CPU 占用高, 且无法支持高效的阻塞式 `recvfrom` 系统调用——后者预期在无数据时将调用进程挂起到等待队列, 直至网络中断或软中断触发唤醒. 若后续要实现完整的 POSIX socket 语义, 必然需要在 `net_recv_packet` 之下增加异步通知机制或在内核中引入网卡中断处理线程<sup>[5](#mod-net-ref-5)</sup>.

### 4. 边角细节与不足

<sup>[6](#mod-net-ref-6)</sup> **指针校验缺失**: `net_send_packet` 和 `net_recv_packet` 接受 `void *` 指针, 但未提供任何关于缓冲区对齐、物理地址连续性的说明. 若底层 DMA 引擎要求缓冲区物理连续或对齐到缓存行, 调用者传入栈上或用户空间地址将导致不可恢复的数据损坏或硬件异常.

<sup>[2](#mod-net-ref-2)</sup> **MTU 硬编码与链路层假设**: `NET_MTU 1514UL` 假设以太网 II 帧, 但未区分有效载荷与帧头长度. 若底层驱动期望包含帧校验序列 (FCS) 的 1518 字节帧, 或使用 VLAN 标签时 MTU 应为 1518, 都会导致报文截断.

<sup>[5](#mod-net-ref-5)</sup> **无错误码语义**: 发送与接收函数返回 `int`, 但未定义错误码枚举. 调用者无法区分“缓冲区太小”“链路断开”或“超时”等不同失败模式, 也无法向上层返回符合 POSIX 规范的 `errno`.

<sup>[4](#mod-net-ref-4)</sup> **socket 层与底层解耦不足**: 即使 socket 实现存在于其他文件, 当前接口缺乏描述一个数据包“所属连接”或“源地址”的元数据. `net_recv_packet` 仅返回原始字节流, 多路分解必须由上层通过自行解析以太网/IP/TCP 头完成, 破坏了协议栈分层.

<sup>[1](#mod-net-ref-1)</sup> **初始化顺序与竞态**: `net_init` 和 `net_available` 均在 `net.h` 中声明为全局可见, 但无文档说明调用顺序或是否可重入. 若在多核环境下某 CPU 在 `net_init` 完成前调用 `net_recv_packet`, 将访问未初始化的硬件寄存器或空指针.

### 5. 与同类教学内核的对比

原版 xv6-riscv 完全没有网络子系统; 本内核在系统调用层面增加了完整的 BSD socket 支持, 属于重大功能扩展. 然而, 与常见的教学内核网络实现 (如 xv6-net 扩展、OSv 或 rCore-Tutorial 的 socket 层) 相比, 这里缺少了文件描述符与 socket 的集成 (`struct file` 的 `type == FD_SOCKET` 分支)、inode 风格的 socket 节点、以及用户态/内核态缓冲区之间的 `copyin`/`copyout` 封装. 如果 socket 系统调用仅是对底层 `net_send_packet` 的简单包装, 则完全无法满足 `sendto`/`recvfrom` 所需的地址/长度参数传递、阻塞/非阻塞语义或 `poll` 集成, 可能导致用户态程序出现难以调试的协议栈故障<sup>[3](#mod-net-ref-3)</sup>.

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 本内核没有正式的设备驱动框架，而是由一组遵循临时约定的独立驱动组成。核心抽象是块设备接口（block_init / block_read_sector）与网络接口（net_init / net_send_packet / net_recv_packet），两者均基于 VirtIO 传输。并存两种传输后端：MMIO（用于 QEMU virt RISC-V）和传统 PCI（用于 LoongArch64）。每个驱动自行管理 virtqueue 环、设备发现和寄存器访问，缺乏共享的 VirtIO 库，导致描述符结构体、队列布局和初始化序列在三个文件中大量重复。UART 驱动是最简形式——仅通过 MMIO 忙等输出字符，初始化函数为空。与 xv6-riscv 的单一 virtio_disk.c 相比，本模块扩展了网络驱动和 PCI 传输，但 VirtIO 样板代码是拷贝而非提取的。

### 1. 核心抽象：块与网络的松散契约

这个内核没有设备模型（如 Linux 的 `struct device` 或 `struct kobject`），驱动以全局函数的形式提供服务。块设备接口在 `block.h` 中声明为三个函数：`block_init`、`block_available`、`block_read_sector` <sup>[4](#mod-drivers-ref-4)</sup>。网络接口未在独立头文件中找到（`net.h` 未提供），但从 `virtio_net.c` 可推断出 `net_init`、`net_available`、`net_send_packet`、`net_recv_packet` 四元组。每个子系统假设全局唯一实例：`blk_ready` 和 `net_ready` 静态变量控制可用性，没有“设备号”或“设备指针”参数。这种设计在单板机教学内核中常见——xv6 同样只有一个磁盘——但扩展性为零。UART 甚至更简单：`uart_putc` 直接向固定地址 `UART0_BASE` 写字节，`uart_init` 为空 <sup>[7](#mod-drivers-ref-7)</sup>，完全依赖引导固件完成的初始化。

### 2. VirtIO 传输层的分裂与内聚缺失

VirtIO 是块设备和网络背后的核心传输协议，但代码中不存在统一的“VirtIO 库”。`virtio_net.c` 定义了 `struct NetQueue`，将 8KB 的 `mem` 数组按固定偏移划分为描述符表（desc）、可用环（avail）和已用环（used），并维护 `used_seen` 游标 <sup>[1](#mod-drivers-ref-1)</sup>。`virtio_blk.c` 用了几乎相同的布局：一个 `queue_mem` 数组 + 三个 volatile 指针 <sup>[2](#mod-drivers-ref-2)</sup>。LoongArch 的 PCI 块驱动 `virtio_pci_blk.c` 再次复制了这套结构 <sup>[3](#mod-drivers-ref-3)</sup>，仅将 queue_mem 扩大到 16KB 以容纳更大的环。

三者都重复定义了 `VirtqDesc`、`VirtqAvail`、`VirtqUsed` 以及 `VIRTIO_STATUS_*` 等常量。初始化流程也高度相似：写 status 寄存器进行 ACKNOWLEDGE → DRIVER → FEATURES_OK 协商，然后设置队列地址。区别仅在于寄存器访问方式：MMIO 驱动使用 `*(volatile u32 *)` 直接解引用，PCI 驱动通过 IO 端口函数 `io_write16` 等。如果能抽象出一个 `struct virtio_transport` 提供 `read_device_features` / `setup_queue` 等方法，三个文件可以共享同一套队列管理代码。当前的做法使任何 VirtIO 规范的修订（例如添加新的 feature bit）都需要修改多处。

### 3. 设备发现：硬编码地址与自包含 PCI 枚举

三个驱动各自实现了设备探测，缺乏总线的概念。`virtio_net.c` 的 `net_init` 从 `VIRTIO0_BASE` 开始，以 4KB 步长扫描一段 MMIO 区域，查找 magic 值 `0x74726976` 且设备 ID 为 1 的 VirtIO 设备 <sup>[10](#mod-drivers-ref-10)</sup>。`virtio_blk.c` 的 MMIO 版本则直接假设 `VIRTIO0_BASE` 地址存在设备 ID 为 2 的块设备，不做扫描。PCI 版本 `virtio_pci_blk.c` 内嵌了完整的 PCI 枚举逻辑：`find_legacy_blk` 遍历 bus 0 的 32 个设备（以及多函数设备），通过读配置空间 vendor/device ID 匹配 `0x1af4:0x1001` <sup>[9](#mod-drivers-ref-9)</sup>。这些地址——`PCI_ECAM_BASE`、`VIRTIO0_BASE`、`UART0_BASE`——均来自 `memlayout.h`，属于编译期硬编码。移植到新平台需要修改这些常量，而非更换设备树 blob。

### 4. 边角细节与不足

**缓冲区泄漏风险**：`net_recv_packet` 在取到 used 描述符后立即递增 `rxq.used_seen`，然后才校验 `id` 和 `len`。若校验失败（例如设备返回损坏的描述符），函数直接 `return 0`，既不回退 `used_seen`，也不调用 `rx_refill_one`，导致该槽位永久丢失 <sup>[5](#mod-drivers-ref-5)</sup>。正确的做法是先校验再递增，或在失败路径上 refill。

**无限等待**：所有 VirtIO 操作均以忙等轮询 `used->idx` 的方式等待设备完成，无超时机制 <sup>[6](#mod-drivers-ref-6)</sup>。如果 QEMU 设备因 bug 或配置错误停止响应，内核将永远挂起。教学内核常省略超时以简化代码，但生产环境中这是致命缺陷。

**单描述符假设**：`net_send_packet` 固定使用 TX 描述符表的槽位 0 <sup>[8](#mod-drivers-ref-8)</sup>，意味着同一时刻只能有一个待发送包。若内核在发送过程中被中断，且中断处理程序试图再次发送，则会破坏正在传输的描述符。好在当前内核为单核且无抢占调度，暂不触发。

**写操作缺失**：块设备接口只有 `block_read_sector`，没有对应的写函数。文件系统或应用程序无法持久化数据，这使内核仅适用于只读演示。

**UART 裸机假设**：`uart_init` 为空，依赖于 UART 已被前一级引导程序（如 U-Boot）配置到合适的波特率。若跳过了该步骤，`uart_putc` 中的 `while ((uart0[UART_LSR] & UART_LSR_EMPTY_MASK) == 0)` 将无限循环。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 本模块基于 RISC-V ecall 陷入机制，将用户态寄存器上下文（TrapFrame）转化为内核系统调用分发。核心包括：1) 在 trap_handler 中根据 scause 判断用户态 ecall，对信号返回 (SYS_rt_sigreturn) 特殊处理，其余调用 syscall_dispatch 查表执行；2) 大量 Linux 兼容的系统调用号定义（400+）及对应的静态资源表（管道、socket、eventfd 等），每个资源表项通过 used 标志和引用计数管理生命周期；3) 与进程调度、信号处理、内存管理的跨模块协同，通过 task_on_trap_return 和后处理实现抢占与信号投递。相比于 xv6 的 20 余个调用，本内核力图支持复杂应用程序，但在静态资源上限、用户指针校验、时间基准硬编码等方面存在明显工程风险。

### 1. 核心理念：从 ecall 到系统调用分发

本内核沿用 RISC-V 标准的 ecall 用户态陷入路径。trap_handler 读取 scause 判断是否为来自用户态的 ecall，然后检查 a7 寄存器：若等于 SYS_rt_sigreturn，则直接调用 task_rt_sigreturn 完成信号返回的上下文恢复；否则交由 syscall_dispatch 进行通用系统调用分发 <sup>[1](#mod-syscall-ref-1)</sup>。这样做的理由是信号返回必须完整还原所有用户寄存器——包括可能已被系统调用覆盖的 a0 返回值——无法通过简单的 tf->a0 赋值和 sepc 偏移实现，因此需要一条专用快速路径。

对于普通系统调用，syscall_dispatch 返回一个 long 值。若该值等于 SYS_RESTART，表明系统调用因信号中断而需要重新执行，此时 sepc 保持不变，用户态将再次执行 ecall；否则将返回值写入 tf->a0 作为用户态结果，并将 sepc 加 4 以跳过 ecall 指令 <sup>[1](#mod-syscall-ref-1)</sup>。这种设计比原版 xv6 的固定 -1 错误返回更加精细，兼容了 Linux 的信号中断重启语义。

syscall.h 中定义了从 17 到 435 共数百个系统调用号，涵盖了文件系统、进程控制、网络 socket、eventfd、timerfd、共享内存等现代 Linux 常用接口 <sup>[2](#mod-syscall-ref-2)</sup>。与之对应，syscall_dispatch 应该维护一张 SyscallDesc 数组，根据 a7 索引函数指针并调用。虽然 syscall.c 的完整实现未在片段中展示，但从其声明的接口 `long syscall_dispatch(struct TrapFrame *tf)` 可以推断，参数通过 TrapFrame 中的 a0~a5 传递，实现中会将其转发给具体处理函数。

### 2. 关键取舍：静态资源管理

为了实现如此大规模的系统调用，内核采用了静态定长数组的方式管理各类资源。例如管道表 `static struct Pipe pipe_table[TASK_MAX_PIPE]`、socket 表 `static struct SocketState socket_table[SOCKET_MAX]`、eventfd 表、epoll 表、timerfd 表以及共享内存段表等 <sup>[3](#mod-syscall-ref-3)</sup>。每个资源表条目使用一个 `used` 标志和引用计数 `ref` 进行生命周期跟踪，典型结构如 `struct EventFdState { int used; int ref; u64 value; int flags; };` <sup>[4](#mod-syscall-ref-4)</sup>。

这种设计完全避免了动态内存分配，对应的分配/释放路径只是遍历固定数组寻找空闲槽，简单且不存在内存泄漏风险。但代价是资源并发量受到硬编码上限的严格约束，例如 SOCKET_MAX 仅为 64，一旦耗尽将直接向用户态返回 -ENFILE 或 -EMFILE。对于旨在运行复杂 Linux 应用程序的内核，这种限制极易成为瓶颈。相比之下，xv6 虽然同样静态分配进程表和文件表，但其规模小得多，而本项目显然希望在更大规模上提供兼容性。

此外，这些全局数组很可能被多个 CPU 核并发访问，但片段中未出现任何自旋锁或互斥体保护。若在多核环境下多个核心同时分配/释放资源，`used` 和 `ref` 的更新将产生数据竞争，导致资源泄漏或重复分配 <sup>[3](#mod-syscall-ref-3)</sup>。

### 3. 用户态指针安全与路径处理风险

内核必须校验所有来自用户态的指针。syscall.c 中声明了 `static int user_range_ok(uintptr_t ptr, usize size, int write);`，预期实现会检查虚拟地址区间是否完全位于用户地址空间且当前可访问 <sup>[5](#mod-syscall-ref-5)</sup>。然而其具体实现并未出现在当前片段中，若未正确完成或未在所有系统调用中强制使用，用户可通过传入内核地址或错误地址引发缺页异常甚至内核崩溃。

在字符串处理方面，内核定义了一系列辅助函数如 `copy_cstr`，它使用 `max_len` 参数限制拷贝长度 <sup>[10](#mod-syscall-ref-10)</sup>。但调用者若从用户态字符串计算长度时未做充分校验，仍可能造成缓冲区溢出。尤为典型的是路径解析函数 `resolve_path_arg`，它接收用户路径，通过 `fs_resolve_task_path` 将其解析到内核分配的缓冲区 `resolved` 中，而缓冲区的尺寸由调用者指定。若该尺寸计算错误或路径超长，将直接导致内核栈溢出 <sup>[6](#mod-syscall-ref-6)</sup>。在 `build_abi_path` 等辅助函数中也存在类似模式，依赖调用者正确传递 `out_size`。

### 4. 跨模块协同：陷入上下文与进程调度

系统调用层并非孤立运转。在 trap 入口，汇编代码 `trap.S` 通过 `sscratch` 寄存器交换用户栈和内核栈，将 32 个通用寄存器连同 `sepc` 和 `sstatus` 保存到内核栈上的 TrapFrame 结构 <sup>[7](#mod-syscall-ref-7)</sup>。这种设计为每个进程提供了独立的内核栈，避免用户栈污染，同时也为系统调用中发生的嵌套陷阱（如缺页）提供了上下文支持。

系统调用返回前必须调用 `task_on_trap_return`，这通常是进程调度的关键触发点。在该函数内部，内核可以检查是否有更高优先级进程就绪，或是否有待投递的信号 <sup>[1](#mod-syscall-ref-1)</sup>。时钟中断则通过 `timer_set_next` 设置下一次中断，并在中断处理中调用 `task_request_resched` 请求调度 <sup>[8](#mod-syscall-ref-8)</sup>。这些机制共同实现了抢占式多任务，使得系统调用层不仅能处理请求，还能作为调度点的触发源。

对于信号，`syscall_dispatch` 返回 `SYS_RESTART` 后，任务子系统会在适当时候安排信号处理，并最终在 `task_on_trap_return` 中设置用户态上下文以重新执行被中断的系统调用。这要求任务模块能够记住当前系统调用号和参数，具体细节依赖 `task` 模块的实现。

### 5. 边角细节与不足

尽管设计规模宏大，已有代码暴露出若干工程隐患：

- **时钟基准硬编码**：`QEMU_VIRT_TIMEBASE` 被硬定义为 10000000UL，与 QEMU virt 平台每秒 10^7 次的时钟频率绑定 <sup>[9](#mod-syscall-ref-9)</sup>。若在不匹配的硬件上运行，所有定时器、sleep 类调用将完全错误，且内核没有提供运行时频率检测的钩子。
- **静态数组边界风险**：例如 epoll 表 `epoll_has_ready` 等函数接收 epoll_id，若传入无效值且未严格校验数组下标，可能导致越界访问。
- **用户指针校验缺失实现**：`user_range_ok` 仅为声明，未见实现，可能导致大量系统调用直接信任用户指针。
- **错误码一致性**：虽然模仿 Linux 返回负错误码，但部分代码可能忽略返回 -ENOSYS 等，用户态期望的 `errno` 机制在内核中如何支持尚未明确。
- **资源清理路径缺失**：当进程异常退出时，其打开的文件描述符、socket 等理应被内核回收，但片段中未出现对应的清理逻辑，可能导致静态表格条目无法重用，长时间运行后资源耗尽。
- **未实现桩函数**：系统调用号列表中大量调用（如 shmget、clone3 等）的实现未知，可能多数为直接返回 -ENOSYS 的桩，这将导致依赖这些调用的程序出错。

这些细节反映了从教学内核迈向实用内核时需应对的种种挑战。系统调用层作为用户与内核的桥梁，其健壮性至关重要。进一步分析需要深入到具体系统调用实现中，逐一审查指针对齐、权限检查、错误路径的资源释放等环节。

### ⚠ 开放问题

- 时钟频率硬编码于 QEMU_VIRT_TIMEBASE，在不同硬件上将导致所有时间相关系统调用错误。
- 全局资源表无锁保护，在多核并发分配/释放时存在数据竞争，可能导致 used 标志或引用计数错乱。
- user_range_ok 仅为声明，未见实现，大量系统调用可能未对用户指针做充分校验。
