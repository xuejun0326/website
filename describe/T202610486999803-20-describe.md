## 📖 一、总览

A20OS 是一个自研的 Unix-like 内核，基于 xv6 但进行了深度的功能扩展，在整体架构上定位为教学与研究兼具的演示级内核。它采用 C 语言编写，管理约 30 万行代码，实现了类 Linux 的进程调度、内存管理、虚拟文件系统、网络协议栈及驱动框架，并支持原生 A20 ABI 与 Linux ABI 的双表系统调用分支。与 xv6 相比，其复杂度跃升了一个数量级，但仍保留了固定的关键数据结构大小、部分未完成的桩函数以及明显的 SMP 安全性隐患（如无锁全局单例），表明项目仍处于积极开发或有选择地裸露技术债的阶段。

最值得评审人注意的特点有三：其一，**双 ABI 架构**（[syscall.ref:1]）——内核同时维护 `a20_syscall_lookup` 和 `linux_syscall_lookup` 两套函数表，使得同一内核能直接运行原生和 Linux 二进制文件；其二，**VFS 层采用短生命周期 vnode 模型**（[fs.ref:3]）——放弃 vnode 缓存以避免悬空指针崩溃，是作者精于稳定性的刻意取舍；其三，**异步网络数据传递的无锁底部半环 bh_ring**（[net.ref:1]）——在 lwIP 回调与系统调用上下文之间设计了一个单生产者单消费者的无锁环形缓冲区，兼顾安全与效率。

## 🎯 二、综合评价

### 整体定位
A20OS 处于教学内核（如 xv6）与生产内核（如 Linux）的中间地带。从目录结构和代码风格（如 `mm_struct_t`, `task_t`, `vnode_ops_t`）来看，它明显师法 Linux，但并没有追求完整的 POSIX 或 Linux API 覆盖。相反，它选择性地实现了足以运行常见用户程序（如 shell、简单 Web 服务器）的子系统，并注入了鲜明的教学印记：大量说明性注释、固定的缓冲区大小（如 `SIGALRM_PIDS_MAX 128`、`wake_list[512]`）、以及“如果出问题就 panic/关闭中断”的开发方式。可以说，A20OS 是一个规模最大化、定位在研究员和高级学生之间的实验平台。

### 真正的创新点/独特设计
1. **双 ABI 系统调用层**（[syscall.ref:1]）：传统兼容内核往往将 Linux syscall 同一映射到原生 ABI，而 A20OS 保留了两张独立的函数指针表。这种设计增加了代码体积，但保留了原生 ABI 的演进自由，同时避免了映射复杂性。缺点是必须同步维护两张表，且部分 Linux syscall 仅返回 ENOSYS。

2. **短生命周期 vnode 模型**（[fs.ref:3]）：ext4 模块在经历基于缓存的悬空指针崩溃后，主动退化为每次 lookup 创建新 vnode、引用计数归零即释放。这彻底消除了缓存一致性的复杂性，换来的是对每个文件元数据操作的频繁分配释放。对于教学系统和有限负载来说，这是一个合理的稳定性优先的权衡。

3. **无锁底部半环 (bh_ring)**（[net.ref:1]）：在 lwIP 的复杂回调语境（可能处于中断或持有 `g_lwip_lock`）与 socket 系统调用的 `g_net_lock` 之间，设计了基于原子 head/tail 索引的单生产者单消费者 ring。这避免了死锁和双向锁依赖，是模块间协调的出色实践。

4. **A20 原生 IPC 事件队列**（[ipc.ref:5]）：通过全局哈希表将对象指针映射到监视条目，实现内核对象的事件通知。虽然当前阻塞等待尚未实现（`a20_eventq_wait` 始终返回 `-EWOULDBLOCK`），但设计意图清晰，为后续提供真·异步事件框架铺路。

### 取舍判断
从代码中能清晰看出作者的精力偏好：**在内存管理、进程调度和 VFS 上投入了极大心血**（实现了 COW、mprotect、mremap、O(1) 调度器、RT 策略、ext4、epoll 等），而 **信号机制、网络栈和驱动框架则采取了“够用即可”的策略**。信号缺少完整并发保护（[signal] 中 `pending` 字段无锁）、网络部分在临时端口分配和 TCP 大数据发送上留下了未竞态的代码、驱动中 StarFive GMAC 的无锁全局单例（[drivers.ref:5]）都是明显的技术债。这表明作者有意将底层接口打通后，优先在上层（系统调用兼容性）展示功能。

另一个显著取舍是**固定大小数组的大量使用**：shm 表（`g_shm[128]`）、信号量集合数（`g_sem[64]`）、socket 表（`NET_MAX_SOCKETS`）、procfs 软链接缓冲区（60B）等。这些硬编码简化了动态内存管理，却限制了可扩展性。教学内核常以此降低复杂度，A20OS 也不例外。

### 完成度与不足
**完成度**：整体功能范围令人印象深刻 — 395 项系统调用涵盖进程、文件、网络、IPC 等多域，核心场景（fork/execve/mmap/socket）均可工作。但深度参差不齐：ext4 和 procfs 内有约 30% 的桩函数（[fs.ref:19]），sysv sem 的 `GETNCNT` 返回 0（[ipc.不足]），信号的核心转储仅为状态赋值（[signal.不足]）。此外，多处错误路径未妥善处理（如 extent 树分裂内存泄漏 [fs.ref:6]，mm munmap 中重复 `page_cache_put` [mm.ref:18][mm.ref:19]），双 ABI 表中大量 Linux syscall 返回 `-ENOSYS`。因此完成度打分 2（合格），尚需填补细节漏洞。

**创新性**：虽然未发明全新技术，但在教学内核范畴内，双 ABI、短生命周期 vnode、无锁 bh_ring 均是富有启发性的设计。它们直接回应了真实系统中稳定性和性能的挑战，适合在教学案例中分析。扣分项在于部分创新（A20 事件队列）并未实现完整语义（阻塞等待缺失），因此创新性评为 2（合格）。

**代码质量**：整体代码风格统一、命名规范，关键路径有注释和 `LOCK_ORDER` 标注（[drivers.ref:1]）。但多处潜在内存泄漏、越界、锁保护缺失、硬编码假设（x86_64 指令长度 [syscall.ref:3]）降低了安全底线。特别是 SMP 环境下，全局自旋锁的粒度较粗（mm 锁、runq 锁等），但仍有不少无保护共享访问（如 signal.pending、g_shm 表并发）。因此代码质量评为 2（合格），需要一次集中的审计与修正。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 内核内存管理模块以 mm_struct_t 为核心抽象，管理每个进程的虚拟地址空间（VMA 链表 + 多级页表），物理内存通过 buddy 分配器 (pfa) 与 slab 分配器分层供给。模块实现了 Linux 兼容的 mmap/munmap/mprotect/mremap/brk 等系统调用，支持 COW fork、文件映射、共享内存、大页降级、ASLR 以及 ELF 加载器。相比 xv6 原版，该实现增加了多级页表递归遍历、VMA 拆分合并、mprotect 权限修改、mremap 地址搬迁、用户态拷贝的缺页按需解析等高级特性，同时有意识地设计了锁模型（mm->lock 配合中断关闭）以应对 SMP 并发。

### 1. 核心抽象：两级内存管理层与 VMA 链表

内存管理模块的核心抽象分为物理页管理（底层）和虚拟地址空间（上层）两层。底层通过 `pfa_alloc_page` / `pfa_free_page`（buddy 分配器）提供连续物理帧，slab 分配器在其上管理小对象（`slab.c`）。`mm_init()` 在 `_bss_end` 之后初始化 buddy，接着初始化 slab <sup>[1](#mod-mm-ref-1)</sup>。上层每个进程持有一个 `mm_struct_t`，包含页表根指针 `pgdir`、VMA 链表 `mmap`、堆边界 `brk`/`start_brk` 以及统计计数 `total_vm`/`rss`/`locked_vm` <sup>[2](#mod-mm-ref-2)</sup>。

VMA 链表以 `vm_area_t` 节点描述一段连续的虚拟地址区间，每个节点记录起始/结束地址、vma 标志（`VM_FILE`/`VM_SHARED`/`VM_STACK` 等）和页表项标志。`mm_insert_vma` 插入新 VMA 时自动尝试与相邻节点合并 <sup>[3](#mod-mm-ref-3)</sup>；`mm_split_vma_at` 在指定地址处切开 VMA，用于 `mprotect` 和 `mremap` 的局部操作。合并/拆分逻辑通过 `vma_can_merge` 检查文件偏移连续性、sysv shm 互斥性等条件，避免非法合并 <sup>[4](#mod-mm-ref-4)</sup>。

### 2. 关键设计取舍：COW fork、大页降级与缺页解析

**COW fork** 是模块最复杂的路径之一。`mm_fork()` 复制父进程 VMA 链表后，在持有 `parent->lock` 且关闭本地中断的条件下遍历父进程页表，调用 `mm_fork_clone_present_level` 逐级克隆叶节点。对于可写页，克隆时使用 `mm_cow_flags()` 清除 `PTE_W`/`PTE_D` 并设置 `PTE_COW`；父进程的 PTE 同样被改写为 COW 标志 <sup>[5](#mod-mm-ref-5)</sup>。这个锁保护是后来添加的回归防护（`MM_FORK_COW_REGRESSION_GUARD` 注释），防止多线程父进程与缺页处理并发修改同一 PTE 导致数据竞争 <sup>[6](#mod-mm-ref-6)</sup>。

**大页降级** (`mm_demote_huge_page`) 解决 PMD 级别大页部分覆盖 VMA 边界的问题。当 `free_vma_pages` 或 `mm_munmap` 遍历到一个 leaf 级别 > 0 且基地址超出目标范围的大页时，先将其拆分为 4K 小页再逐个释放 <sup>[7](#mod-mm-ref-7)</sup>。降级流程分配 `PMD_PAGE_COUNT` 个物理帧，拷贝原大页内容，解除大页映射后逐页建立小页映射，最后刷新 TLB。这个操作在 `mm_mprotect` 和 `mm_mremap` 中也被调用，代码中多处出现 `continue` 回退逻辑，形成一个“遇到大页→降级→重试”的循环模式 <sup>[8](#mod-mm-ref-8)</sup>。

**缺页按需解析**体现在 `copy_from_user`/`copy_to_user` 的实现中。`user_resolve_leaf` 首先调用 `pt_lookup_leaf` 查找 PTE；如果叶节点不存在，调用 `handle_demand_fault` 触发缺页处理；如果叶节点缺写权限（COW 保护），调用 `handle_cow_fault` 完成页复制后再写入 <sup>[9](#mod-mm-ref-9)</sup>。这个设计将用户态拷贝与缺页处理无缝衔接，无需调用方感知页表状态。

### 3. 跨模块协同：ELF 加载器、文件系统、共享脏页同步

**ELF 加载器** (`kernel/mm/elf.c`) 与内存管理模块紧密配合。`elf_load` 通过 `pt_create` + `pt_map_kernel` 建立新页表，调用 `map_segment` 为每个 PT_LOAD 段分配物理帧并复制内容，最后通过 `elf_add_vma` 将映射区间注册为 VMA <sup>[10](#mod-mm-ref-10)</sup>。对 ET_DYN 对象（含 PIE 和动态链接器）应用 ASLR 偏移（`elf_aslr_bias` 返回 11 位随机数左移 16 位 = 64MB 粒度随机化）<sup>[11](#mod-mm-ref-11)</sup>。动态链接器的搜索路径支持精确路径、可执行文件同级目录、musl 风格的 libc.so 替换以及体系结构特定后备路径 <sup>[12](#mod-mm-ref-12)</sup>。

**文件映射与页缓存**：`VM_FILE | VM_SHARED` 的 VMA 在缺页时通过 `mm_shared_file_fault` 从页缓存获取页面。`mm_sync_shared_dirty_for_vnode` 遍历所有进程，扫描与给定 vnode 关联的共享可写 VMA，读取硬件脏位（`PTE_D`）并标记页缓存对应页面为脏 <sup>[13](#mod-mm-ref-13)</sup>。这个函数的锁模型（`proc_lock` → `mm->lock` → `g_page_cache_lock`）有明确的顺序约定，但代码中未验证 `g_page_cache_lock` 是否存在或已初始化。

**brk 与栈管理**：`mm_brk` 扩展堆时调用 `frame_alloc` 分配物理页并映射；收缩时释放页面并更新 RSS 计数 <sup>[14](#mod-mm-ref-14)</sup>。栈映射在 `elf.c` 的 `map_stack` 中分配初始页面（`USER_STACK_INITIAL_PAGES`），`stack_ensure_mapped` 在 `elf_setup_stack` 设置参数/envp/auxv 时按需向下扩展栈底，最多允许 `max_grow` 页的额外增长 <sup>[15](#mod-mm-ref-15)</sup>。

### 4. 边角细节与不足

**<sup>[16](#mod-mm-ref-16)</sup> 整数溢出与截断风险**：`mm_find_gap` 中 `uint64_t prev_end = hint` 后遍历 VMA 累加 `prev_end`，若所有 VMA 覆盖范围到 `USER_VA_LIMIT`，则 `prev_end` 可能回绕到 0 或成为已占用的地址，造成后续 `mmap` 返回非法地址。`mm_mmap` 虽在之后检查 `addr + len < addr` 溢出，但若 `prev_end` 恰好落在已映射区间内，静默的重叠映射将破坏地址空间完整性。

**<sup>[17](#mod-mm-ref-17)</sup> 页表递归销毁中的非法条目跳过**：`pt_destroy_user_recursive` 遇到 `pfn_valid` 失败或物理地址未对齐的非叶 PTE 时，打印错误后直接 `table[i] = 0` 并 `continue`，不会递归进入子页表。这会导致整个子页表树泄漏物理帧。该防御性跳过虽防止了内核崩溃，但掩盖了可能的页表损坏或内存泄露。

**<sup>[18](#mod-mm-ref-18)</sup> 共享文件映射的 page_cache 引用计数异常**：`mm_munmap` 的 `shared_file_vma` 分支中对每个 4K 子页调用两次 `page_cache_get` 和两次 `page_cache_put`，注释未解释为何需要双倍引用。若 `page_cache_get` 实际只获取一次引用，则第二次 `page_cache_put` 可能导致 page_cache 页被过早释放，后续文件操作访问悬空指针 <sup>[19](#mod-mm-ref-19)</sup>。

**<sup>[20](#mod-mm-ref-20)</sup> elf_setup_stack 栈上数组无界限检查**：`arg_ptrs[64]` 和 `env_ptrs[64]` 是固定大小的栈数组，但 `argc` 和 `envc` 只通过 `while (envp[envc] && envc < 63)` 限制，上限为 63。如果用户程序传入 64 个环境变量或参数，将越界写入栈内存，可能破坏返回地址或相邻局部变量。

**<sup>[21](#mod-mm-ref-21)</sup> ELF 加载失败路径资源泄漏**：`elf_load` 的 `fail` 标签使用 `kfree` 而非 `vma_release` 释放已创建的 VMA 链表，不会释放 VMA 中引用的文件描述符（`vma_release_file`）和共享内存引用（`vma_release_ipc`）。如果 `map_segment` 在部分段加载后失败，已分配的文件引用将永久泄漏。

**<sup>[22](#mod-mm-ref-22)</sup> 锁顺序依赖缺乏运行时验证**：`mm_sync_shared_dirty_for_vnode` 持有 `mm->lock` 后调用 `page_cache_get(0)`，注释声称锁顺序 `mm->lock → g_page_cache_lock` 已建立。但代码中没有 `lockdep` 风格验证，若未来重构引入反向嵌套，将产生隐蔽死锁。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块以 task_t 为进程控制核心，围绕多级调度队列（每 CPU 独立 proc_runq_t，含 SCHED_LEVELS 个优先级队列）和双锁模型（proc_lock + runq_lock）构建。解决的核心问题是多任务抢占调度、进程生命周期管理（fork/exec/exit/wait）以及 POSIX 信号递送。与 xv6 原版相比，显著扩展了：O(1) 级调度器（bitmap dispatch + aging）、RT 策略支持（SCHED_FIFO/SCHED_RR）、cgroup CPU 控制器集成、双 ABI（Linux + Native）exec 路径、以及完整的 POSIX 信号语义。

### 1. 核心抽象：task_t 与多级调度队列

进程调度的核心数据结构是 `task_t`（通过 `proc.c` 中的 `proc_alloc_task_slot` 动态分配 <sup>[4](#mod-task-ref-4)</sup>），它承载了完整的 PCB：调度状态（`state`）、运行队列链接（`rq_next/rq_prev`）、调度策略与优先级（`sched_policy`/`priority`）、内存管理（`mm`/`pgdir`）、信号状态（`signals`）、凭据（`cred`）、cgroup 关联（`cgroup`）等。进程全局链表由 `task_list_head/tail` 维护，受 `proc_lock` 保护 <sup>[1](#mod-task-ref-1)</sup>。

调度队列的设计是模块最突出的抽象：每个 CPU 维护一个 `proc_runq_t` <sup>[6](#mod-task-ref-6)</sup>，内部包含 `SCHED_LEVELS` 个双向链表（由 `rq_next/rq_prev` 链接）和一个 `uint32_t bitmap` 用于 O(1) 定位非空队列。RT 任务（`SCHED_FIFO`/`SCHED_RR`）始终进入队列 0，且在该队列内按优先级排序；普通任务根据 `sched_level` 散列到不同队列。`sched_promote_aged_locked` <sup>[10](#mod-task-ref-10)</sup> 实现了老化提升 —— 在队列 1 及以上等待超过 `SCHED_AGING_THRESHOLD` 的非 RT 任务会被提升到队列 0，防止饥饿。

任务状态机清晰：`PROC_READY`→`PROC_RUNNING`→`PROC_BLOCKED`/`PROC_ZOMBIE`/`PROC_STOPPED`。`proc_make_ready` <sup>[5](#mod-task-ref-5)</sup> 负责将任务放回就绪队列并触发跨 CPU 唤醒（`proc_sched_kick_cpu` <sup>[12](#mod-task-ref-12)</sup>），而 `context_switch` <sup>[14](#mod-task-ref-14)</sup> 是唯一将任务状态置为 `PROC_RUNNING` 的路径。

### 2. 关键设计取舍：锁分层、O(1) dispatch 与负载均衡

**锁分层**是该模块最值得分析的设计决策。代码在 `sched.c:70-74` <sup>[8](#mod-task-ref-8)</sup> 显式注释了锁顺序：`proc_lock → runq_lock`，禁止反向。`proc_lock` 保护全局进程链表和状态转换（如 `sched_scan_timers` 遍历全链表 <sup>[13](#mod-task-ref-13)</sup>），而 `runq_lock` （每 CPU 独立）保护入队/出队/选择操作。这种分离避免了全局锁竞争：调度器热路径（`proc_runq_pick_locked` <sup>[11](#mod-task-ref-11)</sup>）只获取本地 CPU 的 `runq_lock`，不碰 `proc_lock`。但这也带来了复杂性 —— `proc_make_ready` 先持 `proc_lock` 选择目标 CPU，再持 `runq_lock` 入队；跨 CPU 唤醒时必须通过 IPI（`smp_send_reschedule`）通知目标 CPU，而目标 CPU 在调度器入口（`sched()`）再检查 `on_rq` 标志。注释中列出的 7 条并发前提（`SCHEDULER_CONCURRENCY_PREREQS`）反映了该模型对时序的严格依赖。

**O(1) dispatch** 方面，`proc_runq_pick_locked` <sup>[11](#mod-task-ref-11)</sup> 使用 bitmap 扫描最高优先级非空队列，复杂度严格 O(SCHED_LEVELS) 而非 O(n)。对于队列 0 的 RT 任务，它还额外遍历链表选择最高 `priority` 的任务 —— 这是权衡：RT 任务数量通常很少，遍历开销可接受。对比 xv6 的简单轮询（全局就绪链表 + O(n) 扫描），这个设计明显更像 Linux 2.6 的 O(1) 调度器。

**负载均衡** 由 `proc_sched_select_cpu` <sup>[9](#mod-task-ref-9)</sup> 实现：遍历所有 CPU 的 `sched_runq[cpu].nr_running`，选择负载最轻的目标。同时考虑 `cpus_allowed` 掩码和 cgroup `cpuset.effective_cpus`。这是一个简单的贪心策略，放弃了迁移成本和缓存亲和性优化 —— 对于教学级内核是合理的取舍。

### 3. 跨文件协同：exec、信号与调度器

进程生命周期中最复杂的跨模块协同发生在 `exec` 路径。`proc_exec` <sup>[17](#mod-task-ref-17)</sup> 的设计灵感来自 Linux `do_execve` / `struct linux_binprm`：先将用户态 argv/envp 通过 `exec_copy_args` <sup>[15](#mod-task-ref-15)</sup> 一次性复制到内核堆上的 `exec_bprm_t`（含边界检查和 `MAX_ARG_BYTES` 总大小限制），然后迭代解析 ELF/shebang（最多 `EXEC_MAX_DEPTH = 4` 层）。关键设计决策在于 **new mm 先行准备，再原子交换**：`exec_install_process` <sup>[16](#mod-task-ref-16)</sup> 先构建好 `mm_struct_t`（含新页表、brk、mmap 链表），确认无误后再通过 `t->mm = new_mm` 交换，最后销毁旧 `mm`。这保证了失败时可回滚，不会丢失旧地址空间。

`exec_install_process` 与信号模块的协同体现在步骤 5：重置所有非 `SIG_IGN`/`SIG_DFL` 的信号处理函数为 `SIG_DFL`，清零 pending 位图 <sup>[16](#mod-task-ref-16)</sup>。这与 POSIX exec 语义一致。同时，步骤 6 设置 `trap_ctx` 时，将 `kernel_tp` 指向当前 task，确保 trap 返回时能正确恢复进程上下文。

信号递送本身是异步的，但实际执行在 trap 边界同步完成。`signal_send_info` <sup>[18](#mod-task-ref-18)</sup> 将信号加入目标进程的 `pending` 位图，如果目标处于 `PROC_BLOCKED` 或 `PROC_STOPPED` 则调用 `proc_make_ready` 唤醒。`signal_make_page_exec` <sup>[19](#mod-task-ref-19)</sup> 是一个引人注目的辅助函数：它通过 `pt_unmap` + `pt_map` 将用户栈上的信号跳板页重新映射为可执行。这是对 W^X 原则的明确破坏 —— 设计者知道这一点，因为 `mmap` 分配的栈页默认无 `VM_EXEC` 标志，而 sigreturn 跳板必须放在栈上。这个取舍选择了兼容性（让 `clone02` 等测试通过）而非安全性。

调度器与 cgroup 的集成在 `context_switch` <sup>[14](#mod-task-ref-14)</sup> 中体现：切换前调用 `cg_cpu_account` 核算当前任务的 CPU 使用时间（转换为纳秒），如果超过配额则标记 `cg_throttled`。被节流的任务在 `proc_runq_pick_locked` 中被跳过（`!t->cg_throttled` 检查），直到下一周期 `cg_cpu_check_unthrottle` 解除。这实现了简单的 CPU 带宽控制。

### 4. 边角细节与不足

**<sup>[2](#mod-task-ref-2)</sup> 全局链表 O(n) 遍历**：`proc_first_task_locked` / `proc_next_task_locked` 构成一个单向遍历宏。`sched_scan_timers`（每次调度器 tick 调用）和 `sched_reap_zombies` 都 O(n) 扫描全链表。没有哈希索引或红黑树优化 —— 进程数一多（>1000），每次 timer 扫描就是灾难。这是教学级内核可接受的简化，但注释中未提及此限制。

**<sup>[3](#mod-task-ref-3)</sup> 固定数组静默截断**：`sched_scan_timers` 中 `sigalrm_pids[128]` 和 `wake_list[512]` 使用固定栈数组 <sup>[13](#mod-task-ref-13)</sup>，当并发超时信号/唤醒任务超过容量时，超出的条目被静默丢弃（`sigalrm_pids` 直接不记录，`wake_list` 通过 `else if` 调整 `next_wake` 以期下次处理）。类似地，`sched_reap_zombies` 中的 `to_reap[32]` <sup>[20](#mod-task-ref-20)</sup> 也有限制。这些都不是安全的降级 —— 信号丢失可能导致进程永远阻塞。

**<sup>[7](#mod-task-ref-7)</sup> exec_copy_args 的用户/内核判断**：`exec_copy_args` 使用 `arch_is_kernel_address(argv)` 来判断 argv 是用户态还是内核态指针 <sup>[15](#mod-task-ref-15)</sup>。这个判断依赖于地址范围静态分割 —— 如果用户态程序通过 mmap 将内存映射到内核地址区域（某些架构上可能），则会绕过 `copy_from_user` 安全路径，直接以 `memcpy` 访问。这是一个潜在的安全盲区。

**<sup>[19](#mod-task-ref-19)</sup> 信号跳板破坏 W^X**：如前所述，`signal_make_page_exec` 将用户栈页重新映射为可执行 <sup>[19](#mod-task-ref-19)</sup>。攻击者若能控制栈内容，即可构造恶意跳板执行任意代码。虽然设计者通过 `mm_pte_flags_make_writable_dirty` 确保页仍可写（W^X 被破坏），但至少应该只在信号处理期间临时添加 X 权限，处理完毕后恢复。当前实现是永久性的。

**<sup>[10](#mod-task-ref-10)</sup> 老化精度**：老化阈值 `SCHED_AGING_THRESHOLD` 基于 `timer_get_ticks()`，但 `ready_since` 只在入队时设置。如果任务多次被切换出再重新入队（`proc_make_ready` 未重置 `ready_since`），累计等待时间可能不准确。这不是严重问题，但可能导致不公平的提前提升。

**<sup>[9](#mod-task-ref-9)</sup> 负载均衡朴素**：`proc_sched_select_cpu` 仅比较 `nr_running`，不考虑任务的历史 CPU 亲和性或缓存热度。这意味着任务可能频繁迁移，导致 cache miss 增加。对于教学级内核这是合理的简化，但生产环境中需要引入 per-task `cpu` 偏好和迁移成本计算。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块实现了类 Linux 的 VFS 层，提供统一的 fd 接口，支持 ext4、procfs、ramfs、fat32、devfs 多种文件系统。核心抽象为 vnode（类 inode）、vfile（打开文件描述）、mount（挂载点）、dcache（目录项缓存）和 ops 操作表。VFS 层实现了 Linux 兼容的 ~70 种系统调用（openat、statx、renameat2、epoll 等）。相比 xv6 原版，显著扩展包括：完整 ext4 文件系统（含 extent 树、位图分配、符号链接）、procfs 虚拟文件系统（~50 种文件类型）、POSIX 和 OFD 文件锁、页缓存和块缓存、管道、fcntl 全套操作。

### 1. VFS 核心抽象与架构

VFS 层以 vnode 为中心抽象，每个 vnode 对应一个文件系统对象，通过 `vnode_ops_t` 操作表（lookup、create、read、write、truncate 等）实现多态 <sup>[1](#mod-fs-ref-1)</sup>。打开文件由 `vfile_t` 描述，持有 vnode 引用和文件偏移等状态，其 `vfile_ops_t` 提供了 read/write/lseek/readdir 等文件级操作。挂载点由 `mount_t` 管理，`vfs_find_mount()` 根据路径前缀查找对应挂载 <sup>[2](#mod-fs-ref-2)</sup>。整个 VFS 采用**短生命周期 vnode** 模型——每次 lookup 创建新 vnode，引用计数归零时释放。这一设计是 ext4 模块在经历了 vnode 缓存导致的悬挂指针崩溃后主动选择的（见注释“amplified stale-pointer bugs into deterministic panics”）<sup>[3](#mod-fs-ref-3)</sup>。路径解析被拆分到 `fs/vfs/path_resolution.c` 中独立实现，支持 `RESOLVE_IN_ROOT`、`RESOLVE_BENEATH` 等安全标记 <sup>[4](#mod-fs-ref-4)</sup>。

### 2. ext4 实现的设计取舍

ext4 模块是 FS 层最复杂的部分，同时支持 extent 树和间接块映射两种块寻址方式 <sup>[5](#mod-fs-ref-5)</sup>。extent 树支持深度≥1 的索引结构，根节点最多 4 个条目，深度≥1 时叶节点按块大小容纳尽可能多的 extent <sup>[6](#mod-fs-ref-6)</sup>。这种方式对连续块分配效率高，但 extent 树分裂（如 `ext4_extent_grow` 中当根节点 4 个条目填满时创建两级索引）的代码路径非常长，且回滚处理不完整——分配了中间块但后续失败时可能泄漏已分配的块 <sup>[6](#mod-fs-ref-6)</sup>。间接块映射则是经典的 12 直接+单重间接+双重间接+三重间接结构，作为无 EXT4_EXTENTS_FL 标记的 inode 的 fallback <sup>[5](#mod-fs-ref-5)</sup>。

分配器使用每块组位图管理，通过 `ext4_alloc_block()` 和 `ext4_alloc_inode()` 进行 <sup>[7](#mod-fs-ref-7)</sup>。在锁设计上有一个显式修复：`ext4_alloc_block` 将 `kmalloc` 移到 `mutex_lock` 之外，避免持有 `alloc_lock` 时可能睡眠（kmalloc 可能触发页面回收/调度），从而防止优先级反转甚至死锁 <sup>[8](#mod-fs-ref-8)</sup>。但该修复不彻底——`ext4_bitmap_alloc` 内部仍会在持锁时调用 `kmalloc`（分配 block_size 的临时缓冲区）<sup>[9](#mod-fs-ref-9)</sup>。

目录项操作（`ext4_dir_find`/`ext4_dir_add`/`ext4_dir_remove`）在遍历目录块时每次 `kmalloc` 一个块大小的缓冲区，大目录遍历将显著增加内存分配压力 <sup>[10](#mod-fs-ref-10)</sup>。读取目录时 `ext4_freaddir` 对每个目录项做了边界验证（`de->rec_len < 8 || de->rec_len > bs - off || (de->rec_len & 3) || de->name_len > de->rec_len - 8`），防止恶意/损坏的 ext4 镜像导致溢出 <sup>[11](#mod-fs-ref-11)</sup>。符号链接只支持快速符号链接（≤60 字节直接存在 inode 的 i_block 中）<sup>[12](#mod-fs-ref-12)</sup>。

### 3. procfs：按需生成的虚拟文件系统

procfs 完全在内存中合成文件和目录结构，通过 `pf_type_t` 枚举区分 ~50 种文件类型 <sup>[13](#mod-fs-ref-13)</sup>。lookup 时动态创建 `pf_entry_t` 节点，read 时调用 `generate_content()` 即时格式化输出 <sup>[14](#mod-fs-ref-14)</sup>。重要文件如 `/proc/meminfo` 汇总了物理页分配器、slab、块缓存、页缓存、透明大页等统计信息 <sup>[15](#mod-fs-ref-15)</sup>；`/proc/<pid>/smaps` 遍历进程 VMA 并逐页查询页表权限和脏位 <sup>[16](#mod-fs-ref-16)</sup>。值得注意的边角细节：`/proc/<pid>/maps` 中获取文件路径通过 vfile 的 `path` 字段间接获得 `<sup>[17](#mod-fs-ref-17)</sup>`，但 `vma->file_fd` 是文件描述符索引而非全局 fd，需要经过 `t->files->fd[]` 转换，如果传入进程的文件描述符表在 VMA 建立后发生了变化可能导致错误路径 <sup>[17](#mod-fs-ref-17)</sup>。`/proc/config.gz` 内嵌了一份 gzip 压缩的内核配置（`g_proc_config_gz` 数组）直接返回给用户 <sup>[18](#mod-fs-ref-18)</sup>。所有 `PF_PID_NS_*` 类型统一返回 `[0]` 标记——命名空间是桩函数 <sup>[19](#mod-fs-ref-19)</sup>。

### 4. 跨模块协同与并发模型

VFS 层与进程子系统紧密耦合：`vfs_open` 通过 `proc_current()` 获取 cwd、root 和 umask <sup>[20](#mod-fs-ref-20)</sup>；文件描述符表（`fdtable`）是进程私有的，`vfs_open()` 调用 `vfs_alloc_fd()` 分配全局 fd 并建立映射 <sup>[21](#mod-fs-ref-21)</sup>。块缓存（`block_cache`）和页缓存（`page_cache`）为 ext4 和 fat32 提供磁盘 I/O 加速，`page_cache_truncate` 在文件 truncate 时联动 <sup>[22](#mod-fs-ref-22)</sup>。文件锁子系统（`fs/locks`）支持 POSIX（F_SETLK/F_SETLKW）和 OFD（F_OFD_SETLK）锁，通过 `vfs_release_open_file_locks` 在关闭文件时自动释放 <sup>[23](#mod-fs-ref-23)</sup>。`vfs_fcntl` 实现了 20+ 种操作，包括管道容量查询/设置（`F_GETPIPE_SZ`/`F_SETPIPE_SZ`）、seal 操作（`F_GET_SEALS`/`F_ADD_SEALS`）、所有者信号等 <sup>[24](#mod-fs-ref-24)</sup>。pipe 通过 `pipe_create`/`pipe_poll_events` 集成到 VFS poll 机制中 <sup>[25](#mod-fs-ref-25)</sup>。

### 5. 边角细节与不足

**风险 1：ext4 位图操作持锁分配内存**。`ext4_bitmap_alloc` 和 `ext4_bitmap_free` 在已持有 `alloc_lock` 时调用 `kmalloc(sb->block_size)`，若 kmalloc 触发内存回收导致睡眠，其他等待 `alloc_lock` 的进程会被长时间阻塞，形成潜在的优先级反转 <sup>[9](#mod-fs-ref-9)</sup>。

**风险 2：extent 树分裂路径的资源泄漏**。`ext4_extent_grow` 在创建两级索引时，如果先分配了 l1 块但后续 `kmalloc` 或 `kfree` 失败，已分配的块（l1、l2）通过局部变量跟踪并回滚；但在更深层的递归中（eh_depth > 1），函数直接返回 -ENOSPC 而不释放可能已分配的中间块 <sup>[6](#mod-fs-ref-6)</sup>。

**风险 3：procfs `generate_content` 中 snprintf 的截断处理**。多数 case 使用 `snprintf` 直接写入固定大小的栈缓冲区（`buf`，大小由调用者提供），如果输出超过 `bufsz`，数据被静默截断。`appendf` 辅助函数对此有保护（`*off = bufsz - 1`），但像 `PF_PID_STATUS` 这样的 case 直接 `snprintf(buf, bufsz, ...)` 后，如果实际内容超过 bufsz 不会返回错误码给用户 <sup>[26](#mod-fs-ref-26)</sup>。

**风险 4：VMA 文件路径解析的竞态条件**。`generate_pid_maps` 中通过 `v->file_fd` 索引 `t->files->fd[]` 获取全局 fd，再通过 `vfs_get_file` 获取 vfile。如果 VMA 建立后进程关闭/重用了该 fd，`vfs_get_file` 返回的可能不是同一文件（甚至可能为 NULL），导致 `maps` 输出与实际内存映射不一致 <sup>[17](#mod-fs-ref-17)</sup>。

**风险 5：ext4 短生命周期 vnode 模式下的父引用计数**。`ext4_make_vnode` 中如果传入了 parent vnode 会执行 `vnode_get(parent)`，但 `ext4_vnode_cache_remove` 是空函数——vnode 释放时无法通知缓存，缓存查询永远返回 NULL。这不会导致错误（因为缓存已被禁用），但代码注释暗示将来可能重新启用缓存，届时需要完整的生命周期管理 <sup>[3](#mod-fs-ref-3)</sup>。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制模块实现了 POSIX 兼容的信号处理，核心抽象是 `signal_state_t` 结构体（每个进程一个，包含动作表、pending 位掩码和排队信息）。它解决了用户态与内核态信号异步传递、处理函数执行、掩码管理等问题。与 xv6 原版相比，增加了 SA_SIGINFO、实时信号、信号栈、线程组发送、signalfd 等 Linux ABI 特性，并通过 `signal_deliver_user` 在陷入边界同步传递信号，避免了信号丢失。

### 1. 核心抽象与外部依赖

信号机制的核心数据结构是 `signal_state_t`（`kernel/include/proc/signal.h:43-48`），它嵌入在每个进程的 `task_t` 中（通过 `signals` 指针引用）。该结构体包含一个 `sigaction_t` 数组（NSIG 即 64 个）、两个 pending 位掩码（进程级和线程级）、以及每个信号的排队信息缓冲区（最多 128 字节）。设计上采用位图而非链表，简化了遍历和掩码操作，但代价是无法支持同一信号的多次排队（只保留最新信息）<sup>[1](#mod-signal-ref-1)</sup>。

关键系统调用通过 `kernel/abi/linux/sys_signal.c` 中的 `sys_kill`、`sys_tgkill`、`sys_sigaction` 等暴露给用户，它们调用 `kernel/proc/signal.c` 中的内部实现函数（如 `signal_send_info`、`sys_sigaction_impl`）。用户态信号传递依赖于陷入边界：`trap_handler` 在每次返回用户态前调用 `signal_deliver_user`，确保信号在指令边界同步传递<sup>[2](#mod-signal-ref-2)</sup>。架构相关部分（如 mcontext 读写、跳板代码）由 `arch_signal_*` 函数抽象，在 `kernel/arch/riscv64/signal.c` 中实现，保证了可移植性<sup>[3](#mod-signal-ref-3)</sup>。

### 2. 关键设计取舍

**位掩码与排队**：内核内部使用位位置等于信号编号的掩码（bit 0 未使用），与用户空间 ABI 不同（信号 N 对应 bit N-1）。转换函数 `signal_mask_from_user` 和 `signal_mask_to_user` 在 `signal.h:68-71` 定义，边界处进行移位。这种设计避免了遍历循环，提高了掩码操作效率，但增加了 syscall 层的转换成本<sup>[4](#mod-signal-ref-4)</sup>。排队信息数组 `pending_info[NSIG][SIGNAL_INFO_SIZE]` 固定 128 字节，意味着同一信号多次发送时，只有最后一次的信息被保留；对于 Linux 标准来说这是简化的，但符合内核初期的需求。

**同步传递点**：信号不在发送时立即传递，而是利用 `signal_task_has_unblocked` 在调度或陷入前检查，最终在 `signal_deliver_user`（用户态）或 `signal_deliver`（内核线程）中集中处理。这种设计避免了递归锁和重入问题，但牺牲了实时性——信号可能滞后若干指令。对于 SA_NODEFER 和嵌套信号，代码在 `signal_deliver_user` 中先更新掩码再设置 `sig_handling`，用注释说明了防止时钟中断嵌套的理由（`signal.c:398-403`）<sup>[5](#mod-signal-ref-5)</sup>。

**特权检查与权限**：`signal_send_permission` 函数在 `sys_kill` 等入口处检查调用者是否有 `CAP_KILL` 或 uid 匹配。对于 pid <= -1 的循环，它遍历所有进程并基于权限过滤，若全部失败则返回 -ESRCH（`sys_signal.c:66-75`）<sup>[6](#mod-signal-ref-6)</sup>。这种实现比列表遍历更完整，但 O(n) 扫描在进程数多时效率低，且未考虑 RCU 或锁，多核下可能不安全。

**信号栈与可执行权限**：`signal_make_page_exec` 在传递信号时动态将包含跳板的用户栈页设为可执行。由于用户栈通常没有 VM_EXEC，而信号跳板放在栈上，必须临时添加权限。它通过 `pt_unmap`/`pt_map` 切换，并刷新 TLB。这种做法能在页粒度控制权限，但每传递一个信号就进行页表操作，性能开销大，且未考虑栈页面可能与其他进程共享的情况（`signal.c:12-26`）<sup>[7](#mod-signal-ref-7)</sup>。

### 3. 跨模块协同

**与进程管理**：信号发送函数 `signal_send_info` 在目标进程被阻塞时调用 `proc_make_ready` 将其唤醒（`signal.c:64`），并与 `proc_block_until`（用于 sigsuspend/sigtimedwait）配合。`proc_force_exit` 和 `proc_exit_group` 在信号默认终止时被调用来结束进程。这些函数在 `kernel/proc/proc.c` 中实现，信号模块通过函数调用直接使用，未经过 IPC 或 lock 辅助，因此必须保证相关锁已在合适时持用。但当前代码中未看到显式锁保护 `signal_state_t`，在多核下 `pending` 和 `pending_info` 可能被并发写入，存在竞态<sup>[8](#mod-signal-ref-8)</sup>。

**与陷入处理**：`signal_deliver_user` 在 `trap_handler` 的返回路径被调用，它修改 `trap_context_t` 中的 sp、epc、a0-a2、ra 等寄存器，以模拟用户态信号处理函数的执行。当信号处理完成后，`int64_t sys_rt_sigreturn_impl` 被用户调用，从栈上的 `sig_rt_frame_t` 恢复上下文。`build_ucontext` 填充异常上下文和信号掩码，与 `arch_signal_build_mcontext` 及 `arch_signal_restore_mcontext` 协同，形成完整的保存/恢复契约（`signal.c:300-320`）<sup>[9](#mod-signal-ref-9)</sup>。

**与页表系统**：`signal_make_page_exec` 直接调用 `pt_translate`、`pt_unmap`、`pt_map` 和 `arch_tlb_flush_page`。这意味着信号模块与虚拟内存子系统紧密耦合，而非通过标准接口。一旦页表实现变化，该函数必须同步修改。此外，`copy_to_user` 和 `copy_from_user` 用于在栈帧之间拷贝数据，这些函数依赖于用户页表是否有效，但未显式检查 `t->pgdir` 是否为 NULL（虽然在调用前已有判断）。

### 4. 边角细节与不足

1. **指针与内存安全**：`signal_send_info` 中对于 info 的拷贝直接 `memcpy(ss->pending_info[signum], info, n)`，未检查 `signum` 是否为有效信号？实际上函数开头已检查 `signum <= 0 || signum >= NSIG`，所以安全。但 `build_siginfo_code` 中访问 `_sifields[0..2]` 时未验证 sender 是否为 NULL？代码中 `if (sender)` 才赋值，所以安全。不足：`signal_send_thread` 中 `memcpy` 时未使用 `SIGNAL_INFO_SIZE` 而是 `sizeof(si)`，但 si 恰好为 128 字节？实际上 `siginfo_t` 大小为 4+4+4+29*4=8+116=124？结构体可能包含对齐填充，但 `SIGNAL_INFO_SIZE` 定义为 128，所以拷贝 `sizeof(si)` 可能不足 128，剩余的 0 已在前面用 `memset` 初始化，安全。但`signal_send_info` 中如果没有 info，则 `*(int *)ss->pending_info[signum] = signum;` 只写前 4 字节，后 124 字节未清零？但之前有 `memset(ss->pending_info[signum], 0, SIGNAL_INFO_SIZE);` 所以清零了。整体看指针使用安全。<sup>[10](#mod-signal-ref-10)</sup>

2. **锁与中断上下文**：模块多处访问 `task_t` 和 `signal_state_t` 字段（如 `pending`, `sig_blocked`, `thread_pending`），但均未使用自旋锁或类似机制。例如在 `signal_send_info` 中，对 `ss->pending` 的读-修改-写（`ss->pending |= signal_mask_bit(signum);`）非原子；这可能在多核上导致丢失信号。此外，`signal_deliver_user` 在循环中多次修改 `ss->pending`，同样缺乏保护。虽然可能假设信号模块在中断关闭下运行或只被单核访问，但从代码中并未看到显式的屏障或锁，这是一个潜在竞争条件。<sup>[8](#mod-signal-ref-8)</sup>

3. **整数溢出**：`signal_mask_bit(sig)` 在 sig=63 时左移 63 位，产生 `1ULL << 63` 合法。但 `signal_mask_from_user` 将用户掩码左移一位，如果用户掩码最高位（bit 63）为 1，则移位后超出 64 位范围？由于被封装在 `uint64_t` 中，左移导致未定义行为（C 标准规定左移达到或超过宽度是 UB）。但实际上用户掩码来自用户空间，如果用户设置了 64 位信号（实际信号号最大 31，但 NSIG=64），系统调用会限制位移量不超过 63，所以用户不会设置最高位（因为信号编号最大 31，对应的 bit 30）。但 Linux ABI 使用 64 位掩码，bit 31-63 理论可用，但本内核 NSIG=64，信号编号 1-63，用户可能对应 bit 0-62。因此 `signal_mask_from_user` 在用户设置 bit 63 时会导致 `user_mask << 1` UB。不过在内核中需要避免。解决方法是使用 `user_mask << 1ULL` 或用 `user_mask << 1`，但 C 标准不允许 64 位移位。应添加检查或使用更安全方式。<sup>[11](#mod-signal-ref-11)</sup>

4. **错误码处理**：`signal_send_info` 中调用 `proc_force_exit` 后直接返回 0，忽略了 `proc_force_exit` 可能失败（虽然通常不返回）。`sys_sigaction` 中 `copy_to_user` 失败时返回 -EFAULT，但 `copy_from_user` 失败时同样返回 -EFAULT，符合预期。但在 `sys_sigtimedwait` 中，用户传入的 timeout 为 NULL 时，设置了 `until=0`（表示不阻塞），然后 `proc_block_until(t, 0)`，这就变成了永久阻塞直到有信号？实际上 `proc_block_until` 的 0 可能表示没有超时，永久等待。但文档说如果 timeout 为 NULL 则等待任意信号，这符合 POSIX。但若 timeout 非空但 sec 和 nsec 都为 0，则返回 -EAGAIN，正确。不过 `until` 的计算可能溢出：`sec * TICKS_PER_SEC + nsec * TICKS_PER_SEC / 1000000000ULL`，如果 sec 很大可能溢出 uint64_t，但实际中 sec 受用户传值限制，不过仍需考虑。<sup>[12](#mod-signal-ref-12)</sup>

5. **页表/地址空间**：`signal_make_page_exec` 中，`pt_translate` 返回的物理地址 pa 为 0 时直接返回，没有判断是否为有效页（如内核页表？），但栈页通常映射有效，所以风险较小。另外，该函数未检查 user 栈是否在有效地址范围内，如果是用户栈的某个栈页，可能因为映射不存在而返回 0。更严重的是，`pt_unmap` 和 `pt_map` 没有检查返回值，如果 `pt_unmap` 失败，后续 `pt_map` 可能覆盖错误的页表项。<sup>[7](#mod-signal-ref-7)</sup>

6. **硬编码常量**：`SIGNAL_INFO_SIZE` 硬编码为 128（`signal.h:10`），与 Linux 的 `struct siginfo_t` 大小（128 字节）一致，但架构相关的扩展（如 x86 的 `_fpxregs`）可能需要更大，当前未考虑。另外 `MINSIGSTKSZ` 为 2048（`signal.h:118`），符合 glibc 最小值，但可能不满足某些架构硬件需求。`NSIG` 定义为 64（`signal.h:9`），与 Linux 用户空间定义一致，但内部位图用 64 位，实际仅支持信号 1-63（信号 0 不用）。<sup>[13](#mod-signal-ref-13)</sup>

7. **TODO/桩函数**：未发现明显的 TODO 或桩函数。但 `signal_deliver` 中对用户进程直接 return（`if (is_user) return;`），说明用户进程的信号传递完全由 `signal_deliver_user` 处理，这是设计而非未完成。不过 `signal_deliver` 中的 `sched()` 调用后未处理调度器返回的可能，但对于内核线程停止后重新调度是合理的。<sup>[14](#mod-signal-ref-14)</sup>

8. **并发安全**：`signal_state_t` 中的 `pending` 字段被多个发送者（进程间 signal_send_info）和接收者（信号传递函数）同时访问，未使用原子操作或锁。`signal_send_info` 在设置 `ss->pending |= ...` 时可能丢失其他 CPU 上同时的发送。建议使用 `__sync_or_and_fetch` 或 `atomic_clear_bit` 等。（参见第 2 点）。<sup>[8](#mod-signal-ref-8)</sup>

9. **数组边界**：`actions[NSIG]` 使用信号号直接索引，且 `signal_send_info` 等函数已检查范围，故安全。`pending_has_info` 和 `pending_info` 同样以信号号索引，超出部分未访问。<sup>[15](#mod-signal-ref-15)</sup>

10. **死代码**：`signal_core_dump_default` 在 `signal_wait_status` 中用于判断是否产生 core dump，但仅用于生成退出状态，未使用该状态在实际中触发 core dump，即未调用 dump 函数。这可能是一个死逻辑，或者未来才实现核心转储。目前该函数仅用于设置状态中的 0x80 位，但并未真正保存 core 文件。<sup>[16](#mod-signal-ref-16)</sup>

### 5. 总结

A20OS 的信号机制在 xv6 基础上大幅扩展，实现了 Linux 风格的信号系统，包括 `rt_sigaction`、线程组信号、排队信息、信号栈等。其设计围绕同步传递点展开，减少了异步路径的复杂度。但当前实现存在显著的并发缺陷（缺少锁）、少量未定义行为以及效率问题，需要在多核环境中进一步加固。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现多种 IPC 机制：System V 共享内存（基于页表映射和引用计数延迟删除）与信号量（基于等待队列和超时重试）；Linux 兼容的 timerfd/eventfd（通过 VFS 文件描述符暴露）；自研 A20 原生 IPC（事件队列和通道，基于句柄类型系统和哈希监视）。相比 xv6 仅有的管道和共享页，该模块提供了更丰富的 IPC 原语，但部分实现处于早期，存在桩函数和未完成的阻塞等待。

### 1. 核心抽象：三类 IPC 机制

该模块的进程间通信设施可划分为三个层次。**System V 兼容层**（`sysv_shm.c`, `sysv_sem.c`）提供共享内存和信号量。共享内存的核心是 `sysv_shm_t` 结构，包含 `pfn_t *pages` 物理页帧数组和 `nattach` 引用计数 <sup>[1](#mod-ipc-ref-1)</sup>；信号量以 `sysv_sem_set_t` 为单位，每个集合最多 64 个信号量值，并内嵌 `wait_queue_t` 用于阻塞等待 <sup>[3](#mod-ipc-ref-3)</sup>。**Linux 兼容层**（`timerfd.c`, `eventfd.c`）将计时器和事件计数器包装为文件描述符，通过 `vfile_ops` 的 read/write 实现用户态接口 <sup>[6](#mod-ipc-ref-6)</sup>。**自研 A20 原生 IPC**（`a20_event.c`, `a20_channel.c`）提供事件队列和双向通道：事件队列通过全局哈希表 `g_evq_hash` 将对象指针映射到监视条目，支持任意内核对象的事件通知 <sup>[5](#mod-ipc-ref-5)</sup>；通道采用双端点（peer）设计，消息队列支持数据+句柄传递 <sup>[4](#mod-ipc-ref-4)</sup>。

### 2. 关键设计取舍

**共享内存的延迟删除**：`sysv_shm_control(IPC_RMID)` 仅设置 `marked_delete` 标志，真正释放物理页面的时机推迟到 `nattach` 归零时 <sup>[2](#mod-ipc-ref-2)</sup>。这避免了进程尚在映射时物理内存被回收，但代价是 `shmid` 槽位在删除后仍被占用（`used` 仍为 1），直到 `nattach==0` 才清空。

**信号量超时睡眠循环**：`sysv_sem_timedop` 采用“检查—阻塞—重试”循环。当操作无法立即满足时，进程通过 `wait_queue` 挂起，并设置 `proc_set_wake_time` 用于超时唤醒 <sup>[4](#mod-ipc-ref-4)</sup>。缺陷是 `deadline` 的比较使用 `int64_t` 可能溢出，且 `GETNCNT`/`GETZCNT` 直接返回 0 未实现等待计数 <sup>[7](#mod-ipc-ref-7)</sup>。

**A20 类型安全**：通道在发送和接收时分别调用 `ch_check_send_types` / `ch_check_recv_types`，检查句柄类型是否被通道类型位掩码允许 <sup>[4](#mod-ipc-ref-4)</sup>。这是微内核 IPC 的典型设计，防止类型混淆。但接收端的类型检查在消息已出队后才执行，若检查失败则消息丢失 <sup>[10](#mod-ipc-ref-10)</sup>。

**eventfd 信号量模式**：通过 `flags & 1` 开启信号量模式，每次读取返回 1 而非全部计数器值 <sup>[6](#mod-ipc-ref-6)</sup>。写操作会在计数器溢出前阻塞（`counter + val > ~0ULL - 1`），使用独立的读者和写者等待队列。

### 3. 跨模块协同

与**虚拟内存系统**的协同最为紧密：`sysv_shm_at` 通过 `pt_map` 将共享内存物理页映射到进程地址空间，使用 `mm_find_gap` 查找可用地址，`mm_insert_vma` 注册 VMA，并在拆除时调用 `pt_unmap_leaf` 和 `frame_put` <sup>[2](#mod-ipc-ref-2)</sup>。与**文件系统**的协同体现在 timerfd/eventfd 通过 `anonfd_install_vfile` 创建文件描述符，使用 `vfs_get_file_ref` / `vfs_put_file_ref` 管理引用计数 <sup>[6](#mod-ipc-ref-6)</sup>。与**进程调度**的协同：所有阻塞原语都基于 `wait_queue_sleep` / `wait_queue_wake_one`，并通过 `proc_set_wake_time` 设置定时唤醒 <sup>[4](#mod-ipc-ref-4)</sup>。与**信号处理**的协同：`sysv_sem_timedop` 在阻塞前检查 `signal_task_has_unblocked`，以响应信号中断。

### 4. 边角细节与不足

- **桩函数**：`sysv_sem_control` 的 `GETNCNT`/`GETZCNT` 直接返回 0 <sup>[7](#mod-ipc-ref-7)</sup>；`sysv_shm_control` 的 `SHM_INFO` 和 `SHM_STAT_ANY` 返回全零结构体 <sup>[9](#mod-ipc-ref-9)</sup>；`sysv_sem_control` 的 `IPC_STAT`/`SEM_STAT` 同样返回零数据。这些功能尚未实现。
- **A20 事件队列阻塞缺失**：`a20_eventq_wait` 在环为空且 `timeout_ns != 0` 时直接返回 `-A20_ERR_WOULD_BLOCK`，未实现真正的阻塞等待 <sup>[8](#mod-ipc-ref-8)</sup>。
- **A20 通道接收消息丢失**：`a20_channel_recv` 先将消息出队（更新 `msg_head`），然后进行 `ch_check_recv_types` 类型检查；若检查失败，消息已被移除且不可恢复 <sup>[10](#mod-ipc-ref-10)</sup>。
- **timerfd 锁循环风险**：`timerfd_read` 在 `spin_lock` 内使用 `wait_queue_sleep`，必须先 `spin_unlock` 再睡眠，然后重新获得锁。当前代码确实这样做了，但若 `wait_queue_sleep` 的实现不符合预期（例如不在唤醒时重新获得调用者期待的锁），会引入死锁 <sup>[6](#mod-ipc-ref-6)</sup>。
- **硬编码限制**：`SYSV_SHM_MAX=32`、`SHM_MAX_PAGES=1024`、`SYSV_SEM_MAX=32`、`SYSV_SEM_PER_SET=64` 均为固定常量，无动态扩容机制 <sup>[1](#mod-ipc-ref-1)</sup>。
- **整数边界**：`sysv_shm_get` 中 `size == 0` 被拒绝，但 `size > SHM_MAX_PAGES * PAGE_SIZE` 检查后若 `PAGE_SIZE` 不是常量乘积可溢出（此处安全）；`eventfd_write` 中 `efd->counter + val > ~0ULL - 1` 可防止溢出，但 `~0ULL` 是 `0xFFFFFFFFFFFFFFFF`，减 1 后为 `0xFFFFFFFFFFFFFFFE`，逻辑正确。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 该模块基于 lwIP 协议栈实现了类 POSIX 的 socket API，支持 AF_INET、AF_INET6、AF_UNIX 和 AF_ALG 域。核心抽象是 net_socket_t 结构体，它封装了 lwIP 的协议控制块（TCP/UDP/RAW）以及 socket 状态，并通过无锁底部半环（bh_ring）在中断上下文和进程上下文中安全传递数据。与 xv6 原版相比，此实现大幅扩展了网络功能，支持完整的 IPv4/IPv6、多播、套接字选项、BPF 过滤（存根）以及加密套接字（AF_ALG）。关键设计取舍包括两锁分离（g_net_lock/g_lwip_lock）、TCP 监听不切换 lwIP LISTEN PCB，以及原子 ring 的无锁通信。

### 1. 核心抽象：net_socket_t 与双缓冲消息传递

网络模块的核心数据结构是 `net_socket_t`（定义在 `socket_internal.h`，使用见 `socket.c` 第 10 行），它同时扮演两个角色：内核 socket 层状态机（domain、type、bound、listening、connected 等标志）和 lwIP 协议控制块的包装器（`s->tcp`/`s->udp`/`s->raw`）。为了在 lwIP 回调上下文（可能处于中断或持 `g_lwip_lock` 的状态）与 socket 系统调用上下文（持 `g_net_lock`）之间安全高效地传递接收数据，引入了底部半环（bottom-half ring）机制`<sup>[1](#mod-net-ref-1)</sup>`。其核心数据结构 `net_bh_ring_t` 包含一个固定大小的 `net_bh_event_t` 数组及原子操作的 head/tail 索引。lwIP 回调（如 `lwip_udp_recv_cb`、`lwip_tcp_recv_cb`）在 `g_lwip_lock` 保护下调用 `bh_ring_prepare` 和 `bh_ring_commit` 将数据分段写入 ring，然后设置 `bh_pending` 标志；socket 下半部分 `net_inet_bottom_half_process_socket_locked`（在 `g_net_lock` 下运行）调用 `bh_ring_consume` 取出事件，将数据复制到 `net_msg_t` 消息队列，并唤醒阻塞的进程。这种设计避免了在两个锁之间嵌套，同时利用原子操作实现了单生产者单消费者的无锁传递，在不增加锁竞争的前提下保证了 SMP 安全性。

### 2. 两锁分离与设计取舍

网络模块采用显式的两锁方案：`g_net_lock` 保护 socket 状态、消息队列和 socket 表，`g_lwip_lock` 保护 lwIP 核心状态。锁的使用规则是：lwIP 回调从不尝试获取 `g_net_lock`；socket 层在持 `g_net_lock` 时不会直接调用 lwIP API（初始化等场景除外，此时先持 `g_lwip_lock`）。这一约定从源代码注释中明确说明`<sup>[2](#mod-net-ref-2)</sup>`。

一个突出的设计取舍是 TCP 监听状态的处理。`net_listen` 函数特意避免将 TCP PCB 转换为 lwIP 的 LISTEN 状态，而是保持为普通 bound PCB，由 socket 层自己维护 accept 队列。注释指出，这是为了应对 LTP（Linux Test Project）中 close-after-accept 测试场景，保证在监听套接字关闭后已经接受但尚未返回给用户空间的子套接字仍能正常生存，从而获得确定的关闭语义`<sup>[3](#mod-net-ref-3)</sup>`。这一选择的代价是失去了 lwIP 内部对 SYN 洪泛的 SYN cookie 保护，但考虑到是教学/实验内核，优先保证兼容性是正确的。

另一个关键设计是字节序处理。`net_sockaddr_loopback` 中直接赋值 `in.sin_addr = 0x0100007fU`，隐含了主机字节序为小端的假设`<sup>[5](#mod-net-ref-5)</sup>`。`net_htons`/`net_ntohs` 也通过字节交换实现，假定主机是小端。整个模块缺少对大端平台的适配层，虽然目前 RISC-V 和 x86 均为小端，但代码中未使用 `htons`/`ntohl` 等标准函数，可移植性存在隐患。

### 3. 跨模块协同与阻塞模型

网络模块与进程调度、文件系统和信号机制紧密协作。阻塞 I/O 通过 `net_block_on_socket_locked` 实现：将当前任务状态设为 `PROC_BLOCKED`，设置 `s->waiter` 指针，并通过 `proc_set_wake_time` 设置超时。当数据到达或连接建立时，下半部分或对端操作调用 `proc_make_ready` 唤醒等待任务`<sup>[4](#mod-net-ref-4)</sup>`。超时检测通过 `net_socket_wait_expired` 比较时间戳实现，避免了定时间隔的精度损失。

套接字作为文件描述符，通过 `net_socket_install_file` 与 VFS 层集成，`net_socket_from_file` 从文件描述符反向查找套接字。`net_poll_events` 实现了 `poll`/`select` 的支持，检查 `rx_head`、`accept_head`、`closed` 等状态。此外，模块通过 `a20_lwip_poll()` 驱动 lwIP 协议栈的定时器和收包处理，该函数由内核的空闲循环或时钟中断周期性调用，从而将 lwIP 的单线程进度模型融入内核的事件循环。

### 4. 边角细节与不足

- **临时端口分配缺少锁保护**：`net_alloc_ephemeral_port_locked` 函数名暗示调用者应持有 `g_net_lock`，但在 `net_bind` 中，它在获取 `g_net_lock` 之前就被调用（`socket.c` 第 270 行附近）。这可能导致多核下端口重复分配`<sup>[6](#mod-net-ref-6)</sup>`。
- **`net_recvfrom_meta` 持锁循环读取**：该函数在持有 `g_net_lock` 的情况下，循环从 `rx_head` 中读取多个消息以填充用户缓冲区（SOCK_STREAM 路径）。如果用户缓冲区很大且消息很多，锁持有时间可能很长，阻塞其他 socket 操作`<sup>[7](#mod-net-ref-7)</sup>`。
- **`net_sendto` 中 TCP 大数据的部分发送问题**：当 TCP 套接字发送大于 `NET_MAX_PAYLOAD` 的数据时，代码将其分割成多个 `net_enqueue_msg_blocking` 调用。若中途发生错误，函数返回已发送的字节数，但调用者难以处理未发送的残留数据，且该路径未考虑 SIGPIPE 信号`<sup>[8](#mod-net-ref-8)</sup>`。
- **`net_shutdown` 的 SHUT_RD 数据丢失**：在 SHUT_RD 模式下，代码立即清空 `rx_head` 队列并释放所有消息，但未通知正在等待读取的进程，可能导致应用层数据丢失`<sup>[9](#mod-net-ref-9)</sup>`。
- **全局 socket 表大小固定**：`NET_MAX_SOCKETS` 决定了最大套接字数，无法动态扩容，在多服务场景下可能成为瓶颈。
- **错误码一致性**：某些错误路径返回 `-1` 而非负的错误码（例如 `net_socket_create` 中 `net_register_socket_locked` 失败后的清理路径），但上层调用者可能期望标准的 Linux 错误码。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块实现了一个分层设备驱动模型，核心抽象为 `driver_t`（驱动描述符，包含 probe/remove 回调、设备 ID 表和 class_ops 类操作表）、`device_t`（设备实例，承载 drv_priv 私有数据指针和资源列表）、以及 `DEV_CLASS_BLOCK`/`DEV_CLASS_NET` 等设备类别。框架通过 `DRIVER_REGISTER()` 宏将驱动静态注册到内核，由总线探测或平台初始化时调用 probe 回调完成设备识别与实例化。与 xv6 原版相比，该框架显式分离了驱动层（`driver_t`）与设备类接口层（`block_dev_ops_t`/`net_dev_ops_t`），并引入了锁层级注释规范（`LOCK_ORDER`），但部分驱动（StarFive GMAC、DW SDIO）仍使用无锁全局单例，是 SMP 安全的隐患。

### 1. 核心抽象：driver_t 与设备类操作表

驱动框架围绕 `driver_t` 结构体展开<sup>[2](#mod-drivers-ref-2)</sup>。每个驱动（如 virtio-blk、virtio-net、starfive-gmac）声明一个静态的 `driver_t` 实例，填充 `name`、`id_table`（PCI/MMIO 设备 ID 匹配表）、`probe` 和 `remove` 回调，以及 `class_ops` 指针与 `class_type` 枚举。`probe` 回调接受 `device_t *dev`，从中提取 MMIO 基址与 IRQ 资源（通过 `device_get_resource`），初始化设备实例，并将其挂接到 `dev->drv_priv`。`class_ops` 是一个设备类专用的函数指针表——`block_dev_ops_t` 包含 `read`/`write`/`capacity`/`sector_size`<sup>[1](#mod-drivers-ref-1)</sup>，`net_dev_ops_t` 包含 `send`/`recv`/`mac`/`poll`<sup>[6](#mod-drivers-ref-6)</sup>。这种设计与 Linux 内核的 `file_operations` 思路一致，将同一类设备的操作多态化，上层文件系统或网络协议栈只需面对统一的 `block_dev_t`/`net_dev_ops_t`，无需感知底层是 VirtIO、GMAC 还是 SDIO。

`DRIVER_REGISTER()` 宏在编译期将驱动静态注册到一个全局链表中，由内核启动阶段的 `driver_init()` 遍历调用 probe。这种静态注册避免了动态内存分配和模块加载器的复杂性，适合教学内核，但代价是驱动数量上限受限于编译期数组（如 `g_insts[VIRTIO_MAX_DEVS]`<sup>[8](#mod-drivers-ref-8)</sup>），且无法在运行时卸载。

### 2. 关键设计取舍：同步轮询 vs 中断驱动与锁层级契约

该框架的显著特点是在单个驱动中混合了中断驱动与同步轮询两种完成模型。VirtIO 块设备的中断处理函数 `virtio_blk_irq_handler`<sup>[3](#mod-drivers-ref-3)</sup> 在关中断下持有 `inst->lock`，调用 `virtio_blk_complete_used_locked` 遍历 used ring，标记 `req->done` 并通过 `proc_make_ready()` 唤醒阻塞的进程。同时，`virtio_blk_wait_req` 在等待超时时会主动调用 `virtio_blk_complete_used_locked` 轮询，形成“中断优先、轮询保底”的策略。`virtio_blk_poll_all()` 则作为调度器 `kernel_progress_poll()` 的兼容桥接<sup>[4](#mod-drivers-ref-4)</sup>。这种冗余设计的原因在注释中已说明：“目标模型是 IRQ 或 bottom-half 完成”，当前实现是过渡期产物。

锁层级是该框架的另一核心设计。每个驱动实例的私有锁（`inst->lock`/`net->lock`）被标记为 innermost，并明确写出锁顺序<sup>[1](#mod-drivers-ref-1)</sup>。VirtIO 网络驱动额外声明 `g_lwip_lock → net->lock` 的全局顺序<sup>[7](#mod-drivers-ref-7)</sup>，防止 IRQ 处理中调用 lwIP 回调时反向加锁导致死锁。这是 C 内核中典型的“锁顺序文档化”实践，与 xv6 的隐式锁顺序相比更为显式。

StarFive GMAC 和 DW SDIO 驱动则选择了完全不同的路径：两者均使用全局单例 `g_gmac`/`g_sdio`，文件头注释明确写着“没有私有自旋锁”<sup>[5](#mod-drivers-ref-5)</sup>，所有收发操作以同步轮询（busy-poll）方式运行。这种设计在单核平台上可以工作，但在 SMP 下 `g_gmac.tx_busy` 和 `g_gmac.rx_busy` 的读写缺乏原子性，存在竞态。

### 3. 跨模块协同：内存屏障、DMA 同步与调度器唤醒

驱动框架与内核其他模块的协同体现在三个关键路径上。

**内存管理与 DMA 同步**：所有 VirtIO 队列描述符地址在提交前必须通过 `va_to_pa()` 转换为物理地址<sup>[1](#mod-drivers-ref-1)</sup>，因为设备通过 IOMMU 或直接物理寻址访问内存。驱动在修改描述符环后调用 `arch_dma_sync_for_device()`，在读取完成状态前调用 `arch_dma_sync_for_cpu()`，确保缓存一致性。这是一个极容易遗漏的环节——若缺少某个 `arch_dma_sync_for_device` 调用（如 `virtio_blk_submit_req` 中连续 9 次显式同步），设备可能读到过时的描述符。

**进程调度与阻塞 I/O**：`virtio_blk_submit_req` 将当前进程（`proc_current()`）记录到 `req->waiter`，然后调用 `proc_set_wake_time` 和 `sched()` 主动让出 CPU。中断处理函数中调用 `proc_make_ready(req->waiter)` 将其唤醒<sup>[3](#mod-drivers-ref-3)</sup>。这里有一个精妙的竞态处理：`virtio_blk_wait_req` 在进入 `PROC_BLOCKED` 前再次检查 `req->done`，防止请求在标记阻塞前已经完成导致丢失唤醒<sup>[4](#mod-drivers-ref-4)</sup>。

**网络协议栈集成**：`virtio_net_irq_handler` 不直接触及 virtio 队列，而是获取 `g_lwip_lock` 后调用 `a20_lwip_process_netif_irq_locked`，将数据交付给 lwIP。`virtio_net_send` 在 `nonblock=1` 模式下提交后立即返回，避免在持有 `g_lwip_lock` 时阻塞等待 TX 完成（否则会死锁：`g_lwip_lock → virtio_net_send → 等待完成 → IRQ → lwIP 回调 → 需要 g_lwip_lock`）<sup>[7](#mod-drivers-ref-7)</sup>。

### 4. 边角细节与不足

**<sup>[5](#mod-drivers-ref-5)</sup> 无锁全局单例**：StarFive GMAC 和 DW SDIO 驱动使用 `static g_gmac`/`g_sdio` 全局变量，所有 TX/RX 操作直接读写全局状态，无自旋锁保护。注释承认“未来 SMP 版本必须加锁”，但当前若在多核上运行，`tx_busy` 和 `rx_busy` 索引的更新可能被中断或另一核破坏。

**<sup>[3](#mod-drivers-ref-3)</sup> 中断注册失败不阻断**：`virtio_blk_driver_probe` 和 `virtio_net_driver_probe` 中 `request_irq` 失败后仅 `printf` 警告，`probe` 仍返回 0 表示成功。这意味着设备可能在没有中断的情况下运行，完全依赖 `virtio_blk_poll_all()` 轮询，而轮询的触发时机由调度器决定，可能导致 I/O 延迟剧增。

**<sup>[8](#mod-drivers-ref-8)</sup> 定长数组与边界检查不统一**：`g_insts[VIRTIO_MAX_DEVS]` 和 `g_net[VIRTIO_NET_MAX_DEVS]` 编译期固定上限。`virtio_blk_read` 的 `idx` 参数使用 `if (idx < 0 || idx >= g_ninst)` 检查，但 `virtio_blk_ready` 中同样检查却访问 `g_insts[idx]`——如果 `g_ninst` 与 `VIRTIO_MAX_DEVS` 不同，数组仍可能越界。更隐蔽的是 `virtio_blk_alloc_req_locked` 中 `req->head = (uint16_t)(i * 3)`，其中 `i` 最大为 `VIRTIO_BLK_REQ_SLOTS - 1`，若 `VIRTIO_QUEUE_SIZE` 不是 3 的倍数，则最后一个请求的 head 可能占用超出正常范围的描述符槽位。

**<sup>[4](#mod-drivers-ref-4)</sup> 可用环标志硬编码**：`virtio_blk_submit_req` 在每次提交时设置 `avail->flags = 1`（即 `VIRTQ_AVAIL_F_NO_INTERRUPT`）。这意味着驱动程序主动抑制了可用环更新时产生的中断，与 IRQ 处理器的设计矛盾——既然开启了中断，就不应设置 NO_INTERRUPT。实际可能是为了兼容性，但未留下注释解释这一矛盾的取舍。

**<sup>[6](#mod-drivers-ref-6)</sup> 回退 MAC 地址硬编码**：`virtio_net_init_instance` 在未获取到设备 MAC 时，使用 `{0x02, 0x20, 0x25, 0xa2, 0x00, idx}` 作为回退 MAC。若系统中存在多个 virtio-net 设备，idx 仅能保证最后一字节不同，存在 MAC 冲突风险。StarFive GMAC 同样使用硬编码 MAC `{0x00, 0x55, 0x7B, 0xB5, 0x7D, 0xF7}`，未从设备树或 OTP 读取。

**<sup>[2](#mod-drivers-ref-2)</sup> `driver_t` 中 `bus` 字段始终为 NULL**：所有驱动实例的 `.bus = NULL`，表明框架尚未实现总线模型。设备探测目前依赖平台初始化函数（如 `arch_virtio_blk_probe`/`arch_virtio_net_probe`）硬编码调用，而非通过设备树或 PCI 枚举自动发现。这是框架的早期状态标志。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该系统调用层是A20OS内核的核心接口，通过syscall_dispatch分发用户态系统调用，支持原生ABI和Linux AB双表机制，并包含x86_64系统号转换。相比xv6，增加了性能剖析（条件编译的syscall_profile）、信号驱动的系统调用重试（-ERESTARTSYS处理）以及路径解析辅助函数syscall_path_at。核心抽象是dispatch函数基于当前任务abi_mode选择A20或Linux处理表，并在返回前处理信号和重调度。

### 1. 核心抽象与双ABI架构
系统调用层以`syscall_dispatch`为总入口，接收`trap_context_t`指针，从中提取系统调用号和参数。首先通过`syscall_profile_now`和`syscall_profile_record`进行性能计数（条件编译控制）。对于x86_64平台，通过`x86_syscall_to_kernel_nr`将原生系统调用号转换为内核内部号，未实现的返回-ENOSYS<sup>[1](#mod-syscall-ref-1)</sup>。

随后根据当前任务的`abi_mode`（通过`proc_current`获取）选择两套ABI处理路径：原生A20 ABI和Linux ABI。A20路径使用`a20_syscall_lookup`查找处理函数；Linux路径使用`linux_syscall_lookup`。这种双表设计使得内核能同时运行原生和Linux二进制，但增加了维护两套表的成本。

### 2. 关键设计取舍
**双表 vs 统一表**：作者选择了保留两套独立的函数指针表，而非将Linux syscall映射到A20内部，可能是为了保留A20 ABI的独立演进能力，同时利用Linux兼容性。代价是内核代码体积增大，且需要同时管理和更新两份表。

**信号恢复逻辑**：当Linux syscall返回-ERESTARTSYS时，dispatch会检查是否有未决信号且对应handler未设置SA_RESTART，决定重启还是返回-EINTR<sup>[3](#mod-syscall-ref-3)</sup>。该逻辑比xv6复杂，但更接近Linux行为。注意这里的`TRAP_CTX_EPC(ctx) -= 4`硬编码了指令长度，假设x86_64指令为4字节，不适用于可变长度指令。

**性能剖析**：使用`__atomic_fetch_add`进行无锁计数，但`syscall_prof_t`包含count和cycles两个字段，原子操作无法保证两者一致性，多核下可能观察到错配的count/cycles。

### 3. 跨模块协同
- **进程管理**：多次调用`proc_current`和`proc_yield`，并在最后进行`proc_check_exit_pending`（检查退出请求），确保系统调用返回前处理进程退出。
- **信号**：调用`signal_deliver_user`递送用户态信号，并在sigsuspend后恢复阻塞掩码<sup>[4](#mod-syscall-ref-4)</sup>。
- **VFS/文件系统**：`syscall_path_at`解析路径时与文件描述符表（`fdtable_get`、`vfs_get_file_ref`）和VFS节点交互，实现基于文件描述符的相对路径查找。

### 4. 边角细节与不足
- **<sup>[2](#mod-syscall-ref-2)</sup> 缓冲区溢出风险**：`syscall_path_at`使用固定大小`MAX_PATH_LEN`的栈缓冲区`logical`，若用户提供超长路径（例如`path`长度接近`outsz`但大于`MAX_PATH_LEN`），`strncpy`和`snprintf`可能导致越界写。`strncpy`在源串过长时不添加空字符，后续`strlen(base)`可能读越界。
- **<sup>[5](#mod-syscall-ref-5)</sup> 错误处理不一致**：`syscall_path_at`中`fdtable_get`返回全局文件描述符，但错误路径上可能遗漏`vfs_put_file_ref`。检查代码见行73-75：当`vf->vnode->type != VFS_FT_DIR`时，虽然调用了`vfs_put_file_ref`，但后续`strncpy(logical, vf->path, ...)`时`vf`已被释放，产生释放后使用。
- **<sup>[1](#mod-syscall-ref-1)</sup> 硬编码指令长度**：信号重试时`TRAP_CTX_EPC(ctx) -= 4`假设x86_64 syscall指令长度为4字节，不适用于x32或其他ABI。
- **<sup>[3](#mod-syscall-ref-3)</sup> 无锁统计不一致**：`syscall_profile_record`中使用`__atomic_fetch_add`单独更新count和cycles，两个原子操作之间可能被抢占，多核下profile数据错乱。
- **<sup>[4](#mod-syscall-ref-4)</sup> 缺少中断上下文保护**：`syscall_dispatch`末尾调用`proc_yield`和`signal_deliver_user`，未显式关中断，若被中断嵌套可能导致内核栈或进程表竞态。

系统调用表中有大量未实现条目仅打印`UNHANDLED`并返回ENOSYS，缺少对linux_syscall_lookup表项完整性的保障。
