## 📖 一、总览

该仓库是一个基于 Rust 的实验型内核，采用三层能力层（L0-L1-L2）和 need 依赖注入机制，旨在实现 Linux 应用接口兼容性。与 rCore-Tutorial 家族基线相比，它更强调模块化分离、显式的用户地址空间生命周期管理以及完整的信号递送框架，但整体处于早期开发阶段，大量系统调用为存根或未实现。

该仓库最值得评审注意的特点包括：
1. **自研 FDT 解析与三层启动管道**：从物理内存布局到用户进程启动分为 L0→L1→L2 三个独立层，每层暴露统一的 init() 接口，且 FDT 解析完全自实现而不依赖第三方 crate，体现了对引导流程的精细控制。
2. **显式的用户地址空间管理**：通过全局 slot 数组管理多个 AddressSpace，提供 new、clone、destroy 完整生命周期操作，并支持惰性缺页回调和写时克隆，比 rCore-Tutorial 的简单进程空间管理更接近 Linux 复杂度。
3. **信号框架的分层实现**：L1 层提供信号位图、信号动作和递送管道，L2 层封装 Linux 兼容系统调用接口，但缺少关键的 sigreturn 和用户态 handler 框架，导致信号无法实际执行用户处理函数。

## 🎯 二、综合评价

### 整体定位
该仓库在 rCore-Rust 家族中属于**实验型内核**，目标是对标 Linux 用户接口，但实现深度大幅降低。它通过分层抽象（L0-L1-L2）和 need 依赖注入展示了模块化内核的设计理念，但大量功能为原型或存根，从完整性看远未达到可用状态。与教学型 rCore-Tutorial 相比，它引入了更多 Linux 特有的概念（信号、futex、procfs），但稳定性与代码质量不如后者。

### 真正的创新与独特设计
该仓库的最独特之处在于其**三层能力层与 need 依赖注入机制**。启动流程中，L0 的 boot_memory 自解析 FDT 并构建物理内存布局，L1 的 user_boot 挂载根文件系统并决定引导模式，L2 的 syscall_entry 注册系统调用表，每层只通过 need.rs 声明所需能力（[boot.ref:2][boot.ref:3][boot.ref:4]）。这种设计使得模块间依赖显式化，易于替换和裁剪。此外，**用户地址空间显式管理**是一大亮点，通过全局 slot 数组（[mm.ref:4]）和惰性缺页回调实现类 Linux 的地址空间操作，包括写时克隆 fork 的雏形。信号模块的**分层设计**（L1 核心层 + L2 syscall 层）也体现了清晰的架构思路，尽管用户态 handler 缺失。

### 取舍判断
该仓库明显将精力投入了**模块化骨架和接口抽象**，而非功能完整性和安全性验证。例如，手动实现 FDT 解析（而非复用 device_tree crate）增加了代码量和风险，但换来了对引导流程的完全控制。内存管理使用了固定数组物理页分配器（[mm.ref:4]），简单但无法应对碎片化。内核堆锁重入检测仅记录日志而不恢复（[mm.ref:5]），暴露了调试阶段快速迭代的意图。驱动框架中两个 virtio 驱动完全重复代码，未抽取公共层（[drv.ref:18]），说明作者在追求“先跑通”而非代码复用。系统调用层大量使用 mock/unimplemented（[syscall.ref:2]），显示优先覆盖接口而非具体实现。

### 完成度与不足
按模块评估：
- **启动**：相对完整，FDT 解析有边界安全问题（[boot.ref:13][boot.ref:14]）。
- **内存管理**：基本可用，但 page_cache 硬编码 4K，物理页分配器可能丢页。
- **任务调度**：简单轮转，存在死代码（[task.ref:2]）和锁安全问题。
- **文件系统**：ext4 驱动有插入/删除实现，但 VFS 枚举约束扩展性。
- **信号**：队列管理和系统调用覆盖较全，但 sigreturn 缺失（[signal.ref:5]），信号无法传回用户态。
- **IPC**：共享内存/信号量空壳，套接字无数据传输（[ipc.ref:1][ipc.ref:2]）。
- **驱动**：两个冗余 virtio 驱动，PCI 扫描受限（[drv.ref:13]）。
- **系统调用**：覆盖几十个调用，但 clone 返回 ENOSYS（[syscall.ref:2]），核心进程创建能力缺失。

整体完成度偏低，创新性因三层管道和地址空间管理而可见，代码质量受安全注释和重复代码影响。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 该模块实现了从引导加载器接手后到首个用户进程进入用户态的全链路初始化。核心抽象是"三层管道"：L0 的 boot_memory 负责自解析 FDT 设备树构建物理内存布局并提供一个 bump allocator；L1 的 user_boot 挂载根 ext4 文件系统并决定 Shell/Tests 引导模式；L2 的 syscall_entry 注册系统调用分发表、通过 exec 加载初始进程并调用 enter_user 返回用户态。与标准 ArceOS 相比，该实现自建 FDT 解析器而非依赖 device_tree crate，嵌入了完整的 Linux 信号交付框架，并使用 AtomicBool 原子标志串联各阶段的 readiness。

### 1. 核心抽象：从物理内存到用户态的三层启动管道

本仓库的启动流程并非集中于单个入口文件，而是分散在 L0→L1→L2 三个能力层中，每个层暴露一个 `init() -> InitResult` 接口供全局初始化框架顺序调用。L0 的 `boot_info_seed` 模块（`l0/00_seed/`）是最上游的契约点：它通过 `crate::arch::boot::raw()` 获取架构层设置的 `BootInfoRaw`，包含 boot CPU 编号、payload 地址和引导源类型（DeviceTree / FirmwareTable / Unknown）<sup>[1](#mod-boot-ref-1)</sup>。这一设计将架构相关的引导信息收集隔离在 `arch/boot.rs` 的 `seed()` 函数中，L0 只做读取和转发。

真正的物理内存初始化落在 `boot_memory` 模块（`l0/20_boot_memory/`）。其 `body.rs` 中的 `init()` 函数执行一个三阶段流程：① 从 `need::boot_raw::raw()` 获取原始信息并填充静态 `BOOT_INFO`；② 若引导源为 DeviceTree 且 payload 非零，则调用自实现的 `parse_fdt()` 解析 Flattened Device Tree 提取内存节点 (memory@...) 和保留区域 (memreserve)，否则回退到 `install_arch_fallback_memory()` 使用条件编译的硬编码地址；③ 调用 `reserve_kernel()` 将内核镜像本身所占物理区间标记为 Reserved，最后通过 `normalize_usable_regions()` 从 Usable 区域中减去所有 Reserved 区间，并对结果按起始地址排序，准备作为 bump allocator 的内存池 <sup>[2](#mod-boot-ref-2)</sup>。

L1 的 `user_boot` 模块（`l1/60_user_boot/`）将启动推入用户态准备阶段。它的 `rootfs::plan_for()` 根据编译环境变量 `SHELL_ENABLE` 决定是进入交互 Shell 模式还是自动化 Test 模式，固定使用 `/dev/vda` 作为根设备、`/dev/vdb` 作为额外挂载设备 <sup>[3](#mod-boot-ref-3)</sup>。`rootfs::mount()` 调用下层 VFS 的 `mount_ext4_root()` 和 `mount_extra_ext4()`，并使用 `ROOT_MOUNTED` / `EXTRA_MOUNTED` 两个 `AtomicBool` 记录挂载结果，供后续 `user_boot::ready()` 轮询。

L2 的 `syscall_entry` 模块（`l2/50_syscall_entry/`）作为启动管道的终点，完成最后两件事：① `body::init()` 注册 `handle_trap` 为系统调用处理函数 <sup>[4](#mod-boot-ref-4)</sup>；② `user_entry_ready::start()` 调用 `exec` 加载初始用户程序，设置 `USER_ENTRY_STARTED` 标志，然后 `enter_current_user()` 通过 `thread_context::enter_user()` 执行特权级切换跳入用户态 <sup>[5](#mod-boot-ref-5)</sup>。至此启动流程完成。

### 2. 关键设计取舍

**取舍 1：自实现 FDT 解析器 vs 使用 device_tree crate。** 代码在 `body.rs` 中完整实现了 `Fdt`、`FdtParser`、`FdtReserveIter` 三个类型，从原始内存字节中读取 big-endian 令牌流 <sup>[6](#mod-boot-ref-6)</sup>。这一选择避免了引入外部依赖，也减少了对 alloc 的早期需求（解析器不需要堆分配），但牺牲了对设备树高级特性（如 interrupt-map、phandle 解析、DTC 编译的 overlay）的支持。当前代码仅关注 `#address-cells`、`#size-cells`、`reg` 和 `linux,initrd-*` 属性，遇到不认识的结构直接跳过。

**取舍 2：静态 mut 全局 bump allocator。** `MEMORY_REGIONS`、`USABLE_REGIONS`、`NEXT_ALLOC` 等全部声明为 `static mut`，`alloc()` 线性扫描可用区域并前移游标 <sup>[7](#mod-boot-ref-7)</sup>。这使得实现极其简单（~60 行），且零初始化开销。代价是无法释放——`BootAllocError::OutOfMemory` 一旦返回便无法恢复，且所有 `static mut` 访问依赖 unsafe 块，缺乏编译时借用检查。幸运的是 boot 阶段只有单核执行，不存在并发竞争。

**取舍 3：架构 fallback 硬编码地址。** 当设备树解析失败或引导源不是 DeviceTree 时，`install_arch_fallback_memory()` 按 `#[cfg]` 为 riscv64/loongarch64/其他架构分别填充 128 MiB 的固定物理地址区间 <sup>[8](#mod-boot-ref-8)</sup>。这是一个合理的调试默认值，但在 QEMU 内存配置非默认时（例如 `-m 2G`），硬编码的起始地址 `0x8000_0000` 可能恰好落在已占用的 MMIO 区域，导致后续 page fault。

**取舍 4：用户引导的同步屏障设计。** 三个关键状态（`ROOT_MOUNTED`、`EXTRA_MOUNTED`、`USER_ENTRY_STARTED`）全部使用 `AtomicBool` + `Ordering::SeqCst` 实现在模块间传递进度信号 <sup>[3](#mod-boot-ref-3)</sup><sup>[5](#mod-boot-ref-5)</sup>。这比回调或显式依赖注入更松散：L2 的 `user_entry_ready::start()` 不直接调用 L1 的函数，而是先检查 `rootfs::is_root_mounted()`（通过 `user_boot::ready()`）。这一设计有利于未来支持多核启动——虽然当前单核顺序执行下 SeqCst 带来的性能开销可忽略。

**取舍 5：信号交付嵌入 syscall_entry 层。** Linux 信号的用户态栈帧构建、sigreturn 恢复、以及 SA_ONSTACK 处理全部实现在 `syscall_entry::api::syscall_ready` 模块中，而非放在独立的 signal 模块 <sup>[9](#mod-boot-ref-9)</sup>。这减少了跨层调用的数量，但使 `syscall_entry` 承担了过多的职责——需要直接处理 `SignalFrame` 的内存布局、用户态读写、以及 TLS/栈寄存器的架构差异。

### 3. 跨模块协同机制

**协同点 A：boot_info_seed → arch::boot → boot_memory 的信息管道。** 架构层的 `boot::seed()` 由汇编入口在启用 MMU 前调用，将 boot CPU、FDT 地址等写入三个 `AtomicUsize`；`boot_info_seed::api::raw()` 读取这些原子变量返回 `BootInfoRaw`；`boot_memory::body::init()` 消费之 <sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>。关键细节是 `seed()` 必须在非常早期（可能还在物理地址空间）执行，而 `raw()` 可能在页表启用后调用——RISC-V 上 FDT 地址是物理地址，但 `parse_fdt()` 直接以该地址指针访问内存，意味着调用前必须保证该区域在页表中可读（当前通过恒等映射假设满足）。

**协同点 B：need 文件作为显式依赖清单。** 每个模块的 `need.rs` 用 `pub use` 列出它依赖的所有外部能力，例如 `syscall_entry/need.rs` 从 `l0` 引用 `syscall_raw` 和 `thread_context`，从 `l1` 引用 `task`、`signal`、`user_memory`、`exec`、`fork` 等 <sup>[10](#mod-boot-ref-10)</sup>。这种模式让启动流程的依赖关系完全显式化——评审者可以清晰地看到 L2 依赖于 L0 的 trap 基础设施和 L1 的任务管理，而 `boot_memory` 的 `need.rs` 只依赖 `boot_raw` 和 `early_console`，说明其确实位于最底层。

**协同点 C：`handle_trap` 中的 execve/clone 特殊路径。** 当用户进程调用 `execve` 且成功后返回 0，`handle_trap` 会调用 `reset_user_entry_frame()` 覆盖当前 trapframe 的 PC、SP 和 TLS，然后直接返回（不走 `advance_pc`）<sup>[11](#mod-boot-ref-11)</sup>。这意味着 syscall_entry 层感知到了 execve 的语义——新进程的入口地址在 exec 完成前就已确定，但 trapframe 的更新必须由入口层完成，因为下层 `table::dispatch` 不知道如何设置 `a0`/`sp` 等寄存器。`clone` 系统调用也类似：`clone_from_trap()` 在 fork 前手动构造子任务的 trapframe。

**协同点 D：信号交付跨层调用链。** `deliver_pending_signal()` 在每个系统调用返回后调用。它首先从 `need::task::current()` 获取当前任务，然后调用 `need::signal::take_pending()` 提取待处理信号 <sup>[9](#mod-boot-ref-9)</sup>。确定 handler 后，直接在栈上布局 `SignalFrame`（magic/version/siginfo/context），通过 `need::user_memory::write_current()` 写入用户栈，最后修改 trapframe 的 PC 为 handler、SP 为 signal frame 地址、ra 为 restorer。当用户从信号处理返回时，rt_sigreturn 系统调用触发 `restore_signal_frame()`，从栈帧读取 saved_mask 恢复信号屏蔽字并恢复寄存器上下文 <sup>[12](#mod-boot-ref-12)</sup>。

### 4. 边角细节与不足

**<sup>[1](#mod-boot-ref-1)</sup> `static mut` 安全契约缺失。** `BOOT_INFO`、`MEMORY_REGIONS` 等 6 个全局变量均为 `static mut`，但 `init()` 和所有读取函数均未标注 `// SAFETY:`。虽然在单核启动阶段不会并发，但未来若引入多核早期初始化或中断 handler 在 boot 期间意外触发，这些 unsafe 块将成为数据竞争入口。

**<sup>[2](#mod-boot-ref-2)</sup> `align_up` 的错误处理被忽略。** `normalize_usable_regions()` 中调用 `align_up(start, PAGE_SIZE)` 返回 `Option<usize>`，但通过 `unwrap_or(start)` 静默处理溢出——若起始地址加页面对齐发生回绕，将使用未对齐的原始地址，后续 `alloc` 可能返回非页对齐的内存 <sup>[13](#mod-boot-ref-13)</sup>。

**<sup>[4](#mod-boot-ref-4)</sup> 系统调用处理器注册使用 `set_handler`，但未检查旧 handler。** `syscall_entry::body::init()` 直接调用 `need::syscall_raw::set_handler(...)` <sup>[4](#mod-boot-ref-4)</sup>。如果该函数允许多次调用，后续模块若重新设置 handler 将静默替换，而 `handle_trap` 内部并未考虑被覆盖的场景。

**<sup>[6](#mod-boot-ref-6)</sup> FDT 解析缺少长度验证。** `Fdt::read_u32` 和 `read_u64` 直接对 `base + offset` 做指针解引用，没有校验 `offset` 是否在 FDT 总量范围内 <sup>[14](#mod-boot-ref-14)</sup>。如果设备树在内存中被破坏或 payload 地址指向无效区域，将导致访问越界触发异常。

**<sup>[8](#mod-boot-ref-8)</sup> fallback 内存与 QEMU 版本耦合。** `install_arch_fallback_memory()` 为不同架构固定分配 128 MiB 起始于硬编码地址，在 QEMU 使用非默认内存布局（如 `-m 2G`）时，可用区域可能被截断或遗漏 <sup>[8](#mod-boot-ref-8)</sup>。

**<sup>[3](#mod-boot-ref-3)</sup> 原子变量同步采用 SeqCst。** 三个 `AtomicBool` 全部使用 `Ordering::SeqCst`，在单核顺序执行下是过防护的 <sup>[3](#mod-boot-ref-3)</sup>。虽然不会出错，但可能掩盖真实的并发意图——若未来引入多核 boot，SeqCst 导致的全栅栏可能掩盖需要 acquire/release 语义的边界条件。

**<sup>[9](#mod-boot-ref-9)</sup> 信号帧写入使用 `write_user_value` 但缺少 `SAFETY` 注释。** `write_user_value` 内部将任意 `T` 按字节切片写入用户空间，依赖调用者保证 `T` 不包含引用或 `Drop` 标志 <sup>[15](#mod-boot-ref-15)</sup>。当前 `SignalFrame` 仅含整数和固定数组，安全；但若未来在 `UserSignalInfo::data` 中引入动态大小的字段可能引入未定义行为。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块涵盖物理页分配 (`page_alloc`)、多级页表操作 (`page_map`)、内核堆 (`kernel_heap`)、用户地址空间 (`AddressSpace` 及其映射管理) 以及文件页缓存 (`page_cache`)。核心抽象是 `AddressSpace`，它封装了独立的 `PageTable` 和 `Mapping` 列表，支持匿名映射、文件惰性加载和写时克隆。与 ArceOS 基线相比，本仓的用户空间管理更显式：提供 `new_address_space`、`clone`、`destroy` 等完整生命周期操作，并使用全局 slot 数组管理多个空间；惰性缺页处理通过回调 `LazyPageLoader` 与文件系统/ELF 加载器协同。内核堆采用 bootstrap 堆 + 页扩展的两阶段策略，锁设计包含重入检测。

### 1. 核心抽象与外部依赖

内存管理模块以**用户地址空间** (`AddressSpace`) 为顶层抽象。每个 `AddressSpace` 是一个 `usize` 索引，指向全局 `SPACES` 数组中的一个 `Space` 结构。`Space` 包含一个 `PageTable`（页表根物理地址）、一个 `Mapping` 列表（记录每个映射区间的虚拟地址范围、权限、物理页数组和来源）以及 `brk` 指针。页表操作由 `l0/30_page/page_map` 模块提供，它封装了 3 级页表的 `walk_create` / `walk_existing` 等底层函数 <sup>[2](#mod-mm-ref-2)</sup><sup>[3](#mod-mm-ref-3)</sup>；物理页的分配/释放则由 `page_alloc` 管理，它维护了一个固定大小的空闲区间数组（最大 64 项）<sup>[4](#mod-mm-ref-4)</sup>。内核堆分配器 (`kernel_heap`) 建立在页分配器之上，采用 bootstrap 堆 + 按需页扩展策略 <sup>[10](#mod-mm-ref-10)</sup>。此外，文件层 `page_cache` 模块提供了基于 LRU 的页粒度缓存。

与同家族标准 ArceOS 相比，本仓没有采用 `axmm` crate 的通用接口，而是为每个用户进程分配独立的 `AddressSpace`，并通过 `new_address_space` / `clone_address_space` / `destroy` 等函数显式管理地址空间生命周期。`Space` 中的 `mappings` 列表完整记录了每个映射的元数据，支持 `mprotect` 对区间权限进行拆分重组。

### 2. 关键设计取舍

**页表级数硬编码**：`page_map` 中 `LEVELS = 3` 且每表 512 项 <sup>[2](#mod-mm-ref-2)</sup>，这是为 RISC-V Sv39 定制的；页表遍历和分配代码未对级数做泛型化。对于 LoongArch64（也使用 3 级页表）可以工作，但若移植到 x86_64（4 级）必须修改。此外，`walk_create` 在中间表中遇到叶子条目时直接返回 `AlreadyMapped`，而非递归拆分大页——表明不支持 2MB/1GB 大页映射。

**物理页分配器简化**：`page_alloc` 使用定长数组 `FREE_RANGES` 存储空闲区间 <sup>[4](#mod-mm-ref-4)</sup>，分配时从低地址区间的起始处切出一页，释放时执行简单合并。这种设计没有伙伴系统的反碎片能力，且 `MAX_FREE_RANGES=64` 在碎片化严重时会拒绝释放。分配器锁是 IRQ 安全自旋锁，带有重入检测，但检测只记录日志而不崩溃，这可能导致静默死锁。

**内核堆的两阶段初始化**：`kernel_heap` 在静态区预分配 16MB `BootstrapHeap` <sup>[10](#mod-mm-ref-10)</sup>，通过 `HeapState` 的 `bootstrap_ready` 标志控制；在 `init_heap` 被调用前，`alloc_global` 仍可使用 bootstrap 堆。锁设计同样有重入检测 <sup>[5](#mod-mm-ref-5)</sup>，但锁内禁止调用可能重入的 `log::info!`，只能使用 `log_early`。

**惰性加载的“精简克隆”**：`clone_address_space` 对于只读的惰性源映射（典型如代码段），跳过物理页复制 <sup>[7](#mod-mm-ref-7)</sup>。父子进程的该映射在新地址空间中均无已分配的物理页，缺页时由同一个 `LazyPageLoader` 回调加载。这避免了无关物理页复制，且 `retain`/`release` 回调维护了文件引用计数。

**用户空间访问模型**：`read`/`write` 通过 `translate` 查用户页表获得物理地址，再经 `phys_to_virt` 转为内核虚拟地址直接拷贝 <sup>[6](#mod-mm-ref-6)</sup>。对于 RISC-V，由于内核与用户空间共享物理地址空间（身份映射），`phys_to_virt` 就是 `paddr`，因此无需切换页表；对于其它架构则需调用 `arch::vm::phys_to_virt`。

### 3. 跨模块协同

**缺页处理链路**：架构 trap handler 调用 `user_memory::handle_current_page_fault` <sup>[8](#mod-mm-ref-8)</sup>，后者定位到当前 `AddressSpace` 和对应的 `Mapping`，若为懒惰源则通过 `LazyPageLoader` 从文件读取数据到新分配的物理页 <sup>[6](#mod-mm-ref-6)</sup>。然后调用 `map_page` 更新页表，并刷新 TLB。若页面具有可执行权限，还会执行指令缓存刷写 (`clean_invalidate`)。该链路串联了物理页分配器、页表模块、文件系统和架构缓存操作。

**堆扩展与物理页分配器**：`kernel_heap` 在 bootstrap 堆用尽后，通过 `expand_unlocked` 调用 `need::pages::alloc_zeroed` 获取新页，再经 `phys_to_virt` 转换为内核虚拟地址并入空闲链表。此过程在 IRQ 安全锁释放后进行，以避免重入。`phys_to_virt` 与 `page_map` 共享相同的架构转换函数。

**用户空间与页缓存的隔离**：`page_cache` 模块服务于 VFS 层的文件读写，它与用户地址空间的“懒惰页”机制是独立的——前者缓存文件数据于内核态，后者记录的是进程页表映射。然而，二者可能对同一文件重复缓存（一次在用户页，一次在内核页缓存），这是当前设计的一个冗余点。

### 4. 边角细节与不足

**<sup>[1](#mod-mm-ref-1)</sup> `KERNEL_IDENTITY_RANGES` 架构覆盖不全**：仅对 `riscv64` 定义了四个物理地址范围，用于在用户页表中映射内核外设/跳板页；LoongArch64 和其他架构为空。若需在用户态中断/系统调用中使用内核跳板，LoongArch64 可能缺失必要映射。

**<sup>[9](#mod-mm-ref-9)</sup> `page_cache` 硬编码 `PAGE_SIZE = 4096`**：与架构 `crate::arch::vm::PAGE_SIZE` 可能不一致（如某些 arm64 配置使用 16KB 页），导致偏移计算错误。

**<sup>[5](#mod-mm-ref-5)</sup> 堆分配器锁重入检测仅记录日志**：`with_lock` 中检测到重入时调用 `log_early::record`，但重入发生后锁仍会被死等（`compare_exchange` 循环），没有超时或恢复机制，最终系统可能挂起。

**<sup>[4](#mod-mm-ref-4)</sup> 物理页分配器固定数组 `MAX_FREE_RANGES=64`**：当空闲区间超过 64 个（如频繁 mmap/munmap）时，后续释放会静默丢失（`push_free_range` 返回 `Err` 但调用者忽略），导致内存泄漏。

**未完成的 `update_present_page_perm`**：`mprotect` 调用此函数更新已存在物理页的页表项权限，但该函数并未出现在 `memory.rs` 的任何片段中（可能在未读部分），若未实现则 `mprotect` 仅修改元数据而不更新页表，权限控制失效。

**`clone_address_space` 中 `skip_present_lazy_pages` 逻辑可能漏掉已加载的干净页**：若父进程已因某种原因加载了只读懒惰页（例如被 `madvise` 提前触发），克隆时仍会跳过复制，子进程将重新从文件加载，这在文件内容未改变时是正确的，但若文件已被修改（MAP_SHARED 且其他进程写入），子进程将看到旧数据。当前假定惰性源总是只读文件映射，但未来若引入写时复制映射，此逻辑需调整。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块采用两层架构：底层 `l0/70_run/scheduler` 提供状态机与上下文切换，上层 `l1/20_task/task_mm_signal` 封装任务生命周期（spawn/exit/exec）并集成信号处理。核心抽象是 `Task` 结构体（包含调度ID、地址空间、信号掩码、内核栈）与调度器 `Scheduler`（就绪队列 + 轮转切换）。相比 rCore-Tutorial 将进程控制块与调度器分离，此仓将调度状态直接嵌入 `Scheduler` 的 `Entry`，任务层仅持有 ID 引用，形成“弱进程、强调度”的扁平设计。独特之处在于：用全局自旋锁保护任务表、通过 `need` 模块解耦架构依赖、fork 实现通过陷阱帧复制直接复用父栈。

### 1. 核心抽象与两层架构

进程与任务调度模块由两个清晰的层级构成。底层是 **调度器层**（`kernel/src/l0/70_run/scheduler/api.rs`），它维护一个 `Scheduler` 结构体，内部有一个 `Vec<Entry>` 作为全局就绪队列。每个 `Entry` 包含 `TaskId`、`State` 枚举（Ready/Running/Blocked/Exited）和 `ThreadContext`（即保存的寄存器现场）。调度器提供 `register_task`、`set_current`、`wake_task`、`block_task`、`exit_task` 等原语，以及 `schedule()` 函数完成实际上下文切换<sup>[1](#mod-task-ref-1)</sup>。`schedule()` 每次被时钟中断触发，先唤醒所有到期的 Blocked 任务，再选择下一个非当前的 Ready 任务，通过 `need::thread_context::switch` 汇编函数切换栈与寄存器<sup>[2](#mod-task-ref-2)</sup>。

上层是 **任务管理层**（`kernel/src/l1/20_task/task_mm_signal/api/task_state.rs`），它定义了 `Task` 结构体，封装了信号处理（`signal_waiting`、`signal_blocked`、`signal_actions` 等）、地址空间（`AddressSpace`）、内核栈、执行路径等进程/线程属性<sup>[3](#mod-task-ref-3)</sup>。任务层通过 `run` 模块（即调度器 API）与底层交互，每个 `Task` 对应调度器中的一个 `Entry`，通过相同的 `TaskId` 关联。这种分离使得调度器可以独立于信号和地址空间等高层语义，而任务层则专注于 POSIX 语义的实现。

### 2. 关键设计取舍

**(1) 全局任务表与自旋锁**：任务表 `TASKS` 是一个 `Vec<Option<Task>>`，通过 `TasksCell` 包装（内部 `UnsafeCell`）和全局 `AtomicBool` 自旋锁保护<sup>[4](#mod-task-ref-4)</sup>。这种设计避免了标准库的 `Mutex` 依赖，适合无 std 内核环境。但自旋锁未关中断，若在中断上下文中调用任务 API 可能死锁（实际中断处理只调用 `schedule`，后者使用调度器自己的锁并会关中断，因此风险可控）。`// SAFETY:` 注释缺失，是明显技术债。

**(2) TaskId 分配策略**：`insert_task` 首先扫描现有槽位寻找 `None`，若没有则 `push` 新槽。任务退出后，槽位通过 `take()` 清空可被重用。但 id 是单调递增的 `usize`，长时间运行可能导致 id 溢出（虽然概率极低）。槽 0 永远为 `None` 且被保留用于表示无效 ID。

**(3) 上下文初始化与架构抽象**：`arch::thread` 模块为每个架构提供 `ThreadContext` 结构体（成员包括 sp/tp/pc/页表等），并实现 `init_user`、`init_kernel`、`init_fork` 等初始化函数<sup>[5](#mod-task-ref-5)</sup>。`init_fork` 的特殊之处在于它直接在子内核栈上拷贝父陷阱帧，并设置入口为架构特定的 `fork_return_entry`，子进程首次返回用户态时 a0=0<sup>[6](#mod-task-ref-6)</sup>。这种“栈上造帧”技术避免了额外的内存分配。

**(4) 信号处理的嵌入**：信号掩码、等待集、动作表直接放在 `Task` 中，信号队列使用 `Vec<SignalInfo>`。`take_task_signal` 通过位运算和线性搜索取出最高优先级信号<sup>[7](#mod-task-ref-7)</sup>。这种设计简单直接，但信号队列的 `remove` 操作是 O(n)，且信号挂起数没有上限，可能被恶意用户耗尽内核内存。

### 3. 跨模块协同与关键流程

**spawn → 调度就绪**：`spawn_user` 分配内核栈，调用 `arch::thread::init_user` 初始化上下文，然后通过 `run::register_task` 将任务加入调度器<sup>[8](#mod-task-ref-8)</sup>。最后 `insert_task` 填充任务表的其他字段（信号、地址空间等）。整个过程需要 task_lock 和调度器锁嵌套，但顺序固定（先 task_lock，再调度器锁）。

**上下文切换**：时钟中断触发 `schedule()`，它在 `SchedulerCell::with` 中持锁遍历、选择下一任务，然后释放锁，最后执行 `need::thread_context::switch` 汇编切换。切换后新任务的 `ThreadContext` 中 `page_table` 字段在 `set_current_task` 时已通过 `memory::activate` 写入页表基址寄存器<sup>[9](#mod-task-ref-9)</sup>。

**exit 与资源回收**：`exit_task` 先在 task_lock 下处理父子关系（子任务重新挂到祖父），然后调用 `run::exit_task` 标记调度器状态为 Exited。父进程通过 `take_exited_child` 回收子任务内核栈和地址空间，最终调用 `run::unregister_task` 移除调度条目<sup>[10](#mod-task-ref-10)</sup>。这种回收分两步：状态标记与资源析构分离，避免父进程在信号处理中长时间持锁。

**exec（replace_image）**：保留原内核栈，通过 `arch::thread::init_user` 重新初始化上下文，更新地址空间和页表，同时更新任务表的 entry/stack/tls/exec_path 等字段。旧地址空间作为返回值由调用者释放<sup>[11](#mod-task-ref-11)</sup>。

### 4. 边角细节与不足

- **`todo!` 残留**：`procfs.rs` 中 `ProcessExe` 的文件内容实现为 `todo!("unimplemented")`<sup>[12](#mod-task-ref-12)</sup>，意味着 `/proc/self/exe` 读取将 panic，这会影响 busybox 等依赖该节点的程序。
- **`schedule` 与 `next_ready` 代码重复**：两者都实现唤醒到期任务并选择下一任务的逻辑，但 `schedule` 额外处理当前任务状态切换并执行上下文切换。`next_ready` 未被使用，是死代码。
- **硬编码时钟周期**：`body.rs` 中 `alarm::set_periodic(100)` 固定 10ms 时间片，未参数化。
- **错误路径吞没**：`with_task` 返回 `Result`，但某些上层调用（如 `procfs::symlink_target`）使用 `ok()` 掩盖错误<sup>[13](#mod-task-ref-13)</sup>，导致文件系统操作静默失败。
- **栈分配 panic 风险**：`alloc_kernel_stack` 使用 `alloc::alloc::alloc`，当堆内存不足时返回空指针，随后 `unwrap()` 或 `?` 传播 `NoMemory`，但 `dealloc_kernel_stack` 中若 bottom==0 则直接返回，逻辑正确但缺少更友好的错误提示。
- **自旋锁无退避**：`task_lock` 和 `SchedulerCell` 的锁都使用 `spin_loop()` 但未实现指数退避，在高度竞争下可能影响性能。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制由两层组成：L1 的 `task_mm_signal` 提供信号类型、位图操作和状态管理核心；L2 的 `sys_signal` 提供 Linux 兼容的系统调用接口（rt_sigaction/rt_sigprocmask/kill 等）。核心抽象是 u64 位图表示的 SigSet、SignalAction 和通过 signal_state 桥接任务状态管理的递送管道。与完整 Linux 信号实现相比，本实现缺少用户态信号框架（sigframe）搭建和 rt_sigreturn 逻辑，进程组发送被 stub 为 ENOSYS，整体处于可处理信号排队/阻塞但无法执行用户态 handler 的早期阶段。

### 1. 核心抽象与外部依赖

信号机制的核心抽象定义在 `kernel/src/l1/20_task/task_mm_signal/api/signal.rs` 中。`Signal(pub usize)` 是信号编号的包装类型，支持 1–64 范围（包含 32 个常规信号和 33 个实时信号）<sup>[1](#mod-signal-ref-1)</sup>。`SigSet(u64)` 是以位图表示的信号集，每一位对应一个信号（bit 0 = SIGHUP, bit 63 = SIGRTMAX），通过 `from_bits`/`bits` 方法与用户态 `sigset_t` 零拷贝互转 <sup>[2](#mod-signal-ref-2)</sup>。`SignalAction` 包含 `handler`（用户态函数地址或 SIG_DFL/SIG_IGN）、`flags`、`restorer` 和 `mask`，其中 `restorer` 字段保留给内核在 sigreturn 时使用——但当前并未实际使用。`SignalInfo` 携带信号编号、errno、code 和 116 字节的数据负载，用于 `rt_sigqueueinfo` 等需要传递额外信息的调用。

模块间的依赖关系通过 `need.rs` 文件声明。L2 的 `sys_signal/need.rs` 引入 `task` 和 `signal` API，以及 `syscall_args` 和 `errno` <sup>[3](#mod-signal-ref-3)</sup>。L1 的 `task_mm_signal/need.rs` 引入更底层的 `run`（调度器唤醒）、`user_copy`（用户态内存读写）、`trap` 等 <sup>[4](#mod-signal-ref-4)</sup>。这种分层符合该仓库统一的“能力分层”架构，每层只依赖下层接口，不跨层调用。

### 2. 关键设计取舍

**位图选择与可移植性假设**。`SigSet` 使用 `u64` 而非 `u128` 或动态位图，隐含地假设 Linux 信号编号不超过 64。Linux 内核虽然也限制为 64 个信号，但某些架构（如 Alpha）有不同约定。代码中通过 `SIGNAL_MAX = 64` 常量声明这一边界 <sup>[1](#mod-signal-ref-1)</sup>，并在 `validate_signal` 中硬编码检查 `signal.0 <= 64` <sup>[5](#mod-signal-ref-5)</sup>。如果未来需要扩展 RT 信号范围，需要同时修改 `SigSet` 的底层类型和所有位操作。

**不可屏蔽信号的强制过滤**。`mask_without_unblockable` 函数在每次设置信号掩码时强制清除 SIGKILL(9) 和 SIGSTOP(19) 的对应位 <sup>[6](#mod-signal-ref-6)</sup>。这个函数被 `rt_sigaction`、`rt_sigprocmask`、`rt_sigsuspend` 三处调用，确保用户态无法通过 sigprocmask 或 sigaction 屏蔽这两个信号。这是一个安全关键设计：如果漏调此函数，用户态任务可能错误地屏蔽 SIGKILL 导致无法被杀死。

**信号递送与任务唤醒的耦合**。`deliver_signal_info` 在 `signal_state.rs` 中将信号压入任务队列后，立即调用 `run::wake_task(task)` <sup>[7](#mod-signal-ref-7)</sup>。这意味着信号递送会强制唤醒任务（如果任务处于睡眠/等待状态）。但这样做丢失了 Linux 的某些语义：例如 SIGSTOP 应当暂停任务而非唤醒，SIGCONT 应当唤醒被暂停的任务——当前实现中所有信号都执行同一套“入队+唤醒”逻辑，没有区分信号类型的行为差异。

**值传递与零拷贝边界**。在 `sys_signal/api.rs` 中，用户态与内核之间的数据结构传递全部通过 `need::syscall_args::read_value`/`write_value` 进行 <sup>[8](#mod-signal-ref-8)</sup>。这些函数封装了 `user_copy` 模块，进行地址校验和权限检查。`LinuxSigAction`、`LinuxStack`、`LinuxSigInfo` 等结构体都标注 `#[repr(C)]` 以匹配 Linux 用户态 ABI。这种设计将 ABI 转换隔离在 syscall handler 层，L1 核心代码只使用内部类型（`SignalAction`、`SignalStack`），不暴露与具体 OS ABI 的耦合。

**进程组信号的显式缺失**。`send_to_pid` 在处理 `signed_pid <= 0`（即负 pid 或 0）时，若 `accept_groups` 为 true 则直接返回 ENOSYS，否则返回 EINVAL <sup>[9](#mod-signal-ref-9)</sup>。这意味着 `kill(2)` 的进程组语义（负 pid 表示进程组）未被实现。这是合理的阶段化取舍：在没有完整进程组管理之前，先提供最基本的有名 pid 发送。

### 3. 跨模块协同

信号机制的完整生命周期涉及三个层级的模块：

- **L2 syscall handler** (`sys_signal/api.rs`)：负责参数解析、合法性校验、错误码映射。它将 Linux ABI 结构体转换为内部类型，然后调用 L1 的 `task` 和 `signal` API。
- **L1 task_mm_signal** (`signal_state.rs` + `task_state.rs`)：`signal_state.rs` 作为信号状态的中转层，`deliver_signal_info` 调用 `task_state::push_task_signal` 将信号入队，然后调用 `run::wake_task` 唤醒目标任务 <sup>[7](#mod-signal-ref-7)</sup>。`take_pending` 调用 `task_state::take_task_signal` 从队列中取出匹配的信号。`terminate_task` 调用 `task_state::exit_task` 用退出码 `128 + signal.0` 终止任务。
- **L0 scheduler** (`run::wake_task`)：作为最底层依赖，提供任务唤醒原语。信号递送通过唤醒机制实现“异步通知”——目标任务在下次被调度时会在 trap 返回路径中检查信号。

虽然 `task_state.rs` 的具体实现未提供，但从接口调用可以推断：每个任务维护一个信号队列（FIFO 或优先级队列），`push_task_signal` 将信号信息插入队列，`take_task_signal` 根据 `wait_set` 和 `blocked` 位图过滤并取出首个匹配信号。`TaskInfo` 中的 `signal_waiting` 和 `signal_blocked` 字段提供了信号状态的快照视图 <sup>[10](#mod-signal-ref-10)</sup>。

### 4. 边角细节与不足

**<sup>[11](#mod-signal-ref-11)</sup> `rt_sigreturn` 是空桩**。`rt_sigreturn` 函数仅打印 `log::warn!` 并返回 EINVAL。这意味着用户态信号 handler 返回后无法恢复上下文——信号框架（sigframe）的构建和恢复完全没有实现。这是信号机制最大的功能缺口：`signal(2)` 可以注册 handler，但信号递送时不会调用它，因为 trap 处理路径中没有插入“检查待处理信号并构建 sigframe”的逻辑。

**<sup>[12](#mod-signal-ref-12)</sup> 硬编码常量 MINSIGSTKSZ 的安全假设**。`sigaltstack` 中检查 `stack.size < MINSIGSTKSZ` 时使用常量 2048 <sup>[12](#mod-signal-ref-12)</sup>。不同架构的 `MINSIGSTKSZ` 不同（如 x86 为 2048，但 ARM64 可能需要更大），这里未做 `#[cfg]` 区分。同时，该检查仅在 `flags == 0` 时执行，而 `flags == SS_DISABLE(2)` 时跳过——但 `SS_DISABLE` 时 `stack.size` 被强制设为 0，因此不会触发。

**<sup>[13](#mod-signal-ref-13)</sup> 错误路径中的 warn 吞错误**。在 `rt_sigprocmask` 和 `rt_sigpending` 中，当 `need::task::current_info()` 失败时，只记录 `log::warn!` 然后返回 EINVAL <sup>[13](#mod-signal-ref-13)</sup>。但如果 `current_info` 返回 `Err` 的原因是 `NotFound`（即当前没有任务上下文），返回 EINVAL 可能掩盖了更严重的状态错误。类似地，`current_task_id` 辅助函数在 `need::task::current()` 返回 `None` 时也返回 EINVAL <sup>[14](#mod-signal-ref-14)</sup>。

**<sup>[5](#mod-signal-ref-5)</sup> 信号范围验证与 Rust 类型脱节**。`validate_signal` 在 `signal_state.rs` 中实现为 `if signal.0 <= 64 { Ok(()) }`，但 `Signal` 的构造函数是 `pub` 的，可以在不调用 validate 的情况下构造编号 >64 的 Signal 实例 <sup>[5](#mod-signal-ref-5)</sup><sup>[1](#mod-signal-ref-1)</sup>。这导致类型系统无法静态保证信号合法性，非法信号只有在运行期经过 validate 才能捕获。

**<sup>[15](#mod-signal-ref-15)</sup> 未使用的 `SA_ONSTACK` 常量**。`api/signal.rs` 中定义了 `SA_ONSTACK: usize = 0x0800_0000` <sup>[1](#mod-signal-ref-1)</sup>，但 `sigaltstack` 的调用和信号处理路径中没有任何代码检查 `SignalAction.flags` 中的 `SA_ONSTACK` 位以决定是否使用备选栈。这是半成品标记——常量已定义，逻辑未接入。

**<sup>[16](#mod-signal-ref-16)</sup> `SigSet::first` 的优先级反转风险**。`first()` 使用 `trailing_zeros` 返回编号最小的待处理信号 <sup>[2](#mod-signal-ref-2)</sup>。在 Linux 中，常规信号（1-31）的优先级高于实时信号（32-64），但实时信号支持排队。当前实现将所有信号同等对待，且不支持队列（多个同号信号会合并），这会导致实时信号语义丢失。

**<sup>[17](#mod-signal-ref-17)</sup> 缺少对 SIGSTOP/SIGCONT 的特殊处理**。`terminate_task` 只处理 SIGKILL 类的终止信号，但对 SIGSTOP（应暂停任务）和 SIGCONT（应恢复暂停任务）没有区别对待 <sup>[7](#mod-signal-ref-7)</sup>。L0 调度器是否有暂停/继续的接口也未在代码中体现。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本仓库的 IPC 模块由 `ipc_net_device` 和 `futex` 两个子模块组成，前者提供共享内存、信号量、套接字等 POSIX 风格原语，后者实现 Linux 兼容的 futex 快速用户空间互斥。核心抽象是通过原子计数器分配句柄（`ShmHandle`、`Semaphore`、`SocketHandle`），套接字基于外部文件描述符（`EXTERNAL_FD_KIND`）与 VFS 集成，futex 基于用户虚拟地址构建等待队列。与同家族基线（ArceOS / rCore-Tutorial）相比，该实现处于早期原型阶段：共享内存和信号量是空壳，套接字仅模拟无数据传输，futex 使用线性扫描等待队列和自旋锁保护。设计偏向快速迭代验证接口，而非性能或完整性。

### 1. 核心抽象与外部依赖

IPC 模块分为两个独立的子系统。`ipc_net_device` 子模块（位于 `l1/40_ipc_net_device/`）提供 `posix_time`、`ipc`（共享内存、信号量）、`socket`、`device_file`、`tty`、`random_file` 等能力。`futex` 子模块（位于 `l1/50_bridge/futex/`）提供 `wait_until`、`wake`、`requeue` 三个核心操作。两个子模块通过自动生成的 `need.rs` 声明外部依赖 <sup>[10](#mod-ipc-ref-10)</sup><sup>[11](#mod-ipc-ref-11)</sup>：`ipc_net_device` 依赖 VFS 文件描述符 (`fd`)、页缓存 (`file`)、路径解析 (`path`)、信号 (`signal`)、用户内存访问 (`user_memory`)、任务等待 (`wait`)、时钟闹钟 (`clock`/`alarm`)、字符/块/随机设备；`futex` 依赖用户内存 (`user_memory`)、任务调度 (`task`)、时钟 (`clock`)、等待队列 (`wait`)、系统错误码 (`sys_error`)。

核心抽象是“句柄”与“外部文件描述符”。共享内存和信号量只通过原子计数器生成唯一 ID，不分配实际资源 <sup>[1](#mod-ipc-ref-1)</sup>；套接字句柄与 `fd::ExternalFile` 绑定，使套接字可以挂载到进程的文件描述符表中，通过 `EXTERNAL_FD_KIND` 标识类型 <sup>[5](#mod-ipc-ref-5)</sup>。Futex 的核心抽象是“等待者”（`Waiter`），包含用户地址和任务 ID，存储在全局 `Store<Vec<Waiter>>` 中 <sup>[7](#mod-ipc-ref-7)</sup>。

### 2. 关键设计取舍

**存根策略 vs 真实同步**：`create_shared_memory` 和 `create_semaphore` 完全忽略参数，仅返回递增 ID <sup>[1](#mod-ipc-ref-1)</sup>。这意味着底层没有分配共享内存页或创建内核信号量。替代方案是立即分配物理页或使用 `axsync` crate 的信号量。当前选择快速通过编译，将真实实现推给上层用户库或未来迭代，代价是进程间无法真正共享数据或同步——这是典型的“先有接口、后有实现”的早期开发策略。

**套接字模拟 vs 网络栈**：`send` 总是返回完整长度并设置 `readable = true`，`recv` 若不可读则返回 `WouldBlock`，否则清空 `readable` 且返回 0（`buf.len().min(0)`恒为0）<sup>[2](#mod-ipc-ref-2)</sup><sup>[3](#mod-ipc-ref-3)</sup>。这构成一个“回环”但无数据的流控制：发送方告知接收方可读，但接收方永远读不到字节。替代方案是引入环缓冲区或借用现有网络协议栈（如 smoltcp）。当前设计允许上层进行系统调用流程调试，但不能进行真实网络通信。

**Futex 的自旋锁与线性扫描**：`Store<T>` 实现了一个基于 `AtomicBool` 的简单自旋锁 <sup>[4](#mod-ipc-ref-4)</sup>，没有退避或阻塞回退。这在多核 CPU 上可用，但单核下若持有锁的任务被抢占，则可能死锁。等待队列是 `Vec<Waiter>`，`wake` 线性扫描全部元素 <sup>[7](#mod-ipc-ref-7)</sup>。替代方案是使用每个地址的哈希表或红黑树。线性扫描在等待者数量少时简单高效，但大量线程竞争同一地址时性能退化。此外，`requeue` 操作用线性扫描迁移等待者，未考虑目的地址是否已有等待者，可能导致未来 `wake` 唤醒错误的任务。

**Futex 超时处理**：`wait_until` 在进入时检查 `deadline_ns` 是否已过，若已过则立即返回 `TimedOut`，不再检查 `*uaddr` <sup>[6](#mod-ipc-ref-6)</sup>。这符合 POSIX 规范（超时优先），但放弃了一次优化机会：若 `*uaddr` 已不等于 `expected`，返回 `WouldBlock` 可能比超时更有用。

### 3. 跨模块协同

**套接字与 VFS 文件描述符**：`socket::create` 调用 `fd::register_external` 注册一个 `ExternalFile`，其中 `kind = EXTERNAL_FD_KIND`，`id` 为套接字句柄 <sup>[5](#mod-ipc-ref-5)</sup>。此后，通过 `socket::fd` 可获取标准 `fd::Fd`，使套接字能用于 `read`/`write` 系统调用（由 VFS 根据 kind 分发到对应处理函数）。但当前套接字本身未实现 `read`/`write` 语义，仅通过 `send`/`recv` 接口暴露——这种分离说明 VFS 集成尚不成熟。

**Futex 与内存管理**：`wait_until` 通过 `user_memory::read_current` 读取用户空间地址 <sup>[8](#mod-ipc-ref-8)</sup>。该函数接受虚拟地址和长度，返回 `Result<Vec<u8>, ...>`，内部需处理页错误和权限检查。若地址无效，返回 `Fault` 错误。futex 依赖此机制确保操作的原子性（实际原子性由用户空间保证，内核只负责比较和等待）。

**Futex 与任务调度**：添加等待者后，调用 `task::mark_blocked_until(current, deadline_ns)` 将当前任务标记为阻塞状态，并设置超时唤醒 <sup>[9](#mod-ipc-ref-9)</sup>。`wake` 调用 `task::mark_ready(task)` 将任务移回就绪队列。错误转换 `map_task` 将 `task::TaskError` 映射为 `sys_error::SysError`，但 `BadAddress` 映射为 `Fault`，`NoMemory` 映射为 `TooManyOpenFiles`（语义不匹配）<sup>[9](#mod-ipc-ref-9)</sup>。

**时钟与超时**：`posix_time` 子模块通过 `clock::now_ns()` 获取单调时间，用于计算超时。`futex` 和套接字均未直接使用 alarm，但 `need.rs` 导入了 `alarm`，可能是为未来信号驱动 I/O 预留。

### 4. 边角细节与不足

**可疑的 `recv` 返回值**：`socket::recv` 中 `buf.len().min(0)` 恒等于 0 <sup>[3](#mod-ipc-ref-3)</sup>，导致接收永远返回 0 字节。这可能是笔误：应为 `buf.len().min(data_len)`，但 `data_len` 未定义。当前实现使所有 recv 调用成功但不读取数据，若上层依赖返回值判断数据量，会陷入无限循环。

**自旋锁的并发安全性**：`Store<T>::with` 使用 `compare_exchange` 自旋，`locked` 标志在 `Relaxed` 顺序下读取，可能导致弱内存模型下的活锁。此外，没有禁用中断或迁移保护，在多核环境若持有锁的任务被中断，其他核可能自旋等待。当前无显式 `unsafe` 注释说明安全契约 <sup>[4](#mod-ipc-ref-4)</sup>。

**句柄无回收**：`NEXT_SHM`、`NEXT_SEM`、`NEXT_SOCKET` 单调递增，从不回收。长期运行可能导致 ID 回绕（64 位约需 5 亿年，但 32 位可能更快），且 `SOCKETS` 的 `Vec` 只增不减，浪费内存。

**错误码语义不一致**：`ipc::create_shared_memory` 返回 `ShmHandle`，无失败路径；`socket::create` 若 `fd::register_external` 失败（如达到进程文件描述符上限），返回 `SysError`，但未处理 `SOCKETS.with` 内部的资源泄漏（fd 已分配，但 socket 未插入）。

**Futex 的竞争条件**：`wait_until` 中检查 `*uaddr == expected` 与添加等待者之间无锁保护。若在检查和 push 之间，另一个线程修改 `*uaddr` 并调用 `wake`，当前线程可能无限阻塞。这是经典 futex 实现问题，通常通过用户空间原子操作加内核序列化解决，但当前代码未处理。

**初始化仅为日志**：两个子模块的 `init` 函数仅打印日志 <sup>[12](#mod-ipc-ref-12)</sup>，未做任何状态初始化（如全局队列预分配）。这符合模块化惰性初始化策略，但首次调用可能因动态分配失败。

**`device_file` 和 `tty` 为空**：`api.rs` 中声明了 `pub mod device_file {}` 和 `pub mod tty {}`，但无任何内容，属于死代码。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块提供了块设备（BlockDevice）和字符设备（CharDevice）两层 trait 抽象，通过全局注册表（Mutex<Vec<Box<dyn ...>>>）管理驱动实例，返回递增的设备 ID 供上层索引。具体实现包含两个 virtio 块设备驱动：virtio_pci_blk（通过 PCI ECAM 配置空间发现）和 virtio_mmio_blk（扫描固定 MMIO 地址段）。与同家族 ArceOS 基线相比，本仓将驱动发现与 DMA 内存分配完全托管给 need 基础设施层，并引入了独立的 validate_geometry/validate_io 校验链，但两个 virtio 驱动存在大量重复逻辑，缺少公共 virtio 核心抽象。

### 1. 核心抽象：设备 trait 与注册机制

驱动框架的核心是两层 trait 抽象——`BlockDevice` 和 `CharDevice`，均定义在 `api.rs` 中。`BlockDevice` 约定块设备的属性（name、block_size、block_count）与块级读写操作（read_blocks、write_blocks），要求实现者满足 `Send` 约束从而可跨线程传递 <sup>[1](#mod-drivers-ref-1)</sup>。`CharDevice` 类似，面向字节流读写 <sup>[2](#mod-drivers-ref-2)</sup>。两种 trait 的设计都极其精简：块设备只有 5 个方法，字符设备只有 3 个，没有任何生命周期参数或关联类型，这降低了实现门槛，但也意味着缺少异步接口、`O_DIRECT` 对齐提示等高级语义。

设备实例通过 `register()` 函数注册到全局静态变量 `REGISTRY`（类型为 `need::lock::Mutex<Vec<Box<dyn BlockDevice>>>`）<sup>[3](#mod-drivers-ref-3)</sup>。注册时首先调用 `validate_geometry()` 检查 name 非空、block_size > 0、block_count > 0，然后分配一个递增的 `BlockDeviceId`（即数组下标）并压入 Vec <sup>[4](#mod-drivers-ref-4)</sup>。这种“递增 ID = 下标”的设计极为简单，但意味着设备一旦注册就不能卸载（pop 会导致后续 ID 错位），且 ID 的生命周期完全由 Vec 的生存期管理。

值得对比的是 `ext4_rs` 文件系统层定义的 `BlockDevice` trait <sup>[5](#mod-drivers-ref-5)</sup>：它使用 `read_offset(offset) -> Vec<u8>` 和 `write_offset(offset, data)` 的字节偏移接口，而非块号接口。这意味着内核驱动框架提供的块设备不能直接被 ext4 使用，中间必须存在一个适配层（将块号与块大小转换为字节偏移）。该适配层在当前仓库中并未出现在驱动框架目录下，而是由文件系统模块自行封装——这是一个跨模块契约的隐性假设，评审应关注适配层是否存在以及其正确性。

### 2. 关键设计取舍：PCI 发现 vs MMIO 扫描，轮询 vs 中断

仓库同时实现了两个 virtio 块设备驱动：`virtio_pci_blk` 和 `virtio_mmio_blk`。两者实现了完全相同的 `BlockDevice` trait，但设备发现路径截然不同：

- **PCI 版本**通过 ECAM 内存映射配置空间遍历总线 <sup>[6](#mod-drivers-ref-6)</sup>：循环扫描 bus 0..=PCI_BUS_MAX（0x7f，仅 128 条总线，而非 PCIe 标准的 256），每个 bus 上设备号 0..32，读取 Vendor ID 和 Device ID 筛选 virtio 块设备。然后通过能力链表（Capabilities List）解析 Virtio PCI 能力结构（common/notify/isr/device 配置），映射到 `DMW0_BASE | paddr` 的直接映射线性区 <sup>[7](#mod-drivers-ref-7)</sup>。
- **MMIO 版本**则假设设备映射在 `0x1000_1000` 起始、步长 0x1000 的 8 个槽位 <sup>[8](#mod-drivers-ref-8)</sup>，通过读取 Magic Number（0x7472_6976）、Version 和 Device ID 判断是否为 virtio 块设备。

两种方式覆盖了 QEMU/经典计算机系统的两种典型配置，但没有 cfg gate 做平台隔离，意味着在只提供 PCI 的硬件上 MMIO 探测会白白扫一遍无意义的地址段（反例：若某嵌入式平台 0x1000_1000 恰好是 RAM，可能导致误判）。

**更大的取舍在于 I/O 模型**：两个驱动的 `submit()` 方法都使用 `loop { core::hint::spin_loop() }` 轮询等待设备完成 <sup>[9](#mod-drivers-ref-9)</sup>，而非注册中断处理函数。这一取舍彻底避开了中断控制器（如 PLIC/APIC）的集成、中断上下文的嵌套、以及下半部（Bottom Half）的调度复杂性，但也带来了严重的 CPU 忙等浪费。对于块设备这类毫秒级操作，轮询意味着整个 CPU 核被锁死在等待循环中，无法执行其他任务。这是典型的“早期原型简化”信号——评审应当追问后续是否有中断驱动路径的计划。

此外，两个驱动都自己实现了 `alloc_contiguous_zeroed()` 函数 <sup>[10](#mod-drivers-ref-10)</sup>，通过 `need::pages::alloc_zeroed()` 逐页分配并检查物理连续性。若某一页不连续则释放并返回 `None`。这是一个脆弱的策略：在高碎片化场景下连续两页的分配成功率迅速下降，且没有回退到不连续 DMA （通过 IOMMU/SWIOTLB 映射）。

### 3. 跨模块协同：need 基础设施与架构抽象层

驱动框架不直接操作硬件寄存器，而是通过两层间接调用：

1. **`need` 模块**（通过 `super::super::need` 引入）提供 `pages::alloc_zeroed()` 分配 DMA 内存、`lock::Mutex` 保护注册表。这是本仓独有的设计：将内存分配与锁原语抽象为 need 接口，允许上层（如裸机 HAL 或特权级封装层）替换实现。驱动不关心页分配来自哪个物理内存分配器，只关心它返回 `Page { paddr, .. }` <sup>[11](#mod-drivers-ref-11)</sup>。
2. **架构层**（`crate::arch::mmio`、`crate::arch::dma`、`crate::arch::vm`）提供 MMIO 读写、DMA 缓存一致性同步、虚实地址转换。`api.rs` 中的 `device::mmio_read32` 和 `dma::sync_for_device` 只是薄封装 <sup>[12](#mod-drivers-ref-12)</sup>，真正的实现下沉到 arch 目录（可能是 RISC-V 或 x86 的 satp/ptes 操作）。

这种双层抽象使得驱动代码与架构相关的位姿解耦——例如 `virtio_pci_blk` 中的 `io_phys_to_virt` 通过 `DMW0_BASE | paddr` 做线性映射 <sup>[7](#mod-drivers-ref-7)</sup>，而 `phys_to_virt` 则委托给 `crate::arch::vm::phys_to_virt`，两者面对不同的地址空间（外设 MMIO vs 常规 DRAM）。这种区分是正确但微妙的：若某个架构的 MMIO 映射不是简单的 OR 操作，`io_phys_to_virt` 就需要条件编译。

### 4. 边角细节与不足

**<sup>[13](#mod-drivers-ref-13)</sup> PCI 总线枚举未覆盖完整范围**：`PCI_BUS_MAX = 0x7f` 只扫描 128 个总线，而 PCIe 规范支持 256。若设备挂载在 Bus 128 之后，驱动将完全不可见。

**<sup>[14](#mod-drivers-ref-14)</sup> MMIO 基地址硬编码**：`VIRTIO_MMIO_BASE = 0x1000_1000` 和槽位数 8 高度依赖 QEMU 的 virt 平台默认布局，在其他硬件（如 RISC-V 的 HiFive Unleashed）上可能无设备或冲突。

**<sup>[15](#mod-drivers-ref-15)</sup> 轮询无超时机制**：`submit()` 中的 `loop { ... core::hint::spin_loop() }` 没有设置超时计数或 deadline。若设备故障/无响应，调用者将永远阻塞。这是潜在的 DoS 点——用户态通过文件系统发起的读写可能通过此路径永久卡死内核线程。

**<sup>[3](#mod-drivers-ref-3)</sup> REGISTRY 的 Mutex 粒度**：`block_device::read()` 和 `write()` 通过 `with_device()` 访问设备 <sup>[16](#mod-drivers-ref-16)</sup>，持锁期间调用设备 trait 方法。若设备实现中触发缺页或分配（例如 virtio 驱动的 `submit` 内部不分配，但 future 版本可能），可能导致持锁重入，但当前实现由于纯轮询而无阻塞，死锁风险较低。

**<sup>[17](#mod-drivers-ref-17)</sup> unsafe 块安全契约模糊**：`write_desc()`、`write_request_header()` 等函数标记为 `unsafe` 但未提供 `// SAFETY:` 注释说明调用者必须保证哪些不变式（例如 ptr 指向已映射的 DMA 区域、queue_vaddr 的生命周期长于操作）。这是技术债，随着维护者更替容易引入内存损坏。

**<sup>[18](#mod-drivers-ref-18)</sup> 驱动代码重复**：两个 virtio 驱动共享完全相同的队列布局常量（DESC_OFFSET、AVAIL_OFFSET、USED_OFFSET 等）、描述符操作函数、状态管理流程（set_status、STATUS_DRIVER_OK 等），却没有任何公共 virtio 核心库的抽取——每一处都是复制粘贴。且 PCI 版本使用物理地址 `queue_paddr + HEADER_OFFSET` 写入描述符，而 MMIO 版本使用 `data as usize`（直接传虚拟地址），两者的地址空间假设并不一致 <sup>[19](#mod-drivers-ref-19)</sup>，这会在 IOMMU 启用的平台上引入隐蔽的错误。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层采用三层+中央调度表架构：底层 `arch/syscall` + `arch/trap` 负责从用户态陷入并提取参数；中间层 `syscall_args` 提供用户态指针的编组（marshal）服务，包含读写缓冲区和 C 字符串；上层 `syscall_handlers` 按功能域（process/memory/time/file/signal/ipc/socket/misc）实现具体系统调用逻辑；中央 `syscall_table` 通过 `match` 将系统调用号映射到各 handler。与同家族基线相比，该仓库将大量系统调用标记为 mock/unimplemented（如 clone 返回 ENOSYS、prlimit64 返回无限资源），并通过 `need.rs` 自动生成的能力依赖图实现模块间解耦。

### 1. 核心抽象与分层架构

系统调用层被明确拆分为三个层级 + 一张中央调度表，每层只通过 `need.rs` 导入所需能力。

- **架构层**（`arch/syscall.rs`、`arch/trap.rs`）：定义 `TrapFrame` 结构体（保存 32 个通用寄存器、PC、SP、状态、故障地址、原因、用户态标志）<sup>[7](#mod-syscall-ref-7)</sup>，提供 `number()`、`arg()`、`set_result()`、`advance_pc()` 等基本操作。该层不关心具体系统调用号含义，仅负责从寄存器中提取和设置值。
- **编组层**（`syscall_args`）：提供 `marshal` 子模块，封装 `read_buffer()`、`write_buffer()`、`read_path()`、`read_value()`、`write_value()` 等函数，将用户态指针转换为内核可安全访问的字节切片 <sup>[1](#mod-syscall-ref-1)</sup>。底层依赖 `user_copy_raw` 进行实际的 `unsafe` 内存操作。
- **处理层**（`syscall_handlers`）：按功能分为 `process`、`memory`、`time`、`file`（在 `sys_fs` 模块）、`signal`（在 `sys_signal` 模块）、`ipc`、`socket`、`misc` 等子模块。每个子模块实现对应的 Linux 系统调用语义，返回 `isize`（正数为成功值，负数为错误码）。
- **调度层**（`syscall_table`）：一个超过 60 路的 `match` 语句，将 `Syscall` 枚举映射到具体的 handler 函数 <sup>[2](#mod-syscall-ref-2)</sup>。未知系统调用号经过 `Unknown(number)` 分支返回 `ENOSYS` <sup>[5](#mod-syscall-ref-5)</sup>。

这种分层使得架构相关代码与平台无关的系统调用逻辑完全隔离，编组层可以在不改动 handler 的情况下替换用户态访问策略。

### 2. 关键设计取舍与替代方案

**（a）Mock 策略：优先保证启动流程通过**

大量系统调用被实现为“假”版本，例如 `prlimit64` 直接写入硬编码的 `RLIM_INFINITY` 而不检查实际资源 <sup>[4](#mod-syscall-ref-4)</sup>；`setpgid`、`getpgid`、`getsid`、`setsid` 均返回 trivial 值；`getuid`/`geteuid`/`getgid`/`getegid` 统一返回 0（root 身份）。这些假实现并不是为了功能正确，而是为了欺骗 musl libc 的启动阶段，使其认为系统调用成功执行而不报错退出。**代价是完全丢失了安全语义**——例如，所有进程都被视为 root，`prlimit64` 返回无限资源可能被后续恶意程序利用。一个替代方案是返回 `ENOSYS` 并让 libc 自行 fallback，但 musl 对某些系统调用（如 `prlimit64`）的重试行为可能导致启动循环。当前设计是早期开发阶段的务实选择。

**（b）用户态拷贝的安全模型**

`user_copy_raw` 模块提供最底层的 `read_bytes` 和 `write_bytes`，直接对用户传入的裸指针作 `unsafe { core::slice::from_raw_parts }` 而不做任何地址合法性校验 <sup>[1](#mod-syscall-ref-1)</sup>。安全契约被推迟到上层 `syscall_args::marshal`——它通过 `user_memory::read_current` / `write_current` 间接调用 `user_copy_raw`，而 `user_memory` 会检查地址是否在当前进程的用户地址空间内。**这种设计将安全责任放在了中间层，但底层模块本身没有提供任何保护屏障**。如果一个 handler 不小心直接调用了 `user_copy_raw`（而不是通过 `marshal`），就可能产生安全漏洞。在 rCore-Tutorial 中，用户态拷贝通常使用 `copy_from_user` / `copy_to_user` 等显式检查的函数，本仓库的分离设计增加了误用的风险。

**（c）自旋锁 vs 休眠锁**

`ExitedChildren` 队列使用 `AtomicBool` 实现的自旋锁来保护 `UnsafeCell<Option<Vec<ChildExit>>>` <sup>[3](#mod-syscall-ref-3)</sup>。自旋锁在单核 CPU 上不会产生实际等待（因为中断处理期间不会被抢占），但在多核环境下，持有锁的线程可能被调度到另一个核上，导致其他核忙等待浪费 CPU 周期。该实现也没有使用 `core::hint::spin_loop` 的 `pause` 指令（在 x86 上），但 RISC-V 上的 `pause` 可能不可用。一个更好的选择是使用 `spin::Mutex` 或基于 futex 的阻塞锁，但本仓库可能运行在单核模拟环境中，自旋锁的简单性可以接受。

**（d）全局 MMAP 分配器**

`memory::mmap` 使用全局 `AtomicUsize` 变量 `NEXT_MMAP` 以 `fetch_add(Ordering::SeqCst)` 分配地址 <sup>[6](#mod-syscall-ref-6)</sup>，既不回收 `munmap` 释放的区域，也不考虑映射文件的偏移对齐。这会导致地址空间快速耗尽（如果频繁 mmap/munmap）且无法在固定地址上重映射。Linux 内核的 mmap 分配器使用红黑树管理 VMA，本仓库显然牺牲了正确性换取实现速度。

### 3. 跨模块协同：从用户态陷入到系统调用返回

完整路径如下：

1. **陷入**：用户态执行 `ecall`/`syscall` 指令，硬件跳转到 `arch::trap::init()` 注册的入口。架构级代码保存 TrapFrame，检测到 `TrapKind::Syscall` 后调用 `arch::syscall::handle_trap(&mut TrapFrame)`。
2. **原始系统调用处理**：`arch::syscall::handle_trap` 从静态 `TRAP_HANDLER` 单元格取出之前注册的 handler 并调用 <sup>[7](#mod-syscall-ref-7)</sup>。如果未注册 handler，则直接设置返回值 -38（ENOSYS）并 advance PC。
3. **参数提取**：handler（通常由上层模块注册）调用 `syscall_raw::number(tf)` 和 `syscall_raw::arg(tf, i)` 从 TrapFrame 的寄存器中读出系统调用号和最多六个参数，构造 `syscall_args::context::Args`。
4. **参数编组**：对于需要读取用户态数据的系统调用（如 `execve`、`mmap`），handler 调用 `syscall_args::marshal::read_path`、`read_buffer` 等函数。这些函数先通过 `user_memory::current()` 获取当前进程的地址空间，再调用 `user_copy_raw` 进行实际的拷贝。
5. **中央调度**：`syscall_table::dispatch(args)` 根据 `args.number` 匹配到具体 handler，如 `process::execve`、`memory::mmap`、`process::clone` 等。
6. **结果返回**：handler 返回 `isize`。调度函数直接返回该值，上层将其写入 TrapFrame 的返回值寄存器（通过 `syscall_raw::set_result(tf, ret, 0)`），然后调用 `advance_pc(tf)` 将 PC 指向下一条指令。
7. **恢复用户态**：架构级代码执行 `eret`/`sret` 返回用户态。

所有步骤中，`need.rs` 文件自动生成，确保每个模块只能使用其显式导入的能力。例如，`syscall_handlers` 的 `need.rs` 导入了 `user_memory`、`exec`、`exit_wait` 等模块，而 `syscall_table` 的 `need.rs` 导入了 `number`、`process`、`memory`、`file`、`signal`、`time`、`ipc`、`socket`、`misc` 等模块。

### 4. 边角细节与不足

本模块存在多处明显的早期开发痕迹和潜在问题：

- **`clone` 完全未实现**：`process::clone` 仅输出警告后返回 `ENOSYS` <sup>[2](#mod-syscall-ref-2)</sup>。这意味着任何依赖 fork 或线程创建的程序（包括 shell）都无法正常工作。该接口的签名虽然完整，但功能缺失使多进程能力为零。
- **自旋锁忙等**：`ExitedChildren` 的 `with` 方法使用 `compare_exchange` 忙等待 <sup>[3](#mod-syscall-ref-3)</sup>。在单核模拟中可能无影响，但若移植到多核平台，没有指数退避或 `yield` 会导致 CPU 资源浪费。
- **用户态指针无底层校验**：`user_copy_raw::read_bytes` 直接解引用用户传入的指针 <sup>[1](#mod-syscall-ref-1)</sup>，`read_cstr` 逐字节读取直到 NUL 或最大长度 <sup>[4](#mod-syscall-ref-4)</sup>。虽然上层 `marshal` 通过 `user_memory` 做了范围检查，但底层的 `unsafe` 块未遵循“最小权限”原则——如果未来有代码绕过 `marshal` 直接调用 `user_copy_raw`，就会引入任意地址读写漏洞。
- **硬编码 argv/envp 上限**：`read_argv` 最多读取 64 个参数 <sup>[8](#mod-syscall-ref-8)</sup>，`read_envp` 最多读取 128 个环境变量。Linux 没有这类硬限制，恶意或畸形程序如果提供超长列表，会截断后续数据，可能导致不确定性行为（例如 `execve` 使用不完整的 argv 启动程序）。
- **全局 NEXT_MMAP 不回收**：mmap 使用静态原子变量递增分配地址 <sup>[6](#mod-syscall-ref-6)</sup>，`munmap` 仅解除映射但不标记地址为空闲。重复 mmap/munmap 会导致地址空间碎片并最终耗尽用户空间（默认从 `0x5000_0000` 开始）。
- **TrapFrame 的 `from_user` 未充分利用**：`arch::trap::TrapFrame` 包含 `from_user` 字段，但 `syscall_raw` 的 handler 未根据该字段做任何隔离判断。恶意用户程序若通过某种方式在内核态构造 TrapFrame 并触发系统调用，本应被拒绝，但当前实现不会检查。
- **prlimit64 忽略进程过滤**：`prlimit64(mock)` 完全不检查 `pid` 参数，即任何进程都可以读取或设置任何其他进程的资源限制（尽管返回值是 mock 的无限值）<sup>[4](#mod-syscall-ref-4)</sup>。这违反了 Linux 的权限检查。

尽管存在上述不足，系统调用层的整体分层设计和能力依赖机制在后续完善中能提供良好的重构基础：每个 mock 函数都可以在不改动签名的情况下替换为真实实现，且 `need.rs` 保证了依赖关系不会循环。

### ⚠ 开放问题

- clone 系统调用返回 ENOSYS，完全不具备进程创建能力，多进程程序无法执行。
- ExitedChildren 自旋锁使用忙等待（无退避、无 yield），多核环境下浪费 CPU。
- user_copy_raw 的 read_bytes/write_bytes 直接解引用用户指针，缺乏底层地址校验，依赖上层模块保证安全。
- 全局 NEXT_MMAP 分配地址不回收已释放区域，反复 mmap/munmap 会导致地址空间耗尽。
