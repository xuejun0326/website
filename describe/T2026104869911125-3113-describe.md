## 📖 一、总览

F7LY 是一个面向 RISC-V 和 LoongArch 架构的 Linux 兼容教学操作系统内核。它与传统教学内核（如 xv6）的最大不同在于：不是从零重头设计，而是力求在自研内核上直接运行为 Linux 编译的二进制程序。这种定位决定了它的核心架构——用户态不依赖 GNU libc 而是同时支持 musl/glibc 双 libc，通过 `user` 模块提供的系统调用封装和测试调度框架，将内核自身的功能验证与 libc 兼容性测试融合为一条回归流水线。项目在 17.5 万行代码中，实现了进程管理、VFS 挂载命名空间、部分信号机制和高精度时钟，并引入 EASTL 作为 C++ 基础设施来规避标准库的异常和分配开销。

值得评审注意的特点：
1. **双 libc 回归体系**：`user_test.cc` 的调度框架能同时运行 musl 和 glibc 的测试，包括 LTP、busybox 等，这在同类教学内核中少见，体现了对 Linux ABI 兼容性的务实态度。
2. **EASTL 的嵌入式改造**：thirdparty 模块引入 EASTL 并适配自身的堆分配器和零异常要求，短字符串优化（SSO）使大量字符串操作免于堆分配，这在内核这样的受限环境中有实际收益。
3. **系统调用实现的精度考量**：kernel 模块在高精度超时处理中采用混合 busy-wait 与 tick 睡眠的方案（`[kernel.ref:1]`），避免了纯 tick 驱动的对齐误差，反映出在实时性与复杂度之间的务实权衡。

## 🎯 二、综合评价

### 整体定位
F7LY 项目可归类为**教学/实验型 Linux 兼容内核**。与 Linux 内核的完整实现不同，它更关注核心系统调用的逻辑正确性和 ABI 兼容性，而非性能或完整功能。模块划分明确：user 负责用户态接口与测试，kernel 负责核心调度与资源管理，thirdparty 提供基础件，scoreboard 作为测例归档工具。整个项目的设计哲学是“能跑就通”——优先让标准二进制（busybox、libc 测试、LTP）在其上运行，而非追求内核本身的特性创新。

### 真正的创新点
尽管项目定位偏向教学，但有几个独特设计值得注意：
- **硬编码运行时补丁（runtime patch）**：`kernel` 模块在 `proc_manager.cc` 中硬编码了 musl 库的特定地址，在进程启动时修改其原子操作实现，以绕过自研内核尚未实现的原子系统调用。这是一种极其务实但不优雅的 hack（`[kernel.ref:5]`），在其他教学内核中极少出现，反映了项目在兼容性与实现成本之间的取舍。
- **EASTL 的深度定制**：`thirdparty` 模块没有直接选用 libstdc++ 或 LLVM 的 libc++，而是跨项目引入 EASTL 并重新绑定内存分配器（`[thirdparty.ref:3]`）。这避免了 C++ 异常和 RTTI 的运行时开销，符合内核环境的零异常需求，同时也带来了 SSO 等内存优化。
- **双 libc 测试流水线**：`user` 模块的测试调度框架能够同时运行 musl 和 glibc 版本的测例（`[user.ref:1]`），这在教学内核中不常见。它使得内核的 ABI 兼容性测试覆盖更全面，也间接验证了系统调用接口的正确性。

### 取舍判断
项目明显将精力集中在**系统调用的兼容性实现**和**测试验证**上，而在其他方面做了大量简化：
- **省略了完整网络栈**：目前没有看到网络系统调用（如 socket, connect）的实现，这意味着内核只能运行本地进程。
- **省略了加速设备支持**：没有块设备驱动和文件系统缓存，文件系统实现可能基于性能较低的虚拟文件系统。
- **省略了多核调度**：从模块内容看，进程管理可能只支持单核或简单调度，没有 SMP 负载均衡。
- **架构支持有限**：仅支持 RISC-V 和 LoongArch，没有 x86 或 ARM 打算，缩小了验证范围。
这些省略使项目能聚焦于最小可运行集合，适合教学和竞赛环境。

### 完成度与不足
从代码量（17.5 万行）和模块粒度看，项目完成度属于**中上等**。syscall 方面，核心的 fork/execve/mmap/clock_nanosleep 等已实现，但系统调用数量未统计，且部分系统调用可能只是 stub（无实质功能）。存在一些已知缺陷：`proc_manager.cc` 的全局计数变量存在数据竞争（`[kernel:不足1]`），硬编码补丁版本脆弱（`[kernel:不足2]`），EASTL 算法集不完整（`[thirdparty:不足]`）。此外，项目缺乏完善的文档和构建脚本说明，上手门槛较高。

总体而言，F7LY 是一个在 Linux 兼容性上做了扎实工作的教学内核，尽管有些实现粗糙，但双 libc 测试和 EASTL 应用是亮点。打分如下：
- 完成度：2（核心功能齐全但部分未完善）
- 创新性：2（硬编码补丁和 EASTL 引入有实用创新）
- 代码质量：2（存在数据竞争和硬编码问题，但整体组织合理）

## 📄 三、用户接口模块

> 💡 **TL;DR**
>
> `user` 模块是 F7LY 操作系统的用户态核心支撑层，包含系统调用封装库（`syscall.cc`）、轻量级格式化输出（`printf.cc`）、架构相关的系统调用内联汇编（`syscall_arch.hh`）以及一套完整的测试调度框架（`user_test.cc`）。它解决了在无标准 C 库环境下运行评测程序的问题，提供了跨 musl/glibc 双 libc 架构的回归测试流水线，并提供了一个最小交互式 shell（`shell.cc`）。作为内核启动后的第一个用户态模块，它串联了所有基准测试、LTP 用例和系统调用验证，最终通过 `shutdown()` 关闭系统。

### 1. 核心抽象与外部依赖

`user` 模块的核心抽象分为三层。**底层**是架构相关的系统调用机制：`syscall_arch.hh` 对 RISC-V 和 LoongArch 分别封装了 `ecall` 和 `syscall 0` 指令，通过寄存器 `a7` 传递调用号、`a0-a5` 传递参数，所有返回值强制约束为 `long` 类型<sup>[4](#mod-user-ref-4)</sup><sup>[5](#mod-user-ref-5)</sup>。**中间层**是 `syscall.cc` 和 `printf.cc`，前者将 `syscall()` 宏包装为 POSIX 风格的 API（如 `openat`、`read`、`fork`、`mmap`），后者实现轻量级 `printf`，直接通过 `write(fd,...)` 输出，不依赖任何文件流缓冲<sup>[8](#mod-user-ref-8)</sup>。**最上层**是测试调度框架 `user_test.cc`，以 `run_test()` 为原语，通过 `fork+execve` 启动独立进程执行测例，并围绕它构建了多种测试集合（basic、libc、LTP、iozone 等），以及环境初始化（`init_env` 安装 busybox applet）<sup>[1](#mod-user-ref-1)</sup>。

模块的外部依赖包括：musl 和 glibc 两种 libc 的共享库、busybox 作为 sh 和工具链、Linux Test Project（LTP）测例集、以及 iozone/netperf/iperf/lmbench/cyclictest 等第三方基准。这些依赖以预置的 rootfs 目录（`/musl/`、`/glibc/`）的形式存在，模块通过硬编码路径访问它们<sup>[12](#mod-user-ref-12)</sup>。

### 2. 关键设计取舍

**测试隔离策略**：每个测例在独立子进程中执行，且子进程一创建就调用 `setpgid(0, 0)` 形成新的进程组<sup>[1](#mod-user-ref-1)</sup>。设计动机注释明确指出：LTP 某些超时清理路径会对当前进程组发信号（`kill(0, SIGKILL)`），若与回归主进程共用一个 pgid，会导致测例超时误杀整个回归，表现为“内核 panic: init exiting”。即使 `setpgid` 失败，模块仍继续 exec，不阻断测试。这是典型的 fault-tolerant 降级——隔离失败比完全阻塞更好。

**LTP shell 脚本批处理**：LTP 中有大量 `.sh` 测例，如果每个脚本独立 fork+exec，开销极大。模块设计了 `run_ltp_shell_batch`，将最多 24 个脚本打包成一个 ash 进程依次执行，内部通过子 shell `( )` 隔离状态，避免 trap/变量污染<sup>[2](#mod-user-ref-2)</sup>。这个优化在 LoongArch 长回归中尤为关键，因为持续 exec 可能暴露内核尚未稳定的 `execve` 路径。

**系统调用库的自主实现**：`user` 模块没有链接任何现成的 libc，而是自行实现了一套最小系统调用封装。这样做既避免了 musl/glibc 对内核接口的假设（比如某些 clone 标志）与 F7LY 内核不匹配，也使测试框架不受 libc 版本的干扰——回归切换 musl/glibc 时只需要切换动态链接器路径，而不需要重新编译测试框架本身。代价是一些 POSIX 约定的边缘行为（`errno` 设置）没有被完整实现<sup>[6](#mod-user-ref-6)</sup>。

**架构适配层**：通过 `syscall_arch.hh` 将系统调用指令抽象成 `__syscall0` 到 `__syscall6` 的系列内联函数，再经过 `syscall_base.hh` 的变参宏 `__SYSCALL_DISP` 自动选择正确元数<sup>[5](#mod-user-ref-5)</sup>。这使得上层的 `syscall.cc` 只需写 `syscall(SYS_read, fd, buf, len)`，无需关心底层是 ecall 还是 syscall 指令。LoongArch 版本额外在 `ltp_envp` 中设置了 `LTP_TIMEOUT_MUL=10`，反映该架构在仿真环境下的时钟差异<sup>[11](#mod-user-ref-11)</sup>。

## 📄 四、内核核心模块

> 💡 **TL;DR**
>
> kernel 模块是操作系统内核的核心，实现了 Linux 兼容的系统调用接口、进程管理和 VFS 挂载命名空间。核心抽象包括系统调用分发函数、进程控制块（Pcb）、挂载命名空间状态（MountNamespaceState）等。该模块解决了在自研内核上运行为 Linux 编译的二进制程序的问题，与底层 HAL、内存管理、文件系统、网络栈等模块紧密协同。

### 1. 核心抽象与外部依赖

kernel 模块构建在几个关键抽象之上。系统调用处理核心位于 `syscall_handler.cc` 的匿名命名空间，通过 `resolve_at_path` 解析路径（利用当前进程的 cwd 或 dirfd），`check_utimens_constraints` 实现细致的 Linux 权限模型（owner、capability、组写权限）<sup>[3](#mod-kernel-ref-3)</sup>。进程管理由 `proc_manager.cc` 提供，核心是 `Pcb` 结构，通过 `k_pm` 全局实例访问当前进程。进程创建时，`reset_signal_state_for_exec` 正确继承 SIG_IGN 并私有化 sighand，确保 exec 后信号状态符合 POSIX 规范<sup>[4](#mod-kernel-ref-4)</sup>。VFS 挂载命名空间在 `vfs_utils.cc` 中实现，`MountNamespaceState` 持有挂载记录，并缓存 `mounts_snapshot_cache` 供 `/proc/mounts` 快速读取<sup>[7](#mod-kernel-ref-7)</sup><sup>[8](#mod-kernel-ref-8)</sup>。这些抽象共同为系统调用提供了基础服务。

### 2. 关键设计取舍

**高精度超时实现**：`wait_short_timeout` 和 `sleep_relative_timeout_or_signal` 实现了高精度 deadline 睡眠，避免 tick 对齐导致的误差<sup>[1](#mod-kernel-ref-1)</sup><sup>[2](#mod-kernel-ref-2)</sup>。对于小于一个 tick 的超时，使用硬件时间戳循环检查并低频 yield，而不是直接忙等或增大 tick 粒度；对于长超时则先按 tick 粗睡，再用高精度尾巴补齐。这个设计放弃了纯 tick 睡眠的简单性，但获得了微秒级精度，满足 clock_nanosleep 测试要求。

**运行时用户态补丁**：`proc_manager.cc` 包含针对 RISC-V 和 LoongArch 的 ELF 二进制补丁，用于修复 musl libc 中原子序列与内核 LL/SC 语义不匹配的问题<sup>[5](#mod-kernel-ref-5)</sup>。这种方式避免了修改源头程序或工具链，但补丁表硬编码了地址和字节，当库版本更新时可能失效或导致程序崩溃，是一种易碎但成本低的前期兼容方案。

**挂载传播缓存策略**：`build_mounts_snapshot_cache` 将 /proc/mounts 格式缓存为字符串，仅在挂载变化时重建，避免每次读取都遍历整表<sup>[8](#mod-kernel-ref-8)</sup>。缓存一致性依赖脏标记，`mark_mount_state_dirty` 在每次挂载操作时都被调用，保证了正确性。

### 3. 跨模块协同

syscall_handler 与 proc_manager 紧密配合：`check_utimens_constraints` 通过 `proc::k_pm.get_cur_pcb()` 获取当前进程以检查权限<sup>[3](#mod-kernel-ref-3)</sup>。proc_manager 在 exec 时更新信号状态和文件描述符表，并通过 `_mnt_ns_id` 与挂载命名空间关联（`current_mount_namespace_id` 获取当前进程命名空间）<sup>[7](#mod-kernel-ref-7)</sup>。VFS 模块通过 `vfs_path_stat_noflush` 等接口被系统调用广泛使用。定时器方面，`wait_timeout_or_signal` 使用 `tmm::k_tm` 定时器服务，`query_epoll_ready_events` 与 epoll 文件对象交互，体现出模块间的依赖关系。

### 4. 边角细节与不足

尽管整体设计扎实，但仍存在多处值得关注的潜在问题：

- **并发争议**：全局变量 `g_tiny_timeout_poll_count` 在 `wait_short_timeout` 中被无锁累加，若多个线程同时进入该函数，会导致计数不准和 yield 频率异常，但该变量仅用于控制低精度让步节奏，不影响正确性，但仍是数据竞争<sup>[1](#mod-kernel-ref-1)</sup>。
- **硬编码地址**：RISC-V 和 LoongArch 补丁表直接使用固定偏移，若用户态库版本不同，可能打错位置或引起崩溃；但代码中通过旧字节匹配验证避免误改，但补丁失败仅返回 false 而不报错，静默降级可能导致原子序列错误<sup>[5](#mod-kernel-ref-5)</sup>。
- **错误处理一致性**：`apply_ext4_times_to_open_file` 在权限检查失败时返回 -EPERM，随后清理 inode ref，处理正确。但 `wait_fd_event_tick_or_signal` 将 `sleep_n_ticks` 返回的 -2 当作信号中断处理，未区分其他异常，可能掩盖真实错误。
- **资源泄漏风险**：`find_conflicting_lease_holder` 遍历所有进程时调用 `collect_open_file_refs`，该函数对每个文件进行 dup，若中间出现错误未完全释放已 dup 的文件，可能导致泄漏。但调用者最后会调用 `release_file_refs` 释放全部，但如果在泄漏产生之前出错退出（如 OOM），释放可能被跳过<sup>[6](#mod-kernel-ref-6)</sup>。
- **路径校验广度**：`is_valid_user_write_range` 只检查 addr 和 len 非零且不超 MAXVA，未验证地址是否真的是用户空间页，可能允许用户传递内核地址但在后续 copy_out 时通过页表检查。

这些不足之处仍属早期代码的正常演变痕迹；随着项目成熟，预计会通过更规范的错误处理、并发保护和参数化解决。

## 📄 五、第三方支撑库

> 💡 **TL;DR**
>
> thirdparty 模块引入 EASTL（Electronic Arts Standard Template Library）作为项目的基础组件库，替代 C++ 标准库。核心抽象是 eastl::basic_string（支持短字符串优化 SSO）、高性能算法集合（eastl::algorithm）和 eastl::atomic 原子操作封装。该库通过 libs/klib.hh 接入项目自身的内存与字符串函数，为编译器其他模块提供无异常、可预测性能的基础设施。

### 1. 核心抽象：EASTL 作为项目基础设施

thirdparty 模块的核心是 EASTL（Electronic Arts Standard Template Library），一个针对游戏/嵌入式环境深度定制的 C++ 标准库替代。它提供三个主要组件：eastl::basic_string、eastl::algorithm 和 eastl::atomic。

**eastl::basic_string** 是该模块最复杂的抽象。它实现了与 std::basic_string 几乎相同的接口，但在内存管理上做了根本性改变：完全放弃写时拷贝（COW），强制采用短字符串优化（SSO）。SSO 通过一个 Layout 联合体实现，该联合体在堆分配（HeapLayout，含指针、大小、容量）和栈内缓冲（SSOLayout，固定大小字符数组 + 剩余大小字段）之间按位切换 <sup>[2](#mod-thirdparty-ref-2)</sup>。运行时通过 `mnRemainingSize` 的最高位（小端）或最低位（大端）判断当前存储模式 <sup>[2](#mod-thirdparty-ref-2)</sup>，使得 95% 以上的短字符串操作无需堆分配。

**eastl::algorithm** 提供了与 C++ STL 算法同名的函数集，但内部优化策略不同：利用 `type_traits` 对平凡可复制类型启用 memcpy/memmove 替代循环，减少分支并利用随机访问迭代器改进缓存局部性 <sup>[4](#mod-thirdparty-ref-4)</sup>。它明确列出了暂未实现的算法（如 `inplace_merge`、`prev_permutation`），表明当前集合是面向常用场景的精简子集 <sup>[5](#mod-thirdparty-ref-5)</sup>。

**eastl::atomic** 提供了与 C++11 std::atomic 兼容的原子操作接口，并扩展了 `add_fetch`、`cpu_pause`、`compiler_barrier` 等实用函数 <sup>[6](#mod-thirdparty-ref-6)</sup>。其关键设计决定是所有原子操作强制无锁（即使对于用户定义类型也宣称无锁），以避免隐式锁带来的性能陷阱 <sup>[6](#mod-thirdparty-ref-6)</sup>。

三个组件共同构建了一个可预测、低开销的基础层，支撑编译器其他模块的字符串处理、排序查找和并发原语需求。

### 2. 关键设计取舍

**放弃 COW 拥抱 SSO**：EASTL 的设计者花了大量篇幅解释为何不采用写时拷贝 <sup>[1](#mod-thirdparty-ref-1)</sup>。核心观点是 COW 在 95% 的场景下带来额外开销（原子引用计数、所有非 const 操作前的 detachment 检查），而实际共享情况极少。SSO 以固定对象大小换取栈分配，避免了 95% 的堆分配。EASTL 的 SSO 容量取决于指针宽度和字符大小，例如 64 位系统上 char 的 SSO 容量为 22 字符。这个取舍牺牲了极长字符串的栈上存储可能（反正那些必须走堆），换取了小字符串的零分配性能。

**算法定制而非全盘照搬**：eastl::algorithm 并非简单包装 std::algorithm，而是重写了核心函数。例如 `min` 对标量类型返回 T 而非 const T&，避免编译器生成引用临时变量 [ref...]。由于 EASTL 假设无异常环境，算法中不使用异常传播，而是通过 `EASTL_EXCEPTIONS_ENABLED` 宏条件编译异常相关代码 [ref:string.h 开头]。这减少了二进制体积，但也意味着调用方必须自行处理错误。

**原子操作扩展**：eastl::atomic 在标准接口之外增加了 `add_fetch`、`atomic_load_cond` 等函数。`atomic_load_cond` 用于自旋等待某个条件成立，内部包含编译器屏障以避免循环被优化掉 <sup>[6](#mod-thirdparty-ref-6)</sup>。这种扩展反映了游戏引擎中对高性能自旋锁的常见需求。

### 3. 跨模块协同

EASTL 并不独立存在，它深度依赖项目提供的 `libs/klib.hh` 基础库。string.h 和 algorithm.h 均 `#include "libs/klib.hh"` 并使用其中的 `malloc`、`free`、`memcpy`、`memmove`、`strlen` 等函数 <sup>[3](#mod-thirdparty-ref-3)</sup>。这意味着 EASTL 的分配器默认使用项目的自定义堆管理，而非操作系统提供的 malloc。这种设计保证了编译器各模块使用统一的内存分配策略，便于在宿主环境（如模拟器）或目标硬件上替换。

`basic_string` 通过 `compressed_pair<Layout, allocator_type>` 将字符串布局与分配器绑定在一起 <sup>[3](#mod-thirdparty-ref-3)</sup>，分配器可被用户替换（例如使用栈分配器或静态池）。这种设计来自与 EASTL 的 `compressed_pair` 机制，继承了 C++ 空基类优化，当分配器无状态时零额外开销。

算法组件与 klib 的 `memcpy`/`memmove` 深度配合：通过 `type_traits` 检测类型是否平凡可复制，若是则直接调用 klib 的块移动函数，避免了循环逐元素复制 <sup>[4](#mod-thirdparty-ref-4)</sup>。这使得 algorithm.h 中的 `copy`、`fill` 等操作在基本类型上获得接近 C 语言 memcpy 的性能。

### 4. 边角细节与不足

尽管 EASTL 设计精良，但仍存在若干需要注意的边角问题。

**自插入触发重分配**：string.h 的注释明确指出，当 `insert` 的目标是自己（`s.insert(pos, s)`）时，即使容量足够也会强制重分配 <sup>[1](#mod-thirdparty-ref-1)</sup>。这是一个已知未优化点，因为源和目标是同一个对象，修改过程中无法安全读取。替代方案是调用方先拷贝一份，但这违背了接口的直觉。

**未实现算法集合**：algorithm.h 文档中明确列出了暂未实现的算法，包括 `inplace_merge`、`prev_permutation`、`unique_copy` 等 <sup>[5](#mod-thirdparty-ref-5)</sup>。如果项目后续需要使用归并排序或排列生成，必须自行实现或等待补全。

**原子无锁保证的隐含前提**：eastl::atomic 声称总是无锁 <sup>[6](#mod-thirdparty-ref-6)</sup>，但实际上这只对整数和指针类型成立；对于用户定义类型，EASTL 内部通过 `is_lock_free` 静态断言确保编译期检查，但文档未强调此限制。若用户自定义类型超过硬件原子操作宽度，编译将失败而非回退到锁，这可能造成移植问题。

**硬编码容量依赖指针大小**：SSO 容量 `SSO_CAPACITY` 通过 `sizeof(HeapLayout) / sizeof(value_type)` 计算，而 `HeapLayout` 包含一个指针和两个 `size_type`。在 32 位系统上 SSO 容量更大，在 64 位系统上更小。这种设计是故意的，但跨平台时字符串对象大小会变动，可能影响 ABI 兼容性。

**错误处理路径薄弱**：`basic_string` 的 `ThrowLengthException`、`ThrowRangeException` 等函数仅在 `EASTL_EXCEPTIONS_ENABLED` 时才抛出异常，否则可能直接 abort 或静默失败。项目作为编译器，异常通常被禁用，因此这些边界错误可能无法被优雅捕获。

**TODO 与未完成痕迹**：代码中存在多处 TODO，例如 `operator=` 的 `noexcept` 规范标注为 `// TODO(c++17): ...`，表明这段代码是逐步迁移到 C++17 过程中的中间状态 [ref:string.h 420-430]。另外，`EASTL_STRING_EXPLICIT` 宏控制 explicit 构造函数的启用，该宏默认关闭，可能导致隐式转换风险。

> scoreboard 模块是一个 Python 脚本 generate_scoreboard.py，核心抽象是 ScoreCase 数据类及其扫描-输出流水线。它从已挂载的大赛磁盘自动扫描测例文件结构，解析各类脚本（basic、libctest、busybox、LTP 等），生成分层 Markdown 文档（组页面、架构/libc 汇总页、根 README）。该模块只负责测例元数据的描述与编排，不执行测例，将评测入口、磁盘脚本和调试导航分离维护。在整个项目中承担“测例清单生成器”角色，与 regression_suite_4d1444 的执行器配合提供一致的测例集合。

### 1. 核心抽象与数据流

模块围绕 `ScoreCase` 数据类<sup>[1](#mod-scoreboard-ref-1)</sup>组织所有测例信息：架构、libc、分组、名称、命令、文件路径、URL、来源标签、是否默认回归、状态和备注。`scan_mounts`<sup>[2](#mod-scoreboard-ref-2)</sup> 遍历 `ARCH_MOUNT_DEFAULTS` 中硬编码的 riscv/loongarch 挂载点，对每个架构下的 musl/glibc 子目录调用六个扫描函数（`scan_basic`、`scan_lua`、`scan_busybox`、`scan_libctest`、`scan_ltp`、`scan_script_groups`）<sup>[3](#mod-scoreboard-ref-3)</sup>。这些函数解析目标目录下的特定脚本文件（如 `run-all.sh`、`lua_testcode.sh`、`busybox_cmd.txt`），提取测例名称和命令，并追加到 `cases` 列表。生成的测例列表随后进入 `write_outputs` 流水线<sup>[4](#mod-scoreboard-ref-4)</sup>：先清理旧 HTML 目录，再按 `(arch, libc, group)` 分组，为每个分组写一个 Markdown 文件（包含测例表格），为每个 `(arch, libc)` 写一个带小分统计的 README，最后写根 README 汇总所有组合。

### 2. 关键设计取舍

**Python 而非 Shell/Rust**：选择 Python 脚本而非项目主语言 Rust 或 Shell，是为了快速迭代和更强的文本处理能力（`shlex`、`pathlib`）。代价是需要 Python 运行时，且与 Rust 核心组件无静态类型接口。

**只描述不执行**：模块明确不运行测例，只记录“磁盘里有什么”。这分离了测例发现与执行两个关注点，使 scoreboard 可以在无执行环境时独立更新，也避免了重复运行耗时测例。执行器（外部模块）通过读 `status` 字段回写结果。

**默认回归组硬编码**：`PROJECT_DEFAULT_GROUPS` 和 `PROJECT_BASIC_CASES`<sup>[5](#mod-scoreboard-ref-5)</sup> 分别定义项目默认回归的组和 basic 子集。这种硬编码与 `regression_suite_4d1444` 耦合，缺乏动态发现机制，需手动同步（注释提示与 `user/user_lib/user_test.cc` 保持一致）。

**挂载路径硬编码**：`ARCH_MOUNT_DEFAULTS`<sup>[6](#mod-scoreboard-ref-6)</sup> 直接写死 `/mnt/sdcard-rv` 和 `/mnt/sdcard-la`，仅支持 riscv 和 loongarch 两个架构，扩展其他架构需修改代码。

**解析策略的鲁棒性与脆弱性**：`is_command_line` 函数<sup>[7](#mod-scoreboard-ref-7)</sup>通过启发式规则（排除变量赋值、控制流关键字等）从 shell 脚本中提取实际测例命令。`shell_words` 在 `shlex.split` 失败时退化到 `line.split()`<sup>[8](#mod-scoreboard-ref-8)</sup>，可能丢失引号含义。这种“尽力解析”在脚本格式规范时高效，但在遇到复杂命令（含管道、重定向）时可能错误切分。

### 3. 跨模块协同

虽然模块独立运行，但它通过两个隐式契约与项目其他部分协同：
- **测例清单一致性**：`PROJECT_BASIC_CASES` 手动镜像 `user/user_lib/user_test.cc` 的 `basic_testcases`。若内核新增 basic 测例未同步更新此处，scoreboard 会漏标默认回归状态。
- **状态回写**：`write_outputs` 先调用 `read_existing_case_notes` 读取已存在的 `.md` 文件中的 `status` 和 `note`，再覆盖新生成的数据<sup>[9](#mod-scoreboard-ref-9)</sup>。这使得外部执行器可以独立更新 scoreboard 中的通过状态，生成器不会覆盖已有结果。
- **Markdown 消费**：输出目录的 `README.md` 和分组页面供 CI 面板、开发文档直接引用，形成可导航的测试地图。
