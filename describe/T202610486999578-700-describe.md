## 📖 一、总览

该仓库属于 arceos-starry 家族中追求 Linux 兼容性的实验分支。它基于 ArceOS 的 axmod 模块化架构，但通过大量扩展实现了 Linux 进程模型、文件系统、信号、IPC、网络等子系统，目标是能够运行 musl 编译的用户态程序并通过 LTP 测试。整体上，它比原版 ArceOS 更接近完整内核，但仍处于实验阶段，许多功能仅为通过特定测试而实现。

本仓最值得评审注意的两三个特点在于：第一，抽象层次上的创新——将地址空间后端策略解耦为 Backend 枚举，支持 COW、共享文件映射等，同时利用 ActiveScope 实现进程资源隔离。第二，系统调用覆盖范围突出——加入了 eBPF 解释器、netlink 协议栈、System V IPC 等大量 Linux 特有机制。第三，测试驱动的设计导向——从编译期嵌入 init.sh（[boot.ref:2]）到硬编码 LTP 接口名（[net.ref:5]），处处可见以通过测试为优先的取舍。

## 🎯 二、综合评价

### 整体定位：面向 Linux 兼容的实验性分支

本仓库是 arceos-starry 家族中一个以 Linux 应用兼容为目标的实验性内核。它不追求从零开始的创新，而是以 ArceOS 的模块化框架为基础，通过大量定制来覆盖 Linux 系统调用集和 POSIX 语义。与同族其他成员相比（如教学型的 rCore-Tutorial），本仓明显更侧重于功能广度与测试通过率：启动阶段嵌入 init.sh 脚本（[boot.ref:2]），硬编码 LTP 网络接口名（[net.ref:5]），以及为 pause 系统调用号跨架构特殊适配（[syscall.ref:3]），都体现了“以能跑 Linux 用例”为首要目标的设计哲学。同时，它仍然保留了 ArceOS 的 `#![no_std]` 风格和组件化架构，在嵌入式场景中具有微内核的轻量优势。

### 真正的创新点

本仓的创新并非发明全新的 OS 概念，而是在 ArceOS 的约束下，用简洁而巧妙的方式实现了 Linux 的复杂语义。以下几个技术选型值得注意：

1. **写时复制与全局物理页引用计数**（[mm.ref:5]）：通过 `FRAME_TABLE` 全局 BTreeMap 维护物理帧的引用计数，使得 fork 时的 COW 映射、克隆、释放得以安全高效地实现，这是 fork 性能的关键支撑。
2. **基于 ActiveScope 的任务资源隔离**：task 模块通过 `TaskExt` trait 在任务切换时自动切换 `ActiveScope`，实现了每个进程独立的文件描述符表、网络命名空间等资源视图，无需修改底层调度器，体现了关注点分离。
3. **Bind mount 的代理 DirEntry 机制**：文件系统层通过创建 `BindMountFs` 和代理 `DirEntry` 实现 bind mount，避免了传统 VFS 的挂载堆叠复杂性，是一种轻量而优雅的设计。
4. **相对完整的 netlink 实现**（[net.ref:3]、[net.ref:4]）：从旧版仅处理少数消息类型到新版完整支持 RTNL 消息族，覆盖了链路、地址、路由等核心操作，在同类微内核中较为罕见。

### 取舍判断：以测试通过为导向的务实选择

本仓在多处有意牺牲了完整性或性能以换取功能覆盖度：

- **启动流程**：舍弃了独立 initramfs 和环境变量传递（[boot.ref:3]），通过编译期嵌入 init.sh 并直接构建 argv，简化了启动路径，但导致子进程无环境变量继承，可能影响依赖 `PATH` 的脚本。
- **信号机制**：CoreDump、Stop、Continue 三种默认动作未实现（[signal.ref:4]），因此 `SIGTSTP`、`SIGQUIT` 等信号的行为与 Linux 不符，明显是对 LTP 测试中较少覆盖的边缘行为所做的妥协。
- **IPC 阻塞等待**：消息队列和 `futex_waitv` 均使用 busy-poll + `yield` 而非条件变量（[ipc.ref:2]），避免了更复杂的 waitqueue 实现，但高竞争场景下会浪费 CPU。
- **网络**：强制创建命名空间并预创建 veth 对（[net.ref:5]），通用性不足，显然是针对 LTP 网络测试的特化。

这些取舍表明作者的精力集中在“让核心路径跑通”，而非追求通用性和高性能。

### 完成度与不足

整体完成度中等偏上。系统调用覆盖 263 项，数量可观；核心机制如 fork/exec/mmap/futex 等已实现，能运行 musl 编译的用户程序。但从质量上看，代码中存在较多未完成部分：大量 TODO（如 signal 模块默认动作未实现 [signal.ref:4]）、实现与注释不一致（[boot.ref:2]）、缺页处理仅记录日志而不 kill 进程的潜在风险（[mm.ref:1]）、以及平台特化代码的 `cfg` 遗漏风险（[boot.ref:4]）。此外，全局静态管理器缺乏锁层级声明，存在死锁风险（如 `sys_kill(-1)` 遍历进程时的循环等待，见 [signal.ref:4] 相关分析）。整体上，这是一个功能广度优先、深度其次的实现——它知道用户空间需要什么，但往往只做到了恰好能通过测试的程度。

### 1. 核心抽象与外部依赖

启动流程的核心抽象是 **“编译期嵌入 init 脚本 + axhal 硬件抽象层”** 的组合。

入口文件 `src/main.rs` 声明为 `#![no_std]` 和 `#![no_main]`<sup>[1](#mod-boot-ref-1)</sup>，表明这是一个无标准库的裸机程序，由链接器脚本或运行时库提供真正的入口点。`extern crate alloc`<sup>[4](#mod-boot-ref-4)</sup> 启用了 Rust 的堆分配支持，为后续的 Vec 和 String 分配提供基础。

关键的启动数据——init.sh 脚本——通过 `include_str!("init.sh")`<sup>[2](#mod-boot-ref-2)</sup> 在编译期直接嵌入到二进制文件的 `.rodata` 段。这种设计避免了早期启动时对文件系统的依赖（此时存储驱动尚未初始化），但同时也放弃了运行时替换脚本的灵活性。`build_init_sh_c_argument()` 函数将嵌入的脚本内容包装为 `String`，注释中提到该脚本会在启动时物化到 `/tmp/starry_init.sh`<sup>[2](#mod-boot-ref-2)</sup>，但实际观察到的调用链中仅将其作为字符串参数直接传递，并未执行物化操作——这是一个值得注意的**实现与注释的偏离**。

`boot_style` 标注为 `axhal`，表明硬件初始化（如页表、异常向量表、多核唤醒）委托给 axhal crate。该 crate 是 ArceOS 生态的标准硬件抽象层，提供统一的 `HalTrait` 接口。本仓的启动流程通过 `starry_kernel::entry::init` 进入内核后，由 axhal 完成 CPU 初始化、Trap 向量安装等底层操作。

### 2. 关键设计取舍

**取舍一：编译期嵌入 vs. 运行时加载。** 将 init.sh 嵌入二进制是最小依赖的启动方案。替代方案包括从 FAT 分区加载（需提前实现 VFS 和 FAT 驱动）或构建 initramfs cpio 归档（需解压逻辑）。本仓选择嵌入，换取了启动路径的极度简化——无需任何存储后端即可执行用户态初始化。代价是每次修改 init.sh 都必须重新编译内核，不适合调试迭代。

**取舍二：argv 直接传递 vs. 环境变量隔离。** `main` 函数构造了 `["/bin/sh", "-c", script]` 的 argv<sup>[3](#mod-boot-ref-3)</sup>，将整个 init.sh 脚本作为 `-c` 参数传给 shell。这利用了 `/bin/sh -c` 从字符串执行命令的能力，但 argv 总长度受限于内核或硬件规定的单次参数最大长度（通常为 128KB~2MB）。`envs` 被设为空数组 `[]`<sup>[3](#mod-boot-ref-3)</sup>，意味着所有用户态进程启动时环境变量完全缺失。这是一个显式的简化决定：启动流程开发时假定 init.sh 不依赖环境变量，但若后续脚本或子进程依赖 `PATH`、`HOME` 等环境变量，将产生难以调试的故障。

**取舍三：统一入口 vs. 平台特化。** 代码中唯一平台相关的部分是 `#[cfg(feature = "vf2")] extern crate axplat_riscv64_visionfive2;`<sup>[5](#mod-boot-ref-5)</sup>。对于 VisionFive 2 开发板，该声明确保平台特定的 PLL 时钟、GPIO、PLIC 等外设驱动被链接。其他架构（x86_64、aarch64、riscv64 通用）则完全依赖 axhal 的默认实现。这意味着若在某未列出的架构上启动，代码依然能编译，但硬件初始化可能遗漏关键步骤——没有 `cfg` 错误提示。

**取舍四：使用 `#[unsafe(no_mangle)]` 语法。** `main` 函数使用 Rust 2024 引入的 `#[unsafe(no_mangle)]`<sup>[3](#mod-boot-ref-3)</sup> 替代传统的 `#[no_mangle]`。这一语法要求显式声明 `no_mangle` 是一个 unsafe 操作（因为导出符号可能破坏 Rust 的名称修饰保证）。本仓率先采用了这一新惯例，但同一代码库的其他函数（如 `build_init_sh_c_argument`）并未使用该属性——说明团队很可能只对入口函数做了最小化转换，尚未全面审计符号导出安全性。

### 3. 跨模块协同

启动流程跨越三个主要模块的边界：

1. **`src/main.rs`（启动入口层）**：负责构造 argv/envs 并调用内核初始化。
2. **`starry_kernel::entry::init`（内核初始化层）**：根据 boot facts 的 call_chain，该函数接收 argv/envs 后，依次完成内存管理器初始化（创建内核页表）、调度器初始化（创建 idle 线程）、中断子系统初始化（安装 Trap 向量），最后创建 init 进程并传入 argv。该函数不在本文件范围内，但其签名 `fn init(args: &[String], envs: &[String])` 与 `main` 中的调用完全匹配。
3. **axhal（硬件抽象层）**：通过 `boot_style: axhal` 关联。axhal 提供 `hal_init()`、`hal_setup_trap()` 等接口，供内核初始化层在 `entry::init` 内部调用。

一个重要的协同点是 **Trap 处理与用户态切换**。boot facts 列出了 `handle_page_fault` 作为页面错误处理函数（位于 `kernel/src/mm/access.rs:258`），同时 `patches/axcpu/src/` 下存在针对 aarch64、loongarch64、riscv、x86_64 的 `trap.S` 和 `user_copy.S`。这表明启动流程完成后，内核通过 axhal 安装的 Trap 向量表直接跳转到这些汇编入口，再由 rust 层的 trap handler 分发。`user_copy.S` 的存在尤其关键——它实现了用户态与内核态之间的数据拷贝，依赖启动阶段建立的页表映射。如果启动阶段页表权限配置不当（如未设置 U 位），`user_copy.S` 中的访存指令将触发 page fault，由 `handle_page_fault` 处理。

### 4. 边角细节与不足

- **<sup>[6](#mod-boot-ref-6)</sup> `build_init_sh_c_argument` 中的 `String::from` 冗余**：`include_str!` 已经返回 `&'static str`，直接传递给 `args.push()` 本可工作，但代码先通过 `String::from` 转换为堆分配的 `String`，再压入 `Vec<String>`。这引入了一次不必要的堆分配和拷贝。若脚本体积较大（例如数百 KB），启动时会出现短暂的堆压力峰值。可直接使用 `args.push(script.into())` 或保持 `&str` 类型。

- **<sup>[3](#mod-boot-ref-3)</sup> 空环境变量数组 `envs = []`**：类型被推断为 `[&str; 0]`，但 `starry_kernel::entry::init` 期望的是 `&[String]`。这里存在一个隐式类型转换——Rust 可以自动将 `[&str; 0]` 转换为 `&[String]` 吗？实际上不能，因为 `&str` 和 `String` 是不同大小。这行代码能够编译通过，说明 `entry::init` 的第二个参数可能是 `&[impl AsRef<str>]` 或使用了某种泛型约束。无论如何，空 envs 意味着所有子进程无法继承任何环境变量，这对于需要 `PATH` 的脚本是一个隐患。

- **<sup>[5](#mod-boot-ref-5)</sup> `#[cfg(feature = "vf2")]` 只处理了单板**：VisionFive 2 是 RISC-V 平台，但通用 RISC-V 平台（如 QEMU virt）并没有对应的 `extern crate` 声明。如果用户使用 `--features vf2` 编译，该行会引入平台 crate；但若不启用任何 feature，所有平台相关的外设驱动都不会被链接，完全依赖 axhal 的默认设备驱动。这种设计假设了 axhal 的默认实现已经足够覆盖通用平台，但若某个平台需要特有的时钟/复位配置，缺少该声明将导致系统启动后外设无响应。

- **<sup>[1](#mod-boot-ref-1)</sup> 缺乏明确的 `panic_handler` 和 `global_allocator` 定义**：`#![no_std]` 要求 crate 提供 `#[panic_handler]` 函数和全局分配器。`src/main.rs` 中并未定义它们，说明这些符号由 `starry_kernel` crate 内部提供，或通过链接器脚本解析。如果 `starry_kernel` 的依赖树发生变动导致分配器未链接，启动时第一个 `Vec::new()` 或 `String::from()` 就会触发分配器缺失的链接错误——这种错误是链接期而非编译期捕获，反馈周期长。

- **死代码风险**：`build_init_sh_c_argument` 函数被定义为独立函数，但仅在 `main` 中调用一次。理论上可以直接内联。分离为函数增加了测试性，但本仓库中不存在对该函数的单元测试。若未来重构引入新的入口点，该函数可能变成死代码而无人察觉。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块基于 `axmm` crate 的 `PageTable` 和 `memory_set` crate 的 `MemorySet` 构建用户地址空间 `AddrSpace`，通过 `Backend` 枚举解耦物理内存管理策略（线性映射、写时复制、文件映射、共享页等）。与原始 ArceOS 相比，核心增强包括：(1) 完善的 COW 后端及全局物理页引用计数 `FRAME_TABLE`，支持 fork 写时复制；(2) 用户态指针 `UserPtr`/`UserConstPtr` 配合 `access_user_memory` 标志位和统一的 page fault handler，实现内核安全访问用户内存；(3) 丰富的 mmap 系统调用支持（PRIVATE/SHARED、ANONYMOUS、FIXED、HUGETLB 等），并集成基于 pid 和时间的 ASLR 随机化。

### 1. 核心抽象：地址空间与后端解耦

用户地址空间由 `AddrSpace` 结构体表示，它组合了 `VirtAddrRange`（有效虚拟地址范围）、`MemorySet<Backend>`（管理 `MemoryArea` 的区间树）和一个 `PageTable` 实例<sup>[9](#mod-mm-ref-9)</sup>。与 rCore-Tutorial 的 `MemorySet` 直接持有 `MapArea` 不同，本设计将物理内存管理抽象为 `Backend` 枚举，支持 `new_alloc`（匿名映射）、`new_linear`（线性映射，用于设备或内核）、`new_cow`（写时复制，基于文件或共享页）、`new_file`（带缓存的共享文件映射）和 `new_shared`（共享匿名页）等多种策略。这种设计使得 `AddrSpace` 的映射、缺页处理、克隆等操作与后端策略解耦：例如 `try_clone` 遍历所有 area，通过 `Backend::clone_map` 实现写时复制或共享映射，而无需关心后端细节。

COW 后端（`cow.rs`）维护一个全局的 `FRAME_TABLE`——一个从物理地址到引用计数的 `BTreeMap`。当创建一个 COW 映射时，物理帧的引用计数初始化为 1；当 fork 克隆 area 时，`clone_map` 会递增该物理帧的引用计数；当 `drop_frame` 递减至 0 时，先通过 `remove_frame` 从全局表中移除条目，再释放物理帧，避免并发分配导致的安全漏洞<sup>[5](#mod-mm-ref-5)</sup>。

### 2. 关键设计取舍：缺页处理与用户态内存访问安全

传统 OS 使用 `copy_from_user`/`copy_to_user` 函数，并在每次访问前检查地址范围。本模块选择了另一种路径：通过 `access_user_memory` 设置当前线程的 `accessing_user_memory` 标志<sup>[2](#mod-mm-ref-2)</sup>，然后直接解引用用户空间指针；若发生缺页，已注册的 `PAGE_FAULT` trap handler 会检查该标志，若是内核态访问用户内存，则调用 `AddrSpace::handle_page_fault` 尝试按需分配物理页<sup>[1](#mod-mm-ref-1)</sup>。这种方案的优点是无须重复边界检查（已在 `UserPtr::get_as_mut` 等入口做一次 `check_region`），并且能自然处理用户内存被换出（当前未实现交换）或按需分配（populate）的场景。代价是每次 `check_region` 都需要获取 `aspace` 锁并查询 `can_access_range`，而 `check_null_terminated` 循环中**每 4K 页面都要重新获取锁**，可能成为性能瓶颈。

`UserPtr` 和 `UserConstPtr` 类型封装了裸指针，通过 `check_region` 进行对齐、范围权限验证和页面预填充，然后以 `unsafe` 返回 `'static` 生命周期引用<sup>[6](#mod-mm-ref-6)</sup>。这种设计假设用户内存的生命周期与线程绑定，但并没有编译时保证，若用户通过另一个线程修改映射后仍持有旧指针，会导致 UAF。当前代码在信号处理等场景依赖此假设。

另一个设计取舍是一批系统调用的空实现：`sys_madvise`、`sys_msync`、`sys_mlock2` 均直接返回 0<sup>[8](#mod-mm-ref-8)</sup>。这显然是早期原型阶段的取舍——保证了 Linux 测试套件中不实现这些功能不会导致崩溃，但牺牲了性能优化和内存锁定的语义。这一选择对运行 glibc 是可行的（glibc 的 `mlock` 失败时静默忽略），但对需要 `MADV_DONTNEED` 的 JVM 等应用不友好。

### 3. 跨模块协同：ELF 加载、信号跳板与 mmap

本模块是用户态初始化和运行时内存管理的核心，与多个子系统紧密协作。

**ELF 加载**（`loader.rs`）：`load_user_app` 首先通过 `ELF_LOADER` 的 LRU 缓存读取并解析 ELF 文件，然后对每个 `PT_LOAD` segment 调用 `addrspace.map` 创建 COW 或文件映射；随后映射信号跳板 `map_trampoline`，将内核提供的 `signal_trampoline_address` 物理页通过 `map_linear` 固定映射到 `SIGNAL_TRAMPOLINE` 虚拟地址<sup>[7](#mod-mm-ref-7)</sup>。加载动态链接器时使用 `ldso_random_slide` 基于单调时间实现 ASLR 偏移，确保 /musl/lib 路径优先于 PT_INTERP 指定的路径（兼容 musl 与 glibc 混合环境）。

**mmap 系统调用**（`mmap.rs`）通过 `AddrSpace::find_free_area` 和 `gap_end_after` 寻找合适区间，然后根据 MAP_SHARED/MAP_PRIVATE、ANONYMOUS 等标志选择相应 `Backend`。ASLR 随机化使用 `monotonic_time_nanos()` 异或 pid 后取模，在空闲间隙内随机选择 slot<sup>[4](#mod-mm-ref-4)</sup>。对于设备映射，通过 `DeviceMmap` 枚举支持 ReadOnly、Physical、Cache 三种特化后端。此外，`sys_mremap` 的实现非常粗糙：先找到旧区域 flags，重新 mmap 新区域，拷贝数据，然后 munmap 旧区域，这会丢失原地址若被 FIXED 指定的情形，且没有处理 MREMAP_MAYMOVE 等标志。

**缺页协同**：`access.rs` 中注册的 PAGE_FAULT handler 检查当前线程的 `accessing_user_memory` 标志，若为真则调用 `aspace.handle_page_fault`<sup>[1](#mod-mm-ref-1)</sup>。`handle_page_fault` 通过 `MemorySet::find` 找到所在 area，检查权限，然后调用 `Backend::populate` 分配物理页并更新页表。若 populate 成功且后端提供了回调，还会回调 `AddrSpace` 做额外处理（如 COW 时需要复制页面）。

### 4. 边角细节与不足

以下是从代码中识别出的具体风险点和早期实现迹象，按严重程度排序：

1. **`sys_mremap` 初期实现**：`mmap.rs:331-367` 的 `sys_mremap` 有 `// TODO: full implementation`。它通过查找旧 area 的 flags，调用 `sys_mmap` 分配新映射，然后手动拷贝数据，最后 `munmap` 旧映射。这完全忽略了 `MREMAP_MAYMOVE`、`MREMAP_FIXED` 等标志，且在新映射地址可能与旧映射冲突时出错。更严重的是，它在 `aspace` 锁释放后重新获取，期间另一个线程可能修改映射，导致 TOCTOU 漏洞。
2. **`check_null_terminated` 的锁开销**：`access.rs:91-105` 的循环中每 4K 页面都调用 `aspace.lock()` 和 `can_access_range`，在复杂循环（如扫描大量命令行参数）中造成巨大的锁竞争和上下文切换。且若页面已经映射，`can_access_range` 可能返回 true 但仍需 populate？但 populate 已在 check_region 中完成，这里多余。
3. **全局 COW 引用计数粒度**：`cow.rs` 的 `FRAME_TABLE` 是全局 `BTreeMap`，每次分配/释放物理帧都需要查询或修改该映射，多核扩展性差。且 `FrameTableRefCount::init_frame` 使用 `assert!` 检查重复，生产环境应优雅处理。
4. **I-cache 刷新缺失**：`loader.rs:100` 注释 `// TDOO: flush the I-cache`（拼写错误），在加载可执行代码后未执行架构相关的 I-cache 刷新。这对自修改代码或动态代码生成（如 JIT）是致命 Bug，但对静态链接程序可能隐藏。
5. **硬编码 LRU 缓存大小**：`loader.rs:369` 的 `LRUCache<ElfCacheEntry, 32>` 容量固定为 32，不支持配置。若系统运行大量短生命期进程，缓存可能频繁驱逐和重新加载，影响性能。
6. **`sys_mmap` 中 `MAP_FIXED_NOREPLACE` 未完全实现**：`mmap.rs:245` 在 `FIXED` 情况下调用 `aspace.unmap` 移除旧映射，但 `FIXED_NOREPLACE` 要求若地址已映射则返回错误，当前代码直接走 unmap 路径（因为 `intersects` 分支包含 `FIXED_NOREPLACE` 时也会执行 unmap），实际上没有检查现有映射。

这些不足体现了该项目仍处于早期演进阶段，优先满足 glibc 和 musl 动态链接器的运行，而性能优化和完全符合 Linux 语义的路径尚待完善。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块围绕用户态进程与线程管理，核心抽象是 `Thread`（单线程）和 `ProcessData`（进程共享数据），通过 `axtask::TaskExt` 扩展底层调度器。实现了 `clone`/`fork`/`vfork`/`exit` 系统调用，支持线程组、进程组、会话以及命名空间（网络、挂载等部分）。与原生 ArceOS 基线相比，增加了多进程隔离（`Scope` 资源作用域）、`/proc` 文件系统集成、POSIX 信号与定时器协同等特性。关键设计包括 `CloneArgs` 的严格校验、`TimeManager` 的并发控制、以及通过 `ActiveScope` 机制在任务切换时自动切换系统资源视图。

### 1. 核心抽象与调度集成

用户任务管理以 `Thread` 结构体为基本单元（`task/mod.rs:78-122`），每个 `Thread` 持有指向 `ProcessData` 的 `Arc` 指针，实现同一进程内线程共享地址空间、文件描述符表等资源。`ProcessData`（`task/mod.rs:228-280`）封装了进程级属性：可执行路径、地址空间（`AddrSpace`）、资源作用域（`Scope`）、信号处理器、资源限制、futex 表、AIO 注册表等。

调度层复用 `axtask` crate 的 `TaskInner` 和 `TaskExt` trait。`Box<Thread>` 实现了 `TaskExt`，在 `on_enter`/`on_leave` 回调中设置和恢复 `ActiveScope`（`task/mod.rs:161-170`）。这种设计允许每个进程拥有独立的系统资源视图（如文件描述符表、网络命名空间），而无需修改底层调度器——调度器只负责 CPU 时间分配，资源隔离由该模块通过作用域切换完成，体现了关注点分离。

### 2. 进程创建与命名空间隔离

`CloneArgs::validate`（`clone.rs:78-114`）对 Linux 兼容的 clone 标志进行严格检查：拒绝 `THREAD` 不带 `VM|SIGHAND`、`VFORK` 与 `THREAD` 共存等非法组合。对于不支持的命名空间标志（`NEWPID`、`NEWUSER` 等）仅输出警告并继续，体现了“尽力兼容”的取舍——既通过数量庞大的 LTP 测试套件，又避免未实现功能阻塞上层程序。

`do_clone`（`clone.rs:143-273`）根据 `THREAD` 标志决定是创建新进程还是新线程。创建新进程时，通过 `aspace.try_clone()` 复制页表（若 `CLONE_VM` 则共享），并利用 `copy_from_kernel` 按需复制内核映射区域。信号动作的继承逻辑处理了 `CLEAR_SIGHAND` 和新命名空间场景（`clone.rs:215-228`），确保守护进程不会继承父进程的信号处理函数（符合 LTP `tst_ns_create` 需求）。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制围绕 `starry-signal` crate 的基础类型（SignalInfo, SignalSet, Signo）构建，将状态分为两个层次：进程级别持有信号动作表（`proc_data.signal.actions`），线程级别持有挂起集、阻塞掩码与替代栈。信号传递采用"入队 + task.interrupt()" 的主动唤醒模式，阻塞等待（sigtimedwait/sigsuspend/pause）依托 `poll_fn + block_on` 这一内核态异步框架实现。与原版 ArceOS 相比，本仓引入了独立的信号系统调用家族（共 17 个入口），并利用异步协作式调度的方式完成信号等待，而非直接阻塞内核线程。

### 1. 核心抽象与分层设计

信号机制的基石是 `starry-signal` crate，它定义了 `SignalInfo`（信号信息荷载）、`SignalSet`（64 位掩码）、`Signo`（枚举信号编号）和 `SignalOSAction`（操作系统默认动作）等核心类型。内核在此基础上做了两层切分：**进程级信号动作**（`proc_data.signal.actions`）是一个由 `Mutex` 保护的数组，按 `Signo` 索引，保存每个信号的处置方式（忽略、捕获或默认）；**线程级信号状态**（`thr.signal`）则持有当前的阻塞掩码、挂起信号队列以及替代信号栈（sigaltstack）。

这种分离符合 POSIX 语义：`sigaction()` 设置的是进程范围内的动作，而 `pthread_sigmask()`/`sigpending()` 查询的是调用线程的状态。`sys_rt_sigaction` 通过 `curr.as_thread().proc_data.signal.actions.lock()` 获取锁后修改动作表，并专门排除了 `SIGKILL`/`SIGSTOP` 的修改 <sup>[8](#mod-signal-ref-8)</sup>；`sys_rt_sigprocmask` 则直接操作 `curr.as_thread().signal` 上的 `blocked()` 与 `set_blocked()` 方法 <sup>[12](#mod-signal-ref-12)</sup>。

值得一提的是，`Signo` 的解析使用了 `Signo::from_repr(signo as u8)` <sup>[7](#mod-signal-ref-7)</sup>，将用户传入的 32 位信号编号截断为 8 位后查找枚举值。这一截断符合 Linux 内核行为（仅使用低 8 位），但若 `starry-signal` 将来扩展超过 256 个信号，该转换将产生语义错误。

### 2. 信号传递与检查的 "Async + Interrupt" 路径

信号传递的核心路径在两个文件中配合完成。系统调用层（`kernel/src/syscall/signal.rs`）解析参数并调用 `send_signal_to_thread`/`send_signal_to_process`/`send_signal_to_process_group`（位于 `kernel/src/task/signal.rs`），这些函数最终汇聚到 `send_signal_thread_inner` <sup>[5](#mod-signal-ref-5)</sup>：

```rust
pub(super) fn send_signal_thread_inner(task: &TaskInner, thr: &Thread, sig: SignalInfo) {
    if thr.signal.send_signal(sig) {
        task.interrupt();
    }
}
```

该函数展示了信号机制与任务调度器的契约：`thr.signal.send_signal()` 将信号入队（内部管理挂起集与排队顺序），返回值指示是否有信号被实际加入（可能因掩码过滤或队列满而丢弃）；若成功，通过 `task.interrupt()` 唤醒目标线程。`interrupt()` 会设置一个线程级标志，使得正在 `poll_fn` 中等待（如 `poll_interrupt`）的线程能够被重新调度。

信号检查由 `check_signals()` <sup>[3](#mod-signal-ref-3)</sup> 完成，它调用 `thr.signal.check_signals(uctx, restore_blocked)`，后者从 `starry-signal` crate 中返回 `(SignalInfo, SignalOSAction)` 对。对于 `SignalOSAction::Handler`，`starry-signal` 会修改 `UserContext`——保存旧上下文到用户栈，并将程序计数器指向注册的信号处理函数——从而使线程在从内核返回用户态后直接进入处理函数。处理函数返回后，用户态通常调用 `rt_sigreturn`，内核侧对应 `sys_rt_sigreturn` <sup>[1](#mod-signal-ref-1)</sup>，它调用 `block_next_signal()` 防止立即重入，再通过 `signal.restore(uctx)` 恢复之前保存的上下文。

### 3. 阻塞等待的异步实现

`sys_rt_sigtimedwait`、`sys_rt_sigsuspend` 和 `sys_pause` 三个系统调用都涉及"等待信号到达"的阻塞语义。Starry 的实现没有直接挂起内核线程，而是利用 `axtask::future::block_on` 驱动 `poll_fn` 构造的异步任务 <sup>[6](#mod-signal-ref-6)</sup>：

```rust
pub fn sys_rt_sigsuspend(...) -> AxResult<isize> {
    let set = unsafe { set.vm_read_uninit()?.assume_init() };
    let old_blocked = thr.signal.set_blocked(set);
    block_until_signal(uctx, Some(old_blocked))
}
```

`block_until_signal` 内部调用 `block_on(poll_fn(|cx| { ... }))`，在每次 poll 中调用 `check_signals()` 判断是否有信号待处理；若无则注册 `curr.poll_interrupt(cx)` 并返回 `Poll::Pending`。当 `task.interrupt()` 被触发时，该 async 任务重新被调度，再次检查信号。这种实现避免了显式的等待队列，将信号等待与内核的协作式调度无缝结合。

`sigtimedwait` 更复杂：它在等待前通过 `old_blocked & !set` 临时解除对指定信号集的阻塞 <sup>[11](#mod-signal-ref-11)</sup>，使其可以被递交；等待结束后恢复原阻塞掩码。超时处理依赖 `future::timeout` 包装器，超时后返回 `AxError::WouldBlock`。需要注意的是，若发生超时或中断，之前 `dequeue_signal` 已经消费的信号不会回退——这是语义上的丢失（一旦出队就无法恢复）。

### 4. 跨模块协同：signalfd 文件抽象

`signalfd` 是信号机制与文件/VFS 层的桥梁，实现于 `kernel/src/file/signalfd.rs`。它同时实现了 `FileLike` 和 `Pollable` trait，将信号变成可以被 `select`/`poll`/`epoll` 监控的文件描述符 <sup>[9](#mod-signal-ref-9)</sup>。其 `read` 方法内部调用 `block_on(poll_io(...))`，从当前线程的信号队列中通过 `signal.dequeue_signal(&mask)` 取出匹配的信号，转换为定长 128 字节的 `SignalfdSiginfo` 结构返回给用户。

`sigaltstack` 的跨模块协同 体现在：线程的替代栈状态保存在 `thr.signal.stack()` / `thr.signal.set_stack()` 中，而 `starry-signal` crate 在 `check_signals` 返回 `Handler` 动作时，会读取此栈地址来决定是否在替代栈上执行处理函数。`sys_sigaltstack` 仅校验用户传入的 `ss.size > MINSIGSTKSZ`（否则返回 `NoMemory`），但未校验 `SIGSTKSZ`（系统建议的最小信号栈大小）——这是一个潜在的兼容性问题，因为某些 glibc 版本可能依赖 `SIGSTKSZ` 的检查。

### 5. 边角细节与不足

**LTP 兼容性 hack <sup>[3](#mod-signal-ref-3)</sup>**：`check_signals` 中 `Terminate` 分支对 `SIGHUP`/`SIGPIPE` 特殊处理——遇到这两个信号时 return true 但不终止进程。注释说明这是为了 LTP `tst_ns_create` 测试中短生命周期父进程退出时子进程收到 `SIGHUP` 的场景。这种 hack 破坏了 POSIX 语义（`SIGHUP` 默认应终止进程），但属于测试合规的务实选择。

**TODO 堆积 <sup>[4](#mod-signal-ref-4)</sup>**：`CoreDump`、`Stop`、`Continue` 三种默认动作均未实现。`CoreDump` 仅以 128+signo 退出，不产生 coredump；`Stop` 直接调用 `do_exit` 而非暂停进程；`Continue` 为空操作。这意味着 `SIGTSTP`/`SIGCONT`/`SIGQUIT` 等信号的行为与 Linux 标准不符。

**unsafe transmute_copy <sup>[2](#mod-signal-ref-2)</sup>**：`starry-signal` 0.3.0 未为 `SignalSet` 实现 `PartialEq`，内核不得不使用 `core::mem::transmute_copy` 将其视为 `u64` 进行比较。注释假设 `SignalSet` 是 `#[repr(transparent)] u64`，但这属于导出 crate 的内部布局细节——若 `starry-signal` 升级后变更了内部表示，此处的 unsafe 将导致静默数据损坏。

**signalfd 信息丢失 <sup>[9](#mod-signal-ref-9)</sup>**：`SignalfdSiginfo::from_signal_info` 将 `ssi_pid`、`ssi_uid`、`ssi_addr` 等字段全部置为 0，仅填充了 `signo`、`errno`、`code`。依赖发送者信息的程序（如 `SIGCHLD` 的 `status`/`pid` 或 `SIGIO` 的 `fd`）将收到不完整的数据。

**进程表全扫描与锁层级 <sup>[3](#mod-signal-ref-3)</sup>**：`sys_kill(-1, ...)` 通过 `processes()` 遍历所有进程，并在闭包内调用 `send_signal_to_process`。`processes()` 返回的迭代器背后持有哪些锁、是否与 `send_signal_to_process` 内部的 `get_process_data` 存在锁顺序依赖，代码中未显式声明——存在潜在的循环等待死锁风险。

**`check_sigset_size` 允许 size==0 <sup>[12](#mod-signal-ref-12)</sup>**：`check_sigset_size(size)` 将 `size == 0` 视为合法。Linux 内核确实允许 `sigsetsize` 参数为 0（视为等同于 sizeof(sigset_t)），但此处的实现完全跳过了对 `set` 指针的解引用——若用户在 `sigsetsize=0` 时仍然传入一个非空指针，`unsafe { set.vm_read_uninit() }` 不会执行，避免了读越界；但若 `oldset` 非空时，会照常读取 `oldset.vm_write(old)`——这种情况下 `size` 检查已过，而写入的 `old` 总是合法的 `SignalSet`。整体行为与 Linux 一致，但错误路径（返回 `InvalidInput`）的 error 码与 Linux 不同（Linux 返回 `EINVAL`）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现了 POSIX 消息队列 (sys_msgget/ctl/rcv/snd)、System V 共享内存 (sys_shmget/at/dt/ctl)、futex 快速用户空间锁 (sys_futex/waitv/robust_list) 以及匿名管道 (pipe)。核心抽象分别围绕全局管理器 (MSG_MANAGER, SHM_MANAGER)、基于页面的共享内存后端 (SharedPages) 和位掩码唤醒的 WaitQueue 构建。与 ArceOS 基线相比，本仓新增了完整的 System V IPC 支持、futex_waitv 以及 robust list，但阻塞等待普遍采用 busy‑poll + yield 而非条件变量，缺少真正的睡眠‑唤醒机制。

### 1. 核心抽象与外部依赖

进程间通信子系统的四个组件共享两个设计模式：全局静态管理器 + 引用计数的内核对象。

- **消息队列**：`MsgManager` 全局实例 (`MSG_MANAGER` <sup>[1](#mod-ipc-ref-1)</sup>) 维护 `key_msqid` 和 `msqid_queues` 两张 BTreeMap，分别提供键→ID 和 ID→`Arc<Mutex<MessageQueue>>` 的查找。每个 `MessageQueue` 内部以 `BTreeMap<i64, Vec<Message>>` 按类型组织消息，支持按正/负/零类型匹配（POSIX msgrcv 语义）。
- **共享内存**：`ShmManager` 全局实例 (`SHM_MANAGER`) 使用 `BiBTreeMap` 实现键↔ID 的双向查找，`shmid_inner` 映射 ID → `Arc<Mutex<ShmInner>>`，并额外维护 `pid_shmid_vaddr` 跟踪每个进程的附加地址。物理页面由 `SharedPages` 管理，首次 `shmat` 时分配，后续附加直接映射同一组页面，实现零拷贝共享 <sup>[12](#mod-ipc-ref-12)</sup>。
- **futex**：`FutexTable` (一个 `HashMap<usize, Arc<FutexEntry>>`) 按地址（私有）或偏移+共享区域标识（共享）提供 wait queue 查找。`FutexEntry` 包含一个 `WaitQueue` 和 `owner_dead` 标志。共享 futex 通过 `SHARED_FUTEX_TABLES` 全局表组织，以 `Weak<SharedPages>` 或文件标识为键 <sup>[7](#mod-ipc-ref-7)</sup>。
- **管道**：`Pipe` 封装 `Arc<Shared>`，其中 `Shared` 包含 `Mutex<HeapRb<u8>>` 环形缓冲区、三个 `PollSet`（读就绪、写就绪、关闭）。它实现 `FileLike` trait，因此可被 `sys_read`/`sys_write` 通过文件描述符访问。

上述组件均依赖 `axsync::Mutex` 进行内部互斥，`axtask::current` 获取当前任务，`starry_vm` 的 `VmPtr`/`VmMutPtr` 进行安全的用户空间内存访问。

### 2. 关键设计取舍

**取舍 1：阻塞等待以 busy‑poll 替代 condition variable**

消息队列的 `sys_msgsnd`（队列满时）和 `sys_msgrcv`（队列空时）以及 futex 的 `wait_if` 都使用了 `block_on(interruptible(async { loop { yield_now(); check_condition(); } }))` 模式 <sup>[2](#mod-ipc-ref-2)</sup><sup>[6](#mod-ipc-ref-6)</sup>。这意味着等待线程不断被调度但立即让出 CPU，实际是轮询而非睡眠。Linux 内核在这类场景下会将任务放入等待队列并切换出，仅在事件到达时唤醒。

该选择的代价是：
- CPU 浪费：轮询间隔完全取决于调度器，没有事件驱动的高效。
- 延迟：唤醒可能延迟一个调度 tick。
- 公平性：高负载下轮询任务可能抢占实际工作者。

收益是实现简单，无需设计复杂的 waitqueue 与信号集成。项目显然接受了这一 trade‑off，并将真正的 waitqueue 列为 TODO <sup>[3](#mod-ipc-ref-3)</sup>。

**取舍 2：消息队列的硬编码 2 秒超时**

`sys_msgsnd` 和 `sys_msgrcv` 的阻塞循环内嵌了 2 秒超时 (`wall_time().as_millis() - start.as_millis() > 2000`)，超时后返回 `EAGAIN`/`ENOMSG`。Linux 的 System V 消息队列在未指定 `IPC_NOWAIT` 时是无限阻塞的（除非被信号中断）。这里的 2 秒硬限制是一个明显的简化，可能会破坏依赖无限等待的应用程序。

**取舍 3：futex_waitv 采用全轮询而非多队列注册**

`sys_futex_waitv` 的实现读取所有 waiter 后进入一个循环，反复 `vm_read` 每个 uaddr 检查值是否变化，期间仅 `yield_now()` <sup>[8](#mod-ipc-ref-8)</sup>。Linux 原版会将任务同时注册到所有目标 futex 的 waitqueue 上，任一 wake 即返回。全轮询方式在 waiter 数量较多时效率极低，且每次迭代都执行用户空间读操作（可能触发页错误），但避免了多队列注册的复杂性。

**取舍 4：共享内存的页面分配时机**

`sys_shmat` 仅在第一个附加进程时才分配物理页面 (`SharedPages::new`)，后续附加直接映射已有的页面。这种 lazy 分配延迟了物理页的占用，但如果多个进程同时 attach，可能存在竞态：第一个进程尚未完成映射时第二个进程发现 `phys_pages` 为 None 而重复分配。不过由于 `sys_shmat` 全程持有 `ShmInner` 的锁，竞态被排除。

**取舍 5：管道关闭检测依赖 Arc 引用计数**

`Pipe::closed()` 通过 `Arc::strong_count(&self.shared) == 1` 判断对端是否关闭 <sup>[9](#mod-ipc-ref-9)</sup>。当一端 Drop 时，强引用减 1；若仅剩当前端，则对端已关闭。这种方法无需额外的标志位或原子变量，但要求两端必须通过 Arc 共享同一块 `Shared`。如果存在通过 `dup` 复制的文件描述符，`strong_count` 会准确反映所有引用。

### 3. 跨模块协同

**进程退出时的资源清理**

共享内存通过 `ShmManager::clear_proc_shm` 在进程退出时被调用 <sup>[4](#mod-ipc-ref-4)</sup>。该函数遍历进程附加的所有 shmid，调用 `detach_process`，若 `rmid` 标志为 true 且附加计数归零则删除整个共享段。消息队列和 futex 的清理则隐式依赖：`MSG_MANAGER` 和 `SHARED_FUTEX_TABLES` 是全局的，但进程持有的 `msqid` 只是整数 ID，进程退出后不会自动回收消息队列（需要显式 `IPC_RMID`）。futex 的 `FutexTable` 是 `proc_data` 的一部分，进程退出时被 drop，但全局共享 futex 表依靠惰性清理（每 100 次操作检查强引用计数）<sup>[7](#mod-ipc-ref-7)</sup>。若进程崩溃，其 waiter 的 `Waker` 可能滞留在共享 waitqueue 中，不过 `wake` 操作会调用已死的 waker（空操作），但条目本身不会被移除，直到惰性清理发现 `strong_count==1` 且队列为空。这存在轻微的内存泄漏风险。

**信号与中断**

所有阻塞路径都使用了 `interruptible` 包装器，它会检查 `current.as_thread().signal.pending()`。消息队列和 futex 的循环也手动检查信号 <sup>[2](#mod-ipc-ref-2)</sup><sup>[6](#mod-ipc-ref-6)</sup>，确保 `EINTR` 能正确传递。但管道使用 `poll_io`，它集成到 axtask 的异步机制中，信号中断由上层处理。

**futex 与 robust list**

`sys_set_robust_list` 和 `sys_get_robust_list` 将用户空间的 robust list 头地址存入线程数据。当线程异常退出时，内核应遍历该链表并标记锁的 `OWNER_DIED` 位。当前代码仅存储了地址 [ref:??]，但未实现 exit 时的遍历逻辑。这是一个半成品：`FutexEntry` 虽包含 `owner_dead` 标志，但从未被设置。

### 4. 边角细节与不足

1. **<sup>[3](#mod-ipc-ref-3)</sup> 消息队列的唤醒未实现**：`sys_msgsnd` 和 `sys_msgrcv` 末尾均有 `warn!("wakeup of waiting ... not implemented")`。这意味着即使有任务在等待，发送/接收后也不会主动唤醒对方，只能靠轮询超时重试。这严重影响了性能。

2. **<sup>[5](#mod-ipc-ref-5)</sup> `BiBTreeMap` 的定位尴尬**：其注释 `// TODO: I don't know where to put this, so I put it here.` 表明作者不确定它的归属。这导致 `shm.rs` 中重复实现了双向映射逻辑，而其他模块（如消息队列）也使用类似模式但用了两个独立 BTreeMap。应当抽取为公共工具。

3. **<sup>[8](#mod-ipc-ref-8)</sup> futex_waitv 的轮询实现缺少睡眠**：与标准 futex 不同，`futex_waitv` 没有将任务注册到任何 waitqueue，而是纯轮询。这意味着即使没有其他线程操作 futex，调用线程也会持续占用 CPU。此外，轮询间隔无退避策略。

4. **消息队列的 `sys_msgctl` 实现不完整**：截断的代码显示 `IPC_INFO` 和 `MSG_INFO` 已完成，但 `IPC_SET`、`IPC_RMID`、`MSG_STAT` 分支在截断之后，可能也未完全实现权限检查。

5. **<sup>[9](#mod-ipc-ref-9)</sup> 管道的 `closed()` 与 SIGPIPE 的竞态**：`Pipe::write` 中先检查 `closed()` 再尝试写入，但在检查后、写入前对端可能关闭，导致数据写入已关闭的管道。此时虽然后续会触发 `raise_pipe()` 并返回 `BrokenPipe`，但已写入的数据可能丢失。这是基于 Arc 引用计数检测的固有问题。

6. **futex 的 `FutexKey::Shared` 使用弱引用**：当 `SharedPages` 被回收（所有进程都 detach），弱引用会失效，但哈希表中可能仍有以该弱引用地址为键的条目。惰性清理依赖 `strong_count`，`Weak::as_ptr` 返回的指针在 `SharedPages` 释放后变为悬垂指针，但作为 `usize` 比较仍可能发生碰撞。不过由于 `Weak` 升级为 `Arc` 会失败，代码可能不会访问无效内存，但 key 的唯一性保证被削弱。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块基于 ArceOS 的 axnet 栈与 smoltcp 协议栈，提供 TCP/UDP socket 及 AF_NETLINK/NETLINK_ROUTE 子系统的双重实现。核心抽象是 `NetNamespace`（命名空间化管理网络接口、地址、sysctl）和 `NetlinkSocket`（通过 FileLike trait 暴露给用户）。与上游 ArceOS 相比，本仓库引入了独立的 netlink 解析层（`rtnetlink.rs` vs `syscall/net/netlink.rs`），并深度绑定 LTP 测试场景（启动时预创建 veth 对、硬编码接口名 fallback）。网络命名空间通过 `scope_local!` 宏实现 per-process 隔离，但接口迁移逻辑存在未完成痕迹。

### 1. 核心抽象与外部依赖

网络模块的核心抽象分为两层。底层是 **axnet-ng**（`patches/axnet-ng/`），它是对 ArceOS 原版 `axnet` 的修补版，依赖于 `smoltcp` 协议栈。Tcp socket 在 `tcp.rs` 中通过 `smol::Socket` 包装，使用固定大小的环形缓冲区 `[0; TCP_RX_BUF_LEN]` 和 `[0; TCP_TX_BUF_LEN]` <sup>[1](#mod-net-ref-1)</sup>。上层是 **netlink 子系统**和 **网络命名空间**，由 `kernel/src/net/` 下的模块实现。

`NetNamespace`（`namespace.rs`）是网络配置的状态中心，管理接口（`Interface`）、IP 地址、模块加载记录、sysctl 值（如 `disable_ipv6`）。每个进程通过 `scope_local!` 宏关联一个 `Arc<RwLock<NetNamespace>>`，实现 per-process 网络命名空间隔离 <sup>[2](#mod-net-ref-2)</sup>。这与 Linux 的 `CLONE_NEWNET` 语义对应，但当前仅支持通过 PID 映射（`PID_NETNS`）来模拟多命名空间。

Netlink 实现有两个版本：较早的简化版 `kernel/src/syscall/net/netlink.rs` 仅处理 `RTM_NEWLINK` veth 创建并返回基本 ACK；较完整的 `kernel/src/net/rtnetlink.rs` 支持 `RTM_NEWLINK`/`DELLINK`/`SETLINK`/`NEWADDR`/`DELADDR`/`GETLINK`/`GETADDR`/`NEWROUTE`/`GETROUTE`，并直接操作 `NetNamespace` <sup>[3](#mod-net-ref-3)</sup>。两个版本共存反映工程上的过渡状态。

### 2. 关键设计取舍

**1. 双 netlink 实现的分工与冲突**
`syscall/net/netlink.rs` 更早实现，只处理 veth 创建，对其他消息返回 `EOPNOTSUPP`。而 `rtnetlink.rs` 实现了完整的 RTNL 协议，但它并不接管所有 netlink 消息的分发；syscall 入口可能仍然调用旧版。这导致功能碎片：例如 `vti` 接口创建在新版中支持，但若 syscall 路由到旧版则返回错误。Linux 工具依赖的 `RTM_GETADDR`、`RTM_NEWROUTE` 等必须走新版路径才能正常工作 <sup>[4](#mod-net-ref-4)</sup>。

**2. 测试驱动的硬编码适配**
大量 `ifname == "ltp_ns_veth1"` 这类硬编码回退出现在 `rtnetlink.rs` 的 `parse_rtm_newlink` 和 `namespace.rs` 的 `bootstrap_ltp_veth` 中 <sup>[5](#mod-net-ref-5)</sup>。内核启动时预创建固定的 veth 对（`ltp_ns_veth1`–`ltp_ns_veth2`），并在 netlink 解析失败时 fallback 到这些名字。这是为了绕过 `busybox ip` 在 loongarch64 上的 SIGILL 问题，而非通用设计。

**3. 接口迁移的语义不完整**
Linux 移动 veth 到另一个 netns 时会将两端一起移动。`NetNamespace` 提供了 `take_iface_with_veth_peer()` 方法 <sup>[6](#mod-net-ref-6)</sup>，但 `move_iface_to_netns`（`rtnetlink.rs` 第 244 行）仅调用 `take_iface`，导致 peer 残留在原命名空间。LTP 测试可能依赖这种非标准行为才通过，但违背了 POSIX 语义。

**4. unsafe 块缺乏安全契约**
`rtnetlink.rs` 在构造 netlink 响应报文时频繁使用 `&mut *(out.as_mut_ptr() as *mut IfAddrMsg)` 这类指针转换，但全文未出现一条 `// SAFETY:` 注释 <sup>[7](#mod-net-ref-7)</sup>。这些转换假设对齐且生命周期有效，但若缓冲区大小不匹配会产生 UB。同一文件中无安全抽象层封装此模式。

**5. 缓冲区大小与错误处理的简化**
`tcp.rs` 使用固定 65536 字节的收发缓冲区，且硬编码在 `consts` 模块中，不支持运行时 `setsockopt` 调整。此外，多个 netlink handler 在失败路径上仅 `return Err(AxError::InvalidInput)` 或 `Err(-22)`，丢失了具体错误原因（如 `ENODEV` 与 `EINVAL` 的区分）<sup>[8](#mod-net-ref-8)</sup>。

### 3. 跨模块协同

网络模块与以下子系统紧密配合：

- **syscall 分发**：socket/bind/connect 等直接对应 `axnet` 的 socket 操作；`socketpair` 由 netlink 子系统模拟。`accept4`、`recvmmsg` 等部分系统调用尚未在上层实现完整 <sup>[9](#mod-net-ref-9)</sup>。
- **文件系统**：`NetlinkSocket` 实现了 `FileLike` trait，因此用户可通过 `read`/`write` 系统调用与 netlink 交互。`Kstat` 结构体被用于伪造统计信息。
- **进程管理**：`register_pid_netns` 将特定 PID 与 netns 绑定，供 `/proc/<pid>/ns/net` 和 `IFLA_NET_NS_PID` 使用。`move_iface_to_pid_netns` 通过 `crate::file::net_ns_for_pid` 查询目标命名空间。
- **sysfs 同步**：`sync_iface_sysfs` 在接口变更时更新 `/sys/class/net/` 下的伪文件，LTP 测试通过读取这些文件获取接口参数 <sup>[10](#mod-net-ref-10)</sup>。

### 4. 边角细节与不足

**<sup>[1](#mod-net-ref-1)</sup>** `tcp.rs` 中缓冲区大小硬编码，未提供动态调整机制，大型传输可能因为缓冲区过小导致性能瓶颈。
**<sup>[2](#mod-net-ref-2)</sup>** `scope_local!` 宏的使用意味着 netns 切换范围是线程粒度的，而非 Linux 的进程/线程组粒度。当线程池复用时可能出现命名空间泄漏。
**<sup>[3](#mod-net-ref-3)</sup>** 两个 netlink 实现的文件位置（`syscall/net/` vs `net/`）暗示设计者试图将纯内核逻辑与 syscall 胶水分离，但实际存在重复的常量定义和解析逻辑，增加了维护成本。
**<sup>[4](#mod-net-ref-4)</sup>** `rtnetlink.rs` 中的 `after_net_change()` 函数为空实现 <sup>[11](#mod-net-ref-11)</sup>，所有接口变更后应触发的回调（如路由表刷新、ARP 通知）均被忽略，可能导致内核状态与用户预期不一致。
**<sup>[5](#mod-net-ref-5)</sup>** `LTP_VETH_ADOPTED` 全局标志在首次迁移后永久置位，若之后有新的 veth 需要迁移则不会生效，这假设 LTP 测试序列是线性的。
**<sup>[6](#mod-net-ref-6)</sup>** `IFLA_NET_NS_FD` 的处理尝试直接解析 fd 为 `NsFile`，但若 fd 不是 net ns 类型（如返回 `ns_kind() != "net"`）则静默跳过，不返回错误给用户 <sup>[12](#mod-net-ref-12)</sup>。

总体而言，网络模块在兼容 LTP 测试集上投入了大量针对性代码，但通用性、错误报告和安全性方面仍有明显改进空间。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层以`handle_syscall`为核心入口，通过`Sysno`枚举匹配系统调用号，将参数从`UserContext`提取并转发到各功能子模块（文件、内存、任务、信号等）。相比原版ArceOS，该实现扩展了大量Linux特有系统调用（如eBPF、fanotify、pidfd、内核模块加载），且BPF子系统完全以内核态解释器实现，而非依赖硬件JIT。模块加载采用了“名称路由”策略，仅解析ELF提取模块名并调用预注册的Rust函数，而非真正加载内核模块。

### 1. 核心抽象与调度架构
系统调用调度中心是`kernel/src/syscall/mod.rs`中的`handle_syscall`函数<sup>[1](#mod-syscall-ref-1)</sup>。该函数接收一个`UserContext`（来自`axhal`），从中提取系统调用号（通过`uctx.sysno()`）和参数（`arg0`~`arg4`），然后通过`Sysno`枚举（来自`syscalls` crate）匹配分发。匹配分支极大，覆盖文件操作、内存管理、任务控制、信号处理、网络、时间等。对于无法识别的系统调用号，返回`ENOSYS`<sup>[2](#mod-syscall-ref-2)</sup>。有几个特殊情况：`pause(2)`的系统调用号不在`syscalls` crate中所有架构都存在，因此通过一个内联函数`pause_syscall_no`根据编译条件返回不同值<sup>[3](#mod-syscall-ref-3)</sup>；另外，MIPS特有的`cacheflush`系统调用被直接忽略<sup>[4](#mod-syscall-ref-4)</sup>；`memfd_secret`在`Sysno::new`可能不识别时通过原始系统号匹配<sup>[5](#mod-syscall-ref-5)</sup>。这种架构确保了平台兼容性，但`#[cfg]`的使用也带来风险：某些系统调用仅在x86_64下且未在其他架构实现，通过条件编译排除，可能导致在RISC-V或LoongArch上调用这些系统调用时意外返回ENOSYS而非正确的stub。

### 2. 独特功能实现：用户态eBPF与模块名称路由
本仓库最引人注目的特性是完整的BPF系统调用实现（`sys_bpf`）<sup>[6](#mod-syscall-ref-6)</sup>，支持`MAP_CREATE`、`MAP_LOOKUP`、`PROG_LOAD`等关键命令，并在内核中内置了一个eBPF验证器和解释器。与标准Linux不同，此实现不依赖硬件JIT，而是用纯Rust解释执行`SOCKET_FILTER`程序。验证器跟踪寄存器类型（`RegType`），特别防范CVE-2021-4204和CVE-2022-23222模式：对`BPF_FUNC_ringbuf_reserve`直接拒绝加载<sup>[7](#mod-syscall-ref-7)</sup>，并实现32位操作数污染标记（`tainted_32bit`）和环形缓冲区指针历史追踪（`was_ringbuf`）<sup>[8](#mod-syscall-ref-8)</sup>。这些措施在仅有约600行代码的情况下达到了令人印象深刻的深度，但仍有简化：验证器是单次线性扫描，不支持循环回边检查，因此任何含后向跳转的程序会被拒绝（`BPF_JMP`的branch target超出范围检查<sup>[9](#mod-syscall-ref-9)</sup>）。此外，`ringbuf_reserve`的占位实现返回固定地址0x1000<sup>[10](#mod-syscall-ref-10)</sup>，导致实际使用ringbuf的程序无法运行。
模块加载子系统（`module.rs`）采用“名称路由”模式：`init_module`/`finit_module`不加载真正的内核模块，而是从用户提供的ELF镜像中提取模块名称<sup>[11](#mod-syscall-ref-11)</sup>，再到预先注册的`MODULE_REGISTRY`中查找对应的Rust handler并调用其`init`函数<sup>[12](#mod-syscall-ref-12)</sup>。这一设计源于开发团队希望在不实现完整ELF模块加载的情况下满足LTP测试套件对`init_module`的需求。其代价是只能运行预先硬编码的模块（如`vcan`、`ltp_block_dev`），任何第三方模块都无法加载。代码中的`finit_module`故意返回ENOSYS以触发busybox `insmod`的回退路径<sup>[13](#mod-syscall-ref-13)</sup>，体现了对用户态工具行为的精细考量。

### 3. 跨模块协同与架构适配
系统调用层与进程/线程模块（`axtask`）紧密协作：每次系统调用时，`handle_syscall`会记录当前线程的最后系统调用号（`thr.set_last_syscall_no(raw_sysno)`）<sup>[14](#mod-syscall-ref-14)</sup>，这用于信号处理和`/proc/self/`信息。内存管理子系统（`starry_vm`）提供`vm_read_slice`、`vm_write_slice`和`vm_load_string`等函数，供系统调用实现安全地在用户空间与内核缓冲区间拷贝数据<sup>[15](#mod-syscall-ref-15)</sup>。在信号处理方面，`sys_rt_sigreturn`通过修改`UserContext`返回到信号跳板<sup>[16](#mod-syscall-ref-16)</sup>。BPF解释器在虚拟内存空间之外运行，完全在核心态数据上进行操作，避免了复杂的用户态指针处理。
架构适配方面，`mod.rs`对每个架构都配置了对应的系统调用表：例如`clone`、`openat`等通用系统调用在所有架构可用，而`fork`、`pipe`、`stat`等在x86_64专属<sup>[17](#mod-syscall-ref-17)</sup>。LoongArch和RISC-V等新架构使用更较新的`clone3`和`openat`，放弃了旧的`fork`/`open`入口点。这种取舍减少了架构相关代码量，但可能破坏期望旧系统调用存在的遗留二进制兼容性。

### 4. 边角细节与不足
尽管系统调用层覆盖面广，仍有多处值得注意的不足：
- **`unsafe` 块缺乏安全注释**：在`bpf.rs`的`BpfInsn`结构体访问函数中没有`// SAFETY:`注释<sup>[18](#mod-syscall-ref-18)</sup>；`module.rs`的`read_shdr`中使用`unsafe`指针转换<sup>[19](#mod-syscall-ref-19)</sup>，但没有解释为什么内存布局和边界检查已经保证安全。
- **锁粒度风险**：`MODULE_REGISTRY`和`LOADED_MODULES`是全局`Mutex`，在初始化时持有锁的时间较短，但在`do_init_module`中，“持有锁→释放锁→重新获取锁”的模式可能导致死锁？实际上代码中`drop(registry)`后重新获取`loaded`锁，没有嵌套，但是`handler.init()`可能间接尝试注册新模块导致锁重入<sup>[20](#mod-syscall-ref-20)</sup>，可能死锁（尽管现有handler是简单函数不会重入）。
- **硬编码常量**：`bpf.rs`中的`PROG_TYPE_SOCKET_FILTER`硬编码为1，未考虑其他程序类型<sup>[21](#mod-syscall-ref-21)</sup>；`module.rs`中最大镜像大小限制16MB是硬编码常量<sup>[22](#mod-syscall-ref-22)</sup>。
- **错误处理吞没风险**：在`bpf.rs`的`BpfMap::lookup`中，对于`RingBuf`类型返回`Unsupported`，但上层调用者可能忽略错误<sup>[23](#mod-syscall-ref-23)</sup>；在`module.rs`的`sys_init_module`中，`vm_read_slice`可能返回`EFAULT`，但没有传递到用户态（被`?`传播）<sup>[24](#mod-syscall-ref-24)</sup>。
- **测试导向的逻辑**：`module.rs`中的`ltp_block_dev`的测试函数（`tc01`-`tc07`）是纯粹的测试代码，但随模块加载系统一起编译进内核，增加了约300行的死代码（除非LTP测试真正加载该模块）<sup>[25](#mod-syscall-ref-25)</sup>。
- **`#[cfg]`覆盖不全**：`mod.rs`中`#[cfg(target_arch = "x86_64")]`的条件在多个系统调用中应用，而`#[cfg(not(target_arch = "riscv64"))]`用于`renameat`，这些条件可能导致在未提及的架构（如aarch64）上丢失系统调用<sup>[26](#mod-syscall-ref-26)</sup>。不过当前目标架构明确为x86_64和loongarch64，因此风险可控。

- BPF验证器不支持循环，含向后跳转的程序会被拒绝（[ref:9]）。
- ringbuf_reserve和ringbuf_submit/discard实现不完整，实际使用时会失败（[ref:10][ref:7]）。
- 模块加载系统仅对预注册的内置模块有效，不支持动态模块（[ref:12]）。
- 多个unsafe块缺少安全说明，存在未声明的安全假设（[ref:19]）。
