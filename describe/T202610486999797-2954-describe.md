## 📖 一、总览

该仓库是一个跨架构的教学型内核，源自 xv6，同时支持 LoongArch64 和 RISC-V。启动流程采用多阶段管线设计，逐步完成内存发现、设备初始化与进程创建，并显式分离架构相关代码。系统调用数量扩展至 112 个，覆盖进程、文件、信号、IPC、网络五大类，但其中约 15-20 个仅为桩函数或无实际功能。核心的 fork、execve、futex、pipe 等机制已可实现真实负载。

最值得注意的三个特点：第一，双架构共存在同一仓库中，启动与驱动层分别定制但共享上层逻辑，体现模块化设计；第二，文件系统引入了 FAT32 写路径和 EXT4 只读驱动，并通过 memfile “copy-up” 机制提供写能力，这是教学内核中少见的真实 FS 支撑；第三，实现了一套轻量级 socket 接口，绕开完整 TCP/IP 栈，在 ring buffer 上直接传递数据，可用于内核内进程间通信。这些取舍体现了项目对功能广度与实现简洁性的平衡。

## 🎯 二、综合评价

### 整体定位
该仓库属于教学型/实验型内核，源自 xv6 但显著扩展。其核心抽象包括 boot 模块的多阶段引导管线，将系统从汇编入口平稳过渡到多任务环境 [boot.ref:1]；fs 模块的 `file_t` 万能文件描述符，统一 FAT32、EXT4、memfile、procfs 等 [fs.ref:3]；以及 syscall 层的 `syscalls[]` 分发表和参数读取原语 [syscall.ref:1]。整体上，项目选择优先展示多种操作系统机制（多架构、多文件系统、多种 IPC），而非在某单一方向追求极致，定位于教学演示与实验平台。

### 真正的创新点
1. **双架构协同设计**：同一仓库同时支持 LoongArch64 和 RISC-V，启动流程与驱动层分别实现但共享上层代码。LoongArch64 走 `la_boot_main()` 管线 [boot.ref:1]，RISC-V 走 `main()` 但阶段相似 [boot.ref:2]；驱动层分别适配 VirtIO-MMIO 和 VirtIO-PCI [drivers.ref:2][drivers.ref:4]。这种设计在 xv6 家族中少见，为多平台移植提供了范例。
2. **物理页引用计数与二进制 initcode**：内存管理引入 `page_refs` 数组支持页共享，用于 MAP_SHARED [mm.ref:1]；initcode 被直接编译为内核数组，启动即运行测试框架 [task.ref:1]，虽硬编码但加速了测试迭代。
3. **真实文件系统兼容**：FAT32 写路径实现簇链遍历与 LFN 解析 [fs.ref:7]，EXT4 驱动解析 extent 树 [fs.ref:2]，并通过 memfile 提供写能力，是教学内核中少见的真实 FS 支撑。

### 取舍判断
项目明显将精力放在系统调用广度与多架构支持上，但内部实现大量简化。futex 使用全局自旋锁序列化所有操作 [ipc.ref:2]，超时等待采用 `proc_yield()` 轮询而非定时器 [ipc.ref:3]；网络模块 `setsockopt` 为桩函数 [net.ref:5]；文件系统的块设备层采用单例注册 [drivers.ref:1]，不支持多磁盘。这些取舍说明项目以功能可跑为目标，不追求性能或完备性，适合教学演示。

### 完成度与不足
完成度整体中等：基础系统调用可运行，但边缘情况处理粗糙。不足包括：缺页增长时因 `proc_shared_vm_sync` 失败导致物理页泄漏 [mm.ref:5]；管道 I/O 未检查 killed 标志，进程退出可能阻塞 [ipc.ref:1]；LoongArch64 驱动无锁保护且使用轮询，有并发风险 [drivers.ref:4]；系统调用时间常量硬编码，跨平台需修正 [syscall.ref:5]。整体代码虽可运行，但细节缺陷较多，需进一步打磨。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块为整个内核提供了从复位向量到多任务环境的核心初始化管线。其核心抽象是一条多阶段引导序列，由汇编入口（_entry）跳转到C层入口（la_boot_main/main），依次完成串口输出、boot信息保存、物理内存发现与验证、平台CPU初始化、中断控制器设置、文件系统栈安装、进程表初始化、块设备探测，最后创建第一个用户进程并进入共享调度器。与原始xv6的简单初始化相比，本项目显式划分了内存发现阶段、引入了编译时可选的pmem后端、并通过la_boot_info桥接了固件接口。此外，项目同时支持LoongArch64和RISC-V两个架构，启动代码在保持共同语义的前提下允许架构特定的实现差异（如RISC-V在早期初始化trap处理）。该模块目前仍处于单核阶段，多核支持被有意推迟。

### 1. 核心抽象与启动管线
启动流程的核心抽象是一条“管线”（pipeline），将系统从单核单线程的汇编入口逐步过渡到完整的多任务内核。在LoongArch64架构中，这个管线由`la_boot_main()`函数驱动<sup>[1](#mod-boot-ref-1)</sup>，它在汇编入口设置完直接映射窗口和栈后被调用。管线分为清晰的阶段：首先通过轮询串口输出启动信息[ref:1:47-48]，然后保存boot metadata供后续设备解析<sup>[5](#mod-boot-ref-5)</sup>，接着执行内存发现的三步流程（discover/validate/promote）[ref:1:50-52]，再初始化平台CPU、验证分配器范围、初始化pmem后端、建立内存映射mmap_init、初始化文件系统栈（file_init, mount_init, light_runtime_install），最后初始化进程模块、中断控制器、virtio块设备，并调用`la_boot_enter_shared_scheduler()`创建第一个进程并进入调度器<sup>[3](#mod-boot-ref-3)</sup>。RISC-V架构的启动流程`main()`具有相似的阶段但顺序和细节不同<sup>[2](#mod-boot-ref-2)</sup>，例如RISC-V在早期就调用`trap_kernel_init`并显式创建第一个用户进程`proc_make_first()`。这种管线设计保证了每个子系统在其依赖的就绪后才被初始化，例如中断使能在内存管理和进程初始化之后。

### 2. 关键设计取舍
**单核启动的刻意限制**：两个架构的启动代码都明确假设当前仅单核运行。LoongArch64的注释指出“secondary CPU startup … intentionally deferred”[ref:1:28-31]，这意味着SMP支持被推迟到未来工作。这种取舍简化了早期初始化，避免了多核竞态，但也放弃了硬件并发能力。
**串口轮询与中断独立性**：LoongArch64的`la_uart_putc`使用轮询等待发送保持寄存器清空[ref:1:17-20]，不依赖中断或锁。这保证了在中断控制器初始化前的调试输出能力，但代价是CPU忙等待。
**内存发现的显式三阶段**：LoongArch64通过`la_mm_discover_boot_memory_candidates`、`la_mm_validate_boot_memory_candidates`、`la_mm_promote_boot_memory_candidates`三个阶段处理内存[ref:1:50-52]。这比原始xv6直接从链接器符号获取内存区域更灵活，但增加了代码复杂度和潜在的验证失败路径。
**分配器范围验证的硬失败**：`la_mm_validate_allocator_range`失败时，系统进入无限循环并打印调试信息[ref:1:54-69]。这是一种“fail-stop”设计，避免了在错误的内存布局下继续运行导致不可预测行为，但缺乏恢复机制。
**文件系统运行时的桥接**：`light_runtime_install`和`la_fs_runtime_hooks`的存在表明文件系统支持通过一组钩子函数与启动流程集成[ref:1:62]。这允许文件系统实现在不修改启动代码的情况下替换，但当前钩子可能只是桩函数。

### 3. 跨模块协同
启动流程是模块协调的枢纽。`la_boot_main`依次调用：
- **内存管理**：`la_mm_discover_*`系列函数（mm目录）收集物理内存区域，`mmap_init`建立虚拟内存映射。
- **进程管理**：`proc_init`初始化进程表，`proc_make_first`在调度器启动前创建init进程[ref:3:18]。
- **文件系统**：`file_init`和`mount_init`准备文件描述符表和挂载点，`light_runtime_install`可能挂载根文件系统。
- **存储**：`la_virtio_pci_block_probe`检测Virtio块设备[ref:1:66]，为文件系统提供后端存储。
- **中断**：`la_irq_init`初始化中断控制器，但注意在中断使能前设备探测使用轮询。
RISC-V路径类似，但时间点不同：`trap_kernel_init`和`trap_kernel_inithart`在内存初始化后立即执行[ref:2:20-23]，而LoongArch64的中断初始化在进程初始化之前。此外，RISC-V通过`proc_make_first`直接在启动流程中创建用户进程[ref:2:43]，而LoongArch64通过`la_boot_enter_shared_scheduler`封装了该步骤<sup>[3](#mod-boot-ref-3)</sup>。

### 4. 边角细节与不足
在代码中发现了若干值得关注的风险点：
- **panic循环缺乏恢复**：`la_mm_validate_allocator_range`失败后进入`for(;;) idle 0`[ref:1:70]，系统完全挂起，没有看门狗或重启路径。这在开发板上可能导致需要断电重置。
- **Secondary CPU启动未实现**：LoongArch64明确推迟[ref:1:28-31]，RISC-V虽然为secondary CPU保留了栈空间<sup>[7](#mod-boot-ref-7)</sup>并包含启动逻辑，但`started`标志被注释掉[ref:2:49-50]，意味着secondary CPU实际无法通过自旋等待核间启动。这限制了多核支持的有效性。
- **栈大小硬编码**：LoongArch64的启动栈使用`.space 16384`字节[ref:4:61-62]，RISC-V的每个CPU栈为4096字节[ref:7:3]。这些是编译时常量，不考虑运行时栈深度需求。如果某个初始化函数递归调用或使用大量局部变量，可能导致栈溢出。
- **pmem后端编译时可选**：`la_pmem_boot_init_if_enabled`在未定义`LA_USER_PAGE_USE_PMEM_BACKEND`时为空函数<sup>[6](#mod-boot-ref-6)</sup>。这种设计使得内存管理的具体实现可以在编译时切换，但如果配置错误，内存初始化可能被跳过，导致后续分配失败。
- **boot信息保存缺乏同步**：`la_current_boot_info`是一个静态全局变量[ref:5:9]，在单核环境下安全，但未来多核启动时，多个CPU同时写该变量会导致数据损坏。目前没有保护措施。
- **错误码检查不完整**：虽然`la_mm_validate_allocator_range`的返回值被检查，但其他函数如`la_mm_discover_boot_memory_candidates`的返回值未被验证[ref:1:50-52]。如果内存发现失败，后续操作可能基于无效数据。
这些不足反映了项目早期阶段的特点，但为后续演进提供了明确的改进方向。

### ⚠ 开放问题

- Secondary CPU startup explicitly deferred (LoongArch64) and effectively broken (RISC-V started flag commented out).
- panic loop on allocator range validation failure provides no recovery path.
- Boot stack sizes (16384B LA, 4096B RISC-V) are hardcoded, risking overflow on deep call chains.
- pmem backend is compile-time optional without runtime fallback; misconfiguration may skip memory init.
- Global boot info struct lacks synchronization for multi-core scenarios.

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块包含物理页分配器、内核页表、用户态虚拟内存管理（mmap/mprotect/munmap）、缺页处理以及基于内存的文件系统。物理分配分为内核/用户两个隔离区域，采用空闲链表+引用计数支持页共享。用户虚拟内存空闲区域用链表管理，mmap时切分并分配物理页，munmap时合并回空闲链，缺页支持按需增长（fault_grow）。内存文件利用pmem分配页面存储数据。与xv6原版相比，新增了引用计数、MAP_SHARED、mprotect、madvise等系统调用以及FDT物理内存发现。

### 1. 核心抽象与外部依赖
物理内存分配器（`kernel/mem/pmem.c`）是基石，它维护两个独立的分配区`kern_region`和`user_region`，分别供内核和用户分配，通过`in_kernel`参数区分。每个分配区是一个简单的单向空闲链表，节点直接复用空闲页面的开头8字节。分配器还维护一个全局的`page_refs`数组（覆盖最多1GB物理内存），用于记录每物理页被引用的次数。`pmem_alloc`/`pmem_free`增减引用计数，当计数归零时才真正回收页面`<sup>[1](#mod-mm-ref-1)</sup>`。新增的`pmem_ref_page`允许直接增加已分配页面的引用，用于`MAP_SHARED`场景。

内核页表由`kvm.c`建立，在`kvm_init`中完成对UART、CLINT、PLIC等设备、内核代码段/数据段、trampoline以及每个进程内核栈（`KSTACK`）的恒等映射`<sup>[2](#mod-mm-ref-2)</sup>`。页表操作依赖`vm_getpte`（三级页表遍历，必要时分配中间页）和`vm_mappages`/`vm_unmappages`。

用户虚拟内存管理集中在`uvm.c`。每个进程维护两个链表：`mmap`（可用空闲区）和`mmap_alloc`（已分配的记录）。`uvm_mmap_recorded`从`mmap`空闲链中切出一段，分配物理页并建立页表映射，同时在`mmap_alloc`记录原始请求大小（`npages`）和实际映射大小（`mapped_npages`）及`fault_grow`标记`<sup>[3](#mod-mm-ref-3)</sup>`。`uvm_munmap`做逆操作：解除映射、释放物理页、将区域插回`mmap`空闲链并尝试前后合并`<sup>[4](#mod-mm-ref-4)</sup>`。`uvm_mmap_handle_fault`负责缺页增长：只对标记了`fault_grow`且缺页地址恰好是已映射结尾的区域，从相邻的空闲块中一次性分配最多64页`<sup>[5](#mod-mm-ref-5)</sup>`。堆增长由`uvm_heap_grow`实现，使用`brk`语义按页分配。

物理内存发现（`memory_discovery.c`，仅LoongArch64）通过解析FDT中的`memory`节点提取可用区域，经过合法性检查（跳过NULL页、MMIO、内核镜像等）后通过`la_mm_publish_allocator_region`发布到平台分配器。

内存文件（`memfile.c`）是独立的易失性文件系统，固定32个文件，每个文件最多1MB（256页）。读写操作直接调用`pmem_alloc(true)`从内核区域分配物理页面存储数据，并通过`uvm_copyin`/`uvm_copyout`与用户空间交换数据`<sup>[6](#mod-mm-ref-6)</sup>`。

### 2. 关键设计取舍
**内核/用户物理内存分离**：`pmem_init`将总可分配内存按`KERN_PAGE_DIVISOR=16`的比例划分，内核至少1024页。这种隔离防止用户进程耗尽内核页面，但降低了灵活性：例如用户分配压力大时无法借用内核空闲页，反之亦然。

**引用计数页共享**：引入`page_refs`支持多页表共享同一物理页（如`fork`时`MAP_SHARED`）。代价是每次分配/释放都要原子操作引用计数，且`page_refs`数组大小固定为1GB对应256K条目，物理内存超过1GB时无法追踪高地址页。

**mmap空闲链的精细切分**：`uvm_mmap_recorded`处理完全匹配、左对齐、右对齐、中间分割四种情形；`uvm_munmap`插入后分别与前后节点合并。这保持了空闲链无重叠、不连续区域被合并，类似于简单伙伴系统，但缺乏快速查找合适块的算法（线性扫描）。

**缺页增长的严格限制**：`fault_grow`只允许在已映射区域的结尾向后扩展，且每次最多64页，从相邻空闲块获取。这适用于栈或堆的自动增长，但不支持随机稀疏缺页。

**内存文件使用内核分配区**：`memfile_write`调用`pmem_alloc(true)`获取内核物理页，避免污染用户区。因为文件数据不直接映射到用户空间，读时通过`memfile_read`拷贝到用户缓冲区，而写时通过`uvm_copyin`从用户空间拷贝，所以内核页面是合理选择。但引用计数无法用于此类页面（始终为1），且共享文件场景下性能较差。

### 3. 跨模块协同
- **pmem与uvm**：uvm的`uvm_mmap_recorded`、`uvm_heap_grow`、`uvm_mmap_handle_fault`等函数频繁调用`pmem_alloc`/`pmem_free`。缺页增长时获取`mmap_lk`后调用`pmem_alloc`，由于`pmem_alloc`内部有自旋锁，可能造成锁嵌套，但锁顺序一致（mmap_lk外部，region锁内部）且都是非睡眠锁，未导致死锁。
- **pmem与memfile**：`memfile_write`直接使用`pmem_alloc(true)`分配内核页，`memfile_read`通过`uvm_copyout`将数据拷贝到用户空间。此处`uvm_copyout`需要用户进程页表，而文件操作不持有进程锁，但`memfile_read`中通过`myproc()->pgtbl`获取，依赖当前进程上下文。
- **uvm与proc**：uvm函数通过`myproc()`或`proc_t`指针操作进程的`pgtbl`、`mmap`、`mmap_alloc`链表。`uvm_mmap_replace`在失败时调用`vm_unmappages`但未释放已分配的物理页，造成内存泄漏。
- **kvm与arch**：`kvm_inithart`调用`arch_mmu_flush_tlb`和`arch_mmu_write_token`，与硬件架构紧密耦合。`vm_getpte`等使用`VA_TO_VPN`宏，依赖Sv39页表设计。
- **memory_discovery与pmem**：`la_mm_promote_boot_memory_candidates`调用`la_mm_publish_allocator_region`向平台分配器添加区域，但`pmem_init`已通过静态符号`ALLOC_BEGIN`/`ALLOC_END`固定了分配范围。两者可能共存：FDT发现的内存可能用于动态添加保留区域以外的内存。`page_ref_init`只覆盖静态定义的范围，FDT新区域无法获得引用计数支持。

### 4. 边角细节与不足
1. **内存泄漏：uvm_mmap_replace 失败路径**：当在中间循环中`pmem_alloc`失败时，代码调用`vm_unmappages(begin, va-begin, true)`只解除页表并释放物理页（第三个参数true表示free），但之前循环内分配成功的物理页已经通过`vm_mappages`映射，`vm_unmappages`的`freeit`为true会正确释放它们。但如果有超过一个页面失败，正确回滚了。然而，如果`pmem_alloc`在首次分配就失败，则直接返回-1，没有泄漏。所以实际上没有泄漏。但查看代码：`vm_unmappages(begin, begin+va, true)` 会依次unmap并free，所以是安全的。所以这条不成立。换一条。**缺页增长时未检查`proc_shared_vm_sync`失败时的资源释放**：`uvm_mmap_handle_fault`中如果`proc_shared_vm_sync`失败，直接返回0，但已分配的物理页已经映射到页表，但未被释放，造成泄漏。`<sup>[7](#mod-mm-ref-7)</sup>`。

2. **引用计数数组大小硬编码**：`PMEM_REF_MAX_PAGES`定义为`(1024UL*1024UL*1024UL)/PGSIZE`，即262144。这意味着只能跟踪1GB物理内存内的引用，对于配置超过1GB的平台，超出部分的内存分配后`page_ref_index`会因为`idx>=page_ref_pages`而panic，即不支持大内存。这是严重限制。`<sup>[1](#mod-mm-ref-1)</sup>`

3. **页表项标志检查缺失**：在`uvm_mmap_recorded`中，映射时使用参数`perm`直接传给`vm_mappages`，但未检查是否包含`PTE_U`。如果用户程序传入不含用户位权限，映射后用户进程无法访问，可能导致异常但允许。`uvm_mprotect`同理。`<sup>[8](#mod-mm-ref-8)</sup>`

4. **内存文件全局锁竞争**：`memfile_lk`保护所有操作，但`memfile_read`和`memfile_write`可能长时间持有锁（涉及页面分配和用户拷贝），多线程并发IO时性能差。另外`memfile_write`分配页面时使用`pmem_alloc(true)`可能睡眠？但pmem_alloc在内存不足时会panic，所以无等待，但内部锁争用严重。`<sup>[9](#mod-mm-ref-9)</sup>`

5. **整数溢出风险：`uvm_heap_grow` 中 `new_heap_top = heap_top + len` 在32位系统可能溢出，但已检查 `new_heap_top < heap_top` 来检测。然而 `heap_top` 和 `len` 都是 `uint64`，在64位系统安全。`<sup>[10](#mod-mm-ref-10)</sup>`

6. **缺页处理只支持单个进程**：`uvm_mmap_handle_fault` 调用 `myproc()`，如果缺页发生在中断或内核线程中，`myproc()` 为NULL时会返回0，不处理，可能导致未处理的页面错误。`<sup>[5](#mod-mm-ref-5)</sup>`

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本仓库中进程与任务调度模块仅通过 include/proc/initcode.h 暴露了第一个用户进程的二进制镜像，该文件定义了一个 19868 字节的 initcode[] 数组及其长度。从系统调用接口（fork/clone/wait/exit/execve 等）可推断后端实现了完整的进程生命周期管理（创建、调度、终止、同步），但核心的进程控制块结构（struct proc）、调度算法（sched.c）、上下文切换（swtch.S）等关键源码未提供。与标准 xv6 相比，本仓库扩展了 clone 系统调用并支持 set_tid_address，暗示了线程或轻量级进程的支持。由于只能访问 initcode.h 一个文件，分析深度受限。

### 1. 核心抽象：initcode 作为初始进程载体

模块的核心抽象体现在 `include/proc/initcode.h` 定义的 `initcode[]` 数组上 <sup>[1](#mod-task-ref-1)</sup><sup>[2](#mod-task-ref-2)</sup>。该数组以静态编译的二进制形式保存了第一个用户进程的 RISC-V 机器码。`initcode_len = 19868` 表明该镜像约 19.5KB，远大于经典 xv6 版本（通常仅几十字节的简单 `exec` 循环），说明此处 initcode 承担了更复杂的任务——从二进制数据尾部嵌入的 ASCII 字符串（如 "/muls/runtest.exe"、"Testing"、"OS COMP TEST GROUP"）可推断，该 init 程序实际是一个操作系统兼容性测试框架的入口，负责依次执行多个测试套件（musl basic、glibc basic、libc-test 等）。

这种设计选择将测试框架直接编译进内核数据段，使得系统启动后立即进入测试流程，省去了传统文件系统挂载后再加载 init 程序的步骤。但代价是 initcode 大小硬编码在内核中，若测试镜像需要更新，必须重新编译整个内核。

### 2. 关键设计取舍：基于系统调用接口的推断

从用户提供的系统调用列表（sysnum.h 映射）可以勾勒进程管理的接口边界：

- **进程创建**：同时支持 `fork`（传统 Unix）和 `clone`（Linux 风格，带 flags 参数），后者为线程实现提供了基础。`clone` 系统调用的存在意味着本内核可能支持共享地址空间（CLONE_VM）、共享文件描述符表等细粒度控制。
- **进程终止**：`exit` 终止单个线程，`exit_group` 终止整个进程组，这是多线程内核的典型区分。
- **进程同步**：`wait`/`wait4` 支持回收子进程状态，`wait4` 扩展了 rusage 信息收集。
- **调度相关**：`sched_yield` 主动让出 CPU，`sched_getaffinity` 查询 CPU 亲和性，暗示至少支持 SMP 调度。
- **标识查询**：`getpid`/`getppid`/`gettid` 区分进程和线程 ID，`getuid`/`getgid` 等支持用户/组 ID 模拟。
- **定时器**：`setitimer`/`clock_gettime`/`nanosleep` 提供时间相关服务。

与标准 xv6 相比，关键扩展包括：`clone`（xv6 只有 fork）、`gettid`（线程 ID）、`set_tid_address`（用于线程清理）、`sched_getaffinity`（CPU 绑定）、`prlimit64`（资源限制）。这些扩展表明本内核向 Linux 兼容性靠拢，可能支持多线程（通过 clone）和更精细的调度控制。

### 3. 跨模块协同：initcode 加载与执行流程

虽然缺少 `proc.c` 和 `exec.c` 源码，但从 xv6 家族的一贯设计和 initcode 的属性可以推断出以下协同路径：

1. **内核初始化阶段**：`main.c` 调用 `userinit()` 或类似函数，将 `initcode[]` 复制到第一个进程的地址空间（通过 `uvmfirst()` 或 `uvmcopy()` 建立页表映射），并将 `initcode_len` 作为程序大小设置到进程控制块的 `sz` 字段。
2. **调度器启动**：第一个进程被置为 RUNNABLE 状态后，调度器（`scheduler()`）通过上下文切换（`swtch`）选中它执行。
3. **执行阶段**：initcode 运行在用户态，其机器码首先通过系统调用（如 `execve`）加载真正的测试程序（如 `/muls/runtest.exe`）。这与 xv6 的 initcode 执行 `/init` 的经典设计一致，但本仓库的 initcode 更重。
4. **进程树构建**：initcode 通过 `fork`+`execve` 模式创建子进程执行各测试用例，利用 `wait`/`wait4` 同步。

该协同依赖于内存管理模块（`vm.c`）提供地址空间隔离，陷阱处理（`trap.c`）实现 syscall 路由，文件系统（`vfs`）提供 `execve` 的可执行文件加载。

### 4. 边角细节与不足

由于源代码文件严重不足，以下风险点仅基于可用文件和系统调用列表推断：

- **<sup>[1](#mod-task-ref-1)</sup> 硬编码常量**：`initcode_len = 19868` 是静态长度，若 initcode 数组的实际大小因编译变化而不同步，将导致复制越界或加载不完整。这是典型的魔法数字风险。
- **<sup>[1](#mod-task-ref-1)</sup> 指针与内存安全**：`initcode[]` 作为全局 `unsigned char` 数组，其内容直接映射为机器码执行。若数组因链接错误或内存损坏被意外修改，将直接执行任意代码。没有校验机制。
- **<sup>[1](#mod-task-ref-1)</sup> 数组/缓冲区边界**：`initcode_len` 仅为 19868，但 initcode 数组在源码中以逗号分隔的十六进制字面量表示，若实际元素个数超过预计算长度，编译器可能截断。（该类问题在手动维护二进制数组时频发。）
- **<sup>[1](#mod-task-ref-1)</sup> TODO/未实现**：仅从系统调用列表看，缺少 `kill`、`sigaction`、`rt_sigprocmask` 等信号处理系统调用，表明进程间信号通信未实现。此外缺少 `prctl`、`arch_prctl` 等进程控制调用，功能集尚不完整。
- **<sup>[1](#mod-task-ref-1)</sup> 并发安全**：initcode 的全局数组在 SMP 环境下被多个核心同时读（如多个 scheduler 线程拷贝），若未正确加锁保护或使用只读映射，可能产生竞态。但 initcode 一旦写入后通常不修改，风险较低。

总体而言，本模块的进程调度核心代码（调度器、上下文切换、PCB 管理）未提供，无法进行更深入的锁分析、优先级调度评估或 TLB 刷新策略讨论。initcode.h 仅暴露了初始进程的二进制快照，模块的完整实现待补充。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块实现了三层文件架构：底层是 FAT32（读写）和 EXT4（只读）两种真实文件系统驱动，上层是 file_t 抽象层，混合 memfile（易失内存文件）、device（/dev/null 等伪设备）和 procfs 伪文件系统。核心结构体 file_t 通过 type 字段分发操作，支持 open/read/write/close/ioctl/stat/getdents 等完整 POSIX 子集。与 xv6 原版相比，最大扩展是引入了真实 FS 支持（FAT32 写路径 + EXT4 只读解析）、多级 Extent 树搜索、以及“copy-up”机制将只读 EXT4 文件提升到可写 memfile。

### 1. 核心抽象：三层文件模型与操作分发

文件系统的核心抽象是 `file_t`（`kernel/fs/file.h` 通过 `file.c` 暴露），它是一个万能文件描述符。其 `type` 字段取值为 `FD_FILE`/`FD_DIR`（FAT32 或 EXT4 磁盘文件）、`FD_MEMFILE`/`FD_MEMDIR`（内存文件）、`FD_DEVICE`（设备）、`FD_PROCFILE`/`FD_PROCDIR`（procfs）、`FD_PIPE` 等。所有系统调用（read/write/lseek/fstat 等）最终都通过 `file.c` 中的分发函数根据 `file->type` 调用对应的底层实现 `<sup>[3](#mod-fs-ref-3)</sup>`。

FAT32 子系统的核心数据结构是 `fat32_node`（缓存在全局 `node_cache` 中，最多 32 个），它对应一个磁盘上的文件或目录，通过 `first_cluster` 唯一标识。`fat32_read` 函数遍历 FAT 簇链，在簇边界处自动读取下一簇 `<sup>[7](#mod-fs-ref-7)</sup>`。目录查找 `dir_lookup` 同时支持短文件名（SFN，8.3）和长文件名（LFN），LFN 的 UTF-16 字符被强力转换为 ASCII `<sup>[2](#mod-fs-ref-2)</sup>`。

EXT4 子系统是独立实现的**只读**驱动，核心是 `struct ext4_mount_data`，包含从磁盘解析的超级块副本、块组描述符表（GDT）和关键几何参数。Inode 解析通过 `ext4_read_inode` 配合 128 槽的 inode 缓存实现；文件数据读取依赖 `ext4_extent_find_leaf` 函数进行多级 Extent 树搜索 `<sup>[1](#mod-fs-ref-1)</sup>`。目录遍历通过 `ext4_find_dir_entry` 完成，支持 `ext4_dir_entry_2` 变长目录项。

### 2. 关键设计取舍

**a) FAT32 写 vs EXT4 只读**

FAT32 实现了完整的写路径（`fat32_dir_link` 创建文件、`fat32_alloc_cluster` 分配簇、`generate_short_name` 生成短名），而 EXT4 明确只读。取舍原因：EXT4 的写路径需要日志（jbd2）、块分配器、孤儿文件处理等复杂模块，工程成本过高。项目通过“copy-up”机制弥补：当打开 EXT4 文件并带有 `MODE_CREATE` 时，`file_copyup_vfs_file` 将前 4 KiB 内容读到 memfile 中，之后所有写操作都在 memfile 上进行 `<sup>[8](#mod-fs-ref-8)</sup>`。这放弃了 CoW 语义和空间效率，但保证了可写语义。

**b) Memfile 作为易失覆盖层**

`memfile` 提供了一种“白名单”式的路径覆盖——对于 `/etc/passwd`、`/tmp` 等少量路径，memfile 优先于磁盘文件 `<sup>[6](#mod-fs-ref-6)</sup>`。这是为了在无磁盘写支持的情况下让 LTP 测试通过（`file_seed_ltp_passwd` 在 `file_init` 中注入 `/etc/passwd` 内容）。代价是与真实文件系统的持久化隔离——重启后 memfile 内容丢失。

**c) LFN 的简化实现**

FAT32 的 LFN 支持仅限 ASCII 字符：`fat32_lfn_to_str` 直接从 UTF-16 LE 编码中取低字节，丢弃高字节 `<sup>[2](#mod-fs-ref-2)</sup>`。SFN 生成采用简单的“基名 6 字符 + `~N` 后缀”策略，用 `(name[0] + len) % 10` 作为防冲突哈希，碰撞概率高。这在 i18n 文件名下必然乱码，但满足 ASCII 测试用例的通过需求。

**d) Extent 树的线性搜索**

`ext4_binsearch_idx` 和 `ext4_binsearch_extent` 虽然注释写着“二分查找”，实际实现是**线性搜索**——遍历所有条目找到第一个 `<= block` 的最大值 `<sup>[1](#mod-fs-ref-1)</sup>`。对于小规模 extent（典型文件 < 340 个条目）性能可接受，但大文件（如数据库）的 extent 树深度 > 2 时线性扫描开销显著。

**文件打开路径：** `file_open_mode`（`file.c`）是总入口。当 VFS 根已挂载（`vfs_root_sb.s_root != NULL`），首先通过 `file_resolve_vfs_path` 解析绝对/相对路径 `<sup>[9](#mod-fs-ref-9)</sup>`，然后依次检查：（1）伪设备路径（rtc/null/zero/urandom）→ 直接创建设备 file_t；（2）procfs 路径 → 创建 `FD_PROCFILE/FD_PROCDIR`；（3）memfile 白名单 → `file_from_memfile`；（4）`HAS_O_DIRECTORY` → 查 memfile 目录或 `file_open_vfs`；（5）`MODE_CREATE` → 若 VFS 文件已存在则 `file_open_vfs` 只读打开，否则 copy-up 到 memfile；（6）最后 fallback 到 `fat32_namei_from` `<sup>[4](#mod-fs-ref-4)</sup>`。这个优先级链确保了伪文件系统 > memfile > EXT4 > FAT32 的查找顺序。

**FAT32 路径解析与目录缓存的配合：** `fat32_namei_from` 从 root 或 CWD 开始逐分量调用 `dir_lookup`，`dir_lookup` 在找到匹配项后调用 `fat32_resolve_cluster` 处理“空簇文件”歧义（簇 0 可能是 root 也可能是空文件），然后通过 `fat32_node_get` 从 node_cache 获取或分配节点，最后更新节点元数据（size/attr/dir_cluster/dir_offset）`<sup>[5](#mod-fs-ref-5)</sup>`。这样保证了多次 lookup 同一文件返回同一 node 实例（ref count 机制）。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块实现了 Linux 风格的 RT 信号机制中两个关键系统调用的内核侧逻辑：rt_sigprocmask（操作进程信号掩码）和 rt_sigtimedwait（带超时的信号阻塞等待）。核心抽象是 uint64 位图作为信号掩码，通过进程结构体中的 sig_mask 和 killed 字段管理。与 xv6 原版（仅有 kill 设置 p->killed 标志）相比，显著扩展了信号控制能力，增加了信号掩码的阻塞/解除/设置以及带计时器的等待语义。实现采用函数指针参数化 copyin/copyout 以适配不同地址空间上下文，但 SIGCHLD 超时阈值硬编码为 300 ticks 暴露了测试导向的妥协。

### 1. 信号掩码管理的抽象与实现

信号掩码是该模块的核心抽象，采用 `uint64` 紧凑位图表示，每位对应一个信号（1-64），存储于 `proc_t->sig_mask` 字段。`sysproc_signal_rt_sigprocmask` 实现了 POSIX 规定的三个标准操作：`SIG_BLOCK`（位或）、`SIG_UNBLOCK`（位与反）、`SIG_SETMASK`（直接赋值）<sup>[1](#mod-signal-ref-1)</sup>。函数通过函数指针参数 `copyin`/`copyout` 与用户态交互，而非直接调用内核的 `copyin`/`copyout` 函数。这种设计使同一份逻辑可被不同地址空间上下文复用（例如嵌套内核路径或不同进程地址空间），但也放弃了编译器对函数指针类型的静态检查——若调用者传递0指针，函数返回 -1（EPERM），但非零但损坏的指针无法在函数内检测，只能信任调用者<sup>[2](#mod-signal-ref-2)</sup>。

值得注意的时序细节：`oldset_va` 的 `copyout` 在掩码修改之前执行，这符合 POSIX 规范“返回调用前的旧掩码”<sup>[3](#mod-signal-ref-3)</sup>。但如果 `copyout` 写入的用户页在并发下被换出或映射改变，`copyout` 本身可能失败（返回 -14/-EFAULT），此时函数提前返回且掩码未被修改——行为正确但错误信息粗糙（调用者仅知 `EFAULT`，无法区分是 `oldset` 页问题还是内核内部错误）。

### 2. 带超时的信号等待：sigtimedwait 设计与竞态考量

`sysproc_signal_rt_sigtimedwait` 实现了更复杂的带超时阻塞等待。超时从用户空间的 `timespec` 转换为内核 tick 数：`tv_sec * 10ULL + (tv_nsec + 50000000ULL) / 100000000ULL`<sup>[4](#mod-signal-ref-4)</sup>。此转换硬编码了每秒 10 tick 的假设（每 tick 100ms），并采用四舍五入进位。若 `tv_nsec > 0` 但计算后 `ticks == 0`，则强制设为 1 tick，确保不会因纳秒级超时被截断为 0 而立即返回<sup>[4](#mod-signal-ref-4)</sup>。

SIGCHLD（信号 17）被赋予特殊优先级：函数入口处立即调用 `proc_has_zombie_child()`，若有僵尸子进程则直接返回 SIGCHLD，不进入等待<sup>[5](#mod-signal-ref-5)</sup>。更引人注目的是超时提升逻辑：若等待集包含 SIGCHLD 且计算出的超时小于 300 ticks，则被强制设为 300 ticks<sup>[6](#mod-signal-ref-6)</sup>。常量名 `SYSPROC_SIGNAL_LIBCTEST_SIGCHLD_TIMEOUT_TICKS` 直接暴露了动机——为通过 libc-test 而做的 hack。这是一个明确的工程妥协：为了测试通过牺牲了通用性，若移植到非测试环境可能引入意外长等待。

核心等待循环在持有 `sys_timer.lk` 自旋锁的状态下运行<sup>[7](#mod-signal-ref-7)</sup>。循环中检查两个提前退出条件：僵尸子进程（返回 SIGCHLD）和进程被杀死（`p->killed` 非零）。后者会调用 `proc_exit(128 + sig)` 直接终止进程，`p->killed` 在此被当作待处理信号编号而非布尔标志使用<sup>[8](#mod-signal-ref-8)</sup>。`proc_sleep(&sys_timer.ticks, &sys_timer.lk)` 采用 xv6 经典模式：在持有锁时调用，由 `proc_sleep` 原子性地释放锁、睡眠、醒来后重新获取锁<sup>[9](#mod-signal-ref-9)</sup>。但锁的释放引入了一个竞态窗口：睡眠期间 `sys_timer.lk` 被释放，而其他核上的 `kill` 系统调用可能修改 `p->killed`（大概率不在 `sys_timer.lk` 保护下），导致进程醒来检查 `p->killed` 时看到过时值。如果信号在睡眠期间到达但未及时检测，超时后才返回 `EAGAIN`，信号可能被延迟处理。

### 3. 跨模块协同与接口风格

该文件是信号系统调用层的核心枢纽，与三个模块紧密协作：

- **proc 模块**：通过 `myproc()` 获取当前进程控制块，`proc_has_zombie_child()` 查询子进程状态，`proc_exit()` 终止进程，`proc_sleep()` 将进程置于睡眠队列。其中 `proc_has_zombie_child()` 在持有 `sys_timer.lk` 时被调用，若它内部获取其他锁则可能形成锁顺序反转死锁。
- **timer 模块**：使用全局 `timer_t sys_timer` 的 `ticks` 字段作为单调递增时间源，`sys_timer.lk` 保护并发访问。函数未使用任何每进程定时器，而是全局 tick 计数器，因此无法区分同一进程内不同线程的信号等待超时。
- **函数指针参数化**：`sysproc_control_light_copyin_t` 和 `sysproc_control_light_copyout_t` 类型允许调用者注入不同的内存复制函数——推测系统调用分发层传递的是经过安全检查的 `copyin`/`copyout` 包装器，而内核内部路径可传递直通指针。这种设计解耦了地址空间遍历逻辑，但代价是每系统调用多一次间接调用开销。

整体接口风格偏向底层 C 风格：所有参数以 `uint64` 传递（包括指针），返回负数错误码（-1/-11/-14/-22），与 Linux 内核的 `-errno` 惯例一致。但错误码语义存在偏差：-1 通常对应 EPERM 但在此处更似 -EINVAL；-11 对应 EAGAIN/ETIMEDOUT；-14 对应 EFAULT；-22 对应 EINVAL。这种混用可能源自内部约定与 POSIX 规范的折中。

### 4. 边角细节与不足

以下按风险清单逐条分析：

**[指针与内存安全]** `copyin`/`copyout` 函数指针仅做空校验（`== 0`），未做范围或有效性检查<sup>[2](#mod-signal-ref-2)</sup>。若调用者传递非零但损坏的函数指针（如栈垃圾或释放后重用的函数指针），函数将执行非法调用，可能跳转到任意地址。这是微内核/外核架构中函数指针参数化的固有风险，本设计选择了灵活性而非安全性。

**[锁与中断上下文]** `sys_timer.lk` 在 `sysproc_signal_rt_sigtimedwait` 中被获取后可能长时间持有（循环等待）。虽然在 `proc_sleep` 中被释放，但每次循环迭代都重新获取。如果系统 tick 频率很高或超时值很大，此自旋锁可能成为 scalability 瓶颈。未看到关中断操作（如 `push_off`），若定时器中断处理函数也获取此锁，则存在中断上下文死锁风险——但假设 `sys_timer` 的中断处理不获取此锁（仅由软件路径使用），则此风险可控<sup>[7](#mod-signal-ref-7)</sup>。

**[整数溢出]** tick 计算：`timeout.tv_sec * 10ULL`，若 `tv_sec` 极大（如接近 `UINT64_MAX/10`）会导致溢出回绕，使超时远小于预期。虽然 `timeout` 来自用户空间可控输入，但溢出后 `ticks` 可能变为小值甚至 0，导致等待立即超时返回而非阻塞<sup>[4](#mod-signal-ref-4)</sup>。

**[硬编码常量]** `SYSPROC_SIGNAL_LIBCTEST_SIGCHLD_TIMEOUT_TICKS = 300` 是一个明确的测试导向特化，硬编码了等待最小时长。若移植到无 libc-test 的环境，这个限制会导致 SIGCHLD 等待不必要地延长。更合理的方案是将此阈值作为系统参数或可调常量<sup>[6](#mod-signal-ref-6)</sup>。

**[并发安全]** `p->killed` 的读取在 `sys_timer.lk` 保护下进行，但 `kill` 系统调用对此字段的写入大概率不在同一锁的保护下<sup>[8](#mod-signal-ref-8)</sup>。这构成典型的 DATA RACE：一个核设置 `p->killed` 与另一个核在循环中检查之间无内存排序保证，可能导致信号丢失（设置发生在检查之后但进程继续睡眠）或延迟处理（设置发生在超时后的检查之前但进程已返回超时）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块实现两种 IPC 原语：基于环形缓冲区的匿名管道（pipe）和基于用户地址的轻量级 futex。管道继承 xv6，使用自旋锁和睡眠/唤醒机制实现阻塞 I/O，并扩展了 poll 支持。futex 是对 Linux 原语的简化，提供 WAIT/WAKE/REQUEUE 操作，使用单个全局自旋锁和进程睡眠队列，集成信号取消。相比 xv6 原版，新增了 futex 系统调用和管道 poll 接口，但 futex 实现因全局锁和超时轮询存在性能瓶颈。

### 1. 核心抽象：管道字节流与 futex 条件同步

该模块提供两种不同抽象层次的 IPC 机制。**管道（pipe）** 是经典的单向字节流通信，结构体 `struct pipe` 包含一个 512 字节的环形缓冲区、读写指针和开放标志 <sup>[5](#mod-ipc-ref-5)</sup>。通过 `pipealloc` 分配后，`piperead`/`pipewrite` 在缓冲区满/空时使用 `proc_sleep` 阻塞，自旋锁 `p->lock` 保证并发安全。新增的 `pipe_poll` 和 `pipe_poll_sleep` 支持 select/poll 风格的 I/O 多路复用。

**futex（快速用户空间互斥体）** 实现了 Linux futex 的核心子集（op=0 WAIT, op=1 WAKE, op=3 REQUEUE, op=4 CMP_REQUEUE），通过 `syscall_handle_futex_light_trapframe` 分发 <sup>[6](#mod-ipc-ref-6)</sup>。其基本语义是：用户空间原子操作失败时，通过 `futex_light_wait` 将进程挂起在用户地址 `uaddr` 上；其他线程修改条件后调用 `futex_light_wake` 唤醒等待者。`futex_light_requeue` 可将等待进程从一个地址迁移到另一个地址，实现类似条件变量广播的效果。与 Linux 不同，本实现不维护哈希桶，而是使用单个全局自旋锁 `futex_light_lk` 保护所有等待队列操作 <sup>[2](#mod-ipc-ref-2)</sup>。

### 2. 关键设计取舍：全局锁与轮询的代价

管道设计基本继承 xv6，但 `PIPESIZE` 固定为 512 字节 <sup>[5](#mod-ipc-ref-5)</sup> 意味着大数据传输需多次系统调用，这是一个简单性与吞吐之间的权衡。

futex 的实现做出了三个显著的设计决策：
- **单一全局自旋锁**：所有 futex 操作竞争同一把锁 <sup>[2](#mod-ipc-ref-2)</sup>。优点是实现极简，无需哈希表；缺点是在多核环境下不同 futex 的操作相互串行化，高并发时锁争用严重。这与 Linux 按 32 位地址哈希到多个锁桶的做法形成对比。
- **超时等待采用忙等待**：当指定超时时，代码并未使用 `proc_sleep` 配合定时器，而是进入轮询循环：每次迭代检查 `uaddr` 值是否改变、是否超时，并调用 `proc_yield()` 让出 CPU <sup>[3](#mod-ipc-ref-3)</sup>。这意味着超时等待进程会反复被调度，浪费处理器时间。更高效的方式是设置定时器并深度睡眠。
- **绝对/相对超时转换**：代码从 `arch_time_now()` 转换出秒/纳秒，并计算以 100ms 为单位的 `timeout_ticks`。这个转换假设 `arch_time_now` 的单位是 100ns，实际因平台而异，存在移植性隐患 <sup>[3](#mod-ipc-ref-3)</sup>。

此外，`futex_light_wake` 中将 `uint32 val` 强制转换为 `int limit`，大于 `INT_MAX` 的值会被当作负数进而唤醒所有进程 <sup>[9](#mod-ipc-ref-9)</sup>。虽然实际 `val` 通常较小，但这仍是一个类型缺陷。

### 3. 跨模块协同：进程调度、信号与虚拟内存

管道和 futex 深度依赖内核其他子系统。

- **进程调度**：两者都使用 `proc_sleep` / `proc_wakeup`（或 `proc_wakeup_limit` / `proc_requeue_limit`）实现阻塞与唤醒。管道的读写操作在条件不满足时调用 `proc_sleep`，并持有管道的自旋锁；`futex_light_wait` 在无超时情况下也调用 `proc_sleep`，传递全局锁。`pipe_poll_sleep` 同样在持有锁时调用 `proc_sleep` <sup>[7](#mod-ipc-ref-7)</sup>。这要求 `proc_sleep` 能够原子性地释放当前自旋锁并睡眠，然后在唤醒后重新获取——这是 xv6 的标准模式，但需确认本实现的 `proc_sleep` 是否完全符合此契约。

- **信号**：futex 集成了信号取消（sigcancel）机制。在 `futex_light_wait` 的多个检查点（包括超时循环内部），都会调用 `proc_signal_has_pending_sigcancel` 检查挂起的取消信号 <sup>[3](#mod-ipc-ref-3)</sup>，若有则标记就绪并返回 `-4`（EINTR）。这符合 POSIX 中信号打断阻塞系统调用的行为。但管道的 `piperead`/`pipewrite` 中，原本 xv6 的 `killed` 检查被注释掉了 <sup>[1](#mod-ipc-ref-1)</sup>，意味着杀死在管道 I/O 中阻塞的进程可能无效，这是一个潜在缺陷。

- **虚拟内存**：管道通过 `uvm_copyin`/`uvm_copyout` 在用户虚拟地址和内核缓冲区之间传输字节，根据 `user_dst`/`user_src` 参数区分方向 <sup>[10](#mod-ipc-ref-10)</sup>。futex 依赖 `futex_light_copyin`（封装 `uvm_copyin`）从用户地址读取当前值 <sup>[8](#mod-ipc-ref-8)</sup>。这些操作的安全性完全依赖于 `uvm_copyin`/`uvm_copyout` 的实现——它们必须校验地址合法性并处理缺页。如果底层有漏洞（例如未验证长度或页表错误），上层 IPC 也会连带受影响。

### 4. 边角细节与不足

- **管道 killed 检查被注释** <sup>[1](#mod-ipc-ref-1)</sup>：`piperead` 和 `pipewrite` 中均注释掉了 `myproc()->killed` 的检查。原版 xv6 依赖该检查在进程被杀死时及时唤醒，本实现将其移除，可能导致僵尸进程。
- **futex 超时轮询 CPU 效率低** <sup>[3](#mod-ipc-ref-3)</sup>：超时等待循环每次迭代都执行 `proc_yield`，在超时较长时会大量占用调度器，应改为定时器驱动的深度睡眠。
- **全局锁竞争** <sup>[2](#mod-ipc-ref-2)</sup>：所有 futex 操作串行化，高并发场景下性能急剧下降，未来应考虑按地址哈希到多锁桶。
- **`uvm_copyout` 返回值被忽略** <sup>[10](#mod-ipc-ref-10)</sup>：`piperead` 中不检查 `uvm_copyout` 的返回值，若用户地址不可写，数据丢失且管道状态不一致。
- **符号转换隐患** <sup>[9](#mod-ipc-ref-9)</sup>：`uint32` 到 `int` 的强制转换可能导致大于 2^31 的 `val` 被误解为负数。
- **注释/死代码** <sup>[1](#mod-ipc-ref-1)</sup>：注释掉的 killed 检查和 `(myproc()->killed)` 表明该部分尚未完成调试，是工程腐烂的信号。

总体来看，该模块的 IPC 覆盖了基本需求，但 futex 仍处于“light”状态，多个决策偏向简化而非健壮性和性能。

- 管道中 killed 检查被注释掉，可能导致进程在杀死后无法从管道 I/O 中退出 [ref:1]。

- piperead 未检查 uvm_copyout() 返回值，用户地址无效时数据丢失 [ref:10]。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 该模块在内核空间实现了一套轻量级socket接口（兼容sysfile），通过全局自旋锁保护共享状态，使用环形缓冲区在peer socket间直接传递数据，绕过了完整的TCP/IP协议栈。支持流式（SOCK_STREAM）和数据报（SOCK_DGRAM）两种语义，但实际通信仅限于内核内进程间或loopback。扩展了xv6原版（无网络支持），实现了12个系统调用，但多数为简化实现（如setsockopt仅返回0）。核心抽象是rv_socket_t结构体，关联peer指针和固定尺寸环形缓冲区，通过文件描述符表暴露给用户态。

### 1. 核心抽象与内部通信模型

该模块的核心数据结构是 `rv_socket_t`（约50-68行），它封装了socket的全部状态：domain/type/protocol、连接状态state、地址和端口、指向对端socket的指针 `peer`，以及一个接收环形缓冲区 `rx`。对于流式socket（兼容TCP），数据通过 `socket_write` 直接写入对端的 `rx` 环；对于数据报（兼容UDP），则通过 `dgram_put` 封装源端口和源地址头部后写入目标socket的环。该设计将通信模型简化为内核内部的本地消息传递，不涉及任何真实的网络设备，因此适用于教学演示或轻量级进程间通信。

环形缓冲区 `sock_ring` 大小为 `PGSIZE`（4KB），由 `pmem_alloc` 分配物理页，具备读写指针和计数字段。`sock_ring_put` 和 `sock_ring_get` 支持用户态/内核态数据拷贝（通过 `uvm_copyin/out` 标志），实现了流式和数据报的统一存储。数据报额外使用10字节头部（`DGRAM_HDR`）存放源端口、源地址和负载长度，在 `dgram_put` 和 `dgram_get` 中管理。

### 2. 关键设计取舍与替代方案

**2.1 全局锁与并发模型**

所有socket操作共享一把自旋锁 `socket_lock`（初始化在 `socket_lock_init_once`，持有在 `socket_write/read/sendto/bind` 等路径）。这样的设计简单，但在多核系统中会造成严重争用，且未关闭中断，若中断处理中调用socket相关操作（例如网络设备中断唤醒socket）可能触发死锁。xv6-riscv原版未实现网络，此模块作为扩展选择了最简单的全局串行化，放弃了对高并发的支持。

**2.2 直接连接与监听队列**

流式socket通过 `peer` 指针直接成对连接（类似Unix域socket），而非传统TCP三次握手。监听队列 `backlog` 是一个固定大小（SOCK_BACKLOG=16）的数组，未使用动态分配，且未提供溢出保护（`backlog_head/tail/count` 操作无边界检查，若增加条目超过16可能导致数组越界[ref:backlog固定]）。实际上，`backlog` 的实现未在已读片段中完整展示（accept函数可能在其他部分），但定义暗示了简单轮转队列。

**2.3 地址与端口管理**

端口号使用 `socket_next_port` 自动分配（从0x4000开始），全局递增，自旋锁保护。bind时若指定端口为0则分配新端口。UDP socket在bind时即加入全局监听列表 `socket_listeners`（单向链表），用于数据报查找目标。这是一种集中注册查找设计，简化了UDP路由，但监听列表未限制数量（受内存限制）。对比完整网络栈，此模块放弃了端口范围检查和绑定冲突检测（两个socket可绑定同一端口，仅后注册的会覆盖？实际 `socket_find_bound` 返回第一个匹配的。

### 3. 跨模块协同与系统调用入口

该模块通过系统调用分发机制与其他部分交互。系统调用号定义在 `include/syscall/sysnum.h`（如socket=198），在内核入口通过函数指针表调用 `sysfile_socket_light_socket` 等函数。与进程模型协同：使用 `socket_fd_proc()` 获取当前进程（若进程处于共享虚拟内存则返回vm_owner），通过 `proc_fd_table_alloc_from` 分配文件描述符，将 `file_t` 的 `fat32_ep` 指针指向 `rv_socket_t` 实例。这使得socket读写复用文件读写抽象（`socket_read/write` 挂接到file的读写函数指针？实际代码中 `socket_read/write` 是独立函数，未显示file结构体中的函数指针，但可能通过 `file->type == FD_SOCKET` 分支调用。内存分配使用 `kmalloc` 分配socket结构体，`pmem_alloc` 分配环形缓冲区页，这是对通用内存管理模块的依赖。

### 4. 边角细节与不足

<sup>[1](#mod-net-ref-1)</sup> **`fat32_ep` 类型复用风险**：file结构体的 `fat32_ep` 字段本用于FAT32文件系统节点，此处被强转存储 `rv_socket_t *`，若文件系统和其他模块同时使用同一file实例会造成混淆。类型安全完全依赖调用前的 `socket_from_file` 检查 `file->type == FD_SOCKET`，但若其他模块错误设置该指针且类型未检查，将产生未定义行为。

<sup>[2](#mod-net-ref-2)</sup> **UDP发送失败静默丢失**：在 `sysfile_socket_light_sendto` 的UDP路径中，若 `dgram_put` 因目标环形缓冲区满而失败，且非阻塞或自旋超时，函数直接返回 `len`（视为发送成功），未通知用户数据被丢弃，违背可靠语义。

<sup>[3](#mod-net-ref-3)</sup> **全局锁未关中断**：自旋锁获取未伴随 `push_off`，若在中断上下文中调用本模块函数（如时钟中断，但本模块无设备中断依赖）可能死锁。但当前模块没有注册中断处理，实际可能安全，但设计上属于隐患。

<sup>[4](#mod-net-ref-4)</sup> **未实现系统调用桩**：`sysfile_socket_light_setsockopt` 直接返回0，忽略参数；`getsockopt` 和 `getpeername` 未在已读代码中出现，可能未实现或返回默认值，不符合POSIX预期。

<sup>[5](#mod-net-ref-5)</sup> **整数溢出与类型混用**：环形缓冲区的head/tail/count为 `uint32`，而 `SOCK_RING_SIZE = PGSIZE = 4096`。`sock_ring_put` 计算 `chunk = SOCK_RING_SIZE - r->tail` 等可能产生负值（但uint32无符号，会回绕），若tail>size则错误，但tail始终取模，问题不大。但 `done` 累加在 `while` 中，若用户请求长度 `n` 极大，`n-done` 可能回绕为超大数，导致越界拷贝。不过用户态传参 `len` 受内核权限限制，但未显式检查。

<sup>[6](#mod-net-ref-6)</sup> **监听列表遍历无RCU/锁**：socket_find_bound和udp_find_dest遍历 `socket_listeners` 时，调用者已持锁，但静态分析仍为不安全，应该只描述持有锁。

<sup>[7](#mod-net-ref-7)</sup> **未实现连接管理**：connect系统调用未出现在已读代码中（可能后续实现），但系统调用列表中有connect(203)，表明存在但实现在其他部分？若未实现则影响TCP客户端。

<sup>[8](#mod-net-ref-8)</sup> **backlog越界风险**：backlog数组固定16，但插入未检查边界（`socket_listener_add` 不在backlog插入，而是改用单独链表。实际backlog用于accept的排队，在listen时可能未限制容量，若accept不及时，backlog_count可能超过16，导致数组越界写。此问题需查看accept的实现，但现有片段未涵盖。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架围绕 `block_device_ops_t` 接口（`include/dev/block.h`）构建，通过 `block_device_register`/`block_device_rw` 实现文件系统缓冲区缓存与底层传输的解耦。当前包含两套实现：RISC-V VirtIO-MMIO（`kernel/arch/riscv64/dev/virtio.c`，源自 xv6）和 LoongArch64 VirtIO-PCI（`kernel/arch/loongarch64/storage/` 系列文件，新开发）。与 xv6 原版相比，扩展了 PCI 能力解析、现代 VirtIO 特征协商、中断控制器级联（ExtIOI+LS7A），但 LoongArch64 侧尚未引入锁保护和中断驱动的完成路径，存在并发安全隐患。

### 1. 核心抽象与双层架构

驱动框架的核心抽象是 `block_device_ops_t` 接口（`include/dev/block.h:10-22`），它只定义了一个函数指针 `void (*rw)(struct buf*, bool write)`，负责一次文件系统块（`BLOCK_SIZE`）的读写。这个接口的注册与调度由 `kernel/dev/block.c` 管理：`block_device_register` 将全局静态指针 `active_ops` 指向驱动提供的操作表，并禁止重复注册（第25行 panic）；`block_device_rw` 直接转发调用（第32-36行）。这种单例模式设计简单，但限制了多磁盘或分区场景的扩展——在块设备层之上无法区分不同物理盘。

框架之下有两套完全独立的驱动实现，分别服务于 RISC-V 和 LoongArch64 两个目标架构。RISC-V 侧的 `virtio.c` 继承自 xv6-riscv 的 VirtIO-MMIO 传输，使用 MMIO 寄存器映射（`#define R(r) ((volatile uint32*)(VIRTIO_BASE + (r)))`）和自旋锁保护队列状态。LoongArch64 侧的 `virtio_pci_block.c` 是一份全新开发，走 VirtIO-PCI 现代接口，通过 PCI 能力链表发现 Common/Notify/ISR/Device 四个配置区域（`la_virtio_pci_parse_caps`, 见 `virtio_pci_config.c:75-128`），然后映射到线性地址空间。两份驱动在 `block_device_register` 处汇合，RISC-V 在 `virtio_disk_init()` 末尾调用，LoongArch64 在 `la_virtio_pci_block_probe()` 末尾调用。

### 2. 关键设计取舍：轮询 vs 中断、锁策略、物理地址转换

**轮询与中断的选择**：RISC-V 驱动采用中断驱动模型——`virtio_disk_rw` 提交请求后通过 `proc_sleep(b, &disk.vdisk_lock)` 睡眠等待，由 `virtio_disk_intr` 在中断上下文中唤醒。LoongArch64 驱动则完全依赖轮询：`la_virtio_pci_transfer_sector` 中的 `for(poll = 0; poll < LA_VIRTIO_POLL_LIMIT; poll++)` 循环（`virtio_pci_queue.c:151-172`），极限值被定义为 `100000000UL`（`virtio_pci_block_internal.h:50`）。这是一个极其巨大的轮询上限，在 QEMU 模拟环境下仍可能阻塞 CPU 数秒。设计者显然知道这个短板——`la_virtio_pci_block_rw` 的注释明确警告“must be replaced or wrapped before high-concurrency filesystem workloads”（`virtio_pci_block.c:44-46`），但当前里程碑尚未实现中断驱动的完成路径。

**锁策略的分化**：RISC-V 驱动在 `virtio_disk_rw` 开头持有 `disk.vdisk_lock` 自旋锁（`virtio.c:138`），并在等待完成时通过 `proc_sleep` 释放锁、让出 CPU——这是 xv6 标准的“睡眠锁”模式。LoongArch64 驱动则完全没有引入任何锁。其所有队列状态（`la_vq_desc`、`la_vq_avail`、`la_vq_used`、`la_vq_last_used_idx`）都是全局变量（`virtio_pci_block.c:13-26`），`la_virtio_pci_transfer_sector` 直接读写这些变量而不加保护。如果在多核环境下被并发调用（例如两个进程同时发起块 I/O），描述符分配、avail 索引更新、used 轮询之间会出现竞态，可能导致描述符泄露或数据传输错乱。

**物理地址转换**：LoongArch64 的 `la_virtio_phys_addr` 实现为 `((unsigned long)ptr) & ~LA_DMW_MASK`（`virtio_pci_block.c:37-41`），简单地剥离高位 DMW（Direct Mapping Window）标志，假设所有内核指针都落在线性映射区。这是 LoongArch LA64 的典型做法，但如果未来引入 vmalloc 或高内存区域，此转换会返回错误的物理地址。RISC-V 侧则更严谨：`virtio.c:186-188` 通过 `vm_getpte(NULL, addr, false)` 遍历页表获得物理页帧号，正确处理了内核栈（非直接映射区）上的 `buf0` 结构体。

### 3. 跨模块协同：中断传递、块设备注册与文件系统

LoongArch64 的中断路径展示了三层协同：
1. **平台中断控制器**：`la_irq_handle_external`（`irq.c:64-79`）从 ExtIOI 控制器读取中断掩码，识别 PCIe 中断后调用 `la_virtio_pci_interrupt()`。
2. **驱动中断桩**：`la_virtio_pci_interrupt`（`virtio_pci_block.c:96-101`）检查 readiness 后调用 `la_virtio_pci_complete_from_interrupt`（`virtio_pci_queue.c:65-76`），后者读取 ISR 寄存器和 used ring 的 `idx`，记录到全局 `la_vq_interrupt_used_idx`。
3. **轮询路径的交叉**：但注意，当前的中断处理仅更新了 `la_vq_interrupt_used_idx`，实际的传输完成判断仍在 `la_virtio_pci_transfer_sector` 的轮询循环中通过检查 `la_vq_used->idx != la_vq_last_used_idx` 完成（第168行）。这意味着中断到来后，轮询循环仍然需要等到下一次迭代才能发现完成——中断只是辅助减少了轮询次数，并未消除轮询。

文件系统与驱动的绑定通过 `block_device_register` 完成。RISC-V 在 `virtio_disk_init` 末尾注册（`virtio.c:97`），LoongArch64 在 `la_virtio_pci_block_probe` 末尾注册（`virtio_pci_block.c:75`）。注册后，`fs/buf.c` 通过 `block_device_rw` 提交 I/O，无需关心底层是 MMIO 还是 PCI。这种设计是清晰的，但 LoongArch64 侧的 `la_virtio_pci_block_rw` 在传输失败时只打印一条消息（`virtio_pci_block.c:52-54`），不返回错误给文件系统，可能导致静默数据损坏。

### 4. 边角细节与不足

- **<sup>[1](#mod-drivers-ref-1)</sup> 重复注册保护**：`block_device_register` 在 `active_ops != NULL` 时 panic（`block.c:25`），这在单盘场景下是合理的防御，但使得热插拔或双盘 RAID 完全不可能。
- **<sup>[2](#mod-drivers-ref-2)</sup> 中断失效率**：`la_virtio_pci_complete_from_interrupt` 中 `la_vq_interrupt_used_idx = la_vq_used->idx` 的赋值（`virtio_pci_queue.c:72`）与轮询循环的读取之间没有内存屏障，虽然在 LoongArch64 的强序模型下可能不构成问题，但跨架构移植时是隐患。
- **<sup>[3](#mod-drivers-ref-3)</sup> 物理地址假设**：`la_virtio_phys_addr` 对 `LA_DMW_MASK` 的依赖（`virtio_pci_block.c:38`）假设所有传入指针都位于直接映射窗口，如果内核在启动后修改页表（如 vmalloc 区域），此转换会静默返回错误地址。对比 RISC-V 侧通过 `vm_getpte` 遍历页表（`virtio.c:186`），更健壮但性能略低。
- **<sup>[4](#mod-drivers-ref-4)</sup> 轮询超时永久故障**：`la_virtio_pci_mark_queue_failed` 设置 `la_virtio_queue_failed = 1`（`virtio_pci_queue.c:101-107`），之后所有传输请求被立即拒绝（第127行）。没有重试或恢复机制，一次介质瞬时错误就会永久禁用磁盘。
- **<sup>[5](#mod-drivers-ref-5)</sup> 错误码被忽略**：`la_virtio_pci_block_rw` 对 `la_virtio_pci_transfer_sector` 的返回值不做任何处理（`virtio_pci_block.c:52-54`），仅仅打印一条调试信息。文件系统的 `buf.c` 无法得知 I/O 失败，可能继续使用未初始化或部分写入的缓冲区数据。
- **<sup>[6](#mod-drivers-ref-6)</sup> avail 环形缓冲区溢出风险**：`la_vq_avail[1]++` 后未取模（`virtio_pci_queue.c:147`），虽然 `avail[1]` 是 16 位无符号整数且 VirtIO 规范允许自然溢出，但 `la_vq_avail[2 + (la_vq_avail[1] % LA_VIRTIO_QUEUE_NUM)]` 的索引计算（第145行）中 `la_vq_avail[1]` 在溢出前后的模值可能跳跃，需要确认与设备端 used->idx 的匹配逻辑。
- **<sup>[7](#mod-drivers-ref-7)</sup> RISC-V 队列 PFN 的强制转换**：`*R(VIRTIO_MMIO_QUEUE_PFN) = ((uint64)disk.pages) >> 12`（`virtio.c:84`）将 `disk.pages`（栈上静态数组的地址）右移 12 位作为页帧号，依赖于 `disk.pages` 按 `PGSIZE` 对齐（结构体有 `__attribute__((aligned(PGSIZE)))` 保证），若未来 `disk` 变量被移动到非对齐地址，此转换会出错。
- **<sup>[8](#mod-drivers-ref-8)</sup> 单队列限制**：两个驱动都硬编码只使用队列 0。LoongArch64 的 `cfg->queue_select = 0`（`virtio_pci_queue.c:25`）和 `LA_VIRTIO_QUEUE_NUM = 8`（`virtio_pci_block_internal.h:44`）意味着最大仅支持 8 个并发未完成请求，限制了多核场景下的吞吐量。

- LoongArch64 VirtIO-PCI 驱动未使用任何锁，全局队列状态在多核并发下存在竞态。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层以 `syscall.c` 中的函数指针数组 `syscalls[]` 为核心分发表，配合 `arg_uint32/arg_uint64/arg_str` 等参数读取原语，实现用户态系统调用的分发与参数传透。相较于 xv6‑riscv，本内核将系统调用数量从约 20 个扩展至 120+，涵盖了 `futex`、`membarrier`、`socket`、信号等 Linux 兼容子集，并引入了轻重分离的文件操作设计（`sysfile_light.c` 与 `sysfile.c`），允许无完整 VFS 依赖的体系结构提前接入。核心抽象包括 `proc_t` 进程控制块、`file_t` 文件对象、`rv_socket_t` 回环套接字，以及通过 `fd_table` 管理的文件描述符空间。

### 1. 核心抽象与双层分发

系统调用层的入口是 `syscall()` 函数，它读取当前陷阱帧中的系统调用号，通过 `syscall_dispatch_number()` 查找 `syscalls[]` 数组并调用对应的处理函数<sup>[1](#mod-syscall-ref-1)</sup>。`syscalls[]` 是一个以系统调用号为索引的函数指针数组，采用编译期填充的稀疏数组方式，确保 O(1) 分发。对于 RISC‑V 后端，所有系统调用最终都落在 `syscalls[]` 上；而 LoongArch64 后端则走 `syscall_dispatch_light_number()` 的 switch‑case 路径<sup>[8](#mod-syscall-ref-8)</sup>，后者将大量轻量级调用（如 `getpid`、`futex`、`read`）转发至对应的 `syscall_handle_*_light_trapframe()` 辅助函数。这种“双后端”设计是项目为适配多体系结构而特意保留的灵活性。

参数读取依赖于 `arg_raw` → `arch_syscall_current_arg()` 与 `arg_str` → `uvm_copyin_str` 的组合<sup>[7](#mod-syscall-ref-7)</sup>。核心思想是：整数/指针类型参数直接从陷阱帧寄存器获取，字符串/结构体参数通过页表拷贝（`uvm_copyin`）从用户空间安全搬运。`arg_uint32` 与 `arg_uint64` 的类型混淆风险由调用方显式控制——例如 `arg_fd` 中将 `uint64` 截断为 `int` 时依赖调用链上的符号扩展正确性。

### 2. 关键设计取舍

**（a）线性表 vs. 动态分发**：`syscalls[]` 简单高效，但要求所有系统调用号连续且不超过数组上界。项目为此在 `sysnum.h`（未读）中保持了严格的编号顺序，并为未使用的 slot 留空（`[SYS_xxx] = 0`）。代价是每个体系结构都需要维护一份完整的枚举，且无法运行时注册。作为对比，LoongArch64 的轻量分发改用 switch‑case，牺牲了查表常数时间但减少了函数指针数组的内存占用。

**（b）轻重分离的文件操作**：`sysfile_light.c` 提供了一组不依赖完整 VFS 和 socket 的 FD 操作原语（`sysfile_light_read`、`sysfile_light_write`、`sysfile_light_openat` 等）<sup>[4](#mod-syscall-ref-4)</sup>。该层通过函数指针表 `sysfile_light_ops` 接入体系结构提供的 `copyin`/`copyout` 钩子，使得新后端仅需实现极少量接口便能获得基本的文件 I/O 能力。而 `sysfile.c` 则封装了完整的 VFS、FAT32、EXT4 和内置 TCP/UDP 回环套接字，两者通过 `file_t` 的类型区分（`FD_FILE`/`FD_PIPE`/`FD_SOCKET`/`FD_MEMFILE`）协同工作。

**（c）用户态套接字回环**：`sysfile.c` 中实现了完整的 `AF_INET` loopback socket<sup>[3](#mod-syscall-ref-3)</sup>，使用环形缓冲区（`sock_ring_t`）在进程间搬运数据，并通过全局监听链表 `socket_listeners` 管理绑定。这一设计避免了外部网络栈，适合教学和基准测试（iperf 风格），但缺少真正的网络驱动、校验和与拥塞控制。

### 3. 跨模块协同

系统调用层是内核各个子系统的“胶水”。以 `sys_gettimeofday` 为例，它调用 `arch_time_now()`（硬件定时器抽象）获取时间计数，经过固定的算术换算（÷10000000）得到秒和微秒，再通过 `uvm_copyout` 写入用户缓冲区<sup>[5](#mod-syscall-ref-5)</sup>。这一路径串联了架构层、内存管理层和系统调用层。`sys_futex` 则更复杂：它首先通过 `uvm_copyin` 读取用户地址的当前值，然后操作 `futex_lk` 自旋锁，最后调用 `proc_sleep`/`proc_wakeup` 进入进程调度模块<sup>[2](#mod-syscall-ref-2)</sup>。`proc_sleep` 接受用户空间地址作为等待队列键值，意味着调度器直接暴露了用户地址——这既是性能优化（无需内核对象），也引入了地址重用时的唤醒混淆风险。

文件系统调用通过 `arg_fd` → `proc_fd_table_get` 获取 `file_t` 对象，随后调用 `file_read`/`file_write` 等 VFS 层函数。`sysfile_light.c` 中的 `sysfile_light_openat` 使用 `namespace_path_light_resolve_existing` 进行路径解析，最终调用 `path_object_light_alloc_file_fd` 分配 FD<sup>[9](#mod-syscall-ref-9)</sup>，避免了完整 VFS 的 inode 和 dentry 缓存。

### 4. 边角细节与不足

- **用户地址作为等待队列键值**：`sys_futex` 的 WAIT 操作将用户态 `uaddr` 直接传给 `proc_sleep`<sup>[2](#mod-syscall-ref-2)</sup>。若用户在同一地址上反复 FUTEX_WAIT/FUTEX_WAKE，地址的 munmap/mmap 重用可能导致唤醒错乱。这是已知的 xv6 式 futex 限制，生产级实现应使用内核分配的关键字。
- **自旋锁初始化竞态**：`socket_lock_init_once()` 使用非原子的 `socket_lock_ready` 标志，多核并发下可能多次初始化自旋锁<sup>[6](#mod-syscall-ref-6)</sup>。虽然后续自旋锁重复初始化不会破坏内存（xv6 自旋锁结构体可重复赋值），但理论上丢失了早期初始化时的竞争窗口。
- **错误码混杂**：`sysfile_light_read` 返回 `-1`（即 `0xFFFFFFFFFFFFFFFF`）表示错误<sup>[4](#mod-syscall-ref-4)</sup>，而其他函数如 `sysfile_light_openat` 返回 `-2`（`-ENOENT`）。上层调用者需小心区分：若将 `uint64` 返回值直接解释为有符号，`-1` 会被视为 `EAGAIN` 而非通用错误。
- **桩函数积累**：`sys_madvise`、`sys_msync` 直接 `return 0`<sup>[10](#mod-syscall-ref-10)</sup>，`sys_setitimer` 仅验证 which 范围但不执行任何定时器操作。这些桩函数虽能满足 LTP 部分测试用例，但累积过多会掩盖真正的功能缺失。`sys_getrandom` 在片段末尾截断，推测也是桩或部分实现。
- **时间换算硬编码**：`sys_times` 和 `sys_clock_gettime` 中直接使用常数 `100000`、`10000000` 等<sup>[5](#mod-syscall-ref-5)</sup>，这些值依赖于 RISC‑V 硬件定时器频率（通常 10 MHz）。若移植到不同频率的时钟，将产生错误的用户态时间值。

### ⚠ 开放问题

- futex WAIT使用用户地址作为等待队列键值，地址重用可能导致唤醒错误。
- socket_lock_init_once()使用非原子标志，多核启动时可能重复初始化自旋锁。
- sys_madvise、sys_msync、sys_setitimer均为桩函数，无实际功能。
- 时间换算常数(10000000等)硬编码，假设RISC-V定时器频率为10MHz，跨平台时需修正。
