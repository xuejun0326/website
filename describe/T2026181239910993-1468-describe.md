## 📖 一、总览

该仓库属于 rcore-rust 家族中“实验性 Linux 兼容内核”类型。与教学型 rCore-Tutorial 不同，其代码规模庞大、模块划分细致，旨在提供更完整 Linux 系统调用兼容性，并接入了 landlock、fanotify、KSM 等高级特性。项目使用 polyhal 封装多架构启动入口，内存管理基于 MapArea trait 实现了惰性分配、写时复制和文件映射，进程管理采用 PCB/TCB 双层结构并支持 landlock 沙箱，文件系统则引入基于 lwext4 的 ext4 和完整 fanotify 框架。然而，项目整体仍处于原型阶段，各模块完成度参差不齐。

该仓最值得评审注意的特点有三：第一，高级特性集成密度高——内存合并（KSM）([mm.ref:5])、文件系统监控（fanotify）、安全域（landlock）等在现代 Linux 中经过多年演进的机制，在此被一并纳入，且 KSM 和 fanotify 的实现并非简单存根，而是具备完整功能流的尝试；第二，网络协议栈完全自主研发——抛弃了 smoltcp 等 crate，手写 ARP/IP/TCP 处理，体现开发团队对协议细节的控制需求，但也因此牺牲了硬件 offload 和协议覆盖面；第三，设计选择中处处可见“先让功能跑起来”的实用主义——从全局 bounce buffer 串行化 DMA 到 futex 超时函数未接入时钟中断，从 pipe 容量调整空存根到设备注册路径被注释，这些痕迹揭示了项目当前所处的演进阶段。

## 🎯 二、综合评价

### 整体定位
该内核定位为 Linux ABI 兼容的多架构实验内核，代码量达 72.9 万行，远超典型教学内核。它借鉴了 rCore 的部分基础设施（polyhal、VirtIO 驱动），但自主实现了内存管理、进程调度、文件系统、网络协议栈等核心子系统。与同一家族的 ArceOS 相比，本仓库更追求特性广度而非稳定性，大量移植了 Linux 5.x 时代引入的新机制（如 landlock、fanotify、KSM），在 Rust 内核社区中独树一帜。但同时也暴露出原型项目的典型缺陷：各模块发展不均，高级功能往往附带未实现的边界条件和硬编码假设。

### 真正的创新点
本节聚焦该项目在 rCore 生态中真正与众不同的设计。

1. **手写 TCP/IP 协议栈**：网络模块完全自建，未依赖社区 crate，包含了从网络设备驱动到应用层 TCP 服务（端口 8080）的完整路径。它以内核线程方式运行 TCP 状态机，支持基础的拥塞控制和重传，展示了在 Rust 中从零实现协议栈的可行性。但其不提供 UDP/TCP 多队列、硬件校验和 offload 等能力，与 smoltcp 相比协议覆盖面有限。

2. **KSM 内存合并**：内存管理模块实验性地实现了内核同页合并（[mm.ref:5]），这是 Linux 服务器常用的内存节省技术。实现中，进程地址空间的可合并区域通过 cow_flag 和 merge_flag 标记，后台线程周期扫描并合并相同页。但由于使用全局锁保护扫描过程，当大量进程并发时可能成为瓶颈，且未处理页表合并后的反向映射更新。

3. **fanotify 文件监控**：文件系统模块实现了 Linux fanotify 框架，支持事件标记、权限检查、队列导出等完整流程。fanotify 子系统是 Linux 安全审计的重要接口，本实现将其与 VFS 层和任务调度模块集成，能够在文件访问事件中插入权限判断。不过当前实现缺少对 mount 事件的监控，且未实现 FAN_OPEN_EXEC_PERM 等复杂事件。

4. **Landlock 安全域**：进程模块通过 landlock 规则集提供可组合的沙箱能力（[syscall.ref:3]），实现了 LANDLOCK_CREATE_RULESET、LANDLOCK_ADD_RULE、LANDLOCK_RESTRICT_SELF 等系统调用。与 Linux 内核实现相比，本项目的 landlock 仅支持文件系统动作（路径访问），不涉及网络、信号等资源，但能在用户态进程中使用，是本项目中完成度较高的创新特性。

### 取舍判断
开发团队在多处做出了明确取舍：

- **启动抽象**：采用 polyhal 的 `#[polyhal::arch_entry]` 属性宏（[boot.ref:1]）代替手写 entry.S，获得了架构无关的入口点，但多核启动逻辑目前仅支持 riscv64，且调试时需要深入 polyhal 内部。这一取舍减少了汇编代码，但增加了对第三方 crate 的依赖与调试复杂度。
- **驱动层**：VirtIO 块驱动使用全局 BlkIoBounce 单例进行 DMA（[drivers.ref:2]），简化了物理内存管理，代价是块 I/O 完全串行化，且固定 4KB 缓冲区无法支持大块请求。同时，PCI 枚举发现的设备未注册到全局块设备表（probe.rs 中注册函数被注释），使得驱动框架目前实际不可用——这可能是优先级调整后的结果。
- **系统调用层**：采用直接解引用用户指针的方式（通过设置 SUM 位），而非逐页拷贝。这能让短路径 syscall 代码简洁，但要求每个 syscall 入口正确设置 SUM 位。如信号模块每个 syscall 开头显式调用 _set_sum_bit（[syscall.ref:1]），而进程模块的其他 syscall 依赖 trap 入口统一设置，这种不一致可能隐藏 bug。另一取舍是优先移植 Linux 高级系统调用（landlock、fanotify）而放弃完善基础 IPC 的时效性（如 futex 超时不可用）。
- **网络协议栈**：手写协议栈放弃了通用性和硬件事务，但换取了完全可控的代码量（约 2k 行）和内核内建 TCP 服务的灵活性。这是一种适合教学和特殊场景但不适合通用部署的典型取舍。

### 完成度与不足
该内核的完成度可评价为中等（2/3）。从宏观看，核心子系统（进程、内存、文件系统、网络）均实现了关键功能路径，系统调用数达 200，能支撑 Linux 用户态的基本执行。但深入微观，许多模块存在“未完成”的标记：

- 内存管理：物理页分配失败直接 panic（vm_area.rs:178），而非适当返回错误；KSM 全局锁长持有，合并粒度粗糙。
- 进程调度：execve 强制单线程（[task.ref:4]）；block_current_and_run_next 中 zombie_flag 的处理可能导致状态切换问题；rlimit_nofile 默认 1024 但无运行时调整（系统调用 prlimit64 已实现但未截断上限）。
- IPC：futex 超时函数 check_futex_timeouts 被 #[allow(unused)] 标记（[ipc.ref:5]），带超时的 FUTEX_WAIT 实际永不返回 ETIMEDOUT；pipe 容量调整写操作是空存根（[ipc.ref:3]）；共享内存 shmctl 部分命令直接使用用户裸指针构造 slice 且未验证可写性（[ipc.ref:4]）。
- 信号机制：wakeup_first_blocked_task 被 #[allow(dead_code)] 标记（[syscall.ref:8]），信号触发唤醒机制未实际启用；信号数组 off-by-one 风险（索引 0 闲置但边界检查不严格）。
- 驱动与文件系统：设备注册通路中断，PCI 发现的 VirtIO 块设备未加入全局表；block_device_for_mount_source 使用硬编码设备名（[syscall.ref:7]）；copy_struct_from_user 的 unsafe 块缺少 SAFETY 注释（[syscall.ref:6]）。
- 安全性：多个模块存在 unsafe 块缺乏安全契约的问题，如 futex 中直接解引用物理地址（[ipc.ref:4]），以及 DMA 虚拟地址假设不一致（[drivers.ref:3]）。

综上，该内核是一个充满活力的原型，其创新性（手写网络、KSM、fanotify、landlock）值得肯定，但代码质量和整体完成度尚有较大提升空间。如果目标是运行真实 Linux 应用，还需要在稳健性和规范化上投入更多精力。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 内存管理模块基于 `MapArea` trait 提供虚拟地址空间管理，使用 `PageTable`（来自 polyhal）进行页表操作，支持惰性分配、写时复制、文件回退页映射以及内核同页合并（KSM）等特性。用户进程地址空间由 `UserVMSet`（包含一组 `UserMapArea`）表示，内核空间由 `KernelVMSet` 管理。与 ArceOS 基线相比，本实现引入了文件映射缺页处理路径、栈自动扩展、更完整的缺页异常处理（包括权限修正），并实验性地实现了 KSM。但存在若干早期实现痕迹，如物理页分配失败直接 panic、RISC-V 特定条件编译、全局锁竞争的 KSM 扫描等。

### 1. 核心抽象：MapArea 与 VMSet

内存管理模块的核心抽象是 trait `MapArea`（定义于 `vm_area.rs:43-65`），它规定了虚拟地址区间管理的基本接口：`range_va` 返回区间范围，`perm` 返回权限，`map_one`/`unmap_one` 映射/取消映射单页，`map`/`unmap` 批量操作。该 trait 有两个实现：`UserMapArea`（`vm_area.rs:129-145`）和 `KernelMapArea`（`vm_area.rs:170-190`）。`UserMapArea` 包含丰富的元信息：类型（`UserMapAreaType` 枚举，区分 ELF/Stack/Heap/Mmap 等）、惰性分配标志 (`lazy_flag`)、写时复制标志 (`cow_flag`)、向下增长标志 (`growdown_flag`)、文件绑定及偏移、映射类型（私有/共享）等。内核区域则使用 `KernelMapArea`，其 `area_type` 区分 Text/Rodata/Data/Bss/PhysMem/MemMappedReg/KernelStack，映射时根据类型选择线性映射（`identical_map`）或独立映射（`frame_map`）<sup>[1](#mod-mm-ref-1)</sup>。

地址空间由泛型结构 `VMSet<A: MapArea>`（`vm_set.rs:290-295`）管理，它包含一个 `PageTable` 和一个 `Vec<A>`。用户态使用 `UserVMSet`（`vm_set.rs:338-341`），内核态使用 `KernelVMSet`（通过 `KERNEL_VMSET` 全局实例 `vm_set.rs:37-40`）。`VMSpace` trait（`vm_set.rs:262-280`）定义了页表访问、新建、激活、翻译等高层操作。这种分离使得用户和内核页表可以独立切换，同时共享相同的 `MapArea` 接口。

### 2. 关键设计取舍：惰性分配、COW 与文件回退页

**惰性分配**：`UserMapArea` 的 `lazy_flag` 控制是否在 `map` 时立即分配物理页。若为 true，则 `map` 仅记录区间，实际物理页在缺页时分配（`handle_unalloc_page_fault`）。实现中根据区域类型区分处理：Heap/Stack/Elf/TrapContext 等直接 `frame_alloc` 分配新页；Mmap/Shm 类型若有文件则从文件缓存读取（`file.get_cache_frame`），否则分配匿名页。对于 `MAP_PRIVATE` 文件映射，还会复制文件内容到私有帧并清零剩余部分<sup>[3](#mod-mm-ref-3)</sup>。这种设计减少了内存浪费，但首次访问增加缺页开销。与立即分配相比，适用于稀疏映射场景。

**写时复制（COW）**：通过 `cow_flag` 和 `COW` trait（`vm_area.rs:200-215`）实现。`cow_mapping_flags` 函数构造只读临时 PTE，写缺页时由 `handle_cow_page_fault` 处理：检查物理帧引用计数 (`Arc::strong_count`)，若为 1 则直接恢复写权限，否则分配新帧并复制内容<sup>[2](#mod-mm-ref-2)</sup>。这种做法避免了 `fork` 后大量复制，但要求帧必须用 `Arc` 包裹以共享引用。值得注意的是，在处理缺页时若发现 PTE 已存在但权限不匹配（例如 `mprotect` 后未更新 PTE），代码会主动修正 PTE 权限并刷 TLB，避免陷入缺页死循环<sup>[3](#mod-mm-ref-3)</sup>。

**文件回退页**：对于 `Mmap` 类型的文件映射，缺页处理由 `mod.rs` 中的 `file_backed_fault_snapshot` 和 `install_file_backed_fault_page` 函数协同完成。`file_backed_fault_snapshot` 在进程锁外快照所需的文件信息（文件指针、偏移、页号、映射类型），然后实际分配帧并调用 `install_file_backed_fault_page` 在锁内再次验证并映射<sup>[4](#mod-mm-ref-4)</sup>。这种“快照-验证”模式避免在持锁期间执行文件 I/O，减少了锁持有时间。验证步骤检查映射是否一致（文件指针、偏移、映射类型），防止竞态条件。

### 3. 跨模块协同：缺页异常处理的完整链路

用户态缺页异常的处理链路涉及 trap handler、`exception` 模块和 `VMSet` 的协作。trap handler 捕获异常后调用 `exception::handle_page_fault`（假设在 `exception.rs`），该函数最终调用 `UserVMSet` 上的 `SetPageFaultException` 实现。`handle_store_page_fault_set` 首先通过 `find_area` 定位包含故障地址的区域，若未找到则尝试自动扩展栈（`try_expand_stack`），否则返回 `None` 导致 SIGSEGV。找到区域后根据 `access_check` 的结果（`ExceptionType`）分派：COW 缺页调用 `handle_cow_page_fault`，其他缺页（读写执行）调用 `handle_unalloc_page_fault`<sup>[3](#mod-mm-ref-3)</sup>。后者在分配物理页后，通过 `page_table.map_page` 建立映射并刷 TLB。

内核地址空间管理通过 `KERNEL_VMSET` 全局实例（`vm_set.rs:37-40`）并以自旋锁保护。内核启动时通过 `for_each_physical_memory_region`（`vm_set.rs:103-121`）探测物理内存，依赖 FDT 或预定义内存区域<sup>[8](#mod-mm-ref-8)</sup>。内核区域的映射在 `KernelMapArea` 中根据 `area_type` 选择线性映射（Identical）或帧映射（KernelStack）。

KSM 支持位于独立的 `ksm.rs` 模块，通过全局状态机 `KSM_STATE` 管理扫描和合并。`mark_range` 函数（`ksm.rs:250-274`）标记进程的匿名可合并区域，由内核线程定期扫描相同内容页并合并为写时复制页<sup>[5](#mod-mm-ref-5)</sup>。合并操作通过 `unmerge_area_pages` 和 `handle_ksm_write_fault` 处理写时复制。该实现目前是实验性的，`tick` 和 `run_until_quiescent` 提供驱动接口，但未与内核线程调度集成。

### 4. 边角细节与不足

**风险 1：`map_one` 中帧分配失败直接 panic**：`UserMapArea::map_one` 在 `frame_alloc` 失败时调用 `panic!`（`vm_area.rs:178-182`）<sup>[7](#mod-mm-ref-7)</sup>。这可以通过用户态 `mmap` 耗尽内存触发，导致内核崩溃而非返回 `ENOMEM`。正确的做法应返回错误码或退出当前进程。

**风险 2：RISC-V 条件编译的覆盖缺失**：多处使用 `#[cfg(target_arch = "riscv64")]`（如信号跳板代码、物理内存区域扫描），未显式覆盖 x86_64 或其他架构时的默认分支。若未来添加新架构，这些 cfg 可能导致未定义行为。

**风险 3：KSM 全局锁竞争**：`KSM_STATE` 使用 `Mutex` 保护，扫描过程中长时间持有锁（`scan_once` 遍历所有进程），可能阻塞其他系统调用。`CandidateGroup` 的分块设计（每块 256 项）试图减少锁粒度，但扫描期间锁仍被持有。

**风险 4：解释器加载使用固定静态缓冲区**：`INTERP_SCRATCH` 大小为 4MB（`vm_set.rs:147-149`），若动态链接器超过此大小则加载失败。此外，`SleepLock` 导致多进程串行加载解释器，影响启动性能。

**风险 5：未实现的 `KernelVMSet` 定义**：`vm_set.rs:39` 引用了 `KernelVMSet::new()`，但在可见代码中并未找到 `KernelVMSet` 结构体定义，可能位于未展示的代码中或被遗漏。若未定义将导致编译错误。

**风险 6：缺页处理中权限修正逻辑可能引入不一致**：`handle_unalloc_page_fault` 在发现已有 PTE 时尝试修正权限（`vm_set.rs:399-422`），但未验证 `area` 权限是否确实已变更（例如 `mprotect` 缩小权限）。修正后仍可能违反新的权限约束，导致安全漏洞。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块实现了基于多线程的进程模型，核心抽象为 `ProcessControlBlock` 与 `TaskControlBlock`，通过 `process::ProcessControlBlockInner` 管理进程资源（文件描述符、信号、内存空间）与线程组，通过 `task::TaskControlBlock` 管理单个线程的执行状态、内核栈和调度上下文。与同类系统（如 rCore-Tutorial）相比，本实现在信号处理、Landlock 安全域、`clone3`/`execve` 语义等方面做了大量扩展，并引入了细粒度的内存回收统计、`defer_drop` 机制以降低锁竞争。

### 1. 进程与线程的双层抽象

模块以 **`ProcessControlBlock`**（PCB）和 **`TaskControlBlock`**（TCB）分别建模进程与线程。PCB 通过 `SpinNoIrqLock<ProcessControlBlockInner>` 保护可变状态，`inner` 包含进程级资源：文件描述符表 (`fd_table`)、子进程列表、信号处理器、`LandlockDomain`、资源限制等。TCB 则持有对 `process` 的 `Weak` 引用，以及调度状态、内核栈、`TaskUserRes`（用户栈和 Trap 上下文）。这种分离使得多线程进程共享同一个 PCB，而每个线程有独立的 TCB，符合 Linux 的线程组模型。

`ProcessControlBlock::new` 创建进程时会调用 `TaskControlBlock::new` 生成主线程，将其加入 `inner.tasks` 并调用 `add_task` 加入调度器。`execve` 仅支持单线程进程（`assert_eq!(thread_count(), 1)`），替换整个用户地址空间并重置信号处理器。`clone` 家族（`clone3` 由 syscall 模块转发）则通过 `TaskControlBlock::new_worker` 和 `ProcessControlBlock::add_task` 创建新线程或进程。

### 2. 调度器与任务切换

调度器的核心入口在 `processor::run_tasks` 和 `schedule`。`suspend_current_and_run_next` 和 `block_current_and_run_next` 是典型的主动/阻塞切换路径。`block_current_and_run_next` 中加入了关键的竞态处理：检查 `zombie_flag`（防止 SIGKILL 后永远阻塞）和 `pending_wakeup` 标志（解决 lost wakeup 问题），这是对经典 `schedule` 模式的改进。

模块使用 `task_entry` 函数作为新线程的入口，该函数会循环调用 `run_user_task` 并在每次返回后检查进程是否变为 zombie，从而支持进程退出时自动清理。`finish_current_zombie_task` 处理孤儿进程的回收。

定时器队列 `TIMER_QUEUE` 保存在 `mod.rs` 中，用于唤醒定时阻塞的任务。`check_timers` 在每次定时器中断时被调用，它先收集过期任务再统一唤醒，避免锁顺序死锁。

### 3. 进程退出与回收的精细处理

`exit_current_and_run_next` 函数执行线程/进程退出。当主线程 (`tid == 0`) 退出时，整个进程变为 zombie，子进程被收养到 `INITPROC`。非主线程退出时，会处理 `clear_child_tid` 和 robust list，并通过 `futex_wake_one` 唤醒等待线程。模块引入了 `defer_drop_exited_task` 机制：将已退出但仍有引用保留的 TCB 放入 `DEFERRED_EXITED_TASKS` 列表，在后续调度循环中安全释放，避免在持有锁时执行复杂 drop 操作。

`task_retention_stats` 提供了全面的内存保留统计，用于 OOM 诊断，显示进程数、僵尸数、子进程引用数、定时器队列长度等。`print_oom_snapshot` 在 `id.rs` 中集成了帧分配器、堆、内核栈、IPC 等多个子系统的统计，是调试资源泄漏的利器。

### 4. ID 分配与资源回收

`id.rs` 实现了 `PidHandle`、`KernelStack`、`TaskUserRes` 等资源的 RAII 包装。`RecycleAllocator` 提供 ID 回收，确保分配号尽可能紧凑。内核栈通过 `KERNEL_VMSET` 线性映射，并在析构时移除映射。`TaskUserRes` 负责为线程分配用户栈和 Trap 上下文页，并在线程退出时释放。`PgidHandle` 用于进程组 ID。

`pid_alloc` 和 `kstack_alloc` 都带有统计计数器，帮助识别泄漏。`print_oom_snapshot_with_kstack_stats` 在分配失败时输出详细的上下文，指导调试。

### 5. 跨模块协同

- **与 mm 模块**：`ProcessControlBlockInner` 持有 `UserVMSet`，`execve` 时替换整个地址空间并释放旧映射。`TaskUserRes::alloc_user_res` 在进程的 VMSet 中插入用户栈和 Trap 上下文区域。
- **与 syscall 模块**：`exit_group`、`wait4`、`clone3` 等系统调用直接操作 PCB/TCB。信号处理中，`handle_default_action` 被 syscall 模块的信号投递路径调用。
- **与 fs 模块**：进程退出时 `close_all_files_on_exit` 关闭文件描述符，并协同 socket 管理器。
- **与 signal 模块**：PCB 中维护了完整的信号处理状态，`execve` 重置所有处理器，`sig_context_stack` 支持信号嵌套。
- **与 futex 模块**：线程退出时清理 futex 等待条目，并处理 robust list。

### 6. 边角细节与不足

1. **`#[cfg]` 条件编译**：`mod.rs` 中 `#[cfg(target_arch = "riscv64")]` 和 `#[cfg(target_arch = "loongarch64")]` 用于获取 `get_tp`，但未覆盖 x86_64 等架构，可能导致编译失败 <sup>[1](#mod-task-ref-1)</sup>。
2. **`assert!` 风险**：`ProcessControlBlockInner::alloc_fd` 中 `assert!(!self.recycled.iter().any(...))` <sup>[2](#mod-task-ref-2)</sup> 可在重复 dealloc 时 panic，但 dealloc 一般只由系统调用内部触发，用户态难以直接触发。
3. **`unsafe` 块缺乏 SAFETY 注释**：`task_entry` 中 `unsafe { task.as_mut().unwrap() }` <sup>[3](#mod-task-ref-3)</sup> 直接假设指针有效，但未解释为什么这是安全的（task 来自 current_task，其生命周期受调度保护）。
4. **硬编码常量**：`rlimit_nofile` 默认 1024 已参数化，但 `exit_signal` 硬编码为 17 (SIGCHLD) <sup>[4](#mod-task-ref-4)</sup>，可能不适合某些进程（如 `clone` 时设置为其他值）。
5. **`SpinNoIrqLock` 死锁风险**：`exit_current_and_run_next` 中先获取 `task.inner`，再获取 `process.inner`，而其他路径（如 sys_clone）可能先获取 process.inner 再获取 task.inner，出现反向锁序。代码通过提前释放 `task_inner` 缓解，但未完全消除死锁可能。
6. **TODO 注释**：代码中有多处 `//` 注释残留，如被注释的  `KERNEL_MEMORY_SPACE` <sup>[5](#mod-task-ref-5)</sup>，表明重构尚未完成。
7. **错误处理路径**：`block_current_and_run_next` 中若 `zombie_flag` 为 true，直接返回而不 schedule，但此时若任务不在运行状态，可能导致调度器状态不一致。

总的来说，模块在 Linux 兼容性方面做了大量工作，但部分边界情况的鲁棒性仍有提升空间。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制核心抽象为 `SignalHandlers` 固定数组，封装在进程控制块中，支持 POSIX 标准信号（1-64）及实时信号。与 rCore-Tutorial 族基线相比，本实现采用 64 元素 `[SigAction; 64]` 而非动态数据结构，并引入 `0xDEAD` 标志位禁止 SIGKILL/SIGSTOP 修改，同时调用 `_set_sum_bit` 与 trap 模块协同。但存在 off-by-one 越界、实时信号 64 静默截断、unsafe 缺安全注释等早期实现痕迹。

### 1. 核心抽象与 POSIX 接口映射

信号模块的核心抽象是 `SignalHandlers` 结构体，它包含一个长度为 64 的 `[SigAction; 64]` 数组，索引 `i` 对应信号编号 `i`（索引 0 闲置）<sup>[1](#mod-signal-ref-1)</sup>。每个 `SigAction` 包含 `sa_handler`（`SigHandler` 枚举）、`sa_mask`（`SignalSet`）、`sa_flags` 和 `sa_restorer`。`SigHandler` 通过指针编码区分 `SIG_DFL`（0）、`SIG_IGN`（1）和自定义函数（非零非一），编码方式与 musl 用户态约定一致 <sup>[2](#mod-signal-ref-2)</sup>。`SignalSet` 使用 `u64` 位掩码，支持 64 个信号，`Signal::from_i32` 限定范围为 1..=64，符合 POSIX 实时信号范围 <sup>[3](#mod-signal-ref-3)</sup>。

模块提供了 POSIX 所需的全部信号常量（`SIGHUP`…`SIGSYS`），并通过 `default_action` 方法映射默认动作（Terminate/Ignore/Stop/Core/Continue）<sup>[4](#mod-signal-ref-4)</sup>。`SignalHandlers::new` 初始化时强制将 SIGKILL（9）和 SIGSTOP（19）的 `sa_flags` 置为 `0xDEAD` 作为不可变标记，后续 `set` 方法检查此标记拒绝修改，模拟内核不可捕获性 <sup>[5](#mod-signal-ref-5)</sup>。

### 2. 关键设计取舍

**固定数组 vs 动态结构**：`SignalHandlers` 使用 `[SigAction; 64]` 而非 `HashMap` 或稀疏向量。这消除了分配开销和查找延迟，但造成了内存浪费（每个进程固定 64×24≈1.5KB）。更严重的是，信号编号从 1 开始，数组索引与编号一致导致索引 0 永远闲置，且信号 64（SIGRTMAX）的索引为 64，超出了数组长度（0..63），`get` 方法通过 `idx < self.actions.len()` 检查会返回默认动作，使信号 64 无法被设置或捕获 <sup>[6](#mod-signal-ref-6)</sup>。这是一个明显的 off-by-one 设计缺陷。

**0xDEAD 标志位不可变机制**：设置 `sa_flags = 0xDEAD` 替代专门的位域或 bool，利用 `SigAction` 结构体中的 `sa_flags` 字段传递元信息。此方案简洁但脆弱：如果用户态通过 `rt_sigaction` 传入 `sa_flags` 恰好为 `0xDEAD` 的信号，`set` 会错误拒绝 <sup>[7](#mod-signal-ref-7)</sup>。此外，`can_catch` 方法已硬编码排除 9 和 19，二次检查冗余却未移除。

**指针编码与 unsafe**：`SigHandler::from_ptr` 使用 `core::mem::transmute` 将 `*const c_void` 转换为函数指针，且 `as_ptr` 返回 `*const c_void`。这种基于地址值 0/1 的判别方式依赖 musl 的 `SIG_DFL`/`SIG_IGN` 定义，假设用户态传入的地址不会低到被误判。若传入地址 2，会被当作自定义函数但 `transmute` 可能产生无效指针，调用时将 UB <sup>[8](#mod-signal-ref-8)</sup>。该函数缺少 `// SAFETY` 注释，是技术债。

### 3. 跨模块协同

信号模块与 trap 模块通过 `crate::trap::_set_sum_bit` 建立协同 <sup>[9](#mod-signal-ref-9)</sup>。`_set_sum_bit` 在 `SignalHandlers::set` 中被调用，意图可能是设置某个状态位（如“sum”位）以允许用户态访问某寄存器或启用快速系统调用。但当前调用位置紧跟在权限检查之后，未使用其结果，若有错误仅被忽略。其命名和语义不透明，推测与 rCore-N 的 `set_sum` 功能相关，用于控制页表属性（如 `SUM` 位允许 supervisor 访问 user 页面）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了管道（Pipe）、futex 和 System V 共享内存三种 IPC 原语。核心抽象包括基于页粒度环形缓冲区的 Pipe 文件描述符、全局等待队列驱动的 futex 表（FutexKey 区分私有/共享）、以及基于固定物理页面的共享内存段管理。与同家族基线（如 rCore-Tutorial）相比，futex 支持了 REQUEUE/CMP_REQUEUE 和超时参数，共享内存实现了完整的 shmctl 命令集（IPC_STAT/IPC_INFO/SHM_INFO/IPC_RMID），pipe 通过 procfs 暴露容量但实际写操作为空存根。信号集成（SIGPIPE）和进程退出清理机制与任务调度模块紧密耦合，但也暴露出多处未完成实现和潜在安全风险。

### 1. 核心抽象与外部依赖

进程间通信模块由三个独立但共享任务调度基础设施的子系统组成。

**管道（Pipe）** 基于 `PipeRingBuffer` 环形缓冲区，使用 `SpinLock<PipeRingBuffer>` 保护并发访问 <sup>[1](#mod-ipc-ref-1)</sup>。`Pipe` 结构体通过 `read_user_slice` 和 `write_user_slice` 方法实现阻塞/非阻塞读写，并集成 SIGPIPE 信号。缓冲区采用按需分配的页（`PipePage = Box<[u8; PAGE_SIZE]>`），允许动态扩容。

**futex** 以全局 `BTreeMap<FutexKey, VecDeque<FutexWaiter>>` 作为等待队列 <sup>[2](#mod-ipc-ref-2)</sup>，`FutexKey` 分为进程私有（`(pid, uaddr)`）和跨进程共享（物理地址）两类。`FutexWaiter` 记录任务、位掩码和超时截止时间。超时通过 `check_futex_timeouts` 轮询实现，但该函数被 `#[allow(unused)]` 标记。

**System V 共享内存** 由全局 `SHM_STATE`（`SpinNoIrqLock<ShmState>`）管理，每个 `ShmSegment` 持有分配的物理页面 `Vec<Arc<FrameTracker>>`、引用计数 `shm_nattch` 和销毁标志。`sys_shmget`/`sys_shmat`/`sys_shmdt`/`sys_shmctl` 实现了完整的 SysV 语义。

三个子系统均依赖 `TaskControlBlock` 的阻塞唤醒机制（`block_current_and_run_next`/`wakeup_task`）、`PageTable` 地址翻译、`UserBuffer` 用户内存读写以及信号模块的 `deliver_signal`。

### 2. 关键设计取舍

**管道页粒度缓冲区 vs 传统连续缓冲区**：页分配避免了预先分配大块连续物理内存，允许最大容量达到 1MB（`PIPE_MAX_SIZE`），但在多页读写时需要分散-收集拷贝。`resize` 方法通过新建页面并拷贝数据实现扩容，但该功能仅能通过 procfs 写触发，而 procfs 的 `PipeMaxSizeFile::write` 为空操作，直接返回 `Ok(_buf.len())` 不进行任何校验 <sup>[3](#mod-ipc-ref-3)</sup>，导致动态扩容实际不可用。

**futex key 设计**：私有 key 使用 `(pid, uaddr)` 避免了同一进程不同地址空间的冲突；共享 key 使用物理地址以保证跨进程唯一性。但 `read_user_u32_mapped` 在持有 `FUTEX_TABLE` 锁时直接解引用物理地址指针 <sup>[4](#mod-ipc-ref-4)</sup>，依赖 polyhal 的物理内存直接映射假设，在非直接映射架构（如 x86 SMAP）上会触发故障，且缺少 `// SAFETY` 注释。

**超时唤醒策略**：futex 超时通过 `check_futex_timeouts` 函数遍历全局表检查截止时间，该函数仅被 `#[allow(unused)]` 修饰 <sup>[5](#mod-ipc-ref-5)</sup>，暗示未接入时钟中断处理链，导致带超时的 `FUTEX_WAIT` 实际永远不返回 `ETIMEDOUT`，而是阻塞直到被信号或 `FUTEX_WAKE` 唤醒。这种粗粒度轮询也带来了周期性的全局表扫描开销。

**共享内存引用计数与生命周期**：`shmctl IPC_RMID` 采用标记删除（`destroyed = true`），当所有进程 detach 后（`shm_nattch == 0`）才释放页面。这要求进程退出时准确调用 `release_shm_attaches` 递减计数。`shmat` 将段页面直接插入进程的 `vm_set` 并手动映射页表 <sup>[6](#mod-ipc-ref-6)</sup>，其中 `lazy_flag = true` 但 `data_frames` 预填充，这种做法与缺页处理机制协同，但若缺页路径未正确处理 `shmid` 关联，会导致映射不一致。

**信号集成**：`Pipe::write_user_slice` 在检测到读端全部关闭时，先释放环形缓冲区锁，再调用 `deliver_signal` 发送 `SIGPIPE` <sup>[7](#mod-ipc-ref-7)</sup>，避免在持有锁的情况下调用信号发送函数，防止锁反转。但信号发送仍可能失败（如进程已退出），当前仅返回 `EPIPE` 错误。

### 3. 跨模块协同

**管道与任务调度**：`Pipe::read_user_slice` 和 `write_user_slice` 在数据不足/满时释放缓冲区锁，通过 `PipeRingBuffer::register_read_waiter` 将当前任务加入等待队列，然后 `block_current_and_run_next` 让出 CPU。唤醒时由 `wake_read_waiters`/`wake_write_waiters` 调用 `wakeup_task`。这种模式与 futex 的等待机制同源，但管道检查 `interrupted_after_block` 以响应进程退出和信号 <sup>[8](#mod-ipc-ref-8)</sup>。

**futex 与进程/线程退出**：`remove_task_from_futex_table` 遍历全局 futex 表，移除指定任务的所有等待条目 <sup>[9](#mod-ipc-ref-9)</sup>，防止残留引用导致内存泄漏。该函数在任务退出路径（`exit_current_and_run_next`）中被调用，与 `futex_wake_one`（用于 `clear_child_tid`）协同。但遍历复杂度为 O(N*K)，高频创建/销毁线程时可能成为瓶颈。

**共享内存与地址空间**：`sys_shmat` 将段物理页直接插入进程的 `UserMapArea` 列表，并调用 `page_table.map_page` 建立映射 <sup>[6](#mod-ipc-ref-6)</sup>。`sys_shmdt` 通过 `vm_set.areas` 查找匹配的 Shm 区域，解映射并移除。`release_shm_attaches` 在进程执行 `execve` 或 `exit` 时被调用，确保所有 Shm 区域 detach，并递减段引用计数 <sup>[10](#mod-ipc-ref-10)</sup>。这要求 `UserMapAreaType::Shm` 类型和 `shmid` 字段在整个地址空间生命周期内正确维护。

**pipe 与 procfs**：`PipeMaxSizeFile` 作为 procfs 文件暴露当前 pipe 最大容量，读取返回固定字符串 `"1048576\n"`，但写入直接丢弃，未与 `PipeRingBuffer::resize` 连接 <sup>[3](#mod-ipc-ref-3)</sup>。这表明该接口处于早期存根状态。

### 4. 边角细节与不足

**动态扩容不可用**：`PipeMaxSizeFile::write` 空操作意味着用户无法通过 `/proc/sys/fs/pipe-max-size` 调整容量，`resize` 方法虽已实现但永不触发。<sup>[3](#mod-ipc-ref-3)</sup>

**超时轮询未接入**：`check_futex_timeouts` 被 `#[allow(unused)]` 标记 <sup>[5](#mod-ipc-ref-5)</sup>，时钟中断路径未调用此函数，带超时的 `FUTEX_WAIT` 将无限阻塞（除非信号或 wake 唤醒），与 Linux 语义严重不符。

**用户指针未校验长度**：`sys_shmctl` 中 `IPC_INFO`/`SHM_INFO`/`IPC_STAT` 等命令直接从用户提供的裸指针构造 `&mut [u8]` 写入，没有验证缓冲区可写性（未经过 `translated_byte_buffer_for_write`）<sup>[11](#mod-ipc-ref-11)</sup>，恶意用户程序可传入任意地址导致内核内存破坏或信息泄露。

**锁顺序未文档化**：`shmat` 先获取进程 `inner` 锁再获取 `SHM_STATE` 锁 <sup>[12](#mod-ipc-ref-12)</sup>，而 `release_shm_attaches` 从进程退出路径调用，若退出路径持有 `SHM_STATE` 锁并等待 `inner` 锁则死锁。当前虽未触发，但缺乏锁顺序注解。

**Drop 中的锁获取**：`Pipe::drop` 在回收时获取 `ring_buffer` 锁 <sup>[13](#mod-ipc-ref-13)</sup>，若另一个线程正持锁且恰好被阻塞在 `block_current_and_run_next` 中，Drop 将自旋等待，可能导致进程退出延迟或死锁（尤其当 Drop 发生在信号处理或异常路径中）。

**物理地址直接解引用缺少安全契约**：`read_user_u32_mapped` 的 `unsafe` 块依赖 polyhal 的物理内存映射假设，未在 `// SAFETY` 中说明前提条件 <sup>[4](#mod-ipc-ref-4)</sup>，若移植到非直接映射平台将导致 UB。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 该模块实现了一个从 VirtIO 网卡驱动到 TCP/IP 协议栈的完整网络子系统，核心抽象是 `NetDevice` trait 与 `Skb` 包缓冲区。它采用手写协议栈（而非 smoltcp 等现成 crate），包含 ARP、IP、ICMP、UDP、TCP 处理，并通过 `KERNEL_TCP_CONNS` 在内核内建了一个 TCP 服务（端口 8080）。与同家族基线（ArceOS/smoltcp）相比，该项目放弃通用性和硬件 offload 支持，换取对内核 TCP 服务的完全控制。

### 1. 核心抽象与外部依赖

网络模块的核心抽象是 `Skb`（Socket Buffer）和 `NetDevice` trait。`Skb` 在所有协议层间传递报文，支持 `push`/`pull`/`reserve_head` 操作，其内部通过 `Vec<u8>` 管理数据。`NetDevice` trait 定义了 `hard_start_xmit`、`set_rx_handler`、`poll_rx`、`mac_addr`、`ip_addr` 等接口，`VirtIONetDevice` 是唯一实现者，通过 PCI 探测 VirtIO 网卡并进行 MMIO 寄存器配置。

模块外部依赖极少：仅使用 `alloc`、`spin::Mutex`、`log` 和 `polyhal`（提供 `VIRT_ADDR_START` 虚实地址转换）。没有使用 `smoltcp`、`embassy-net` 等社区 crate，而是手写了整个 TCP/IP 协议栈。这一选择换取了极小代码量约 2000 行，但放弃了硬件校验和 offload、GSO、多队列等能力。

### 2. 关键设计取舍

**手写 TCP 协议栈 vs smoltcp**

`tcp.rs` 手工定义了 `TcpHeader`（ packed 结构）、校验和计算 `tcp_checksum`、序列号推进 `tcp_seq_advance`，完整实现了有限状态机（SYN_RECV→ESTABLISHED→LAST_ACK→CLOSE）。`dispatch_kernel_tcp_service` 为内核 TCP 服务（硬编码端口 8080 <sup>[1](#mod-net-ref-1)</sup>）维护连接表 `KERNEL_TCP_CONNS`，而用户 socket 连接通过 `try_dispatch` 委托给 `crate::socket::tcp::dispatch_tcp_segment` <sup>[5](#mod-net-ref-5)</sup>，形成双路径分发。这一设计使得内核可以抢占性地回应特定端口的 TCP 请求，但两份状态机实现之间缺乏共享代码。

**VirtIO 探测策略**

`probe.rs` 首先尝试从 FDT（平坦设备树）提取 ECAM 基址，该代码只对 `loongarch64` 生效 <sup>[3](#mod-net-ref-3)</sup>；其他架构（riscv、x86_64）使用硬编码的 `0x30000000`，假设 QEMU virt 平台。MMIO BAR 在 `ensure_mmio_bars_assigned` 中动态分配，从 `0x40100000` 开始线性增长，未考虑预分配的 BAR 冲突。

**RX 采用轮询而非中断**

`VirtIONetDevice` 的 `poll_rx_once` 方法检查 used ring 的 `idx` 变化来接收报文，并未注册中断处理。`start_rx_thread` 包含一个 TODO <sup>[6](#mod-net-ref-6)</sup>，原本期望用异步任务循环调用 `poll_rx_once`，但当前仅提供轮询接口给上层驱动使用。

### 3. 跨模块协同

**IP 分发链**

`ip_rcv` 在验证校验和后，根据 `protocol` 字段将报文派发给 `tcp_rcv`、`udp_rcv` 或 `icmp_rcv`，同时调用 `deliver_raw_packet` 将副本投递到 raw socket 层 <sup>[4](#mod-net-ref-4)</sup>。`ip_queue_xmit` 先查路由表，再通过 `neighbour_output` 进行 ARP/邻居解析，最终调用 `NetDevice::hard_start_xmit`。

**TCP 发送/接收闭环**

`tcp_rcv` 解析 TCP 头后，根据 SYN、ACK、FIN、RST 标志进入不同路径：SYN 未 ACK 的报文视为新建连接请求，递交给 `try_dispatch_or_rst`；已建立连接的数据或 FIN/ACK 同样进入 `try_dispatch_or_rst`，该函数优先尝试内核 TCP 服务 `dispatch_kernel_tcp_service`，失败再尝试用户 socket 层 <sup>[5](#mod-net-ref-5)</sup>，若无匹配则发送 RST。`tcp_send_segment` 构造 TCP 报文头，填入校验和，然后调用 `ip_queue_xmit` 递交给 IP 层。

**ARP 与邻居子系统**

`arp.rs` 维护全局 `ARP_CACHE`（`BTreeMap<u32, ArpEntry>`），`arp_lookup` 用于邻居解析，`arp_request` 发送广播请求。约束：仅支持以太网硬件类型和 IPv4 协议。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块基于 BlockDevice trait 抽象块设备接口，通过 VirtIOBlk（virtio_drivers crate）实现具体驱动。核心是为 riscv64（MMIO）和 loongarch64（PCI）提供统一的 VirtIO 块设备支持。与 rCore-Tutorial 基线相比，本仓库为 loongarch64 实现了完整的 PCI 枚举和 BAR 分配逻辑，使用全局 bounce buffer 中转 DMA，并显式管理锁层次以避免死锁，但存在设备注册通路中断、FDT 地址硬编码、bounce buffer 单例限制并发等不足。

### 1. 核心抽象与外部依赖

驱动框架的核心抽象是 `BlockDevice` trait（定义于 `crate::devices`，未在已读文件中展示），它提供了 `size()`、`block_size()`、`read_block()`、`write_block()` 四个方法，构成一个最小块设备接口。具体实现是 `VirtIOBlock` 结构体，它通过条件编译为不同架构选择不同的底层传输层：riscv64 使用 `MmioTransport`，loongarch64 使用 `PciTransport` <sup>[1](#mod-drivers-ref-1)</sup>。`VirtIOBlock` 内部包装了 `virtio_drivers` crate 的 `VirtIOBlk`，并通过自定义的 `VirtioHal` 实现了 `virtio_drivers::Hal` trait，负责 DMA 内存分配/释放、物理地址到虚拟地址的转换以及设备内存共享 <sup>[3](#mod-drivers-ref-3)</sup><sup>[9](#mod-drivers-ref-9)</sup>。

与同家族基线（如 rCore Tutorial Chapter 7 的 virtio_blk 驱动）相比，本仓库的最大差异在于：（1）为 loongarch64 实现了完整的 PCI 总线枚举和 BAR 分配逻辑（`pci.rs`），不依赖固件或 BootLoader 完成配置；（2）使用全局预分配的 `BlkIoBounce` 结构体作为所有 I/O 的 DMA 缓冲区，而非逐请求分配或直接映射用户缓冲区；（3）通过 `SleepLock` 和 `SpinLock` 组成的两层锁保护共享状态，并在注释中显式声明锁顺序以避免死锁 <sup>[7](#mod-drivers-ref-7)</sup>。

### 2. 关键设计取舍

**全局 Bounce Buffer vs 直接用户缓冲区映射**

所有块读写操作都通过一个全局的 `BlkIoBounce` 单例进行中转，其包含三个固定大小（4KB，由 `BLK_BOUNCE_SIZE = PAGE_SIZE` 定义）且 4KB 对齐的缓冲区：`DmaReq`、`DmaResp`、`DmaBuffer` <sup>[2](#mod-drivers-ref-2)</sup>。读写时将数据从用户缓冲区拷贝到 `DmaBuffer`，再由 VirtIO 引擎 DMA 到设备。这种设计舍弃了零拷贝的可能性，引入了额外 memcpy 开销，但换来了 DMA 内存管理的简化：无需保证用户缓冲区物理连续、无需处理用户缓冲区的页表映射变化，且避免了反复分配/回收小片物理页。缺点是全局单例 `BLK_IO_BOUNCE` 配合 `SleepLock` 使所有 I/O 请求串行化，无法利用 VirtIO 的多队列并行能力。

**自旋等待完成 vs 中断驱动**

提交读写请求后，`read_block` 和 `write_block` 使用 `while blk.peek_used() != Some(token) { core::hint::spin_loop(); }` 自旋等待设备完成 <sup>[4](#mod-drivers-ref-4)</sup>。这是典型的轮询模式，适合高频短 I/O 或中断延迟不可预测的系统。但此实现隐含一个风险：如果 `SleepLock` 内部在持锁期间不响应中断（当前实现确实不处理），那么自旋等待会独占 CPU 直到硬件完成，这在单核环境下阻塞了所有其他任务。更健壮的方案应利用 `VirtIOBlk` 的中断使能能力，通过 WFI 或信号量等待来释放 CPU。

**条件编译适配不同硬件架构**

`#[cfg(target_arch)]` 被广泛用于隔离平台差异：riscv64 使用 MMIO 方式访问 VirtIO 设备（固定地址 `0x10001000 + VIRT_ADDR_START`），而 loongarch64 则通过设备树解析 PCI 节点并使用 ECAM 配置空间枚举 VirtIO 设备 <sup>[1](#mod-drivers-ref-1)</sup>。`VirtioHal::share` 方法也做了架构分叉：riscv64 版直接通过页表翻译得到物理地址；loongarch64 版则先判断虚拟地址的高 60 位是否匹配 `VIRT_ADDR_START`——若匹配则认为是恒等映射区，直接减去偏移，否则走页表翻译 <sup>[9](#mod-drivers-ref-9)</sup>。这种条件分支反映了 loongarch64 内核线性映射段和动态映射段共存的现实，但缺少对其他架构（如 x86_64、aarch64）的默认分支，若将来移植可能会在编译期静默失败。

### 3. 跨模块协同

**与内存管理模块的协同**

`VirtioHal::dma_alloc` 调用 `frame_alloc_contiguous(pages)` 分配物理连续页面，并将返回的 `FrameTracker` 存入全局 `QUEUE_FRAMES` 列表中 <sup>[3](#mod-drivers-ref-3)</sup>。`dma_dealloc` 则遍历该列表找到对应的 `FrameTracker` 并 `swap_remove` 后 drop 释放。源码注释1明确指出了锁获取顺序的重要性：`dma_alloc` 先获取 `QUEUE_FRAMES` 锁再进入帧分配器（帧分配器内部可能也有锁），而 `dma_dealloc` 先释放 `QUEUE_FRAMES` 锁再 drop `FrameTracker`（触发帧回收），从而避免 ABBA 死锁 <sup>[7](#mod-drivers-ref-7)</sup>。这种显式的锁层次设计是优秀但易碎的工程实践——任何未来代码改动若打乱此顺序将引入隐蔽的死锁。

`share` 方法通过 `KERNEL_VMSET.lock().token()` 获取内核页表，然后调用 `page_table.translate_va` 将用户/内核虚拟地址转换为物理地址 <sup>[9](#mod-drivers-ref-9)</sup>。这建立了驱动与页表系统的桥梁，但也引入了运行时锁竞争：KERNEL_VMSET 的 `lock()` 可能在并发路径上成为瓶颈。

**与板级支持模块的协同**

`block/mod.rs` 中通过 `use crate::board::BlockDeviceImpl` 将具体设备实现延迟到板级包 <sup>[10](#mod-drivers-ref-10)</sup>。全局 `BLOCK_DEVICE` 是一个 `lazy_static!` 初始化的 `Arc<dyn BlockDevice>`，由 `BlockDeviceImpl::new()` 创建。这种设计遵循了 Rust 常见的类型状态模式：board crate 决定使用哪个具体驱动（例如当前只有 VirtIOBlock），上层代码只通过 trait 方法调用。

### 4. 边角细节与不足

**DMA 虚拟地址假设的不一致性**：`dma_alloc` 返回的第二个参数（内核使用的虚拟地址指针）通过 `NonNull::new(pa.get_mut::<u8>()).unwrap()` 获得，注释声称“内核页表还是恒等映射”。然而 loongarch64 的 `mmio_phys_to_virt` 使用 `paddr + VIRT_ADDR_START`，表明内核使用线性偏移映射而非恒等映射。如果 polyhal 的 `PhysAddr::get_mut` 依赖特定页表映射（例如直接解引用物理地址），则在 loongarch64 上可能访问到错误的虚拟地址 <sup>[3](#mod-drivers-ref-3)</sup>。这是一个值得单独跟踪的技术债。

**FDT 地址硬编码**：loongarch64 初始化中 FDT 基址被硬编码为 `0x9000_0000_0010_0000`，而非从 BootLoader 传递的寄存器（如 `a1`）中获取 <sup>[6](#mod-drivers-ref-6)</sup>。注释掉的 `get_fdt_addr` 函数表明开发者已经意识到这个问题，但尚未切换。这导致内核只能在该特定内存布局的 QEMU 模拟环境中运行。

**设备注册通路中断**：`probe.rs` 的 `_register_block_device` 函数体被完全注释，`_virtio_device` 创建 `VirtIOBlock` 后并未更新全局 `BLOCK_DEVICE` <sup>[8](#mod-drivers-ref-8)</sup>。这意味着通过 PCI 枚举发现的 VirtIO 块设备虽然被实例化，但上层文件系统无法访问它。当前系统仅能通过 `block/mod.rs` 中预初始化的 `BLOCK_DEVICE` 工作，该设备在编译期由 `BlockDeviceImpl` 决定。

**Bounce Buffer 性能瓶颈**：全局 `BLK_IO_BOUNCE` 单例配合 `SleepLock` 使所有读写操作串行执行。若上层文件系统尝试并发 I/O（例如同时读取不同块），它们将逐一排队。此外，`BLK_BOUNCE_SIZE` 固定为 4KB，对大块 I/O（如 64KB 文件读取）会被切分为 16 次 4KB 操作，每次都需要 bounce 拷贝和 VirtIO 请求提交，效率低下。

**错误路径全部 panic**：`read_block` 和 `write_block` 中，无论是提交请求失败还是完成校验失败，都直接调用 `panic!` 打印详细错误信息后终止 <sup>[4](#mod-drivers-ref-4)</sup>。没有重试机制、没有降级策略，甚至没有返回 `Result` 给调用者。这在生产环境中意味着任何磁盘瞬时错误都会导致内核 panic。

### 5. 与族基线的对比总结

与 rCore-Tutorial 的 virtio_blk 驱动相比，本仓库的 loongarch64 支持是最大亮点：实现了 `pci.rs` 中完整的 PCI 总线枚举、BAR 分配和配置逻辑，这是 rCore-Tutorial 目前尚未覆盖的。但代价是代码复杂度显著增加，且存在上述多个早期实现特征（硬编码、注释掉的注册、panic-only 错误处理）。若与 ArceOS（另一个知名 Rust OS 项目）对比，ArceOS 使用更简洁的 `virtio-drivers` 高层 API 和更灵活的 DMA 管理策略，本仓库目前仍处于“先跑通”阶段，性能优化和错误处理尚未充分展开。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层位于 os/src/syscall，核心抽象是将每个 Linux 系统调用实现为独立的 public 函数，直接映射 Linux ABI，通过 `_set_sum_bit()` 和翻译辅助函数（translated_*）访问用户指针。与同家族基线（如 rCore-Tutorial）相比，该实现增加了 landlock 安全模块、fanotify/inotify 通知、mount 传播、LTP 过滤等 Linux 高级特性，代码量更大，但也引入了硬编码假设、遗留死代码和 unsafe 块缺乏契约等问题。

### 1. 核心抽象：独立函数与 ABI 直译

系统调用层位于 `os/src/syscall`，每个系统调用被实现为一个独立的 `pub fn`，直接接受与 Linux 系统调用相同的参数（例如 `sys_openat(dirfd: isize, path: *const u8, flags: u32, mode: u32)`）。这种设计没有引入额外的抽象层（如统一的 Syscall trait），而是让每个函数处理自己的逻辑。这样做的动机是直接兼容 Linux ABI，避免在 dispatch 时进行参数重排。代价是代码冗余：每个 syscall 都需要重复设置 SUM 位、翻译用户指针等 boilerplate。例如，在信号模块中，每个函数开头都调用 `_set_sum_bit()` <sup>[1](#mod-syscall-ref-1)</sup>，而在 process 模块中则依赖 trap 入口的统一设置，这种不一致可能隐藏 bug（若某个 syscall 通过非标准路径进入，SUM 位可能未设置）。fs 和 process 模块的许多函数也直接操作用户指针，体现了“信任用户页表”的设计哲学。

### 2. 关键设计取舍

- **直接用户指针访问**：通过设置 SUM 位，内核可以直接解引用用户指针。这简化了短路径读写，但要求每次 syscall 入口必须确保 SUM 位已设置。signal 模块中每个 syscall 开头都调用 `_set_sum_bit()` <sup>[1](#mod-syscall-ref-1)</sup>，而 `sys_execve` 和 `sys_brk` 却没有调用，却依然使用 `translated_str` 等。这可能是因为 trap 入口在进入系统调用前已设置 SUM 位，但 signal 模块为了安全再次设置。这种不一致是设计未统一的表现。

- **信号模型：进程级为主，线程级为辅**：大多数信号通过 `deliver_signal` 进程级投递，但 `sys_tkill` 支持向特定线程定向发送自定义信号 <sup>[2](#mod-syscall-ref-2)</sup>。这符合 POSIX 线程信号要求，但导致投递路径的复杂化：自定义 handler 直接添加到线程 pending，否则走进程级。此外，`finish_signaled_process` 在终止进程时需清理每个线程的资源 <sup>[5](#mod-syscall-ref-5)</sup>，增加了复杂性。

- **安全机制的可组合性**：在 `sys_execve` 中，先后进行了 landlock 检查、mount noexec 检查、fanotify 权限检查 <sup>[3](#mod-syscall-ref-3)</sup>。这些检查虽然保证了安全性，但造成多次路径遍历，性能可能受影响。

### 3. 跨模块协同

以 `sys_execve` 为例：它从用户空间读取参数（使用 `translated_str`），调用文件系统模块的 `open_file` 打开可执行文件，读取整个文件到内核缓冲区（大小受 `EXEC_IMAGE_MAX_SIZE` 限制 <sup>[4](#mod-syscall-ref-4)</sup>），然后调用进程控制块的 `execve` 方法加载 ELF。如果非 ELF，回退到 busybox 解释执行。权限检查涉及 landlock、mount flags、fanotify 等多个模块。

另一个例子是 `finish_signaled_process`，它协调 futex、定时器、共享内存、文件写回等多个模块的清理 <sup>[5](#mod-syscall-ref-5)</sup>，展示了系统调用层作为事件中心的角色。

### 4. 边角细节与不足

- **unsafe 块缺乏安全契约**：`copy_struct_from_user` 使用 `unsafe { value.assume_init() }` 但未提供 SAFETY 说明 <sup>[6](#mod-syscall-ref-6)</sup>。如果 `translated_byte_buffer` 返回的长度不足以覆盖结构体，将读取未初始化内存，构成信息泄露风险。
- **硬编码设备路径**：`block_device_for_mount_source` 直接匹配 "/dev/vda" 等字符串 <sup>[7](#mod-syscall-ref-7)</sup>，无法适应不同磁盘命名，移植性差。
- **死代码残留**：`wakeup_first_blocked_task` 被 `#[allow(dead_code)]` 标记 <sup>[8](#mod-syscall-ref-8)</sup>，表明信号唤醒机制未实际启用，可能是早期实现的遗留。
- **TLB 刷新粒度过大**：`sys_brk` 在缩小堆时调用 `TLB::flush_all()` <sup>[9](#mod-syscall-ref-9)</sup>，刷新所有 TLB 而不是仅失效被回收的页面，影响性能。
- **信号结构体 ABI 假设**：`LinuxSigInfo` 的 `_pad` 硬编码为 116 字节 <sup>[10](#mod-syscall-ref-10)</sup>，依赖于特定的 glibc/musl ABI，不同版本可能导致兼容性问题。
