## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本仓库的“内存管理”模块当前仅提供一个用于调试的内存布局打印函数 `print_memory_map`，它通过 `extern "C"` 引用链接器脚本定义的符号（如 `stext`、`etext` 等）来输出内核各个段和启动栈的物理地址范围。该函数不涉及页表、虚拟地址空间或分配器等核心内存管理抽象，而是作为初始化阶段的辅助诊断工具。与 rCore-Tutorial 系列通常包含的完整分页、堆分配器相比，此实现处于极早期或简化状态，可能将真正内存管理逻辑留待后续实现。

### 1. 核心抽象与外部依赖
内存管理模块唯一的公开接口是 `print_memory_map` 函数 <sup>[1](#mod-mm-ref-1)</sup>，它不依赖任何 Rust 内存管理 crate（如 `axmm`），而是直接使用 `extern "C"` 块声明一组链接器符号。这些符号由内核链接脚本（通常是 `linker.ld`）定义，代表各段的起始和结束地址。函数将符号地址转换为 `usize` 并打印 `[start, end)` 格式的范围 <sup>[2](#mod-mm-ref-2)</sup>。这种设计意味着模块的核心抽象不是 Rust 中的结构体或 trait，而是隐式的、由链接脚本保证的内存布局契约。这种做法的优势是零开销——编译期即可确定所有地址；劣势是缺乏类型安全和运行时验证，且无法独立于特定目标平台。

### 2. 关键设计取舍
选择通过外部符号而非 Rust 常量或函数暴露地址，是权衡了简单性与可移植性的结果。替代方案包括：使用 `include!` 或构建脚本生成地址常量，或使用 `lazy_static` 运行时计算。当前做法更接近 C 风格内核的惯例，但放弃了 Rust 提供的编译时类型检查。此外，函数未使用 `unsafe` 块，尽管将函数符号转换为裸指针并解引用（通过 `println!` 间接涉及格式化输出）在严格意义上需要 `unsafe` <sup>[3](#mod-mm-ref-3)</sup>。这可能是为了减少噪声，但也掩盖了潜在的未定义行为风险——若链接脚本错误定义符号，或符号地址未对齐，则 `println!` 的格式化可能触发异常。

### 3. 跨模块协同
`print_memory_map` 的设计意图是在内核初始化早期被调用，通常紧接在清除 BSS 段之后、启用分页之前。虽然无法直接读取调用方代码，但可以推断它可能被置于 `rust_main` 或 `entry` 函数的开头，用于验证内存布局是否符合预期。它依赖全局的 `println!` 宏，后者需要控制台或串口驱动已初始化。这种强耦合意味着如果控制台未准备工作，调用该函数将导致静默失败或恐慌。与 rCore-Tutorial 中通常将内存布局信息记录在 `MemorySet` 结构体中不同，这里直接将信息输出到终端，牺牲了复用性却简化了早期调试。

### 4. 边角细节与不足
- **缺失 `unsafe`**：`extern "C"` 块本身在 safe Rust 中允许，但将函数项转换为裸指针（`stext as *const ()`）实际上是不安全的操作。当前代码未显式标记 `unsafe`，可能在 strict Rust 规则下引发编译警告或未来的拒绝 <sup>[4](#mod-mm-ref-4)</sup>。
- **重入与并发**：函数未使用 `#[inline]` 或同步原语，若在多个 CPU 核心上同时调用（例如在 SMP 初始化中），打印可能交错。
- **硬编码依赖**：`boot_stack_lower_bound` 和 `boot_stack_top` 符号假设存在且非零；若链接脚本未导出，则编译通过但运行时地址为 0。
- **无错误路径**：`println!` 可能失败（如控制台未初始化），但当前实现直接忽略错误。
- **TODO 信号**：整个模块仅有一个调试函数，缺乏真正的内存分配或虚拟内存管理，暗示该仓库的内存管理功能尚未完整实现 <sup>[5](#mod-mm-ref-5)</sup>。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块基于批处理模型（batch）实现极简的“进程”抽象，核心数据结构是 TrapContext（保存用户态/内核态切换的 CPU 状态），系统调用仅支持 exit(93)、getpid(172)、sched_yield(124)。与 rCore-Tutorial 基线相比，本仓库未实现任务控制块（TCB）、抢占式调度或优先级调度；yield 系统调用被直接绕过，调度仅由批处理队列顺序推进，属于早期实验性质的实现。

### 1. 核心抽象：批处理执行模型与 TrapContext

本仓库的“进程与任务调度”模块并未引入传统的进程控制块（PCB）或任务结构体，而是采用极简的**批处理（batch）模型**。用户程序按照编译时确定的顺序逐个执行，前一个退出后自动加载下一个。

整个上下文切换的核心数据结构是 `TrapContext` <sup>[1](#mod-task-ref-1)</sup>，它保存了 32 个通用寄存器、sstatus、sepc、内核页表地址（kernel_satp）、内核栈指针（kernel_sp）以及 trap_handler 函数地址。该结构体以 `#[repr(C)]` 布局，确保与底层汇编的字段偏移一致。`app_init` 方法创建初始上下文时，将 sstatus 的 SPIE 位（bit 5）置 1、SPP 位清 0，表示从用户模式进入且中断使能 <sup>[2](#mod-task-ref-2)</sup>。

值得注意的是，`kernel_satp` 字段在注释中明确标注为“暂未使用”，而 `kernel_sp` 和 `trap_handler` 在初始化时也被设为 0，由调用者后续填充。这暗示了当前实现仅完成了最基本的上下文保存/恢复路径，完整的多任务抢占仍需这些字段就绪。

### 2. 关键设计取舍：极简的调度器与未完成的 yield

调度策略上，本模块选择了最简单可行的方案：**无调度器**。批处理模块 `batch` 维护一个应用队列和一个 `current_pid` 计数器。`sys_exit` 直接调用 `batch::run_next_app()` <sup>[3](#mod-task-ref-3)</sup>，切换到下一个应用；`sys_getpid` 返回 `batch::current_pid()` <sup>[4](#mod-task-ref-4)</sup>。

最激进的取舍体现在 `sys_yield` <sup>[5](#mod-task-ref-5)</sup>：
```rust
pub fn sys_yield() -> isize {
    // 当前简单实现：直接返回 0，不触发调度
    0
}
```
这完全绕过了调度的核心语义——主动放弃 CPU。虽然注释写着“当前简单实现”，但实际效果是 **sched_yield 系统调用变成了空操作**，用户程序无法通过它让出处理器。这在协同调度场景下会直接导致：若某应用进入死循环，后续所有应用都无法得到执行。

该取舍的代价是放弃了与 rCore-Tutorial 教程的兼容性。教程在类似阶段会实现基于 `task_info` 的抢占式调度或至少实现 `sys_yield` 的立即重调度。本仓库选择将 yield 降级为 no-op，推测是为了优先验证 trap 上下文切换的正确性，将调度策略推迟到后续迭代。

### 3. 跨模块协同：TrapContext 与系统调用流程

虽然代码片段有限，但系统调用与 TrapContext 的协作路径仍可勾勒：

1. **用户态发出 syscall**：如 `exit(93)` 触发 `ecall`，CPU 跳转到 `stvec` 指向的 trap handler（保存在 `TrapContext.trap_handler` 中）。
2. **Trap 保存**：汇编入口将当前寄存器压入当前任务的 `TrapContext` 中（通过 `kernel_sp` 找到结构体地址）。
3. **分发**：trap handler 根据 cause 寄存器分发到 `syscall/mod.rs`，进而调用 `process.rs` 中的对应函数。
4. **系统调用处理**：`sys_exit` 调用 `batch::run_next_app()`，后者修改 `current_pid` 并加载下一个应用的 `TrapContext`（设置 sepc 为入口地址、sp 为用户栈顶），最后执行 `sret` 返回到用户态。

当前实现中，所有进程共享一个 trap handler 和内核栈（batch 模式下同一时刻只有一个活跃应用），因此无需 `kernel_satp` 和独立的 `kernel_sp`。但一旦引入多任务并发，每个 TCB 必须持有自己的 `TrapContext`、内核栈和页表，届时这些预留字段就会变得必要。

### 4. 边角细节与不足

以下通过对代码细节的审查，列出当前实现中的 5 个具体风险点：

**① 硬编码 sstatus 掩码 <sup>[2](#mod-task-ref-2)</sup>**：`sstatus: 1 << 5` 直接写死了 SPIE 位。虽然在 RISC-V 标准中 bit 5 固定为 SPIE，但不同特权级下的 sstatus 字段布局可能因扩展（如 hypervisor）而不同。更好的做法是使用常量定义（如 `1 << SSTATUS_SPIE`）并附架构文档引用。

**② sys_yield 吞掉调度请求 <sup>[5](#mod-task-ref-5)</sup>**：将 yield 实现为无操作，意味着用户程序无法主动让出 CPU。若后续不修改，任何依赖 yield 的协同调度程序（如协程库、事件驱动应用）都会失效。这是一个功能缺失（TODO），但当前没有 TODO 注释，容易被维护者忽略。

**③ kernel_satp 字段未初始化但占用空间 <sup>[1](#mod-task-ref-1)</sup>**：`TrapContext` 为 `kernel_satp` 分配了 usize 空间，`app_init` 中将其设为 0，注释说明“暂未使用”。但在上下文切换时，若汇编代码意外读取该字段（例如在切换到不同地址空间时），会使用错误的值，可能导致访存异常。属于未来扩展预留的典型“死字段”风险。

**④ #![allow(dead_code)] 掩盖未使用项 <sup>[1](#mod-task-ref-1)</sup>**：整个文件顶部使用 `allow(dead_code)`，允许未被引用的字段和方法不产生警告。这可能导致 `TrapContext` 中新增的字段在未完全接入时被编译器放过，直到运行时才暴露问题。社区最佳实践是尽量不使用全局 allow，而是逐项标注。

**⑤ trap_handler 由调用者设置但无安全检查 <sup>[1](#mod-task-ref-1)</sup>**：`app_init` 将 `trap_handler` 初始化为 0，调用者需在之后显式赋值。若调用者忘记赋值，trap 发生时将从零地址取指令，触发非法指令异常。虽然当前 batch 模型只有一个调用点，但代码缺少防御性检查（如 assert!）。

以上问题反映了该模块尚处于早期实现阶段，后续需要补充真正的任务结构、实现 yield 语义、删除未使用字段，并增加硬件抽象层的常量定义。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 当前文件系统模块仅实现了面向控制台的极简 I/O，通过 sys_write 输出字节到串口，sys_read 则直接返回 0 未实现。它依赖 sbi crate 进行物理输出，未引入任何文件系统抽象（如 vnode、磁盘驱动、文件描述符表），与 rCore-Tutorial 系列内核相比尚处于功能起点。

### 1. 核心抽象与外部依赖

本模块的核心抽象并非传统文件系统，而是一对系统调用 `sys_write` 和 `sys_read`，它们直接映射到平台串口（UART）。文件 `src/syscall/fs.rs` 通过 `use crate::sbi` 调用底层 SBI 固件的 `console_putchar` 完成单个字符的输出<sup>[1](#mod-fs-ref-1)</sup>。这一依赖关系意味着当前实现完全托管于 RustSBI 提供的超特权接口，没有自己的缓冲、中断或 DMA 支持。

从抽象层级看，模块没有定义任何文件描述符表、inode 或路径解析类型。`sys_write` 仅对 `fd == 1`（标准输出）敏感，其他文件描述符一概返回 `-1`，相当于 Linux 的 `EBADF` 但未区分具体错误<sup>[3](#mod-fs-ref-3)</sup>。`sys_read` 则直接返回 0，相当于总是读到文件末尾<sup>[4](#mod-fs-ref-4)</sup>。这意味着整个“文件系统”仅覆盖了 stdout 输出，stdin 输入、磁盘文件、目录操作均不存在。

### 2. 关键设计取舍

与同家族基线（如 rCore-Tutorial ch2 的 `sys_console_putchar`）相比，本模块将系统调用号直接对应 Linux 风格的 `read(63)`/`write(64)`，而非独立的 `putchar`。这种命名选择暗示了未来扩展为通用文件系统的意图，但目前保留了最低兼容性：**用 `write` 代替专用输出调用**。代价是调用者必须理解 `fd=1` 的特殊含义，且 `sys_write` 的签名为 `(fd, buf, len)` 却只能处理 stdout，限制了其通用性。

设计上完全放弃错误传播：`sys_write` 总是返回 `len as isize`，从不报告部分写入或失败；`sys_read` 返回 0 而不区分“无数据”与“未实现”<sup>[4](#mod-fs-ref-4)</sup>。这符合早期内核的“make it work”策略，但长期会隐藏 bug——例如当用户程序误写 `fd=2`（stderr）时不会得到错误提示，只会静默失败。

### 3. 跨模块协同

模块与 `crate::sbi` 紧密耦合，`sys_write` 的逐字节输出循环直接调用 `sbi::console_putchar`，而非通过任何抽象层或缓冲区<sup>[2](#mod-fs-ref-2)</sup>。这意味着每次 `write` 调用都会产生和字节数相等的 SBI 调用，性能极低（对于 4KiB 数据需 4096 次超陷入）。此外，`sys_write` 中的 `unsafe` 块从用户提供的裸指针 `buf` 读取数据，但没有页表验证注释 `// TODO(ch2+): 引入页表后需验证 buf 指向用户地址空间，防止内核内存泄露` <sup>[2](#mod-fs-ref-2)</sup>。这提示当前阶段内核与用户程序共享同一地址空间（或均运行在 S 模式），一旦引入页表，此处的 `unsafe` 将可能成为任意内核内存读的漏洞入口。

与系统调用分发模块的协同方式可从行号推断：`syscall/mod.rs` 根据系统调用号（63/64）分别调用 `sys_read` 和 `sys_write`。但本模块不维护任何文件描述符状态，所有状态隐含在寄存器参数中，因此无需锁或并发控制——这是单 CPU 裸机环境的简化假设。

### 4. 边角细节与不足

**① 无文件描述符表与资源管理**：`fd` 参数直接硬编码判断 `1`，其他值全部拒绝。没有 `fd_alloc`、`fd_close` 机制，无法打开文件或管道。这是当前最大的功能空白。
**② unsafe 安全契约缺失**：第 12 行的 `unsafe { buf.add(i).read() }` 没有 `// SAFETY:` 注释，且前提条件（`buf` 必须有效、对齐、可读）未得到验证。`TODO` 注释表明作者意识到该问题，但当前代码完全信任用户传入的指针<sup>[2](#mod-fs-ref-2)</sup>。
**③ 错误路径沉默**：`sys_read` 返回 0 而不是 `-ENOSYS` 或 `-EINVAL`，用户程序无法区分“设备无数据”与“功能未实现”，可能导致无意义的忙等。`sys_write` 对非法 `fd` 返回 `-1`，但未设置 `errno`（此内核尚无 errno 机制）。
**④ 硬编码 magic number**：`fd=1` 作为标准输出是硬编码的<sup>[3](#mod-fs-ref-3)</sup>，未定义如 `const STDOUT: usize = 1` 的常量，不利于后续扩展 stderr 或文件。
**⑤ 无缓冲与性能问题**：逐字节 SBI 调用，无任何软件或硬件缓冲，大块输出时性能灾难。
**⑥ 无架构条件编译**：模块唯一依赖的 `sbi` 模块假设目标平台为 RISC-V (SBI 规范)，但 `#[cfg]` 未限制仅对 riscv 编译；若移植到 x86，此处代码会因缺少 `sbi` 模块而无法编译，但错误信息可能令人困惑。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 采用 RISC-V SBI 经典 ecall 陷入模型，通过 trap_handler 分发 5 个基础系统调用（read/write/exit/getpid/yield）。核心抽象是 TrapContext 寄存器的保存与恢复，系统调用参数通过 a0-a2 传入，a7 传递调用号，返回值放回 a0。相比 rCore-Tutorial Chapter 3 基线，本实现取消了 TaskManager 的自动调度耦合，改用 batch::run_next_app() 的粗粒度切换，且完全未实现信号、多进程与文件描述符表。

### 1. 核心抽象与外部依赖

系统调用层的核心抽象是 **ecall 陷入 + TrapContext 保存恢复** 的经典二阶段模型。第一阶段由汇编入口 `__alltraps`（通过 `entry.asm` 全局嵌入 <sup>[4](#mod-syscall-ref-4)</sup>）将用户态寄存器完整压栈，构造 `TrapContext` 结构体；第二阶段进入 Rust 函数 `trap_handler`，根据 `scause` CSR 的值进行分发。

外部依赖极为精简：
- **batch 模块**：`batch::run_next_app()` 是唯一的进程调度接口，在异常杀死应用后无条件切换到下一个应用 <sup>[3](#mod-syscall-ref-3)</sup>，不涉及优先级、时间片等概念。
- **syscall 子模块**：`fs` 和 `process` 两个子模块通过 `mod` 声明聚合 <sup>[2](#mod-syscall-ref-2)</sup>，但具体实现在本仓库外无法读取，从调用签名可推断 `sys_write` 和 `sys_read` 直接使用裸指针操作内存。

这种无抽象层的设计带来极低的开销 —— 整个系统调用路径从 ecall 到返回值写入寄存器仅需 ~30 条指令（含 TrapContext 的压栈/出栈），但代价是放弃了所有安全隔离和资源管理。

### 2. 关键设计取舍

**寄存器传递约定**：采用 RISC-V Linux 标准 calling convention，a0-a2（x10-x12）传参数，a7（x17）传系统调用号 <sup>[1](#mod-syscall-ref-1)</sup>。这与 rCore-Tutorial 完全一致，但与 ArceOS 的 `axsyscall!` 宏封装的抽象层不同。放弃高级抽象意味着每个系统调用实现都必须手动处理寄存器编号，移植到 x86_64 时需要重写整个 __alltraps 和参数提取逻辑。

**错误处理策略的差异化**：
- 对 **已知系统调用** 出现内部错误时返回 `isize` 负数（依赖具体实现），未知系统调用返回 `-1` <sup>[2](#mod-syscall-ref-2)</sup> —— 这是积极的失败恢复，不 panic。
- 对 **异常类 trap**（page fault、非法指令等）直接打印信息并调用 `batch::run_next_app()` 切换到下一应用 <sup>[3](#mod-syscall-ref-3)</sup>，不尝试修复或信号通知。
- 对 **来自 S-mode 的 ecall**（cause=9）直接 panic <sup>[1](#mod-syscall-ref-1)</sup>，隐含的假设是内核自身的 bug 不应静默恢复。

这三种不同的错误处理路径在同一个 match 分支中明确区分，但缺少对 `ecall from U-mode` (cause=8) 与 `ecall from S-mode` (cause=9) 的对称处理 —— 前者正常分发，后者 panic，这与 RISC-V 特权级规范一致，但若内核某处误用 ecall 将导致难以诊断的 panic。

**Trap 入口的 Direct 模式**：`stvec` 被设置为 Direct 模式（指向 `__alltraps`）<sup>[4](#mod-syscall-ref-4)</sup>，而非 Vectored 模式。这是 rCore-Tutorial 的标准选择，优点是实现简单，缺点是无法为不同 trap 原因设置不同入口，所有异常统一走同一个保存恢复路径。对于嵌入式场景这可以接受，但缺少了利用硬件向量化减少中断延迟的可能性。

### 3. 跨模块协同

**trap_handler ↔ syscall::syscall** 的协作链是系统调用层的核心数据流：

```
ecall → __alltraps (保存上下文) → trap_handler (读取scause) 
        → syscall::syscall(id, args) → fs::sys_write/process::sys_exit 等
        → 返回值写入 cx.x[10] (a0) → __restore (恢复上下文) → eret
```

这里的关键隐藏约定是 **`cx.sepc += 4`** 必须在系统调用分发前执行 <sup>[1](#mod-syscall-ref-1)</sup>。如果把这个操作放在 syscall 返回后或 __restore 中，会破坏 context 的一致性 —— 因为 `sys_write` 等实现可能阻塞或修改 sepc。当前代码将其放在 match 分支入口处，确保用户态返回时 PC 越过 ecall 指令。

**与 batch 模块的协同** 更为粗粒度：当异常发生时（page fault、非法指令等），`trap_handler` 直接调用 `batch::run_next_app()` <sup>[3](#mod-syscall-ref-3)</sup>，该函数预期会清空当前应用状态并加载下一个应用的 TrapContext。这个调用不返回，意味着异常分支后不会有寄存器写回操作。这是有意为之的简化 —— 应用异常即终止，不提供信号恢复机制。
