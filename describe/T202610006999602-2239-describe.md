## 📖 一、总览

该内核是一个面向 Linux 兼容性的 Rust 实验性操作系统，属于 rcore-rust 家族中偏向“完整兼容”的类型，而非基础教学内核。它通过 157 个系统调用覆盖进程、文件、网络、信号等领域，支持 RISC‑V 与 LoongArch64 双架构及多核启动，目标显然是运行 musl libc 乃至复杂用户态工作负载。整体架构为宏内核，内部模块（VFS/ext4、futex、信号、smoltcp 套接字等）耦合较紧密，但各子系统均力求符合 POSIX/Linux 语义。

值得评审注意的特点有三：一是 syscall 层遍布 `const _: [(); N] = [(); size_of::<T>()]` 编译期断言，强制保证内核‑用户 ABI 结构体布局与 Linux 完全一致（[syscall.ref:5]），这在 Rust 内核中实属罕见；二是在 futex 等 IPC 路径中密集调用 `mm_hotspot` 进行内存分配热点追踪（[ipc.ref:4]），透露出面向存储场景的可观测性设计意图；三是信号子系统以 64 位掩码配合架构特化栈帧实现 ucontext 风格交付（[signal.ref:1][signal.ref:4]），完整性远超教学基线。这些特征表明该内核并非简单课程的作业，而是承载了具体应用设想的探索性原型。

## 🎯 二、综合评价

### 整体定位
内核在 rcore-rust 家族中属于“重兼容、轻抽象”的实践派：它直接面向 Linux ABI，用薄封装层将系统调用号映射到内部模块（[syscall.ref:1]），而不追求微内核或形式化验证。启动模块的引导 hart / 辅助 hart 双轨路径（[boot.ref:3]）和 task 模块的线程组、信号、资源限制（[task.ref:1]）共同支撑起丰富的多线程环境。与 rCore-Tutorial 相比，该仓明显超越了“演示原理”的阶段，试图在真实硬件上运行非平凡应用，但尚无法脱离 QEMU 虚拟平台（依赖 SBI 与 VirtIO）。

### 真正的创新点
除了概述中提到的编译期 ABI 断言，该内核在资源管理上展现了若干精巧设计。ASID 分配器使用 RAII 句柄并在回收时广播 TLB 刷新所有 hart（mm 模块）；内核栈延迟回收机制利用 idle 循环安全释放，避免上下文切换中释放锁定的栈（task 模块）。此外，内存分配热点追踪 `mm_hotspot` 并非孤立的调试设施，它在 futex 入队/唤醒/移队时记录分配来源与容量（[ipc.ref:4]），为存储型工作负载的性能剖析提供了内核级埋点——这通常只在专用 OS 中出现。信号子系统虽然依赖架构相关代码，但已做到完整支持 signalfd、altstack 和实时信号（[signal.ref:2]），且帧布局包含 siginfo 与 ucontext，较之多数教学内核仅能投递默认动作，这是质的区别。

### 取舍判断
该仓库将主要精力投入到**广度覆盖**而非**深度优化**。文件系统通过 `lwext4_rust` 绑定获得了 ext4 读写能力，但 VFS 的 page cache 仅为桩实现；网络基于 smoltcp 提供 TCP/UDP 套接字，但 AF_UNIX 域声明而不实现（[net.ref:5]），且全局 SOCKET_FILES 广播唤醒为 O(n) 扫描（[net.ref:4]）。IPC 的 futex 实现了 wait/wake/requeue，却将 WAKE_OP 和定向信号入队直接标记为 ENOSYS（[ipc.ref:5]），而管道读写采用逐字节循环加锁（[ipc.ref:2]），明显未考虑吞吐量。这些选择揭示出内核的首要目标是**跑通 POSIX 应用逻辑**，而尽量延迟性能与边界情况的打磨。

### 完成度与不足
完成度总体上合格但缺口明显：核心进程/线程原语（fork/exec/clone）与文件 I/O 较完整，信号递送亦超越基线；然而网络部分缺少 `sendmsg`/`recvmsg`，信号中 SA_RESTART 未实现自动重启（[signal.ref:17]），部分 futex 操作缺失直接削弱 pthread 条件变量的效率。代码质量堪忧：多处 unsafe 裸指针偏移无 SAFETY 注释（[boot.ref:1][signal.ref:4]），错误处理常以 `expect`/`panic` 终结（boot 的启动失败静默继续、VirtIO 错误直接 panic），管道与 futex 存在 TOCTOU 窗口（[ipc.ref:5]），且多处存在泄漏风险（如 Drop 中吞掉错误）。这些隐患表明内核尚处于“功能优先”的早期开发阶段，鲁棒性与安全性远未达到生产标准。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 该模块实现了多 hart 的 Rust OS 内核启动流程。核心抽象是引导 hart（rust_main）的全局初始化与辅助 hart（secondary_main_rust）的 per-CPU 最小初始化两条分离路径，通过 SBI HSM 机制完成多核唤醒。启动序列严格线性化：从 BSS 清除、架构特定数据段拷贝开始，依次构建内存管理、中断控制、设备驱动，最后进入调度器。与 rCore-Tutorial 基线相比，本仓库增加了多核支持、LoongArch64 架构适配、启动阶段冒烟测试等生产级特性，但架构差异处理依赖编译期 feature flag 而非统一抽象。

### 1. 启动入口的双轨架构

内核明确区分两条启动路径：`rust_main`（引导 hart，第 68 行）与 `secondary_main_rust`（辅助 hart，第 136 行）<sup>[3](#mod-boot-ref-3)</sup><sup>[7](#mod-boot-ref-7)</sup>。引导 hart 承担全部全局初始化——从内存子系统、中断控制器到设备驱动与网络栈——最后通过 SBI HSM 逐个唤醒其他 hart（第 117-129 行），再进入调度器 <sup>[6](#mod-boot-ref-6)</sup>。辅助 hart 仅执行最小化的 per-CPU 初始化：激活共享内核页表、配置 trap、使能中断，随后直接汇入同一个调度循环。这种非对称多处理器（AMP）启动模型避免了多核并发初始化的同步复杂度，但代价是辅助 hart 完全依赖引导 hart 已完成的内存映射与数据结构，两者间通过内存隐式通信而无显式同步原语。

### 2. 架构差异的编译期处理：init_data 的取舍

最醒目的架构差异体现在 `init_data` 的实现上：LoongArch64 平台需要将 `.data` 段从加载地址（`sidata`）显式拷贝到链接地址（`sdata`），而 RISC-V 等架构直接返回空函数（第 49-50 行）<sup>[1](#mod-boot-ref-1)</sup><sup>[8](#mod-boot-ref-8)</sup>。在 LoongArch64 上，固件可能将内核镜像以只读方式映射，或加载地址与链接地址物理上不一致，因此内核必须通过 `core::ptr::copy_nonoverlapping` 自行完成数据段重定位（第 40-44 行）。该设计放弃了统一抽象，转而用 `#[cfg(feature = "arch-loongarch64")]` 做条件编译——这在新架构接入时容易遗漏，且无法在编译期检测到"某架构确实需要 init_data 但忘记实现"的错误。此外，`sidata as usize == sdata as usize` 的提前返回（第 37-39 行）在无任何诊断信息的情况下跳过拷贝，调试困难。

### 3. 初始化顺序的信任链

启动序列构成一条严格的信任链，任何重排都可能导致难以调试的竞态或 use-after-free <sup>[3](#mod-boot-ref-3)</sup><sup>[4](#mod-boot-ref-4)</sup><sup>[5](#mod-boot-ref-5)</sup>。其顺序逻辑如下：
- `clear_bss` 必须在任何持有静态变量的 Rust 代码前执行（第 55-62 行）<sup>[2](#mod-boot-ref-2)</sup>，否则 `static` 变量的初始值被污染；
- `mm::init` 必须先于 `cpu::init_this_cpu_processor`（第 84-88 行），因为 CPU 子系统的 per-CPU 数据结构依赖堆分配器；
- `trap::init` 必须先于 `trap::enable_timer_interrupt`（第 90-92 行），确保中断处理函数已注册；
- `hal::arch::irq::init_external_interrupts` 必须先于 `timer::set_next_trigger`（第 94-96 行），确保中断控制器能转发定时器中断。
代码中通过依次调用与紧随其后的 `klog_debug!` 构成一个显式的"检查点序列"——这是一种轻量级的手工 tracing 机制，在早期启动阶段（尚未有完整调试设施时）提供可观测性，但缺乏编译期顺序强制。

### 4. 跨模块协同：从裸机到调度器的跃迁

辅助 hart 的启动路径揭示了一个关键协同点：`mm::KERNEL_SPACE.exclusive_access().activate()`（第 139 行）必须在其他初始化之前调用，否则内核栈（映射在高虚拟地址的 TRAMPOLINE 附近）不可访问 <sup>[7](#mod-boot-ref-7)</sup>。这一行依赖引导 hart 在 `mm::init` 中构建的共享内核页表——两个 hart 之间通过物理内存隐式通信，其正确性建立在"引导 hart 在唤醒辅助 hart 之前已完成 mm::init"这一时序假设上。`init::run()`（第 114 行）位于设备初始化之后、调度器启动之前，很可能是加载并执行第一个用户态进程（initproc），但其具体实现隐藏在未公开的 `init` 模块中，构成分析边界。`cpu::signal_boot_complete()`（第 115 行）与 `cpu::mark_online()`（第 98 行）配合，可能用于同步栅栏，确保所有 hart 就绪后才释放用户态入口。

### 5. 边角细节与不足

- **`init_data` 空实现风险**：非 LoongArch64 架构的 `init_data` 为空函数（第 49-50 行），若未来移植到需要数据段拷贝的架构（如 ARM 从 NOR Flash 启动），会静默产生未初始化的全局变量，且该错误在运行时几乎无法定位 <sup>[8](#mod-boot-ref-8)</sup>。
- **`_hart_id` 被忽略**：`secondary_main_rust` 接收 `_hart_id` 参数但完全未使用（第 136 行），无法执行 hart 特定的初始化（如 per-hart 的 PLIC 上下文配置），所有辅助 hart 执行完全相同的初始化路径 <sup>[7](#mod-boot-ref-7)</sup>。
- **hart_start 无错误处理**：`firmware::hart_start` 的返回值仅用于条件日志输出（第 127-129 行），启动失败时系统静默继续，缺少重试或 panic 机制——若某 hart 启动失败，系统将以减少的核心数运行，可能违反调度器的 CPU 数量假设 <sup>[6](#mod-boot-ref-6)</sup>。
- **`unsafe` 块缺少安全注释**：`init_data` 中的 `copy_nonoverlapping`（第 40-44 行）和 `clear_bss` 中的 `from_raw_parts_mut`（第 59-61 行）均涉及裸指针操作，但未提供 `// SAFETY:` 注释说明为何这些操作是安全的——长度计算依赖链接器符号的正确性，若链接脚本有误，此处将产生未定义行为 <sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>。
- **MAX_CPUS 硬编码**：`config::MAX_CPUS` 作为编译期常量用于 hart 遍历循环（第 118 行），无运行时与设备树或 SBI 查询的实际 hart 数校验，超出硬件范围会触发对不存在 hart 的 `hart_get_status` 和 `hart_start` 调用，其行为取决于 SBI 实现 <sup>[6](#mod-boot-ref-6)</sup>。
- **日志子系统的前置依赖隐忧**：`logging::init()` 在第 78 行调用后立即大量使用 `klog_info!`/`klog_debug!`，若日志子系统内部依赖尚未初始化的内存分配器或其他服务，可能在早期启动阶段触发未定义行为。

**2. 核心抽象与设计取舍**

内存管理模块有几个突出的设计抽象。首先是 **`MemorySet`** ——一个虚拟地址空间的完整表示，内部维护一个按起始地址排序的 `MapArea` 集合。`MemorySet` 既是用户进程地址空间的复刻，也用于内核自身的地址空间管理（如 `KERNEL_SPACE`）。它通过 `BTreeMap` 维护虚拟页到物理帧的映射，并利用 `FrameTracker` 实现物理帧的引用计数与自动回收，有效避免了显式的物理页手动释放。其次，模块采用了 **COW（写时复制）** 策略来支持 `fork` 等操作。当进程克隆时，父进程与子进程共享只读页，直到一方尝试写入才触发缺页异常并实际复制物理页。这一设计权衡了创建新地址空间的开销和后续写入成本，是类 Unix 内核的经典选择。此外，针对固定大小对象的分配，`slab` 分配器利用对象本身的空闲内存构建侵入式空闲链表，由此在不额外占用内存的情况下实现快速分配与回收，并提供了引用计数的 `Arc`‑like 包装，使共享对象的生命周期管理更为安全。`layout_allocator` 则选择将不同 size class 的分配内部化，以支持 `allocator_api` 要求，避免传统全局分配器的锁竞争和碎片，为 B‑Tree 等数据结构提供了更可预测的性能。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以 `TaskControlBlock` 为核心每线程控制块, 结合 `ThreadGroup` 实现 POSIX 兼容的进程/线程模型, 支持 `clone`/`exec`/`exit`/`wait` 等系统调用. 调度就绪队列采用每 CPU FIFO + work stealing 的简单策略, 等待队列通过 `SleepToken` 防御过期唤醒. PID/ASID/内核栈均使用 RAII 句柄管理生命周期, 其中 ASID 回收时刷新所有 hart 的 TLB, 内核栈则通过延迟回收机制在 idle 循环中安全释放. 模块广泛使用 `SpinNoIrq` 保护临界区, 要求不得跨上下文切换持锁. 与同家族基线相比, 本仓库引入了线程组、信号、资源限制、robust list 等更强的 Linux 兼容性, 并同时支持 RISC‑V 与 LoongArch 双架构.

### 1. 进程线程抽象：双级控制块与锁序

`TaskControlBlock` (TCB) 是每线程的控制块, 包含不可变字段 (pid、kernel_stack 等) 和由 `SpinNoIrq` 保护的可变内部状态 `TaskControlBlockInner` <sup>[1](#mod-task-ref-1)</sup>. `ThreadGroup` 则封装了线程间共享的资源: 地址空间、信号处理、资源限制等. 通过 `is_thread` 标志区分线程与进程组长, 当 `clone` 时指定 `CLONE_THREAD` 标志即创建线程, 共享同一线程组. 这种设计使得大多数 POSIX 进程语义得以自然映射.

锁的使用遵循严格顺序: 先获取 TCB 内部锁, 再获取线程组锁. 文档明确警告内部锁不可跨上下文切换持有, 因为其实现为 `SpinNoIrq`, 若在持锁期间发生调度将导致死锁或数据损坏. 作为典型示例, `exec` 在替换地址空间前调用 `exec_current_group_leader_cleanup`, 该函数需要杀死同一线程组的其他线程, 此清理过程在未持 TCB 锁的状态下执行, 从而满足锁序约束 <sup>[2](#mod-task-ref-2)</sup>.

### 2. 调度与阻塞：就绪队列与等待队列协同

调度就绪队列由每 CPU 的 `Processor` 维护, 操作通过 `cpu::this_processor().exclusive_access()` 进行 <sup>[4](#mod-task-ref-4)</sup>. 任务加入当前 CPU 队列, `fetch_task` 优先从本地 `fetch_local` 获取, 空时执行 `steal_from_others` 从 CPU 0 窃取, 采用 `steal_back` 以降低缓存争用 <sup>[5](#mod-task-ref-5)</sup>. 这种 work stealing 策略避免了全局队列的锁竞争, 适合中低负载场景.

阻塞机制的核心是 `WaitQueue` 与 `SleepToken`. 任务每次进入阻塞时 `SleepToken` 自增, 唤醒时仅当 token 匹配才会真正移入就绪队列, 从而消除过期的定时器或信号唤醒 <sup>[6](#mod-task-ref-6)</sup>. 可中断睡眠由 `InterruptibleSleep` 封装: `prepare_current_interruptible_sleep` 将当前任务从 `Processor` 取出并标记为 `Interruptible`, 同时记录上下文指针; 调用者随后可将其推入等待队列并注册定时器, 最终调用 `InterruptibleSleep::schedule` 让出 CPU <sup>[7](#mod-task-ref-7)</sup>. 在 `schedule` 返回后, 任务会检查是否收到信号, 若存在待递送信号则返回 `Err(EINTR)` 供上层处理 <sup>[8](#mod-task-ref-8)</sup>. 此路径中信号模块通过 `has_deliverable_signal` 与任务模块互动, 定时器模块则通过 `sleep_task_until_token` 协调超时唤醒, 体现了多个子系统间的紧密协同.

### 3. 资源生命周期与延迟回收

PID 和 ASID 的分配器均为全局单例, 由 `SpinNoIrq` 保护. `PidHandle` 和 `AsidHandle` 在 drop 时自动回收资源. ASID 回收前必须调用 `paging::flush_tlb_all_harts` 刷新所有 hart 的 TLB, 以确保该 ASID 不会被重新使用后产生地址混淆 <sup>[10](#mod-task-ref-10)</sup>. 由于 TLB 刷新代价高昂, 其执行上下文需要安全——不应在中断上下文或任务切换点进行.

内核栈的回收同样面临此问题. `KernelStack::drop` 会检查当前 PID 是否在 `DEFERRED_KERNEL_STACK_RECYCLE` 列表中, 若在则推迟 unmap 操作, 仅记录物理块的归还 <sup>[11](#mod-task-ref-11)</sup>. 延迟的操作最终在 idle 循环中由 `recycle_deferred_kernel_stacks` 统一执行, 该函数在 `run_tasks` 中被调用, 此时所有 hart 的页表已一致, TLB 维护安全无虞. 这一设计精巧地平衡了资源回收的及时性与同步代价.

### 4. 边角细节与不足

尽管整体架构合理, 代码中仍存在若干可改进之处:

- **PID 回收效率**: `PidAllocator::dealloc` 使用 `Vec` 存储回收的 PID 并通过线性遍历断言重复, 时间复杂度 O(n). 频繁创建/销毁进程时可能成为瓶颈, 且重复释放的 panic 可能被异常触发 <sup>[9](#mod-task-ref-9)</sup>.
- **ASID 耗尽无降级**: `AsidAllocator::alloc` 在池满时直接 `panic!`, 若系统允许创建大量进程(如通过 `clone` 不回收)则构成 DoS 入口 <sup>[12](#mod-task-ref-12)</sup>.
- **多处 `expect` 导致内核崩溃**: `TaskControlBlock::new` 和 `exec` 中大量使用 `expect`, 例如注册 executor 失败或打开控制台失败时直接 panic, 缺少向用户态返回错误的机制 <sup>[13](#mod-task-ref-13)</sup>.
- **Work stealing 不够灵活**: `steal_from_others` 仅从 CPU 0 窃取, 在多核系统中可能导致 CPU 0 过载或窃取失败率高, 且硬编码 hart ID 的假设限制了可移植性 <sup>[5](#mod-task-ref-5)</sup>.
- **可中断睡眠中的 TOCTOU 窗口**: `InterruptibleSleep::schedule` 在检测到信号后立即调用 `wake_task_if_blocked`, 但此时任务可能已被其他核心以新 token 唤醒, 造成重复但无害的唤醒尝试, 虽正确却增加了不必要的调度开销 <sup>[8](#mod-task-ref-8)</sup>.
- **死代码残留**: 常量 `ELF_AUXV_ENTRY_COUNT` 定义后未使用, `base_size` 字段标记 `#[allow(unused)]`, 表明早期设计的遗留, 增加了维护者的理解成本 <sup>[14](#mod-task-ref-14)</sup>.
- **日志宏掩盖主逻辑**: 大量 `#[cfg(klog_task)]` 和 `klog_debug!` 使核心流程被冗长的日志分支包裹, 降低了代码可读性.

### ⚠ 开放问题

- PID 回收使用 Vec 且线性查找, 大规模进程创建/销毁时性能不佳, 重复释放的 assert 可能被异常触发.
- ASID 分配器池满时直接 panic, 若用户态可控进程数量则构成 DoS 入口, 缺乏向用户空间报告 ENOMEM 的路径.
- 多处 .expect() (如注册 executor、打开 /dev/console) 在内核初始化或 exec 时直接 panic, 鲁棒性不足.
- Work stealing 仅从 CPU 0 窃取, 多核场景下可能造成 CPU 0 过载或饥饿其他核, 缺乏可配置的窃取策略.
- InterruptibleSleep::schedule 存在 TOCTOU 窗口, 可能导致重复唤醒尝试, 虽正确但增加不必要的调度开销.

## 1. 模块定位与文件构成

该模块实现内核的虚拟文件系统（VFS）及 ext4 物理后端，位于 `os/src/fs/` 下。核心职责包括：提供标准的文件／目录操作接口、路径解析、挂载管理，并通过适配器将请求转接到具体文件系统实现。主要文件有：`adapter/ext4.rs` 负责将 `lwext4_rust` 库的 ext4 能力封装为内核所需的 `KernelFileSystem` 接口；`inode.rs` 定义了操作文件在 VFS 层的代理 `OSInode`，承担路径查找、符号链接跟踪、锁管理与读写代理；`page_cache.rs` 则定义了文件页缓存的身份与 API 边界，目前处于第一阶段的桩实现，旨在后续引入真实缓存树和全局 LRU 管理。此外还要结合未列出的 `ops.rs`、`mount.rs` 等定义共享 trait 与挂载逻辑。整个模块向上为系统调用提供文件抽象，向下通过块设备驱动访问存储介质，是内核 I/O 通路的中枢。

## 2. 核心抽象与设计取舍

最关键的抽象是 `KernelFileSystem` 与 `KernelInode` trait，它们构建了后端无关的 VFS 契约。`Ext4Adapter` 实现 `KernelFileSystem`，内部持有 `Ext4Context`、根 inode 和磁盘块封装，将文件系统级操作（如 open、stat、readdir）翻译为 lwext4 调用。这种设计允许替换不同后端，例如未来可添加 tmpfs 或 FAT 适配器。另一个重要抽象是 `OSInode`，它组合了读写能力开关、内层状态锁与可选的写保护 guard（`BusyFileGuard`），将 `KernelInode` 包装成符合 POSIX 语义的文件句柄。路径解析通过 `ResolveOptions` 显式控制符号链接跟踪深度，防止无限循环。一个明显的取舍是在 ext4 适配器中通过 `unsafe impl Send/Sync` 标记 `Ext4Adapter`（见 `adapter/ext4.rs:40-43` 附近），注释说明因内核当前为单处理器且 lwext4 使用全局 C 状态，此标记是安全的；若将来支持多核，这一假设需重新审视，可能引入细粒度锁。

## 3. 具体不足与待办事项

第一处明确未完成的工作在 `page_cache.rs`，文件头注释声明当前是第一阶段的桩实现，所有缓存操作均直接透传至后端（`page_cache.rs:4-7`），真正的缓存树、脏页回写、全局 LRU 回收均待后续阶段实现。这意味着目前所有文件读写都会穿透到存储设备，无法享受内存缓存带来的性能提升，在频繁元数据访问场景下会显著增加延迟。第二，`adapter/ext4.rs` 中的错误码转换函数 `mount_error_to_fs_error` 仅将 `EEXIST` 映射为 `AlreadyExists`，其余一律映射为 `Invalid`（`adapter/ext4.rs:67-76`），丢失了如权限拒绝、磁盘满、只读文件系统等语义，上层调用者无法获得精确的错误原因，影响系统调用的错误报告。另外，`inode.rs` 中 `OSInode` 的 `_write_guard` 以 `Option` 持有但未见到完整的忙文件防冲突逻辑，可能尚未彻底实现多 writer 互斥或文件范围的字节锁。page_cache 模块中还定义了多个 workder（flusher、kswapd、readahead）但启动主循环的代码尚未在提供的片段出现，推测仍是待完成项。

## 4. 跨文件协同

文件系统模块的协同链路清晰：`inode.rs` 中的 `OSInode` 通过 `KernelInode` trait 调用后端，通常由 `ActiveFileSystem` 根据挂载表将请求路由到 `Ext4Adapter`。具体读写时，`OSInode` 会检查 page_cache 的命中情况，但目前由于 `page_cache.rs` 直接透传，其实质仍是调用 `KernelInode` 的底层读写方法。挂载与路径解析流程跨多个文件：`mount.rs` 负责维护挂载表，`path.rs` 提供 `VfsPath` 拆分与遍历逻辑，再由 `inode.rs` 的 `ResolveOptions` 控制符号链接跟踪。设备节点注册通过 `DeviceNodeInfo` 与 `devices` 模块交互，使得 `OSInode` 可以为设备文件创建转发句柄。这样的分层使得后端替换和缓存增强可以局部演进，但当前阶段要特别注意 page_cache 桩实现导致的穿透特性，一旦后续写入真实缓存需确保在整个调用链中正确刷新和失效，避免跨层数据不一致。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块以 64 位掩码为信号挂起/阻塞表示, 通过 `KernelSigAction` 数组存储每信号处理配置, 并由架构相关代码 (`loongarch64`/`riscv64`) 负责用户栈上的 ucontext 风格信号帧构造与恢复. 核心流程为: `enqueue_signal` 置位并唤醒任务 → 返回用户态前由 `take_next_deliverable_signal` 择取可递送信号 → 按 disposition 执行默认终止或构造栈帧跳转 handler → 用户态通过 `rt_sigreturn` 恢复上下文. 相比 rCore-Tutorial 基线的简易信号支持, 该实现额外提供了 signalfd、线程组递送、altstack、超时等待等机制.

### 1. 核心抽象: 64 位掩码 + 架构分派的栈帧

信号子系统的平台无关核心在 `os/src/signal.rs`, 它将信号表示为一个 `u64` 的 pending 位图, 支持 1..64 号信号 (`MAX_SIGNAL=64`) <sup>[1](#mod-signal-ref-1)</sup>. 每个任务的 `TaskControlBlock` 内嵌 `signal` 状态: `pending`(挂起位图)、`mask`(阻塞位图)、`wait_mask`(等待位图) 以及 `actions[65]` 数组 <sup>[2](#mod-signal-ref-2)</sup>. 每信号的处理配置由 `KernelSigAction` 承载: `handler`(用户态地址或 SIG_DFL/SIG_IGN)、`mask`(处理期间的阻塞集)、`flags`、`restorer`(固定为 `USER_SIGRETURN_TRAMPOLINE` 跳板地址) <sup>[3](#mod-signal-ref-3)</sup>.

架构相关层 (`hal/arch/{loongarch64,riscv64}/signal.rs`) 负责信号帧的物理布局: 在用户栈上依次放置 core 寄存器区、`siginfo` 区、`ucontext` 区 (含 `sigmask`、`mcontext` 的 PC/GRPs/FP regs) <sup>[4](#mod-signal-ref-4)</sup>. 帧大小在 LoongArch64 上为 `FRAME_UCONTEXT_OFFSET + UCONTEXT_SIZE + 16`, 在 RISC-V 上无末尾的 +16 对齐填充 <sup>[5](#mod-signal-ref-5)</sup>. 这种分层设计使得信号语义与具体寄存器布局解耦 — 新增架构只需实现 `write_frame`/`read_restored_context` 两个函数.

### 2. 递送流水线: 入队 → 唤醒 → 惰性择取 → 栈帧构造

信号递送采用"即时入队、延迟择取"模式. `enqueue_signal` 仅做三件事: 有效性校验、pending 位置位、条件唤醒 — 若任务处于 `Interruptible` 睡眠且信号未被阻塞(或在 `wait_mask` 中), 则将状态改为 `Ready` 并调用 `add_task` 唤醒 <sup>[6](#mod-signal-ref-6)</sup>. 

真正的择取发生在返回用户态前夕: `take_next_deliverable_signal` 遍历 1..=64 号信号, 跳过 `SIG_IGN` 和默认忽略 (SIGCHLD), 对 `SIG_DFL` 返回 `DefaultTerminate`, 对用户 handler 返回 `Handler {signum, action}` 并清零对应 pending 位 <sup>[7](#mod-signal-ref-7)</sup>. 调用方(陷阱处理) 据此决定是终止任务还是通过 arch `write_frame` 构造栈帧: 将当前寄存器、旧 mask、siginfo、ucontext 写入用户栈 (可能使用 altstack), 设置用户 PC 为 handler 地址、RA 为 restorer 跳板.

`signal_disposition` 函数按需计算信号处置, 不做缓存 <sup>[8](#mod-signal-ref-8)</sup>. 这意味着每次查询都重新读取 `actions[]` 和 `mask` — 简单但存在 TOCTOU 窗口: `is_ignored` 在 `enqueue_signal` 内先持锁检查再释放, 随后 `enqueue_signal` 再次持锁写入, 期间 action 可能被其他线程修改. 不过实际影响有限, 因为递送路径上的择取与入队共享同一锁保护.

### 3. 跨模块协同: 全局注册表 + 中断睡眠 + signalfd

**任务注册表**: `TASK_REGISTRY` 是一个 `BTreeMap<tid, TaskWeak>` 全局表, 受 `SpinNoIrq` 保护 <sup>[9](#mod-signal-ref-9)</sup>. `find_task` 通过 tid 查表并尝试 `upgrade` 弱引用, 用于 `sys_kill`/`sys_tkill` 的目标定位. `sys_kill` 对 pid>0 先查精确 tid 匹配, 再遍历 `registered_tasks()` 按 tgid 匹配 — 这模拟了 Linux 的线程组信号投递 <sup>[10](#mod-signal-ref-10)</sup>. `enqueue_signal_to_thread_group` 优先选择未阻塞信号的线程, 回退到线程组 leader. 使用 `TaskWeak` 避免了全局表阻止任务释放.

**中断睡眠**: `sys_rt_sigtimedwait` 和 `sys_rt_sigsuspend` 通过 `prepare_current_interruptible_signal_sleep` 进入带 `wait_mask` 的可中断睡眠 <sup>[11](#mod-signal-ref-11)</sup>. 当 `enqueue_signal` 检测到 `wait_mask & bit != 0` 时唤醒任务, 由此将信号子系统与调度器耦合. `rt_sigtimedwait` 还支持带 deadline 的超时: 先计算绝对 tick, 进入睡眠前调用 `sleep_task_until_tick_token` 注册定时器, 超时返回 `EAGAIN` <sup>[12](#mod-signal-ref-12)</sup>.

**signalfd**: `SignalFd` 实现 `File` trait, 将信号抽象为可读文件描述符. `read` 方法调用 `take_pending_signal_in_set` 消费一个匹配信号并返回 `SignalfdSiginfo` 结构 <sup>[13](#mod-signal-ref-13)</sup>. `poll` 方法直接检查 `pending & mask` 以返回 `POLLIN`. 该实现支持通过 `signalfd4` 系统调用创建的 mask 动态更新. 与 Linux 的一个显著差异: 阻塞模式下若无信号, `read` 返回 `Ok(0)` 而非阻塞 — 这可能导致依赖 blocking read 的用户态程序误判 EOF.

### 4. 边角细节与不足

- **`read_sigset` 吞错误**: `copy_from_user(addr).unwrap_or(0)` 在用户指针无效时返回 0 mask, 掩盖了 `EFAULT` <sup>[14](#mod-signal-ref-14)</sup>. 调用方 `sys_signalfd4`、`sys_rt_sigprocmask` 等依赖此行为可能接收空掩码而不报错.
- **`sys_rt_sigqueueinfo` 完全未实现**: 直接返回 `ENOSYS`, 意味着带 `siginfo` 的定向信号入队不可用 <sup>[15](#mod-signal-ref-15)</sup>.
- **unsafe 块无 SAFETY 注释**: 两个架构的 `write_frame` 和 `read_restored_context` 中包含大量 `unsafe { frame_ptr.add(...) }` 裸指针运算, 均未附 `// SAFETY:` 说明其契约 <sup>[4](#mod-signal-ref-4)</sup><sup>[5](#mod-signal-ref-5)</sup>. 虽然这些偏移量由编译期常量保证不越界, 但缺少文档会增加维护风险.
- **signalfd 阻塞语义偏差**: 如上所述, 阻塞 `read` 返回 0 而非阻塞或 `EAGAIN`, 不符合 Linux signalfd(2) 规范 <sup>[16](#mod-signal-ref-16)</sup>.
- **`SA_RESTART` 未被 plumb**: 该标志位已定义 (`0x1000_0000`) 并通过 `rt_sigaction` 存入 `KernelSigAction.flags`, 但 `take_next_deliverable_signal` 到陷阱处理的整条路径未见任何 syscall restart 逻辑 <sup>[17](#mod-signal-ref-17)</sup>. 这意味着被信号中断的系统调用不会自动重启.
- **altstack 状态跟踪的 TOCTOU**: `sys_sigaltstack` 在设置新栈时检查 `altstack_onstack` 防止重入, 但 `rt_sigreturn` 的 onstack 判定在恢复上下文之后 — 若 handler 永不返回, onstack 状态可能长期滞留 <sup>[18](#mod-signal-ref-18)</sup>.

### ⚠ 开放问题

- `sys_rt_sigqueueinfo` 直接返回 ENOSYS, 带 siginfo 的定向信号入队完全不可用 [ref:15].
- `read_sigset` 以 unwrap_or(0) 吞掉 copy_from_user 错误, 用户态传非法指针不会得到 EFAULT [ref:14].
- signalfd 阻塞模式 read 无信号时返回 Ok(0) 而非阻塞, 与 Linux signalfd(2) 语义不一致, 可能误导 poll/epoll 使用者 [ref:16].
- 两个架构的 write_frame/read_restored_context 中含大量 unsafe 裸指针偏移, 无 SAFETY 注释说明安全契约 [ref:4][ref:5].
- SA_RESTART 标志已定义并存入 flags, 但整个 syscall 返回路径未见自动重启逻辑, 被信号中断的慢速系统调用不会重试 [ref:17].

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了三种 IPC 机制: (1) Linux futex—基于全局 BTreeMap + 侵入式链表的内核等待队列, 支持 WAIT/WAKE/REQUEUE/CMP_REQUEUE 及 waitv 多地址等待, 但 WAKE_OP 与若干次要 op 返回 ENOSYS; (2) 匿名管道—作为 File trait 实现, 以物理帧为环形缓冲区, 支持阻塞/非阻塞读写、poll 和 SIGPIPE 信号, 但每字节一次锁操作; (3) System V 共享内存—shmget/shmat/shmdt/shmctl 四个 syscall 委托给 mm 子系统, 显式拒绝 SHM_REMAP, 部分统计字段填零. 三者通过 syscall 层统一分发, 并依赖 task 调度器与 mm 地址空间管理完成睡眠/唤醒/映射协同.

### 1. 核心抽象: 三条独立的 IPC 路径

该模块并非单一切面, 而是将三种截然不同的 IPC 范式集成在同一内核中. futex 面向用户态锁原语, 核心抽象是 `FutexKey`—以 `(tgid, uaddr)` 或文件映射的 `SharedFutexKey` 作为等待队列的键 <sup>[1](#mod-ipc-ref-1)</sup>. 匿名管道实现为 `PipeEnd: File`, 通过 `Arc<Pipe>` 共享环形缓冲区, 以 `readers`/`writers` 引用计数感知对端关闭 <sup>[2](#mod-ipc-ref-2)</sup>. System V 共享内存则完全委托给 `crate::mm` 模块, syscall 层仅负责参数校验、权限转换和错误码映射 <sup>[3](#mod-ipc-ref-3)</sup>. 三条路径在 `os/src/syscall.rs` 中通过分发编号 (98/449, 194-197) 汇聚, 但各自依赖不同的内核子系统.

与同家族基线 (ArceOS / rCore-Tutorial) 相比, 本仓库的独特之处在于: futex 实现中遍布 `crate::mm::mm_hotspot` 调用——这是一种内存分配热点追踪机制, 在每次 futex 入队、唤醒、移队时记录容量变化与分配来源, 体现了该系统作为 storage 型 OS 对内存行为可观测性的强需求 <sup>[4](#mod-ipc-ref-4)</sup>.

### 2. Futex 设计取舍: 全局锁、侵入式链表与 TOCTOU

futex 最关键的架构决策是使用单一全局 `FutexQueueMap`: 一个 `SpinNoIrq<BTreeMap<FutexKey, FutexQueue, KernelLayoutAllocator>>` <sup>[5](#mod-ipc-ref-5)</sup>. 这放弃了 Linux 的 per-mm 哈希表设计, 换来极简的实现复杂度, 代价是所有进程的 futex 操作争用同一把自旋锁. 在 storage 场景下, 用户态 IPC 压力通常远小于内核 I/O 路径, 这个取舍是合理的, 但若运行高并发数据库类负载 (如 PostgreSQL 的锁管理器) 则可能成为瓶颈.

等待者链表没有使用标准库的 `LinkedList`, 而是手工维护 `NonNull<FutexWaiter>` 侵入式链表 <sup>[6](#mod-ipc-ref-6)</sup>. 这并非风格偏好: waiter 的分配走 `Slab<FutexWaiter, 1>` slab 分配器, 通过 `SlabBox::into_raw_parts` 将分配器元数据与链表指针解耦 <sup>[7](#mod-ipc-ref-7)</sup>. 唤醒路径在持锁期间将 `TaskRef` 收集到栈上的 `Vec` 中, 释放锁后再调用 `add_task` 将任务重新入队 <sup>[8](#mod-ipc-ref-8)</sup>——这是典型的”锁内收集, 锁外调度“模式, 避免了持锁期间重入调度器的死锁风险.

然而 `sys_futex` 的 `FUTEX_CMP_REQUEUE` 分支存在 TOCTOU 隐患: 代码在计算 source key 之后、获取全局锁之前, 调用 `read_user_u32` 检查 futex 字的值 <sup>[9](#mod-ipc-ref-9)</sup>. 从读到实际执行 `futex_requeue` 之间存在窗口, 另一个线程可以修改 uaddr 处的值, 使得条件比较失效. Linux 内核通过在持有 futex_hash_bucket 自旋锁之后才读取用户值来解决此问题; 本实现因为 key 计算和值检查分离, 无法保证原子性.

`FUTEX_WAKE_OP` 直接返回 `ENOSYS`, 另外 op 2/6/7/8/11/12/13 同样未实现 <sup>[10](#mod-ipc-ref-10)</sup>. WAKE_OP 是 glibc `pthread_cond_signal` 在特定情况下使用的关键操作, 缺失意味着条件变量可能退化为纯 futex_wake + futex_wait, 功能正确但性能下降.

### 3. 管道: 基于物理帧的字节级环形缓冲

管道实现选择了直接操作物理页帧的方案: `PipeInner` 持有 `Vec<FrameTracker>`, 通过 `ppn.get_bytes_array()[offset]` 读写字节 <sup>[11](#mod-ipc-ref-11)</sup>. 这绕过了虚拟地址映射, 避免了内核需要为管道单独建立页表的复杂性, 但也将管道与物理内存分配器紧耦合. 环形缓冲的读写指针在 `capacity = PIPE_BUFFER_PAGES * PAGE_SIZE` 范围内取模, 利用 `readers`/`writers` 计数器在 `Drop` 时唤醒对端 <sup>[12](#mod-ipc-ref-12)</sup>.

`PipeEnd::read` 和 `write` 按**单字节**循环处理: 每次迭代获取 `SpinNoIrq` 锁, 检查条件, 读写一个字节, 释放锁, 然后唤醒等待者 <sup>[13](#mod-ipc-ref-13)</sup>. 对于典型的 shell 管道 (如 `cat file | wc -c`), 这种逐字节传输会导致系统调用级别的吞吐量极低. 与此对比, Linux 内核的 pipe 实现通过 `pipe_buf_operations` 支持零拷贝的页面窃取 (splice), 且在 `pipe_read`/`pipe_write` 中进行批量复制. 本实现更适合作为教学或嵌入式场景的最小可用管道.

`poll` 方法被标注为 `SYSCALL-UNREVIEWED` <sup>[14](#mod-ipc-ref-14)</sup>——这是一个自我声明的技术债标记, 说明该实现是快速原型, 尚未经过充分的非阻塞语义审查. 当前逻辑在判断 `POLLHUP` 和 `POLLERR` 时与读写路径的检查条件存在细微差异 (例如 poll 在 `writers == 0` 时同时返回 `POLLIN | POLLHUP`, 但 read 在 `writers == 0` 时直接返回已读字节数而不设置 HUP), 可能导致 epoll 边缘触发下的行为不一致.

### 4. System V 共享内存: 惰性实现与 ABI 校验

SysV shm 的实现遵循“syscall 层只做编解码”原则: `sys_shmget`/`sys_shmat`/`sys_shmdt`/`sys_shmctl` 将核心逻辑全部委托给 `crate::mm` 中的 `shmget_segment`、`shm_attach_segment`、`shm_detach_segment` 等函数 <sup>[15](#mod-ipc-ref-15)</sup>. 唯一在 syscall 层完成的复杂操作是 `sys_shmat` 中的地址选择: 当 `shmaddr == 0` 时调用 `memory_set.find_mmap_area(size)` 寻找空闲虚拟地址区间, 并将返回地址传给 `insert_shared_memory_area` <sup>[16](#mod-ipc-ref-16)</sup>. 如果映射失败, 代码会显式调用 `shm_detach_segment` 回滚附着计数——这是正确的资源清理路径.

两处编译期结构体大小断言 (`const _: [(); 48]` 和 `const _: [(); 112]`) 强制 `Ipc64Perm` 和 `Shmid64Ds` 与 Linux ABI 布局一致 <sup>[17](#mod-ipc-ref-17)</sup>, 这是一种廉价但有效的跨版本兼容性检查. 但 `shmid64_ds_from_snapshot` 将 `shm_perm.seq` 硬编码为 0, `__unused4`/`__unused5` 也未填充 <sup>[18](#mod-ipc-ref-18)</sup>, 依赖这些字段的应用程序 (如 `ipcs -m` 的某些实现) 可能获取到无效信息.

`SHM_REMAP` 被显式拒绝 <sup>[19](#mod-ipc-ref-19)</sup>——该标志允许替换指定地址的已有映射, 在 Linux 2.6 后引入. 不支持此标志意味着无法运行依赖 `shm_open` + `mmap(MAP_FIXED)` 语义的应用, 但鉴于该 OS 的 storage 定位, 这类需求可能并不迫切.

### 5. 跨模块协同

三个 IPC 机制都重度依赖 `crate::task` 的睡眠基础设施. futex 的 `prepare_current_interruptible_sleep` 返回 `SleepToken`, waiter 入队后调用 `sleep.schedule()` 将当前任务置为阻塞态; 唤醒路径则通过 `mark_task_ready_if_blocked` 检查 token 是否匹配, 防止虚假唤醒 <sup>[20](#mod-ipc-ref-20)</sup>. 管道使用相同的 `prepare_current_interruptible_sleep` 但搭配 `WaitQueue::sleep_interruptible_locked`——这会将 waiter 注册到管道的等待队列上, 使得 `Drop` 时可以批量唤醒 <sup>[21](#mod-ipc-ref-21)</sup>.

信号集成方面: 管道写端在检测到 `readers == 0` 时, 通过 `crate::signal::enqueue_signal(&task, SIGPIPE)` 向当前任务投递 `SIGPIPE` <sup>[22](#mod-ipc-ref-22)</sup>. 这是 POSIX 要求的语义——写入无读端的管道应产生 SIGPIPE. 共享内存分离操作 `sys_shmdt` 在成功解除映射后, 调用 `sync_munmap_to_thread_group` 进行跨线程 TLB 一致性维护 <sup>[23](#mod-ipc-ref-23)</sup>.

## 🌐 九、网络

> 💡 **TL;DR**
>
> 基于 smoltcp 协议栈实现的用户态套接字层, 通过 SocketFile 将 TCP/UDP 套接字封装为文件抽象 (实现 File trait). 核心机制包括: (1) 全局 SOCKET_FILES 注册表用于广播唤醒; (2) 基于 SpinNoIrq 的粗粒度锁; (3) TCP 监听池预分配策略以支撑并发 accept. AF_UNIX 声明但未实质实现, 所有操作最终回退到 FsError::Invalid.

### 1. 核心抽象: SocketFile 作为文件接口的 smoltcp 封装

该模块将 smoltcp 协议栈的套接字句柄 (`SocketHandle`) 包装为 `SocketFile`, 并实现 `File` trait, 使其可通过文件描述符表统一操作 <sup>[1](#mod-net-ref-1)</sup>. `SocketFile` 内部持有一个 `SpinNoIrq<SocketFileInner>` 保护所有状态字段, 包括协议类型、端点信息、shutdown 标志、超时设置及多播组成员等 <sup>[2](#mod-net-ref-2)</sup>.

套接字的生命周期由全局 `SOCKET_FILES` 注册表 (`Static<SpinNoIrq<Vec<Weak<SocketFile>>>>`) 追踪 <sup>[3](#mod-net-ref-3)</sup>. 该注册表同时承担两个职责: (a) 在 `Drop` 时确保 smoltcp 句柄被正确关闭与移除; (b) 在网络事件 (如 poll 返回变更) 后广播唤醒所有注册套接字的等待队列. 这是一种简化设计——省去了事件分发层的额外抽象, 但代价是每次广播都需 O(n) 遍历全部存活套接字 <sup>[4](#mod-net-ref-4)</sup>.

`SocketOpenConfig::parse()` 负责将用户态传入的 `domain/type/protocol` 三元组转换为内部枚举. 值得注意: `AF_UNIX` 被接受但映射为 `SocketDomain::Unix`, 且强制要求 `SOCK_STREAM` 与 protocol=0; 然而后续 `create_socket_file` 对该域不会调用 `with_stack` 创建实际句柄 (`handle = None`), 导致任何后续操作均因 `snapshot()` 无法取得 `SocketHandleState::Single` 而返回 `FsError::Invalid` <sup>[5](#mod-net-ref-5)</sup>.

### 2. TCP 监听池: 预分配与饱和增长策略

该模块为 TCP `listen` 实现了一个精巧的预分配池 `TcpListenerPool`. 当用户调用 `listen` 时, 并不直接让单个 smoltcp socket 进入 LISTEN 状态, 而是创建一批监听 socket (初始 `target_watermark = min(backlog, LISTEN_PREALLOC_THRESHOLD)` 个), 放入 `VecDeque<SocketHandle>` <sup>[6](#mod-net-ref-6)</sup>. accept 时从池中寻找第一个已有连接就绪的 handle, 取出后立即调用 `replenish_listener_pool_locked` 补回目标水位 <sup>[7](#mod-net-ref-7)</sup>.

当池中所有 handle 都仍在监听而无就绪连接时, `grow_listener_pool_if_saturated_locked` 被触发: 若当前 `target_watermark` 未达 `backlog` 且无空闲监听者, 则递增水位 (步长为 `LISTEN_PREALLOC_THRESHOLD=8`) 并补货 <sup>[8](#mod-net-ref-8)</sup>. 这实现了 backlog 的惰性增长——只有负载确实超出当前池容量时才扩大, 避免一次性创建过多 smoltcp socket 浪费内存. 水位上限被 `MAX_TCP_LISTEN_BACKLOG=64` 硬封顶.

这一设计放弃了标准的"单一 listen socket + accept 队列"模型, 换来的是: 多个 smoltcp socket 并发监听同一端点, accept 无需在用户态与协议栈之间搬运连接状态; 但代价是每个预分配 socket 都占用独立的协议栈资源, 且池的管理逻辑 (replenish/grow/retune) 在每次 accept 和 listen 时都会执行, 增加了锁内代码路径.

### 3. 阻塞 I/O 与超时机制

`SocketFile` 的 `send_to`/`recv_from`/`accept` 均在循环中调用对应的 `try_*` 方法, 若返回 `WouldBlock` 且套接字非 `NONBLOCK`, 则进入等待 <sup>[9](#mod-net-ref-9)</sup>. 等待路径通过 `wait_on` 实现: 首先调用 `prepare_current_interruptible_sleep` 创建可中断睡眠上下文, 然后以 `SOCKET_SLEEP_POLL_MS=10ms` 为轮询间隔设置定时器, 在定时器到期后重新调度并再次尝试 I/O <sup>[10](#mod-net-ref-10)</sup>. 这种"定时唤醒+重试"模式避免了依赖 smoltcp 提供异步事件回调, 但引入了 10ms 的平均延迟.

超时支持基于 `recv_timeout_us`/`send_timeout_us` 字段, 通过 `timeout_deadline_ms` 将微秒超时转换为绝对毫秒截止时间, 并在 `wait_on` 入口处检查是否已过期. 若 `prepare_current_interruptible_sleep` 失败 (通常因信号), 返回 `FsError::Interrupted` 以支持信号驱动中断.

### 4. 跨模块协同: 与文件系统及任务调度的集成

`SocketFile` 实现 `File` trait 的核心方法是 `read` 与 `write`: `read` 在内核堆分配临时 `Vec<u8>`, 调用 `recv_from` 填入数据, 再通过 `unsafe { dst.write_volatile(*byte) }` 逐字节拷贝到用户态 `UserBuffer` <sup>[11](#mod-net-ref-11)</sup>; `write` 反之, 从用户缓冲区 `read_volatile` 读入内核 Vec 后调用 `send_to`. 这里的两处 `unsafe` 均无 `// SAFETY:` 注释, 安全契约隐含依赖 `UserBuffer` 构造时对地址有效性的校验.

网络事件触发后的任务调度协同: 当 smoltcp `poll()` 返回变更时, `wake_all_socket_waiters()` 遍历全局注册表唤醒所有套接字的四个等待队列 (read/write/poll_read/poll_write), 且若当前上下文有任务则调用 `suspend_current_and_run_next()` 主动让出 CPU——这使得网络数据到达后等待中的用户任务能尽快被调度 <sup>[12](#mod-net-ref-12)</sup>.

### 5. 边角细节与不足

- **AF_UNIX 实质为空实现** <sup>[5](#mod-net-ref-5)</sup>: `SocketDomain::Unix` 路径不会创建 smoltcp 句柄, 导致 bind/connect/listen 等操作必然失败. 该常量与枚举分支的存在暗示设计预留, 但当前属于死代码路径.
- **Drop 中的静默失败** <sup>[13](#mod-net-ref-13)</sup>: `SocketFile::drop` 内 `with_stack(...).unwrap_or(false)` 在协议栈不可用时直接吞掉错误, 导致 smoltcp 句柄泄露且无日志告警.
- **Ephemeral port 分配的非原子回绕** <sup>[14](#mod-net-ref-14)</sup>: `next_ephemeral_endpoint_with_addr` 使用 `Relaxed` 的 `fetch_add`, 回绕逻辑先检测越界再 `store`, 中间窗口存在竞争 (尽管当前单核或协作式调度下风险较低).
- **错误映射丢失诊断信息** <sup>[15](#mod-net-ref-15)</sup>: 全部 `map_*_error` 函数将 smoltcp 的细分错误统一映射为 `FsError::Invalid`/`WouldBlock`/`BrokenPipe`, 丢弃了原始错误类型, 增加了调试难度.
- **`unsafe` 无安全契约文档** <sup>[11](#mod-net-ref-11)</sup>: `File::read` 和 `File::write` 中的 `write_volatile`/`read_volatile` 均未附 `// SAFETY:` 注释说明为何用户态指针在此处有效.
- **全局注册表的 O(n) 广播** <sup>[4](#mod-net-ref-4)</sup>: `wake_all_socket_waiters` 每次调用都 drain 并重建全局 Vec, 套接字数量大时可能成为性能瓶颈.
- **硬编码常量缺乏平台参数化**: `SOCKET_SLEEP_POLL_MS=10`、`LISTEN_PREALLOC_THRESHOLD=8`、`MAX_TCP_LISTEN_BACKLOG=64` 均为经验值, 未见根据内存容量或 CPU 频率自适应调整的机制.

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该模块以 `BlockDevice` trait 为核心抽象，统一所有块设备驱动（VirtIO-PCI、loop、ramdisk 等），固定 512 字节块大小。辅以 `CharDevice` trait 覆盖控制台与 loop-control 字符设备。上层通过 `request_queue` 提供 LRU 读缓存、写合并、延迟刷写与内核工作线程协同；驱动侧以 SpinNoIrq 实现内部同步，采用同步轮询等待 I/O 完成。整体面向 LoongArch/QEMU 平台，与 rCore-Tutorial 基线相比新增了 loop 设备、请求队列缓存层与 TTY 集成。

### 1. 核心抽象：BlockDevice 与 CharDevice 双 trait 体系

驱动框架围绕两个核心 trait 展开。`BlockDevice` 定义了统一的块设备接口：`read_block`/`write_block` 以 512 字节单块为原子操作，`read_blocks`/`write_blocks` 提供连续多块读写（默认回退到逐块调用），`read_blocks_sg`/`write_blocks_sg` 支持 scatter-gather 批量提交 <sup>[1](#mod-drivers-ref-1)</sup>。所有方法通过 `&self` 共享借用调用，强制驱动内部自行处理同步——本仓库统一使用 `SpinNoIrq` 自旋锁 <sup>[2](#mod-drivers-ref-2)</sup>。

`CharDevice` trait（在 console.rs 和 loop_device.rs 中实现）提供字符设备语义：`read`/`write`、`ioctl`、`poll` 和 `register_poll_wait`。它不追求 POSIX 完备性，而是为 `/dev/console` 和 `/dev/loop-control` 等特殊设备提供最小可行接口。

与同家族基线（rCore-Tutorial ch9）相比，该模块在两个方向做了显著扩展：一是引入了 Linux 兼容的 loop 设备（支持 `LOOP_SET_FD`/`LOOP_CONFIGURE` 等 ioctl），让 LTP mount-device 测试流程可以直接跑在本内核上；二是新增了 `request_queue` 层，将读写请求做合并缓存后再下发驱动。

### 2. 关键设计取舍：同步轮询、panic 式错误处理与硬编码常量

**同步轮询 vs 中断驱动**：VirtIO-PCI 驱动在提交请求后，通过 `spin_loop()` 忙等 `used_idx` 变化来等待设备完成 <sup>[3](#mod-drivers-ref-3)</sup>。这避免了异步回调的复杂度，但牺牲了 CPU 效率——在内核态持有自旋锁期间忙等，违背了 virtio 规范推荐的中断通知模式。`reset_device` 同样用 100,000 次 `spin_loop()` 作为超时轮询 <sup>[4](#mod-drivers-ref-4)</sup>。

**panic 式错误处理**：`VirtIOPciBlock::read_block` 在 I/O 失败时直接 `.expect("Error when reading VirtIO PCI block")`，将设备错误升级为内核 panic <sup>[5](#mod-drivers-ref-5)</sup>。这与 `BlockDevice` trait 中 `read_block` 无返回值的设计一致——trait 本身不提供错误传播通道，驱动只能选择 panic 或吞掉错误。对比 `request_queue` 层的 `queued_read_blocks` 则返回 `FsResult<()>`，体现了上层对错误处理的更高要求。

**硬编码常量**：块大小 512 字节在至少四个文件中独立定义为 `const BLOCK_SIZE: usize = 512` <sup>[6](#mod-drivers-ref-6)</sup>。virtio 队列大小固定为 16，loop 设备数量固定为 8，读缓存上限为 256 个块（128 KiB）。这些常量缺乏集中管理，若需调整需要 grep 多处修改。

**loop 设备的容错策略**：未绑定的 loop 设备读取返回全零缓冲区，写入静默丢弃 <sup>[7](#mod-drivers-ref-7)</sup>。这种"优雅降级"避免了 panic，但也掩盖了用户态误操作（如对未绑定设备执行 mkfs），可能造成难以排查的静默数据丢失。

### 3. 跨模块协同：请求队列、读缓存与内核工作线程

`request_queue` 模块是本框架的"中间件"。它提供三个协同机制：

**(1) LRU 读缓存**：`BlockReadCache` 以 `(device_id, block_id)` 为键，用 `BTreeMap` 存储最近读取的块数据，LRU 淘汰通过 `VecDeque` 实现 <sup>[8](#mod-drivers-ref-8)</sup>。`queued_read_blocks` 先查缓存命中则跳过设备 I/O，未命中则合并连续块请求（最多 14 个块）通过 scatter-gather 一次下发。写入路径会调用 `invalidate_range` 使缓存失效，保证读后写的一致性。

**(2) 延迟刷写与内核工作线程**：`note_block_request_enqueue` 设置 `PENDING_FLUSH` 原子标志并注册 5ms 定时器，同步请求额外唤醒 `block-request-flush` 内核工作线程 <sup>[9](#mod-drivers-ref-9)</sup>。工作线程的 `poll_block_request_worker` 遍历所有已注册设备调用 `flush_blocks`，将脏数据写出。`block_request_timer_tick` 由平台定时器中断触发，检查截止时间并唤醒工作线程——这构成了"写缓冲 + 定期刷写"的简易 write-back 策略。

**(3) 设备注册与全局句柄**：`register_block_request_device` 将 `Arc<dyn BlockDevice>` 推入全局 `REGISTERED_DEVICES` 列表 <sup>[10](#mod-drivers-ref-10)</sup>。`BLOCK_DEVICE` lazy_static 则通过 `expect_first_block_device()` 获取首个注册设备作为根文件系统块设备——若未注册直接 panic <sup>[11](#mod-drivers-ref-11)</sup>，这是一条隐式的初始化顺序约束。

控制台侧则有另一条协同链路：`UartConsoleDevice` 实现 `IrqHandler`，其 `handle_irq` 中 drain UART FIFO 后唤醒 `input_wait` 和 `poll_wait` 等待队列，与 TTY 层的 `ConsoleTty` 配合处理 `ICRNL`/`ECHO` 等 termios 设置 <sup>[12](#mod-drivers-ref-12)</sup>。LoongArch 架构的特殊分支使用 `sleep_current_interruptible_until(1ms)` 轮询替代等待队列 <sup>[13](#mod-drivers-ref-13)</sup>，揭示了该平台中断子系统尚未完全就绪的状态。

### 4. 边角细节与不足

**1. `#[cfg]` 架构分支的覆盖率风险**：`console.rs` 中 `read` 方法通过 `#[cfg(feature = "arch-loongarch64")]` 分为轮询睡眠和等待队列两条路径 <sup>[13](#mod-drivers-ref-13)</sup>。非 LoongArch 架构走到 `prepare_current_interruptible_sleep` 的等待队列分支——但如果未来移植到未实现该函数的平台，会导致编译错误或运行时未定义行为。

**2. `unsafe` 块缺乏 SAFETY 注释**：`virtio_pci_blk.rs` 中大量 `unsafe` 块用于 MMIO 读写（`read_volatile`/`write_volatile`）、裸指针运算（`self.avail.add(2)`）和 `unsafe impl Send` <sup>[14](#mod-drivers-ref-14)</sup>，但全文件未出现任何 `// SAFETY:` 注释，安全契约完全隐式依赖程序员心智负担，属于显著技术债。

**3. 潜在的 TOCTOU 竞态**：`LoopDevice::binding_snapshot()` 在持锁期间克隆 `Arc<dyn KernelInode>` 后释放锁，随后 `read_binding_at` 通过该克隆 inode 进行读写 <sup>[15](#mod-drivers-ref-15)</sup>。若并发执行 `clear_fd` 解绑，读写操作可能作用到已释放的 inode 上（尽管 `Arc` 保证内存安全，但语义上可能读到过期数据）。

**4. 破坏性测试代码暴露**：`block_device_test()` 向前 512 块写入测试模式并回读校验，会覆盖块设备内容 <sup>[16](#mod-drivers-ref-16)</sup>。该函数仅标记 `#[allow(unused)]` 而未置于 `#[cfg(test)]` 或 feature gate 之后，任何链接该模块的代码均可调用。

**5. spin_loop 忙等无超时保护**：`PciVirtQueue::wait_used` 在 `used_idx` 不更新时无限 `spin_loop()` <sup>[3](#mod-drivers-ref-3)</sup>。若 virtio 设备故障或配置错误，该函数将永久占用 CPU 且持有 `SpinNoIrq` 锁，导致整个核无法响应中断，构成 DoS 入口。

**6. 错误吞没**：`flush_block_requests` 对每个设备的 `flush_blocks` 返回值使用 `let _ =` 显式丢弃 <sup>[17](#mod-drivers-ref-17)</sup>，刷写失败时上层完全无感知，可能在磁盘满或硬件故障时静默丢失数据。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块是内核向用户态暴露的 POSIX/Linux 兼容系统调用接口层, 按功能域拆分为 fs (文件/IO)、process (进程/内存/调度)、net (套接字)、signal (信号) 四个源文件. 核心抽象是每个 syscall 函数直接将 Linux 系统调用号映射为 Rust 函数, 遵循统一模式: 用 `copy_from_user`/`copy_cstr_from_user` 从用户空间提取参数, 校验后调用内部 crate (如 `crate::fs`, `crate::task`, `crate::mm`) 执行操作, 最终返回 `isize`(0 成功, 负 errno 失败). 与同家族基线相比, 本仓库的关键独特之处在于 (1) 大量使用 `const _: [(); N] = [(); size_of::<T>()]` 编译期 struct 大小断言保证 ABI 兼容, (2) `write_timespec_fast` 直接物理地址写入以优化时钟 syscall 路径, (3) 实现了 Linux 5.x+ 的 `openat2`/`clone3`/`statx` 等较新接口.

### 1. 整体架构与分域组织

系统调用层按功能域水平拆分为四个文件: `fs.rs`(~3470 行) 覆盖文件、目录、IO 多路复用, `process.rs`(~3301 行) 覆盖进程/线程生命周期、内存映射、调度与时钟, `net.rs`(~904 行) 覆盖 socket 操作, `signal.rs`(~473 行) 覆盖信号管理. 每个 syscall 函数遵循一套固定模式: 先从用户空间原始指针参数中通过 `copy_from_user` / `copy_cstr_from_user` 安全提取数据 <sup>[1](#mod-syscall-ref-1)</sup>, 然后调用 `crate::fs` / `crate::task` / `crate::mm` 等内部 crate 完成实际操作, 最终返回 `isize` — 0 表示成功, 负 errno 常量表示失败 <sup>[2](#mod-syscall-ref-2)</sup>.

这种 "薄封装" 设计意味着 syscall 层自身几乎不含业务逻辑: 权限检查在 VFS 层 (`fs.rs` 中 `has_permission` 等辅助函数实为对 `crate::fs` 状态的查询), 内存策略在 `mm` 子系统, 调度决策在 `task` 子系统. 该层的主要价值是 **Linux ABI 兼容性适配**: 将 Linux 风格的 flag 常量 (如 `AT_SYMLINK_NOFOLLOW`, `CLONE_VM`, `MAP_PRIVATE`) 翻译为内核内部枚举, 并在失败时把内部 `FsError`/`MsyncError` 映射到正确的 errno <sup>[3](#mod-syscall-ref-3)</sup><sup>[4](#mod-syscall-ref-4)</sup>.

### 2. 关键设计取舍

**ABI 兼容性断言.** 每个 `#[repr(C)]` 结构体后都紧跟编译期大小断言, 例如 `const _: [(); 128] = [(); size_of::<KStat>()]` <sup>[5](#mod-syscall-ref-5)</sup>. 这放弃了灵活性 (struct 字段顺序和填充必须严格匹配 Linux `struct kstat`), 换取用户态程序 (尤其是静态链接的 musl/glibc) 无需重编译即可正确解析内核返回的结构. 一旦 Linux 内核修改这些结构体, 此处必须同步更新.

**`write_timespec_fast` 优化.** `process.rs` 中 `sys_nanosleep`、`sys_clock_nanosleep` 等时钟 syscall 的 `rem` 参数写入路径使用了一个特化的 `write_timespec_fast` 函数 <sup>[6](#mod-syscall-ref-6)</sup>: 它通过虚拟地址直接查页表获取物理地址 (`page_table.translate_va(start)`), 然后直接解引用 `*dst = value`. 这绕过了标准 `copy_to_user` 的权限检查和跨页处理, 仅在精确对齐且单页内时生效 (fallback 到标准路径). 这是一个为高频时钟 syscall 做的微优化, 代价是维护了两套用户空间写入逻辑.

**`/proc/self/fd/` 特殊处理.** `fs.rs` 中的 `proc_self_fd_number` 函数硬编码了对 `/proc/self/fd/` 路径的解析 <sup>[7](#mod-syscall-ref-7)</sup>, 在 `fchmodat`/`fchownat` 等操作中自动转换为对 fd 的直接操作. 这并非完整的 procfs 实现, 而是一个最小化 workaround — 允许 shell 重定向 (`>/dev/stderr`) 等常见操作正常工作, 但不支持其他 `/proc/` 路径.

**poll/select 与 smoltcp 的协作.** `fs.rs` 中 poll/ppoll/select 的实现引入了一个 `SOCKET_EVENT_POLL_MS: usize = 10` 常量 <sup>[8](#mod-syscall-ref-8)</sup>, 因为底层 smoltcp 网络栈需要周期性轮询来驱动状态机. 当检测到 fd 集合中包含 socket 时 (`file.as_socket_file().is_some()`), poll 等待循环会设置一个 10ms 的超时唤醒点, 而非无限期睡眠. 这体现了用户态网络栈集成时的一个典型妥协.

### 3. 跨模块协同

**文件系统 syscall → VFS 层.** `fs.rs` 中的 `sys_openat`/`sys_read`/`sys_write` 等并不直接操作磁盘, 而是通过 `File` trait 和 `KernelInode` trait 调用 VFS 层. 关键流程: `current_cwd_and_fd_table_snapshot()` 获取当前任务的工作目录和 fd 表快照 <sup>[9](#mod-syscall-ref-9)</sup>, `file_from_fd()` 根据 fd 和读写需求取出 `Arc<dyn File>`, 然后调用 `file.read(buf)` 等多态方法. 这种设计使 socket、pipe、普通文件统一通过 `dyn File` 接口处理, 也是 `sys_read` 等函数中需要特判 `file.as_socket_file()` 的原因 (socket 的 `read` 语义与普通文件不同).

**内存映射 syscall → MM 层.** `sys_mmap` 的实现展示了典型的跨层协同 <sup>[10](#mod-syscall-ref-10)</sup>: 先解析 `prot`/`flags` 为 `MapPermission` 和 `VmaMapFlags`, 然后通过 `task.thread_group.address_space()` 获取地址空间锁, 调用 `inner.memory_set.mmap_user_range()` 建立映射, 成功后调用 `sync_mmap_to_thread_group()` 将映射信息同步到线程组的其他成员 — 这确保了多线程进程的地址空间一致性.

**信号 syscall → signal 子系统 + arch 层.** `sys_rt_sigreturn` 是最复杂的跨模块协同案例 <sup>[11](#mod-syscall-ref-11)</sup>: 它从 trap 帧中提取用户栈指针 (`frame_sp`), 通过 `arch::signal::read_restored_context()` 读取架构相关的信号上下文, 恢复寄存器 (`cx.x = restored.regs`)、PC (`cx.sepc = restored.pc`)、信号掩码, 并调用 `arch::trap::signal_return_value()` 设置返回值. 这个流程涉及 `task` (获取 trap 上下文)、`arch::trap` (读取/写入寄存器)、`arch::signal` (平台相关的 signal frame 布局) 三个模块的精密配合.

### 4. 边角细节与不足

**<sup>[12](#mod-syscall-ref-12)</sup> `#[allow(dead_code)]` 与半成品接口.** `sys_syslog` 函数标记了 `#[allow(dead_code)]`, 但其内部调用的 `copy_klog_to_user` 同样带有此标记. 这表明 syslog 接口可能未被系统调用表注册, 属于预留但未完全集成的代码.

**<sup>[13](#mod-syscall-ref-13)</sup> 硬编码的 TCP 默认 MSS.** `net.rs` 中 `TCP_DEFAULT_MAXSEG: i32 = 1460` 是写死的, 而非从网络接口 MTU 动态计算. 在非以太网场景 (如 loopback、VPN 隧道) 中, 这会导致 TCP 分段大小不最优.

**<sup>[14](#mod-syscall-ref-14)</sup> `sys_socketpair` 完全未实现.** 对 `AF_UNIX`、`AF_INET`、`AF_INET6` 一律返回 `EOPNOTSUPP`, 但 `AF_UNIX` 域其实是 `validate_socket_args` 中唯一支持的 `SOCK_STREAM` protocol 0 的域. 这意味着 Unix domain socket 的 `socket()` 可创建, 但 `socketpair()` 不可用 — 两者支持度不对称.

**<sup>[15](#mod-syscall-ref-15)</sup> `sys_sendmsg`/`sys_recvmsg` 返回 `EOPNOTSUPP`.** scatter-gather IO 的缺失意味着 `sendmsg`/`recvmsg` 系列调用完全不可用, 影响依赖 `sendmsg` 传递 ancillary data (如 `SCM_RIGHTS`) 的应用程序.

**<sup>[16](#mod-syscall-ref-16)</sup> `UNSUPPORTED_CLONE_FLAGS` 覆盖了所有 namespace 相关 flag.** 包括 `CLONE_NEWNS`、`CLONE_NEWUSER`、`CLONE_NEWPID` 等, 这意味着容器运行时 (Docker/containerd) 无法在此内核上运行. 这是有意的范围裁剪, 但对期望完整 Linux 兼容性的用户是一个显著限制.

**<sup>[17](#mod-syscall-ref-17)</sup> `sys_rt_sigqueueinfo` stub.** 直接返回 `ENOSYS`, 无实际实现. 这影响需要携带附加数据的信号发送 (如 `sigqueue()` 的 `si_value`).

**<sup>[18](#mod-syscall-ref-18)</sup> `openat2` resolve flag 的 `RESOLVE_IN_ROOT` 处理有简化.** 当设置 `RESOLVE_IN_ROOT` 且路径以 `/` 开头时直接返回 `ENOENT` [ref:18_src], 而非 Linux 的 "假装 root 为基目录" 语义. 这是一个 breaking difference.

**<sup>[19](#mod-syscall-ref-19)</sup> `pselect6` 的信号掩码原子替换借用 `replace_signal_mask`.** 该函数在 poll 等待前替换信号掩码, 完成后恢复. 但如果 poll 过程被信号中断且 handler 内部修改了掩码, `restore_signal_mask` 会覆盖 handler 的修改 — 这是 POSIX 要求的行为, 但值得在代码中标注.

### ⚠ 开放问题

- sys_socketpair 对所有 domain 返回 EOPNOTSUPP, 但 AF_UNIX 域 socket() 已支持, 两者支持度不对称.
- sys_sendmsg/sys_recvmsg 均为 EOPNOTSUPP stub, 缺失 scatter-gather IO 和 ancillary data 传递.
- openat2 RESOLVE_IN_ROOT 以 '/' 开头的路径直接返回 ENOENT 而非重基到 root, 与 Linux 语义不一致.
- TCP_DEFAULT_MAXSEG 硬编码为 1460, 在非以太网 MTU 场景下 TCP 分段大小可能不最优.
