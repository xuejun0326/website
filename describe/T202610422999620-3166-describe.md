## 📖 一、总览

该仓库是 rCore-Tutorial 家族中一个实验性内核实现，定位介于教学型与 Linux 兼容型之间。它保留了 rCore-Tutorial 的微内核教学抽象（如 `MemorySet`、`TaskControlBlock`），但大幅扩展了系统调用覆盖（217 项）和模块深度，试图在简化教学的同时兼容更多 POSIX 和 Linux 语义。

最值得注意的三个特点：
1. **统一进程/线程模型与三层优先级调度器**：`TaskControlBlock` 不区分进程与线程，通过 `CLONE_THREAD` 决定资源共享；调度器实现 RT 100 级、普通 40 级、Idle 三层队列，使用 bitmap 实现 O(1) 取最高优先级任务 [task.ref:3][task.ref:4]。
2. **广泛的系统调用骨架但实现深度参差**：覆盖编号 5–447 共 217 个 syscall，包含 `io_uring`、`fspick`、`pidfd_open` 等现代 Linux 扩展，但大量仅返回 `ENOSYS` 或简单 stub；核心路径如 `fork/execve/futex/epoll` 则完整实现 [syscall.ref:1][task.ref:5]。
3. **信号机制的精细模拟与 musl 兼容 hack**：实现了普通帧与 RT 帧分离、完整信号递送循环，并引入 `is_disabled_musl_sigcancel` 硬编码 tp-152 偏移来绕过 musl libc 的信号自取消特性 [signal.ref:1][task.ref:17]。

## 🎯 二、综合评价

### 整体定位
该仓库是 rCore-Tutorial 家族中一个**教学型但力求 Linux 兼容的实验性内核**。与原始 rCore-Tutorial（单 crate、简单 RR 调度、基本文件系统）相比，它显著扩展了模块划分和系统调用覆盖，但并未追求完整 Linux 兼容（如缺少 SA_RESTART、core dump、完整 IPC 和网络协议栈）。其架构核心仍保持教学清晰性：两级启动、大 match 分发 syscall、基于 trait 的 VFS 和设备驱动。这种定位使得代码风格介于教学和产品之间——既有清晰的抽象（如 `MemorySet`、`TaskControlBlock`），又有大量 `unsafe` 块和全局锁等早期优化痕迹。

### 真正的创新点
本仓并非从头设计，但在 rCore-Tutorial 基础上引入了若干值得注意的设计：
1. **三层优先级调度器**：与基线简单的 RR 不同，实现了 100 级 RT + 40 级 Nice + Idle 三层，使用 bitmap 和 `leading_zeros` 实现 O(1) 取高优先级任务 [task.ref:4]。虽然硬件支持不完全（如硬编码双核），但设计本身复杂度远超教学要求。
2. **信号帧结构分离**：普通帧仅存寄存器，RT 帧包含额外 `siginfo`，通过 `FrameFlags` 标记辅助 `sigreturn`，减少了内核栈压力 [signal.ref:1]。
3. **musl 兼容 hack**：`is_disabled_musl_sigcancel` 硬编码线程指针偏移，是一种非常规但有效的 ABI 适配手段 [task.ref:17]。此类 hack 在教学内核中罕见，体现了对用户态行为的深层次关注。
4. **共享匿名映射立即分配策略**：在 `mm` 中，`MAP_SHARED | MAP_ANONYMOUS` 立即分配物理页而非惰性，简化了 COW 但增加了内存使用 [mm.ref:5]。

### 取舍判断
该仓的精力分配明显偏向**系统调用覆盖的广度**和**调度/信号等“内核感”强的模块**，而**文件系统和 IPC 的深度与正确性**则相对薄弱。例如：
- syscall 层实现了 217 个入口，但许多仅返回 `ENOSYS` 或直接 `panic!` [syscall.ref:1]。
- ext4 文件系统通过全局锁串行化所有操作，并发读性能差；procfs inode 编号冲突（`PROC_CPUINFO_INO` 与 `PROC_SELF_FD_INO` 均为 12）[fs.ref:0]。
- IPC 的共享内存段无显式清理机制，匿名管道缺少非阻塞和超时 [ipc.ref:2]。
- 网络模块中 `get_ephemeral_port` 在端口耗尽时 `panic!`，且 IPv6 静默降级为 IPv4 [net.ref:3][net.ref:4]。
这些取舍表明开发者的主要目标是跑通基本 Linux 用户态（尤其是 musl libc），而非构建健壮的生产级内核。

### 完成度与不足
**完成度 (2/3)**：核心路径（进程创建、调度、信号、基本文件读写、TCP 通信）完整可运行，能支持 musl libc 编译的 busybox、nginx 等简单 workload。但大量 syscall 是 stub，且 `epoll`、`signalfd`、`timerfd` 等重要异步机制未实现或仅占位。
**创新性 (2/3)**：调度器分层、信号帧分离、musl hack 等设计在教学系列中有一定原创性，但整体架构仍继承自 rCore-Tutorial 和 ArceOS，底层内存管理、驱动框架等未有根本突破。
**代码质量 (1/3)**：多处 `unsafe` 块缺乏 `SAFETY` 注释，存在明显的竞争窗口（如 `shared_file_frame` 双重检查 [mm.ref:4]），全局锁粒度不合理（EXT4_OP_LOCK 串行化所有 ext4 操作 [fs.ref:2]），以及 `tid` 永不回收、`switch_to_next_task_after_exit` 在空队列时 `spin_loop` 等低级隐患 [task.ref:15][task.ref:20]。代码可读性尚可，但健壮性明显不足以投入实际使用。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块采用两级初始化抽象（rust_main → rust_main_high），通过条件编译分离 loongarch64 与 riscv64 的早期启动路径。核心职责包括 BSS 清零、页表启用、地址空间切换（仅 loongarch64）以及按精确依赖顺序调用 trap、mm、net、task 等子系统初始化。与同家族基线（ArceOS）相比，本实现显式处理了 LoongArch 的两级跳转，并依赖编译期嵌入 link_app.S 来绑定用户程序，而非运行时扫描。

### 1. 核心抽象与两级启动模型

启动流程模块的核心抽象是**两级初始化**模式：`rust_main` → `rust_main_high`。`rust_main` 是 Rust 重写的内核入口，由汇编引导代码（如 `entry.asm`）跳转而来，负责完成架构依赖的早期初始化；`rust_main_high` 则执行平台无关的内核子系统初始化，最终调用 `task::run_tasks()` 进入调度循环。

对于 loongarch64 架构，两级启动是**必要**的：在 `rust_main` 中调用 `arch::enable_boot_paging()` 启用启动页表，再通过 `arch::jump_to_high_half()` 跳转到高地址空间执行 `rust_main_high` <sup>[1](#mod-boot-ref-1)</sup>。这反映了 LoongArch 地址空间设计的特殊性——内核启动时位于物理地址空间，必须通过页表切换到虚拟地址空间，且跳转过程需要在关闭中断和缓存一致性的状态下完成。相比之下，riscv64 架构直接调用 `rust_main_high()` <sup>[1](#mod-boot-ref-1)</sup>，因为 RISC-V SBI 启动后 CPU 已经处于虚拟地址空间或直接映射区域，无需额外切换。

这种两级抽象的设计代价是**架构耦合**：`rust_main` 内部的条件编译分支必须为每种架构显式处理早期初始化。如果未来新增 x86_64 或 aarch64 支持，开发者需要理解每个架构的地址空间特性并添加对应的 `#[cfg]` 分支。

### 2. 关键设计取舍

**取舍 1：手动 BSS 清零 vs 通用 memset。** `clear_bss()` 函数为 loongarch64 使用内联汇编 `st.d`/`st.b` 指令逐字清零，为 riscv64 使用 volatile 写操作迭代清零 <sup>[3](#mod-boot-ref-3)</sup>。设计者选择内联汇编而非调用 `memset`，原因在于启动初期栈和全局函数调用可能尚不可靠——`memset` 本身可能依赖栈或全局状态。但这也意味着代码维护成本增加：每新增一个架构就必须添加对应的清零实现，且 loongarch64 的汇编实现使用了 `options(nostack, preserves_flags)` 约束，要求编译器确保不生成栈访问且不破坏标志位——这对优化器是一个严格的限制，也增加了误用风险。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `MemorySet` 为核心抽象，管理进程虚拟地址空间，由 `MapArea` 列表和 `PageTable` 构成。物理页帧通过 `FrameTracker` RAII 包装，使用栈式分配器回收。内核采用恒等线性映射（偏移 `KERNEL_BASE`）。用户态拷贝通过页表翻译直接读写物理页帧，避免未映射虚拟地址的解引用风险。与 ArceOS 基线相比，本仓库新增了 brk 堆的惰性 VMA 创建、共享匿名映射的立即分配策略（非完整 shmem），以及 COW fork 时父进程 PTE 的原子性处理。

### 1. 核心抽象与外部依赖

内存管理模块建立在四层抽象之上：**地址类型**（`VirtAddr`/`PhysAddr`/`VirtPageNum`/`PhysPageNum`，定义于 `address.rs`）、**物理帧分配器**（`FrameTracker` + `StackFrameAllocator`，定义于 `frame_allocator.rs`）、**内核堆分配器**（基于 `buddy_system_allocator`，定义于 `heap_allocator.rs`）、以及**地址空间**（`MemorySet` + `MapArea`，定义于 `memory_set.rs`）。

`MemorySet` 持有一个 `PageTable` 和一个 `Vec<MapArea>`。`MapArea` 记录一段连续的虚拟页范围（`VPNRange`）、映射类型（`MapType`）、权限（`MapPermission`）、是否为共享/文件映射，以及已分配的物理帧（`data_frames: BTreeMap<VirtPageNum, Arc<FrameTracker>>`）。这种设计将页表与 VMA 分离：页表负责硬件翻译，`MapArea` 负责软件层面的 VMA 管理（权限检查、惰性分配、COW 等）。内核地址空间 `KERNEL_SPACE` 采用全局 `lazy_static` 构造，使用 `Arc<Mutex<MemorySet>>` 实现跨进程共享（所有用户进程通过内核态切换访问同一内核映射）<sup>[3](#mod-mm-ref-3)</sup>。

物理帧分配器 `StackFrameAllocator` 维护一个连续区间 `[current, end)` 和回收栈 `recycled`。`FrameTracker` 在 `drop` 时自动回收帧，利用 Rust 所有权机制杜绝内存泄漏<sup>[10](#mod-mm-ref-10)</sup>。但 `recycled_bitmap` 的长度被初始化为 `r.0`（即物理页总数），对于大内存配置可能造成显著的静态内存开销<sup>[9](#mod-mm-ref-9)</sup>。

### 2. 关键设计取舍

**2.1 用户态拷贝绕过虚拟地址缺页**

`mod.rs` 中的 `copy_user_bytes_to_kernel` 和 `copy_kernel_bytes_to_user` 没有直接解引用用户虚拟地址，而是通过 `MemorySet` 的页表逐页翻译到物理页帧（`PTE → ppn → get_bytes_array`），再执行拷贝<sup>[7](#mod-mm-ref-7)</sup>。这一设计的动机是：用户页可能是惰性分配的（匿名 mmap），虚拟地址上尚无物理页映射，直接解引用会触发 kernel page fault，而内核缺页处理复杂且可能睡眠。通过绕行物理帧，避免了映射延迟，但也带来了每次拷贝至少查一次页表的性能开销。拷贝前还会调用 `ensure_user_page_access` 确保对应页已分配，这隐含了物理帧的预分配。

**2.2 共享匿名映射的早期分配**

`insert_shared_framed_area_va` 注释明确标注 `TODO[ABI-COMPAT]`：当前实现立即为共享匿名映射分配全部物理页，而不是采用 Linux 的 `shmem/tmpfs` 语义<sup>[5](#mod-mm-ref-5)</sup>。原因是仓库的 fork 实现使用 COW 机制，如果共享页是惰性分配的，父子进程会各自缺页出不同帧，破坏共享语义。早期分配牺牲了启动时内存效率，但保证 fork 后共享正确。这是典型的功能完整性优先于性能的取舍。

**2.3 恒等内核映射与地址掩码**

`address.rs` 中的 `From<usize>` 实现通过掩码 `& ((1 << PA_WIDTH) - 1)` 截断地址，而非 `assert` 检查有效性<sup>[1](#mod-mm-ref-1)</sup>。注释掉的旧实现（带 `assert`）表明开发者曾考虑严格校验，但因性能或灵活性放弃了。同时 `VirtAddr → usize` 转换使用符号扩展（`(((v.0 << shift) as isize) >> shift) as usize`），正确处理 Sv39 高地址 canonical 需求。`PhysAddr::kernel_addr` 在 loongarch64 下动态判断是否已启用分页，而 riscv64 下恒加 `KERNEL_BASE`，反映了两种架构的启动流程差异。

**2.4 栈式分配器的回收位图**

`StackFrameAllocator::dealloc` 使用 `recycled_bitmap` 检测重复释放，但位图被分配为 `vec![false; r.0]`，其大小等于物理页总数<sup>[9](#mod-mm-ref-9)</sup>。在 512MB 内存、4KB 页的场景下，位图占用 131072 字节（约128KB），尚可接受；但若未来支持大内存（如 64GB），位图将膨胀至 16MB，应考虑压缩位图或动态分配。

### 3. 跨模块协同

**3.1 与 Trap 处理的缺页协作**

`MemorySet::handle_page_fault` 是缺页中断的入口（由 `trap` 模块调用）。它根据 `stval` 定位 `MapArea`，处理 COW（复制物理帧并清写位）和惰性分配（调用 `frame_alloc` 并映射）。该函数与 `copy_user_bytes_to_kernel` 形成闭环：后者先通过 `ensure_user_page_access` 确保页存在（可能触发缺页的预分配），而缺页处理则负责运行时填充。

**3.2 与 Syscall 的 mmap 接口**

`MemorySet::mmap_area` 是 `mmap` 系统调用的底层实现。它接受 `MmapBacking` 枚举（lazy/共享/文件等），通过 `prepare_mmap` 选择地址并处理重叠，然后根据 backing 类型执行不同映射策略。特殊地，`MmapBacking::SharedFrames` 允许绕过页表直接插入预分配的帧数组，用于共享内存 attach 操作。`mmap_file_backing` 函数在 `mod.rs` 中被 `pub(crate)` 导出，供文件映射路径调用[ref:?]。

**3.3 与任务模块的指针检查**

`check_user_buffer` 和 `copy_cstr_from_user` 等函数通过 `current_task().op_memory_set_write` 获取当前任务的地址空间，执行权限验证。这种设计将用户指针的安全检查与页表操作绑定，避免了从任务外部直接操作 `MemorySet` 可能引入的竞争。

### 4. 边角细节与不足

**4.1 unsafe 块缺乏安全契约**

`address.rs` 中 `PhysAddr::get_mut`、`PhysPageNum::get_bytes_array` 等函数通过 `kernel_addr` 计算虚拟地址后直接生成 `&'static mut` 引用，没有任何 `// SAFETY:` 注释<sup>[2](#mod-mm-ref-2)</sup>。调用者需自行保证：1) 该物理地址对应的虚拟地址已被映射；2) 没有别名引用；3) 生命周期为 `'static` 不会超过对象实际存活时间。在当前内核线性映射的上下文中这些条件成立，但无注释意味着未来维护者易引入 UB。

**4.2 shared_file_frame 的竞争窗口**

`shared_file_frame` 函数先从全局缓存 `SHARED_FILE_PAGES` 查找，未命中则分配新帧并再次检查后插入<sup>[4](#mod-mm-ref-4)</sup>。两个锁操作非原子：在第一次解锁和第二次加锁之间，另一线程可能已插入相同键的帧。当前代码通过 `Arc::new` 和 `downgrade` 保证了最终一致性，但会浪费一个短期分配的帧（随后被丢弃回收）。更严谨的做法是使用 `BTreeMap::entry` API 或外部传入的锁。

**4.3 全局 FRAME_ALLOCATOR 锁的中断安全**

`FRAME_ALLOCATOR` 是普通的 `spin::Mutex`，不支持中断嵌套。若在中断上下文（如缺页处理中）调用 `frame_alloc` 而此时锁已被占（例如被同一 CPU 的中断低优先级路径持有），将导致死锁。当前 `handle_page_fault` 确实调用了 `frame_alloc`，但缺页中断通常未嵌套另一种分配路径；然而，在支持优先级中断的架构上存在隐患。

**4.4 brk 堆的惰性 VMA 创建效率**

`ensure_private_writable_anonymous_range` 在调整堆大小时会逐双遍历当前 VMA 并插入缺失区间，最后调用 `merge_adjacent_private_anonymous_areas` 合并。该合并操作每次都对 `areas` 排序（`sort_by_key`），时间复杂度 O(n log n)。若进程频繁调用 `sbrk` 微调堆尾，此行为将退化。

**4.5 未完成的共享文件页缓存**

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以 `TaskControlBlock` 为核心抽象（类似 Linux `task_struct`），统一表示进程与线程，通过 `CLONE_THREAD` 标志区分创建语义。调度器实现三层优先级队列（RT 100 级、普通 40 级、Idle），使用 bitmap 实现 O(1) 取最高优先级任务。futex 模块完整支持 WAIT/WAKE/REQUEUE/CMP_REQUEUE/WAIT_BITSET 及超时和信号中断。相比 rCore-Tutorial v3，本仓的特点是：(1) 调度策略从简单 RR 扩展为完整 POSIX RT+Nice 分层；(2) 新增 `interruptible`/`interrupted` 原子标志协调信号与阻塞系统调用；(3) 内核栈 slot 回收复用 TLB 热地址；(4) 信号投递包含 musl libc 特定 hack。

### 1. 核心抽象：TaskControlBlock — 统一进程/线程描述符

本模块的支柱是 `TaskControlBlock` <sup>[1](#mod-task-ref-1)</sup>，一个 50+ 字段的结构体，其设计理念直接继承 Linux 的 `task_struct`：**不区分进程与线程，仅通过资源共享程度来定义关系**。`tgid`（线程组 ID）标识进程，`tid`（线程 ID）标识可调度实体；当 `CLONE_THREAD` 被设置时，clone 创建的是同一进程内的新线程（共享 `thread_group`、`memory_set`、`fd_table`、`sig_handler`），否则创建独立进程（COW 地址空间、拷贝 fd_table、新线程组）<sup>[7](#mod-task-ref-7)</sup>。

这一取舍的直接后果是：`TaskControlBlock` 的体积膨胀到约 100 字节（仅字段指针和原子变量），每个线程都持有完整的进程级资源引用（`Arc<RwLock<MemorySet>>`、`Arc<SpinLock<SigHandler>>`），而非分层结构（如 seL4 的 TCB 分离架构）。好处是访问路径短——线程切换无需间接寻址进程结构；坏处是每个线程内存开销大（含内核栈 + slot 元数据），且 `Arc` 引用计数在 fork 执行 exec 时仍需判断 `strong_count == 1` 才释放页 <sup>[19](#mod-task-ref-19)</sup>。

`TaskControlBlock` 的 `#[repr(C)]` 布局 <sup>[1](#mod-task-ref-1)</sup> 要求第一个字段必须是 `kernel_stack: KernelStack`，这是因为汇编层 `__switch` 通过 TCB 指针直接偏移访问内核栈顶 <sup>[12](#mod-task-ref-12)</sup>。这是一个编译期契约：若未来添加字段在 `kernel_stack` 之前，会静默破坏上下文切换。

### 2. 调度器设计：三层优先级 + bitmap O(1) 取队首

调度器 `Scheduler` <sup>[3](#mod-task-ref-3)</sup> 维护三组队列：(1) RT 队列 100 级（对应 `SCHED_FIFO`/`SCHED_RR` 优先级 1-99）；(2) Normal 队列 40 级（nice -20..19 映射为 0..39）；(3) Idle 队列。使用 `u128` 和 `u64` bitmap 标记非空队列，`fetch()` 时通过 `leading_zeros()`（RT，大索引高优先级）和 `trailing_zeros()`（Normal，小索引高优先级）实现 O(1) 取最高优先级任务 <sup>[4](#mod-task-ref-4)</sup>。

调度策略的语义遵循 POSIX 但做了简化：`SCHED_FIFO` 任务运行至阻塞或主动 yield，`SCHED_RR` 在时钟中断 `preempt_current_task()` 时被放回同优先级队尾轮转 <sup>[5](#mod-task-ref-5)</sup>；`SCHED_IDLE` 仅在 RT 和 Normal 队列全空时才会被调度。`SchedState` 全部使用 `Atomic*` 实现无锁访问 <sup>[6](#mod-task-ref-6)</sup>，但这是一个**伪无锁设计**：`from_parent()` 读取多个字段时未用快照锁，子进程可能在 fork 时观察到中间状态（例如 `policy` 已更新但 `priority` 未更新）。目前单核假设下竞态窗口极窄，多核扩展时需引入 seqlock。

`DEFAULT_CPU_AFFINITY_MASK = 0b11` <sup>[2](#mod-task-ref-2)</sup> 硬编码暗示双核目标机器。当运行在单核或 4+ 核平台时，该掩码不会造成错误（`AtomicUsize` 无范围校验），但 `sched_setaffinity` 的校验逻辑仅在 `capable(CAP_SYS_NICE)` 时执行。

### 3. 跨模块协同：信号中断 + futex 等待的可打断机制

futex 等待路径与信号模块的协同是本模块最精巧的部分。futex 引入了两个原子标志 `interruptible` 和 `interrupted` <sup>[8](#mod-task-ref-8)</sup> 来替换传统的“检查 pending 信号”模式。其工作流程如下：

1. `futex_wait_common()` 在持有 `FUTEX_QUEUES` 锁的临界区内，先 `read_futex_value` 比较期望值，再将当前线程的 `FutexQ` 入队，然后调用 `prepare_current_task_blocked()` 将任务标记为 blocked <sup>[10](#mod-task-ref-10)</sup>。
2. 关键时序保护：`set_interruptible(true)` **必须在入队之前执行** <sup>[9](#mod-task-ref-9)</sup>，防止信号在入队和阻塞之间的窗口到达丢失唤醒。
3. 信号到达时，`receive_siginfo()` 不仅将信号加入 pending 队列，还检查 `check_signal_interrupt()`，若返回 true 则设置 `interrupted = true` 并调用 `wakeup_task()` <sup>[14](#mod-task-ref-14)</sup>。
4. 被唤醒的 futex 等待者醒来后检查 `is_interrupted()`，若为 true 则负责将自己从 `FUTEX_QUEUES` 中移除并返回 `EINTR` <sup>[11](#mod-task-ref-11)</sup>。

这种设计相比 rCore-Tutorial v3 的“在 syscall 入口处统一检查信号”更精细：它允许阻塞操作在等待中途被信号中断，而不是仅在下一次进入内核时检查。代价是每个阻塞站点都必须手动维护 `interruptible`/`interrupted` 的配对——若有路径忘记在阻塞前设置 `interruptible=true`，信号将永远无法中断该阻塞。

futex 的 key 构造 <sup>[13](#mod-task-ref-13)</sup> 也体现了跨模块协同：对于非私有 futex，先通过 `memory_set.shared_futex_key()` 尝试获取基于物理页面的共享 key（支持跨进程共享），回退到进程内 `(tgid, uaddr)`。这里隐含的假设是 `shared_futex_key` 当前仅对匿名映射实现，文件映射的 futex 会退化到进程内可见。

### 4. 边角细节与不足

**<sup>[15](#mod-task-ref-15)</sup> TID 不复用**：`TidHandle::drop` 是一个空实现，附 TODO 说明复用 TID 会与 futex wakeup 路径中的 tid 索引产生竞态。这意味着长时间运行的系统的 tid 会单调递增直至 usize 溢出（约 180 亿年，理论安全），但调试时 tid 不再反映活跃线程数，`/proc` 直觉会失效。

**<sup>[16](#mod-task-ref-16)</sup> 退出路径差异**：`task_exit` 的注释明确写道“当前退出线程组主线程时会退出整个线程组，与 Linux 实现存在差异”。Linux 允许线程组主线程退出后，其他线程继续运行（以 tgid 不变为代价），但当前实现直接调用 `task_group_exit` 杀掉所有线程。这会破坏期望主线程可先退出的 pthread 程序。

**<sup>[17](#mod-task-ref-17)</sup> 信号投递的 musl hack**：`is_disabled_musl_sigcancel` 函数硬编码了 musl libc 内部数据结构偏移（`tp - 152` 处的 `cancel_state` 字节）<sup>[18](#mod-task-ref-18)</sup>。这是一个脆弱且不可移植的依赖：不同 musl 版本或非 riscv64/loongarch64 架构的偏移量可能不同，且该代码针对信号 33（SIGCANCEL）做了特殊豁免。若未来切换 libc 实现，此路径需彻底重写。

**<sup>[20](#mod-task-ref-20)</sup> 无任务时的忙等**：`switch_to_next_task_after_exit()` 在就绪队列为空时执行 `core::hint::spin_loop()` 而非进入 WFI 休眠。由于该路径仅在最后一个任务退出时触发（此时系统应关机），忙等功耗问题不大，但若 `check_all_task_timers()` 返回后仍无任务，将造成 CPU 满载空转。

**<sup>[21](#mod-task-ref-21)</sup> 调度器选型局限**：当前的 bitmap 多级队列算法（类似 Linux O(1) 调度器）缺少 CFS 的虚拟运行时公平性。两个同 nice 值的 CPU 密集型任务在时间片分配上完全依赖 RR 轮转，没有考虑实际运行时间累积。对于交互型负载（短阻塞 + 短运行），`rt_priority` 需要用户手动调优。

**<sup>[22](#mod-task-ref-22)</sup> `prepare_current_task_blocked()` 的失败路径**：当就绪队列为空时该函数直接返回 false，调用者（如 futex wait）必须回退处理。futex 的 `futex_wait_timed_common` 在 `!prepare_current_task_blocked()` 失败后调用 `yield_current_task()` 并 retry <sup>[23](#mod-task-ref-23)</sup>，这是个合理但高昂的回退——它可能导致 futex 在系统负载低时频繁 yield-spin。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 实现了一个 Linux 兼容的 VFS 层，核心抽象是 `InodeOp` trait（统一 ext4 和 procfs）和 `FileOp` trait（文件操作接口）。路径解析器 `namei.rs` 模拟 Linux 的 `path_openat` 语义，支持符号链接跟随（上限 40 层）、挂载点穿越、sticky bit 检查和 glibc 库路径重映射。ext4 后端包装 `lwext4_rust` C 库并通过全局 `EXT4_OP_LOCK` 序列化所有操作；procfs 完全在内存中构造，使用静态 inode 编号分配。与基线 ArceOS 相比，此仓显著扩展了 procfs 深度（`/proc/sys/net/ipv4/conf/lo/tag` 等多层目录）、增加了 `glibc_default_lib_alias` 路径透明替换、以及 `orphan_regular_file` 机制处理已删除但仍打开的文件。

### 1. 核心抽象与外部依赖

文件系统模块以两个 trait 为骨架：`InodeOp` 和 `FileOp`。`InodeOp` 定义 inode 级操作（`lookup`, `readdir`, `read_at`, `write_at`, `stat`, `create`, `link`, `unlink`, `symlink` 等），由 ext4 真实文件系统和 procfs 伪文件系统各自实现。`FileOp` 定义打开文件后的操作（`read`, `write`, `seek`, `get_stat`, `fsync`, `truncate`），由 `File` 结构体实现，后者内部持有 `Arc<dyn InodeOp>` 和 `Mutex<FileInner>`（含偏移、标志、页缓存引用、时间戳覆盖等状态）。`Path` 结构体（`Arc<VfsMount>` + `Arc<Dentry>`）作为路径解析的结果在系统调用之间传递。

外部依赖方面，ext4 后端通过 `lwext4_rust` 调用 C 库 `lwext4` 的 FFI 接口，所有 ext4 操作都受全局 `EXT4_OP_LOCK: Mutex<()>` 保护 <sup>[2](#mod-fs-ref-2)</sup>。该锁确保底层 C 库的状态不会因并发而损坏，但将所有 ext4 操作（包括只读的 `readdir` 和 `lookup`）串行化，是明显的性能瓶颈。`Ext4Inode` 实现了 `Send` 和 `Sync`（通过 `unsafe impl`），其安全性完全依赖 `EXT4_OP_LOCK` 的持有 <sup>[13](#mod-fs-ref-13)</sup>。

### 2. 关键设计取舍

**2.1 路径名解析：重走 Linux 老路而非简化**  
`namei.rs` 的 `resolve_path_from` 函数模拟了 Linux 内核的 `path_openat` 逻辑：使用 `Nameidata` 结构体维护 `mnt` + `dentry` + `path_segments` + `depth`，循环逐层遍历。对 `..` 的处理会调用 `follow_dotdot` 检查是否跨挂载点边界 <sup>[5](#mod-fs-ref-5)</sup>；对中间符号链接会递归展开（上限 `MAX_SYMLINK_FOLLOWS=40` <sup>[1](#mod-fs-ref-1)</sup>）；对最后一级的跟随行为可通过 `follow_final_symlink` 和 `follow_final_mount` 两个布尔参数控制，从而复用同一核心为 `openat`、`lstat`、`umount` 等不同语义服务。这种设计的代价是代码复杂度高（`resolve_path_from` 约 80 行），但换来了与 Linux 应用程序的精确兼容。

**2.2 全局 DentryCache 与 Mutation Lock**  
`lookup_dentry` 先查全局 DentryCache（由 `dentry_cache` 模块管理），未命中才调文件系统 `lookup` <sup>[1](#mod-fs-ref-1)</sup>。缓存命中可避免重复的目录项遍历 I/O。然而在 `open_last_lookups` 的 O_CREAT 路径中，使用了 `NAMEI_MUTATION_LOCK: Mutex<()>` 保护整个创建流程 <sup>[1](#mod-fs-ref-1)</sup>：先持锁，再二次检查 ENOENT 以避免 TOCTOU 竞态。这个锁粒度较粗，但在多线程高频创建文件的场景下会成为瓶颈。

**2.3 ext4 的合成 inode 与孤儿文件处理**  
`Ext4Inode` 引入了一个特殊设计：`synthetic_created_inode`（`ino >= CREATED_INODE_BASE = 1<<62`）<sup>[8](#mod-fs-ref-8)</sup>。当通过 `O_TMPFILE` 等机制创建无名文件时，inode 编号从高位分配，`stat`, `set_times`, `set_owner` 等操作完全在内存缓存中完成，不写回磁盘。对应的 `orphan_regular_file` 机制将已 unlink 但仍打开的文件重命名为 `{old_path}.respos_orphan_{ino}` <sup>[7](#mod-fs-ref-7)</sup>，保留磁盘数据直到最后关闭后清理。这与 Linux 内核的 `dentry_open` + `inode->i_nlink` 语义不同：Linux 在 unlink 时直接减少 nlink，不移动文件；此仓选择物理移动文件，可能是为了在 `lwext4` C 库不支持 nlink 语义妥协下的 workaround。

**2.4 procfs 的静态 inode 编号与硬编码内容**  
`dirs.rs` 为所有 `/proc` 节点分配了编译期常量 inode 编号（`PROC_ROOT_INO=1`, `PROC_SELF_INO=2`, ...）<sup>[0](#mod-fs-ref-0)</sup>。这种做法简单但缺乏灵活性：枚举扩散到 30+ 个常量，且新增节点需手动分配编号，容易冲突（`PROC_SELF_FD_INO` 和 `PROC_CPUINFO_INO` 都等于 12，明显是 bug <sup>[14](#mod-fs-ref-14)</sup>）。另外，`PID_MAX_CONTENT`、`DOMAINNAME_CONTENT`、`CONFIG_GZ_CONTENT` 等数据被硬编码为字节数组或字符串常量 <sup>[4](#mod-fs-ref-4)</sup>，不随运行时状态变化。

**2.5 glibc 库路径透明重映射**  
`resolve_path` 在 ENOENT 后触发 `glibc_default_lib_alias` <sup>[9](#mod-fs-ref-9)</sup>：将 `/lib/`、`/lib64/`、`/usr/lib/`、`/usr/lib64/` 前缀的路径自动重试为 `/glibc/lib/{rest}`。这允许在单一文件系统镜像中同时提供标准路径和 glibc 兼容库，是此仓特有的设计——基线 ArceOS 无此机制。

### 3. 跨模块协同

**3.1 路径解析 → ext4 → dentry cache 的三角协作**  
用户调用 `openat` → `path_open` → `resolve_path` → `resolve_path_from` → `link_path_walk` → `lookup_dentry` <sup>[1](#mod-fs-ref-1)</sup>。`lookup_dentry` 先查询全局 DentryCache；未命中则调用 `nd.dentry.get_inode().lookup()`（即 `Ext4Inode::lookup`），后者通过 `lwext4` 的 `ext4_dir_entry_next` 扫描目录块 <sup>[3](#mod-fs-ref-3)</sup>。找到后创建 `Arc<Dentry>`，插入父子关系树并加入缓存。后续同一路径的查找直接从缓存返回，避免重复的 ext4 目录扫描。

**3.2 File ↔ PageCache ↔ ext4 的写回协同**  
`File::write` 调用时，若命中 `PageCache` 则先写入缓存；缓存满或 O_SYNC/O_DSYNC 时调用 `flush_page_cache_if_needed` 刷入 `Ext4Inode::write_at` <sup>[2](#mod-fs-ref-2)</sup>。文件关闭时若 `write_back` 标志为 true，也会触发刷写。`File::get_stat` 返回的 `size` 取 `page_cache.len()` 和 `self.inode.stat().size` 的最大值 <sup>[2](#mod-fs-ref-2)</sup>，因为缓存可能包含尚未写回的数据。这种设计在内存和磁盘之间做了延迟写，但若系统在写回前崩溃会丢失数据。

**3.3 procfs ↔ TASK_MANAGER 的进程列表实时生成**  
`ProcDirInode::readdir` 遍历 `TASK_MANAGER` 获取所有进程（只取 `tid == tgid` 的 leader 线程）<sup>[0](#mod-fs-ref-0)</sup>，动态生成 `/proc/{pid}/` 目录项。每个 `ProcPidDirInode` 的 lookup 又会委托给 `proc_fs_inode` 等具体实现。这种设计让 `/proc` 真实反映了当前进程状态，但每次 readdir 都要遍历整个任务表，在进程数很多时可能有性能问题。

### 4. 边角细节与不足

**<sup>[0](#mod-fs-ref-0)</sup> procfs inode 编号冲突**: `PROC_SELF_FD_INO` 和 `PROC_CPUINFO_INO` 都被定义为 12 <sup>[14](#mod-fs-ref-14)</sup>，这会导致 `readdir` 返回的目录项 inode 不唯一，可能引发用户态程序逻辑异常。

**<sup>[1](#mod-fs-ref-1)</sup> 全局 `NAMEI_MUTATION_LOCK` 并发限制**: 整个 `filename_create` 和 `open_last_lookups` 的 O_CREAT 路径都持有该锁 <sup>[1](#mod-fs-ref-1)</sup>，在高并发创建文件场景下构成串行化瓶颈。

**<sup>[2](#mod-fs-ref-2)</sup> `EXT4_OP_LOCK` 锁粒度过粗**: 所有 ext4 读写元数据操作均序列化 <sup>[2](#mod-fs-ref-2)</sup>，即使并发只读读取目录也互相阻塞。当前实现靠 `unsafe impl Send/Sync` 绕过 Rust 编译检查，但安全契约完全依赖该锁。

**<sup>[3](#mod-fs-ref-3)</sup> 大量 `unsafe` 块缺少 SAFETY 注释**: 例如 `lookup_dirent_locked` 中的 `ext4_dir_open` / `ext4_dir_entry_next` / `ext4_dir_close` 调用 <sup>[3](#mod-fs-ref-3)</sup>，以及 `file_link` / `file_symlink` / `file_readlink` 中的 `bindings::ext4_*` 调用 <sup>[13](#mod-fs-ref-13)</sup>，没有任何 `// SAFETY:` 注释说明 invariants 的维护方式。

**<sup>[4](#mod-fs-ref-4)</sup> `CONFIG_GZ_CONTENT` 硬编码过时风险**: procfs 中的 `config.gz` 内容是一个编译期嵌入的 gzip 字节数组 <sup>[4](#mod-fs-ref-4)</sup>，内核配置变化后不会自动更新，读取该文件的程序会得到过时信息。

**<sup>[6](#mod-fs-ref-6)</sup> 时间戳修复的 ABI 兼容 TODO**: `normalize_lower_time` 将 ext4 镜像中的未来时间戳（由于 CLOCK_REALTIME 仍为开机时间）归零 <sup>[6](#mod-fs-ref-6)</sup>，注释标注了 `TODO[ABI-COMPAT]`，说明这是一个临时 workaround，长期需要正确实现墙上时钟。

**<sup>[9](#mod-fs-ref-9)</sup> `glibc_default_lib_alias` 无缓存/重试限制**: 每次 ENOENT 后都会尝试四个前缀的匹配 <sup>[9](#mod-fs-ref-9)</sup>，没有命中缓存，也没有嵌套层数限制——虽然 path 本身有长度限制，但在深度嵌套路径下可能多次触发不必要的重试。

**<sup>[10](#mod-fs-ref-10)</sup> `ProcReadOnlyBytesInode` 只读但通过 `Arc<Vec<u8>>` 共享**: procfs 的只读文件使用 `ProcReadOnlyBytesInode` 包装 `&'static [u8]` <sup>[10](#mod-fs-ref-10)</sup>，但若传入非静态生命周期数据可能在运行时失效。

**<sup>[11](#mod-fs-ref-11)</sup> xattr 完全在内存中实现**: `Ext4Inode::set_xattr` 仅操作 `self.xattrs: Mutex<HashMap<…>>` <sup>[11](#mod-fs-ref-11)</sup>，没有持久化到 ext4 磁盘的 xattr 块。文件关闭后 xattr 立即丢失。

**<sup>[12](#mod-fs-ref-12)</sup> `PageCache` 引用计数共享可能导致未写回丢失**: 普通文件的 `PageCache` 挂载在 `Ext4Inode` 上，所有 `File` 实例共享同一个 `Arc<PageCache>` <sup>[12](#mod-fs-ref-12)</sup>。若一个 `File` 通过 write 修改了缓存但未刷写，另一个 `File` 的读操作会看到脏数据还是旧数据取决于实现，存在一致性风险。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 基于Linux信号模型，通过`Sig`/`SigSet`/`SigPending`/`SigHandler`等结构管理异步信号，支持自定义处理、默认动作及`SA_SIGINFO`等标志。与基线rCore-Tutorial相比，实现了普通帧与RT帧的分离，并引入`FrameFlags`标记辅助`sigreturn`，但`SA_RESTART`和`core dump`尚未完成。

### 1. 核心抽象与数据结构

信号机制的核心是模拟POSIX异步信号模型。模块通过`Sig`（信号编号）、`SigSet`（信号集位图）、`SigPending`（每个任务的待决信号和掩码）以及`SigHandler`（每个信号的动作数组）组成。与标准Linux一致，使用64位位图支持最多64个信号（`sig_struct.rs:1-322`）。`SigSet`使用`bitflags` crate定义，并显式包含了32个实时信号（SIGRT1-SIGRT31）的位定义，尽管当前仅作占位（`sig_struct.rs:143-186`）。`SigPending`使用`BTreeMap`存储每个信号的`SigInfo`（`sig_struct.rs:64-142`）。信号动作`SigAction`包含处理函数指针、标志、掩码等（`sig_handler.rs:32-42`）。`SigAction`的初始化通过`ActionType::default`决定默认行为（忽略、终止、停止、继续、core），其中`Core`与`Term`当前处理相同（`TODO: core dump`，`handle_signal:88-93`）。`SigHandler`提供`reset_user_handlers_for_exec`用于exec后恢复用户自定义处理程序，符合Linux规范（`sig_handler.rs:29-31`）。

### 2. 信号处理流程与用户栈帧构建

信号处理的入口是`handle_signal()`函数（`mod.rs:49-173`），它被设计为每次trap返回用户态之前调用。函数从任务的`SigPending`中`fetch_signal`取出一个信号，判断动作类型。若为用户处理程序，则构建用户栈帧。`make_sig_context`根据架构（riscv64、loongarch64）生成上下文（`mod.rs:13-28`）。根据`SA_SIGINFO`标志，构建普通帧`SigFrame`或RT帧`SigRTFrame`，通过`copy_to_user`写入用户栈（`mod.rs:110-153`）。帧结构均包含`FrameFlags`（`0x66666666`或`0x77777777`）供`sigreturn`系统调用区分（`sig_struct.rs:48-55`）。最后，修改trap上下文中的`ra`为`TRAMPOLINE`、`sp`调整到帧底、`sepc`设置为用户handler地址（`mod.rs:162-168`）。代码中一次只处理一个信号（`break;` at `mod.rs:170`），避免重入。

### 3. 关键设计取舍：普通帧与RT帧的分离

采用两种帧结构的设计简化了`sigreturn`的实现：`FrameFlags`提供签名机制，`sigreturn`可根据`flag`高效判断帧类型，无需遍历链表（`sig_struct.rs:48-55`）。替代方案（如Linux使用`uc_link`形成上下文链表）会增加内存访问和复杂度。此外，`copy_to_user`一次拷贝整个帧，保证了原子性。放弃的是支持嵌套信号处理的多帧栈，但当前实现仅处理一个信号，因此是合理的简化。

### 4. 跨模块协同：与任务管理和陷阱处理的接口

信号模块通过`task`模块的闭包API操作任务字段：`op_sig_pending_mut`和`op_sig_handler`读写信号状态；`get_trap_cx`获取trap上下文；`set_wait_event`和`notify_parent_sigchld`实现`SIGSTOP`/`SIGCONT`导致的任务状态变化（`mod.rs:60,68,96`）。特别地，`take_sigsuspend_saved_mask`用于恢复`sigsuspend`覆盖的掩码（`mod.rs:58`）。系统调用接口（如`rt_sigaction`、`rt_sigreturn`）在`syscall/mod.rs`中处理，它们调用信号模块提供的方法修改动作或执行恢复。`TRAMPOLINE`常量来自配置模块，假设为一个固定映射的代码页。

### 5. 边角细节与不足

- **`SA_RESTART`未实现**：`mod.rs:155`标注`TODO`，缺少自动重试被打断的系统调用。
- **`Core`与`Term`等价**：`mod.rs:88-93`中`ActionType::Core`直接终止，未执行core dump（`TODO`）。
- **`fetch_signal`的`unwrap`风险**：`sig_struct.rs:121`和`sig_struct.rs:141`中移除信号后直接`.unwrap()`获取`SigInfo`，若状态不一致将导致panic。
- **`BTreeMap`无大小限制**：尽管最大64个信号，但`info` map在pending清空后仍可能残留条目（`sig_struct.rs:93`的`clear_pending`清空info，但部分路径未调用）。
- **用户栈指针有效性验证缺失**：`mod.rs:108-111`仅对齐`user_sp`，未校验是否属合法用户地址，依靠`copy_to_user`返回错误后终止进程。
- **`FrameFlags`硬编码值**：`sig_struct.rs:48-55`中0x66666666/0x77777777非标准，依赖用户态`sigreturn`正确匹配。
- **`Sig`的`From<usize>`截断**：`sig_struct.rs:231`将`usize`直接转为`i32`，若值过大将成负数，虽当前调用处确保<64，但属隐藏陷阱。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本仓 IPC 覆盖匿名管道、命名管道 (FIFO) 和 System V 共享内存，辅以 POSIX 共享内存的 VFS 骨架。管道基于共享环形缓冲区和手动 TID 等待队列实现阻塞读写，共享内存通过 `Arc<FrameTracker>` 实现多进程物理页共享，映射由 `axmm` crate 的 `MmapBacking::SharedFrames` 完成。与 rCore-Tutorial 相比，增加了完整的 shm 系统调用和命名管道支持；与 ArceOS 相比，手动等待队列减少了内核对象依赖，但缺少非阻塞标志和超时机制。

### 1. 核心抽象：两条平行路径

该仓的 IPC 实现覆盖了三种机制：**匿名管道（pipe）** + **命名管道（FIFO）** + **System V 共享内存（shm）**，以及 **POSIX 共享内存（/dev/shm）** 的 VFS 骨架。前两者基于共享的环形缓冲区，后两者基于物理页面共享。所有 IPC 机制都复用了同一个 `FileOp` trait 和系统调用分发路径。

匿名管道通过 `make_pipe` 创建一对 `Pipe` 对象，共享一个 `PipeRingBuffer`。`Pipe` 实现了 `FileOp`，从而无缝接入已有的文件读写框架。与原生 ArceOS 相比，这里定义了完整的 `set_capacity` 和 `pipe_max_size` 全局限制 <sup>[2](#mod-ipc-ref-2)</sup>，更贴近 Linux `fcntl(F_SETPIPE_SZ)` 的行为。命名管道则维护一个全局 `NAMED_FIFOS` 表，通过 `open_named_fifo` 打开或创建 FIFO，支持 O_NONBLOCK 打开时的 FD 语义。

System V 共享内存完全在 `os/src/syscall/ipc.rs` 中由软件实现：`ShmTable` 管理所有段，每个 `ShmSegment` 持有 `Vec<Arc<FrameTracker>>`，引用计数使物理页面可在多进程间安全共享。`sys_shmat` 将页面通过 `MmapBacking::SharedFrames` 映射到进程地址空间，`sys_shmdt` 则解映射并更新引用。POSIX 共享内存 `/dev/shm` 通过 VFS 模拟：`ShmDirInode` 和 `ShmFileInode` 提供目录和文件操作，但系统调用层缺乏直接的 `shm_open`/`shm_unlink` 绑定，实际使用可能依赖文件系统路径操作 <sup>[6](#mod-ipc-ref-6)</sup>。

### 2. 关键设计取舍：手动等待队列 vs. 内核同步原语

管道阻塞最易读的设计是使用条件变量或 `WaitQueue`，但这里选择了完全手动管理 `read_waiters` / `write_waiters`（`VecDeque<usize>`）。每个等待者以其 TID 入队，阻塞时通过 `prepare_current_task_blocked` 将自己标记为阻塞，切换前在锁外再次确认。这种设计避免了引入额外的内核同步对象，减少间接开销，但要求每个路径都显式唤醒和清除等待者。例如，`Pipe::drop` 在关闭一端时，必须手动唤醒所有对端等待者（`wake_waiters.append(&mut buffer.write_waiters);`），防止死锁。此处的唤醒不持有 buffer 锁（drop 中先获取锁填充 `wake_waiters`，释放锁后再唤醒），避免了持锁唤醒，但可能存在 ABA 问题：等待者可能在释放锁后被其他线程调度并重新入队，不过由于唤醒后等待者会立即检查条件，安全。

另一个设计取舍是共享内存的段引用计数策略。`ShmSegment` 不维护自己的 `nattch` 计数器，而是每次需要时遍历所有任务的 `MemorySet::shm_attach_ids_for_frames` 来扫描 <sup>[5](#mod-ipc-ref-5)</sup>。这是 O(n) 的操作，但简化了数据结构的一致性，因为 attach 信息分散在任务的内存描述里。对于 `IPC_RMID` 和 `IPC_STAT` 这类需要准确 attach 计数的操作，这种遍历可以接受，但频繁调用会有性能风险。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块基于 smoltcp 库实现了 TCP、UDP 和 UNIX 域套接字。核心抽象是 `Socket` 结构体，它实现 `FileOp` trait 从而融入内核的 VFS 文件描述符机制。内部根据协议类型分派到 `TcpSocket`、`UdpSocket` 或 `UnixSocket`，所有协议操作通过全局 `SocketSetWrapper` 封装对 smoltcp `SocketSet` 的访问。与同家族基线相比，该实现使用循环自旋 + 主动 yield 的阻塞模型而非等待队列，并通过 `AtomicU8` 维护粗粒度 TCP 状态机以简化同步。UNIX 域套接字在用户态用 `VecDeque` 模拟管道，未集成文件系统路径。

### 1. 核心抽象：多协议套接字与 smoltcp 集成

网络模块的入口是 `os/src/net/socket.rs` 中的 `Socket` 结构体。它枚举了三种内部实现：`TcpSocket`、`UdpSocket`、`UnixSocket`，并通过实现 `FileOp` trait 接入内核的 VFS 层，使得套接字可以像普通文件一样通过文件描述符进行 read/write/poll 操作 <sup>[1](#mod-net-ref-1)</sup>。系统调用层（如 `sys_socket`、`sys_bind`）直接与 `Socket` 的方法交互，无需关心底层协议细节。

所有 TCP/UDP 协议逻辑委托给 smoltcp 库。全局单例 `SocketSetWrapper` 封装了 smoltcp 的 `SocketSet`，通过 `SpinLock` 提供线程安全的访问 <sup>[2](#mod-net-ref-2)</sup>。回环通信由 `LoopbackDev` 设备驱动，仅支持 IPv4 回环地址 127.0.0.1/8。网络栈的初始化（`net::init`）仅通过访问 lazy_static 变量确保单例就绪，并无硬件探测或配置步骤。这种设计简洁但放弃了多网卡或物理网络支持。

UNIX 域套接字完全在用户态模拟：`UnixSocket` 使用 `VecDeque<u8>` 作为接收缓冲区，通过共享 `Arc<SpinLock<VecDeque>>>` 实现双端通信。`socketpair` 调用直接创建一对互相连接 `peer_rx` 的 `UnixSocket`。路径绑定通过全局 `UNIX_LISTENERS` 表实现，不与文件系统真实路径关联，因此不支持 `unlink` 或权限检查 <sup>[3](#mod-net-ref-3)</sup>。

### 2. 关键设计取舍：状态机、阻塞模型与并发安全

#### 粗粒度 TCP 状态机

`TcpSocket` 定义了一个上层状态机（`STATE_CLOSED` / `STATE_BUSY` / `STATE_CONNECTING` / `STATE_CONNECTED` / `STATE_LISTENING`），与 smoltcp 内部的 TCP 状态机解耦 <sup>[4](#mod-net-ref-4)</sup>。状态转换通过 `AtomicU8` 的 CAS 操作实现，如 `update_state` 方法先将状态置为 `BUSY`，执行操作后再设置为目标状态。这种设计简化了阻塞操作中的状态同步：用户只需检查 `is_connected()` 而无需关心 smoltcp 的 `SynSent`、`Established` 等子状态。代价是丢失了细粒度状态反馈（如 `TIME_WAIT`），且 `BUSY` 中间态若因 panic 未恢复会导致状态永久卡死。

#### block_on 阻塞模型

所有可能阻塞的操作（connect、accept、recv、send）都通过 `block_on` 闭包实现 <sup>[5](#mod-net-ref-5)</sup>。该闭包在非阻塞模式下直接执行并返回；在阻塞模式下循环调用 `poll_interfaces()` 驱动 smoltcp，若操作返回 `EAGAIN` 则调用 `yield_current_task()` 让出 CPU，同时检查信号中断和实时定时器。这种“自旋 + 让步”的模式在单核协作式调度下是高效的，因为没有阻塞等待队列的开销。但若引入多核或抢占式调度，`yield_current_task()` 无法保证其他进程推进网络栈，可能导致忙等。

#### 并发安全假设

`TcpSocket` 和 `UdpSocket` 均 `unsafe impl Sync`，其关键在于所有字段（`handle`、`local_addr`、`remote_addr`）使用 `UnsafeCell` 包装，并依赖单核协作调度下同一时刻只有一个线程访问 socket 的假设 <sup>[6](#mod-net-ref-6)</sup>。SAFETY 注释仅出现在 impl 块上，而每个 `unsafe { handle.get().read() }` 调用点缺少独立的 safety 论证。一旦未来引入多核或中断抢占，这种假设就会打破，导致数据竞争。相比之下，使用 `RefCell` 或 `spin::RwLock` 会更明确。

#### 临时端口分配

TCP 和 UDP 各自实现了一个 `get_ephemeral_port` 函数（分别在 `tcp.rs` 和 `udp.rs`），采用线性扫描从 `0xc000` 递增寻找空闲端口。TCP 版本同时检查 `LISTEN_TABLE` 和 TCP socket 集合，UDP 版本仅检查 UDP socket 集合。当端口耗尽时，两个函数都会 `panic!` <sup>[7](#mod-net-ref-7)</sup>。这个路径可通过用户态创建大量连接触发，构成 DoS 攻击入口。更健壮的做法应返回 `EADDRNOTAVAIL` 错误。

### 3. 跨模块协同：系统调用、任务调度与 ListenTable

#### 系统调用入口

系统调用层（如 `os/src/syscall/mod.rs`）调用 `Socket` 的公开方法。例如 `sys_socket` 调用 `Socket::new(domain, kind, protocol)` 并返回文件描述符；`sys_bind` 调用 `socket.bind(addr)`。`Socket` 的方法内部再分派到 `TcpSocket`、`UdpSocket` 或 `UnixSocket`。这种分层使得 syscall 层无需了解网络协议细节。

#### 任务调度协作

`block_on` 闭包与 task 模块紧密配合：获取当前任务后设置 `interruptible` 标志，在每次循环中检查 `check_signal_interrupt()` 和 `is_interrupted()` <sup>[5](#mod-net-ref-5)</sup>。若收到信号则返回 `EINTR`。`yield_current_task()` 触发调度器切换到其他任务。这种设计使网络阻塞操作可被信号中断，符合 POSIX。但注意 `set_interruptible(true)` 在非阻塞路径上不会被设置，因此非阻塞操作不会响应信号。

#### ListenTable 与 accept 流程

`ListenTable` 是端口到监听队列的映射，固定大小为 65536 的 `Box<[SpinLock<Option<Box<ListenTableEntry>>>]>` <sup>[8](#mod-net-ref-8)</sup>。`TcpSocket::listen` 将监听句柄注册到 `LISTEN_TABLE`，并在 `accept` 方法中通过 `block_on` 轮询 `LISTEN_TABLE.lock().accept(port)`。`ListenTable` 内部有一个 `promote_listener` 方法（具体实现被截断），推测其作用是从 smoltcp 的监听 socket 中提取已完成握手的连接并放入 `accept_queue`。`unlisten` 会移除监听句柄并清理所有未 accept 的连接句柄。这种设计将端口监听与连接队列管理集中化，但缺乏对惊群问题的处理（多个线程在同一端口 listen），不过单核下不存在此问题。

### 4. 边角细节与不足

1. **`get_ephemeral_port` panic 风险**：TCP 和 UDP 的临时端口分配函数在耗尽时均调用 `panic!`，用户态可通过连续创建 socket 耗尽端口范围，导致内核崩溃 <sup>[7](#mod-net-ref-7)</sup>。应改为返回 `Errno::EADDRNOTAVAIL`。

2. **`unsafe` 块缺乏细粒度安全论证**：`TcpSocket` 和 `UdpSocket` 中大量使用 `unsafe { self.handle.get().read().unwrap() }` 等操作，但每个调用点没有独立 `// SAFETY:` 注释，仅依靠整个类型的 `unsafe impl Sync` 注释 <sup>[6](#mod-net-ref-6)</sup>。这不仅违反 Rust 惯例，也使得审计困难。

3. **`poll_stream` 强制设置 `readable = true`**：在 `TcpSocket::poll_stream` 中，若 `!readable && !writeable`，代码将 `readable` 强制置为 true <sup>[9](#mod-net-ref-9)</sup>。此分支对应 socket 处于关闭或错误状态，强制可读可能导致 `select/poll` 返回并触发 read 调用后获得错误（如 `ECONNRESET`），但若 read 返回 0 则用户可能误判为 EOF。更清晰的做法是直接报告 `POLLHUP` 或 `POLLERR`。

4. **`AF_INET6` 仅存于注释**：`SocketDomain::AF_INET6` 被定义为值 10，且 `parse_domain` 接受 10，但内部实现完全回退到 AF_INET 逻辑 <sup>[10](#mod-net-ref-10)</sup>。这意味着创建 IPv6 socket 实际上得到一个 IPv4 回环 socket，违反 POSIX。

5. **死代码字段**：`Socket` 中的 `send_buf_size`、`recv_buf_size`、`recvtimeout`、`sendtimeout` 均为 `#[allow(dead_code)]` 的预备字段，未真正参与逻辑 <sup>[11](#mod-net-ref-11)</sup>。`SO_RCVTIMEO` 和 `SO_SNDTIMEO` 等选项不会被处理。

6. **锁顺序一致性**：TCP 的 `get_ephemeral_port` 先获取 `LISTEN_TABLE.lock()` 再获取 `socket_set().lock()`，而在 `listen` 方法中也是先获取 `LISTEN_TABLE.lock()` 再获取 `socket_set().lock()`，顺序一致 <sup>[12](#mod-net-ref-12)</sup>。但 UDP 的 `bind` 中直接获取 `socket_set().lock()` 后可能间接调用 `LISTEN_TABLE.lock()`？不，UDP 不涉及 LISTEN_TABLE。因此当前无死锁风险，但未来新增路径时需小心。

```rust
fn block_on<F, T>(&self, mut f: F) -> Result<T, Errno>
where
    F: FnMut() -> Result<T, Errno>,
{
    if self.is_nonblocking() {
        f()
    } else {
        let task = current_task().ok_or(Errno::ESRCH)?;
        task.set_interruptible(true);
        let result = loop {
            poll_interfaces();
            task.check_real_timer();
            if task.check_signal_interrupt() || task.is_interrupted() {
                task.clear_interrupted();
                break Err(Errno::EINTR);
            }
            match f() {
                Ok(res) => break Ok(res),
                Err(res) => {
                    if res == Errno::EAGAIN {
                        yield_current_task();
                        ...
                    } else { break Err(res); }
                }
            }
        };
        task.set_interruptible(false);
        result
    }
}
```
