## 📖 一、总览

该仓库是 ArceOS 家族中面向 Linux 兼容的完整宏内核实现，定位为生产可用级别的教学/实验型内核。它以 axruntime 两级初始化为基础，通过 StarryTaskExt 将 axtask 与 axprocess 绑定，构建了完整的进程模型；在 axfs_ng 上自研 MemoryFs 和 ProcMountTable，并通过 axsignal + POST_TRAP 实现异步信号投递；系统调用层基于单一大 match 分发 185 个入口，支持多架构（riscv64/loongarch64）。

最值得评审注意的 2-3 个特点：
1. **双物理页缓存策略**：ELF text 段缓存 [mm.ref:4] 和共享文件映射页面缓存 [mm.ref:5] 显著减少了重复加载和物理页浪费，这在 ArceOS 生态中是独创设计。
2. **信号处理的 Post-trap 机制**：利用 POST_TRAP 钩子替代轮询，并引入 `BLOCK_NEXT_SIGNAL_CHECK` 原子标志防止 handler 重入风暴 [task.ref:2]，实现了轻量级信号投递。
3. **系统调用层的条件编译兼容**：通过大量 `#[cfg]` 处理 x86_64/riscv64/loongarch64 差异（如 clone flags、ucontext 布局），支撑了单内核镜像多架构编译 [syscall.ref:5]。

## 🎯 二、综合评价

### 整体定位

该仓库属于**完整 Linux 兼容型** ArceOS 变种，目标是在保留宏内核性能优势的同时，最大化 Linux ABI 兼容性以运行 Busybox、musl 工具链等用户态程序。与标准 ArceOS 仅提供内核线程不同，本仓库实现了 fork/execve/exit 进程模型、mm/mmap/mprotect 内存管理、POSIX 信号、System V IPC 以及 socket 网络接口。其整体设计思路是“增量扩展”：在 ArceOS 模块化 crate 基础上，通过 `StarryTaskExt`、`UserPtr`、自定义 VFS 后端等抽象层补全 Linux 语义，而非重新设计架构。

### 真正的创新点

**1. 增量式物理页缓存（[mm.ref:4]、[mm.ref:5]）**
ELF text 段缓存和共享文件映射页面缓存是该仓库最突出的独创设计。text 段缓存通过 `(device, inode, offset, size)` 键实现文件粒度的物理页共享，避免了同一 ELF 多进程运行时重复加载；共享文件映射缓存则让 MAP_SHARED 的物理页在进程间自动共享。这种设计在缺页路径中直接插入缓存检查，与 ArceOS 原版“每缺页必分配”相比，显著降低内存占用，尤其适用于多实例容器场景。

**2. 基于 POST_TRAP 的异步信号投递（[task.ref:1]、[task.ref:2]）**
信号检查被推迟到 trap 返回前的 POST_TRAP 阶段，而非每次系统调用后轮询或硬中断发信号。配合 `current_handler_signo` 原子变量和 `BLOCK_NEXT_SIGNAL_CHECK` 标志，有效防止了 handler 重入和信号丢失。这种设计耦合度低，对现有系统调用路径改动小，是宏内核中实现信号的一种轻量级方案。

**3. 条件编译解决多 arch 兼容（[syscall.ref:5]、[ipc.ref:3]）**
通过约 20 处 `#[cfg]` 适配了 riscv64、loongarch64 的调用约定、字节序、结构体布局（如信号上下文 ucontext 长度）。共享内存模块专门为 loongarch64 调整对齐到 64 字节 [shm.ref:3]。这种细粒度适配使得单一仓库能维护多个主流 ISA，避免分支分叉。

### 取舍判断

该仓库将工程精力集中于**用户态兼容性的核心路径**，而在以下方面做了明确省略：
- **驱动框架**完全沿用 ArceOS 基线（未实现自家代码），这一取舍合理——基线已覆盖常见外设，重写性价比低。
- **内存管理**采用逐页缺页而非批量预映射，简化了 COW 和 VMA 管理，但牺牲了短时大量缺页场景的性能。
- **系统调用 stub**：约 10% 的 syscall 为 stub（如 set_mempolicy、mbind），这些调用在轻量级测试中极少触发，优先保证 fork/execve/futex/socket 等核心路径的实现完整。
- **死代码与硬编码**：某些功能如 `find_vaddr_by_shmid` 标记为未使用，条件编译缺少默认分支（[boot.ref:1]），反映项目可能以特定架构（riscv64）为主要开发目标，对其他架构的投入有限。

### 完成度与不足

**完成度**：核心子系统基本可用——进程管理支持 fork/clone/exit/execve，信号覆盖全部标准信号，mmap/brk 通过内存压力测试，TCP/UDP socket 可通信，tmpfs 挂载正常。整体能够启动 Busybox 并运行 shell 脚本。

**不足**：
1. 代码硬编码多：EMBEDDED_LIBGCC_S_SO 路径 [boot.ref:2]、TEXT_CACHE_MIN_SEG_SIZE [mm.ref:5] 等缺少参数化，跨环境编译易失败。
2. 并发安全存在隐患：时间统计 [syscall.ref:5] 和文件系统操作 [boot.ref:9] 未使用适当的内存屏障，多核环境下可能出现数据竞争。
3. 部分协议栈接口残缺：Unix socket 未实现 O_NONBLOCK，SO_ERROR 硬编码为 0，netfilter 规则无效 [net.ref:4]。
4. 模块间耦合度较高：条件编译配置未形成全局目标特征表，新架构移植需逐一审计所有 `#[cfg]`。

综合来看，该仓库在 ArceOS 框架下实现了较高水平的 Linux 兼容性，代码质量尚可但仍有优化空间，是一个颇具特色的教学与研究原型。

### 1. 核心抽象：axruntime 驱动的两级初始化体系

启动流程采用 ArceOS 家族经典的两级初始化模式。第一级由 `axruntime` 负责：从汇编入口跳转后初始化页表、中断向量、内存分配器、多核环境等底层设施，然后调用用户定义的 `main()`。第二级即本文件的 `main()`（行 167），负责操作系统语义层的初始化。这种分离将平台相关逻辑收束在 axruntime 内部，而 `main()` 只关注进程模型、文件系统、用户态加载等 OS 概念。

`main()` 的签名不返回（行 165-167 标记 `#[unsafe(no_mangle)]`），代表内核持续运行至关机，这与 Unix 内核 `start_kernel` 永不返回的设计一致。注意到 `#![no_main]`（行 14）禁用了标准 `main` 入口，显式声明了裸机环境 —— axruntime 接管了 lang_start 职责，`main()` 实质上是 axruntime 回调的函数指针，而非 CRT 入口。

与标准 ArceOS 基线相比，本仓库的 main() 做了大量扩展：标准实现通常只挂载根文件系统后直接启动 shell，而这里增加了 ELF 预缓存（行 215-221）、libgcc 自动部署（行 184）、环境变量透传（行 191-197）、init.sh 编译时嵌入与模板替换（行 206-208）等测试套件专用逻辑。可以说，启动流程已经从"通用内核初始化"演化为"测试运行引擎"。

### 2. 关键设计取舍：工程约束驱动的决策

**取舍 1：ELF 预缓存 —— 以内存换启动速度。** 行 76-88 和行 101-110 定义了按架构区分的预缓存列表，包含 busybox、动态链接器、libc.so 等。替代方案是按需从 ext4 读取，但测试中频繁 fork+exec 会导致反复解析 ELF 头。预缓存将解析结果（`insert_elf_cache`，行 218）缓存在内存中，典型场景下 glibc 的 ld-linux 每次都要 open+read+mmap 约 2MB 的 libc.so.6（见行 82-84 中文注释）。代价是启动时增加约几 MB 的内存占用，且个别文件缺失时通过 `if let Err` 降级为 warn（行 218-219）而不阻塞启动 —— 这是有意识地区分"关键初始化"（mount_all 失败必须 panic）与"性能优化"（缓存缺失只是变慢）。

**取舍 2：libgcc_s.so.1 嵌入内核 —— 解决评测盘只读约束。** 行 112-121 定义了三路条件编译的 `EMBEDDED_LIBGCC_S_SO`：riscv64 从 `/usr/riscv64-linux-gnu/lib/libgcc_s.so.1` 嵌入，loongarch64 从 `/opt/gcc-13.2.0-...` 嵌入，其他架构为空。`ensure_libgcc_runtime()`（行 123-149）在 rootfs 挂载后检查目标路径是否存在，若不存在则创建父目录并写入嵌入数据。这个机制的动机在行 124-125 注释中说明得清楚："评测盘是系统提供的，不能直接修改"。替代方案是在编译镜像时预装，但评测盘不可控。嵌入方案将 libgcc 与内核绑定，代价是内核二进制增大约 500KB（libgcc_s.so.1 典型大小），且路径硬编码为特定工具链安装位置（行 113, 116-117），跨环境构建时路径不存在会导致编译失败。

**取舍 3：init.sh 编译时嵌入 vs 从文件系统读取。** 行 206-208 使用 `include_str!("init.sh")` 将脚本编译进内核，并运行时替换 `__RV_LTP_CASES__` / `__LA_LTP_CASES__` 占位符。好处是启动不依赖文件系统完整性 —— 即使 rootfs 损坏，内核仍能执行测试脚本。代价是修改 init.sh 需要重新编译内核，且 init.sh 较长（包含数百个 LTP 测试用例）时会显著增加内核二进制体积。

**取舍 4：条件编译仅覆盖 riscv64/loongarch64。** `CACHED_ELFS` 只有 `#[cfg(target_arch = "riscv64")]`（行 76）和 `#[cfg(target_arch = "loongarch64")]`（行 101）两个版本，缺少默认分支。这意味着在 x86_64、aarch64 等架构上编译会直接失败，因为 `CACHED_ELFS` 未定义。这是一个明确的架构锁定决策 —— Starry Mix OS 当前只针对 RISC-V 和 LoongArch 的测试环境部署，放弃了跨平台通用性。相比之下，`EMBEDDED_LIBGCC_S_SO` 则有 `not(any(...))` 回退（行 120-121），至少不会编译失败。

### 3. 跨模块协同：从内核到用户态的执行链

`main()` 的 7 个步骤编织了多个模块的协作：

**进程模型初始化（行 173）。** `axprocess::Process::new_init(axtask::current().id().as_u64() as _).build()` 将当前硬件任务（由 axtask 管理）绑定为 PID 相等的 init 进程。这是 Unix 进程树根的形成时刻。`axtask::current()` 返回当前 CPU 的 task 结构，其 `id()` 取自底层调度器分配的唯一 ID。`as_u64() as _` 的转换暗示进程 ID 类型与任务 ID 类型之间存在兼容性假设 —— 若未来两者类型定义不同步，这里会静默截断。

**VFS 挂载（行 183）。** `starry_core::vfs::mount_all().expect(...)` 调用 starry_core 封装的 VFS 层，按配置挂载 ext4 rootfs、procfs、tmpfs。这个调用必须成功，因为后续所有文件操作（`ensure_libgcc_runtime` 中的 `metadata`/`create_dir`/`write`、ELF 缓存中的 `insert_elf_cache`、最终 `run_user_app` 加载 busybox）都依赖文件系统。`expect` 在此处是合理的 —— 没有文件系统的内核没有继续运行的意义。

**libgcc 部署与 ELF 缓存（行 184, 215-221）。** 这两个步骤是性能与兼容性优化，它们的失败不会阻止内核继续运行。但注意顺序：`ensure_libgcc_runtime` 在 mount_all 之后立即执行，而 ELF 缓存在其之后。这是因为 libgcc 写入需要文件系统就绪，而 ELF 缓存理论上可以在挂载前预热（缓存的是解析结果，不是原始文件），但当前实现仍然放在挂载后。

**用户态执行（行 232）。** `entry::run_user_app(&args, &envs)` 是进入用户态的关键跳转。它接收 args 为 `["/musl/busybox", "sh", "-c", &init]`，这意味着用户态进程以 busybox sh 解释器运行 init.sh。run_user_app 内部会加载 ELF、创建页表、设置用户栈、将 CPU 控制权转移到用户态。此后，syscall 处理（`handle_syscall`，见 boot facts）和页错误处理（`handle_page_fault`）接管内核与用户态的交互。

### 4. 边角细节与不足

**<sup>[1](#mod-boot-ref-1)</sup> 条件编译架构锁定风险。** `CACHED_ELFS` 仅对 riscv64（行 76）和 loongarch64（行 101）定义，无默认分支。若在 x86_64 或 aarch64 上编译，行 215 的 `for elf in CACHED_ELFS` 将因未定义常量而报错。`EMBEDDED_LIBGCC_S_SO` 虽然有 `not(any(...))` 回退（行 120-121），但行为是静默置空，可能导致 `ensure_libgcc_runtime` 打印 warn 后跳过，而调用者未必检查了 warn 日志。

**<sup>[2](#mod-boot-ref-2)</sup> 宿主机路径硬编码。** 行 113 的 `/usr/riscv64-linux-gnu/lib/libgcc_s.so.1` 和行 116-117 的 `/opt/gcc-13.2.0-loongarch64-linux-gnu/...` 是特定开发环境的绝对路径。若 CI/CD 使用不同工具链安装位置，`include_bytes!` 会在编译期 panic。替代方案是通过 build.rs 或环境变量传递路径，但当前实现选择了最简单直接的硬编码。

**<sup>[5](#mod-boot-ref-5)</sup> mount_all 失败无降级。** 行 183 的 `.expect()` 在文件系统挂载失败时直接终止内核。虽然这符合"关键路径必须成功"的原则，但在测试环境中若能提供 fallback（如挂载一个空 tmpfs 作为 rootfs）也许能提供更好的调试体验。

**<sup>[6](#mod-boot-ref-6)</sup> insert_elf_cache 的静默降级可能掩盖问题。** 行 218-219 在缓存失败时只打印 warn 并继续。如果文件系统布局异常导致全部缓存失败，启动仍能继续但性能大幅下降，且没有显式告警机制。此外，`insert_elf_cache` 本身的错误类型被 `{:?}` 格式化，错误信息可能不够直观。

**<sup>[3](#mod-boot-ref-3)</sup> 锁 FS_CONTEXT 持锁时间较长。** 行 124 获取 `FS_CONTEXT.lock()` 后，`ensure_libgcc_runtime` 执行了多次 `metadata`、`create_dir` 和 `write` 操作（行 125-141），持锁时间可能较长。虽然当前系统是单核或单任务环境，未来若引入多核并发文件访问，这个锁可能成为瓶颈。

**<sup>[9](#mod-boot-ref-9)</sup> 权限常量 0o755 硬编码。** 行 134 和 137 的 `0o755` 是 Unix 常规权限，但缺少参数化。若未来需要修改目录权限，需要 grep 所有出现 `0o755` 的位置。

**总体评价：** 启动流程模块的设计紧密围绕测试运行需求，在标准 ArceOS 框架上叠加了大量工程 hack 和优化。这些 hack（如 libgcc 嵌入、ELF 预缓存）在通用场景中属于非标准做法，但在评测盘环境受限的约束下是合理且必要的中短期方案。长远来看，这些硬编码路径和架构锁定需要通过构建脚本参数化和设备树配置来消除。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 基于 axmm crate 的 `AddrSpace` 抽象，实现用户态虚拟地址空间的生命周期管理、ELF 加载、mmap/brk 等系统调用以及缺页异常处理。与族基线（原版 ArceOS）相比，本模块引入了共享文件映射写回缓存、ELF text 段物理页缓存、信号跳板线性映射，并在 mmap 缺省基地址、COW 缺页路径等细节上做了大量 Linux ABI 兼容性调整。

### 1. 核心抽象与外部依赖

内存管理模块以 `axmm::AddrSpace` 作为核心抽象 <sup>[1](#mod-mm-ref-1)</sup>。每个用户进程持有一个 `AddrSpace`，内部管理一组 VMA（虚拟内存区域），每个 VMA 关联一个 `Backend` 枚举（`Alloc`、`Shared`、`Linear` 等）。`Alloc` 后端在缺页时按需分配物理页，`Shared` 后端支持多进程共享物理页。模块通过 `axhal::paging` 提供的页表操作接口与硬件交互。

与 ArceOS 基线相比，本模块额外引入了两层缓存：
- **ELF text 段缓存** (`TEXT_PAGE_CACHE`)：当 `ENABLE_TEXT_CACHE` 开启且段为只读且大小 ≥ 4 页时，`map_elf` 会先检查缓存，若命中则直接 `map_shared` 插入已有 `SharedPages`，避免重复加载和物理页浪费 <sup>[4](#mod-mm-ref-4)</sup>。缓存 key 包含 `(device, inode, offset, size)`，确保不同文件或同一文件的不同段不会混淆。加载 ld.so 时会独立获取 `file_key`，防止 text 段错存到主二进制 key 下 <sup>[4](#mod-mm-ref-4)</sup>。
- **共享文件映射页面缓存** (`SHARED_FILE_PAGE_CACHE`)：在 `sys_mmap` 创建 `MAP_SHARED` 文件映射时，通过 `SharedFileCacheKey` 查询是否已有相同后端文件页的 `SharedPages`，实现跨进程物理页共享。

### 2. 关键设计取舍

#### 2.1 mmap 缺省基地址选择

Linux 下 `mmap(NULL, ...)` 内核会优先在地址空间底部附近搜索空闲区域，但这导致某些 LTP 用例（如 loongarch64 上的 `grep/sed/hush`）频繁获得 `0x1000` 等低地址映射，暴露异常。本模块在 `sys_mmap` 中，当 `addr == 0` 时，将搜索基地址从 `aspace.base()` 改为 `heap_bottom + heap_size` <sup>[2](#mod-mm-ref-2)</sup>，使得匿名映射集中在堆区之后，避免低地址空洞被占用。这一修改属于兼容性 hack，牺牲了与严格 Linux 行为的一致性，但通过了相关 LTP 测试。

#### 2.2 COW 与用户内存访问标志的交互

写时复制（COW）机制需要内核在系统调用中写用户缓冲区时也能触发缺页。`core/src/mm.rs` 提供了 `access_user_memory` 函数，通过 percpu 标记 `ACCESSING_USER_MEM` 来允许内核上下文中的缺页被正常处理 <sup>[8](#mod-mm-ref-8)</sup>。`src/mm.rs` 中的 `handle_page_fault` 检查该标志：若缺页发生在内核态且 `is_accessing_user_memory()` 为 false 且地址在用户空间之外，才返回 false <sup>[3](#mod-mm-ref-3)</sup>。这保证了 `sys_write`、`copy_to_user` 等路径在访问 COW 页时不会 panic。

#### 2.3 brk() 的简化实现

`sys_brk` <sup>[7](#mod-mm-ref-7)</sup> 只维护一个 `heap_top` 游标，不真正分配或释放物理页。堆区在进程创建时通过 `load_user_app` 预映射了 `USER_HEAP_SIZE` 大小 <sup>[1](#mod-mm-ref-1)</sup>。因此 `brk` 只是移动边界，缺页完全由 `AddrSpace::handle_page_fault` 按需补页。这种简化避免了 VMA 分裂合并的复杂性，但意味着堆区永远保留满 `USER_HEAP_SIZE` 的 VMA，无法归还未使用的部分（类似 `MAP_NORESERVE` 语义）。

#### 2.4 text 段缓存最小粒度硬编码

`TEXT_CACHE_MIN_SEG_SIZE` 硬编码为 4 页 <sup>[5](#mod-mm-ref-5)</sup>，注释说明这是为了 `fs_bind` 系列测试中频繁拉起的小型 ELF 也能命中缓存。但该值未参数化，若未来运行超大 text 段或极小的二进制集合，可能需要调整，但修改需全局替换。

### 3. 跨模块协同

#### 3.1 缺页处理与信号机制

用户缺页流程：CPU 触发 `PAGE_FAULT` → `src/mm.rs` 中 `handle_page_fault` 被 `register_trap_handler` 注册为回调 <sup>[3](#mod-mm-ref-3)</sup>。它先校验是否在内核态且非用户内存访问上下文，然后调用 `handle_user_page_fault` → `aspace.lock().handle_page_fault(vaddr, access_flags)`。若缺页无法满足（如栈扩展超过 rlimit 或地址无映射），则发送 `SIGSEGV` 到当前进程。这里特别避免在信号发送前打印任务名，因为 `splice02` 崩溃现场中任务名元数据可能已损坏，避免二次 panic <sup>[3](#mod-mm-ref-3)</sup>。

#### 3.2 共享文件映射与页同步

`sys_mmap` 中创建 `MAP_SHARED` 文件映射时，会注册 `SharedFileMapping` 到全局 `SHARED_FILE_MAPPINGS` <sup>[6](#mod-mm-ref-6)</sup>。当调用 `msync` 或进程退出时，通过 `sync_mapping_to_file` 将脏页写回文件。`flush_process_file_mappings_on_exit` 在进程退出前同步所有共享文件映射 <sup>[6](#mod-mm-ref-6)</sup>。`munmap` 也触发部分同步，并更新映射记录，确保写回完整性。

#### 3.3 信号跳板映射

`core/src/mm.rs` 中的 `map_trampoline` 将信号处理返回的跳板代码（物理页）线性映射到用户空间的固定地址 `SIGNAL_TRAMPOLINE` <sup>[1](#mod-mm-ref-1)</sup>。这避免了每次信号传递时复制跳板代码，并且由于映射是 `USER | READ | EXECUTE`，用户态可直接执行。该地址在编译期固定，与 trap 处理中的返回流程配合。

### 4. 边角细节与不足

1. **`mlock` 范围管理仅记录，未真正锁定** <sup>[1](#mod-mm-ref-1)</sup>：`api/src/imp/mm/mmap.rs` 中存在 `mlock` 相关逻辑（`MLOCKED_RANGES`、`remove_mlock_overlap` 等），但实际只记录锁定范围，并未调用 `axhal` 的页锁定 API。因此 `mlock` 返回成功但物理页仍可能被换出（本模块无 swap 支持）。这是明显的半成品。
2. **text 段缓存最小粒度硬编码** <sup>[5](#mod-mm-ref-5)</sup>：4 页的阈值可能不适合所有场景，且未暴露为可配置选项。
3. **`mmap` 标志检查 TODO** <sup>[6](#mod-mm-ref-6)</sup>：`sys_mmap` 中有一处 `// TODO: check illegal flags for mmap`，后续仅检查了 `PRIVATE` 与 `SHARED` 互斥以及必须包含一种类型，但未检查如 `MAP_STACK` 与 `MAP_GROWSDOWN` 的结合等 Linux 具体限制。
4. **共享文件页面缓存无淘汰机制** <sup>[1](#mod-mm-ref-1)</sup>：`SHARED_FILE_PAGE_CACHE` 使用 `Weak` 引用，当所有映射解除后自动清理，但 `Weak` 在 `Arc` 引用归零前会一直占用缓存条目。未实现 LRU 或主动淘汰，可能在某些模式下长期持有大量条目。
5. **`brk` 不支持收缩** <sup>[7](#mod-mm-ref-7)</sup>：`sys_brk` 只允许向上移动，不允许向下移动（缩小堆）。Linux 允许 `brk` 至低于当前堆顶的地址来释放页，但本模块直接拒绝，可能导致内存浪费。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 核心抽象是 `StarryTaskExt`（`TaskExt` trait 的实现），它将 `axtask::TaskInner` 与 `axprocess::Thread`/`Process` 绑定，使每个内核任务对应一个用户线程。该模块解决了在单内核中支持 Linux 进程模型（fork/clone/exit、进程组/会话、信号传递、时间统计）的核心问题。与 ArceOS 基线相比，Starry 增加了完整的用户态进程生命周期管理、系统调用实现、procfs 集成以及信号与调度的协同机制。

### 1. 核心抽象：StarryTaskExt 与 axprocess 的融合

进程与任务调度模块的核心是 `StarryTaskExt` 结构体，它实现了 `axtask::TaskExt` trait，为每个 `axtask::TaskInner` 附加一个指向 `axprocess::Thread` 的引用 <sup>[1](#mod-task-ref-1)</sup>。`Thread` 和 `Process` 来自外部 crate `axprocess`，分别代表用户线程和进程。扩展数据 `ThreadData` 和 `ProcessData` 存储每个线程/进程的私有状态：信号管理器、时间管理器、futex 位集、资源限制、用户/组 ID 等 <sup>[2](#mod-task-ref-2)</sup>。这种设计使得底层的协作式调度器（`axtask`）与上层的 Linux 进程模型解耦：`axtask` 负责 CPU 上下文切换，而 `axprocess` 提供进程树、线程组、进程组/会话的抽象。

与 ArceOS 原版相比，Starry 的这一层完全是新增的。ArceOS 的任务仅支持内核线程，没有用户态地址空间、信号、系统调用等概念。Starry 通过 `StarryTaskExt` 将每个内核任务“包装”为用户线程，并在 trap handler 中通过 `current()` 获取当前任务后提取 `Thread` 引用，进而访问进程数据。

### 2. 关键设计取舍

#### 2.1 clone/fork 的地址空间管理

`sys_clone` 是进程创建的核心，它根据 `CloneFlags` 决定地址空间是共享（`CLONE_VM`）还是复制。对于 `CLONE_VFORK` 场景，代码选择分配独立的地址空间副本，而不是直接共享 <sup>[3](#mod-task-ref-3)</sup>。注释解释了原因：vfork 子进程如果执行 `execve()`，会替换地址空间，如果此时共享页表，父进程恢复后写回 `waitpid()` 状态时可能访问到已被新程序改写的用户栈。替代方案可以是临时标记写保护，但当前实现通过完整副本避免了复杂性。

地址空间复制使用 `try_clone()`，如果因内存不足失败，会先调用 `reclaim_all_tmpfs()` 回收 tmpfs 后再重试 <sup>[4](#mod-task-ref-4)</sup>。这是一个实用的回退策略，但全局回收可能影响其他进程。

#### 2.2 fork 风暴防护

模块维护一个全局 `FORK_COUNT` 静态变量，每 500 次 fork 主动回收一次 tmpfs 内存 <sup>[5](#mod-task-ref-5)</sup>。这防止了短时间内大量 fork 导致的内存耗尽。设计上假设 fork 频率不会极端，但 500 的间隔是硬编码常量，在 SMP 系统中可能不精确（存在竞争，但仅用于计数，可以接受）。

#### 2.3 退出时的资源清理

`do_exit` 处理线程和进程退出。当最后一个线程退出时，进程会关闭所有文件描述符（通过遍历 `FD_TABLE` 并调用 `close()`），然后刷新文件映射（`flush_process_file_mappings_on_exit`），并根据架构决定是否 unmap 用户地址空间 <sup>[6](#mod-task-ref-6)</sup>。对于 `exit_group`，除了发送 SIGKILL 给所有线程，还主动唤醒私有 futex 等待者，防止线程在 `futex_wait` 中永久阻塞 <sup>[7](#mod-task-ref-7)</sup>。

需要注意的是，地址空间 unmap 只在 `loongarch64`、`aarch64` 和 `riscv64` 上执行，x86_64 被排除，因为共享页表下 unmap 用户页可能导致 QEMU 静默退出 <sup>[8](#mod-task-ref-8)</sup>。这反映了硬件差异带来的实现约束。

#### 2.4 时间统计与信号传递的绑定

`time_stat_from_kernel_to_user` 和反函数在上下文切换时被调用，更新时间统计（用户/内核态时间、ITIMER 实时定时器），并在超时时通过回调发送信号 <sup>[9](#mod-task-ref-9)</sup>。这种设计将时间管理和信号产生紧密耦合：当用户态调度器切换到当前线程时，时间统计更新可能触发 `SIGPROF`、`SIGVTALRM` 等信号。回调函数使用 `SignalInfo::new(signo, SI_KERNEL)` 标记为内核来源。

### 3. 跨模块协同

该模块与多个子系统深度协同：

- **信号模块**：`ThreadData.signal` 和 `ProcessData.signal` 分别保存 `ThreadSignalManager` 和 `ProcessSignalManager`。exit 时向父进程发送 `SIGCHLD`，填充正确的 `si_code` 和 `si_status` <sup>[10](#mod-task-ref-10)</sup>。时间统计超时也通过信号管理器的 `send_signal` 投递信号。
- **VFS / procfs**：`/proc/[pid]` 目录通过 `ProcFsHandler` 动态查询 `PROCESS_TABLE`，访问 `ThreadData` 的 `oom_score_adj`、`ProcessData` 的 `coredump_filter` 等字段 <sup>[11](#mod-task-ref-11)</sup>。这实现了运行时进程状态的可观察性。
- **futex 系统**：`ProcessData` 包含私有 `FutexTable`，`futex_table_for` 根据 key 类型选择私有或共享表。`exit_group` 调用 `wake_private_futex_waiters` 防止线程挂起 <sup>[7](#mod-task-ref-7)</sup>。
- **文件描述符**：通过 `Scope`（资源作用域）和 `FD_TABLE`/`FD_CLOEXEC_TABLE` 管理文件描述符。fork 时根据 `CLONE_FILES` 决定共享还是复制。`do_exit` 显式关闭所有 fd 并清空 cloexec 表 <sup>[6](#mod-task-ref-6)</sup>。

### 4. 边角细节与不足

1. **`clear_child_tid` 写入失败被静默忽略**：`do_exit` 中写入 `clear_child_tid` 时，若 `write_bytes` 返回 `Err`，只忽略不处理 <sup>[12](#mod-task-ref-12)</sup>。用户态可能因此无法收到线程退出通知。攻击者可故意映射只读页到该地址，导致内核永远无法清除 TID，虽然不影响内核稳定性，但违反 POSIX 语义。

2. **`AssumeSync<RefCell<TimeManager>>` 的安全性依赖非形式化约束**：`ThreadData.time` 使用 `AssumeSync` 包装 `RefCell`，其 `Sync` impl 依赖“仅上下文切换时可变借用”的假设 <sup>[13](#mod-task-ref-13)</sup>。若中断处理程序或调度器意外访问该字段，可能导致数据竞争。当前代码中该字段仅在 `time_stat_*` 函数中借用，这些函数在进程上下文调用，因此安全，但缺少编译期保证。

3. **`sys_sched_getaffinity` 不完全符合 Linux 语义**：当前只支持 `pid=0`（当前线程），对非零 pid 返回 `EPERM` <sup>[14](#mod-task-ref-14)</sup>。Linux 允许查询任意线程的 CPU affinity。对于需要多线程调度的用户态监控工具不兼容。

4. **`oom_score_adj` 默认值 200**：`ThreadData::new` 中将 `oom_score_adj` 初始化为 200，而 Linux 默认值为 0 <sup>[15](#mod-task-ref-15)</sup>。这可能导致 OOM 时该进程被优先杀死，与用户期望不符。虽然当前 OOM 实现可能未使用该值，但为兼容性应设为 0。

5. **fork 回收间隔硬编码**：`FORK_RECLAIM_INTERVAL = 500` 是全局常数，无法动态调整 <sup>[5](#mod-task-ref-5)</sup>。在内存极低的系统上可能不够激进，在内存充足时又造成不必要的回收开销。

总体而言，该模块设计围绕 Linux 兼容性目标，在 ArceOS 基础之上增量开发，权衡了实现复杂度与性能，但部分细节仍有提升空间。

>
> 文件系统模块基于 `axfs_ng` 与 `axfs_ng_vfs` crate 构建 VFS 抽象层，通过 `core/src/vfs/` 自实现 `MemoryFs`、`ProcMountTable` 及设备框架，在 `api/src/imp/fs/` 中实现 Linux 兼容的系统调用（读写、控制、挂载）。与标准 ArceOS 相比，本项目扩展了完整的挂载传播（MS_SHARED/MS_SLAVE/unbindable）、sparse file 语义（SEEK_DATA/HOLE）、loop 与 RTC 设备 ioctl，以及通过 `UserPtr` 强制内核缓冲解决非 x86 架构用户态直接访问问题。

### 1. 核心抽象：VFS 层与多后端协作

文件系统模块的核心抽象是定义在 `axfs_ng_vfs` 中的 `Location`、`Filesystem`、`Metadata` 等 trait，通过全局单例 `FS_CONTEXT` 暴露文件系统上下文。项目在 `core/src/vfs/mod.rs` 中自行实现了 `MemoryFs`（基于 tmpfs）<sup>[1](#mod-fs-ref-1)</sup>，以及用于记录 `/proc/mounts` 的 `ProcMountTable`。后端文件系统通过 `axfs_ng` 的 `Filesystem` trait 接入，目前仅支持 tmpfs（mount.rs 中将 vfat 也 fallback 为 tmpfs）<sup>[5](#mod-fs-ref-5)</sup>。

`api/src/imp/fs/` 下的三个文件分别实现系统调用的不同职责：`io.rs` 处理读写类调用（read/write/lseek/fallocate 等），`ctl.rs` 处理控制类（ioctl/chdir/mkdir/link/unlink 等），`mount.rs` 处理挂载相关（mount/umount2）。所有系统调用通过 `crate::syscall.rs` 中的分发逻辑进入对应函数。

设备框架通过 `core/src/vfs/dev` 模块注册（我们未看到完整源码，但从 ctl.rs 可知设备如 RTC、LoopDevice 通过 `cast_file_like_to_device` 转换后匹配），并使用 `downcast_ref` 进行类型识别 <sup>[2](#mod-fs-ref-2)</sup>。这种设计牺牲了部分性能但避免了在 `Device` trait 中定义统一方法集合，属于布局灵活的取舍。

### 2. 关键设计取舍

**用户态缓冲区处理**：在 `io.rs` 的 `sys_read` 中，使用 `vec![0; len]` 在内核分配临时缓冲，再通过 `write_user_buf` 写入用户指针。注释明确说明 loongarch64/aarch64 下不能直接写用户虚拟地址 <sup>[3](#mod-fs-ref-3)</sup>。这牺牲了一次拷贝，但避免了架构差异且防止了 TOCTOU 竞争。类似地，`write_user_buf` 仅当数据非空时调用 `write_bytes`，符合 Linux 对零长度写入的语义 <sup>[4](#mod-fs-ref-4)</sup>。

**挂载传播实现**：`mount.rs` 实现了完整的传播语义，包括 `infer_new_mount_peer_group` 函数根据 source 与 target 的传播属性决定新挂载是否分配新 peer group <sup>[6](#mod-fs-ref-6)</sup>。这与 Linux 内核行为一致，且针对 LTP 的 `fs_bind05`、`fs_bind10/11/17` 等测试用例有专门的注释说明。在 `propagate_bind_to_shared_peers` 中区分了父 peer group 和源 peer group 两个概念，是该项目相比 ArceOS 基线最显著的增强。

**稀疏文件语义**：`seek_data_or_hole` 通过逐块扫描非零字节实现 SEEK_DATA/SEEK_HOLE <sup>[7](#mod-fs-ref-7)</sup>，复杂度为 O(n) 且硬编码 4096 字节块大小，但兼容缺乏原生支持的底层文件系统。这种做法在轻量内核中常见，但在大规模稀疏文件上性能较差。

**目录操作权限检查**：`ctl.rs` 中的 `check_chdir_perm`、`check_dir_write_search_perm` 以及 sticky bit 处理 <sup>[8](#mod-fs-ref-8)</sup> 完整模拟 Linux 权限模型。值得注意的是 `gid_matches` 函数检查了 supplementary gids，这在简单实现中常被忽视。

### 3. 跨模块协同

与进程/任务模块的协作：权限检查时通过 `starry_core::task` 获取当前进程的 euid/egid/umask/rlimit。例如 `sys_write` 中会检查 RLIMIT_FSIZE 限制，若超过则发送 SIGXFSZ 信号 <sup>[9](#mod-fs-ref-9)</sup>。`core/src/vfs/mod.rs` 中的 `ProcMountTable` 结构体使用 `BTreeMap<String, Vec<usize>>` 作为 target 索引，保留稳定槽位避免 `umount` 操作使下标失效 <sup>[10](#mod-fs-ref-10)</sup>。这保证了 `peer_group_index` 的可靠性。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了 System V 共享内存、匿名管道和 futex 三种进程间通信机制。共享内存通过 `ShmInner` + `ShmManager` 全局管理器维护，物理页面使用 `axmm::SharedPages` 引用计数；管道使用固定 64KiB 环形缓冲区和 `WaitQueue` 实现阻塞读写；futex 通过 `FutexKey` 区分私有和共享地址，支持 `FUTEX_WAIT/WAKE/REQUEUE` 及健壮的 robust list。与 rCore-Tutorial 相比，该实现完整支持了 `shmat/shmid_ds` 用户态兼容布局和 loongarch64 对齐差异。

### 1. 核心抽象与外部依赖

进程间通信模块由三个子系统构成，共享相同的基础设施（`axsync::Mutex`、`axtask::WaitQueue`、`axmm::SharedPages`）和星型内核的任务扩展 `StarryTaskExt`。

**System V 共享内存** 以 `ShmInner` <sup>[1](#mod-ipc-ref-1)</sup> 为核心结构，记录段大小、物理页引用、每进程映射地址范围 (`va_range`) 和标准 `ShmidDs` 控制结构。所有段由全局 `ShmManager` <sup>[2](#mod-ipc-ref-2)</sup> 统一管理，后者通过三重映射（key↔shmid、shmid→`ShmInner`、pid→(shmid↔vaddr)）提供 O(log n) 查找，并实现完整的引用计数生命周期管理。

**匿名管道** 基于 `PipeRingBuffer`（固定 64 KiB 数组 + head/tail）和 `Pipe` 封装 <sup>[4](#mod-ipc-ref-4)</sup>，利用 `WaitQueue` 实现阻塞 I/O，同时支持非阻塞模式和 `O_NONBLOCK` 标志。写端关闭时自动向读端发送 `SIGPIPE` 信号 <sup>[5](#mod-ipc-ref-5)</sup>。

**Futex** 以 `FutexKey` <sup>[6](#mod-ipc-ref-6)</sup> 区分进程私有地址和共享内存中的地址，核心等待队列由 `FutexEntry` 包装，每个进程维护一个 `FutexTable`（`BTreeMap<usize, Arc<FutexEntry>>`）。`FutexGuard` 的 Drop 实现自动清理空条目 <sup>[7](#mod-ipc-ref-7)</sup>。

### 2. 关键设计取舍

**物理页面的延迟分配**：共享内存的 `phys_pages: Option<Arc<SharedPages>>` 在 `map_to_phys` 时才填充，而非创建段时立即分配。这允许内核推迟物理页分配直到首次 attach，与 Linux 的惰性语义一致，但代价是首次 `shmat` 可能引入延迟。

**全局单锁 vs 分片**：`SHM_MANAGER` 使用单一的 `Mutex<ShmManager>` 保护所有操作。在高并发下，对 `shmctl/stat` 等遍历操作会阻塞其他进程的 `shmat/det`。更细粒度的锁（如 per-segment 锁）会引入复杂度，当前实现优先正确性。

**BiBTreeMap 的双向索引**：共享内存模块使用自定义的双向映射 <sup>[8](#mod-ipc-ref-8)</sup> 同时在两个方向上查询，避免维护两个独立 `BTreeMap` 的不一致风险。该 HashMap 风格的中间层使得 `remove_shmaddr`（按 vaddr 删除）可直接找到 shmid。

**Futex 的地址类型动态识别**：`FutexKey::new` 通过检测当前进程地址空间中的区域后端是否为 `SharedPages`，自动决定键类型 <sup>[6](#mod-ipc-ref-6)</sup>。这使得用户代码无需显式区分 `FUTEX_PRIVATE_FLAG`，内核自动选择正确的比较语义（进程内地址 vs 物理页偏移）。

**loongarch64 对齐兼容**：`shmat_round_align` 在 loongarch64 上根据传入地址的低位判断采用 64K 还是 4K 对齐 <sup>[3](#mod-ipc-ref-3)</sup>，以兼容 glibc 与 musl 的不同 `SHMLBA` 值。这是一种工程上的“最坏兼容”策略，避免了在 musl 上因对齐过严而拒绝合法地址。

### 3. 跨模块协同

**进程退出与共享内存清理**：当进程退出时，`clear_proc_shm` <sup>[9](#mod-ipc-ref-9)</sup> 遍历进程关联的所有 shmid，对每个段执行 `detach_process` 递减 `nattch`，若段已被标记 `rmid` 且无其他 attach，则彻底删除段。`inherit_proc_shm` 在 `fork` 时复制父进程的映射，保证子进程继承共享段。

**Futex 与共享内存**：若 futex 操作地址落在共享内存区域，`FutexKey::Shared` 使用 `Weak<SharedPages>` 作为区域标识。这保证了共享内存段释放后，对应 futex 键自动失效，避免悬空指针。`futex_table_for(&key)` 利用 `FutexKey` 的 `as_usize` 输出统一哈希（私有用地址，共享用偏移），因此同一物理页的不同虚拟地址会正确串扰。

**Pipe 与信号传递**：`write_broken_pipe` 在写入已无读端的管道时，不仅返回 `EPIPE`，还发送 `SIGPIPE` 信号（通过 `StarryTaskExt::thread_data().signal.send_signal`）<sup>[5](#mod-ipc-ref-5)</sup>。这需要与信号模块的 `SignalInfo/Signo` 类型配合，是跨模块协作的典型。

**Robust list 与 futex 死亡通知**：`exit_robust_list` 遍历用户空间链表，调用 `handle_futex_death` 将对应 futex 标记 `owner_dead` 并唤醒等待者 <sup>[7](#mod-ipc-ref-7)</sup>。这保证持有 futex 的进程崩溃时，等待者不会永久阻塞。

### 4. 边角细节与不足

- **`find_vaddr_by_shmid` 死代码**：该函数被 `#[allow(dead_code)]` 隐藏，未在任何系统调用路径中调用 <sup>[2](#mod-ipc-ref-2)</sup>。可能是调试保留的辅助函数，构成工程腐烂信号。
- **`phys_pages` 初始化路径不透明**：`ShmInner::new` 中 `phys_pages` 为 `None`，但 `map_to_phys` 必须在某处被调用。在系统调用实现中（未完全展示），若 `shmget` 后直接 `shmat` 而未调用 `map_to_phys`，会导致后续 `shmat` 映射无物理页面。这个管道缺失未在代码中明确断言。
- **`pipe set_capacity` 的数据竞争**：`available_read_len()` 在锁外计算，而 `set_capacity` 内部再次上锁计算 `used`，存在 TOCTOU。虽然 Rust 类型系统保证了 `BTreeMap` 安全，但业务逻辑上 `new_capacity < used` 可能在竞态下误判。
- **Futex `FUTEX_REQUEUE` 实现简化**：仅处理 `value` 个唤醒并迁移剩余等待者，但未实现位掩码过滤（`FUTEX_WAIT_BITSET` 的 `bitset` 参数仅在 `WAIT/WAKE` 中使用）。完整的 `FUTEX_CMP_REQUEUE` 在检查 `*uaddr == value3` 后即调用普通 `REQUEUE` 逻辑，忽略了位掩码约束。

> 💡 **TL;DR**
>
> 该模块实现了完整的 POSIX socket 系统调用层（~20 个 syscall），核心抽象为 `Socket` 枚举（包装 axnet 的 TcpSocket/UdpSocket）和 `UnixSocket`（进程间通信）。通过 `SocketAddrExt` trait 统一了 sockaddr 的 user/kernel 空间安全转换，并依赖 `UserPtr` 封装处理边界检查。与原版 ArceOS 相比，此仓新增了 Unix domain socket 支持（含 socketpair/abstract address）、完整的错误码映射、信号打断阻塞 I/O、以及多架构 sun_path 类型适配。

### 1. 核心抽象：Socket 枚举与 SocketAddrExt trait

网络模块的核心抽象是 `api/src/file/net.rs` 中定义的 `Socket` 枚举，它将底层 `axnet` crate 的 `TcpSocket` 和 `UdpSocket` 包装进 `Mutex` 中，并通过 `impl_socket!` 宏对各方法做统一派发 <sup>[2](#mod-net-ref-2)</sup>。这种“枚举+宏”的设计替代了原版 ArceOS 可能使用的 trait object 方式——优点是编译期确定变体、零动态分发开销；代价是每新增一个协议族（如 Raw Socket）都需要修改枚举定义和所有 match 分支。

地址处理层是 `api/src/socket.rs` 的 `SocketAddrExt` trait，它为 `SocketAddr`、`SocketAddrV4`、`SocketAddrV6` 和 `UnixAddr` 统一实现了 `read_from_user` / `write_to_user` / `write_to_user_limited` 方法 <sup>[1](#mod-net-ref-1)</sup>。设计要点：`copy_sockaddr_from_user` 先整体清零再部分拷贝，应对 AF_UNIX 场景下用户传的 `sockaddr_un` 长度不足完整结构体的情况；`read_sockaddr_family` 只读 family 头部 2 字节而非整个 `sockaddr`，避免 LTP 测试中把最小结构体放在页尾时触发 EFAULT。

### 2. 关键设计取舍

**UDP 自动 bind 与 TCP sendto 忽略目标地址**：`api/src/file/net.rs:100-110` 中，UDP socket 在未 bind 时调用 `sendto()` 会自动补一个临时端口（`0.0.0.0:0`），匹配 Linux 行为；而已连接 TCP 的 `sendto()` 则直接忽略用户传入的目标地址，返回 `EPIPE` 而非去解析 sockaddr <sup>[3](#mod-net-ref-3)</sup>。这避免了 TCP 场景下误将合法地址解析为 EAFNOSUPPORT 的风险。

**connect 错误码映射**：`api/src/imp/net/socket.rs:198-206` 中，底层 axnet 返回 `EAGAIN` 时映射为 Linux 语义的 `EINPROGRESS`（非阻塞连接进行中），`EEXIST` 映射为 `EISCONN`（已连接）<sup>[6](#mod-net-ref-6)</sup>。这种映射暴露了 axnet 与 Linux 错误体系之间的语义鸿沟——axnet 用 `AlreadyExists` 表示连接已建立，但 POSIX 要求 `EISCONN`。

**阻塞 accept 的信号打断**：`api/src/file/net.rs:143-170` 的 `accept()` 循环在 `WouldBlock` 时调用 `time_stat_refresh_itimer_real_in_kernel()` 刷新 real timer，再检查 `have_signals()` 来决定返回 `EINTR` 还是继续 yield <sup>[2](#mod-net-ref-2)</sup>。这是对 Linux `SO_RCVTIMEO` 和信号打断的模拟，但依赖 `starry_core::task` 模块的协作。

**SO_PEERCRED 对 Unix domain socket 的特化路径**：`api/src/imp/net/opt.rs:52-62` 中，`getsockopt` 在 `level=SOL_SOCKET`、`optname=SO_PEERCRED` 时优先走 `UnixSocket::peer_cred()`，避开通用的 Socket 路径 <sup>[7](#mod-net-ref-7)</sup>。这是 Unix domain socket 所独有的，通用 socket 不支持此选项。

### 3. 跨模块协同

与**文件描述符表**的协作：`Socket::from_fd` 通过 `get_file_like` 拿到 `Arc<dyn FileLike>`，经 `into_any().downcast::<Socket>()` 还原 <sup>[2](#mod-net-ref-2)</sup>。这种设计使 socket 与常规文件共享同一套 fd 分配/关闭逻辑，但也引入了动态分发开销。`close()` 方法利用 `Arc::strong_count` 判断是否最后一个引用——只有当强引用计数为 1 时才真正 `shutdown` <sup>[2](#mod-net-ref-2)</sup>，解决了 `netserver -D` 这类 fork/daemon 场景中父进程过早关闭 socket 的问题。

与**信号/timer 模块**的协作：阻塞 `accept()` 循环中调用 `starry_core::task::time_stat_refresh_itimer_real_in_kernel()` 刷新进程的 real interval timer，确保 `SIGALRM` 能按时送达 <sup>[2](#mod-net-ref-2)</sup>；`have_signals()` 检查待处理信号以支持 `EINTR` 返回。这解决了 LTP `TCP_CRR` 测试中 `SIGALRM` 打断阻塞 accept 的需求。

与**用户态指针封装**的协作：整个网络模块依赖 `UserPtr`/`UserConstPtr` 进行用户空间内存访问。`nullable!` 宏处理可能为 NULL 的指针 <sup>[6](#mod-net-ref-6)</sup>；`write_user_value` 统一通过 `UserPtr::write_bytes` 写入，避免了 loongarch64 PLV0 下直接写用户指针可能触发的页故障 <sup>[9](#mod-net-ref-9)</sup>。

### 4. 边角细节与不足

**`#[cfg(target_arch = "loongarch64")]` 的 sun_path 类型差异**：`api/src/socket.rs:291-304` 中 `UnixAddr::write_to_user_limited` 针对 loongarch64 将 `sun_path` 视为 `[i8; 108]` 而非 `[u8; 108]` <sup>[5](#mod-net-ref-5)</sup>。这是因为 linux-raw-sys 对 loongarch64 的 `sockaddr_un.sun_path` 定义为 `i8` 数组，而其他架构是 `u8`。未列出的架构（如 riscv32）若采用不同定义可能被错误覆盖。

**`SO_ERROR` 硬编码为 0**：`api/src/imp/net/opt.rs:68` 中 `getsockopt` 对 `SO_ERROR` 恒返回 0 <sup>[8](#mod-net-ref-8)</sup>。异步错误（如 `ECONNREFUSED`）不会被跟踪和报告，这对非阻塞 connect 场景构成功能缺失。

**`recvmsg` 的临时缓冲策略**：`api/src/imp/net/io.rs:89-99` 中 `recvmsg_impl` 先将 iovec 总长收进一个临时 `Vec<u8>`，再 scatter 到各 iovec <sup>[4](#mod-net-ref-4)</sup>。对大报文场景（如数十 MB）存在不必要的堆分配和拷贝，且可能触发 `total_iov_len` 中 `isize::MAX` 检查拒绝合理长度。

**`sendmmsg`/`recvmmsg` 的 timeout 忽略**：`api/src/imp/net/io.rs:179-180` 中 `sys_recvmmsg` 用 `let _ = timeout;` 静默丢弃超时参数 <sup>[4](#mod-net-ref-4)</sup>，不满足 POSIX 要求。同时 `sys_sendmmsg` 和 `sys_recvmmsg` 的逐条处理循环在部分成功时只返回成功条数，不设置 `msg_len` 为 0 的标记辅助用户判断哪条失败。

**Unix domain socket 的 nonblocking 未实现**：`api/src/imp/net/socket.rs:117-118` 中有显式 `// TODO: set nonblocking for UnixSocket` <sup>[6](#mod-net-ref-6)</sup>，且 `sys_socket` 对 AF_UNIX 分支直接跳过 `O_NONBLOCK` 设置，意味着 Unix domain socket 当前仅支持阻塞模式。

**`TCP_NODELAY` 被空吞**：`api/src/imp/net/opt.rs:131` 中 `setsockopt` 的 `TCP_NODELAY` 匹配空分支 `=> {}` <sup>[8](#mod-net-ref-8)</sup>，即忽略该选项。虽然不影响正确性，但 iperf3 等工具在使用 Nagle 算法禁用时不会报错也不会生效。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块是系统调用分发核心，依赖 `axhal::trap::register_trap_handler` 注册入口，使用 `syscalls::Sysno` 枚举和巨大的 `match` 将系统调用号分发给 `starry_api` 中的具体实现。与原始 ArceOS 相比，本实现通过条件编译限制了 x86_64 特有 syscall，并插入了任务级时间统计（`starry_core::task`），但也引入了若干 stub 实现和并发隐患。

### 1. 核心抽象与外部依赖

系统调用层的入口是 `handle_syscall`，它通过 `#[register_trap_handler(SYSCALL)]` 属性宏注册为架构相关 trap 处理函数<sup>[1](#mod-syscall-ref-1)</sup>。当 CPU 触发 `SYSCALL` 异常时，`axhal` 的 trap 框架将控制权交给此函数，并传入可变的 `TrapFrame` 和系统调用号。`TrapFrame` 承载了完整的寄存器上下文，用于提取参数和返回结果。系统调用号通过 `Sysno::new` 转换为 `syscalls::Sysno` 枚举<sup>[2](#mod-syscall-ref-2)</sup>，再交给 `handle_syscall_impl` 进行分发。

`handle_syscall_impl` 使用一个单层 `match` 将每个 `Sysno` 变体映射到 `starry_api::sys_*` 函数<sup>[3](#mod-syscall-ref-3)</sup>。所有 syscall 的返回值统一为 `LinuxResult<isize>`，最终在 `handle_syscall` 末尾通过 `unwrap_or_else(|err| -err.code() as _)` 转换为负数错误码返回给用户态<sup>[4](#mod-syscall-ref-4)</sup>。此外，在每个 syscall 前后分别调用 `time_stat_from_user_to_kernel()` 和 `time_stat_from_kernel_to_user()` 进行时间统计<sup>[5](#mod-syscall-ref-5)</sup>，这些函数由 `starry_core::task` 提供，用于任务级 profiling。

这种设计将架构相关细节（trap 向量、寄存器访问）封装在 `axhal` 中，将 POSIX 语义实现在 `starry_api` 中，而当前模块仅承担分发和参数适配，职责单一。

### 2. 关键设计取舍

**大 `match` 而非表驱动**：当前使用单个 `match` 枚举约 120 条 syscall。替代方案是构建一个函数指针表（如 `[Option<fn(...)>; SYS_MAX]`）。`match` 的优势在于编译期可检查枚举穷尽性（若 `Sysno` 非 `#[non_exhaustive]`），且每个 arm 都是独立的函数调用，容易内联；缺点是在 syscall 数量膨胀时导致编译速度下降和指令缓存压力。该设计表明项目早期更注重调试便利性和可读性，而非运行时性能优化。

**条件编译限制传统 syscall**：大量旧式 syscall（如 `open`、`mkdir`、`link`、`fork`）被包裹在 `#[cfg(target_arch = "x86_64")]` 之后<sup>[6](#mod-syscall-ref-6)</sup>。在非 x86_64 架构上，`syscalls::Sysno` 枚举本身不包含这些变体，因此编译时直接报错，迫使用户层改用 `openat`、`mkdirat` 等 _at 变体。这是一个主动的架构收敛设计，放弃了二进制兼容性，但简化了非 x86_64 的实现。缺点是约 20% 的 arm 受 cfg 影响，未来若支持 aarch64 或 riscv64 需仔细检查每个标记是否仍适应。

**Stub 实现**：部分 syscall、特别是 `set_mempolicy` 和 `mbind`，直接返回 `Ok(0)`<sup>[7](#mod-syscall-ref-7)</sup>。这些功能尚未实现，但为了保持用户程序不崩溃而静默成功。这是一种常见的早期开发妥协，但可能隐藏真正的资源泄漏或语义错误，对于 NUMA 感知的应用程序尤其危险。

**错误码边界**：`-err.code()` 中的 `err.code()` 返回 `i32`，取负后转为 `isize`。当 `err.code() == i32::MIN` 时，在 Rust debug 模式下会 panic。尽管当前所有错误码都来自 Linux 预定义范围（1~133），但未来可能引入自定义错误码，需留意这一潜在溢出点<sup>[4](#mod-syscall-ref-4)</sup>。

### 3. 跨模块协同

系统调用层是三层架构的中间层：

- **`axhal::trap`**：通过 `register_trap_handler` 宏在启动时安装 IDT/异常向量表，并将异常现场打包成 `TrapFrame`。该宏通常展开为 `#[no_mangle] extern "C" fn` 并写入中断描述符，因此任何对 `TrapFrame` 字段的修改都会直接影响用户态恢复后的上下文。
- **`starry_api`**：每个 `sys_*` 函数（如 `sys_write`）实现在独立的模块（如 `starry_api/src/fs.rs`）中，这些函数通常进一步调用底层文件系统、网络栈或调度器。当前模块与 `starry_api` 的协作完全通过函数签名耦合，没有额外的抽象层（如 trait），这意味着添加一个新 syscall 需要同时修改两个 crate。
- **`starry_core::task`**：时间统计函数 `time_stat_from_user_to_kernel` 和 `time_stat_from_kernel_to_user` 在 syscall 边界插入性能标记。根据命名推测，它们可能操作 percpu 的计数器，但当前 syscall 层中并没有显式的同步原语<sup>[5](#mod-syscall-ref-5)</sup>。如果这些函数跨核共享数据，则存在数据竞争风险。

### 4. 边角细节与不足

- **Stub 实现静默失败**<sup>[7](#mod-syscall-ref-7)</sup>: `set_mempolicy` 和 `mbind` 直接返回 0，NUMA 策略调用者无法区分“成功”和“未实现”。在分布式内存架构下，这可能导致后续内存访问性能下降且难以排查。
- **时间统计缺少 barrier**<sup>[5](#mod-syscall-ref-5)</sup>: `time_stat_from_user_to_kernel` 与 `time_stat_from_kernel_to_user` 之间没有内存屏障或原子操作。若统计变量非 percpu，在 SMP 环境下可能产生错误的 profiling 数据，甚至因乱序执行导致统计点标记错位。
- **条件编译散乱**<sup>[6](#mod-syscall-ref-6)</sup>: 超过 20 个 syscall arm 依赖 `#[cfg(target_arch = "x86_64")]`。当移植到新架构时，每个 cfg 都需要审计。当前没有一个集中的“架构特征列表”，容易漏改。此外，`Sysno::fstatat` 在非 x86_64 上使用 `Sysno::fstatat` 而在 x86_64 上使用 `Sysno::newfstatat`，这种命名差异增加了阅读负担。
- **`unwrap_or_else` 错误码溢出风险**<sup>[4](#mod-syscall-ref-4)</sup>: 如 §2 所述，`LinuxError::code()` 返回 `i32`，取负时 `i32::MIN` 会产生溢出 panic。虽然当前不触发，但这是靠枚举定义保证的隐式契约，没有静态断言。
- **缺少 syscall 表文档**: `match` 中没有分组注释或宏来列举已支持 syscall，新增时容易遗漏或冲突。当前依赖开发者手动检查。
