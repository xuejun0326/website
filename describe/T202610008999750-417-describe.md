## 📖 一、总览

`starry` 是一个面向 libc 兼容性测试的实验型内核，属于 arceos-starry 家族中“让测例先通过”的激进分支。它并非通用操作系统，而是围绕一套编译期注入的测例列表，以串行方式在 ArceOS 微内核基底上尽可能多地执行 Linux 用户程序 [boot.ref:3]。整体架构定位介于教学演示与 Linux 兼容层之间：代码量达 117k 行，涉及 101 个系统调用，覆盖文件、进程、内存、信号、网络、IPC 六大领域，但每一层都采用了面向测试的简化语义，而非追求 POSIX 完整性。

三个最值得评审注意的特点：
1. **内核任务至用户进程的 1:1 身份转换**：启动时直接将当前 `axtask` 的 ID 作为 init 进程 PID，无需 fork 或构造初始栈帧，极大地简化了进程创建路径，但也将内核调度实体与用户进程的生命周期强行耦合 [boot.ref:2]。
2. **编译期测例注入与 libc 自适应**：通过环境变量 `AX_TESTCASES_LIST` 在编译期指定要运行的 ELF 列表，并依据路径中的 `/glibc/` 或 `/musl/` 前缀动态选择对应的 libc 根文件系统，使得同一内核可同时测试两种 libc 的测例 [boot.ref:5]。
3. **围绕 fd 表的全局状态补丁**：由于底层 axfs/axnet 缺乏某些元数据字段，该仓通过全局 `TimestampOverrides`（跨进程 fd 编号冲突）、`HardlinkManager`、静态 loopback 队列等机制强行模拟 POSIX 行为，暴露出内核内部缺乏命名空间隔离的架构债务 [fs.ref:4][net.ref:25]。

## 🎯 二、综合评价

### 整体定位
`starry` 并非为生产或通用场景设计，而是一个 **libc 测试框架的内核载体**。其启动流程完全被 `main` 中的测试编排器支配——不提供 shell、没有用户交互，只负责批量执行预定义的 ELF 测例并收集结果 [boot.ref:3]。这种定位决定了整个系统的取舍：凡是测例需要的功能（如 shebang、BusyBox applet 别名、lmbench 路径重写）会被深入实现，而测例不触及的角落（如 ioctl、lstat 实际行为、完整的信号权限）则大量留白 [task.ref:8][fs.ref:3]。在该家族中，它比 ArceOS 基线离真正 Linux 兼容的目标更近，但距离一个能运行通用发行版的内核仍很远。

### 真正的创新点
多数模块在已有 crate（axprocess、axtask、axmm）上做了增量工作，但有三处设计体现了对测试场景的深度适配：
- **per-CPU 用户内存访问标记**：`ACCESSING_USER_MEM` 标志使得内核态能够安全地临时访问用户空间，支撑了 `atomic cmpxchg` 等需在内核中操作用户内存的系统调用，这是对传统宏内核用户拷贝机制的巧妙替代 [mm.ref:3]。
- **延迟信号投递与 POST_TRAP 钩子**：信号并非在发送时立即处理，而是在返回用户态前的陷阱钩子中统一下发。这种异步模型规避了微内核中信号抢占上下文切换的复杂性，且与测例的事件驱动特征吻合 [signal.ref:5]。
- **进程生命周期的 proc_trace 诊断**：通过 `proc_trace` 子系统输出进程创建、退出、僵尸收割等事件，为测试框架中的资源泄漏排查提供了关键的可观测性，这在学校内核中极为罕见 [task.ref:10]。

### 取舍判断
该仓把精力主要花在了 **让现有测例集尽可能多地通过**，而非构建通用的抽象。以下取舍尤为明显：
- **确定性优先于完整性**：启动采用编译期串行列表，避免并发带来的不确定性，这使测例结果可重复，但也导致内核完全无法作为通用系统启动 [boot.ref:3]。
- **全局状态优先于隔离性**：文件系统的时间戳覆盖、硬链接计数、网络 loopback 均使用全局表，丧失了多进程隔离性，但在测例串行执行的假设下不会暴露冲突 [fs.ref:4][net.ref:25]。
- **功能模拟优先于正确语义**：管道 `set_nonblocking` 为空实现、`sys_ioctl` 总是返回成功、`lstat` 行为与 `stat` 一致，这些简化使得部分程序“看上去能工作”，但破坏了 POSIX 承诺 [fs.ref:3][ipc.ref:9]。

### 完成度与不足
尽管系统调用覆盖面较广（101 个），完成度仍处于初期阶段：
- **内存管理** 缺少按需分页，堆区采用全量预映射，无法应对稀疏内存场景 [mm.ref:2]。
- **信号子系统** 的 Stop/Continue/CoreDump 均未实现，作业控制完全失效 [signal.ref:4]。
- **网络模块** 仅支持 AF_INET，且内置 loopback 表用无界 Vec 实现，存在内存耗尽风险 [net.ref:25]。
- **代码质量** 普遍存在 unsafe 裸指针操作未添加 SAFETY 注释、硬编码常量（如管道 1ms 轮询间隔、256 字节缓冲区）、以及大量静默忽略错误的 `warn!()` + `Ok(0)` 模式，不利于维护和演进 [boot.ref:1][ipc.ref:4]。
总体而言，`starry` 是一个高效达成“跑测试”目标的工程原型，若想演进为通用内核，需在架构隔离、错误处理及 POSIX 语义完整性上进行大量重构。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 本项目的启动流程是一个编译期配置的串行测例执行器: 内核通过 axhal 完成硬件初始化后进入 main, 以当前内核任务 ID 构建 init 进程的 1:1 映射, 随后从编译期环境变量 `AX_TESTCASES_LIST` 解析测例列表, 为每个测例按 libc 类型 (/glibc/ 或 /musl/) 构造环境变量并调用 `entry::run_user_app` 执行, 全程无并发、无文件系统依赖. 与标准 ArceOS 的宏内核启动相比, 本仓将 init 进程创建提前到用户程序执行之前, 并把启动后的控制流完全交给测试框架而非 shell 或 init 脚本.

### 1. 核心抽象: 内核任务到用户进程的 1:1 映射

启动流程的第一个关键动作是构建 init 进程. 在 `main` 入口处, 代码直接调用 `axprocess::Process::new_init(axtask::current().id().as_u64() as _).build()` <sup>[2](#mod-boot-ref-2)</sup>. 这行代码揭示了一个重要的设计抽象: ArceOS 的调度单元 `axtask` 与进程抽象 `axprocess::Process` 在此处形成了一一对应关系 — init 进程的标识符直接取自当前内核任务的 ID.

传统 Unix 系统中 init 进程 (PID=1) 由内核在启动末尾手工构造, 其 PID 是硬编码常量. 而本项目的做法是将"当前正在执行的调度实体"提升为第一个进程, 这意味着在进入 `main` 之前, ArceOS 的运行时 (`axruntime`) 已经创建了一个内核任务并开始调度. 这个任务在调用 `Process::build()` 后并未被替换或 fork — 它只是获得了一个进程外壳. 这种设计的优势是简化了进程创建路径: 不需要实现 fork 语义, 不需要构造虚假的初始栈帧; 劣势则是 init 进程与内核任务的生命周期耦合, 当后续用户程序退出时, 这个初始的 task-process 绑定如何回收成为隐含的约束.

### 2. 关键设计取舍: 编译期测例注入与串行执行模型

启动流程的第二阶段 — 也是占据 `main` 函数主体的部分 — 是一个编译期确定的测例循环. 测例列表通过 `option_env!("AX_TESTCASES_LIST")` 在编译时注入 <sup>[3](#mod-boot-ref-3)</sup>, 这意味着内核镜像本身是面向特定测试套件定制的, 而非通用操作系统. 这种取舍带来了以下连锁效应:

**编译期 vs 运行时配置.** 使用 `option_env!` 而非解析运行时配置文件 (如 /etc/init 或 bootargs), 使得内核无需在启动阶段挂载文件系统或读取块设备. 对于网络类型的系统软件, 这意味着网络栈尚未初始化之前就能开始执行用户程序 — 但也意味着更换测例需要重新编译内核.

**串行执行与错误隔离.** 测例通过 `for testcase in testcases` 循环逐个执行, 每个测例的退出码被单独记录但不会中止后续测例 <sup>[4](#mod-boot-ref-4)</sup>. 这种设计类似于测试框架的 "fail-fast: false" 策略: 一个崩溃的测例不会阻止其他测例的运行. 但串行模型也限制了系统的并发能力 — 即便硬件支持多核, 用户程序也只能逐个执行.

**libc 自适应环境构造.** `libc_root` 函数通过路径前缀匹配 (`/glibc/` vs `/musl/`) 来推断测例所需的 C 运行时库类型, 并据此设置 `LD_LIBRARY_PATH` <sup>[6](#mod-boot-ref-6)</sup> 和 `SHELL` 等环境变量 <sup>[5](#mod-boot-ref-5)</sup>. 这个机制的巧妙之处在于利用了测例路径的命名约定而非显式的元数据, 但其脆弱性也显而易见: 路径前缀的语义是隐含的, 若测例路径不符合 `/glibc/...` 或 `/musl/...` 模式, 动态链接器将无法找到正确的共享库.

### 3. 跨模块协同: 从 main 到用户程序的陷阱处理链路

虽然 `main` 函数本身只有约 70 行, 但它所触发的跨模块协同覆盖了整个内核的异常处理体系. boot facts 揭示的 handler 注册表说明了这一点:

当 `entry::run_user_app(&args, &envs)` 被调用时, 控制流从内核态切换到用户态. 此后发生的每一类异常都有对应的处理器:
- **SYSCALL**: `src/syscall.rs:10` 的 `handle_syscall` — 用户程序通过 `ecall` 指令发出的系统调用在此被分发;
- **PAGE_FAULT**: `src/mm.rs:12` 的 `handle_page_fault` — 缺页异常在此被处理, 可能触发页分配或 COW;
- **IRQ**: `vendor/arceos/modules/axhal/src/irq.rs:38` 的 `handler_irq` — 外设中断 (包括网卡) 在此被分发;
- **POST_TRAP**: `api/src/signal.rs:46` 的 `post_trap_callback` — 每次陷阱返回前检查是否有待投递的信号.

这四条路径共同构成了用户程序运行的"安全网". 值得注意的是, 这些 handler 的注册发生在 `main` 之前 (由 `axruntime` 和 `axhal` 完成), 而 `main` 本身并不感知这些机制的存在 — 它只是调用 `run_user_app` 然后等待返回. 这种"初始化与执行分离"的设计是 ArceOS 家族的典型模式: 启动流程只负责构造用户态上下文并跳转, 异常处理逻辑已在更早的阶段就位.

### 4. 边角细节与不足

**`#[unsafe(no_mangle)]` 缺少 SAFETY 注释** <sup>[1](#mod-boot-ref-1)</sup>. Rust 2024 edition 将 `no_mangle` 标记为 unsafe, 因为它要求符号名全局唯一且与 C ABI 兼容. 本代码使用了该属性但未附加 `// SAFETY:` 注释说明为何不会发生符号冲突. 在当前的单二进制场景下风险较低, 但若未来链接其他 object 文件, 符号名冲突将成为隐患.

**`unwrap_or_else` 中的死字符串** <sup>[3](#mod-boot-ref-3)</sup>. 当编译期未定义 `AX_TESTCASES_LIST` 时, `option_env!` 返回 `None`, 代码使用 `unwrap_or_else(|| "Please specify...")` 提供一个默认字符串. 但该字符串随后被 `.split(',')` 和 `.filter(|&x| !x.is_empty())` 处理, 结果是一个仅包含提示信息的单元素迭代器 — 而 `shlex::split("Please specify...")` 必然会解析失败, 最终整个循环体什么也不执行. 这个"错误提示"永远不会以有意义的方式呈现给用户, 属于误导性的 defensive programming.

**`shlex` 解析失败的静默处理** <sup>[4](#mod-boot-ref-4)</sup>. 对每个测例字符串调用 `shlex::split` 可能因引号不匹配等原因返回 `None`. 当前代码仅调用 `error!` 宏记录日志后 `continue`. 但对于一个包含上百测例的列表, 某些测例解析失败不会被汇总 — 开发者需要手动检查日志才能发现, 且退出码不会反映这些失败.

**硬编码的 libc 路径前缀** <sup>[5](#mod-boot-ref-5)</sup><sup>[6](#mod-boot-ref-6)</sup>. `/glibc/` 和 `/musl/` 两个字符串直接嵌入在 `libc_root` 函数的条件分支中. 如果需要支持新的 libc (如 `cosmopolitan` 或静态链接的 `musl-static`), 必须修改源代码. 更灵活的设计是使用一个 `(prefix, root, lib_path)` 的静态映射表.

**类型强转的隐式约定** <sup>[2](#mod-boot-ref-2)</sup>. `axtask::current().id().as_u64() as _` 中的 `as _` 让 Rust 编译器推断目标类型, 这依赖于 `Process::new_init` 的参数类型声明. 若该接口未来改为接受 `u32` 或 `usize`, 此处可能在 64 位平台上发生截断而没有任何编译警告. 显式标注目标类型或使用 `try_from` 会是更安全的做法.

**缺乏测例间的资源清理.** `main` 中的循环在每次 `run_user_app` 返回后直接进入下一次迭代, 没有显式的进程清理、内存回收或文件描述符关闭步骤. 如果 `run_user_app` 内部没有完善的资源回收机制, 连续运行大量测例可能导致资源泄漏 — 尤其在网络类型的测例中, socket 和缓冲区是有限的.

> 💡 **TL;DR**
>
> 该模块以 ArceOS 的 axtask/axprocess 为基础, 通过三层扩展结构 (TaskExt / ProcessData / ThreadData) 将裸任务骨架粘合为 Linux 兼容的进程-线程模型. 核心解决三个问题: (1) clone/fork 的地址空间与资源复用或复制, (2) execve 的镜像加载与地址空间原子替换, (3) exit/wait 的僵尸收割与父子信号通知. 与 ArceOS 基线相比, 本仓库新增了 shebang 展开、BusyBox applet 别名解析、lmbench 路径重写等面向测评镜像的兼容层, 并通过 proc_trace 子系统输出进程生命周期事件以辅助诊断资源泄漏.

### 1. 核心抽象: 三层扩展结构与全局表

进程调度模块并非从零构建, 而是在 `axtask::TaskInner` 之上叠加三层扩展数据 <sup>[8](#mod-task-ref-8)</sup>. 最底层是 `TaskExt`, 通过 `axtask::def_task_ext!` 注册为全局任务扩展, 持有一个 `Arc<Thread>` 和 `TimeStat` 时间统计 <sup>[1](#mod-task-ref-1)</sup>. `Thread` 自身来自 `axprocess` crate, 可附加 `ThreadData` — 包含 `clear_child_tid`、`robust_list` 元数据、线程级信号管理器. `Process` 则可附加 `ProcessData` — 包含地址空间 (`Arc<Mutex<AddrSpace>>`)、命名空间 (`AxNamespace`)、进程级信号管理器、futex 表、子进程退出等待队列等.

四张全局弱引用表 (`THREAD_TABLE`, `PROCESS_TABLE`, `PROCESS_GROUP_TABLE`, `SESSION_TABLE`) 使用 `WeakMap` 维护 PID/TID 到对象的映射 <sup>[10](#mod-task-ref-10)</sup>. 选择 `Weak` 而非 `Arc` 是为了避免循环引用导致进程无法回收: 当所有强引用释放后, 表项自动消失. 代价是查询时需通过 `get()` 升级为 `Arc`, 若对象已回收则返回 `ESRCH`.

### 2. 关键设计取舍

**clone 的分岔逻辑.** `sys_clone` 根据 `CLONE_THREAD` 标志选择两条路径 <sup>[2](#mod-task-ref-2)</sup>. 若置位, 新任务直接复用当前进程的地址空间 (仅复制页表根指针) 和 `Process` 对象, 成为同一进程的新线程. 否则走 `Process::fork()` 创建新进程, 并根据 `CLONE_VM/SIGHAND/FILES/FS` 等标志决定地址空间、信号处理器、文件描述符表、当前目录的共享或复制. 地址空间复制使用 `AddrSpace::try_clone()` 浅拷贝页表再按需触发 COW, 失败时通过 `proc_trace` 打点后返回错误 <sup>[11](#mod-task-ref-11)</sup>. 一个值得注意的细节: `CloneFlags` 用 `from_bits_truncate` 解析, 未识别的标志位会被静默丢弃, 这降低了新 flag 引入时立即报错的概率, 但也避免了因未知 flag 导致 clone 整体失败.

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 该模块在 ArceOS axfs 层之上构建 Linux 兼容文件系统接口, 核心抽象是 `FileLike` trait (统一 File/Directory/Pipe/Socket) 与基于 `FlattenObjects` 的进程 fd 表 `FD_TABLE`. 它通过内存态的 `TimestampOverrides` 记录 utimensat 时间戳覆盖 (因 lwext4 不暴露可写时间戳), 以全局 `HardlinkManager` 管理硬链接引用计数, 并通过 BusyBox applet 别名机制将 `/bin/ls` 等标准路径映射到真实 BusyBox ELF. 与原版 ArceOS 相比, 该仓新增了路径追踪 (path_trace)、进程级 fd 限制 (FdLimit)、O_TMPFILE 命名兜底、以及 pselect6 协作式调度适配.

### 1. 核心抽象: FileLike 与 FD_TABLE

文件系统模块以 `FileLike` trait 作为统一抽象, 将所有可暴露为文件描述符的对象 — 普通文件 `File`、目录 `Directory`、管道 `Pipe`、套接字 `Socket` — 收敛到同一接口: `read`/`write`/`stat`/`poll`/`set_nonblocking` 以及用于向下转型的 `into_any` <sup>[1](#mod-fs-ref-1)</sup>. 每个打开的文件由 `FdEntry` 包裹, 同时存储 fd 级别标志 (`FD_CLOEXEC`) 和文件状态标志 (`O_APPEND`/`O_NONBLOCK` 等) <sup>[2](#mod-fs-ref-2)</sup>.

进程 fd 表 `FD_TABLE` 使用 `flatten_objects::FlattenObjects` — 一个带 slot 复用功能的稀疏数组, 上限 `AX_FILE_LIMIT` (1024). 与 Linux 的位图+指针数组不同, `FlattenObjects` 提供 `is_assigned`/`add_at`/`remove` 实现 O(1) 分配回收, 代价是内部维护空闲链表. fd 软硬上限通过 `FD_LIMIT` (`ResArc<RwLock<FdLimit>>`) 管理, 支持 `F_DUPFD` 从指定最小值开始分配 <sup>[3](#mod-fs-ref-3)</sup>. `ResArc` (来自 axns crate) 是命名空间感知的引用计数资源包装, 使得 fork 时可通过 `copy_inner` 浅拷贝 fd 条目实现父子进程共享文件对象.

### 2. 关键设计取舍

**时间戳覆盖 (Timestamp Overrides).** 底层 lwext4 的 axfs 封装未暴露可写的 atime/mtime/ctime 接口. 模块选择在内存中维护全局 `TimestampOverrides` 表 (路径→时间戳, fd→时间戳), `sys_utimensat` 写入覆盖值, `sys_stat`/`sys_fstat`/`sys_statx` 读取时叠加 <sup>[4](#mod-fs-ref-4)</sup>. 这放弃了持久化语义 — 重启后时间戳覆盖丢失 — 但避免了侵入式修改底层文件系统. `futimens` 调用时额外将 fd 形式的时间戳同步写入路径映射 (`path_of_fd`), 保证后续按路径 stat 能看到更新时间 <sup>[5](#mod-fs-ref-5)</sup>.

**硬链接管理器 (HardlinkManager).** 同样因底层 fs 不直接支持硬链接, 模块引入全局 `HARDLINK_MANAGER` 维护 `BTreeMap<String, String>` (新路径→目标路径) 及引用计数 <sup>[6](#mod-fs-ref-6)</sup>. 但 `create_link` 中的 `!dst.is_dir()` 判断依赖 `FilePath::is_dir()` — 该函数仅检查路径字符串是否以 `/` 结尾 <sup>[7](#mod-fs-ref-7)</sup>, 而非查询文件系统元数据. 对 `/bin/ls` 这类路径, `is_dir()` 返回 false, 导致 `!dst.is_dir()` 为 true, 直接返回 `LinkError::NotFile`, 与注释"目标路径不是文件则返回 NotFile"的意图相悖 — 该判断实际上拒绝了所有不以 `/` 结尾的目标路径.

**O_TMPFILE 兜底.** Linux `O_TMPFILE` 创建无链接匿名 inode; 当前 VFS 无此能力, 模块改为在目标目录创建命名文件 `".starry-tmpfile-{pid}-{tid}-{seq}"` <sup>[8](#mod-fs-ref-8)</sup>. 这不是原子操作, 进程崩溃会留下垃圾文件.

**BusyBox applet 别名.** 为减少对完整 procfs/symlink 层的依赖, 当 `openat`/`stat` 解析 `/bin/ls` 等标准路径失败时, 根据当前进程 libc 根 (`/musl` 或 `/glibc`) 将其映射到对应 BusyBox ELF <sup>[9](#mod-fs-ref-9)</sup>. 白名单 `BUSYBOX_APPLETS` 硬编码 38 个命令, `bash` 特殊映射为 `sh`. 这使测评脚本无需独立二进制即可运行常见命令, 但白名单外的命令直接返回原始错误.

### 3. 跨模块协同

**路径解析链.** `handle_file_path` 是核心路径解析函数, 根据 `dirfd` 和 `path` 计算 `FilePath`: 绝对路径直接规范化, 空路径取 fd 对应路径, 相对路径与 dirfd 对应目录拼接 <sup>[10](#mod-fs-ref-10)</sup>. `FilePath::new` 内部调用 `axfs::api::canonicalize` 规范化, 再经 `HARDLINK_MANAGER.real_path` 解析硬链接重定向. 所有 fs 系统调用 (`openat`/`stat`/`mkdirat`/`unlinkat`/`renameat2`/`readlinkat`) 均经由此函数统一入口, 确保 dirfd 语义一致.

**poll/pselect6 的协作式适配.** `sys_ppoll` 仅执行即时就绪查询, 不处理 timeout 等待或信号屏蔽 <sup>[11](#mod-fs-ref-11)</sup>. `sys_pselect6` 在无 fd 就绪时根据 timeout 调用 `axtask::sleep` 或 `axtask::yield_now`, 其中 `timeout=0` 时额外让出一次 CPU, 以避免 lmbench 等 busy-poll 用户态在协作式调度下饿住写端任务 <sup>[12](#mod-fs-ref-12)</sup>.

**rename 的类型匹配.** `sys_renameat2` 在目标路径已存在时进行四象限匹配: 目录→目录先 rmdir 再 rename, 目录→文件返回 `ENOTDIR`, 文件→目录返回 `EISDIR`, 文件→文件先 unlink 再 rename <sup>[13](#mod-fs-ref-13)</sup>. 当前仅支持 `RENAME_NOREPLACE`; `RENAME_EXCHANGE` 和 `RENAME_WHITEOUT` 直接返回 `EINVAL`.

### 4. 边角细节与不足

1. **`sys_ioctl` 是无声空操作** <sup>[14](#mod-fs-ref-14)</sup>: 任何 fd 的 ioctl 调用均返回 0 且仅打 warn 日志, 导致依赖 ioctl 的应用 (如终端控制) 静默失败.

2. **`sys_lstat` 与 `sys_stat` 行为完全相同** <sup>[15](#mod-fs-ref-15)</sup>: 注释标注 `// TODO: symlink`, 对符号链接调用 lstat 会穿透而非返回链接自身属性, 违反 POSIX 语义.

3. **`sys_readlinkat` 仅处理 `/proc/self/exe`** <sup>[16](#mod-fs-ref-16)</sup>: 对其他任何路径返回 `EINVAL`, 完全未实现通用符号链接读取. 注释说明仅为满足 glibc 动态链接器探测需求.

4. **`sys_faccessat` 不校验权限** <sup>[17](#mod-fs-ref-17)</sup>: 注释承认"当前文件系统权限模型尚不完整", `R_OK`/`W_OK`/`X_OK` 仅验证参数合法性, 实际不检查调用者是否有访问权限.

5. **`sys_fcntl` 对不支持的命令静默返回 0** <sup>[18](#mod-fs-ref-18)</sup>: `F_GETOWN`/`F_SETOWN` 等与信号驱动 I/O 相关的命令会误判成功, 可能导致应用程序行为异常.

6. **`statfs_for_path` 返回硬编码近似值** <sup>[19](#mod-fs-ref-19)</sup>: `f_blocks`/`f_bfree`/`f_files` 等字段是固定常量, `df` 等工具能运行但输出不准确.

7. **`init_stdio` 中 `panic!()` 作为错误处理** <sup>[20](#mod-fs-ref-20)</sup>: 若 fd 0/1/2 初始化失败, 内核直接 panic, 在生产环境不可接受.

8. **`TimestampOverrides` 的 fd 索引全局冲突** <sup>[4](#mod-fs-ref-4)</sup>: `RwLock<BTreeMap<c_int, FileTimestamps>>` 以 fd 为键, 但 fd 是进程内资源; 多进程场景下不同进程的相同 fd 编号会互相污染时间戳覆盖.

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块在 arceos-starry 的微内核任务模型上实现了 Linux 兼容的信号子系统。核心抽象来自 axsignal crate (SignalInfo/SignalSet/Signo/SignalOSAction), 信号状态按线程 (阻塞掩码/挂起队列/altstack) 与进程 (共享 handler 表) 分层存储, 通过 axtask 的 TaskExtRef 机制访问。信号投递采用延迟模型: 发送方仅入队, 实际派发发生在目标线程返回用户态前的 POST_TRAP 钩子中。与标准 ArceOS 相比, 该仓新增了完整的 rt_sig* 系列系统调用及 kill 变体的权限/存在性语义, 但权限检查、CoreDump、Stop/Continue 仍多处标注 TODO。

### 1. 核心抽象: axsignal 类型体系与星云架构的融合

信号子系统建立在 axsignal crate 提供的四个核心类型上: `SignalInfo` (信号实例, 含编号/发送者/附加数据), `SignalSet` (位掩码, 表示阻塞或挂起集合), `SignalStack` (替代栈描述符, 含栈底指针与大小), `Signo` (信号编号枚举, 实现 `from_repr`)。这些类型通过 `linux_raw_sys::general` 的 C ABI 类型 (`kernel_sigaction`, `siginfo`, `timespec`) 与用户态交互, 在系统调用边界进行 `TryInto`/`to_ctype` 转换 <sup>[1](#mod-signal-ref-1)</sup>。

与 Linux 的 `task_struct` 将信号状态内嵌不同, arceos-starry 采用分层设计: 每线程拥有独立的 `ThreadData.signal` (阻塞掩码、挂起信号集、替代栈), 每进程共享 `ProcessData.signal.actions` (信号处理动作表, 受 Mutex 保护) <sup>[2](#mod-signal-ref-2)</sup> <sup>[3](#mod-signal-ref-3)</sup>。通过 `axtask::TaskExtRef` trait, 当前任务可统一通过 `current().task_ext().thread_data()` / `process_data()` 访问这些扩展字段。这种设计使调度器 crate (axtask) 无需感知 POSIX 语义, 同时保证信号状态有明确的类型和所有权边界。

`SignalOSAction` 枚举 (Terminate / CoreDump / Stop / Continue / Handler) 将用户的 `sa_handler` 配置转化为内核执行决策。关键在于 `Handler` 动作: 它表示用户态处理函数已由 axsignal 的跳板 (trampoline) 安装, 此时上层 `check_signals` 只需返回 `true` 让线程继续运行 —— 实际的信号栈帧构造和 TrapFrame 篡改发生在 `thread_data().signal.check_signals()` 内部 <sup>[4](#mod-signal-ref-4)</sup>。

### 2. 信号投递: "发送-检查"两阶段与 POST_TRAP 延迟派发

信号投递被拆分为两个阶段。**发送阶段** (`send_signal_thread` / `send_signal_process` / `send_signal_process_group`) 仅将 `SignalInfo` 入队到目标线程的挂起集, 不立即触发处理 <sup>[5](#mod-signal-ref-5)</sup>。发送函数通过 `thr.data::<ThreadData>()` 获取线程扩展数据, 失败时返回 `EPERM` —— 这发生在目标线程尚未初始化 `ThreadData` 的竞态窗口。

**检查阶段** 发生在目标线程即将返回用户态时。`post_trap_callback` 通过 `#[register_trap_handler(POST_TRAP)]` 注册为 axhal 的陷阱后回调, 仅在 `from_user == true` 时调用 `check_signals(tf, None)` <sup>[6](#mod-signal-ref-6)</sup>。`check_signals` 调用 `thread_data().signal.check_signals(tf, restore_blocked)` 从挂起集中弹出最高优先级的未被阻塞信号, 根据 `SignalOSAction` 分派: 若是 `Terminate` 或 `CoreDump`, 直接调用 `do_exit(128 + signo)` 终结进程; 若是 `Stop`, 当前错误地以 `do_exit(1)` 处理; 若是 `Continue`, 则为空操作 <sup>[7](#mod-signal-ref-7)</sup>。

这种延迟模型避免了发送方在未知目标调度状态时执行复杂的状态修改, 与 Linux 的 `TIF_SIGPENDING` + `do_signal()` 路径理念一致。唯一的同步等待发生在 `sys_rt_sigsuspend`: 它原子地替换阻塞掩码为传入集合 (显式移除 SIGKILL/SIGSTOP), 设置 TrapFrame 返回值为 `-EINTR`, 然后进入 `loop { check_signals(tf, Some(old_blocked)); process_data().signal.wait_signal() }` <sup>[8](#mod-signal-ref-8)</sup>。当信号到达且被处理时, `check_signals` 返回 `true` 跳出循环; 之后 sigsuspend 恢复旧阻塞掩码并返回 `-EINTR`。

### 3. 关键系统调用的实现取舍

**sigaction** (行 58-78): 操作进程级 `actions` 表, 双层互斥 —— 外层通过 `Mutex<actions>` 保证线程安全, 内层通过 `TryInto` 拒绝不支持的 `sa_flags` 而非静默忽略 <sup>[3](#mod-signal-ref-3)</sup>。SIGKILL 和 SIGSTOP 的修改被明确拒绝 (`EINVAL`), 符合 POSIX。

**sigprocmask** (行 34-56): 线程级操作, 使用 `with_blocked_mut` 闭包模式持有锁仅跨临界区, 避免跨用户指针访问时持锁 <sup>[2](#mod-signal-ref-2)</sup>。`how` 参数支持 `SIG_BLOCK` (或)、`SIG_UNBLOCK` (与非)、`SIG_SETMASK` (赋值) 三种操作, 集合运算直接使用 `|=` / `&=` / `=`。

**kill 系列** (行 112-189): `sys_kill` 实现了完整的四向 pid 语义 (pid>0: 单进程; pid==0: 当前进程组; pid==-1: 广播除 init 外; pid<-1: 指定进程组) <sup>[9](#mod-signal-ref-9)</sup>。一个值得注意的妥协是: 当目标进程已退出但发送的是真实信号 (sig.is_some()) 时, 返回 `Ok(0)` 而非 `ESRCH`, 这是为兼容 BusyBox 固定 PID 测例引入的变通 <sup>[10](#mod-signal-ref-10)</sup>。`sys_tkill` 和 `sys_tgkill` 分别向线程和线程组内特定线程发送信号, 均标注了权限检查 TODO。

**sigreturn** (行 240-248): 完全委托给 `signal.restore(tf)`, 从用户栈恢复保存的 TrapFrame, 返回值取自 `tf.retval()` <sup>[11](#mod-signal-ref-11)</sup>。

**sigaltstack** (行 304-324): 设置替代栈前进行两重验证: 大小 ≥ `MINSIGSTKSZ` (Linux 定义), 栈指针指向的内存区域可访问 (通过 `get_as_slice` 探测) <sup>[12](#mod-signal-ref-12)</sup>。这防止了安装指向无效内存的栈, 但探测只确认页存在 —— 不保证信号投递时该区域仍可写。

### 4. 边角细节与不足

**<sup>[13](#mod-signal-ref-13)</sup> 权限检查全局缺失**: `sys_kill` (行 124), `sys_tkill` (行 166), `sys_tgkill` (行 178) 三处均有 `// TODO: 未实现 Linux 权限检查` 注释。Linux 要求发送者 uid/gid 与目标匹配或具备 `CAP_KILL`, 当前实现允许任意进程向任意目标发送任意信号, 构成权限旁路。

**<sup>[14](#mod-signal-ref-14)</sup> 信号编号 u8 截断**: `parse_signo` (行 31) 将 `u32` 强转为 `u8` 再通过 `Signo::from_repr` 查找。虽然 SIGRTMIN(32)~SIGRTMAX(64) 在 u8 范围内, 但若传入超大值 (如 512), u8 截断为 0 后 `from_repr` 返回 `None` → `EINVAL`, 行为与 Linux 的 `EINVAL` 一致但路径隐蔽; 若 `Signo` 未来扩展至 128+ 信号, u8 将成为瓶颈。

**<sup>[15](#mod-signal-ref-15)</sup> CoreDump/Stop/Continue 残缺**: `check_signals` (行 29-39) 中 `CoreDump` 降级为普通 `do_exit`, `Stop` 错误地调用 `do_exit(1)` 而非进入停止状态, `Continue` 为空操作。这意味着 SIGSTOP/SIGCONT 作业控制完全失效, 影响 shell 的 `Ctrl-Z` / `fg` / `bg` 功能。

**<sup>[16](#mod-signal-ref-16)</sup> sigsuspend 忙等风险**: `sys_rt_sigsuspend` (行 290-297) 的 `loop` 中每次唤醒都重入 `check_signals`, 若无期望信号则再调用 `wait_signal()` 阻塞。若 `wait_signal()` 是 yield 而非 sleep, 或者被虚假唤醒频繁触发, 将造成内核态空转。

**<sup>[17](#mod-signal-ref-17)</sup> kill -1 的 O(N) 信号克隆**: `sys_kill` 的 `-1` 分支 (行 139-145) 遍历全部进程并对每个调用 `sig.clone()`。`SignalInfo` 的 clone 可能涉及堆分配; 在进程数极多的系统上, 单次 `kill -1` 即可触发大量分配, 在 OOM 条件下形成 DoS 向量。

**<sup>[18](#mod-signal-ref-18)</sup> ESRCH 静默吞掉导致错误不可观测**: `sys_kill` (行 134-135) 中, 目标进程不存在但信号非零时返回成功, 调用者无法区分"信号已投递"和"目标已退出"。这虽然解决了 BusyBox 兼容性问题, 但掩盖了真实竞态 —— 例如监控程序可能误认为信号已被目标接收。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块实现了两种 IPC 机制: (1) Unix 管道 (Pipe) — 基于 256 字节固定环形缓冲区 + WaitQueue 的阻塞式字节流, 实现 FileLike trait 接入文件描述符表; (2) Linux futex — 以进程为粒度的地址→WaitQueue 映射表 (BTreeMap), 支持 WAIT/WAKE/REQUEUE/CMP_REQUEUE 四个基本操作, 通过 WaitQueueGuard 的 RAII 回收空闲队列. 与 ArceOS 基线相比, 本仓独立实现了完整的管道层 (含信号中断路径) 和 futex 表, 但缺少 PI/BITSET 等扩展操作, 且管道采用 1ms 轮询等待而非真阻塞.

### 1. 核心抽象: 管道与 futex 共享的 WaitQueue 原语

进程间通信模块围绕 `axtask::WaitQueue` 这一共同原语构建了两套上层接口. 管道路径将 WaitQueue 封装在 `PipeShared` 中, 分别维护读端等待队列 `read_wq` 和写端等待队列 `write_wq` <sup>[1](#mod-ipc-ref-1)</sup>; futex 路径则将 WaitQueue 以用户态地址为键存入 `FutexTable` (底层为 `Mutex<BTreeMap<usize, Arc<WaitQueue>>>`) <sup>[5](#mod-ipc-ref-5)</sup>. 二者均通过 `Arc<WaitQueue>` 共享所有权, 使得等待者与被等待者之间不存在生命周期耦合.

管道与 futex 的目标场景完全不同: 管道面向父子/兄弟进程间的流式数据传输, 依托文件描述符体系暴露给用户态; futex 面向线程同步, 直接操作用户态内存地址. 但二者在"阻塞—唤醒"这一核心路径上共享了完全相同的机制: 调用 `WaitQueue::wait()` 或 `wait_timeout()` 挂起当前任务, 由对端通过 `notify_one()` / `notify_all()` 唤醒 <sup>[2](#mod-ipc-ref-2)</sup><sup>[7](#mod-ipc-ref-7)</sup>. 这种共享避免了在两个子系统间重复实现调度原语, 但也意味着 WaitQueue 的任何缺陷会同时影响两条 IPC 路径.

### 2. 管道设计: 小环缓冲区 + 轮询伪阻塞

管道实现全部集中在 `api/src/file/pipe.rs`. 其数据结构为经典的环形缓冲区: `PipeRingBuffer` 维护一个 `[u8; 256]` 数组与 head/tail 指针, 状态机包含 Full/Empty/Normal 三态 <sup>[3](#mod-ipc-ref-3)</sup>. `PipeShared` 将缓冲区置于 `Mutex` 保护下, 同一时刻只允许一个读/写者进入临界区 <sup>[1](#mod-ipc-ref-1)</sup>.

该设计最关键的取舍在于"阻塞"语义的实现方式. 当读端发现缓冲区空且写端未关闭时, 它并非将任务真正挂起等待事件通知, 而是**释放锁后调用 `wait_timeout(PIPE_WAIT_SLICE)`** — 其中 `PIPE_WAIT_SLICE` 硬编码为 1 毫秒 <sup>[4](#mod-ipc-ref-4)</sup>. 这意味着管道等待本质上是 **1ms 粒度的轮询**: 任务每隔 1ms 被唤醒、重新获取锁、检查缓冲区状态, 若仍不满足则继续睡眠. 这种设计放弃了真阻塞带来的即时唤醒优势 (延迟上限为 1ms), 换取了实现简单性 — 不需要在环形缓冲区状态变化时精确唤醒对应等待者 (虽然代码中确实在读写完成后调用了 `notify_one`, 但 `wait_timeout` 的超时机制使得即使 notify 丢失, 任务也会在 1ms 后自行重试).

轮询设计的一个直接后果是 CPU 唤醒频率与管道吞吐量之间形成张力: 减小 `PIPE_WAIT_SLICE` 可降低延迟但增加上下文切换开销; 增大则反之. 当前 1ms 取值未见平台相关的参数化说明. 另一个后果是 `set_nonblocking` 方法为空实现 <sup>[9](#mod-ipc-ref-9)</sup> — 因为底层根本不区分阻塞与非阻塞模式, 所有 I/O 都走同一套 `wait_timeout` 循环.

### 3. Futex 设计: 进程级地址键控与 RAII 回收

Futex 实现分为两层: `core/src/futex.rs` 提供数据结构 `FutexTable`, `api/src/imp/futex.rs` 负责解析系统调用参数并编排操作. `FutexTable` 是一个包装了 `Mutex<BTreeMap<usize, Arc<WaitQueue>>>` 的新类型, 提供 `get` 和 `get_or_insert` 两个方法, 均返回 `WaitQueueGuard` <sup>[5](#mod-ipc-ref-5)</sup>.

`WaitQueueGuard` 是本模块最精巧的设计: 它通过 `Deref` 暴露底层 `Arc<WaitQueue>`, 允许调用者直接使用其方法; 同时在 `Drop` 中检查 `Arc::strong_count == 1 && is_empty()`, 若成立则从 BTreeMap 中移除该条目 <sup>[8](#mod-ipc-ref-8)</sup>. 这实现了**自动垃圾回收** — 当所有等待者都离开且无其他引用时, 队列被静默清理, 避免 BTreeMap 无限增长. 但这也引入了一个隐式约束: 持有 `WaitQueueGuard` 的代码不能在 guard 存活期间让出 CPU 并期望 futex 地址映射保持不变 (实际路径中 guard 生命周期很短, 不会跨 `wait()` 持有, 因此安全).

`sys_futex` 的 REQUEUE 路径值得仔细审视 <sup>[7](#mod-ipc-ref-7)</sup>. 它先从源队列唤醒最多 `value` 个等待者; **只有当实际唤醒数等于 `value` (即全部唤醒成功) 时**, 才会将剩余最多 `value2` 个等待者迁移到目标队列. 这与 Linux 手册中 "wake up to val waiters, then requeue up to val2" 的语义一致. 但 `value2` 的取值来自 `timeout.address().as_usize() as u32` — 这是 Linux futex ABI 中复用 timeout 指针传递 `val2` 的标准约定, 代码正确遵循了这一约定.

一个显著的架构限制: `FutexTable` 存储在 `current().task_ext().process_data()` 中 <sup>[6](#mod-ipc-ref-6)</sup>, 即**每个进程拥有独立的 futex 表**. 这意味着两个进程即使通过 `mmap` 共享了同一物理页, 也无法在该共享内存上使用 futex 进行跨进程同步 — 因为它们的 futex 表互不可见. 对于需要跨进程 futex 的场景 (如基于共享内存的进程间互斥锁), 这是致命的. 这一设计选择可能是为了简化实现 (避免全局 futex 表的地址空间冲突), 但严重限制了 futex 的适用范围.

### 4. 跨模块协同: 文件系统、信号与调度

管道路径展示了三个模块的紧耦合:

- **文件描述符层**: `sys_pipe2` <sup>[10](#mod-ipc-ref-10)</sup> 创建 `Pipe` 对后, 分别调用 `add_to_fd_table()` 将读/写端注册到当前进程的 fd 表. 注册失败时通过 `inspect_err` 闭包调用 `close_file_like` 回滚已分配的描述符, 保证原子性. `Pipe` 实现 `FileLike` trait, 这使得 `read`/`write`/`poll` 系统调用可以统一分派, 无需专门管道处理路径.

- **信号系统**: 读写循环每次获取锁后都调用 `has_pending_signal()` <sup>[12](#mod-ipc-ref-12)</sup> 检查是否有待处理信号. 检查对象是 `current().task_ext().thread_data().signal.pending()`. 若发现信号, 立即返回 `EINTR`. 这支持了 alarm/setitimer 等定时信号打断阻塞管道的场景 (如 lmbench 的 alarm 依赖).

- **Drop 语义**: `Pipe::drop` <sup>[11](#mod-ipc-ref-11)</sup> 在任一端释放时同时通知 `read_wq` 和 `write_wq`, 确保对端能及时感知到管道破裂 — 读端看到 EOF (read 返回 0), 写端看到 EPIPE (closed 检查触发).

Futex 路径同样涉入信号: `FUTEX_WAIT` 在比较 `*uaddr` 与 `value` 失败时返回 `EAGAIN` 而非阻塞, 这与 Linux 语义一致 <sup>[6](#mod-ipc-ref-6)</sup>. 但 futex 等待路径中**没有显式的信号检查** — 这依赖于 `WaitQueue::wait()` 内部是否处理信号唤醒. 若底层不支持, 则信号无法打断 futex 等待, 构成一个潜在的 POSIX 合规性问题.

### 5. 边角细节与不足

<sup>[4](#mod-ipc-ref-4)</sup> **硬编码常量缺乏参数化**: `RING_BUFFER_SIZE = 256` 字节远小于 Linux 默认的 64KB, 且无任何配置机制. `PIPE_WAIT_SLICE = 1ms` 同样硬编码, 未与调度器 tick 频率对齐.

<sup>[9](#mod-ipc-ref-9)</sup> **非阻塞模式未实现**: `set_nonblocking` 是空函数. 任何通过 `fcntl(fd, F_SETFL, O_NONBLOCK)` 设置的管道非阻塞标志都会被静默忽略, 导致依赖非阻塞 I/O 的应用 (如 shell 管道链) 行为异常.

<sup>[6](#mod-ipc-ref-6)</sup> **futex 表进程级作用域**: 如前所述, 这阻止了基于共享内存的跨进程 futex. 替代方案 (全局表 + 地址命名空间) 需要处理不同进程地址重叠问题, 实现复杂度更高, 本实现选择了简单性.

<sup>[7](#mod-ipc-ref-7)</sup> **REQUEUE 路径的边界条件**: `if count == value as isize` 中的 `count` 和 `value` 分别为 `isize` 和 `u32`. 当 `value > isize::MAX` 时比较将溢出; 实际上 `value` 来自用户态 `u32`, Linux 定义 `val` 为 `u32`, 但本处转换未做饱和处理.

<sup>[10](#mod-ipc-ref-10)</sup> **`sys_pipe2` 未实现 flags**: 函数签名接受 `flags: i32` 参数, 但仅对非零值输出警告后忽略. 这意味着 `O_CLOEXEC` 和 `O_DIRECT` (pipe2 的核心价值) 均无效, 导致 fd 泄漏风险.

<sup>[8](#mod-ipc-ref-8)</sup> **WaitQueueGuard 的 ABA 回收窗口**: 在 `get_or_insert` 中获取 guard 后, 若另一个线程同时删除了同一地址的队列并创建了新的, 引用计数检查 `strong_count == 1` 可能在新队列上误触发回收. 不过由于外层的 `Mutex` 保护 BTreeMap, 这一问题实际不会发生 — 但代码未添加注释说明这一安全性依赖.

此外, 系统调用表中列出了 `get_robust_list`/`set_robust_list` <sup>[13](#mod-ipc-ref-13)</sup>, 但其实现文件未在已读范围内, 可能为存根或委派到其他 crate.

## 🌐 九、网络

> 💡 **TL;DR**
>
> 模块以 `Socket` 枚举封装 axnet 的 TCP/UDP socket，通过 `FileLike` trait 挂入 fd 表实现与文件系统的统一访问。为绕过 axnet/smoltcp 不支持同机 loopback 的限制，内置四张全局静态表模拟最小 UDP/TCP 本地回环，支撑 libctest 测例。地址转换层定义 `SocketAddrExt` trait，处理 sockaddr 与用户态的读写边界。整体呈现'先让测例通过'的早期实现风格，多处存在语义简化和未完成标记。

### 1. 核心抽象：Socket 枚举与 FileLike 的适配层

模块的根抽象是 `Socket` 枚举，其两个变体分别包裹 `axnet::UdpSocket` 和 `axnet::TcpSocket`，外层再套一层 `axsync::Mutex` 提供内部可变性与线程安全 <sup>[1](#mod-net-ref-1)</sup>。`Socket` 没有继承任何 trait hierarchy，而是直接 `impl FileLike for Socket` <sup>[2](#mod-net-ref-2)</sup>，使得每个 socket 实例可以像普通文件一样被分配 fd、参与 poll/select。

`read/write` 方法分别映射到 `recv/send`，保持与 POSIX 语义的对齐。`stat()` 返回一个硬编码的 stub：`S_IFSOCK | 0o777` 权限加 `blksize: 4096` <sup>[3](#mod-net-ref-3)</sup>，注释坦承"not really implemented"。`into_any` 直接返回 `Arc<Self>`，利用 `Socket: Send + Sync` 的事实避免了额外的 type-erased wrapper 分配。

`sys_socket` 是用户态入口 <sup>[4](#mod-net-ref-4)</sup>，解析 `socktype` 中的 `O_NONBLOCK` 和 `SOCK_CLOEXEC` flag 位后，根据 `SOCK_DGRAM`/`SOCK_STREAM` 匹配底层协议。domain 检查仅允许 `AF_INET`，对 IPv6 返回 `EAFNOSUPPORT`——尽管 `socket.rs` 已完整实现 `SocketAddrV6` 的解析逻辑 <sup>[5](#mod-net-ref-5)</sup>。

### 2. 最小本地回环：为什么需要、放弃了什么

这是本模块最体现"取舍"的设计。代码注释直接说明：axnet/smoltcp 路径不负责把发往本机监听端口的数据绕回同一内核实例 <sup>[6](#mod-net-ref-6)</sup>。为了让 libctest 中同机 socket 测例跑通，模块实现了一套极简本地回环：

- 四张全局 `Mutex<Vec<...>>` 表分别追踪 UDP 绑定端口、UDP 待投递包、TCP 监听端口、TCP 待 accept 连接 <sup>[6](#mod-net-ref-6)</sup>。
- UDP 路径：`sendto` 检查目标端口是否在 `UDP_BOUND_PORTS` 中，命中则直接 `push` 到 `UDP_LOOPBACK_PACKETS` <sup>[7](#mod-net-ref-7)</sup>；`recvfrom` 优先消费 loopback 队列再 fallback 到真实 axnet 收包 <sup>[8](#mod-net-ref-8)</sup>。
- TCP 路径：`listen` 记录监听端口到 `TCP_LISTEN_PORTS` <sup>[9](#mod-net-ref-9)</sup>；`connect` 查表命中后 push 一条 `TcpLoopbackConnection` <sup>[10](#mod-net-ref-10)</sup>；`accept` 优先消费该队列，返回一个裸 `TcpSocket::new()` <sup>[11](#mod-net-ref-11)</sup>。

这套机制的取舍非常清晰：**放弃真实协议栈语义，换取测试用例可运行**。它不处理并发连接、三次握手状态迁移、窗口控制、重传或真实网卡收发。TCP loopback accept 返回的是全新未连接 socket，仅靠存在性模拟"连接已建立"——这在单进程测例中足够，但并发场景下可能出现多个 accept 消费同一连接、backlog 被忽略等问题。

### 3. 用户态地址转换：SocketAddrExt trait

`socket.rs` 定义了 `SocketAddrExt` trait <sup>[12](#mod-net-ref-12)</sup>，为 `SocketAddr`/`SocketAddrV4`/`SocketAddrV6` 提供 `read_from_user` 和 `write_to_user` 方法。`read_from_user` 的流程分三步：(1) 长度校验，`addrlen` 必须在 `sizeof(sa_family_t)` 到 `sizeof(sockaddr)` 之间 <sup>[13](#mod-net-ref-13)</sup>；(2) 通过 `copy_sockaddr_from_user` 将用户态原始字节拷贝到内核栈上的 `MaybeUninit<sockaddr>` <sup>[14](#mod-net-ref-14)</sup>；(3) 读 `ss_family` 字段分派到 v4/v6 解析器。

`copy_sockaddr_from_user` 的核心操作是一个 `unsafe { ptr::copy_nonoverlapping }` <sup>[14](#mod-net-ref-14)</sup>，但其安全前提——`addr.get_as_ref()?` 已验证指针可读、`addrlen` 由上层保证不超过 `sockaddr` 大小——没有用 SAFETY 注释显式文档化。类似地，`read_from_user` 中 `&*(storage.as_ptr() as *const sockaddr_in)` 的裸指针解引用 <sup>[19](#mod-net-ref-19)</sup> 也缺少契约声明。

写入方向 (`write_to_user`) 在空指针检查后构造对应 `sockaddr_in`/`sockaddr_in6` 结构体，调用 `copy_nonoverlapping` 写回用户空间。IPv4 的 `s_addr` 使用 `from_ne_bytes` <sup>[24](#mod-net-ref-24)</sup>，IPv6 使用 `octets()` 直接赋值——两处的字节序处理均正确，但未来若扩展到其他架构需再次验证。

### 4. 系统调用层的语义映射与"先让测例通过"策略

`sys_connect` 中有一个精心设计的错误码映射：底层返回 `EAGAIN` 时转换为 `EINPROGRESS` <sup>[16](#mod-net-ref-16)</sup>，匹配 Linux 对非阻塞 TCP connect 的语义。`sys_sendto` 对 `addr.is_null()` 的分支处理正确区分了"已连接 socket 的 send"和"指定地址的 sendto" <sup>[18](#mod-net-ref-18)</sup>。

`sys_setsockopt` 集中体现了"先让测例通过"的早期实现风格：它只校验 fd 有效性，对 `SO_REUSEADDR`/`SO_RCVTIMEO`/`SO_SNDTIMEO` 三个已知选项接受但不实际设置 <sup>[17](#mod-net-ref-17)</sup>。更重要的是，**所有未知 level/optname 组合也返回 `Ok(0)`**，注释明确写道"后续接入完整 socket 选项表时应收紧语义" <sup>[23](#mod-net-ref-23)</sup>——这意味着当前实现会静默吞掉非法的 setsockopt 调用，包括用户传入明显错误的 option value 的情况。

`sys_accept` 的返回路径中，loopback 路径返回的 `new_socket` 被包装为 `Socket::Tcp(Mutex::new(new_socket))` 加入 fd 表 <sup>[11](#mod-net-ref-11)</sup>。这个新建 socket 的 `peer_addr()` 将返回错误——只有 accept 返回值中的 `peer_addr` 携带对方地址——上层调用者若通过 `getpeername` 查询将得到不一致的结果 <sup>[20](#mod-net-ref-20)</sup>。

### 5. 边角细节与不足

**<sup>[19](#mod-net-ref-19)</sup> `unsafe` 块普遍缺少 SAFETY 注释**：`copy_sockaddr_from_user`（socket.rs:42-51）和 `read_from_user` 中 `&*(... as *const sockaddr_in)` 式解引用（socket.rs:87, socket.rs:123）均未以 `// SAFETY:` 形式文档化安全前提。这在 Rust 安全实践中属于技术债。

**<sup>[21](#mod-net-ref-21)</sup> 隐式锁顺序带来死锁风险**：`sendto` 在持有 `udpsocket.lock()` 期间调用 `has_local_udp_bound_port`（获取 `UDP_BOUND_PORTS` 锁）和 `push_udp_loopback_packet`（获取 `UDP_LOOPBACK_PACKETS` 锁）。`recvfrom` 同理在持 socket 锁期间访问全局队列。当前所有路径遵循"先 socket 锁、后全局锁"的顺序，但这是隐式约定而非编译期保证——未来任何反向获取将立即死锁。

**<sup>[22](#mod-net-ref-22)</sup> UDP connect 路径绕过 loopback**：`connect()` 对 UDP 直接透传到 `axnet::UdpSocket::connect()`（net.rs:207），不经 loopback 检查。同进程内两个 UDP socket 的 connect+send（而非 sendto 指定地址）路径不会触发 loopback，可能导致某些测例模式失败。

**<sup>[5](#mod-net-ref-5)</sup> IPv6 解析已实现但被入口阻断**：`socket.rs` 完整实现了 `SocketAddrV6::read_from_user/write_to_user`，包括 flowinfo 和 scope_id 的正确字节序转换，但 `sys_socket` 在 socket.rs:50 只接受 `AF_INET`。这造成了"代码准备好了但功能没打开"的半成品状态。

**<sup>[26](#mod-net-ref-26)</sup> 硬编码常量存在跨平台隐患**：`SOCK_CLOEXEC` 硬编码为 `0o2000000`（socket.rs:27），注释说明 linux-raw-sys 未导出该常量。但此值在非 x86 Linux 上可能不同（如 alpha 架构）；`S_IFSOCK | 0o777` 和 `blksize: 4096` 也是任意选择，对需要正确 stat 信息的场景（如 `/proc/self/fd` 遍历）会产生误导 <sup>[3](#mod-net-ref-3)</sup>。

**<sup>[25](#mod-net-ref-25)</sup> loopback 连接队列无容量限制**：`UDP_LOOPBACK_PACKETS` 和 `TCP_LOOPBACK_CONNECTIONS` 均为无界 `Vec`，攻击者或 buggy 代码可通过快速 sendto 撑爆内核内存。
