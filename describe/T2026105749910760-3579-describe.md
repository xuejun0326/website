## 📖 一、总览

该仓库是 xv6-riscv 的深度扩展版本，保留了 xv6 简洁的教学内核骨架，同时大幅扩展功能：引入多架构支持（RISC-V 和 LoongArch）、线程模型（进程/线程分离）、基于 VFS 的 ext4 文件系统、完整的信号子系统以及 futex 等 IPC 原语。它定位为“Linux 兼容型教学内核”，在保持代码量可控的前提下尽可能覆盖 POSIX 接口，但并非强调生产级别的健壮性。多架构支持通过直接映射窗口（DMW）适配 LoongArch，多线程通过 `struct tcb` 和 `struct proc` 分离实现。

该仓库最值得评审注意的 2-3 个特点：
1. **线程模型与调度器分离**：完全重新设计了调度器，引入 `struct tcb` 抽象，支持 `clone`、线程组、线程级信号投递，这在 xv6 家族中罕见。
2. **文件系统**：弃用 xv6 原生 fs 架构，引入 VFS 层和 lwext4，支持 ext4 extent 树和日志，提供更现代的文件操作（`linkat`、`renameat2` 等）。
3. **信号机制**：从零搭建完整的 POSIX 信号处理，包括实时信号、SA_SIGINFO、sigaltstack 等，实现了信号帧构造与恢复。

## 🎯 二、综合评价

### 整体定位
该内核在教学与实验性扩展之间取得平衡，以 xv6 为基础，目标并非操作系统竞赛获奖，而是探索 Linux 兼容性。相比原版，它更复杂但未达到 Linux 内核的复杂度。它引入多架构，但 LoongArch 部分明显未完成；扩展系统调用，但很多是浅层封装。整体上，它适合用于学习如何扩展 xv6，但不适合作为可靠的基础设施。

### 真正的创新点
- **线程模型与调度器分离**：task 模块将进程状态简化为 UNUSED/USED/ZOMBIE，可运行状态下放到线程（`struct tcb`），实现线程组和 `clone` 系统调用（task.ref:4）。一个真正的设计创新是 `struct mm_struct` 管理用户地址空间，支持 VMA 链表（`kernel/mm/mmap.c`），实现 `mmap/munmap/mprotect`（mm.ref:5），这在 xv6 家族中少见。
- **LoongArch 架构支持**：boot 模块采用 DMW 直接映射窗口，替代 RISC-V 的 SATP 页表切换，启动流程在入口阶段即完成地址映射（boot.ref:1）。然而对应的异常处理代码（`exception.S`）未使用（boot.ref:3），显示该支持处于早期阶段。
- **信号子系统完全重写**：从零实现 _NSIG=64、sigaction、sigpending 队列、sigtimedwait 等（signal.ref:1-5）。具有双层信号帧结构（signal.ref:4），但存在内存泄漏和 pending 队列膨胀（signal.ref:5-6）。

### 取舍判断
该仓库将精力集中在扩展功能接口而非内部优化。例如，物理分配器保持 xv6 原始的空闲链表（mm.ref:1），未引入 slab；调度器忙等待无 idle（task.ref:1）；系统调用参数提取简单但安全（syscall.ref:4）。这些取舍使得代码量控制在 37K LOC，适合教学。被刻意省略的部分包括网络模块完全缺失、多核优化不足（所有锁均全局）、以及很多系统调用以 stub 形式存在（如 CLONE_FS/CLONE_FILES 仅打印调试信息，task.ref:5）。这种选择使核心功能（fork/exec/pipe/ext4）可用，但高级功能如 robust futex、资源限制只是一层空壳。

### 完成度与不足
完成度：中等偏下。核心模块（进程、内存、文件系统、系统调用）基础路径基本实现，可以运行简单的用户程序（shell、hello）。但边缘情况处理粗糙：例如 mmap 中的 max_vma 地址空间永不回收（mm.ref:6），do_munmap 可能死锁（mm.ref:8），信号 default 处理仅 3 个信号（signal.ref:7），futex 仅 WAIT/WAKE（ipc.ref:3），驱动框架的 blockdev.c 是骨架（drivers.ref:1）。代码中存在死代码（boot.ref:3）、未解决的 TODO、以及潜在的竞态（task.ref:2）。整体完成度约 60-70%，创新性在 xv6 生态中处于中上，代码质量因仓促扩展而参差不齐。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 本模块是 xv6-c 内核的入口与初始化阶段，核心抽象包括汇编入口 `_entry` 与 C 入口 `start`、`main`。解决从硬件复位到 C 环境就绪（栈、页表、中断向量）的过渡问题。扩展原版 xv6-riscv 对 LoongArch 架构的支持，引入直接映射窗口（DMW）替代 RISC-V 的 SATP 页表切换，并内联禁用/废弃了 `timerinit`。

### 1. 核心抽象与多架构入口
启动流程的核心抽象是两级入口：汇编层 `_entry` 与 C 层 `start`。`_entry` 是链接脚本指定的第一条指令（`kernel.ld` 将其放置在 `0x9000000000000000` 或 `0x80000000`）<sup>[1](#mod-boot-ref-1)</sup><sup>[2](#mod-boot-ref-2)</sup>。其职责是极简的：为每个 CPU 设置栈指针（通过 CPUID 偏移 `stack0`），然后跳转到 `start()`。在 RISC-V 实现中，`_entry` 额外将 `a0`（hart id）保存到 `tp` 寄存器<sup>[2](#mod-boot-ref-2)</sup>；在 LoongArch 实现中，则通过 CSR 读取 CPUID 并计算偏移<sup>[1](#mod-boot-ref-1)</sup>。两种架构均使用静态分配的 `stack0`（`kernel/init/start.c` 中定义 `char stack0[4096 * NCPU]`）<sup>[4](#mod-boot-ref-4)</sup>，避免了动态分配，但限制了最大 CPU 数。

### 2. 关键设计取舍：内联 `timerinit` 与页表策略
原版 xv6-riscv 在 `start()` 后调用 `timerinit()` 来设置机器模式定时器中断。本仓库中该函数被整个注释掉<sup>[4](#mod-boot-ref-4)</sup>，且 `start()` 在 RISC-V 上仅设置 `SSTATUS_SIE` 与 `SIE` 寄存器，不再进入机器模式。这是一个有意的简化：放弃 SBI 委托之外的 M 模式中断处理，将定时器中断完全交由 S 模式处理。代价是失去了原版 `CLINT` 直接编程的能力，但换来了架构无关性——LoongArch 根本不存在 CLINT。
另一个设计决策是页表初始化策略。LoongArch 的 `_entry` 通过向 `CRMD` 写入 `0xb0`（PLV=0, IE=0, PG=1）直接启用分页，并使用直接映射窗口（DMW0/DMW1）将物理地址空间 `0x8000xxxxxxxxxxxx` 与 `0x9000xxxxxxxxxxxx` 映射到虚拟地址<sup>[1](#mod-boot-ref-1)</sup>。这避免了像 RISC-V 那样先进入 `start()` 再通过 `w_satp(0)` 临时关闭分页的步骤。两种方案各有利弊：LoongArch 的 DMW 方式更简洁（无需页表项），但灵活性低；RISC-V 的 SATP 方式需要后续在 `main()` 中建立页表，但支持更细粒度的权限控制。

### 3. 跨模块协同：从 `start` 到 `main` 的接力
`start()` 执行完毕后直接调用 `main()`<sup>[4](#mod-boot-ref-4)</sup>，后者位于 `kernel/main.c`（未在 excerpt 中提供，但可通过调用链推断）。`main()` 完成进程初始化、文件系统挂载、用户态首次切换等。启动流程与内存管理模块的协同体现在：LoongArch 的 DMW 必须在 `_entry` 中完成，否则后续 C 代码中的全局符号引用会因虚拟地址未映射而崩溃。与中断模块的协同则通过 `start()` 中设置的 `SIE` 位——它们直接决定了 `kernelvec.S` 中的异常向量能否被触发。
### 4. 边角细节与不足
- **<sup>[1](#mod-boot-ref-1)</sup> 行号风险**：LoongArch 的 `_entry` 中 `la.abs $sp, stack0+4096` 假设 `stack0` 位于可绝对寻址的虚拟地址空间，若链接脚本变更或 DMW 窗口未覆盖该地址，将导致栈不可用。没有明显的保护检查。
- **<sup>[2](#mod-boot-ref-2)</sup> 整数溢出**：RISC-V 的 `_entry` 中 `li t0, 1024*4` 然后 `mul t0, t0, a0`，若 `a0`（hart id）很大（如 512），乘积可能超过 32 位，但 `t0` 是 64 位寄存器，实际溢出风险低，但 `addi a0, a0, 1` 后若 `a0=1023` 则 `1024*4*1024=4MB`，仍安全。
- **<sup>[3](#mod-boot-ref-3)</sup> 死代码**：`kernel/arch/loongarch/exception.S` 等 5 个汇编文件在启动流程中未被引用（仅 `entry.S` 和 `trampoline.S` 被使用），这些文件可能是未完成的异常处理存根。
- **<sup>[4](#mod-boot-ref-4)</sup> 注释掉的 timerinit**：`timerinit()` 整个被注释但未删除，且 `timer_scratch` 数组和 `timervec` 声明仍存在，是工程腐烂信号。若未来需要 M 模式中断，需手动恢复。
- **硬编码常量**：`stack0` 大小 `4096 * NCPU` 是硬编码的 4KB 每栈，对于复杂内核可能不足（如文件系统操作需要深层调用栈），但缺乏编译时断言。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块由三个子模块组成：物理页分配器 (kalloc.c)、虚拟内存管理 (vm.c) 和文件映射子系统 (mmap.c)。核心抽象是 4K 页大小的空闲链表物理分配器、Sv39 三级页表操作、以及基于 VMA 链表实现的 mmap/munmap/mprotect。与 xv6-riscv 原版相比，显著扩展了 mmap 系统调用族，引入进程级 mm_struct 管理用户地址空间，并支持多线程内核栈和 LoongArch 架构。

### 1. 核心抽象：物理分配器与虚拟内存管理分层

内存管理模块由三层抽象构成。底层是物理页分配器 (`kernel/mm/kalloc.c`)，采用最简单的空闲链表结构：`struct run` 嵌入空闲页中，通过 `kmem.freelist` 头插维护，分配时摘取头结点，释放时头插回链表 <sup>[1](#mod-mm-ref-1)</sup>。该设计没有伙伴系统或 slab，意味着所有内核内存分配（页表、管道缓冲区、进程内核栈）都以 4K 粒度进行，内部碎片由调用方自行消化。`kzalloc()` 与 `kalloc()` 的区别仅在于 `memset` 清零与否 <sup>[2](#mod-mm-ref-2)</sup>，后者分配后并未清零（注释误导为 "fill with junk"，实际填 0），这是调试意图不连贯的小瑕疵。

中间层是虚拟内存管理 (`kernel/mm/vm.c`)，提供页表遍历 (`walk`)、批量建立映射 (`mappages`)、解除映射 (`uvmunmap`)、进程堆增减 (`uvmalloc`/`uvmdealloc`) 等原语 <sup>[3](#mod-mm-ref-3)</sup>。`walk` 函数按 Sv39 三级页表结构逐级下降，若中间页表不存在且 `alloc` 标志为真时分配新页表页；若 `alloc` 为假则返回 0，调用方需处理未映射情况。`walkaddr` 封装了用户页的虚拟地址到物理地址转换，并校验 `PTE_U` 标志，确保仅能查询用户页 <sup>[4](#mod-mm-ref-4)</sup>。`copyout`/`copyin`/`copyinstr` 基于 `walkaddr` 实现逐页拷贝，循环处理跨页边界，是用户态与内核态数据传输的核心路径。

### 2. 关键设计取舍：mmap 的 VMA 机制与懒惰分配

`mmap` 子系统的核心数据结构是 `vma_struct` (`kernel/include/mmap.h`)，记录映射的起始地址、长度、权限、标志（共享/私有/匿名）、关联文件描述符等 <sup>[5](#mod-mm-ref-5)</sup>。每个进程的 `mm_struct` 持有 `vma_list` 双向链表保护自旋锁 `lock`。`do_mmap()` 仅分配并初始化 VMA 结构体，插入链表，**并不分配物理页或建立页表映射**（即懒惰分配）<sup>[6](#mod-mm-ref-6)</sup>。物理页的实际分配推迟到缺页异常处理程序 (page fault handler) 中，这从 `mmap_writeback_unmapf` 的注释也能确认（"the page could not be mapped and allocated because never access by the thread, then never reach mmap pgfault handler"）<sup>[7](#mod-mm-ref-7)</sup>。这种设计节省了未访问页的内存，但使缺页处理路径必须能识别 VMA 并据此分配正确权限的页。

`do_munmap` 调用 `mmap_writeback_unmapf` 回写脏页（对于 `MAP_SHARED` 文件映射）并解除页表映射、释放物理页。回写操作通过 `filewrite` 完成，涉及文件系统锁，这引出了明显的死锁风险：注释指出持 `mm.lock` 时睡眠等待文件 inode 锁可能导致死锁，但当前调度器 `thread_sched()` 可能不允许持自旋锁睡眠 <sup>[8](#mod-mm-ref-8)</sup>。这是早期实现中未解决的并发问题。

### 3. 跨模块协同：缺页处理、进程生命周期与文件系统

内存管理模块与进程管理、文件系统紧密协作。在 fork 时，`proc_copy_vma` 遍历父进程 VMA 链表，对每个 VMA 分配新结构并插入子进程链表，同时递增文件引用计数 <sup>[9](#mod-mm-ref-9)</sup>。在进程销毁时，`freeprocvm` 遍历 VMA 链表，对每个 VMA 调用 `mmap_writeback_unmapf` 回写并解除映射，然后删除 VMA、释放内存 <sup>[10](#mod-mm-ref-10)</sup>。

`mmap_writeback_unmapf` 是 mmap 与文件系统的接口：对于 `MAP_SHARED` 且 `VMA_FILE` 类型的脏页（`PTE_D` 标志），通过 `filewrite` 将页内容写回文件，体现了写时回写 (write-back) 语义 <sup>[11](#mod-mm-ref-11)</sup>。这与 xv6 原版没有 mmap 的设计形成对比：原版文件访问仅通过 read/write 系统调用。

用户地址空间布局在 `memlayout.h` 中定义，从高地址向下布局：trampoline、SIGRETURN、TRAPFRAME、多线程独占的 trapframe 区域 (`THREAD_TRAPFRAME`)、内核栈区域 (`KSTACK`)，以及 mmap 区域（从 `MMAP_MAX_ADDR_START` 向下增长）<sup>[12](#mod-mm-ref-12)</sup>。mmap 的 VMA 通过递减 `mm.max_vma` 实现地址分配，但注释承认 "max_vma is always going down"，没有回收机制 <sup>[6](#mod-mm-ref-6)</sup>。

### 4. 边角细节与不足

**<sup>[1](#mod-mm-ref-1)</sup> 物理分配器无 slab 支持**：`kmalloc(uint size)` 和 `kcalloc` 仅支持 `size <= PGSIZE`，否则返回 0 <sup>[13](#mod-mm-ref-13)</sup>。这意味着所有小于一页的内核对象（如 VMA 结构体、inode、文件描述符等）仍占用完整 4K 物理页，内部碎片严重。替代方案可考虑引入 slab 分配器（如 Linux kmem_cache），但当前实现舍弃了这一复杂度。

**<sup>[8](#mod-mm-ref-8)</sup> do_munmap 中的潜在死锁**：代码在持有 `p->mm.lock` 时调用 `mmap_writeback_unmapf`，后者会调用 `filewrite` 可能睡眠等待文件 inode 锁。注释指出 "just never hold a spinlock when sleep right now in thread_sched()"，但未作保护，可能触发死锁。这是并发安全缺陷。

**<sup>[6](#mod-mm-ref-6)</sup> VMA 地址空间永不回收**：`mm.max_vma` 仅递减，即使中间区域的 VMA 被 munmap 释放，高地址空洞也无法被后续 mmap 重用。当前实现仅通过 `max_vma < MMAP_END_ADDRESS` 检查是否耗尽，地址空间泄漏风险存在。

**<sup>[14](#mod-mm-ref-14)</sup> uvmfirst 的循环分配**：`uvmfirst` 被注释掉的旧代码只映射一页，新代码用循环按页分配，但未检查 `sz` 是否超限；若 `sz` 很大（例如大于一个进程应有的 initcode 大小），会浪费大量内存。原版 xv6 限制 `sz < PGSIZE`，这里放松了限制却无边界校验。

**<sup>[15](#mod-mm-ref-15)</sup> kfree 的边界检查仅限页对齐**：`kfree` 检查 `pa % PGSIZE != 0` 和地址范围，但未校验 `pa` 是否已被释放（double-free），仅靠填充 "junk"（1）难以检测。并发环境下 double-free 可能破坏链表结构。

**<sup>[3](#mod-mm-ref-3)</sup> walk 中 alloc 路径的 panic 风险**：当 `va >= MAXVA` 时 `walk` 直接 `panic`，而非返回错误。这意味着内核代码若传入非法虚拟地址会导致系统崩溃，容错性弱于返回 -1 让调用方处理的方式。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 该模块以 struct proc 为进程资源容器（地址空间、文件表、信号处理等），struct tcb 为可调度线程实体。调度器在 thread_scheduler() 中循环从 TCB_RUNNABLE 队列弹出一个线程，通过 swtch 切换执行。相比原版 xv6-riscv，主要扩展是引入了线程模型：clone 系统调用支持 CLONE_THREAD/CLONE_VM 等 Linux 标志，进程状态简化为 UNUSED/USED/ZOMBIE（可运行/睡眠状态移到线程），实现了线程组、wait/channel 睡眠、超时唤醒以及基于线程的信号投递。

### 1. 核心抽象与数据结构

进程与任务调度模块的核心抽象是**进程**（`struct proc`）和**线程**（`struct tcb`）。

- **struct proc**（`kernel/include/proc.h:28-74`）持有进程的全局标识：pid、pgid、状态（UNUSED/USED/ZOMBIE）、内核栈地址 kstack、用户内存大小 sz、文件描述符表 ofile、进程树指针（parent、first_child、sibling_list）、线程组 tg、内存管理 mm_struct 以及资源限制 rlim。注意，proc 不再包含可运行/睡眠状态，这些下放到线程。
- **struct tcb**（`kernel/include/thread.h:67-110`）则代表一个可调度实体：全局 tid、组内索引 tidx、所属进程 p、状态（TCB_UNUSED/USED/RUNNABLE/RUNNING/SLEEPING）、trapframe 指针、swtch 上下文、睡眠通道 chan、超时 ticks、信号相关字段（sighand、blocked、sig_pending）。
- **struct thread_group**（`kernel/include/thread.h:52-64`）包含锁、组长指针、线程链表、线程计数，使得进程可以管理多个线程，clone 时通过 CLONE_THREAD 在同一进程内新增线程。
- 调度器使用两组全局队列：进程队列（`g_pcb_queues[]`，`kernel/sched/sched.c:10-13`）和线程队列（`g_tcb_queues[]`，`kernel/sched/thread.c:22-28`）。状态变更通过 `pcb_q_change_state`/`tcb_q_change_state` 在队列间原子移动元素。

这种分离使得进程的角色转变为资源容器，而线程承担执行流的职责，是原版 xv6 所不具备的。

### 2. 关键设计取舍

**2.1 进程状态简化为三态**
原版 xv6 的 proc 状态包括 UNUSED/USED/SLEEPING/RUNNABLE/RUNNING/ZOMBIE 六种。此实现将 SLEEPING/RUNNABLE/RUNNING 移到了线程上，进程只保留 UNUSED/USED/ZOMBIE。这简化了进程状态机，但也意味着“进程正在运行”的概念由线程体现，`myproc()` 通过 CPU 当前线程反向获取。

**2.2 线程调度与锁设计**
每个线程和进程都有自己的自旋锁。修改状态时必须持有对应锁。调度器 `thread_scheduler()`（`kernel/sched/sched.c:142-174`）先关闭中断，然后循环尝试从 TCB_RUNNABLE 队列中 pop 一个线程，若队列为空则 `continue` 忙等待。获取线程锁后将其状态设为 TCB_RUNNING，通过 `swtch` 切换到该线程。线程主动让出 CPU 时调用 `thread_yield()`（`kernel/sched/sched.c:120-125`），将自己移回 RUNNABLE 队列并调用 `thread_sched()` 进入调度器。

**2.3 线程组与 clone 语义**
`do_clone()`（`kernel/sched/proc.c:600-700`）是核心实现，支持多种 Linux 标志：
- `CLONE_THREAD`：在相同进程内创建线程，共享进程资源，返回 tid。
- `CLONE_VM`：新进程共享父进程页表。
- `CLONE_SETTLS`：设置 tp 寄存器。
- `CLONE_CHILD_SETTID`/`CLONE_CHILD_CLEARTID`：用于线程库的 tid 清置。
- `CLONE_SIGHAND`：共享 sighand 结构。
未实现 `CLONE_FS`/`CLONE_FILES`（只留下 debug 打印，`<sup>[8](#mod-task-ref-8)</sup>`）。

**2.4 睡眠与唤醒机制**
线程通过 `thread_sleep()`（`kernel/sched/thread.c:350-360`）原子地释放传入的锁、将状态置为 TCB_SLEEPING、记录 chan，然后调用 `thread_sched()`。唤醒函数 `thread_wakeup_chan()`（`kernel/sched/thread.c:270-290`）遍历整个 `tcb_pool`，匹配 chan 并置为 RUNNABLE。这种方法简单但 O(n) 扫描所有线程，且需要依次获取每个线程的锁，可能造成锁竞争。

### 3. 跨模块协同

**3.1 与内存管理**
- `fork()`/`do_clone()` 调用 `uvmcopy()` 复制用户页表（`kernel/sched/proc.c:340`）。
- 每个线程的 trapframe 通过 `uvm_thread_trapframe()` 分配并映射到进程页表的固定虚拟地址 `THREAD_TRAPFRAME(tidx)`。
- 进程退出时 `proc_freepagetable()` 解除 trampoline、sigreturn 以及各线程 trapframe 的映射。
- `growproc()` 通过 `uvmalloc`/`uvmdealloc` 调整堆大小。

**3.2 与文件系统**
- `fork()`/`do_clone()` 中遍历 `p->ofile[]` 调用 `filedup()` 增加引用计数。
- `proc_exit()` 中关闭所有打开文件。
- 进程的 `cwd` 和 `cinfo.path` 在 exec 时更新，clone 时拷贝。

**3.3 与信号处理**
- 每个线程的 `struct sighand` 通过 `sighandinit()` 分配，`CLONE_SIGHAND` 时共享父线程的 sighand（增加引用计数）。
- `thread_send_signal()`（`kernel/sched/thread.c:605-615`）将信号加入目标线程的 pending 队列，若线程处于 SLEEPING 状态则唤醒，对于 SIGKILL 设置 `killed` 标志。
- 信号处理与睡眠超时联动：`thread_wakeup_chan_timeout()` 用于唤醒等待特定信号的线程。

**3.4 与陷阱处理**
- 线程第一次被调度时执行 `thread_forkret()`（`kernel/sched/thread.c:90-105`），释放线程锁、初始化文件系统（仅一次），然后调用 `usertrapret()` 返回用户空间。
- 时钟中断处理会调用 `thread_wakeup_timeout()` 唤醒超时睡眠的线程。

### 4. 边角细节与不足

**4.1 调度器忙等待** <sup>[1](#mod-task-ref-1)</sup> `thread_scheduler()` 在无就绪线程时 `continue` 无限循环，未使 CPU 进入低功耗状态，在 idle 时浪费电力。

**4.2 状态切换中 RUNNING 的特殊处理** <sup>[2](#mod-task-ref-2)</sup> `tcb_q_change_state()` 在旧状态为 TCB_RUNNING 时使用 `queue_remove`（非原子）而不是 `queue_remove_atomic`，因为 RUNNING 线程不在队列中。但该函数也在 `thread_yield()` 中用于从 RUNNING 转到 RUNNABLE，这可能导致竞态（中断可能在此期间发生）。

**4.3 `freeproc` 与 `thread_exit` 的竞争** <sup>[3](#mod-task-ref-3)</sup> `freeproc()` 在释放进程前获取 `lth_exitlock`，然后遍历 `cur_proc->tg.threads` 释放所有线程。但若某个线程正在 `thread_exit` 中且已减少计数但尚未从链表删除，`freeproc` 可能重复释放或漏释放。实际路径要求最后一个线程退出时才调用 `proc_exit`，但代码中有 `thread_cnt==1` 的判断，可能导致时序问题。

**4.4 `allocproc`/`create_proc` 返回时隐式持有锁** <sup>[4](#mod-task-ref-4)</sup> `allocproc()` 返回时持有 `p->lock`，`create_proc()` 调用后只释放了线程锁，进程锁仍保持，调用者必须手动释放。这种隐式约定容易遗漏，增加死锁风险。

**4.5 `clone` 中 `CLONE_FILES`/`CLONE_FS` 未实现** <sup>[5](#mod-task-ref-5)</sup> 代码只打了 debug 日志而无实际功能，这意味着使用这些标志时行为不符合 Linux 预期。

**4.6 `thread_wakeup_chan` 遍历全部 TCB 池** <sup>[6](#mod-task-ref-6)</sup> 每次唤醒都要扫描 `NTHREADS` 个线程并获取锁，若线程数较大（默认 64?）效率低，且锁顺序可能死锁（但代码避免了自己唤醒自己）。

**4.7 `mappages` 中物理地址类型转换** <sup>[7](#mod-task-ref-7)</sup> `mappages(kpgtbl, va + i * PGSIZE, PGSIZE, (uint64)pa, PTE_R | PTE_W)` 将 `char*` 直接转为 `uint64`，未校验物理地址是否在合法范围内，在嵌套页表或 IOMMU 场景可能有隐患。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 该模块实现了 POSIX 信号的核心基础设施，包括信号发送 (signal_send)、挂起队列管理 (sigpending)、动作配置 (do_sigaction/do_sigprocmask)、默认处理 (signal_default) 以及实时信号帧的构造与恢复 (setup_rt_frame/signal_frame_restore)。核心抽象是 per-thread 的 struct sighand（动作表）和 struct sigpending（挂起队列），配合 per-signal 的 sigaction 描述。相比 xv6-riscv 原版（无信号机制），这是一个从零搭建的完整子系统，支持 _NSIG=64 个信号、SA_SIGINFO、sigprocmask 掩码操作、以及 sigtimedwait 同步等待。架构上通过 sigret.S 提供的跳板与 trap 处理协同，利用 copyin/copyout 在用户栈构造 rt_sigframe。

### 1. 核心抽象与数据结构

信号模块围绕两条主线组织：**信号动作配置**与**挂起信号管理**。前者通过 per-thread 的 `struct sighand`（定义于 `signal.h:119-123`）持有 64 个 `struct sigaction` 条目和一个自旋锁 `siglock`；后者通过 per-thread 的 `struct sigpending`（`signal.h:125-129`）维护一个挂起信号的双向链表（`struct sigqueue` 节点）和一个原子信号集 `signal`（位图）。这种分离设计允许 `do_sigaction`/`do_sigprocmask` 操作动作表时只锁 `sighand.siglock`，而 `signal_send`/`signal_handle` 操作挂起队列时锁 `sigpending.siglock`，减少锁竞争。

信号编号定义遵循 Linux 惯例，范围 1-31（标准信号），`_NSIG=64` 预留了实时信号空间 <sup>[1](#mod-signal-ref-1)</sup>。`sigset_t` 仅用一个 `uint64` 位图表示，配套的宏 `sig_add_set`/`sig_del_set`/`sig_is_member` 等可直接位操作 <sup>[2](#mod-signal-ref-2)</sup>。`struct sigaction` 采用匿名 union 区分 `sa_handler` 和 `sa_sigaction`，由 `sa_flags & SA_SIGINFO` 决定激活哪个 <sup>[3](#mod-signal-ref-3)</sup>。

`struct rt_sigframe`（`signal.h:194-198`）是信号交付的核心载体，包含三个成员：`siginfo_t info`、`ucontext_t uc_riscv`（glibc 兼容的完整浮点/向量上下文）、以及内核自定义的 `struct ucontext`（保存 trapframe、sigmask 和当前处理信号编号）。这个双层设计兼容了 libc 的 `ucontext_t` 标准布局，同时内核只使用精简的 `struct ucontext` 恢复上下文 <sup>[4](#mod-signal-ref-4)</sup>。

### 2. 关键设计取舍

**2.1 信号去重策略**
`signal_send` 在入队前检查 `sig_existed(t, sig)`，如果信号已挂起则直接返回 -1，不再次入队 <sup>[5](#mod-signal-ref-5)</sup>。这是与 Linux 的重要差异——Linux 标准信号不排队（多次投递只记录一次），但实时信号（SIGRTMIN+）需排队。当前实现对所有信号都去重，简化了队列管理但丢失了实时信号的语义。此外，去重检查在未持 `sig_pending.siglock` 时读取 `t->sig_pending.signal`，存在与 `signal_handle` 删除并发的读写竞态。

**2.2 忽略信号不删除**
`signal_handle` 中当 `sig_ignored(t, sig_no) || sig_act.sa_handler == SIG_IGN` 时，代码仅 `continue`，注释掉了删除和释放逻辑 <sup>[6](#mod-signal-ref-6)</sup>。这意味着被忽略的信号永远不会从挂起队列中移除，`pending_cnt` 也不同步减少，造成挂起队列无限膨胀。这显然是一个**未完成的实现**——开发者可能打算在信号被取消忽略时递送，但当前状态会泄漏内存且无法处理新信号。

**2.3 默认动作部分实现**
`signal_default` 只实现了 `SIGKILL`、`SIGCHLD`、`SIGIOT` 的规范行为，其余信号（包括 `SIGTERM`、`SIGQUIT`、`SIGSEGV` 等）均落入 `default` 分支打印 Warn 并忽略 <sup>[7](#mod-signal-ref-7)</sup>。特别地，`signal_send` 却对 `SIGTERM` 设置了 `t->killed = 1`（`signal.c:377`），与默认处理器的忽略行为矛盾。这种不一致会导致用户注册 `SIG_DFL` 后信号被静默吞掉，而 `SIGKILL` 虽正确终止但 `SIGTERM` 则行为异常。

**2.4 信号帧栈布局**
`get_sigframe` 直接在当前栈指针 `tf->sp` 上减去 `framesize` 并 16 字节对齐 <sup>[8](#mod-signal-ref-8)</sup>。代码中注释掉了备选信号栈（`on_sig_stack`）的溢出检查和 `sigsp` 切换，意味着**不具备 `sigaltstack` 支持**。对于需要可靠信号栈的应用（如协程库、Go runtime），在现有栈上覆写 frame 可能因栈溢出触发 SIGSEGV。

### 3. 跨模块协同与完整流程

信号交付的完整路径跨越多个子系统：

**发送端**：系统调用（如 `kill`）到达后构造 `siginfo_t`，调用 `signal_send` → 分配 `struct sigqueue` → 加 `sig_pending.siglock` → 入队并设位图。`SIGKILL`/`SIGSTOP`/`SIGTERM` 还会直接设置 `t->killed` 标记，这是一个**紧急出口**，即使目标线程不在内核中也能被调度循环检查。

**检测与递送**：在 `trap.c` 的异常/中断返回路径中（推测），调用 `signal_handle`。该函数遍历 `sig_pending.list`，对每个信号检查 `sig_ignored`（即 `t->blocked` 位图），未被屏蔽的信号按 `sigaction` 动作分类：SIG_IGN 跳过（但未删除）、SIG_DFL 执行默认动作、自定义 handler 调用 `do_handle_signal` → `setup_rt_frame`。

**帧构造**：`setup_rt_frame` 在用户栈上通过 `get_sigframe` 计算位置，然后调用 `signal_frame_setup` 将当前 `trapframe`、`blocked` 掩码复制到 `struct rt_sigframe` 中 <sup>[9](#mod-signal-ref-9)</sup>。关键技巧是：修改 `t->trapframe` 的 `epc`（或 `era`）为 handler 地址，`sp` 为 frame 基址，`ra` 固定为 `SIGRETURN` 跳板地址。用户态信号 handler 执行完毕后，通过 `__user_rt_sigreturn`（`riscv/sigret.S` 或 `loongarch/sigret.S`）触发 `SYS_rt_sigreturn` 系统调用。

**恢复**：`SYS_rt_sigreturn` 在内核中调用 `signal_frame_restore`，从帧中 `copyin` 出 `struct ucontext`，恢复 `t->blocked` 和 `t->trapframe`，然后检查 `uc_riscv` 的 `__gregs[0]` 是否为非零（libc 兼容的 MC_PC 调整），最终通过 `trapframe` 恢复用户态寄存器 <sup>[10](#mod-signal-ref-10)</sup>。

**同步等待**：`do_sigtimedwait` 展示了一种特殊的等待路径——它先调用 `sig_first_member` 从用户传入的 `set` 中取出第一个信号编号，然后检查是否已挂起；若否，通过 `thread_sleep` 阻塞在 `sig_pending.siglock` 上等待信号到来 <sup>[11](#mod-signal-ref-11)</sup>。但此处的 `thread_sleep` 第三个参数 `timeout` 仍是用户空间指针（`__kernel_space`），没有做 `copyin` 转换，直接传入内核睡眠函数会访问错误地址。

### 4. 边角细节与不足

- **<sup>[5](#mod-signal-ref-5)</sup> 内存泄漏风险**：`signal_send` 在 `sig_existed` 返回真时直接 return -1，但此前已通过 `kalloc` 分配了 `struct sigqueue`（`signal.c:369`），未释放即返回，造成每次重复投递泄漏 144 字节（sizeof sigqueue）。
- **<sup>[6](#mod-signal-ref-6)</sup> 忽略信号队列堆积**：`signal_handle` 对 SIG_IGN 的跳过操作不删除节点（`signal.c:154-157` 注释掉的删除代码），导致 `pending_cnt` 只增不减，且无效条目阻塞后续处理。
- **<sup>[7](#mod-signal-ref-7)</sup> 默认动作残缺**：`signal_default` 只实现了 3/64 个信号的标准行为，`SIGTERM`、`SIGSEGV`、`SIGFPE` 等关键信号的默认动作被错误地当作忽略处理，不符合 POSIX。
- **<sup>[12](#mod-signal-ref-12)</sup> 整数溢出隐患**：`sig_first_member` 宏（`signal.h:215-222`）在信号集为空时循环结束后 i=64，返回 i+1=65，而 `valid_signal(65)` 为假，调用者 `do_sigtimedwait` 未预先检查空集（虽开头检查了 `sig_test_empty`，但后续直接调用 `sig_first_member`，若 set 传空指针则未定义）。
- **<sup>[8](#mod-signal-ref-8)</sup> 备选信号栈缺失**：`get_sigframe` 注释掉了备选栈逻辑，且 `struct ucontext` 中 `uc_stack` 成员被注释掉（`signal.h:183-189`），信号处理时无法切换栈，递归信号或因栈不足导致崩溃。
- **<sup>[11](#mod-signal-ref-11)</sup> 用户态指针直接传入**：`do_sigtimedwait` 中 `thread_sleep((void *) sig_no, &t->sig_pending.siglock, timeout)` 的 `timeout` 声明为 `__kernel_space`，但未做 `copyin` 拷贝到内核空间，直接传入内核可能导致访问错误或睡眠行为异常。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 该模块提供四种进程间通信机制：管道（pipe）、futex、信号（signal）和信号量（semaphore）。管道基于 xv6 经典的环形缓冲区模型，通过 thread_sleep/wakeup 与调度器协同。Futex（快速用户空间互斥锁）通过哈希表将用户地址映射到内核等待队列节点，实现了 FUTEX_WAIT/FUTEX_WAKE 基本操作。信号处理系统调用（sigaction/rt_sigprocmask/sigtimedwait/sigreturn）桥接用户空间与内核信号核心逻辑。信号量基于条件变量实现 P/V 操作。与 xv6-riscv 原版相比，扩展了 futex 和信号量机制，但 futex 仅实现了最基本的操作子集。

### 1. 核心抽象与外部依赖

该 IPC 模块包含四个子机制，每个机制面向不同层次的同步与通信需求。

**管道 (pipe)** 是传统的字节流通信原语，基于 `struct pipe` 环形缓冲区（512 字节 `<sup>[4](#mod-ipc-ref-4)</sup>`），通过自旋锁保护并发访问。写入端满时调用 `thread_sleep` 阻塞，读取端唤醒写入端；读空时类似地阻塞写入端。这种设计与 xv6 原版一致，依赖调度器的 `thread_sleep`/`thread_wakeup_chan` 接口完成线程状态切换。

**Futex (快速用户空间互斥锁)** 是 Linux 兼容的同步原语，核心结构体 `struct futex` 包含一个等待队列 `<sup>[7](#mod-ipc-ref-7)</sup>`。内核维护一个 32 桶哈希表（`FUTEX_NUM=32`），以用户空间地址为键进行映射。`do_futex` 是系统调用入口，当前仅实现 `FUTEX_WAIT` 和 `FUTEX_WAKE` 两个命令 `<sup>[3](#mod-ipc-ref-3)</sup>`。Futex 依赖 `copyin` 读取用户空间的值 `<sup>[2](#mod-ipc-ref-2)</sup>`，依赖 `get_futex` 在哈希表中查找或创建内核 futex 对象。

**信号 (signal)** 通过 `sys_sigaction`、`sys_rt_sigprocmask`、`sys_sigreturn` 和 `sys_sigtimedwait` 四个系统调用暴露给用户空间。它们调用内核核心信号处理函数 (`do_sigaction`、`do_sigprocmask`、`do_sigtimedwait`) `<sup>[8](#mod-ipc-ref-8)</sup>`，这些函数位于 `kernel/signal.c`（未在此处提供）。信号处理涉及用户栈上的信号帧（`rt_sigframe`）恢复，通过 `t->trapframe->sp` 获取 `<sup>[8](#mod-ipc-ref-8)</sup>`。

**信号量 (semaphore)** 是一个内核级计数信号量，使用自旋锁保护内部状态，并通过条件变量 (`struct cond`) 实现阻塞和唤醒 `<sup>[5](#mod-ipc-ref-5)</sup>`。`sema_wait` 递减 value，若为负则阻塞；`sema_signal` 递增 value，若 ≤0 则唤醒一个等待线程。

### 2. 关键设计取舍

**Futex 哈希表 vs 红黑树**：实现采用固定大小（32 桶）的哈希表，将用户地址直接作为哈希键（`__futex_hash_key` 返回 `(uint64)key`）`<sup>[1](#mod-ipc-ref-1)</sup>`。这比 Linux 内核使用的红黑树（优化大量 futex）实现简单，但哈希冲突严重时退化为链表遍历，且不支持范围查询。对于教学内核或嵌入式场景，32 桶是合理的空间-时间权衡。

**Futex 锁复用策略**：`struct futex` 中定义了一个自旋锁 `lock`，但注释明确指出 `// uncessary, using p->lock is ok` `<sup>[7](#mod-ipc-ref-7)</sup>`。实际代码中该锁被初始化但从未获取。设计者认为可以复用 `p->lock`（进程锁）来保护 futex 操作，避免了额外的锁层级。这减少了锁冲突的可能性，但意味着 futex 的并发保护完全依赖于调度器的进程锁，可能限制细粒度并发。

**管道固定缓冲区 vs 动态扩展**：`PIPESIZE=512` 是硬编码常量 `<sup>[4](#mod-ipc-ref-4)</sup>`，继承自 xv6 原版。如果写入速度远快于读取，小缓冲区会频繁触发调度器切换，降低吞吐。动态扩展缓冲区（如 Linux 的 pipe 使用 page 级缓冲）可以缓解，但增加了复杂度和内存管理开销。

**信号量基于条件变量 vs 原子操作 + 忙等待**：`sema_wait` 在 value < 0 时调用 `cond_wait` 让出 CPU `<sup>[5](#mod-ipc-ref-5)</sup>`，而不是自旋等待。这在内核上下文中是高效选择，因为自旋会浪费 CPU 时间且容易导致死锁（特别是在单核或关中断环境下）。代价是每次 P/V 操作都需要两次上下文切换。

**信号处理系统调用的参数传递**：`sys_sigaction` 使用 `argint`/`argaddr` 获取参数，然后手动 `copyin`/`copyout` 从用户空间拷贝 `struct sigaction` 结构体 `<sup>[8](#mod-ipc-ref-8)</sup>`。这绕过了更高级的抽象（如 `copyin_struct`），但给予开发者对拷贝粒度的完全控制。

### 3. 跨模块协同

**Futex + 页表 + 调度器**：`futex_wait` 首先调用 `copyin` 通过进程页表读取用户地址处的值 `<sup>[2](#mod-ipc-ref-2)</sup>`，若值与期望不符则立即返回 `-EAGAIN`。值匹配后，调用 `get_futex` 获取内核节点，通过 `tcb_q_change_state` 将线程置为 `TCB_SLEEPING` 并加入等待队列，最后调用 `thread_sched` 触发调度。当超时发生时，通过 `queue_remove_atomic` 从等待队列移除 `<sup>[6](#mod-ipc-ref-6)</sup>`。整个流程涉及页表、进程管理、定时器和调度器四个子系统。

**管道 + 文件系统 + 调度器**：`pipealloc` 分配两个 `struct file` 并通过 `filealloc()` 注册到文件系统 `<sup>[9](#mod-ipc-ref-9)</sup>`。读写操作通过 `either_copyin`/`either_copyout` 自动处理用户/内核空间拷贝，不必关心当前上下文。关闭管道时，`pipeclose` 设置 `readopen`/`writeopen` 标志并唤醒等待线程，当双方都关闭时释放内存。

**信号 + 进程异常处理**：`sys_sigreturn` 在信号处理完成后恢复用户上下文，它从 `t->trapframe->sp` 获取信号帧指针，调用 `signal_frame_restore` 恢复寄存器 `<sup>[8](#mod-ipc-ref-8)</sup>`。`sig_del_set_mask` 清除正在处理的信号掩码。这需要与陷阱处理（trap handler）紧密配合，确保用户栈上信号帧布局一致。

**信号量 + 条件变量**：`sema_wait` 和 `sema_signal` 通过 `cond_wait`/`cond_signal` 与条件变量模块交互 `<sup>[5](#mod-ipc-ref-5)</sup>`。条件变量内部使用自旋锁和等待队列，`cond_wait` 会原子地释放 `sem_lock` 并阻塞线程，被唤醒后重新获取 `sem_lock`。这种分层设计使得信号量可以复用条件变量的阻塞/唤醒机制。

### 4. 边角细节与不足

**`<sup>[1](#mod-ipc-ref-1)</sup>` 内存释放逻辑不确定性**：`__futex_hash_delete` 中注释 `// Should I free futex here?` 表明开发者对何时释放 `futex` 对象存疑。代码中先通过 `futex_hash_lookup_node_locked` 查找节点，然后依次 `kfree(fp)` 和 `kfree(fhn)`。如果 `futex` 正在被其他线程使用（例如在等待队列中），此释放可能导致释放后使用。当前仅在 `futex_wake` 检查队列为空后调用 `futex_free`，但并行路径下仍可能存在竞态。

**`<sup>[6](#mod-ipc-ref-6)</sup>` 锁获取顺序不一致**：在 `futex_wait` 中，先 `acquire(&timeout_lock)`，然后 `acquire(&t->lock)`，接着立即 `release(&timeout_lock)` `<sup>[6](#mod-ipc-ref-6)</sup>`。锁获取顺序不是对称的（先 A 后 B，然后释放 A），如果其他代码路径以相反顺序获取这两个锁，则会造成死锁。此外，`timeout_lock` 在此处的实际作用不明确 —— 它似乎只是作为一个全局屏障，可能意图防止超时相关竞态，但注释缺失。

**`<sup>[3](#mod-ipc-ref-3)</sup>` Futex 功能严重不完整**：`do_futex` 的 switch 语句中，只有 `FUTEX_WAIT` 和 `FUTEX_WAKE` 两个 case 是激活的，其余（`FUTEX_WAIT_BITSET`、`FUTEX_REQUEUE`、`FUTEX_CMP_REQUEUE`、`FUTEX_WAKE_OP`、`FUTEX_LOCK_PI` 等）全部被 `//` 注释掉 `<sup>[3](#mod-ipc-ref-3)</sup>`。这意味着依赖 `FUTEX_REQUEUE` 的 POSIX 条件变量实现（如 glibc 的 `pthread_cond_wait`）无法正常工作。

**`<sup>[7](#mod-ipc-ref-7)</sup>` 冗余的自旋锁开销**：`struct futex` 包含一个从未使用的 `struct spinlock lock` `<sup>[7](#mod-ipc-ref-7)</sup>`。`futex_init` 调用 `initlock(&fp->lock, name)` 初始化它，但 `lock` 从未在 `acquire`/`release` 中出现。这增加了 `struct futex` 的大小（在 64 位平台上 spinlock_t 通常为 4 字节，加上 padding 可能更多），并且初始化操作浪费了指令周期。

**`<sup>[8](#mod-ipc-ref-8)</sup>` sys_sigaction 拷贝 bug**：在 `copyout` 调用中，第三个参数是 `(char*) &oldact_addr`，但 `oldact_addr` 是 `uint64` 类型的栈变量，而不是 `struct sigaction` 类型的 `oldact`。正确的写法应该是 `(char*) &oldact`。这一错误导致无论 `do_sigaction` 返回什么内容，用户空间都只会收到 `oldact_addr` 本身的 8 字节值，而不是完整的信号动作结构体。此问题会导致用户态无法获取旧的信号处理函数地址。

**超时处理可能遗漏移除**：`futex_wait` 中超时唤醒后，检查 `t->wait_chan_entry != NULL` 决定是否从队列移除 `<sup>[6](#mod-ipc-ref-6)</sup>`。但如果 `futex_wake` 在超时之前刚好唤醒了该线程（`wait_chan_entry` 被设为 NULL），超时处理会跳过移除。此时 `release(&t->lock)` 后函数返回 0，进程继续执行，逻辑正确。但若 `futex_wake` 和超时同时发生，可能存在线程同时被两个路径处理的风险，取决于 `timeout_lock` 和 `t->lock` 的精确顺序。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架模块围绕块设备开关表（bdev_table）和具体设备驱动（virtio磁盘、UART）构建，提供设备注册、打开及中断处理的标准接口。与原始xv6相比，扩展了struct bdev_ops抽象，允许通过register_bdev动态注册设备，并引入了ext4块设备适配层（blockdev.c）。核心数据结构包括bdev_table全局数组和disk全局结构体。该模块解决了内核统一管理多种块设备并支持多请求并发的问题。

### 1. 核心抽象：bdev表与驱动接口

bdev表是一个全局数组`bdev_table.bdeventry[MAXBDEV]`，每个条目是指向`struct bdev_ops`的指针，包含open和close函数指针<sup>[1](#mod-drivers-ref-1)</sup><sup>[2](#mod-drivers-ref-2)</sup>。驱动通过`register_bdev`注册，主设备号作为索引。`bdev_open`根据inode中的主设备号查找ops并调用open。这种设计借鉴了Unix传统的主/次设备号分离，但将静态的devsw数组改为动态注册，便于扩展。

virtio磁盘驱动通过`virtio_disk_init`初始化，分配两个页作为描述符表、可用环和已用环，并写入MMIO寄存器<sup>[3](#mod-drivers-ref-3)</sup>。UART驱动则直接操作内存映射寄存器，使用循环缓冲区并配合中断<sup>[4](#mod-drivers-ref-4)</sup>。

### 2. 关键设计取舍：自旋锁与中断协同

所有设备共享一个全局自旋锁`disk.vdisk_lock`保护virtio磁盘的并发访问。`virtio_disk_rw`在持锁期间可能睡眠等待描述符或完成，而`virtio_disk_intr`中断处理也获取同一锁。这种设计导致中断处理可能被阻塞，但xv6采用轮询+等待模式，实际中中断频率低，不会造成性能瓶颈<sup>[5](#mod-drivers-ref-5)</sup>。UART驱动则使用单独的`uart_tx_lock`，并在中断处理中调用`uartstart`发送缓冲字符<sup>[6](#mod-drivers-ref-6)</sup>。

关键点：`thread_sleep`在持锁时释放锁并在返回前重新获取，避免死锁<sup>[7](#mod-drivers-ref-7)</sup>。中断处理中获取锁后立刻ACK中断，避免丢失后续中断<sup>[8](#mod-drivers-ref-8)</sup>。

### 3. 跨模块协同：VFS与ext4适配

块设备驱动通过`blockdev.c`适配ext4文件系统。`blockdev_bread`和`blockdev_bwrite`封装了xv6的`bread`/`bwrite`接口，将ext4的块读写转换为缓冲区操作<sup>[9](#mod-drivers-ref-9)</sup>。这种适配层使得内核可以同时支持xv6原始文件系统和外部ext4库。

中断控制器PLIC初始化设置UART和VIRTIO的中断优先级与使能<sup>[10](#mod-drivers-ref-10)</sup>。`plic_claim`/`plic_complete`在`devintr`中被调用，连接硬件中断与驱动处理函数。

### 4. 边角细节与不足

- <sup>[11](#mod-drivers-ref-11)</sup> `blockdev_open`等函数为骨架，直接返回EOK，未实现实际锁与状态管理，可能导致并发访问文件系统时出现竞态。
- <sup>[12](#mod-drivers-ref-12)</sup> `bdev_open`读取`bdev_table.bdeventry`时未加锁，与`register_bdev`存在竞态。
- <sup>[13](#mod-drivers-ref-13)</sup> `virtio_disk_intr`中未验证`used->ring[id].id`是否小于NUM，设备可能返回越界id导致数组越界。
- <sup>[14](#mod-drivers-ref-14)</sup> `virtio_disk_init`使用裸指针`(uint64)disk.pages`直接写入MMIO，依赖内核线性映射，若未来引入非直接映射需修改。
- <sup>[15](#mod-drivers-ref-15)</sup> `blockdev_bwrite`调用`bget`后未检查返回NULL（内存不足时），且`bwrite`与`brelse`之间存在锁重复获取风险（`bget`已获得锁）。

这些问题表明驱动框架仍处于早期实现，健壮性有待加强。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层以 `syscall.c` 为分发核心，通过静态函数指针数组 `syscalls[]` 将调用号映射到处理函数。`sysproc.c` 实现进程、线程、内存、时间等核心系统调用处理函数。相比原版 xv6-riscv，本内核扩展了多线程支持（clone/exit_group/tkill/tgkill/set_tid_address）、高精度睡眠（nanosleep/clock_nanosleep）、资源限制（prlimit64）以及 brk 系统调用。核心抽象包括：通过 `thread_sleep` 实现的等待机制、基于 `tickslock` 的时间管理、通过 `copyin`/`copyout` 完成的用户态数据安全访问。

### 1. 系统调用分发与参数提取

系统调用的统一入口在 `syscall()` 函数 (`syscall.c:32`)。它从当前线程的 trapframe 的 `a7` 寄存器读取调用号，通过 `syscalls[]` 数组（行27）查找处理函数并调用，返回值写回 `a0`。数组通过 `#include "sysfunc.h"` 在编译时展开，由多个系统调用函数指针组成。参数提取函数系列 `argint`、`argaddr`、`argstr` 等最终调用 `argraw`（行42），从 trapframe 的 `a0-a5` 寄存器读取原始值。与 xv6 原版不同的是，`argraw` 使用 `mythread()` 而非 `myproc()`，因为参数存储在**线程**的 trapframe 中，这直接体现了对多线程的支持。

`fetchaddr`（行9）在读取用户空间指针前检查 `addr >= p->sz` 和加法溢出条件，双重验证防止用户传入越界地址。但 `argaddr` 不执行此类检查，完全依赖后续的 `copyin`/`copyout` 进行页表权限验证 <sup>[1](#mod-syscall-ref-1)</sup>。这种分层设计在性能与安全性之间做了权衡：快速路径（参数提取）不校验，慢速路径（内存拷贝）才校验。

### 2. 进程与线程管理的关键实现

`sys_clone`（`sysproc.c:196`）是线程创建的核心，它调用 `do_clone` 传递 `flags`、`stack`、`ptid`、`tls`、`ctid` 等参数。`do_clone` 的实现位于 `proc.c`，负责分配线程控制块、设置 trapframe、复制页表等。`sys_fork` 退化地以全零参数调用 `do_clone`，行为与原版 fork 兼容。`sys_exit_group`（行174）处理线程组退出：若当前线程是组 leader，则先清理组内其他线程再退出；否则只退出当前线程，符合 Linux 语义 <sup>[5](#mod-syscall-ref-5)</sup>。

信号发送系列 `sys_kill`、`sys_tkill`、`sys_tgkill` 分别通过 `proc_kill`、`thread_kill`、`thread_group_kill` 传递信号。值得注意的是，`sys_tkill` 和 `sys_tgkill` 使用 `arguint64` 读取信号值，这是因为 `sig_t` 类型可能为 64 位。`sys_set_tid_address`（行367）将用户空间地址存入线程的 `clear_child_tid`，供线程退出时通知父进程。

### 3. 时间管理与睡眠实现中的精巧设计

`sys_nanosleep`（行128）和 `sys_clock_nanosleep`（行72）实现了 POSIX 高精度睡眠。它们将 `timespec` 转换为 ticks 粒度，在 `tickslock` 保护下调用 `thread_sleep(&ticks, &tickslock, NULL)` 阻塞等待。`thread_sleep` 在等待时会释放锁、被唤醒后重新获取，从而避免忙等。中断处理：若进程被杀死 (`proc_killed`)，则将剩余时间通过 `copyout` 写回 `rem` 并返回 `-EINTR`；正常完成时写零。`sys_clock_nanosleep` 额外支持 `TIMER_ABSTIME` 标志和时钟选择 <sup>[6](#mod-syscall-ref-6)</sup>。

`sys_sleep`（行243）是简化版，使用 `10 * n` 作为等待 ticks 数，隐含假设每秒 10 个 tick。若 `TICKS_PER_SECOND` 不是 10，将导致睡眠时间错误，这是硬编码的隐患。

### 4. 内存管理：brk 与资源限制

`sys_brk`（行221）支持绝对地址设置堆断点：若 `addr == 0` 返回当前断点；若 `addr > oldsz` 则扩容；否则不操作（返回 `oldsz`）。边界检查 `addr >= MAXVA || addr >= BRKTOP` 防止设置过高的断点。值得注意的是，`addr <= oldsz` 时既不回收页面也不返回错误，这符合 Linux 行为（允许降低堆顶但内核不立即回收）。`sys_prlimit64`（行419）实现资源限制查询与设置：从用户空间读取 `rlimit64` 后转换为内核的 `rlimit`，通过遍历 `proc[]` 数组找到目标进程（或当前进程），持锁后调用 `do_prlimit` 修改限制 <sup>[7](#mod-syscall-ref-7)</sup>。当前仅支持 `RLIMIT_NOFILE` 和 `RLIMIT_STACK`。

### 5. 边角细节与不足

- **竞态条件**：`sys_prlimit64` 和 `sys_getpgid` 在 `proc[]` 数组中查找进程时未持锁，虽然后续操作会加锁，但查找过程和加锁之间存在时间窗，进程状态可能变化（如变为 UNUSED），导致后续操作基于过时状态。`sys_getpgid` 在加锁前检查了 `p->state == UNUSED`，但检查与加锁非原子，仍有竞态 <sup>[2](#mod-syscall-ref-2)</sup>。

- **未实现系统调用**：`sys_set_robust_list` 直接返回 0，未做任何操作。这意味着用户态 robust futex 可能无法正确清理，影响死锁检测 <sup>[9](#mod-syscall-ref-9)</sup>。

- **时间单位硬编码**：`sys_sleep` 使用 `10 * n` 隐含每秒 10 ticks 的假设，与 `TICKS_PER_SECOND` 可能不一致，导致实际睡眠时间偏差。

- **资源限制静默忽略**：`do_prlimit` 对未支持的资源返回 `-EINVAL`，但调用方 `sys_prlimit64` 在加锁后调用，若资源非法则返回错误后释放锁，处理正确。然而对于合法但未实际实现的资源（如 `RLIMIT_AS`），`do_prlimit` 会返回 0 但不做任何操作，导致用户设置无效果也无错误提示 <sup>[10](#mod-syscall-ref-10)</sup>。

- **`sys_brk` 不释放内存**：当用户降低断点时，内核仅调整 `sz` 而不调用 `uvmdealloc` 回收页面。这虽然符合 POSIX（允许但不要求回收），但可能造成内存浪费。
