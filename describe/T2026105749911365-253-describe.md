## 📖 一、总览

本仓库基于xv6-riscv教学内核，面向Rv64平台，在保留原始内核核心机制（SV39页表、轮转调度、日志文件系统）的基础上，进行了多项功能扩展。与原始版本相比，本实现显著增强了存储栈（引入ext4只读+部分写入支持）、内存管理（懒分配与mmap/mprotect/brk）以及系统调用兼容性（采用Linux RISC-V调用号编排）。但多核初始化不完整（BSS未清零、AP缺少timerinit），信号与网络子系统完全缺失，整体定位为面向教学验证与特定测试场景的实验扩展型内核。

本仓库最值得关注的两个特点为：其一，双栈文件系统设计，通过全局标志动态切换xv6原生与ext4格式，在未引入VFS层的前提下实现了多磁盘格式共存，虽写入路径缺乏日志保护，但拓展了xv6的应用场景；其二，上下文感知的I/O完成等待策略，在驱动层根据当前上下文（进程或init）采用sleep/wakeup或轮询，体现了对xv6原始设计的实用主义改造。此外，内核级shebang解析和MEMFS编译模式也展示了开发者在兼容性与部署灵活性上的思考。

## 🎯 二、综合评价

### 整体定位
本仓库是xv6-riscv的一个功能扩展分支，其核心仍遵循MIT xv6的教学设计：简单、可读、无抽象层。但与原始版本“最小可运行”目标不同，本仓致力于拓展兼容能力——支持运行musl编译的动态ELF（通过Linux调用号表），并引入ext4文件系统以读写常见Linux磁盘镜像。然而扩展的代价是部分原始设计的精简（如BSS清零）和未完成子系统的遗留（信号、网络）。因此，其定位更接近“教学扩展型”内核：在xv6框架下探索功能增量，但未追求商业级完备。

### 真正的创新点
本仓最独特的创新在于双栈文件系统。它通过[fs.ref:1]中的魔数检测，在初始化时决定磁盘格式，然后在内存inode层统一表示。这种设计避免了实现完整VFS的复杂性，但让xv6原生日志与ext4写入路径共存。`ext4_bmap`[fs.ref:5]遍历extent树支持多级深度，表明开发者对ext4磁盘布局有深入理解。另一处巧妙设计是驱动层的上下文感知同步[drivers.ref:5]：在进程上下文中使用sleep/wakeup出让CPU，而在init上下文中直接轮询，避免了内核启动阶段的锁依赖。此外，懒分配机制[mm.ref:4]在缺页时按需分配，结合copyout/copyin中的自动触发[mm.ref:5]，在保持接口简单的同时减少了物理页浪费。MEMFS模式[drivers.ref:1]则是一种实用的测试优化，将文件系统镜像嵌入内核，加速迭代开发。

### 取舍判断
从实现细节可以看出明确的取舍策略。首先，系统调用层[syscall.ref:2]采用Linux调用号并包含大量stub（如SYS_lseek、SYS_ioctl返回0），优先确保musl引导流程通过，而非完整POSIX合规。其次，多核初始化被简化：entry.S未清BSS[boot.ref:1]，AP不调用timerinit[boot.ref:2]，开发者可能假设BSS由加载器清零、且仅BSP处理定时器中断以节省实现。再次，信号和网络子系统的完全缺失表明项目聚焦于存储、内存和进程的基础操作。最后，代码质量上多处表现出“够用即可”：管道计数器回绕[pipe.ref:8]、virtio轮询超时死锁[drivers.ref:4]、uvmunmap页表页泄漏[mm.ref:14]等悬而未决，反映了实验代码的常态。

### 完成度与不足
从功能完成度看，核心子系统（启动、内存、进程调度、文件系统、管道、驱动）均有基本实现，但信号和网络完全空白，系统调用仅约半数有实质性内容（约49个调用中~20个返回有效结果）。代码质量参差不齐：内存管理部分的懒分配和copyout协作相对完善，但存在walk中间页表未回滚[mm.ref:15]等瑕疵；驱动层虽有modern virtio支持，但超时检测逻辑失效[drivers.ref:4]；进程管理中的forkret无条件判断[task.ref:2]和锁竞争风险。总体而言，本仓体现了对xv6的深入理解和有价值的扩展尝试，但距离生产就绪尚有较大差距。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 本模块实现了从 OpenSBI 到 xv6 内核调度器入口的启动链条。核心抽象包含汇编入口 _entry、C 函数 start() 和 main()，以及全局 started 标志用于多核同步。它解决了在 RISC-V 监督模式下初始化内核子系统（内存、陷阱、进程、设备）的问题。与 xv6-riscv 原版相比，本实现缺少 BSS 清零循环、AP 的 timerinit 调用和 DTB 地址的实际使用，在多核完备性和健壮性上存在明显差距。

### 1. 核心抽象与启动链

启动流程从 OpenSBI 加载内核至 0x80000000 并跳转到 `_entry` 开始。汇编入口 `entry.S` 承担了极简的 CPU 本地状态初始化：将 hartid 保存至 tp 寄存器（供 `cpuid()` 使用），将 DTB 物理地址写入全局 `dtb_addr`，并为每个 CPU 分配指向 `stack0` 数组的初始栈指针，然后调用 `start()` <sup>[1](#mod-boot-ref-1)</sup>。`start()` 立即通过 `w_satp(0)` 关闭分页，避免早期代码使用不一致的地址映射，随后调用 `main()` <sup>[2](#mod-boot-ref-2)</sup>。

`main()` 是 C 级初始化的核心 <sup>[3](#mod-boot-ref-3)</sup>。引导 CPU（BSP）在 `started == 0` 分支中依次调用 `consoleinit`、`printfinit`、`kinit`、`kvminit`、`kvminithart`、`procinit`、`trapinit`、`trapinithart`、`timerinit`、`plicinit`、`plicinithart`、`binit`、`iinit`、`fileinit`、`virtio_disk_init`、`userinit`，最后通过原子 fence 设置 `started = 1`。辅助 CPU（AP）在 else 分支中等待 `started` 变为 1，然后执行 per-hart 的 `kvminithart`（加载页表）、`trapinithart`（安装陷阱向量）、`plicinithart`（配置 PLIC）。所有 CPU 最终进入无限循环 `scheduler()`。

### 2. 关键设计取舍

**栈分配方案**：`entry.S` 中 `la sp, stack0` 后通过 `(hartid+1)*4096` 计算偏移，使每个 CPU 获得不重叠的 4 KB 栈区域 <sup>[1](#mod-boot-ref-1)</sup>。这一设计省去了动态栈分配，但 4 KB 栈在嵌套中断或深度调用链下极易溢出；连续排列的栈区域更导致溢出会直接损坏邻接 CPU 的栈或数组边界之外的数据。原版 xv6 同样采用 4 KB，但本仓库缺少溢出保护措施。

**多核启动同步**：使用 `volatile static int started` 配合 `__atomic_thread_fence(__ATOMIC_SEQ_CST)` 实现轻量同步 <sup>[3](#mod-boot-ref-3)</sup>。BSP 单次写入，AP 自旋读取。在 RISC-V 弱序模型下，fence 保证了写操作在 started=1 之前对所有 CPU 可见，以及读操作不会提前看到 started==0 之后的初始化结果。这一方案避免了锁的引入，但若硬件实现不保证原子性（尽管原子操作指令可用），则存在竞态的可能。

**BSS 段未清零**：与 xv6 原版不同，`entry.S` 没有执行清零 BSS 的循环 <sup>[1](#mod-boot-ref-1)</sup>。C 标准要求静态变量零初始化，但若 bootloader 未清零 BSS（例如 QEMU 可能已清零，但物理硬件未必），`stack0` 中的残留数据可能破坏栈行为，`started` 可能不为 0 导致 BSP 跳过初始化。这一缺失是本启动流程最突出的隐患。

**timerinit 仅 BSP**：`timerinit()` 只在 BSP 路径被调用 <sup>[2](#mod-boot-ref-2)</sup><sup>[3](#mod-boot-ref-3)</sup>。AP 没有设置 `stimecmp`，因此无法接收 S 模式时钟中断。在多核调度中，AP 上运行的进程将无法被时钟中断抢占，可能导致死锁或无限循环。这明显是多核支持不完整的信号。

### 3. 跨模块协同

启动流程与多个子系统形成显式协同：
- **内存管理**：`start()` 关闭分页后，`main()` 立即通过 `kvminit()` 建立内核页表（物理恒等映射 + 跳板页），再通过 `kvminithart()` 写入 satp 启用分页。这一顺序确保了从物理寻址到虚拟寻址的无感过渡 <sup>[2](#mod-boot-ref-2)</sup><sup>[3](#mod-boot-ref-3)</sup>。
- **陷阱处理**：`trapinit()` 设置内核陷阱向量表，`trapinithart()` 写入 `stvec`。AP 在 else 分支中重复后者，因为每个 CPU 有独立的 `stvec` 寄存器 <sup>[3](#mod-boot-ref-3)</sup>。
- **进程管理**：`procinit()` 初始化进程表（含锁），`userinit()` 创建第一个用户进程，使得 scheduler 有就绪任务可调度 <sup>[3](#mod-boot-ref-3)</sup>。
- **设备栈**：`plicinit/plicinithart` 配置 PLIC，`virtio_disk_init` 初始化磁盘。值得注意的是，`entry.S` 保存的 DTB 地址（`dtb_addr`）在后续所有初始化中未被引用，设备树信息完全被忽略，依赖硬编码的地址和设备类型 <sup>[1](#mod-boot-ref-1)</sup>。

### 4. 边角细节与不足

1. **栈溢出风险**：每个 CPU 仅 4 KB 栈，`entry.S` 无边界检查；若 call tree 深入或中断嵌套，栈可能写穿至相邻区域甚至数组外 <sup>[1](#mod-boot-ref-1)</sup>。
2. **BSS 未清零**：缺失 xv6 原版中的 BSS 清零循环，全局变量初始化依赖 bootloader，存在隐晦 bug 可能 <sup>[1](#mod-boot-ref-1)</sup>。
3. **AP 无 clock 中断**：`timerinit` 只被 BSP 调用，AP 上 scheduler 无法获得时钟事件，多核调度的正确性存疑 <sup>[2](#mod-boot-ref-2)</sup><sup>[3](#mod-boot-ref-3)</sup>。
4. **dtb_addr 死赋值**：`entry.S` 写入了 DTB 地址，但 `start.c` 仅声明变量而无读取代码，设备树信息丢失 <sup>[1](#mod-boot-ref-1)</sup>。
5. **`w_satp(0)` 在 AP 上的语义模糊**：AP 执行 `start()` 时可能 BSP 已启用分页，此操作短暂关闭分页，但由于代码仍在物理地址运行，无实际危害，但表明设计者未完全考虑多核分页状态 <sup>[2](#mod-boot-ref-2)</sup>。
6. **`spin` 死循环**：若 `start()` 返回，`entry.S` 的 `spin` 形成死循环。虽然正常流程不会到达，但说明没有异常恢复路径 <sup>[1](#mod-boot-ref-1)</sup>。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块基于 xv6-riscv 的 Sv39 三级页表与空闲链表物理内存分配器，实现了用户进程地址空间管理。核心抽象是 pagetable_t 页表和 kmem 物理分配器。与 xv6 原版相比，关键扩展包括：通过 vmfault 实现懒分配（lazy allocation），在 copyout/copyin 中自动按需分配页面；支持 mmap/munmap/mprotect/brk 系统调用。模块采用直接映射的内核页表，并将跳板页固定在 TRAMPOLINE 虚拟地址以加速陷阱处理。

### 1. 核心抽象：页表与物理内存分配器

内存管理模块由两个紧密配合的组件构成：物理页面分配器（`kernel/kalloc.c`）和虚拟地址空间管理器（`kernel/vm.c`）。物理分配器基于最简单的空闲链表结构，全局变量 `kmem` 包含一个自旋锁 `lock` 和一个 `freelist` 指针，所有空闲页框通过 `struct run` 中的 `next` 字段串接 <sup>[1](#mod-mm-ref-1)</sup>。初始化时 `kinit()` 调用 `freerange()` 将内核结束地址 `end` 到 `PHYSTOP` 之间的所有物理页面逐个 `kfree()` 加入空闲链表 <sup>[2](#mod-mm-ref-2)</sup>。`kalloc()` 摘取链表头并返回，`kfree()` 将释放的页面清空为 0x01 后插回链表头部。这种分配器实现简单，分配和释放均为 O(1)，但不会合并相邻空闲页，长期运行后可能面临链表碎片化问题。

虚拟地址管理围绕 `pagetable_t`（本质是 `uint64*`，指向一个 4096 字节的页表页）展开。`kvmmake()` 构建内核的直接映射页表：设备地址（UART0、VIRTIO0、PLIC）、内核代码段（`KERNBASE~etext`，可读可执行）、内核数据段（`etext~PHYSTOP`，可读写）、跳板页（`TRAMPOLINE`，映射到 `trampoline` 符号）均使用 `kvmmap()` 映射 <sup>[3](#mod-mm-ref-3)</sup>。`kvmmap()` 封装 `mappages()`，若映射失败直接 `panic`，这种设计适合启动阶段，但在运行时无法优雅处理错误。

### 2. 关键设计取舍：懒分配与按需调页

本模块最显著的扩展是实现了懒分配（lazy allocation）机制。原版 xv6-riscv 的 `sys_sbrk()` 直接调用 `uvmalloc()` 立即分配物理页面，而此内核的 `vmfault()` 函数承担了按需分配的角色 <sup>[4](#mod-mm-ref-4)</sup>。在 `copyout()` 和 `copyin()` 中，当 `walkaddr()` 返回 0（页面未映射）时，会调用 `vmfault()` 尝试分配物理页面并建立映射 <sup>[5](#mod-mm-ref-5)</sup>。`vmfault()` 检查虚拟地址是否超出进程大小 `p->sz`，若未超出且页面尚未映射，则分配一个清零的物理页，并以 `PTE_R | PTE_W | PTE_U` 权限映射到进程页表 <sup>[6](#mod-mm-ref-6)</sup>。这种设计推迟了物理内存的分配，节省了空闲进程的地址空间开销，但代价是每次第一次访问缺页页面时触发额外的页表遍历和分配。

值得注意的是，`vmfault()` 硬编码了 `PTE_W` 权限，即使 `read` 参数指示是读访问。这意味着通过 `vmfault()` 分配的页面总是可写的，与 `mprotect` 系统调用可能设置的只读权限冲突——用户可以先 `mmap` 一个只读区域，但首次访问时 `copyout`/`copyin` 会通过 `vmfault` 将其映射为可写，破坏权限语义。此外，`copyout()` 在调用 `vmfault()` 后，额外通过 `walk()` 检查 PTE 的 `PTE_W` 标志 <sup>[7](#mod-mm-ref-7)</sup>，若页面只读则返回 -1，这似乎与 `vmfault` 总是设置 `PTE_W` 的行为矛盾，但可能用于防止非法写只读页面。

### 3. 跨模块协同：系统调用与陷阱处理

虽然无法直接读取陷阱处理代码，但从 `vmfault` 的调用上下文可以推断其协作方式。`vmfault()` 在 `copyout` 和 `copyin` 中被调用，这意味着用户态陷入内核执行 I/O 操作时，如果目标地址尚未分配物理页面，内核会按需补全映射。这与 `sys_sbrk()` 配合：`brk` 系统调用仅更新进程的 `sz` 字段和 `p->trapframe->sp` 栈指针，而不实际分配页面，真正的物理分配延迟到首次访问时由 `vmfault` 触发。`mmap`/`munmap` 系统调用则需要更精细的页表操作，`uvmunmap()` 可批量解除映射并选择是否释放物理页面 <sup>[8](#mod-mm-ref-8)</sup>，`uvmalloc()` 用于按指定权限分配内存 <sup>[9](#mod-mm-ref-9)</sup>。`mprotect` 需修改 PTEs 的权限标志，但当前代码未见专门函数——可能通过 `walk()` 定位 PTE 后直接修改标志位实现。

模块间的协同还体现在进程生命周期中：`uvmcreate()` 创建空白用户页表 <sup>[10](#mod-mm-ref-10)</sup>；`uvmfree()` 释放所有用户页面并递归释放页表页 <sup>[11](#mod-mm-ref-11)</sup>；`uvmcopy()` 在 fork 时逐页拷贝物理页面，但未实现 COW <sup>[12](#mod-mm-ref-12)</sup>。内核栈通过 `proc_mapstacks()` 逐进程映射到内核页表。跳板页 `TRAMPOLINE` 被映射到最高虚拟地址，与每个进程的用户页表共享相同的物理页，确保陷入/返回时无需切换页表。

### 4. 边角细节与不足

以下从代码安全与设计权衡角度指出若干风险点：

- **`vmfault` 与已映射页面的交互缺陷** <sup>[13](#mod-mm-ref-13)</sup>：`vmfault()` 在调用 `mappages()` 前会检查 `ismapped()`，若页面已映射则返回 0。但 `copyout()` 和 `copyin()` 将 `vmfault()` 返回 0 视为失败，导致合法访问（如第二次访问同一页）被错误拒绝。实际上，已映射的页面根本不应进入 `vmfault` 处理——`walkaddr` 已经发现页面存在，但 `walkaddr` 调用 `walk`，如果 PTE 存在但 `PTE_V` 为 0（如被 `uvmunmap` 清空但不释放），`walkaddr` 也会返回 0，进而调用 `vmfault`，但 `vmfault` 的 `ismapped` 同样返回 0，从而正常分配。真正的风险在于：若某页 PTE 有效但 `PTE_U` 为 0（如用户栈保护页），`walkaddr` 返回 0，`vmfault` 会重新映射该页，覆盖原保护页的语义。
- **`uvmunmap` 不释放页表页** <sup>[14](#mod-mm-ref-14)</sup>：与 `freewalk` 配合，`uvmunmap` 仅解除叶节点 PTE，不回收中间页表页。`uvmfree` 先调用 `uvmunmap` 释放所有物理页，再调用 `freewalk` 递归释放页表页。但如果中途调用 `uvmunmap` 后不调用 `freewalk`，则中间页表页会泄漏。`uvmdealloc` 在收缩进程时调用 `uvmunmap` 但不回收页表页，这会导致地址空间降级后页表页残留，浪费内存。
- **`walk` 的分配失败不回滚** <sup>[15](#mod-mm-ref-15)</sup>：当 `walk` 的 `alloc=1` 时，如果中间级页表分配失败（`kalloc` 返回 0），函数返回 0，但之前分配的更高级页表页不会被释放。调用者 `mappages` 会返回 -1，但已分配的中间页表页被孤立。这可能在内存紧张时造成页表页泄漏。
- **物理分配器的双重释放检查缺失** <sup>[16](#mod-mm-ref-16)</sup>：`kfree` 只检查地址是否在合法范围和对齐，不检查该页是否已在空闲链表中。调用者错误地多次 `kfree` 同一页会导致链表成环，后续 `kalloc` 返回已被使用的页面，造成数据损坏。
- **硬编码物理地址边界** <sup>[2](#mod-mm-ref-2)</sup>：`PHYSTOP` 被假定为物理内存上限，且与内核数据段 `etext` 的边界通过 `end` 符号确定。这种假设不适用于可变内存配置的物理机或非 QEMU 平台。
- **`uvmcopy` 未实现写时复制（COW）** <sup>[12](#mod-mm-ref-12)</sup>：fork 时立即拷贝所有物理页面，对于大地址空间但实际使用少的情况，效率很低。原版 xv6 也未实现 COW，但本内核既然实现了懒分配，理应后续扩展 COW 以减少内存压力。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 进程模块以 struct proc 为核心，管理进程生命周期（创建、调度、退出、等待），采用基于自旋锁保护的状态机模型。调度器实现简单的轮转调度，通过 swtch 汇编进行上下文切换。相比 xv6-riscv 原版，扩展支持 clone 系统调用（kfork_ex 带子栈指针），增加了竞赛测试框架字段（contest_ticks/path/argv），并将 init 进程启动逻辑移至 forkret 中通过 kexec 实现，同时移除了传统的 usertrapret 调用。

### 1. 核心抽象：进程结构与生命周期
进程的核心是 `struct proc`，包含状态、锁、页表、内核栈、trapframe 等字段。进程表 `proc[NPROC]` 是全局静态数组，每个 CPU 对应一个 `struct cpu` 记录当前运行的进程。初始化阶段 `procinit` 为每个进程分配内核栈并映射到高地址，设置锁和初始状态为 UNUSED <sup>[1](#mod-task-ref-1)</sup>。`allocproc` 遍历进程表找到一个 UNUSED 条目，为其分配 trapframe 页、创建用户页表（包含跳板和 trapframe 映射），并初始化上下文使首次调度时跳转到 `forkret` <sup>[2](#mod-task-ref-2)</sup>。`userinit` 创建第一个 init 进程，分配 proc 结构并设置为 RUNNABLE，`cwd` 指向根目录 <sup>[3](#mod-task-ref-3)</sup>。进程状态包括 UNUSED、USED、RUNNABLE、RUNNING、ZOMBIE 等，通过 `acquire`/`release` 锁保护状态迁移，确保并发安全。

### 2. 调度与上下文切换机制
调度器 `scheduler` 是每个 CPU 上的无限循环：开启中断后遍历进程表，选择 RUNNABLE 进程，获取其锁后通过 `swtch` 切换到该进程的上下文（保存当前 CPU 的寄存器，加载目标进程的寄存器）<sup>[4](#mod-task-ref-4)</sup> <sup>[5](#mod-task-ref-5)</sup>。`swtch` 汇编仅保存/恢复 callee-saved 寄存器（ra, sp, s0-s11），返回地址通过 ra 传递，切换后目标进程从 `forkret` 开始执行 <sup>[6](#mod-task-ref-6)</sup>。`sched` 函数用于进程主动让出 CPU（如 `yield`、`kexit`），要求持有 `p->lock` 且中断关闭，通过 `swtch` 切换回调度器 <sup>[7](#mod-task-ref-7)</sup>。`yield` 将状态设为 RUNNABLE 后调用 `sched`，通常由定时器中断触发 <sup>[8](#mod-task-ref-8)</sup>。整个调度器无优先级、无时间片计数，是纯粹的轮转调度。

### 3. 进程创建与退出：fork/clone/exit/wait
`kfork_ex` 实现 fork（child_sp=0）和 clone（child_sp≠0），分配新进程，通过 `uvmcopy` 拷贝用户内存，拷贝 trapframe 并将子进程 a0 置 0，复制文件描述符表和当前目录，设置父进程后状态变为 RUNNABLE <sup>[9](#mod-task-ref-9)</sup>。`kexit` 关闭所有打开的文件、释放当前目录，将子进程通过 `reparent` 托管给 init，唤醒父进程，设置 xstate 和 ZOMBIE 状态后调用 `sched` 不再返回 <sup>[10](#mod-task-ref-10)</sup>。`kwait` 在 `wait_lock` 下循环扫描子进程，找到 ZOMBIE 后通过 `copyout` 将退出状态（编码为 (status << 8) 以兼容 Linux ABI）写回用户空间，然后释放子进程 <sup>[11](#mod-task-ref-11)</sup>。`forkret` 在首次调度时执行 `kexec` 启动 init 进程；对于竞赛测试进程，则根据 `contest_path` 构造参数并执行相应 busybox 命令 <sup>[12](#mod-task-ref-12)</sup>。

### 4. 边角细节与不足
【风险点 1: 整数溢出】`growproc` 中 `sz + n` 在 n 为负时可能回绕：虽然 `uvmdealloc` 内部处理缩小，但回绕后可能误判为巨大值，导致未定义行为 <sup>[13](#mod-task-ref-13)</sup>。【风险点 2: forkret 返回未定义行为】普通 fork 子进程进入 `forkret` 后既不满足 first 条件也无 contest_path，函数直接结束并返回，但此时 ra 指向不确定地址（因缺失 `usertrapret` 调用），必然崩溃或执行错误代码路径 <sup>[14](#mod-task-ref-14)</sup>。【风险点 3: 首次调度竞态】`forkret` 中的 `static int first` 未受锁保护，多 CPU 同时调度新进程可能同时判定 first=1 并两次执行 `kexec`，虽然后续 `kexec` 可能返回错误，但存在重复初始化风险 <sup>[15](#mod-task-ref-15)</sup>。【风险点 4: sched 断言过于严格】`sched` 检查 `mycpu()->noff != 1` 确保持锁数为 1，但若进程嵌套持锁（例如文件系统操作中持有额外锁），会触发 panic，限制了锁模型 <sup>[16](#mod-task-ref-16)</sup>。【风险点 5: kwait 退出编码未校验】`enc_status = pp->xstate << 8` 未检查 `xstate` 是否超出 uint8 范围，若状态码大于 255 会导致截断且不符合 Linux 规范 <sup>[17](#mod-task-ref-17)</sup>。此外，`kexit` 中 init 进程退出时直接调用 SBI 关机而非 panic，表明竞赛环境下的特殊处理，但通用实现中 init 应循环等待。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 文件系统模块实现了一个双栈架构，同时支持 xv6 原生的 FSMAGIC 格式和 Linux ext4 只读（含部分写入）格式。核心抽象是 struct inode（内存 inode 表，通过 itable 保护）和 struct file（文件描述符表）。xv6 层使用经典的块位图分配器、日志事务（log.c）和间接块映射；ext4 层则绕过日志，直接解析 ext4 超级块、块组描述符、extent 树和变长目录项。路径名解析（namex）统一处理两种格式，通过全局标志 ext4_fs 动态切换。与 xv6-riscv 原版相比，该实现新增了 ext4 磁盘格式支持，但 ext4 写入路径缺乏日志保障，且符号链接被退化为普通文件。

### 1. 核心抽象：双文件系统栈

文件系统模块在 VFS 层面统一使用 xv6 的 `struct inode` 作为内存中 inode 表示，但磁盘格式支持两套：xv6 原生格式（`FSMAGIC`）和 Linux ext4。初始化时通过 `readsb()` 读取磁盘超级块，根据魔数决定 `ext4_fs` 标志<sup>[1](#mod-fs-ref-1)</sup><sup>[2](#mod-fs-ref-2)</sup>。在 ext4 模式下，超级块解析完成后立即分配内存加载块组描述符表（GDT），并通过 `iget` + `ilock` 验证根 inode 可读<sup>[3](#mod-fs-ref-3)</sup>。此后所有文件操作均通过 `ext4_fs` 分支进行：磁盘 inode 的读取由 `ext4_read_inode()` 完成，它将 ext4 的 `struct ext4_inode` 字段转换为 xv6 的 `ip->type`/`ip->size`/`ip->nlink`<sup>[4](#mod-fs-ref-4)</sup>；数据块映射使用 `ext4_bmap()` 遍历 extent 树，支持 1~4 级深度<sup>[5](#mod-fs-ref-5)</sup>；目录遍历由 `dirlookup()` 的 ext4 分支处理变长 `ext4_dir_entry`<sup>[6](#mod-fs-ref-6)</sup>。路径名解析 `namex()` 在顶层统一处理，但起始根 inode 号根据 `ext4_fs` 选择 `EXT4_ROOT_INO` (2) 或 `ROOTINO` (1)<sup>[7](#mod-fs-ref-7)</sup>；它复用了 xv6 的 `ilock`/`iunlockput` 锁协议来保证并发安全。

### 2. 关键设计取舍：ext4 只读与性能权衡

最大的设计决策是 **复用 xv6 的 inode 缓存与锁机制，但为 ext4 绕过日志系统**。由于 ext4 是日志文件系统（jbd/jbd2），而 xv6 的 log.c 只能重放自己的日志格式，直接混合将导致互不兼容。实现者选择在 `ext4_fs=1` 时让 `begin_op()`/`end_op()` 空转<sup>[8](#mod-fs-ref-8)</sup>，`log_write()` 退化为直接 `bwrite()`<sup>[9](#mod-fs-ref-9)</sup>，这意味着 ext4 写入操作不经过任何崩溃原子性保护。这是明显的**取舍**：为了快速获得 ext4 读取能力，放弃了写入安全。替代方案可以是实现 ext4 日志的重放，或者使用 xv6 的日志包装 ext4 的元数据更新，但前者工作量巨大，后者会破坏 ext4 磁盘结构与 Linux 的兼容性。此选择导致 ext4 写入路径（如 `ialloc`、`writei` 触发的块分配）在系统崩溃后可能留下不一致状态。另一个取舍是 **固定 BSIZE=1024 且强制 ext4 块大小为 BSIZE 的整数倍**（见 `readsb` 中的检查 `ext4_block_size % BSIZE != 0`<sup>[10](#mod-fs-ref-10)</sup>）。这是由于 xv6 的块缓冲层（bio.c）以 BSIZE 为粒度操作，若 ext4 块大于 BSIZE，代码将每个 ext4 块拆解为多个扇区进行处理，增加了复杂度但避免了重写缓冲层。代价是只兼容 1024、2048、4096 等 1024 倍数的 ext4 块大小。

### 3. 跨模块协同：缓冲、日志与 VFS 层

模块间的数据流如下：系统调用层（sysfile.c）调用 `file.c` 的 `fileread`/`filewrite`，这些函数通过 `readi`/`writei` 与 inode 交互。`readi`/`writei` 调用 `bmap()` 获取磁盘扇区号，然后通过 `bio.c` 的 `bread`/`bwrite` 读写缓冲块<sup>[11](#mod-fs-ref-11)</sup>。块缓冲层维护 LRU 链表并使用自旋锁保护元数据，每个缓冲块有独立的睡眠锁，避免持锁时睡眠<sup>[12](#mod-fs-ref-12)</sup>。对于 xv6 格式，`log_write` 将修改后的块号记入日志，提交时由 `commit()` 执行写日志、写盘、清除三步<sup>[13](#mod-fs-ref-13)</sup>；对于 ext4 格式，`log_write` 直接调用 `bwrite` 将数据写入原始位置<sup>[9](#mod-fs-ref-9)</sup>，完全绕过日志。文件描述符层（file.c）通过 `fileclose` 在关闭时调用 `begin_op`/`end_op` 包裹 `iput`，确保 xv6 格式下的日志事务边界正确<sup>[14](#mod-fs-ref-14)</sup>。路径名解析 `namex` 需要在遍历过程中多次调用 `ilock`/`iunlockput`，复用 xv6 的睡眠锁协议以避免死锁——它先锁定父目录，找到子 inode 后释放父目录再加锁子 inode，这是经典的逐层锁传递模式。

### 4. 边角细节与不足

- **<sup>[15](#mod-fs-ref-15)</sup> ext4 写入路径缺失日志**：如前所述，`log_write` 在 ext4 模式下直接写盘，若在 `writei` 过程中系统崩溃，可能出现已分配的块未计入位图、或 inode 元数据与数据不一致。 `ialloc` 和 `ext4_bmap_alloc`（在 `bmap` 中被调用）的实现在当前片段中仅存声明，实际定义在未提供的 `ext4_alloc.c` 中，这可能是一个未完成的桩。
- **<sup>[16](#mod-fs-ref-16)</sup> 符号链接被退化为普通文件**：`ext4_read_inode` 将 `EXT4_S_IFLNK` 映射为 `T_FILE`，且不读取链接目标路径。当用户读取符号链接文件时将返回文件内容而非目标路径，导致 `readlinkat` 等系统调用行为异常。
- **<sup>[17](#mod-fs-ref-17)</sup> 固定 NINODE 与潜在 OOM**：`itable.inode[NINODE]` 是固定大小数组（NINODE 默认 50），当同时打开大量文件时 `iget` 会 panic<sup>[18](#mod-fs-ref-18)</sup>。在 ext4 模式下，`iget` 行为相同，同样受此限制。
- **<sup>[19](#mod-fs-ref-19)</sup> 块分配器在位图操作中的竞态**：ext4 模式下 `iput` 直接修改 inode 位图`<sup>[20](#mod-fs-ref-20)</sup>`，既不通过日志也不持有 `bcache` 锁（只持有 `itable.lock` 且短暂释放），若两个进程同时释放不同的 inode 可能破坏位图一致性。
- **<sup>[21](#mod-fs-ref-21)</sup> 用户地址校验依赖外部函数**：`readi`/`writei` 使用 `either_copyout`/`either_copyin`，这些函数需要正确检查用户空间地址边界。片段中未见其实现，若实现不完整可能导致任意内核内存读写。
- **<sup>[22](#mod-fs-ref-22)</sup> ext4_gdt 内存泄漏**：`ext4_gdt` 通过 `kalloc()` 分配且没有释放机制，若文件系统被卸载（当前不支持卸载，但 `umount2` 系统调用已声明，实现可能为空），描述符表内存将永久泄漏。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 核心抽象为管道（pipe），基于固定大小环形缓冲区、自旋锁配合 `sleep`/`wakeup` 实现阻塞式读写。管道通过文件描述符接口暴露，用户进程通过 `read()`/`write()` 操作。该实现与 xv6-riscv 原版一致，未引入信号、共享内存或 socket 等扩展。

### 1. 核心抽象与数据结构

管道由 `struct pipe` 表示（`pipe.c:6-14`），包含 512 字节环形缓冲区 `data`、两个累计计数器 `nread`/`nwrite` 以及开关标志 `readopen`/`writeopen`。分配函数 `pipealloc` 同时创建两个 `struct file` 对象（一个只读、一个只写）并共享同一 `struct pipe` 实例。这种设计将管道融入已有文件描述符框架，用户无需感知底层差异。`pipealloc` 采用“先分配所有可能失败资源，再组装”的风格，若中间失败则回滚已分配资源，避免泄漏 <sup>[1](#mod-ipc-ref-1)</sup>。

### 2. 阻塞 I/O 与同步机制

管道读写均采用阻塞语义。自旋锁 `&pi->lock` 保证互斥访问；条件变量通过 `sleep`/`wakeup` 实现。

- **写满阻塞**：当缓冲区满时（`nwrite == nread + PIPESIZE`），写者调用 `sleep(&pi->nwrite, &pi->lock)` 释放锁并睡眠 <sup>[2](#mod-ipc-ref-2)</sup>。读者读取后通过 `wakeup(&pi->nwrite)` 唤醒写者 <sup>[5](#mod-ipc-ref-5)</sup>。
- **读空阻塞**：当缓冲区空时，读者调用 `sleep(&pi->nread, &pi->lock)` <sup>[3](#mod-ipc-ref-3)</sup>。写者写入后通过 `wakeup(&pi->nread)` 唤醒读者 <sup>[4](#mod-ipc-ref-4)</sup>。
- **唤醒语义**：`wakeup` 唤醒所有等待指定 channel 的进程，但只有一个进程能获得锁并继续，其余进程重新检查条件后再睡眠。这避免了丢失唤醒，但可能引发“惊群”效应，不过对于管道场景影响较小。

### 3. 跨模块协同

- **文件系统**：`pipealloc` 返回的 `struct file` 通过 `filealloc()` 从全局文件表分配，经由 `fileclose` 管理引用计数，确保 `fork` 继承时生命周期正确。
- **进程调度**：`sleep`/`wakeup` 依赖调度器（`proc.c`），`sleep` 原子性地释放锁并置进程为 SLEEPING 状态，被唤醒后重新获取锁。
- **内存管理**：数据通过 `copyin`/`copyout` 在用户页表与内核缓冲区之间传输 <sup>[6](#mod-ipc-ref-6)</sup><sup>[7](#mod-ipc-ref-7)</sup>。若用户地址无效，返回 -1；但 `pipewrite` 对拷贝失败的处理不一致：它在 `copyin` 失败时仅 `break` 并返回已写入字节数（可能为 0），而非返回 -1 <sup>[6](#mod-ipc-ref-6)</sup>；而 `piperead` 在 `copyout` 失败且未读取任何字节时返回 -1 <sup>[7](#mod-ipc-ref-7)</sup>。这种差异可能令用户程序无法区分地址错误与正常短返回。

### 4. 边角细节与不足

- **整数溢出风险**：`nread`/`nwrite` 为 `uint`，无限递增。当写入超过 2^32 字节时，`nwrite` 回绕零，但 `nread` 未回绕，导致 `nwrite == nread + PIPESIZE` 满条件判断出错 <sup>[8](#mod-ipc-ref-8)</sup>。尽管受限于 512 缓冲区触发概率低，但属潜在缺陷。
- **copyin 失败处理不一致**：如前所述，`pipewrite` 在拷贝失败时返回短计数而非错误码，不符合 POSIX 规范 <sup>[6](#mod-ipc-ref-6)</sup>。
- **管道大小硬编码**：`PIPESIZE` 定义为 512（`pipe.c:3`），无法动态调整。
- **持锁期间调用 copyin/copyout**：自旋锁持有期间（中断关闭）调用可能阻塞的拷贝函数，若未来缺页逻辑引入睡眠则可能导致死锁。当前 xv6 的缺页处理不会在 copyin/copyout 中阻塞，风险可控。
- **无非阻塞模式**：不支持 `O_NONBLOCK`，无法用于多路复用场景。

总体而言，该管道实现符合 xv6 的简洁教学风格，正确性经过经典验证，但距离生产级可靠性尚有距离。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 该仓库的驱动框架延续 xv6 的简约风格——无统一抽象层，每个设备直接操作 MMIO 寄存器并通过中断与内核交互。核心包括：① virtio 块设备驱动，基于 VirtQueue 描述符环，同时支持 legacy (pre-v1.0) 和 modern (v1.0+) MMIO 传输协议，并提供 MEMFS 编译模式将文件系统镜像嵌入内核；② 16550a UART 串口驱动，提供同步轮询输出和基于睡眠锁的阻塞写两种路径；③ RISC-V PLIC 中断控制器驱动。与原版 xv6-riscv 相比，扩展体现在现代 virtio 支持、MEMFS 模式以及上下文感知的 I/O 完成等待策略（进程上下文用 sleep/wakeup，init 上下文直接轮询）。

### 1. 核心抽象：MMIO 寄存器与 VirtQueue 环

该仓库的驱动框架没有统一的设备抽象层，每个设备驱动以独立 C 文件存在，直接操作 MMIO 寄存器并通过中断与内核上层交互。这种设计延续了 xv6 的极简风格，每个驱动内部封装了完整的协议栈。

**virtio 块设备** 的核心抽象是 VirtQueue 描述符环：驱动在内存中维护 desc（描述符表）、avail（可用环）和 used（已用环）三个结构体，通过 MMIO 寄存器与设备共享。对寄存器的访问通过 `#define R(r) ((volatile uint32 *)(VIRTIO0 + (r)))` 宏实现，直接对物理地址进行 volatile 指针转换 <sup>[2](#mod-drivers-ref-2)</sup>。驱动在提交请求前后插入 `asm volatile("fence rw, rw" ::: "memory")` 内存屏障，保证描述符写入对设备可见。

MEMFS 编译模式是一个有趣的扩展：当定义 `MEMFS` 宏时，`virtio_disk_init` 和 `virtio_disk_intr` 变成空函数，`virtio_disk_rw` 通过 `memmove` 直接操作嵌入内核二进制中的文件系统镜像 <sup>[1](#mod-drivers-ref-1)</sup>。这完全绕过了硬件 virtio 交互，适合早期启动测试或嵌入式场景，但放弃了 virtio 设备的热插拔和中断驱动能力。

**UART 串口驱动** 使用 `#define Reg(reg) ((volatile unsigned char *)(UART0 + (reg)))` 宏进行 MMIO 寄存器访问，配合 `ReadReg`/`WriteReg` 封装。驱动维护一个全局自旋锁 `tx_lock` 和发送忙标志 `tx_busy`，配合 `sleep`/`wakeup` 实现阻塞写。

**PLIC 中断控制器** 提供最简接口：`plicinit` 设置所有 IRQ 优先级、`plicinithart` 使能 hart 级中断、`plic_claim` 获取待处理中断号、`plic_complete` 通知处理完成。

### 2. 关键设计取舍：兼容性、上下文感知与同步策略

**Legacy vs Modern virtio**：驱动在初始化时通过检测设备特征寄存器高 32 位是否为零来判定传输协议版本 (`disk.legacy = (fhi == 0)`) <sup>[2](#mod-drivers-ref-2)</sup>。Legacy 模式使用 `VIRTIO_MMIO_QUEUE_PFN` 传递队列物理页号，要求 desc、avail、used 三部分在物理内存中连续且 used ring 页对齐。驱动为此静态分配了 `char pages[2*PGSIZE] __attribute__((aligned(PGSIZE)))` 数组 <sup>[3](#mod-drivers-ref-3)</sup>——这种做法的代价是无论队列实际大小，都固定占用 2 页物理内存。Modern 模式则使用独立的描述符寄存器（`QUEUE_DESC_LOW/HIGH` 等），每个队列部分由 `kalloc()` 分配独立页，更加灵活。这一取舍反映了对旧版 QEMU 设备的兼容性需求与内存效率之间的平衡。

**上下文感知的同步策略**：virtio 驱动在 `virtio_disk_rw` 中根据 `myproc() != 0` 判断当前执行上下文 <sup>[5](#mod-drivers-ref-5)</sup>：在进程上下文中，驱动调用 `sleep(b, &disk.vdisk_lock)` 等待中断处理程序通过 `wakeup(b)` 唤醒；在 init 上下文中（无调度器），驱动直接轮询 `used->idx` 直到设备完成。这种设计避免了为 init 阶段引入调度器依赖，但意味着 init 阶段的 I/O 是忙等待，CPU 利用率低且不存在超时保障。

**睡眠锁 vs 自旋锁**：UART 驱动选择睡眠锁 `tx_lock`，使 `uartwrite` 在 UART 忙时可以睡眠等待发送完成中断 <sup>[6](#mod-drivers-ref-6)</sup>。virtio 驱动则使用自旋锁 `vdisk_lock`，因为中断处理程序 `virtio_disk_intr` 也需持有该锁，中断上下文中不能睡眠。这意味着进程上下文持有 `vdisk_lock` 时，当前 CPU 的中断必须处于关闭状态——xv6 的 `acquire` 内部调用 `push_off` 隐式保证这一点，但这一约束未被文档化。

### 3. 跨模块协同：中断分发、调度器配合与文件系统接口

**与 PLIC 的协同**：设备中断通过 PLIC 路由到 CPU 的 S 模式。`plic_claim()` 假定在 `devintr()` 中被调用，返回中断号后分发到对应设备的处理函数。`plicinit` 将所有 63 个可用 IRQ 优先级设为 1 <sup>[7](#mod-drivers-ref-7)</sup>，未做任何差异化——这意味着高延迟敏感的 UART 中断可能被吞吐型 virtio 中断淹没。

**与调度器的协同**：virtio 和 UART 驱动都深度嵌入 xv6 的 sleep/wakeup 机制。在 `virtio_disk_rw` 中，驱动持有 `vdisk_lock` 时调用 `sleep(b, &disk.vdisk_lock)`，释放锁并阻塞进程；中断处理程序在处理完 I/O 后调用 `wakeup(b)` 唤醒等待进程 <sup>[5](#mod-drivers-ref-5)</sup>。类似地，UART 的 `uartwrite` 在 `tx_busy` 为 1 时睡眠在 `&tx_chan` 上，中断处理程序 `uartintr` 在检测到 LSR 发送空闲时调用 `wakeup(&tx_chan)` 唤醒。

**与文件系统/控制台的协同**：virtio 驱动通过 `struct buf` 与文件系统层交互，`virtio_disk_rw` 接收 `struct buf *b`，将其 `b->data` 字段的物理地址直接填入描述符（假设恒等映射）。UART 驱动的中断处理程序接收输入字符后调用 `consoleintr(c)` 处理控制台输入，连接了硬件中断与终端子系统。

### 4. 边角细节与不足

**<sup>[4](#mod-drivers-ref-4)</sup> 轮询超时检测逻辑 bug**：在 init 上下文轮询循环中，超时检测代码存在明显的逻辑错误：
```c
if (poll == 100000 || poll == 500000 || poll == 1000000)
if (poll > 10000000)
    panic("virtio_disk_rw: poll timeout");
```
两个 if 语句在 C 语法下嵌套组合，实际效果是仅当 `poll` 等于三个幻数之一 **且** `poll > 10000000` 时才 panic。由于外层 if 已将 poll 限定在 ≤1000000，内层条件永远不可能满足，panic 无法触发。这很可能是调试代码残留（首个 if 本应包含进度打印，被删除后留下了空 if）。在正常 init 流程中 I/O 通常快速完成，此 bug 不易暴露，但在慢速设备或虚拟化竞态下可能导致无超时保障的死循环。

**<sup>[7](#mod-drivers-ref-7)</sup> PLIC 硬编码 IRQ 数量**：`plicinit` 中的循环 `for (int i = 1; i < 64; i++)` 假设 PLIC 始终支持 63 个 IRQ。RISC-V PLIC 规范允许更少的 IRQ 源，硬编码限制了移植性。此外，所有 IRQ 优先级统一为 1，没有为 UART 等延迟敏感设备设置更高优先级。

**<sup>[6](#mod-drivers-ref-6)</sup> UART 同步输出的 panic 死循环**：`uartputc_sync` 中 `if (panicked) { for (;;); }` 在系统 panic 后进入死循环，这是 xv6 的标准做法，但意味着 panic 后的调试输出无法通过 UART 完成，systemd 风格的看门狗也无法介入。

**<sup>[1](#mod-drivers-ref-1)</sup> MEMFS 模式的竞态风险**：MEMFS 模式下的 `virtio_disk_rw` 直接通过 `memmove` 操作全局镜像缓冲区，没有使用任何互斥机制。在多核 CPU 同时访问文件系统时（即使是在 init 阶段之后），可能发生数据竞争。边界校验 `off + BSIZE > size` 也仅防止单次越界，若 `blockno` 源自被破坏的缓冲区，仍可能导致非连续越界访问。

**<sup>[3](#mod-drivers-ref-3)</sup> 静态数组的内存浪费与地址假设**：Legacy 模式分配的 `pages[2*PGSIZE]` 在 `disk` 结构体中永远占用 2 页，即使用户配置的队列深度 `NUM` 远小于页容量（典型值 32 个描述符仅占约 512 字节）。更严重的是，`disk.pages` 的物理地址被直接左移 12 位作为 `QUEUE_PFN`，依赖恒等映射的假设；若系统启用了非直接页表，此写法将产生错误地址。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层由 syscall() 分发函数和 syscalls[] 函数指针表构成，表以 Linux RISC-V 调用号索引。它抽象了用户陷入到内核操作的路由（核心结构体为 struct proc 的 trapframe 和 ofile 数组）。相比原版 xv6，此实现大幅扩展了系统调用兼容性，添加了 openat、wait4、brk 等 Linux 调用，并实现了内核级 shebang 解析以支持脚本执行。

### 1. 核心抽象与外部依赖
系统调用层位于 `kernel/syscall.c`，是用户态与内核态间的仲裁者。其核心是 `syscall()` 函数，从当前进程 `trapframe` 的 `a7` 寄存器提取调用号，并查表调用对应处理函数 <sup>[1](#mod-syscall-ref-1)</sup>。参数提取通过 `argraw()` 从 `a0`–`a5` 直接获取，并包装为 `argint`、`argaddr`、`argstr` 等函数。文件描述符参数由 `sysfile.c` 的 `argfd()` 校验并查找 `struct proc` 中的 `ofile[]` 表 <sup>[3](#mod-syscall-ref-3)</sup>。该层依赖 `proc.h`、`file.h`、`fs.h` 等模块，所有系统调用最终委托给文件系统、进程管理或设备驱动。

### 2. 关键设计取舍
- **Linux ABI 兼容**：`syscalls[]` 表编排为 Linux RISC-V 调用号 <sup>[2](#mod-syscall-ref-2)</sup>，包含 40+ 个条目，许多是返回 0 的 stub（如 `SYS_lseek`、`SYS_ioctl`）。这种设计使 musl 编译的动态 ELF 能直接运行，但错误码被静默，可能导致应用误判。
- **sys_write 后备输出**：当 `fd=1/2` 且文件未打开时，直接写串口 <sup>[4](#mod-syscall-ref-4)</sup>。这避免了文件系统未就绪时的死锁，但 `copyin` 失败时误报成功写入（函数返回 `n`，调用者认为全部写入）。
- **shebang 内核解析**：`sys_exec` 中完全解析 `#!` 行 <sup>[5](#mod-syscall-ref-5)</sup>，灵活但粗糙：路径映射、参数构造每个都用 `kalloc` 分配，却未检查返回值 <sup>[6](#mod-syscall-ref-6)</sup>，存在空指针风险。
- **内存管理双模式**：`sys_sbrk` 支持懒分配（`SBRK_LAZY`）和积极分配（`SBRK_EAGER`），通过额外参数 `t` 选择 <sup>[7](#mod-syscall-ref-7)</sup>，但整数溢出检查在有符号/无符号混用下可能不可靠。

### 3. 跨模块协同
- 文件操作委托 `file.c`（`filealloc`、`fileread` 等）和 `fs.c`（`namei`、`readi`、`writei`）。
- 进程控制委托 `proc.c`（`kexit`、`kfork_ex`、`growproc`）。
- 地址验证依赖 `copyin`/`copyout`；`fetchaddr` 检查用户地址范围 <sup>[8](#mod-syscall-ref-8)</sup>。
- 时钟相关系统调用（`sys_pause`、`sys_uptime`）使用 `tickslock`，与 `trap.c` 的中断处理同步。

### 4. 边角细节与不足
1. **整教溢出风险** (`fetchaddr` <sup>[8](#mod-syscall-ref-8)</sup>)：`addr + sizeof(uint64) > p->sz` 在 `addr` 接近 `UINT64_MAX` 时加法回绕可能绕过检查。
2. **`kalloc` 返回值未检查** (shebang 路径 <sup>[6](#mod-syscall-ref-6)</sup>)：`new_argv[new_argc] = kalloc(); safestrcpy(...)` 未判断返回 `NULL`，可能导致空指针解引用。
3. **桩函数无操作** (`sys_munmap`、`sys_mprotect` <sup>[9](#mod-syscall-ref-9)</sup>)：直接返回 0，应用得不到反馈，内存映射错误静默。
4. **sys_write 后备错误语义** <sup>[4](#mod-syscall-ref-4)</sup>：`copyin` 失败时仅 `break`，函数仍返回 `n`，调用者误以为全部写入。
5. **整型溢出与符号混用** (`sys_sbrk` <sup>[7](#mod-syscall-ref-7)</sup>)：`addr + n < addr` 中 `uint64` 与 `int` 隐式转换，负数 `n` 变为大无符号数，可能产生意外回绕。
6. **硬编码常量** (`sys_getcwd` <sup>[10](#mod-syscall-ref-10)</sup>)：始终返回 `"/"`，不支持真实当前工作目录。
