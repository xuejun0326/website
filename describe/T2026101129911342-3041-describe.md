## 📖 一、总览

该仓库是一个面向 LTP 兼容性测试的单进程顺序执行实验内核，属于 rcore-rust 家族中的“测试兼容层”类型。它通过全局静态变量、unsafe 代码和激进简化，在约 3500 行代码内实现了内存分配、virtio-blk 块设备驱动、全内存文件系统（含 ext4 只读加载）、极简网络栈和 ~100+ 系统调用分发。尽管缺乏任务调度、信号、IPC 等核心模块，启动流程也未独立模块化，但通过 global_asm 入口和 khal/kunikos workspace 组织，勉强构成一个可运行特定 LTP 测试子集的内核镜像。

评审最需关注的三个特点：其一，文件系统模块引入扁平化路径名线性搜索与“胖枚举” Fd 设计 [fs.ref:1][fs.ref:3][fs.ref:4]，在消除分层遍历开销的同时，将六种后端统一在一个 match 派发中，这对 LTP 测试的单文件路径模式极高效；其二，系统调用层摒弃 trait 或宏注册机制，采用单一巨型 match 分支内联所有逻辑 [syscall.ref:1]，相当比例的调用退化为常量返回（如 `96 => 1`），这种“先让测试通过再补实现”的策略直接塑造了代码形态；其三，网络栈为每个 Socket 同时携带 UDP 数据报队列和 TCP backlog，完全放弃流式语义 [net.ref:1][net.ref:2]，将 socket 相关系统调用压缩在 159 行内，这是为通过 libc-test 套件所做的极致权衡。

## 🎯 二、综合评价

### 整体定位

本仓库在 rcore-rust 家族中属于“功能验证原型”而非教学或生产内核。它与 rCore-Tutorial 的典型实现形成鲜明对比：教程用 trait 抽象 Syscall、Buddy 分配器、完善的进程模型；而这里全部退化为 `static mut` + `unsafe` 的最简实现。其目标不是展示清晰的 OS 架构，而是以最小代价撑起一批系统调用，以便通过 LTP 的特定测试组。因此，精力的分配明显偏向文件系统、网络和 syscall 层的横向广度，牺牲了纵向的正确性和可维护性。代码中随处可见的“桩实现”和缺失的并发保护正是这种定位的结果。

### 真正的创新点

与家族基线相比，本仓库在几个方面展示了针对性设计，尽管大多以健壮性为代价。一是文件系统的 ext4 只读加载：[fs.ref:5] 中所述的“惰性从 ext4 加载”策略使得内核无需实现完整的目录索引或日志，却能支撑持久化文件 I/O 测试；它从镜像解析 inode 和 extent 后将数据 copy 到内存 Vec，后续修改只影响内存副本，实际上将 ext4 降格为预填充器。然而，`ext4.rs` 中的递归入栈无深度限制 [fs.ref:8] 和硬编码偏移 [fs.ref:7] 又暴露了健壮性的缺失。

二是内存分配器自制 free-list：内嵌块头 [mm.ref:2] 的设计完全内聚在一块 64 MiB 静态数组上，惰性初始化 [mm.ref:5] 省去了早期裸指针操作，这种极简方案比标准 buddy 更容易在无 crate 依赖时独立运作；但它忽略对齐要求 [mm.ref:10] 和 TOCTOU 风险 [mm.ref:12]，表明设计者仅考虑单 hart 顺序执行且调用者不会请求高对齐的场景。

三是网络栈的“单 Socket 双语义”：[net.ref:1] 中每个 Socket 同时携带 UDP 收报队列和 TCP backlog，connect() 不建立流通道而仅填写 backlog [net.ref:5]，这是一种为通过 socket 相关测试而做的极端压缩，其创新在于用最少代码模拟了套接字 API 的最小可用集，但 TCP 数据路径完全缺失。

### 取舍判断

该仓库明确选择在“横向兼容性”（更多 syscall）和“纵向正确性”（POSIX 语义、并发、性能）之间偏向横向。典型表现为：fork/clone 在无调度器时即时完成子进程；futex/ppoll 不作真阻塞；getdents64 对非根目录返回空列表 [fs.ref:10]；connect() 不返回客户端 fd [net.ref:5]。这些取舍使系统调用行为与 Linux 差异巨大，但换取了极低的代码量，只能应付简单的单线程 C 程序。

另一个根本性的取舍是完全依赖裸金属 virtio-blk 驱动并假设 VA=PA，从而避免了实现页表和 MMU 的复杂度。这使得内核与硬件紧耦合，几乎无移植性，但实现了块设备驱动的快速集成。

### 完成度与不足

从完整 OS 视角看，完成度处于基础水平：无任务调度、信号、IPC，无法运行多进程程序；启动流程仅 global_asm 接力；内存管理无页式虚拟内存；系统调用大量 stub（例如 `getdents64` 在非根目录无数据 [fs.ref:10]），无法通过严格的 POSIX 测试。代码质量方面，多处 unsafe 无安全注释 [mm.ref:13]，静态可变全局变量随意访问 [fs.ref:1][net.ref:2]，平台相关偏移硬编码 [net.ref:4]，且错误返回全部使用裸负数，无错误类型枚举。两个 workspace 成员 khal 和 kunikos 松散组合，未形成清晰的硬件抽象层或驱动模型。但在“以最小代价跑通 LTP 子集”的窄目标下，它成功集成了块设备、ext4 解析、socket 系统调用等基础设施，达到了原型验证的目的。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块实现了一个静态数组上的空闲链表堆分配器（首次适配 + 相邻块合并），通过 `#[global_allocator]` 注册为内核全局分配器。核心抽象是 `FreeList` 结构体，实现 `GlobalAlloc` trait。它解决了单 hart 内核的动态内存分配需求，用嵌入式空闲块头（size/next）管理 64 MiB 堆区。与 rCore-Tutorial 常用 buddy/linked_list_allocator 不同，这是一个自制的、极简的 free-list 实现。

### 1. 核心抽象：嵌入式空闲链表

内存管理的唯一组件是 `kernel/src/heap.rs`，它提供了一个全局堆分配器。核心数据结构为 `FreeList`，内部维护一个按地址排序的单向空闲链表 `head` <sup>[1](#mod-mm-ref-1)</sup>。每个空闲块的前 16 字节被复用为块头：低 8 字节存块大小 `size`，高 8 字节存下一块地址 `next`（0 表示链表尾） <sup>[2](#mod-mm-ref-2)</sup>。分配器不依赖任何外部 crate（如 `buddy_system_allocator`），而是直接实现 `core::alloc::GlobalAlloc` trait，通过 `#[global_allocator]` 属性注册为 Rust 内核的全局分配器 <sup>[3](#mod-mm-ref-3)</sup>。

堆内存本身是编译期静态分配的 64 MiB 零初始化数组 `HEAP`，以 16 字节对齐 <sup>[4](#mod-mm-ref-4)</sup>。`FreeList` 的 `ensure_init` 方法在首次分配时惰性初始化：将整个堆区作为一个大空闲块插入链表 <sup>[5](#mod-mm-ref-5)</sup>。这种惰性初始化避免了在静态构造阶段操作裸指针的麻烦。

### 2. 关键设计取舍

**取舍 1：自旋锁 vs 无锁。** 代码注释明确标注“单 hart 顺序执行”，因此不存在真正的并发竞争 <sup>[6](#mod-mm-ref-6)</sup>。但为实现 `GlobalAlloc`（其方法签名为 `unsafe` 且要求线程安全），分配器仍使用 `AtomicBool` CAS 自旋锁。作者称此举“仅为满足并发安全形态”。此选择放弃了无锁方案可能带来的零开销，换取了与 Rust 分配器 API 的直接兼容。替代方案（如用 `UnsafeCell` 裸指针 + `#[global_allocator]` 标记为单线程）在实践中并不可行，因为 Rust 的 `GlobalAlloc` 要求 `Sync`。

**取舍 2：首次适配 vs 最佳适配。** 分配策略采用简单的首次适配（first-fit）：遍历空闲链表，返回第一个足够大的块 <sup>[7](#mod-mm-ref-7)</sup>。相比最佳适配（best-fit），首次适配在碎片化场景下可能浪费更多内存，但分配路径更短、代码量更小。对于内核启动阶段的低频分配场景，这个取舍是合理的。

**取舍 3：内嵌块头 vs 外置元数据。** 空闲块的元数据（size/next）直接存放在空闲块自身的起始位置 <sup>[2](#mod-mm-ref-2)</sup>，而非使用独立的管理数组。此设计的优势是零额外内存开销，缺点是在块被分配后，其前 16 字节可被用户数据覆盖，回收时需要调用方提供 `Layout` 来恢复大小 <sup>[8](#mod-mm-ref-8)</sup>。这也意味着分配器不记录已分配块的元数据——它完全信任 `dealloc` 传入的 `layout.size()`。

**取舍 4：固定堆大小 vs 可配置。** `HEAP_SIZE` 硬编码为 `0x400_0000`（64 MiB） <sup>[9](#mod-mm-ref-9)</sup>，注释说明是为了容纳 `ftruncate` 测试的大文件及 ELF 加载。相比通过链接脚本或 bootloader 传递堆边界，硬编码简化了初始化流程，但牺牲了跨平台灵活性。

**取舍 5：无对齐保证的 `alloc`。** `alloc` 方法仅将请求大小向上对齐到 `MIN`（16 字节） <sup>[10](#mod-mm-ref-10)</sup>，但完全忽略 `Layout::align()`。这意味着如果内核某处请求大于 16 字节对齐的分配（如某些 SIMD 类型要求 32 字节对齐），此分配器无法满足。注释称“满足 align<=16 的全部内核分配”，表明作者知晓此限制并认为当前内核无需更大对齐。

### 3. 跨模块协同

作为 `#[global_allocator]`，`ALLOC` 静态变量被 Rust 编译器插入到所有堆分配代码路径中。内核中任何使用 `Box`、`Vec`、`String`、`Arc` 等智能指针的模块，其底层内存均由此分配器提供 <sup>[3](#mod-mm-ref-3)</sup>。

`dealloc` 方法依赖调用方传入正确的 `Layout` 来确定释放块的大小 <sup>[8](#mod-mm-ref-8)</sup>。这意味着上层代码（如 `Vec` 的 `Drop` 实现）必须准确记住原始分配的 size 和 align——这是 Rust 标准库的契约，分配器本身无条件信任。若未来引入自定义分配器包装层，此处契约可能被破坏。

分配器的 `insert` 合并逻辑是系统正常运行的关键：如果只分配不合并，64 MiB 堆会在频繁分配/释放后迅速碎片化 <sup>[11](#mod-mm-ref-11)</sup>。合并的正确性依赖于空闲链表始终按地址排序的不变式（invariant），该不变式由 `insert` 的插入逻辑和 `alloc` 的分裂逻辑共同维护。

### 4. 边角细节与不足

**<sup>[12](#mod-mm-ref-12)</sup> `ensure_init` 的 TOCTOU 隐患：** 第 46-50 行的初始化检查是非原子的“检查-执行”序列。在单 hart 环境下安全，但若未来扩展为多核，两个 hart 可能同时通过 `if !*self.init.get()` 检查，导致双重初始化。正确的多核版本需要 `AtomicBool::compare_exchange` 或 `Once` 原语。

**<sup>[13](#mod-mm-ref-13)</sup> 所有 `unsafe` 块缺乏 SAFETY 注释：** 文件中包含 6 处 `unsafe` 块（`rd`/`wr` 函数体、`ensure_init`、`insert`、`alloc`、`dealloc`），没有一处附带 `// SAFETY:` 说明安全契约。例如 `rd` 函数（第 16 行）接收任意 `addr` 和 `off`，直接解引用裸指针，调用方需保证地址落在堆区内，但这一前置条件未被文档化。

**<sup>[14](#mod-mm-ref-14)</sup> `alloc` 返回未初始化内存：** 分配时未对返回的块做零初始化。虽然内核中很多分配会立即被写入，但某些场景（如 `Vec::reserve` 后再 `set_len`）可能暴露未初始化数据，构成信息泄露风险。

**<sup>[15](#mod-mm-ref-15)</sup> `dealloc` 对 null 指针的静默处理：** 第 119-121 行，`dealloc` 在 `ptr.is_null()` 时直接返回，这与 POSIX `free(NULL)` 的语义一致，但 Rust 的 `GlobalAlloc::dealloc` 文档未明确承诺此行为。如果上层代码传入非堆区的野指针（非 null），`insert` 会将其插入空闲链表，导致堆元数据损坏且无法检测。

**<sup>[16](#mod-mm-ref-16)</sup> 空闲链表遍历无环检测：** `alloc` 的 while 循环（第 93 行）和 `insert` 的 while 循环（第 57 行）均无条件遍历链表。若堆元数据因 bug 或内存损坏形成环，将导致死循环。生产级分配器通常会在遍历中设置上限或使用哨兵值。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 全内存文件系统, 以 `Node` 结构存储扁平化路径名与数据, 6 种 `Fd` 后端统一文件/管道/设备/socket. ext4 层提供只读磁盘加载, 写入仅改内存副本. 面向单进程顺序执行的 LTP 测试框架, 通过 `reset()` 清空全局状态, 无锁无并发控制.

### 1. 核心抽象：全内存节点与多后端 Fd 枚举
本模块的核心是一对 `static mut` 全局容器：`FILES: Vec<Node>` 持有所有已打开文件的内存副本, `FDS: [Option<Fd>; 64]` 为进程文件描述符表 <sup>[1](#mod-fs-ref-1)</sup><sup>[2](#mod-fs-ref-2)</sup>。`Node` 以完整路径名字符串作为唯一标识, 目录节点额外维护 `names: Vec<String>` 用于 `getdents64`, 但模块内部不维护树状层级——`node_idx()` 在全 `FILES` 向量上按路径名线性搜索 <sup>[3](#mod-fs-ref-3)</sup>, 放弃了层级查找的对数效率以换取极简性。

`Fd` 枚举统一了六种后端：普通文件(带读写位置)、管道读写端、控制台、`/dev/null`、`/dev/zero` 和 socket <sup>[4](#mod-fs-ref-4)</sup>。这种「胖枚举」设计让 `read()`/`write_file()` 可以在单个 `match` 中派发所有后端, 避免了虚函数表或 trait object 的开销, 但每增一种新后端都需修改所有 I/O 函数。

### 2. 关键设计取舍：单进程顺序模型的激进简化
整个模块建立在「单进程顺序执行, 每程序 reset()」的强假设上 <sup>[5](#mod-fs-ref-5)</sup>。所有全局状态通过 `static mut` + `unsafe` 裸指针访问, 无 `Mutex` 或 `RwLock`。`reset()` 在每次测试程序启动前清空 fd 表、内存文件、工作目录、网络状态和 PID 表 <sup>[6](#mod-fs-ref-6)</sup>, 将系统恢复至干净初始态。这彻底消除了并发控制复杂度, 但任何多线程场景都会导致数据竞争。

管道实现同样体现这一取舍：`PipeBuf` 仅为一个 `Vec<u8>` 加读偏移 `rpos`, 无阻塞语义——读端数据耗尽时直接返回 0（EOF）, 依赖顺序执行保证写端先于读端完成 <sup>[7](#mod-fs-ref-7)</sup>。这与 POSIX 管道的阻塞/非阻塞语义完全不同, 是面向 LTP 框架的务实妥协。

ext4 层被设计为只读：`mount()` 解析超级块并缓存块组描述符表, `load()` 按名从磁盘读取文件到 `Vec<u8>` <sup>[8](#mod-fs-ref-8)</sup>。写入仅修改内存中的 `Node.data`, 永不回写磁盘。`extent_runs()` 用显式栈遍历多级 extent 树 <sup>[9](#mod-fs-ref-9)</sup>, 能正确读取大文件和稀疏文件, 比仅支持内联 extent 的实现更完整。

### 3. 跨模块协同：从 virtio-blk 到 /proc 合成文件
首次 `openat()` 时, `node_idx()` 调用 `ext4::global().load(name)`, 后者经 `lookup_root()` 在根目录 inode 2 中搜索, 再通过 `read_file()` 批量读取数据块 <sup>[10](#mod-fs-ref-10)</sup>。`read_file()` 将连续物理块聚合为单次 `virtio_blk::read()` 请求（上限 `DATA_MAX/BS` 块）, 把 1 MiB ELF 文件的 256 次磁盘往返压缩到约 16 次 <sup>[11](#mod-fs-ref-11)</sup>。

`/proc` 伪文件由 `synthetic()` 以硬编码字节串模拟, 覆盖 `proc/meminfo`、`proc/cpuinfo`（按 `khal::ARCH` 区分架构）、`proc/self/stat` 等 LTP 必需节点 <sup>[12](#mod-fs-ref-12)</sup>。`normalize_proc()` 将 `/proc/<数字pid>/X` 归一化到 `/proc/self/X`, 用字符串替换而非真实进程视图 <sup>[13](#mod-fs-ref-13)</sup>。

Socket 集成通过 `install_socket()` 桥接 `crate::net::SOCKETS` <sup>[14](#mod-fs-ref-14)</sup>; fork 的 fd 表隔离通过 `FD_SAVE` 栈实现, `fork_push()` 克隆快照, `fork_pop()` 恢复 <sup>[15](#mod-fs-ref-15)</sup>。

### 4. 边角细节与不足
1. **悬垂引用风险** <sup>[16](#mod-fs-ref-16)</sup>：`file_bytes()` 返回 `Option<&'static [u8]>`, 实际指向 `FILES` 内 `Vec<u8>` 数据；后续写入可能触发 `push()` 重分配, 使已返回的引用悬垂。标注 `'static` 是事实上的生命周期谎言。
2. **硬编码结构偏移** <sup>[17](#mod-fs-ref-17)</sup>：`statx()` 和 `fstat()` 以硬编码偏移量手动布局 `struct statx`/`struct stat`（如 `buf+0x1c` 写 `stx_mode`）, 平台 ABI 变动将使这些偏移全部失效且无编译期检查。
3. **路径规范化不一致** <sup>[18](#mod-fs-ref-18)</sup>：`chdir()` 支持 `.` / `..` 规范化, 但 `openat()` 仅做 `trim_start_matches` 简单修剪, 多斜杠、末尾点号等边界情况处理不完整。
4. **错误码裸数字** <sup>[19](#mod-fs-ref-19)</sup>：返回 `-2`、`-9`、`-24` 等裸负数, 无命名常量或错误类型枚举, 可读性和可维护性差。
5. **伪文件数据静态不变** <sup>[12](#mod-fs-ref-12)</sup>：`proc/uptime` 固定返回 `100.00`, `proc/self/stat` 返回硬编码的假进程数据, 依赖准确 /proc 信息的程序将得到错误结果。
6. **extent 树无深度限制** <sup>[9](#mod-fs-ref-9)</sup>：`extent_runs()` 对 depth>0 的索引节点递归入栈, 无深度或循环引用检查, 恶意镜像可导致栈溢出。
7. **目录节点未更新** <sup>[20](#mod-fs-ref-20)</sup>：`mkdir` 创建的目录节点 `names` 初始化为空且无更新机制, `getdents64` 在非根目录上永远返回空列表。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 内核 net 模块实现了一个极简的单进程内存回环网络栈，使用全局 Socket 向量管理 UDP 数据报收发和 TCP 监听/连接配对。无硬件依赖，不与执行模型耦合，仅为满足 libc-test 的 socket 类测试及 iperf/netperf 回环基准而存在。相比标准 ArceOS 或 rCore 的网络子系统，本模块砍掉了完整的 TCP 状态机、网卡驱动层和异步 I/O，退化成端口到端口的队列转发模型。

### 1. 核心抽象：全局 Socket 向量与两种投递语义

模块的唯一核心数据结构是 `Socket`<sup>[1](#mod-net-ref-1)</sup>，它不区分 UDP/TCP 类型，而是通过字段组合模拟两种传输语义：`.dgrams: Vec<Vec<u8>>` 作为 UDP 收包队列，`.backlog: Vec<usize>` 存储 TCP 监听者待 accept 的对端 socket 下标。所有 socket 实例集中在一个全局 `static mut SOCKETS: Vec<Socket>` 中<sup>[1](#mod-net-ref-1)</sup>，通过裸指针 cast 提供访问函数 `socks()`<sup>[2](#mod-net-ref-2)</sup>，这是典型的「先跑通功能，后考虑安全」的早期内核写法。

此抽象放弃了标准 socket 的类型安全（一个 Socket 同时携带 UDP 队列和 TCP backlog 字段），但换取了实现的极度压缩——整个模块仅 159 行。端口分配从 `NEXT_PORT = 20000` 开始自增<sup>[1](#mod-net-ref-1)</sup>，避免与知名端口冲突，但没有任何端口重用或冲突检测逻辑。

### 2. 关键设计取舍：牺牲 POSIX 语义换取最小可用集

本模块在多个维度做了激进简化，每条都值得评审关注：

- **TCP 非流式**：`connect()` 不建立真正的字节流通道，只是在监听 socket 的 `backlog` 中填入一个新创建的 socket 下标<sup>[5](#mod-net-ref-5)</sup>。此「连接」本质是索引配对，无序列号、无窗口、无重传，因此不存在 `send()`/`recv()`——TCP 数据路径完全缺失，只能跑三次握手后的 close 测试。
- **connect 的隐式 socket 创建**：标准 POSIX 要求先 `socket()` 后 `connect()`，但本模块 `connect(addr)` 直接新建 socket 并入 backlog，调用者无从获取该 socket 的 fd<sup>[5](#mod-net-ref-5)</sup>。这意味着客户端发送数据的能力完全依赖于外部 syscall 层的额外适配代码（如先调 `socket()` 拿 fd，再调 `connect()` 配对），但 `connect` 本身的签名不支持传入 fd，存在语义断层。
- **UDP 发送无需 fd**：`sendto()` 仅凭目标端口寻址<sup>[7](#mod-net-ref-7)</sup>，不校验发送方是否 bind 了该端口。这模拟了 UDP 无连接特性，但丢失了源端口信息，也无法支持多 socket 绑定同一端口的场景。
- **同步阻塞假象**：注释写「设了 SO_RCVTIMEO，同步模型下数据报已先到」<sup>[8](#mod-net-ref-8)</sup>，暗示上层通过某种忙等/超时机制在调用 `recvfrom` 前确保数据就绪，但模块本身无任何等待/调度原语，`EAGAIN` 只是一个硬错误码，真正的同步语义由 syscall 层或测试框架外部驱动。
- **unsafe 遍地**：从 `static mut` 的直接访问<sup>[2](#mod-net-ref-2)</sup>到 sockaddr_in 的裸指针构造<sup>[4](#mod-net-ref-4)</sup><sup>[6](#mod-net-ref-6)</sup>，安全契约完全依靠「单线程执行」这一隐含假设。如果未来引入中断/多核，所有 `unsafe` 块都是数据竞争和 UB 的入口。

### 3. 跨模块协同：与文件描述符系统的寄生关系

网络模块自身不维护 fd 表，而是通过 `crate::fs::install_socket()` 和 `crate::fs::socket_of()` 寄生在 fs 模块之上<sup>[3](#mod-net-ref-3)</sup><sup>[9](#mod-net-ref-9)</sup>。Socket 在全局向量中的下标成为两个模块之间的「密钥」：`socket()` 创建 socket 后将其下标交给 fs 分配 fd；后续所有操作（`bind`、`listen`、`accept`、`recvfrom`）都先通过 `socket_of(fd)` 反查下标再索引 `SOCKETS`<sup>[9](#mod-net-ref-9)</sup>。

这种设计将 socket 的生命周期管理交给了 fs 模块，但 `reset()` 函数的注释明确指出需要「与 fs::reset 同步调用」<sup>[3](#mod-net-ref-3)</sup>，说明程序重启时两个全局状态必须同时清零。若未来某个路径调了 `fs::reset()` 而忘记调 `net::reset()`，残留的 socket 下标将导致 fd 表指向已释放或错位的 socket 数据。

模块还暴露了 `cloexec(sidx)` 和 `nonblock(sidx)` 两个查询接口，供 fs 模块在 `dup`/`fork`/`exec` 时判断是否关闭 socket 或设置非阻塞标志——这些是典型的 Unix 文件描述符语义，说明本模块虽简，但仍试图满足 libc-test 的基础要求。

### 4. 边角细节与不足

**<sup>[1](#mod-net-ref-1)</sup> 全局可变状态无并发防护**：`static mut SOCKETS` 与 `static mut NEXT_PORT` 是典型的「先让它跑」写法。即便在单线程内核中，若中断处理函数间接触发 socket 操作（如信号驱动的 I/O），也会导致重入与数据竞争。后续迭代至少应改为 `Mutex<Vec<Socket>>`。

**<sup>[4](#mod-net-ref-4)</sup> sockaddr_in 解析的硬件假设**：`parse_port()` 以固定偏移 +2/+3 读取大端序端口，`getsockname()` 写入 `0x0100_007f` 作为 127.0.0.1 的网络序表示<sup>[6](#mod-net-ref-6)</sup>。后者依赖小端主机将 `u32` 的字节序反转来「正好」得到正确的网络序字节排列（7f 00 00 01）。代码注释意识到了这一点<sup>[6](#mod-net-ref-6)</sup>，但若迁移到大端架构（如某些 MIPS），IP 地址将被错误编码为 1.0.0.127。

**<sup>[5](#mod-net-ref-5)</sup> connect 语义残缺**：`connect()` 新建 socket 但不返回 fd，且强制将 `cloexec` 和 `nonblock` 设为 false。若调用者期望非阻塞 connect，此实现直接忽略。此外，它不检查目标端口是否已被绑定（只查 `listening`），这意味着可以向一个仅 bind 未 listen 的端口「连接」，与标准行为不符。

**<sup>[7](#mod-net-ref-7)</sup> sendto 缺少 fd 参数与源端口绑定**：发送数据报时无法指定源端口，且不检查是否有 socket 绑定到目标端口之外的其他条件（如目标 socket 是否允许接收）。如果两个不同 socket 绑定同一端口（理论上不应发生，但无代码阻止），投递目标将是不确定的第一个匹配项。

**<sup>[8](#mod-net-ref-8)</sup> recvfrom 的无序交付假设**：UDP 数据报以 `Vec<u8>` 队列存储，取走即删，无 peek 接口，无 MSG_TRUNC 支持。注释中的「同步模型下数据报已先到」暴露了外部依赖——若数据未到达，返回 -11（EAGAIN），但 libc-test 可能期望真正的阻塞直到数据就绪。

**额外隐患**：
- 无 `close()`/`shutdown()`，socket 泄露无可避免。
- `bind()` 中端口 0 自动分配逻辑与 `NEXT_PORT` 自增存在竞态（虽在当前单线程模型下无害）。
- 无任何错误注入或日志埋点，调试完全依赖 libc-test 的 pass/fail 反馈。
- 所有 `unsafe` 块均无 `// SAFETY:` 注释，安全契约不可追溯。

总体而言，该模块是一个「刚好够用」的 libc-test 适配层，距离真正的网络栈还有 TCP 流控、拥塞控制、设备抽象、并发模型四道鸿沟需要跨越。
