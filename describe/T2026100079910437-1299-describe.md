## 📖 一、总览

该仓库在 ArceOS 模块化内核基础上，通过 starry-api/starry-core/starry 三层 workspace 结构，构建了一个面向 Linux 应用兼容的宏内核框架。其核心定位并非从零研发新内核，而是以 “Linux syscall 兼容层” 为目标，在 ArceOS 的 axhal/axtask/axmm/axnet 等基底之上，注入进程模型、信号、完整 VFS、SysV IPC 和 epoll 等典型 POSIX 子系统，从而支持运行动态链接的 Linux 用户程序。启动流程中同时保留标准路径与实验性的 vsched2 调度器（[boot.ref:1]），反映出该仓库兼具实用与探索的双重性质。

最值得注意的三点设计抉择：第一，项目在系统调用层采用巨型 match 分发而放弃可扩展性，换取编译期穷尽检查与零开销静态调用（[syscall.ref:?]），这与 Linux 的动态表方案形成鲜明对比，是 Rust 安全哲学在内核上的典型投影；第二，信号模块存在两套并行的实现——分别用于核心调度器与遗留执行器（[signal.ref:5]），这种双重架构虽暴露了演进痕迹，却意外提供了观察 “从 hand-rolled 执行器迁移至模块化任务系统” 的变迁窗口；第三，文件系统与内存管理模块通过 Backend 多态将设备 mmap、共享内存、匿名映射等异构后端统一到同一 VMA 框架下（[mm.ref:3][fs.ref:3]），展现了比基线更成熟的内存-文件协同设计。然而，全局单锁 FsContext、IPC 管理器等设计也说明并发可扩展性尚未成为首要关注点。

## 🎯 二、综合评价

### 整体定位

本仓库不是传统意义上的全新内核，而是在 ArceOS 模块化操作系统框架上构建的 Linux 兼容运行时。它利用 ArceOS 已有的硬件抽象（axhal）、任务调度（axtask）、网络协议栈（axnet）等底层组件，通过 starry-api 层封装出近 200 个 Linux 系统调用，并引入进程地址空间、信号、完整文件系统、SysV IPC 等典型 POSIX 语义，最终以单个二进制（starry）形式运行用户程序。这种设计使它能以较小成本获得较为广泛的 Linux 应用兼容性，同时保留了 ArceOS 组件化带来的可定制性。但该仓库仍高度依赖 ArceOS 生态，驱动框架直接沿用基线，且启动流程中保留着实验性调度器 vsched2 的切换开关，显示其定位介于“实用”与“探索”之间。

### 真正的创新点

尽管多数模块是对 Linux 系统调用的“移植实现”，但本仓库在若干设计点上体现了超越机械复刻的思考：

1. **内存管理 Backend 多态与统一映射**：`sys_mmap` 并未为不同映射类型编写独立路径，而是通过 `Backend` 枚举将匿名页、文件、共享内存、设备物理内存等映射后端统一为同一接口（[mm.ref:3]），再由 `AddrSpace` 的内核缺页处理自动适配。这种设计使 VMA 管理层保持简洁，新增一种映射类型只需扩展 Backend 变体，符合开闭原则。

2. **信号系统的双重实现与架构演进**：仓库内同时存在两套信号处理路径——一套基于 `starry_core::task` 的现代化异步模型，另一套位于 `core/executor` 的遗留执行器中（[signal.ref:5]）。虽然这带来了代码重复和不一致性风险，但它忠实记录了该项目从手写调度器向 ArceOS 标准任务模型迁移的过程，为后续合并或淘汰提供了参照。

3. **文件系统与套接字的统一 `FileLike` 抽象**：无论是磁盘文件、管道、eventfd 还是套接字，均通过实现 `FileLike` trait 融入文件描述符表（[net.ref:2]），使得 `select`/`poll`/`epoll` 能透明地作用于所有 I/O 资源。自定义 epoll 同时支持 Level/Edge/Oneshot 三种触发模式，在同类 Rust OS 中较为少见。

4. **架构感知的编译期适配**：系统调用分发及陷阱处理大量使用 `#[cfg(target_arch)]` 条件编译，配合 RISC-V 与 x86_64 两套 TrapFrame 定义，在编译期消除死代码而非运行时判断，既保持了性能又避免了链接臃肿。虽然当前 vsched2 路径仍硬编码 RISC-V 指令，但整体架构已预留扩展空间。

### 取舍判断

该仓库做出一系列务实的工程取舍：它将**系统调用广度**置于首位，197 个（实际 195）系统调用覆盖了进程管理、文件 I/O、网络、IPC 和信号等主要领域，但为了快速达成这一目标，**部分实现停留在“功能演示”深度**——消息队列阻塞收发直接标 TODO（[ipc.ref:5]），信号 Stop/Cont 默认动作触发 `unimplemented!()`（[signal.ref:1]），`madvise`/`mlock` 等为空操作。并发模型选择了**大锁简化**：文件系统全局 `FS_CONTEXT` 锁、IPC 全局管理器单 Mutex 保护（[ipc.ref:4]），牺牲多核扩展性换取开发敏捷。安全问题则采用**局部安全封装配合全局 unsafe**的策略：`UserPtr` 对外暴露安全接口，但内部实现仍包含大量未经 SAFETY 注释的裸指针操作（[boot.ref:6]），且错误路径普遍以 `panic` 处理，这在原型阶段可接受，但距离生产环境尚有距离。这些取舍共同决定了当前版本适合作为研究平台或中等复杂度的应用沙箱，而非高可靠性服务器系统。

### 完成度与不足

仓库在“跑起来”方面已达到一定成熟度：能够加载 ELF（含动态链接）、处理缺页、调度多线程、进行文件读写和网络通信。但从高质量实现角度看，明显短板包括：

- **系统调用语义缺失**：除前述阻塞 IPC 和作业控制信号外，`memfd_create` 误用真实文件系统、`membarrier` 无 IPI 支持、`stat()` 对套接字返回硬编码值等，将导致依赖这些行为的应用出现不可预料错误。
- **健壮性不足**：多处 `unwrap()` 和 `panic!` 可被用户态输入触发（如无效 shmid、套接字选项查询失败），构成拒绝服务漏洞（[ipc.ref:?]）。内核应返回 `-EINVAL` 而非崩溃。
- **安全契约缺失**：`core::mem::zeroed()` 用于非平凡类型可能造成未定义行为（[boot.ref:9]）；信号处理直接操作用户栈无地址验证（[signal.ref:3]）；unsafe 块普遍缺少 SAFETY 注释，使得外部审查困难。
- **可维护性问题**：系统调用中央 `match` 分发器长达 640 行，增删调用需修改同一文件；遗留与现行信号实现并存增加维护负担；硬编码的架构指令（如 SBI ecall）破坏可移植性（[boot.ref:8]）。

**评分依据**：完成度 2 分——核心子系统可用，但大量细节缺失或错误，尚不能稳定运行复杂应用；创新性 2 分——部分设计有亮点，但整体框架未超出现有 OS 研究实践；代码质量 1 分——多处不安全实践和未处理错误，生产级代码标准远未达到。

### 1. 启动路径的双轨制：编译期开关的取舍

`main()` 中通过 `const USE_VSCHED2: bool = false` <sup>[1](#mod-boot-ref-1)</sup> 控制两条截然不同的 init 启动路径。这不是 Cargo feature 或 `#[cfg]` 条件编译，而是一个需要手动修改源码才能切换的布尔常量。这种设计的代价是：两条路径无法在同一构建产物中共存，测试 vsched2 路径必须重新编译；好处则是避免了两套调度器共存带来的符号冲突与链接膨胀。从代码注释 `// axlog::ax_println!("main: USE_VSCHED2={} args={:?}", ...)` 被注释掉 <sup>[2](#mod-boot-ref-2)</sup> 可以看出，开发者曾在两条路径间频繁切换调试，但最终将默认值固定为 `false`，说明 vsched2 仍处于实验阶段，尚未达到可替代原有 `entry::run_initproc` 的成熟度。

当 `USE_VSCHED2` 为 `false` 时，`entry::run_initproc(&args, envs)` 被调用。该函数位于 `mod entry`（`src/entry.rs`）中，其内部实现虽不可见，但从调用签名推断，它封装了完整的「创建用户地址空间→加载 ELF→构造初始任务→进入调度器」流程。这符合 ArceOS 一贯的「单入口函数完成所有初始化」风格。

### 2. 内核初始化序列：`starry_api::init` 与 vDSO 的前置依赖

`main()` 最先调用 `starry_api::init()` 和 `vdso::vdso_init()` <sup>[3](#mod-boot-ref-3)</sup>，而后才构造命令行参数。这一顺序确保了文件系统、内存分配器、中断处理等内核子系统先于任何用户态操作就绪。boot facts 显示 IRQ handler 注册在 `arceos/modules/axhal/src/irq.rs:35`，页故障 handler 在 `api/src/mm.rs:237`，这些正是 `starry_api::init` 内部完成的平台初始化工作的冰山一角。vDSO 初始化紧随其后，为后续用户态提供虚拟动态共享对象的映射——尽管 init 进程 /bin/sh 是否实际使用 vDSO 取决于其自身链接，但内核侧必须在任何用户代码执行前建立映射。

值得注意的是，`CMDLINE` 通过 `include_str!("init.sh")` <sup>[1](#mod-boot-ref-1)</sup> 在编译期将 shell 脚本嵌入二进制。这意味着 init 进程的命令行参数在编译时即已固定，无法通过 bootloader 或外部配置修改。对于 database 类型的系统软件，这可能是刻意为之——生产环境的 init 脚本通常不应被随意篡改；但这也牺牲了现场调试的灵活性。

### 3. vsched2 路径的初始任务构造：手工装配与 unsafe 契约

`create_vsched_init_task` 函数 <sup>[4](#mod-boot-ref-4)</sup> 是 vsched2 路径的核心，展示了一条「全手动」的用户任务构造流水线：

1. **地址空间**：通过 `new_user_aspace_empty()` 创建空用户地址空间，随后调用 `copy_from_kernel()` 将内核镜像映射到用户态（典型用法是让用户态通过 vDSO 或固定映射访问内核数据）。
2. **ELF 加载**：通过 `FS_CONTEXT.lock().resolve()` 解析可执行文件路径，调用 `load_user_app()` 加载 ELF 到用户地址空间，得到入口虚拟地址与用户栈顶。
3. **任务构造**：`new_user_task()` 创建内核任务对象，`Process::new_init()` 建立进程模型，`N_TTY.bind_to()` 绑定终端，`ProcessData::new()` 组装进程元数据。
4. **调度器注册**：`axtask::into_ref()` 将任务转为引用计数指针，`add_task_to_table()` 注册到全局任务表，`register_task()` 向 vsched2 调度器注册。
5. **TrapFrame 装配**：手动构造 `UserTrapFrame`，将 `sepc` 设为入口地址，`regs.sp` 设为用户栈顶，`regs.ra` 设为 0——注释解释「ra=0 so a stray 'ret' crashes cleanly instead of jumping to a kernel address」<sup>[5](#mod-boot-ref-5)</sup>，这是一种防御性编程：若用户程序意外执行 `ret`，将触发地址 0 处的页故障而非跳转到内核地址。
6. **元数据写入**：通过裸指针将栈指针、TrapFrame 指针、页表根、地址空间指针以 `Ordering::Release` 写入 vsched2 任务结构体的原子字段 <sup>[6](#mod-boot-ref-6)</sup>。

整段代码充斥着 `unsafe` 块，且无一附带 `// SAFETY:` 注释，安全契约完全依赖程序员对整体生命周期的隐式理解。

### 4. 跨模块协同：文件系统、内存管理与调度器的交汇

`create_vsched_init_task` 是内核多个子系统的交汇点：

- **文件系统**：`FS_CONTEXT` 全局锁被获取以解析路径，该锁在 `axfs` crate 中定义，是 ArceOS 文件系统层的核心同步原语。
- **内存管理**：`new_user_aspace_empty()`、`copy_from_kernel()`、`load_user_app()` 来自 `starry_core::mm`，操作页表与地址空间。其中 `proc_data.aspace` 被包裹在 `Arc<Mutex<AddrSpace>>` 中，既用于页表操作，又通过裸指针逃逸给 vsched2 调度器用于后续 `copy_mappings_from`——注释中提及的 `aspace_mutex_ptr` <sup>[7](#mod-boot-ref-7)</sup> 正是为跨调度上下文共享地址空间预留的。
- **任务管理**：`axtask` crate 提供 `AxTaskExt` trait，`starry_core::task` 提供 `Thread` 和 `ProcessData`。`AxTaskExt::from_impl(thr)` 的 unsafe 源自 trait 对象的类型擦除：`Thread` 的具体类型在运行时被抹去，调度器通过 trait 对象操作任务。
- **调度器**：`starry_core::vsched` 模块接收裸指针并存储于原子变量，这种设计避免了锁竞争，但将生命周期管理的全部责任转移给调用方。

### 5. 边角细节与不足

1. **架构绑定风险**：`main()` 末尾的 SBI shutdown ecall <sup>[8](#mod-boot-ref-8)</sup> 硬编码了 RISC-V SBI SRST 扩展的指令序列 (`li a7, 0x53525354` = "SRST")，且未用 `#[cfg(target_arch = "riscv64")]` 守卫。若该项目未来支持其他架构（如 aarch64），此处将直接编译失败。这与 `axhal` crate 通常提供的架构抽象层做法相悖。

2. **unsafe 泛滥且无 SAFETY 注释**：`create_vsched_init_task` 中至少 6 处 `unsafe` 块 <sup>[6](#mod-boot-ref-6)</sup><sup>[4](#mod-boot-ref-4)</sup>，包括裸指针解引用、`core::mem::zeroed()`、trait object 构造等，没有一处写明安全契约。这是显著的技术债务，尤其 `&*vti` 直接解引用 `register_task` 返回的裸指针而不检查空指针。

3. **`zeroed()` 的 UB 隐忧**：`UserTrapFrame.regs` 通过 `core::mem::zeroed()` 初始化 <sup>[9](#mod-boot-ref-9)</sup>，随后被选择性覆盖。若该寄存器结构体任何字段具有非零有效性不变式（如 Rust 的 `NonZeroUsize` 或某个枚举的判别式），此行即为未定义行为。虽然目前 `regs` 很可能是 `#[repr(C)]` 的普通整数数组，但缺乏文档约束。

4. **双重加锁与死代码**：`proc_data.aspace.lock()` 在相邻几行内被调用两次 <sup>[10](#mod-boot-ref-10)</sup>，一次获取 AddrSpace 裸指针，一次获取页表根。虽然同线程递归加锁在 `Mutex` 上会死锁（若为非递归锁），此处 ArceOS 的 `Mutex` 很可能基于自旋锁实现且不允许递归，因此第二次 `lock()` 要么 deadlock（如果锁已被持有），要么此处实际持有锁的 guard 已提前 drop（Rust 的临时生命周期）。第 78 行的 `guard` 在块末尾 drop，因此第 86 行的 `lock()` 是安全的，但这种紧邻的二次加锁是低效的，且极易在重构时引入死锁。

5. **未使用的变量**：`entry_ra` 和 `entry_sp` 在读取后从未使用 <sup>[11](#mod-boot-ref-11)</sup>，可能是调试残留或预留接口，但编译器会发出 warning（除非被 `_` 前缀抑制），在生产代码中属于工程腐化信号。

6. **错误处理一律 panic**：从地址空间创建到 stdio 绑定，所有 `expect`/`unwrap` 均直接 panic <sup>[12](#mod-boot-ref-12)</sup>。对于 database 场景，若 init 进程因磁盘错误无法加载，更合理的做法可能是进入救援 shell 或输出诊断信息后停机，而非直接 panic 刷屏。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `axmm::AddrSpace` + `Backend` 为核心抽象，管理用户态虚拟地址空间的生命周期：创建空地址空间、加载 ELF（含动态链接器与 shebang 解释）、映射栈/堆/vDSO、按需缺页填充。上层通过 `UserPtr`/`UserConstPtr` 安全封装内核访问用户内存，底层缺页 handler 与 `access_user_memory` 标志位协同实现懒分配。与 ArceOS 基线相比，新增了 LRU ELF 缓存、vDSO 预留间隙填充、设备 mmap 的 Physical/ReadOnly/Cache 三态分发等特性。

### 1. 核心抽象：AddrSpace + Backend + UserPtr 三层模型

用户态内存管理构建在三层抽象之上。最底层是 `axmm::AddrSpace`，封装了页表操作（map/unmap/protect/populate）和 VMA 区域查询 <sup>[1](#mod-mm-ref-1)</sup>。每个进程的 `proc_data.aspace` 持有一个 `Mutex<AddrSpace>`，所有地址空间操作均需获取该锁 <sup>[2](#mod-mm-ref-2)</sup>。

中间层是 `Backend` 枚举，决定映射的物理页来源：`Backend::new_alloc` 用于匿名映射（堆、栈、mmap 匿名），`Backend::new_cow` 实现文件私有映射的写时复制，`Backend::new_file`/`new_shared` 处理共享文件/匿名映射，`Backend::new_linear` 将设备物理内存线性映射到用户空间 <sup>[3](#mod-mm-ref-3)</sup>。这种多态 Backend 使得 `sys_mmap` 仅需根据文件类型和标志选择对应 Backend，无需在地址空间层区分映射来源。

最上层是 `UserPtr<T>` / `UserConstPtr<T>`，它们是对用户态裸指针的安全包装：在转换为 Rust 引用前，先通过 `check_region` 验证地址范围属于合法 VMA 且权限匹配，并调用 `populate_area` 触发懒分配物理页 <sup>[4](#mod-mm-ref-4)</sup>。这种设计使得系统调用实现中可以使用几乎正常的 Rust 引用语法，而无需在每处手动处理缺页。

### 2. 关键设计取舍：懒分配与 access_user_memory 标志

该系统采用“按需分配”策略：`load_user_app` 创建栈/堆映射时只建立 VMA 而不立即分配物理页（堆除外，其 `populate` 参数为 `true`）<sup>[5](#mod-mm-ref-5)</sup>。物理页的实际分配发生在首次访问时的缺页处理中。

缺页 handler 位于 `api/src/mm.rs`，通过 `#[register_trap_handler(PAGE_FAULT)]` 注册 <sup>[6](#mod-mm-ref-6)</sup>。它不无条件处理所有缺页，而是检查当前线程的 `is_accessing_user_memory()` 标志。该标志由 `core/src/mm.rs` 中的 `access_user_memory` 函数以 RAII 方式设置/清除 <sup>[7](#mod-mm-ref-7)</sup>。这意味着：
- **内核访问用户内存时**（如 `Vm::read` 内部调用 `user_copy`），标志被设置，缺页 handler 会调用 `aspace.handle_page_fault` 尝试填充页面。
- **用户态代码自身触发缺页时**，标志为 false，handler 返回 false，将控制权交还给 trap 分发层（通常向用户进程发送 SIGSEGV）。

这一设计的取舍是：**简单性优于完备性**。一个布尔标志无法区分“内核访问用户内存时的正常懒分配缺页”与“内核代码中的真正 bug 导致访问了未映射地址”。如果内核代码在 `access_user_memory` 范围内意外访问了一个未映射内核地址，缺页 handler 会尝试在用户地址空间中处理，最坏情况下可能静默失败。但标志方案的实现复杂度远低于维护一个“当前正在访问的用户地址列表”。

### 3. 跨模块协同：从 ELF 加载到首次缺页的完整路径

ELF 加载过程展示了多个子系统的紧密协同 <sup>[8](#mod-mm-ref-8)</sup>：

1. **文件系统 → ELF 解析**：`ElfLoader::load` 通过 `FS_CONTEXT.lock().resolve(path)` 获取文件位置，创建 `CachedFile` 缓存文件内容。`ElfCacheEntry` 使用 `ouroboros` 自引用结构将文件数据和解析后的 ELF headers 绑定在一起，支持零拷贝解析 <sup>[9](#mod-mm-ref-9)</sup>。
2. **ELF → 地址空间**：`map_elf` 遍历 PT_LOAD 段，为每段创建 `Backend::new_cow`（文件私有映射），确保写操作不会修改页缓存中的原始文件数据 <sup>[10](#mod-mm-ref-10)</sup>。段偏移未对齐到 4K 时由 Backend 内部处理，而非在映射层做 padding。
3. **动态链接器处理**：如果 ELF 包含 `PT_INTERP` 段，`ElfLoader` 会额外加载动态链接器到 `USER_INTERP_BASE`，入口点重定向到链接器 <sup>[11](#mod-mm-ref-11)</sup>。
4. **vDSO 集成**：`map_so(uspace_ptr)` 将 vDSO ELF 直接装入地址空间，随后通过 gap-filling 逻辑用 `Backend::new_alloc` 填满 `VDSO_SIZE` 保留区域，防止 musl 的 mmap 在 vDSO 预留间隙分配内存 <sup>[12](#mod-mm-ref-12)</sup>。vDSO BSS 区域通过主动触发 `handle_page_fault` 预填充，避免用户态首次访问时产生缺页 <sup>[13](#mod-mm-ref-13)</sup>。

`sys_mincore` 是另一个跨层协同的例子：它调用 `aspace.page_table().query(addr)` 直接查询页表，利用返回的连续映射 size 做批量判断（连续已映射页一次性填充结果数组），避免了逐页查询的开销 <sup>[14](#mod-mm-ref-14)</sup>。

### 4. 边角细节与不足

**`assert!` / `unwrap()` 暴露的健壮性风险** <sup>[15](#mod-mm-ref-15)</sup>：`map_elf` 中 `assert_eq!(seg_pad, ph.offset as usize % PAGE_SIZE_4K)` 假设 ELF 文件的文件偏移与虚拟地址的低 12 位对齐。如果加载了恶意构造的 ELF，这会直接 panic 内核（DoS）。`ElfLoader` 中 `self.0.front().unwrap()` 在 cache 为空时会 panic，虽在正常流程中不会发生，但缺乏防御性编程。

**TOCTOU 竞态** <sup>[16](#mod-mm-ref-16)</sup>：`check_null_terminated` 在 `access_user_memory` 闭包内反复获取/释放 `aspace` 锁来验证页权限，但实际的 `ptr.read_volatile()` 发生在锁外。另一个线程可能在两次操作之间 `munmap` 该页，造成读访问越界。

**Stub 系统调用** <sup>[17](#mod-mm-ref-17)</sup>：`sys_madvise`、`sys_msync`、`sys_mlock`/`sys_mlock2` 仅返回 `Ok(0)` 而无实际行为。依赖这些调用的应用（如通过 `madvise(MADV_DONTNEED)` 释放页或 `mlock` 锁定敏感内存）会静默失效，可能导致内存压力下的非预期行为。

**`sys_mremap` 朴素实现** <sup>[18](#mod-mm-ref-18)</sup>：采用 mmap+copy+munmap 三步走策略，而非操作页表项重映射。这导致：(a) 无法在原地扩展映射，(b) 物理页被复制而非共享，浪费内存，(c) 不支持 `MREMAP_FIXED` 等标志。注释中也标注了 `// TODO: full implementation`。

**`memfd_create` 实现错误** <sup>[19](#mod-mm-ref-19)</sup>：它在 `/tmp` 下创建真实临时文件而非匿名内存文件，函数自身注释为 `// This is cursed`。这不符合 memfd 的语义（密封文件、不关联磁盘路径），且在并发场景下存在竞争条件。

**`membarrier` 仅为编译屏障** <sup>[20](#mod-mm-ref-20)</sup>：`sys_membarrier` 对非 QUERY 命令只执行 `compiler_fence(Ordering::SeqCst)`，没有通过 IPI 或其他机制同步多核的 store buffer。在多核系统上这无法保证内存屏障语义。

**`cfg` 门闘可能遗漏 riscv64** <sup>[21](#mod-mm-ref-21)</sup>：`copy_from_kernel` 仅在 `not(aarch64 | loongarch64)` 时复制内核页表。riscv64 通常也使用独立页表（通过 ASID 或不同 satp），但未被排除，可能导致内核映射被错误复制到用户页表中。

**未完成的 I-cache 刷新** <sup>[22](#mod-mm-ref-22)</sup>：`map_elf` 末尾有拼写错误的 `// TDOO: flush the I-cache` 注释，表明在映射可执行段后未刷新指令缓存。在 ARM 等哈佛架构上，这可能导致执行到陈旧指令。

**`Vm` 结构体上的 `#[allow(dead_code)]`** <sup>[23](#mod-mm-ref-23)</sup>：暗示 `Vm` 可能未被充分使用或处于迁移状态，是潜在的工程腐烂信号。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块基于 axfs_ng_vfs 的 VFS 抽象层, 向上提供 Linux 兼容的文件系统调用 (open/read/write/ioctl/epoll 等 70+ 个 syscall). 核心包含三套机制: (1) 全局 FsContext 锁管理的目录树解析与路径操作, (2) 以 MemoryFs (tmpfs) 为内存后端的 inode 式文件系统实现 — 但其实际 I/O 被委托给页缓存层, (3) 基于 Weak 指针的自定义 epoll 实现, 支持 level/edge/oneshot 三种触发模式. 与 ArceOS 基线相比, starry 的差异化在于引入了完整的 Linux syscall 编号兼容层与用户态指针安全封装 (starry_vm), 并将设备文件 (如 evdev) 通过统一的 DeviceOps 接入 VFS.

### 1. 核心抽象: VFS trait 层次与全局上下文

文件系统模块的核心入口是 `axfs::FS_CONTEXT` — 一个全局的 `Mutex<FsContext>`, 它持有当前进程的根目录与工作目录引用 <sup>[1](#mod-fs-ref-1)</sup>. 所有路径解析型 syscall (chdir, mkdirat, linkat, renameat2 等) 均以 `FS_CONTEXT.lock()` 为起点, 调用 `fs.resolve()` 获得 `DirEntry` 后执行操作 <sup>[2](#mod-fs-ref-2)</sup>. 这种单锁设计简单直接, 但意味着所有文件系统操作被串行化: 一个进程在 `getcwd` 中持锁时, 另一个进程的 `chdir` 会被阻塞.

VFS 抽象由 `axfs_ng_vfs` crate 提供, 其 trait 层次分为三层: `NodeOps` (inode 元数据与 sync), `FileNodeOps` (read_at/write_at/append/set_len), `DirNodeOps` (lookup/create/link/unlink/rename). `MemoryFs` 中的 `MemoryNode` 同时实现了这三个 trait <sup>[3](#mod-fs-ref-3)</sup>. 值得注意的是, `FileNodeOps::read_at` / `write_at` / `append` 在 tmpfs 中直接 `unreachable!()` — 注释声明

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 本模块提供 POSIX 信号机制的完整实现，核心抽象围绕 `starry_signal` 库的 `SignalSet`/`Signo`/`kernel_sigaction` 等类型，每个线程通过 `Thread::signal` 维护独立的信号掩码、待决队列和处理器。系统调用层支持 kill/tkill/sigaction/sigwait 等十余个调用，并集成 signalfd 作为文件描述符。与同家族基线相比，该仓库同时存在两套信号处理路径：一是基于 `starry_core::task` 的现代化异步实现（用于系统调用），二是遗留的 `core/executor` 任务级框架，后者包含直接操作用户栈的 unsafe 代码且未完成 STOP/CONT 支持。

### 1. 核心抽象与分层架构

信号系统的底层类型由 `starry_signal` 库定义，包括 `SignalSet`（信号集位图）、`Signo`（信号编号）和 `kernel_sigaction`（信号动作描述符）。线程级的信号状态封装在 `starry_core::task::Thread` 中，通过 `signal()` 方法暴露。系统调用实现（`api/src/syscall/signal.rs`）直接操作这些对象，例如 `sys_rt_sigprocmask` 用 `current().as_thread().signal` 读写阻塞掩码，`sys_rt_sigaction` 更新动作表。这一层实现了 `kill(2)`、`tkill(2)`、`sigqueue(3)` 等 15+ 个系统调用，覆盖进程组、线程定向发送、siginfo 队列等 POSIX 细节。

另一套实现存在于 `core/src/executor/src/signal.rs`：每个执行器任务拥有一个 `SignalModule` 结构，独立保存最后陷入帧、信号处理器句柄、退出信号等。该模块通过 `async fn handle_signals()` 在任务返回用户态前检查并分派信号，采用直接修改 `TrapFrame`（PC、SP、参数寄存器）的方式将控制流转向用户态处理函数。两套实现并存反映出项目从早期执行器模型向通用线程抽象的迁移，但也带来状态同步与代码重复的风险 [ref:dual_impl]。

### 2. 异步等待与系统调用集成

`sigtimedwait` 和 `sigsuspend` 等阻塞调用通过 Rust 异步运行时实现零开销的等待。以 `sys_rt_sigtimedwait` 为例，它构造一个 `poll_fn` 闭包，在每次轮询时检查当前线程的待决信号是否匹配给定集合；若无信号，则调用 `curr.poll_interrupt(cx)` 将当前协程挂起，直到信号到达时由发送方唤醒或超时。`block_on(future::timeout(timeout, fut))` 组合提供了带超时的等待语义。这种设计避免了传统内核中每个线程一个等待队列的复杂同步，复用 Rust 的 `Waker` 机制即可与上层执行器无缝配合 [ref:sigtimedwait]。

信号发送方在 `core::send_signal_to_thread` 中，向目标线程的 `signal_set` 添加信号后，会检查该线程是否处于阻塞状态（`task.is_blocked()`），若是则调用 `taskctx::wakeup_task` 将其唤醒，从而触发重新检查信号。这一“投递-唤醒-检查”链条构成了信号异步机制的基础。值得注意的是，`api` 层发送信号最终也调用 `starry_core::task::send_signal_to_thread`，内部机制与 `core` 版本类似，但使用了不同的数据结构。

### 3. 用户态信号调用的栈操控与 Trampoline

当信号动作指定为用户态处理函数（非 SIG_DFL/IGN），内核需构造调用栈帧。`handle_signals` 通过修改当前任务的 `TrapFrame` 实现：PC 被设为 `sa_handler`，SP 指向调整后的用户信号栈（若未设置 `SA_ONSTACK` 则用原栈减去保护区域 512 字节）。返回地址被设为 `axconfig::SIGNAL_TRAMPOLINE`，即一段固定在进程地址空间的跳板代码，用于在执行完 handler 后调用 `sigreturn`。若设置 `SA_SIGINFO`，内核还会在栈上依次构造 `SigInfo` 和 `ucontext` 结构，并更新参数寄存器 [ref:trampoline]。

这些栈写入操作均通过 `unsafe` 裸指针完成，因为内核需要直接操作用户态内存。`handle_signals` 调用 `process.manual_alloc_range_for_lazy` 预分配栈页，但这并未保证地址合法性或防止跨任务破坏；一旦用户传递恶意 sp 值，可能导致内核内存被篡改 [ref:unsafe_stack]。`signal_return` 调用 `load_trap_for_signal` 还原保存的陷阱帧，并从 handler 返回原中断点继续执行。

### 4. signalfd：信号的文件描述符抽象

`signalfd` 机制将信号接收抽象为文件读取，这在 Linux 中常用于与 `epoll` 配合，实现统一的事件循环。仓库通过 `Signalfd` 结构实现 `FileLike` trait，其 `read` 方法阻塞地从线程信号队列中取出匹配掩码的信号，转换为 128 字节的 `SignalfdSiginfo` 结构写入用户缓冲区 [ref:signalfd_read]。内部使用 `block_on(poll_io(...))` 实现阻塞 I/O，并利用 `Pollable` trait 注册 waker，使得 signalfd 可以集成到 epoll 等多路复用器中 [ref:signalfd_poll]。`sys_signalfd4` 系统调用负责创建或更新 signalfd，处理 `CLOEXEC` 和 `NONBLOCK` 标志，并将其加入当前进程的文件描述符表。这种“一切皆文件”的设计保持了接口的统一性，但 `read` 内部的 `block_on` 调用要求调用上下文必须处于异步执行器内，可能在某些内核路径中引发死锁。

### 5. 边角细节与不足

- **未实现的默认动作**：`core` 模块中对 `SIGSTOP` 和 `SIGCONT` 的处理直接 `unimplemented!()`，任何触发将导致任务 panic；`api/src/signal.rs` 中虽不会 panic，但只是简单退出进程而非正确暂停/恢复，违反了 POSIX 语义 [ref:unimplemented]。
- **嵌套信号保护缺陷**：`handle_signals` 仅当已有未处理的信号且新信号为 `SIGSEGV`/`SIGBUS` 时才终止进程，其他信号的嵌套被静默丢弃（直接 return），可能造成信号丢失或导致上层误以为已处理 [ref:nested_protection]。
- **用户栈写入无校验**：多处 `unsafe` 块根据用户可控的 sp 值直接写入整个 `SigInfo`/`ucontext`，没有边界检查或映射验证，成为潜在的安全漏洞 [ref:unsafe_write]。
- **双重实现并存**：`core/executor` 与 `api` 系统调用层各自维护信号状态和发送路径，数据结构（`SignalModule` vs `starry_signal::Signal`）和行为存在分歧，长期维护成本高 [ref:dual_impl]。
- **硬编码常数 `USER_SIGNAL_PROTECT = 512`**：作为默认栈保护区域的大小，缺乏注释说明其依据，可能在特定 ABI 或信号帧较大时不足，且未随架构参数化 [ref:magic_number]。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块涵盖四套 IPC 机制: (1) System V 消息队列——以消息类型为键的 BTreeMap 存储, 支持 MSG_COPY 扩展但阻塞收发未实现; (2) System V 共享内存——基于 SharedPages 的惰性物理页分配, 通过 BiBTreeMap 维护 key↔shmid 双向映射; (3) futex——位集感知的 WaitQueue, 支持 Private/Shared 两种 FutexKey 及 requeue 操作; (4) 匿名管道——基于 ringbuf::HeapRb 的 poll 驱动异步 I/O. 所有 SysV IPC 共用 IpcPerm 权限模型, 全局管理器由单一 Mutex 保护.

### 1. 核心抽象与外部依赖

该模块并非单一"进程间通信子系统", 而是四套独立 IPC 机制的松散集合: System V 消息队列 (`api/src/syscall/ipc/msg.rs`)、System V 共享内存 (`core/src/shm.rs` + `api/src/syscall/ipc/shm.rs`)、futex (`core/src/futex.rs` + `api/src/syscall/sync/futex.rs`)、以及匿名管道 (`api/src/file/pipe.rs`). 它们共享的唯一基础设施是 `IpcPerm` 权限结构体 <sup>[1](#mod-ipc-ref-1)</sup>——一个 `#[repr(C)]` 的 9 位 Unix 权限模型, 被消息队列的 `msqid_ds` 与共享内存的 `ShmidDs` 同时嵌入。权限检查通过 `has_ipc_permission` 辅助函数统一实现, 对 root (uid==0) 无条件放行 <sup>[10](#mod-ipc-ref-10)</sup>.

从外部依赖看, 消息队列与 futex 重度依赖 `starry_vm` 提供的 `VmPtr/VmMutPtr` trait 来实现用户态指针的安全读写; 共享内存则依赖 `axmm::backend::SharedPages` 作为物理页的跨进程共享载体; 管道使用 `ringbuf::HeapRb` 作为环形缓冲区, 并通过 `axpoll` 的 `PollSet` 实现异步等待唤醒。

### 2. System V 消息队列: 类型索引与阻塞缺口

消息队列的核心数据结构 `MessageQueue` 使用 `BTreeMap<i64, Vec<Message>>` 按消息类型 (mtype) 分组存储 <sup>[2](#mod-ipc-ref-2)</sup>, 并通过 `total_bytes` 与 `msqid_ds` 中的 `msg_cbytes/msg_qnum` 双重追踪队列占用。消息匹配逻辑实现了 POSIX 标准要求的四种模式: mtype==0 取首条, mtype>0 精确匹配, mtype<0 取类型 ≤|mtype| 的最小值, 以及 Linux 扩展 MSG_EXCEPT (取类型不等于指定值的首条). 此外还实现了 Linux 特有的 MSG_COPY 模式——按索引拷贝消息而不移除 <sup>[3](#mod-ipc-ref-3)</sup>, 该模式强制要求与 IPC_NOWAIT 联用且与 MSG_EXCEPT 互斥.

全局消息队列管理器 `MSG_MANAGER` 是一个 `Mutex<MsgManager>`, 内部维护 key→msqid 和 msqid→queue 两张 BTreeMap <sup>[4](#mod-ipc-ref-4)</sup>. 每次 `sys_msgget/sys_msgsnd/sys_msgrcv` 都需要获取该全局锁, 这在多进程并发 IPC 场景下构成潜在瓶颈。更关键的是, **阻塞收发完全未实现**: `sys_msgsnd` 在队列满时若未指定 IPC_NOWAIT, 本应阻塞等待, 但当前实现直接返回 EAGAIN 并打印 warn 日志 <sup>[5](#mod-ipc-ref-5)</sup>; `sys_msgrcv` 同理, 无消息时直接返回 ENOMSG <sup>[6](#mod-ipc-ref-6)</sup>. 这两个 TODO 意味着依赖阻塞语义的应用程序 (如 Shell 管道脚本、数据库的消息队列通知) 将无法正常工作.

IPC_RMID 的"标记删除"机制值得注意: 当队列非空时仅设置 `mark_removed` 标志, 待所有消息被取走后由后续操作触发删除 <sup>[7](#mod-ipc-ref-7)</sup>. 但标记删除后并不会唤醒等待的读写者, 违反了 Linux man-page 要求的"返回 EIDRM 错误"语义 <sup>[8](#mod-ipc-ref-8)</sup>.

### 3. 共享内存: 惰性分配与跨进程页面复用

共享内存子系统的设计亮点在于物理页的惰性分配。`ShmInner` 在创建时不立即分配物理页 (`phys_pages: None`), 而是在首个进程调用 `sys_shmat` 时通过 `SharedPages::new` 分配, 并通过 `Backend::new_shared` 映射到进程地址空间 <sup>[9](#mod-ipc-ref-9)</sup>. 后续进程 attach 同一 shmid 时, 直接克隆已有的 `Arc<SharedPages>` 并映射, 实现零拷贝共享 <sup>[11](#mod-ipc-ref-11)</sup>. 这种设计避免了预分配造成的物理内存浪费.

`ShmManager` 维护三层映射关系: key↔shmid (通过 `BiBTreeMap`——一个自维护的双向映射 <sup>[12](#mod-ipc-ref-12)</sup>)、shmid→ShmInner (标准 BTreeMap)、以及 pid→(shmid↔vaddr) (每进程独立的 BiBTreeMap) <sup>[13](#mod-ipc-ref-13)</sup>. 进程退出时 `clear_proc_shm` 遍历该进程所有附属段, 执行 detach 并在 `rmid` 标志置位且引用计数归零时彻底删除 <sup>[14](#mod-ipc-ref-14)</sup>. 但 `BiBTreeMap` 自身的 TODO 注释 ("I don't know where to put this") 暴露了模块边界划分的临时性.

`sys_shmat` 中 `shm_manager.get_inner_by_shmid(shmid).unwrap()` <sup>[15](#mod-ipc-ref-15)</sup> 是一个危险的 `unwrap()`——若用户传入无效 shmid, 将直接触发 kernel panic 而非返回 EINVAL. `attach_process` 与 `detach_process` 内部的 `assert!` <sup>[16](#mod-ipc-ref-16)</sup> 同样将本应优雅处理的重复 attach/detach 变成了 panic 入口.

### 4. Futex: 位集唤醒与健壮列表

Futex 子系统是本模块实现最完整的 IPC 机制。`WaitQueue` 以 `SpinNoIrq<VecDeque<(Waker, u32)>>` 存储等待任务及其 bitset, 支持 `FUTEX_WAIT_BITSET/FUTEX_WAKE_BITSET` 的位掩码匹配语义 <sup>[17](#mod-ipc-ref-17)</sup>. `requeue` 操作允许将等待者从一个 futex 转移到另一个——这是 glibc 条件变量实现 (pthread_cond) 的关键原语.

`FutexKey` 的创新在于区分 Private 与 Shared 两种 futex <sup>[18](#mod-ipc-ref-18)</sup>: Private futex 仅以虚拟地址为键, 用于同一进程的线程同步; Shared futex 通过遍历地址空间的 VMA 查找所属的 `SharedPages` 或文件后端, 以偏移量为键, 实现跨进程的 futex. 这种设计使得同一个 `FutexTable` 可以安全地混合存储两种 futex (因为键的语义空间不同).

`FutexGuard` 的 Drop 实现 <sup>[19](#mod-ipc-ref-19)</sup> 采用引用计数清理策略: 当 `Arc` 强引用计数 ≤2 (即只剩 table 中的一份和当前 guard) 且等待队列为空时, 自动从 table 中移除该条目, 防止内存泄漏. 健壮列表 (robust list) 通过 `owner_dead` 原子标志位实现, 当 futex 所有者异常终止时, 内核可检测该标志并唤醒等待者返回 `EOWNERDEAD`.

### 5. 边角细节与不足

**<sup>[5](#mod-ipc-ref-5)</sup><sup>[6](#mod-ipc-ref-6)</sup> 阻塞语义缺失**: 消息队列的阻塞收发是 System V IPC 的核心语义, 当前直接返回 EAGAIN/ENOMSG 使多数实际用例无法工作. 需要引入等待队列并将当前任务挂起.

**<sup>[15](#mod-ipc-ref-15)</sup> 可触发 panic 的 unwrap**: `sys_shmat` 第 78 行的 `unwrap()` 接受用户态可控的 shmid 参数, 构成 DoS 入口.

**<sup>[16](#mod-ipc-ref-16)</sup> assert 代替错误处理**: `ShmInner::attach_process` 与 `detach_process` 使用 `assert!` 验证前置条件, 若因竞态条件 (如 SYS_shmdt 重复调用) 触发将导致 kernel panic.

**<sup>[8](#mod-ipc-ref-8)</sup> IPC_RMID 不唤醒等待者**: 标记删除后等待的读写者永远不会被唤醒, 可能导致进程永久阻塞 (一旦阻塞机制实现后).

**<sup>[12](#mod-ipc-ref-12)</sup> BiBTreeMap 放置不当**: `BiBTreeMap` 被放在 `core/src/shm.rs` 中且带有 "I don't know where to put this" 的 TODO, 说明缺乏通用工具模块的规划.

**<sup>[4](#mod-ipc-ref-4)</sup> 全局锁粒度**: `MSG_MANAGER` 和 `SHM_MANAGER` 均使用单一 `Mutex` 保护, 在高并发场景下可能成为瓶颈. 消息队列的 `sys_msgsnd` 在持锁期间还执行用户态内存拷贝 (`vm_load`), 延长了临界区.

**<sup>[7](#mod-ipc-ref-7)</sup> mark_removed 的 TOCTOU**: `sys_msgrcv` 在获取队列锁后检查 `mark_removed`, 但 `sys_msgctl(IPC_RMID)` 在队列非空时仅置标志位而不删除——这意味着后续 `sys_msgrcv` 拿走最后一条消息后, 队列仍残留在全局 map 中, 没有触发自动清理逻辑.

## 🌐 九、网络

> 💡 **TL;DR**
>
> 网络模块将 axnet 协议栈能力封装为 POSIX 套接字系统调用. 核心抽象有两层: (1) `SocketAddrExt` trait 实现用户态 sockaddr 与内核地址类型的双向转换, 支持 IPv4/IPv6/Unix/VSOCK 四族; (2) `Socket` 通过 `FileLike` trait 融入文件描述符体系, 使 read/write/poll 透明作用于套接字. 与 ArceOS 基线相比, starry 增加了 Unix Domain Socket (含 SCM_RIGHTS 控制消息) 和 VSOCK, 并通过 IoVec 支持 scatter-gather I/O.

### 1. 核心抽象: FileLike 与 SocketAddrExt 的双层适配

模块并非直接实现协议栈, 而是在底层 `axnet` crate 之上构建 POSIX 兼容层. 第一层抽象是 `Socket` 类型 — 一个 `axnet::Socket` 的 newtype 包装, 通过 `Deref` 透明暴露底层方法 <sup>[1](#mod-net-ref-1)</sup>. `Socket` 实现 `FileLike` trait, 其 `read`/`write` 分别委托给 `recv`/`send`, 使套接字可被 `select`/`poll`/`epoll` 统一管理 <sup>[2](#mod-net-ref-2)</sup>. 同时实现 `Pollable`, 将异步事件注册到 `axpoll` 框架.

第二层抽象是 `SocketAddrExt` trait, 定义 `read_from_user`/`write_to_user`/`family` 三个方法 <sup>[3](#mod-net-ref-3)</sup>. 其核心技巧是: 先通过 `read_family` 从用户态 sockaddr 头部偷看 `sa_family_t` 以确定地址族 <sup>[4](#mod-net-ref-4)</sup>, 再分派到具体类型 (V4/V6/Unix/VSOCK) 的反序列化实现. 写回用户态时使用 `fill_addr` 辅助函数, 以 `min(addrlen, data.len())` 截断防止缓冲区溢出, 并将实际长度写回 `addrlen` <sup>[5](#mod-net-ref-5)</sup>. 这种 "peek-then-dispatch" 模式避免了为每种地址族维护独立的 syscall 入口.

### 2. 关键设计取舍: 薄封装与协议分派的代价

syscall 层保持极薄: `sys_bind`/`sys_connect` 仅做地址解析后立即委托给 `Socket::from_fd(fd)?.bind(addr)` <sup>[6](#mod-net-ref-6)</sup>. 协议族与类型的分派集中在 `sys_socket` 的一个 `match (domain, ty)` 中 <sup>[7](#mod-net-ref-7)</sup>. 这个设计放弃了内核内统一协议切换表, 换来代码直观性, 但导致每增加一个协议族需修改 match 分支. 实际分支覆盖: TCP/UDP over IPv4, Unix stream/dgram, 以及 feature-gate 的 VSOCK stream.

`sys_listen` 接收但完全忽略 `backlog` 参数 <sup>[8](#mod-net-ref-8)</sup> — 底层 `listen()` 未暴露积压队列配置能力, 这是 TCP 监听语义的明显残缺. `send_impl` 接收 `flags: u32` 却在构造 `SendOptions` 时硬编码 `SendFlags::default()` <sup>[9](#mod-net-ref-9)</sup>, 意味着 MSG_OOB/MSG_DONTROUTE 等标志被静默丢弃; `recv_impl` 虽处理了 `MSG_PEEK` 和 `MSG_TRUNC` <sup>[10](#mod-net-ref-10)</sup>, 但 MSG_WAITALL 等仍缺失. 这是协议完整性 vs. 实现复杂度的取舍 — starry 选择了快速可用而非完整 POSIX 合规.

`socketpair` 仅支持 `AF_UNIX` <sup>[11](#mod-net-ref-11)</sup>, 且将 `SOCK_SEQPACKET` 降级为 `DgramTransport::new_pair` <sup>[12](#mod-net-ref-12)</sup>, 语义上有微妙偏差 (SEQPACKET 应保证记录边界和顺序, 而 dgram 不保证顺序).

### 3. 跨模块协同: 用户态指针、文件描述符与控制消息

模块跨四个子系统协同工作. 其一, `crate::mm` 提供的 `UserPtr`/`UserConstPtr` 封装了用户态内存的安全访问 — 所有 `get_as_ref`/`get_as_mut`/`get_as_slice` 调用均经过页表边界检查, 防止内核被非法用户指针触发缺页 <sup>[13](#mod-net-ref-13)</sup>. 其二, `crate::file` 的 `add_file_like`/`get_file_like` 实现了 fd 整数与 `Arc<dyn FileLike>` 的双向映射, `Socket::from_fd` 通过 downcast 确保 fd 确实是套接字 <sup>[14](#mod-net-ref-14)</sup>.

其三, `CMsg`/`CMsgBuilder` 实现了 SCM_RIGHTS 的完整传递闭环. 发送端: `sys_sendmsg` 遍历 `msghdr.msg_control` 中的 cmsghdr 链, 调用 `CMsg::parse` 将每个 SCM_RIGHTS 消息中的 fd 数组解析为 `Vec<Arc<dyn FileLike>>` <sup>[15](#mod-net-ref-15)</sup>, 再装箱为 `CMsgData` trait object 塞入发送选项. 接收端: `recv_impl` 将收到的 `CMsgData` downcast 回 `CMsg::Rights`, 对每个 `FileLike` 重新分配 fd 并写回用户态 cmsghdr 缓冲区 <sup>[16](#mod-net-ref-16)</sup>. 这里有一个微妙行为: 如果用户缓冲区不足以容纳所有 cmsg, `CMsgBuilder::push` 返回 `Ok(false)` 并提前终止, 与 Linux 的 `EMSGSIZE` 行为不同 (Linux 会直接失败).

其四, Unix socket 创建依赖 `starry_core::task::AsThread` 获取进程 PID, 传递给 `StreamTransport::new(pid)` <sup>[17](#mod-net-ref-17)</sup>, 用于 `SO_PEERCRED` 等凭证传递 — 这是 starry 多进程模型与网络栈的耦合点.

### 4. 边角细节与不足

- **`cast_to_slice` 无 SAFETY 注释** <sup>[18](#mod-net-ref-18)</sup>: 该 `unsafe fn` 将任意 `&T` 重新解释为 `&[u8]`, 调用者需保证 `T` 无内部填充字节且对齐正确. 当前所有调用点传入的是 `#[repr(C)]` 结构体, 但这一契约未文档化, 未来维护者可能误用于带 padding 的 Rust 结构体.
- **`SocketAddrEx::family()` 始终返回 `AF_INET`** <sup>[19](#mod-net-ref-19)</sup>: 该方法对 `Unix` 和 `Vsock` 变体也返回 `AF_INET`, 显然是一个未完成的桩实现. 所幸在当前代码路径中 `family()` 未被实际调用.
- **`nonblocking()` 中的 `unwrap()`** <sup>[20](#mod-net-ref-20)</sup>: `get_option(GetSocketOption::NonBlocking(&mut result)).unwrap()` 在底层出错时直接 panic, 构成用户态可达的内核崩溃入口 (尽管 `GetSocketOption::NonBlocking` 实现大概率不会失败).
- **`stat()` 硬编码权限 0o777**<sup>[21](#mod-net-ref-21)</sup>: 注释 `TODO(mivik): implement stat for sockets` 明确标记为未完成; 所有套接字均返回世界可读写执行权限, 不区分 Unix socket 文件与 inet socket.
- **UnixSocketAddr 路径名解析的 UTF-8 假设** <sup>[22](#mod-net-ref-22)</sup>: `str::from_utf8(&data[..end])` 要求路径为合法 UTF-8, 但 Linux 允许文件系统路径为任意字节序列 (除 NUL 和 `/`). starry 返回 `InvalidInput` 而非更准确的 `EINVAL`, 且限制了非 UTF-8 路径名的使用.
- **`sys_accept4` 中错误命名 `remote_addr`** <sup>[23](#mod-net-ref-23)</sup>: 实际调用的是 `socket.local_addr()` 获取本地地址, 却将变量命名为 `remote_addr`. 这不影响正确性但易导致后续维护误读.
- **`sockaddr_vm` 手动定义** <sup>[24](#mod-net-ref-24)</sup>: 因 `linux_raw_sys` 未提供 VSOCK 的 sockaddr 结构体, 代码自行定义并附 GitHub issue 链接. 若上游修复后此处将成为重复定义, 且手动维护的 layout 可能与内核头文件不同步.

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块以 `api/src/syscall/mod.rs` 中的 `handle_syscall` 为核心，使用 `syscalls::Sysno` 枚举进行巨型 match 分发，将约 180+ 个 Linux 系统调用映射到各子模块的 `sys_*` 处理函数。与 ArceOS 基线相比，本仓库新增了 vsched2 调度框架的 trap 拦截路径、大量 `#[cfg(target_arch)]` 条件编译以区分 x86_64 与 riscv64 支持，以及通过 `UserTrapFrame`/`VschedTrapInfoImpl` 实现的自定义 trap 分发器注册机制。

### 1. 核心抽象：巨型 match 分发器与 UserContext

`handle_syscall` 是整个系统调用层的唯一入口 <sup>[1](#mod-syscall-ref-1)</sup>。它接收 `&mut UserContext`，从中提取系统调用号与 6 个参数，通过 `Sysno::new()` 将原始编号转换为枚举变体，若转换失败则返回 `ENOSYS` <sup>[2](#mod-syscall-ref-2)</sup>。分发逻辑是一个覆盖 ~180 个变体的 `match` 语句，每个 arm 调用对应的 `sys_*` 函数并将 `uctx.argN()` 强转为所需类型。最终结果通过 `uctx.set_retval()` 写回用户态上下文，错误码经 `LinuxError::from(err).code()` 转换为负的 Linux errno <sup>[3](#mod-syscall-ref-3)</sup>。

这种设计的取舍很明确：零抽象成本的静态分发，所有 syscall handler 在编译期即确定调用目标，无虚表开销。代价是 `mod.rs` 文件膨胀至 640 行且可维护性差——每增删一个 syscall 需修改中央 match。与 Linux 内核的 `sys_call_table` 数组索引方案相比，本方案放弃了运行时可扩展性（如 seccomp 过滤无法 hook），换取了 Rust 模式匹配的穷尽性检查与类型安全。

### 2. 关键设计取舍：条件编译与架构差异化

模块中存在大量 `#[cfg(target_arch = "x86_64")]` 和 `#[cfg(not(target_arch = "riscv64"))]` 的条件编译 <sup>[4](#mod-syscall-ref-4)</sup>。例如 `open`、`mkdir`、`stat`、`lstat`、`chown` 等传统 syscall 仅在 x86_64 下可用，而 `openat`、`mkdirat`、`fstatat` 等 `*at` 变体作为 POSIX 新标准在 riscv64 上优先支持 <sup>[5](#mod-syscall-ref-5)</sup>。这反映了一个工程权衡：riscv64 是优先支持的新平台，因此跳过旧版 syscall 以减少实现负担，但 `renameat` 却被显式排除在 riscv64 之外 (`#[cfg(not(target_arch = "riscv64"))]`) 而直接使用 `renameat2` <sup>[6](#mod-syscall-ref-6)</sup>，说明对 riscv64 的 syscall 兼容策略尚不一致。

更微妙的取舍在 `newfstatat` vs `fstatat` 的处理上：x86_64 和 riscv64 使用 `Sysno::newfstatat`，其他架构走 `Sysno::fstatat` <sup>[7](#mod-syscall-ref-7)</sup>。这暴露了 `Sysno` 枚举对不同架构 ABI 差异的适应——64 位 arch 的 `stat` 结构不同导致系统调用号差异，本层选择用 cfg gate 硬编码而非运行时检测。

### 3. 跨模块协同：从硬件 trap 到 syscall 分发的完整链路

系统调用层的调用链路涉及三个 crate 的跨模块配合：

**trap 入口 → 上下文保存** (`core/src/vsched/trap_vector.rs`)：当用户态执行 `ecall` 指令，CPU 陷入 S-mode，跳转到 `vsched2_trap_vector`。汇编代码先将 `sp` 与 `sscratch` 交换以切换到预设的内核栈，保存 t0-t4 暂存寄存器，然后检测是否为 vsched2 特殊 ecall（`a7 == 0xdead`），若是则走 `vsched2_direct_entry_stub` 快速路径 <sup>[8](#mod-syscall-ref-8)</sup>。普通 trap 则继续保存完整寄存器上下文到 `UserTrapFrame`，设置 `kind = 1 (Trap)`，调用 Rust 函数 `vsched2_trap_entry_stub` <sup>[9](#mod-syscall-ref-9)</sup>。

**trap 分发 → syscall 处理** (`core/src/vsched/trap.rs`)：`VschedTrapInfoImpl` 实现了 `libvsched2::TrapInfo` trait，其 `handle()` 方法从全局 `TRAP_DISPATCHER` 原子变量读取事先注册的函数指针并调用 <sup>[10](#mod-syscall-ref-10)</sup>。在分发前，它记录最后一次陷入的用户任务指针到 `LAST_TRAPPED_USER_TASK`，用于 page fault 回退时定位地址空间 <sup>[11](#mod-syscall-ref-11)</sup>。`new_handler()` 为 trap 处理创建新的 axtask 协程，通过 `register_task` 注册到 vsched2 调度器 <sup>[12](#mod-syscall-ref-12)</sup>。

**syscall 参数提取** (`api/src/syscall/mod.rs`)：当上层（调度框架）最终调用 `handle_syscall` 时，`UserContext` 已从 `UserTrapFrame` 的 `a0`-`a7` 寄存器中提取了 syscall 参数。`handle_syscall` 首行检查 `VSCHED2_INTO_KERNEL_SYSNO`——这是 vsched2 的用户态→内核态切换协议的私有 ecall 编号，被拦截后直接调用 `enter_raw_trap_entry()` 而不走标准 syscall 路径 <sup>[2](#mod-syscall-ref-2)</sup>。

**trap 帧恢复** (`core/src/vsched/trapframe.rs`)：提供三种恢复路径：`restore_and_sret`（S→U 全量恢复+sret）、`restore_and_sret_user`（恢复+切用户页表+sret）、`restore_and_jump`（根据 `kind` 选择 Yield 轻量恢复或 Trap 全量恢复+jr）<sup>[13](#mod-syscall-ref-13)</sup>。`restore_and_jump(Trap)` 的设计精妙之处在于：它不使用 `sscratch` 暂存 `sepc`，而是将 `sepc` 放在 trap frame offset 256 处，先恢复 31 个 GPR（offset 8~248），最后再从 offset 256 读取 `sepc` 到 t0 并 `jr t0`。这避免了 `sscratch` 污染导致嵌套异常时 trap 向量误用 <sup>[14](#mod-syscall-ref-14)</sup>。
