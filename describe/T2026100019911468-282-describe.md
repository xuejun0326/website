## 📖 一、总览

该仓库是 xv6-riscv 的深度定制变体，定位为面向多平台的教学型实验内核，但实际已远超原版教学范畴：它在 RISC-V (K210/QEMU/VF2) 和 LoongArch64 上实现了一套包含写时拷贝、惰性页分配、三级可切换调度器和 FAT32/ext4 双文件系统等现代 OS 特性的混合体，更独特的是内嵌了一个 Linux-RISC-V ABI 兼容层，使未修改的 Linux 测试二进制可直接运行。整体上，这是一个为了验证多项 OS 扩展而拼合而成的竞技场式实现，而非追求生产质量的完整内核。

最值得评审注意的三个特点：1) Linux ABI 兼容层——通过进程名启发式 (`is_linux_abi_proc`) 将 Linux 系统调用号动态翻译为本地语义，这在 xv6 衍生中极为罕见，直接体现了‘让现成 Linux 测试套件在自制内核上跑起来’的野心 [syscall.ref:4]；2) 调度器的可切换设计——进程结构内嵌了 MLFQ 的动态优先级字段，并允许运行时通过专用系统调用改变调度算法，这在教学内核中引入了实时系统才有的灵活性 [task.ref:5]；3) 双页表与 COW 的组合——每进程独立内核页表 (`kpagetable`) 与写时拷贝 fork 共存，兼顾了内核栈隔离和内存效率，但引入了复杂的竞态管理 [mm.ref:4][mm.ref:6]。

## 🎯 二、综合评价

### 整体定位

该仓库是 xv6-riscv 教学内核的激进扩展版，远超出“教学演示”范畴：它在启动层同时支持 RISC-V 和 LoongArch64 双架构 [boot.ref:1][boot.ref:10]，内存管理引入 COW 与惰性分配 [mm.ref:4][mm.ref:5]，进程调度实现可切换的 RR/PRIORITY/MLFQ 策略 [task.ref:1][task.ref:5]，文件系统吞下 FAT32 与只读 ext4 [fs.ref:3][fs.ref:4]，并直接内嵌 Linux 测试套件运行框架 [syscall.ref:2][syscall.ref:4]。这些组件并非从零设计，而是以“移植+整合”方式堆积而成，因此它更像一个面向竞赛或深度实验的 OS 特性验证平台，而非追求产品级完整性的通用内核。其代码规模达 25k 行，却极度缺乏防御性编程与一致性清理，反映出典型的“快速原型”风格。

### 真正的创新点

仓库最突出的原创性设计是 Linux ABI 兼容层。它不依赖 ELF 解析或 AUXV 信息，仅凭进程名中的模式匹配 (`is_linux_abi_proc`) 激活 Linux syscall 翻译 [syscall.ref:4]，这种轻量启发式方法在 xv6 衍生中独一无二。它将数十个 Linux 系统调用号映射到本地语义，使未经修改的 Linux 测试二进制可直接在单地址空间的内核中执行，大幅降低了评测门槛。其次，调度器支持运行时算法切换，结构体内嵌 MLFQ 所需的 `mlfq_cpu_ticks` 等动态字段 [task.ref:1]，并通过 `SYS_set_timeslice` 等系统调用暴露控制接口，在教学内核中引入了接近实时系统的灵活性。第三，每进程独立内核页表 (`kpagetable`) 与 COW fork 的组合 [mm.ref:4][mm.ref:6] 尝试在内存效率和内核栈隔离之间取得平衡——尽管实现上存在竞态，但设计意图清晰，在同类小型内核中并不多见。

### 取舍与代价

该仓库的选择性投入非常明显：精力集中在调度器、文件系统后端和 Linux 测试支持上，而信号机制完全空白，网络栈缺失。驱动层仅针对 K210 的 DMA/SPI 做了硬编码移植 [drivers.ref:1][drivers.ref:2]，毫无可移植抽象。文件系统层虽然提供了 ext4 支持，却仅实现了只读路径，写入直接返回错误。内存管理上，为了 COW 和惰性分配，代码容忍了多核竞态窗口（`uvmcopy` 先改 PTE 后增引用）[mm.ref:4] 和整数溢出漏洞（`copyin2` 边界检查）[mm.ref:6]。测试代码直接嵌入核心调度模块（约 200 行测试数据在 `proc.c` 中）[task 不足]，造成了生产代码与测试基础设施的强耦合。这些取舍符合“快速验证想法”的竞赛场景需求，但留下了大量技术债务。

### 完成度与不足

完成度呈两极分化：核心路径（fork/execve/管道/基本文件 I/O）尚能支撑 lua、busybox、libctest 等复杂测试 [task 测试嵌入]，但边缘路径缺陷密集。ext4 卷无写入能力，FAT 表缺少缓存层 [fs 不足]。Linux ABI 中 `kill`、`nanosleep`、`mount` 等大量函数仅为空桩 [syscall.ref:5]。管道计数器使用无溢界保护的 `uint`，可能导致死锁 [ipc.ref:4]。启动入口对 hart ID 未做边界检查 [boot.ref:10]，异常入口未保存浮点寄存器 [boot.ref:2]。驱动层多处 `kalloc()` 返回值未检查 [drivers.ref:5]，SPI DMA 等待通道共享单一唤醒点 [drivers.ref:3]。更有大量注释残留、隐式 ERA 传递约定 [boot.ref:3] 等工程性问题。整体而言，该仓库是一个“功能概念验证”的激进原型，展示了丰富的可能性，但在健壮性、安全性和可维护性上需大量打磨。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程由两族汇编入口构成：LoongArch64 的 la-minimal（`la-minimal/entry.S`）与 RISC-V 的 xv6-k210 系列（`entry_k210.S`/`entry_qemu.S`/`entry_vf2.S`）。核心抽象是“从机器复位到 C 入口的栈与调用约定桥接”：每个入口根据 hart ID 计算 per-hart 栈顶，转入 `la_main` 或 `main`。la-minimal 额外内联了完整异常入口 `la_exception_entry` 和一个编译期段表驱动的用户态探针 `la_user_probe`，可直接在特权态内核中顺序执行用户二进制并通过 syscall 输出。与标准 xv6-riscv 相比，多平台入口引入了 hart-aware 栈分配、平台特定空闲指令（`wfi`/`idle`）以及非统一入口符号（`_entry` 与 `_start` 混用）。

### 1. 双族启动骨架：LoongArch64 与 RISC-V 的入口抽象

仓库包含两条独立的启动路径。LoongArch64 侧 (`la-minimal/entry.S`) 以 `_start` 为固件入口，仅做两件事：通过 `la.local $sp, boot_stack_top` 装载初始栈指针，随后 `bl la_main` 跳入 C 代码 <sup>[1](#mod-boot-ref-1)</sup>。若 `la_main` 返回，则执行 `idle 0` 后无限循环——这是一个“不归路”约定，内核永不返回。栈空间 `boot_stack` 大小为 16 KiB，位于 `.bss.stack` 节并以 4 KiB 对齐 <sup>[9](#mod-boot-ref-9)</sup>。

RISC-V 侧三文件 (`entry_k210.S`, `entry_qemu.S`, `entry_vf2.S`) 共享同一模式：利用固件传入的 hart ID（`a0` 寄存器）计算栈偏移 `(a0 + 1) << 14`，即每 hart 获得 16 KiB 的栈窗口。随后 `la sp, boot_stack; add sp, sp, t0` 将栈顶定位到对应窗口顶部，再 `call main` 进入 C 入口 <sup>[10](#mod-boot-ref-10)</sup><sup>[13](#mod-boot-ref-13)</sup><sup>[14](#mod-boot-ref-14)</sup>。这实质上是一个编译期参数化的 per-hart 启动桩，放弃了硬件线程发现（如 DeviceTree 解析）的灵活性，换取了极简的代码尺寸与确定性。

两族入口共同遵循“汇编只做栈、C 做其余”的职责切分：汇编不初始化 BSS、不配置页表、不设置陷入向量——这些全部交给 `main`/`la_main` 完成。这简化了汇编层，但也意味着若 C 入口在早期崩溃，汇编层没有任何回退输出（无 panic、无 UART 打印），系统静默挂起。

### 2. 异常/陷入处理的早期绑定：la_exception_entry 的寄存器保存与 CSR 交互

`la-minimal/entry.S` 在同一个翻译单元内直接提供了完整的异常入口 `la_exception_entry`，标记于 `.text.exception` 节并以 12 位对齐（满足 LoongArch 异常基址对齐要求）<sup>[2](#mod-boot-ref-2)</sup>。该入口在栈上一次性开辟 256 字节帧，按 `r1` 到 `r31` 的顺序保存全部通用寄存器（跳过 `r0` 恒零寄存器与 `r3`/sp 自身），每个寄存器占 8 字节 <sup>[2](#mod-boot-ref-2)</sup>。保存完毕后，连续读取四个关键 CSR——`0x5` (ERA, 异常返回地址)、`0x6` (PRMD, 特权模式信息)、`0x7` (BADV, 坏地址)、`0x8` (ESTAT, 异常状态)——存入 `r4`-`r7`，再将当前栈指针 `r3` 复制到 `r8` 作为 C 函数第一个参数，随后调用 `la_trap_handler` <sup>[3](#mod-boot-ref-3)</sup>。

返回后，入口执行一次 `csrwr $r4, 0x6` 将 `r4`（可能被 `la_trap_handler` 修改过的 ERA/PRMD）写回 PRMD，然后从栈帧中逐寄存器恢复，最后 `ertn` 返回 <sup>[4](#mod-boot-ref-4)</sup>。这里存在一个精细的隐含约定：`la_trap_handler` 返回前必须将期望的 ERA 值留在 `r4` 中，否则 `ertn` 的目标地址不可控。这种通过寄存器而非栈传递 CSR 改写的模式，避免了在栈帧中额外保留 ERA 槽位，但也让调用约定变得脆弱——若 `la_trap_handler` 的实现未遵循此约定，返回地址将不可预测。

值得注意的是，该异常入口未保存任何浮点寄存器（32 个 FPR 及 FCSR）。如果 `la_trap_handler` 内部使用浮点运算，或被中断的用户代码正在使用浮点，状态会静默损坏。这在教学/最小化内核中是可接受的简化，但对任何需要运行浮点密集型负载的场景都是隐患 <sup>[2](#mod-boot-ref-2)</sup>。

### 3. 用户态探针机制：编译期段表驱动的小批量测试框架

`la-minimal/entry.S` 的独特功能是一套“用户态探针”（user probe），用于在特权态内核环境中执行预链接的用户二进制并收集输出。它由三部分协同构成：

- **进入用户态**：`la_enter_user_probe` 将 `r4`（预设 PRMD 值）写入 CSR `0x6`，将常数 `3` 写入 CSR `0x1`（CRMD 的 PLV 字段，设为用户模式），然后将栈指针替换为 `r5` 指定的用户栈，最后 `ertn` 切换到用户态执行 <sup>[5](#mod-boot-ref-5)</sup>。
- **用户态代码**：`la_user_probe` 位于 `.text.user` 节，在用户态下遍历 `la_user_segment_table` 段表。每个表项为一对 8 字节指针（起始/结束），它计算长度后通过 `syscall 0` 发起写系统调用（`r11=64`，文件描述符 `r4=1`），将段内容输出 <sup>[7](#mod-boot-ref-7)</sup>。全部段输出完毕后，发起 `exit` 系统调用（`r11=93`）。
- **返回内核**：exit 系统调用触发异常，经 `la_exception_entry` 路由到 `la_trap_handler`，后者识别出探针结束时调用 `la_after_user_probe`，该函数重置栈指针到 `boot_stack_top`，调用 `la_user_probe_done` 通知 C 层，然后进入空闲循环 <sup>[6](#mod-boot-ref-6)</sup>。

段表本身硬编码于 `.rodata.user`，包含 12 组测试二进制（glibc/musl 变体），涵盖 busybox、LTP、iperf、cyclictest 等 <sup>[8](#mod-boot-ref-8)</sup>。这套机制的取舍非常鲜明：它避免了实现完整的 ELF 加载器和用户态内存管理，而是利用链接器顺序将用户二进制直接嵌入内核映像，在编译期完成地址解析。代价是弹性为零——二进制集合在编译时固定，无法在运行时动态加载或替换。同时，用户代码运行在内核的页表中（没有独立的用户页表），系统调用通过 `syscall` 指令触发异常回到同一个内核处理，安全性完全依赖 `la_trap_handler` 中对用户态指针的校验。

### 4. 跨平台差异与多 hart 协同

三个 xv6 入口文件虽然模式一致，但在细节上暴露了面向不同运行时的妥协：

- **入口符号**：K210 与 VF2 使用 `_start`，而 QEMU 使用 `_entry` <sup>[13](#mod-boot-ref-13)</sup>。这是因为 QEMU `virt` 机器的 OpenSBI 固件跳转到 `_entry` 而非 `_start`（与 BBL 的遗留约定有关）。这意味着链接脚本必须为不同平台提供不同的入口符号，增加了维护复杂度。
- **栈容量**：K210 与 QEMU 的 `boot_stack` 均为 `4096 * 4 * 2 = 32768` 字节，即支持 2 个 hart 各 16 KiB <sup>[12](#mod-boot-ref-12)</sup>；VF2 则为 `4096 * 4 * 4 = 65536` 字节，适配其 4 个 hart <sup>[15](#mod-boot-ref-15)</sup>。当 hart 数超出分配时，栈指针会越界写入 `.bss` 后继区域，可能破坏内核静态数据。
- **空闲指令**：K210 与 QEMU 使用 `j loop` 忙等，而 VF2 使用 `wfi` 等待中断 <sup>[14](#mod-boot-ref-14)</sup>。`wfi` 可以降低功耗，但要求后续中断能唤醒 hart；若中断控制器未初始化，hart 可能永久睡眠。这暗示 VF2 的 `main` 函数后期依赖中断驱动调度。

K210 入口中有一段被注释掉的代码：`csrr t1, mhartid; mv tp, t1` <sup>[11](#mod-boot-ref-11)</sup>。这显示早期版本曾尝试在汇编层设置 `tp` 寄存器作为 hart-local 存储基址，但后来被放弃（可能是将 tp 初始化移到了 C 代码中）。遗留的注释增加了代码噪声，但未影响功能。

### 5. 边角细节与不足

**<sup>[2](#mod-boot-ref-2)</sup> 浮点寄存器未保存**：`la_exception_entry` 仅保存通用寄存器，未保存 `$f0`-`$f31` 及 FCSR。若 `la_trap_handler` 或用户态负载使用浮点，状态会损坏。

**<sup>[10](#mod-boot-ref-10)</sup> 未验证 hart ID 上界**：栈偏移公式 `(a0+1)<<14` 未做边界检查。在 K210 上若固件传入 `a0=3`，栈指针将超出 `boot_stack` 范围，写入相邻内存。

**<sup>[5](#mod-boot-ref-5)</sup> 用户态探针的页表缺失**：`la_enter_user_probe` 切换到用户模式 (PLV=3) 但未加载独立页表。用户代码运行在内核页表下，虽然简化了设计，但丧失了内存隔离——用户态可通过构造指针访问内核数据。

**<sup>[8](#mod-boot-ref-8)</sup> 段表符号的链接时脆弱性**：`la_user_segment_table` 引用了大量外部符号（如 `la_busybox_glibc`）。若某符号在链接时未定义且无弱定义回退，链接器将报错；若使用了弱符号且回退为零地址，用户态写零地址将触发异常。

**<sup>[11](#mod-boot-ref-11)</sup> 遗留注释/死代码**：K210 入口中被注释的 `tp` 初始化代码表明一次未完成的清理，是典型的工程腐化信号。

**<sup>[1](#mod-boot-ref-1)</sup><sup>[13](#mod-boot-ref-13)</sup><sup>[14](#mod-boot-ref-14)</sup> 入口返回无诊断**：所有入口在 C 函数返回后均进入无限循环（`idle 0` 或 `j loop`），不输出任何诊断信息。在早期启动失败时，调试者只能依赖 JTAG 或外部仿真器。

**<sup>[3](#mod-boot-ref-3)</sup> CSR 传递约定未文档化**：`la_trap_handler` 需通过 `r4` 返回 ERA 值是隐式约定，未在注释中说明，容易在重构时被破坏。

### ⚠ 开放问题

- la_exception_entry 未保存浮点寄存器 (32 FPR + FCSR)，一旦 la_trap_handler 或用户态使用 FP，状态静默损坏 [ref:2]
- xv6 入口的 hart ID 未做边界检查，hart 数超预期时栈指针越界写入内核静态数据 [ref:10]
- la_enter_user_probe 切换到 PLV=3 但未加载独立用户页表，用户代码运行在内核地址空间，无内存隔离 [ref:5]
- K210 入口遗留被注释的 tp 初始化代码，表明未完成的清理 [ref:11]
- 所有入口在 C 函数返回后静默挂起 (idle/j loop)，无任何诊断输出，早期启动失败极难定位 [ref:1][ref:13][ref:14]
- la_trap_handler 通过 r4 返回 ERA 的隐式约定无文档注释，重构时易被破坏 [ref:3]
- la_user_segment_table 引用大量外部链接符号，若任一符号缺失或弱符号回零，将导致运行时异常 [ref:8]

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 基于 Sv39 三级页表的虚拟内存管理，核心包括：内核/用户页表创建与映射 (walk/mappages)、物理页引用计数分配器 (kmem)、写时拷贝 fork (PTE_COW)、惰性页分配 (lazy_alloc_page)、以及每进程独立内核页表 (kpagetable)。与 xv6-riscv 原版相比，新增 COW、惰性分配、双页表与引用计数，支持 K210/QEMU/VisionFive2 多平台条件编译。

### 1. 核心抽象与外部依赖

内存管理模块围绕三个核心抽象：**页表** (`pagetable_t`，即 `uint64*`，指向 Sv39 三级页表根 <sup>[1](#mod-mm-ref-1)</sup>)、**物理页分配器** (`kmem` 结构体，包含自由链表、自旋锁与引用计数数组 <sup>[2](#mod-mm-ref-2)</sup>)、以及**每进程双页表** (用户页表 `p->pagetable` 与内核页表 `p->kpagetable`，后者通过 `proc_kpagetable` 从全局 `kernel_pagetable` 拷贝顶层映射并附加私有内核栈 <sup>[6](#mod-mm-ref-6)</sup>)。

地址空间布局由 `memlayout.h` 集中定义：`KERNBASE`、`PHYSTOP`、`TRAMPOLINE`、`TRAPFRAME`、`VKSTACK` 等常量 <sup>[3](#mod-mm-ref-3)</sup>。平台差异 (K210 / QEMU / VisionFive2) 通过 `#ifdef` 处理 UART、CLINT、PLIC 等设备物理地址的映射偏移量 `VIRT_OFFSET` (0x3F00000000)，使得内核可直接访问 MMIO 寄存器 <sup>[3](#mod-mm-ref-3)</sup>。

### 2. 关键设计取舍：COW、惰性分配与双页表

**写时拷贝 (COW) fork**：`uvmcopy` 在复制进程地址空间时，不立即拷贝物理页，而是将父子双方的 PTE 标记为只读并设置 `PTE_COW` 标志 (利用 RISC-V PTE 的 RSW 保留位) <sup>[4](#mod-mm-ref-4)</sup>。随后调用 `kaddref` 增加物理页引用计数。当任一进程写入 COW 页时，触发 store page fault (scause=15)，`uvm_handle_page_fault` 转调 `cow_alloc_page`：若引用计数为 1，则直接清除 COW 位并恢复可写；否则分配新页、拷贝内容、递减旧页引用 <sup>[5](#mod-mm-ref-5)</sup>。这一设计用更精细的引用计数换取 fork 的 O(1) 延迟，避免了原版 xv6 的全量拷贝，但代价是增加了一个自定义 PTE 标志位与故障路径的复杂度。

**惰性页分配**：`lazy_alloc_page` 在缺页时按需分配物理页，支持堆的稀疏增长 (地址低于 `p->brk`) 与用户栈的动态扩展 (地址在 `p->brk` 和 `p->trapframe->sp` 之间) <sup>[7](#mod-mm-ref-7)</sup>。然而，该函数仅凭地址范围判断是否为“合法”惰性分配——任何落在该区间的缺页都被视为需分配新页，这可能会掩盖真正的非法访问（如野指针恰好落在该范围）。同时，读取 `p->brk` 和 `p->trapframe->sp` 时未持有 `p->lock`，存在竞态风险 <sup>[7](#mod-mm-ref-7)</sup>。

**每进程内核页表**：`proc_kpagetable` 将全局 `kernel_pagetable` 的顶层 512 个 PTE 完整复制到新页表，然后映射独立的 2 页内核栈 (`VKSTACK` + `KSTACK_PAGES`)，取代原版 xv6 的共享内核页表 <sup>[6](#mod-mm-ref-6)</sup>。这使得用户内存可以同时映射到内核页表（用于 `copyin`/`copyout` 直接物理地址访问），但也意味着每次 `uvmalloc`/`uvmcopy` 必须同步操作两个页表，加倍映射开销且容易在错误路径上不一致 <sup>[8](#mod-mm-ref-8)</sup>。

### 3. 跨模块协同：与陷阱处理、进程管理的交互

缺页异常从陷阱入口 (trampoline) 进入后，最终调用 `uvm_handle_page_fault`，依据 `scause` 和 PTE 标志分发到 COW 处理或惰性分配 <sup>[9](#mod-mm-ref-9)</sup>。这一函数是内存管理与 CPU 异常机制的桥梁。

`ensure_user_page` 被 `copyout`/`copyin`/`copyinstr` 等内核-用户传输函数调用 <sup>[10](#mod-mm-ref-10)</sup>：在逐页执行 `memmove` 前，检查 PTE 是否存在且用户可访问，若缺失或标记为 COW，则触发 `uvm_handle_page_fault` 补齐页面。这消除了内核对用户内存的预映射要求，但引入了一个隐式假设：缺页处理可能阻塞 (分配物理页时可能睡眠等待)，因此这些传输函数只能在进程上下文中调用，不可在中断上下文中使用。

物理页引用计数 (`kaddref`/`kgetref`/`kfree` 递减到零才释放) 是 COW 实现的基础 <sup>[11](#mod-mm-ref-11)</sup>。`kalloc` 分配时设置引用计数为 1，`uvmcopy` 共享页面时递增，`cow_alloc_page` 通过 `kgetref` 判断是否为最后持有者来决定复用或拷贝 <sup>[5](#mod-mm-ref-5)</sup>。这一机制将内存管理器的职责从简单的分配/释放延伸到共享跟踪，要求所有页面使用者严格遵循“引用计数对称”的契约。

### 4. 边角细节与不足

- **COW 竞态窗口**：`uvmcopy` 先修改父进程 PTE 为 COW，再调用 `kaddref` 增加引用。若多核场景下另一 hart 在该窗口内触发 COW 缺页，`cow_alloc_page` 会看见引用计数为 0 (尚未递增) 而错误复用页面，导致数据损坏 <sup>[4](#mod-mm-ref-4)</sup>。当前单核运行无影响，但属潜在 SMP 缺陷。

- **惰性分配策略粗糙**：`lazy_alloc_page` 仅凭 `va >= PGROUNDUP(p->brk) && va < PGROUNDDOWN(p->trapframe->sp)` 判断是否为合法增长，任何位于该区间的缺页均会分配新页，可能隐藏 use-after-free 或野指针错误 <sup>[7](#mod-mm-ref-7)</sup>。

- **`uvmalloc` 错误恢复路径脆弱**：当 `mappages(kpagetable, ...)` 失败时，清理代码 `vmunmap(pagetable, oldsz, npages + 1, 1)` 与 `vmunmap(kpagetable, oldsz, npages, 0)` 中 `npages+1` 与 `npages` 的差异依赖于当前页映射失败的精确语义，未来修改极易引入双重释放或泄漏 <sup>[8](#mod-mm-ref-8)</sup>。

- **TLB 刷新缺失**：`vmunmap` 自身不执行 `sfence_vma`，依赖调用方刷新。`uvmcopy` 成功路径末尾有 `sfence_vma`，但错误路径 (`goto err`) 未刷新 TLB。若错误发生在多核场景下，其他 hart 可能仍持有已释放页面的陈旧 TLB 项 <sup>[12](#mod-mm-ref-12)</sup>。

- **`copyin2`/`copyout2` 整数溢出**：边界检查 `dstva + len > sz` 在 `len` 极大时可能回绕，绕过检查 <sup>[13](#mod-mm-ref-13)</sup>。虽 `sz` 受 MAXVA 限制，仍为经典安全漏洞模式。

- **`vmprint` 无边界遍历**：函数通过 `pte++` 遍历各级页表，若 PTE 指向非法物理地址，循环可能越界读取并触发异常，缺少防御性编程 <sup>[14](#mod-mm-ref-14)</sup>。

- **`kvmfree` 部分未完成**：`kvmfreeusr` 仅释放用户地址范围 (`PX(2, MAXUVA)`) 内的下级页表，但对于 `proc_kpagetable` 创建的 2 页内核栈映射，其错误路径 (`goto fail`) 调用 `kvmfree(kpt, 1)` 会通过 `vmunmap` 释放栈物理页，但 `kfreewalk` 仅递归释放无叶子页表，若下级页表仍有残留映射会跳过而非 panic，与 `freewalk` 的 `panic("freewalk: leaf")` 行为不一致 <sup>[6](#mod-mm-ref-6)</sup>。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以静态进程表 `struct proc proc[NPROC]` 与每 CPU 结构 `struct cpu cpus[NCPU]` 为核心, 实现进程生命周期管理 (fork/exit/wait/kill) 与三级可切换调度策略 (SCHED_RR / SCHED_PRIORITY / SCHED_MLFQ). 与 xv6-riscv 原版相比, 该模块显著扩展了调度器: 支持运行时通过 `proc_set_sched_algo()` 切换算法; 新增 MLFQ 动态优先级调整; 引入 `clone_proc` 支持用户态指定子进程栈; 并将大量自动化测试用例 (init_tests/lua/busybox/libctest) 直接嵌入 init 进程执行流. 进程内核栈采用 `VKSTACK` 固定虚拟地址 + 每进程独立 `kpagetable` 映射的方案, 取代原版 per-process 固定偏移的 KSTACK(i) 分配.

### 1. 核心抽象: 进程结构与调度器框架

进程描述符 `struct proc` 聚合了调度状态 (`state`, `chan`, `killed`)、内存空间 (`pagetable`, `kpagetable`, `sz`, `brk`)、文件表 (`ofile[]`, `cwd`)、调度参数 (`timeslice`, `slice_remain`, `priority`, `mlfq_cpu_ticks`, `mlfq_sleep_ticks`) 以及上下文切换所需的 `trapframe` 和 `context` <sup>[1](#mod-task-ref-1)</sup>. 进程表是编译期定长的全局数组 `proc[NPROC]`, 分配器 `allocproc()` 线性扫描寻找 `UNUSED` 槽位——这是 xv6 经典的 O(N) 分配策略, 简单但扩展性受限 <sup>[2](#mod-task-ref-2)</sup>.

每 CPU 结构 `struct cpu` 持有当前运行进程指针 `proc` 和调度器上下文 `context`, 以及中断关闭嵌套深度 `noff` 与中断使能状态 `intena` <sup>[3](#mod-task-ref-3)</sup>. `mycpu()` 通过读取 RISC-V `tp` 寄存器获取 CPU ID, 要求调用者在关中断下执行以避免迁移竞争 <sup>[4](#mod-task-ref-4)</sup>.

进程状态机定义五态: `UNUSED → RUNNABLE → RUNNING → {SLEEPING, ZOMBIE}`. 其中 `SLEEPING` 进程通过 `chan` 指针等待事件, `ZOMBIE` 则由父进程 `wait()` 收割 <sup>[5](#mod-task-ref-5)</sup>. 状态转换均在持有 `p->lock` 自旋锁的前提下进行, 保证调度器、系统调用、中断处理三者间的互斥.

### 2. 三级调度策略: 从 RR 到 MLFQ 的工程取舍

调度器入口 `scheduler()` 是一个无限循环: 开中断 → `pick_next_proc()` 选进程 → 切页表到目标进程的 `kpagetable` → `swtch(&c->context, &p->context)` → 回来后切回内核页表并释放进程锁 <sup>[6](#mod-task-ref-6)</sup>. 无就绪进程时执行 `wfi` 等待中断.

`pick_next_proc()` 根据全局 `sched_algo` 分发到两个选择函数 <sup>[7](#mod-task-ref-7)</sup>. SCHED_RR 使用 `pick_rr_proc()`: 从 `rr_cursor` 开始环形扫描, 取第一个 `RUNNABLE` 进程, 并推进游标避免饥饿 <sup>[8](#mod-task-ref-8)</sup>. SCHED_PRIORITY 和 SCHED_MLFQ 共用 `pick_priority_proc()`: 全表扫描, 取 `priority` 数值最小 (即优先级最高) 的进程, 打破平局时兼顾游标位置 <sup>[9](#mod-task-ref-9)</sup>. MLFQ 的动态优先级调整由 `mlfq_maybe_update_locked()` 在时钟中断中完成: 以 `MLFQ_WINDOW` (5 个 tick) 为周期, 若 CPU 时间多于睡眠时间则降级 (`priority+1`), 反之升级 (`priority-1`) <sup>[10](#mod-task-ref-10)</sup>.

相比原版 xv6-riscv 仅有的简单 RR, 这套三级调度有三个值得注意的取舍: (1) **运行时切换** — `proc_set_sched_algo()` 遍历全部进程重置 `mlfq_cpu_ticks`/`mlfq_sleep_ticks` 或补正 `slice_remain`, 代价是 O(NPROC) 的持锁遍历 <sup>[11](#mod-task-ref-11)</sup>; (2) **MLFQ 简化** — 仅一级队列 + 动态优先级, 无多队列的严格老化机制, I/O 密集型进程的响应延迟取决于窗口大小; (3) **优先级语义** — 低数值 = 高优先, 范围 `[PRIO_MIN=1, PRIO_MAX=31]`, 与 Linux nice 值相反, 是教学内核常有的约定 <sup>[12](#mod-task-ref-12)</sup>.

### 3. 进程生命周期与上下文切换的协同

**创建**: `fork()` 和 `clone_proc()` 统一汇聚到 `fork_at(child_stack)`. 该函数通过 `allocproc()` 获取进程槽位, 调用 `uvmcopy()` 复制父进程用户态内存到子进程的 `pagetable` 和 `kpagetable`, 复制 `trapframe` 并将子进程的 `a0` 寄存器设为 0 (fork 返回值), 复制文件描述符和当前目录 <sup>[13](#mod-task-ref-13)</sup>. 区别在于 `clone_proc` 允许指定子进程的用户栈指针 `sp`, 这为 `clone` 系统调用提供了基础.

**退出与收割**: `exit(status)` 关闭所有文件、释放 `cwd`, 将子进程过继给 `initproc`, 唤醒父进程, 置 `ZOMBIE` 后调用 `sched()` 切换到调度器 <sup>[14](#mod-task-ref-14)</sup>. `wait(addr)` 扫描进程表寻找 `ZOMBIE` 子进程, 将退出状态通过 `copyout2` 写入用户空间, 调用 `freeproc()` 释放资源 <sup>[15](#mod-task-ref-15)</sup>. `reparent()` 绕过锁顺序死锁的经典注释表明设计者对锁协议有清醒认识: 只检查 `pp->parent` 而不持锁, 因为只有父进程能修改该字段 <sup>[16](#mod-task-ref-16)</sup>.

**睡眠/唤醒**: `sleep(chan, lk)` 实现原子释放锁并睡眠的经典原语: 先获取 `p->lock`, 释放传入锁 `lk`, 置 `SLEEPING`, 调用 `sched()`, 醒来后重新获取 `lk` <sup>[17](#mod-task-ref-17)</sup>. `wakeup(chan)` 遍历进程表将匹配 `chan` 的 `SLEEPING` 进程置为 `RUNNABLE` <sup>[18](#mod-task-ref-18)</sup>. `wakeup1(p)` 是 `exit`/`kill` 内部使用的变体, 要求调用者持有 `p->lock`, 且仅当 `p->chan == p` (即 `wait()` 中以自身为 channel 的睡眠) 时才唤醒 <sup>[19](#mod-task-ref-19)</sup>.

**上下文切换**: `sched()` 必须在持有 `p->lock` 且中断关闭的前提下调用, 检查 `noff==1` (恰好一层锁)、`state != RUNNING`、中断已关闭, 然后执行 `swtch(&p->context, &mycpu()->context)` 切换到调度器上下文 <sup>[20](#mod-task-ref-20)</sup>. 调度器在 `swtch` 返回后释放进程锁, 完成一次完整的调度握手.

**时钟协同**: `proc_on_timer_tick()` 由时钟中断调用, 对 RR 递减 `slice_remain`, 对 MLFQ 累加 `mlfq_cpu_ticks` 并触发优先级重算 <sup>[21](#mod-task-ref-21)</sup>. `proc_should_yield_on_tick()` 判断是否需要让出 CPU, 对 RR 检查 `slice_remain <= 0` <sup>[22](#mod-task-ref-22)</sup>. 这种将调度决策分散到时钟中断中的做法, 比原版 xv6 的 `yield()` 仅在 `usertrap` 中调用更精细.

### 4. 边角细节与不足

**VKSTACK 的隐式约定** <sup>[2](#mod-task-ref-2)</sup>: `allocproc()` 将 `p->kstack` 设为固定 `VKSTACK`, 而非像注释掉的 `procinit()` 代码那样为每个进程分配独立的 `KSTACK(i)`. 这意味着多进程共享同一内核栈虚拟地址, 隔离性完全依赖 `kpagetable` 为该虚拟地址映射不同物理页. 若 `kpagetable` 分配或映射失败, 多进程将运行在同一物理栈上, 导致不可预测的相互覆盖. 该路径缺乏运行时检查.

**procinit 的半迁移** <sup>[23](#mod-task-ref-23)</sup>: 约 20 行内核栈分配代码被整体注释, 但 `procinit` 仍保留循环初始化锁的逻辑. 这种“半迁移”暗示该项目正在从固定偏移方案过渡到 per-process 页表方案, 但过渡不完整, 遗留代码增加了阅读理解难度.

**测试逻辑侵入内核** <sup>[24](#mod-task-ref-24)</sup>: `run_next_init_test()` 及配套的 `init_tests[]`/`lua_scripts[]`/`busybox_cmds[]`/`libctest_cases[]` 数组（合计约 200 行）直接嵌入 `proc.c`, 由 `initproc` 的 `exit()` 路径驱动. 这使内核进程管理模块混杂了用户态测试编排职责, 且大量 `static` 全局变量 (`init_test_idx`, `lua_pending`, `busybox_cmd_idx` 等) 管理状态机, 测试失败时难以调试.

**整数溢出风险** <sup>[25](#mod-task-ref-25)</sup>: `growproc()` 对正增长使用 `sz + (uint64)n < sz` 检测溢出, 正确. 但 `n` 为 `int` 类型, 调用方传入用户可控的 `n` 若为 `INT_MIN` 且路径进入正增长分支 (虽然通常不会), 强制转换后为巨大正数. 负增长走 else 分支调用 `uvmdealloc`, 尺寸参数依赖该函数内部边界检查.

**slice_remain 的 fork 语义** <sup>[13](#mod-task-ref-13)</sup>: `fork_at()` 中子进程的 `slice_remain` 被设为 `p->timeslice` 而非 `p->slice_remain`, 即子进程获得完整时间片. 这保证了 fork 后的公平性, 但也意味着进程可以通过反复 fork 来重置时间片, 存在被恶意利用的可能.

**MLFQ 窗口硬编码** <sup>[10](#mod-task-ref-10)</sup>: `MLFQ_WINDOW` 固定为 5 ticks, 无系统调用或配置接口可调整. 对于 I/O 密集型负载, 5 ticks 可能使优先级提升滞后; 对于 CPU 密集型负载, 降级可能过于激进.

**pid 回绕未处理** <sup>[26](#mod-task-ref-26)</sup>: `allocpid()` 单调递增 `nextpid`, 无回绕重用或溢出检查. 在长时间运行系统中 `nextpid` 溢出到负数 (作为有符号 `int`), 将产生非法 PID.

**procdump 无锁** <sup>[27](#mod-task-ref-27)</sup>: `procdump()` 遍历进程表时不持锁, 注释称“避免加深卡死”, 但读取 `p->state` 和 `p->pid` 时可能与并发 `exit`/`freeproc` 竞争, 导致打印出中间态数据.

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块实现类 Unix VFS 抽象，通过 struct file、struct dirent 和文件描述符管理层，统一支持 FAT32 与 ext4 两种后端。核心数据结构包括全局文件表 ftable、磁盘目录项缓存 ecache/ext4_cache 以及描述文件自身元数据的 struct dirent。除标准文件 I/O 外，还提供了 pipe、null、zero 等特殊设备。相比原版 xv6，扩展了 ext4 只读支持、内置虚拟文件覆层（用于测试），以及 FAT32 长文件名、目录项缓存等特性，但写操作在 ext4 上未实现。

### 1. 核心抽象与数据结构

文件系统层的核心抽象围绕 `struct file` 和 `struct dirent` 构建。`struct file` 表示一个打开的文件，可以是普通文件、目录、管道或设备，通过类型字段 `type` 区分<sup>[1](#mod-fs-ref-1)</sup>。每个进程通过文件描述符表（在 proc 结构中，未直接展示）引用这些 file 对象，通过引用计数管理生命周期<sup>[2](#mod-fs-ref-2)</sup>。`struct dirent` 则是“目录项”的内存表示，不仅用于目录本身，也用于普通文件，它直接持有底层文件系统的元数据：起始簇号、当前簇号、文件大小、属性等<sup>[3](#mod-fs-ref-3)</sup>。FAT32 和 ext4 共用同一个 `struct dirent` 定义，但内部字段含义有微小差异——例如 ext4 使用 `first_clus` 字段存储 inode 号，并通过特殊高位标记虚拟文件<sup>[4](#mod-fs-ref-4)</sup>。

全局文件表 `ftable` 是一个静态数组，包含 `NFILE` 个 `struct file`，所有打开文件均从此分配<sup>[5](#mod-fs-ref-5)</sup>。为避免并发冲突，访问该表需持有自旋锁 `ftable.lock`。类似地，FAT32 的目录项缓存 `ecache` 和 ext4 的 `ext4_cache` 也使用自旋锁保护各自的双向链表，实现 LRU 淘汰策略<sup>[6](#mod-fs-ref-6)</sup><sup>[7](#mod-fs-ref-7)</sup>。这种设计将内存缓存与底层磁盘 I/O 分离，使同一份缓冲区块（`struct buf`）可用于文件数据和目录项的读写。

文件系统初始化时先尝试识别 FAT32 卷；若失败则探测 ext4。一旦 ext4 初始化成功，全局标志 `ext4_active()` 将返回真，后续所有文件操作都会分流到 ext4 对应的函数<sup>[8](#mod-fs-ref-8)</sup>。这种运行时多后端支持比原版 xv6 单一文件系统更灵活，但增加了每个接口的条件分支，也埋下了 ext4 写功能未实现的隐患。

### 2. 跨模块协同与数据流

一次典型的 `read` 系统调用从用户空间指针开始，经过 `sys_read`（未在片段中）获取 `struct file`，然后调用 `fileread`。`fileread` 根据文件类型分发：管道走 `piperead`，设备走 `devsw` 表，普通文件则调用 `eread`（FAT32 路径）或 `ext4_eread`<sup>[9](#mod-fs-ref-9)</sup>。这里有一个关键细节：`eread` 和 `ewrite` 都要求调用者已经持有 `dirent` 的睡眠锁，`fileread` 在调用前通过 `elock` 获取该锁，保证了并发安全性，但锁粒度较粗（整个读操作期间持有）<sup>[10](#mod-fs-ref-10)</sup>。

`eread` 消耗 `off` 和 `n`，通过 `reloc_clus` 在 FAT 链上移动当前簇指针，然后调用 `rw_clus` 完成实际磁盘 I/O<sup>[11](#mod-fs-ref-11)</sup>。`reloc_clus` 的设计值得注意：它维护 `cur_clus` 和 `clus_cnt` 来跟踪已遍历的簇数，避免了每次读取都从文件头开始扫描，这是性能上的优化。然而，`clus_cnt` 仅在向前移动时递增，若用户执行随机读且偏移回退，则会触发重新遍历，效率可能下降。

写操作路径更为复杂：`ewrite` 在偏移超出当前文件大小时自动扩展文件，必要时通过 `reloc_clus` 分配新簇。它通过在 FAT 表中写入链指针实现动态增长，但缺少空闲簇缓存，`alloc_clus` 每次都从头扫描 FAT 表，这在大型卷上可能成为瓶颈<sup>[12](#mod-fs-ref-12)</sup>。此外，`ewrite` 在写入成功后才更新 `file_size`，若中途失败可能导致部分写入但大小不变的不一致状态。

目录操作依赖 `dirlookup` 和 `enext` 等函数。`enext` 用于遍历目录项，跳过空条目（标记为 0xe5 或 0x00），并维护 `count` 参数以返回跳过的条目数<sup>[13](#mod-fs-ref-13)</sup>。但 `enext` 的实现并不在提供的片段中（可能在 fat32.c 后半部分），需要进一步阅读。同样，`dirlookup` 负责在目录中查找指定文件名，返回对应的 `struct dirent`（可能来自缓存）。这些函数与 `ealloc`（创建新目录项）、`emake`（写入磁盘目录项）协同完成文件系统的命名操作。

### 3. 关键设计取舍

**FAT32 vs. ext4 的运行时选择**：系统在 `fat32_init` 中读取 BPB，若签名非 “FAT32” 则尝试 ext4 探测。一旦 ext4 挂载成功，后续所有操作都切换到 ext4 路径，例如 `eread` 和 `ewrite` 的开头都有 `if (ext4_active()) return ext4_eread(...)` 这样的分流语句<sup>[14](#mod-fs-ref-14)</sup>。这种设计避免了在高层使用函数指针表的复杂性，代价是每个入口函数都需要显式判断，且 ext4 只读支持不完整（`ext4_ewrite` 仅返回 0 或 -1？需要验证）。

**虚拟文件覆层**：ext4 实现中包含了一个“虚拟文件”系统，通过 `virtual_spec` 函数判断文件名（如 `test_mmap.txt`），直接返回内存中的数据，或创建仅存在于内存的目录项（`dirty=1` 且 `first_clus` 高位标记）<sup>[15](#mod-fs-ref-15)</sup>。这显然是用于测试目的，例如 `test_chdir` 目录使得 `chdir` 等系统调用可以在无磁盘 ext4 卷的情况下验证。它通过 `virtual_lookup` 和 `virtual_create` 实现，绕过了真正的 ext4 io。这种覆层允许在 QEMU 等环境中快速验证高层语义，但增加了 ext4 实现的混乱度——真正的 ext4 读写函数与虚拟函数交织，且虚拟文件只能创建不能删除，可能造成资源泄漏。

**缓存策略**：FAT32 的 `ecache` 使用 LRU 双向链表，在 `eget` 中寻找命中或未使用的槽位，并维护 `parent` 指针作为快速校验（因为 FAT32 无 inode 号，同名文件可能出现在不同目录）<sup>[16](#mod-fs-ref-16)</sup>。ext4 的 `ext4_cache` 则更简单：直接线性扫描，依赖 `dirty` 标志区分虚拟文件。两者都未实现脏条目回写（FAT32 的回写可能在 `eupdate` 中，但未展示）。LRU 算法的实现在释放条目时仅检查 `ref == 0`，未考虑 `dirty` 位，可能导致脏数据在缓存中被覆盖后丢失（尽管 FAT32 似乎习惯在写入后立即更新磁盘）。

**设备抽象**：`devsw` 数组提供了简单的设备驱动接口，每个设备有 `read` 和 `write` 函数指针。当前实现了 `null` 和 `zero` 设备，前者读写即丢弃/返回 0，后者返回零字节流<sup>[17](#mod-fs-ref-17)</sup>。这种设计使管道（`pipe`）也可视为一种设备，但在代码中管道是单独处理的，这可能是因为管道有独特的阻塞唤醒逻辑。

### 4. 边角细节与不足

尽管文件系统实现了丰富的功能，但多处代码暴露出不完整或危险的做法：

1. **ext4 写操作未实现**：`ext4_ewrite` 在头文件中声明，但在 ext4.c 中很可能只是一个桩函数，返回 -1 或 0。提供的片段未显示实现，但在 `ewrite` 中如果 `ext4_active()` 为真，则调用 `ext4_ewrite`，如果其实现不完整，会导致所有写操作失败。这限制了 ext4 后端只能用于只读场景。<sup>[14](#mod-fs-ref-14)</sup>
2. **未对齐访问风险**：在 K210 平台上，未对齐读写会触发异常，因此代码中多处使用 `memmove` 替代直接指针解引用，例如 `fat32_init` 中读取 BPB 字段时注释了 `avoid misaligned load on k210`<sup>[18](#mod-fs-ref-18)</sup>。但并非所有地方都遵守了该规则——例如 `read_fat` 中 `*(uint32 *)(b->data + fat_offset_of_clus(cluster))` 可能仍然未对齐，因为 `fat_offset_of_clus` 基于 `(cluster << 2) % byts_per_sec`，当 `byts_per_sec=512` 时该值总是 4 的倍数，安全；但如果未来更改扇区大小，则可能产生未对齐访问。<sup>[19](#mod-fs-ref-19)</sup>
3. **缓冲区溢出风险**：`formatname` 对文件名进行修饰后未检查长度是否超过 `FAT32_MAX_FILENAME`，尽管调用处如 `ealloc` 会使用 `strncpy` 并强制截断，但 `formatname` 自身操作的缓冲区是传入的指针，长度未知。如果用户传入超长路径名，可能引起栈溢出。<sup>[20](#mod-fs-ref-20)</sup>
4. **整数溢出**：`eread` 和 `ewrite` 中都有 `if (off + n < off)` 检查，试图防止溢出，但 `off` 和 `n` 都是 `uint`，在加法前应确保 `n` 不会过大。`ewrite` 还检查 `(uint64)off + n > 0xffffffff`，这暗示文件大小被限制在 4G 以内，但 FAT32 理论上可支持 4G-1 的文件，这里的判断可能误伤合法写入。<sup>[11](#mod-fs-ref-11)</sup>
5. **锁的正确性**：`eget` 在持有 `ecache.lock` 自旋锁时可能会睡眠（如果 `ep->lock` 睡眠锁被其他进程持有？实际上 `eget` 并不获取 `ep->lock`，它只是分配缓存条目并返回，但调用者需要 `elock`）。然而，`eget` 中当找到一个空闲条目时，会 `release(&ecache.lock)` 然后返回，但在释放锁和返回之间没有保护，可能两个 CPU 同时拿到同一个空闲条目，导致竞争。虽然 `ecache.lock` 释放后可能存在并发问题，但由于条目状态在释放前已设置为 `ref=1`，且其他 CPU 的 LRU 查找会跳过该条目，风险较低。但若同时多个 CPU 执行 `eget` 并都发现同一个条目 `ref==0`，则可能发生双重分配。因为释放锁发生在设置 `ref` 之后，但设置 `ref` 时仍持有锁，所以只有第一个 CPU 将其设为 1 并释放锁，第二个 CPU 看到 `ref==0` 时实际上可能已经被第一个 CPU 修改——等等，释放锁后才返回，但设置 `ref` 是在持有锁时完成的。所以如果两个 CPU 同时进入 for 循环检查 `ep->ref == 0`，可能都认为同一个条目可用（因为 ref 还未被修改），但持有锁的 CPU 会先修改 `ref` 为 1，另一个 CPU 进入后看到 `ref` 已经为 1 而跳过。所以不会冲突。但释放锁后才返回，中间无问题。然而，在 ext4_cache 里，`virtual_create` 和 `alloc_entry` 的分配算法没有考虑 `valid` 或 `dirty` 状态的复杂情况，可能分配出仍在使用的条目。<sup>[7](#mod-fs-ref-7)</sup>
6. **未实现的 FAT 缓存**：注释中提到 “here should be a cache layer for FAT table, but not implemented yet.” 这表明开发者意识到频繁读 FAT 扇区的性能问题，但尚未实现。<sup>[12](#mod-fs-ref-12)</sup>
7. **ext4 的虚拟文件容量硬编码**：`EXT4_VFILE_CAP` 设为 16384，但写入虚拟文件时没有边界检查，如果写入超过该大小会溢出栈上全局数组。<sup>[15](#mod-fs-ref-15)</sup>

### 5. 与原版 xv6 的对比扩展

与原版 xv6-riscv 相比，本文件系统有显著增强：
- **双文件系统支持**：原版仅实现了一种简单的日志文件系统，这里同时支持 FAT32 和 ext4，且通过运行时探测自动选择。
- **长文件名**：FAT32 实现支持 VFAT 长文件名（最多 255 字符），而原版 xv6 文件名限制为 14 字符。这体现在 `generate_shortname`、`emake` 中的长名条目生成等逻辑。
- **ext4 只读能力**：新增了 ext4 的超级块、块组描述符、inode、extent 树解析，能读取 ext4 文件系统的完整目录结构，但写操作未实现。
- **虚拟文件测试框架**：通过内存覆层提供了与磁盘无关的测试支持，这在教学或 CI 环境中很有价值。
- **设备抽象**：`/dev/null` 和 `/dev/zero` 的实现使系统调用测试更完整。

尽管扩展很多，但代码质量参差不齐，ext4 与 FAT32 的集成略显生硬，全局变量和魔法数字较多，错误处理也常以 `panic` 代替恢复。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> xv6-k210 的唯一 IPC 机制是传统 Unix 管道: 一个 512 字节环形缓冲区, 通过一对 `struct file` (FD_PIPE) 融入 VFS, 由自旋锁 + sleep/wakeup 实现阻塞式读写. 与 xv6-riscv 原版的差异在于将 `copyin`/`copyout` 替换为 `copyin2`/`copyout2`, 适配 K210 平台内存访问.

### 1. 核心抽象: 以文件为壳的环形管道

管道 (`struct pipe`) 聚合了自旋锁、512 字节字符数组 `data[PIPESIZE]`、两个无符号偏移计数器 `nread`/`nwrite` 以及两端开闭标志 `readopen`/`writeopen` <sup>[1](#mod-ipc-ref-1)</sup>. 它不直接暴露给用户程序: `pipealloc` 分配两个 `struct file`, 分别标记为 `FD_PIPE` 并设置单向可读/可写, 二者共享同一个 `pipe` 指针 <sup>[2](#mod-ipc-ref-2)</sup>. 这种设计让 `read`/`write` 系统调用无需感知底层实现.

与 xv6-riscv 原版相比, 本仓库将 `copyin(pr->pagetable, ...)` 替换为 `copyin2(&ch, addr+i, 1)` <sup>[4](#mod-ipc-ref-4)</sup>, 并同样替换了 `copyout`. 注释掉的旧代码表明 K210 移植阶段用户态内存访问接口发生了变更, 但旧路径的残留也暗示移植尚未完全收尾.

### 2. 同步模型: 持锁 sleep/wakeup 的代价

管道同步完全依赖 xv6 经典的 `sleep`/`wakeup` 机制. 写者在缓冲区满时 (`nwrite == nread + PIPESIZE`) 调用 `sleep(&pi->nwrite, &pi->lock)`, 睡眠前先唤醒可能阻塞的读者 <sup>[4](#mod-ipc-ref-4)</sup>. 读者在空管道且写端尚开时 (`nread == nwrite && writeopen`) 阻塞于 `&pi->nread` <sup>[5](#mod-ipc-ref-5)</sup>. 两侧关闭时 `pipeclose` 负责唤醒对方 <sup>[3](#mod-ipc-ref-3)</sup>.

关键取舍: 整个 I/O 操作全程持有 `pi->lock` 自旋锁. 优点是消除了生产者-消费者竞态, 缺点是即使传输 10 字节也会遍历循环多次, 且每字节伴随一次 `copyin2`/`copyout2` 调用. 在单核 K210 上这不会死锁, 但自旋锁持有时间与 `n` 成正比 — 若用户传入大 `n`, 系统事实上冻结 <sup>[4](#mod-ipc-ref-4)</sup>.

### 3. 跨模块协同

管道子系统虽仅约 120 行, 却牵涉五个内核模块: (a) **文件层** — `filealloc`/`fileclose` 管理管道两端的 file 对象, 管道生命周期与文件描述符绑定 <sup>[2](#mod-ipc-ref-2)</sup>; (b) **进程管理** — `myproc()` 获取当前进程以检查 `pr->killed` 标志, `sleep`/`wakeup` 本身也是进程调度原语 <sup>[4](#mod-ipc-ref-4)</sup><sup>[5](#mod-ipc-ref-5)</sup>; (c) **内存管理** — `kalloc` 分配整页物理内存 (却只用了 ~540 字节), `copyin2`/`copyout2` 执行用户内核之间的字节传输; (d) **同步原语** — `initlock`/`acquire`/`release` 保护管道内部状态. 任何一个子系统的实现变更 (例如 `copyin2` 改为可睡眠) 都会直接威胁管道正确性.

### 4. 边角细节与不足

1. **`uint` 计数器无限增长** <sup>[4](#mod-ipc-ref-4)</sup>: 满管判定 `nwrite == nread + PIPESIZE` 在 `nread` 接近 2³² 时加法溢出, 可能造成死锁. 尽管 4GB 数据不易触发, 但计数器永不回卷是设计缺陷.
2. **自旋锁持有期间调用 `copyin2`/`copyout2`** <sup>[4](#mod-ipc-ref-4)</sup><sup>[5](#mod-ipc-ref-5)</sup>: 当前假定这两个函数不睡眠. 若未来因缺页处理而睡眠, 将直接触发 "sleeping while holding spinlock" panic. 没有文档或断言保护这一假设.
3. **部分写入时错误码丢失** <sup>[4](#mod-ipc-ref-4)</sup>: `pipewrite` 在 `copyin2` 失败后 `break` 返回已写字节数 `i`. 调用者无法区分 "完整写入 n 字节" 与 "写入 i 字节后遇到错误". 标准做法是若 `i==0` 应返回 -1.
4. **`piperead` 被杀时不唤醒写者** <sup>[5](#mod-ipc-ref-5)</sup>: 读者在空管道上被 `kill` 后直接返回 -1, 未调用 `wakeup(&pi->nwrite)`. 虽然 `pipeclose` 最终会补救, 但写者可能在进程退出前被长时间悬挂.
5. **整页内存浪费** <sup>[2](#mod-ipc-ref-2)</sup>: `kalloc` 分配 4096 字节, 管道结构仅约 540 字节. 教学场景可接受, 但工程上应使用 slab 或对象缓存.
6. **`n` 参数无上限** <sup>[4](#mod-ipc-ref-4)</sup><sup>[5](#mod-ipc-ref-5)</sup>: 恶意传入 `INT_MAX` 可导致系统长期持有自旋锁, 实际上冻结. 应限制 `n` 不超过 `PIPESIZE` 或设置合理上界.
7. **注释掉的旧代码残留** <sup>[4](#mod-ipc-ref-4)</sup><sup>[5](#mod-ipc-ref-5)</sup>: `// if(copyin(pr->pagetable, ...))` 表明移植尚未完成. 若 `copyin2` 方案已确定, 应清理旧路径.

- `uint` 计数器无限增长, `nwrite == nread + PIPESIZE` 在 nread 接近 UINT_MAX 时加法溢出, 可能导致死锁.

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 本模块并非传统意义上的统一驱动框架，而是一组从 Canaan K210 SDK 移植的裸机外设驱动集合，以内存映射寄存器结构体 (dmac_t/spi_t) 为核心抽象，通过位域联合体实现类型安全的寄存器访问。DMAC 驱动封装了 6 通道 DMA 传输，SPI 驱动提供 FIFO 轮询与 DMA 两种传输模式。与 xv6 原版无驱动层相比，这是面向 RISC-V K210 开发板的完整硬件抽象层，但缺少统一的设备模型与资源管理。

### 1. 核心抽象：寄存器映射与传输模型

驱动层的核心抽象是两组内存映射寄存器结构体：`dmac_t` 及其子结构 `dmac_channel_t` 对应 DMA 控制器 <sup>[1](#mod-drivers-ref-1)</sup>，`spi_t` 对应 SPI 控制器 <sup>[2](#mod-drivers-ref-2)</sup>。两者均采用 `__attribute__((packed, aligned))` 保证与硬件寄存器布局严格一致，通过位域联合体 (`dmac_cfg_u_t` 等) 提供“读-改-写”模式下的类型安全操作。DMA 传输模型以通道为中心：`dmac_set_channel_param()` 配置 SAR/DAR/CTL/CFG 四个关键寄存器后，`dmac_set_single_mode()` 统一执行“清中断→禁通道→等空闲→设参数→使能中断→启通道”的标准序列 <sup>[3](#mod-drivers-ref-3)</sup>。SPI 侧则分化为 `_normal` (纯 FIFO 轮询) 与 `_dma` (DMA 驱动) 两套 API，后者通过 `spi_receive_data_normal_dma()` 将命令缓冲区与数据缓冲区分开处理，利用 `sysctl_dma_select()` 将 SPI 的 TX/RX 请求线绑定到指定 DMAC 通道 <sup>[4](#mod-drivers-ref-4)</sup>。

### 2. 关键设计取舍：中断/轮询混合与缓冲区策略

DMA 完成等待采用“中断通知 + 睡眠唤醒”的混合模式：`dmac_wait_done()` 直接调用 `dmac_wait_idle()`，后者检查 `chen` 寄存器判断通道是否空闲——若空闲则立即返回，否则通过 xv6 的 `sleep()` 在全局等待通道 `dmac_chan` 上阻塞 <sup>[5](#mod-drivers-ref-5)</sup>。中断处理函数 `dmac_intr()` 仅做两件事：清除通道中断状态，然后 `wakeup(dmac_chan)` 唤醒所有等待者 <sup>[6](#mod-drivers-ref-6)</sup>。这种设计的取舍很明显：**放弃了中断回调的精确匹配**（所有通道共享同一个 wait channel），换来的是与 xv6 调度器的最小耦合——不需要 PLIC 中断注册表或回调函数列表。SPI 层在 DMA 路径上同样追求简洁：`spi_send_data_standard_dma()` 用 `kalloc()` 分配临时缓冲区，将 8/16 位数据手动打包为 32 位以满足 DMA 宽度要求 <sup>[7](#mod-drivers-ref-7)</sup>，避免了为每种帧格式单独配置 DMA 描述符的复杂性，但引入了内核堆分配开销。

### 3. 跨模块协同：sysctl、PLIC 与 kalloc 的三角依赖

驱动层并非孤立存在。DMAC 初始化 `dmac_init()` 必须首先调用 `sysctl_clock_enable(SYSCTL_CLOCK_DMA)` 使能 DMA 外设时钟，然后通过置位 `dmac->reset` 并轮询等待复位完成 <sup>[8](#mod-drivers-ref-8)</sup>。SPI DMA 路径中，`spi_send_data_normal_dma()` 调用 `sysctl_dma_select()` 将 SPI 外设的 DMA 请求 (如 `SSI0_TX_REQ`) 路由到指定 DMAC 通道 <sup>[9](#mod-drivers-ref-9)</sup>，这依赖于 sysctl 模块对外设互联矩阵的正确配置。中断链路则从 PLIC 出发：外部 DMAC 中断触发 PLIC handler，最终调用 `dmac_intr()`，后者通过 `wakeup()` 将等待进程放回就绪队列——这是典型的“自底向上”通知路径 <sup>[6](#mod-drivers-ref-6)</sup>。内存管理方面，SPI 函数大量使用 `kalloc()`/`kfree()` 作为暂存区分配器，表明驱动运行在内核态、可直接访问内核堆，但也意味着每次 SPI 传输都可能触发堆分配，对实时性不友好。

### 4. 边角细节与不足

**<sup>[10](#mod-drivers-ref-10)</sup> 内存分配无 NULL 检查**：`spi_send_data_standard()`、`spi_send_data_standard_dma()`、`spi_receive_data_standard_dma()` 等多处直接对 `kalloc()` 返回值解引用，若内核堆耗尽则直接触发缺页异常。

**<sup>[11](#mod-drivers-ref-11)</sup> 硬编码地址与平台绑定**：`is_memory()` 通过三个硬编码地址范围判断指针是否指向“内存”，包括一个神秘常量 `0x50450040`——它实际上是某个外设的寄存器地址，却被当作内存类地址处理，一旦地址布局改变，DMA 流控与握手选择全部失效。

**<sup>[12](#mod-drivers-ref-12)</sup> 全局等待通道导致惊群**：所有 DMAC 通道共享同一个 `dmac_chan` wait channel，任一通道的中断都会唤醒所有等待者，虽然 `dmac_wait_idle()` 会重新检查条件，但在多进程并发 DMA 场景下会导致不必要的调度抖动。

**<sup>[13](#mod-drivers-ref-13)</sup> 已注释的大段死代码**：`spi.h` 中 `spi_send_data_multiple_dma`、`spi_dup_send_receive_data_dma`、`spi_slave_config` 等 6 个函数原型被整块注释 <sup>[14](#mod-drivers-ref-14)</sup>，`dmac.h` 中 `dmac_set_irq`、`dmac_irq_register` 也被注释 <sup>[15](#mod-drivers-ref-15)</sup>，说明 SPI 从机模式、全双工 DMA、链表传输等高级特性移植未完成。

**<sup>[16](#mod-drivers-ref-16)</sup> FIFO 忙等无限循环风险**：`spi_send_data_normal()` 末尾的 `while((spi_handle->sr & 0x05) != 0x04);` 没有超时机制，若硬件异常导致状态位永远不满足，内核将在此死循环。

**<sup>[17](#mod-drivers-ref-17)</sup> 注释掉的断言暴露工程债务**：SPI 函数中大量 `// configASSERT(...)` 被注释而非替换为运行时错误返回，表明原 SDK 的调试断言被简单禁用，调用者传入非法参数时行为完全未定义。

**<sup>[18](#mod-drivers-ref-18)</sup> 通道编号无越界保护**：`dmac_channel_enable()` 的 `switch` 仅覆盖 `DMAC_CHANNEL0~5`，对非法 `channel_num` 直接 `break` 不处理——调用者传入越界值将导致未初始化硬件或静默失败。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层是用户态与内核态之间的协议边界, 核心入口为 `syscall()` (syscall.c:1690), 由 trap 处理路径 (`usertrap()`, trap.c:68) 在 `scause==8` 时调用. 参数通过 per-process trapframe (trap.h:9) 的寄存器槽位传递, 并借助 `fetchaddr`/`argraw` 等辅助函数跨地址空间拷贝. 相比原版 xv6-riscv, 本仓库最显著的扩展是内嵌了一个 Linux-RISC-V ABI 兼容层: 通过进程名启发式 (`is_linux_abi_proc`) 将 Linux syscall 号翻译为本地语义, 使未修改的 Linux 测试二进制可直接运行. 调度器、文件系统、信号模拟等新增 syscall (SYS_set_timeslice, SYS_trace, SYS_getpgcnt 等) 均挂在同一张 `syscalls[]` 跳转表上, 构成一个既保留 xv6 教学血统又面向竞赛/测试场景的混合系统调用接口.

### 1. 核心抽象: 陷入-派发-退出的闭环

系统调用层的生命周期严格遵循 RISC-V 特权架构的 `ecall` 路径 <sup>[5](#mod-syscall-ref-5)</sup>. 用户态执行 `ecall` 后, 硬件将 PC 存入 `sepc`, 将 `scause` 设为 8, 并跳转到 `stvec` 指向的 trampoline. 经过汇编蹦床的寄存器保存, 控制权到达 `usertrap()` (trap.c:68). 这里内核做三个关键动作: (a) 保存用户 `epc` 并 +4 指向下一条指令, (b) 打开中断 (`intr_on()`) 使系统调用期间可抢占, (c) 调用 `syscall()`. 这种 "在系统调用处理期间开中断" 的设计与 xv6 一致, 代价是 `syscall()` 内部访问的 per-process 状态必须假定可能被调度改变, 但实际依赖 `myproc()` 返回当前 CPU 上运行的进程, 在非抢占式调度下是安全的 <sup>[1](#mod-syscall-ref-1)</sup>.

`syscall()` 本身是一个双路径派发器: 先从 trapframe 的 `a7` 寄存器读取 syscall 号 <sup>[2](#mod-syscall-ref-2)</sup>, 若当前进程被 `is_linux_abi_proc()` 判定为 Linux 测试二进制, 则优先走 `linux_dispatch()` 分支; 否则回退到 xv6 原生的 `syscalls[]` 跳转表. 派发结束后返回值写入 `trapframe->a0`, 由 `usertrapret()` 恢复上下文并通过 `sret` 返回用户态. 这一闭环的每一环节——参数在哪、返回值写回哪、何时开中断——都精确地耦合到 trapframe 的寄存器布局 (trap.h:9-32) <sup>[3](#mod-syscall-ref-3)</sup>.

### 2. 关键设计取舍: Linux ABI 兼容层的代价

本模块最突出的取舍是把 Linux syscall 模拟直接做在内核里, 而不是通过用户态 libc 或 vDSO. 动机很明确: 让来自 `testsuits-for-oskernel` 的预编译 RISC-V Linux 二进制不经过任何修改即可运行 <sup>[4](#mod-syscall-ref-4)</sup>. 具体做法是维护一个二进制名白名单 (`is_linux_abi_proc`, syscall.c:421), 对匹配的进程启用 Linux 号空间. 这带来几个后果:

- **号空间冲突**: Linux 和 xv6 的 syscall 编号有重叠 (例如 `SYS_read` 在 xv6 是 5, Linux 是 63). 解决方式是在 `syscall()` 中对 Linux 进程优先调用 `linux_dispatch()`, 只有未被该函数处理的号才 fallback 到 xv6 表. 这要求 `linux_dispatch()` 的返回语义精确 (1=已处理, 0=未识别), 否则会错误地落入 xv6 路径 <sup>[2](#mod-syscall-ref-2)</sup>.
- **参数语义转换**: 许多 Linux syscall 的调用约定与 xv6 不一致. 例如 `openat` 有 dirfd + flags + mode 四个参数, 而 xv6 的 `sys_open` 仅从 `a0`/`a1` 取路径和标志. 适配器 `linux_sys_openat()` (syscall.c:490) 的做法是临时改写 trapframe 的 `a0`/`a1`, 调用 `sys_open()`, 再恢复原值——这是一种对现有代码侵入最小的 "寄存器垫片" 模式, 但依赖于 `sys_open` 不检查 trapframe 的其他字段 <sup>[6](#mod-syscall-ref-6)</sup>.
- **不完全实现**: `linux_dispatch()` 覆盖了约 50 个 Linux syscall, 但其中大量是桩函数直接返回 0 或 -1 (例如 `linux_sys_mount`, `linux_sys_kill`, `linux_sys_mprotect`), 意味着这些调用在测试中要么被跳过, 要么期望的错误码恰好匹配. 这是工程上典型的 "按需实现" 策略, 但留下了明显的功能缺口 <sup>[7](#mod-syscall-ref-7)</sup><sup>[8](#mod-syscall-ref-8)</sup>.

### 3. 跨模块协同: trapframe 作为参数/返回的枢纽

系统调用层与 trap 机制、虚拟内存、进程管理之间存在紧密的跨模块耦合:

- **与 trap.c 的接口**: `usertrap()` 在调用 `syscall()` 前执行 `p->trapframe->epc += 4` (trap.c:77), 确保返回用户态时从 `ecall` 的下一条指令继续. 同时它打开中断, 使得 `syscall()` 内部可以安全地调用可能阻塞的 `sleep()` 或 `yield()`. 这一开中断的时机是精心选择的——必须在读取完 `scause` 和保存 `epc` 之后, 因为中断可能改变这些 CSR <sup>[1](#mod-syscall-ref-1)</sup>.
- **与虚拟内存的接口**: `fetchaddr` 和 `fetchstr` 是对用户态指针的安全封装. `fetchaddr` (syscall.c:25) 先做边界检查 (`addr >= p->sz || addr+sizeof(uint64) > p->sz`), 再调用 `copyin2` 从用户页表拷贝. 这里 `copyin2` 替代了原版 xv6 的 `copyin(p->pagetable, ...)`, 省去显式传递页表的繁琐, 但隐式依赖 `myproc()->pagetable`, 这意味着它只能在进程上下文中使用 <sup>[9](#mod-syscall-ref-9)</sup>.
- **与调度模块的接口**: 新增的调度相关 syscall (`SYS_set_timeslice`, `SYS_set_priority`, `SYS_set_sched_algo`) 直接挂载在 `syscalls[]` 表中, 表明调度参数的调整被设计为可通过系统调用动态改变, 而非常量编译期固定. 这为竞赛场景的调度策略实验提供了便利 <sup>[10](#mod-syscall-ref-10)</sup>.
- **与文件系统的接口**: Linux 兼容层的 `linux_begin_dirfd_cwd` / `linux_end_dirfd_cwd` 对 (syscall.c:470-488) 临时替换进程的 `cwd` 来模拟 `*at()` 系统调用的相对目录语义. 这种借用进程 `cwd` 字段的做法虽然简洁, 但在并发场景下如果同一进程多线程并发执行 `*at` 调用, 会导致 cwd 竞争——不过当前内核显然假设单线程进程 <sup>[11](#mod-syscall-ref-11)</sup>.

### 4. 边角细节与不足

**指针与内存安全**: `fetchaddr` 中的边界检查 `addr+sizeof(uint64) > p->sz` (syscall.c:28) 存在整数溢出风险: 若 `addr` 接近 `UINT64_MAX`, 加法将回绕, 绕过检查. 好在用户态地址空间远小于此值, 实际不会触发, 但作为防御性代码不够严谨 <sup>[12](#mod-syscall-ref-12)</sup>.

**锁与并发**: `linux_rlimit_nofile` 是文件级静态变量 (syscall.c:295), 通过 `linux_sys_prlimit64` 修改时无锁保护. 多核场景下可能存在竞争, 但当前内核可能尚未真正支持 SMP, 故风险有限 <sup>[13](#mod-syscall-ref-13)</sup>.

**参数校验缺失**: `argraw()` 对 `n > 5` 直接 `panic` (syscall.c:65), 而 `syscall()` 在调用各 `sys_*` 函数前并不校验 syscall 号是否合法, 只是依赖 `syscalls[]` 数组的边界和 NULL 检查. 如果某函数内部错误地调用 `argint(6, ...)`, 将触发内核 panic——这是 xv6 的 "fail-fast" 哲学, 但在生产系统中不可接受 <sup>[14](#mod-syscall-ref-14)</sup>.

**`linux_sys_brk` 整数下溢**: `if(addr >= p->sz - 2 * PGSIZE)` (syscall.c:1194) 在 `p->sz < 2*PGSIZE` 时会导致无符号下溢, 使得条件永远为真, 任何 `addr` 都会被拒绝. 对于极小的用户进程这可能是 bug <sup>[15](#mod-syscall-ref-15)</sup>.

**`linux_sys_nanosleep` 的 sleep 循环**: 该函数持有 `tickslock` 自旋锁进入 `sleep()`, 这是 xv6 中合法的用法 (sleep 会原子释放锁并让出 CPU), 但循环内部在 `sleep` 返回后未重新检查 `killed` 标志就再次进入 `sleep`, 理论上如果进程在睡眠期间被 kill, 可能多睡一轮 <sup>[16](#mod-syscall-ref-16)</sup>.

**TODO/桩函数**: 大量 Linux syscall 实现为空壳: `linux_sys_mount`, `linux_sys_umount2`, `linux_sys_faccessat`, `linux_sys_syslog`, `linux_sys_sendfile`, `linux_sys_utimensat`, `linux_sys_renameat2`, `linux_sys_mprotect` 均直接 `return 0;` 或 `return -1;` <sup>[7](#mod-syscall-ref-7)</sup><sup>[8](#mod-syscall-ref-8)</sup>. `linux_sys_kill` 更是完全忽略信号发送请求直接返回 0——这意味着测试中依赖 `kill` 的用例实际从未执行到信号逻辑 <sup>[17](#mod-syscall-ref-17)</sup>.

**硬编码常量**: `TIMEBASE_FREQ` (390000000ULL) 和 `USEC_PER_TICK` (5000ULL) 写死为 Kendryte K210 平台的值, 虽然后续有 QEMU 和 VF2 的条件编译分支, 但这些时间常量并未随平台切换, 可能导致计时偏差 <sup>[18](#mod-syscall-ref-18)</sup>.

**`sysnames[]` 数组对齐风险**: `syscalls[]` 和 `sysnames[]` 均使用相同的 `SYS_*` 索引, 但 trace 打印时仅检查 `p->tmask & (1 << num)` 而不验证 `num` 是否在 `sysnames` 范围内——如果两表长度不一致, 可能越界读取字符串指针 <sup>[19](#mod-syscall-ref-19)</sup>.
