## 📖 一、总览

该仓库是一个面向双架构（LoongArch、RISC-V）的教学/实验型 Unix-like 内核，采用汇编+C 两级启动、进程-线程双层调度、VMA 链表内存管理、嵌入式 ext4 文件系统 (lwext4) 以及 Slab 分配器等组件，意图在 xv6 基础上大幅扩展 Linux 兼容性。

最值得注意的 2-3 个特点包括：① **双架构启动与底层设计** — 实现了 LoongArch 的直接窗口映射和 RISC-V 的 M/S 模式切换（boot [ref:1][ref:2]），为内核适配不同 CPU 提供了完整的基础；② **Linux 兼容的系统调用集** — 定义了 119 个系统调用，涵盖文件（openat、read/write）、进程（fork、execve、clone）、内存（mmap、munmap）、IPC（pipe、futex）及信号/网络框架，意图运行标准 Linux 用户程序；③ **嵌入式 ext4 文件系统集成** — 直接引入 lwext4 库，支持 ext4 挂载、读写、JBD 日志（fs [ref:1][ref:5]），在教学内核中属罕见的完整文件系统实现。

## 🎯 二、综合评价

### 整体定位与架构档次
该内核从 xv6 启发但独立发展，定位介于教学原型与实验性兼容内核之间。它在保持简洁启动和调度模型的同时，尝试引入 Linux 风格的复杂子系统（ext4、futex、多架构），但各模块完成度不均：进程调度、基础内存管理、文件 I/O 相对可用；而信号、网络、并发安全则严重欠缺。整体属于“广度优先、深度不足”的设计，适合在 qemu 中演示基本 POSIX 语义，但远未达到稳定或生产水平。

### 真正的创新点
1. **双架构支持**：不同于 xv6 的单架构，该内核在启动阶段即分化为 LoongArch 的直接窗口映射（boot [ref:1]）和 RISC-V 的中断委派（boot [ref:2]），驱动层也分别实现 PCI 总线扫描（la）和固定 MMIO（riscv），体现了跨平台内核的早期架构设计。
2. **futex 原语**：实现了基于地址的等待/唤醒队列（ipc [ref:3][ref:4]），尽管存在无锁竞态和超时桩，但这是教学内核中少见的对 Linux 同步机制的移植尝试，意图支撑用户态线程库。
3. **ext4 文件系统**：通过 lwext4 库完整支持 ext4 的扩展区、JBD 日志和事务管理（fs [ref:1][ref:4]），这在内核教学中通常被简化为虚拟文件系统或极简 fs，此处选择了实际文件系统，提升了实验可用的真实性。

### 取舍判断
该仓库的开发精力明显集中在**功能覆盖**和**多架构**上，而非深度优化或安全加固。具体取舍包括：
- **立即映射 vs. 懒加载**：mmap 直接分配物理页（mm [ref:3]），舍弃懒加载的复杂度，但浪费内存且无法处理大文件映射。
- **并发安全排在其次**：全局无锁（futex、fs 挂载表、信号等）或锁使用不当（futex [ref:5]），表明作者优先功能正确性而非并发正确性，适合单核场景。
- **静态数据结构**：进程池、futex 队列、块设备表等均使用固定大小数组（fs [ref:1]），简化实现但限制扩展性。
- **信号/网络仅建框架**：信号仅实现配置接口（signal [ref:2]），网络只完成 bind 桩（net [ref:3]），其余递送、协议栈均省略，聚焦于基础路径。

### 完成度与不足
- **完成度**：约 50% 功能可运行。进程调度、管道、文件读写（ext4）、基本内存管理正常工作；但信号发送/递送完全缺失，网络只有 bind，驱动存在未完成注释（drivers [ref:6]），futex 超时是桩。
- **代码质量**：存在多处严重缺陷：slab 链表操作错误导致 slab 丢失（mm [ref:5]）、futex 无锁竞态（ipc [ref:5]）、信号位操作语义相反（signal [ref:5]）、pipe 整数溢出（ipc [ref:2]）、mmap 缺乏 copyin（mm 模块 noted），且缺少边界检查（signal [ref:2]）。整体代码属于早期原型阶段，需大量修复。
- **多核支持**：几乎未考虑 SMP，自旋锁使用不普遍，无原子操作，多核下必然崩溃。

总体而言，该内核是富有雄心但未成熟的实验品，为操作系统课程或个人学习提供了丰富的扩展案例，但距离可用的 Linux 兼容系统仍有巨大差距。

## 🚀 三、启动流程

> 💡 **TL;DR**
>
> 启动流程模块涵盖从硬件复位到内核主函数 `sos_start_kernel()` 执行完毕的完整引导路径。核心抽象是通过汇编入口（`_entry`）与 C 入口（`start` / `sos_start_kernel`）两级跳转，完成栈设置、BSS 清零、页表窗口配置（LoongArch）、中断委派（RISC‑V）等低级初始化，再依次调用 `thread_init`、`proc_init`、`pmem_init` 等子系统初始化，最后启动首个用户进程并进入调度器。相比 xv6 的单架构单核启动，本实现支持 RISC‑V 与 LoongArch 双架构，且集成了 slab 分配器、VFS/ext4 文件系统、虚拟内存管理等更丰富的模块初始化序列。

### 1. 核心抽象与外部依赖

启动流程以**两级入口**为抽象：底层汇编入口 `_entry`（位于 `hal/riscv/entry.S` 或 `hal/loongarch/entry.S`）负责架构相关的微量准备——设置栈指针（`sp`）、清零 BSS 段（仅主核）、配置直接映射窗口（LoongArch 的 `DMWIN`），然后跳转到 C 入口；C 入口根据架构分为 `start()`（RISC‑V，位于 `hal/riscv/start.c`）或直接 `sos_start_kernel()`（LoongArch 汇编末尾直接 `bl sos_start_kernel`）。`start()` 进一步配置 M 模式下的中断委派、时间中断初始化，最终通过 `mret` 切换到 S 模式并进入 `sos_start_kernel()`。

`<sup>[1](#mod-boot-ref-1)</sup>` 展示了 LoongArch 汇编中通过 `li.d` 向 `CSR_DMWIN0` 写入固定值来建立内核空间的直接映射，这一设计选择了硬件窗口（而非页表）来覆盖线性地址空间，放弃了对大内存区域的精细权限控制，但换取了 TLB 缺失的零开销。`<sup>[2](#mod-boot-ref-2)</sup>` 则显示 RISC‑V 的 `start()` 通过设置 `MSTATUS_MPP_S` 并调用 `mret` 迁跃到 S 模式，这是一种标准的 xv6 风格引导方式，但本实现在 SBI 环境下也保留了直接跳转的路径（`#if defined SBI` 分支），体现了对模拟器和硬件的兼容性考量。

`sos_start_kernel()` 本身是一个中枢函数（`<sup>[3](#mod-boot-ref-3)</sup>`），它按顺序调用了十余个初始化函数：字符设备、printf、线程、进程、物理内存、虚拟内存、slab 分配器、陷阱处理、PLIC、磁盘、文件系统（VFS/ext4）、inode 缓存，最后创建 init 进程并进入调度器。这种线性的、无错误检查的初始化方式简化了早期启动逻辑，但意味着任何一步失败都会导致系统静默挂起或崩溃。

### 2. 关键设计取舍

**多核启动的竞态处理**：`<sup>[1](#mod-boot-ref-1)</sup>` 中 LoongArch 的 `_entry` 通过 `csrrd $a1, LOONGARCH_CSR_CPUID` 获取核心 ID，并仅在 `$tp`（即 hart ID）为零时执行 BSS 清零。其他核心跳过清零后直接进入 `sos_start_kernel()`。这意味着所有核心都会同时执行全套初始化函数，包括 `pmem_init()` 等可能修改全局状态的过程。这放弃了单核顺序初始化的简单性，换取多核并行启动的速度。但当前未看到任何核间同步或锁保护，可能导致物理内存管理、slab 分配器等在初始化阶段产生竞态。这是早期实现中常见的风险点。

**栈分配与溢出风险**：每个核心的栈被静态分配在 `entry_stack` 区域，大小为 4096 字节乘以（hart ID+1）。`<sup>[1](#mod-boot-ref-1)</sup>` 中 `la $sp, entry_stack; ... mul.d $a0, $a0, $a1; add.d $sp, $sp, $a0` 这段代码未检查 `hart ID` 是否超出预分配范围。如果硬件核心数超过编译期设定的栈空间上限，会导致栈指针越界，覆盖其他关键内存。这是一个硬编码常量配合运行时输入的风险。

**中断上下文与锁的隐式假设**：`sos_start_kernel()` 在其线性序列中调用了 `hsai_trap_init()`，但在此之前中断仍处于关闭状态（LoongArch 在 `_entry` 中通过 `CSR_CRMD` 关闭了全局中断，RISC‑V 在 `start()` 中未显式打开 S 模式中断）。直到调度器启动后，中断才被使能。因此初始化期间所有操作都在中断关闭下进行，自旋锁的获取不需要考虑中断嵌套，这简化了实现但降低了中断响应实时性。`<sup>[4](#mod-boot-ref-4)</sup>` 中 `init_process` 在设置 `p->state = RUNNABLE` 后立即 `release(&p->lock)`，而此时中断尚未使能，因此不会发生锁的并发问题。

**架构抽象的边界**：启动流程通过 `#if defined RISCV` / `#else` 预编译宏来区分架构，而不是通过函数指针或 HAL 接口。`<sup>[3](#mod-boot-ref-3)</sup>` 中 `#if defined RISCV` 分支调用 `plicinit()` 等，LoongArch 分支则调用 `virtio_probe()`。这种直接在 C 代码中嵌入条件编译的方式耦合了架构细节，增加了维护成本，但避免了抽象层的性能开销——注释也明确提到“使 la 运行速度变慢”，说明这是一个有意识的性能‑可维护性取舍。

### 3. 跨模块协同

启动流程与后续模块的协同体现在 `init_process()` 函数中。`<sup>[5](#mod-boot-ref-5)</sup>` 显示 `init_process` 调用了 `uvminit(p, init_code, len)` 将用户初始代码加载到进程地址空间，再通过 `hsai_set_trapframe_epc` 和 `hsai_set_trapframe_user_sp` 设置用户态入口和栈指针。这建立了从内核启动到用户态第一个进程的桥梁。`init_code` 是由 `rv_init_code.h` 或 `la_init_code.h` 提供的二进制数组，这种静态嵌入方式省去了文件系统加载的依赖，但使得修改初始程序必须重新编译内核。

文件系统初始化的顺序也体现了依赖关系：`init_fs()` 先注册文件系统类型，然后 `binit()` 初始化块缓存，`fileinit()` 准备文件描述符表，`inodeinit()` 分配 inode 缓存，最后 `vfs_ext4_init()` 挂载根文件系统。`<sup>[3](#mod-boot-ref-3)</sup>` 中该序列确保了在创建 init 进程时 `p->cwd.fs` 可以正确指向 EXT4 文件系统。如果任意一步失败，后续的 `get_fs_by_type(EXT4)` 可能返回 NULL，但代码未对此进行检查。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 该模块管理进程虚拟地址空间和内核小对象内存分配，核心抽象为VMA（虚拟内存区域）双向链表和Slab分配器。VMA链表支持mmap/munmap/mprotect系统调用，实现用户堆、栈、文件映射的灵活管理；Slab分配器为内核提供固定大小的快速内存分配，降低外部碎片。与xv6原版相比，本项目扩展了多VMA支持、文件映射、共享映射、mprotect权限修改等特性，并引入了独立的Slab分配器替代简单页分配。

### 1. 核心抽象：VMA双向链表与Slab分配器

进程地址空间由VMA双向环形链表管理，每个VMA（`struct vma`，定义于`include/kernel/vma.h:17-28`）记录一段连续虚拟地址区的类型（MMAP/STACK/SHARE）、权限、文件映射信息。链表头为哨兵节点（`type=NONE`），插入操作保持地址升序。该设计允许每个进程拥有多个不连续映射段，突破xv6单一`sz`的限制。

内核小对象分配依赖Slab分配器，全局`struct slab_allocator`（`include/kernel/slab.h:40-46`）维护8个固定大小`kmem_cache`（8B~1024B）。每个`kmem_cache`管理一组slab页，页内通过自由对象链表分配，释放时回插。Slab初始化使用`simple_alloc`从boot页分配元数据，形成自举依赖。<sup>[1](#mod-mm-ref-1)</sup><sup>[2](#mod-mm-ref-2)</sup>

### 2. 关键设计取舍：立即映射、权限转换与Slab缺陷

**映射策略**：`mmap`当前采用立即分配（`MAP_ALLOC`标志），在系统调用中直接分配物理页并填充文件内容（`kernel/vma.c:143-259`）。注释显示曾考虑懒加载（“逻辑有问题”被注释），但最终选择简单实现——这导致映射大文件时内存浪费，但避免了缺页异常处理的复杂性。<sup>[3](#mod-mm-ref-3)</sup>

**权限转换**：`get_mmapperms`将`PROT_READ/WRITE/EXEC`转换为硬件PTE标志。RISC-V下`PROT_NONE`被映射为`perm=0`，但注释坦言“似乎只可读和只可写也能通过”，暗示U模式页表项必须至少设置`PTE_R`才能访问，`PTE_U|PTE_R=0`可能引发异常。<sup>[4](#mod-mm-ref-4)</sup> LA64架构通过清除`PTE_NR`实现读权限，反映平台差异的硬编码适配。

**Slab缺陷**：固定大小表（8/16/32/…/1024）覆盖多数内核对象，但对>1024B的分配直接panic（`slab_common.c:247`）。`kmem_cache_create`接口未实现（返回0），无法添加自定义cache。`slab_free`中对已满slab的回收存在链表断裂bug：当slab从`full_slab`链表迁移回`free_slab`时，仅假设slab为链表头（`full_slab = slab->next`），若slab在非头位置，链表后半部分全部丢失。<sup>[5](#mod-mm-ref-5)</sup>

### 3. 跨模块协同：文件映射、进程复制与页表切换

`mmap`的文件映射路径涉及VFS层：调用`vfs_ext4_lseek`定位文件偏移，然后通过`get_file_ops()->read()`读取内容到映射的物理页（`kernel/vma.c:230`）。关键危险：`read`的第二个参数是用户虚拟地址`start+i`，但内核文件读操作期望的是内核缓冲区地址或用户缓冲区（需要`copyin`），直接传递用户VA可能导致内核访问无效页或绕过权限检查——这是一个明显的安全与正确性漏洞。<sup>[6](#mod-mm-ref-6)</sup>

进程复制通过`vma_copy`（`kernel/vma.c:462-480`）遍历原VMA链表，为每个VMA分配新结构并复制内容，对文件映射VMA调用`dup`增加文件引用计数。`alloc_vma_stack`为进程分配固定大小的用户栈（`USER_STACK_TOP`向下`100*PGSIZE`），并插入链表末尾，与`get_proc_sp`配合获取栈顶。<sup>[7](#mod-mm-ref-7)</sup>

页表切换依赖架构抽象层：`hsai_mem.c`的`hsai_config_pagetable`写入`satp`寄存器（RISC-V）或配置页表遍历寄存器（LA64），并执行TLB刷新。`TRAMPOLINE`和`TRAPFRAME`在编译期固定映射到最高虚拟地址（`riscv_memlayout.h:53-54`），实现内核与用户页表共享跳板页。<sup>[8](#mod-mm-ref-8)</sup>

### 4. 边角细节与不足

1. **Slab full链表操作错误**：`slab_free`中`kmem_cache->full_slab = slab->next`未维护链表完整性，若slab非头节点则导致后续节点丢失。<sup>[5](#mod-mm-ref-5)</sup>
2. **mmap文件读取越权**：`get_file_ops()->read(f, start+i, to_read)`直接使用用户VA作为目标缓冲区，可能造成内核内存破坏。<sup>[6](#mod-mm-ref-6)</sup>
3. **对齐检查失效**：`slab_free`中的对齐检查`(addr-sizeof(struct slab))%slab->size`未减去`page_start`，实际无法捕获未对齐object，形同虚设。<sup>[9](#mod-mm-ref-9)</sup>
4. **并发安全缺失**：所有VMA链表操作和Slab分配均未加锁，SMP环境下list corruption、double-free等竞态问题必然发生。<sup>[10](#mod-mm-ref-10)</sup>
5. **断言代替检查**：`alloc_vma`使用`assert(p != NULL && p->vma != NULL)`，Release编译下断言被移除，空指针解引用可能直接崩溃。<sup>[11](#mod-mm-ref-11)</sup>
6. **TLB刷新遗漏**：`experm`修改PTE权限后未立即刷TLB，若调用方（如`mmap`）未后续刷TLB，旧权限可能仍然生效。<sup>[12](#mod-mm-ref-12)</sup>

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 本模块以进程（struct proc）为资源容器、线程（thread_t）为调度实体，实现了双层调度模型。核心包括固定大小的进程池、线程队列、上下文切换（hsai_swtch）、睡眠唤醒（sleep_on_chan/wakeup）、fork/clone/exit/wait 等完整生命周期管理。相比 xv6 原版，扩展了多线程支持（clone/clone3）、进程组/会话（setpgid/setsid）、CPU 亲和性（sched_setaffinity）、futex 同步、POSIX 定时器（itimers）和信号框架，同时保留 xv6 经典的自旋锁与中断关闭约束。

### 1. 进程-线程双层调度模型

本模块的核心抽象是进程（`struct proc`）与线程（`thread_t`）的双层结构：进程作为资源容器（地址空间、文件描述符表、信号设置、定时器），线程作为调度实体。每个进程维护一个线程队列 `thread_queue`，主线程 `main_thread` 在进程创建时分配并常驻 <sup>[1](#mod-task-ref-1)</sup>。调度器 `scheduler()` 遍历全局进程池 `pool[NPROC]`，对每个 `RUNNABLE` 状态的进程，在其线程队列中搜索第一个 `t_RUNNABLE` 或 `t_TIMING`（定时唤醒）的线程，然后通过 `hsai_swtch` 进行上下文切换 <sup>[8](#mod-task-ref-8)</sup>。

## 📁 六、文件系统

> 💡 **TL;DR**
>
> 本模块基于 lwext4 嵌入式 ext4 文件系统库，实现了完整的 ext4 文件系统支持，包括块设备管理、挂载点管理、inode 操作、目录遍历、扩展区 (extent) 管理和 JBD (Journaling Block Device) 日志恢复。核心抽象为 `ext4_mountpoint`、`ext4_blockdev`、`ext4_inode_ref` 和 `ext4_trans`，分别管理挂载状态、块设备 I/O、inode 引用和事务生命周期。与 xv6 原始设计相比，本模块引入了完整的 ext4 兼容日志 (JBD) 和扩展区树，支持大文件和高效空间管理，并提供了丰富的系统调用 (mount/umount/openat/read/write 等) 接口，但保留了 lwext4 的全局静态数组设计和部分未实现的桩函数。

### 1. 核心抽象与外部依赖

文件系统模块围绕 lwext4 库构建，核心数据结构包括：
- **`ext4_block_devices`**：全局静态数组 `s_bdevices[CONFIG_EXT4_BLOCKDEVS_COUNT]`，存储注册的块设备名称和句柄。设备通过 `ext4_device_register` 注册，查找依赖线性遍历 <sup>[1](#mod-fs-ref-1)</sup>。
- **`ext4_mountpoint`**：全局静态数组 `s_mp[CONFIG_EXT4_MOUNTPOINTS_COUNT]`，管理挂载点状态。`ext4_mount` 在数组空位中创建挂载点，初始化块设备、inode 缓存和日志；`ext4_umount` 反向清理 <sup>[2](#mod-fs-ref-2)</sup>。
- **`ext4_inode_ref`**：inode 引用封装，通过 `ext4_fs_get_inode_ref` / `ext4_fs_put_inode_ref` 成对操作，保护 inode 缓存一致性 <sup>[6](#mod-fs-ref-6)</sup>。
- **`ext4_trans`**：事务（Journal）抽象，在 `__ext4_trans_start` / `__ext4_trans_stop` 中管理 `jbd_trans` 的生命周期，确保元数据原子性 <sup>[5](#mod-fs-ref-5)</sup>。

依赖关系：`ext4.c` 调用 `ext4_blockdev.c` 的块 I/O、`ext4_inode.c` 的 inode 读取、`ext4_fs.c` 的块组管理以及 `ext4_journal.c` / `ext4_extent.c` 的日志和扩展区操作。系统调用层 (如 `fs/syscall.c`) 通过 `_ext4_get_mount`、`ext4_file_open` 等接口桥接用户态请求与内核文件系统实现。

### 2. 关键设计取舍

**取舍 1：全局静态数组 vs 动态分配**
`s_bdevices` 和 `s_mp` 以固定大小静态数组实现 <sup>[1](#mod-fs-ref-1)</sup><sup>[2](#mod-fs-ref-2)</sup>，避免了内核堆分配器的依赖和碎片问题，简化了代码。代价是最大设备数和挂载点数由编译期常量 `CONFIG_EXT4_BLOCKDEVS_COUNT` 和 `CONFIG_EXT4_MOUNTPOINTS_COUNT` 硬编码，无法运行时扩展。若超出限制，`ext4_device_register` 返回 `ENOSPC`，`ext4_mount` 返回 `ENOMEM`。

**取舍 2：中断上下文安全**
锁机制通过宏 `EXT4_MP_LOCK` / `EXT4_MP_UNLOCK` 封装 OS 依赖的锁 <sup>[3](#mod-fs-ref-3)</sup>，支持在中断上下文中使用（例如 `__ext4_recover` 在持锁时调用了 `ext4_fs_get_block_group_ref`）。但 `ext4_mount` 和 `ext4_umount` 等路径未显式加锁，存在竞态风险——若两个 CPU 同时调用 `ext4_mount` 操作同一设备，可能破坏 `s_mp` 状态。

**取舍 3：日志完整性 vs 性能**
JBD 日志实现支持 checksum (crc32c) 进行元数据校验 <sup>[7](#mod-fs-ref-7)</sup>，但 `CONFIG_META_CSUM_ENABLE` 控制的条件编译意味着若未开启则无校验。日志在 `jbd_block_get` 中强制标记 `BC_FLUSH` 和 `BC_TMP` <sup>[4](#mod-fs-ref-4)</sup>，确保元数据块直接写回而非延迟缓存——这保证了崩溃一致性，但牺牲了块缓存效率。

**取舍 4：扩展区树 vs 传统块映射**
`ext4_extent.c` 实现了完整的 ext4 扩展区树（B-树）管理 <sup>[8](#mod-fs-ref-8)</sup>，支持初始化/未初始化（预分配）状态，最大单个 extent 覆盖 32768 块（初始化）/ 32767 块（未初始化）。这比 xv6 的直接块+间接块映射更节省元数据空间，适合大文件，但引入了树遍历和分裂/合并开销。`ext4_fs.c` 保留了传统块映射路径作为后备。

**取舍 5：全局锁粒度粗化**
挂载点搜索 `ext4_get_mount` 遍历全局数组且未加锁 <sup>[9](#mod-fs-ref-9)</sup>。在多线程并发挂载/卸载场景下，`s_mp[i].mounted` 的读取可能看到中间状态，导致挂载点误判。但考虑到本内核可能未启用多核或用户态线程，当前实现优先保证了简单性。

### 3. 跨模块协同

**协同一：日志事务生命周期 (ext4.c + ext4_journal.c)**
`ext4.c` 中 `__ext4_trans_start` / `__ext4_trans_stop` 负责事务开启和提交。它们调用 `ext4_journal.c` 的 `jbd_journal_new_trans` 和 `jbd_journal_commit_trans`。关键路径是 `ext4_mount` 中通过 `__ext4_journal_start` 初始化 `jbd_fs`，如果超级块标记 `EXT4_FCOM_HAS_JOURNAL` 则创建日志 <sup>[5](#mod-fs-ref-5)</sup>。写操作在 `ext4_link`、`ext4_unlink` 等函数中通过 `ext4_trans_set_block_dirty` 标记脏块，最终由 commit 原子写盘。

**协同二：扩展区块分配 (ext4_extent.c + ext4_balloc.c)**
`ext4_extent.c` 的 `ext4_allocate_single_block` 委托 `ext4_balloc.c` 的 `ext4_balloc_alloc_block` 进行物理块分配 <sup>[10](#mod-fs-ref-10)</sup>。分配的块号被填入 extent 结构（`ext4_ext_store_pblock`）。若需要分裂节点，`ext4_ext_split` 调用 `ext4_new_meta_blocks` 分配新元数据块，更新树结构。释放路径则调用 `ext4_balloc_free_blocks`。这种分工隔离了物理位图操作与逻辑树管理。

**协同三：系统调用桥接 (fs/syscall.c + ext4.c)**
系统调用如 `openat`、`read`、`write`、`mount` 通过 `_ext4_get_mount` 定位挂载点 <sup>[9](#mod-fs-ref-9)</sup>，然后调用 lwext4 提供的 ext4 级 API（如 `ext4_file_open`、`ext4_file_read`、`ext4_file_write`）。这些 API 内部再调用 inode 操作和事务接口。桥接层不直接操作 ext4 内部数据结构，保持了一定抽象。

### 4. 边角细节与不足

- **<sup>[11](#mod-fs-ref-11)</sup>** `ext4_unlink` 中有两个 `TODO: Update timestamps` 注释，时间戳更新被省略。这意味着 `ctime`/`mtime` 不会在删除时更新，违反 POSIX 要求，但挂载时若设置了 `read_only` 则无影响。
- **<sup>[12](#mod-fs-ref-12)</sup>** `jbd_inode_bmap` 只处理了 `iblock` 到 `fblock` 的映射，但注释 `TODO: journal device.` 表明外部日志设备未实现。所有日志块存储在主文件系统的 inode 中，可能占用大量元数据空间。
- **<sup>[13](#mod-fs-ref-13)</sup>** `ext4_mount` 检查挂载点末尾必须是 `/`（`mount_point[mp_len - 1] != '/'` 返回 `ENOTSUP`），这限制了挂载点字符串格式，与 Linux 内核允许无尾部斜杠的路径不符。
- **<sup>[14](#mod-fs-ref-14)</sup>** `ext4_device_register` 中用 `strcpy` 和 `strlen` 操作用户提供的 `dev_name`，但未验证输入长度是否超过 `CONFIG_EXT4_MAX_BLOCKDEV_NAME`——仅检查了 `strlen` 返回值，若等于最大长度则可能溢出（`strcpy` 会写结尾 `\0` 越界）。
- **<sup>[15](#mod-fs-ref-15)</sup>** `ext4_unlink` 在 `ext4_inode_get_links_cnt` == 0 时不递减链接计数，但此分支可能隐藏逻辑错误：若 inode 已无硬链接但仍被打开，删除后应保留 inode 直到所有引用关闭。当前实现没有处理此场景。
- **并发风险**：全局 `s_bdevices` 和 `s_mp` 在 `ext4_device_register`、`ext4_mount`、`ext4_umount` 中读写均未加锁。若多核同时挂载/卸载，可能破坏数组一致性（如同时查找空位并写入）。
- **错误码忽略**：`ext4_has_children` 中 `ext4_dir_iterator_init` 失败时直接返回 `rc`，但调用方 `ext4_unlink` 未检查该返回值即继续操作，可能导致对疑似空目录的错误判断 <sup>[16](#mod-fs-ref-16)</sup>。

### 5. 与 xv6 的对比

本文件系统模块与经典 xv6 有本质区别：
- **文件系统类型**：xv6 使用简单日志型文件系统（类似于 ext2 的变体），而本模块实现了完整的 ext4，包括扩展区树、日志、Htree 目录索引、元数据校验和等高级特性。
- **日志系统**：xv6 的日志位于磁盘固定区域，仅记录块改写；本模块采用 JBD2 兼容日志，支持事务提交、撤销、校验和，恢复路径 `__ext4_recover` 会遍历日志进行重放。
- **缓存策略**：xv6 使用固定大小的块缓存（`bcache`），替换策略为 LRU；本模块使用 `ext4_bcache`，支持 `BC_FLUSH` 强制写回和 `BC_TMP` 临时标记，更灵活但复杂度更高。
- **扩展性**：xv6 的 inode 使用直接/间接块指针，限制单文件最大大小；ext4 扩展区树可支持超大文件（最大 16 TB）。
- **不足**：本模块缺少 dentry 缓存、内存映射文件支持、以及类似 xv6 的 `ide` 驱动抽象层，块设备接口直接调用 `ext4_blockdev`。

## 📡 七、信号机制

> 💡 **TL;DR**
>
> 信号机制围绕进程控制块 `proc_t` 中的 `sigaction[]` 数组和 `__sigset_t` 类型信号掩码 `sig_set` 构建，提供 `set_sigaction` 和 `sigprocmask` 两个核心函数，分别对应 `rt_sigaction` 与 `rt_sigprocmask` 系统调用的底层逻辑。当前实现仅覆盖信号配置面（设置处理动作和阻塞掩码），完全缺失信号发送（kill/tkill/tgkill）、挂起信号队列、递送时机（trap 返回检查）和执行跳板。相比 xv6‑riscv（无信号机制）是全新的 POSIX 兼容扩展，但处于极早期原型阶段，存在多处安全与正确性缺陷。

### 1. 核心抽象：sigaction 表与信号掩码

信号机制的核心数据结构嵌入在进程控制块 `proc_t` 中：`sigaction[]` 数组为每个信号（索引为信号编号）存储处理动作——通过 `__sigaction_handler` 联合体中的 `sa_handler` 函数指针指定忽略(SIG_IGN)、默认(SIG_DFL)或用户自定义处理函数；`__sigset_t` 类型的 `sig_set` 字段则用位图记录当前被阻塞的信号集。

`set_sigaction()` <sup>[2](#mod-signal-ref-2)</sup> 是配置信号处理动作的唯一入口。它从 `myproc()` 获取当前进程的 `proc_t`，然后根据输入输出语义：若 `oldact` 非 NULL 且该信号已有处理函数，将旧配置拷贝回用户空间；若 `act` 非 NULL，则将新配置写入 `sigaction[signum]`。这个接口直接对应 `rt_sigaction` 系统调用的 `SIGACTION_READ | SIGACTION_WRITE` 两阶段操作。

`sigprocmask()` <sup>[3](#mod-signal-ref-3)</sup> 操作信号掩码。它遍历 `__sigset_t` 内部的 `__val[]` 数组（长度为 `SIGSET_LEN`），根据 `how` 参数执行三种 POSIX 标准操作：`SIG_BLOCK`（或运算添加）、`SIG_UNBLOCK`（与非运算移除）、`SIG_SETMASK`（直接覆盖）。`oldset` 参数支持原子地获取旧掩码，这是线程库（如 pthread）实现的关键需求 <sup>[4](#mod-signal-ref-4)</sup>。

### 2. 关键设计取舍与风险

**用户指针直接解引用 vs 安全拷贝。** `set_sigaction` 直接对用户传入的 `act` 和 `oldact` 指针进行结构体赋值（第 8、10 行），没有调用 `copyin`/`copyout` 进行地址空间校验。这意味着：(1) 若用户传入内核空间的地址，内核可能被诱导读取或覆写敏感内存；(2) 若指针指向未映射的用户页，将触发缺页异常而非优雅返回错误。xv6 风格的安全实践要求所有用户态指针必须经过 `copyin`/`copyout` 检查，这里的设计选择了简洁性而放弃了安全底线 <sup>[2](#mod-signal-ref-2)</sup>。

**无锁并发控制。** 两个函数均通过 `myproc()` 获取 `proc_t` 指针后直接读写 `sigaction` 和 `sig_set`，未持有任何自旋锁或关闭中断。在多核环境下，若两个 CPU 同时调用 `sigprocmask`（例如主线程和信号处理线程），`__val[i] |= set->__val[i]` 这样的读‑改‑写操作不是原子的，会导致掩码丢失更新。此外，若在中断上下文中调用（虽然当前未实现信号递送），缺乏 `push_off` 保护可能造成死锁 <sup>[2](#mod-signal-ref-2)</sup><sup>[3](#mod-signal-ref-3)</sup>。

**信号编号边界缺失。** `signum` 直接作为 `sigaction[]` 数组的下标（第 8、10 行），没有任何 `if (signum < 1 || signum >= NSIG)` 检查。若用户传入负值或大于数组长度的数，将造成栈或堆越界写入。这是 C 内核中典型的“信任用户输入”缺陷 <sup>[2](#mod-signal-ref-2)</sup>。

**始终返回 0 的错误处理。** 两个函数在路径上均返回 0，从不返回负错误码。这意味着用户态 `rt_sigaction` 或 `rt_sigprocmask` 无法区分成功与失败——传入非法 `signum`、`how` 或无效指针时，调用者仍会看到成功返回值，静默吞噬错误 <sup>[2](#mod-signal-ref-2)</sup><sup>[3](#mod-signal-ref-3)</sup>。

### 3. 跨模块协同与缺失链路

当前代码暴露出与三个模块的依赖关系：

- **进程管理 (`process.h`)**：通过 `myproc()` 获取 `proc_t`。信号掩码和动作表存储在进程控制块中，意味着 `fork` 实现必须正确复制这些字段（否则子进程会丢失父进程的信号配置），而 `exec` 需要将 `sigaction` 重置为 SIG_DFL。这些协同逻辑在当前文件中没有体现。
- **CPU 抽象 (`cpu.h`)**：仅通过头文件包含引入，可能用于获取当前 CPU 核心 ID 或中断控制。代码中未直接使用，是为未来多核或中断上下文扩展埋的钩子。
- **系统调用分发层**：虽然无法读取 `syscall.c`，但从 syscall ID 列表（`rt_sigaction=134`, `rt_sigprocmask=135`）可知，内核期望在系统调用入口处调用 `set_sigaction` 和 `sigprocmask`。然而，`kill(129)`、`tgkill(131)`、`tkill(130)`、`rt_sigtimedwait(137)` 等发送与等待系统调用的底层实现在 `kernel/signal.c` 中完全空缺——没有任何 `send_signal`、`sigismember` 或等待队列逻辑，表明整个信号递送路径（`trap.c` 中检查 `pending` 标志、构建 `sigframe`、跳转至信号跳板）均未接入。

这种“只有配置面、没有递送面”的实现意味着，即使通过 `rt_sigaction` 注册了用户处理函数，进程也永远不会收到信号——`kill` 系统调用最多修改一个全局位图，而 `trap_return` 中的递送检查根本不存在。

### 4. 边角细节与不足

以下逐条分析代码中的具体风险与缺陷，每条对应取舍清单中的一种风险模式：

- **<sup>[2](#mod-signal-ref-2)</sup> 用户指针直接解引用（指针与内存安全）**：`set_sigaction` 第 8 行 `*oldact = p->sigaction[signum]` 和第 10 行 `p->sigaction[signum] = *act` 直接对用户传入的指针进行结构体赋值，没有经过 `copyin`/`copyout`。攻击者可以通过传入内核地址实现任意内存写入。
- **<sup>[2](#mod-signal-ref-2)</sup><sup>[3](#mod-signal-ref-3)</sup> 无锁并发（锁与中断上下文）**：`set_sigaction` 和 `sigprocmask` 在读写 `proc_t` 字段时未加锁。多核环境下 `sigprocmask` 中的 `|=` 和 `&=` 操作不是原子的，可能产生竞态。
- **<sup>[2](#mod-signal-ref-2)</sup> signum 未校验（整数溢出/类型）**：`sigaction[signum]` 直接使用用户传入的 `signum` 作为下标，无上下界检查。可导致越界写入。
- **<sup>[2](#mod-signal-ref-2)</sup><sup>[3](#mod-signal-ref-3)</sup> 始终返回 0（错误码处理）**：两个函数均返回 0，即使指针无效或 `how` 参数非法也静默通过。
- **<sup>[5](#mod-signal-ref-5)</sup> 位操作语义反转（硬编码常量/bug）**：`sigprocmask` 末尾的 `p->sig_set.__val[0] &= 1ul << SIGTERM | 1ul << SIGKILL | 1ul << SIGSTOP` 本意是“确保这三个信号不被阻塞”，但 `&= mask` 的效果是只保留这三个位（其他的清零）。正确的写法应为 `&= ~(…)` 清除这三个位。当前实现的效果是：所有除 SIGTERM/SIGKILL/SIGSTOP 之外的信号都被强制解除阻塞，而这三个信号若之前被阻塞则继续保持阻塞——与注释完全相反。

此外，`SIGSET_LEN` 的定义未知，若 `SIGSET_LEN > 1`，则循环修改了所有 `__val[i]` 后，第 55 行只修复了 `__val[0]`，其他元素的最终状态与预期不一致（其他数组元素被循环正确修改，但忽略了“强制不被阻塞”的策略）。

## 🔄 八、进程间通信

> 💡 **TL;DR**
>
> 本模块提供进程间通信的基本机制，包括基于循环缓冲区的管道（pipe）和用户级等待原语futex。管道继承自xv6，使用自旋锁保护，以单字节为单位在用户和内核空间之间复制数据。futex基于固定大小2048的等待队列，支持等待、唤醒和重新排队操作，但并发控制存在严重缺陷，且超时功能未真正实现。与xv6相比，管道实现基本一致，而futex为新增特性，旨在支持用户态线程同步。但整体实现处于早期阶段，存在整数溢出、锁竞争和资源泄漏等风险。

### 1. 核心抽象与外部依赖
进程间通信模块的核心是**管道（pipe）**和**futex**两种抽象。

**管道**提供字节流通信，通过`struct pipe`结构体表示，内部包含一个512字节的循环缓冲区、读写位置计数器`nread`/`nwrite`、以及读/写端是否打开的标志。每个管道由两个`struct file`对象关联，分别代表读端和写端，通过文件操作表与虚拟文件系统集成。管道的读写操作依赖`copyin`/`copyout`在用户空间和内核缓冲区之间传输数据，并使用自旋锁`pi->lock`保护内部状态。当缓冲区满或空时，通过`sleep_on_chan`和`wakeup`实现等待和通知。

**futex**提供基于地址的等待/唤醒机制，核心数据结构是`futex_queue_t`数组`futex_queue[2048]`。每个条目记录用户空间地址`addr`、等待线程指针`thread`和有效标志`valid`。`futex_wait`将当前线程加入队列并进入睡眠；`futex_wake`唤醒指定地址上的一个或多个线程；`futex_requeue`将等待同一地址的线程迁移到新地址；`futex_clear`在线程退出时清理其队列条目。futex依赖线程调度函数`sched`、进程锁`th->p->lock`以及定时器结构`timespec_t`。

这两种抽象服务于不同场景：管道用于父子进程或兄弟进程间的数据传输，futex用于用户态实现互斥锁、条件变量等高级同步原语。

### 2. 关键设计取舍
**管道单字节传输**：每个读写系统调用在循环中逐字节调用`copyin`/`copyout`，实现简单但效率极低。每次系统调用需要多次陷入和复制，尤其在大数据量时性能损失明显。这是典型的早期教学实现，放弃了批量和DMA优化。

**固定大小futex队列**：`futex_queue`数组长度硬编码为2048，当一个地址上有超过2048个等待线程，或全局并发等待数超过该值时，`futex_wait`直接触发panic。替代方案是链表或哈希表动态扩展，但当前设计优先了简单性。

**futex无锁并发策略**：最严重的缺陷是`futex_wake`和`futex_clear`不获取任何锁，直接修改队列条目和线程状态。而`futex_wait`在设置队列条目后立即调用`sched`切换，期间可能被`futex_wake`抢占，导致“lost wakeup”——线程状态被设置为RUNNABLE后又马上被设为SLEEPING。这种放弃同步的做法无法保证正确性，是明显的早期错误。

**超时实现的桩函数性质**：`futex_wait`接收`timespec_t`参数并将其转换为微秒存储在`th->awakeTime`，设置线程状态为`t_TIMING`后即调用`sched`。但没有任何定时器中断或内核线程检查该字段，线程实际上会永久睡眠。超时功能仅停留在接口层面，未与时钟子系统集成。

**futex_requeue的两次遍历**：先唤醒n个线程，再遍历剩余线程修改其地址。这两个遍历不是原子的，如果在两次遍历之间又有新线程等待原地址，它们可能被遗漏或错误迁移。

### 3. 跨模块协同
管道通过`filealloc`和`get_file_ops()->close`与文件系统集成。`pipealloc`分配两个`struct file`，其`f_flags`分别设为`O_RDONLY`和`O_WRONLY`，`f_data.f_pipe`指向共享的`struct pipe`。当通过`read`/`write`系统调用读写管道时，最终调用`pipewrite`/`piperead`。这种设计复用了xv6的文件描述符机制，使管道能无缝融入进程的文件表。

futex与线程系统的接口较为粗糙。`futex_wait`直接调用`myproc()`获取当前进程，操作线程状态后调用`sched`切换。正常情况下，用户程序通过`sys_futex`系统调用（未提供源码）进入内核，而当前实现缺少系统调用处理例程。此外，进程退出时应当调用`futex_clear`清理残留的等待条目，但代码中未找到调用点，可能导致僵尸条目占用队列槽位。

`sleep_on_chan`和`wakeup`是管道中使用的条件等待原语，基于进程表的锁和等待通道实现。futex则绕过这一机制，使用自己的队列和`sched`直接切换。这种设计不一致增加了维护成本。

### 4. 边角细节与不足
1. **Futex队列并发无锁保护**：`futex_wake`直接写`futex_queue[i].thread->state`和`futex_queue[i].valid`，而`futex_wait`在设置队列后还要获取进程锁并修改状态，两者之间没有同步。若`futex_wake`在`futex_wait`设置状态前执行，会导致唤醒丢失 [ref: futex.c:50-53, futex.c:84-89]。

2. **整数溢出风险**：`pipewrite`中`pi->nwrite == pi->nread + PIPESIZE`的判定，当`nread`接近`UINT32_MAX`时，加法溢出导致条件恒不成立或恒成立，可能造成死锁或缓冲区越界 [ref: pipe.c:122]。

3. **Futex队列资源耗尽**：固定2048个槽位，`futex_wait`找不到空槽时直接panic，未提供动态扩展或失败返回值，系统容错性差 [ref: futex.c:73]。

4. **超时功能桩函数**：`futex_wait`设置线程状态为`t_TIMING`并存储唤醒时间，但没有任何代码在定时器中断中检查并唤醒超时线程，导致超时参数被忽略 [ref: futex.c:33]。

5. **管道单字节拷贝**：每次循环调用`copyin`/`copyout`仅复制1字节，引发大量地址空间切换开销，是明显的早期实现，应改为批量复制 [ref: pipe.c:129, pipe.c:158]。

6. **锁顺序违反**：`futex_wait`在持有进程锁`th->p->lock`的情况下调用`sched`，而`sched`可能尝试获取其他锁（如调度器锁），违反已建立的锁层级顺序 [ref: futex.c:56-60]。

## 🌐 九、网络

> 💡 **TL;DR**
>
> 本仓库的网络模块提供了一套极简的 socket 抽象层，核心数据结构是 `struct socket`，它封装了协议族、套接字类型、状态机（UNBOUND→BOUND→…→CLOSED）以及本地/远程地址。当前仅实现了 `sock_bind` 函数，其余网络系统调用（socket/connect/sendto/recvfrom 等）均为空桩或未接入。与 xv6 相比，这套接口尝试引入 POSIX socket 风格，但未附带真正的 TCP/IP 协议栈，网络层收发函数被整体注释，属于早期骨架。

### 1. 核心抽象：socket 结构体与状态机
网络模块的基石是 `include/kernel/socket.h` 中定义的 `struct socket`<sup>[1](#mod-net-ref-1)</sup>。该结构体聚合了协议族（domain）、套接字类型（type，仅支持 SOCK_STREAM 和 SOCK_DGRAM）、协议号（protocol）、一个五元状态机（`enum socket_state`<sup>[2](#mod-net-ref-2)</sup>：UNBOUND→BOUND→LISTENING→CONNECTED→CLOSED）、本地/远端地址以及控制字段（backlog/rcv_timeout）。状态机是套接字生命周期的核心，`sock_bind` 函数严格检查状态必须为 UNBOUND 才允许绑定<sup>[3](#mod-net-ref-3)</sup>，绑定后迁移至 BOUND。这种设计隐含了后续 listen/connect 等操作对其他状态的依赖。

值得注意的是，`struct socket` 中没有包含文件描述符索引或持有自旋锁，这意味着 socket 对象的管理和并发保护完全依赖上层调用方——这与 xv6 的 `struct file` 包装方式不同，后者将 socket 作为文件的一种类型。此处 socket 是独立结构，可能通过全局表或进程 fd 表间接引用。

### 2. 关键设计取舍：简化实现与未完成的协议栈
当前实现只有 `sock_bind` 是真实代码，其余网络功能均为注释或缺失。`kernel/socket.c` 顶部注释掉了 `net_send_packet` 和 `net_recv_packet` 两个模拟网络层函数<sup>[4](#mod-net-ref-4)</sup>，它们用静态测试数据模拟收发，并打印日志。这表明开发者曾计划分层（socket 层→网络层），但网络层尚未接入真实协议栈。

`sock_bind` 内部有一个值得讨论的取舍：当 `addr->sin_port` 为 0 时，硬编码赋予默认端口 2000<sup>[5](#mod-net-ref-5)</sup>。这在玩具内核中可接受，但缺少端口冲突检测和分配算法，放弃了对端口复用的支持。此外，`memmove` 复制地址时未校验 `addrlen` 与 `sockaddr_in` 大小的一致性（虽然用 `(void)addrlen` 忽略了参数），可能埋下兼容性隐患。

模块暴露的常量组（`SOL_SOCKET`, `SO_*`, `SOCK_NONBLOCK` 等）表明它试图兼容 Linux socket API 风格，但实际实现中没有任何选项处理逻辑——`setsockopt` 系统调用虽在 syscall ID 列表中，却无对应内核处理函数。

### 3. 跨模块协同：与系统调用入口和文件管理的关系
网络系统调用（socket/bind/connect/sendto/recvfrom/accept/listen/getsockname/setsockopt）在 `include/kernel/syscall_ids.h` 中有定义，但内核并未在 `kernel/socket.c` 中提供完整实现。`sock_bind` 是唯一被引用的函数，它由对应的 `sys_bind` 系统调用处理程序调用。由于没有读取 syscall 分发代码，可以推测调用方在获取 `struct socket*` 时可能通过文件描述符表查找，或者依赖全局 socket 数组——无论哪种方式，当前缺少 `sock_alloc`/`sock_close` 等配套函数。

`struct socket` 中存在 `pending_conn` 和 `pending_remote_addr` 字段<sup>[1](#mod-net-ref-1)</sup>，显然是为 accept 准备的待处理连接队列，但 accept 未实现，这些字段成为死域。同样，`struct simple_packet` 和 `MAX_PACKETS` 宏<sup>[6](#mod-net-ref-6)</sup>定义了数据包缓冲结构，但从未被使用，暗示了原始设计中的本地环回队列方案被放弃。

### 4. 边角细节与不足
1. **指针安全与用户地址校验缺失**<sup>[7](#mod-net-ref-7)</sup>：`sock_bind` 直接解引用传入的 `struct sockaddr_in *addr` 指针，没有通过 `copyin` 从用户空间复制，也未检查 addr 是否为内核可访问地址。若此函数被系统调用路径直接调用，将导致内核可被用户程序传入任意地址读写。
2. **并发安全空白**<sup>[1](#mod-net-ref-1)</sup>：`struct socket` 没有锁字段，状态转换无原子保护。多核环境下两个线程同时对同一 socket 调用 bind 可能造成状态错乱。
3. **硬编码常量**<sup>[5](#mod-net-ref-5)</sup><sup>[6](#mod-net-ref-6)</sup>：默认端口 2000、`MAX_PACKET_SIZE` 1500、`MAX_PACKETS` 10 均为魔法数，未参数化或根据配置生成。
4. **资源泄漏与错误处理**<sup>[3](#mod-net-ref-3)</sup>：`sock_bind` 在状态错误时返回 -EINVAL，但上层若忽略该返回值继续使用 socket，可能导致后续操作在未绑定状态下异常。另外，`addr->sin_port` 被修改为 2000 后，调用方感知不到端口变更。
5. **死代码与注释残留**<sup>[4](#mod-net-ref-4)</sup>：`net_send_packet` 和 `net_recv_packet` 完全注释掉，但没有删除；`struct simple_packet` 及相关宏未与任何逻辑连接，属于工程腐烂信号。
6. **未实现的系统调用**<sup>[8](#mod-net-ref-8)</sup>：socket/connect/accept/sendto/recvfrom/listen/getsockname/setsockopt 均有 syscall ID，但内核无对应实现，调用将直接返回错误或 panic，用户程序不可用。

- 除 sock_bind 外所有网络系统调用（socket/connect/sendto/recvfrom/accept/listen/getsockname/setsockopt）均无内核实现，调用将返回错误或 panic。

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 驱动框架模块实现了 LoongArch 和 RISC-V 两个平台的 VirtIO 块设备驱动。核心抽象是静态全局结构体 `struct disk`，它封装了三个 virtqueue 环（描述符、可用、使用）以及请求跟踪数组 `info` 和 `ops`。LoongArch 版本通过 PCI 传输层（`pci.c` + `virtio_pci.c`）进行设备发现和 MMIO 映射，而 RISC-V 版本使用固定的 MMIO 寄存器地址。两种实现均遵循 VirtIO 块设备协议，但同步策略不同：LoongArch 采用轮询（忙等待）并持有自旋锁，RISC-V 采用中断驱动。相比 xv6-riscv 原版，LoongArch 版本引入了完整的 PCI 初始化和 BAR 分配过程，但代码成熟度较低，存在多处 TODO 和未检查的错误路径。

### 1. 核心抽象与架构分层

驱动框架的核心是 `struct disk`，它是一个页对齐的全局静态结构体，包含 4 页（LoongArch）或 2 页（RISC-V）的原始内存 `pages`，用于存放三个 virtqueue 环（描述符 `desc`、可用 `avail`、使用 `used`）。<sup>[1](#mod-drivers-ref-1)</sup> 此外，`disk` 还维护描述符空闲位图 `free[NUM]`、已消费索引 `used_idx`、每个请求的信息 `info[NUM]`（包括 `buf` 指针和状态字节）以及命令头部 `ops[NUM]`。LoongArch 版本额外包含 `struct spinlock vdisk_lock` 用于并发保护。<sup>[1](#mod-drivers-ref-1)</sup>

驱动分为两层：**传输层**和**VirtIO 协议层**。LoongArch 的传输层由 `pci.c`（PCI 配置空间访问、设备探测、MMIO 分配）和 `virtio_pci.c`（VirtIO PCI 能力扫描、MMIO 寄存器读写）实现。<sup>[5](#mod-drivers-ref-5)</sup><sup>[6](#mod-drivers-ref-6)</sup> RISC-V 的传输层则通过 `virt.h` 中定义的 MMIO 寄存器宏（如 `*R(VIRTIO_MMIO_STATUS)`）直接操作。<sup>[4](#mod-drivers-ref-4)</sup> 协议层在 `virtio_disk.c`（LoongArch）和 `virt.c`（RISC-V）中实现，完成描述符链构建、请求提交和完成等待。

### 2. 关键设计取舍

**同步策略**：LoongArch 的 `la_virtio_disk_rw` 在提交请求后执行 `while (disk.used_idx == disk.used->id) { }` 忙等待，直到使用环索引改变。<sup>[3](#mod-drivers-ref-3)</sup> 该循环无超时保护，若设备故障或无响应将永久阻塞。RISC-V 的 `virtio_rw` 则通过 `while (b->disk == 1) { }` 等待中断处理函数 `virtio_disk_intr` 清除标志。<sup>[4](#mod-drivers-ref-4)</sup> 两种方式均未禁止中断，LoongArch 选择轮询可能是因为其中断子系统尚未完善，但也带来了 CPU 占用问题。

**锁保护**：LoongArch 在 `la_virtio_disk_rw` 全程持有 `disk.vdisk_lock`，但未关闭中断，若在中断上下文中调用同一函数或时钟中断触发重入，可能导致死锁。<sup>[3](#mod-drivers-ref-3)</sup> RISC-V 版本完全没有锁，假设单核或由外部保证串行化。

**PCI 初始化**：LoongArch 通过 `pci_device_probe` 遍历 255 个总线、32 个设备，匹配 vendor/device ID 0x1af4/0x1001，然后通过 `pci_device_init` 配置命令寄存器、解析 BAR 并分配 MMIO 空间。<sup>[5](#mod-drivers-ref-5)</sup> `virtio_pci_read_caps` 遍历能力链表（PCI 配置空间偏移 0x34），定位 common/notify/device/isr 四种能力类型，计算 MMIO 基址并填充 `virtio_pci_hw_t`。<sup>[6](#mod-drivers-ref-6)</sup> RISC-V 则直接检查固定 MMIO 地址的 Magic/Version/DeviceID/VendorID 寄存器，若不符仅打印消息而不终止初始化。<sup>[14](#mod-drivers-ref-14)</sup> 这种硬编码方式简单但无法应对平台变化。

**描述符管理**：两个平台均使用 `alloc_desc` 从 `free` 位图中分配描述符，每次 I/O 固定使用 3 个描述符（类型头、数据、状态）。`alloc3_desc` 在部分分配失败时会回滚已分配的描述符。<sup>[7](#mod-drivers-ref-7)</sup> 当没有空闲描述符时，`alloc_desc` 返回 -1，调用侧不做阻塞等待而直接重试，空耗 CPU。<sup>[13](#mod-drivers-ref-13)</sup>

### 3. 跨模块协同

驱动模块与以下模块紧密配合：
- **内存管理模块**：`pci_map` 调用外部函数 `walk_device` 和 `mappages_device`（来自 `vm.c`）将 PCI ECAM 和 MMIO 区域映射到内核页表，使用 `PTE_MAT | PTE_W | PTE_P | PTE_D` 属性。<sup>[9](#mod-drivers-ref-9)</sup> LoongArch 特有的 `PA2VA` 宏用于将物理地址转换为直接映射虚拟地址。<sup>[10](#mod-drivers-ref-10)</sup>
- **PCI 服务**：`pci.c` 提供 `pci_device_probe`、`pci_config_read/write` 系列函数以及 `pci_alloc_mmio`，被 `virtio_disk.c` 和 `virtio_pci.c` 广泛调用。<sup>[8](#mod-drivers-ref-8)</sup> `virtio_pci.c` 进一步封装了 VirtIO PCI 传输层操作，如 `virtio_pci_set_status`、`virtio_pci_get_device_features`。<sup>[6](#mod-drivers-ref-6)</sup>
- **中断子系统**：RISC-V 版本依赖 PLIC 和 `trap.c` 路由 VirtIO 中断，`virtio_disk_intr` 在中断上下文中应答中断并更新使用环。<sup>[4](#mod-drivers-ref-4)</sup> LoongArch 版本当前不使用中断，但代码中保留了 `// intr_off()` 的注释。<sup>[3](#mod-drivers-ref-3)</sup>

### 4. 边角细节与不足

- **<sup>[10](#mod-drivers-ref-10)</sup>** `PA2VA` 宏 `#define PA2VA(pa) ((pa) & (~(DMWIN_MASK)))` 简单丢弃高位地址，若传入非直接映射区的物理地址将产生错误虚拟地址，存在地址空间混淆风险。
- **<sup>[3](#mod-drivers-ref-3)</sup>** LoongArch 的 `while (disk.used_idx == disk.used->id)` 轮询无超时，且未在持锁期间关闭中断，若时钟中断在轮询中发生并尝试获取 `vdisk_lock` 将死锁。
- **<sup>[11](#mod-drivers-ref-11)</sup>** `pci_alloc_mmio` 使用静态变量 `s_off` 累加分配，无锁保护且可能溢出（64 位地址回绕），多核并发挥调用将导致竞态。
- **<sup>[12](#mod-drivers-ref-12)</sup>** `la_virtio_disk_init` 调用 `virtio_pci_read_caps` 后未检查返回值（-1），若能力扫描失败仍继续使用未初始化的 `gs_virtio_blk_hw`，可能访问无效 MMIO 地址。
- **<sup>[14](#mod-drivers-ref-14)</sup>** RISC-V 的 `virtio_disk_init` 中设备 ID 校验失败仅打印消息，不 panic 也不返回错误，后续操作将基于无效硬件状态进行，导致不可预测行为。
- **<sup>[13](#mod-drivers-ref-13)</sup>** `alloc_desc` 无空闲时返回 -1，`alloc3_desc` 直接返回失败，调用侧（`la_virtio_disk_rw` 和 `virtio_rw`）的死循环 `while(1) { if(alloc3_desc==0) break; }` 会持续占用 CPU，没有睡眠机制。

此外，代码中存在多处工程欠账：`// 之后应该放到virtio_pci.h` 表明头文件组织尚未完成；LoongArch 的 `pci_device_init` 硬编码了 6 次 BAR 循环；轮询路径中遗留了 `printf` 调试代码。这些不足暗示该驱动框架仍处于早期开发阶段。

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 系统调用层封装了用户态与内核的陷入接口，由 syscall.c 实现系统调用分派（依托 trapframe）和具体系统调用函数，包括 sys_openat、ioctl 等。核心抽象是系统调用分发器从 trapframe 提取 syscall 编号和参数，调用具体实现。与 xv6 原版相比，本实现扩展了大量 Linux 兼容的系统调用（如 openat 支持 AT_FDCWD、O_TMPFILE 等）、/proc 虚拟文件系统（适配 busybox）、只读挂载仿真、ext4 文件系统支持、进程 OOM 调整、终端 ioctl 等，并提供 futex、socket 等原语，显著增强了兼容性和功能集。

### 1. 核心抽象与外部依赖
系统调用层是用户态与内核交互的唯一同步途径。核心抽象是 `syscall(struct trapframe *trapframe)` 函数（声明于 `trap.h`），它从陷阱帧中读取系统调用编号和参数，并分派到 `syscalls[]` 数组中的具体处理函数。本模块依赖大量下层设施：陷阱处理（`trap.c`）、进程与内存管理（`process.h`, `vmem.h`）、文件系统（`vfs_ext4.h`）、用户态地址拷贝（`copyin`/`copyout`）以及锁机制（`cpu.h` 中的 `acquire`/`release`）。

syscall.c 通过 `syscall_ids.h` 定义系统调用编号（如 `SYS_openat`），并通过 `defs.h` 声明了所有 sys_* 函数原型。模块内部封装了许多辅助函数来处理虚拟 /proc 路径、只读挂载仿真、OOM 调整等，这些函数仅在系统调用实现内部使用，保持了接口的简洁。

跨模块协同的关键点：trap 处理中断后调用 `syscall`，后者需要操作进程的页表（`p->pagetable`）执行 `copyin`/`copyout`；文件操作最终调用 VFS 层（`vfs_ext4_*`），通过 `filesystem_t` 结构体与底层文件系统交互；全局的进程表 `proc pool` 和 `file` 表被系统调用直接访问（如 `myproc()`、`filealloc()`、`fdalloc()`）。<sup>[1](#mod-syscall-ref-1)</sup><sup>[6](#mod-syscall-ref-6)</sup>

### 2. 关键设计取舍
**虚拟 /proc 文件系统**：为兼容 busybox 等用户程序，系统调用层在用户空间路径与真实 ext4 文件系统之间插入了一层虚拟路径识别。`is_busybox_virtual_path()` <sup>[2](#mod-syscall-ref-2)</sup> 和 `fill_busybox_virtual_kstat()` <sup>[3](#mod-syscall-ref-3)</sup> 等函数拦截对 `/proc/self/stat`、`/proc/meminfo` 等路径的访问，直接在内核态构造 stat 结构或内容，而不经过磁盘。这种设计避免了在嵌入式环境中挂载真正的 procfs，但也带来维护负担（每增加一个虚拟路径需同步修改多个函数）。

**只读挂载仿真**：通过静态全局变量 `synthetic_ro_mount` 和 `synthetic_mount_readonly` <sup>[4](#mod-syscall-ref-4)</sup> 实现一个“伪只读挂载点”。当检测到该路径上的写操作时，系统调用直接返回 `-EROFS`。这是对 ext4 缺少真正挂载功能的权宜之计，放弃了对复杂挂载选项的支持，但足够通过部分测试。

**OOM 调整兼容性**：通过 `normalize_proc_oom_score_adj_path` 将用户请求的 `/proc/self/oom_score_adj` 转换为进程 PID 对应的路径，并依赖 busybox 虚拟路径函数处理。这模仿了 Linux 的 OOM 接口，但只是解析路径，并没有实际调整 OOM 分数（实际实现为桩）。这是一个明显的取舍：保证应用兼容但功能缺失。

**ioctl 终端实现**：`ioctl_is_tty()` 和 `ioctl_copyout_value()` <sup>[5](#mod-syscall-ref-5)</sup> 封装了终端判断和用户态拷贝，支持常见的窗口大小查询（`TIOCGWINSZ`）等操作，但实现仅限于少量 ioctl 命令，其他命令可能直接返回错误。

### 3. 跨模块协同
- **文件系统协同**：`sys_openat` 通过 `get_fs_from_path()` 获取文件的文件系统类型，若为 EXT4 或 VFAT 则调用 `vfs_ext4_*` 系列函数；还需调用 `vfs_ext4_flush()` 实现 sync 系统调用。`fsync_regular_ext4_file` 和 `ftruncate_regular_ext4_file` 则直接委托给 ext4 的 flock 操作。这种设计将文件系统细节封装在 VFS 层，系统调用仅协调路径解析、权限检查和 FD 分配。
- **内存与用户拷贝**：系统调用频繁使用 `copyin`/`copyout`，它们依赖当前进程的页表。`ioctl_copyout_value` 显式调用 `copyout`，并在失败时返回 `-EFAULT`。<sup>[1](#mod-syscall-ref-1)</sup> 展示了 `sys_openat` 中通过 `copyinstr` 从用户空间读取路径字符串。
- **进程调度与 FD 管理**：`sys_openat` 调用 `filealloc()`（返回 `struct file*`）和 `fdalloc()`（分配文件描述符索引），它们分别管理文件对象表中进程的文件描述符表。系统调用需持有 `p->lock` 或相关锁，但已读片段未显式展示（推测在 `filealloc` 和 `fdalloc` 内部加锁）。
