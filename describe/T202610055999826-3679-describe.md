## 📖 一、总览

该仓库是一个面向 RISC-V 和 LoongArch64 平台的实验性 Rust 内核，定位为从底层手写硬件交互的快速原型，而非通用教学或生产型系统。它完全避开了 rcore-rust 家族常见的 trait 抽象层，采用全局静态变量与编译期架构分支，在特定 QEMU virt 环境上实现多进程 Unix 环境。架构上，内核无启动引导，无虚拟文件系统，无统一驱动框架，任务调度以单一全局事件槽替代就绪队列，体现出“极简可跑”的直接性。值得评审注意的三个特点：其一，驱动模块手工实现 virtio-mmio 与 virtio-pci，从寄存器到 virtqueue 全自写，绕过了现成 crate，深度展示了硬件操作细节，但牺牲了可移植性；其二，文件系统采用嵌入式 ext4 解析，内置页面缓存机制能将预置数据锁在物理帧中，使测试程序无需真实磁盘即可运行，这在演示内核中颇为实用；其三，调度器的事件驱动模型通过静态 Option 传递调度事件，极简却高效，但单槽设计严重限制了并发扩展。

## 🎯 二、综合评价

### 整体定位

该内核可归类为“演示验证型”原型，以最小代码量在特定硬件模拟环境中跑通自备测试集为核心目标。它刻意偏离了 rCore-Tutorial 等教学核的分层抽象之路：驱动无 `BlockDevice` trait [drivers.ref:1]，文件系统无 VFS 层，系统调用无 `Syscall` trait 分发。这一激进简化使得架构极度扁平，代码直观，便于理解底层机制，但也导致各模块紧密耦合，难以复用或跨平台。内核针对每种架构独立编译，启动流程完全缺位（可能依赖外部固件），因此它更像是一份“硬件直达应用”的展示代码，而非可演进的 OS 骨架。

### 真正的创新或独特设计

尽管工程上粗糙，该内核仍闪现出若干有启发性的设计。内存管理模块用 `PageTable` trait 统一了 RISC-V Sv39 和 LoongArch 4-level 页表 [mm.ref:1]，并预留了 COW 引用计数基础设施（虽未接入），在跨架构内核中展现了抽象意愿。文件系统是其最大亮点：`ext4.rs` 直接解析 ext4 磁盘结构的同时，引入 `page_cache_preload` 机制，将预置的应用数据锁定在物理帧中 [fs.ref:3]，使得内核可“原地”运行 ELF 程序而无需块设备参与，这在测试或嵌入式场景中极为巧妙。任务模块的 `SCHEDULE_EVENT_SLOT` 用一个 `Option<ScheduleEvent>` 实现事件驱动调度 [task.ref:3]，代码量极小，却成功替代了传统调度队列，体现了“用 Rust 枚举做状态机”的简练风格。但这些创新通常停留在“功能可用”阶段，伴生大量未处理边界：COW 仍为 TODO，全局变量无保护，硬编码地址和掩码，限制了它们的实际健壮性。

### 取舍判断

编写者明显将精力倾注于两点：一是手写 virtio、页表 walk 等硬件直驱，二是让文件系统能直接加载执行测试程序。为此，大量子系统被刻意省略或简陋化：启动模块完全缺失，信号处理只见于系统调用号，IPC、网络为零，系统调用中约三分之一为 stub，甚至 `sys_nanosleep` 以忙等自旋实现。内存管理仅实现基础分页，物理帧回收栈满即丢弃，`brk` 只增不减，`mprotect` 路径潜在物理内存泄漏。这些取舍使得内核在不足万行的代码量内跑通了多进程测试，但一旦脱离该预定义场景，内存泄漏、并发缺失、硬编码假设等问题将立刻暴露。整体看，这是一个“快速走通全流程”的原型，而非目标通用化的 OS。

### 完成度与不足

核心三子系统（进程、内存、文件）在限定条件下可运转，69 个系统调用中 fs 域达 32 项，支持基本的 ELF 加载、fork/exec/exit 和文件读写，可运行内置脚本。然而，信号、IPC、网络完全空白，系统服务层严重残缺，大量 syscall 返回 0（如 `fcntl`），导致应用兼容性极低。代码质量方面，`static mut` 近乎滥用，无任何互斥机制；多处硬编码魔数（如 LoongArch 直接映射窗口掩码 0x9000_0000_0000_0000）；驱动层 virtio 双路径存在大量重复逻辑；轮询方式等待设备、忙等实现延时，均反映鲁棒性不足。综合来看，该内核在“演示特定功能”维度上达成目标，但工程严密性、安全性和可维护性均处于初级阶段，更适合用作硬件交互与内核机制的教学分析案例。

## 🧠 四、内存管理

> 💡 **TL;DR**
>
> 本模块以 `PageTable` trait 为核心抽象，在 RISC-V Sv39 与 LoongArch 4-level 页表之上构建 `MemorySet`（地址空间）与 `MapArea`（连续映射区域）两层模型。物理帧分配器采用带回收栈的栈式分配，内核堆使用侵入式空闲链表。模块同时提供 fork 全拷贝、brk/mmap/mprotect 用户态接口，以及 copy_to_user/copy_from_user 的权限检查版本。相比 rCore-Tutorial，本仓增加了 LoongArch64 支持、COW 引用计数基础设施（尚未接入）、以及更防御性的堆分配器完整性检查，但多处仍留有早期实现标记。

### 1. 核心抽象与分层

内存管理模块的核心抽象是 `PageTable` trait <sup>[1](#mod-mm-ref-1)</sup>，定义了 `map`/`unmap`/`translate`/`token` 四个原语。两个架构分别实现：`RiscvPageTable` 操作 Sv39 三级页表 <sup>[2](#mod-mm-ref-2)</sup>，`LoongArchPageTable` 操作 4-level 页表并通过 CSR 配置硬件页表 walker <sup>[3](#mod-mm-ref-3)</sup>。上层 `MemorySet` 封装一个 `ArchPageTable`（通过 `crate::arch::ArchPageTable` 类型别名在编译期选择）和一个 `MapArea` 列表 <sup>[4](#mod-mm-ref-4)</sup>，每个 `MapArea` 描述一段连续的虚拟地址范围，携带映射类型（`Identical` 内核等值映射 或 `Framed` 动态分配帧）、权限位和区域类型标签 <sup>[5](#mod-mm-ref-5)</sup>。

`MapArea` 的设计体现了"区域即分配单元"的思路：`map()` 遍历 `VPNRange` 逐页建立映射，`append_to()` 支持在区域末端扩展（用于堆增长），`copy_data_to()` 在 fork 时逐页拷贝物理页 <sup>[6](#mod-mm-ref-6)</sup>。区域类型 `MapAreaType` 枚举区分了内核各段、MMIO、用户栈/堆/mmap 等不同语义，这在调试输出和 fork 跳过内核区域时起到过滤作用。

### 2. 关键设计取舍

**帧分配器：栈式回收 vs 伙伴系统。** `StackFrameAllocator` 使用简单的线性扫描 + 固定大小回收栈（65536 项）<sup>[7](#mod-mm-ref-7)</sup>。选择栈式而非伙伴系统意味着：分配/释放 O(1)，但外部碎片无法控制，且回收栈满时直接丢弃帧（静默泄漏）<sup>[8](#mod-mm-ref-8)</sup>。此外，模块声明了 COW 引用计数数组 `FRAME_REFCOUNT`（每个物理帧一个 `u16` 计数）及 `frame_refcount_inc/dec/frame_is_shared` 接口 <sup>[9](#mod-mm-ref-9)</sup>，但 `fork_user_space()` 仍执行全量物理页拷贝，COW 机制完全未接入——`MapArea::copy_data_to` 对 `Framed` 类型只是空 match 分支后直接遍历拷贝。

**内核堆：侵入式空闲链表 + 分配头。** `HeapAllocator` 在每次分配时在用户指针前嵌入 `AllocHeader`（记录分配起始与大小），释放时读取头信息回收 <sup>[10](#mod-mm-ref-10)</sup>。空闲链表按地址排序，合并相邻块。值得注意的取舍：每次 alloc/dealloc 都调用 `check_free_list()` 遍历整个链表做完整性校验 <sup>[11](#mod-mm-ref-11)</sup>——这在调试期非常有价值（检测 double-free、链表循环、越界写），但会导致大堆下分配性能随空闲块数量线性下降。

**LoongArch 物理地址映射差异。** 与 RISC-V 的 VA=PA 等值映射不同，LoongArch64 内核通过 `phys_to_kvirt(pa) = pa | 0x9000_0000_0000_0000` 访问物理内存 <sup>[12](#mod-mm-ref-12)</sup>，这个偏移量在 `frame_allocator`、`memory_set`、`mm.rs` 三处重复定义而非集中到 arch crate。`LoongArchPageTable::map()` 在覆写用户页时执行 `invtlb` 全刷 <sup>[13](#mod-mm-ref-13)</sup>，而 `RiscvPageTable::map()` 使用 `sfence.vma` <sup>[14](#mod-mm-ref-14)</sup>——两者都未区分单页刷新与全刷，在频繁 mmap/mprotect 场景下 TLB 冲刷开销较大。

**`Locked<T>` 的 Sync 假实现。** 内核堆全局分配器 `HEAP_ALLOCATOR` 是 `static` 变量，需要 `Sync`。`Locked<T>` 用 `UnsafeCell` + `unsafe impl Sync` 绕过编译器检查 <sup>[15](#mod-mm-ref-15)</sup>，注释明确标注"当前阶段为单核 -smp 1；后续多核时这里要换成真正的锁"。这是典型的早期实现技术债。

### 3. 跨模块协同

**fork 全路径：MemorySet → page_table → frame_allocator。** `fork_user_space()` <sup>[16](#mod-mm-ref-16)</sup> 遍历 `self.areas`，跳过内核区域（`!area.perm().user`），对每个用户区域：调用 `clone_metadata()` 复制元数据（不包含 `data_frames`），在新页表中 `map()` 分配新帧，然后 `copy_data_to()` 逐页调用 `copy_physical_page()` ——后者通过 `phys_to_kvirt` 获取内核虚拟地址，以 `u64` 为单位执行 `read_volatile/write_volatile` 拷贝 <sup>[17](#mod-mm-ref-17)</sup>。整个过程持 `&self`（不可变引用），仅在创建的新 `MemorySet` 上做可变操作，避免了 fork 期间对原进程地址空间的锁竞争。

**brk 系统调用链路。** `sys_brk` <sup>[18](#mod-mm-ref-18)</sup> 通过 `task::with_current_memory_set` 获取当前进程的 `MemorySet` 可变引用，调用 `MemorySet::brk()`。`brk()` 做四重检查：new_brk 必须 >= heap_start、< USER_STACK_BOTTOM、不等于当前值、>= 当前值（不支持收缩）。通过后调用 `grow_user_heap()`，后者查找 `MapAreaType::UserHeap` 区域并调用 `append_to()` 扩展映射 <sup>[19](#mod-mm-ref-19)</sup>。初始堆大小为 256KB <sup>[20](#mod-mm-ref-20)</sup>，在 `init_user_heap()` 中预分配。

**mmap 的 malloc 兼容性设计。** `mmap_user()` <sup>[21](#mod-mm-ref-21)</sup> 在请求长度上额外加一页（`actual_len = len + 0x1000`），并将返回地址偏移 4KB（`start + 0x1000`）。这是因为 musl libc 的 `mallocng` 实现在 `(返回地址 - 16)` 处写入元数据——该设计保证元数据落在已映射页内。`mmap_fixed()` <sup>[22](#mod-mm-ref-22)</sup> 支持 `MAP_FIXED`：对范围内每个 VPN，如果已有映射则更新权限（用于 guard page 等场景），否则分配新帧。

**copy_to_user/copy_from_user 的安全版本。** 模块提供两个层次的数据传输：`copy_to_user` 按物理页分块拷贝，仅依赖 `translate_va` <sup>[23](#mod-mm-ref-23)</sup>；`copy_to_user_checked` 逐字节检查目标地址所在 `MapArea` 的权限标志（user + write）<sup>[24](#mod-mm-ref-24)</sup>。前者追求性能，后者追求安全性——调用方可根据场景选择。

### 4. 边角细节与不足

`<sup>[25](#mod-mm-ref-25)</sup>` **LoongArch 与 RISC-V 初始化路径不一致。** `MemorySet::new_kernel()` 对非 LoongArch 调用 `map_kernel_sections()` <sup>[26](#mod-mm-ref-26)</sup>，而 LoongArch 走 `new_bare()` 跳过内核段映射（依赖直接映射窗口 0x9000...）。但 `mm::init()` 中又显式写了 `#[cfg(feature = "arch_loongarch64")]` 分支 <sup>[27](#mod-mm-ref-27)</sup>，导致同一个条件判断在两个层级重复。如果未来新增架构（如 aarch64），需要在三处同步修改。

`<sup>[28](#mod-mm-ref-28)</sup>` **用户栈的 guard 区域未主动保护。** `USER_STACK_SIZE` 为 8MB，但 `USER_STACK_ALLOC` 仅 512KB <sup>[29](#mod-mm-ref-29)</sup>。剩余的 ~7.5MB 虚拟地址空间虽未映射，访问会触发 page fault，但模块没有显式创建不可访问的 guard page——完全依赖地址空间空洞被动防御。

`<sup>[30](#mod-mm-ref-30)</sup>` **页表覆写不回收旧帧。** `RiscvPageTable::map()` 和 `LoongArchPageTable::map()` 在 `perm.user` 且 PTE 已存在时直接覆写 PTE 而不释放旧物理帧 <sup>[31](#mod-mm-ref-31)</sup><sup>[13](#mod-mm-ref-13)</sup>。`mprotect` 和 `MAP_FIXED` 的现有映射覆写路径会导致物理内存泄漏。

`<sup>[32](#mod-mm-ref-32)</sup>` **帧分配器的静默泄漏与弱防御。** `StackFrameAllocator::dealloc()` 在回收栈满时直接 `return` 丢弃帧 <sup>[33](#mod-mm-ref-33)</sup>；double-free 检测仅打印日志后 `return`，不触发 panic <sup>[34](#mod-mm-ref-34)</sup>。在生产场景下，这两者会将内存错误转化为难以追踪的静默泄漏。

`<sup>[35](#mod-mm-ref-35)</sup>` **`unsafe` 块普遍缺少 SAFETY 注释。** 包括 `Locked::lock()` <sup>[36](#mod-mm-ref-36)</sup>、`copy_physical_page` 内的 volatile 指针操作 <sup>[17](#mod-mm-ref-17)</sup>、`activate_page_table` 的内联汇编 <sup>[37](#mod-mm-ref-37)</sup> 等关键 unsafe 块均无契约说明，增大了后续维护时的审计难度。

`<sup>[38](#mod-mm-ref-38)</sup>` **`check_free_list()` 在每次 alloc/dealloc 完整遍历链表。** 调试期有利，但未用 `#[cfg(debug_assertions)]` 或 feature gate 控制 <sup>[11](#mod-mm-ref-11)</sup>。堆碎片化严重时（空闲块数量多），分配路径的 O(n) 遍历叠加每次 O(n) 的完整性检查会显著拖慢性能。

`<sup>[39](#mod-mm-ref-39)</sup>` **`mmap_fixed` 的帧追踪不完整。** 该函数将新分配的帧推入第一个 `MapAreaType::Mmap` 区域的 `data_frames` <sup>[40](#mod-mm-ref-40)</sup>，但如果范围内不存在 Mmap 区域（例如对 UserHeap 区域使用 MAP_FIXED），帧不会被任何 `MapArea` 追踪。这些"孤儿帧"在 `MemorySet` 析构时不会被 `MapArea::drop()` 回收，造成泄漏。

`<sup>[41](#mod-mm-ref-41)</sup>` **COW 引用计数使用 `saturating_add`。** `frame_refcount_inc` <sup>[42](#mod-mm-ref-42)</sup> 在计数溢出时饱和而非 panic——这意味着如果引用计数意外达到 u16::MAX（65535），后续 fork 将无法正确追踪共享，导致提前释放或 use-after-free 隐患。虽然当前 COW 未启用，但基础设施的防御深度不足。

## 🧵 五、进程与任务调度

> 💡 **TL;DR**
>
> 核心抽象为 `TaskControlBlock`(TCB), 以 `TaskKind` 区分用户进程与内核线程, 调度器通过静态全局 `SCHEDULE_EVENT_SLOT` 实现事件驱动调度. 进程生命周期覆盖 fork/clone/exec/exit/wait4, 信号仅支持 pending 位掩码设置无实际投递. 与 rCore-Tutorial 基线相比, 本仓引入了内核线程抽象、基于静态槽位的事件调度器、以及 EXT4 磁盘 ELF 流式加载等独特扩展.

### 1. 核心抽象: 双态 TCB 与静态全局调度器

进程与任务模块的中心是 `TaskControlBlock`(TCB)<sup>[1](#mod-task-ref-1)</sup>, 它统一承载了用户进程 (`UserProcess`) 和内核线程 (`KernelThread`) 两种 `TaskKind`. 用户进程持有 `MemorySet`(页表/地址空间)和 TrapContext(用户态寄存器快照), 内核线程则仅依赖内核栈上的 `TaskContext` 参与调度<sup>[2](#mod-task-ref-2)</sup>. 这种双态模型避免了为内核线程单独设计一套调度基础设施, 但也意味着所有访问 `memory_set()` 的代码路径都必须预先检查 `kind()`.

调度器不采用传统的就绪队列+定时器轮询模式, 而是通过一个全局 `static mut SCHEDULE_EVENT_SLOT: Option<ScheduleEvent>`<sup>[3](#mod-task-ref-3)</sup> 实现

> 💡 **TL;DR**
>
> 本模块实现了一个无 VFS 抽象层的嵌入式 ext4 直读/写引擎, 核心为 1218 行的 `src/fs/ext4.rs`. 它直接从块设备解析 ext4 超级块、组描述符、inode、extent 树与目录项, 提供路径查找、文件读写、目录列表功能. 独特之处在于:(1) 内建 VFS 页面缓存 (`page_cache_preload`) 将预置应用数据锁定在物理帧中以绕过磁盘读取; (2) 写支持虽简陋 (硬编码块号与 inode 号), 但在同类教学 OS 中少见; (3) 深度耦合测试框架, 包含 `find_test_scripts` / `parse_run_all_tests` 等专用解析器.

### 1. 核心抽象：无 VFS 的嵌入式 ext4 解析器

与 ArceOS 的 `axfs` 或 rCore-Tutorial 的 VFS 多文件系统架构不同, 本模块完全省略了虚拟文件系统层 <sup>[1](#mod-fs-ref-1)</sup>. `src/fs/ext4.rs` 将 ext4 磁盘结构解析、路径遍历、文件读写、目录操作全部耦合在单一文件中. 所有公开接口直接暴露 ext4 语义 (如 `Ext4InodeInfo`, `Ext4DirEntryInfo`), 没有 `struct File` 或 `trait FileSystem` 抽象.

状态通过两个 `static mut` 全局变量持有: `EXT4_INFO: Option<Ext4SuperBlockInfo>` 缓存超级块信息 <sup>[2](#mod-fs-ref-2)</sup>, `PAGE_CACHE_ENTRIES: Vec<PageCacheEntry>` 缓存预加载页面 <sup>[3](#mod-fs-ref-3)</sup>. 这种设计在单核内核中避免了锁开销, 但丧失了并发安全性与多文件系统实例能力.

初始化入口 `init()` 顺序完成: 读取超级块 → 验证魔数 `0xEF53` → 检查块大小 (仅支持 1024/2048/4096) → 读取组描述符 → 验证根 inode 为目录 → 扫描根目录项 <sup>[4](#mod-fs-ref-4)</sup>. 任一阶段失败则跳过 ext4 挂载, 内核仍可运行 (降级为无持久化文件系统).

### 2. 关键设计取舍：页面缓存与硬编码写路径

**页面缓存 (VFS Page Cache).** `page_cache_preload(name, data)` 在启动时被调用, 将内置应用二进制数据逐页分配物理帧、通过 `phys_to_kvirt` 映射到内核虚拟地址、逐字节 `write_volatile` 拷贝 <sup>[5](#mod-fs-ref-5)</sup>. 运行时 `read_file_range` 先查缓存 (`page_cache_lookup`), 命中后直接构造 `&[u8]` 切片从物理页拷贝, 完全绕过 ext4 磁盘读取 <sup>[6](#mod-fs-ref-6)</sup>. 这一机制为测试框架中的高频文件访问 (如 `/musl/xxx_testcode.sh`) 提供了零堆分配、零磁盘 I/O 的快速路径. 代价是: 缓存仅对预加载文件生效, 不自动填充; 路径规范化使用 `trim_start_matches("./")` 是个粗糙的 hack <sup>[6](#mod-fs-ref-6)</sup>.

**硬编码写路径.** `create_file` 从 `NEXT_INO = 48` 开始分配 inode 号, `write_file` 将数据固定写入物理块 4000 <sup>[7](#mod-fs-ref-7)</sup><sup>[8](#mod-fs-ref-8)</sup>. 这些魔数显然针对特定测试镜像, 不具备通用性. 目录条目插入时采用线性扫描寻找空洞 (`if rec > existing_size + new_entry_size`), 无法处理目录块满的情况 (无限循环风险) <sup>[9](#mod-fs-ref-9)</sup>. 写路径的存在本身在同类教学 OS 中罕见, 但实现质量处于早期原型阶段.

**稀疏文件处理.** `read_file_range` 正确区分 extent 覆盖块与空洞 (unallocated block): 有 extent 时读磁盘扇区, 否则填零 <sup>[10](#mod-fs-ref-10)</sup>. 这对 ELF 二进制 BSS 段至关重要, 避免因读未分配块返回垃圾数据. 粒度精细到 512 字节扇区, 而非整个块.

**`..` 显式不支持.** `lookup_path` 遇到 `..` 直接返回 `ENOENT`, 而非实现父目录回溯 <sup>[11](#mod-fs-ref-11)</sup>. 这简化了路径解析但限制了目录遍历能力 (用户态 `cd ..` 失效).

### 3. 跨模块协同

**块设备驱动.** 所有磁盘 I/O 通过 `crate::drivers::block_read_bytes(offset, buf)` 和 `block_write_bytes` 进行 <sup>[12](#mod-fs-ref-12)</sup>. 这两个函数签名表明驱动层负责字节级寻址, ext4 层自行管理块到字节的转换 (`physical_block * block_size`). 读写未对齐至块大小的场景 (如读取 256 字节 inode) 直接调用字节接口.

**内存管理.** 页面缓存依赖 `crate::mm::frame_allocator::frame_alloc()` 分配物理帧, `crate::mm::memory_set::phys_to_kvirt(pa)` 将物理地址转换为内核可访问的虚拟地址 <sup>[5](#mod-fs-ref-5)</sup>. `read_file_range` 的缓存命中路径使用 `unsafe { core::slice::from_raw_parts(...) }` 直接从物理页构造引用, 隐含假设该物理页在内核地址空间中恒等映射 (或至少可读) <sup>[6](#mod-fs-ref-6)</sup>.

**系统调用层.** `src/fs/mod.rs` 仅做条件编译分发 <sup>[1](#mod-fs-ref-1)</sup>, 实际系统调用 (如 `openat`, `read`, `write`, `getdents64`) 的实现位于 `src/syscall/mod.rs` 并直接调用 ext4 的公开函数. 这种直接耦合意味着更换文件系统需同时修改 syscall 层与 fs 层.

**测试扫描器.** `test_scanner` 模块 (未读到源码) 与 `ext4::find_test_scripts` / `find_basic_test_programs` 协同, 后者从 `/musl` 与 `/glibc` 目录收集以 `_testcode.sh` 结尾的脚本 <sup>[13](#mod-fs-ref-13)</sup>, 并解析 `run-all.sh` 中的 `tests="..."` 列表 <sup>[14](#mod-fs-ref-14)</sup>. 这暴露了该内核的主要用途: 批量运行 libc 兼容性测试.

### 4. 边角细节与不足

**<sup>[15](#mod-fs-ref-15)</sup> `static mut` 无同步.** `EXT4_INFO` 与 `PAGE_CACHE_ENTRIES` 均为 `static mut`, 所有访问包裹在 `unsafe` 块中. 单核环境下功能正确, 但若未来引入多核或中断抢占, 将产生数据竞争. 无 `Mutex`/`RwLock` 包装.

**<sup>[16](#mod-fs-ref-16)</sup> `unsafe` 块缺 SAFETY 注释.** 全文件共约 8 处 `unsafe` 块 (全局变量访问、裸指针解引用、`NEXT_INO` 自增), 无一处附带 `// SAFETY:` 说明前置条件. 这对代码审查和后续维护构成技术债.

**<sup>[17](#mod-fs-ref-17)</sup> `write_file` 仅支持 ≤4096 字节.** `write_file` 分配 `block_buf.resize(4096, 0u8)` 并写入单一块 4000, 数据超过 4KB 将静默截断. 且每次写入覆盖同一物理块, 多文件写入互相破坏.

**<sup>[18](#mod-fs-ref-18)</sup> 深度不支持.** `parse_extent_header` 后检查 `header.depth != 0` 即返回 `EINVAL`, 意味着仅支持叶子 extent 节点. 多层 extent 树 (大文件、碎片化文件系统) 无法读取. 这限制了可挂载的 ext4 镜像类型.

**<sup>[19](#mod-fs-ref-19)</sup> `create_file` 无重复名称检查.** 直接在目录块中追加条目, 不检查同名文件是否存在. 同时 `NEXT_INO` 单调递增, 不查询 inode 位图, 可能与已有 inode 冲突.

**<sup>[20](#mod-fs-ref-20)</sup> 路径规范化不一致.** `page_cache_lookup` 对路径做 `trim_start_matches('/').trim_start_matches("./")` 处理, 而 `lookup_path` 不做此类规范化, 可能导致缓存命中失败 (例如 `/musl/test` 与 `musl/test` 被视为不同路径但缓存键不一致).

**<sup>[21](#mod-fs-ref-21)</sup> `scan_root_dir` 在 `init` 期间无条件执行.** 即使后续不需要, 初始化时也会遍历根目录所有条目并打印日志, 对大型目录可能显著拖慢启动.

## 🔌 十、驱动框架

> 💡 **TL;DR**
>
> 本模块并非传统意义上的驱动框架——没有统一的 Device/Driver trait 抽象, 也没有总线枚举层或设备树. 它本质是一个按架构 feature gate 平铺的驱动集合: RISC-V 走 virtio-mmio 传统传输, LoongArch 走 virtio-pci 现代传输, 外加一个平台相关的 NS16550A 串口. 核心机制是对 virtio-blk 的 minimal 实现 (单队列, 8 个 descriptor, 轮询 used ring), 对上暴露 `block_read_bytes`/`block_write_bytes` 字节级接口. 与 rCore-Tutorial 的 `virtio-drivers` crate 相比, 本仓完全绕过了现成 virtio 库, 从头手写寄存器级交互, 牺牲可移植性换取极简依赖.

### 1. 无抽象的架构门控

本模块没有 `trait BlockDevice` 或 `trait Driver`, 而是在 `mod.rs` 中用 `#[cfg(feature = "arch_riscv64")]` 和 `#[cfg(feature = "arch_loongarch64")]` 做编译期二选一 <sup>[1](#mod-drivers-ref-1)</sup>. 这种设计意味着: (1) 不存在运行时设备发现与驱动绑定——内核镜像在编译时已确定唯一的块设备后端; (2) 如果未来增加新架构 (如 aarch64), 必须在每个 `cfg` 分支处追加代码, 否则函数体将没有 return 值而导致编译失败; (3) 串口驱动采用了不同的策略: `#[cfg(feature = "arch_loongarch64")]` 用一个地址, `#[cfg(not(...))]` 用另一个, 这意味着 LoongArch 之外的所有平台 (包括 RISC-V 和未预期的 mips) 都被静默覆盖到同一个 QEMU virt 串口地址 <sup>[9](#mod-drivers-ref-9)</sup>. 这是一个隐式的"默认分支即 RISC-V QEMU"假设.

### 2. virtio 双传输路径的代码复制

`virtio_mmio.rs` (570 行) 与 `virtio_pci.rs` (384 行) 共享约 70% 的逻辑: virtqueue 结构体定义、描述符链构建、`read_block`/`write_block` 的主循环几乎逐字相同 <sup>[6](#mod-drivers-ref-6)</sup>. 真正的差异仅在于: (a) 传输层——MMIO 用 `read32`/`write32` 访问固定基址, PCI 用 ECAM 扫描 capability 链后获得 common/notify/ISR 三组 BAR <sup>[5](#mod-drivers-ref-5)</sup>; (b) 地址转换——`virtio_mmio` 直接将 `static mut` 变量的虚拟地址填入 descriptor (依赖 QEMU virt 的恒等映射), 而 `virtio_pci` 通过 `ptr_pa()` 显式转换为物理地址 <sup>[2](#mod-drivers-ref-2)</sup>. 这意味着该仓放弃了复用 virtqueue 操作逻辑的机会——一个 `struct VirtQueue<Transport: VirtioTransport>` 抽象即可消除重复, 但作者选择了两份独立实现, 代价是任何 virtio 协议修改 (如增加 QUEUE_SIZE) 必须同步两处.

此外, PCI 枚举硬编码了 bus=0, device 0..31 <sup>[5](#mod-drivers-ref-5)</sup>, 且 capability 扫描只循环 32 次 (`for _ in 0..32`). 这对单设备 QEMU 环境足够, 但在真实 LoongArch 硬件上可能遗漏多功能设备或挂在非零 bus 上的 virtio 设备.

### 3. 全局可变状态与 unsafe 的默许

两个 virtio 驱动均将全部设备状态放在 `static mut` 全局变量中: `QUEUE`, `REQ`, `DATA_BUF`, `BATCH_BUF`, `STATUS`, `LAST_USED_IDX`/`LAST_USED` <sup>[8](#mod-drivers-ref-8)</sup>. 这种模式在单核、单线程、无中断的内核启动阶段可以工作, 但存在三个风险: (a) 没有任何并发保护——如果将来引入中断处理或 SMP, 对 `QUEUE.avail.idx` 的竞争写入会直接导致数据损坏; (b) 所有 `unsafe` 块均未携带 `// SAFETY:` 注释, 读者无法判断作者是否意识到 `static mut` 的别名分析风险; (c) `BATCH_BUF` 是 4096 字节的静态数组, 作为 "批量 DMA 缓冲区" 复用——每次 `read_block`/`write_block` 调用都会覆写, 如果未来出现重入调用 (如中断中触发块 I/O), 数据会被静默破坏.

### 4. 轮询等待与中断确认的割裂

两个 virtio 驱动都在提交 descriptor 后进入 `while Q.used.idx == LAST_USED { core::hint::spin_loop(); }` 的忙等循环 <sup>[8](#mod-drivers-ref-8)</sup>. 这意味着每次 I/O 都会独占 CPU 直到设备完成. 有趣的是, 循环退出后代码又去读了 `MMIO_INTERRUPT_STATUS` 并回写 ACK (仅 MMIO 版本, PCI 版本只读了一次 ISR 在 init 时). 这说明硬件确实会产生中断, 但内核选择不启用中断驱动路径——可能是因为该内核尚未实现中断子系统, 或者 trap handler 尚不能分发设备中断. 这种"确认中断但不使用"的模式说明驱动处于中间态: 寄存器交互已完备, 只差中断处理框架的接入.

### 5. 跨模块协同与边界陷阱

`mod.rs` 暴露的公共接口是 `block_read_bytes(offset, buf)` 和 `block_write_bytes(offset, buf)` <sup>[7](#mod-drivers-ref-7)</sup>. 这两个函数负责将任意字节偏移的 I/O 请求拆解为 512 字节对齐的扇区操作. 读路径相对安全 (有 `checked_add` 防溢出, 分批读入 `sec_buf` 后 `copy_from_slice`). 但写路径存在一个典型的早期实现陷阱: 当写入跨扇区边界时, 它执行 **read-modify-write**——先 `read_block` 读出整个扇区, 修改局部, 再 `write_block` 写回 <sup>[7](#mod-drivers-ref-7)</sup>. 此操作没有任何锁保护, 如果文件系统层或未来多任务环境并发写入同一扇区, 后完成的 RMW 会覆盖先完成者的写入. 这是块设备驱动常见的丢写 bug 来源.

串口驱动通过 `_print` 函数接入内核日志宏 <sup>[9](#mod-drivers-ref-9)</sup>. 它在 LoongArch 上有一个额外的 LSR (Line Status Register) 轮询 (`while (read(UART0+5) & (1<<5)) == 0`), 等待发送 FIFO 就绪, 而 RISC-V 分支没有这个等待——这说明 RISC-V 的 QEMU virt NS16550A 可能接受无条件写入 (或依赖更底层的 SBI 输出), 两种行为不对称, 可能在某些慢速模拟器上导致丢字符.

## ⚙️ 十一、系统调用层

> 💡 **TL;DR**
>
> 该模块基于 match 手动路由实现约 60 个 Linux 兼容系统调用，以 UserFile 结构体统一管理文件描述符，通过 static mut 全局变量模拟文件系统状态。核心取舍是放弃 trait 抽象以换取实现简洁性，代价是并发安全缺失和硬编码限制。特色包括 EXT4 延迟加载和匿名管道全局数组。

### 1. 系统调用分发机制：基于 match 的手动路由

内核采用单一入口函数 `syscall(id, args)`，内部通过 `match id` 直接跳转到对应处理函数 <sup>[2](#mod-syscall-ref-2)</sup>。这种设计与 Linux 内核早期的系统调用表类似，但放弃了宏生成的表或基于 trait 对象的动态分发（如 rCore 的 `Syscall` trait）。优点是无虚函数开销且易于跟踪，缺点是对每个新系统调用都需要手动添加分支，且无法复用通用逻辑。

系统调用号定义为一组 `usize` 常量，直接对应 RISC-V Linux 的标准号，并额外保留了 `SYS_FORK` (4000) 用于兼容旧测试 <sup>[2](#mod-syscall-ref-2)</sup>。调用约定遵循标准：从 `a7` 寄存器获取 id，`a0-a5` 作为 6 个参数，由 trap handler 在上层提取 <sup>[3](#mod-syscall-ref-3)</sup>。

### 2. 文件描述符管理：全局状态与 UserFile 抽象

与主流微内核或 ArceOS 将文件系统状态封装在独立对象中不同，本模块使用多个 `static mut` 全局变量管理运行时状态：`CREATED_DIRS` 记录动态创建的目录，`CREATED_FILES` 存储写入过的文件数据，`DELETED_PATHS` 标记被删除的路径，`PIPE_BUFFERS` 作为全局管道缓冲区池，`MOUNT_POINTS` 记录挂载点 <sup>[1](#mod-syscall-ref-1)</sup>。这些全局变量无任何同步原语保护，在多任务环境下存在数据竞争风险，是典型的早期原型特征。

文件描述符本身由 `UserFile` 结构体表示，每个任务通过 `alloc_user_file` 和 `close_user_file` 管理自己的描述符表。`UserFile` 包含路径、数据缓冲区、偏移量、类型标志（目录、管道、stdout、ext4 延迟加载等），而不是通过 trait 多态区分不同文件类型 <sup>[4](#mod-syscall-ref-4)</sup>。这种“一个结构体包含所有可能字段”的设计虽然简单，但每个实例携带冗余字段，且扩展新文件类型需要增加更多标志。

`sys_openat` 体现了路径解析的复杂性：它需要依次检查是否为 `/dev/null`、是否为已删除路径、是否为目录、是否为 EXT4 文件（并设置 `on_disk` 标志实现延迟加载）、是否为虚拟文件、是否为内置应用 <sup>[6](#mod-syscall-ref-6)</sup>。这种线性检查流程难以组合，且路径缓冲区硬编码为 128 字节 <sup>[6](#mod-syscall-ref-6)</sup>，路径过长时无法处理。

### 3. 跨模块协同：trap 入口、内存管理和文件系统

系统调用层的三个关键外部依赖：
- **trap 处理**：`rust_trap_handler` 在 `scause` 为 `USER_ENV_CALL` 时，从当前 `TrapContext` 提取 id 和参数，调用 `crate::syscall::syscall` 后写回返回值到 `x[10]` (a0)，并递增 `sepc` 跳过 ecall 指令 <sup>[3](#mod-syscall-ref-3)</sup>。对于 `SYS_YIELD` 和 `SYS_EXECVE` 两个特例，handler 需直接操作上下文：`SYS_YIELD` 可能不返回，因此必须先修正 `sepc` 和 `a0` 防字段覆盖 <sup>[3](#mod-syscall-ref-3)</sup>。
- **内存管理**：系统调用实现大量使用 `crate::task::with_current_memory_set` 和 `copy_to_user_checked`/`copy_from_user` 安全访问用户空间，每次访问都校验指针并返回 `EFAULT` <sup>[4](#mod-syscall-ref-4)</sup>。写操作如 `sys_write` 采用 256 字节分块循环，避免一次性分配大内核缓冲区，但引入了多次用户态拷贝的开销 <sup>[4](#mod-syscall-ref-4)</sup>。
- **文件系统与驱动**：`sys_read` 在延迟加载场景下调用 `read_file_from_rootfs` 读取 EXT4 数据；`sys_write` 通过 `crate::drivers::serial::Serial::putchar` 输出到串口 <sup>[4](#mod-syscall-ref-4)</sup>。模块没有统一的 VFS 层，而是通过条件判断直接调用底层函数，导致耦合紧密。

### 4. 边角细节与不足

1. **并发缺失**：5 个 `static mut` 全局变量 <sup>[1](#mod-syscall-ref-1)</sup> 和 `PIPE_BUFFERS` 的 `unsafe` 访问 <sup>[4](#mod-syscall-ref-4)</sup> 均无并发保护，多核环境下会产生竞态。管道读写在无数据时通过忙等 `loop` + `yield_current` 实现阻塞，不够高效。
2. **缓冲区硬编码**：读写采用 256 字节固定块 <sup>[4](#mod-syscall-ref-4)</sup>，路径解析 128 字节 <sup>[6](#mod-syscall-ref-6)</sup>，`stat` 结构 128 字节均硬编码，缺乏适应性，修改需要全局搜索。
3. **错误处理简化**：许多系统调用 stub（如 `SYS_FCNTL`、`SYS_SIGACTION`）直接返回 0 <sup>[2](#mod-syscall-ref-2)</sup>，`SYS_FACCESSAT` 永远成功，这虽然绕过了应用启动检查，但破坏了语义完整性。`sys_unlinkat` 无论是否真实删除都返回 0。
4. **性能陷阱**：`sys_nanosleep` 使用 `spin_loop` 忙等直到目标时间 <sup>[5](#mod-syscall-ref-5)</sup>，在单线程环境中占用 CPU 100%，未利用 SBI 的定时器中断休眠。
5. **安全性注释缺失**：多处 `unsafe` 块无 `// SAFETY:` 解释，如 `PIPE_BUFFERS.get_mut` <sup>[4](#mod-syscall-ref-4)</sup>，增加了维护风险。
